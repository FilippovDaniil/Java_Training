## 📚 О репозитории

Это **учебный** репозиторий: не библиотека и не приложение, а **тренажёр**. В каждой теме — объяснение и набор задач-каркасов с пометками `// TODO`. **Готовых решений нет** — код пишет студент. Прогресс идёт по нарастающей: от `System.out.println` до микросервисов, от пузырьковой сортировки до Ахо-Корасик, от одного класса до гексагональной архитектуры с DDD.

Два последних раздела — **системный анализ** и **system design** — стоят особняком: они **не про написание кода**. Системный анализ отвечает на вопрос «**что** строить» (требования, моделирование, данные, архитектура, интеграции, поставка), а system design — «**как** построить это на масштабе под нагрузкой» (количественная оценка, строительные блоки, распределённые системы, референсные архитектуры, подготовка к design-интервью). Вместо кода — артефакты в `.md` (диаграммы C4/BPMN/UML, расчёты, ADR, спецификации, SQL, OpenAPI), вместо компиляции — проверка по **критериям приёмки** и валидаторам.

Все пять разделов построены по одной переносимой методике — [`LEARNING-METHODOLOGY.md`](LEARNING-METHODOLOGY.md):

> **Знание дробится на маленькие темы; каждая тема = короткая теория + набор задач с нарастающей сложностью; задачи даются без готовых решений; каждый результат изолирован, дописывается по частям и объективно проверяется.**

## 🧩 Анатомия задачи (кодовые разделы)

Каждая задача — это **условие + пустой каркас**, который дописывает студент. Условие лежит в JavaDoc-шапке точки входа:

```java
/**
 * Задача NN — Тема NN: <название>
 *
 * ЗАДАНИЕ:        что нужно сделать (1–3 предложения).
 * ПРИМЕР / ВЫВОД: как выглядит правильный результат.
 * ТРЕБОВАНИЯ:     ограничения и обязательные условия.
 * ПОДСКАЗКА:      направление, но НЕ ответ.
 */
public class TaskNN {
    public static void main(String[] args) {
        // TODO: ваш код
    }
}
```

**Две оси нарастающей сложности:** внутри темы (задачи `01 → 07`, где 7-я — мини-проект) и между темами (от азов к продвинутому уровню).

### Структура темы

```
<раздел>/
+-- README.md                  # карта раздела
+-- module-NN-topic/           # одна папка на тему
    +-- theory.md              # объяснение + схемы + «подводные камни» + «что дальше»
    +-- practice/
        +-- Task01.java        # задача из одного класса
        +-- ...
        +-- Task07/            # задача с несколькими типами → отдельная папка
            +-- Task07.java    #   точка входа + условие в JavaDoc
            +-- ...            #   вспомогательные типы — каждый своим файлом
```

> **Конвенция:** один top-level класс — файл `TaskNN.java`; несколько типов — папка `TaskNN/`. Каждый файл объявляет пакет по своему модулю: `mNN_…/practice/TaskNN.java` → `package mNN_….practice;` (папки-модули — валидные идентификаторы пакетов, поэтому работает нативный ▶ в IDE).

> 🟧🟥 **Системный анализ и System Design — другой формат.** Кода нет: задача — это `.md`-файл (`ЗАДАНИЕ → ВХОДНЫЕ ДАННЫЕ → ТРЕБОВАНИЯ → ФОРМАТ РЕЗУЛЬТАТА → КРИТЕРИИ ПРИЁМКИ → ПОДСКАЗКА`); решение пишется текстом, таблицами, диаграммами (C4/BPMN/UML/PlantUML), расчётами, ADR, SQL и OpenAPI. Проверка — не компиляция, а **критерии приёмки** и валидаторы (например, Swagger Editor для OpenAPI). Структура темы та же: `module-NN-topic/` → `theory.md` + `practice/Task01..Task10.md` + `questions.md` (25 вопросов в формате собеседования middle+/senior + ключ эталонных ответов).

## ▶️ Как запускать

Каждая тема лежит в папке-модуле `mNN_…` (валидный пакет), поэтому задачи запускаются **нативно из IntelliJ IDEA**:

1. Откройте `<курс>/mNN_…/practice/TaskNN.java`.
2. Нажмите **зелёную ▶** слева от `main` → **Run 'TaskNN.main()'**. Вывод — в консоли Run.

Тест-классы (JUnit, без `main`) и темы со Spring / Hibernate / Testcontainers запускаются так же из IDEA либо через **Gradle** (`./gradlew test`).

**Без IDE** — скрипт `run.ps1` (single-file launch + classpath Spring/JUnit):

```powershell
.\run.ps1 m05_loops Task01                              # по имени модуля и задаче
.\run.ps1 java-course\m05_loops\practice\Task01.java    # или по полному пути
```

> ⚙️ **Окружение:** **JDK 21 LTS** — важно: на JDK 25 IntelliJ 2025.2 краснит стандартную библиотеку, поэтому в IDE поставьте **Gradle JVM = 21** и **Project SDK = 21** (проект уже пинит toolchain 21). IntelliJ IDEA (Community достаточно) · Docker — для модулей контейнеризации `java-course`.

> 🟧 **Системный анализ ничего не запускает:** задачи решаются текстом и диаграммами прямо в `.md`, проверка — по критериям приёмки (нотации BPMN/UML, SQL, OpenAPI в [Swagger Editor](https://editor.swagger.io)). JDK/IDE для этого раздела не нужны.

## 🚀 Как учиться

1. **Идите сверху вниз** — темы выстроены по нарастанию сложности.
2. **Сначала теория** (`theory.md`): разберите примеры, схемы и блок «подводные камни».
3. **Потом практика** (`Task01 → Task07`): пишите сами, подглядывая в подсказку, а не в ответ.
4. **Ведите сквозные проекты** — к финалу из мелких задач собирается реальная система.
5. **Проверяйте объективно** — компиляция, запуск, тесты, ожидаемый вывод в шапке задачи.

## 📐 Принципы методики

| # | Принцип | Зачем |
|---|---------|-------|
| 1 | Атомарная единица = одна тема | Пройти за один присест, не утонув |
| 2 | Теория → практика в каждой теме | Сразу закрепляем прочитанное |
| 3 | Фиксированный набор задач (5–7) | Предсказуемый ритм, рост сложности |
| 4 | Задача без готового решения | Учится тот, кто пишет сам |
| 5 | Последняя задача — мини-проект | Синтез темы в одно целое |
| 6 | Изоляция и самодостаточность | Любую задачу можно сделать/проверить отдельно |
| 7 | Сквозные проекты | Изолированные темы собираются в реальную систему |
| 8 | Объективная проверяемость | Компиляция/запуск/тест дают честное «да/нет» |

Полное описание и чек-лист переноса методики на другой язык/предмет — в [`LEARNING-METHODOLOGY.md`](LEARNING-METHODOLOGY.md).
