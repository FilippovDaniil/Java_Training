# Тема 09. Быстрая сортировка (Quick Sort) и её вариации

Quick sort — тоже «разделяй и властвуй» и тоже O(n log n) в среднем, но в отличие от merge sort работает **in-place** (без доп. массива) и на практике обычно **быстрее**. Цена: O(n²) в худшем случае и **нестабильность**. Идея: выбрать **опорный элемент** (pivot), переставить массив так, чтобы слева были меньшие, справа — большие, и рекурсивно отсортировать обе части.

```
Разбиение вокруг pivot = 5:
[3 7 2 8 5 1 6 4]  → переставляем →  [3 2 1 4 | 5 | 7 8 6]
                                      меньшие  pivot  большие
Теперь pivot на своём финальном месте; рекурсивно сортируем левую и правую части.
```

---

## 1. Разбиение Ломуто (Lomuto)

Простое в реализации. Pivot — последний элемент. Указатель `i` отмечает границу «меньших».

```java
int partition(int[] a, int lo, int hi) {
    int pivot = a[hi], i = lo;
    for (int j = lo; j < hi; j++)
        if (a[j] < pivot) swap(a, i++, j);
    swap(a, i, hi);          // pivot на своё место
    return i;                // индекс pivot
}
```

---

## 2. Разбиение Хоара (Hoare)

Два указателя идут навстречу. Делает в среднем **меньше обменов**, чем Ломуто, и лучше ведёт себя на массивах с дубликатами. Возвращает точку разделения (не финальную позицию pivot).

```
lo→                    ←hi
[3 7 2 8 5 1 6 4]   pivot = a[lo] = 3 (или средний)
i ищет слева элемент >= pivot, j справа <= pivot, меняем, пока не пересеклись.
```

---

## 3. Сложность и выбор pivot

| Случай | Сложность | Когда |
|--------|-----------|-------|
| Средний | O(n log n) | случайные данные, хороший pivot |
| Худший | **O(n²)** | pivot всегда минимум/максимум (напр. уже отсортированный массив + pivot=последний) |

**Выбор pivot решает всё:**
- *первый/последний* — просто, но O(n²) на отсортированных данных;
- *медиана из трёх* (first, mid, last) — устойчивее;
- *рандомизированный* — случайный pivot делает худший случай маловероятным независимо от входа.

> **Глубина рекурсии** в худшем случае O(n) → риск `StackOverflowError`. Рекурсию делают сначала на меньшей половине, либо ограничивают глубину.

---

## 4. Оптимизации

- **Insertion sort для малых подмассивов** (n < 10–16): на маленьких данных insertion быстрее из-за меньших накладных расходов. Классический гибрид (так устроена `Arrays.sort` для примитивов — dual-pivot quicksort).
- **Хвостовая рекурсия** в цикл: рекурсия только на одной половине.
- **3-way partition** (Dijkstra) для массивов с многими дубликатами: делим на «<pivot», «=pivot», «>pivot».

---

## 5. Quick Select — поиск k-го элемента за O(n) в среднем

Если нужен не весь отсортированный массив, а лишь **k-й по порядку** (например, медиана), не нужно сортировать полностью. Quick Select делает разбиение и рекурсивно идёт **только в ту половину**, где лежит k → O(n) в среднем.

```
ищем k-й: partition; если pivot-индекс == k — нашли;
если k < pivot-индекс — ищем слева, иначе справа. Одна ветвь вместо двух.
```

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Pivot=last на отсортированном | O(n²), `StackOverflowError` | рандомизация или медиана из трёх |
| Неверные границы в partition | бесконечная рекурсия / пропуск | внимательно с `<`, `<=`, возвратом индекса |
| Hoare: рекурсия по `[lo..p]` и `[p+1..hi]` | путаница границ (Hoare ≠ Lomuto!) | для Hoare: `qsort(lo,p)` и `qsort(p+1,hi)` |
| Нет порога на insertion | медленнее на малых n | переключение при n < ~10 |
| Сравнение `==` для поиска pivot-индекса в Quick Select | путаница | сравнивайте k с индексом pivot |
