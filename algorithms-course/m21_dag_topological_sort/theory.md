# Тема 21. Алгоритмы на DAG (направленный ациклический граф)

**DAG** (Directed Acyclic Graph) — ориентированный граф **без циклов**. Он моделирует зависимости: задачи проекта, сборку модулей, расписание курсов, формулы в таблице. Главная операция — **топологическая сортировка** (линейный порядок, согласованный со стрелками). На топопорядке легко считаются «самый длинный путь», «число путей», «критический путь» — потому что когда мы доходим до вершины, все её предшественники уже обработаны (это и есть динамическое программирование на графе).

```
Зависимости задач (стрелка = «должно идти раньше»):
   A → B → D          Топопорядок: A, C, B, D, E
   ↓        ↑         (каждая задача после всех своих зависимостей)
   C -------+ E
```

---

## 1. Топологическая сортировка — алгоритм Кана (BFS по in-degree)

1. Посчитать **in-degree** (число входящих рёбер) каждой вершины.
2. В очередь — все вершины с in-degree 0 (нет зависимостей).
3. Доставать вершину, добавлять в порядок, у соседей уменьшать in-degree; если стал 0 — в очередь.

```java
int[] indeg = new int[n];
for (each edge u->v) indeg[v]++;
Queue<Integer> q = вершины с indeg==0;
while (!q.isEmpty()) {
    int u = q.poll(); order.add(u);
    for (int v : adj.get(u)) if (--indeg[v] == 0) q.add(v);
}
// если order.size() < n -> в графе есть ЦИКЛ (топосортировки нет)
```

> Бонус Кана: если в итоге обработаны **не все** вершины — граф содержит **цикл** (это и проверка ацикличности).

Альтернатива — **DFS + стек** (post-order, развернуть), см. тему 18.

## 2. Самый длинный путь в DAG — DP на топопорядке

В произвольном графе longest path — NP-трудная задача, но **в DAG** решается за O(V+E): обрабатываем вершины в топопорядке и релаксируем «на максимум».

```java
// dist[v] = самый длинный путь, заканчивающийся в v
for (int u : topoOrder)
    for (Edge (u->v, w) : adj.get(u))
        dist[v] = max(dist[v], dist[u] + w);
```

Это та же **релаксация**, что в кратчайших путях (тема 19), но с `max` и в гарантированном порядке (топологическом), поэтому без кучи.

> Трюк: longest path = shortest path при инвертированных (отрицательных) весах. Но на топопорядке DP проще и нагляднее.

## 3. Количество путей между s и t в DAG

DP по топопорядку: `ways[s] = 1`; для каждой вершины в топопорядке прибавляем её число путей всем её преемникам.

```java
ways[s] = 1;
for (int u : topoOrder)
    for (int v : adj.get(u)) ways[v] += ways[u];
// ways[t] — число различных путей s -> t
```

## 4. Критический путь (project scheduling)

Задачи с длительностями и зависимостями. **Критический путь** — самая длинная по суммарной длительности цепочка зависимостей; она определяет минимальное время всего проекта. Считается как longest path в DAG, где вес вершины — длительность задачи.

```
earliest[v] = max по предшественникам (earliest[u] + dur[u])
длина проекта = max earliest[v] + dur[v]
```

---

## 5. Конденсация компонент сильной связности (обзор)

Произвольный орграф можно превратить в DAG: найти **компоненты сильной связности** (SCC — максимальные подграфы, где из любой вершины достижима любая другая; алгоритмы Косарайю/Тарьяна) и «схлопнуть» каждую в одну супервершину. Граф супервершин — всегда DAG (конденсация). Это позволяет применять DAG-алгоритмы к любому орграфу. Детали SCC — за рамками темы; здесь важна идея.

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Топосортировка графа с циклом | неполный/неверный порядок | Кан: проверить order.size()==n |
| Longest path в общем графе | задача NP-трудна | работает только на DAG (по топопорядку) |
| DP не в топопорядке | используются необработанные предшественники | строго в топологическом порядке |
| in-degree посчитан неверно | неверный старт Кана | indeg[v]++ для каждого ребра u->v |
| Несколько компонент/истоков | пропуск вершин | в очередь — ВСЕ вершины с indeg 0 |
