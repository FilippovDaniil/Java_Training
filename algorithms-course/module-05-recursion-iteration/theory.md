# Тема 05. Рекурсия и итерация

Рекурсия — это когда метод вызывает сам себя на «меньшей» версии задачи, пока не дойдёт до базового случая. Это не магия: каждый вызов кладёт **кадр на стек вызовов**, и когда базовый случай достигнут, кадры сворачиваются обратно. Понимание стека вызовов объясняет и красоту рекурсии, и её опасности (`StackOverflowError`, экспонента).

```
factorial(4)
 └ 4 * factorial(3)
        └ 3 * factorial(2)
               └ 2 * factorial(1)
                      └ 1            ← базовый случай
        свёртка: 1 → 2 → 6 → 24
Стек вызовов: 4 кадра в самой глубокой точке.
```

---

## 1. Два обязательных элемента рекурсии

1. **Базовый случай** — условие остановки (иначе бесконечная рекурсия → `StackOverflowError`).
2. **Шаг к базовому случаю** — каждый вызов должен приближать к базе (уменьшать n, делить пополам и т.п.).

```java
int factorial(int n) {
    if (n <= 1) return 1;          // базовый случай
    return n * factorial(n - 1);   // шаг: n -> n-1
}
```

---

## 2. Стек вызовов и память

Каждый незавершённый вызов хранит свои локальные переменные в кадре стека. Глубина рекурсии `d` → **O(d) памяти**, даже если работа в каждом кадре копеечная. JVM-стек ограничен (обычно ~10⁴–10⁵ кадров) → глубокая рекурсия падает.

> **Хвостовая рекурсия** — когда рекурсивный вызов является последней операцией. В некоторых языках компилятор превращает её в цикл (O(1) памяти). **JVM этого НЕ делает** — в Java хвостовая рекурсия всё равно растит стек. Поэтому в Java для глубоких задач предпочитают итерацию или явный стек.

---

## 3. Рекурсия vs итерация

| | Рекурсия | Итерация |
|---|----------|----------|
| Читаемость для «древовидных» задач | отличная (обходы, бэктрекинг) | громоздкая (свой стек) |
| Память | O(глубины) на стек вызовов | обычно O(1) |
| Риск | `StackOverflowError` | нет |
| Скорость | чуть медленнее (вызовы) | чуть быстрее |

Любую рекурсию **можно** переписать итеративно (со своим явным стеком). Вопрос в читаемости: для факториала — итерация очевиднее, для обхода дерева — рекурсия.

## 4. Опасность наивной рекурсии: Фибоначчи

```java
int fib(int n) {                       // O(2^n) — КАТАСТРОФА
    if (n < 2) return n;
    return fib(n-1) + fib(n-2);        // одни и те же значения считаются заново
}
```

Дерево вызовов ветвится надвое → экспонента. `fib(50)` наивно — минуты. Лечение — **мемоизация**: запоминать уже вычисленные значения (HashMap или массив) → O(n).

```
fib(5) наивно пересчитывает fib(2) пять раз, fib(3) трижды...
с мемоизацией каждое fib(k) считается ОДИН раз.
```

Это мостик к динамическому программированию (тема 22).

---

## 5. Бэктрекинг (перебор с возвратом)

Рекурсивный шаблон «попробовать вариант → углубиться → откатить»: генерация подмножеств, перестановок, комбинаций, решение судоку, расстановка ферзей.

```
subsets([1,2,3]): на каждом элементе — два выбора (взять / не взять)
             ┌─ взять 1 ─┐         ┌─ не взять 1 ─┐
           ... рекурсия на оставшихся [2,3] ...
2^n листьев — все подмножества.
```

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Нет базового случая | `StackOverflowError` сразу | добавить условие остановки |
| Шаг не приближает к базе | бесконечная рекурсия | убедиться, что аргумент «уменьшается» |
| Наивный Фибоначчи/подобное | экспоненциальное время | мемоизация / табуляция |
| Глубокая рекурсия в Java | `StackOverflowError` на больших n | итерация или явный стек |
| Бэктрекинг без отката | «грязное» состояние между ветвями | откатывать изменения после рекурсии |
