# Тема 14. Хеш-таблицы (HashMap)

Деревья (12–13) дают поиск за O(log n). Хеш-таблица идёт дальше — **O(1) в среднем**. Идея: вычислить из ключа число (**хеш**), превратить его в индекс «корзины» (bucket) массива и положить значение туда. Тогда поиск — это «посчитать индекс и заглянуть в корзину», без сравнений по дереву. Цена: нет порядка элементов, а в худшем случае (плохой хеш/много коллизий) — O(n).

```
ключ "cat" --hash--> 9843  --% 8--> индекс 3
                                       ▼
buckets:  0   1   2   3        4   5   6   7
              [ ] [ ] [cat→...][ ][ ][ ][ ]
                       └ корзина (цепочка) для индекса 3
```

---

## 1. Хеш-функция и индекс корзины

```java
int index = (hash(key) & 0x7fffffff) % buckets.length;   // неотрицательный индекс
```

Хорошая хеш-функция **равномерно** распределяет ключи по корзинам. В Java у каждого объекта есть `hashCode()`; `HashMap` дополнительно «перемешивает» биты, чтобы убрать закономерности.

Для строк — **полиномиальный хеш**: `h = s[0]·p^(n-1) + s[1]·p^(n-2) + ... + s[n-1]` (p — небольшое простое, напр. 31). Именно так устроен `String.hashCode()` (с p=31).

---

## 2. Коллизии: два способа разрешения

**Коллизия** — два разных ключа попали в одну корзину. Это неизбежно (корзин меньше, чем возможных ключей). Два классических решения:

### а) Цепочки (separate chaining)
Каждая корзина — это связный список (или дерево) пар. При коллизии добавляем в список.
```
buckets[3] -> ("cat",1) -> ("act",2) -> null
```

### б) Открытая адресация (open addressing)
Всё хранится в самом массиве. При коллизии ищем следующую свободную ячейку (линейное пробирование: index+1, index+2, ...).
```
hash("cat")=3 занято -> пробуем 4, 5, ... пока не найдём пусто
```

| | Цепочки | Открытая адресация |
|---|---------|--------------------|
| Память | списки (доп. ссылки) | компактнее (один массив) |
| Удаление | простое | хитрое («надгробия») |
| При высокой загрузке | деградирует плавно | резко (кластеризация) |
| Кэш | хуже (узлы разбросаны) | лучше (локальность) |

---

## 3. Коэффициент загрузки и рехеширование

**Load factor** = число элементов / число корзин. Когда он превышает порог (в Java по умолчанию **0.75**), таблица **рехешируется**: создаётся массив корзин **вдвое больше**, и все элементы перераспределяются. Это держит средние цепочки короткими → O(1).

```
size/capacity > 0.75  ->  capacity *= 2, перераспределить все элементы
```

Рехеширование — O(n), но происходит редко → амортизированно O(1) на вставку (как рост `ArrayList`, тема 02).

---

## 4. HashMap в Java (как устроен)

- Массив корзин (`Node[] table`), размер — степень двойки (индекс через `hash & (n-1)`).
- Цепочки — связные списки; **при длине цепочки ≥ 8** и размере таблицы ≥ 64 список превращается в **красно-чёрное дерево** (тема 13) → худший случай корзины O(log k) вместо O(k).
- `hashCode()` «домешивается»: `h ^ (h >>> 16)` — чтобы старшие биты влияли на индекс.

---

## 5. Сложности

| Операция | Средний | Худший |
|----------|---------|--------|
| put / get / remove | **O(1)** | O(n) (плохой хеш) / O(log n) (древовидные корзины) |
| перебор всех | O(n + capacity) | — |
| память | O(n + capacity) | — |

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Отрицательный индекс из hash | `ArrayIndexOutOfBounds` | `(h & 0x7fffffff) % n` или `h & (n-1)` |
| Переопределён `equals`, но не `hashCode` | элементы «теряются» в map | переопределяйте оба согласованно |
| Изменяемый ключ | объект «пропал» после мутации | ключи должны быть неизменяемыми |
| Нет рехеширования | длинные цепочки, O(n) | resize при load factor > порога |
| Плохая хеш-функция (все в одну корзину) | деградация до O(n) | равномерное распределение, домешивание битов |
