# Тема 25. Сжатие данных: Хаффман и LZW

Финальная тема. Сжатие убирает избыточность данных. **Хаффман** — оптимальное **префиксное кодирование**: частым символам — короткие коды, редким — длинные (жадный алгоритм на куче, тема 15). **LZW** — **словарное** сжатие: повторяющиеся последовательности заменяются короткими кодами из растущего словаря (применялось в GIF, `compress`). Обе техники соединяют идеи курса: жадность, кучи, деревья, хеш-таблицы.

```
Хаффман для "aaabbc" (частоты a:3, b:2, c:1):
        (6)               Коды (частому — короче):
       /   \                a -> 0
      a    (3)              b -> 10
     (3)   /  \             c -> 11
          c    b            "aaabbc" = 000 10 10 11 = 9 бит
         (1)  (2)           вместо 6*8 = 48 бит
```

---

## 1. Префиксный код и почему он нужен

**Префиксный код** — ни один код не является префиксом другого. Тогда поток битов декодируется однозначно, без разделителей. Дерево Хаффмана автоматически даёт префиксный код: символы — только в листьях, путь от корня (0 — влево, 1 — вправо) и есть код.

---

## 2. Алгоритм Хаффмана (жадный)

1. Посчитать **частоты** символов.
2. Создать по листу на символ; положить все в **min-heap** по частоте.
3. Пока в куче > 1 узла: извлечь два минимальных, объединить в новый узел (частота = сумма), вернуть в кучу.
4. Оставшийся узел — корень. Коды — пути от корня к листьям.

```java
PriorityQueue<Node> pq = листья по частоте;
while (pq.size() > 1) {
    Node a = pq.poll(), b = pq.poll();
    pq.add(new Node(a.freq + b.freq, a, b));   // внутренний узел
}
Node root = pq.poll();
```

**Кодирование:** заменить каждый символ его кодом. **Декодирование:** идти по дереву от корня битам потока; дошли до листа — выдать символ, вернуться в корень.

> Хаффман **оптимален** среди посимвольных префиксных кодов: даёт минимальную суммарную длину. Сложность построения — O(n log n) (куча).

---

## 3. Хранение таблицы кодов

Сжатый файл бесполезен без таблицы кодов (или дерева) для декодирования. В формат сохраняют либо **частоты** символов (по ним восстанавливают то же дерево), либо само дерево (обходом). Это «накладные расходы» сжатия — на маленьких данных они могут перевесить выгоду.

---

## 4. LZW — словарное сжатие

Не требует знать частоты заранее и не хранит дерево. Словарь строится **на ходу** и одинаково восстанавливается при распаковке.

**Сжатие:**
```
словарь = все одиночные символы (коды 0..255)
w = ""
для каждого символа c:
    если (w+c) в словаре: w = w+c
    иначе: вывести код(w); добавить (w+c) в словарь с новым кодом; w = c
вывести код(w)
```

**Распаковка** зеркально восстанавливает тот же словарь по кодам. Хорошо жмёт данные с повторами; словарь обычно ограничивают (сбрасывают при переполнении).

---

## 5. Коэффициент сжатия и пределы

**Коэффициент сжатия** = размер исходных / размер сжатых данных (или в процентах). Зависит от избыточности:
- текст на естественном языке — Хаффман даёт ~40–60% экономии;
- уже сжатые/случайные данные — почти не жмутся (предел — энтропия Шеннона);
- данные с длинными повторами — LZW особенно эффективен.

> Ни один алгоритм не сжимает **все** входы (принцип Дирихле): часть данных неизбежно несжимаема. Сжатие выигрывает на «типичных», избыточных данных.

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Код одного символа — префикс другого | неоднозначное декодирование | строить префиксный код (дерево Хаффмана) |
| Не сохранили таблицу/дерево | невозможно распаковать | хранить частоты или дерево в формате |
| Один уникальный символ во входе | дерево из одного узла, код пустой | спец-случай: код "0" для единственного символа |
| LZW: рассинхрон словарей | распаковка выдаёт мусор | словарь строить ИДЕНТИЧНО при сжатии и распаковке |
| Сравнение размеров без учёта таблицы | «сжатие» больше оригинала | учитывать накладные расходы (таблица/дерево) |
