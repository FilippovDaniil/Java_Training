Блок Spring Test закрыт ([модули 101–110](../m110_spring_test_security_async/theory.md)). Финальная часть курса — **контейнеризация**. Приложение, упакованное в Docker-образ, запускается одинаково на ноутбуке, в CI и на проде — «у меня работает» перестаёт быть отговоркой. ## Контейнер vs виртуальная машина

```
   ВИРТУАЛЬНАЯ МАШИНА                     КОНТЕЙНЕР (Docker)
   ┌───────────────────┐                 ┌───────────────────┐
   │  App A  │  App B   │                 │  App A  │  App B   │
   │  Bins/Libs         │                 │  Bins/Libs         │
   │  Гостевая ОС (полная)│               │  (общее ядро хоста) │
   ├───────────────────┤                 ├───────────────────┤
   │   Гипервизор       │                 │  Docker Engine     │
   │   Хостовая ОС      │                 │  Хостовая ОС       │
   └───────────────────┘                 └───────────────────┘
   тяжёлая (ГБ), старт минуты            лёгкий (МБ), старт секунды
   своё ядро на каждую VM               общее ядро ОС, изоляция процессов
```

Контейнер — это изолированный процесс с собственной файловой системой, но использующий **ядро хоста**. Отсюда лёгкость и скорость.

## Образ vs контейнер

```
   ОБРАЗ (image)            =  шаблон (read-only): слои файловой системы + метаданные
   КОНТЕЙНЕР (container)    =  запущенный экземпляр образа (+ writable-слой сверху)

   образ  ──docker run──►  контейнер 1
                       └─►  контейнер 2   (из одного образа — много контейнеров)
```

| Понятие | Аналогия в Java |
|---------|-----------------|
| образ (image) | класс |
| контейнер (container) | объект (экземпляр) |
| реестр (registry, Docker Hub) | Maven Central для образов |
| тег (tag) | версия (`postgres:16`, `nginx:latest`) |

## Базовые команды CLI

```bash
docker pull nginx:latest          # скачать образ из реестра (Docker Hub)
docker images                     # список локальных образов
docker run nginx                  # создать и запустить контейнер (foreground)
docker run -d --name web -p 8080:80 nginx   # detached + имя + проброс порта
docker ps                         # запущенные контейнеры
docker ps -a                      # все (включая остановленные)
docker logs web                   # логи контейнера
docker logs -f web                # логи в реальном времени (follow)
docker exec -it web sh            # выполнить команду внутри (интерактивная оболочка)
docker stop web                   # остановить
docker start web                  # запустить снова
docker rm web                     # удалить контейнер (сначала остановить)
docker rmi nginx                  # удалить образ
```

### Частые флаги `docker run`

| Флаг | Значение |
|------|----------|
| `-d` | detached — в фоне |
| `-p host:container` | проброс порта (`-p 8080:80` → localhost:8080 → контейнер:80) |
| `--name <имя>` | имя контейнера (иначе случайное) |
| `-e KEY=value` | переменная окружения |
| `-it` | интерактивный + TTY (для оболочки) |
| `--rm` | удалить контейнер после остановки |
| `-v host:container` | монтировать том (данные) |

## Проброс портов

```
   docker run -d -p 8080:80 nginx
                  │    │
                  │    └─ порт ВНУТРИ контейнера (nginx слушает 80)
                  └────── порт на ХОСТЕ (открываем localhost:8080)

   браузер → http://localhost:8080  →  [хост:8080]→[контейнер:80]  →  nginx
```

Без `-p` сервис внутри контейнера недоступен снаружи.

## Жизненный цикл контейнера

```
   docker run ──► [Running] ──docker stop──► [Stopped] ──docker start──► [Running]
                     │                          │
                  docker logs/exec          docker rm ──► [удалён]
```

> Остановленный контейнер не исчезает — он виден в `docker ps -a` и хранит writable-слой. Чтобы освободить место — `docker rm`. Образ удаляется отдельно (`docker rmi`).

## Запуск Java-приложения в контейнере (предпросмотр)

```bash
# официальный образ JRE, монтируем jar и запускаем:
docker run --rm -p 8080:8080 -v "$(pwd)/app.jar:/app.jar" eclipse-temurin:17-jre java -jar /app.jar
```

В [модуле 112](../m112_docker_dockerfile/theory.md) jar будет «запекаться» в собственный образ через `Dockerfile` — это правильный путь упаковки Spring Boot.

## Подводные камни

| Проблема | Причина | Решение |
|----------|---------|---------|
| Сервис недоступен на localhost | не пробросили порт | `-p host:container` |
| `docker rm` отказывает | контейнер запущен | сначала `docker stop`, потом `rm` (или `rm -f`) |
| Контейнеров много, место кончилось | остановленные не удалены | `docker ps -a`, `docker rm`; `docker system prune` |
| `docker run` каждый раз новый контейнер | `run` создаёт новый | повторно — `docker start <имя>` |
| Изменения в контейнере пропали | writable-слой удалён с контейнером | данные — в томах (`-v`), модуль [114](../m114_docker_configuration/theory.md) |
| `latest` тащит непредсказуемую версию | плавающий тег | фиксировать версию (`postgres:16`) |
| Нет доступа к Docker | демон не запущен | поднять Docker/Rancher Desktop |
