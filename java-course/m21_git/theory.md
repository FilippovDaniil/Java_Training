# Модуль 21. Основы Git

**Git** — распределённая система контроля версий. Она запоминает историю изменений проекта, позволяет откатываться назад, вести параллельные ветки разработки и работать в команде.

> Этот модуль — практический. Задачи (`practice/`) — это **упражнения для терминала**: каждый `TaskXX.java` содержит в комментарии задание, которое нужно выполнить командами Git. Сам Java-файл — лишь носитель задания (он компилируется с пустым `main`).

## Зачем нужен контроль версий

| Без Git | С Git |
|---------|-------|
| `project_final_v2_итог.zip` | одна папка, вся история внутри |
| страшно что-то менять | в любой момент откат назад |
| правки затирают чужие | слияние изменений нескольких людей |

## Три состояния файла

```
рабочая директория      индекс (staging)        репозиторий
   (Working Dir)            (Index)               (.git)
       │                       │                      │
       │   git add             │   git commit         │
   изменения  ──────────▶  подготовлено  ──────────▶  сохранено
```

| Состояние | Что значит |
|-----------|-----------|
| Modified | файл изменён, но не подготовлен |
| Staged | файл добавлен в индекс (`git add`) |
| Committed | изменения зафиксированы в истории (`git commit`) |

## Базовые команды

### Инициализация и настройка

```bash
git init                              # создать репозиторий в текущей папке
git config --global user.name "Имя"   # имя автора коммитов
git config --global user.email "x@y.z"
```

### Ежедневный цикл работы

```bash
git status                 # что изменилось
git add file.java          # добавить файл в индекс
git add .                  # добавить все изменения
git commit -m "Описание"   # зафиксировать изменения
git log                    # история коммитов
git log --oneline          # история кратко
```

### Просмотр изменений

```bash
git diff                   # незакоммиченные изменения
git diff --staged          # изменения в индексе
git show <hash>            # содержимое конкретного коммита
```

## Ветки (branches)

Ветка — независимая линия разработки. По умолчанию — `main` (или `master`).

```bash
git branch                  # список веток
git branch feature-login    # создать ветку
git checkout feature-login  # переключиться на ветку
git switch feature-login    # современный аналог checkout
git checkout -b feature-x   # создать и сразу переключиться
git merge feature-login     # влить ветку в текущую
git branch -d feature-login # удалить ветку
```

```
main:     A───B───C────────F   (merge)
                   \       /
feature:            D───E
```

> Ветки дёшевы: создавайте отдельную ветку под каждую задачу, затем сливайте в `main`.

## `.gitignore`

Файл со списком того, что Git должен **игнорировать** (не отслеживать):

```
# .gitignore для Java
*.class
target/
.idea/
*.iml
```

> Скомпилированные файлы, артефакты сборки и настройки IDE не коммитят.

---

## Отмена изменений

```bash
git restore file.java         # отменить незакоммиченные изменения файла
git restore --staged file.java  # убрать файл из индекса
git revert <hash>             # создать коммит, отменяющий другой коммит
git reset --hard <hash>       # ОПАСНО: вернуться к коммиту, теряя изменения
```

---

## Работа с удалённым репозиторием (обзор)

```bash
git clone <url>            # склонировать репозиторий
git remote -v              # список удалённых репозиториев
git push                   # отправить коммиты на сервер
git pull                   # забрать изменения с сервера
```

---

## Типичный рабочий процесс

```
1. git switch -c feature-task   # новая ветка под задачу
2. ... пишем код ...
3. git add .
4. git commit -m "Реализовал X"
5. git switch main
6. git merge feature-task       # вливаем результат
```

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| коммит без `git add` | новые/изменённые файлы не попадут в коммит |
| коммит `target/`, `.class` | замусоривают репозиторий; используйте `.gitignore` |
| `git reset --hard` без нужды | безвозвратно теряет изменения |
| невнятные сообщения коммитов | пишите осмысленно: что и зачем изменено |
| работа только в `main` | заводите ветки под задачи |
