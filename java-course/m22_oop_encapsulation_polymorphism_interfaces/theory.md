# Модуль 22. ООП: инкапсуляция, полиморфизм, интерфейсы

Углубляем ООП: разбираем интерфейсы, полиморфизм и связи между объектами (композиция/агрегация).

## Интерфейс — контракт поведения

**Интерфейс** описывает, *что* объект умеет делать, не указывая *как*. Это набор методов без реализации (по умолчанию). Класс «подписывает контракт» через `implements`.

```java
interface Flyable {
    void fly();              // абстрактный метод (без тела)
}

class Bird implements Flyable {
    public void fly() {       // обязаны реализовать
        System.out.println("Птица летит, махая крыльями");
    }
}

class Plane implements Flyable {
    public void fly() {
        System.out.println("Самолёт летит на двигателях");
    }
}
```

> Методы интерфейса неявно `public abstract`. Реализующий класс **обязан** их определить (иначе сам станет абстрактным).

---

## Реализация нескольких интерфейсов

Класс может наследоваться только от одного класса, но реализовывать **много интерфейсов**:

```java
interface Swimmable {
    void swim();
}

class Duck implements Flyable, Swimmable {   // и летает, и плавает
    public void fly()  { System.out.println("Утка летит"); }
    public void swim() { System.out.println("Утка плывёт"); }
}
```

```
        Flyable      Swimmable
           \           /
            \         /
              Duck (implements обоих)
```

## Полиморфизм

**Полиморфизм** — способность работать с объектами разных классов через общий тип (интерфейс или суперкласс). Какой именно метод вызовется — решается **во время выполнения** по реальному типу объекта (динамическая диспетчеризация).

```java
Flyable[] flyers = {
    new Bird(),
    new Plane(),
    new Duck()
};

for (Flyable f : flyers) {
    f.fly();   // вызовется fly() конкретного класса!
}
// Птица летит, махая крыльями
// Самолёт летит на двигателях
// Утка летит
```

> Переменная типа `Flyable` может хранить **любой** объект, реализующий `Flyable`. Это и есть полиморфизм.

### Upcasting и проверка типа

```java
Flyable f = new Duck();        // upcasting — объект как Flyable
f.fly();                        // ок
// f.swim();                    // ❌ через Flyable метод swim не виден

if (f instanceof Swimmable) {   // проверка реального типа
    Swimmable s = (Swimmable) f; // downcasting
    s.swim();
}

// Java 16+: pattern matching
if (f instanceof Swimmable s) {
    s.swim();
}
```

## Методы по умолчанию (default) в интерфейсе

С Java 8 интерфейс может иметь методы с реализацией — `default`:

```java
interface Greetable {
    String name();
    default void greet() {                    // готовая реализация
        System.out.println("Привет, " + name());
    }
}
```

Реализующий класс может использовать `greet()` как есть или переопределить.

> Интерфейсы также могут содержать `static`-методы и константы (`static final`).

## Композиция и агрегация

Это способы «иметь» другие объекты как части (отношение **HAS-A**, в отличие от наследования IS-A).

### Композиция — «часть целого», жёсткая связь

Часть не существует без целого. Объект создаёт свои части сам.

```java
class Engine {
    void start() { System.out.println("Двигатель запущен"); }
}

class Car {
    private final Engine engine = new Engine();   // Car ВЛАДЕЕТ Engine

    void start() {
        engine.start();                            // делегирование
        System.out.println("Машина поехала");
    }
}
```

```
Car ◆------▶ Engine     (композиция: Engine — часть Car, гибнет вместе с ним)
```

### Агрегация — «использует», слабая связь

Часть может существовать отдельно. Передаётся извне.

```java
class Player {
    String name;
    Player(String name) { this.name = name; }
}

class Team {
    private List<Player> players;       // игроки существуют независимо
    Team(List<Player> players) {        // получаем готовых извне
        this.players = players;
    }
}
```

```
Team ◇------▶ Player    (агрегация: игроки живут и без команды)
```

| | Композиция | Агрегация |
|--|-----------|-----------|
| Связь | сильная (часть-целое) | слабая (использование) |
| Жизненный цикл | часть гибнет с целым | часть живёт отдельно |
| Кто создаёт часть | само целое | приходит извне |

> **«Композиция предпочтительнее наследования»** — гибче и снижает связанность. Наследуйтесь только при истинном отношении IS-A.

## Интерфейс vs абстрактный класс (превью)

| | Интерфейс | Абстрактный класс |
|--|-----------|-------------------|
| Множественность | много (`implements A, B`) | один (`extends`) |
| Состояние (поля) | только константы | обычные поля |
| Методы | абстрактные + default/static | любые |

Подробно об абстрактных классах — в [модуле 23](../m23_oop_overloading_overriding_abstract/theory.md).

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| забыли `public` в реализации метода интерфейса | методы интерфейса `public`; нельзя сужать видимость |
| `ClassCastException` при downcasting | проверяйте `instanceof` перед приведением |
| создание объекта интерфейса | `new Flyable()` нельзя — нет реализации |
| наследование вместо композиции | если связь не IS-A, используйте композицию |
