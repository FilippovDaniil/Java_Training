# Модуль 29. Сборка мусора и типы ссылок

В Java память освобождается автоматически: **сборщик мусора (Garbage Collector, GC)** находит объекты, на которые больше нет ссылок, и удаляет их. Понимание GC и типов ссылок помогает писать программы без утечек памяти.

## Куча (heap) и стек (stack)

```
   СТЕК (stack)                    КУЧА (heap)
 ┌──────────────┐              ┌────────────────────┐
 │ локальные    │              │  ОБЪЕКТЫ            │
 │ переменные   │  ───ссылка──▶│  (new ...)         │
 │ примитивы    │              │  собираются GC      │
 └──────────────┘              └────────────────────┘
  своё у потока                 общая, управляется GC
```

- **Стек**: локальные переменные и примитивы, у каждого потока свой.
- **Куча**: все объекты (`new`). За ней следит GC.

---

## Как работает сборщик мусора

GC удаляет объекты, **недостижимые** из «корней» (GC roots: локальные переменные, статические поля и т.д.).

```
GC root ──▶ A ──▶ B          C ──▶ D
            (достижимы)      (недостижимы → будут собраны)
```

### Поколения объектов (generational GC)

Большинство объектов живут недолго. GC делит кучу на поколения:

```
   Young Generation              Old Generation
 ┌─────────────────────┐      ┌──────────────────┐
 │ Eden │ S0 │ S1       │ ───▶ │  долгоживущие    │
 └─────────────────────┘      └──────────────────┘
   новые объекты               пережившие много GC
   (Minor GC — часто)          (Major/Full GC — редко)
```

| Сборка | Что чистит | Частота |
|--------|-----------|---------|
| Minor GC | Young Generation | часто, быстро |
| Major/Full GC | Old Generation | редко, дольше |

> `System.gc()` — лишь **просьба** запустить GC, не гарантия. В реальном коде вызывать не нужно (только для учебных экспериментов).

## Четыре типа ссылок

Java различает «силу» ссылки — это влияет на то, когда GC соберёт объект. Классы в пакете `java.lang.ref`.

```
сила:  Strong  >  Soft  >  Weak  >  Phantom
       (никогда   (при     (при      (уже
        не соберёт нехватке  любом GC  собран,
        живой)     памяти)   собирается) только уведомление)
```

### 1. Strong (обычная) ссылка

```java
Object obj = new Object();   // strong reference
// пока obj указывает на объект — GC его не тронет
obj = null;                  // объект стал недостижим → может быть собран
```

### 2. SoftReference — «мягкая»

Объект живёт, **пока хватает памяти**. GC соберёт его только при нехватке памяти. Идеально для **кешей**.

```java
import java.lang.ref.SoftReference;

SoftReference<byte[]> cache = new SoftReference<>(new byte[1024 * 1024]);
byte[] data = cache.get();   // вернёт объект или null, если уже собран
```

### 3. WeakReference — «слабая»

Объект собирается при **ближайшем** GC, если на него нет strong-ссылок.

```java
import java.lang.ref.WeakReference;

WeakReference<String> weak = new WeakReference<>(new String("данные"));
System.out.println(weak.get());   // "данные"
System.gc();
System.out.println(weak.get());   // вероятно null — собран
```

### 4. PhantomReference — «фантомная»

`get()` всегда возвращает `null`. Используется для уведомления о том, что объект **уже собран** (через `ReferenceQueue`) — для тонкой очистки ресурсов.

```java
import java.lang.ref.*;

ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(new Object(), queue);
// после сборки ссылка попадёт в queue
```

## `WeakHashMap` — карта со слабыми ключами

Если на ключ больше нет strong-ссылок — запись **автоматически удаляется** из карты при GC. Полезно для кешей и метаданных.

```java
import java.util.WeakHashMap;

Map<Object, String> map = new WeakHashMap<>();
Object key = new Object();
map.put(key, "значение");
key = null;        // убрали strong-ссылку на ключ
System.gc();
// запись исчезнет из map автоматически
```

## Наблюдение за памятью

```java
Runtime rt = Runtime.getRuntime();
long total = rt.totalMemory();    // выделено JVM
long free = rt.freeMemory();      // свободно
long used = total - free;         // занято
System.out.println("Использовано: " + used / 1024 / 1024 + " МБ");
```

## Утечки памяти в Java

GC не спасает от **логических утечек** — когда ненужные объекты остаются достижимыми:

| Причина утечки | Решение |
|----------------|---------|
| объекты в `static`-коллекции, которые не убирают | удалять вовремя или `WeakHashMap` |
| незакрытые ресурсы (потоки, соединения) | try-with-resources (см. [модуль 17](../m17_exceptions/theory.md)) |
| слушатели, которые не отписали | хранить через `WeakReference` |

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| вера, что `System.gc()` точно соберёт мусор | это лишь рекомендация JVM |
| использование `finalize()` | устарел и ненадёжен; используйте `try-with-resources`/`Cleaner` |
| `SoftReference` как обычная ссылка | объект может исчезнуть при нехватке памяти |
| `weak.get()` без проверки на null | объект мог быть уже собран |
