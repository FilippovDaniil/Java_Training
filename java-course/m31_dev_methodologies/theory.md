# Модуль 31. Методологии разработки ПО

> ⚠️ Это **теоретический** модуль — практических задач нет.

**Методология разработки** — это подход к организации процесса создания ПО: как планировать, разрабатывать, тестировать и поставлять продукт.

## Жизненный цикл разработки (SDLC)

Большинство методологий проходят одни и те же этапы, отличается лишь их **порядок и повторяемость**:

```
Требования → Проектирование → Реализация → Тестирование → Внедрение → Сопровождение
```

---

## Waterfall (каскадная модель)

Последовательные этапы: каждый начинается после завершения предыдущего. Возврата назад почти нет.

```
Требования
    +-▶ Проектирование
            +-▶ Реализация
                    +-▶ Тестирование
                            +-▶ Внедрение
```

| Плюсы | Минусы |
|-------|--------|
| простота, чёткие этапы | негибкость к изменениям |
| хорошая документация | поздняя обратная связь |
| предсказуемость | ошибки требований всплывают в конце |

> Подходит для проектов с **стабильными, известными** требованиями (госзаказ, встроенные системы).

---

## V-Model (V-образная модель)

Развитие Waterfall: каждому этапу разработки соответствует этап тестирования.

```
Требования ---------------- Приёмочное тестирование
   Анализ ------------- Системное тестирование
     Проектирование -- Интеграционное тестирование
        Детальное -- Модульное тестирование
             \    Кодирование    /
```

---

## Итеративная и инкрементальная модели

Продукт создаётся **по частям** (инкрементам) или **уточняется по кругу** (итерациям), а не целиком за раз.

```
Итерация 1 → Итерация 2 → Итерация 3 → ...
(каждая = мини-цикл: план→код→тест→оценка)
```

---

## Agile (гибкая разработка)

**Agile** — это не методология, а набор **ценностей и принципов** (Agile Manifesto, 2001).

### Манифест Agile — 4 ценности

| Ценим больше | Чем |
|--------------|-----|
| Люди и взаимодействие | процессы и инструменты |
| Работающий продукт | исчерпывающую документацию |
| Сотрудничество с заказчиком | согласование контракта |
| Готовность к изменениям | следование плану |

> Agile реализуется конкретными фреймворками: **Scrum**, **Kanban**, **XP**.

---

## Scrum

Итеративный фреймворк: работа идёт **спринтами** (обычно 1–4 недели).

### Роли

| Роль | Ответственность |
|------|-----------------|
| **Product Owner** | приоритеты, backlog, ценность для бизнеса |
| **Scrum Master** | процесс, устранение препятствий |
| **Команда разработки** | создание инкремента продукта |

### Артефакты

- **Product Backlog** — общий список задач (что нужно сделать).
- **Sprint Backlog** — задачи на текущий спринт.
- **Increment** — рабочий результат спринта.

### Церемонии (события)

```
        +------------ СПРИНТ (1–4 недели) ------------+
Sprint  |  Daily   Daily   Daily ...                   |  Sprint    Sprint
Planning|  Standup Standup Standup                      |  Review  Retrospective
   --------------------------------------------------------------------------▶
```

| Событие | Зачем |
|---------|-------|
| Sprint Planning | планирование спринта |
| Daily Standup | синхронизация (15 мин/день) |
| Sprint Review | демонстрация результата |
| Retrospective | улучшение процесса |

---

## Kanban

Визуализация потока работ на доске; ограничение задач «в работе» (WIP limit). Нет фиксированных спринтов — непрерывный поток.

```
+----------+--------------+--------------+----------+
|  To Do   | In Progress  |   Review     |   Done   |
|          |  (WIP ≤ 3)   |              |          |
+----------+--------------+--------------+----------+
| задача 4 | задача 2     | задача 1     | задача 0 |
| задача 5 | задача 3     |              |          |
+----------+--------------+--------------+----------+
```

| Scrum | Kanban |
|-------|--------|
| фиксированные спринты | непрерывный поток |
| роли определены | роли необязательны |
| планирование на спринт | по мере поступления |

---

## XP (Extreme Programming)

Инженерные практики качества кода:

- **Парное программирование** (двое за одним кодом).
- **TDD** — тесты до кода (см. концепцию в [модуле 28](../m28_junit/theory.md)).
- **Непрерывная интеграция** (CI).
- **Рефакторинг**, простота дизайна, частые релизы.

---

## DevOps

Культура и практики, объединяющие **разработку (Dev)** и **эксплуатацию (Ops)** для быстрой и надёжной поставки.

```
   +--------------- бесконечный цикл ---------------+
   Plan → Code → Build → Test → Release → Deploy → Operate → Monitor
   +--------------------------------------------------------------+
                  CI/CD автоматизирует этот цикл
```

Ключевые практики: CI/CD, инфраструктура как код (IaC), мониторинг, автоматизация.

---

## Сравнение подходов

| Критерий | Waterfall | Scrum | Kanban |
|----------|-----------|-------|--------|
| Гибкость к изменениям | низкая | высокая | высокая |
| Обратная связь | в конце | каждый спринт | непрерывно |
| Подходит для | стабильных требований | развивающихся продуктов | поддержки/потока задач |
| Документация | подробная | минимально достаточная | по необходимости |

---

## Глоссарий ключевых терминов

| Термин | Значение |
|--------|----------|
| **Backlog** | приоритизированный список задач |
| **Sprint** | фиксированный временной отрезок работы |
| **User Story** | требование «как пользователь, я хочу…» |
| **Velocity** | объём работ команды за спринт |
| **Definition of Done** | критерии завершённости задачи |
| **MVP** | минимально жизнеспособный продукт |
| **Stand-up** | короткая ежедневная синхронизация |
| **Burndown chart** | график оставшейся работы |
