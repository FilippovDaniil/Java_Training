# Модуль 37. Архитектура ПО

> ⚠️ Это **теоретический** модуль — практических задач нет.

**Архитектура ПО** — это набор фундаментальных решений о структуре системы: из каких частей она состоит, как они взаимодействуют и какими качествами обладают. Хорошая архитектура облегчает развитие, поддержку и масштабирование.

## Клиент-серверная модель

Базовая модель распределённых систем: **клиент** запрашивает, **сервер** обслуживает.

```
   ┌──────────┐   запрос    ┌──────────┐
   │  КЛИЕНТ  │ ──────────▶ │  СЕРВЕР  │
   │ (браузер,│ ◀────────── │ (логика, │
   │  приложение)  ответ    │  данные) │
   └──────────┘             └──────────┘
```

| Сторона | Роль |
|---------|------|
| Клиент | интерфейс, запросы |
| Сервер | бизнес-логика, хранение данных |

## Многоуровневая (трёхуровневая) архитектура

Система делится на горизонтальные слои с чёткими обязанностями. Каждый слой общается только с соседними.

```
 ┌─────────────────────────────────┐
 │ Presentation (представление)     │  UI, контроллеры
 ├─────────────────────────────────┤
 │ Business Logic (бизнес-логика)   │  сервисы, правила
 ├─────────────────────────────────┤
 │ Data Access (доступ к данным)    │  репозитории, БД
 └─────────────────────────────────┘
```

| Слой | Ответственность |
|------|-----------------|
| Presentation | взаимодействие с пользователем |
| Business Logic | бизнес-правила, обработка |
| Data Access | работа с хранилищем (БД) |

> Преимущество: изменения в одном слое слабо затрагивают другие. Это разделение мы реализуем в Spring (контроллер → сервис → репозиторий).

## MVC (Model-View-Controller)

Паттерн разделения представления, данных и управления:

```
   Пользователь
        │ действие
        ▼
   ┌─────────────┐  обновляет  ┌─────────┐
   │ Controller  │ ──────────▶ │  Model  │  (данные + логика)
   └─────────────┘             └─────────┘
        │ выбирает                  │
        ▼                           │ данные
   ┌─────────────┐ ◀────────────────┘
   │    View     │  (отображение)
   └─────────────┘
```

| Компонент | Роль |
|-----------|------|
| Model | данные и бизнес-логика |
| View | отображение данных пользователю |
| Controller | обработка запросов, связь View и Model |

## Монолит vs Микросервисы

```
   МОНОЛИТ                        МИКРОСЕРВИСЫ
 ┌────────────────┐         ┌──────┐ ┌──────┐ ┌──────┐
 │ UI             │         │Заказы│ │Оплата│ │Склад │
 │ Логика         │         └──┬───┘ └──┬───┘ └──┬───┘
 │ Данные         │            └────────┼────────┘
 │ (одно прилож.) │              сеть (API/события)
 └────────────────┘         (отдельные сервисы + БД)
```

| Критерий | Монолит | Микросервисы |
|----------|---------|--------------|
| Сложность старта | низкая | высокая |
| Развёртывание | целиком | независимо по сервисам |
| Масштабирование | всё сразу | точечно по сервисам |
| Отказоустойчивость | один сбой = всё | сбой изолирован |
| Накладные расходы | малы | сеть, согласованность данных |
| Когда | старт, малые/средние команды | крупные системы, много команд |

> **Совет:** начинайте с монолита («Monolith First»). Переходите к микросервисам, когда монолит реально мешает масштабированию команды/нагрузки.

## Другие архитектурные стили

| Стиль | Суть |
|-------|------|
| **Hexagonal (порты и адаптеры)** | ядро бизнес-логики изолировано от внешнего мира через интерфейсы |
| **Event-Driven** | компоненты общаются через события (очереди, брокеры — Kafka) |
| **Layered** | строгие горизонтальные слои |
| **CQRS** | разделение операций чтения и записи |
| **Serverless** | функции в облаке без управления серверами |

## Критерии качества архитектуры (атрибуты)

| Атрибут | Вопрос |
|---------|--------|
| **Масштабируемость** | выдержит ли рост нагрузки? |
| **Доступность** | какой процент времени система работает? |
| **Производительность** | насколько быстры отклики? |
| **Надёжность** | как ведёт себя при сбоях? |
| **Сопровождаемость** | легко ли менять и развивать? |
| **Безопасность** | защищены ли данные? |
| **Тестируемость** | легко ли покрыть тестами? |

> Архитектура — это всегда **компромиссы** (trade-offs). Нельзя максимизировать всё сразу; выбирайте важное для конкретной задачи.

---

## Связанность и связность

| Понятие | Хотим | Значение |
|---------|-------|----------|
| **Coupling (связанность)** | низкую | насколько компоненты зависят друг от друга |
| **Cohesion (связность)** | высокую | насколько элементы внутри компонента связаны одной задачей |

> Цель: **низкая связанность, высокая связность**. Это делает систему гибкой и понятной.

## Глоссарий

| Термин | Значение |
|--------|----------|
| Компонент | самостоятельная часть системы |
| API | контракт взаимодействия между частями |
| Слой (layer) | горизонтальный уровень ответственности |
| Trade-off | компромисс между качествами |
| Scalability | способность расти под нагрузкой |
| Latency | задержка отклика |
| SPOF | единая точка отказа (single point of failure) |
