# Модуль 42. Веб-сервисы

> ⚠️ **Только теория.** Практика будет добавлена позже.

**Веб-сервис** — это программный компонент, доступный по сети и предоставляющий функциональность другим программам (а не людям) через стандартные протоколы. Так системы общаются друг с другом.

## Зачем нужны веб-сервисы

```
   Приложение A ──запрос──▶ Веб-сервис ──ответ──▶ Приложение A
   (например,              (на другом
    мобильное)              сервере/в облаке)
```

| Без веб-сервисов | С веб-сервисами |
|------------------|-----------------|
| системы изолированы | интеграция через сеть |
| дублирование функций | переиспользование сервисов |
| жёсткая связанность | слабая связанность через API |

---

## REST — современный стандарт

**REST** (Representational State Transfer) — архитектурный стиль поверх HTTP. Ресурсы адресуются URL, действия — HTTP-методами (см. [модуль 38](../m38_http_protocol/theory.md)).

```
GET    /users        → список пользователей
GET    /users/1      → один пользователь
POST   /users        → создать
PUT    /users/1      → заменить
PATCH  /users/1      → частично обновить
DELETE /users/1      → удалить
```

Данные обычно в **JSON**:

```json
{ "id": 1, "name": "Иван", "email": "ivan@example.com" }
```

| Принцип REST | Значение |
|--------------|----------|
| Ресурсы | сущности адресуются URL |
| Stateless | сервер не хранит состояние клиента |
| HTTP-методы | действия через GET/POST/PUT/DELETE |
| Представления | JSON, XML |

---

## SOAP — классический стандарт

**SOAP** (Simple Object Access Protocol) — протокол на основе XML, с жёстким контрактом (WSDL). Тяжелее REST, но строже.

```xml
<soap:Envelope>
    <soap:Body>
        <getUser><id>1</id></getUser>
    </soap:Body>
</soap:Envelope>
```

| | REST | SOAP |
|--|------|------|
| Формат | JSON (обычно) | только XML |
| Протокол | HTTP | HTTP, SMTP и др. |
| Контракт | OpenAPI (необязателен) | WSDL (строгий) |
| Вес | лёгкий | тяжёлый |
| Применение | веб, мобильные API | enterprise, банки |

> Сегодня для большинства задач выбирают **REST + JSON**. SOAP — в legacy и строго регулируемых сферах.

## JSON и XML

```
JSON                              XML
{                                 <user>
  "name": "Иван",                   <name>Иван</name>
  "age": 30                         <age>30</age>
}                                 </user>
```

| | JSON | XML |
|--|------|-----|
| Читаемость | выше | ниже (многословный) |
| Размер | меньше | больше |
| Схемы | JSON Schema | XSD |
| Где | веб, REST | SOAP, конфиги, документы |

## Обмен сообщениями: JMS и брокеры

Помимо синхронных вызовов (запрос-ответ), системы общаются **асинхронно** через очереди сообщений.

**JMS** (Java Message Service) — стандарт обмена сообщениями в Java.

```
Отправитель ──▶ [ Очередь / Топик ] ──▶ Получатель
   (Producer)      (брокер: Kafka,        (Consumer)
                    RabbitMQ, ActiveMQ)
```

| Модель | Суть |
|--------|------|
| **Queue** (точка-точка) | одно сообщение — одному получателю |
| **Topic** (публикация-подписка) | сообщение — всем подписчикам |

> Асинхронные сообщения повышают устойчивость: получатель может быть временно недоступен, сообщение подождёт в очереди.

## Облачные сервисы (обзор)

Веб-сервисы часто разворачивают в облаке:

| Модель | Что предоставляет | Пример |
|--------|-------------------|--------|
| **IaaS** | виртуальные машины, сеть | AWS EC2 |
| **PaaS** | платформа для приложений | Heroku, App Engine |
| **SaaS** | готовое ПО | Gmail, Slack |
| **FaaS / Serverless** | функции по вызову | AWS Lambda |

## API и контракты

| Понятие | Значение |
|---------|----------|
| **API** | контракт взаимодействия |
| **OpenAPI/Swagger** | стандарт описания REST API |
| **WSDL** | описание SOAP-сервиса |
| **Endpoint** | конкретный адрес сервиса |
| **Idempotency** | повтор запроса не меняет результат |
