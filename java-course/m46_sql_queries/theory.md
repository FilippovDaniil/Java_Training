# Модуль 46. Выборка данных: JOIN, GROUP BY, индексы

Реальные данные хранятся в нескольких связанных таблицах. Чтобы получать из них осмысленную информацию, нужны **объединения (JOIN)**, **агрегация (GROUP BY)** и **индексы** для скорости.

> Практика — SQL-скрипты (H2/любая СУБД).

## Агрегатные функции

Считают значение по группе строк:

```sql
SELECT COUNT(*)    FROM orders;            -- количество строк
SELECT SUM(amount) FROM orders;            -- сумма
SELECT AVG(amount) FROM orders;            -- среднее
SELECT MIN(amount), MAX(amount) FROM orders; -- мин/макс
```

| Функция | Что считает |
|---------|-------------|
| `COUNT(*)` | число строк |
| `COUNT(col)` | число НЕ-NULL значений |
| `SUM(col)` | сумма |
| `AVG(col)` | среднее |
| `MIN/MAX(col)` | минимум/максимум |

---

## GROUP BY — группировка

Считает агрегаты **по группам**:

```sql
SELECT category, COUNT(*) AS cnt, AVG(price) AS avg_price
FROM products
GROUP BY category;
```

```
   products                       результат
 +----------+-------+          +----------+-----+-----------+
 | Напитки  | 100   |          | category | cnt | avg_price |
 | Напитки  | 200   |   ---▶   +----------+-----+-----------+
 | Еда      | 50    |          | Напитки  | 2   | 150       |
 | Еда      | 80    |          | Еда      | 2   | 65        |
 +----------+-------+          +----------+-----+-----------+
```

> Все столбцы в `SELECT` (кроме агрегатов) должны быть в `GROUP BY`.

---

## HAVING — фильтр групп

`WHERE` фильтрует строки **до** группировки, `HAVING` — группы **после**:

```sql
SELECT category, COUNT(*) AS cnt
FROM products
WHERE price > 0           -- фильтр строк (до группировки)
GROUP BY category
HAVING COUNT(*) > 5;      -- фильтр групп (после агрегации)
```

| | WHERE | HAVING |
|--|-------|--------|
| Когда | до группировки | после группировки |
| С чем работает | отдельные строки | группы/агрегаты |

## JOIN — объединение таблиц

Связывает строки разных таблиц по общему полю (обычно FK = PK).

```
   orders                customers
 +----+-------------+   +----+--------+
 | id | customer_id |   | id | name   |
 +----+-------------+   +----+--------+
 | 1  | 10          |   | 10 | Иван   |
 | 2  | 20          |   | 20 | Мария  |
 +----+-------------+   +----+--------+
        JOIN ON orders.customer_id = customers.id
```

### INNER JOIN — только совпадения

```sql
SELECT o.id, c.name, o.amount
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id;
```

Вернёт только заказы, у которых есть клиент.

### LEFT JOIN — все из левой + совпадения из правой

```sql
SELECT c.name, o.id
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id;
```

Вернёт **всех** клиентов, даже без заказов (у них `o.id` будет `NULL`).

### Типы JOIN

```
INNER JOIN        LEFT JOIN         RIGHT JOIN        FULL JOIN
  A ∩ B           всё A + (A∩B)     всё B + (A∩B)     всё A + всё B
   ▟▙               ▟▙▙               ▟▟▙               ▟▟▟▙
```

| JOIN | Возвращает |
|------|-----------|
| `INNER` | только совпадающие строки |
| `LEFT` | все из левой + совпавшие из правой |
| `RIGHT` | все из правой + совпавшие из левой |
| `FULL` | все строки обеих таблиц |
| `CROSS` | декартово произведение (все со всеми) |

## Подзапросы (subqueries)

Запрос внутри запроса:

```sql
-- товары дороже среднего
SELECT * FROM products
WHERE price > (SELECT AVG(price) FROM products);

-- клиенты, у которых есть заказы
SELECT * FROM customers
WHERE id IN (SELECT customer_id FROM orders);
```

## Индексы — ускорение поиска

**Индекс** — структура (обычно B-дерево), ускоряющая поиск по столбцу, как алфавитный указатель в книге.

```sql
CREATE INDEX idx_products_category ON products(category);
```

```
Без индекса:  перебор ВСЕХ строк (full scan)  — медленно на больших данных
С индексом:   быстрый поиск по дереву          — быстро
```

| Плюсы индекса | Минусы индекса |
|---------------|----------------|
| быстрый SELECT/WHERE/JOIN | замедляет INSERT/UPDATE |
| быстрая сортировка | занимает место |

> Индексируйте столбцы, по которым часто ищете и соединяете (FK, часто используемые в WHERE). Не индексируйте всё подряд.

> Первичный ключ индексируется автоматически.

---

## Порядок выполнения SELECT (логический)

```
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

> Понимание порядка объясняет, почему в `WHERE` нельзя использовать псевдоним из `SELECT` (SELECT выполняется позже).

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| столбец не в GROUP BY и не агрегат | ошибка/неопределённый результат |
| HAVING вместо WHERE для строк | HAVING — для групп, медленнее |
| JOIN без условия ON | декартово произведение (взрыв строк) |
| забыли, что LEFT JOIN даёт NULL | учитывайте NULL в правой таблице |
| индекс на всё подряд | замедляет запись, ест память |
