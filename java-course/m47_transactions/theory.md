# Модуль 47. Транзакции БД. ACID, уровни изоляции

**Транзакция** — группа операций, которые выполняются как **единое целое**: либо все успешно (COMMIT), либо ни одной (ROLLBACK). Это защищает данные от частичных изменений при сбоях.

> Практика — SQL-скрипты (H2/любая СУБД).

## Классический пример: перевод денег

```
Перевести 100 со счёта A на счёт B:
  1. UPDATE A: баланс − 100
  2. UPDATE B: баланс + 100

Если сбой между шагами 1 и 2 → деньги списаны, но не зачислены!
Транзакция гарантирует: либо ОБА шага, либо НИ ОДНОГО.
```

```sql
BEGIN;                                      -- начало транзакции
UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';
COMMIT;                                      -- зафиксировать оба изменения
-- при ошибке: ROLLBACK; — отменить всё
```

---

## ACID — свойства транзакций

```
A — Atomicity   (Атомарность)
C — Consistency (Согласованность)
I — Isolation   (Изолированность)
D — Durability  (Долговечность)
```

| Свойство | Значение |
|----------|----------|
| **Atomicity** | всё или ничего (нет частичного выполнения) |
| **Consistency** | данные переходят из одного корректного состояния в другое |
| **Isolation** | параллельные транзакции не мешают друг другу |
| **Durability** | после COMMIT данные сохранены навсегда (даже при сбое) |

---

## Управление транзакциями (TCL)

```sql
BEGIN;            -- или START TRANSACTION — начать
-- ... операции ...
COMMIT;           -- зафиксировать изменения
-- ИЛИ
ROLLBACK;         -- отменить все изменения транзакции
```

### SAVEPOINT — точка отката

Позволяет откатиться к промежуточной точке, а не отменять всё:

```sql
BEGIN;
INSERT INTO log VALUES ('шаг 1');
SAVEPOINT sp1;
INSERT INTO log VALUES ('шаг 2');
ROLLBACK TO sp1;     -- отменит только "шаг 2", "шаг 1" останется
COMMIT;
```

## Проблемы параллельного доступа (аномалии)

Когда несколько транзакций работают одновременно, возможны аномалии:

| Аномалия | Что происходит |
|----------|----------------|
| **Грязное чтение** (dirty read) | читаем данные, которые другая транзакция ещё не зафиксировала (и может откатить) |
| **Неповторяющееся чтение** (non-repeatable read) | повторное чтение той же строки даёт другое значение (её изменили) |
| **Фантомное чтение** (phantom read) | повторный запрос возвращает новые строки (их добавили) |

```
Транзакция 1            Транзакция 2
читает баланс = 1000
                        UPDATE баланс = 500 (не закоммичено)
читает баланс = 500  ← грязное чтение! (T2 может откатиться)
```

## Уровни изоляции

Чем выше изоляция — тем меньше аномалий, но ниже параллелизм (медленнее).

```
READ UNCOMMITTED  <  READ COMMITTED  <  REPEATABLE READ  <  SERIALIZABLE
(меньше изоляции)                                          (больше изоляции)
```

| Уровень | Грязное чтение | Неповтор. чтение | Фантомы |
|---------|:--------------:|:----------------:|:-------:|
| READ UNCOMMITTED | возможно | возможно | возможно |
| READ COMMITTED | нет | возможно | возможно |
| REPEATABLE READ | нет | нет | возможно |
| SERIALIZABLE | нет | нет | нет |

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

> По умолчанию: PostgreSQL/Oracle — `READ COMMITTED`, MySQL (InnoDB) — `REPEATABLE READ`. `SERIALIZABLE` — самый строгий и медленный.

---

## Блокировки (locks)

СУБД использует блокировки для изоляции:

| Блокировка | Значение |
|------------|----------|
| Shared (S) | на чтение, несколько одновременно |
| Exclusive (X) | на запись, эксклюзивно |

> **Взаимоблокировка (deadlock)**: T1 ждёт ресурс T2, а T2 — ресурс T1. СУБД обнаруживает это и откатывает одну из транзакций.

## Транзакции в Java (превью)

В JDBC транзакциями управляют через `Connection` (см. [модуль 50](../module-50-jdbc-2/theory.md)):

```java
connection.setAutoCommit(false);   // выключить авто-коммит
try {
    // ... несколько запросов
    connection.commit();
} catch (SQLException e) {
    connection.rollback();
}
```

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| забыли COMMIT | изменения не сохранятся / заблокируют строки |
| слишком длинные транзакции | долгие блокировки, падение параллелизма |
| `SERIALIZABLE` везде | сильно замедляет систему |
| отсутствие ROLLBACK при ошибке | частичные изменения, неконсистентность |
| автокоммит в Java по умолчанию | каждый запрос — отдельная транзакция; отключайте для группировки |
