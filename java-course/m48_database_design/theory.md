# Модуль 48. Проектирование БД. Нормальные формы

Хорошая структура БД устраняет дублирование, противоречия и аномалии. Этого достигают **нормализацией** — последовательным приведением таблиц к нормальным формам.

> Практика — проектирование схем и SQL (H2/любая СУБД).

## ER-модель (сущность-связь)

Перед таблицами проектируют модель: **сущности** (объекты), их **атрибуты** (свойства) и **связи**.

```
 +----------+         покупает        +----------+
 | КЛИЕНТ   |◄-------- 1 : N --------►|  ЗАКАЗ   |
 | id       |                         | id       |
 | имя      |                         | дата     |
 | email    |                         | сумма    |
 +----------+                         +----------+
```

| Понятие | Значение |
|---------|----------|
| Сущность | объект предметной области (Клиент, Заказ) |
| Атрибут | свойство сущности (имя, дата) |
| Связь | отношение между сущностями |

---

## Виды связей (кардинальность)

```
1 : 1   один к одному    (человек -- паспорт)
1 : N   один ко многим   (клиент -- много заказов)
M : N   многие ко многим (студенты -- курсы)
```

| Связь | Реализация в БД |
|-------|-----------------|
| 1:1 | FK с UNIQUE, или объединить таблицы |
| 1:N | FK в таблице «многие» |
| M:N | промежуточная (связующая) таблица |

### M:N через связующую таблицу

```
students          student_courses           courses
+----+            +------------+----------+  +----+
| id |◄-----------| student_id | course_id|--►| id |
+----+            +------------+----------+  +----+
                  (разбивает M:N на два 1:N)
```

---

## Ключи

| Ключ | Значение |
|------|----------|
| **Первичный (PK)** | уникально идентифицирует строку |
| **Внешний (FK)** | ссылка на PK другой таблицы |
| **Кандидатный** | потенциальный PK (уникален) |
| **Составной** | PK из нескольких столбцов |

---

## Нормализация: зачем

Денормализованная таблица порождает **аномалии**:

```
   orders (плохо — всё в одной таблице)
 +----+---------+-----------+----------+---------+
 | id | клиент  | телефон   | товар    | цена    |
 +----+---------+-----------+----------+---------+
 | 1  | Иван    | 111-22-33 | Кофе     | 350     |
 | 2  | Иван    | 111-22-33 | Чай      | 200     |  ← дублирование клиента
 +----+---------+-----------+----------+---------+
```

| Аномалия | Проблема |
|----------|----------|
| Вставки | нельзя добавить клиента без заказа |
| Обновления | сменился телефон — править во всех строках |
| Удаления | удалив заказ, теряем данные о клиенте |

---

## Первая нормальная форма (1НФ)

**Правило:** все значения **атомарны** (неделимы), нет повторяющихся групп и массивов в ячейке.

```
❌ НЕ 1НФ                          ✅ 1НФ
+----+------------------+         +----+-------+
| id | телефоны          |        | id | телефон|
+----+------------------+        +----+-------+
| 1  | 111, 222, 333     |        | 1  | 111   |
+----+------------------+        | 1  | 222   |
  (несколько в одной ячейке)     | 1  | 333   |
                                  +----+-------+
```

---

## Вторая нормальная форма (2НФ)

**Правило:** соответствует 1НФ + каждый неключевой атрибут зависит от **всего** составного ключа, а не от его части (нет частичных зависимостей).

```
❌ НЕ 2НФ (ключ = order_id + product_id)
order_items(order_id, product_id, product_name, quantity)
            product_name зависит только от product_id → частичная зависимость!

✅ 2НФ — вынести product_name в отдельную таблицу products
order_items(order_id, product_id, quantity)
products(product_id, product_name)
```

---

## Третья нормальная форма (3НФ)

**Правило:** соответствует 2НФ + нет **транзитивных** зависимостей (неключевой атрибут не зависит от другого неключевого).

```
❌ НЕ 3НФ
employees(id, name, dept_id, dept_name)
          dept_name зависит от dept_id (не от PK) → транзитивная зависимость!

✅ 3НФ — вынести отдел
employees(id, name, dept_id)
departments(dept_id, dept_name)
```

---

## Сводка нормальных форм

| Форма | Требование (упрощённо) |
|-------|------------------------|
| 1НФ | атомарные значения, нет повторяющихся групп |
| 2НФ | 1НФ + нет частичных зависимостей от составного ключа |
| 3НФ | 2НФ + нет транзитивных зависимостей |
| БКНФ | усиленная 3НФ |

> На практике обычно достаточно **3НФ**. Каждая форма опирается на предыдущую.

---

## Денормализация

Иногда ради **скорости** намеренно вводят дублирование (меньше JOIN-ов):

| Нормализация | Денормализация |
|--------------|----------------|
| меньше дублирования | быстрее чтение (меньше JOIN) |
| целостность данных | риск рассогласования |
| для OLTP (транзакции) | для аналитики/отчётов (OLAP) |

> Сначала нормализуйте. Денормализуйте осознанно, когда есть реальная проблема производительности.

---

## Процесс проектирования

```
1. Определить сущности и атрибуты (из требований)
2. Установить связи (1:1, 1:N, M:N)
3. Назначить ключи (PK, FK)
4. Нормализовать до 3НФ
5. Создать схему (CREATE TABLE с ключами/ограничениями)
```

---

## Подводные камни

| Ошибка | Объяснение |
|--------|-----------|
| хранение списка в одной ячейке | нарушение 1НФ |
| дублирование данных | аномалии обновления |
| M:N без связующей таблицы | невозможно корректно связать |
| избыточная нормализация | слишком много JOIN, сложность |
| отсутствие FK | потеря ссылочной целостности |
