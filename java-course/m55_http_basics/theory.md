## HTTP-запрос и ответ: контракт

Каждый HTTP-вызов — договор между клиентом и сервером: клиент говорит **что** и **как**,
сервер отвечает **результатом** и **метаданными**.

```
КЛИЕНТ                                           СЕРВЕР
  │  POST /notes HTTP/1.1                          │
  │  Host: api.example.com                         │
  │  Content-Type: application/json                │
  │  Authorization: Bearer eyJhbGc...              │
  │                                                 │
  │  {"title":"Купить молоко","text":"2 литра"}    │
  │ ──────────────────────────────────────────────▶│
  │                                                 │
  │  HTTP/1.1 201 Created                           │
  │  Content-Type: application/json                 │
  │  Location: /notes/42                            │
  │                                                 │
  │  {"id":42,"title":"Купить молоко","text":"..."}│
  │ ◀──────────────────────────────────────────────│
```

## HTTP-методы: назначение, идемпотентность, безопасность

### Таблица методов

| Метод    | Назначение                     | Тело запроса | Идемпотентный | Безопасный |
|----------|-------------------------------|:------------:|:-------------:|:----------:|
| `GET`    | Прочитать ресурс               | нет          | да            | да         |
| `POST`   | Создать новый ресурс           | да           | **нет**       | нет        |
| `PUT`    | Полная замена ресурса          | да           | да            | нет        |
| `PATCH`  | Частичное обновление           | да           | условно       | нет        |
| `DELETE` | Удалить ресурс                 | обычно нет   | да            | нет        |
| `HEAD`   | Только заголовки (без тела)    | нет          | да            | да         |
| `OPTIONS`| Узнать доступные методы        | нет          | да            | да         |

**Идемпотентный** — повторный вызов с теми же данными даёт тот же результат.
`PUT /notes/42` с тем же телом — заметка остаётся неизменной.
`POST /notes` — каждый вызов создаёт новую заметку.

**Безопасный** — не изменяет состояние сервера (только чтение).

## Статус-коды: выбор правильного

### Группы

```
2xx — успех
3xx — перенаправление
4xx — ошибка клиента (запрос неверный)
5xx — ошибка сервера (сервер не справился)
```

### Детальная таблица

| Ситуация                                  | Код | Пояснение                             |
|------------------------------------------|-----|---------------------------------------|
| Успешное чтение                          | 200 | OK — ресурс возвращён в теле          |
| Ресурс создан                            | 201 | Created — тело + заголовок `Location` |
| Успех, тела нет                          | 204 | No Content — DELETE, PATCH без тела   |
| Ресурс не изменился (кеш)                | 304 | Not Modified — клиент берёт из кеша   |
| Неверный формат запроса                  | 400 | Bad Request — ошибка валидации         |
| Нет аутентификации                       | 401 | Unauthorized — нужен токен/логин      |
| Нет прав (токен есть, роль не та)        | 403 | Forbidden — доступ запрещён           |
| Ресурс не найден                         | 404 | Not Found                             |
| Метод не разрешён для этого пути         | 405 | Method Not Allowed                    |
| Тело в неподдерживаемом формате          | 415 | Unsupported Media Type                |
| Превышен лимит запросов                  | 429 | Too Many Requests                     |
| Внутренняя ошибка сервера                | 500 | Internal Server Error                 |
| Сервис временно недоступен               | 503 | Service Unavailable                   |

> **401 vs 403:** 401 — «я не знаю, кто ты», 403 — «знаю, кто ты, но не пущу».

## Заголовки (Headers)

Метаданные, описывающие запрос или ответ.

| Заголовок          | Направление    | Назначение                                      |
|--------------------|----------------|-------------------------------------------------|
| `Content-Type`     | запрос/ответ   | Формат тела: `application/json`, `text/html`    |
| `Accept`           | запрос         | Какой формат ответа хочет клиент                |
| `Authorization`    | запрос         | Токен: `Bearer eyJ...` или `Basic dXNlcjpwYXNz`|
| `Location`         | ответ          | URL созданного ресурса при 201                  |
| `Cache-Control`    | запрос/ответ   | Управление кешем: `no-cache`, `max-age=3600`    |
| `ETag`             | ответ          | Версия ресурса для условных запросов            |
| `If-None-Match`    | запрос         | Клиент отправляет ETag — сервер даёт 304/200    |
| `Accept-Language`  | запрос         | Желаемый язык ответа: `ru`, `en-US`             |

### Stateless — HTTP не хранит состояние

Каждый запрос **самодостаточен**: сервер не помнит предыдущих. Всё необходимое
(токен, параметры) клиент обязан передавать в каждом запросе заново.

```
Запрос 1: GET /notes  + Authorization: Bearer TOKEN   → 200
Запрос 2: GET /notes  (без токена)                    → 401
Сервер «не помнит» запрос 1 при обработке запроса 2.
```

## REST-дизайн URL

### Правила

1. **Существительные, множественное число**: `/notes`, `/users`, `/products`
2. **Вложенность максимум 2 уровня**: `/notes/{id}/tags` — хорошо, `/a/b/c/d` — плохо
3. **Нет глаголов в URL**: действие выражает HTTP-метод, не путь
4. **Нет расширений** в пути: не `/notes.json`, а заголовок `Accept: application/json`

### Сравнение: правильно vs неправильно

| Операция                   | Неправильно                  | Правильно                   |
|---------------------------|------------------------------|-----------------------------|
| Получить список заметок    | `GET /getNotes`              | `GET /notes`                |
| Создать заметку            | `POST /createNote`           | `POST /notes`               |
| Получить заметку           | `GET /note?id=42`            | `GET /notes/42`             |
| Обновить заметку           | `POST /notes/update/42`      | `PUT /notes/42`             |
| Удалить заметку            | `GET /deleteNote/42`         | `DELETE /notes/42`          |
| Сменить статус             | `POST /notes/42/setActive`   | `PATCH /notes/42`           |
| Получить теги заметки      | `GET /tags/ofNote/42`        | `GET /notes/42/tags`        |

### CRUD-контракт для ресурса `/notes`

```
GET    /notes           → 200  список заметок (с пагинацией: ?page=0&size=20)
POST   /notes           → 201  создать заметку, Location: /notes/{id}
GET    /notes/{id}      → 200  / 404
PUT    /notes/{id}      → 200  полная замена
PATCH  /notes/{id}      → 200  частичное обновление (например, только title)
DELETE /notes/{id}      → 204  / 404
GET    /notes/{id}/tags → 200  теги заметки (вложенный ресурс)
```

## Подводные камни

| Ошибка                                    | Правильный подход                                     |
|------------------------------------------|-------------------------------------------------------|
| `GET` с побочным эффектом (изменяет БД)  | Изменения — только через POST/PUT/PATCH/DELETE        |
| `DELETE` возвращает 200 с телом           | Используйте 204 No Content                            |
| `POST` вместо `PATCH` для обновления      | PATCH — частичное, PUT — полная замена                |
| `404` когда нужен `403`                  | Не скрывайте `403` за `404` без причины               |
| Глаголы в URL: `/api/deleteUser/5`        | `DELETE /users/5`                                     |
| Вложенность глубже 2: `/a/b/c/d/e`       | Упростите: `/d/{id}` с query-параметром               |
| Игнорирование заголовка `Content-Type`   | Всегда указывайте формат — сервер вернёт 415          |
| Нет заголовка `Location` при 201         | RFC требует `Location` при создании ресурса           |
