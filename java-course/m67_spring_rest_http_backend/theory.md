# Модуль 67. HTTP глазами backend-разработчика

Прежде чем проектировать REST-API, нужно глубоко понимать **HTTP** — протокол, на котором это API живёт. В [модуле 55](../m55_http_basics/theory.md) мы смотрели на HTTP со стороны клиента. Теперь — со стороны **сервера**: что приходит в запросе, что мы обязаны вернуть, какой статус и заголовки соответствуют ситуации.

Этот блок (67–76) строит сквозной проект **Task Tracker API** — сервис управления задачами (задачи, статусы, исполнители).

> Практика — задачи в `practice/`. Зависимости: `spring-boot-starter-web`. Bare-`javac` НЕ компилируется — запускайте в IDE/Gradle.

## Анатомия HTTP-запроса и ответа

```
ЗАПРОС                                  ОТВЕТ
-----------------------------          -----------------------------
POST /api/tasks HTTP/1.1                HTTP/1.1 201 Created
Host: tracker.local                     Location: /api/tasks/42
Content-Type: application/json          Content-Type: application/json
Accept: application/json                Content-Length: 84
Authorization: Bearer eyJ...
                                        { "id": 42, "title": "...",
{ "title": "Купить кофе",                 "status": "NEW" }
  "assignee": "ivan" }
-----------------------------          -----------------------------
 ▲стартовая строка                       ▲статусная строка
 ▲заголовки                              ▲заголовки
 ▲(пустая строка)                        ▲(пустая строка)
 ▲тело                                   ▲тело
```

Backend получает **запрос** (метод + URI + заголовки + тело) и обязан вернуть **ответ** (статус + заголовки + тело). Всё проектирование REST — про правильный выбор этих частей.

## HTTP-методы и их семантика

| Метод | Смысл | Безопасный | Идемпотентный | Тело запроса |
|-------|-------|:----------:|:-------------:|:------------:|
| `GET` | прочитать ресурс | ✅ | ✅ | нет |
| `POST` | создать / выполнить действие | ❌ | ❌ | да |
| `PUT` | полностью заменить | ❌ | ✅ | да |
| `PATCH` | частично обновить | ❌ | ❌* | да |
| `DELETE` | удалить | ❌ | ✅ | нет |
| `HEAD` | как GET, но без тела | ✅ | ✅ | нет |
| `OPTIONS` | какие методы доступны | ✅ | ✅ | нет |

- **Безопасный (safe):** не изменяет состояние сервера. Браузеры/прокси могут кэшировать и повторять.
- **Идемпотентный:** повтор того же запроса даёт тот же эффект, что и один вызов. `DELETE /tasks/42` дважды → задача удалена (второй раз — уже нет, но состояние то же).

> *PATCH в общем случае не идемпотентен (зависит от семантики), хотя его часто проектируют идемпотентным.

```
   Идемпотентность на практике:
   PUT  /tasks/42  {status:"DONE"}  ×3  →  задача в DONE  (одинаково)
   POST /tasks     {title:"X"}      ×3  →  СОЗДАНО 3 задачи! (опасно при ретраях)
```

## Коды статуса ответа

| Класс | Значение | Примеры |
|-------|----------|---------|
| **1xx** | информационные | 100 Continue |
| **2xx** | успех | 200 OK, 201 Created, 204 No Content |
| **3xx** | перенаправление | 301 Moved, 304 Not Modified |
| **4xx** | ошибка клиента | 400 Bad Request, 401, 403, 404, 409 Conflict, 422 |
| **5xx** | ошибка сервера | 500 Internal Server Error, 503 |

Ключевые для REST:

| Ситуация | Код |
|----------|-----|
| Прочитали / выполнили действие с телом | **200 OK** |
| Создали ресурс | **201 Created** (+ заголовок `Location`) |
| Выполнили, тела нет | **204 No Content** |
| Неверные данные клиента | **400 Bad Request** |
| Не аутентифицирован | **401 Unauthorized** |
| Аутентифицирован, но нет прав | **403 Forbidden** |
| Ресурс не существует | **404 Not Found** |
| Конфликт состояния (дубликат, версия) | **409 Conflict** |
| Семантически невалидные данные | **422 Unprocessable Entity** |

> **Анти-паттерн:** возвращать `200 OK` с телом `{"error": "..."}`. Клиент должен узнавать об ошибке по **статусу**, а не парся тело.

## Важные заголовки

| Заголовок | Сторона | Назначение |
|-----------|---------|------------|
| `Content-Type` | оба | формат тела (`application/json`) |
| `Accept` | запрос | какой формат хочет клиент (основа content negotiation) |
| `Authorization` | запрос | токен/учётные данные (`Bearer ...`) |
| `Location` | ответ | URI созданного ресурса (с 201) |
| `Cache-Control` | ответ | правила кэширования |
| `ETag` / `If-None-Match` | оба | условные запросы, оптимистичная проверка |

### Content negotiation (согласование формата)

Клиент в `Accept` указывает желаемый формат, сервер выбирает подходящий:

```
Accept: application/json  →  сервер отдаёт JSON
Accept: application/xml   →  сервер отдаёт XML (если умеет)
Accept: */*               →  сервер отдаёт формат по умолчанию
```

В Spring это управляется атрибутами `produces`/`consumes`:

```java
@GetMapping(value = "/tasks/{id}", produces = "application/json")
@PostMapping(value = "/tasks", consumes = "application/json")
```

## Stateless — без состояния

HTTP **не хранит состояние** между запросами. Каждый запрос самодостаточен: сервер не «помнит» предыдущий. Поэтому аутентификацию (токен) шлют в **каждом** запросе. Это ключевое ограничение REST — оно даёт масштабируемость (любой инстанс сервера может обработать любой запрос).

```
   Запрос 1 --► [сервер A]   (токен в заголовке)
   Запрос 2 --► [сервер B]   (тот же токен — B обработает не хуже A)
   Сессия на сервере НЕ нужна.
```

## Как backend выбирает ответ

```
   Пришёл запрос
        |
        +- метод поддерживается? нет → 405 Method Not Allowed
        +- тело валидно?         нет → 400 / 422
        +- аутентифицирован?     нет → 401
        +- есть права?           нет → 403
        +- ресурс существует?    нет → 404
        +- конфликт состояния?   да  → 409
        |
        +- всё ок → 200 / 201 / 204 + тело/заголовки
```

## Подводные камни

| Проблема | Причина | Решение |
|----------|---------|---------|
| Клиент не понимает, что произошла ошибка | вернули 200 с `{"error":...}` | вернуть корректный 4xx/5xx |
| Повторный POST создаёт дубликаты | POST не идемпотентен, клиент ретраит | идемпотентный ключ или PUT с известным id |
| `415 Unsupported Media Type` | нет/неверный `Content-Type` в запросе | клиент шлёт `Content-Type: application/json` |
| `406 Not Acceptable` | `Accept` требует формат, который сервер не умеет | согласовать форматы / убрать жёсткий `produces` |
| После создания клиент не знает URI ресурса | нет заголовка `Location` | при 201 ставить `Location: /api/tasks/{id}` |
| 200 вместо 201 при создании | вернули объект без статуса | `ResponseEntity.status(CREATED)` или `@ResponseStatus` |
