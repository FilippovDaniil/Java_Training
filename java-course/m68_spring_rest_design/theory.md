# Модуль 68. REST как стиль проектирования API

В [модуле 67](../m67_spring_rest_http_backend/theory.md) мы разобрали HTTP. **REST** (REpresentational State Transfer) — это архитектурный стиль, который использует HTTP «по назначению»: данные представляются как **ресурсы**, а действия — как **HTTP-методы** над ними. Хорошо спроектированный REST-API предсказуем: зная одну часть, разработчик угадывает остальное.

> Практика — задачи в `practice/`. Зависимости: `spring-boot-starter-web`. Продолжаем проектировать **Task Tracker API**.

---

## Ресурс — центральное понятие

**Ресурс** — это «вещь», которой можно дать имя: задача, пользователь, проект. У ресурса есть **URI** (адрес) и **представление** (например, JSON).

```
   Ресурс            URI                         Представление (JSON)
   ─────             ───                         ────────────────────
   коллекция задач   /api/tasks                  [ {...}, {...} ]
   одна задача       /api/tasks/42               { "id":42, "title":... }
   задачи проекта    /api/projects/7/tasks       [ {...} ]
```

Действие выражается **не** в URI, а в **методе**: не `/api/getTask`, а `GET /api/tasks/42`.

---

## Правила проектирования URI

| Правило | ❌ Плохо | ✅ Хорошо |
|---------|---------|----------|
| Без глаголов в URI | `POST /createTask` | `POST /tasks` |
| Существительные во мн. числе | `/task/42` | `/tasks/42` |
| Действие — это метод | `GET /tasks/42/delete` | `DELETE /tasks/42` |
| Смена статуса — PATCH | `PUT /tasks/42/complete` | `PATCH /tasks/42` + `{"status":"DONE"}` |
| Вложенность ≤ 2 уровней | `/projects/7/tasks/42/comments/3/author` | `/comments/3` (или `/tasks/42/comments`) |
| Фильтры — query-параметры | `/tasks/status/done` | `/tasks?status=DONE` |
| kebab-case в путях | `/taskComments` | `/task-comments` |

```
   Хорошая иерархия Task Tracker:

   /api/tasks                     коллекция
   /api/tasks/{id}                элемент
   /api/tasks/{id}/comments       подресурс (комментарии задачи)
   /api/projects/{id}/tasks       задачи проекта (2 уровня — максимум)
   /api/tasks?status=DONE&page=0  фильтрация и пагинация — параметрами
```

---

## Коллекция vs элемент

| URI | GET | POST | PUT | DELETE |
|-----|-----|------|-----|--------|
| `/tasks` (коллекция) | список | **создать** | заменить всю коллекцию (редко) | удалить все (редко) |
| `/tasks/{id}` (элемент) | один | — (не создают так) | заменить элемент | удалить элемент |

**Создание** — это `POST` на **коллекцию** (`POST /tasks`), сервер сам присваивает id и возвращает его в `Location`.

---

## Семантика методов в дизайне

```
   GET    /tasks/42         «дай задачу 42»            (безопасно, кэшируемо)
   POST   /tasks            «создай новую задачу»      (вернёт 201 + Location)
   PUT    /tasks/42         «замени задачу 42 целиком» (нужны ВСЕ поля)
   PATCH  /tasks/42         «измени часть задачи 42»   (только нужные поля)
   DELETE /tasks/42         «удали задачу 42»
```

**PUT vs PATCH:**

| | PUT | PATCH |
|-|-----|-------|
| Что шлём | полное представление | только изменяемые поля |
| Отсутствующее поле | обнуляется/сбрасывается | не трогается |
| Пример | `{ "title":"X", "status":"NEW", "assignee":"ivan" }` | `{ "status":"DONE" }` |

---

## Действия, не укладывающиеся в CRUD

Иногда нужно действие-глагол («запустить», «отправить», «архивировать»). Подходы:

1. **Через смену состояния (предпочтительно):** `PATCH /tasks/42` с `{"status":"ARCHIVED"}`.
2. **Подресурс-действие как существительное:** `POST /tasks/42/assignments` (назначить исполнителя).
3. **Контролируемое исключение** (когда никак иначе): `POST /tasks/42/actions` с телом `{"action":"reopen"}`.

> Старайтесь моделировать действие как изменение ресурса, а не как RPC-вызов.

## Уровни зрелости — Richardson Maturity Model

```
   Уровень 0:  один URL, один метод (обычно POST) — это RPC, не REST
   Уровень 1:  есть ресурсы (разные URI), но методы не используются по смыслу
   Уровень 2:  ресурсы + правильные HTTP-методы + статусы   ← цель большинства API
   Уровень 3:  + HATEOAS (гиперссылки в ответах)            ← «полный» REST
```

Большинство «REST-API» в индустрии — **уровень 2**, и это нормальная цель.

### HATEOAS (уровень 3, обзорно)

Ответ содержит **ссылки** на возможные следующие действия — клиент не «зашивает» URI, а следует ссылкам:

```json
{
  "id": 42,
  "title": "Купить кофе",
  "status": "NEW",
  "_links": {
    "self":     { "href": "/api/tasks/42" },
    "complete": { "href": "/api/tasks/42", "method": "PATCH" }
  }
}
```

Spring предлагает `spring-boot-starter-hateoas` (`EntityModel`, `Link`). В курсе мы делаем упрощённую ручную версию (ссылки в DTO).

## API — это контракт

После публикации API его форму нельзя ломать: клиенты на неё рассчитывают. Поэтому:

- продумывайте имена ресурсов и поля **заранее**;
- расширяйте **аддитивно** (добавлять поля можно, удалять/переименовывать — ломает клиентов);
- несовместимые изменения — через **версионирование** (`/api/v2/...`), подробно в [модуле 75](../m75_spring_rest_config_openapi/theory.md).

## Подводные камни

| Проблема | Причина | Решение |
|----------|---------|---------|
| URI с глаголами (`/getAllTasks`) | мышление в стиле RPC | ресурс + метод (`GET /tasks`) |
| Глубокая вложенность (`/a/1/b/2/c/3`) | моделируют всё дерево | ≤ 2 уровней, ссылаться по id |
| Фильтр в пути (`/tasks/done`) | путают данные и фильтр | query-параметр (`?status=DONE`) |
| `PUT` используют для частичного апдейта | путают с PATCH | PUT — целиком, PATCH — частично |
| Единственное число (`/task/42`) | непоследовательность | везде множественное (`/tasks/42`) |
| Действие как новый endpoint на каждый чих | RPC-стиль | моделировать как смену состояния |
