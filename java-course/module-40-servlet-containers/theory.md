# Модуль 40. Контейнеры сервлетов: Tomcat, развёртывание

> ⚠️ **Только теория.** Практика будет добавлена позже.

**Сервлет-контейнер** (Servlet Container, Web Container) — это программа-сервер, которая запускает сервлеты (см. [модуль 39](../module-39-servlets/theory.md)), управляет их жизненным циклом, принимает HTTP-запросы и направляет их нужному сервлету.

## Зачем нужен контейнер

Сервлет — это просто класс, он не умеет сам слушать сеть. Контейнер берёт на себя «инфраструктуру»:

```
        ┌──────────────── Контейнер (Tomcat) ────────────────┐
HTTP ──▶ │ принимает TCP-соединения (порт 8080)               │
запрос   │ парсит HTTP                                         │
         │ создаёт HttpServletRequest/Response                 │
         │ управляет потоками (пул)                            │ ──▶ ваш Сервлет
         │ управляет жизненным циклом (init/service/destroy)   │
         │ хранит сессии                                        │
         └─────────────────────────────────────────────────────┘
```

| Контейнер делает | Вы пишете |
|------------------|-----------|
| сеть, потоки, HTTP-парсинг | бизнес-логику в сервлетах |
| жизненный цикл, сессии | обработку запросов |
| безопасность, пулы | конфигурацию |

---

## Популярные контейнеры

| Контейнер | Особенность |
|-----------|-------------|
| **Apache Tomcat** | самый популярный, лёгкий, только сервлеты/JSP |
| **Jetty** | лёгкий, встраиваемый |
| **Undertow** | высокопроизводительный (использует WildFly) |
| **GlassFish / WildFly** | полноценные сервера приложений (Jakarta EE целиком) |

> Tomcat — это **сервлет-контейнер**, а не полный сервер приложений (нет EJB, JMS из коробки). Для большинства веб-приложений этого достаточно.

## Структура каталогов Tomcat

```
tomcat/
├── bin/        # скрипты запуска (startup.sh, shutdown.sh, catalina)
├── conf/       # конфигурация (server.xml, web.xml)
├── lib/        # общие библиотеки
├── logs/       # логи
├── webapps/    # СЮДА кладут приложения (.war)
└── work/       # временные файлы (скомпилированные JSP)
```

---

## WAR — упаковка веб-приложения

**WAR** (Web Application Archive) — zip-архив с веб-приложением, имеющий стандартную структуру:

```
my-app.war
├── index.html, *.jsp        # публичные ресурсы
├── css/, js/, images/
└── WEB-INF/
    ├── web.xml              # дескриптор развёртывания
    ├── classes/             # скомпилированные .class
    └── lib/                 # jar-зависимости
```

> Всё в `WEB-INF/` **недоступно** напрямую из браузера — это защита кода и конфигурации.

---

## Развёртывание (deployment)

```
1. Собрать приложение в .war (mvn package)
2. Скопировать my-app.war в tomcat/webapps/
3. Tomcat автоматически распакует и развернёт его
4. Приложение доступно по http://localhost:8080/my-app/
```

| Способ развёртывания | Как |
|----------------------|-----|
| Авторазвёртывание | положить `.war` в `webapps/` |
| Manager App | через веб-интерфейс Tomcat |
| Maven-плагин | `tomcat7-maven-plugin` / `cargo` |
| IDE | IntelliJ IDEA подключает Tomcat и деплоит автоматически |

---

## Контекст приложения (context path)

Имя war-файла становится **контекстным путём**:

```
my-app.war       → http://localhost:8080/my-app/
ROOT.war         → http://localhost:8080/         (корень)
```

## Встроенный (embedded) vs автономный (standalone)

```
АВТОНОМНЫЙ (классика)          ВСТРОЕННЫЙ (современно)
┌──────────────┐               ┌──────────────────────┐
│   Tomcat     │               │  Ваше приложение     │
│  ┌────────┐  │               │  ┌────────────────┐  │
│  │app.war │  │               │  │ Tomcat внутри  │  │
│  └────────┘  │               │  │ (Spring Boot)  │  │
└──────────────┘               │  └────────────────┘  │
ставим Tomcat отдельно         └──────────────────────┘
                                java -jar app.jar
```

> **Spring Boot** использует встроенный Tomcat: приложение — обычный `.jar`, запускается через `java -jar`, контейнер уже внутри. Это упрощает развёртывание (см. [модуль 56](../module-56-spring-ecosystem/theory.md)).
