# Тема 01. Принципы ООП

Прежде чем браться за паттерны, надо твёрдо владеть четырьмя китами объектно-ориентированного проектирования. Паттерны GoF — это всего лишь грамотные комбинации этих четырёх принципов плюс одного правила («предпочитай композицию наследованию»). Если киты шатаются — паттерны превращаются в карго-культ.

```

   ┌───────────┬───────────┬───────────┐
Инкапсуляция Наследование Полиморфизм Абстракция
 (скрыть      (переиспольз.(один        (выделить
  состояние)   через is-a)  интерфейс —   суть, скрыть
                            много форм)   детали)
```

## 1. Инкапсуляция — скрыть состояние за поведением

Поля — `private`, доступ — только через методы, которые **защищают инварианты** (правила, которые всегда должны быть истинны).

```java
class Account {
    private long balance;                 // никто снаружи не трогает напрямую

    public void deposit(long amount) {
        if (amount <= 0) throw new IllegalArgumentException("Сумма > 0");
        balance += amount;                // инвариант: balance не уходит в минус «случайно»
    }
    public long getBalance() { return balance; }
}
```

> Плохая инкапсуляция — это `public` поля или «глупые» геттеры/сеттеры на всё подряд: тогда класс становится просто структурой данных, а правила размазываются по вызывающему коду (это путь к **Anemic Domain Model**, тема 22).

**Признак хорошей инкапсуляции:** снаружи объект нельзя привести в некорректное состояние.

## 2. Наследование — переиспользование через `is-a`

`class B extends A` означает «B **является** частным случаем A». Наследник получает поля и методы родителя, может добавлять свои и переопределять (`@Override`) существующие.

```java
class Employee {
    protected String name;
    public double salary() { return 50_000; }
}
class Manager extends Employee {
    @Override public double salary() { return super.salary() + 30_000; }  // расширили
}
```

Наследование — мощный, но **опасный** инструмент: оно жёстко связывает наследника с реализацией родителя. Поэтому есть правило (см. §5): сначала думай о композиции.

**Тест на наследование:** подставь фразу «B is a A». «Менеджер — это сотрудник» ✅. «Машина — это двигатель» ❌ (у машины **есть** двигатель → композиция).

---

## 3. Полиморфизм — один интерфейс, много реализаций

Переменная типа родителя/интерфейса может ссылаться на любой подтип, а вызов метода уходит в **реальную** реализацию (динамическая диспетчеризация).

```java
List<Shape> shapes = List.of(new Circle(2), new Rectangle(3, 4));
for (Shape s : shapes) {
    System.out.println(s.area());   // у каждого — своя area(), решается в рантайме
}
```

Полиморфизм — это то, что убивает длинные `if (type == ...) else if (...)`. Вместо ветвления по типу — вызов метода, который у каждого типа свой. Почти все паттерны GoF держатся на полиморфизме.

---

## 4. Абстракция — выделить суть, спрятать детали

Абстракция — это контракт «что делает», без «как именно». В Java выражается через `interface` и `abstract class`.

| Инструмент | Когда |
|------------|-------|
| `interface` | чистый контракт поведения; нет общего состояния; класс может реализовать несколько |
| `abstract class` | есть общее состояние/код для наследников; нужен общий «скелет» |

```java
interface PaymentMethod {            // контракт: «чем-то можно заплатить»
    void pay(long amount);
}
```

> Абстракция и инкапсуляция — не одно и то же. Инкапсуляция прячет **данные**, абстракция прячет **реализацию** за контрактом. Часто работают вместе.

## 5. Композиция vs наследование — главное правило проектирования

> **Favor composition over inheritance** — предпочитай композицию наследованию.

| | Наследование (`is-a`) | Композиция (`has-a`) |
|---|----------------------|----------------------|
| Связь | жёсткая, на этапе компиляции | гибкая, можно менять в рантайме |
| Переиспользование | через родителя | через делегирование вложенному объекту |
| Риск | хрупкий базовый класс, «взрыв» иерархии | чуть больше «проводки» вручную |

```java
// ❌ наследование ради переиспользования кода — хрупко
class LoggingList extends ArrayList<String> { /* перекрыли все методы? */ }

// ✅ композиция — оборачиваем и делегируем
class LoggingList {
    private final List<String> inner = new ArrayList<>();
    public void add(String s) { System.out.println("add " + s); inner.add(s); }
}
```

Композиция — фундамент паттернов Strategy, Decorator, Adapter и почти всей «части 2–4».

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| `public` поля / сеттеры на всё | объект приводят в кривое состояние извне | инварианты в методах, поля `private final` где можно |
| Наследование «ради экономии кода» | хрупкий базовый класс, переопределяем половину методов | замени на композицию |
| Глубокие иерархии (5+ уровней) | непонятно, откуда метод; сложно менять | максимум 1–2 уровня; выноси в интерфейсы |
| Проверка типа `instanceof` + ветвление | длинные `if/else` по типам | полиморфизм: метод у каждого типа свой |
| Абстракция «на всякий случай» | интерфейс с одной реализацией навсегда | абстрагируй, когда есть **вторая** причина (или явно ожидается) |
