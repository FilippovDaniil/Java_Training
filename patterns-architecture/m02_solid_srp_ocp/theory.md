# Тема 02. SOLID: SRP и OCP

**SOLID** — пять принципов, которые превращают «работающий код» в «код, который не страшно менять». В этой теме — первые два: **S** (Single Responsibility) и **O** (Open/Closed). Оставшиеся три — в теме 03.

```
S — Single Responsibility   одна причина для изменения
O — Open/Closed             открыт для расширения, закрыт для модификации
L — Liskov Substitution     ← тема 03
I — Interface Segregation   ← тема 03
D — Dependency Inversion    ← тема 03
```

---

## 1. SRP — Single Responsibility Principle

> У класса должна быть **ровно одна причина для изменения**.

«Причина для изменения» = заинтересованная сторона (актор). Если класс меняется и когда меняется формула расчёта, и когда меняется формат вывода, и когда меняется способ хранения — у него **три** причины меняться, а значит три ответственности.

```java
// ❌ Нарушение SRP: один класс делает три разные вещи
class Invoice {
    double calculateTotal() { ... }          // бизнес-правило (меняет бухгалтерия)
    String renderHtml() { ... }              // представление (меняет фронт)
    void saveToDatabase() { ... }            // хранение (меняет DBA)
}
```

```java
// ✅ Три ответственности — три класса
class InvoiceCalculator { double total(Invoice i) { ... } }
class InvoiceRenderer   { String html(Invoice i) { ... } }
class InvoiceRepository { void save(Invoice i) { ... } }
```

**Симптом нарушения:** в описании класса есть «И» — «считает И печатает И сохраняет». Каждое «И» — кандидат на отдельный класс.

> SRP ≠ «один метод на класс». Это про **связность** (cohesion): всё в классе меняется по одной причине и работает с одними данными.

## 2. OCP — Open/Closed Principle

> Класс открыт для **расширения**, но закрыт для **модификации**.

Добавление нового поведения не должно требовать правок в уже работающем, протестированном коде. Главный враг OCP — ветвление по типу:

```java
// ❌ Нарушение OCP: новый способ доставки → лезем в этот switch
class ShippingCalculator {
    double cost(String type, double weight) {
        switch (type) {
            case "standard": return weight * 5;
            case "express":  return weight * 10;
            // новый тип "drone" → правим РАБОТАЮЩИЙ код, рискуем сломать старое
            default: throw new IllegalArgumentException();
        }
    }
}
```

```java
// ✅ OCP через полиморфизм: новый способ = новый класс, старое не трогаем
interface ShippingMethod { double cost(double weight); }

class StandardShipping implements ShippingMethod { public double cost(double w){ return w*5; } }
class ExpressShipping  implements ShippingMethod { public double cost(double w){ return w*10; } }
// новый DroneShipping добавляется БЕЗ правок существующих классов
```

Механизмы OCP: интерфейсы + полиморфизм, паттерн **Strategy** (тема 10), реестр/фабрика реализаций.

```
Закрыт для модификации          Открыт для расширения
+--------------------+          +----------------------+
| ShippingCalculator |  --использует--▶  ShippingMethod |
|  (не трогаем)      |          |  Standard / Express / |
+--------------------+          |  Drone (добавляем)    |
                                 +----------------------+
```

## SRP и OCP вместе

Они усиливают друг друга: разделив ответственности (SRP), вы получаете маленькие классы, каждый из которых легко расширить новой реализацией интерфейса (OCP), не задевая остальные.

| Принцип | Вопрос, который задаём | Решение |
|---------|------------------------|---------|
| SRP | «Сколько причин у класса меняться?» | разнести ответственности по классам |
| OCP | «Что я правлю, добавляя новый случай?» | свести правки к добавлению нового класса |

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| God Object | класс на 1000 строк делает всё | разнести по SRP (тема 22 — антипаттерны) |
| `switch (type)` повсюду | новый тип → правки в N местах | OCP через интерфейс + полиморфизм |
| Over-engineering OCP | абстракции «на будущее», которое не наступит | абстрагируй по факту второго случая, не раньше |
| Дробление до абсурда | классы из одного поля без поведения | SRP про связность, а не про «класс на каждую строку» |
| OCP без полиморфизма | добавили интерфейс, но всё равно `if (x instanceof ...)` | вызов метода через интерфейс, не проверку типа |
