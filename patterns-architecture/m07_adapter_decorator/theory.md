# Тема 07. Adapter и Decorator

Первые **структурные** паттерны GoF. Структурные паттерны отвечают на вопрос «как собирать объекты в более крупные структуры», не переписывая их код. **Adapter** делает несовместимые интерфейсы совместимыми, **Decorator** наращивает поведение, оборачивая объект.

```
Структурные паттерны (композиция объектов)
  Adapter    — «переходник» между несовместимыми интерфейсами
  Decorator  — динамически добавляет поведение через обёртку
  Proxy, Facade        (тема 08)
  Composite, Bridge    (тема 09)
```

Оба держатся на **композиции** (тема 01): обёртка хранит ссылку на оборачиваемый объект и делегирует ему вызовы.

---

## 1. Adapter — переходник

Преобразует интерфейс класса в другой, ожидаемый клиентом. Нужен, когда есть готовый/чужой класс с «неправильным» интерфейсом.

```java
// есть чужой класс с неудобным методом
class LegacyPayment {
    void doPayment(int cents) { System.out.println("оплачено " + cents); }
}

// клиент ожидает такой интерфейс
interface PaymentGateway { boolean pay(long amountCents); }

// адаптер: реализует ожидаемый интерфейс, внутри делегирует чужому классу
class LegacyPaymentAdapter implements PaymentGateway {
    private final LegacyPayment legacy;                       // композиция
    LegacyPaymentAdapter(LegacyPayment legacy) { this.legacy = legacy; }
    public boolean pay(long amountCents) {
        legacy.doPayment((int) amountCents);                  // переводим вызов
        return true;
    }
}
```

```
   Клиент --▶ PaymentGateway (ожидаемый интерфейс)
                    ▲
              LegacyPaymentAdapter --делегирует--▶ LegacyPayment (чужой класс)
```

**Object adapter** (через композицию, как выше) — предпочтительный. **Class adapter** (через наследование) в Java ограничен отсутствием множественного наследования классов.

Частый случай — **конвертация данных/единиц**: адаптер датчика в Фаренгейтах под интерфейс «верни Цельсий».

> Где встречается: `Arrays.asList()` (массив → List), `InputStreamReader` (байты → символы), обёртки над сторонними SDK.

## 2. Decorator — обёртка с поведением

Добавляет объекту новые обязанности динамически, оборачивая его в объект-декоратор с тем же интерфейсом. Альтернатива «взрыву» подклассов.

```java
interface Coffee { long costCents(); String description(); }

class SimpleCoffee implements Coffee {
    public long costCents() { return 10000; }
    public String description() { return "кофе"; }
}

// базовый декоратор: реализует тот же интерфейс и хранит обёрнутый объект
abstract class CoffeeDecorator implements Coffee {
    protected final Coffee inner;                             // композиция
    CoffeeDecorator(Coffee inner) { this.inner = inner; }
}

class Milk extends CoffeeDecorator {
    Milk(Coffee inner) { super(inner); }
    public long costCents() { return inner.costCents() + 2000; }       // +поведение
    public String description() { return inner.description() + " + молоко"; }
}
```

Использование — «обёртка вокруг обёртки»:

```java
Coffee c = new Milk(new Sugar(new SimpleCoffee()));   // кофе + сахар + молоко
c.costCents();  c.description();
```

```
   Milk( Sugar( SimpleCoffee ) )
    |     |        |
    |     |        + базовая цена/описание
    |     + добавляет сахар к результату вложенного
    + добавляет молоко к результату вложенного
```

**Порядок обёрток влияет на результат** (например, сжать-потом-зашифровать ≠ зашифровать-потом-сжать).

> Где встречается: классические потоки java.io — `new BufferedReader(new InputStreamReader(in))`; `Collections.unmodifiableList(...)`.

---

## Adapter vs Decorator

| | Adapter | Decorator |
|---|---------|-----------|
| Цель | поменять интерфейс (сделать совместимым) | добавить поведение, интерфейс тот же |
| Интерфейс на входе/выходе | разный | одинаковый |
| Меняет ли поведение | нет (только «переводит») | да (расширяет) |
| Можно ли стопкой | обычно один | да, много обёрток подряд |

Оба — обёртки на композиции; разница в **намерении**.

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Адаптер с бизнес-логикой | в адаптере появились правила, а не только «перевод» | адаптер только конвертирует, логика — снаружи |
| Декоратор не реализует общий интерфейс | нельзя вложить друг в друга | декоратор и объект — один интерфейс |
| Декоратор не делегирует inner | потеряно базовое поведение | всегда зови inner.method() и дополняй |
| Глубокая стопка декораторов | трудно отладить, неясный порядок | следи за порядком обёрток, не злоупотребляй |
| Adapter там, где нужен Decorator (и наоборот) | путаница намерений | интерфейс меняется → Adapter; расширяется → Decorator |
