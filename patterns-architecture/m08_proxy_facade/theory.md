# Тема 08. Proxy и Facade

Ещё два структурных паттерна. **Proxy** ставит «посредника» перед объектом, чтобы контролировать доступ к нему. **Facade** прячет сложную подсистему за одним простым интерфейсом.

```
Proxy   — заместитель: тот же интерфейс, но контролирует доступ к реальному объекту
Facade  — единая «дверь» к подсистеме из многих классов
```

---

## 1. Proxy — заместитель

Proxy реализует **тот же интерфейс**, что и реальный объект, и стоит «перед» ним, добавляя контроль: ленивую загрузку, проверку прав, кеширование, логирование, удалённый вызов.

```java
interface Image { void display(); }

class RealImage implements Image {                 // «тяжёлый» объект
    RealImage(String file) { System.out.println("загрузка " + file); /* дорого */ }
    public void display() { System.out.println("показ"); }
}

class ImageProxy implements Image {                // заместитель с тем же интерфейсом
    private RealImage real;
    private final String file;
    ImageProxy(String file) { this.file = file; }
    public void display() {
        if (real == null) real = new RealImage(file);   // ленивая загрузка при первом показе
        real.display();
    }
}
```

```
   Клиент ──▶ Image ──▶ ImageProxy ──(создаёт лениво / проверяет / кеширует)──▶ RealImage
```

### Виды Proxy

| Вид | Зачем |
|-----|-------|
| Virtual (ленивый) | откладывает создание дорогого объекта до первого использования |
| Protection (защитный) | пропускает вызов только при достаточных правах |
| Logging / Caching | логирует обращения / кеширует результаты |
| Remote | прячет сетевой вызов за локальным интерфейсом (RMI, gRPC-стаб) |

> Где встречается: Spring AOP-прокси (`@Transactional`, `@Cacheable`), Hibernate ленивые сущности, динамические прокси `java.lang.reflect.Proxy`.

---

## 2. Facade — фасад

Facade даёт **один упрощённый интерфейс** к группе классов. Клиенту не нужно знать порядок вызовов и зависимости внутри подсистемы.

```java
// сложная подсистема из нескольких классов
class Cpu    { void boot()  { ... } }
class Memory { void load()  { ... } }
class Disk   { void read()  { ... } }

// фасад прячет порядок и детали запуска
class Computer {
    private final Cpu cpu = new Cpu();
    private final Memory memory = new Memory();
    private final Disk disk = new Disk();
    public void start() {                          // одна простая операция
        cpu.boot();
        memory.load();
        disk.read();
    }
}
// клиент: new Computer().start();  — и всё
```

```
              ┌──────── Facade (Computer.start) ────────┐
   Клиент ──▶ │  cpu.boot() → memory.load() → disk.read()│
              └──────────────────────────────────────────┘
                 (клиент не знает про CPU/Memory/Disk)
```

Facade **не запрещает** доступ к подсистеме напрямую — он лишь предлагает удобный «вход» для типового сценария.

---

## Proxy vs Facade vs Decorator

| | Proxy | Facade | Decorator |
|---|-------|--------|-----------|
| Интерфейс | как у реального объекта | новый, упрощённый | как у оборачиваемого |
| Сколько объектов оборачивает | один (тот же) | целую подсистему | один (тот же) |
| Цель | контроль доступа | упрощение входа | добавить поведение |

Все три — обёртки на композиции; различаются **намерением**.

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Proxy с бизнес-логикой | посредник «решает» доменные правила | proxy только контролирует доступ, логика — в реальном объекте |
| Кеш-proxy без инвалидации | устаревшие данные | продумай срок/сброс кеша |
| Facade превращается в God Object | фасад знает и делает слишком много | фасад только координирует, не реализует логику подсистемы |
| Facade прячет всё «намертво» | нельзя обратиться к подсистеме, когда нужно | фасад — удобный вход, не единственный |
| Proxy не реализует общий интерфейс | его нельзя подставить вместо реального | proxy и объект — один интерфейс |

## 🔗 Связь с другими темами

- **Decorator (тема 07)** — родственная обёртка; Proxy отличается намерением (контроль, а не расширение).
- **Facade** — частый «вход» в слой бизнес-логики (тема 15 Layered) и в гексагон (тема 17).
- **DIP (тема 03)** — клиент зависит от интерфейса, не зная, реальный это объект или proxy.
- **Spring AOP** — прокси как механизм `@Transactional`/`@Cacheable` (см. курс Spring).
