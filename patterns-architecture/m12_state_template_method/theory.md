# Тема 12. State и Template Method

Ещё два поведенческих паттерна. **State** меняет поведение объекта при смене его внутреннего состояния (вместо россыпи `if`). **Template Method** задаёт скелет алгоритма в базовом классе, оставляя отдельные шаги подклассам.

```
State           — поведение зависит от состояния; состояния — отдельные классы
Template Method — фиксированный каркас алгоритма + переопределяемые шаги
```

---

## 1. State — состояние как объект

Когда поведение объекта сильно зависит от его состояния, и состояний много, ветвление `if (state == ...)` расползается. State выносит каждое состояние в отдельный класс с собственным поведением и правилами перехода.

```java
interface OrderState {
    OrderState pay();      // что происходит при оплате
    OrderState ship();     // что происходит при отгрузке
    String name();
}

class NewState implements OrderState {
    public OrderState pay()  { return new PaidState(); }       // переход NEW → PAID
    public OrderState ship() { return this; }                  // нельзя отгрузить неоплаченный
    public String name() { return "NEW"; }
}
class PaidState implements OrderState {
    public OrderState pay()  { return this; }                  // уже оплачен
    public OrderState ship() { return new ShippedState(); }    // переход PAID → SHIPPED
    public String name() { return "PAID"; }
}

class Order {                                     // контекст
    private OrderState state = new NewState();
    public void pay()  { state = state.pay(); }   // делегирует текущему состоянию
    public void ship() { state = state.ship(); }
    public String status() { return state.name(); }
}
```

```
   NEW --pay()--▶ PAID --ship()--▶ SHIPPED
    |ship()=нельзя   |pay()=уже оплачен
   (каждое состояние знает свои переходы)
```

Контекст не содержит `switch` — он делегирует текущему состоянию, а состояние само решает, во что перейти.

> Где встречается: жизненный цикл заказа/документа/задачи; конечные автоматы; парсеры; UI-режимы.

### State vs Strategy

Структурно похожи (делегирование объекту), но: **Strategy** подставляют извне ради алгоритма; **State** объект переключает сам по ходу жизни, и состояния знают о переходах между собой.

## 2. Template Method — каркас алгоритма

Базовый класс задаёт **скелет** алгоритма (последовательность шагов) в одном `final`-методе, а отдельные шаги объявляет абстрактными/переопределяемыми. Подклассы меняют шаги, не трогая структуру.

```java
abstract class DataProcessor {
    public final void run() {        // ШАБЛОННЫЙ МЕТОД — каркас, не переопределяется
        String raw = read();
        String result = process(raw);
        write(result);
        if (shouldLog()) log(result);   // hook (необязательный шаг)
    }
    protected abstract String read();              // шаги — за подклассами
    protected abstract String process(String s);
    protected abstract void write(String s);
    protected boolean shouldLog() { return false; }  // hook со значением по умолчанию
    private void log(String s) { System.out.println("log: " + s); }
}
```

```
   run() (фиксированный порядок)
     +- read()      ← подкласс
     +- process()   ← подкласс
     +- write()     ← подкласс
     +- shouldLog()? log()   ← hook
```

**Hook** — шаг с реализацией по умолчанию, который подкласс может (но не обязан) переопределить. Каркас (`run`) делают `final`, чтобы порядок шагов нельзя было сломать.

> Где встречается: `AbstractList`, фреймворки тестов (`setUp`/`test`/`tearDown`), сервлеты (`doGet`/`doPost`), Spring `JdbcTemplate`.

---

## State vs Template Method

| | State | Template Method |
|---|-------|-----------------|
| Механизм | композиция (объект-состояние) | наследование (шаги в подклассе) |
| Что меняется | поведение при смене состояния | отдельные шаги фиксированного алгоритма |
| Кто управляет порядком | переходы между состояниями | базовый класс (скелет) |
| Гибкость в рантайме | да (состояние меняется) | нет (тип задан при создании) |

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| `switch (state)` вместо классов состояний | ветвление расползается при росте состояний | вынеси состояния в классы (State) |
| Контекст сам решает переходы | дублирование логики переходов | переход возвращает следующее состояние из самого состояния |
| Переопределяемый шаблонный метод | подкласс ломает порядок шагов | делай каркас `final` |
| Слишком много абстрактных шагов | подклассы перегружены | часть шагов — hooks со значением по умолчанию |
| Template Method ради одной реализации | наследование без необходимости | если вариант один — не плоди иерархию |

## 🔗 Связь с другими темами

- **Strategy (тема 10)** — структурно близок State; разница в инициаторе смены поведения.
- **State machine** — основа жизненного цикла агрегатов (DDD, тема 19) и saga (тема 21).
- **Template Method** — каркас, который во фреймворках сочетается с **hook-методами** (инверсия управления).
- **Factory Method (тема 04)** — часто вызывается как один из шагов Template Method.
