# Тема 21. Saga — распределённые транзакции

В монолите согласованность даёт транзакция БД (всё или ничего). Но когда бизнес-процесс растянут на несколько сервисов/агрегатов, **общей транзакции нет** — нельзя «откатить» чужой сервис. **Saga** решает это: процесс разбивается на последовательность локальных шагов, а при сбое выполняются **компенсирующие действия**, отменяющие уже сделанное.

```
Saga — последовательность локальных шагов + компенсации при сбое
  + шаг1 - шаг2 - шаг3 ✗
  + при сбое: компенс2 - компенс1  (откат в обратном порядке)
Два стиля координации: ОРКЕСТРАЦИЯ и ХОРЕОГРАФИЯ
```

> ⚠️ Saga даёт **eventual consistency**, а не атомарность. В промежутке состояние может быть «частично применённым»; компенсации возвращают систему в согласованное состояние (но это не мгновенно).

---

## 1. Шаги и компенсации

Каждый шаг имеет **прямое** действие и **компенсирующее** (откатывающее). Если шаг N упал — выполняем компенсации шагов N-1, N-2, ... 1 в обратном порядке.

```java
interface SagaStep {
    void execute();      // прямое действие
    void compensate();   // откат (обратное действие)
}
```

Компенсация — не «rollback БД», а **новое действие**, семантически отменяющее эффект: списали — вернуть; зарезервировали — снять резерв; отправили письмо — послать письмо-отмену.

```
   reserveStock  -- execute --▶  ✓
   chargeCard    -- execute --▶  ✓
   ship          -- execute --▶  ✗ упал
   → compensate: chargeCard.compensate() (вернуть деньги),
                 reserveStock.compensate() (снять резерв)
```

---

## 2. Оркестрация (Orchestration)

Единый **оркестратор** (координатор) знает все шаги и явно ими управляет: вызывает по очереди, при сбое запускает компенсации.

```java
class TransferSaga {                          // оркестратор
    private final List<SagaStep> steps;
    void run() {
        List<SagaStep> done = new ArrayList<>();
        try {
            for (SagaStep s : steps) { s.execute(); done.add(s); }
        } catch (Exception e) {
            for (int i = done.size() - 1; i >= 0; i--) done.get(i).compensate();  // откат назад
        }
    }
}
```

**Плюсы:** логика процесса в одном месте, понятно и контролируемо. **Минусы:** оркестратор знает обо всех, центральная точка.

---

## 3. Хореография (Choreography)

Нет центрального координатора: каждый шаг **реагирует на событие** предыдущего и публикует своё. Процесс «собирается» из реакций.

```
   OrderCreated -▶ (Payment слушает) -▶ PaymentDone -▶ (Shipping слушает) -▶ Shipped
   при сбое: ShippingFailed -▶ (Payment слушает) -▶ откат оплаты
```

**Плюсы:** нет центральной точки, сервисы слабо связаны (через события). **Минусы:** общий ход процесса «размазан», труднее проследить и отлаживать.

| | Оркестрация | Хореография |
|---|-------------|-------------|
| Координация | центральный оркестратор | реакции на события |
| Видимость процесса | в одном месте | распределена |
| Связанность | оркестратор знает всех | сервисы знают только события |
| Когда | сложный процесс, нужен контроль | простые цепочки, максимум развязки |

---

## 4. Идемпотентность и порядок компенсаций

- **Компенсации — в обратном порядке** выполненных шагов.
- Шаги и компенсации желательно **идемпотентны** (повторный вызов не навредит) — сообщения в распределённой системе могут доставляться повторно.
- Не каждый шаг откатывается «начисто»: иногда компенсация — это «отметить как отменённое», а не «как будто не было».

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Ожидание атомарности | «или всё, или ничего» не выполняется | Saga = eventual consistency, не ACID |
| Компенсации в прямом порядке | откат ломает зависимости шагов | компенсировать в ОБРАТНОМ порядке |
| Нет компенсации у шага с эффектом | «зависшие» резервы/списания | у каждого шага с внешним эффектом — компенсация |
| Неидемпотентные шаги | повтор сообщения = двойное списание | делай шаги/компенсации идемпотентными |
| Хореография для сложного процесса | невозможно проследить ход | сложный процесс → оркестрация |
| Компенсация может упасть | застряли в полу-состоянии | ретраи, ручное вмешательство, dead-letter |
