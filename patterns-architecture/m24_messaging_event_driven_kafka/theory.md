# Тема 24. Асинхронные архитектуры: очереди, pub-sub, события

Тема 23 показала: сервисы можно связать синхронно (REST/gRPC) или асинхронно. Асинхронная связь через **сообщения/события** развязывает сервисы: отправитель не ждёт получателя и не зависит от его доступности. Это фундамент событийных и микросервисных систем.

```
Message Queue (точка-точка) — одно сообщение получает ОДИН потребитель
Pub-Sub (топик)             — событие получают ВСЕ подписчики
Event-Driven                — система реагирует на события, а не на прямые вызовы
Гарантии доставки + DLQ     — что делать с потерями и «ядовитыми» сообщениями
```

> Реализуем на чистом Java in-memory (очереди/топики — коллекции, доставка — вызов метода). Kafka/RabbitMQ добавляют сеть, персистентность и масштаб — но модель «продюсер → брокер → консьюмер» та же.

---

## 1. Message Queue — точка-точка

Очередь: продюсер кладёт сообщения, **один** консьюмер их забирает (FIFO). Отправитель и получатель развязаны во времени.

```java
class MessageQueue {
    private final Queue<String> q = new ArrayDeque<>();
    void send(String msg) { q.offer(msg); }          // продюсер
    String receive() { return q.poll(); }             // консьюмер (один)
}
```

```
   Producer --send--▶ [ msg msg msg ] --receive--▶ Consumer
```

---

## 2. Pub-Sub — публикация/подписка (топик)

Топик рассылает каждое событие **всем** подписчикам. Продюсер не знает, кто и сколько слушает.

```java
class Topic {
    private final List<Consumer<String>> subscribers = new ArrayList<>();
    void subscribe(Consumer<String> sub) { subscribers.add(sub); }
    void publish(String event) { subscribers.forEach(s -> s.accept(event)); }  // всем
}
```

```
   Publisher --publish--▶ Topic --▶ Subscriber A
                                 --▶ Subscriber B
                                 --▶ Subscriber C
```

> Очередь vs топик: очередь — «работу сделает один из»; топик — «уведомить всех заинтересованных». В Kafka это сочетается: топик с партициями + группы консьюмеров.

---

## 3. Event-Driven — реакция на события

Система строится вокруг **событий** (факты в прошедшем времени, тема 19): компоненты публикуют события и реагируют на чужие, а не вызывают друг друга напрямую. Это даёт слабую связанность и расширяемость (новый подписчик — без правок издателя, как Observer, тема 11).

---

## 4. Гарантии доставки

| Гарантия | Смысл | Цена |
|----------|-------|------|
| **at-most-once** | не более одного раза (возможна потеря) | просто, но теряет |
| **at-least-once** | хотя бы раз (возможны дубли) | надёжно, но нужны **идемпотентные** консьюмеры |
| **exactly-once** | ровно один раз | дорого/сложно, часто эмулируется at-least-once + дедуп |

На практике чаще всего **at-least-once + идемпотентность** (тема 21): консьюмер дедуплицирует по id сообщения.

```java
Set<String> processed = new HashSet<>();
void onMessage(String id, String payload) {
    if (!processed.add(id)) return;   // дубликат — пропускаем (идемпотентность)
    handle(payload);
}
```

---

## 5. Dead Letter Queue (DLQ)

«Ядовитое» сообщение, которое не удаётся обработать (ошибка, битые данные), нельзя крутить вечно — после N повторов его отправляют в **DLQ** для разбора, не блокируя остальную очередь.

```
   Очередь --▶ обработка --ошибка--▶ retry (до N раз) --всё ещё падает--▶ DLQ
```

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Консьюмер не идемпотентен при at-least-once | дубли эффектов (двойное списание) | дедуп по id сообщения |
| Нет DLQ | «ядовитое» сообщение блокирует очередь / бесконечный retry | DLQ после N повторов |
| Очередь вместо топика (и наоборот) | «уведомить всех» сделал один; «работу» сделали все | очередь — один из; топик — всем |
| Расчёт на порядок/мгновенность | события приходят с задержкой/переупорядоченно | проектируй под eventual consistency |
| Публикация события до фиксации состояния | подписчики видят то, чего ещё нет | публиковать после сохранения (тема 19) |
| Связанность через формат события | смена схемы ломает потребителей | версионируй события, не ломай контракт |
