# Тема 26. Рефакторинг legacy (capstone)

Финал раздела. Знать паттерны мало — в реальности чаще приходится **улучшать существующий «запущенный» код**, не останавливая систему и не ломая работающее. Эта тема — про безопасные техники крупного рефакторинга, которые сводят воедино весь раздел.

```
Strangler Fig       — постепенно «обвить и задушить» старое новым за фасадом
Branch by Abstraction — ввести абстракцию, под ней заменить реализацию
Feature Toggle      — переключатель старое/новое в рантайме
Parallel Run        — гонять старое и новое параллельно, сверять результаты
Anti-Corruption Layer — слой-переводчик между чужой/старой и новой моделью
```

> Главный принцип: **маленькие безопасные шаги под защитой тестов**, а не «большой взрыв» переписывания.

---

## 1. Strangler Fig — «фиговое удушение»

Имя — от фикуса-душителя, который обвивает дерево и постепенно его замещает. Новую реализацию вводят **за фасадом**, перенаправляя на неё функциональность по кусочку, пока старое не исчезнет.

```java
interface OrderApi { String process(String orderId); }

class StranglerFacade implements OrderApi {
    private final OrderApi legacy;
    private final OrderApi modern;
    private final Set<String> migrated;            // что уже «переехало» на новое
    public String process(String orderId) {
        return migrated.contains(orderId) ? modern.process(orderId)   // новое
                                          : legacy.process(orderId);  // пока старое
    }
}
```

Клиент видит один `OrderApi`; внутри трафик постепенно переключается на новую реализацию. Старое «отмирает», когда всё переехало.

---

## 2. Branch by Abstraction — ветка через абстракцию

Чтобы заменить реализацию безопасно: (1) вводим **абстракцию** (интерфейс) над текущим кодом, (2) переключаем клиентов на абстракцию (со старой реализацией), (3) пишем новую реализацию того же интерфейса, (4) подменяем. Всё — маленькими шагами, система всё время рабочая.

```java
interface PaymentProcessor { boolean pay(long amount); }
class LegacyPaymentProcessor  implements PaymentProcessor { ... }   // шаг 1–2
class ModernPaymentProcessor  implements PaymentProcessor { ... }   // шаг 3
// шаг 4: подменили реализацию в точке сборки — клиент не изменился
```

---

## 3. Feature Toggle — переключатель

Флаг (в конфиге/БД) выбирает, какая реализация активна. Позволяет включать новое постепенно, быстро откатываться, тестировать на части трафика.

```java
class ToggledService {
    private final Service legacy, modern;
    private final boolean useModern;            // флаг
    String run() { return useModern ? modern.run() : legacy.run(); }
}
```

> Не копи тумблеры бесконечно: после успешной миграции toggle и старую ветку удаляют (иначе — Lava Flow, тема 22).

---

## 4. Parallel Run — параллельный прогон

Перед переключением гоняют **оба** варианта на одних данных и **сверяют** результаты: новое не должно расходиться со старым. Расхождения логируют, источником истины пока остаётся старое.

```java
String legacyResult = legacy.run(input);
String modernResult = modern.run(input);
if (!legacyResult.equals(modernResult)) log("расхождение: " + input);
return legacyResult;                  // пока доверяем старому
```

## 5. Anti-Corruption Layer (ACL) — слой-переводчик

Чтобы старая/чужая модель не «протекала» в новый чистый домен, между ними ставят **переводчик**: он конвертирует данные чужой модели в понятия нового домена (и обратно). Новый код не зависит от формата старого.

```java
// чужая/старая модель: "ИВАНОВ;1990" → новый Customer(name, birthYear)
class LegacyCustomerAdapter {
    Customer translate(String legacyRecord) {
        String[] p = legacyRecord.split(";");
        return new Customer(normalize(p[0]), Integer.parseInt(p[1]));   // перевод в наш домен
    }
}
```

ACL — это Adapter (тема 07) на границе контекстов (тема 18), защищающий чистоту новой модели.

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| «Большой взрыв» переписывания | месяцы без релизов, всё ломается | маленькие шаги, Strangler/Branch by Abstraction |
| Рефакторинг без тестов | regression, «починили-сломали» | характеризующие тесты до изменений |
| Toggle навсегда | россыпь мёртвых веток (Lava Flow) | удалять toggle и старый код после миграции |
| Чужая модель протекла в домен | новый код завязан на формат старого | Anti-Corruption Layer |
| Нет сверки при миграции | новое тихо расходится со старым | Parallel Run перед переключением |
