# Тема 06. Prototype и Object Pool

Завершаем порождающие паттерны. **Prototype** создаёт объект копированием готового образца, **Object Pool** переиспользует «дорогие» объекты вместо создания новых.

```
Prototype    — новый объект = копия существующего (а не сборка с нуля)
Object Pool  — взять готовый из пула → попользовать → вернуть обратно
```

---

## 1. Prototype — клонирование

Когда создание объекта дорого или его конфигурация сложна, проще скопировать готовый образец и чуть подправить, чем собирать заново.

### Способ 1 — copy-конструктор (рекомендуемый в Java)

```java
class Document {
    private final String title;
    private final List<String> tags;

    Document(Document other) {                       // copy-конструктор
        this.title = other.title;
        this.tags  = new ArrayList<>(other.tags);    // КОПИЯ списка, не ссылка!
    }
}
Document copy = new Document(original);
```

Явно, безопасно, не требует `Cloneable`. Предпочтительный путь (Блох, *Effective Java*: «избегайте `clone()`»).

### Способ 2 — Cloneable / clone()

```java
class Point implements Cloneable {
    int x, y;
    @Override public Point clone() {
        try { return (Point) super.clone(); }        // поверхностная копия полей
        catch (CloneNotSupportedException e) { throw new AssertionError(e); }
    }
}
```

`super.clone()` делает **поверхностную** копию. Механизм исторический и с ловушками — знать нужно, но в новом коде чаще берут copy-конструктор.

### Поверхностная vs глубокая копия — ключевая ловушка

```
Поверхностная (shallow):           Глубокая (deep):
 copy.tags ──┐                       copy.tags ──▶ [новый список]
             ├──▶ [тот же список]     orig.tags ──▶ [исходный список]
 orig.tags ──┘                       (изменения не пересекаются)
```

При поверхностной копии оба объекта ссылаются на **один и тот же** вложенный список — изменишь у копии, изменится у оригинала. Для изменяемых вложенных объектов нужна **глубокая** копия (копировать и их).

## 2. Prototype-реестр

Хранилище готовых образцов; новый объект получают, клонируя образец по ключу:

```java
class ShapeRegistry {
    private final Map<String, Shape> prototypes = new HashMap<>();
    void register(String key, Shape proto) { prototypes.put(key, proto); }
    Shape create(String key) { return prototypes.get(key).copy(); }   // клон образца
}
```

Удобно для «шаблонов»: зарегистрировали типовой заказ/документ — дальше штампуем копии.

## 3. Object Pool — пул объектов

Создание/уничтожение «тяжёлых» объектов (соединения с БД, потоки, большие буферы) — дорого. Пул держит набор готовых объектов: клиент **берёт** (`acquire`), пользуется и **возвращает** (`release`).

```java
class ConnectionPool {
    private final Queue<Connection> free = new ArrayDeque<>();

    Connection acquire() {
        return free.isEmpty() ? new Connection() : free.poll();   // переиспользуем, если есть
    }
    void release(Connection c) {
        c.reset();                                                // очистить состояние перед возвратом!
        free.offer(c);
    }
}
```

```
   [ acquire ] ──▶ объект у клиента ──▶ [ release ] ──▶ обратно в пул
        ▲                                                     │
        └─────────────── переиспользование ───────────────────┘
```

**Важно:** при возврате объект нужно **сбрасывать** (`reset`), иначе чужое состояние «протечёт» к следующему пользователю. Часто задают **лимит** размера пула.

> Где встречается: пулы соединений (HikariCP), пулы потоков (`ThreadPoolExecutor`), пулы буферов. В прикладном коде свой пул пишут редко — обычно берут готовый.

---

## Когда что применять

| Паттерн | Решает | Признак |
|---------|--------|---------|
| Prototype | дорого/сложно создавать с нуля | есть «образец», от которого штампуют похожие |
| Prototype-реестр | нужны типовые заготовки | «шаблоны» объектов по ключу |
| Object Pool | создание объекта дорого, их переиспользуют | соединения, потоки, тяжёлые буферы |

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Поверхностная копия изменяемого поля | правка копии меняет оригинал | глубокая копия вложенных изменяемых объектов |
| `clone()` без копирования вложенных | то же самое, скрыто | copy-конструктор + явное копирование коллекций |
| Pool без `reset()` при возврате | чужие данные у следующего клиента | очищать состояние перед `release` |
| Pool без лимита | бесконтрольный рост памяти | ограничить размер, блокировать/отказывать при исчерпании |
| Объект не возвращён в пул | «утечка», пул пустеет | строгий контракт acquire/release (try-finally) |
| Pool ради лёгких объектов | переусложнение, GC и так быстр | пул только для реально дорогих объектов |
