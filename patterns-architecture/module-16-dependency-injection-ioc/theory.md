# Тема 16. DI и IoC

Слои (тема 15) и принцип DIP (тема 03) требуют, чтобы классы зависели от абстракций, а конкретику получали **извне**. Механизм, который это обеспечивает, — **Dependency Injection** (внедрение зависимостей), а более общий принцип — **Inversion of Control** (инверсия управления). Это «клей», которым собираются слоистые и гексагональные приложения.

> Здесь мы реализуем DI/IoC **на чистом Java** — чтобы понять механику. В реальных проектах эту работу берёт на себя контейнер (Spring), но делает он ровно то же самое.

```
DIP  (тема 03) — ПРИНЦИП: завись от абстракции
DI            — ТЕХНИКА: зависимость передаётся снаружи (через конструктор и т.п.)
IoC           — ИДЕЯ шире: управление созданием/связыванием отдано наружу (контейнеру)
```

---

## 1. Inversion of Control — кто управляет

Обычно объект сам создаёт свои зависимости (`new`). При **инверсии управления** это решение «переворачивается»: не объект тянет зависимости, а кто-то снаружи их ему даёт.

```java
// ❌ объект сам управляет созданием зависимости — жёсткая связь
class OrderService {
    private final EmailSender sender = new SmtpEmailSender();   // прибито намертво
}

// ✅ управление инвертировано: зависимость приходит снаружи
class OrderService {
    private final EmailSender sender;
    OrderService(EmailSender sender) { this.sender = sender; }   // кто-то снаружи решает, какой sender
}
```

DI — самый частый способ реализовать IoC.

---

## 2. Виды внедрения

### Constructor injection (рекомендуемый)

```java
class OrderService {
    private final OrderRepository repo;          // final → неизменяемо
    OrderService(OrderRepository repo) {         // обязательная зависимость
        this.repo = repo;
    }
}
```

**Плюсы:** зависимость обязательна и неизменяема (`final`), объект всегда в валидном состоянии, легко тестировать (передал мок в конструктор).

### Setter injection

```java
class OrderService {
    private OrderRepository repo;
    void setRepository(OrderRepository repo) { this.repo = repo; }  // можно забыть вызвать
}
```

Годится для **необязательных** зависимостей. Минус: объект может пожить без зависимости (NPE).

| | Constructor | Setter |
|---|-------------|--------|
| Обязательность | да | нет |
| Неизменяемость | да (`final`) | нет |
| Когда | основной выбор | опциональные зависимости |

> В Spring предпочтителен constructor injection (тот же принцип). Field injection (`@Autowired` на поле) считается плохой практикой: скрывает зависимости и мешает тестам.

## 3. IoC-контейнер — своими руками

**Контейнер** — это объект, который умеет создавать другие объекты и связывать их зависимости. Минимальная реализация — реестр «рецептов создания»:

```java
class Container {
    private final Map<Class<?>, Supplier<?>> recipes = new HashMap<>();
    <T> void register(Class<T> type, Supplier<T> recipe) { recipes.put(type, recipe); }
    @SuppressWarnings("unchecked")
    <T> T resolve(Class<T> type) { return (T) recipes.get(type).get(); }
}

Container c = new Container();
c.register(OrderRepository.class, InMemoryOrderRepository::new);
c.register(OrderService.class, () -> new OrderService(c.resolve(OrderRepository.class)));
OrderService service = c.resolve(OrderService.class);   // контейнер собрал граф зависимостей
```

Spring делает то же самое, но автоматически (читает аннотации/конфиг, строит граф бинов, внедряет). Понимая этот мини-контейнер, понимаешь и Spring «под капотом».

## 4. Service Locator — родственник и часто антипаттерн

Альтернатива DI: вместо того чтобы получать зависимость, объект **сам идёт за ней** в глобальный «локатор».

```java
class OrderService {
    void process() {
        EmailSender sender = ServiceLocator.get(EmailSender.class);  // объект сам просит
        ...
    }
}
```

**Почему обычно хуже DI:** зависимость спрятана (её не видно в сигнатуре), труднее тестировать (нужно подменять глобальный локатор), появляется неявная связь с локатором. DI делает зависимости явными. Service Locator оправдан редко (например, в plugin-системах).

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| `new ConcreteDep()` внутри класса | нельзя подменить/протестировать | внедряй через конструктор |
| Field injection / setter для обязательной зависимости | объект «полусобран», NPE | constructor injection + `final` |
| Глобальный Service Locator везде | скрытые зависимости, хрупкие тесты | явный DI |
| Контейнер протёк в доменную логику | домен зависит от Spring | внедряет «край» (конфиг), домен про контейнер не знает |
| Циклические зависимости A↔B | контейнер не может собрать граф | разорви цикл (выдели третий класс/интерфейс) |
