# Тема 20. Event Sourcing и CQRS

Domain Events (тема 19) — это факты. **Event Sourcing** доводит идею до предела: хранить **не текущее состояние, а поток событий**, и восстанавливать состояние, проигрывая их. **CQRS** разделяет модель **записи** (команды) и модель **чтения** (запросы). Часто идут вместе, но это разные приёмы.

```
Event Sourcing — источник истины = неизменяемый ЛОГ событий; состояние = свёртка лога
CQRS           — отдельная модель для команд (write) и для запросов (read/проекции)
```

---

## 1. Event Sourcing — состояние как свёртка событий

Обычно мы храним текущее состояние и затираем его при изменении. В Event Sourcing мы **дописываем события** в лог (append-only) и никогда их не меняем. Текущее состояние — результат «проигрывания» всех событий.

```java
// события (факты, неизменяемы)
record AccountOpened(String id) {}
record MoneyDeposited(long amount) {}
record MoneyWithdrawn(long amount) {}

class Account {
    private long balance = 0;
    void apply(Object event) {                        // как событие меняет состояние
        if (event instanceof MoneyDeposited e) balance += e.amount();
        else if (event instanceof MoneyWithdrawn e) balance -= e.amount();
    }
    static Account replay(List<Object> events) {      // восстановление из лога
        Account a = new Account();
        for (Object e : events) a.apply(e);
        return a;
    }
    long balance() { return balance; }
}
```

```
   Лог:  [Opened] → [Deposited 100] → [Withdrawn 30] → [Deposited 50]
   replay:   0    →      100        →      70        →     120   = текущий баланс
```

**Плюсы:** полная история (аудит «из коробки»), можно восстановить состояние на любой момент, «машина времени», легко строить новые представления. **Минусы:** сложнее, лог растёт (нужны снимки/snapshots), запросы по текущему состоянию неудобны → отсюда CQRS.

> Команда (намерение) — в повелительном наклонении: `Deposit`. Событие (факт) — в прошедшем: `MoneyDeposited`. Команда может быть отклонена; событие уже произошло.

---

## 2. CQRS — разделение команд и запросов

**Command Query Responsibility Segregation:** операции **изменения** (команды) и операции **чтения** (запросы) обслуживаются разными моделями.

```java
// сторона записи: принимает команды, меняет состояние/пишет события
class AccountCommandHandler {
    void handle(DepositCommand cmd) { /* проверки, запись события */ }
}
// сторона чтения: оптимизированная модель только для запросов
class AccountBalanceView {
    long balanceOf(String id) { /* читает из read-модели/проекции */ }
}
```

```
   Команды ──▶ Write-модель ──события──▶ Проекция ──▶ Read-модель ──▶ Запросы
   (меняют состояние)                  (строит представления)     (только чтение)
```

CQRS можно применять **без** Event Sourcing (просто разные модели чтения/записи), но вместе они особенно естественны: write-сторона пишет события, read-сторона строит из них проекции.

---

## 3. Projection — построение read-модели из событий

**Проекция** — обработчик, который из потока событий собирает удобное для чтения представление (таблицу, счётчик, отчёт). Можно построить **несколько** разных проекций из одного лога.

```java
class BalanceProjection {
    private long balance = 0;
    void on(Object event) {                           // «проецируем» событие в read-модель
        if (event instanceof MoneyDeposited e) balance += e.amount();
        if (event instanceof MoneyWithdrawn e) balance -= e.amount();
    }
    long balance() { return balance; }
}
// другую проекцию (например, число операций) строим из тех же событий, не трогая лог
```

Если нужно новое представление — пишем новую проекцию и проигрываем по ней весь лог. Лог при этом не меняется.

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Мутация/удаление событий | сломана история, «переписали прошлое» | лог append-only; ошибку исправляют новым событием (компенсация) |
| Событие в повелительном (`Deposit`) | путаница команда/факт | факт — прошедшее время (`MoneyDeposited`) |
| Лог растёт бесконечно | долгий replay | снимки (snapshots) состояния + события после снимка |
| Логика в проекции | бизнес-правила в read-модели | правила — на write-стороне; проекция только собирает представление |
| CQRS ради CRUD | переусложнение простого приложения | применяй, когда модели чтения/записи реально расходятся |
| Ожидание мгновенной согласованности | read-модель «отстаёт» от write | помни про eventual consistency между сторонами |
