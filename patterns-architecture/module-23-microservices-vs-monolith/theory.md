# Тема 23. Microservices vs Monolith

Предыдущие темы — про внутреннее устройство. Эта — про систему **целиком**: монолит, микросервисы и «золотая середина» — модульный монолит. Главный вопрос не «что модно», а **где провести границы** и **какой ценой** их пересекать.

```
Монолит             — одно приложение, один деплой
Модульный монолит   — один деплой, но чёткие модули с границами
Микросервисы        — много независимо разворачиваемых сервисов
```

> На чистом Java мы моделируем **границы и контракты** (модуль = пакет/набор классов с публичным API и скрытыми внутренностями; сервис = интерфейс-контракт). Реальные микросервисы добавляют сеть, отдельные деплои и БД — но принципы границ те же.

---

## 1. Спектр, а не выбор «или-или»

| | Монолит | Модульный монолит | Микросервисы |
|---|---------|-------------------|--------------|
| Деплой | один | один | много независимых |
| Границы | размытые | чёткие (модули) | жёсткие (сеть) |
| Связи | вызовы методов | вызовы через API модулей | сеть (REST/gRPC/события) |
| Транзакции | локальные (ACID) | локальные | распределённые (Saga, тема 21) |
| Сложность эксплуатации | низкая | низкая | высокая (сеть, мониторинг, версии) |
| Независимое масштабирование | нет | частично | да |

**Вывод:** микросервисы — не «лучше», а **другой набор trade-offs**. Начинать почти всегда стоит с (модульного) монолита; выделять сервисы — когда есть реальная причина (независимое масштабирование/деплой/команды).

---

## 2. Модуль: публичный API + скрытые внутренности

Модуль предоставляет **узкий публичный контракт** (интерфейс/фасад), а реализацию прячет. Другие модули зависят только от контракта.

```java
// публичный API модуля «Счета»
interface AccountModule {
    long balance(String accountId);
}
// внутренняя реализация — скрыта (другие модули её не видят/не трогают)
class AccountModuleImpl implements AccountModule { /* свои репозитории, сущности */ }
```

Это тот же приём, что инкапсуляция (тема 01) и порты (тема 17), но в масштабе **модуля**. Чёткие границы внутри монолита — заготовка для будущего выделения в сервис.

---

## 3. Контракт сервиса и способы связи

Граница между сервисами — это **контракт** (интерфейс). За ним может быть локальная реализация (модульный монолит) или сетевой вызов (микросервис) — клиент не меняется.

```java
interface PricingService { long priceOf(String sku); }            // контракт
class LocalPricingService  implements PricingService { ... }        // in-process
class RemotePricingClient  implements PricingService { ... }        // по сети (REST/gRPC-стаб)
```

**Способы межсервисной связи:**

| Способ | Когда | Минусы |
|--------|-------|--------|
| **REST** | простые синхронные запросы | связанность по доступности, latency |
| **gRPC** | быстрый типизированный RPC | бинарный контракт, инфраструктура |
| **Messaging/события** (тема 24) | асинхронность, развязка | eventual consistency, сложнее отладка |

---

## 4. Границы по бизнес-возможностям

Сервисы/модули режут **по бизнес-возможностям** (Bounded Context, тема 18), а не по техническим слоям. Признак хорошей границы: внутри — высокая связность, между — низкая связанность; модуль владеет своими данными.

```
   ✅ по возможностям: [Счета] [Платежи] [Уведомления]
   ❌ по слоям:        [Все контроллеры] [Все сервисы] [Все репозитории]
```

---

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Микросервисы «по моде» (Golden Hammer) | распределённый монолит, боль без выгоды | начни с модульного монолита |
| Распределённый монолит | сервисы нельзя деплоить/менять независимо | реальная развязка по данным и контрактам |
| Общая БД на все сервисы | скрытая жёсткая связь | каждый сервис владеет своими данными |
| Границы по слоям, а не возможностям | изменение фичи задевает все модули | резать по Bounded Context |
| Синхронные цепочки вызовов | каскадные отказы, latency | асинхронность/события там, где уместно |
| Нет контракта (лезут во внутренности) | модули жёстко сцеплены | общаться только через публичный API |
