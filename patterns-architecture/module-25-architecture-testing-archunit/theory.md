# Тема 25. Тестирование архитектуры

Мы выстроили правила: слои зависят сверху вниз (тема 15), ядро не зависит от адаптеров (тема 17), domain не импортирует Spring (тема 18). Но правила, которые **никто не проверяет**, со временем нарушаются. Эта тема — как сделать архитектурные правила **исполняемыми тестами**, чтобы нарушение ломало сборку.

```
Архитектурные тесты — проверяют СТРУКТУРУ кода, а не поведение:
  • направление зависимостей между слоями/пакетами
  • отсутствие циклов между модулями
  • соглашения именования и расположения
  • запрет нежелательных зависимостей
```

> В реальных проектах это делает **ArchUnit** (читает байт-код и проверяет правила в JUnit-тестах). Здесь мы реализуем мини-проверки **вручную на чистом Java**: представим архитектуру как граф зависимостей (`класс/пакет → на что зависит`) и напишем функции-правила. Принцип тот же — формализованное правило, дающее «да/нет».

---

## 1. Идея: архитектура как проверяемый граф

Зависимости можно представить данными: для каждого компонента — множество тех, от кого он зависит. Тогда правило — это функция над этим графом, возвращающая список **нарушений**.

```java
Map<String, Set<String>> deps = Map.of(
    "domain",  Set.of(),                  // домен ни от кого не зависит
    "service", Set.of("domain"),
    "web",     Set.of("service", "domain")
);
// правило: domain не должен зависеть ни от чего «верхнего»
boolean ok = deps.get("domain").isEmpty();
```

ArchUnit делает то же самое, но извлекает граф из байт-кода автоматически.

---

## 2. Типовые правила

### Направление зависимостей (слои)

> «`domain` не зависит от `infrastructure`/`web`». Нарушение — если в зависимостях domain есть запрещённый пакет.

### Отсутствие циклов

> Между модулями не должно быть циклических зависимостей (A→B→A). Цикл = сильная связанность, мешает менять и тестировать. Проверяется обходом графа (поиск цикла).

### Соглашения именования/расположения

> «Все классы-репозитории называются `*Repository`», «контроллеры лежат в пакете `web`». Нарушение — класс, не следующий конвенции.

### Запрет зависимости

> «Никто, кроме `persistence`, не импортирует JDBC», «`domain` не импортирует Spring». Нарушение — компонент с запрещённой зависимостью.

```
   Правило ──проверяет──▶ граф зависимостей ──▶ список нарушений (пусто = тест прошёл)
```

---

## 3. Снимок архитектуры в CI

Архитектурные тесты гоняются в CI вместе с обычными: нарушил правило — **красная сборка**, как при упавшем unit-тесте. Так структура не «протухает» со временем. Дополнительно используют **Testcontainers** (поднять реальную БД/брокер в Docker для интеграционных тестов) — но это про поведение, а не про структуру.

## ⚠️ Подводные камни

| Ошибка | Симптом | Лечение |
|--------|---------|---------|
| Правила есть в голове, но не в тестах | архитектура «плывёт» со временем | формализуй правила как тесты в CI |
| Слишком строгие правила | ложные срабатывания, тесты мешают | правила про реальные инварианты, не про вкусовщину |
| Проверяешь поведение вместо структуры | архитектурный тест дублирует unit | арх-тест — про зависимости/структуру |
| Цикл «не замечают» | связанность растёт незаметно | автоматическая проверка на циклы |
| Правило без понятного сообщения | непонятно, что и где нарушено | в нарушении — кто и какое правило нарушил |
