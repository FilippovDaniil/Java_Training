# Системный анализ — практический курс (middle+)

Четвёртый раздел репозитория `Java_Training` — рядом с [`../java-course`](../java-course/README.md), [`../patterns-architecture`](../patterns-architecture/README.md) и [`../algorithms-course`](../algorithms-course/README.md).

В отличие от трёх предыдущих, этот курс **не про написание кода**. Он про **работу системного аналитика уровня middle+**: проектирование архитектуры, сервисов и интеграций, моделирование процессов и данных, требования, API-контракты, нефункциональные требования, распределённые системы, безопасность и документация.

Построен по той же переносимой методике — [`../LEARNING-METHODOLOGY.md`](../LEARNING-METHODOLOGY.md):

> **Маленькая тема → большая теория → задачи без готовых решений → проверяемый результат → следующая тема.**

- **39 атомарных модулей**, сквозная нумерация `01 → 39` (профессия → требования → моделирование → данные → архитектура → интеграции → качество/поставка → capstone → бонус ИИ).
- На каждый модуль — **большая `theory.md`**, **10 задач** (`Task01 → Task10`, десятая — мини-проект) и **`questions.md` с 25 вопросами** для самопроверки (с ключом ответов).
- **Готовых решений нет:** условие + каркас артефакта + критерии приёмки. Решение пишет студент.
- **Два сквозных проекта** проходят через весь курс + новый домен в capstone (см. ниже).

> Статус генерации модулей (что готово, что нет) и детальный план — в [`PROGRESS.md`](PROGRESS.md). Это источник истины: при потере контекста работа продолжается оттуда.

---

## 🎯 Для кого и зачем

Цель курса: после прохождения можно обоснованно сказать — **«я погружён в системный анализ на уровне middle+»**. Курс собран на основе анализа трёх рыночных программ (Практикум-стиль, проф. переподготовка, компактный курс) с **добавлением слоя глубины middle+**, которого в них нет: измеримые NFR, распределённые паттерны (Saga, Outbox, идемпотентность, согласованность), contract-first и версионирование API, гарантии доставки сообщений и DLQ, безопасность интеграций (OAuth2/OIDC/JWT), масштабирование данных (репликация/шардинг/индексы), наблюдаемость, ADR.

Предполагается, что базовый junior-уровень (что такое требование, БД, HTTP) уже есть — курс быстро проходит азы и уходит в глубину проектирования.

```
system-analysis-course/

    |   +-- ...
    |   +-- Task10.md          # мини-проект темы (привязан к сквозному проекту)
```

Каждая задача — это **markdown-файл** с шапкой и местом для ответа (артефакта: текст требований, ER/BPMN/UML-описание, SQL, OpenAPI/JSON, таблица решений и т.д.). Готового ответа нет.

## 🧩 Анатомия задачи (методика, §3)

Шапка задачи в `TaskNN.md`:

```
# Задача NN — Модуль NN: <название>

## ЗАДАНИЕ
что нужно спроектировать/описать (1–3 предложения).

## ВХОДНЫЕ ДАННЫЕ
контекст: фрагмент домена, требования заказчика, схема, регламент.

## ТРЕБОВАНИЯ
ограничения и обязательные условия (нотация, формат, объём).

## ФОРМАТ РЕЗУЛЬТАТА
в каком виде сдать (таблица / диаграмма в нотации X / SQL / OpenAPI-фрагмент / спецификация).

## КРИТЕРИИ ПРИЁМКИ
объективный чек-лист «сделано/нет» — это и есть проверка вместо компиляции.

## ПОДСКАЗКА
направление, но НЕ ответ.

---
<место для твоего решения>
```

**Правило:** готового решения нет. Есть условие, каркас и критерии, по которым студент сам проверяет себя.

### Две оси нарастающей сложности (методика, §4)

```
Внутри темы (01 → 10):
  01–03  базовый приём в нотации/инструменте («повтори по образцу», разбор готового артефакта)
  04–07  комбинирование, нестандартные случаи, поиск дефектов в чужих артефактах, сравнение
  08–09  сложные/спорные кейсы, trade-offs, конфликты
  10     мини-проект — синтез темы в сквозной проект

Между модулями (01 → 39):
  Часть 1–2   профессия, процессы, требования
  Часть 3–4   моделирование процессов и данных
  Часть 5–6   архитектура, распределённые системы, интеграции и API
  Часть 7–9   качество/поставка/документация, capstone, ИИ
```

## ✅ Объективная проверяемость без кода (методика, §10, §12)

Вместо «компилируется/нет» — критерии под тип артефакта:

| Тип задачи | Чем проверяется |
|------------|-----------------|
| SQL | схема БД + ожидаемый результат → запрос реально выполним (PostgreSQL/SQLite/онлайн-песочница) |
| OpenAPI / JSON / XML | валидация линтером/схемой (Swagger Editor, JSON Schema validator) |
| BPMN / UML / ER / DFD | чек-лист обязательных элементов + правила корректности нотации |
| Требования / ТЗ / NFR | чек-лист «КРИТЕРИИ ПРИЁМКИ»: измеримость, полнота, непротиворечивость, трассируемость |
| Решения с trade-offs | обоснование по заданным критериям + сравнительная таблица |
| 25 вопросов модуля | ключ с ответами в конце `questions.md` |

## 🧵 Сквозные проекты

Изолированные темы собираются в реальные системы (методика, §5). Мини-проект (задача 10) каждого модуля относится к одному из сквозных проектов.

| Проект | Домен | Роль |
|--------|-------|------|
| **FoodExpress** | агрегатор доставки еды: клиенты, рестораны, курьеры, заказы, оплата, уведомления | основная линия — богат на процессы, данные, асинхронные интеграции, нагрузку |
| **MedConnect** | телемедицина: записи, врачи, видеоконсультации, рецепты | контраст — регуляторика (152-ФЗ), безопасность, иной домен |
| **Capstone (новый домен)** | предлагается «Умный дом / IoT-платформа» (уточняется при подходе к модулю 38) | проверка самостоятельного прохождения полного цикла «с нуля» |

## 🚀 Как учиться

1. **Идите сверху вниз** — модули выстроены по нарастанию сложности.
2. **Сначала теория** (`theory.md`): разберите примеры, схемы, «подводные камни».
3. **Потом практика** (`Task01 → Task10`): проектируйте сами, сверяясь с критериями приёмки, а не с ответом.
4. **Ведите сквозные проекты** (FoodExpress, MedConnect) — к финалу складывается полный аналитический пакет.
5. **Проверяйте себя 25 вопросами** модуля перед переходом дальше.
6. **Capstone (38)** — пройдите полный цикл самостоятельно на новом домене.
