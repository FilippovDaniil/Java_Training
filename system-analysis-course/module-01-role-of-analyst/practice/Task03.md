# Задача 03 — Модуль 01: Роль системного аналитика

## ЗАДАНИЕ
Постройте RACI-матрицу для разработки одной фичи в проекте **FoodExpress** (агрегатор доставки еды).

## ВХОДНЫЕ ДАННЫЕ
Фича: «Промокоды на заказ» — клиент вводит промокод при оформлении, система проверяет его валидность и применяет скидку.
Роли в команде: **СА, PO, Архитектор, QA, Разработчик, Маркетолог** (владелец промо-кампаний).
Активности по фиче:
1. Определить бизнес-правила промокодов (типы скидок, ограничения, сроки).
2. Приоритизировать фичу в бэклоге.
3. Спроектировать модель данных промокодов.
4. Спроектировать API применения промокода и его влияние на расчёт суммы.
5. Решить, где хранить и как валидировать промокоды (производительность при пиковой нагрузке).
6. Определить критерии приёмки.
7. Написать тест-кейсы (включая негативные: истёкший/несуществующий код).
8. Реализовать логику в коде.
9. Принять готовую фичу.

## ТРЕБОВАНИЯ
- Для каждой активности проставьте R, A, C, I по всем ролям.
- **Ровно один A** на активность.
- Поясните 3 самых спорных назначения (почему именно так).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: строки — активности, столбцы — роли, в ячейках R/A/C/I. Ниже — пояснения к 3 назначениям.

## КРИТЕРИИ ПРИЁМКИ
- [ ] В каждой строке ровно один A и хотя бы один R.
- [ ] Бизнес-правила промокодов: Маркетолог/PO задаёт, СА формализует (R у СА, C у Маркетолога).
- [ ] Решение о хранении/валидации под нагрузкой: A у архитектора, C у СА.
- [ ] Приоритизация и приёмка: A у PO.
- [ ] Нет активности, где СА стоит A за техническое архитектурное решение.
- [ ] Даны пояснения к 3 назначениям.

## ПОДСКАЗКА
Accountable — тот, с кого спросят за результат, а не тот, кто делает руками. R и A могут быть на одной роли (СА сам делает и сам отвечает за спецификацию).
