# Задача 06 — Модуль 01: Роль системного аналитика

## ЗАДАНИЕ
Составьте чек-лист **Definition of Ready (DoR)** — критериев, при которых задача от системного аналитика готова к тому, чтобы команда разработки взяла её в работу.

## ВХОДНЫЕ ДАННЫЕ
Типичные проблемы, из-за которых задачи «отскакивают» обратно к аналитику:
- разработчик не понимает, что считать успешным результатом;
- не описаны ошибочные сценарии (нет данных, нет прав, превышен лимит);
- неясно, какие поля приходят и в каком формате;
- нет критериев приёмки — QA не знает, что проверять;
- задача слишком крупная, её нельзя сделать за спринт;
- непонятно, как фича влияет на смежные модули.

## ТРЕБОВАНИЯ
- Чек-лист из 8–12 пунктов, каждый — проверяемый (можно ответить да/нет).
- Сгруппируйте пункты по категориям (например: понятность, полнота, проверяемость, размер, зависимости).
- Для 3 пунктов приведите контрпример «как выглядит НЕ готовая задача».

## ФОРМАТ РЕЗУЛЬТАТА
Сгруппированный чек-лист (Markdown-список с `[ ]`). Ниже — 3 контрпримера.

## КРИТЕРИИ ПРИЁМКИ
- [ ] 8–12 пунктов, все проверяемы (нет «задача должна быть хорошей»).
- [ ] Есть пункты про критерии приёмки, ошибочные сценарии, формат данных, размер задачи, зависимости.
- [ ] Пункты сгруппированы по категориям.
- [ ] Приведены 3 контрпримера НЕ готовой задачи.
- [ ] Чек-лист применим к реальной задаче (проверьте на любой фиче из задач 03–04).

## ПОДСКАЗКА
DoR — это «входной контроль» для команды. Хороший критерий не зависит от того, кто его проверяет: ответ да/нет одинаков у СА, разработчика и QA.
