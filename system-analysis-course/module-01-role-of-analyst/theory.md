# Модуль 01 — Роль системного аналитика

## Зачем этот модуль

Прежде чем проектировать требования, данные и API, нужно понять **свою роль**: что от вас ждут, какие артефакты вы производите, где проходят границы с соседними ролями и за что вы отвечаете. На собеседовании middle+ первый блок вопросов — почти всегда про роль, разграничение с BA/PO/архитектором и про то, какие артефакты вы делаете на каждом этапе. Этот модуль закрывает именно это.

---

## 1. Кто такой системный аналитик

**Системный аналитик (СА)** — это специалист, который превращает **потребности бизнеса** в **точное, проверяемое и реализуемое описание системы** для команды разработки.

Короткая формула роли:

```
Бизнес говорит "ЧТО и ЗАЧЕМ"  →  СА формулирует "КАК это должна делать система"  →  Разработка строит
```

СА — это «переводчик» и «инженер требований» одновременно. Он:
- понимает бизнес-домен и проблему пользователя;
- понимает, как устроены ИТ-системы (БД, API, интеграции, архитектура) — настолько, чтобы спроектировать решение и говорить с разработкой на одном языке;
- закрывает разрыв между этими двумя мирами так, чтобы получившееся ПО решало реальную задачу.

> Ключевое отличие от бизнес-аналитика: СА **спускается до уровня системы** — сущности данных, контракты API, статусы, интеграции, нефункциональные требования. Он отвечает на вопрос «как это технически устроено и описано», а не только «зачем бизнесу это нужно».

## 2. Что делает системный аналитик (основные активности)

| Группа | Что конкретно делает |
|--------|----------------------|
| **Выявление (elicitation)** | интервью со стейкхолдерами, воркшопы, анализ документов/регламентов/НПА, изучение текущих систем |
| **Анализ** | структурирование требований, поиск противоречий и пробелов, выделение бизнес-сущностей, моделирование процессов |
| **Проектирование решения** | модель данных (ER), use cases / user stories, проектирование API и интеграций, диаграммы (BPMN, UML, sequence) |
| **Спецификация** | SRS / ТЗ, описание функциональных и нефункциональных требований, критерии приёмки |
| **Сопровождение разработки** | ответы на вопросы команды, уточнение требований, ведение изменений, участие в груминге/планировании |
| **Проверка** | участие в приёмке, ревью тест-кейсов, демонстрация заказчику, трассировка «требование → реализация → тест» |

СА участвует **на всём жизненном цикле**, а не только «в начале». Его вовлечённость максимальна на этапах выявления/проектирования и не падает до нуля на разработке и поддержке.

## 3. Артефакты СА по фазам жизненного цикла

Это таблица, которую полезно держать в голове — её вариации спрашивают на собеседовании.

| Фаза | Типичные артефакты СА |
|------|------------------------|
| **Инициация / pre-sale** | бизнес-цели, границы (scope), карта стейкхолдеров, верхнеуровневые требования, BMC, оценка осуществимости |
| **Выявление требований** | протоколы интервью, глоссарий (ubiquitous language), список требований, контекстная диаграмма |
| **Анализ и моделирование** | модель процессов (BPMN), use cases / user stories, модель данных (ER), модель состояний, прототипы |
| **Проектирование решения** | спецификация API (OpenAPI), sequence-диаграммы, описание интеграций, NFR, ADR (совместно с архитектором) |
| **Спецификация** | SRS / ТЗ (ГОСТ 34.602), критерии приёмки, матрица трассировки |
| **Разработка** | уточнения, ответы на вопросы, декомпозиция задач, поддержка бэклога |
| **Тестирование / приёмка** | ревью тест-кейсов, участие в UAT, проверка соответствия требованиям |
| **Внедрение / сопровождение** | план миграции, пользовательская документация, обработка change requests |

> **Подсказка для запоминания:** артефакт всегда отвечает на вопрос «кому он нужен и какое решение помогает принять». Глоссарий — чтобы все говорили одинаково; ER — чтобы разработка спроектировала БД; критерии приёмки — чтобы QA знал, что проверять, а заказчик — что принимать.

## 4. Компетенции системного аналитика

### Hard skills (твёрдые)

| Область | Что нужно уметь |
|---------|------------------|
| Требования | уровни/типы требований, техники выявления, спецификация, управление изменениями |
| Моделирование | UML (use case, class, activity, state, sequence), BPMN, DFD, ER |
| Данные | реляционные модели, нормализация, SQL, базовое понимание NoSQL |
| Интеграции / API | REST, форматы (JSON/XML), OpenAPI, очереди/брокеры, синхрон/асинхрон |
| Архитектура | стили (монолит/микросервисы), клиент-сервер, NFR, основы распределённых систем |
| Качество | тест-дизайн, критерии приёмки, трассировка |

### Soft skills (гибкие)

| Навык | Зачем |
|-------|-------|
| Коммуникация | вытащить из заказчика реальную потребность, согласовать спорное |
| Структурное мышление | разложить хаос на части, увидеть пробелы и противоречия |
| Критическое мышление | задавать «почему», не принимать «потому что так надо» |
| Управление конфликтами | стейкхолдеры хотят разного — нужно свести к решению |
| Письменная ясность | требование, которое нельзя понять двояко |

> Middle+ отличается от junior не объёмом знаний, а **самостоятельностью и системностью**: middle+ сам выявляет проблему, проектирует решение целиком (данные + процессы + интеграции + NFR), берёт ответственность за качество требований и умеет защищать решение перед бизнесом и архитектором.

## 5. Соседние роли и границы

Самая частая путаница на собеседовании — границы между ролями. Разберём.

| Роль | За что отвечает | Чем отличается от СА |
|------|-----------------|----------------------|
| **Бизнес-аналитик (BA)** | бизнес-процессы, бизнес-проблема, «зачем», окупаемость | смотрит на бизнес, обычно не спускается до сущностей данных, API, статусов |
| **Product Owner (PO)** | ценность продукта, приоритеты, бэклог, решение «что делаем» | владеет приоритетами и видением; СА детализирует и проектирует «как» |
| **Scrum Master (SM)** | процесс, устранение препятствий команды | про процесс, не про содержание требований |
| **Архитектор** | технические решения системного уровня: стиль, технологии, нефункциональные качества | принимает финальное техническое решение; СА проектирует логику и контракты, согласуя с архитектором |
| **QA / тестировщик** | проверка соответствия, поиск дефектов | проверяет то, что СА специфицировал; СА даёт критерии приёмки, QA — тест-кейсы |
| **Разработчик** | реализация в коде | реализует по спецификации; СА не пишет код, но проектирует, что должно получиться |
| **UX/UI-дизайнер** | интерфейс, удобство | проектирует экраны; СА даёт требования к поведению и данным за интерфейсом |

```
        Бизнес-цель / проблема
                |
   BA / PO  ->  ЗАЧЕМ, ЧТО, приоритеты
                |
        СА  ->  КАК система это делает: требования, данные, процессы, API, NFR
              /        \
   Архитектор          QA            Разработчик
   (как технически      (как проверим)  (реализация)
    в целом)
```

> На практике границы **подвижны** и зависят от компании: в маленькой команде СА может закрывать часть BA и часть QA; в большой — роли строго разделены. На собесе хороший ответ: «формально граница такая-то, но на проекте она зависит от состава команды — я уточняю зоны ответственности в начале».

## 6. RACI — кто за что отвечает

**RACI** — матрица распределения ответственности за задачу/артефакт:

| Буква | Значение | Смысл |
|-------|----------|-------|
| **R** — Responsible | исполнитель | кто делает работу |
| **A** — Accountable | ответственный | кто отвечает за результат (всегда ровно один!) |
| **C** — Consulted | консультируемый | с кем советуются (двусторонняя связь) |
| **I** — Informed | информируемый | кого уведомляют (односторонняя связь) |

Пример RACI для артефакта «Спецификация требований к фиче»:

| Активность | СА | PO | Архитектор | QA | Dev |
|------------|----|----|-----------|----|----|
| Сбор и анализ требований | R | A | C | I | I |
| Проектирование модели данных | R | I | C | I | C |
| Определение критериев приёмки | R | A | I | C | I |
| Решение по архитектуре | C | I | A/R | I | C |

> Зачем СА строит RACI: чтобы убрать вопрос «а кто это решает / кто за это отвечает». Самая частая ошибка — два Accountable или ни одного.

## 7. СА в разных контекстах

Роль выглядит по-разному в зависимости от модели разработки (детали — в [модуле 02](../module-02-sdlc-models/theory.md)):

| Контекст | Как работает СА |
|----------|------------------|
| **Waterfall / ГОСТ** | большой объём документации заранее (ТЗ), требования фиксируются до разработки, СА — «держатель ТЗ» |
| **Agile / Scrum продуктовый** | требования итеративно, СА работает с бэклогом, user stories, участвует в груминге, документация «достаточная», тесная связь с PO и командой |
| **Аутсорс / проектный** | акцент на согласованных артефактах (ТЗ, приёмка), формальная фиксация изменений |

---

## ⚠️ Подводные камни

- **«Аналитик-секретарь».** СА просто записывает то, что сказал заказчик, не анализируя. Признак: требования = дословный пересказ хотелок без выявления противоречий и пробелов. Лечение: задавать «почему», «а что если», проверять на полноту.
- **«Аналитик — узкое горло».** Вся коммуникация идёт только через СА, команда без него парализована. Лечение: фиксировать решения письменно, делать артефакты самодостаточными.
- **Размывание роли.** СА начинает решать за архитектора (выбор технологий) или за PO (приоритеты), не имея на это полномочий. Лечение: знать границы, выносить решения тем, кто за них Accountable.
- **Аналитический паралич.** Бесконечное уточнение и моделирование без выхода в реализацию. Лечение: достаточный уровень детализации под контекст (Agile ≠ ГОСТ).
- **Требование без проверяемости.** «Система должна быть удобной» — нельзя проверить. Лечение: каждое требование формулировать так, чтобы можно было ответить «да/нет, выполнено».
