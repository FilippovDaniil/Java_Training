# Задача 02 — Модуль 02: ЖЦ ПО и модели разработки

## ЗАДАНИЕ
Для трёх проектов выберите модель разработки и обоснуйте выбор по факторам контекста.

## ВХОДНЫЕ ДАННЫЕ
**Проект 1 — Госзаказ.** Система учёта обращений граждан для ведомства. Требования зафиксированы в техзадании, приёмка по ГОСТ, бюджет и сроки утверждены, изменения — через допсоглашения.

**Проект 2 — Стартап.** Мобильное приложение для поиска репетиторов. Бизнес-модель не проверена, требования будут меняться по мере обратной связи от первых пользователей, нужно быстро выпустить MVP и учиться на нём.

**Проект 3 — Банк.** Новый модуль расчёта процентов по вкладам в действующей банковской системе. Высокая цена ошибки, строгая регуляторика, но есть давление выпускать улучшения регулярно.

## ТРЕБОВАНИЯ
- Для каждого проекта выберите модель (Waterfall, V, итеративная/инкрементная, Scrum, Kanban или обоснованный гибрид).
- Обоснуйте по факторам: стабильность требований, регуляторика, цена ошибки, потребность в ранней обратной связи.
- Укажите, как при выбранной модели работает СА (момент фиксации требований, объём документации).

## ФОРМАТ РЕЗУЛЬТАТА
Для каждого проекта блок: `Модель → Обоснование по 4 факторам → Как работает СА`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проект 1 → Waterfall/V (стабильные требования, ГОСТ-приёмка).
- [ ] Проект 2 → Agile/Scrum (меняющиеся требования, MVP, ранняя обратная связь).
- [ ] Проект 3 → обоснованный гибрид или V с регулярными релизами (высокая цена ошибки + давление на частоту).
- [ ] Каждый выбор обоснован по всем 4 факторам, а не «потому что модно».
- [ ] Описана работа СА в каждом случае.

## ПОДСКАЗКА
Нет «правильной» модели в вакууме — есть подходящая под контекст. Самый слабый ответ — «везде Scrum, потому что Agile хорошо». Сильный — связать выбор с конкретными факторами проекта.
