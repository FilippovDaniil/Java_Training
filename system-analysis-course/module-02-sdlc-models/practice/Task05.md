# Задача 05 — Модуль 02: ЖЦ ПО и модели разработки

## ЗАДАНИЕ
Для потока задач технической поддержки выберите между Scrum и Kanban и обоснуйте выбор. Спроектируйте базовую доску с лимитами WIP.

## ВХОДНЫЕ ДАННЫЕ
Команда сопровождения MedConnect получает разнородный поток:
- инциденты («не приходит SMS с кодом» — срочно);
- заявки на доработки («добавить экспорт в PDF» — несрочно);
- запросы на консультацию от других команд;
- баги разной критичности.
Поток непредсказуем по объёму и приоритету, появляется в любой момент, приоритеты меняются в течение дня.

## ТРЕБОВАНИЯ
- Выберите Scrum или Kanban, обоснуйте по характеру потока.
- Объясните, почему фиксированные спринты плохо подходят для такого потока.
- Спроектируйте доску: колонки + предложите лимиты WIP.
- Предложите, как обрабатывать срочные инциденты, не ломая поток.

## ФОРМАТ РЕЗУЛЬТАТА
Обоснование выбора + схема доски (колонки и WIP-лимиты) + правило для срочных задач.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбран Kanban с обоснованием (непрерывный непредсказуемый поток, меняющиеся приоритеты).
- [ ] Объяснено, почему спринты Scrum плохо ложатся на инциденты.
- [ ] Доска содержит осмысленные колонки и WIP-лимиты.
- [ ] Предложен механизм для срочных задач (например, expedite-полоса/класс обслуживания).
- [ ] Упомянута связь WIP → lead time (закон Литтла).

## ПОДСКАЗКА
Scrum любит стабильный план на спринт; поток поддержки этот план постоянно ломает. Kanban с лимитами WIP и классами обслуживания (срочный/обычный) подходит лучше. Лимит WIP не даёт команде распыляться на всё сразу.
