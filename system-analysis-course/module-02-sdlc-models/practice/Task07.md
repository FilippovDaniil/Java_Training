# Задача 07 — Модуль 02: ЖЦ ПО и модели разработки

## ЗАДАНИЕ
Проект ведётся по Waterfall, но требования начали меняться. Определите риски и предложите меры их снижения, не ломая каркас модели.

## ВХОДНЫЕ ДАННЫЕ
Проект автоматизации документооборота ведётся по Waterfall (контракт с фиксированным ТЗ). Через 4 месяца, на этапе разработки, заказчик понял, что часть процессов описана неверно, а законодательство изменилось — нужны правки в уже «замороженных» требованиях. До сдачи 2 месяца.

## ТРЕБОВАНИЯ
- Определите риски, которые создаёт изменение требований на поздней фазе Waterfall.
- Предложите меры снижения: процессные (как обработать изменения) и содержательные (что сделать с требованиями).
- Оцените, какие элементы Agile можно «подмешать», не разрушая контрактную природу проекта.
- Сформулируйте, как можно было бы снизить эти риски **изначально** при выборе модели.

## ФОРМАТ РЕЗУЛЬТАТА
Блоки: `Риски → Меры снижения (процесс + содержание) → Элементы Agile → Что сделать в следующий раз`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Названы риски: срыв срока, рост бюджета, рассогласование ТЗ и реальности, конфликт по контракту.
- [ ] Предложен формальный процесс управления изменениями (импакт-анализ, допсоглашение, переприоритизация).
- [ ] Предложены элементы Agile (промежуточные демо, инкрементная сдача), совместимые с контрактом.
- [ ] Есть вывод: при нестабильных требованиях изначально стоило выбрать итеративную модель.
- [ ] Меры конкретны и реализуемы за оставшееся время.

## ПОДСКАЗКА
Главная проблема Waterfall — поздняя обратная связь и дорогие изменения. Нельзя «отменить» контракт, но можно ввести управляемый процесс изменений и промежуточные демонстрации, чтобы ловить расхождения раньше.
