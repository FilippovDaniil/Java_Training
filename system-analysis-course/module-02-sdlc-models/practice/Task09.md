# Задача 09 — Модуль 02: ЖЦ ПО и модели разработки

## ЗАДАНИЕ
Проинтерпретируйте метрики потока разработки: что они говорят о процессе, где проблема и что предложить.

## ВХОДНЫЕ ДАННЫЕ
Команда работает по Kanban. За квартал собраны данные:
- среднее **lead time** = 18 дней, среднее **cycle time** = 4 дня;
- среднее число задач в работе (**WIP**) = 25 при команде из 5 человек;
- **throughput** = 10 задач/неделя;
- большинство задач подолгу «висят» в колонке To Do и в колонке Code Review.

## ТРЕБОВАНИЯ
- Объясните разницу между lead time и cycle time и что означает разрыв 18 vs 4 дня.
- Оцените, адекватен ли WIP=25 для команды из 5 человек, со ссылкой на закон Литтла.
- Определите узкие места по описанию (где задачи «висят»).
- Предложите 3 конкретные меры улучшения потока.

## ФОРМАТ РЕЗУЛЬТАТА
Анализ по пунктам + 3 меры с ожидаемым эффектом.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснён разрыв lead/cycle time: задачи долго ждут начала работы (очередь), а не долго делаются.
- [ ] Дана оценка WIP с использованием связи `lead time ≈ WIP / throughput` (25/10 ≈ 2.5 недели ≈ 18 дней — сходится).
- [ ] Узкие места определены (To Do — переполнен бэклог/нет лимита; Code Review — бутылочное горло).
- [ ] Предложены меры: ограничить WIP, разгрузить review, приоритизация очереди.
- [ ] Меры связаны с метриками (как именно они снизят lead time).

## ПОДСКАЗКА
Большой разрыв lead/cycle time почти всегда = задачи долго **ждут**, а не долго **делаются**. Лечится ограничением WIP и расшивкой бутылочного горла (здесь — review), а не «работать быстрее».
