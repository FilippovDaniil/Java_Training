# Задача 10 (мини-проект) — Модуль 02: ЖЦ ПО и модели разработки

## ЗАДАНИЕ
Выберите модель разработки и спроектируйте рабочий процесс для телемед-платформы **MedConnect**. Учтите, что регуляторика влияет на выбор. Это синтез всей теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** — платформа телемедицины: онлайн-запись к врачу, видеоконсультации, электронные рецепты, хранение медицинских данных. Контекст:
- работа с персональными и медицинскими данными → требования 152-ФЗ и отраслевой регуляторики (высокая цена ошибки, аудит);
- продукт новый, часть требований будет уточняться по обратной связи врачей и пациентов;
- бизнес хочет регулярно выпускать улучшения, но критичные модули (рецепты, данные пациентов) требуют строгой проверки;
- команда: PO, СА, архитектор, 5 разработчиков, 2 QA, специалист по ИБ.

## ТРЕБОВАНИЯ
Подготовьте решение, включающее:
1. **Выбор модели** (или обоснованный гибрид) с привязкой к 4 факторам: стабильность требований, регуляторика, цена ошибки, потребность в ранней обратной связи.
2. **Дифференциацию по типам модулей**: как обращаться с критичными (рецепты, ПДн) и с некритичными (профиль, отзывы) — можно ли единым процессом.
3. **Процесс работы СА**: какие артефакты, в какой момент фиксируются, объём документации, как обрабатываются изменения.
4. **Обеспечение проверяемости и аудита**: как регуляторные требования встраиваются в процесс (где критерии приёмки, где участвует ИБ).
5. **Риски выбранной модели** и меры их снижения.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный документ из 5 разделов. Выбор модели — с явным обоснованием по 4 факторам (можно таблицей).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбор модели обоснован по всем 4 факторам, а не лозунгом.
- [ ] Учтено противоречие: гибкость для нового продукта vs строгость для регулируемых данных → обоснован гибрид/дифференциация.
- [ ] Критичные и некритичные модули обработаны по-разному с объяснением.
- [ ] Описан процесс СА: момент фиксации требований, объём документации, обработка изменений.
- [ ] Регуляторика встроена в процесс (критерии приёмки, участие ИБ, аудируемость), а не упомянута для галочки.
- [ ] Указаны ≥3 риска с мерами снижения.

## ПОДСКАЗКА
Чистый Waterfall негибок для нового продукта; чистый Scrum рискован для регулируемых данных. Реалистичный ответ — гибрид: строгий V-подход с полной документацией и участием ИБ для критичных модулей (рецепты, ПДн) и более гибкий итеративный процесс для некритичных. Свяжите это с разграничением из задачи 02 и гибридом из задачи 08.
