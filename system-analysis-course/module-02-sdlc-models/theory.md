# Модуль 02 — Жизненный цикл ПО и модели разработки

## Зачем этот модуль

В [модуле 01](../module-01-role-of-analyst/theory.md) вы увидели, что артефакты СА появляются на разных фазах. Теперь разберём сами эти фазы (жизненный цикл ПО) и **модели разработки** — разные способы пройти эти фазы. От выбранной модели зависит, **когда** фиксируются требования, **сколько** документации нужно и **как** работает СА. На собеседовании middle+ обязательно спрашивают разницу между Waterfall/V/итеративной/Agile и критерии выбора.

---

## 1. Жизненный цикл ПО (SDLC)

**Жизненный цикл ПО (Software Development Life Cycle)** — последовательность фаз от идеи до вывода системы из эксплуатации.

Классические фазы:

```
Анализ        →  Проектирование  →  Разработка  →  Тестирование  →  Внедрение  →  Сопровождение
требований                                                                          (и вывод из
                                                                                     эксплуатации)
```

| Фаза | Что происходит | Где СА |
|------|----------------|--------|
| **Анализ требований** | выявление и формализация потребностей | основная роль |
| **Проектирование** | архитектура, модель данных, контракты | проектирует логику; архитектор — техрешения |
| **Разработка** | написание кода | уточнения, ведение изменений |
| **Тестирование** | проверка соответствия | критерии приёмки, ревью тестов, UAT |
| **Внедрение** | развёртывание, миграция, обучение | план миграции, документация |
| **Сопровождение** | поддержка, доработки, исправления | обработка change requests |

> **Модель разработки** — это способ организовать прохождение этих фаз: один раз подряд (Waterfall), циклами (итеративная), по частям (инкрементная), гибко (Agile). Сами фазы при этом не исчезают — меняется их **порядок, повторяемость и момент фиксации**.

---

## 2. Waterfall (каскадная модель)

Фазы идут строго **последовательно**, каждая начинается после полного завершения предыдущей. Возврат назад дорог.

```
Анализ ──► Проектирование ──► Разработка ──► Тестирование ──► Внедрение
(каждая фаза завершается полностью, прежде чем начинается следующая)
```

| Плюсы | Минусы |
|-------|--------|
| предсказуемость, чёткие этапы и документы | поздняя обратная связь — ошибки видны в конце |
| удобно для фиксированного контракта/ГОСТ | плохо переносит изменения требований |
| легко планировать бюджет и сроки заранее | работающий продукт — только в самом конце |

**Когда применять:** требования стабильны и хорошо известны заранее; жёсткий контракт/регуляторика (госзаказ); цена ошибки в проде очень высока. **Роль СА:** большой объём требований и ТЗ заранее, до начала разработки.

---

## 3. V-модель

Развитие Waterfall: каждой фазе проектирования соответствует **уровень тестирования**. Левая ветка (что строим) зеркальна правой (как проверяем).

```
Требования (бизнес) ───────────────► Приёмочное тестирование (UAT)
   Системные требования ───────────► Системное тестирование
      Архитектура ─────────────────► Интеграционное тестирование
         Детальный дизайн ─────────► Модульное (unit) тестирование
                        \
                         Кодирование
```

Идея: тесты **планируются вместе с требованиями**, а не после кода. Требование без способа проверки — дефект уже на левой ветке.

**Когда применять:** там же, где Waterfall, но с высокими требованиями к качеству/проверяемости (медицина, банки, госсектор). **Роль СА:** формулирует требования сразу с критериями проверки; бизнес-требования ↔ приёмочные тесты.

---

## 4. Итеративная и инкрементная модели

Их часто путают — разберём разницу.

- **Инкрементная:** продукт строится **по частям** (инкрементам). Каждый инкремент добавляет новую готовую функциональность. Аналогия: дом строят комната за комнатой.
- **Итеративная:** продукт **улучшается циклами** — сначала грубая версия целиком, затем её дорабатывают. Аналогия: картину рисуют от наброска к деталям.

```
Инкрементная:  [часть1] + [часть2] + [часть3]  = растёт вширь
Итеративная:   [грубо всё] → [лучше] → [ещё лучше] = растёт вглубь
```

На практике Agile сочетает оба подхода: каждый спринт даёт инкремент и одновременно уточняет ранее сделанное.

**Когда применять:** требования известны не полностью, нужна ранняя обратная связь. **Роль СА:** требования прорабатываются порциями под текущую итерацию/инкремент.

---

## 5. Спиральная модель

Итеративная модель с акцентом на **управление рисками**. Каждый виток спирали: определение целей → анализ рисков → разработка/прототип → планирование следующего витка. Дорогая, применяется для крупных, рискованных проектов.

---

## 6. Agile

**Agile** — не одна модель, а семейство подходов на основе [Agile-манифеста](https://agilemanifesto.org/iso/ru/) (2001). Ценности:

```
Люди и взаимодействие         важнее  процессов и инструментов
Работающий продукт            важнее  исчерпывающей документации
Сотрудничество с заказчиком   важнее  согласования условий контракта
Готовность к изменениям       важнее  следования плану
```

> «Важнее» ≠ «вместо». Документация и план всё ещё нужны — но не самоцель. Для СА это значит: документация **достаточная**, а не максимальная; требования уточняются по ходу.

### Scrum

Фреймворк для итеративной разработки спринтами (обычно 1–4 недели).

| Роли | События | Артефакты |
|------|---------|-----------|
| Product Owner | Sprint Planning (планирование) | Product Backlog |
| Scrum Master | Daily Scrum (ежедневный синк) | Sprint Backlog |
| Developers (команда) | Sprint Review (демо) | Increment (инкремент) |
| | Sprint Retrospective (ретро) | |
| | Backlog Refinement (груминг)* | |

\* формально не «событие Scrum», но на практике ключевой для СА.

**Где включается СА:** Backlog Refinement (детализация историй, критерии приёмки), Sprint Planning (уточнения), в течение спринта (ответы команде), Sprint Review (проверка соответствия). СА в Scrum обычно работает в связке с PO.

### Kanban

Метод управления потоком задач без фиксированных спринтов. Ключевое:
- **визуализация потока** (доска с колонками: To Do → In Progress → Review → Done);
- **лимит WIP** (work in progress) — ограничение числа задач в работе, чтобы не распыляться;
- **вытягивание** (pull) — задача берётся, когда освободилась мощность;
- **непрерывный поток** вместо итераций.

| | Scrum | Kanban |
|--|-------|--------|
| Ритм | фиксированные спринты | непрерывный поток |
| Роли | заданы (PO, SM, Dev) | не предписаны |
| Изменения | в рамках спринта стараются не менять | можно в любой момент |
| Лучше для | продуктовой разработки фичами | потока разнородных задач (поддержка, заявки) |

## 7. Как выбрать модель

| Фактор | Тяготеет к Waterfall/V | Тяготеет к Agile |
|--------|------------------------|------------------|
| Стабильность требований | высокая, известны заранее | меняются, неизвестны полностью |
| Регуляторика/контракт | жёсткие, фиксированный объём | гибкие |
| Цена ошибки в проде | очень высокая | управляемая, можно быстро откатить |
| Нужна ранняя обратная связь | нет | да |
| Размер/срок | крупный, долгий | можно дробить на релизы |

> Универсального «лучше» нет. На собеседовании сильный ответ: «модель выбирается под контекст — стабильность требований, регуляторику, цену ошибки и потребность в ранней обратной связи».

---

## 8. Метрики процесса

СА полезно понимать метрики потока (особенно в Kanban/Scrum):

- **Lead time** — время от появления задачи (запроса) до её поставки. Меряет «сколько ждёт заказчик».
- **Cycle time** — время от взятия задачи в работу до готовности. Меряет «сколько реально делаем».
- **Throughput** — сколько задач завершается за период.
- **WIP** — сколько задач в работе одновременно.

Закон Литтла (упрощённо): `Lead time ≈ WIP / Throughput`. Чем больше задач берём одновременно (WIP), тем дольше каждая идёт до поставки. Отсюда практика лимитов WIP.

## ⚠️ Подводные камни

- **Путать инкрементную и итеративную.** Инкрементная — по частям (вширь), итеративная — улучшение целого (вглубь). Классический вопрос на собесе.
- **«Мы делаем Agile» = хаос.** Отсутствие документации и планирования прикрывают словом Agile. Agile — это дисциплина, а не её отсутствие.
- **Waterfall там, где требования меняются.** Фиксируем всё заранее, а к концу проекта мир изменился — продукт неактуален.
- **Scrum для потока поддержки.** Спринты плохо ложатся на непредсказуемый поток инцидентов — там лучше Kanban.
- **Игнорировать момент фиксации требований.** В Waterfall изменение в конце стоит дорого; в Agile — норма. СА должен понимать, в какой модели работает, и соответственно строить процесс.
