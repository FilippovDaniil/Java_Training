# Задача 09 — Модуль 03: Продукт и контекст

## ЗАДАНИЕ
Разрешите конфликт интересов стейкхолдеров: бизнес хочет быстрее и проще, безопасность/комплаенс — строже. Найдите решение, а не «победителя».

## ВХОДНЫЕ ДАННЫЕ
MedConnect, фича «вход по номеру телефона и SMS-коду».
- **Бизнес (PO, маркетинг):** хотят максимально простой вход в один шаг — чем меньше трения, тем выше активация. Просят убрать пароль вовсе.
- **Безопасность/комплаенс (ИБ, юрист):** медицинские данные требуют усиленной защиты; одного SMS недостаточно, нужен второй фактор и согласие на обработку ПДн; SMS-перехват — реальный риск.
- **Пользователи:** хотят быстро и без сложностей, но боятся за приватность медданных.

## ТРЕБОВАНИЯ
- Сформулируйте реальные интересы за позициями каждой стороны (не путать позицию и интерес).
- Предложите 2–3 варианта решения и оцените их по двум осям: трение для пользователя vs уровень защиты.
- Выберите компромисс и обоснуйте.
- Укажите, кто Accountable за финальное решение и какие требования это порождает.

## ФОРМАТ РЕЗУЛЬТАТА
Блок «позиция vs интерес» по сторонам + сравнение вариантов (таблица трение/защита) + выбранное решение с обоснованием + кто решает и какие требования возникают.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Для каждой стороны отделена позиция от интереса (бизнес: интерес — активация, не «убрать пароль любой ценой»).
- [ ] Предложены ≥2 варианта с оценкой trade-off трение/защита.
- [ ] Учтены обязательные требования регуляторики (согласие на ПДн нельзя выкинуть).
- [ ] Выбран обоснованный компромисс (например, простой вход + усиление для доступа к медданным).
- [ ] Указан Accountable и порождённые требования.

## ПОДСКАЗКА
Конфликт разрешается не «кто главнее», а поиском интереса за позицией. Бизнесу важна активация, а не отсутствие пароля; безопасности важна защита медданных, а не сложность ради сложности. Часто решение — разноуровневый доступ: лёгкий вход в общую часть, усиленная аутентификация для медицинских данных.
