# Задача 04 — Модуль 04: Уровни и типы требований

## ЗАДАНИЕ
Найдите дефекты в наборе требований и перепишите их в качественные (однозначные, проверяемые, атомарные, полные).

## ВХОДНЫЕ ДАННЫЕ
Требования из черновика спецификации FoodExpress:
1. «Система должна быстро обрабатывать заказы».
2. «Пользователь может зарегистрироваться, войти и восстановить пароль».
3. «Интерфейс должен быть удобным и современным».
4. «При оплате система списывает деньги».
5. «Система должна поддерживать много пользователей».
6. «Заказ можно отменить» (без указания, когда и кем).
7. «Нужно сделать кнопку повтора заказа в правом верхнем углу».

## ТРЕБОВАНИЯ
- Для каждого требования определите дефект(ы) по типам из теории.
- Перепишите его в качественное (или несколько атомарных).
- Где не хватает данных для метрики — укажите уточняющий вопрос вместо выдуманного числа.
- Для требования 7 вытащите потребность за «решением».

## ФОРМАТ РЕЗУЛЬТАТА
Для каждого: `Исходное → Дефект(ы) → Переписанное (или вопрос для уточнения)`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 7 требований разобраны.
- [ ] 1, 3, 5 — неоднозначность/непроверяемость → переписаны измеримо (или с уточняющим вопросом).
- [ ] 2 — неатомарность → разбито на 3 требования.
- [ ] 6 — неполнота → добавлены условия (кто, когда, в каком статусе).
- [ ] 7 — «решение вместо потребности» → вытащена потребность (быстро повторить заказ), а размещение кнопки оставлено дизайну/UX.
- [ ] Где нет данных — заданы вопросы, а не выдуманы числа.

## ПОДСКАЗКА
«Быстро/удобно/много» — всегда сигнал непроверяемости. «И…и…и» — сигнал неатомарности. «Сделайте кнопку в углу» — это решение; задача СА — найти потребность за ним и не фиксировать преждевременное UI-решение в требованиях.
