# Задача 06 — Модуль 04: Уровни и типы требований

## ЗАДАНИЕ
Заполните карточки требований с полным набором атрибутов. Карточка — это требование плюс метаданные для управления им.

## ВХОДНЫЕ ДАННЫЕ
Три требования FoodExpress:
1. «Клиент может оплатить заказ сохранённой картой».
2. «Система выдерживает 1000 одновременных оформлений заказа».
3. «Промокод нельзя применить к заказу со скидкой пенсионера».

Атрибуты карточки: ID, формулировка, уровень, тип, источник, приоритет (MoSCoW), статус, критерий приёмки, связь с бизнес-целью, зависимости.

## ТРЕБОВАНИЯ
- Для каждого требования заполните все атрибуты карточки.
- Приоритет проставьте по MoSCoW (Must/Should/Could/Won't) с кратким обоснованием.
- Критерий приёмки — проверяемый.
- Свяжите с правдоподобной бизнес-целью.
- Где источник неизвестен — укажите, у кого бы вы его уточнили.

## ФОРМАТ РЕЗУЛЬТАТА
Три карточки (по одной на требование), каждая — таблица «атрибут: значение».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 3 карточки заполнены по всем атрибутам.
- [ ] Уровень и тип проставлены корректно (1 — пользовательское/FR; 2 — NFR; 3 — бизнес-правило).
- [ ] Критерий приёмки проверяем для каждого.
- [ ] Приоритет MoSCoW обоснован.
- [ ] Указаны источник (или у кого уточнить) и связь с бизнес-целью.
- [ ] Для требования 1 отмечена зависимость (нужна сохранённая карта → требование про сохранение карты).

## ПОДСКАЗКА
Карточка превращает требование в управляемый объект: по ней можно приоритизировать, отслеживать статус и оценивать влияние изменений (модуль 10). Без приоритета и источника требованием нельзя управлять.
