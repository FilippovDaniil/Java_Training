# Задача 10 (мини-проект) — Модуль 04: Уровни и типы требований

## ЗАДАНИЕ
Составьте каталог требований верхнего уровня для FoodExpress: иерархию бизнес → пользовательские → функциональные, с типизацией и трассировкой. Это синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
FoodExpress, ключевой сценарий: **клиент находит ресторан, оформляет и оплачивает заказ, отслеживает доставку**. Продуктовый контекст (цели, стейкхолдеры) — из мини-проекта модуля 03 (можно использовать свои наработки).

## ТРЕБОВАНИЯ
Подготовьте каталог, включающий:
1. **2–3 бизнес-требования** (по SMART, связаны с целями проекта).
2. **Пользовательские требования** для основных классов пользователей (клиент, ресторан, курьер) — минимум по 2 на класс.
3. **Функциональные требования** — детализация пользовательских (минимум 8 FR).
4. **Нефункциональные требования и ограничения** — минимум 4 (производительность, надёжность, безопасность, регуляторика).
5. **Границы системы** — кратко, что in/out of scope.
6. **Матрица трассировки** — каждое FR ведёт к пользовательскому и к бизнес-требованию.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный каталог по разделам выше. Требования пронумерованы (BR-1, UR-1, FR-1, NFR-1…). Матрица трассировки — таблица `FR | UR | BR`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все уровни заполнены, требования атомарны и проверяемы.
- [ ] Бизнес-требования по SMART; пользовательские покрывают 3 класса пользователей.
- [ ] ≥8 функциональных и ≥4 нефункциональных/ограничений (с категориями).
- [ ] Границы системы указаны (in/out of scope).
- [ ] Матрица трассировки связывает каждое FR с UR и BR; «висящих» требований нет.
- [ ] Каталог внутренне непротиворечив (нет конфликтующих требований).
- [ ] Учтено минимум одно регуляторное ограничение (152-ФЗ / хранение данных в РФ).

## ПОДСКАЗКА
Это «скелет» спецификации, который дальше детализируется use cases (модуль 07) и user stories (модуль 08). Главные проверки: каждое требование проверяемо, типизировано и прослеживается вверх до бизнес-цели. Опирайтесь на задачи 01, 02, 05 этого модуля.
