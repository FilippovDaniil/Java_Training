# Модуль 06 — Бизнес-моделирование: BMC, контекст, персоны

## Зачем этот модуль

Вы научились добывать ([модуль 05](../module-05-elicitation/theory.md)) и типизировать ([модуль 04](../module-04-requirement-levels/theory.md)) требования. Но разрозненные требования нужно уложить в **связную картину бизнес-контекста**: как устроен бизнес (Business Model Canvas), где граница системы и кто вокруг неё (контекстная диаграмма), для кого мы делаем продукт (классы пользователей и персоны). Эти модели — мост между «зачем бизнесу» и «что проектируем». На собеседовании middle+ спрашивают про BMC, контекст системы и различие «класс пользователей vs персона».

---

## 1. Business Model Canvas (BMC)

**BMC** — одностраничная модель бизнеса из 9 блоков. СА использует её, чтобы быстро понять, **как бизнес создаёт, доставляет и захватывает ценность** — и вывести отсюда требования.

```
┌──────────────┬─────────────┬──────────────┬──────────────┬─────────────┐
│ Ключевые     │ Ключевые    │  Ценностное  │ Отношения с  │ Сегменты    │
│ партнёры     │ активности  │  предложение │ клиентами    │ клиентов    │
│              ├─────────────┤  (Value      ├──────────────┤             │
│              │ Ключевые    │  Proposition)│ Каналы       │             │
│              │ ресурсы     │              │              │             │
├──────────────┴─────────────┴──────────────┴──────────────┴─────────────┤
│ Структура издержек              │ Потоки доходов                        │
└─────────────────────────────────┴───────────────────────────────────────┘
```

| Блок | Вопрос | Пример (FoodExpress) |
|------|--------|----------------------|
| **Сегменты клиентов** | для кого? | клиенты-едоки, рестораны, курьеры |
| **Ценностное предложение** | какую ценность даём? | быстрая доставка еды из любимых ресторанов |
| **Каналы** | как доставляем ценность? | мобильное приложение, сайт |
| **Отношения с клиентами** | как взаимодействуем/удерживаем? | поддержка, программа лояльности, push |
| **Потоки доходов** | как зарабатываем? | комиссия с ресторанов, плата за доставку |
| **Ключевые ресурсы** | что нужно иметь? | платформа, сеть курьеров, бренд |
| **Ключевые активности** | что делаем? | подбор курьера, обработка заказов, маркетинг |
| **Ключевые партнёры** | кто помогает? | рестораны, платёжный провайдер, картографический сервис |
| **Структура издержек** | на что тратим? | разработка, маркетинг, поддержка, выплаты курьерам |

> BMC для СА — источник требований и контекста: сегменты → классы пользователей; ценностное предложение → ключевые фичи; партнёры → внешние интеграции; потоки доходов → бизнес-правила расчётов.

## 2. Контекстная диаграмма системы

**Контекстная диаграмма (System Context)** показывает систему как «чёрный ящик» и её окружение: внешних акторов и системы, с которыми она обменивается данными. Это визуализация **границ системы** ([модуль 04](../module-04-requirement-levels/theory.md), §4).

```
   [Клиент] ──заказ──►  ┌─────────────────┐  ──запрос оплаты──► [Платёжный
            ◄─статус──  │                 │  ◄──результат─────   провайдер]
   [Курьер] ──геопоз──► │   FoodExpress   │  ──SMS──►          [SMS-шлюз]
            ◄─задания─  │  (наша система) │
[Ресторан] ──меню────► │                 │  ──гео-запрос──►    [Картографический
            ◄─заказы──  └─────────────────┘  ◄──маршрут──────   сервис]
```

Элементы:
- **система** в центре (один блок, без внутренностей);
- **внешние акторы** (люди/роли): клиент, курьер, ресторан;
- **внешние системы**: провайдер оплаты, SMS-шлюз, карты;
- **потоки** (что и в каком направлении передаётся).

Зачем СА: фиксирует границы (что наше, что внешнее), выявляет все интеграции и точки обмена, предотвращает scope creep. Отличие от DFD ([модуль 14](../module-14-dfd-idef0/theory.md)): контекстная диаграмма — это «уровень 0», система целиком, без декомпозиции внутренностей.

## 3. Классы пользователей

**Класс пользователей** — группа пользователей со схожими потребностями, правами и поведением. Для каждого класса СА фиксирует:
- кто это и сколько их (примерно);
- цели и задачи в системе;
- уровень прав/доступа;
- частота использования и техническая грамотность;
- особые потребности (например, доступность).

Пример (FoodExpress): «Клиент», «Менеджер ресторана», «Курьер», «Оператор поддержки», «Администратор». У каждого — свои функции и права.

> Классы пользователей напрямую дают **роли** в use cases ([модуль 07](../module-07-use-cases/theory.md)) и в модели доступа/прав ([модуль 34](../module-34-api-security/theory.md)).

## 4. Персоны

**Персона** — собирательный образ конкретного представителя класса пользователей: «живой» портрет с именем, целями, болями и контекстом. Если класс пользователей — это категория, то персона — её человеческое лицо.

Структура персоны:
- имя, краткий портрет (возраст, занятие, контекст);
- цели (чего хочет достичь);
- боли/фрустрации (что мешает);
- поведение и привычки (как пользуется, на каком устройстве);
- сценарий использования (типичная история).

Пример персоны FoodExpress:
> **Анна, 32, маркетолог.** Работает допоздна, готовить некогда. Цель: быстро заказать ужин, не тратя время на выбор. Боль: долгие сроки доставки, скрытые комиссии. Поведение: заказывает с телефона по дороге домой, ценит повтор прошлого заказа.

| | Класс пользователей | Персона |
|--|---------------------|---------|
| Что это | категория, группа | конкретный собирательный человек |
| Уровень | абстрактный | конкретный, «живой» |
| Зачем | права, функции, охват | эмпатия, приоритеты, UX-решения |

> Персоны помогают принимать решения «а как бы Анна это восприняла» и приоритизировать фичи под реальные потребности, а не абстрактного «пользователя вообще».

---

## 5. Как это связано вместе

```
BMC (сегменты клиентов)  ──►  Классы пользователей  ──►  Персоны
        │                            │                       │
        ▼                            ▼                       ▼
ключевые фичи,            роли в use cases,        эмпатия, приоритеты,
партнёры → интеграции     права доступа            UX-решения

Контекстная диаграмма ──► границы системы + все внешние интеграции
```

Эти модели — вход для всей последующей работы: use cases (07), user stories (08), модель данных (16), API и интеграции (часть 6).

## ⚠️ Подводные камни

- **BMC ради BMC.** Заполнили канву и забыли. Ценность — в выводах: какие фичи, интеграции, правила следуют из неё.
- **Контекстная диаграмма с внутренностями.** На контекстном уровне система — один блок. Декомпозиция — это уже DFD/архитектура, не путать.
- **Забыть внешнюю систему.** Пропущенная на контексте интеграция = пропущенные требования и сорванные сроки.
- **Персона-выдумка.** Персона без опоры на реальные данные/интервью — фантазия, ведущая к неверным решениям.
- **Путать класс пользователей и персону.** Класс — категория (для прав/функций), персона — конкретный образ (для эмпатии/UX). На собесе различают.
- **Один «пользователь вообще».** Без классов/персон требования усредняются и не подходят никому конкретно.
