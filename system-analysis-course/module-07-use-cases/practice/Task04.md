# Задача 04 — Модуль 07: Use Cases

## ЗАДАНИЕ
Дополните use case «Оформить заказ» альтернативными потоками и исключениями. Это то, что отличает middle-описание от happy-path-заготовки.

## ВХОДНЫЕ ДАННЫЕ
Возьмите основной поток из задачи 03. Известные «развилки» процесса FoodExpress:
- клиент применяет промокод;
- клиент меняет адрес доставки на лету;
- оплата отклонена банком;
- выбранный товар закончился, пока клиент оформлял;
- ресторан закрылся/перестал принимать заказы;
- потеряна связь с платёжным провайдером (таймаут);
- клиент отменяет оформление до оплаты.

## ТРЕБОВАНИЯ
- Для каждой развилки определите: это альтернативный поток (другой путь к успеху) или исключение (ошибка/отказ).
- Привяжите каждую к номеру шага основного потока (например, 5a — на шаге 5).
- Опишите поведение системы в каждом случае.
- Отметьте, какие исключения требуют отдельных требований (например, идемпотентность при таймауте оплаты — связь с модулем 25).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 7 развилок разобраны и привязаны к шагам.
- [ ] Корректно разделены альтернативы (промокод, смена адреса) и исключения (отклонена оплата, нет товара, закрыт ресторан, таймаут).
- [ ] Для каждой описано поведение системы.
- [ ] Отмечено, что таймаут оплаты требует идемпотентности/повтора (не двойное списание).
- [ ] Отмена до оплаты обработана (возврат в исходное состояние).

## ПОДСКАЗКА
Альтернативный поток всё равно ведёт к успеху (промокод → заказ оформлен). Исключение прерывает или меняет исход (нет товара → заказ не оформлен). Таймаут оплаты — классическая ловушка: нельзя списать дважды (идемпотентность, модуль 25).
