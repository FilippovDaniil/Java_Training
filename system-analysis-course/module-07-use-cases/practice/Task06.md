# Задача 06 — Модуль 07: Use Cases

## ЗАДАНИЕ
Найдите ошибки в чужой модели use cases и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Черновик модели use cases для MedConnect:
1. Use case «Создать пациента».
2. Use case «Прочитать пациента».
3. Use case «Обновить пациента».
4. Use case «Удалить пациента».
5. Use case «Нажать кнопку Записаться».
6. Use case «Работать с системой» (актор: Пользователь).
7. Use case «Валидировать номер телефона».
8. Актор: «Доктор Иванов».
9. Use case «Экран расписания врача».
10. Use case «Провести видеоконсультацию» — описан только успешный сценарий.

## ТРЕБОВАНИЯ
- Для каждого пункта определите ошибку (CRUD-зоопарк, неверный уровень, актор-человек, use case = UI, тех-функция, happy-path-only).
- Предложите исправление.
- Покажите, как сгруппировать CRUD-операции в осмысленные use cases.
- Объясните, почему «Нажать кнопку» и «Экран расписания» — не use cases.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `№ | Ошибка | Исправление`. Ниже — пример группировки CRUD и пояснение про UI/кнопки.

## КРИТЕРИИ ПРИЁМКИ
- [ ] 1–4 опознаны как CRUD-зоопарк → сгруппированы (например, «Вести карту пациента» или разнесены по реальным целям).
- [ ] 5 («Нажать кнопку») и 9 («Экран расписания») опознаны как UI/действия, не use cases.
- [ ] 6 («Работать с системой») — слишком высокий уровень → разбить на цели.
- [ ] 7 («Валидировать телефон») — подфункция (fish-level), не самостоятельный use case.
- [ ] 8 (актор «Доктор Иванов») исправлен на роль «Врач».
- [ ] 10 дополнен альтернативами/исключениями (обрыв связи и т.д.).

## ПОДСКАЗКА
Use case = цель актора с ценностью, уровень sea-level. CRUD сам по себе не цель — цель «вести карту пациента». «Нажать кнопку», «экран», «валидировать поле» — это UI/шаги/подфункции, а не варианты использования. Актор — роль, не имя.
