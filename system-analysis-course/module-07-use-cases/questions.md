# Вопросы для самопроверки — Модуль 07: Use Cases

25 вопросов в стиле собеседования middle+. Сначала ответьте сами, потом сверьтесь с ключом.

1. Что такое use case? Чем он отличается от функции системы?
2. Что такое актор? Актор — это человек или роль?
3. В чём разница между первичным и вторичным актором?
4. Может ли внешняя система или планировщик быть актором?
5. Из каких элементов состоит use case diagram?
6. Какие поля содержит текстовое описание use case?
7. Что такое основной поток, альтернативный поток и исключение?
8. Что важнее для качества use case — happy path или ветвления?
9. В чём разница между include и extend?
10. Куда направлена стрелка в include и в extend?
11. Что такое generalization между use cases?
12. Какие уровни use case выделяет Коберн?
13. Какой уровень use case целевой и почему?
14. Почему «Создать/Прочитать/Обновить/Удалить» по отдельности — плохие use cases?
15. Почему «Нажать кнопку» — не use case?
16. Почему «Экран корзины» — не use case?
17. Что такое предусловие и постусловие use case?
18. Как пред/постусловия связывают use cases между собой?
19. Как use cases связаны с требованиями?
20. Как use case связан с тест-кейсами?
21. Чем use case отличается от user story?
22. Что такое функциональная декомпозиция как ошибка в use cases?
23. Почему актором нельзя делать конкретного человека («Иван»)?
24. Как из use case вывести функциональные требования?
25. Как обнаружить, что use case слишком крупный или слишком мелкий?

---

## Ключ (эталонные ответы)

1. **Use case** — описание того, как актор использует систему для достижения значимой цели, дающей ему ценность. Функция системы («сохранить в БД») сама по себе ценности актору не даёт и use case'ом не является.
2. **Актор** — это роль, а не конкретный человек. Один человек может играть несколько ролей (менеджер ресторана может быть и клиентом).
3. **Первичный** актор инициирует use case ради своей цели. **Вторичный** помогает выполнить, но не инициирует (например, платёжный провайдер при оплате).
4. Да. Актором может быть внешняя система (SMS-шлюз, провайдер) или время/планировщик (для задач по расписанию: «ежедневно формировать отчёт»).
5. Граница системы, акторы (снаружи), use cases (овалы внутри), ассоциации актор—use case, отношения include/extend/generalization.
6. Название, актор (первичный/вторичные), цель, предусловия, постусловия, триггер, основной поток, альтернативные потоки, исключения.
7. **Основной поток** — успешный сценарий (happy path). **Альтернативный** — другой путь к успеху. **Исключение** — обработка ошибки/отказа, меняющая или прерывающая исход.
8. Ветвления (альтернативы и исключения). Happy path пишет и junior; ценность middle — в полноте «что если» (нет товара, отклонена оплата, таймаут).
9. **include** — базовый use case ВСЕГДА использует включаемый (обязательная часть). **extend** — расширяющий use case ИНОГДА добавляется к базовому при условии (опционально).
10. **include**: стрелка от базового к включаемому. **extend**: стрелка от расширяющего к базовому. Направления противоположны.
11. **Generalization** — специализация: дочерний use case — частный случай родительского («Оплатить картой» — частный случай «Оплатить»).
12. Cloud/kite (стратегический), sea-level (цель пользователя), fish/sub-function (подфункция).
13. **Sea-level** — одна значимая цель пользователя за один присест. Cloud — это бизнес-процесс (слишком крупно), fish — техподфункция (слишком мелко).
14. Потому что это CRUD-зоопарк: отдельные операции не отражают цель актора. Их объединяют в осмысленный use case («Вести меню ресторана»), где CRUD — это шаги.
15. «Нажать кнопку» — это UI-действие/шаг, а не цель с ценностью. Use case описывает цель, а не конкретный элемент интерфейса.
16. «Экран корзины» — это интерфейс, а use case — цель и поведение независимо от конкретного UI. Один use case может затрагивать несколько экранов и наоборот.
17. **Предусловие** — что истинно ДО запуска (клиент авторизован). **Постусловие** — что истинно ПОСЛЕ успеха (заказ создан и оплачен). Оба должны быть проверяемы.
18. Постусловие одного use case часто служит предусловием другого, образуя цепочку через состояния (создан → можно назначить курьера → доставлен). Это мост к модели состояний.
19. Каждый use case реализует одно/несколько пользовательских требований; из него выводятся функциональные требования. Трассировка: требование → use case → FR → тест.
20. Из основного потока, альтернатив и исключений выводятся тест-кейсы (позитивные и негативные). Каждая ветка сценария — кандидат на тест.
21. **Use case** — целостный сценарий с ветвлениями (для полного понимания). **User story** — мелкая единица для бэклога и планирования. Часто один use case декомпозируется на несколько user stories.
22. Когда вместо целей актора описывают технические шаги/функции («валидировать поле», «записать в БД») как отдельные use cases. Это ломает суть — use case должен быть про цель, а не про реализацию.
23. Актор — это роль, а не личность. «Иван» сегодня уволится, роль «Клиент» останется. Моделируем устойчивые роли, а не конкретных людей.
24. Разложить шаги основного потока, альтернатив и исключений в требования «система должна …». Каждый значимый шаг и обработка ветки порождает функциональное требование.
25. Слишком крупный (cloud) — охватывает бизнес-процесс/несколько целей, не завершается за один присест → разбить. Слишком мелкий (fish) — техподфункция без самостоятельной ценности → поднять в родительский sea-level use case.
