# Модуль 07 — Use Cases (варианты использования)

## Зачем этот модуль

Вы выявили требования и контекст. Теперь нужно описать **как пользователь взаимодействует с системой для достижения цели** — это вариант использования (use case). Use case — один из самых распространённых артефактов СА: он связывает акторов (классы пользователей из [модуля 06](../module-06-business-modeling/theory.md)) с конкретными сценариями. На собеседовании middle+ обязательно: нотация use case diagram, структура текстового описания, include/extend, уровни и типичные ошибки.

---

## 1. Что такое use case

**Use case (вариант использования)** — описание того, как актор использует систему для достижения значимой **цели**. Ключевое слово — **цель пользователя**, а не функция системы.

```
Актор  ──имеет цель──►  Use case (сценарий взаимодействия)  ──►  результат, ценный для актора
```

Пример: «Клиент оформляет заказ» — это use case (цель клиента). А «Сохранить запись в БД» — это **не** use case, это внутренняя функция, не имеющая ценности для актора сама по себе.

Use case существует в двух формах:
- **диаграмма** (use case diagram) — обзор: кто и что может делать;
- **текстовое описание** — детали: пошаговый сценарий.

---

## 2. Акторы

**Актор** — роль, которая взаимодействует с системой (человек, внешняя система, время/планировщик). Актор — это **роль**, а не конкретный человек: один человек может быть в разных ролях.

| Тип актора | Кто это | Пример |
|------------|---------|--------|
| **Первичный (primary)** | инициирует use case ради своей цели | Клиент оформляет заказ |
| **Вторичный (secondary)** | помогает выполнить, но не инициирует | Платёжный провайдер списывает деньги |
| **Внешняя система** | другая система | SMS-шлюз, картографический сервис |
| **Время/триггер** | планировщик | «ежедневно в 00:00 формировать отчёт» |

> Акторы берутся из классов пользователей ([модуль 06](../module-06-business-modeling/theory.md)) и внешних систем с контекстной диаграммы.

---

## 3. Use case diagram

Нотация (UML):

```
        ┌─────────────────────────────────────┐
        │           FoodExpress                │  ← граница системы
        │                                      │
        │   (Оформить заказ)                   │
  ☺ ────┤        │ «include»                   │
Клиент  │        ▼                             │
        │   (Оплатить заказ)──────────────────┼──── ☺ Платёжный
        │                                      │      провайдер
        │   (Отследить доставку)               │
        │        ▲ «extend»                    │
        │   (Связаться с курьером)             │
        └─────────────────────────────────────┘
```

Элементы:
- **актор** (человечек/прямоугольник) — снаружи границы;
- **use case** (овал) — внутри границы системы;
- **граница системы** (прямоугольник) — что внутри системы;
- **ассоциация** (линия актор—use case) — кто участвует;
- **include, extend, generalization** — отношения между use cases (см. §5).

> Диаграмма даёт **обзор** функциональности: кто какие цели может реализовать. Детали — в текстовом описании.

---

## 4. Текстовое описание use case

Главная ценность — в детальном текстовом описании. Стандартная структура:

| Поле | Что содержит |
|------|--------------|
| **Название** | глагол + объект: «Оформить заказ» |
| **Актор** | первичный (и вторичные) |
| **Цель** | что хочет достичь актор |
| **Предусловия** | что должно быть истинно ДО начала (напр.: клиент авторизован) |
| **Постусловия** | что истинно ПОСЛЕ успеха (заказ создан, оплата проведена) |
| **Триггер** | что запускает сценарий |
| **Основной поток** | пошаговый успешный сценарий (happy path) |
| **Альтернативные потоки** | другие пути к успеху |
| **Исключения** | что при ошибках (нет товара, отклонена оплата) |

Пример основного потока «Оформить заказ»:
```
1. Клиент выбирает ресторан и блюда.
2. Система показывает корзину и итоговую сумму.
3. Клиент подтверждает адрес доставки.
4. Клиент выбирает способ оплаты.
5. Система проводит оплату (include: Оплатить заказ).
6. Система создаёт заказ и отправляет его ресторану.
7. Система показывает клиенту номер и статус заказа.

Альтернативный 4a: клиент применяет промокод → система пересчитывает сумму.
Исключение 5a: оплата отклонена → система предлагает другой способ.
Исключение 1a: ресторан закрыт → система не даёт оформить, предлагает другой.
```

> Именно альтернативы и исключения — то, что отличает middle от junior. Junior пишет happy path; middle покрывает «что если».

## 5. Отношения: include, extend, generalization

| Отношение | Смысл | Когда | Пример |
|-----------|-------|-------|--------|
| **include** | use case ВСЕГДА использует другой (обязательная часть) | вынести общий обязательный шаг | «Оформить заказ» **include** «Оплатить заказ» |
| **extend** | use case ИНОГДА расширяется другим (опционально, при условии) | необязательное/условное поведение | «Оформить заказ» **extend** «Применить промокод» |
| **generalization** | специализация общего use case | разновидности | «Оплатить картой» — generalization «Оплатить» |

```
include:  базовый ──«include»──► обязательный (всегда вызывается)
extend:   расширяющий ──«extend»──► базовый (вызывается при условии)
```

> Частая путаница: include — обязательно и всегда; extend — опционально и при условии. Направление стрелок противоположно: include указывает ОТ базового К включаемому; extend — ОТ расширяющего К базовому.

---

## 6. Уровни use case (по Коберну)

| Уровень | Метафора | Пример |
|---------|----------|--------|
| **Cloud / kite** (стратегический) | облако | «Управлять цепочкой поставок» |
| **Sea-level** (пользовательская цель) | уровень моря | «Оформить заказ» |
| **Fish / sub-function** (подфункция) | под водой | «Авторизоваться», «Проверить промокод» |

> **Целевой уровень для большинства use case — sea-level**: одна значимая цель пользователя за один «присест». Слишком высоко (cloud) — это бизнес-процесс, слишком низко (fish) — техническая подфункция. На собесе ловят, когда «Нажать кнопку» выдают за use case (это слишком мелко).

---

## 7. Типичные ошибки

| Ошибка | Пример | Как правильно |
|--------|--------|---------------|
| **CRUD как use case** | «Создать запись», «Удалить запись» | объединить в цель: «Управлять меню ресторана» |
| **Функциональная декомпозиция** | «Валидировать поле», «Записать в БД» | это шаги, не use cases; цель — «Оформить заказ» |
| **Слишком высокий уровень** | «Работать с системой» | разбить до sea-level целей |
| **Актор = конкретный человек** | «Иван оформляет заказ» | актор = роль «Клиент» |
| **Нет альтернатив/исключений** | только happy path | добавить «что если» |
| **Use case = экран UI** | «Экран корзины» | use case — цель, не интерфейс |

---

## 8. Связь use cases с требованиями

Use cases — мост между требованиями и реализацией:
- каждый use case реализует одно/несколько пользовательских требований;
- из use case выводятся функциональные требования и тест-кейсы;
- трассировка: требование → use case → тест-кейс ([модуль 35](../module-35-testing-for-analyst/theory.md)).

> Use case и user story ([модуль 08](../module-08-user-job-stories/theory.md)) — не конкуренты: use case даёт целостный сценарий с ветками, user story — мелкую единицу для бэклога. Часто use case декомпозируется на несколько user stories.

---

## ⚠️ Подводные камни

- **Happy path only.** Без альтернатив и исключений use case бесполезен для проектирования и тестирования.
- **CRUD-зоопарк.** Каждая операция как отдельный use case раздувает модель и теряет цель. Группировать по цели.
- **Путать include и extend.** include — обязательно/всегда, extend — опционально/при условии.
- **Неверный уровень.** Слишком крупно (бизнес-процесс) или слишком мелко (тех-функция). Цельтесь в sea-level.
- **Актор как должность/человек.** Актор — роль; один человек может играть несколько ролей.
- **Use case = UI-экран.** Use case описывает цель и поведение, а не конкретный интерфейс.
