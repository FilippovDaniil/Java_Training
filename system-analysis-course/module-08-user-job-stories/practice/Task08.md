# Задача 08 — Модуль 08: User / Job Stories

## ЗАДАНИЕ
Очистите критерии приёмки от деталей реализации и дизайна: оставьте проверяемое поведение, уберите «как именно сделать».

## ВХОДНЫЕ ДАННЫЕ
Критерии приёмки из черновика истории «Клиент применяет промокод»:
1. «Поле промокода — серое, в правом нижнем углу корзины».
2. «Промокод проверяется запросом GET /promo/{code} к сервису промокодов».
3. «Если промокод валиден, сумма пересчитывается со скидкой».
4. «Скидка хранится в таблице order_discounts с полем type=PROMO».
5. «Если промокод истёк, показывается сообщение об ошибке».
6. «Кнопка применения становится активной шрифтом Roboto 14px».
7. «Промокод нельзя применить дважды к одному заказу».

## ТРЕБОВАНИЯ
- Определите, какие пункты — настоящие критерии приёмки (проверяемое поведение), а какие — детали реализации/дизайна.
- Перепишите валидные критерии в чистом виде (по возможности в Gherkin).
- Объясните, почему детали реализации/UI не должны быть в критериях приёмки.
- Укажите, где этим деталям место (дизайн-макет, техзадание разработке).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Пункты 1, 6 опознаны как дизайн (цвет, шрифт, расположение).
- [ ] Пункты 2, 4 опознаны как детали реализации (конкретный эндпоинт, имя таблицы).
- [ ] Пункты 3, 5, 7 признаны валидными критериями приёмки (поведение).
- [ ] Валидные критерии переписаны в чистом виде.
- [ ] Объяснено: критерий описывает «что проверить», реализация — забота команды; дизайн — макета.
- [ ] Указано, куда отнести детали (дизайн → макет, эндпоинт/таблица → решение разработки).

## ПОДСКАЗКА
Критерий приёмки отвечает на «как проверить, что готово» с точки зрения поведения, и не диктует, КАК это сделать. Конкретный эндпоинт, имя таблицы, цвет кнопки — это решения команды/дизайна, не критерии приёмки. Они меняются без изменения смысла истории.
