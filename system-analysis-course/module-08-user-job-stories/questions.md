# Вопросы для самопроверки — Модуль 08: User / Job Stories

25 вопросов в стиле собеседования middle+. Сначала ответьте сами, потом сверьтесь с ключом.

1. Что такое user story и каков её формат?
2. Зачем нужна часть «чтобы…» в user story?
3. Что такое 3C (Card, Conversation, Confirmation)?
4. Расшифруйте INVEST.
5. Какие критерии INVEST нарушаются чаще всего?
6. Что такое job story и чем она отличается от user story?
7. Когда job story предпочтительнее user story?
8. Что такое критерии приёмки?
9. Что такое формат Gherkin?
10. Должны ли критерии приёмки описывать реализацию?
11. Что такое эпик?
12. Как декомпозировать эпик на истории?
13. Что значит «вертикальный срез» при декомпозиции?
14. Почему нельзя резать истории по слоям (БД, UI отдельно)?
15. Что такое story map?
16. Как story map связана с MVP?
17. Почему MVP — это горизонтальный, а не вертикальный срез story map?
18. Что такое story points и почему оценивают не в часах?
19. Как понять, что история слишком крупная?
20. В чём разница между Product Backlog и Sprint Backlog?
21. Чем user story отличается от use case?
22. Может ли один use case породить несколько user stories?
23. Почему «Как разработчик, я хочу отрефакторить код» — плохая user story?
24. Что делать с неопределённостью в истории («оплата способом, который придумаем»)?
25. Как критерии приёмки связаны с тестированием?

---

## Ключ (эталонные ответы)

1. **User story** — краткое описание требования глазами пользователя: «Как <роль>, я хочу <действие>, чтобы <ценность>». Это единица планирования в Agile.
2. Часть «чтобы…» объясняет **ценность** — зачем это нужно. Без неё история превращается в задачу без понимания цели; ценность помогает приоритизировать и не делать лишнего.
3. **Card** — краткая карточка-напоминание; **Conversation** — детали уточняются в разговоре с командой; **Confirmation** — критерии приёмки. История — обещание разговора, а не полная спецификация.
4. **I**ndependent (независимая), **N**egotiable (обсуждаемая), **V**aluable (ценная), **E**stimable (оцениваемая), **S**mall (малая), **T**estable (проверяемая).
5. Чаще всего **Small** (эпик выдают за историю) и **Valuable** (техническая задача без пользовательской ценности). Также Testable (нет критериев приёмки).
6. **Job story** — формат «Когда <ситуация>, я хочу <мотив>, чтобы <результат>». Акцент на ситуации/контексте, а не на роли пользователя.
7. Когда поведение определяется контекстом сильнее, чем ролью: один человек в разных ситуациях ведёт себя по-разному, и «как пользователь» не объясняет мотив.
8. **Критерии приёмки** — условия, при которых история считается выполненной (Confirmation из 3C). Описывают проверяемое поведение.
9. **Gherkin** — структура «Дано (Given) / Когда (When) / Тогда (Then)»: предусловие, действие, ожидаемый результат. Удобен для критериев приёмки и автотестов.
10. Нет. Критерии приёмки описывают **что проверить** (поведение), а не **как реализовать**. Конкретный эндпоинт, имя таблицы, цвет кнопки — не критерии приёмки.
11. **Эпик** — крупная история, которая не влезает в спринт и требует декомпозиции на несколько историй.
12. По паттернам: по шагам процесса, по бизнес-правилам, по вариациям данных, по операциям (CRUD), по платформам/интерфейсам, через spike (исследование). Каждый кусок сохраняет ценность.
13. **Вертикальный срез** — кусок, который сам по себе даёт ценность и проходит через все слои (UI → логика → данные), его можно показать пользователю.
14. Потому что слой («вся БД», «весь UI») не даёт пользовательской ценности и его нельзя продемонстрировать/протестировать как работающую функцию. Срез должен быть сквозным.
15. **Story map** — двумерная карта бэклога: горизонтально — последовательность действий пользователя (backbone), вертикально — детализация/приоритет по релизам.
16. MVP — это первый горизонтальный срез story map: по одной базовой истории на каждую активность, чтобы пользователь мог пройти весь путь целиком (пусть и просто).
17. Потому что пользователю нужен весь путь, а не одна развитая часть. «Вся оплата, но без возможности оформить заказ» бесполезна. Тонкая сквозная линия даёт работающий продукт.
18. **Story points** — относительная мера объёма/сложности. Оценивают не в часах, потому что относительное сравнение точнее и устойчивее к различиям в скорости людей; цель — сравнить истории, а не предсказать время.
19. Если её нельзя оценить, она не влезает в спринт, получает высокую оценку (13, 21) или содержит много правил/вариаций — это эпик, нужна декомпозиция.
20. **Product Backlog** — общий приоритизированный список всех историй продукта. **Sprint Backlog** — подмножество, взятое в текущий спринт.
21. **User story** — мелкая единица для бэклога, детали в разговоре и критериях. **Use case** — целостный сценарий с потоками и ветвлениями. Они дополняют друг друга.
22. Да. Один use case (целостный сценарий) обычно декомпозируется на несколько user stories для планирования и реализации по частям.
23. Нарушает Valuable: нет пользовательской ценности, это техническая задача. Рефакторинг оформляют как техзадачу/tech debt, а не как user story (или связывают с ценностью явно).
24. Вынести неопределённость в исследование (spike) или уточнить до оценки. Историю с «придумаем потом» нельзя оценить (нарушает Estimable), пока неизвестность не снята.
25. Критерии приёмки (особенно в Gherkin) напрямую превращаются в приёмочные тест-кейсы: каждый сценарий Given/When/Then — это проверка. Они задают, что QA будет проверять.
