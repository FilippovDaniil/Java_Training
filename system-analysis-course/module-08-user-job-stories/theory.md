# Модуль 08 — User Stories и Job Stories

## Зачем этот модуль

Use case ([модуль 07](../module-07-use-cases/theory.md)) даёт целостный сценарий. Но в Agile нужна **мелкая единица планирования** — то, что влезает в спринт, приоритизируется и оценивается. Это user story. СА в продуктовой команде работает с историями ежедневно: пишет их, проверяет по INVEST, формулирует критерии приёмки, декомпозирует и складывает в story map. На собеседовании middle+ это один из самых частых блоков: формат истории, INVEST, критерии приёмки (Gherkin), декомпозиция, отличие от use case.

---

## 1. User Story

**User story (пользовательская история)** — короткое описание требования с точки зрения пользователя, в формате:

```
Как <роль>,
я хочу <действие/возможность>,
чтобы <ценность/цель>.
```

Пример (FoodExpress):
> Как клиент, я хочу повторить прошлый заказ в один тап, чтобы не искать блюда заново.

Ключевое — третья часть («чтобы…»): она объясняет **ценность**. История без ценности — это просто задача, непонятно зачем.

### 3C — три измерения истории
- **Card** (карточка) — краткая формулировка (напоминание о разговоре);
- **Conversation** (разговор) — детали уточняются в диалоге с командой;
- **Confirmation** (подтверждение) — критерии приёмки, по которым проверяют готовность.

> История — это **обещание разговора**, а не исчерпывающая спецификация. Детали живут в критериях приёмки и обсуждении.

---

## 2. INVEST — критерии хорошей истории

| Буква | Критерий | Что значит |
|-------|----------|-----------|
| **I** — Independent | независимая | можно делать отдельно от других |
| **N** — Negotiable | обсуждаемая | не жёсткий контракт, детали уточняются |
| **V** — Valuable | ценная | даёт ценность пользователю/бизнесу |
| **E** — Estimable | оцениваемая | команда может оценить объём |
| **S** — Small | малая | влезает в один спринт |
| **T** — Testable | проверяемая | есть критерии приёмки |

> INVEST — чек-лист качества истории. Чаще всего нарушаются **Small** (история-эпик) и **Valuable** (техническая задача без пользовательской ценности).

---

## 3. Job Stories

**Job story** — альтернатива user story, делающая акцент на **ситуации (контексте)**, а не на роли. Из подхода Jobs To Be Done.

```
Когда <ситуация>,
я хочу <мотив/действие>,
чтобы <ожидаемый результат>.
```

Пример:
> Когда я заказываю ужин после тяжёлого дня и не хочу думать, я хочу быстро повторить прошлый заказ, чтобы поесть без лишних решений.

| | User story | Job story |
|--|-----------|-----------|
| Фокус | роль пользователя | ситуация/контекст |
| Сильна, когда | роли чётко различимы | важен контекст и мотивация, роли размыты |
| Слабость | «как пользователь» вырождается в штамп | сложнее для бэклога |

> Job story полезна, когда «роль» не объясняет поведение (один человек в разных ситуациях ведёт себя по-разному). На практике команды используют то, что удобнее; СА должен уметь оба.

---

## 4. Критерии приёмки (Acceptance Criteria)

**Критерии приёмки** — условия, при которых история считается выполненной. Это «Confirmation» из 3C. Частый формат — **Gherkin (Given/When/Then)**:

```
Дано (Given)   <предусловие/контекст>
Когда (When)   <действие>
Тогда (Then)   <ожидаемый результат>
```

Пример для «повторить заказ»:
```
Сценарий: успешный повтор заказа
  Дано клиент с историей заказов
  Когда клиент нажимает «Повторить» на прошлом заказе
  Тогда система формирует корзину с теми же блюдами
  И показывает актуальные цены

Сценарий: блюдо больше недоступно
  Дано в прошлом заказе есть блюдо, которого нет в меню
  Когда клиент нажимает «Повторить»
  Тогда система добавляет доступные блюда
  И сообщает, какие блюда недоступны
```

> Критерии приёмки описывают **что проверить**, а не **как реализовать**. «Кнопка синяя в углу» — это не критерий приёмки, это дизайн. Критерий — проверяемое поведение.

---

## 5. Декомпозиция историй

Слишком крупную историю (эпик) разбивают, сохраняя ценность в каждом куске. Паттерны разбиения (SPIDR и др.):

| Паттерн | Как разбить | Пример |
|---------|-------------|--------|
| По шагам процесса (workflow) | выделить этапы | оформление → отдельно адрес, оплата, подтверждение |
| По бизнес-правилам | разные правила — разные истории | оплата картой / оплата бонусами |
| По вариациям данных | разные типы данных | заказ из 1 ресторана / из нескольких |
| По операциям (CRUD) | создание/просмотр/изменение | сначала просмотр меню, потом редактирование |
| По интерфейсам/платформам | веб / мобайл | сначала мобайл |
| Spike (исследование) | вынести неизвестность | прото-исследование интеграции |

> Правило: каждый кусок после разбиения должен **сам по себе давать ценность** и быть тестируемым (вертикальный срез, а не «слой БД отдельно, UI отдельно»).

---

## 6. Story map

**Story map (карта историй)** — двумерная визуализация бэклога: горизонтально — последовательность действий пользователя (backbone), вертикально — детализация и приоритет (релизы).

```
Backbone:   Найти ресторан → Оформить заказ → Оплатить → Отследить
              │                  │               │           │
Релиз 1:    поиск по имени    корзина        картой      статус текстом
(MVP)         │                  │               │           │
Релиз 2:    фильтры           промокоды      бонусами    карта курьера
              │                  │               │           │
Релиз 3:    рекомендации      повтор заказа  подписка    чат с курьером
```

> Story map показывает **целостный путь пользователя** и помогает нарезать релизы «горизонтально» (тонкий сквозной срез ценности), а не «вертикально» (всё про оплату сразу). Связь с MVP ([модуль 03](../module-03-product-context/theory.md)): первый горизонтальный срез = MVP.

---

## 7. Бэклог и оценка

- **Product Backlog** — приоритизированный список историй (приоритизация — [модуль 10](../module-10-requirements-management/theory.md)).
- **Sprint Backlog** — истории, взятые в спринт.
- **Оценка (estimation):** относительная (story points, через Planning Poker), а не в часах. Цель — сравнить объём историй, а не предсказать точное время.

> Слишком крупная история (не оценивается или не влезает в спринт) — сигнал к декомпозиции.

---

## 8. User story vs Use case

| | User story | Use case |
|--|-----------|----------|
| Размер | мелкая (часть функции) | целостный сценарий |
| Детали | в разговоре + критерии приёмки | в самом описании (потоки) |
| Ветвления | в критериях приёмки | альтернативы/исключения |
| Для чего | планирование, бэклог, спринты | понимание сценария целиком |

> Они дополняют друг друга: use case даёт картину, user stories — нарезку для работы. Один use case → несколько user stories.

---

## ⚠️ Подводные камни

- **История без «чтобы».** Потеряна ценность — непонятно, зачем делаем. Всегда указывать цель.
- **Технические истории без ценности.** «Как разработчик, я хочу отрефакторить модуль» — это задача, не user story (нарушает Valuable).
- **Эпик под видом истории.** Не влезает в спринт, не оценивается — декомпозировать.
- **Критерии приёмки = дизайн.** «Кнопка справа» — не критерий; критерий — проверяемое поведение.
- **Горизонтальная нарезка слоёв.** «Сделать всю БД», «весь UI» — куски без самостоятельной ценности. Нарезать вертикально (сквозной срез).
- **Критерии приёмки описывают реализацию.** Должны описывать «что», а не «как».
