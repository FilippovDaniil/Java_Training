# Задача 02 — Модуль 09: Нефункциональные требования

## ЗАДАНИЕ
Превратите размытые пожелания в измеримые NFR по шаблону «метрика + значение + условие». Где данных нет — задайте уточняющий вопрос.

## ВХОДНЫЕ ДАННЫЕ
Пожелания бизнеса FoodExpress:
1. «Приложение должно работать быстро».
2. «Система должна быть надёжной».
3. «Поиск ресторанов должен быть мгновенным».
4. «Система должна выдерживать нагрузку в час пик».
5. «Оформление заказа должно быть удобным».
6. «Данные должны быть в безопасности».

## ТРЕБОВАНИЯ
- Для каждого пожелания сформулируйте измеримое NFR (метрика + значение + условие).
- Используйте перцентили для производительности, где уместно.
- Где нет данных для значения — сформулируйте уточняющий вопрос бизнесу вместо выдуманного числа.
- Укажите, как каждое NFR будет проверяться.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `Пожелание | Измеримое NFR | Способ проверки | Уточняющий вопрос (если нужно)`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 6 переформулированы в измеримые NFR (метрика + значение + условие).
- [ ] Для производительности использованы перцентили (p95/p99), а не «среднее».
- [ ] Указано условие нагрузки (при каком RPS/числе пользователей).
- [ ] Для каждого — способ проверки (нагрузочный тест, мониторинг, аудит, юзабилити-тест).
- [ ] Где значение неизвестно — задан вопрос, а не выдумано число.

## ПОДСКАЗКА
«Быстро» без условия нагрузки бессмысленно: ≤200 мс при 10 пользователях ≠ при 10 000. Шаблон: «<операция>: p95 ≤ X мс при нагрузке Y RPS». «Надёжно» → доступность в %. «Безопасно» → конкретные меры (аутентификация, шифрование, аудит).
