# Задача 03 — Модуль 09: Нефункциональные требования

## ЗАДАНИЕ
Сформулируйте полный набор NFR производительности для ключевых операций FoodExpress, используя перцентили и условия нагрузки.

## ВХОДНЫЕ ДАННЫЕ
Ключевые операции FoodExpress и их характер:
- поиск ресторанов (частая, чтение, важна скорость);
- открытие меню ресторана (частая, чтение);
- оформление заказа (реже, запись, критична надёжность);
- оплата (критична, внешний провайдер);
- отслеживание курьера (частые обновления геопозиции).
Ожидаемая нагрузка: 1000 RPS в обычное время, до 5000 RPS в часы пик.

## ТРЕБОВАНИЯ
- Для каждой операции задайте NFR: latency (p50/p95/p99), throughput, условие нагрузки.
- Объясните, почему для разных операций разные требования (поиск vs оплата).
- Учтите, что оплата зависит от внешнего провайдера (как это влияет на NFR).
- Объясните, почему вы используете перцентили, а не среднее.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `Операция | p50 | p95 | p99 | Throughput | Условие нагрузки`. Ниже — пояснения про разницу операций, внешний провайдер и перцентили.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Для всех операций заданы перцентили latency и условие нагрузки.
- [ ] Требования к частым операциям чтения жёстче по скорости, чем к редким записям.
- [ ] Для оплаты учтена зависимость от внешнего провайдера (таймаут, что считать «своим» временем).
- [ ] Объяснено использование перцентилей вместо среднего.
- [ ] Учтён пик 5000 RPS (деградация допустима/недопустима).

## ПОДСКАЗКА
Разные операции — разные NFR: поиск должен быть очень быстрым (частый, влияет на впечатление), оплата может быть медленнее (реже, зависит от провайдера). Для операций с внешними системами разделяйте «наше» время и время провайдера. p99 показывает, как плохо «худшему» проценту пользователей.
