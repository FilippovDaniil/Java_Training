# Задача 07 — Модуль 09: Нефункциональные требования

## ЗАДАНИЕ
Свяжите NFR с архитектурными решениями: покажите, как каждое нефункциональное требование влияет на устройство системы.

## ВХОДНЫЕ ДАННЫЕ
NFR FoodExpress:
1. Поиск ресторанов: p95 ≤ 300 мс при 5000 RPS.
2. Доступность сервиса заказов ≥99.95%.
3. Рост до 50 000 заказов/день без переписывания.
4. Отсутствие двойного списания при сбоях оплаты.
5. Данные карт не хранятся в системе.
6. Восстановление после сбоя ≤5 минут.

## ТРЕБОВАНИЯ
- Для каждого NFR укажите архитектурное решение, которое его обеспечивает.
- Отметьте, что из этого — зона архитектора, а что — вклад СА (требование/ограничение).
- Покажите, какие NFR требуют распределённых паттернов (связь с модулем 25).
- Объясните, почему эти NFR нельзя «добавить потом».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Для каждого NFR указано осмысленное архитектурное решение (1 — кэш/индексы; 2, 6 — резервирование/health checks; 3 — горизонтальное масштабирование/stateless; 4 — идемпотентность; 5 — токенизация/вынос на провайдера).
- [ ] Разграничены зоны: СА фиксирует требование, архитектор выбирает решение.
- [ ] Отмечено, что NFR 4 требует идемпотентности (модуль 25).
- [ ] Объяснено, почему производительность/безопасность/масштабируемость нельзя прикрутить в конце.
- [ ] Указаны связанные модули курса.

## ПОДСКАЗКА
NFR — главный драйвер архитектуры. «5000 RPS» → кэш и масштабирование; «нет двойного списания» → идемпотентность; «не хранить карты» → токенизация на стороне провайдера. СА ставит планку, архитектор выбирает, как её взять. Эти качества закладываются в фундамент, а не докручиваются.
