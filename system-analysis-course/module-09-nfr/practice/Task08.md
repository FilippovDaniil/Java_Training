# Задача 08 — Модуль 09: Нефункциональные требования

## ЗАДАНИЕ
Для набора NFR определите способ проверки: как объективно убедиться, что требование выполнено.

## ВХОДНЫЕ ДАННЫЕ
NFR FoodExpress:
1. Поиск: p95 ≤ 300 мс при 5000 RPS.
2. Доступность сервиса заказов ≥99.95% в месяц.
3. Новый пользователь оформляет заказ за ≤3 минуты без подсказок.
4. Доступ к админ-панели только по 2FA.
5. Система масштабируется до 50 000 заказов/день.
6. Восстановление после сбоя ≤5 минут.

## ТРЕБОВАНИЯ
- Для каждого NFR опишите конкретный способ проверки (инструмент/метод).
- Укажите, на каком этапе проверяется (тестирование, эксплуатация, аудит).
- Отметьте, что проверяется единоразово, а что — постоянно мониторится.
- Определите, какое NFR проверить труднее всего и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `NFR | Способ проверки | Этап | Разовая проверка / мониторинг`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Для всех NFR указан конкретный способ проверки (нагрузочный тест для 1, 5; мониторинг SLI для 2; юзабилити-тест для 3; пентест/проверка конфигурации для 4; chaos/тест восстановления для 6).
- [ ] Указан этап проверки.
- [ ] Разделены разовые проверки и постоянный мониторинг (доступность — мониторинг).
- [ ] Определено самое трудное для проверки NFR с обоснованием (например, удобство или масштабируемость).
- [ ] Способы реалистичны и объективны.

## ПОДСКАЗКА
NFR без способа проверки — декларация. Производительность → нагрузочное тестирование; доступность → мониторинг SLI в проде; удобство → юзабилити-тест; безопасность → пентест/аудит; восстановление → тест отказа. Часть NFR проверяется один раз, часть — мониторится постоянно.
