# Задача 10 (мини-проект) — Модуль 09: Нефункциональные требования

## ЗАДАНИЕ
Составьте полную спецификацию NFR для FoodExpress: по всем ключевым категориям качества, измеримые, со способами проверки и привязкой к бизнес-приоритетам. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
FoodExpress — агрегатор доставки еды. Обычная нагрузка 1000 RPS, пики до 5000 RPS. Критичны для денег: заказы и оплата. Работа с ПДн клиентов и платёжными данными. Команда хочет, чтобы спецификация NFR стала разделом ТЗ.

## ТРЕБОВАНИЯ
Подготовьте спецификацию NFR, охватывающую минимум 8 категорий:
1. **Производительность** (перцентили, throughput, условия нагрузки).
2. **Надёжность/доступность** (SLO по сервисам, восстановление).
3. **Масштабируемость** (рост, автоскейлинг).
4. **Безопасность** (аутентификация, авторизация, аудит).
5. **Приватность/регуляторика** (152-ФЗ, ПДн, платёжные данные).
6. **Удобство** (измеримые цели юзабилити).
7. **Сопровождаемость** (тестируемость, наблюдаемость).
8. **Совместимость/переносимость** (платформы, развёртывание).

Для каждого NFR: измеримое значение + условие + способ проверки. Отметьте конфликты NFR и выбранный баланс. Выделите NFR, критичные для бизнеса.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированная спецификация по категориям. Каждое NFR в формате «метрика + значение + условие + способ проверки». Отдельный раздел — конфликты и баланс. Пометка приоритета (критично/важно/желательно).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Покрыты все 8 категорий, ≥1 измеримое NFR в каждой.
- [ ] Производительность — с перцентилями и условием нагрузки; учтён пик 5000 RPS.
- [ ] Доступность — разные SLO для критичных и некритичных сервисов.
- [ ] Безопасность и приватность — с учётом ПДн/платёжных данных и проверяемостью.
- [ ] Для каждого NFR — способ проверки.
- [ ] Отмечены ≥2 конфликта NFR и выбранный баланс.
- [ ] Проставлены приоритеты; критичные для бизнеса выделены.
- [ ] Спецификация внутренне непротиворечива и готова стать разделом ТЗ.

## ПОДСКАЗКА
Это раздел ТЗ «Нефункциональные требования», по которому архитектор проектирует, а QA проверяет. Главные проверки middle+: всё измеримо (нет «быстро/надёжно»), есть условия нагрузки и способы проверки, осознанный баланс конфликтов, приоритеты расставлены. Собирайте из задач 02–09 этого модуля.
