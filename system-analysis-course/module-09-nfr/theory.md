# Модуль 09 — Нефункциональные требования (NFR)

## Зачем этот модуль

Функциональные требования и истории описывают, **что** система делает. Но система может делать «то», но плохо: медленно, ненадёжно, небезопасно, неудобно. За это отвечают **нефункциональные требования (NFR)** — требования к **качествам** системы. Именно NFR чаще всего определяют архитектуру и чаще всего пропускаются junior-аналитиками. Для middle+ это критический навык: уметь выявить, классифицировать и, главное, **сделать NFR измеримыми**. На собеседовании это один из самых показательных блоков — отличает того, кто проектирует системы, от того, кто пишет «список функций».

---

## 1. Что такое NFR и почему они критичны

**Нефункциональное требование (NFR)** — требование к свойству/качеству системы, а не к её функции.

```
FR:  «система формирует чек»            → ЧТО делает
NFR: «чек формируется за ≤2 с при 95%   → КАК ХОРОШО делает
      запросов на нагрузке 500 RPS»
```

Почему NFR критичны:
- они **определяют архитектуру** (требование «100k одновременных пользователей» меняет всё — БД, кэш, масштабирование);
- их **дорого добавлять потом** (безопасность или производительность нельзя «прикрутить» в конце);
- их **чаще всего забывают**, и они всплывают как провал в проде (система «работает», но падает под нагрузкой).

> Запоминалка: FR можно увидеть в демо («вот кнопка работает»). NFR проявляются под нагрузкой, при сбое, при атаке, при росте — то есть тогда, когда уже поздно их добавлять.

---

## 2. Классификация: ISO/IEC 25010

Стандарт качества ПО ISO/IEC 25010 выделяет характеристики качества:

| Характеристика | О чём | Пример NFR |
|----------------|-------|------------|
| **Производительность (Performance efficiency)** | скорость, ресурсы, ёмкость | отклик ≤2 с при 1000 RPS |
| **Надёжность (Reliability)** | доступность, отказоустойчивость, восстановление | доступность 99.9%, восстановление ≤5 мин |
| **Безопасность (Security)** | конфиденциальность, целостность, аутентификация, аудит | все платежи логируются; данные карт не хранятся |
| **Удобство (Usability)** | простота освоения, доступность | новый пользователь оформляет заказ за ≤3 мин |
| **Совместимость (Compatibility)** | взаимодействие, сосуществование | поддержка браузеров N, интеграция по REST |
| **Сопровождаемость (Maintainability)** | модульность, тестируемость, изменяемость | покрытие тестами ≥70%, время сборки ≤10 мин |
| **Переносимость (Portability)** | установка, адаптируемость | разворачивается в Docker/K8s |
| **Функциональная пригодность (Functional suitability)** | полнота, корректность функций | (пограничная с FR) |

> На практике для СА главные группы NFR: производительность, надёжность/доступность, безопасность, масштабируемость, удобство. Остальные — по контексту.

---

## 3. Главное правило: NFR должны быть измеримы

NFR без числа — не требование, а пожелание. Шаблон измеримого NFR:

```
<качество> : <метрика> <оператор> <значение> при <условие/нагрузка>
```

| Плохо (непроверяемо) | Хорошо (измеримо) |
|----------------------|-------------------|
| «система быстрая» | «95% запросов поиска ресторанов выполняются за ≤500 мс при нагрузке 1000 RPS» |
| «надёжная» | «доступность сервиса заказов ≥99.9% в месяц» |
| «безопасная» | «все обращения к медданным логируются; доступ только у лечащего врача» |
| «масштабируемая» | «горизонтальное масштабирование до 10k одновременных заказов добавлением инстансов» |
| «удобная» | «новый пользователь оформляет первый заказ за ≤3 минуты в юзабилити-тесте (8 из 10)» |

> Если для NFR нельзя придумать способ проверки — оно сформулировано плохо. «Как мы это измерим?» — главный вопрос middle+ к каждому NFR.

---

## 4. Производительность подробнее

Ключевые метрики:
- **Latency (время отклика):** обычно в перцентилях, а не в среднем. p50 (медиана), **p95**, **p99** — «95%/99% запросов быстрее, чем…». Среднее обманчиво: оно скрывает «хвосты».
- **Throughput (пропускная способность):** RPS (запросов в секунду), TPS (транзакций).
- **Нагрузка:** при каком числе пользователей/запросов держатся метрики.

```
Почему перцентили, а не среднее:
запросы: 10мс, 10мс, 10мс, 10мс, 2000мс
среднее = 408мс (выглядит плохо, но...)
p50 = 10мс (типичный пользователь доволен)
p99 = 2000мс (1% страдает — вот где проблема)
```

> «Среднее время отклика 200 мс» почти бесполезно. «p95 ≤ 300 мс, p99 ≤ 800 мс» — содержательно. Перцентили — маркер зрелости аналитика.

---

## 5. Доступность (availability)

**Доступность** — доля времени, когда система работает. Измеряется в «девятках»:

| Доступность | Простой в год | Простой в месяц | Простой в неделю |
|-------------|---------------|------------------|-------------------|
| 99% («две девятки») | ~3.65 дня | ~7.3 ч | ~1.7 ч |
| 99.9% («три девятки») | ~8.77 ч | ~43.8 мин | ~10 мин |
| 99.99% («четыре девятки») | ~52.6 мин | ~4.4 мин | ~1 мин |
| 99.999% («пять девяток») | ~5.26 мин | ~26 с | ~6 с |

Связанные понятия (подробно — [модуль 26](../module-26-nfr-architecture/theory.md)):
- **SLA** (Service Level Agreement) — обязательство перед клиентом (с санкциями);
- **SLO** (Service Level Objective) — внутренняя цель;
- **SLI** (Service Level Indicator) — измеряемый показатель.

> Каждая «девятка» стоит дорого (резервирование, инфраструктура). 99.999% для агрегатора доставки — избыточно и дорого; для платёжного шлюза банка — оправдано. NFR доступности всегда соизмеряют с ценой.

---

## 6. Безопасность и приватность

Для СА (детально — [модуль 34](../module-34-api-security/theory.md)):
- **аутентификация** (кто) и **авторизация** (что разрешено);
- **конфиденциальность** (данные видят только те, кому положено);
- **целостность** (данные нельзя незаметно изменить);
- **аудит** (кто что делал — логирование);
- **приватность / регуляторика:** 152-ФЗ (РФ), GDPR (ЕС) — согласие на обработку ПДн, право на удаление, хранение данных.

> Безопасность и приватность — почти всегда **ограничения** + NFR. Для MedConnect это критично: медданные — особая категория ПДн.

---

## 7. Конфликты NFR (trade-offs)

NFR часто конфликтуют — нельзя максимизировать всё:

| Конфликт | Суть |
|----------|------|
| Безопасность ↔ Удобство | больше проверок = больше трения |
| Производительность ↔ Согласованность | кэш быстрее, но данные могут устареть |
| Доступность ↔ Стоимость | каждая «девятка» дороже предыдущей |
| Масштабируемость ↔ Простота | распределённость = сложность эксплуатации |
| Безопасность ↔ Производительность | шифрование/проверки добавляют задержку |

> Задача СА — не «сделать всё идеально», а **осознанно выбрать баланс** под бизнес-приоритеты и зафиксировать решение. Это прямой мост к архитектуре ([модули 25–26](../module-26-nfr-architecture/theory.md)) и теореме CAP ([модуль 19](../module-19-nosql-cap/theory.md)).

---

## 8. Привязка к архитектуре и проверка

- Каждый серьёзный NFR влияет на архитектурное решение (нагрузка → кэш/реплики/шардинг; доступность → резервирование; безопасность → шифрование/контроль доступа).
- Для каждого NFR должен быть **способ проверки**: нагрузочное тестирование (производительность), мониторинг доступности (SLI), пентест/аудит (безопасность), юзабилити-тест (удобство).

> NFR без способа проверки = декларация. Указывайте не только значение, но и как его проверить.

---

## ⚠️ Подводные камни

- **NFR без числа.** «Быстро/надёжно/удобно» — не требования. Всегда метрика + значение + условие.
- **Среднее вместо перцентилей.** Среднее прячет хвосты; используйте p95/p99.
- **Забыть NFR вообще.** Система «работает» в демо и падает в проде под нагрузкой. NFR выявляют наравне с FR.
- **«Максимум всего».** Нельзя одновременно максимизировать доступность, безопасность, скорость и дешевизну. Осознанный баланс.
- **NFR без условий нагрузки.** «Отклик ≤200 мс» — при каком числе пользователей? Без условия бессмысленно.
- **Избыточные девятки.** 99.999% «потому что круто» — это огромные затраты. Соизмерять с бизнес-ценностью.
- **Безопасность/приватность «потом».** Их нельзя прикрутить в конце; закладывать с начала.
