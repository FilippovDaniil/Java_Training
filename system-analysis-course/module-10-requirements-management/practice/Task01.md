# Задача 01 — Модуль 10: Приоритизация и управление требованиями

## ЗАДАНИЕ
Приоритизируйте бэклог FoodExpress по методу MoSCoW для запуска MVP в одном городе.

## ВХОДНЫЕ ДАННЫЕ
Фичи-кандидаты:
1. Регистрация и вход.
2. Поиск ресторанов.
3. Оформление заказа.
4. Оплата картой.
5. Оплата бонусами.
6. Отслеживание курьера на карте.
7. Push-уведомления о статусе.
8. Промокоды.
9. Программа лояльности.
10. Отзывы о ресторанах.
11. Согласие на обработку ПДн.
12. Рекомендации на основе истории.

## ТРЕБОВАНИЯ
- Отнесите каждую фичу к Must / Should / Could / Won't (this time).
- Доля Must — не более ~60% объёма (обоснуйте, что без неё MVP не работает).
- Обоснуйте 3 самых спорных решения.
- Отметьте, какая фича — обязательна по регуляторике (не обсуждается).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `Фича | MoSCoW | Обоснование`. Ниже — пояснение к 3 спорным + пометка про регуляторику.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 12 фич распределены по MoSCoW.
- [ ] Must содержит ядро (регистрация, поиск, заказ, оплата картой) и не раздут (~60% макс).
- [ ] Согласие на ПДн (11) — Must по регуляторике.
- [ ] Лояльность, рекомендации, отзывы, оплата бонусами — Should/Could/Won't с обоснованием.
- [ ] Обоснованы 3 спорных решения.
- [ ] MVP из Must реально работоспособен (можно пройти весь путь).

## ПОДСКАЗКА
Must = «без этого MVP не запустится». Проверьте: если убрать фичу, можно ли вообще заказать еду? Если да — это не Must. Регуляторные требования (согласие на ПДн) — всегда Must, они не обсуждаются.
