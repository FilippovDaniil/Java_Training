# Задача 05 — Модуль 10: Приоритизация и управление требованиями

## ЗАДАНИЕ
Обработайте запрос на изменение (change request) через импакт-анализ: оцените последствия и подготовьте решение.

## ВХОДНЫЕ ДАННЫЕ
Проект FoodExpress в середине разработки. Заказчик присылает change request:
> «Нужно добавить оплату через СБП (Систему быстрых платежей) в дополнение к картам. Очень важно, хотим к запуску».

Контекст: оплата картой уже реализуется, есть интеграция с платёжным провайдером, спринт расписан, до запуска 4 недели.

## ТРЕБОВАНИЯ
- Проведите импакт-анализ: что затронет это изменение (требования, интеграции, модель данных, тесты, документация, сроки).
- Используйте трассировку: какие существующие артефакты связаны.
- Оцените влияние на сроки/бюджет/риски.
- Подготовьте варианты решения (принять сейчас / отложить / частично) с последствиями каждого.
- Укажите, кто Accountable за решение и что нужно обновить после принятия.

## ФОРМАТ РЕЗУЛЬТАТА
Импакт-анализ (что затронуто) + оценка влияния + варианты решения с последствиями + кто решает + что обновить.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Импакт-анализ покрывает: требования, интеграцию (новый провайдер СБП), модель данных (новый способ оплаты), тесты, документацию, сроки.
- [ ] Использована трассировка (что связано с оплатой).
- [ ] Оценено влияние на срок запуска (4 недели — реалистично ли).
- [ ] Предложено ≥2 варианта (например, в MVP только карты, СБП во 2-й релиз) с последствиями.
- [ ] Указан Accountable (PO/заказчик) и список обновлений (требования, трассировка, версия, бэклог).

## ПОДСКАЗКА
«Очень важно, хотим к запуску» — не основание принять без анализа. Импакт-анализ показывает истинную цену: новая интеграция = новый провайдер, новые сценарии, тесты, риски сертификации. Часто верное решение — карты в MVP, СБП в следующий релиз. Решает Accountable, СА даёт обоснованную картину.
