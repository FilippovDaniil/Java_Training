# Модуль 10 — Приоритизация и управление требованиями

## Зачем этот модуль

Требований всегда больше, чем ресурсов. Нельзя сделать «всё и сразу» — нужно решить, **что делать первым** (приоритизация) и **как управлять требованиями во времени**: отслеживать связи, обрабатывать изменения, не терять контроль. Это завершает Часть 2: вы научились выявлять, типизировать, моделировать и измерять требования — теперь учитесь ими управлять. На собеседовании middle+ спрашивают методы приоритизации (MoSCoW, RICE, Kano), трассировку и работу с изменениями.

---

## 1. Зачем приоритизация

```
Хотелок: 100      Ресурсов на: 20      →   Что из 100 делать первым?
```

Приоритизация отвечает на вопрос «что важнее» **объективно**, а не «кто громче просит». Без неё:
- делается то, что просит самый влиятельный, а не самое ценное;
- MVP размывается;
- команда распыляется.

> Задача СА — дать прозрачный, обоснованный способ ранжирования, чтобы решение принималось по ценности и затратам, а не по эмоциям.

---

## 2. Методы приоритизации

### MoSCoW

| Категория | Смысл |
|-----------|-------|
| **Must have** | без этого продукт не работает / не выйдет |
| **Should have** | важно, но можно временно без этого |
| **Could have** | желательно, если останется время |
| **Won't have (this time)** | сознательно не сейчас |

Простой, быстрый, понятен бизнесу. Минус: субъективность («всё Must»). Правило: Must — не более ~60% объёма.

### RICE

Формула приоритета:
```
RICE = (Reach × Impact × Confidence) / Effort
```
| Фактор | Что |
|--------|-----|
| **Reach** | сколько пользователей затронет (за период) |
| **Impact** | насколько сильно повлияет (шкала, напр. 0.25–3) |
| **Confidence** | уверенность в оценках (%) |
| **Effort** | трудозатраты (человеко-недели/месяцы) |

Плюс: числовой, сравнимый. Минус: оценки субъективны, точность мнимая.

### Kano

Классифицирует фичи по влиянию на удовлетворённость:

| Тип | Описание | Пример |
|-----|----------|--------|
| **Базовые (must-be)** | их отсутствие злит, наличие не радует | оплата работает |
| **Линейные (performance)** | чем больше, тем лучше | скорость доставки |
| **Восторг (delighters)** | неожиданная радость | бесплатный десерт в подарок |
| **Безразличные** | никому не важны | убрать |
| **Обратные** | раздражают | навязчивые уведомления |

> Базовые — обязательны (но не дают конкурентного преимущества), линейные — дифференциация, восторг — «вау», но со временем превращается в базовое.

### WSJF (Weighted Shortest Job First)

Из SAFe, для потока задач:
```
WSJF = Cost of Delay / Job Size
Cost of Delay = ценность + срочность + снижение риска/ценность возможности
```
Делать первым то, у чего высокая стоимость задержки и малый размер. Хорош для последовательности работ.

### Value vs Effort (матрица)

```
        Низкий Effort      Высокий Effort
Высокая │ Quick Wins      │ Big Bets       │
Value   │ (делать сразу)  │ (планировать)  │
        ├─────────────────┼────────────────┤
Низкая  │ Fill-ins        │ Не делать      │
Value   │ (по остатку)    │ (time sinks)   │
```

## 3. Трассировка требований (traceability)

**Матрица трассировки** связывает требования с их источниками и реализацией.

```
Бизнес-цель ──► Требование ──► Use case / Story ──► Дизайн/код ──► Тест-кейс
   (backward: откуда?)              (forward: где реализовано и проверено?)
```

- **Forward (вперёд):** от требования к реализации и тестам — всё ли сделано и проверено.
- **Backward (назад):** от требования к источнику/цели — зачем оно, нет ли «висящих».

Зачем СА:
- **покрытие:** видно, что не реализовано/не протестировано;
- **импакт-анализ:** при изменении видно, что затронется;
- **обоснованность:** «висящее» требование без цели — кандидат на удаление.

> Матрица трассировки — главный инструмент управления изменениями: меняя одно требование, по связям видишь всё затронутое.

---

## 4. Управление изменениями

Требования меняются — это норма. Важно управлять изменениями, а не запрещать их.

**Change request (запрос на изменение)** обрабатывается через **импакт-анализ**:

```
Запрос на изменение
   │
   ▼
Импакт-анализ: что затронется (по трассировке)?
   │   — какие требования/use cases/тесты/модули?
   │   — влияние на сроки, бюджет, риски?
   ▼
Решение (тот, кто Accountable): принять / отклонить / отложить
   │
   ▼
Обновить требования, трассировку, версию, бэклог
```

> В Waterfall изменение формально (допсоглашение), в Agile — переприоритизация бэклога. Но импакт-анализ нужен везде: «дешёвое» на вид изменение может потянуть цепочку.

---

## 5. Версионирование и baseline

- **Версионирование требований:** у требования есть история изменений (что, когда, кем, почему).
- **Baseline (базовая линия):** зафиксированный согласованный срез требований на момент (например, для релиза/контракта). От него считают изменения.
- **Статусы требования:** черновик → на согласовании → утверждено → реализуется → реализовано → отклонено.

> Baseline нужен, чтобы было от чего отсчитывать изменения и что считать «согласованным». Без него спор «а мы так не договаривались» неразрешим.

---

## ⚠️ Подводные камни

- **«Всё Must».** MoSCoW вырождается, если всё критично. Ограничивайте долю Must.
- **Мнимая точность RICE.** Числа выглядят объективно, но собраны из субъективных оценок. Используйте как поддержку решения, не как истину.
- **Приоритет по громкости.** Делается то, что просит влиятельный, а не ценное. Метод приоритизации защищает от этого.
- **Нет трассировки.** Изменение «дёшево», а тянет цепочку, которую не видно. Без матрицы импакт-анализ невозможен.
- **Запрет изменений.** «Требования заморожены» в меняющемся мире = неактуальный продукт. Управлять, а не запрещать.
- **Gold-plating.** Делается то, что никто не просил («было бы круто»). Лечится трассировкой к цели и приоритизацией.
- **Нет baseline.** Невозможно понять, что изменилось относительно согласованного.
