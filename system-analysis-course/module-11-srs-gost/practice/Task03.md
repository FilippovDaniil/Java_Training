# Задача 03 — Модуль 11: Спецификация (SRS, ТЗ ГОСТ)

## ЗАДАНИЕ
Найдите дефекты в фрагменте чужого ТЗ и предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент ТЗ на MedConnect:
1. «Система должна быть удобной и современной».
2. «Система должна работать быстро и надёжно».
3. «Система должна хранить данные пациентов» (без указания требований к защите).
4. «Система должна, по возможности, поддерживать видеосвязь».
5. «Пользователь нажимает синюю кнопку в правом углу для записи».
6. «Система должна обрабатывать заказы» (в ТЗ на телемедицину — копипаста из другого проекта).
7. Раздел «Порядок контроля и приёмки» отсутствует.

## ТРЕБОВАНИЯ
- Для каждого пункта определите дефект (непроверяемость, неполнота, неоднозначность «по возможности», UI в ТЗ, копипаста, отсутствие раздела).
- Предложите исправление.
- Отметьте дефекты, критичные для приёмки по ГОСТ (без чего нельзя принять систему).
- Укажите, какой обязательный раздел ГОСТ пропущен и чем это грозит.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `№ | Дефект | Исправление`. Ниже — критичные для приёмки дефекты + последствие пропуска раздела приёмки.

## КРИТЕРИИ ПРИЁМКИ
- [ ] 1, 2 — непроверяемость → измеримые формулировки.
- [ ] 3 — неполнота (нет требований к защите медданных) → добавить безопасность/152-ФЗ.
- [ ] 4 — «по возможности» недопустимо в ТЗ (неоднозначность приоритета) → чёткое требование или вынести.
- [ ] 5 — UI-детали в ТЗ → убрать в дизайн.
- [ ] 6 — копипаста из чужого проекта → удалить/исправить.
- [ ] 7 — отсутствие раздела «Порядок контроля и приёмки» опознано как критичный пробел.
- [ ] Отмечено, что без раздела приёмки систему формально нельзя сдать.

## ПОДСКАЗКА
В формальном ТЗ «по возможности», «удобно», «быстро» недопустимы — по ним нельзя принять работу. Раздел «Порядок контроля и приёмки» (р.6 ГОСТ) обязателен: без критериев приёмки заказчик и исполнитель не договорятся, что считать сдачей. Копипаста («обрабатывать заказы» в телемедицине) — частый и опасный дефект.
