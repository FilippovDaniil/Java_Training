# Задача 04 — Модуль 11: Спецификация (SRS, ТЗ ГОСТ)

## ЗАДАНИЕ
Для разных проектов выберите формат спецификации (SRS, ТЗ ГОСТ 34.602, бэклог+критерии в Agile) и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Проекты:
1. Госзаказ: система учёта обращений граждан, закупка по 44-ФЗ, приёмка по формальным критериям.
2. Коммерческий стартап: мобильное приложение, Scrum, требования меняются по обратной связи.
3. Крупный банк, новый модуль в регулируемой системе, нужна полная прослеживаемость и аудит.
4. Внутренний инструмент для своей команды, минимум формальностей.

## ТРЕБОВАНИЯ
- Для каждого проекта выберите формат спецификации и обоснуйте.
- Свяжите выбор с моделью разработки (модуль 02) и с регуляторикой.
- Объясните, почему для стартапа полный ТЗ ГОСТ — избыточен, а для госзаказа — обязателен.
- Укажите, какой минимально достаточный набор артефактов требований нужен в каждом случае.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: `Проект | Формат спецификации | Обоснование | Минимальный набор артефактов`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проект 1 → ТЗ по ГОСТ (госзакупка требует); 2 → бэклог+критерии приёмки (Agile); 3 → SRS/ТЗ с полной трассировкой; 4 → лёгкий формат (README/бэклог).
- [ ] Выбор связан с моделью разработки и регуляторикой.
- [ ] Объяснено, почему ГОСТ избыточен для стартапа и обязателен для госзаказа.
- [ ] Для каждого указан минимально достаточный набор артефактов.
- [ ] Нет ответа «везде SRS» без обоснования.

## ПОДСКАЗКА
Формат спецификации диктуется контекстом: госзакупка → ГОСТ (обязателен по закону/контракту); Agile-стартап → «достаточная» документация (бэклог + критерии приёмки); регулируемый банк → полная трассировка для аудита. «Достаточно» ≠ «ничего»: даже в Agile нужны критерии приёмки.
