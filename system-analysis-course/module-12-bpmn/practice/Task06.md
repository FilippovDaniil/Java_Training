# Задача 06 — Модуль 12: BPMN 2.0

## ЗАДАНИЕ
Найдите ошибки в чужой BPMN-схеме и предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
BPMN-схема процесса записи на приём MedConnect (с дефектами):
1. Два start-события без пояснения, оба ведут в одну задачу.
2. После XOR-шлюза «слот свободен?» ветка «нет» никуда не ведёт (тупик).
3. Параллельный шлюз (AND) развёл потоки «уведомить пациента» и «уведомить врача», а слил их XOR-шлюзом.
4. Между пулом «Пациент» и пулом «MedConnect» проведена сплошная стрелка (поток управления).
5. Шаг «ожидать оплату» нарисован как задача (task), хотя это ожидание события.
6. У одной из веток нет end-события.
7. Задача называется «Оплата» (существительное).

## ТРЕБОВАНИЯ
- Для каждого пункта определите ошибку и её тип.
- Предложите корректное исправление.
- Объясните, почему AND нельзя сливать через XOR.
- Объясните, почему между пулами не может быть потока управления.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 7 ошибок опознаны и классифицированы.
- [ ] 2, 6 — висящие/незавершённые ветки → довести до end.
- [ ] 3 — рассогласование шлюзов (AND развёл, XOR слил) → слить AND.
- [ ] 4 — поток управления между пулами → заменить на сообщение (пунктир).
- [ ] 5 — ожидание как задача → промежуточное событие (message/timer).
- [ ] 7 — задача названа существительным → глагол+объект («Провести оплату»).
- [ ] Объяснено, почему AND→XOR-слияние ломает логику (поток зависнет в ожидании второй ветки, которой XOR не дождётся).

## ПОДСКАЗКА
AND-слияние ждёт ВСЕ входящие ветки; XOR — любую одну. Если развели AND (два потока пошли), а слили XOR, логика рассинхронизируется. Между пулами — только сообщения. «Ожидать» — это событие, а не работа (задача).
