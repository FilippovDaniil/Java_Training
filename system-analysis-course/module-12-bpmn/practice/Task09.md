# Задача 09 — Модуль 12: BPMN 2.0

## ЗАДАНИЕ
Смоделируйте бизнес-процесс из текстового регламента: переведите описание «как есть» в корректную BPMN-схему.

## ВХОДНЫЕ ДАННЫЕ
Регламент MedConnect «Выписка электронного рецепта»:
> После завершения видеоконсультации врач решает, нужен ли пациенту рецепт. Если нужен — врач формирует рецепт в системе и подписывает его электронной подписью. Система проверяет подпись и отправляет рецепт в государственную систему рецептов. Если госсистема приняла рецепт, пациент видит его в приложении и получает уведомление. Если госсистема недоступна, система повторяет отправку каждые 5 минут до 3 раз; если все попытки неудачны — уведомляет врача для ручной обработки. Если рецепт не нужен — консультация просто завершается.

## ТРЕБОВАНИЯ
- Постройте BPMN-схему по регламенту.
- Выделите корректные шлюзы (нужен ли рецепт; принят ли госсистемой).
- Смоделируйте повтор отправки (до 3 раз, интервал 5 мин) с помощью цикла/таймера.
- Покажите оба исхода для «рецепт не нужен» и аварийный путь (ручная обработка).
- Отметьте, какие задачи — врача (user), какие — системы (service), какие — взаимодействие с внешней госсистемой (send/receive).

## ФОРМАТ РЕЗУЛЬТАТА
BPMN-схема (рисунок или текстовая нотация) + перечень шлюзов с типами + типы задач.

## КРИТЕРИИ ПРИЁМКИ
- [ ] XOR «нужен ли рецепт» с обеими ветками (нужен / не нужен → завершить).
- [ ] XOR «госсистема приняла?» с веткой повтора.
- [ ] Повтор отправки до 3 раз с интервалом 5 минут (timer + счётчик).
- [ ] Аварийный путь: 3 неудачи → уведомить врача (ручная обработка).
- [ ] Типы задач: формирование/подпись — user (врач); проверка подписи — service; отправка в госсистему — send/receive.
- [ ] Все ветки завершены корректно.

## ПОДСКАЗКА
Переводя регламент в BPMN, ищите маркеры: «если…то» → XOR-шлюз; «повторяет каждые 5 минут до 3 раз» → цикл с timer и счётчиком попыток; «если недоступна» → аварийная ветка. Взаимодействие с госсистемой — это send/receive (или message events), т.к. это внешний участник.
