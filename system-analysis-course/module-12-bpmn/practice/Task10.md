# Задача 10 (мини-проект) — Модуль 12: BPMN 2.0

## ЗАДАНИЕ
Постройте полную BPMN-модель ключевого бизнес-процесса MedConnect: «Запись → консультация → рецепт» — с участниками, ветвлениями, событиями и обработкой ошибок. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, сквозной процесс: пациент находит врача и записывается на слот → оплачивает → в назначенное время проходит видеоконсультацию → врач при необходимости выписывает рецепт → пациент получает итоги. Участники: Пациент, Система MedConnect, Врач, внешние системы (платёжный провайдер, провайдер видеосвязи, госсистема рецептов). Реальные сложности: врач может перенести приём; пациент может опоздать/не прийти; оплата может не пройти; видеосвязь может оборваться; госсистема рецептов может быть недоступна.

## ТРЕБОВАНИЯ
Постройте модель, включающую:
1. **Пулы и дорожки** — Пациент, MedConnect, Врач, внешние системы; межпуловые сообщения.
2. **Основной поток** — happy path от записи до получения итогов.
3. **Ветвления** (корректные шлюзы) — нужен ли рецепт, прошла ли оплата, состоялась ли консультация.
4. **События** — timer (опоздание, таймаут оплаты), error/сбои (обрыв видео, недоступность госсистемы), message (между участниками).
5. **Аварийные пути и компенсации** — отмена с возвратом денег, перенос, ручная обработка.
6. **Корректность** — все ветки завершены, шлюзы сведены, между пулами только сообщения.

## ФОРМАТ РЕЗУЛЬТАТА
Полная BPMN-схема (рисунок из bpmn.io/draw.io ИЛИ детальная текстовая нотация: пулы, дорожки, элементы, потоки, сообщения, типы шлюзов/событий). Можно с декомпозицией в подпроцессы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все участники в пулах; между пулами — только потоки сообщений.
- [ ] Основной поток покрывает запись → оплата → консультация → рецепт → итоги.
- [ ] Корректные шлюзы (нужен рецепт, оплата прошла, консультация состоялась) с обоснованным типом.
- [ ] Использованы timer (опоздание/таймаут), error (обрыв видео, госсистема), message события.
- [ ] Аварийные пути с компенсациями (возврат денег при отмене, перенос, ручная обработка).
- [ ] Все ветки завершены end-событиями; шлюзы корректно сведены.
- [ ] Схема читаема (при необходимости — подпроцессы).
- [ ] Учтена работа с медданными/регуляторика хотя бы в одном месте (например, доступ к рецепту).

## ПОДСКАЗКА
Это сквозной процесс MedConnect, который дальше детализируется в state machine (модуль 13) и sequence (модуль 33). Главные проверки middle+: реальные сложности (не только happy path), корректная семантика (XOR vs AND, сообщения между пулами, события для таймаутов/ошибок), читаемость через подпроцессы.
