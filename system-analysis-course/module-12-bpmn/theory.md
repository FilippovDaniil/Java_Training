# Модуль 12 — BPMN 2.0 (моделирование бизнес-процессов)

## Зачем этот модуль

Часть 2 описывала **что** система должна делать (требования). Часть 3 — про **как устроены процессы и поведение**, которые стоят за требованиями. Первый и главный инструмент — **BPMN 2.0** (Business Process Model and Notation): стандарт моделирования бизнес-процессов, понятный и бизнесу, и разработке. СА рисует BPMN постоянно: чтобы понять процесс «как есть», спроектировать «как должно быть», согласовать с заказчиком. На собеседовании middle+ обязательно: элементы BPMN, типы шлюзов и событий, пулы/дорожки, типичные ошибки.

---

## 1. Что такое BPMN

**BPMN** — графическая нотация для описания бизнес-процессов: последовательности действий, участников, развилок, событий. Главное достоинство — **единый язык** для бизнеса и ИТ.

```
[Старт] ──► (Действие) ──► <Развилка?> ──► (Действие) ──► [Конец]
 событие      задача         шлюз            задача         событие
```

BPMN описывает процесс как поток: что происходит, в каком порядке, кто делает, что ветвит ход, какие события влияют.

---

## 2. Основные элементы

| Группа | Элемент | Обозначение | Смысл |
|--------|---------|-------------|-------|
| **События (Events)** | старт / промежуточное / конец | кружок (тонкий/двойной/жирный) | что-то происходит |
| **Активности (Activities)** | задача (task), подпроцесс | прямоугольник со скруглением | работа, которую делают |
| **Шлюзы (Gateways)** | ветвление/слияние | ромб | управление потоком |
| **Потоки (Flows)** | поток управления | сплошная стрелка | последовательность |
| | поток сообщений | пунктирная стрелка | обмен между участниками |
| **Пулы/дорожки (Pools/Lanes)** | участники | прямоугольные области | кто выполняет |
| **Артефакты** | данные, аннотации | | дополнительная информация |

### Типы задач (важно для middle+)
- **User task** — выполняет человек через систему;
- **Service task** — выполняет система автоматически;
- **Send/Receive task** — отправка/получение сообщения;
- **Manual task** — ручная работа вне системы;
- **Script/Business rule task** — скрипт / правило.

> Различение user/service task важно: оно показывает, что делает человек, а что — система (граница автоматизации).

---

## 3. События (Events)

| Тип события | Когда | Пример |
|-------------|-------|--------|
| **Start** (тонкий круг) | начало процесса | поступил заказ |
| **End** (жирный круг) | завершение | заказ доставлен |
| **Intermediate** (двойной круг) | в ходе процесса | получено подтверждение |

Триггеры событий (значок внутри круга):
- **Message** (конверт) — пришло/отправлено сообщение;
- **Timer** (часы) — по времени/таймауту;
- **Error** (молния) — ошибка;
- **Signal** — широковещательный сигнал;
- **Conditional** — по условию.

> Timer и Error события — ключевые для моделирования реальных процессов: «отменить заказ, если ресторан не подтвердил за 10 минут» (timer), «откатить при сбое оплаты» (error).

---

## 4. Шлюзы (Gateways)

Шлюз управляет потоком — ветвит или сливает.

| Шлюз | Значок | Смысл | Пример |
|------|--------|-------|--------|
| **Exclusive (XOR)** | ✕ или пустой ромб | ровно ОДИН путь по условию | оплата успешна? да/нет |
| **Parallel (AND)** | ✛ | ВСЕ пути одновременно | параллельно: уведомить ресторан И клиента |
| **Inclusive (OR)** | ◯ | ОДИН ИЛИ НЕСКОЛЬКО по условиям | применить скидку и/или промокод |
| **Event-based** | ◇ со значком | путь по первому наступившему событию | ждём оплату ИЛИ таймаут |

```
XOR (исключающий):        AND (параллельный):       Event-based:
   ──<X>── да ──►            ──<+>──► задача А          ──<◇>──► оплата получена
       └── нет ──►               └──► задача Б              └──► таймаут истёк
   (один путь)              (оба пути)                  (первое событие)
```

> Правило: что **разветвили** одним шлюзом, нужно корректно **слить** (часто шлюзом того же типа). Несведённые ветки или XOR-слияние параллельных потоков — частая ошибка.

---

## 5. Пулы и дорожки (Pools / Lanes)

- **Пул (Pool)** — отдельный участник процесса (организация/система). Между пулами — **потоки сообщений** (пунктир).
- **Дорожка (Lane)** — роль/подразделение внутри пула. Внутри пула — **потоки управления** (сплошные).

```
┌─Пул: FoodExpress ──────────────────────────────────┐
│ Дорожка: Система   [старт]►(принять заказ)►...      │
│ Дорожка: Оператор             (разобрать проблему)  │
└──────────────────▲──────────────────│──────────────┘
                   │ сообщение         │ сообщение (пунктир)
┌─Пул: Клиент ─────┴──────────────────▼──────────────┐
│                  (оформить заказ)                    │
└─────────────────────────────────────────────────────┘
```

**Оркестрация vs хореография:**
- **Оркестрация** — процесс внутри одного пула (как дирижёр управляет);
- **Хореография** — взаимодействие между пулами через сообщения (кто кому что шлёт).

> Между пулами ходят только сообщения (пунктир), внутри пула — поток управления (сплошная). Смешение — типичная ошибка.

---

## 6. Подпроцессы

**Подпроцесс** — свёрнутая часть процесса, которую можно раскрыть отдельно. Нужен, чтобы:
- не загромождать главную схему;
- переиспользовать процесс;
- декомпозировать сложное.

```
(Оформить заказ) [+]   ← свёрнутый подпроцесс (плюсик),
                          раскрывается в отдельную детальную схему
```

> Если процесс на одной схеме не читается (слишком много элементов) — декомпозируйте в подпроцессы. Уровень детализации подбирается под аудиторию.

---

## 7. Типичные ошибки

| Ошибка | Описание | Как правильно |
|--------|----------|---------------|
| **Висящий шлюз** | ветвление без слияния / тупиковая ветка | свести ветки корректно |
| **Поток управления между пулами** | сплошная стрелка между участниками | между пулами — только сообщения (пунктир) |
| **XOR-слияние параллельных** | развели AND, слили XOR | слить тем же типом |
| **Нет конца у ветки** | ветка ни к чему не ведёт | каждый путь к end-событию |
| **Задача вместо события** | «ждать оплату» как задача | это промежуточное событие |
| **Слишком детально/крупно** | вся схема либо мусор, либо «чёрный ящик» | подбирать уровень, использовать подпроцессы |
| **Несколько start без смысла** | непонятный запуск | обычно один явный старт |

## ⚠️ Подводные камни

- **BPMN как блок-схема.** BPMN — не просто «квадратики и стрелки»: у элементов строгая семантика (типы шлюзов, событий). Использовать по назначению.
- **Перепутать XOR и AND.** XOR — один путь, AND — все. Самая частая смысловая ошибка, меняет логику процесса.
- **Поток управления вместо сообщений между пулами.** Между участниками — только message flow (пунктир).
- **Игнорировать ошибки и таймауты.** Реальный процесс имеет сбои и сроки; модель только happy path бесполезна.
- **Моделировать «как хочется», не проверив «как есть».** Сначала понять реальный процесс (наблюдение, модуль 05), потом проектировать «как должно».
- **Перегруз деталями.** Схема, которую нельзя прочитать, не нужна. Декомпозиция в подпроцессы.
