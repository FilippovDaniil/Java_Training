# Задача 01 — Модуль 13: UML поведенческие

## ЗАДАНИЕ
Постройте activity diagram (диаграмму деятельности) процесса обработки заказа в FoodExpress — поток действий от оформления до завершения, с ветвлением и параллельными шагами.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress.** Обработка заказа: клиент оформляет заказ → система проверяет оплату → если оплата прошла, заказ передаётся ресторану, и **одновременно** клиенту отправляется подтверждение, а курьеру — предзаказ на доставку; после готовности заказ забирает курьер и доставляет; если оплата не прошла — заказ отменяется. Дорожки: Клиент, Система, Ресторан, Курьер.

## ТРЕБОВАНИЯ
- Начальный и конечный узлы.
- Действия-глаголы («Оформить заказ», «Проверить оплату», …).
- Узел решения (decision) с guard'ами на ветке оплаты (`[оплата прошла]` / `[оплата не прошла]`).
- Развилку/слияние (fork/join) для параллельных действий после успешной оплаты.
- Дорожки (swimlanes): кто выполняет каждое действие.
- Все ветки завершены.

## ФОРМАТ РЕЗУЛЬТАТА
Activity diagram (рисунок из draw.io/PlantUML ИЛИ текстовая нотация: узлы, переходы, guard'ы, fork/join, дорожки).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Есть начальный и конечный узлы.
- [ ] Действия названы глаголами.
- [ ] Decision на оплате с взаимоисключающими guard'ами.
- [ ] Параллельные действия оформлены через fork/join (не через decision).
- [ ] Действия распределены по дорожкам (Клиент/Система/Ресторан/Курьер).
- [ ] Обе ветки оплаты завершены (доставка / отмена).

## ПОДСКАЗКА
Decision (ромб) = выбор одной ветки по условию. Fork (черта) = параллелизм, синхронизируется join. «Передать ресторану» + «уведомить клиента» + «предзаказ курьеру» идут одновременно → это fork, а не три decision-ветки.
