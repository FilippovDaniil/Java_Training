# Задача 02 — Модуль 13: UML поведенческие

## ЗАДАНИЕ
Постройте state machine diagram (диаграмму состояний) жизненного цикла заказа FoodExpress: все статусы заказа и переходы между ними по событиям.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress.** Заказ проходит статусы: Создан → Оплачен → Принят рестораном → Готовится → Готов → В доставке → Доставлен. На пути возможны: отмена клиентом (до передачи в ресторан) с возвратом средств; отказ ресторана; неуспешная оплата; недоставка (курьер не довёз) с разбором. Финальные исходы: Доставлен, Отменён.

## ТРЕБОВАНИЯ
- Начальное псевдосостояние и финальные состояния.
- Все статусы как состояния (существительные/причастия, не действия).
- Переходы, подписанные **событиями** (`оплата получена`, `принят рестораном`, `отмена`, …).
- Где уместно — guard и действие при переходе (например, `отмена [до передачи] / вернуть оплату`).
- Корректность: у каждого нефинального состояния есть выход; есть путь к финалу.

## ФОРМАТ РЕЗУЛЬТАТА
State machine diagram (рисунок/PlantUML ИЛИ текстовая нотация: список состояний + переходы вида `СостояниеA --событие[guard]/действие--> СостояниеB`).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Состояния названы статусами, а не действиями.
- [ ] Есть начальное псевдосостояние и финальные состояния (Доставлен, Отменён).
- [ ] Переходы подписаны событиями.
- [ ] Указаны guard/действие хотя бы на переходе отмены (возврат средств).
- [ ] Нет тупиковых (нефинальных без выхода) и недостижимых состояний.
- [ ] Из каждого статуса есть путь к финалу.

## ПОДСКАЗКА
State machine ≠ activity: узлы — статусы, переходы — события. «Оплатить» — это не состояние; «Оплачен» — состояние, а переводит в него событие «оплата получена». Эта же модель дальше станет валидацией смены статуса на бэкенде (модуль 29, PATCH).
