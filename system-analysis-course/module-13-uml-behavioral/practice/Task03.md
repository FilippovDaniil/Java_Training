# Задача 03 — Модуль 13: UML поведенческие

## ЗАДАНИЕ
Спроектируйте корректные guard-условия и события переходов для набора ситуаций. Покажите, что guard'ы взаимоисключающие и полные.

## ВХОДНЫЕ ДАННЫЕ
Фрагменты логики (для FoodExpress), которые нужно выразить переходами с событиями и guard'ами:
1. После оплаты: если сумма ≥ 2000 ₽ — заказ получает бесплатную доставку, иначе — платную.
2. При отмене: если заказ ещё не передан ресторану — полный возврат; если готовится — возврат 50%; если в доставке — отмена невозможна.
3. По таймеру: если ресторан не подтвердил заказ за 10 минут — автоотмена с возвратом.
4. Повторная оплата: попытка оплаты разрешена не более 3 раз, после — заказ отменяется.

## ТРЕБОВАНИЯ
- Для каждого пункта запишите переход(ы) в нотации `Состояние --событие [guard] / действие--> Состояние`.
- Guard'ы для одного события из одного состояния — взаимоисключающие и полные (используйте `[else]` где уместно).
- Явно отметьте, где переход **запрещён** (нет стрелки) — пункт 2 «в доставке».
- Для пункта 4 покажите счётчик попыток (guard по счётчику).

## ФОРМАТ РЕЗУЛЬТАТА
Список переходов по пунктам + краткое пояснение, почему guard'ы взаимоисключающие/полные.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каждый пункт выражен переходом(ами) с событием и guard.
- [ ] Guard'ы по одному событию взаимоисключающие и покрывают все случаи.
- [ ] Запрещённый переход (отмена в доставке) отмечен как отсутствие перехода/недопустимость.
- [ ] Действия при переходе указаны (возврат полный/50%/нет).
- [ ] Счётчик попыток оплаты отражён в guard (например, `[попытка < 3]` / `[попытка = 3]`).

## ПОДСКАЗКА
Guard — логическое условие в `[ ]` на переходе. Несколько переходов из одного состояния по одному событию различаются guard'ами; они не должны пересекаться (иначе недетерминизм) и должны покрывать все варианты (иначе зависание). Счётчик — это либо данные объекта, либо отдельный атрибут в guard.
