# Задача 08 — Модуль 13: UML поведенческие

## ЗАДАНИЕ
Свяжите state machine с бизнес-правилами: оформите статусную модель как таблицу допустимых переходов и покажите, как она транслируется в валидацию и права.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, статусы заказа: Создан, Оплачен, Принят, Готовится, В доставке, Доставлен, Отменён. Бизнес-правила:
- отменить заказ может клиент, но только до статуса «Принят»; оператор — на любом статусе до «В доставке»;
- перевести «Готов» → «В доставке» может только курьер;
- возврат средств: полный — при отмене до «Готовится»; невозможен — после «В доставке»;
- из «Доставлен» и «Отменён» переходов нет (финальные).

## ТРЕБОВАНИЯ
- Таблица переходов: `Из статуса | Событие | В статус | Guard (бизнес-правило) | Кто инициирует | Действие`.
- Явно перечислите **запрещённые** переходы (что нельзя).
- Покажите, как таблица превращается в:
  1. валидацию допустимости смены статуса (бэкенд);
  2. правила доступа (кто может инициировать переход).
- 1–2 примера запроса на недопустимый переход и ожидаемую реакцию (отказ).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица переходов + список запрещённых переходов + краткое описание трансляции в валидацию и права + примеры отказов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Таблица переходов покрывает все статусы и события.
- [ ] Guard'ы отражают бизнес-правила (сроки отмены, условия возврата).
- [ ] Указано, кто инициирует каждый переход (клиент/оператор/курьер/система).
- [ ] Перечислены запрещённые переходы.
- [ ] Показана трансляция в валидацию смены статуса и в права доступа.
- [ ] Приведены примеры отказа на недопустимый переход.

## ПОДСКАЗКА
Хорошая state machine = таблица «откуда-куда-при каком условии-кто». Запрещённый переход — это отсутствие строки в таблице → бэкенд возвращает ошибку (например, 409/422). Права на переход (кто может) ложатся в авторизацию. Эта таблица — прямой вход для модулей 29 (PATCH статуса) и 34 (доступ).
