# Задача 10 (мини-проект) — Модуль 13: UML поведенческие

## ЗАДАНИЕ
Постройте полную поведенческую модель ключевой сущности FoodExpress — заказа: **state machine** (жизненный цикл со всеми статусами, событиями, guard'ами, действиями) **+ activity diagram** одного нетривиального этапа обработки. Свяжите модель с бизнес-правилами. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress.** Сущность «Заказ». Полный жизненный цикл: оформление → оплата (с повторными попытками и таймаутом) → подтверждение рестораном (с таймаутом автоотмены) → готовка → передача курьеру → доставка → завершение. Возможны: отмена (с правилами возврата), отказ ресторана, неуспешная доставка (разбор/возврат), параллельные действия после оплаты (резерв на кухне + назначение курьера + начисление бонусов). Бизнес-правила отмены/возврата — как в задачах 03 и 08.

## ТРЕБОВАНИЯ
Сдайте пакет из 3 частей:
1. **State machine заказа** — все статусы, переходы по событиям, guard'ы и действия (возвраты, таймауты), начальное и финальные состояния. Корректность: нет тупиков/недостижимости, есть путь к финалу, переходы детерминированы.
2. **Activity diagram этапа «обработка после оплаты»** — с fork/join для параллельных действий (резерв/курьер/бонусы) и decision где нужно.
3. **Таблица переходов + бизнес-правила** — `Из | Событие | В | Guard | Кто | Действие`, список запрещённых переходов; короткое пояснение, как это станет валидацией статуса и правами (связь с модулями 29, 34).

## ФОРМАТ РЕЗУЛЬТАТА
Три артефакта: state machine (рисунок/PlantUML/текст) + activity diagram + таблица переходов с правилами. Краткое сопроводительное описание (что моделирует каждая диаграмма и как они связаны).

## КРИТЕРИИ ПРИЁМКИ
- [ ] State machine покрывает полный жизненный цикл заказа с финальными исходами.
- [ ] Учтены таймауты (оплата, подтверждение рестораном) и повторные попытки оплаты.
- [ ] Guard'ы и действия отражают бизнес-правила отмены/возврата.
- [ ] Нет тупиковых/недостижимых состояний; переходы детерминированы (различающие guard).
- [ ] Activity diagram этапа после оплаты использует fork/join корректно (синхронизация перед продолжением).
- [ ] Таблица переходов + запрещённые переходы + связь с валидацией/правами.
- [ ] Объяснено, чем здесь различаются роли state machine и activity.

## ПОДСКАЗКА
Это синтез: state machine из задач 02/03/08, activity с fork/join из задач 01/07, таблица правил из 08. Главные проверки middle+: правильное разделение «статус (state machine) vs шаги (activity)», корректность (тупики/недостижимость/детерминизм), реальные сложности (таймауты, повторы, компенсации), прямая связь модели с реализацией (валидация переходов + права). Модель дальше детализируется в sequence (модуль 33) и ляжет в основу API смены статуса (модуль 29).
