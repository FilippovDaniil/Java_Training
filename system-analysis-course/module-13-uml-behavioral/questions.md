## Ключ (эталонные ответы)

1. **Activity** моделирует поток действий/алгоритм (что делается и в каком порядке). **State machine** моделирует жизненный цикл одного объекта (в каких статусах он бывает и что его переводит между ними).
2. Начальный узел, действия (actions), узлы решения/слияния (decision/merge), развилка/слияние (fork/join), конечный узел (activity final), конец потока (flow final), дорожки (partitions), объектные узлы.
3. **Decision** (ромб) выбирает ОДНУ ветку по условию (или-или, аналог BPMN XOR). **Fork** (толстая черта) запускает ВСЕ ветки параллельно (и-и, аналог AND).
4. **Merge** (ромб) объединяет альтернативные ветки без синхронизации (просто сводит поток). **Join** (черта) синхронизирует параллельные ветки — ждёт завершения всех входящих.
5. **Guard** — логическое условие в квадратных скобках `[условие]` на переходе/ветке; переход срабатывает, только если guard истинен.
6. **Взаимоисключающими** (не пересекаются — иначе недетерминизм) и **полными** (покрывают все случаи — иначе зависание). Часто добавляют `[else]`.
7. **Fork** разделяет поток на параллельные ветки, **join** их синхронизирует. Join — барьер: дальше поток пойдёт, только когда завершатся ВСЕ входящие ветки.
8. Join не дождётся всех веток и не пропустит поток — вся деятельность встанет (зависнет). Поэтому в реальных системах нужны таймауты/компенсации.
9. **Activity final** (◉) завершает ВСЮ деятельность (останавливает все потоки). **Flow final** (круг с ✕) завершает только ОДНУ ветку, остальные продолжают выполняться.
10. Начальное псевдосостояние, состояния (states), переходы (transitions), финальное состояние, составные состояния, параллельные регионы; на переходах — события, guard, действия.
11. `событие [guard] / действие`: переход срабатывает по **событию**, если истинен **guard**, и при срабатывании выполняется **действие** (эффект).
12. **Состояние** — устойчивый статус объекта (существительное: «Оплачен»), в котором он находится во времени. **Действие** — выполняемый шаг (глагол: «оплатить»). State machine оперирует состояниями, activity — действиями.
13. **entry** — действие при входе в состояние, **exit** — при выходе, **do** — деятельность, выполняемая пока объект в состоянии.
14. **Internal transition** — реакция на событие внутри состояния без его смены (entry/exit не срабатывают, объект остаётся в том же состоянии).
15. Состояние, в которое нет ни одного входящего перехода (кроме явного начального) — в него невозможно попасть. Ошибка моделирования.
16. Нефинальное состояние без исходящих переходов — объект в нём «застревает», дальше двигаться некуда.
17. Если из одного состояния по одному и тому же событию есть два перехода без различающих (или с пересекающимися) guard — непонятно, какой сработает. Это недетерминизм.
18. **Составное состояние** содержит вложенные подсостояния. **Параллельные регионы** (состояние, разделённое пунктиром) — объект одновременно находится в подсостоянии каждого региона.
19. Activity — часть UML, описывает алгоритм/логику (в т.ч. внутри системы), беднее по событиям. BPMN — отдельный стандарт для бизнес-процессов: пулы/дорожки, message flow между участниками, богатая семантика событий (timer/message/error), единый язык для бизнеса и ИТ.
20. Если есть «статус» сущности и список его значений → state machine. Если описывается последовательность шагов/алгоритм → activity. State machine — про «в каком состоянии объект», activity — про «что делается».
21. State machine формализует бизнес-правила: разрешённые переходы = что допустимо; guard = бизнес-условия; действия при переходе = бизнес-эффекты; отсутствие стрелки = запрещённый переход.
22. Переходом по событию-таймеру: из состояния выходит переход «по истечении N времени» (например, `after 10 мин`), уводящий по аварийному пути (автоотмена, «не состоялась»).
23. Существительные-статусы («становится действующим») → состояния; глаголы/«можно сделать» → события; «если … то» → guard; «возвращается на …» → переход обратно (цикл). Сопоставлять фразы текста с элементами.
24. Таблица допустимых переходов (откуда-куда-событие-guard-кто) превращается в валидацию смены статуса на бэкенде (недопустимый переход → ошибка, напр. 409/422) и в правила доступа (кто может инициировать переход).
25. Иначе модель некорректна: нет выхода → объект застревает (тупик); нет входа → состояние недостижимо. Корректная модель гарантирует, что объект из любого статуса может дойти до завершения жизненного цикла.
