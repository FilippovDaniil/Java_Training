# Модуль 13 — UML поведенческие диаграммы (Activity, State Machine)

## Зачем этот модуль

В модуле 12 вы моделировали бизнес-процессы в BPMN. UML предлагает два дополняющих взгляда на **поведение**:
- **Activity diagram** (диаграмма деятельности) — поток действий/алгоритм: близка к BPMN, но это часть UML и чаще описывает логику системы, а не бизнес-процесс с участниками.
- **State machine diagram** (диаграмма состояний) — **жизненный цикл одного объекта**: в каких состояниях он бывает и по каким событиям переходит.

СА использует их постоянно: activity — чтобы показать алгоритм/процесс обработки; state machine — чтобы зафиксировать статусную модель сущности (заказ, заявка, консультация). На собеседовании middle+ обязательно спрашивают: чем activity отличается от BPMN, что такое guard, fork/join, как читать диаграмму состояний и где у неё тупики.

---

## 1. Activity diagram (диаграмма деятельности)

Описывает **поток работ/алгоритм**: действия, их порядок, ветвления, параллелизм. Похожа на блок-схему и BPMN, но с UML-семантикой.

### Основные элементы

| Элемент | Обозначение | Смысл |
|---------|-------------|-------|
| **Начальный узел** (initial) | закрашенный круг ● | старт потока |
| **Действие** (action) | прямоугольник со скруглением | выполняемый шаг |
| **Узел решения** (decision) | ромб с 1 входом, N выходами | ветвление по условию |
| **Узел слияния** (merge) | ромб с N входами, 1 выходом | объединение веток (без синхронизации) |
| **Развилка** (fork) | толстая черта, 1 вход → N выходов | старт параллельных потоков |
| **Слияние** (join) | толстая черта, N входов → 1 выход | синхронизация параллельных потоков |
| **Конечный узел** (activity final) | круг в круге ◉ | завершение всей деятельности |
| **Конец потока** (flow final) | круг с ✕ | завершение одной ветки (остальные живут) |
| **Дорожки** (swimlanes/partitions) | вертикальные/горизонтальные полосы | кто/что выполняет действие |
| **Объектный узел** (object node) | прямоугольник | данные, передаваемые между действиями |

```
●──►[Принять заказ]──►◇ оплата прошла? ──нет──►[Отменить заказ]──►◉
                       │ да
                       ▼
                  [Передать в ресторан]──►◉
```

### Decision/merge vs fork/join — ключевое различие

- **Decision (ромб)** — выбирается **ОДНА** ветка по условию (аналог BPMN XOR). Сливается **merge** (ромб).
- **Fork (черта)** — запускаются **ВСЕ** ветки параллельно (аналог BPMN AND). Синхронизируется **join** (черта): ждёт завершения всех входящих.

> Частая ошибка: путать ромб и черту. Ромб = выбор (или-или), черта = параллелизм (и-и). Decision сливают merge, fork синхронизируют join — не смешивать.

### Guard-условия на переходах решения

На стрелках из узла решения пишут **guard** в квадратных скобках — условие выбора ветки:

```
◇ ──[сумма > 1000]──► [применить скидку]
  └─[сумма ≤ 1000]──► [без скидки]
```

Guard'ы должны быть **взаимоисключающими и полными** (покрывать все случаи), иначе поток зависнет или пойдёт двумя путями. Полезен `[else]` для «всех остальных».

## 2. State machine diagram (диаграмма состояний)

Описывает **жизненный цикл ОДНОГО объекта/сущности**: множество состояний и переходы между ними по событиям. Это не про порядок шагов алгоритма, а про «в каком статусе находится объект и что переводит его в следующий».

### Основные элементы

| Элемент | Обозначение | Смысл |
|---------|-------------|-------|
| **Начальное псевдосостояние** | закрашенный круг ● | точка входа в жизненный цикл |
| **Состояние** (state) | прямоугольник со скруглением + имя | устойчивый статус объекта |
| **Переход** (transition) | стрелка между состояниями | смена состояния |
| **Финальное состояние** | круг в круге ◉ | конец жизненного цикла |
| **Составное состояние** (composite) | состояние с вложенными | группировка/вложенность |
| **Регион параллелизма** | состояние, делённое пунктиром | одновременные подсостояния |

### Синтаксис перехода

```
        событие [guard] / действие
СостояниеA ─────────────────────────► СостояниеB
```

- **Событие (trigger)** — что инициирует переход (`оплата получена`, `таймаут`, `отмена`).
- **Guard [условие]** — логическое условие, при котором переход разрешён.
- **Действие (effect) / ...** — что выполняется при переходе (side effect).

Внутри состояния возможны:
- **entry / действие** — выполняется при входе в состояние;
- **exit / действие** — при выходе;
- **do / деятельность** — пока объект в состоянии;
- **internal transition** — реакция на событие без смены состояния.

### Пример: жизненный цикл заказа FoodExpress

```
● ──► [Создан] ──оплата получена──► [Оплачен] ──принят рестораном──► [Готовится]
         │                              │                                │
    отмена/возврат                 не подтверждён 10 мин/возврат      готов
         ▼                              ▼                                ▼
    [Отменён] ◄────────────────────────┘                          [В доставке]
         │                                                              │
         ▼                                                          доставлен
        ◉                                                              ▼
                                                                  [Доставлен] ──► ◉
```

> Каждое состояние — существительное/причастие («Оплачен», «В доставке»), а не действие. Переход подписан событием (и при необходимости guard/действием).

## 3. Activity vs State machine — когда что

| Критерий | Activity | State machine |
|----------|----------|---------------|
| **Что моделирует** | поток действий/алгоритм | жизненный цикл одного объекта |
| **Узлы** | действия (глаголы) | состояния (статусы) |
| **Двигатель** | завершение предыдущего действия | наступление события |
| **Вопрос** | «что делается и в каком порядке?» | «в каком статусе объект и что его меняет?» |
| **Аналогия** | как готовят блюдо (шаги) | статус заказа в приложении |
| **Параллелизм** | fork/join | параллельные регионы |

> Простая эвристика: если в ТЗ есть слово **«статус»/«состояние»** сущности и список значений (Создан/Оплачен/…) — это **state machine**. Если описывается **последовательность шагов** обработки — это **activity**.

---

## 4. Activity vs BPMN — когда что

Activity и BPMN визуально близки (действия, ветвления, параллелизм, дорожки). Различия:

| Критерий | Activity (UML) | BPMN 2.0 |
|----------|----------------|----------|
| **Назначение** | алгоритм/логика, в т.ч. внутри системы | бизнес-процесс с участниками |
| **Аудитория** | разработка, проектирование | бизнес + ИТ (единый язык) |
| **Участники** | дорожки (partitions) | пулы/дорожки, message flow между пулами |
| **События** | беднее (нет богатой типологии) | богатая семантика (timer/message/error/signal) |
| **Стандарт** | часть UML | отдельный стандарт OMG для процессов |

> Практика: для **бизнес-процесса** (особенно межролевого, с таймаутами/сообщениями) — BPMN. Для **алгоритма/логики системы** или быстрой схемы внутри проектной документации — activity. Многие команды для процессов используют только BPMN, а UML — для state machine и структурных диаграмм.

## 5. Тупиковые и недостижимые состояния (корректность state machine)

Проверки качества статусной модели:

- **Недостижимое состояние** — в него нет ни одного входящего перехода (кроме явного начального). Ошибка моделирования.
- **Тупиковое состояние (deadlock)** — из него нет исходящих переходов, и оно **не финальное**. Объект «застревает».
- **Неполнота переходов** — для состояния не описана реакция на возможное событие (что будет при «отмене» в статусе «Готовится»?).
- **Неоднозначность** — два перехода по одному событию без различающих guard → недетерминизм.
- **Нет пути к финалу** — из некоторых состояний невозможно дойти до завершения жизненного цикла.

```
[Создан] ──► [Оплачен] ──► [Завис]     ← тупик: нет выхода, но не финал
                              ✕
[Архивный]   ← недостижимо: нет входящих переходов
```

> На собеседовании любят дать «битую» диаграмму и попросить найти недостижимые/тупиковые состояния и неполные реакции. Проверяйте: у каждого нефинального состояния есть выход; в каждое (кроме начального) есть вход; на каждое возможное событие есть реакция.

## 6. Параллелизм: fork/join и параллельные регионы

- **Activity:** `fork` запускает параллельные действия, `join` синхронизирует (ждёт все). Между ними потоки независимы.
- **State machine:** **составное состояние с регионами** (разделено пунктиром) — объект одновременно находится в подсостоянии каждого региона.

```
Activity fork/join:                State machine параллельные регионы:
        ┃ fork                     ┌─[Заказ оформляется]──────────┐
   ┌────╋────┐                     │ Оплата: [Ожидает]→[Оплачено] │
[готовить] [уведомить]             │╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌│
   └────╋────┘                     │ Документы: [Нет чека]→[Чек]  │
        ┃ join                     └──────────────────────────────┘
```

> join — это барьер: дальше поток пойдёт, только когда **все** параллельные ветки завершатся. Если хоть одна зависла — застрянет весь поток (аналог BPMN AND-слияния).

## 7. Связь state machine с бизнес-правилами

Статусная модель — формализация бизнес-правил перехода:
- **Разрешённые переходы** = что бизнес допускает («отменить можно только до передачи в ресторан»).
- **Guard** = бизнес-условие («возврат полный, если отмена до начала готовки»).
- **Действия при переходе** = бизнес-эффекты («при отмене инициировать возврат средств»).
- **Запрещённые переходы** = отсутствие стрелки (нельзя из «Доставлен» в «Готовится»).

> Хорошая state machine — это таблица «из какого статуса в какой можно/нельзя и при каком условии». Она напрямую транслируется в валидацию на бэкенде (проверка допустимости смены статуса) и в права (кто может инициировать переход).

## ⚠️ Подводные камни

- **Путать decision (ромб) и fork (черта)** в activity. Ромб — выбор одной ветки, черта — параллелизм. Меняет логику.
- **Состояние ≠ действие.** В state machine узлы — статусы (существительные: «Оплачен»), а не шаги («оплатить»). Шаги — это activity.
- **Незакрытый параллелизм.** Развели fork — синхронизируйте join; иначе activity-final преждевременно закроет «живые» ветки.
- **Неполные/пересекающиеся guard.** Условия на переходах должны покрывать все случаи и не пересекаться (недетерминизм).
- **Тупики и недостижимость.** Каждое нефинальное состояние имеет выход; в каждое — вход; есть путь к финалу.
- **Моделировать activity там, где нужна state machine** (и наоборот). «Список статусов» → state machine; «последовательность шагов» → activity.
- **Слишком много состояний.** Если статусов десятки — часть, вероятно, не статусы, а данные/флаги. Состояние должно менять поведение объекта.
