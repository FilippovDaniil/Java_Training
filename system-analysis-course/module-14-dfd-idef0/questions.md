# Вопросы для самопроверки — Модуль 14: DFD и IDEF0

## Ключ (эталонные ответы)

1. DFD моделирует **движение данных**: процессы (преобразования), потоки, хранилища, внешние сущности. НЕ показывает порядок во времени, условия и поток управления (это не блок-схема).
2. Внешняя сущность (источник/приёмник), процесс (преобразование данных), хранилище данных (data store), поток данных (стрелка).
3. **Контекстная диаграмма** — вся система как ОДИН процесс + внешние сущности + потоки между ними. Показывает границы системы и её окружение.
4. На уровне 1 единый процесс разложен на несколько пронумерованных процессов, и появляются **хранилища данных**.
5. **Хранилище** — место, где данные покоятся (БД, файл, реестр). Пассивно: само данные не двигает, доступ только через процессы.
6. **Баланс**: входные и выходные потоки процесса на родительском уровне должны совпадать с потоками его детальной диаграммы. Декомпозиция не «теряет» и не «добавляет» граничные потоки.
7. **Чёрная дыра** — процесс, у которого есть вход, но нет ни одного выхода (данные входят, ничего не выходит). Ошибка.
8. **Чудо** — процесс, у которого есть выход, но нет входов (результат «из ниоткуда»). Ошибка.
9. Хранилища пассивны — данные не перемещаются сами. Между двумя хранилищами всегда должен стоять процесс, который читает из одного и пишет в другое.
10. Обмен между двумя внешними сущностями происходит вне системы, поэтому в DFD системы не моделируется (он не касается нашей системы).
11. **Поток данных** — конкретные данные (заказ, чек). **Поток управления** — команда/сигнал («начать»). В DFD моделируются только потоки данных, не управление.
12. Процессы уровня 1 — 1, 2, 3…; их детализация на уровне 2 — 1.1, 1.2…; и так далее (иерархическая нумерация).
13. **IDEF0** — функциональная модель: система как набор функций, связанных стрелками ICOM. Описывает, что система делает, чем управляется и чем выполняется.
14. **ICOM**: Input (вход, слева), Control (управление, сверху), Output (выход, справа), Mechanism (механизм, снизу).
15. **Input** — то, что преобразуется функцией в выход (обрабатываемые данные/сырьё). **Control** — правила/ограничения, по которым идёт обработка (регламент, НПА, протокол); они не «расходуются».
16. **Mechanism** — ресурс-исполнитель: кто/что выполняет функцию (люди, система, оборудование). Рисуется снизу.
17. **A-0** — контекстная функция (вся система). **A0** — её декомпозиция на 3–6 подфункций (A1, A2…). Каждая декомпозируется дальше (A1 → A11, A12…), с сохранением граничных стрелок.
18. 3–6 функций — для читаемости. Меньше — нет смысла декомпозировать; больше — диаграмма нечитаема.
19. DFD — про потоки данных и хранилища, без времени/порядка. BPMN — про процесс во времени с участниками (пулы), последовательностью, событиями и шлюзами.
20. DFD фокусируется на данных (потоки + хранилища). IDEF0 — на функциях с явным разделением управления (Control) и ресурсов (Mechanism); хранилищ данных в IDEF0 нет явно.
21. DFD — когда нужно показать движение/хранение данных (мост к БД). IDEF0 — строгое функциональное описание с управлением и ресурсами (госсектор/ГОСТ). BPMN — последовательность шагов бизнес-процесса с участниками.
22. Хранилище → сущность/таблица; состав потока данных → атрибуты сущности; процесс, соединяющий хранилища, → связь. Один поток с вложенностью может дать несколько сущностей (Заказ 1:N Позиция).
23. Существительные-данные → потоки/хранилища; «система делает/создаёт/фиксирует» → процессы; «хранится/ведётся» → хранилища; внешние участники → внешние сущности. Сначала границы (контекст), затем декомпозиция.
24. **Справочное** хранилище в основном читается (каталог, меню, расписание). **Транзакционное** — активно пишется и читается (заказы, платежи, записи).
25. Они отвечают на «какие данные/функции», а не «в каком порядке». Это минус, когда важна именно последовательность и условия выполнения (тогда нужен BPMN/activity/sequence).
