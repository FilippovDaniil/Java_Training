# Модуль 14 — DFD и IDEF0 (потоки данных и функции)

## Зачем этот модуль

BPMN (модуль 12) и UML activity/state (модуль 13) показывают **поток управления** и **поведение**. Но процесс можно смотреть и через **потоки данных** (что куда течёт и где хранится) и через **функции** (что система делает и чем это управляется/обеспечивается). Для этого есть две классические нотации:
- **DFD (Data Flow Diagram)** — диаграмма потоков данных: процессы, потоки, хранилища, внешние сущности; уровни 0/1/2.
- **IDEF0** — функциональная модель: функции и стрелки ICOM (Input/Control/Output/Mechanism).

Обе старше BPMN, но активно встречаются в госпроектах, ГОСТ-документации, при описании «как есть» и при переходе от процессов к модели данных. СА должен уметь читать/строить DFD (особенно как мост к ER-модели в модуле 16) и понимать IDEF0. На собеседовании middle+ спрашивают: элементы DFD, правила корректности, что такое ICOM, чем DFD отличается от BPMN.

## 1. DFD — диаграмма потоков данных

Показывает, **как данные движутся** через систему: откуда приходят, какие процессы их обрабатывают, где хранятся, кому уходят. **Не показывает** порядок во времени и поток управления (это не блок-схема).

### Элементы (нотация Гейна–Сарсона / Йордана–ДеМарко)

| Элемент | Обозначение | Смысл |
|---------|-------------|-------|
| **Внешняя сущность** (external entity) | прямоугольник | источник/приёмник данных вне системы (клиент, банк) |
| **Процесс** (process) | круг/прямоугольник со скруглением | преобразование данных |
| **Хранилище данных** (data store) | две параллельные линии / открытый прямоугольник | где данные лежат (БД, файл, реестр) |
| **Поток данных** (data flow) | стрелка с подписью | что за данные движутся |

```
[Клиент] ──заказ──► (1. Принять заказ) ──данные заказа──► |D1 Заказы|
                          │
                    ──подтверждение──► [Клиент]
```

### Уровни декомпозиции

- **Контекстная диаграмма (уровень 0)** — вся система как **один** процесс + внешние сущности + потоки между ними. Показывает границы системы.
- **Уровень 1** — система разложена на основные процессы (1, 2, 3…), появляются хранилища.
- **Уровень 2+** — каждый процесс детализируется дальше при необходимости.

> Контекстная DFD ≈ контекстная диаграмма из модуля 06: один «чёрный ящик» системы и его окружение. Дальше «ящик» раскрывается.

### Правила корректности (нумерация процессов, баланс)

- **Нумерация:** процессы уровня 1 — 1, 2, 3; их детализация — 1.1, 1.2 и т.д.
- **Баланс (balancing):** потоки, входящие/выходящие на родительском уровне, должны совпадать с потоками детальной диаграммы.
- **Каждый процесс** имеет хотя бы один вход и один выход.

### Типичные ошибки DFD

| Ошибка | Образное название | Суть |
|--------|-------------------|------|
| Процесс только с входом, без выхода | **«чёрная дыра» (black hole)** | данные входят, ничего не выходит |
| Процесс только с выходом, без входа | **«чудо» (miracle)** | выход без входных данных |
| Поток данных без источника/приёмника | «висящий поток» | стрелка из ниоткуда/в никуда |
| Поток напрямую между двумя хранилищами | — | данные не двигаются сами, нужен процесс |
| Поток напрямую между двумя внешними сущностями | — | это вне системы, не моделируется |
| Поток управления вместо данных | — | DFD про данные, не про команды/control |

```
(Процесс) ◄── только вход, нет выхода   = чёрная дыра
(Процесс) ──► только выход, нет входа    = чудо
|D1| ──► |D2|   (хранилище → хранилище)  = нельзя, нужен процесс между ними
```

## 2. IDEF0 — функциональная модель

Описывает систему как набор **функций** и их связи через стрелки **ICOM**. Стандарт (восходит к SADT), популярен в госсекторе и для строгого описания деятельности.

### Функция и ICOM-стрелки

Функция — прямоугольник (глагол: «Обработать заказ»). К нему подходят 4 типа стрелок по сторонам:

| Буква | Сторона | Стрелка | Смысл |
|-------|---------|---------|-------|
| **I — Input** | слева | вход | что преобразуется (сырьё, данные на обработку) |
| **C — Control** | сверху | управление | правила, по которым выполняется (регламент, НПА, политика) |
| **O — Output** | справа | выход | результат функции |
| **M — Mechanism** | снизу | механизм | чем выполняется (люди, система, оборудование) |

```
              Control (регламент, НПА)
                      │
                      ▼
   Input ──────► ┌──────────────┐ ──────► Output
 (заявка)        │  Обработать  │       (решение)
                 │    заявку    │
                 └──────────────┘
                      ▲
                      │
              Mechanism (оператор, система)
```

> Ключевое отличие IDEF0 от DFD/BPMN: явное разделение **управления (Control)** и **входа (Input)**. Регламент/правило — это Control (сверху), а обрабатываемые данные — Input (слева). Их путают чаще всего.

### Декомпозиция IDEF0

- **A-0** — контекстная функция (вся система, одна функция).
- **A0** — её декомпозиция на 3–6 подфункций (A1, A2, A3…).
- Каждая подфункция декомпозируется дальше (A1 → A11, A12…).
- Стрелки родителя должны сохраняться у детей (как баланс в DFD).

> Правило «3–6 функций на диаграмме» — для читаемости. Меньше — мало смысла декомпозировать, больше — нечитаемо.

## 3. DFD vs IDEF0 vs BPMN — когда что

| Критерий | DFD | IDEF0 | BPMN |
|----------|-----|-------|------|
| **Фокус** | потоки данных | функции и их управление | поток управления процесса |
| **Узлы** | процессы (преобразуют данные) | функции (с ICOM) | задачи/события/шлюзы |
| **Время/порядок** | нет (не про последовательность) | нет (про функции, не порядок) | да (явная последовательность) |
| **Участники** | внешние сущности | механизмы (M) | пулы/дорожки |
| **Хранилища данных** | да (data store) | нет явно | нет (через данные/артефакты) |
| **Управление/правила** | нет явно | да (Control) | через шлюзы/события |
| **Где встречается** | анализ данных, мост к ER | госсектор, строгое описание | бизнес-процессы, повсеместно |

> Эвристика: нужно показать **движение данных и где они хранятся** (мост к БД) → DFD. Строгое описание **функций с управлением и ресурсами** (ГОСТ/госпроект) → IDEF0. **Последовательность шагов бизнес-процесса с участниками** → BPMN.

## 4. DFD как мост к модели данных

DFD особенно ценна тем, что **хранилища данных (data store)** и **потоки данных** напрямую подсказывают будущую модель БД:
- **хранилище** → таблица/набор сущностей (модуль 16, ER);
- **поток данных** → состав атрибутов, которые передаются (что входит в сущность/сообщение);
- **процесс, пишущий в хранилище** → операция создания/изменения сущности.

> Поэтому DFD часто строят перед ER: выявив хранилища и потоки, легче перейти к сущностям, атрибутам и связям. Поток «данные заказа» → атрибуты сущности «Заказ»; хранилище «Заказы» → таблица.

## ⚠️ Подводные камни

- **DFD ≠ блок-схема.** DFD не показывает порядок и условия (нет if/then, нет последовательности во времени). Это про данные, не про управление.
- **Чёрные дыры и чудеса.** У каждого процесса есть и вход, и выход. Только вход = чёрная дыра; только выход = чудо.
- **Хранилище↔хранилище и сущность↔сущность напрямую.** Между ними всегда процесс; между внешними сущностями потоки вообще не моделируются.
- **Путать Input и Control в IDEF0.** Обрабатываемые данные — Input (слева); правила/регламенты — Control (сверху). Это главная ошибка IDEF0.
- **Нарушение баланса при декомпозиции.** Входы/выходы родителя должны сохраняться на детальной диаграмме (и в DFD, и в IDEF0).
- **Слишком много процессов/функций на уровне.** Держите 3–6 (IDEF0) и разумное число (DFD) — иначе нечитаемо, декомпозируйте.
- **Выбор нотации «по привычке».** DFD — данные, IDEF0 — функции с управлением, BPMN — процесс во времени. Под задачу.
