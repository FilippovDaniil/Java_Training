# Задача 04 — Модуль 15: UML структурные

## ЗАДАНИЕ
Смоделируйте наследование (обобщение) в домене: выделите суперкласс и подклассы, распределите атрибуты/связи. Решите, где наследование оправдано, а где нет.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress / MedConnect**, кандидаты на иерархии:
1. Пользователь → Клиент, Курьер, Оператор (различаются атрибутами и связями).
2. Платёж → Оплата картой, Оплата бонусами, Оплата при получении (различается способом).
3. Заказ → Обычный заказ, Предзаказ (различаются только датой исполнения).
4. (MedConnect) Пользователь → Пациент, Врач (различаются ролями и данными).

## ТРЕБОВАНИЯ
- Для каждой группы постройте обобщение: суперкласс (общие атрибуты) + подклассы (специфичные).
- Решите, где наследование действительно нужно, а где достаточно атрибута-типа.
- Покажите нотацию (треугольник к суперклассу).
- Объясните принцип «не плодить иерархии ради иерархий».

## ФОРМАТ РЕЗУЛЬТАТА
Диаграммы обобщений (рисунок/текст) + таблица `Группа | Наследование нужно? | Обоснование`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Пользователь → Клиент/Курьер/Оператор: наследование оправдано (разные атрибуты/связи).
- [ ] Платёж: обоснованно (разные способы → разное поведение/атрибуты) — или альтернатива через атрибут-тип.
- [ ] Заказ → Обычный/Предзаказ: наследование НЕ нужно (различие только в значении атрибута «дата»).
- [ ] Общие атрибуты вынесены в суперкласс, специфичные — в подклассы.
- [ ] Нотация обобщения показана.
- [ ] Принцип «атрибут vs подкласс» объяснён.

## ПОДСКАЗКА
Наследование оправдано, когда подтипы реально различаются атрибутами/связями/поведением, важными для домена. Если разница только в значении одного поля («тип = предзаказ») — это атрибут, а не подкласс. Общее → вверх (суперкласс), особенное → вниз (подкласс).
