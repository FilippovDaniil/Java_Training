# Задача 06 — Модуль 15: UML структурные

## ЗАДАНИЕ
Найдите ошибки в готовой («битой») class diagram: неверные кратности, лишние/недостающие связи, классы-глаголы, атрибуты-списки, смешение уровней. Предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
Class diagram (текстовая нотация) FoodExpress:
- Класс `Заказ`: атрибуты `номер`, `сумма`, `client_id BIGINT`, `позиции: List<Позиция>`.
- Класс `ОформитьЗаказ`: атрибуты `шаг1`, `шаг2`.
- Связь `Клиент 0..* —— 0..* Заказ`.
- Связь `Заказ 1 —— 1 Курьер` (курьер обязателен всегда).
- Класс `Блюдо` связан с `Рестораном`, но кратности не указаны.
- Класс `Адрес`: атрибут `клиент: Клиент` (вместо связи).

## ТРЕБОВАНИЯ
- Найдите и классифицируйте каждую ошибку.
- Для каждой укажите тип (неверная кратность / класс-глагол / атрибут-список / FK в концептуальной модели / отсутствующая кратность / атрибут-ссылка вместо связи).
- Предложите исправление.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Фрагмент | Тип ошибки | Почему неверно | Исправление`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] `client_id BIGINT` — смешение уровней (FK/тип БД в концептуальной модели) → заменить связью.
- [ ] `позиции: List<Позиция>` — атрибут-список → связь Заказ 1—1..* Позиция.
- [ ] `ОформитьЗаказ` — класс-глагол (это процесс/операция, не сущность).
- [ ] `Клиент 0..* — 0..* Заказ` — неверная кратность (должно быть 1 — 0..*).
- [ ] `Заказ 1 — 1 Курьер` — неверно (курьер не всегда назначен → 0..1).
- [ ] Блюдо—Ресторан без кратностей → проставить.
- [ ] `Адрес.клиент: Клиент` — атрибут-ссылка → заменить ассоциацией.

## ПОДСКАЗКА
Чек-лист: классы — существительные; нет типов БД/FK (это ER, не концепт); списки объектов = связи, а не атрибуты; ссылки на другой класс = связь; кратности проставлены и осмыслены. M:N между Клиент и Заказ почти всегда ошибка — у заказа один владелец.
