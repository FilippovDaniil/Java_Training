# Задача 07 — Модуль 15: UML структурные

## ЗАДАНИЕ
Различите три уровня моделей на одной сущности: покажите, как «Заказ» выглядит как концептуальный класс, как ER-сущность и как класс кода. Объясните, что куда относится.

## ВХОДНЫЕ ДАННЫЕ
Сущность «Заказ» FoodExpress. Известно: у заказа есть номер, дата, сумма, статус, клиент, позиции; в БД заказ — таблица; в коде — класс с методами.

## ТРЕБОВАНИЯ
- Представьте «Заказ» на трёх уровнях:
  1. **Концептуальный** (class diagram аналитика) — сущность, атрибуты по смыслу, связи, без типов/ключей.
  2. **ER/логический** (модуль 16) — таблица, PK, FK (client_id), типы намечены, разрешение M:N если есть.
  3. **Класс кода** — поля с типами ЯП, методы (рассчитатьИтог, сменитьСтатус), паттерны.
- Для каждого уровня укажите, что в нём ЕСТЬ и чего НЕТ.
- Сформулируйте правило: какие детали на каком уровне появляются.

## ФОРМАТ РЕЗУЛЬТАТА
Три представления «Заказа» + таблица `Деталь | Концепт | ER | Код` (где появляется) + правило.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Концептуальный уровень: смысл, атрибуты, связи; нет типов БД/FK/методов.
- [ ] ER-уровень: PK/FK, типы намечены, связи как ключи, разрешение M:N.
- [ ] Код: типы ЯП, методы, поведение.
- [ ] Для каждого уровня указано, что есть/чего нет.
- [ ] Сформулировано правило появления деталей по уровням.
- [ ] Показано, что это ОДНА сущность на разных уровнях абстракции.

## ПОДСКАЗКА
Уровни нельзя смешивать. Концепт — смысл домена (для бизнеса). ER — структура хранения (PK/FK, нормализация). Код — реализация (методы, типы ЯП). FK появляется на ER, методы — в коде, а в концептуальной модели — только смысл и связи. Это частый вопрос на собеседовании middle+.
