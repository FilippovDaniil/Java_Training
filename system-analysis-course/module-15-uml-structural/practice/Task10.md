# Задача 10 (мини-проект) — Модуль 15: UML структурные

## ЗАДАНИЕ
Постройте полную концептуальную модель предметной области FoodExpress: class diagram со всеми сущностями, атрибутами, связями (вкл. агрегацию/композицию и обобщение), кратностями и инвариантами. Синтез теории модуля и мост к ER (модуль 16).

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** целиком: пользователи (клиенты, курьеры, операторы), рестораны с меню (блюда, категории), заказы (позиции, статусы, суммы), доставка (адреса, назначения курьеров), оплаты (способы), отзывы/рейтинги. Бизнес-правила — из задач 02, 03, 08.

## ТРЕБОВАНИЯ
Сдайте концептуальную модель:
1. **Сущности и атрибуты** — все ключевые классы домена (без типов БД/FK), атрибуты по смыслу.
2. **Связи и кратности** — ассоциации с multiplicity на обоих концах; роли где нужно.
3. **Агрегация/композиция** — там, где есть «часть-целое» (Заказ◆Позиция, Команда◇Курьер и т.п.) с обоснованием.
4. **Обобщение** — иерархия Пользователь → Клиент/Курьер/Оператор (если оправдано).
5. **Инварианты** — кратностями + текстом (≥1 позиция, уникальность email, сумма = сумме позиций, лимиты).
6. **Корректность** — классы-существительные, нет атрибутов-списков и FK, верные стороны кратностей, M:N разрешены через сущность-связку где нужно.

## ФОРМАТ РЕЗУЛЬТАТА
Concept class diagram (рисунок/PlantUML/текстовая нотация) + список инвариантов + краткое сопроводительное описание (какие решения по агрегации/наследованию приняты и почему).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все ключевые сущности домена с осмысленными атрибутами.
- [ ] Связи с кратностями на обоих концах, корректные стороны.
- [ ] Агрегация/композиция применены и обоснованы (жизненный цикл части).
- [ ] Обобщение Пользователь→роли (или обоснован отказ от него).
- [ ] Инварианты: кратностями + текстом.
- [ ] Модель концептуальная (нет типов БД, FK, методов).
- [ ] M:N осмыслены (например, через «Позицию заказа»).
- [ ] Сопроводительное описание объясняет ключевые решения.

## ПОДСКАЗКА
Синтез: сущности/атрибуты из задач 01/09, связи/кратности из 02, агрегация/композиция из 03, наследование из 04, инварианты из 08, уровни из 07. Главные проверки middle+: чистый концептуальный уровень (не ER и не код), корректные кратности и «часть-целое», осмысленные M:N. Эта модель — прямой вход в ER-модель (модуль 16): сущности → таблицы, связи → ключи.
