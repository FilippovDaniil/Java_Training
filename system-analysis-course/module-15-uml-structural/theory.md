# Модуль 15 — UML структурные диаграммы (Class / Object, концептуальная модель)

## Зачем этот модуль

Поведение (модуль 13) и потоки данных (модуль 14) описывают, что система делает. Теперь — **структура**: из каких сущностей состоит предметная область, какие у них свойства и как они связаны. Главный инструмент — **UML class diagram** (диаграмма классов). Для аналитика это прежде всего **концептуальная модель предметной области** — словарь сущностей домена, а не модель БД и не классы кода.

СА строит концептуальную модель, чтобы зафиксировать понятия и связи домена (единый язык с заказчиком и командой), и она напрямую питает ER-модель (модуль 16) и DDD (модуль 24). На собеседовании middle+ обязательно: элементы class diagram, кратности, разница агрегации и композиции, чем концептуальная модель отличается от ER и от классов кода.

---

## 1. Class diagram — основные элементы

**Класс** — прямоугольник из трёх секций: имя, атрибуты, операции. На концептуальном уровне операции часто опускают (важны сущности и связи).

```
┌─────────────────┐
│      Заказ       │   ← имя (существительное)
├─────────────────┤
│ номер           │   ← атрибуты
│ дата            │
│ сумма           │
├─────────────────┤
│ рассчитатьИтог()│   ← операции (на концептуальном уровне обычно опускаются)
└─────────────────┘
```

| Элемент | Смысл |
|---------|-------|
| **Класс** | понятие/сущность домена (Клиент, Заказ, Ресторан) |
| **Атрибут** | свойство сущности (имя, сумма, статус) |
| **Операция** | поведение (чаще на уровне проектирования, не концептуальном) |
| **Ассоциация** | связь между классами (линия) |
| **Кратность** (multiplicity) | сколько объектов участвует в связи |
| **Роль** | имя конца связи (что значит эта сторона) |

---

## 2. Связи и кратности

### Виды связей

| Связь | Обозначение | Смысл | Пример |
|-------|-------------|-------|--------|
| **Ассоциация** | простая линия | сущности связаны | Клиент — Заказ |
| **Агрегация** | линия с пустым ромбом | «часть-целое», части живут отдельно | Команда ◇— Сотрудник |
| **Композиция** | линия с закрашенным ромбом | «часть-целое», части не живут без целого | Заказ ◆— Позиция заказа |
| **Обобщение/наследование** | линия с пустым треугольником | «является разновидностью» | Физлицо —▷ Клиент |
| **Зависимость** | пунктир со стрелкой | использует/зависит | — |

### Кратности (multiplicity)

| Запись | Значение |
|--------|----------|
| `1` | ровно один |
| `0..1` | ноль или один (необязательно) |
| `*` или `0..*` | ноль и более |
| `1..*` | один и более |
| `2..5` | от 2 до 5 |

```
Клиент 1 ──────── 0..* Заказ
        (у клиента 0+ заказов, у заказа ровно 1 клиент)

Заказ 1 ◆──────── 1..* ПозицияЗаказа
        (заказ состоит из 1+ позиций; позиция не существует без заказа — композиция)
```

> Кратность читается «с другого конца»: возле `Заказ` стоит число, означающее «сколько заказов у одного клиента». Самая частая ошибка — перепутать стороны.

---

## 3. Агрегация vs композиция

Обе — «часть-целое», разница в **жизненном цикле части**:

| Критерий | Агрегация (пустой ромб) | Композиция (закрашенный ромб) |
|----------|--------------------------|-------------------------------|
| Часть существует без целого? | **Да** | **Нет** |
| Часть может принадлежать нескольким целым? | Да (обычно) | Нет (одно целое) |
| Удаление целого | часть остаётся | часть удаляется вместе |
| Пример | Команда и Сотрудник (сотрудник останется при роспуске команды) | Заказ и Позиция заказа (позиция без заказа бессмысленна) |

> Тест: «если удалить целое, часть имеет смысл/живёт сама?» Да → агрегация. Нет → композиция. На концептуальном уровне различие часто упрощают до ассоциации, но middle+ должен уметь его обосновать.

---

## 4. Обобщение (наследование) в домене

**Обобщение** — «является разновидностью» (is-a). Подкласс наследует атрибуты/связи суперкласса и добавляет свои.

```
            Пользователь
           (id, имя, email)
            ▲          ▲
            │          │
        Клиент      Курьер
      (адреса)    (транспорт, рейтинг)
```

> Применять, когда подтипы реально различаются поведением/атрибутами и это важно для домена. Не плодить иерархии ради иерархий: если разница только в значении атрибута («тип = …»), часто достаточно атрибута, а не подкласса.

---

## 5. Object diagram (диаграмма объектов)

**Object diagram** — «снимок» (snapshot) системы в конкретный момент: экземпляры классов с конкретными значениями. Показывает пример состояния, проверяет корректность модели классов.

```
заказ_1024 : Заказ            клиент_77 : Клиент
  номер = 1024        ────────  имя = "Иванов"
  сумма = 1500                   email = "i@mail.ru"
```

> Class diagram — про типы (Заказ вообще), object diagram — про экземпляры (конкретный заказ №1024). Object diagram полезна, чтобы на примере показать, как «оживает» модель классов, и поймать ошибки кратностей.

---

## 6. Концептуальная модель vs ER vs классы кода — уровни моделей

Одни и те же сущности живут на разных уровнях абстракции. Их **нельзя смешивать**:

| Уровень | Что моделирует | Чем оперирует | Кто автор |
|---------|----------------|---------------|-----------|
| **Концептуальная модель** (class diagram аналитика) | понятия предметной области, единый язык | сущности, атрибуты, связи (без типов БД, без кода) | аналитик |
| **Логическая/ER-модель** (модуль 16) | структура данных | таблицы, ключи (PK/FK), нормализация | аналитик/архитектор данных |
| **Физическая модель БД** | реализация в СУБД | типы столбцов, индексы, движок | DBA/разработчик |
| **Классы кода** | реализация в ЯП | классы, методы, паттерны, наследование реализации | разработчик |

> Концептуальная модель — про **смысл домена**, а не про хранение или код. В ней нет «id типа BIGINT», нет внешних ключей как столбцов, нет геттеров. Эти детали появляются ниже (ER → физика → код). Частая ошибка middle− : рисовать «классы» сразу с типами столбцов и FK — это уже не концептуальная модель.

---

## 7. Инварианты и бизнес-правила

К концептуальной модели прикладывают **инварианты** — правила, которые всегда истинны:
- «Сумма заказа = сумма позиций» (производное правило);
- «У заказа хотя бы одна позиция» (отражается кратностью `1..*`);
- «Email клиента уникален»;
- «Курьер не может вести более N активных доставок».

Часть инвариантов выражается прямо нотацией (кратности, обязательность атрибута), часть — текстом или в OCL (Object Constraint Language). Для аналитика обычно достаточно текстовых ограничений рядом с моделью.

> Кратность `1..*` — это уже бизнес-правило («минимум одна позиция»). Остальные правила (уникальность, лимиты, производные значения) фиксируйте текстом — они станут валидацией и ограничениями БД.

---

## ⚠️ Подводные камни

- **Смешивать уровни.** Концептуальная модель ≠ ER ≠ классы кода. Не тащить типы столбцов, FK и геттеры в концептуальную модель.
- **Путать стороны кратности.** Кратность ставится на «дальнем» конце связи; перепутанные стороны меняют смысл (1:N ↔ N:1).
- **Агрегация vs композиция наугад.** Решает жизненный цикл части: живёт без целого → агрегация, нет → композиция.
- **Лишнее наследование.** Если различие только в значении атрибута — это атрибут, а не подкласс.
- **Класс-«глагол».** Класс — существительное (сущность), не действие. «ОформитьЗаказ» — это операция/процесс, а не класс.
- **M:N без осмысления.** Связь «многие-ко-многим» часто скрывает отдельную сущность (например, «Позиция заказа» между Заказом и Блюдом) — это всплывёт в ER (модуль 16).
- **Атрибуты-списки.** Атрибут, который «на самом деле список объектов», — это связь с другой сущностью, а не атрибут.
