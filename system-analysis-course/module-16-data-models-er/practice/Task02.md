# Задача 02 — Модуль 16: Модели данных и ER

## ЗАДАНИЕ
Определите первичные (PK) и внешние (FK) ключи для сущностей FoodExpress: выберите тип PK и правильно расставьте FK по связям.

## ВХОДНЫЕ ДАННЫЕ
Сущности из задачи 01 и их связи (1:N, M:N). Известно: email клиента уникален; у блюда есть артикул ресторана; заказы нумеруются сквозным номером.

## ТРЕБОВАНИЯ
- Для каждой сущности выберите PK (суррогатный/натуральный/составной) с обоснованием.
- Расставьте FK по связям 1:N (на правильной стороне — «многие»).
- Для M:N покажите, что FK уходят в промежуточную сущность (детально — задача 03).
- Объясните, почему email/номер — не лучший PK.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сущность | PK (тип) | Обоснование | FK (куда ссылается)`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] У каждой сущности определён PK с типом (суррогатный/натуральный/составной).
- [ ] Преобладают суррогатные ключи с обоснованием стабильности.
- [ ] FK в связях 1:N стоят на стороне «многие» (например, client_id в Заказе).
- [ ] Email/номер рассмотрены как unique-атрибуты, а не PK (обоснование).
- [ ] Для M:N показано, что FK в промежуточной сущности.
- [ ] Нет FK не на той стороне.

## ПОДСКАЗКА
Суррогатный PK (автоинкремент/UUID) обычно лучше: стабилен и нейтрален. Натуральный (email, номер) меняется/реиспользуется → плохой PK, но хороший unique-constraint. В 1:N внешний ключ всегда на стороне «многие»: в Заказ кладём client_id, не наоборот.
