# Задача 03 — Модуль 16: Модели данных и ER

## ЗАДАНИЕ
Реализуйте связи 1:1, 1:N, M:N и разрешите все «многие-ко-многим» через промежуточные сущности с собственными атрибутами.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, связи:
1. Заказ — Оплата (1:1, у заказа одна оплата).
2. Клиент — Заказ (1:N).
3. Заказ — Блюдо (M:N: заказ включает много блюд, блюдо встречается во многих заказах).
4. Курьер — Зона доставки (M:N: курьер обслуживает несколько зон, зона — несколькими курьерами).
5. Блюдо — Ингредиент (M:N, для аллергенов).

## ТРЕБОВАНИЯ
- Для 1:1 и 1:N покажите расстановку FK.
- Для каждой M:N введите промежуточную сущность с именем и собственными атрибутами.
- Определите PK промежуточных сущностей (составной из FK или суррогатный).
- Отметьте, какие промежуточные сущности — «настоящие» сущности домена, а какие чисто связочные.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица связей `Связь | Тип | Реализация (FK / промежуточная сущность + атрибуты + PK)`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] 1:1 (Заказ—Оплата) реализована FK + unique.
- [ ] 1:N (Клиент—Заказ) — FK на стороне «многие».
- [ ] Заказ—Блюдо разрешена через «Позицию заказа» с атрибутами (количество, цена).
- [ ] Курьер—Зона и Блюдо—Ингредиент разрешены промежуточными сущностями.
- [ ] PK промежуточных сущностей определены (составной/суррогатный).
- [ ] Различены «настоящие» сущности (Позиция) и чисто связочные.

## ПОДСКАЗКА
M:N в реляционной модели не существует напрямую — всегда таблица-связка с двумя FK. Если у связки есть свои атрибуты (количество, цена на момент заказа) — это полноценная сущность («Позиция заказа»), а не техническая. PK связки: либо составной (oba FK), либо суррогатный id.
