# Задача 04 — Модуль 16: Модели данных и ER

## ЗАДАНИЕ
Определите слабые сущности и идентифицирующие связи: найдите сущности, которые не идентифицируются без родителя, и оформите их ключи.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress / общие** кандидаты:
1. Строка чека (существует только в рамках конкретного чека).
2. Позиция заказа (в рамках заказа).
3. Платёж (имеет собственный идентификатор от провайдера).
4. Адрес доставки (привязан к клиенту, но может иметь свой id).
5. История статусов заказа (запись истории в рамках заказа).

## ТРЕБОВАНИЯ
- Определите, какие сущности слабые, какие сильные.
- Для слабых покажите идентифицирующую связь и составной ключ (родитель + локальный различитель).
- Объясните разницу «слабая сущность» vs «сильная с FK».
- Отметьте, где на практике всё равно дают суррогатный id и почему это допустимо.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сущность | Слабая/сильная | Идентификация | Ключ` + краткое пояснение.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Строка чека, История статусов, Позиция — распознаны как слабые (идентификация через родителя).
- [ ] Платёж — сильная (имеет собственный идентификатор провайдера).
- [ ] Для слабых показан составной ключ (родитель + локальный номер).
- [ ] Объяснена разница «слабая» vs «сильная с FK».
- [ ] Отмечено, где суррогатный id применяют на практике и почему.

## ПОДСКАЗКА
Слабая сущность не идентифицируется сама — только в контексте родителя (строка чека = чек + № строки). Идентифицирующая связь даёт часть ключа. На практике часто добавляют суррогатный id для удобства, но концептуально зависимость от родителя сохраняется (каскадное удаление).
