# Задача 06 — Модуль 16: Модели данных и ER

## ЗАДАНИЕ
Найдите ошибки в готовой («битой») ER-модели: отсутствие ключей, неверные кратности, неразрешённый M:N, FK не на той стороне, многозначные атрибуты. Предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
Логическая ER (текстовая нотация) FoodExpress:
- `Клиент(имя, email, телефоны)` — нет PK; `телефоны` — несколько значений в одном поле.
- `Заказ(id PK, дата, сумма)` — нет связи с клиентом вообще.
- `Клиент(id) 1 ──── 1 Заказ(id)` — кратность 1:1 (у клиента только один заказ?).
- `Заказ M ──── N Блюдо` — связь M:N не разрешена.
- `Блюдо(id PK, название, order_id FK→Заказ)` — FK заказа стоит в Блюде.
- `Ресторан(id PK, ..., меню: список блюд)` — меню как атрибут-список.

## ТРЕБОВАНИЯ
- Найдите и классифицируйте каждую ошибку.
- Укажите тип (нет PK / многозначный атрибут / отсутствующая связь / неверная кратность / неразрешённый M:N / FK не на той стороне / атрибут-список).
- Предложите исправление.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Фрагмент | Тип ошибки | Почему неверно | Исправление`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Клиент без PK — найдено, добавлен PK.
- [ ] `телефоны` — многозначный атрибут → отдельная таблица/сущность.
- [ ] Клиент—Заказ 1:1 — неверная кратность → 1:N.
- [ ] Заказ—Блюдо M:N — не разрешён → промежуточная таблица «Позиция».
- [ ] FK заказа в Блюде — неверная сторона → FK блюда в Позиции, не наоборот.
- [ ] Меню-список → связь Ресторан 1—N Блюдо.
- [ ] Для каждой ошибки есть исправление.

## ПОДСКАЗКА
Чек-лист ER: у каждой таблицы есть PK; нет многозначных атрибутов (1NF, модуль 17); M:N всегда через промежуточную таблицу; FK в 1:N — на стороне «многие»; нет атрибутов-списков; кратности осмыслены. Блюдо не «принадлежит заказу» — оно встречается в позициях многих заказов.
