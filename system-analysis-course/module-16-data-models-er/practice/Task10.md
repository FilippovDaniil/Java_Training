# Задача 10 (мини-проект) — Модуль 16: Модели данных и ER

## ЗАДАНИЕ
Постройте полную ER-модель FoodExpress на двух уровнях: концептуальную и логическую. Все связи разрешены, ключи определены, слабые сущности оформлены, нотация Crow's foot. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** целиком: пользователи (клиенты, курьеры), рестораны, меню (блюда, категории), заказы (позиции, статусы, история статусов), доставка (адреса, зоны, назначения), оплаты, отзывы. Бизнес-правила и связи — из задач 01–09.

## ТРЕБОВАНИЯ
Сдайте пакет:
1. **Концептуальная ER** — сущности, атрибуты, связи с кратностями (без ключей и типов).
2. **Логическая ER** — таблицы, PK (с выбором типа), FK на верных сторонах, M:N разрешены промежуточными таблицами, обязательность/уникальность.
3. **Слабые сущности** — оформлены (История статусов, Позиция) с идентифицирующими связями.
4. **Crow's foot** — модель (или ключевой фрагмент) в этой нотации.
5. **Корректность** — у всех таблиц PK; нет многозначных атрибутов и атрибутов-списков; нет неразрешённого M:N; FK на верных сторонах; кратности осмыслены.
6. **Мост к нормализации** — отметьте места, требующие проверки на нормальные формы (модуль 17).

## ФОРМАТ РЕЗУЛЬТАТА
Две модели (концептуальная + логическая) + фрагмент Crow's foot + список решений (ключи, разрешения M:N, слабые сущности) + сопроводительное описание.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Концептуальная ER: сущности/атрибуты/связи без ключей и типов.
- [ ] Логическая ER: PK с типами, FK на верных сторонах.
- [ ] Все M:N разрешены промежуточными сущностями (с собственными атрибутами где есть).
- [ ] Слабые сущности оформлены с идентифицирующими связями.
- [ ] Crow's foot применён (хотя бы для ключевого фрагмента).
- [ ] Корректность: PK у всех, нет многозначных/списков, кратности верны.
- [ ] Отмечены места под нормализацию (модуль 17).
- [ ] Сопроводительное описание объясняет ключевые решения.

## ПОДСКАЗКА
Синтез: концепт из задачи 01, ключи из 02, разрешение M:N из 03, слабые сущности из 04, переход концепт→логика из 05, Crow's foot из 07. Главные проверки middle+: чистое разделение уровней, корректное разрешение M:N с распознаванием сущностей-связок, верные PK/FK, читаемая нотация. Эта модель идёт в нормализацию (модуль 17), затем в SQL-запросы (модуль 18) и индексы/перформанс (модуль 20).
