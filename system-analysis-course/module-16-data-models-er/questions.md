---

## Ключ (эталонные ответы)

1. **Концептуальная** (сущности/связи домена, без ключей и типов), **логическая** (структура: атрибуты, PK/FK, нормализация, СУБД-независимо), **физическая** (реализация в конкретной СУБД: типы, индексы).
2. **ER (Entity-Relationship)** — модель данных, описывающая предметную область через сущности, их атрибуты, ключи и связи с кратностями.
3. Сущности, атрибуты, ключи (PK/FK), связи, кратности.
4. **Сущность** — объект учёта, о котором хранят данные (Клиент, Заказ). **Атрибут** — свойство сущности (имя, сумма).
5. Простой/составной, однозначный/многозначный, хранимый/производный, ключевой.
6. **PK** — атрибут(ы), однозначно идентифицирующий запись; уникален и не NULL.
7. **Натуральный** — из данных домена (ИНН, email): может меняться и быть неуникальным. **Суррогатный** — искусственный (автоинкремент/UUID): стабилен и нейтрален. Обычно лучше суррогатный.
8. Ключ из нескольких атрибутов, идентифицирующий запись в совокупности (например, два FK в таблице-связке).
9. **FK** — атрибут, ссылающийся на PK другой сущности; реализует связь между таблицами.
10. На стороне «многие»: в таблицу-«многие» добавляется FK на «один» (в Заказ — client_id).
11. FK на любой из сторон (часто с ограничением UNIQUE), чтобы гарантировать единственность связи.
12. В реляционной модели нет прямого M:N: одна строка не может ссылаться на множество строк через одно поле. Нужна отдельная таблица.
13. Через **промежуточную (ассоциативную) сущность** с двумя FK на связываемые сущности; её PK — составной из FK или суррогатный.
14. Когда у неё есть собственные атрибуты (количество, цена на момент заказа, дата) — это сущность домена («Позиция заказа»), а не чисто техническая связка.
15. **Слабая сущность** не имеет достаточного собственного ключа и идентифицируется только через связь с родительской (сильной) сущностью.
16. Связь, через которую слабая сущность получает часть своего ключа (родитель + локальный различитель = ключ слабой сущности).
17. Слабая сущность зависит от родителя в самой идентификации (ключ включает родителя). Сильная с FK имеет собственный независимый ключ, а FK — лишь ссылка.
18. Chen (ромбы-связи), Crow's foot (вороньи лапки — отраслевой стандарт), IDEF1X (госсектор), UML class diagram как ER.
19. Вилка («лапка») = «много»; кружок = «ноль допустим»; палочка = «обязательно один». `─||─` ровно один, `─o|─` ноль/один, `─|<` один+, `─o<` ноль+.
20. Сущности → таблицы; атрибуты → столбцы + выбрать PK; 1:N → FK на «многие»; M:N → промежуточная таблица; слабые сущности → FK на родителя; многозначные атрибуты → отдельные таблицы; решить судьбу производных атрибутов.
21. Вынести в отдельную сущность/таблицу (несколько телефонов → таблица «Телефон» со связью 1:N). Хранение нескольких значений в одном поле нарушает 1NF.
22. По контексту: **хранить** — быстрее читать, но риск рассинхрона и нужно поддерживать; **вычислять** — всегда корректно, но дороже на чтении. Выбор зависит от частоты чтения/записи и требований к согласованности.
23. Float — двоичная плавающая точка, теряет точность на десятичных дробях (копейки «плывут»). Для денег — numeric/decimal с фиксированной точностью.
24. Концептуальный — про смысл домена (сущности/связи, без ключей и типов). Физический — про реализацию в СУБД (типы столбцов, индексы, ограничения, движок).
25. Нет PK; неразрешённый M:N; потерянная сущность-связка; FK не на той стороне; натуральный изменчивый ключ как PK; многозначные атрибуты/атрибуты-списки; перепутанные стороны кратностей; смешение уровней.
