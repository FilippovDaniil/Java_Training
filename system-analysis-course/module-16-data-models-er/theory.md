# Модуль 16 — Модели данных и ER (Entity-Relationship)

## Зачем этот модуль

Часть 3 завершилась концептуальной моделью (модуль 15) — словарём сущностей домена. Часть 4 «Данные» начинается с перевода этого словаря в **модель данных**: как сущности, их свойства и связи будут структурированы для хранения. Главный инструмент — **ER-модель** (Entity-Relationship): сущности, атрибуты, ключи, связи и их кратности.

ER — базовый навык аналитика: на её основе строят схему БД, проектируют API, проверяют полноту требований к данным. Дальше идут нормализация (модуль 17) и SQL (модуль 18). На собеседовании middle+ обязательно: уровни моделей данных, элементы ER, первичный/внешний ключ, разрешение связи «многие-ко-многим», слабые сущности, нотация Crow's foot.

---

## 1. Три уровня модели данных

| Уровень | Что описывает | Аудитория | Детали |
|---------|---------------|-----------|--------|
| **Концептуальная** | сущности и связи домена | бизнес + аналитик | без ключей и типов; «что за данные есть» |
| **Логическая** | структура: атрибуты, ключи (PK/FK), нормализация | аналитик/архитектор данных | ключи, нормальные формы; СУБД-независимо |
| **Физическая** | реализация в конкретной СУБД | DBA/разработчик | типы столбцов, индексы, партиции, движок |

```
Концептуальная → Логическая → Физическая
 (смысл)          (структура)   (реализация в PostgreSQL/…)
```

> Концептуальный уровень ER ≈ концептуальная class diagram (модуль 15). Логический добавляет ключи и нормализацию. Физический — типы и индексы под конкретную СУБД. Аналитик чаще всего работает на концептуальном и логическом уровнях.

---

## 2. Элементы ER

| Элемент | Смысл | Пример |
|---------|-------|--------|
| **Сущность** (entity) | объект учёта, о котором хранят данные | Клиент, Заказ, Блюдо |
| **Атрибут** (attribute) | свойство сущности | имя, сумма, дата |
| **Ключ** (key) | атрибут(ы), идентифицирующие экземпляр | id заказа |
| **Связь** (relationship) | как сущности соотносятся | Клиент *размещает* Заказ |
| **Кратность** (cardinality) | сколько экземпляров участвует | 1:N, M:N |

### Виды атрибутов
- **Простой / составной** (адрес = город + улица + дом);
- **Однозначный / многозначный** (телефоны — несколько);
- **Хранимый / производный** (возраст из даты рождения; сумма заказа из позиций);
- **Ключевой** — входит в идентификатор.

> Многозначный и составной атрибуты — сигнал, что, возможно, нужна отдельная сущность (несколько телефонов → сущность «Телефон»). Производный атрибут — кандидат не хранить, а вычислять.

---

## 3. Ключи: PK и FK

- **Первичный ключ (PK, Primary Key)** — атрибут(ы), однозначно идентифицирующий запись. Уникален, не NULL.
  - **Натуральный** ключ — из данных домена (ИНН, номер паспорта). Риск: меняется, бывает не уникален.
  - **Суррогатный** ключ — искусственный (автоинкремент, UUID). Обычно предпочтителен: стабилен, нейтрален.
  - **Составной** ключ — из нескольких атрибутов.
- **Внешний ключ (FK, Foreign Key)** — атрибут, ссылающийся на PK другой сущности. Реализует связь.

```
Клиент(  id (PK), имя, email )
Заказ(   id (PK), дата, сумма, client_id (FK → Клиент.id) )
```

> На логическом уровне связь «1:N» реализуется так: FK ставится на стороне «многие» (в Заказ кладём client_id). Частая ошибка — поставить FK не на ту сторону.

---

## 4. Связи и кратности

| Кратность | Смысл | Реализация |
|-----------|-------|------------|
| **1:1** | одному экземпляру A — один B | FK на любой из сторон (часто с unique) |
| **1:N** | одному A — много B | FK на стороне «многие» (B) |
| **M:N** | многим A — много B | **через промежуточную сущность** (таблицу-связку) |

### Разрешение M:N

«Многие-ко-многим» **нельзя** реализовать напрямую — вводится **ассоциативная (промежуточная) сущность**:

```
Заказ M ──── N Блюдо     ──разрешается──►   Заказ 1 ── N ПозицияЗаказа N ── 1 Блюдо

ПозицияЗаказа( order_id (FK), dish_id (FK), количество, цена )
              PK = (order_id, dish_id)  ← составной, или суррогатный id
```

> Часто промежуточная сущность имеет **собственные атрибуты** (количество, цена на момент заказа) — это и есть полноценная сущность домена, а не «техническая таблица». Умение распознать её — признак middle+.

---

## 5. Слабые сущности и идентифицирующие связи

- **Слабая сущность** (weak entity) — не имеет собственного достаточного ключа, идентифицируется только через связь с «сильной» (родительской).
- **Идентифицирующая связь** — связь, через которую слабая сущность получает часть своего ключа.

```
Заказ(id PK)  ◄─идентифицирующая─  ПозицияЗаказа( № позиции, ... )
   PK позиции = (order_id, № позиции)   ← зависит от заказа
```

> Пример: «Строка чека» сама по себе не идентифицируется — только в рамках конкретного чека (чек + номер строки). На практике часто всё равно дают суррогатный id, но понимать концепт слабой сущности нужно.

---

## 6. Нотации ER

| Нотация | Кратность изображается | Где встречается |
|---------|------------------------|-----------------|
| **Chen** | ромбы-связи, надписи 1/N/M | академическая, классика |
| **Crow's foot** (вороньи лапки) | символы на концах линий | индустрия, большинство инструментов |
| **IDEF1X** | — | госсектор, стандарты |
| **UML** (class diagram как ER) | кратности `1`, `0..*` | команды на UML |

### Crow's foot — чтение символов

```
─||─   ровно один (обязательно один)
─o|─   ноль или один
─|<    один или много (обязательно ≥1)
─o<    ноль или много
```

Пример: `Клиент ─||────o<─ Заказ` = «у клиента 0+ заказов; у заказа ровно один клиент».

> Crow's foot — отраслевой стандарт; «лапка» (вилка) = «много», кружок = «ноль допустим», палочка = «обязательно один». Уметь читать обязательно.

## 7. От концептуальной модели к логической ER

Алгоритм перехода (модуль 15 → 16):
1. Каждая **сущность** концептуальной модели → таблица.
2. **Атрибуты** → столбцы; выбрать **PK** (обычно суррогатный).
3. Связь **1:N** → FK на стороне «многие».
4. Связь **M:N** → промежуточная таблица с двумя FK.
5. **Композиция/слабая сущность** → таблица с FK на родителя (часто с каскадным удалением на физическом уровне).
6. **Многозначные/составные атрибуты** → отдельные таблицы при необходимости.
7. **Производные атрибуты** → решить: хранить или вычислять.

> Это механический, но ответственный шаг: ошибки здесь (не та сторона FK, неразрешённый M:N, потерянная сущность-связка) тиражируются в схему БД и API.

---

## ⚠️ Подводные камни

- **Смешивать уровни.** Концептуальный ER — без ключей и типов; типы и индексы — это физический уровень.
- **Не разрешённый M:N.** «Многие-ко-многим» всегда через промежуточную сущность; прямого M:N в реляционной БД нет.
- **Потерять сущность-связку.** Промежуточная таблица часто имеет свои атрибуты (количество, цена, дата) — это полноценная сущность.
- **FK не на той стороне.** В 1:N внешний ключ ставится на стороне «многие».
- **Натуральный ключ, который меняется.** Email/телефон/ИНН как PK — риск; предпочтителен суррогатный ключ.
- **Многозначный атрибут как одно поле.** «Телефоны» одной строкой через запятую — нарушение; нужна отдельная сущность/таблица (см. 1NF, модуль 17).
- **Путать стороны кратности** (как и в class diagram).
