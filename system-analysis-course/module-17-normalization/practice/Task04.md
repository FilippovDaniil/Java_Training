# Задача 04 — Модуль 17: Нормализация

## ЗАДАНИЕ
Распознайте нарушение BCNF и декомпозируйте: случай, когда таблица в 3NF, но детерминант ФЗ не является ключом-кандидатом.

## ВХОДНЫЕ ДАННЫЕ
Таблица «НазначениеВрача» (MedConnect): запись о том, какой врач в какой день в каком кабинете ведёт приём по специализации.
- Атрибуты: `(врач, дата) → кабинет`; `кабинет → специализация` (кабинет оборудован под одну специализацию); `(врач, дата) → специализация`.
- Ключи-кандидаты: `(врач, дата)`.
- Проблема: есть ФЗ `кабинет → специализация`, где `кабинет` — не ключ-кандидат.

## ТРЕБОВАНИЯ
- Покажите, что таблица в 3NF (нет частичных/транзитивных нарушений относительно ключа).
- Покажите нарушение BCNF (детерминант `кабинет` — не суперключ).
- Декомпозируйте так, чтобы детерминант каждой ФЗ стал ключом.
- Объясните, чем BCNF строже 3NF.

## ФОРМАТ РЕЗУЛЬТАТА
Анализ ФЗ + доказательство 3NF + нарушение BCNF + декомпозиция (таблицы с ключами) + пояснение разницы 3NF/BCNF.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выписаны ФЗ и ключи-кандидаты.
- [ ] Показано, что таблица удовлетворяет 3NF.
- [ ] Найдено нарушение BCNF (`кабинет → специализация`, детерминант не суперключ).
- [ ] Декомпозиция: `Кабинет(кабинет → специализация)` + `Назначение(врач, дата → кабинет)`.
- [ ] Объяснено, что BCNF требует: детерминант ЛЮБОЙ ФЗ — суперключ.
- [ ] Отмечено, что 3NF этот случай пропускает.

## ПОДСКАЗКА
BCNF строже 3NF: для каждой ФЗ `X → Y` X обязан быть суперключом. 3NF допускает ФЗ от неключевого детерминанта, если зависимый атрибут — часть ключа-кандидата (как `специализация`). BCNF это запрещает. Декомпозируем по детерминанту: `кабинет` становится ключом своей таблицы.
