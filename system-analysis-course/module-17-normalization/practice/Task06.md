# Задача 06 — Модуль 17: Нормализация

## ЗАДАНИЕ
Обоснуйте денормализацию под отчётность: выберите, где и как добавить контролируемую избыточность ради производительности, и опишите, как поддерживать согласованность.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, нормализованная схема (3NF). Проблемы производительности:
1. Дашборд «выручка по ресторанам за день» — тяжёлый JOIN заказов, позиций, блюд + агрегация, выполняется часто.
2. Карточка заказа показывает сумму — каждый раз пересчёт из позиций.
3. Список клиентов с числом заказов — постоянный COUNT по заказам.

## ТРЕБОВАНИЯ
- Для каждой проблемы предложите денормализацию (предрасчёт/дублирование/витрина).
- Опишите, как и когда обновлять избыточные данные (триггер/пересчёт/событие).
- Укажите риски (рассинхрон) и как их контролировать.
- Отметьте, где денормализация НЕ нужна (преждевременная оптимизация).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Проблема | Денормализация | Как поддерживать | Риски` + вывод, где оставить нормализованным.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Для дашборда выручки предложена витрина/агрегатная таблица (обновление по расписанию/событию).
- [ ] Для суммы заказа — хранимое поле `total` с пересчётом при изменении позиций.
- [ ] Для счётчика заказов — денормализованный счётчик или материализованное представление.
- [ ] Для каждой денормализации описан механизм поддержания актуальности.
- [ ] Названы риски рассинхрона и контроль.
- [ ] Отмечено, что денормализуют под измеренную проблему, а не везде.

## ПОДСКАЗКА
Денормализация — компромисс: быстрее чтение ценой сложности записи и риска рассинхрона. Сначала нормализуй, денормализуй точечно под доказанную проблему. Витрины/материализованные представления — для отчётов; хранимые агрегаты (total, счётчики) — с чётким правилом обновления. Подробнее — модуль 20.
