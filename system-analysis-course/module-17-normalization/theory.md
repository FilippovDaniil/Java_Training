# Модуль 17 — Нормализация и словарь данных

## Зачем этот модуль

Модуль 16 дал ER-модель. Но «нарисовать таблицы» мало — их структура может содержать **избыточность**, из-за которой возникают **аномалии** (данные ломаются при вставке/обновлении/удалении). **Нормализация** — формальный метод привести схему к виду без избыточности, опираясь на **функциональные зависимости**. Параллельно аналитик ведёт **словарь данных** — реестр атрибутов с типами, ограничениями и смыслом.

Для аналитика это рабочий инструмент: проверить чужую схему, обосновать декомпозицию, объяснить, почему «плоская таблица» — плохо, и когда наоборот оправдана **денормализация**. На собеседовании middle+ обязательно: что такое 1NF/2NF/3NF/BCNF, функциональная зависимость, какие аномалии устраняет каждая форма, когда денормализуют.

---

## 1. Функциональные зависимости (ФЗ)

**Функциональная зависимость** `X → Y` означает: значение X однозначно определяет значение Y. «Зная X, всегда знаю Y».

```
order_id → дата_заказа, client_id     (по заказу однозначно известны дата и клиент)
client_id → имя_клиента, email        (по клиенту — его имя и email)
(order_id, dish_id) → количество      (по паре заказ+блюдо — количество)
```

- **Полная ФЗ** — Y зависит от всего составного X, а не от части.
- **Частичная ФЗ** — Y зависит только от части составного ключа (проблема для 2NF).
- **Транзитивная ФЗ** — X → Y → Z, где Z зависит от неключевого Y (проблема для 3NF).

> ФЗ — фундамент нормализации. Сначала выявляем зависимости (что от чего зависит), потом по ним декомпозируем. Ключ — это атрибут(ы), от которого функционально зависят все остальные.

---

## 2. Аномалии (зачем нормализовать)

Избыточность (одни данные хранятся многократно) порождает три аномалии. Пример «плоской» таблицы заказов:

| order_id | client_id | имя_клиента | email | dish | цена |
|----------|-----------|-------------|-------|------|------|
| 1 | 77 | Иванов | i@m.ru | Ролл | 300 |
| 1 | 77 | Иванов | i@m.ru | Суп | 150 |
| 2 | 77 | Иванов | i@m.ru | Пицца | 500 |

| Аномалия | Суть | Пример в таблице |
|----------|------|------------------|
| **Вставки (insert)** | нельзя добавить факт без «лишнего» | нельзя завести клиента без заказа |
| **Обновления (update)** | менять надо во многих местах | сменить email Иванова → 3 строки, риск рассинхрона |
| **Удаления (delete)** | удаляя одно, теряем другое | удалив заказ 2, можем потерять данные клиента |

> Аномалии — практическое следствие плохой структуры. Нормализация их устраняет, разнося данные так, чтобы каждый факт хранился ровно один раз.

---

## 3. Нормальные формы

Нормальные формы — последовательные требования к структуре. Каждая следующая включает предыдущую.

### 1NF (первая нормальная форма)
- Все атрибуты **атомарны** (одно значение в ячейке, не список).
- Нет повторяющихся групп; есть ключ.

```
❌ телефоны = "+7..., +7..., +7..."        (список в одном поле)
✅ отдельная строка/таблица на каждый телефон
```

### 2NF (вторая нормальная форма)
- 1NF выполнена **И** нет **частичных** зависимостей: неключевые атрибуты зависят от **всего** составного ключа, а не от его части.

```
Таблица ПозицияЗаказа(order_id, dish_id, количество, название_блюда)
❌ название_блюда зависит только от dish_id (части ключа) → частичная ФЗ
✅ вынести название_блюда в таблицу Блюдо(dish_id → название)
```

### 3NF (третья нормальная форма)
- 2NF выполнена **И** нет **транзитивных** зависимостей: неключевой атрибут не зависит от другого неключевого.

```
Таблица Заказ(order_id, client_id, имя_клиента)
❌ имя_клиента зависит от client_id (неключевого) → транзитивная ФЗ
✅ вынести имя_клиента в таблицу Клиент(client_id → имя)
```

> Практическое правило 3NF: «каждый неключевой атрибут зависит от ключа, от всего ключа и ни от чего, кроме ключа» (the key, the whole key, and nothing but the key).

### BCNF (нормальная форма Бойса–Кодда)
- Усиленная 3NF: для **любой** ФЗ `X → Y` детерминант X должен быть **суперключом**.
- Решает редкие случаи, когда в 3NF остаётся аномалия из-за пересекающихся составных ключей-кандидатов.

```
Нарушение BCNF: есть ФЗ, где детерминант — не ключ-кандидат,
но атрибуты входят в другой ключ-кандидат. Декомпозировать так,
чтобы детерминант каждой ФЗ стал ключом.
```

| Форма | Устраняет | Кратко |
|-------|-----------|--------|
| 1NF | неатомарность, повторяющиеся группы | атомарные значения |
| 2NF | частичные зависимости | зависеть от всего ключа |
| 3NF | транзитивные зависимости | не зависеть от неключевого |
| BCNF | аномалии при пересекающихся ключах-кандидатах | детерминант = суперключ |

> В большинстве бизнес-систем целятся в **3NF** — её достаточно. BCNF/4NF/5NF — для специфичных случаев. Аналитику важно уверенно владеть 1NF–3NF и понимать смысл BCNF.

## 4. Денормализация

**Денормализация** — осознанное добавление избыточности ради производительности чтения. Обратна нормализации.

| Когда оправдана | Цена |
|-----------------|------|
| тяжёлые отчёты/аналитика (много JOIN) | риск рассинхрона данных |
| read-heavy системы, дорогие соединения | нужно поддерживать копии в актуальности |
| предрасчитанные агрегаты (сумма заказа, счётчики) | сложнее запись |
| витрины данных / DWH | дублирование |

```
Нормализовано: сумма заказа считается из позиций каждый раз (JOIN + SUM).
Денормализовано: храним order.total — быстрее читать, но надо обновлять при изменении позиций.
```

> Денормализация — не «лень нормализовать», а инженерный компромисс. Правило: **сначала нормализуй, денормализуй осознанно** под конкретную проблему производительности, зафиксировав, как поддерживать согласованность. Связано с модулем 20 (производительность данных).

## 5. Словарь данных (data dictionary)

**Словарь данных** — реестр всех элементов данных системы: для каждого атрибута — имя, тип, ограничения, смысл, источник. Артефакт аналитика, мост между моделью и реализацией.

| Поле словаря | Пример |
|--------------|--------|
| Имя атрибута | `order.total` |
| Тип | numeric(10,2) |
| Обязательность | NOT NULL |
| Ограничения | ≥ 0 |
| Описание/смысл | итоговая сумма заказа с учётом скидок |
| Источник/правило | = сумма(позиции) − скидка |
| Допустимые значения | — |
| Связи | FK → … |

> Словарь данных снимает разночтения: что значит атрибут, в каких единицах, какие значения допустимы. Особенно важен для статусов (перечислить все значения), денег (единицы, точность), дат (таймзона), кодов/справочников. Дополняет глоссарий (модуль 11).

---

## ⚠️ Подводные камни

- **Нормализовать «на глаз», без ФЗ.** Сначала выявить функциональные зависимости, потом декомпозировать по ним — иначе легко ошибиться.
- **Остановиться на 1NF/2NF.** Транзитивные зависимости (имя клиента в заказе) — частая недоработка; целиться в 3NF.
- **Путать частичную и транзитивную зависимость.** Частичная — от части составного ключа (2NF). Транзитивная — через неключевой атрибут (3NF).
- **Денормализация по умолчанию.** Избыточность без причины = аномалии. Денормализуют осознанно, под измеренную проблему.
- **Хранить производные значения без плана поддержки.** Если храните сумму/счётчик — определите, как и когда они обновляются.
- **Словарь данных «в голове».** Несогласованные единицы/значения статусов/таймзоны — источник дефектов. Фиксировать письменно.
- **Многозначные/составные атрибуты остаются.** «Телефоны через запятую», «адрес одной строкой» — нарушение 1NF.
