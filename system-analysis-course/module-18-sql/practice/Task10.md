# Задача 10 (мини-проект) — Модуль 18: SQL

## ЗАДАНИЕ
Подготовьте набор аналитических запросов — «отчёт по FoodExpress» — закрывающий типовые потребности бизнеса. Синтез всех приёмов модуля.

## ВХОДНЫЕ ДАННЫЕ
Схема и данные — из `theory.md` модуля.

## ТРЕБОВАНИЯ
Напишите запросы, отвечающие на вопросы (каждый — с ожидаемым результатом, который вы проверяете в СУБД):
1. **Выручка и число заказов по городам ресторанов** (доставленные) — GROUP BY + агрегаты.
2. **Топ-1 клиент по суммарной выручке** — агрегация + сортировка + LIMIT (или оконная RANK).
3. **Доля доставленных заказов от всех** (по числу) — агрегаты/условный COUNT.
4. **Клиенты без доставленных заказов** — LEFT JOIN/NOT EXISTS.
5. **Нарастающий итог выручки по датам** — оконная `SUM() OVER (ORDER BY дата)`.

## ФОРМАТ РЕЗУЛЬТАТА
5 SQL-запросов + полученные результаты + краткие комментарии (какой приём использован).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Запрос 1: выручка/число по городам (Москва: Суши-бар+Пиццерия; Казань: 0 доставленных).
- [ ] Запрос 2: топ-клиент — Сидорова (1950).
- [ ] Запрос 3: доставлено 5 из 6 заказов (~83%).
- [ ] Запрос 4: Петров (только отменён) и Кузнецов (нет заказов) — без доставленных.
- [ ] Запрос 5: нарастающий итог: 900, 1500, 3000, 3450, 3750.
- [ ] Использованы разные приёмы (агрегаты, JOIN, подзапрос/NOT EXISTS, оконная функция).
- [ ] Каждый результат проверен в СУБД и совпадает.

## ПОДСКАЗКА
Синтез: GROUP BY/HAVING (зад.03), JOIN (02), подзапрос (04), CTE (05), LEFT JOIN (06), оконные (07–08). Нарастающий итог = `SUM(rev) OVER (ORDER BY дата)` по дневной выручке (900→1500→3000→3450→3750). Доля = `COUNT(...) FILTER (WHERE status='Доставлен')` / `COUNT(*)`. Это типовой отчётный набор аналитика.
