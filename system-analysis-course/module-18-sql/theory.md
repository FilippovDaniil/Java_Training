# Модуль 18 — SQL для аналитика

## Зачем этот модуль

Схема спроектирована (модули 16–17) — теперь её надо **запрашивать**. SQL для системного аналитика — не «писать продакшен-запросы», а: проверять данные, готовить аналитику и отчёты, формулировать требования к выборкам, читать чужой SQL, обсуждать с разработкой логику. Без SQL аналитик слеп к реальным данным.

Этот модуль — практический. Задачи **проверяемые**: даны схема и наполнение, для каждого запроса есть ожидаемый результат — можно выполнить в любой СУБД (PostgreSQL/SQLite/онлайн-песочнице) и сверить. На собеседовании middle+ обязательно: типы JOIN, GROUP BY/HAVING, подзапросы vs CTE, оконные функции (ROW_NUMBER/RANK/LAG/LEAD).

```sql
CREATE TABLE client (
    id      INT PRIMARY KEY,
    name    VARCHAR(50),
    city    VARCHAR(50)
);
CREATE TABLE restaurant (
    id      INT PRIMARY KEY,
    name    VARCHAR(50),
    city    VARCHAR(50)
);
CREATE TABLE "order" (
    id            INT PRIMARY KEY,
    client_id     INT REFERENCES client(id),
    restaurant_id INT REFERENCES restaurant(id),
    created_at    DATE,
    status        VARCHAR(20),
    total         NUMERIC(10,2)

CREATE TABLE order_item (
    order_id  INT REFERENCES "order"(id),
    dish      VARCHAR(50),
    qty       INT,
    price     NUMERIC(10,2),
    PRIMARY KEY (order_id, dish)
);
```

## 1. SELECT / WHERE / ORDER BY

```sql
SELECT name, city FROM client
WHERE city = 'Москва'
ORDER BY name;
```

- `SELECT` — какие столбцы; `*` — все.
- `WHERE` — фильтр строк (`=`, `<>`, `>`, `IN`, `BETWEEN`, `LIKE`, `IS NULL`).
- `ORDER BY ... ASC|DESC` — сортировка; `LIMIT n` — ограничение.

> `NULL` — не значение, а «неизвестно». Сравнение `= NULL` не работает — только `IS NULL` / `IS NOT NULL`.

## 2. JOIN — соединение таблиц

| JOIN | Что возвращает |
|------|----------------|
| **INNER JOIN** | только совпадающие строки в обеих таблицах |
| **LEFT JOIN** | все из левой + совпавшие из правой (нет пары → NULL) |
| **RIGHT JOIN** | все из правой + совпавшие из левой |
| **FULL JOIN** | все из обеих (нет пары → NULL с любой стороны) |
| **SELF JOIN** | таблица соединяется сама с собой (иерархии) |

```sql
SELECT o.id, c.name
FROM "order" o
JOIN client c ON c.id = o.client_id;     -- INNER: заказы с клиентами
```

> **LEFT JOIN** — ключевой для аналитика: найти «осиротевшие»/отсутствующие связи (клиенты без заказов: `LEFT JOIN ... WHERE order.id IS NULL`).

## 3. GROUP BY + агрегаты + HAVING

Агрегаты: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`.

```sql
SELECT restaurant_id, SUM(total) AS revenue
FROM "order"
WHERE status = 'Доставлен'
GROUP BY restaurant_id
HAVING SUM(total) > 500;
```

- `GROUP BY` — группировка строк.
- агрегат считается **внутри группы**.
- `WHERE` фильтрует **до** группировки, `HAVING` — **после** (по агрегату).

> Порядок логического выполнения: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY`. Поэтому в WHERE нельзя использовать агрегат (он ещё не посчитан) — для этого HAVING.

## 4. Подзапросы и CTE

**Подзапрос** — запрос внутри запроса:
```sql
SELECT name FROM client
WHERE id IN (SELECT client_id FROM "order" WHERE total > 1000);
```

**CTE (Common Table Expression)** — именованный подзапрос через `WITH`, читаемее для многошаговых расчётов:
```sql
WITH delivered AS (
    SELECT * FROM "order" WHERE status = 'Доставлен'
)
SELECT client_id, SUM(total) FROM delivered GROUP BY client_id;
```

> CTE предпочтительнее вложенных подзапросов, когда логика многошаговая или подзапрос используется несколько раз — читается сверху вниз. Коррелированный подзапрос (ссылается на внешний запрос) считается для каждой строки — дороже.

## 5. Оконные функции ⭐

**Оконная функция** считает по «окну» строк, **не схлопывая** их в группы (в отличие от GROUP BY). Синтаксис: `функция() OVER (PARTITION BY ... ORDER BY ...)`.

| Функция | Назначение |
|---------|------------|
| `ROW_NUMBER()` | сквозной номер строки в окне |
| `RANK()` / `DENSE_RANK()` | ранг с пропусками / без пропусков |
| `LAG(x)` / `LEAD(x)` | значение из предыдущей / следующей строки |
| `SUM(x) OVER(...)` | нарастающий итог / сумма по разделу |

```sql
SELECT id, client_id, total,
       ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY created_at) AS n
FROM "order";
-- нумерует заказы внутри каждого клиента по дате
```

```sql
SELECT created_at, total,
       LAG(total) OVER (ORDER BY created_at) AS prev_total
FROM "order";
-- prev_total = сумма предыдущего по дате заказа (динамика)
```

> Главное отличие от GROUP BY: оконная функция **сохраняет все строки** и добавляет вычисленный столбец. Используется для рейтингов, нумерации, динамики «период к периоду», нарастающих итогов. Частый вопрос middle+ — «GROUP BY vs window».

## ⚠️ Подводные камни

- **NULL в сравнениях.** `= NULL` всегда неизвестно; только `IS NULL`. NULL в агрегатах: `COUNT(col)` пропускает NULL, `COUNT(*)` — нет.
- **WHERE vs HAVING.** Фильтр по агрегату — только в HAVING; обычный фильтр — в WHERE (быстрее, до группировки).
- **INNER vs LEFT JOIN.** INNER теряет строки без пары — для поиска «чего нет» нужен LEFT JOIN + IS NULL.
- **GROUP BY и не-агрегатные столбцы.** В SELECT при GROUP BY можно только сгруппированные столбцы или агрегаты.
- **Оконная функция vs GROUP BY.** Window сохраняет строки, GROUP BY схлопывает. Путаница — частая ошибка.
- **Дубли при JOIN.** Соединение 1:N размножает строки левой таблицы — агрегаты могут «задвоиться»; следить за гранулярностью.
- **Деньги и деление.** Целочисленное деление, округление, NULL в суммах — проверять типы.
