# Задача 01 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Подберите тип хранилища под 5 кейсов и обоснуйте выбор моделью доступа.

## ВХОДНЫЕ ДАННЫЕ
Кейсы:
1. Пользовательские сессии с TTL (быстрый доступ по ключу, частая запись/чтение).
2. Каталог блюд с разной структурой у разных ресторанов (атрибуты варьируются).
3. Рекомендации «с этим заказывают», обход связей между блюдами/клиентами.
4. Заказы и платежи (нужны транзакции и целостность).
5. Поток телеметрии с устройств умного дома (огромный объём записи, time-series).

## ТРЕБОВАНИЯ
- Для каждого кейса выберите семейство (реляционное / KV / document / column-family / graph).
- Обоснуйте через модель доступа (как читают/пишут данные).
- Отметьте, что было бы плохим выбором и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Кейс | Хранилище | Обоснование (модель доступа) | Плохой выбор и почему`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сессии → key-value (доступ по ключу, TTL).
- [ ] Каталог → document (гибкая структура).
- [ ] Рекомендации → graph (обход связей).
- [ ] Заказы/платежи → реляционная (ACID, целостность).
- [ ] Телеметрия → column-family/time-series (масштаб записи).
- [ ] Для каждого указан плохой вариант с обоснованием.

## ПОДСКАЗКА
Выбор = под модель доступа: «по ключу» → KV; «гибкий объект целиком» → document; «обход связей» → graph; «транзакции и целостность» → реляционная; «гигантские потоки записи» → column-family. Плохой выбор обычно — реляционная под телеметрию или document под сильно связанные транзакционные данные.
