# Задача 03 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Сравните документную и реляционную модель для каталога: спроектируйте оба варианта и обоснуйте выбор.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, каталог блюд. У блюд разный набор атрибутов: у пиццы — размер и тесто, у роллов — состав и граммовка, у напитков — объём и газация. Часто читают «блюдо целиком со всеми опциями»; пишут реже. Нужен показ карточки блюда с вариациями и отзывами.

## ТРЕБОВАНИЯ
- Спроектируйте **реляционный** вариант (таблицы блюд, атрибутов, опций, отзывов — с JOIN).
- Спроектируйте **документный** вариант (один JSON-документ блюда со вложенными опциями/отзывами).
- Сравните по: гибкость схемы, число обращений на чтение карточки, целостность, дублирование.
- Обоснуйте выбор под сценарий «читаем блюдо целиком».

## ФОРМАТ РЕЗУЛЬТАТА
Реляционная схема (таблицы) + документный JSON + сравнительная таблица + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Реляционный вариант: нормализованные таблицы + JOIN для сборки карточки.
- [ ] Документный вариант: один документ блюда с вложенными опциями/отзывами.
- [ ] Сравнение по гибкости, числу чтений, целостности, дублированию.
- [ ] Обоснован выбор document для «чтения целиком» с разной структурой.
- [ ] Отмечен минус document (дублирование/рассинхрон, например отзывы).

## ПОДСКАЗКА
Document выигрывает, когда объект читают целиком и структура варьируется (карточка блюда = один документ, одно чтение). Реляционная выигрывает при сильных связях, целостности и аналитике. Минус document — дублирование и рассинхрон (например, общие отзывы лучше не дублировать в каждый документ).
