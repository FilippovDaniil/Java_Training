# Задача 04 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Спроектируйте графовую модель для рекомендаций/связей: узлы, рёбра и запрос обхода.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, рекомендации. Связи: клиенты заказывают блюда; блюда принадлежат ресторанам; блюда бывают похожи; клиенты «похожи» по вкусам. Нужны рекомендации «с этим блюдом часто заказывают», «блюда, которые любят похожие на вас клиенты».

## ТРЕБОВАНИЯ
- Определите узлы (Клиент, Блюдо, Ресторан) и рёбра (ЗАКАЗАЛ, ПОХОЖЕ_НА, ОТНОСИТСЯ_К).
- Опишите 2 запроса обхода графа словами/псевдо-Cypher (рекомендации двух типов).
- Объясните, почему graph лучше реляционной для этих запросов (глубина связей, JOIN-ад).
- Отметьте, где graph НЕ нужен (простые FK).

## ФОРМАТ РЕЗУЛЬТАТА
Описание графа (узлы/рёбра) + 2 запроса обхода + обоснование выбора graph.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Определены узлы и типы рёбер.
- [ ] Запрос 1: «с этим часто заказывают» (обход через ЗАКАЗАЛ).
- [ ] Запрос 2: «нравится похожим клиентам» (обход через похожих).
- [ ] Объяснено преимущество graph (обход связей дешевле многократных JOIN).
- [ ] Отмечено, где graph избыточен (простые связи 1:N).

## ПОДСКАЗКА
Graph силён в обходе связей переменной глубины («друзья друзей», «с этим берут»), где реляционная требует многих рекурсивных JOIN. Модель: `(Клиент)-[ЗАКАЗАЛ]->(Блюдо)`, `(Блюдо)-[ПОХОЖЕ_НА]->(Блюдо)`. Для простых связей (заказ→клиент) graph избыточен — хватит FK.
