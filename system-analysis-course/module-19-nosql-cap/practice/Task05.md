# Задача 05 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Обоснуйте key-value/кэш для сессий и быстрого доступа: спроектируйте модель ключей и политику TTL/инвалидации.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, потребности быстрого доступа: пользовательские сессии (токен → данные сессии), корзина (до оформления), кэш карточек популярных блюд, счётчик «активных заказов» клиента.

## ТРЕБОВАНИЯ
- Спроектируйте схему ключей (паттерн именования) для каждого случая.
- Задайте TTL там, где уместно (сессии, корзина, кэш).
- Опишите политику инвалидации кэша блюд (когда сбрасывать).
- Отметьте, какие данные НЕЛЬЗЯ хранить только в KV (без надёжного хранилища).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Данные | Ключ | Значение | TTL | Инвалидация` + замечание о надёжности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Схема ключей с понятным паттерном (`session:{token}`, `cart:{client}`...).
- [ ] TTL задан для сессий/корзины/кэша.
- [ ] Политика инвалидации кэша блюд описана (по изменению блюда/по времени).
- [ ] Счётчик активных заказов — с оговоркой об источнике истины.
- [ ] Отмечено: заказы/платежи нельзя хранить ТОЛЬКО в KV (нужен durable-стор).

## ПОДСКАЗКА
KV (Redis) — быстрый доступ по ключу + TTL. Паттерн ключа: `тип:идентификатор`. Кэш — копия, не источник истины: инвалидация по изменению или TTL. Важная граница: транзакционные данные (заказы, деньги) живут в надёжном хранилище; KV — ускорение и эфемерное состояние (сессии, корзина).
