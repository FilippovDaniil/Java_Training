# Задача 06 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Определите, где eventual consistency допустима, а где нет: классифицируйте операции и обоснуйте через последствия рассинхрона.

## ВХОДНЫЕ ДАННЫЕ
Операции (FoodExpress / MedConnect):
1. Счётчик просмотров блюда.
2. Списание средств с карты.
3. Лента отзывов (новые отзывы появляются с задержкой).
4. Остаток мест/слотов у врача (MedConnect).
5. Профиль пользователя после редактирования (видит ли он сразу свои изменения).
6. Баланс бонусных баллов при оплате.

## ТРЕБОВАНИЯ
- Для каждой операции укажите: допустима ли eventual consistency.
- Где нужна strong consistency — обоснуйте последствием рассинхрона.
- Где помогает read-your-writes — отметьте (UX).
- Сформулируйте правило: когда eventual недопустима.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Операция | Согласованность (strong/eventual/RYW) | Обоснование/последствие`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Просмотры, лента → eventual (неточность некритична).
- [ ] Списание средств, бонусный баланс → strong (иначе двойное списание/неверный баланс).
- [ ] Остаток слотов → strong (иначе овербукинг).
- [ ] Профиль после редактирования → read-your-writes (пользователь видит своё изменение сразу).
- [ ] Сформулировано правило недопустимости eventual (деньги/остатки/бронь).
- [ ] Для strong-случаев указано последствие рассинхрона.

## ПОДСКАЗКА
Eventual допустима, где временный рассинхрон не вредит (просмотры, лента). Недопустима, где из устаревших данных следует неверное действие: деньги, остатки, бронь — нужна strong. Read-your-writes — компромисс для UX: глобально eventual, но свои изменения пользователь видит немедленно.
