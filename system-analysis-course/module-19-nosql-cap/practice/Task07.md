# Задача 07 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Спроектируйте документную (денормализованную) модель под сценарий чтения: соберите данные в документ под реальный паттерн доступа.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, экран «История заказов клиента»: показывает список заказов с ресторанами, позициями, статусом и суммой — всё на одном экране, читается часто. В реляционной модели это JOIN 4 таблиц.

## ТРЕБОВАНИЯ
- Спроектируйте JSON-документ заказа, денормализованный под этот экран (вложить позиции, имя ресторана, нужные поля).
- Решите, что денормализовать (скопировать) в документ, а что оставить ссылкой.
- Опишите риск рассинхрона (например, переименование ресторана) и как с ним жить.
- Сравните число обращений: document (1 чтение) vs реляционная (JOIN).

## ФОРМАТ РЕЗУЛЬТАТА
JSON-документ заказа + пояснение «что денормализовано и почему» + риск/митигация + сравнение чтений.

## КРИТЕРИИ ПРИЁМКИ
- [ ] JSON-документ содержит заказ + вложенные позиции + имя ресторана + статус/сумму.
- [ ] Явно указано, что денормализовано (скопировано) под чтение.
- [ ] Описан риск рассинхрона (переименование ресторана в старых заказах) и решение (снимок на момент заказа — это нормально для истории).
- [ ] Показано: document = 1 чтение vs реляционная = JOIN.
- [ ] Отмечено, что денормализация оправдана read-heavy сценарием.

## ПОДСКАЗКА
Документную модель проектируют **под запрос**, а не «по справочникам». Для «истории заказов» один документ = один заказ со всем нужным → одно чтение. Скопированное имя ресторана/цена — это «снимок на момент заказа», и рассинхрон тут даже желателен (история не должна меняться задним числом).
