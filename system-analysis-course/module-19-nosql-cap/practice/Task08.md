# Задача 08 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Примените PACELC к геораспределённой системе: разберите trade-off и при разделении (A/C), и в нормальном режиме (Latency/Consistency).

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** работает в нескольких регионах с репликацией данных между ЦОД. Подсистемы: каталог блюд (читается везде), заказы/платежи, профиль пользователя. Репликация между регионами имеет задержку.

## ТРЕБОВАНИЯ
- Для каждой подсистемы определите PACELC-профиль: при Partition — A или C; в Else — Latency или Consistency.
- Объясните, что значит выбор Latency vs Consistency в обычной работе (синхронная vs асинхронная репликация).
- Покажите, чем PACELC честнее CAP (учитывает нормальный режим).
- Приведите пример системы с профилем PA/EL и PC/EC.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Подсистема | при Partition (A/C) | в Else (L/C) | Обоснование` + пояснение PACELC vs CAP.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каталог → PA/EL (доступность + низкая задержка, eventual репликация).
- [ ] Заказы/платежи → PC/EC (согласованность важнее задержки).
- [ ] Профиль → разумный профиль (часто PA/EL + read-your-writes).
- [ ] Объяснён trade-off Latency vs Consistency в нормальном режиме (синхр./асинхр. репликация).
- [ ] Показано, что PACELC учитывает и обычную работу (честнее CAP).
- [ ] Приведены примеры PA/EL (Cassandra) и PC/EC (RDBMS).

## ПОДСКАЗКА
PACELC: *if Partition → A или C; Else → Latency или Consistency*. Даже без сбоев строгая согласованность (синхронная репликация на все регионы) стоит задержки. Каталог терпит устаревание → PA/EL. Платежи требуют корректности → PC/EC. PACELC честнее CAP, потому что описывает компромисс и в нормальном режиме.
