# Задача 09 — Модуль 19: NoSQL и CAP

## ЗАДАНИЕ
Спроектируйте полиглот-персистентность: распределите данные системы по подходящим хранилищам и опишите согласование между ними.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** целиком. Данные: заказы и платежи; каталог блюд (гибкая структура); сессии и корзина; рекомендации (связи); поиск по блюдам; лента событий/телеметрия доставки.

## ТРЕБОВАНИЯ
- Сопоставьте каждый тип данных с хранилищем (реляц/document/KV/graph/column-family/поисковый индекс).
- Обоснуйте каждый выбор моделью доступа.
- Опишите, как данные синхронизируются между хранилищами (события/CDC) — например, заказ в реляционной → индекс в поиске.
- Назовите операционную цену полиглота и когда его НЕ стоит вводить.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Данные | Хранилище | Обоснование` + описание синхронизации + замечание о цене.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Заказы/платежи → реляционная; каталог → document; сессии/корзина → KV; рекомендации → graph; поиск → поисковый индекс; телеметрия → column-family.
- [ ] Каждый выбор обоснован моделью доступа.
- [ ] Описана синхронизация между хранилищами (события/CDC, источник истины).
- [ ] Назван источник истины для ключевых данных (заказы — реляционная).
- [ ] Указана операционная цена полиглота.
- [ ] Отмечено, когда полиглот избыточен (малый масштаб → одна БД).

## ПОДСКАЗКА
Полиглот = каждый тип данных в подходящее хранилище, с явным источником истины. Синхронизация — через события/CDC (заказ создан → проиндексировать в поиске, обновить рекомендации). Цена — операционная сложность (несколько систем + согласование). Для небольшой системы достаточно одной реляционной БД — полиглот вводят под реальный масштаб.
