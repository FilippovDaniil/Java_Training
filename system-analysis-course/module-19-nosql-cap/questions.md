## Ключ (эталонные ответы)

1. Реляционные — строгая схема, связи через JOIN, ACID-транзакции, вертикальное масштабирование. NoSQL — гибкая/без схемы, денормализация, часто мягкая согласованность, горизонтальное масштабирование. Это разные наборы компромиссов.
2. «Not only SQL» — не «анти-SQL», а «не только реляционная модель»: семейство хранилищ под задачи, где реляционная неудобна.
3. **Key-value** (доступ по ключу: кэш/сессии), **document** (гибкие объекты: каталоги/профили), **column-family** (огромные объёмы записи: телеметрия/логи), **graph** (связи: рекомендации/соцсети).
4. Быстрый доступ по ключу: сессии, кэш, корзина, счётчики, конфигурация (часто с TTL).
5. Гибкая структура и чтение объекта целиком: каталоги, профили, контент, документы с вложенностью.
6. Очень большие объёмы и интенсивная запись по строкам с тысячами столбцов: телеметрия, логи, time-series.
7. Данные с богатыми связями и обходом графа переменной глубины: соцсвязи, рекомендации, маршруты, графы знаний.
8. **Schema-on-write** — структура задана при записи (реляционные: данные обязаны соответствовать схеме). **Schema-on-read** — структура интерпретируется при чтении (гибкость NoSQL).
9. В распределённой системе при сетевом разделении нельзя одновременно гарантировать согласованность (C) и доступность (A) — приходится выбирать.
10. **C** — все узлы видят последние данные; **A** — каждый запрос получает ответ; **P** — система работает при разрыве связи между узлами.
11. Потому что в распределённой системе сеть ненадёжна — разделение (P) неизбежно. «CA» возможна только без распределения (один узел). При наличии P выбор сводится к CP или AP.
12. **CP** при разделении жертвует доступностью (отказывает ради корректности). **AP** жертвует согласованностью (отвечает, возможно, устаревшими данными ради отзывчивости).
13. **CP** — деньги, остатки товара, бронь (корректность критична). **AP** — лента, лайки, рекомендации (важнее ответ, чем его свежесть).
14. **PACELC**: при Partition выбираем A или C; Else (в нормальном режиме) — Latency или Consistency. Учитывает компромисс и без сбоев.
15. Строгая согласованность требует синхронной репликации на узлы → растёт задержка. Низкая задержка достигается асинхронной репликацией ценой временного устаревания данных.
16. Гарантия, что любое чтение видит последнюю подтверждённую запись (как будто данные в одном месте).
17. Реплики сходятся «со временем»; чтение может вернуть устаревшие данные. Допустима там, где временный рассинхрон безвреден: счётчики, лента, рекомендации.
18. Где из устаревших данных следует неверное действие: списание средств, остаток/стоп-лист, бронирование слота (двойное списание, овербукинг).
19. Пользователь всегда видит **свои** изменения сразу (даже если глобально система eventual). Частый компромисс для UX.
20. Гарантия порядка для причинно-связанных операций (комментарий виден только после поста, к которому относится).
21. **ACID** (Atomicity, Consistency, Isolation, Durability) — строгие гарантии транзакций (реляционные). **BASE** (Basically Available, Soft state, Eventually consistent) — мягкая модель многих NoSQL: доступность и итоговая согласованность вместо немедленной.
22. Использование разных хранилищ под разные данные одной системы — каждое под свою модель доступа (реляционная для заказов, document для каталога, KV для сессий, graph для рекомендаций и т.д.).
23. Операционная сложность: несколько систем для эксплуатации/мониторинга, необходимость синхронизировать данные между ними и поддерживать согласованность.
24. Через события/CDC (Change Data Capture): изменение в источнике истины публикуется событием, потребители обновляют свои хранилища (поиск, рекомендации, кэш).
25. При малом масштабе и сильно связанных транзакционных данных — достаточно одной реляционной БД. NoSQL/полиглот вводят под реальную потребность (масштаб, специфичный доступ), а не превентивно.
