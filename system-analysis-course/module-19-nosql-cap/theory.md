# Модуль 19 — Реляционные vs NoSQL, теорема CAP

## Зачем этот модуль

Реляционная БД (модули 16–18) — отличный дефолт, но не универсальный. Каталоги с разной структурой, графы связей, кэши, потоки событий, петабайты телеметрии — для них реляционная модель бывает неудобна или не масштабируется. Аналитик должен **понимать семейства хранилищ**, уметь обосновать выбор и оперировать **теоремой CAP/PACELC** — почему в распределённой системе нельзя «всё сразу».

Это модуль про **решения и trade-offs**, а не про синтаксис. На собеседовании middle+ обязательно: типы NoSQL и под что они, что такое CAP, чем strong отличается от eventual consistency, когда eventual недопустима, что такое полиглот-персистентность.

---

## 1. Реляционные vs NoSQL

| Критерий | Реляционные (SQL) | NoSQL |
|----------|-------------------|-------|
| Схема | строгая (schema-on-write) | гибкая/без схемы (schema-on-read) |
| Связи | JOIN, нормализация | денормализация, вложенность |
| Транзакции | ACID | часто BASE (мягкая согласованность) |
| Масштабирование | вертикальное (сложнее горизонтальное) | горизонтальное (шардирование «из коробки») |
| Запросы | мощный SQL | ограничены под модель доступа |
| Когда | сложные связи, целостность, транзакции | масштаб, гибкость, специфичный доступ |

> NoSQL ≠ «лучше». Это «не только SQL» — другой набор компромиссов. Реляционная БД остаётся дефолтом для большинства бизнес-данных с целостностью и транзакциями.

---

## 2. Семейства NoSQL

| Семейство | Модель данных | Сильно в | Примеры | Кейс |
|-----------|---------------|----------|---------|------|
| **Key-Value** | ключ → значение | быстрый доступ по ключу | Redis, DynamoDB | сессии, кэш, счётчики |
| **Document** | JSON-документы | гибкая структура, вложенность | MongoDB, Couchbase | каталоги, профили, контент |
| **Column-family** | широкие столбцы по строкам | огромные объёмы, запись | Cassandra, HBase | телеметрия, логи, time-series |
| **Graph** | узлы + рёбра | связи и обход графа | Neo4j, JanusGraph | соцсети, рекомендации, маршруты |

```
Key-Value:  "session:123" → {user, expires}
Document:   { _id, name, options:[...], reviews:[{...}] }   ← всё в одном документе
Column:     row_key → {col1, col2, ... тысячи столбцов}
Graph:      (Клиент)-[ЗАКАЗАЛ]->(Блюдо)-[ПОХОЖЕ]->(Блюдо)
```

> Выбор семейства = под **модель доступа**: «достаю по ключу» → KV; «храню гибкий объект целиком» → document; «пишу гигантские потоки» → column-family; «обхожу связи» → graph.

## 3. Теорема CAP

В распределённой системе при **сетевом разделении (Partition)** нельзя одновременно гарантировать и согласованность, и доступность. Три свойства:

- **C (Consistency)** — все узлы видят одни и те же данные одновременно (любое чтение видит последнюю запись).
- **A (Availability)** — каждый запрос получает ответ (не ошибку), даже если часть узлов недоступна.
- **P (Partition tolerance)** — система продолжает работать при разрыве связи между узлами.

```
         C
        / \
   CA  /   \  CP        Сеть может разорваться всегда (P обязательна
      /     \           в распределённой системе) → реальный выбор
     A ───── P          между CP и AP при разделении.
        AP
```

> Главное понимание: **P неизбежна** в распределённой системе (сеть ненадёжна). Значит, при разделении выбираем: **CP** (жертвуем доступностью — отказываем в ответе ради консистентности) или **AP** (жертвуем согласованностью — отвечаем, возможно, устаревшими данными). «CA» существует только без распределения (один узел).

| Выбор | Что жертвуем при разрыве | Когда |
|-------|--------------------------|-------|
| **CP** | доступность (часть запросов падает) | деньги, остатки, бронь — важна корректность |
| **AP** | согласованность (видны старые данные) | лента, лайки, корзина — важна отзывчивость |

---

## 4. PACELC — расширение CAP

CAP описывает только поведение при разделении. **PACELC**: *if Partition → choose A or C; Else (нормальная работа) → choose Latency or Consistency.*

> Даже без сбоев есть компромисс **задержка vs согласованность**: строгая согласованность (синхронная репликация на все узлы) стоит latency. PACELC честнее CAP: система выбирает trade-off и в обычном режиме (например, Cassandra = PA/EL — доступность и низкая задержка; традиционная RDBMS = PC/EC — согласованность).

---

## 5. Модели согласованности

| Модель | Гарантия | Пример |
|--------|----------|--------|
| **Strong consistency** | чтение всегда видит последнюю запись | банковский баланс |
| **Eventual consistency** | реплики сойдутся «когда-нибудь»; чтение может быть устаревшим | счётчик лайков, лента |
| **Causal consistency** | причинно-связанные операции видны по порядку | комментарий после поста |
| **Read-your-writes** | пользователь видит свои изменения сразу | свой профиль после редактирования |

> Eventual consistency — норма для масштабируемых AP-систем. Вопрос аналитика: **где она допустима, а где нет**. Лайки/просмотры — да. Списание денег, остаток товара, бронь слота — нет (нужна strong). Промежуточные варианты (read-your-writes) часто спасают UX.

---

## 6. Полиглот-персистентность

**Polyglot persistence** — использовать **разные хранилища под разные данные** одной системы, по их модели доступа.

Пример FoodExpress:
```
Заказы, платежи        → реляционная (ACID, целостность)
Каталог блюд/меню      → document (гибкая структура, вложенность)
Сессии, корзина        → key-value (Redis, быстрый доступ, TTL)
Рекомендации «с этим берут» → graph
Лента событий/телеметрия    → column-family / лог-стор
Поиск по блюдам        → поисковый индекс (OpenSearch)
```

> Полиглот — не «зоопарк ради моды», а осознанное распределение: каждый тип данных — в подходящее хранилище. Цена — операционная сложность (несколько систем, согласование между ними). Решение принимают по реальным потребностям, а не превентивно.

---

## ⚠️ Подводные камни

- **«NoSQL = быстрее/лучше».** Это другой набор компромиссов (масштаб/гибкость ценой целостности/запросов), а не апгрейд. Дефолт — реляционная.
- **«CA-система».** В распределённой системе P обязательна; реальный выбор — CP или AP. «CA» = один узел.
- **Eventual там, где нужна strong.** Деньги/остатки/бронь требуют сильной согласованности; eventual приведёт к двойным списаниям/овербукингу.
- **Документная модель под сильно связанные данные.** Если всюду нужны «JOIN», document обернётся дублированием и рассинхроном — возможно, нужна реляционная или graph.
- **Графовая БД ради «связей вообще».** Graph оправдан для обхода связей (друзья друзей, маршруты), не для любых FK.
- **Полиглот без нужды.** Каждое хранилище — операционные расходы; вводить только под реальную потребность.
- **Игнорировать согласованность между хранилищами.** При полиглоте данные синхронизируют (события/CDC) — это отдельная задача (модуль 25, 32).
