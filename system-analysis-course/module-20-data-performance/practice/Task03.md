# Задача 03 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Спроектируйте стратегию репликации (read replicas) и разберите риск устаревшего чтения.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: мастер перегружен чтением (дашборды, история заказов, каталог). 90% операций — чтение, 10% — запись. Нужно снять нагрузку чтения. Есть требование: пользователь после оформления заказа должен сразу видеть его в своём списке.

## ТРЕБОВАНИЯ
- Предложите схему master-replica: что идёт в мастер, что на реплики.
- Выберите синхронную/асинхронную репликацию и обоснуйте.
- Разберите риск replication lag (устаревшее чтение с реплики).
- Решите проблему «пользователь видит свой свежий заказ» (read-your-writes).

## ФОРМАТ РЕЗУЛЬТАТА
Схема репликации + распределение операций (мастер/реплики) + решение проблемы read-your-writes.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Запись → мастер, тяжёлые чтения (дашборды, каталог) → реплики.
- [ ] Выбран тип репликации (асинхронная для масштаба чтения) с обоснованием.
- [ ] Описан риск replication lag (устаревшее чтение).
- [ ] Решён read-your-writes: свежие данные пользователя читать с мастера (или sticky/sync для них).
- [ ] Отмечено, что критичные по свежести чтения не идут на лаговую реплику.

## ПОДСКАЗКА
Read replicas масштабируют чтение, но асинхронная репликация даёт лаг → реплика отстаёт. Каталог/дашборды лаг терпят. «Свой только что созданный заказ» — нет: читать с мастера или направлять чтения этого пользователя на мастер некоторое время (read-your-writes). Связь с eventual consistency (модуль 19).
