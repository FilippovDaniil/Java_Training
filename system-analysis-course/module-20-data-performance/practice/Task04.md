# Задача 04 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Выберите ключ шардирования и оцените перекос для большой таблицы.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, таблица заказов выросла до сотен миллионов строк, нужно шардировать. Кандидаты на ключ шардирования:
1. `created_at` (дата заказа).
2. `client_id`.
3. `restaurant_id`.
4. хеш от `order_id`.
Основные паттерны доступа: заказы конкретного клиента; заказы конкретного ресторана; свежие заказы (последние сутки) читаются чаще всего.

## ТРЕБОВАНИЯ
- Оцените каждый кандидат: равномерность, перекос (hot shard), стоимость основных запросов.
- Выберите ключ и обоснуйте под паттерны доступа.
- Разберите проблему «горячего шарда» для ключа по дате.
- Отметьте, какие запросы станут кросс-шардовыми (дорогими).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Ключ | Равномерность | Hot shard? | Стоимость запросов` + выбор с обоснованием.

## КРИТЕРИИ ПРИЁМКИ
- [ ] `created_at` → перекос: свежий шард горячий (вся свежая нагрузка в один шард).
- [ ] `client_id` → равномернее, дёшевы запросы по клиенту; запросы по ресторану кросс-шардовые.
- [ ] хеш(order_id) → равномерно, но запросы по клиенту/ресторану кросс-шардовые.
- [ ] Выбран ключ с обоснованием под основной паттерн доступа.
- [ ] Разобран hot shard для ключа по дате.
- [ ] Названы запросы, которые станут кросс-шардовыми.

## ПОДСКАЗКА
Хороший ключ шардирования: высокая кардинальность, равномерное распределение, совпадает с основным паттерном доступа (чтобы запрос бил в один шард). Ключ по дате концентрирует свежую нагрузку в одном шарде (hot shard). `client_id` локализует запросы клиента, но «по ресторану» расходится по шардам. Идеального ключа нет — выбор под доминирующий паттерн.
