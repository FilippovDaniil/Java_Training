# Задача 05 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Спроектируйте кэширование по паттерну cache-aside с продуманной инвалидацией.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: карточка ресторана с меню читается очень часто (главная, поиск), меняется редко (ресторан правит меню несколько раз в день). Сейчас каждый показ — тяжёлый запрос к БД с JOIN. Нужно кэшировать.

## ТРЕБОВАНИЯ
- Опишите поток cache-aside на чтение (кэш → промах → БД → положить в кэш).
- Опишите поток на запись (изменение меню → что делать с кэшем).
- Выберите стратегию инвалидации (TTL / по событию / write-through) и обоснуйте.
- Разберите риски: stale cache, cache stampede; как смягчить.
- Отметьте, что НЕ стоит кэшировать (остатки/стоп-лист — почему).

## ФОРМАТ РЕЗУЛЬТАТА
Описание потоков чтения/записи + стратегия инвалидации + риски и митигация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Поток чтения cache-aside описан (промах → БД → запись в кэш).
- [ ] Поток записи: при изменении меню кэш инвалидируется/обновляется.
- [ ] Выбрана инвалидация (по событию изменения меню + TTL как страховка) с обоснованием.
- [ ] Разобран stale cache и cache stampede + смягчение (TTL+jitter, блокировка перестроения).
- [ ] Отмечено, что стоп-лист/остатки кэшировать с осторожностью (нужна свежесть).

## ПОДСКАЗКА
Cache-aside: приложение само ходит в кэш, при промахе грузит из БД и кладёт в кэш. Меню меняется редко → кэшируем с инвалидацией по событию «меню изменено» + TTL как страховка. Stampede (лавина промахов) смягчают TTL c разбросом (jitter) и единичным перестроением. Остатки — про свежесть, кэш тут опасен.
