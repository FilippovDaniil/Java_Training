# Задача 06 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Разберите ACID на примере перевода средств: покажите, как каждое свойство защищает корректность, и что ломается без него.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, операция «оплата заказа бонусными баллами»: списать N баллов со счёта клиента и зачислить эквивалент ресторану (две записи в разных строках). Параллельно клиент может тратить баллы в другом заказе.

## ТРЕБОВАНИЯ
- Для каждого свойства ACID покажите, как оно защищает эту операцию.
- Для каждого — опишите конкретный сбой, если свойства нет.
- Укажите уровень изоляции/блокировку, чтобы два параллельных списания не ушли в минус.
- Отметьте роль идемпотентности при повторе запроса (связь с модулем 25).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Свойство ACID | Как защищает | Сбой без него` + решение по изоляции + замечание об идемпотентности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Atomicity: списание+зачисление вместе; без него — баллы «исчезли»/«удвоились» при сбое посередине.
- [ ] Consistency: не уйти в минус, инвариант баланса; без него — отрицательный баланс.
- [ ] Isolation: параллельные списания не видят промежуточных значений; без него — двойное списание (lost update).
- [ ] Durability: после commit списание не потеряется при сбое.
- [ ] Указан уровень изоляции/блокировка (Serializable или `SELECT FOR UPDATE`) против гонки.
- [ ] Отмечена идемпотентность для повторного запроса оплаты.

## ПОДСКАЗКА
Перевод — классический пример ACID. Atomicity = «оба или ни одного». Isolation против lost update: два параллельных списания без изоляции могут оба прочитать старый баланс и увести в минус — нужен Serializable или `FOR UPDATE`. Idempotency key защищает от повторного списания при ретрае (модуль 25).
