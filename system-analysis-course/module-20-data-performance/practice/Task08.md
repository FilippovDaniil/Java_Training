# Задача 08 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Найдите узкое место по словесному описанию плана запроса и предложите оптимизацию.

## ВХОДНЫЕ ДАННЫЕ
Медленный запрос (FoodExpress): «выручка по ресторанам Москвы за месяц». Словесный план выполнения:
- Seq Scan (полное сканирование) таблицы `order` (200 млн строк) с фильтром `created_at BETWEEN ... AND status='Доставлен'`.
- Hash Join с `restaurant` по `restaurant_id`.
- Фильтр `restaurant.city='Москва'` применяется ПОСЛЕ соединения.
- Сортировка большого промежуточного результата для GROUP BY.

## ТРЕБОВАНИЯ
- Найдите узкие места плана (полное сканирование, поздний фильтр, сортировка).
- Предложите оптимизации: индексы, порядок фильтрации, предагрегация/витрина.
- Объясните, как индекс по `(created_at)` или `(restaurant_id, created_at)` изменит план.
- Отметьте, когда стоит перейти к денормализованной витрине (связь с модулем 17).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Узкое место | Причина | Оптимизация` + вывод о витрине для частого тяжёлого отчёта.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Найден Seq Scan по 200 млн строк (нет подходящего индекса).
- [ ] Предложен индекс под фильтр по дате/ресторану.
- [ ] Отмечен поздний фильтр по городу (раньше сузить через restaurant или денормализацию города в заказ).
- [ ] Предложена предагрегация/материализованная витрина для частого отчёта.
- [ ] Объяснено изменение плана при добавлении индекса (Index Scan вместо Seq Scan).
- [ ] Указано, когда витрина оправдана (частый тяжёлый отчёт).

## ПОДСКАЗКА
Признаки проблем в плане: Seq Scan большой таблицы вместо Index Scan; фильтр применяется поздно (после JOIN); сортировка огромного промежуточного набора. Лечение: индекс под фильтр, сузить данные раньше, для повторяющегося тяжёлого отчёта — материализованное представление/витрина (денормализация, модуль 17). `EXPLAIN` показывает реальный план.
