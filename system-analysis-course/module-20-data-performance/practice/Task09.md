# Задача 09 — Модуль 20: Производительность данных

## ЗАДАНИЕ
Разберите риски согласованности кэша и БД: найдите сценарии рассинхрона и предложите, как их избежать.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** кэширует профиль клиента и его баланс баллов в Redis (cache-aside). Сценарии:
1. Баланс обновлён в БД, но кэш не инвалидирован — оплата читает старый баланс.
2. Два параллельных запроса: один пишет в БД, другой только что положил старое значение в кэш.
3. Запись в БД прошла, инвалидация кэша упала (сеть) — кэш остался устаревшим.
4. Кэш используется как источник истины для списания баллов.

## ТРЕБОВАНИЯ
- Для каждого сценария опишите риск (что увидит пользователь/система).
- Предложите решение (порядок операций, write-through, инвалидация, не кэшировать критичное).
- Сформулируйте правило: что нельзя обслуживать из кэша.
- Отметьте компромисс TTL: короткий (свежее, больше нагрузки) vs длинный.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Риск | Решение` + правило про критичные данные + замечание про TTL.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сценарий 1: устаревший баланс → инвалидация/обновление кэша при записи.
- [ ] Сценарий 2: race put старого значения → правильный порядок (сначала БД, потом инвалидация; версия).
- [ ] Сценарий 3: упавшая инвалидация → TTL как страховка / повтор инвалидации.
- [ ] Сценарий 4: кэш как источник истины для баллов — недопустимо (списание из durable-БД в транзакции).
- [ ] Сформулировано правило: критичное (деньги/баланс/остатки) не обслуживать из кэша как истины.
- [ ] Отмечен компромисс TTL.

## ПОДСКАЗКА
Кэш — копия, а не источник истины. Списание баллов идёт из БД в транзакции (модуль 06), кэш — лишь для отображения. Порядок: писать БД → инвалидировать кэш (не наоборот). Упавшая инвалидация → TTL спасает (данные протухнут сами). Для критичной свежести — короткий TTL или вовсе не кэшировать.
