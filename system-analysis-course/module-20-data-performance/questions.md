## Ключ (эталонные ответы)

1. **Транзакция** — группа операций, выполняемых как единое целое (либо все, либо ни одной).
2. **ACID**: Atomicity (атомарность), Consistency (согласованность), Isolation (изоляция), Durability (долговечность).
3. «Всё или ничего»: либо все операции транзакции применяются, либо при сбое откатываются все.
4. После `COMMIT` данные сохранены надёжно и переживут сбой/перезагрузку системы.
5. Полная изоляция дорога (блокировки, низкий параллелизм); уровни — компромисс между корректностью и производительностью.
6. Dirty read, non-repeatable read, phantom read (и lost update).
7. Чтение незафиксированных изменений другой транзакции (которая может откатиться).
8. Повторное чтение той же строки в одной транзакции даёт другое значение (её изменила и зафиксировала другая транзакция).
9. Повторный запрос по условию возвращает новые строки, появившиеся из-за вставок другой транзакции.
10. Read Uncommitted → Read Committed → Repeatable Read → Serializable.
11. Обычно **Read Committed** (PostgreSQL, Oracle, SQL Server по умолчанию; MySQL InnoDB — Repeatable Read).
12. **Оптимистическая** — не блокирует, при фиксации проверяет версию (хороша при редких конфликтах). **Пессимистическая** — блокирует строку/таблицу на время работы (хороша при частых конфликтах).
13. Ситуация, когда две транзакции взаимно ждут ресурсы друг друга; СУБД обнаруживает и откатывает одну.
14. **Индекс** — структура для быстрого поиска по столбцу. Цена: замедляет запись (нужно обновлять индексы), занимает место.
15. Индексируют: FK, столбцы в WHERE/JOIN/ORDER BY с высокой селективностью. Не индексируют: низкоселективные (статус с 2-3 значениями), часто меняющиеся, маленькие таблицы.
16. Составной индекс `(a, b)` работает по префиксу: помогает фильтру по `a` и по `a,b`, но не по одному `b`. Порядок столбцов критичен.
17. Запись идёт в master, чтение распределяется по репликам (копиям); даёт масштабирование чтения и отказоустойчивость.
18. **Синхронная** — commit после подтверждения репликой (нет потери данных, выше latency). **Асинхронная** — реплика догоняет позже (быстрее, но возможен лаг).
19. **Replication lag** — отставание реплики от мастера при асинхронной репликации; чтение с реплики может вернуть устаревшие данные.
20. **Партиционирование** — деление одной таблицы на части в пределах одной БД (управляемость, скорость). **Шардирование** — разнос данных по разным узлам по ключу (масштаб записи/объёма).
21. Ключ с высокой кардинальностью и равномерным распределением, совпадающий с паттерном доступа. **Hot shard** — перегруженный шард из-за неравномерного ключа (например, по дате — вся свежая нагрузка в один шард).
22. Паттерн кэширования: читать из кэша; при промахе — из БД, положить в кэш, вернуть; при записи — обновить БД и инвалидировать/обновить кэш.
23. Сложно держать кэш согласованным с БД: stale cache (устаревшие данные), cache stampede (лавина промахов в БД), порядок операций и упавшая инвалидация. Решают TTL, инвалидацией по событию, write-through.
24. Критичные данные, где устаревание ведёт к неверному действию: баланс/деньги, остатки/стоп-лист, бронь. Их источник истины — durable-БД, кэш только для отображения.
25. Нормализация устраняет избыточность, но добавляет JOIN (дороже чтение). Денормализация/кэш/индексы ускоряют чтение ценой места, записи и риска рассинхрона — это компромисс производительность/целостность.
