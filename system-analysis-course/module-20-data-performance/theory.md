# Модуль 20 — Производительность данных

## Зачем этот модуль

Схема спроектирована и нормализована (16–17), запросы написаны (18), выбор хранилища обоснован (19). Теперь — **как сделать так, чтобы данные работали быстро и надёжно под нагрузкой**. Это завершение Части 4: транзакции и ACID, уровни изоляции, индексы, репликация, шардирование, партиционирование, кэширование.

Аналитику это нужно, чтобы формулировать NFR к данным, понимать ограничения («почему так нельзя», «откуда тормоза»), обсуждать с разработкой и архитектором компромиссы. На собеседовании middle+ обязательно: ACID, уровни изоляции и аномалии, зачем/какие индексы и их цена, разница репликации и шардирования, паттерн cache-aside и инвалидация.

---

## 1. Транзакции и ACID

**Транзакция** — группа операций, выполняемых как единое целое. Гарантии ACID:

| Свойство | Смысл | Пример |
|----------|-------|--------|
| **A — Atomicity** | всё или ничего | перевод: списать И зачислить, либо откат |
| **C — Consistency** | переход между корректными состояниями | не нарушаются ограничения/инварианты |
| **I — Isolation** | параллельные транзакции не мешают | две оплаты не видят промежуточных состояний |
| **D — Durability** | зафиксированное не теряется | после commit данные переживут сбой |

```
BEGIN;
  UPDATE account SET balance = balance - 1000 WHERE id = 1;
  UPDATE account SET balance = balance + 1000 WHERE id = 2;
COMMIT;   -- атомарно: либо оба UPDATE, либо ни одного
```

> ACID — фундамент для денег, остатков, броней. Если операцию нельзя «оставить наполовину» — она должна быть в транзакции.

## 2. Уровни изоляции и аномалии

Полная изоляция дорога, поэтому СУБД предлагают **уровни** — компромисс между корректностью и параллелизмом. Аномалии, возникающие при слабой изоляции:

| Аномалия | Суть |
|----------|------|
| **Dirty read** | чтение незафиксированных чужих изменений |
| **Non-repeatable read** | повторное чтение строки даёт другое значение (её изменили) |
| **Phantom read** | повторный запрос по условию возвращает новые строки |

| Уровень изоляции | Dirty | Non-repeatable | Phantom |
|------------------|-------|----------------|---------|
| **Read Uncommitted** | возможен | возможен | возможен |
| **Read Committed** | нет | возможен | возможен |
| **Repeatable Read** | нет | нет | возможен* |
| **Serializable** | нет | нет | нет |

\* В некоторых СУБД (PostgreSQL) Repeatable Read устраняет и фантомы.

> Чем выше изоляция — тем меньше аномалий, но больше блокировок/конфликтов (медленнее, риск deadlock). Дефолт в большинстве СУБД — **Read Committed**. Для денежных операций часто нужен Serializable или явные блокировки. Аналитик должен уметь сказать, какие аномалии недопустимы для операции.

---

## 3. Блокировки

- **Оптимистическая** — не блокируем, но при фиксации проверяем, не изменил ли данные кто-то ещё (версия/timestamp). Хороша при редких конфликтах.
- **Пессимистическая** — блокируем строку/таблицу на время работы (`SELECT ... FOR UPDATE`). Хороша при частых конфликтах.
- **Deadlock** — две транзакции ждут ресурсы друг друга. СУБД обнаруживает и откатывает одну.

> Оптимистическая блокировка через `version` — частый паттерн в приложениях (и в ORM). Аналитику важно заложить поле версии/правило разрешения конфликтов для конкурентно изменяемых сущностей.

---

## 4. Индексы

**Индекс** — структура для быстрого поиска строк по столбцу(ам), как алфавитный указатель в книге. Ускоряет чтение, **замедляет запись** и занимает место.

| Тип | Когда |
|-----|-------|
| **B-tree** | дефолт: равенство и диапазоны (`=`, `<`, `BETWEEN`, `ORDER BY`) |
| **Hash** | только равенство (`=`) |
| **Составной** | фильтр/сортировка по нескольким столбцам (порядок важен!) |
| **Частичный** | индекс по подмножеству строк (`WHERE active`) |
| **Полнотекстовый / GIN** | поиск по тексту, JSON, массивам |

```sql
CREATE INDEX idx_order_client_date ON "order"(client_id, created_at);
-- ускорит: WHERE client_id=? ORDER BY created_at
```

**Что индексировать:** FK, столбцы в WHERE/JOIN/ORDER BY с высокой селективностью.
**Что НЕ индексировать:** низкоселективные столбцы (пол, статус с 2 значениями), часто меняющиеся, маленькие таблицы.

> Цена индекса: каждая вставка/обновление обновляет и индексы. «Проиндексировать всё» = быстрые чтения, медленная запись и раздутая БД. Правило: индекс под реальные запросы, а не превентивно. Составной индекс работает по **префиксу** — порядок столбцов критичен.

## 5. Репликация

**Репликация** — копии данных на нескольких узлах.

| Схема | Суть | Зачем |
|-------|------|-------|
| **Master-replica** | запись в master, чтение с реплик | масштабирование чтения, отказоустойчивость |
| **Синхронная** | commit после записи в реплику | нет потери данных, выше latency |
| **Асинхронная** | реплика догоняет | быстрее, но возможен лаг (устаревшее чтение) |

```
        ┌──► Replica 1 (чтение)
Master ─┤──► Replica 2 (чтение)
(запись)└──► Replica 3 (failover)
```

> Read replicas снимают нагрузку чтения с мастера. Асинхронная репликация → **replication lag**: чтение с реплики может вернуть устаревшие данные (связь с eventual consistency, модуль 19). Read-your-writes для пользователя часто требует читать его свежие данные с мастера.

## 6. Шардирование и партиционирование

- **Партиционирование** — разбиение **одной** таблицы на части в пределах одной БД (по диапазону дат, по списку). Управляемость и скорость запросов к части.
- **Шардирование (sharding)** — горизонтальное разбиение данных по **разным** узлам/БД по **ключу шардирования**. Масштабирование записи и объёма.

| Стратегия ключа | Плюс | Риск |
|-----------------|------|------|
| по диапазону (range) | простые диапазонные запросы | перекос (hot shard) на свежих данных |
| по хешу (hash) | равномерность | диапазонные запросы дороже |
| по справочнику (directory) | гибкость | доп. слой маппинга |

```
Партиционирование: orders → orders_2026_q1, orders_2026_q2 (одна БД)
Шардирование:       client_id % N → shard_0 ... shard_N (разные узлы)
```

> Ключ шардирования — критическое решение: плохой ключ → **перекос** (один шард перегружен) и дорогие кросс-шардовые запросы/джойны. Выбирают ключ с высокой кардинальностью и равномерным распределением, под основной паттерн доступа.

---

## 7. Кэширование

**Кэш** — быстрое хранилище (обычно в памяти, Redis) для горячих данных.

**Cache-aside (lazy loading)** — основной паттерн:
```
1. читать из кэша
2. промах → читать из БД → положить в кэш → вернуть
3. запись → обновить БД + инвалидировать/обновить кэш
```

**Инвалидация** — самое сложное:
- **TTL** — истечение по времени (простое, но возможны устаревшие данные);
- **по событию** — сбросить ключ при изменении данных;
- **write-through** — писать в кэш и БД одновременно.

| Проблема | Суть |
|----------|------|
| **Stale cache** | в кэше устаревшие данные после изменения в БД |
| **Cache stampede** | массовый промах → лавина запросов в БД |
| **Согласованность** | кэш и БД рассинхронизированы |

> «Две сложные вещи в IT: инвалидация кэша и именование». Кэш — копия, не источник истины. Аналитик задаёт: что кэшируем, на какой TTL, когда инвалидируем, допустимо ли устаревание (для денег/остатков — обычно нет).

---

## ⚠️ Подводные камни

- **Игнорировать изоляцию для денег.** Read Committed по умолчанию допускает non-repeatable/phantom — для финансов нужны Serializable или явные блокировки.
- **Индексировать всё.** Индексы ускоряют чтение ценой записи и места; под реальные запросы, не превентивно.
- **Неверный порядок в составном индексе.** Работает по префиксу: `(a,b)` помогает фильтру по `a` и `a,b`, но не по одному `b`.
- **Читать с асинхронной реплики там, где нужна свежесть.** Replication lag → устаревшие данные; для read-your-writes читать с мастера.
- **Плохой ключ шардирования.** Перекос (hot shard) и дорогие кросс-шардовые операции; выбирать по кардинальности и паттерну доступа.
- **Кэш как источник истины.** Кэш эфемерен; критичные данные — в durable-хранилище, кэш — ускорение с продуманной инвалидацией.
- **Денормализация/кэш без плана согласованности.** Любая копия данных требует ответа «как поддерживать актуальной».
