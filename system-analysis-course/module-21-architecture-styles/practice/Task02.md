# Задача 02 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Выделите границы сервисов из домена FoodExpress: предложите декомпозицию по bounded context, а не по техническим слоям.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, функциональность: каталог ресторанов и блюд, оформление заказа, оплата, организация доставки (назначение курьеров, трекинг), уведомления, отзывы/рейтинги, профили пользователей.

## ТРЕБОВАНИЯ
- Предложите набор сервисов по бизнес-доменам.
- Для каждого: ответственность, какие данные владеет, ключевые операции.
- Покажите, почему это границы по домену, а не по слоям.
- Отметьте связи между сервисами (через контракты/события, не через общую БД).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сервис | Ответственность | Свои данные | Контракты/связи` + пояснение принципа границ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сервисы выделены по доменам (Каталог, Заказы, Платежи, Доставка, Уведомления, Отзывы, Профили).
- [ ] Для каждого указаны ответственность и собственные данные.
- [ ] Явно сказано, что границы — по бизнес-возможностям, не по слоям.
- [ ] Связи через контракты/события, не через общую БД.
- [ ] Высокая связность внутри, слабая связанность снаружи (показано).
- [ ] Нет «сервиса контроллеров»/«сервиса БД».

## ПОДСКАЗКА
Граница сервиса = автономная бизнес-способность (bounded context, модуль 24). «Заказы», «Платежи», «Доставка» — домены; «слой контроллеров» — нет. Каждый сервис владеет своими данными. Проверка: изменение бизнес-правила домена затрагивает один сервис.
