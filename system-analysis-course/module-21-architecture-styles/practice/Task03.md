# Задача 03 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Обоснуйте, когда монолит лучше микросервисов: на конкретных контекстах покажите, что выбор стиля зависит от ситуации.

## ВХОДНЫЕ ДАННЫЕ
Контексты:
1. Стартап из 3 разработчиков, MVP, гипотезы меняются еженедельно.
2. Внутренний инструмент на 50 пользователей, стабильные требования.
3. Крупный маркетплейс, 200 инженеров, неравномерная нагрузка по доменам.
4. Банковская система с жёсткими транзакционными требованиями в одном домене.

## ТРЕБОВАНИЯ
- Для каждого контекста выберите стиль (монолит / модульный монолит / микросервисы).
- Обоснуйте через факторы: размер команды, скорость изменений, нагрузка, транзакции, операционная зрелость.
- Сформулируйте правило «когда монолит лучше».
- Отметьте, что выбор можно менять со временем (эволюция).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Контекст | Стиль | Обоснование` + правило выбора + замечание об эволюции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Стартап/MVP → монолит (скорость, мало людей, меняющиеся гипотезы).
- [ ] Внутренний инструмент → монолит (простота, малый масштаб).
- [ ] Маркетплейс (200 инженеров, неравномерная нагрузка) → микросервисы (автономия команд, точечный масштаб).
- [ ] Банк с транзакциями в домене → монолит/модульный (ACID проще в одной БД).
- [ ] Сформулировано правило «когда монолит лучше».
- [ ] Отмечена возможность эволюции стиля.

## ПОДСКАЗКА
Монолит лучше при: малой команде, быстро меняющихся требованиях, небольшом/среднем масштабе, сильной потребности в ACID-транзакциях, низкой операционной зрелости. Микросервисы оправданы при большой организации (много автономных команд) и реальной потребности в независимом масштабе/деплое. Конвейской закон: архитектура отражает структуру команд.
