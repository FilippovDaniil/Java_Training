# Задача 04 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Распознайте «распределённый монолит» и исправьте: найдите признаки антипаттерна в описании системы и предложите лечение.

## ВХОДНЫЕ ДАННЫЕ
Система «микросервисов» FoodExpress с симптомами:
- сервисы Заказы, Платежи, Доставка читают и пишут в **одну общую БД**;
- любой релиз требует деплоить все три сервиса вместе (общие версии);
- оформление заказа = синхронная цепочка Заказы→Платежи→Доставка→Уведомления, при отказе любого падает весь запрос;
- изменение схемы таблицы заказов ломает Платежи и Доставку;
- общая библиотека с моделями, обновляется одновременно у всех.

## ТРЕБОВАНИЯ
- Назовите признаки распределённого монолита в этом описании.
- Объясните, почему это «худшее из двух миров».
- Предложите лечение каждого симптома (database per service, async, версионирование контрактов).
- Укажите альтернативу: когда честнее вернуться к модульному монолиту.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Симптом | Почему плохо | Лечение` + вывод о выборе (исправить границы или вернуться к монолиту).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Общая БД распознана как ключевой признак → database per service.
- [ ] Совместный деплой → независимое версионирование/деплой.
- [ ] Синхронная цепочка с каскадными отказами → асинхронность/события, таймауты, circuit breaker.
- [ ] Жёсткая связь через схему/общую библиотеку → устойчивые контракты, версионирование.
- [ ] Объяснено «худшее из двух миров» (цена микросервисов без автономии).
- [ ] Предложена альтернатива возврата к модульному монолиту.

## ПОДСКАЗКА
Распределённый монолит = сервисы, которые нельзя менять/деплоить независимо. Маркеры: общая БД, совместные релизы, синхронные цепочки, общие модели. Лечение: database per service, асинхронные события вместо цепочек, версионированные контракты. Если автономии всё равно нет — модульный монолит даст те же границы дешевле.
