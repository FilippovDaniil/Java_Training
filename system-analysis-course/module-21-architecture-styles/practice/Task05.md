# Задача 05 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Декомпозируйте монолит на сервисы: предложите план поэтапного выделения сервисов из растущего монолита MedConnect.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** — монолит, который перестал справляться: видеоконсультации создают пиковую нагрузку, команда выросла, релизы медленные. Домены внутри: профили, расписание/записи, видеоконсультации, рецепты, оплаты, уведомления. Видеоконсультации — узкое место по нагрузке и независимому масштабу.

## ТРЕБОВАНИЯ
- Предложите порядок выделения сервисов (что вырезать первым и почему).
- Обоснуйте кандидата №1 на выделение (узкое место/независимый масштаб).
- Опишите стратегию миграции (strangler fig: постепенно, не «большой взрыв»).
- Учтите данные: как разделять БД (database per service), что с транзакциями.
- Отметьте 152-ФЗ для медданных при разделении.

## ФОРМАТ РЕЗУЛЬТАТА
План декомпозиции (очередь сервисов + обоснование) + стратегия миграции + работа с данными.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Первым выделяется узкое место (видеоконсультации — независимый масштаб).
- [ ] Обоснован порядок выделения (по нагрузке/автономии/связности).
- [ ] Применён strangler fig (постепенная замена, не «большой взрыв»).
- [ ] Разделение данных: database per service, разбор транзакций (saga/eventual где нужно).
- [ ] Отмечена регуляторика 152-ФЗ для медданных.
- [ ] Указано, что не всё нужно выделять (часть остаётся монолитом).

## ПОДСКАЗКА
Не «большой взрыв», а strangler fig: оборачиваем монолит, постепенно вырезаем сервисы, начиная с того, что даёт наибольшую выгоду (узкое место по нагрузке — видеоконсультации). Каждый выделенный сервис получает свои данные; общие транзакции заменяются saga (модуль 25). Часть доменов может остаться в монолите.
