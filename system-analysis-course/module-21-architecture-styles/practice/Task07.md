# Задача 07 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Спроектируйте модульный монолит как промежуточный шаг: организуйте монолит FoodExpress в модули с границами, готовые к будущему выделению в сервисы.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** — монолит без внутренней структуры («большой ком»). Домены: Каталог, Заказы, Платежи, Доставка, Уведомления. Нужно навести порядок, не переходя пока на микросервисы.

## ТРЕБОВАНИЯ
- Предложите модульную структуру (модуль на домен) с явными границами.
- Опишите правила взаимодействия модулей (через интерфейсы/API модуля, не через чужие таблицы напрямую).
- Покажите, как организовать данные (схемы/неймспейсы на модуль внутри одной БД).
- Объясните, почему такая структура облегчает будущее выделение сервисов.
- Отметьте сохранённые плюсы монолита (общие транзакции, один деплой).

## ФОРМАТ РЕЗУЛЬТАТА
Модульная структура (модули + границы + правила взаимодействия + организация данных) + пояснение готовности к выделению.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Модуль на бизнес-домен с явными границами.
- [ ] Взаимодействие через интерфейсы модуля, не через прямой доступ к чужим таблицам.
- [ ] Данные разделены логически (схемы/неймспейсы на модуль).
- [ ] Объяснено, почему модули с границами легко выделить в сервисы позже.
- [ ] Сохранены плюсы монолита (ACID, один деплой, простота).
- [ ] Подчёркнуто: это осознанный промежуточный шаг.

## ПОДСКАЗКА
Модульный монолит = «микросервисы по дисциплине, монолит по деплою». Каждый модуль владеет своими данными (логически) и предоставляет интерфейс; другие модули не лезут в его таблицы. Это даёт чистые границы (как у сервисов) при простоте монолита. Когда домен потребует независимого масштаба — он уже готов к вырезанию.
