# Задача 08 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Назначьте данные сервисам по принципу database per service: распределите таблицы по сервисам и разрешите конфликты владения.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервисы: Каталог, Заказы, Платежи, Доставка, Профили. Таблицы: client, restaurant, dish, order, order_item, payment, delivery, courier, review. Проблема: заказу нужны данные клиента и блюд (которыми владеют другие сервисы).

## ТРЕБОВАНИЯ
- Назначьте каждую таблицу одному сервису-владельцу.
- Решите, как сервис Заказы получает нужные данные клиента/блюд (не общая БД): копия ключевых полей, запрос по API, событие.
- Разберите проблему ссылочной целостности между сервисами (нет FK через границы).
- Покажите, что считать «снимком на момент» (название/цена блюда в заказе).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Таблица | Сервис-владелец` + решение по доступу к чужим данным + замечание о целостности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каждая таблица назначена одному владельцу (Каталог: restaurant/dish; Заказы: order/order_item; Платежи: payment; Доставка: delivery/courier; Профили: client; Отзывы: review).
- [ ] Заказы не читают чужие таблицы напрямую — через API/события/копию.
- [ ] Цена/название блюда в заказе — снимок на момент (денормализация, осознанная).
- [ ] Разобрана ссылочная целостность через границы (нет межсервисных FK → проверки/контракты).
- [ ] Нет общей БД между сервисами.
- [ ] client_id в заказе — ссылка-значение, не FK на чужую таблицу.

## ПОДСКАЗКА
Database per service: у каждого сервиса своя БД, никаких межсервисных FK. Заказу нужны имя клиента/цена блюда → хранит снимок на момент заказа (это и правильнее для истории) либо запрашивает по API. Целостность через границы обеспечивается контрактами и проверками, а не внешними ключами. Связь с модулем 19 (денормализация под чтение).
