# Задача 09 — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Найдите антипаттерны декомпозиции в предложенной нарезке сервисов и предложите корректную.

## ВХОДНЫЕ ДАННЫЕ
Предложенная «нарезка» FoodExpress на сервисы:
1. `UI-сервис`, `Контроллеры-сервис`, `БД-сервис` (по слоям).
2. `Сервис-создания-заказа`, `Сервис-чтения-заказа`, `Сервис-отмены-заказа` (по операциям над одной сущностью).
3. Все сервисы пишут в общую таблицу `order`.
4. На каждый показ заказа — синхронная цепочка из 5 сервисов.
5. `Сервис-валидации-email` (нано-сервис на одну функцию).

## ТРЕБОВАНИЯ
- Для каждого пункта назовите антипаттерн.
- Объясните, чем он вредит.
- Предложите корректную декомпозицию (по доменам, с владением данными).
- Сформулируйте принципы хорошей границы.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Нарезка | Антипаттерн | Вред | Как правильно` + принципы границ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Нарезка по слоям (1) → антипаттерн; правильно по доменам.
- [ ] Дробление одной сущности по операциям (2) → искусственные границы; объединить в сервис Заказы.
- [ ] Общая таблица order (3) → общая БД; database per service.
- [ ] Синхронная цепочка из 5 (4) → каскадные отказы/латентность; async/агрегация/пересмотр границ.
- [ ] Нано-сервис валидации email (5) → чрезмерное дробление; в библиотеку/модуль.
- [ ] Сформулированы принципы границ (связность, владение данными, устойчивый контракт).

## ПОДСКАЗКА
Антипаттерны: по слоям, по CRUD-операциям над одной сущностью, общая БД, длинные синхронные цепочки, нано-сервисы. Правильно: границы по бизнес-доменам, сервис владеет своими данными, высокая связность внутри, слабая связанность снаружи, общение через устойчивые контракты/события.
