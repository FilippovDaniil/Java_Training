# Задача 10 (мини-проект) — Модуль 21: Архитектурные стили

## ЗАДАНИЕ
Составьте карту сервисов FoodExpress с границами, владением данными, контрактами и обоснованием выбранного стиля. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** вырос: несколько команд, неравномерная нагрузка (каталог и трекинг доставки — пики), требование независимого деплоя ряда доменов. Домены: Каталог, Заказы, Платежи, Доставка, Уведомления, Отзывы, Профили. Часть доменов стабильна и проста, часть — высоконагружена.

## ТРЕБОВАНИЯ
Сдайте архитектурную карту:
1. **Выбор стиля** — обоснование (почему микросервисы/гибрид, а не чистый монолит) под контекст.
2. **Карта сервисов** — для каждого: ответственность, свои данные, ключевые операции.
3. **Границы** — по bounded context; высокая связность/слабая связанность; что осталось в монолите/модуле.
4. **Контракты и связи** — как сервисы общаются (sync API / async события); где async и почему.
5. **Данные** — database per service; разбор «снимков на момент» и межсервисной целостности.
6. **Антипаттерны** — явно покажите, что НЕ сделали (общая БД, распределённый монолит, дробление).
7. **Операционная цена и риски** — что нужно вокруг (CI/CD, observability, saga для распределённых транзакций — модуль 25).

## ФОРМАТ РЕЗУЛЬТАТА
Карта сервисов (таблица/диаграмма) + обоснование стиля + контракты + работа с данными + раздел «избегнутые антипаттерны» + операционные риски.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбор стиля обоснован контекстом (команды, нагрузка, независимый деплой).
- [ ] Карта сервисов: ответственность + свои данные + операции.
- [ ] Границы по доменам; показаны связность/связанность; что осталось монолитом.
- [ ] Контракты: где sync, где async (события) и почему.
- [ ] Database per service; снимки на момент; межсервисная целостность через контракты.
- [ ] Явно перечислены избегнутые антипаттерны.
- [ ] Названа операционная цена и связь с saga/устойчивостью (модуль 25).

## ПОДСКАЗКА
Синтез: сравнение стилей (зад.01), границы (02), выбор под контекст (03), распределённый монолит (04), декомпозиция (05), цена (06), модульность (07), данные (08), антипаттерны (09). Главные проверки middle+: стиль под контекст (не «по моде»), границы по домену, database per service, осознанные sync/async, честный учёт операционной цены и распределённых транзакций. Стыкуется с DDD (24), распределёнкой (25), интеграциями (27), NFR-арх (26).
