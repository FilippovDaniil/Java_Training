## Ключ (эталонные ответы)

1. Монолит, модульный монолит, SOA, микросервисы (спектр от простоты к гибкости/масштабу).
2. **Монолит** — всё приложение в одном кодовом и деплой-юните. Плюсы: простота разработки/деплоя/отладки, ACID-транзакции в одной БД, нет сетевых вызовов, быстрый старт.
3. Сложно масштабировать по частям, один большой рискованный деплой, при росте без границ — «большой ком грязи», технологический lock-in.
4. **Модульный монолит** — единый деплой, но код разбит на модули с явными границами по доменам. Даёт чистые границы при простоте монолита; облегчает будущее выделение сервисов.
5. **SOA** — архитектура из крупных переиспользуемых сервисов, исторически через ESB с централизованной оркестрацией.
6. **Микросервисы** — много мелких автономных сервисов: владеют своими данными, деплоятся независимо, общаются по сети, могут иметь свой стек.
7. Дают независимый деплой/масштаб, автономию команд, изоляцию отказов, технологическую свободу. Ценой: сетевая латентность, распределённые сбои, отсутствие общих ACID-транзакций, сложность наблюдаемости/отладки и эксплуатации.
8. Принцип «у каждого сервиса своя БД»: сервисы не делят таблицы, что обеспечивает независимость схемы и развёртывания.
9. По **бизнес-доменам (bounded context)** — автономным бизнес-возможностям, а не по техническим слоям.
10. Тогда один бизнес-запрос дёргает все «слоевые» сервисы (UI/контроллеры/БД), границы не совпадают с изменениями — теряется автономия и растёт связанность.
11. **Связность** — насколько элементы внутри модуля логически едины (хотим высокую). **Связанность** — насколько модули зависят друг от друга (хотим низкую).
12. Высокая связность внутри, слабая связанность снаружи; сервис владеет своими данными; изменение бизнес-правила домена затрагивает один сервис; устойчивый контракт на границе.
13. **Распределённый монолит** — сервисы, которые нельзя менять/деплоить независимо (общая БД, совместные релизы, синхронные цепочки): операционная цена микросервисов без их автономии.
14. Создаёт скрытую связанность: нельзя независимо менять схему, сервисы влияют друг на друга через данные — теряется главный смысл микросервисов.
15. Латентность складывается, и отказ/замедление любого звена каскадно роняет весь запрос; усложняется отладка. Лечится async/агрегацией/пересмотром границ.
16. Слишком мелкие сервисы, где сетевое общение и оркестрация дороже самой бизнес-логики; усложнение без выгоды.
17. При малой команде, быстро меняющихся требованиях, небольшом/среднем масштабе, сильной потребности в ACID, низкой операционной зрелости.
18. CI/CD на каждый сервис, observability (логи/метрики/трейсы), оркестрация (K8s), обработка распределённых сбоев, версионирование контрактов, интеграционное тестирование.
19. **Закон Конвея**: структура системы повторяет структуру коммуникаций организации. Архитектура и границы сервисов тяготеют к структуре команд.
20. **Strangler fig** — постепенная замена монолита: оборачиваем его, поэтапно вырезаем функциональность в сервисы, не делая рискованный «большой взрыв».
21. Нет распределённых ACID — используют **saga** (последовательность локальных транзакций с компенсациями), eventual consistency, идемпотентность (модуль 25).
22. Сохраняет простоту монолита (один деплой, общие транзакции), но даёт чистые границы; модули с хорошими границами легко выделяются в сервисы при реальной необходимости.
23. Не через общую БД, а через API-запрос к владельцу, подписку на его события, или хранение копии/снимка нужных полей.
24. Копия данных (название, цена), зафиксированная на момент операции (например, цена блюда в заказе): не меняется задним числом, что и правильно для истории, и убирает межсервисную зависимость.
25. Нарезка по слоям; дробление одной сущности по CRUD-операциям; общая БД; длинные синхронные цепочки; нано-сервисы (чрезмерное дробление); распределённый монолит.
