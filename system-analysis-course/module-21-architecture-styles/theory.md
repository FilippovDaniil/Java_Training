# Модуль 21 — Архитектурные стили

## Зачем этот модуль

Части 1–4 закрыли требования, моделирование процессов/поведения, структуру и данные. Часть 5 — **архитектура**: как система устроена крупными блоками. Первый вопрос архитектуры — **стиль**: монолит, модульный монолит, SOA, микросервисы. Это не «что круче», а **выбор компромиссов** под контекст: команда, нагрузка, скорость изменений, операционная зрелость.

Аналитик участвует в этом выборе: помогает выделить границы по бизнес-домену, фиксирует требования, влияющие на стиль, и предостерегает от антипаттернов (распределённый монолит, общая БД). На собеседовании middle+ обязательно: trade-offs стилей, как проводить границы сервисов, чем плох распределённый монолит, когда монолит лучше микросервисов.

---

## 1. Спектр стилей

| Стиль | Суть | Развёртывание |
|-------|------|---------------|
| **Монолит** | одно приложение, один деплой | единый артефакт |
| **Модульный монолит** | один деплой, но чёткие модули с границами | единый артефакт, внутренняя модульность |
| **SOA** | крупные сервисы, часто общая шина (ESB) | несколько сервисов |
| **Микросервисы** | много мелких автономных сервисов | независимые деплои |

```
Монолит  →  Модульный монолит  →  (SOA)  →  Микросервисы
простота ────────────────────────────────► гибкость/масштаб
                                            ▲ операционная сложность растёт
```

> Это не «эволюция к лучшему». Каждый стиль — точка компромисса. Многие успешные системы остаются монолитом/модульным монолитом. Микросервисы платят операционной сложностью за независимость.

---

## 2. Монолит

**Монолит** — всё приложение в одном кодовом и деплой-юните.

| Плюсы | Минусы |
|-------|--------|
| просто разрабатывать/деплоить/отлаживать | сложно масштабировать по частям |
| транзакции в одной БД (ACID) | один большой деплой = риск |
| нет сетевых вызовов между модулями | рост → «большой ком грязи» (big ball of mud) |
| быстрый старт (стартапы, MVP) | технологический lock-in (один стек) |

> Монолит — правильный дефолт для старта и многих систем. Проблема не в монолите как таковом, а в **отсутствии внутренних границ** (модульности) при росте.

---

## 3. Модульный монолит

**Модульный монолит** — единый деплой, но код разбит на **модули с явными границами** (по бизнес-доменам), которые общаются через определённые интерфейсы, а не лезут друг другу в внутренности.

> Часто лучший промежуточный шаг: сохраняет простоту монолита (один деплой, общие транзакции), но даёт чёткие границы. Если позже понадобятся микросервисы — модули с хорошими границами выделяются в сервисы относительно легко. Рекомендуемая стратегия: «начни с модульного монолита, вырезай сервисы по необходимости».

---

## 4. SOA и микросервисы

- **SOA (Service-Oriented Architecture)** — система из крупных переиспользуемых сервисов, исторически связанных через ESB (enterprise service bus) с централизованной логикой/оркестрацией.
- **Микросервисы** — много мелких автономных сервисов, каждый: владеет своими данными (**database per service**), деплоится независимо, общается по сети (REST/gRPC/события), может иметь свой стек.

| Критерий | Микросервисы дают | Ценой |
|----------|-------------------|-------|
| Независимый деплой/масштаб | каждый сервис отдельно | сетевые вызовы, латентность |
| Автономия команд | владеют сервисом целиком | согласование контрактов |
| Изоляция отказов | падение одного ≠ падение всего | распределённые сбои, сложность отладки |
| Технологическая свобода | свой стек на сервис | разнородность, дублирование |
| Масштабирование | точечное (узкие места) | нет распределённых ACID-транзакций (нужны saga) |

> Микросервисы решают проблемы **масштаба и организации** (много команд, огромная нагрузка), но добавляют распределённую сложность: сеть ненадёжна, нет общих транзакций (модуль 25 — saga, идемпотентность), сложнее наблюдаемость и отладка.

---

## 5. Границы сервисов — по bounded context

Главный вопрос декомпозиции: **где резать**. Правильный критерий — **бизнес-домен (bounded context из DDD, модуль 24)**, а не технический слой.

```
✅ По домену:        Заказы | Каталог | Доставка | Платежи | Уведомления
❌ По слоям:         «сервис контроллеров» | «сервис БД» | «сервис UI»
```

Признаки хорошей границы:
- высокая **связность (cohesion)** внутри, слабая **связанность (coupling)** снаружи;
- сервис владеет своими данными (не делит таблицы);
- изменение бизнес-правила домена затрагивает один сервис;
- у границы — устойчивый контракт (API/события).

> Резать по бизнес-возможностям, а не по техническим слоям. Каждый сервис = автономная бизнес-способность со своими данными. Связь с DDD (модуль 24): bounded context ≈ кандидат в сервис.

## 6. Антипаттерны декомпозиции

| Антипаттерн | Суть | Чем плох |
|-------------|------|----------|
| **Распределённый монолит** | сервисы, но жёстко связаны (деплоятся вместе, синхронные цепочки) | сложность микросервисов без их плюсов |
| **Общая БД** | несколько сервисов пишут в одни таблицы | скрытая связанность, нельзя менять схему независимо |
| **Чрезмерное дробление** | слишком мелкие сервисы («нано») | сеть и оркестрация дороже бизнес-логики |
| **Цепочки синхронных вызовов** | A→B→C→D на каждый запрос | латентность множится, каскадные отказы |
| **Анемичные сервисы по слоям** | сервис на каждый слой | один запрос дёргает все сервисы |

> **Распределённый монолит** — худшее из обоих миров: операционная цена микросервисов без автономии. Если для изменения одной фичи надо менять и деплоить несколько сервисов вместе — границы проведены неверно. Лечится возвратом к модульному монолиту или перепроведением границ.

---

## ⚠️ Подводные камни

- **Микросервисы как «по умолчанию».** Для старта/малой команды это преждевременная сложность; начинай с (модульного) монолита.
- **Резать по техническим слоям.** Границы — по бизнес-доменам (bounded context), иначе один запрос дёргает все сервисы.
- **Общая БД между сервисами.** Скрытая связанность убивает независимость; database per service.
- **Распределённый монолит.** Сервисы, которые нельзя деплоить/менять независимо — антипаттерн; признак неверных границ.
- **Игнорировать операционную цену.** Микросервисы требуют CI/CD, observability, оркестрации, обработки распределённых сбоев (модули 25, 26, 36).
- **Забыть про распределённые транзакции.** Нет общих ACID между сервисами — нужны saga/eventual consistency (модули 25).
- **Дробить слишком мелко.** Если общение дороже работы — сервис слишком мал.
