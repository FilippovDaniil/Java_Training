# Задача 02 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Выберите толстый или тонкий клиент для набора сценариев и обоснуйте через требования.

## ВХОДНЫЕ ДАННЫЕ
Сценарии:
1. Приложение курьера, работающее в зонах со слабой связью (нужен офлайн).
2. Админ-панель оператора FoodExpress (всегда онлайн, частые обновления функций).
3. Калькулятор стоимости заказа с мгновенным пересчётом при изменении состава.
4. Приложение для медицинских показаний с критичными расчётами доз (MedConnect).

## ТРЕБОВАНИЯ
- Для каждого сценария выберите толстый/тонкий клиент.
- Обоснуйте через требования (офлайн, отзывчивость, обновляемость, безопасность).
- Для сценария 4 подчеркните: критичные расчёты нельзя оставлять только на клиенте.
- Сформулируйте правило выбора.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Клиент | Обоснование` + правило + замечание о серверной валидации.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Курьер с офлайн → толстый клиент (локальная логика/данные).
- [ ] Админ-панель → тонкий (простое обновление, всегда онлайн).
- [ ] Калькулятор → толще (мгновенный пересчёт на клиенте), но итог проверяется сервером.
- [ ] Мед-расчёты доз → логика/проверка на сервере (клиент можно обойти).
- [ ] Сформулировано правило выбора.
- [ ] Подчёркнуто дублирование критичной валидации на сервере.

## ПОДСКАЗКА
Толстый клиент: офлайн, отзывчивость, разгрузка сервера — ценой обновляемости и безопасности. Тонкий: простое обновление, контроль, тонкие устройства — ценой зависимости от сети. Критичные правила (деньги, дозы) считаются/проверяются на сервере независимо от клиента — клиент недоверенная среда.
