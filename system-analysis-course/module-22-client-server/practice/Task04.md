# Задача 04 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Спроектируйте BFF (Backend for Frontend) для мобильного и веб-клиента: покажите, зачем разные backend под разные фронтенды.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** на микросервисах (Каталог, Заказы, Доставка, Профили). Два клиента: мобильное приложение (узкий экран, экономия трафика, нужен компактный набор данных) и веб (большой экран, богатая карточка, больше данных за раз). Сейчас один общий API неудобен обоим: мобайл получает лишнее, веб делает много запросов.

## ТРЕБОВАНИЯ
- Предложите по BFF на каждый клиент (Mobile-BFF, Web-BFF).
- Покажите, как BFF агрегирует данные нескольких сервисов под нужды своего клиента.
- Сравните: было (общий API) vs стало (BFF) — число запросов, объём данных.
- Отметьте риск дублирования логики между BFF и как его ограничить.

## ФОРМАТ РЕЗУЛЬТАТА
Схема с двумя BFF + примеры агрегации под каждый клиент + сравнение до/после + замечание о дублировании.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Mobile-BFF и Web-BFF как отдельные backend под клиентов.
- [ ] BFF агрегирует ответы нескольких сервисов в один под нужды клиента.
- [ ] Mobile-BFF отдаёт компактные данные, Web-BFF — расширенные.
- [ ] Сравнение «общий API vs BFF» (overfetching/число запросов).
- [ ] Отмечен риск дублирования бизнес-логики (BFF — про агрегацию/подгонку, не про правила).
- [ ] Бизнес-правила остаются в сервисах, не в BFF.

## ПОДСКАЗКА
BFF = backend, заточенный под конкретный фронтенд: агрегирует вызовы сервисов и подгоняет форму ответа (мобайлу — меньше и компактнее, вебу — больше за раз). Решает проблему «один универсальный API неудобен всем» (overfetching/underfetching, модуль 31). Риск: бизнес-логику в BFF не тащить — она в сервисах; BFF только агрегация/адаптация.
