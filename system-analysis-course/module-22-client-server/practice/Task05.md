# Задача 05 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Опишите роль API Gateway: какие сквозные задачи он решает и чем отличается от BFF.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** на микросервисах с несколькими типами клиентов (веб, мобайл, партнёрские интеграции). Нужна единая точка входа: аутентификация, маршрутизация, ограничение частоты запросов, TLS, скрытие внутренней структуры сервисов.

## ТРЕБОВАНИЯ
- Перечислите функции API Gateway (auth, routing, rate limiting, TLS-терминация, агрегация, логирование).
- Покажите, как Gateway скрывает внутреннюю топологию сервисов от клиентов.
- Объясните разницу Gateway vs BFF (общая точка/сквозные задачи vs подгонка под клиента).
- Отметьте антипаттерн «бизнес-логика в Gateway».

## ФОРМАТ РЕЗУЛЬТАТА
Список функций Gateway + схема «клиенты → Gateway → сервисы» + сравнение Gateway vs BFF + замечание об антипаттерне.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Перечислены сквозные функции Gateway (auth, routing, rate limiting, TLS, логирование/агрегация).
- [ ] Показано скрытие внутренней структуры сервисов за единой точкой входа.
- [ ] Объяснена разница Gateway (сквозные задачи для всех) vs BFF (адаптация под конкретный клиент).
- [ ] Отмечено: бизнес-логика не место в Gateway.
- [ ] Указано, что Gateway и BFF могут сосуществовать.

## ПОДСКАЗКА
API Gateway — единая входная дверь: маршрутизация, аутентификация, лимиты, TLS, скрытие топологии. BFF — про подгонку API под конкретный клиент. Они не взаимоисключающи: клиенты → Gateway (сквозное) → BFF/сервисы. Антипаттерн: складывать бизнес-правила в Gateway (он инфраструктурный, не доменный).
