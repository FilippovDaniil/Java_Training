# Задача 06 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Найдите нарушения слоистости в описании системы и предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
Описание реализации FoodExpress с проблемами:
1. JS на странице оформления напрямую формирует SQL-запрос и шлёт его на сервер.
2. Контроллер HTTP содержит расчёт скидок и проверку бизнес-правил.
3. Репозиторий заказов внутри метода сохранения отправляет email клиенту.
4. UI-компонент меню напрямую читает таблицу `dish` из БД.
5. Бизнес-сервис формирует HTML-разметку ответа.

## ТРЕБОВАНИЯ
- Для каждого пункта назовите нарушение слоистости.
- Объясните, чем это вредит (заменяемость, тестируемость, безопасность).
- Предложите, в какой слой перенести логику.
- Сформулируйте правило направления зависимостей.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Нарушение | Чем вредит | Куда перенести` + правило слоистости.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) UI формирует SQL → presentation лезет в data: вынести в data access слой/репозиторий; UI шлёт намерение, не SQL.
- [ ] (2) бизнес-логика в контроллере → перенести в бизнес-слой.
- [ ] (3) отправка email из репозитория → побочный эффект не в data access (в бизнес/сервис уведомлений).
- [ ] (4) UI читает БД напрямую → через бизнес/data слой.
- [ ] (5) HTML в бизнес-сервисе → представление в presentation.
- [ ] Сформулировано правило: зависимости сверху вниз, слой не делает чужую работу.

## ПОДСКАЗКА
Нарушение слоистости — когда слой делает не свою работу или зависимости идут «не туда». UI не формирует SQL и не читает БД; бизнес-логика — не в контроллере и не в репозитории; представление (HTML) — не в бизнес-слое; побочные эффекты (email) — не в data access. Зависимости: presentation → business → data, не наоборот.
