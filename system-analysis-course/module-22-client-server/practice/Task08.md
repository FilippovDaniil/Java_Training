# Задача 08 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Спроектируйте клиент/сервер для офлайн-режима: как приложение работает без связи и синхронизируется при восстановлении.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, приложение выездного врача: работает в районах без устойчивой связи. Врач должен: смотреть список пациентов на сегодня, вносить осмотры/назначения офлайн, синхронизировать данные при появлении сети. Возможны конфликты (данные изменились и на клиенте, и на сервере).

## ТРЕБОВАНИЯ
- Выберите тип клиента (толстый, с локальным хранилищем) и обоснуйте.
- Опишите офлайн-работу: локальное кэширование данных дня, локальная запись изменений (очередь).
- Опишите синхронизацию при восстановлении связи (push локальных изменений, pull обновлений).
- Разберите разрешение конфликтов (last-write-wins / версии / ручное), что выбрать для медданных.
- Учтите 152-ФЗ: безопасность локально хранимых медданных.

## ФОРМАТ РЕЗУЛЬТАТА
Архитектура офлайн-клиента + поток синхронизации + стратегия конфликтов + замечание о безопасности данных.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбран толстый клиент с локальным хранилищем (офлайн).
- [ ] Описано локальное кэширование данных дня и локальная очередь изменений.
- [ ] Синхронизация: push локальных изменений + pull обновлений при появлении сети.
- [ ] Разобрано разрешение конфликтов; для медданных — осторожная стратегия (версии/ручное, не слепой LWW).
- [ ] Отмечена безопасность локальных медданных (шифрование, 152-ФЗ).
- [ ] Сервер остаётся источником истины после синхронизации.

## ПОДСКАЗКА
Офлайн → толстый клиент с локальной БД и очередью отложенных операций. Синхронизация: при восстановлении связи отправить накопленные изменения и забрать чужие. Конфликты: last-write-wins прост, но опасен для медданных (можно потерять важное) — лучше версии/ручное разрешение. Локальные медданные шифровать (152-ФЗ). Источник истины — сервер.
