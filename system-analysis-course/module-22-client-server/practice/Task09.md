# Задача 09 — Модуль 22: Клиент-серверная архитектура

## ЗАДАНИЕ
Сравните SSR и SPA с точки зрения архитектуры: для набора страниц выберите подход и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, типы страниц:
1. Публичная страница ресторана (важен SEO, быстрый первый показ, индексация).
2. Личный кабинет с интерактивным управлением заказами (много взаимодействия после входа).
3. Главная-лендинг с акциями (SEO, скорость, слабые мобильные устройства).
4. Интерактивная карта трекинга курьера в реальном времени.

## ТРЕБОВАНИЯ
- Для каждой страницы выберите SSR / SPA / гибрид.
- Обоснуйте через: SEO, скорость первого показа, интерактивность, нагрузка на сервер/клиент.
- Покажите, как выбор смещает логику (на сервер при SSR, на клиент при SPA).
- Сформулируйте правило выбора.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Страница | SSR/SPA/гибрид | Обоснование` + правило выбора.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Публичная страница ресторана → SSR (SEO, быстрый первый показ).
- [ ] Личный кабинет → SPA (интерактивность после входа, SEO не нужен).
- [ ] Лендинг с акциями → SSR/статика (SEO, скорость, слабые устройства).
- [ ] Трекинг в реальном времени → SPA (динамика, обновления на клиенте).
- [ ] Объяснено смещение логики (сервер при SSR / клиент при SPA).
- [ ] Сформулировано правило выбора (SEO+первый показ → SSR; интерактивность+за логином → SPA).

## ПОДСКАЗКА
SSR рендерит HTML на сервере (быстрый первый показ, SEO, тонкий клиент) — для публичных/контентных страниц. SPA собирает UI в браузере (отзывчивость, динамика, толстый клиент) — для интерактивных экранов за логином. Архитектурно: SSR грузит сервер, SPA — клиент. Гибриды (Next.js) комбинируют под страницу.
