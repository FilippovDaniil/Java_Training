## Ключ (эталонные ответы)

1. Модель, где **клиент** инициирует запросы, а **сервер** их обрабатывает и отвечает (часто обслуживая многих клиентов).
2. **Слой (layer)** — логическое разделение ответственности (организация кода). **Звено (tier)** — физическое размещение/развёртывание (на каких машинах). Layer — про код, tier — про развёртывание.
3. Presentation (представление/UI), business/application (бизнес-логика), data access (доступ к данным).
4. **1-tier** — всё на одной машине; **2-tier** — клиент + сервер БД; **3-tier** — клиент + сервер приложений + сервер БД.
5. Разделяет представление, логику и данные по звеньям: независимое масштабирование, безопасность (БД скрыта за app-сервером), управляемость.
6. Да: логические слои не требуют физического разделения. Монолит на одной машине имеет все 3 слоя в 1 звене.
7. **Толстый клиент** — много логики и данных на стороне клиента (desktop, тяжёлый SPA): отзывчивость, офлайн, разгрузка сервера.
8. **Тонкий клиент** — минимум логики, основное на сервере (браузер с SSR, терминал): простое обновление, тонкие требования к устройству.
9. Плюсы: отзывчивость, офлайн, разгрузка сервера. Минусы: сложное обновление/дистрибуция, дублирование логики, безопасность (логика доступна клиенту).
10. Клиент — недоверенная среда: его код можно изменить/обойти. Критичные проверки и правила (деньги, дозы, права) должны выполняться/проверяться на сервере.
11. UI, отображение, ввод/вывод. Не должен содержать бизнес-правил.
12. Бизнес-логика, правила, расчёты, оркестрация. Не должен знать про конкретный UI или детали SQL.
13. Работа с хранилищем: запросы, маппинг данных. Не должен содержать бизнес-правил.
14. Сверху вниз: presentation → business → data access (не наоборот).
15. Когда слой делает чужую работу или зависимость идёт «не туда»: например, UI напрямую формирует SQL/читает БД мимо бизнес-слоя; бизнес-логика в контроллере; HTML в бизнес-сервисе.
16. **API Gateway** — единая точка входа для клиентов: маршрутизация к сервисам, аутентификация, rate limiting, TLS-терминация, скрытие внутренней топологии.
17. **BFF (Backend for Frontend)** — отдельный backend под конкретный тип клиента: агрегирует данные нескольких сервисов и подгоняет ответ под нужды именно этого фронтенда.
18. **Gateway** — общая входная точка и сквозные задачи (auth/routing/limits) для всех. **BFF** — адаптация API под конкретный клиент. Могут сосуществовать (клиент → Gateway → BFF/сервисы).
19. Разным клиентам нужны разные данные/агрегация: мобайл получает лишнее (overfetching) или делает много запросов (underfetching). BFF решает это, подгоняя API под клиента.
20. **N-tier** — 3-tier плюс инфраструктурные звенья: балансировщик, кэш, read replicas, очередь сообщений, CDN — под масштаб и доступность.
21. **SSR** — рендеринг HTML на сервере (быстрый первый показ, SEO, тонкий клиент). **SPA** — сборка UI в браузере (отзывчивость, динамика, толстый клиент).
22. **SSR** — публичные/контентные страницы, где важны SEO и быстрый первый показ. **SPA** — интерактивные экраны (за логином), где важна динамика. Гибриды комбинируют.
23. Толстый клиент с локальным хранилищем: кэшировать данные, копить изменения в локальной очереди, синхронизировать при восстановлении связи (push/pull) с разрешением конфликтов.
24. В 2-tier клиент ходит напрямую в БД — она доступна снаружи, выше риск. В 3-tier БД скрыта за сервером приложений, доступ контролируется.
25. Gateway — инфраструктурный компонент для сквозных задач; бизнес-правила в нём размывают границы и дублируют логику сервисов. Логика должна жить в доменных сервисах.
