# Модуль 22 — Клиент-серверная архитектура

## Зачем этот модуль

Стиль выбран (модуль 21). Теперь — **как организовано взаимодействие клиента и сервера** и как разложена система на слои/звенья. Клиент-серверная модель лежит под почти всеми системами: веб, мобильные, desktop. Аналитик должен понимать звенья (tiers), слои (presentation/business/data), толстый/тонкий клиент, роли API Gateway и BFF — чтобы проектировать компоненты, разграничивать ответственность и формулировать требования к развёртыванию.

На собеседовании middle+ обязательно: чем tier отличается от layer, что такое 3-tier, толстый vs тонкий клиент, зачем нужны API Gateway и BFF, SSR vs SPA с точки зрения архитектуры.

---

## 1. Клиент-сервер: базовая модель

**Клиент** запрашивает — **сервер** обрабатывает и отвечает. Клиент инициирует, сервер обслуживает (часто многих клиентов).

```
[Клиент] ──запрос──► [Сервер] ──► [Хранилище]
         ◄─ответ────
```

Варианты: веб-браузер ↔ веб-сервер, мобильное приложение ↔ API, desktop ↔ БД. Ключевые вопросы: **где выполняется логика** (на клиенте или сервере) и **как система разбита на части**.

---

## 2. Звенья (tiers) vs слои (layers)

Это **разные** понятия, их часто путают:

| Понятие | Что это | Измерение |
|---------|---------|-----------|
| **Слой (layer)** | логическое разделение ответственности | **логическое** (как организован код) |
| **Звено (tier)** | физическое размещение/развёртывание | **физическое** (на каких машинах) |

- **Слои:** presentation (представление) → business/logic (бизнес-логика) → data access (доступ к данным).
- **Звенья:** 1-tier (всё на одной машине), 2-tier (клиент + сервер БД), 3-tier (клиент + сервер приложений + сервер БД), N-tier.

```
Слои (логика):              Звенья (физика, 3-tier):
┌─ Presentation ─┐          [Клиент] ── [Сервер приложений] ── [Сервер БД]
├─ Business ─────┤            tier 1         tier 2              tier 3
└─ Data access ──┘
```

> Можно иметь 3 слоя в 1 звене (монолит на одной машине) или те же 3 слоя в 3 звеньях. **Layer — про код, tier — про развёртывание.** Это любимый вопрос на собеседовании.

---

## 3. Архитектура по звеньям

| Звенья | Состав | Когда |
|--------|--------|-------|
| **1-tier** | всё (UI+логика+данные) на одной машине | desktop с локальной БД |
| **2-tier** | клиент (UI+часть логики) ↔ сервер БД | классические клиент-серверные приложения |
| **3-tier** | клиент ↔ сервер приложений ↔ сервер БД | веб-приложения (дефолт) |
| **N-tier** | + доп. звенья (кэш, очередь, gateway, балансировщик) | масштабируемые системы |

> **3-tier** — стандарт для веба: тонкий клиент (браузер), сервер приложений (логика), сервер БД (данные). Разделение звеньев даёт независимое масштабирование и безопасность (БД не торчит наружу).

---

## 4. Толстый vs тонкий клиент

| | Толстый (fat/rich) клиент | Тонкий (thin) клиент |
|--|---------------------------|----------------------|
| Логика | много на клиенте | минимум, основное на сервере |
| Пример | desktop-приложение, тяжёлый SPA | браузер с серверным рендерингом, терминал |
| Плюсы | отзывчивость, работа офлайн, разгрузка сервера | простое обновление (всё на сервере), тонкие требования к устройству, безопаснее |
| Минусы | сложное обновление/дистрибуция, дублирование логики, безопасность (логика у клиента) | зависимость от сети, нагрузка на сервер |

> Выбор зависит от требований: офлайн и отзывчивость → толще клиент; простота обновления и контроль → тоньше. Важно: **критичную бизнес-логику и проверки нельзя оставлять только на клиенте** (его можно обойти) — валидация дублируется на сервере.

---

## 5. Слои приложения (детально)

| Слой | Ответственность | Не должен |
|------|-----------------|-----------|
| **Presentation** | UI, отображение, ввод | содержать бизнес-правила |
| **Business/Application** | бизнес-логика, правила, оркестрация | знать про конкретный UI или SQL |
| **Data access** | работа с хранилищем (запросы, маппинг) | содержать бизнес-правила |

Правило: зависимости идут **сверху вниз** (presentation → business → data), не наоборот. Нарушение слоистости — когда, например, UI напрямую дёргает SQL, минуя бизнес-слой.

> Чёткие слои = заменяемость (сменить UI или БД, не трогая логику) и тестируемость. Связь с гексагональной архитектурой/DDD (модуль 24): бизнес-ядро не зависит от деталей (UI, БД).

---

## 6. API Gateway и BFF

В системах с несколькими сервисами/клиентами появляются посредники:

- **API Gateway** — единая точка входа для клиентов: маршрутизация к сервисам, аутентификация, rate limiting, агрегация, TLS-терминация. Скрывает внутреннюю структуру сервисов.
- **BFF (Backend for Frontend)** — отдельный backend под конкретный тип клиента (один для веба, другой для мобильного): агрегирует и подгоняет данные под нужды именно этого фронтенда.

```
[Web]    ─┐                      ┌─► Сервис Заказы
[Mobile] ─┤─► API Gateway ──────►├─► Сервис Каталог
[Partner]─┘     (auth, routing)  └─► Сервис Платежи

BFF: [Mobile] ─► Mobile-BFF ─► (несколько сервисов, ответ под мобильный экран)
     [Web]    ─► Web-BFF    ─► (те же сервисы, ответ под веб)
```

> **Gateway** — общая точка входа и сквозные задачи (auth, лимиты). **BFF** — подгонка API под конкретный клиент (мобильному нужно меньше данных и другая агрегация, чем вебу). BFF убирает «один универсальный API, неудобный всем».

## 7. SSR vs SPA (архитектурный взгляд)

| | SSR (server-side rendering) | SPA (single-page application) |
|--|------------------------------|-------------------------------|
| Где рендер | на сервере (готовый HTML) | в браузере (JS собирает UI) |
| Клиент | тоньше | толще |
| Плюсы | быстрый первый показ, SEO, слабые устройства | отзывчивость после загрузки, меньше нагрузки на сервер при навигации |
| Минусы | нагрузка на сервер на каждый рендер | тяжёлый первый старт, SEO-сложности, логика на клиенте |

> С точки зрения архитектуры: SSR смещает работу на сервер (тонкий клиент), SPA — на клиент (толстый). Гибриды (Next.js и пр.) комбинируют. Выбор влияет на нагрузку, SEO, отзывчивость и распределение логики.

---

## ⚠️ Подводные камни

- **Путать tier и layer.** Layer — логическое разделение кода; tier — физическое размещение. 3 слоя ≠ 3 звена.
- **Бизнес-логика только на клиенте.** Толстый клиент можно обойти — критичные проверки/правила обязаны быть на сервере.
- **Нарушение слоистости.** UI, лезущий напрямую в БД мимо бизнес-слоя, ломает заменяемость и тестируемость.
- **Один универсальный API для всех клиентов.** Мобайлу и вебу нужны разные данные/агрегация — отсюда BFF.
- **Gateway как «свалка логики».** Gateway — для сквозных задач (auth, маршрутизация, лимиты), не для бизнес-логики (она в сервисах).
- **БД «торчит» наружу.** В 2-tier клиент ходит прямо в БД — риск безопасности; 3-tier прячет данные за сервером приложений.
- **SPA по умолчанию.** Для контентных/SEO-сайтов SSR часто уместнее; выбор под требования, не по моде.
