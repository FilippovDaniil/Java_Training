# Задача 03 — Модуль 23: Сети

## ЗАДАНИЕ
Подберите HTTP-статусы под сценарии: для каждого случая выберите корректный код и обоснуйте, отличив похожие.

## ВХОДНЫЕ ДАННЫЕ
Сценарии API FoodExpress:
1. Заказ успешно создан.
2. Заказ обновлён, тела в ответе нет.
3. Запрошен заказ с несуществующим id.
4. Тело запроса — невалидный JSON (синтаксис).
5. JSON валиден, но нарушено бизнес-правило (qty = 0).
6. Пользователь не передал токен.
7. Токен есть, но нет прав на чужой заказ.
8. Клиент шлёт слишком много запросов.
9. Конфликт: попытка отменить уже доставленный заказ.
10. Внутренняя ошибка сервера.

## ТРЕБОВАНИЯ
- Для каждого сценария выберите статус-код.
- Отличите похожие: 401 vs 403, 400 vs 422, 404 vs 204, 409.
- Кратко обоснуйте выбор.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Статус | Обоснование`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Создан → 201; обновлён без тела → 204.
- [ ] Несуществующий id → 404 (не 200/204).
- [ ] Невалидный JSON → 400; нарушение бизнес-правила → 422.
- [ ] Нет токена → 401; нет прав → 403.
- [ ] Слишком много запросов → 429; конфликт статуса → 409.
- [ ] Ошибка сервера → 500.

## ПОДСКАЗКА
Ключевые различения: 401 (не аутентифицирован) vs 403 (аутентифицирован, но нет прав); 400 (синтаксис/формат) vs 422 (валиден, но бизнес-правило); 404 (ресурса нет) vs 204 (есть, но тела нет); 409 (конфликт состояния, напр. недопустимый переход статуса). Эти коды — часть контракта REST (модуль 29).
