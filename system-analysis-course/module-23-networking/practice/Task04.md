# Задача 04 — Модуль 23: Сети

## ЗАДАНИЕ
Определите идемпотентность HTTP-методов и решите, как сделать создание заказа безопасным при ретраях.

## ВХОДНЫЕ ДАННЫЕ
Операции FoodExpress и их методы:
1. `GET /orders/1024` — получить заказ.
2. `PUT /orders/1024` — полностью заменить заказ.
3. `DELETE /orders/1024` — удалить заказ.
4. `POST /orders` — создать заказ.
5. `PATCH /orders/1024` — сменить статус.
Проблема: при сетевом сбое клиент не получил ответ и повторяет запрос.

## ТРЕБОВАНИЯ
- Для каждого метода укажите, идемпотентен ли он, и что произойдёт при повторе.
- Объясните, почему POST создаёт проблему дублей при ретрае.
- Предложите механизм идемпотентности для POST (idempotency key).
- Отметьте безопасность (safe) методов (не меняют состояние).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Метод | Идемпотентен? | Что при повторе` + решение для POST.

## КРИТЕРИИ ПРИЁМКИ
- [ ] GET — идемпотентен и безопасен.
- [ ] PUT, DELETE — идемпотентны (повтор → то же состояние).
- [ ] POST — не идемпотентен (повтор → дубль заказа).
- [ ] PATCH — обычно не идемпотентен (зависит от семантики).
- [ ] Предложен idempotency key для POST (сервер хранит ключ → повтор не создаёт дубль).
- [ ] Отмечены safe-методы (GET).

## ПОДСКАЗКА
Идемпотентность = повтор даёт тот же результат. GET/PUT/DELETE идемпотентны (можно безопасно повторять при сбое). POST — нет (каждый раз новый ресурс). Решение: клиент шлёт `Idempotency-Key`, сервер запоминает его и при повторе возвращает прежний результат, не создавая дубль (модуль 25).
