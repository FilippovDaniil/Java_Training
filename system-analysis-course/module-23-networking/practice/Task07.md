# Задача 07 — Модуль 23: Сети

## ЗАДАНИЕ
Выберите стратегию балансировки нагрузки под сценарии и разберите L4 vs L7.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сценарии балансировки:
1. Серверы приложений одинаковой мощности, запросы независимы (stateless).
2. Серверы хранят сессию пользователя в памяти (stateful).
3. Серверы разной мощности (старые и новые).
4. Нужно маршрутизировать `/api/v2/*` на новые серверы, остальное — на старые.
5. Долгие запросы создают перекос (одни серверы заняты дольше).

## ТРЕБОВАНИЯ
- Для каждого сценария выберите стратегию (round-robin / least connections / sticky / weighted) или уровень (L4/L7).
- Обоснуйте выбор.
- Объясните разницу L4 (по IP/порт) и L7 (по содержимому HTTP).
- Отметьте, почему stateless-серверы предпочтительнее (модуль 26) и роль health checks.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Стратегия/уровень | Обоснование` + пояснение L4 vs L7.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Stateless равные серверы → round-robin.
- [ ] Stateful сессии → sticky (или вынести сессии вовне, чтобы stateless).
- [ ] Разная мощность → weighted.
- [ ] Маршрутизация по пути `/api/v2/*` → L7 (по содержимому).
- [ ] Долгие запросы/перекос → least connections.
- [ ] Объяснены L4 vs L7 и роль health checks; отмечено преимущество stateless.

## ПОДСКАЗКА
Round-robin — равномерно по очереди (равные stateless серверы). Least connections — на наименее загруженный (долгие запросы). Weighted — с учётом мощности. Sticky — привязка к серверу (нужно при stateful, но лучше stateless + внешние сессии). L4 балансирует по IP/портам (быстро), L7 — по HTTP (путь/заголовки → маршрутизация). Health checks убирают мёртвые серверы.
