# Задача 08 — Модуль 23: Сети

## ЗАДАНИЕ
Спроектируйте HTTP-кэширование для ресурсов: подберите Cache-Control/ETag под типы данных и объясните 304.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, ресурсы:
1. Логотип и статика (CSS/JS), меняются редко.
2. Карточка блюда (меняется несколько раз в день).
3. Текущий статус заказа (меняется часто, важна свежесть).
4. Профиль пользователя (приватный, индивидуальный).

## ТРЕБОВАНИЯ
- Для каждого ресурса предложите политику Cache-Control (max-age, public/private, no-cache).
- Где уместно — используйте ETag и опишите механизм 304 Not Modified.
- Объясните разницу public vs private кэша.
- Отметьте, что нельзя агрессивно кэшировать (статус заказа) и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Ресурс | Cache-Control | ETag? | Обоснование` + описание механизма 304.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Статика → длинный max-age, public (можно версионировать имена файлов).
- [ ] Карточка блюда → умеренный TTL + ETag (валидация изменений).
- [ ] Статус заказа → no-cache/короткий TTL (важна свежесть).
- [ ] Профиль → private (не кэшировать в общих прокси/CDN).
- [ ] Описан механизм ETag + If-None-Match → 304 Not Modified (без тела).
- [ ] Объяснена разница public vs private.

## ПОДСКАЗКА
Cache-Control задаёт политику: `max-age` (сколько свежо), `public/private` (можно ли в общих кэшах/CDN), `no-cache` (всегда проверять). ETag — версия ресурса: клиент шлёт `If-None-Match`, сервер отвечает 304 без тела, если не менялось (экономия трафика). Статика — длинный кэш; статус заказа — почти не кэшировать; приватное — private.
