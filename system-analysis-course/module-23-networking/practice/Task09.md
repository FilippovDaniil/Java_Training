# Задача 09 — Модуль 23: Сети

## ЗАДАНИЕ
Проведите диагностику сетевой проблемы: по симптомам определите вероятный уровень/причину (DNS / TLS / таймаут / балансировщик / приложение).

## ВХОДНЫЕ ДАННЫЕ
Инциденты (FoodExpress):
1. «Не удаётся найти сервер» — имя не резолвится.
2. Браузер: «Сертификат недействителен / истёк».
3. Запрос висит и обрывается через 30 секунд (504 Gateway Timeout).
4. Часть запросов идёт успешно, часть падает с 502 — нестабильно.
5. После переключения на новый сервер часть пользователей всё ещё попадает на старый.
6. 200 OK, но тело ответа с ошибкой бизнес-логики.

## ТРЕБОВАНИЯ
- Для каждого инцидента назовите вероятный уровень/причину.
- Предложите, чем диагностировать (nslookup/dig, проверка сертификата, логи, health checks, TTL).
- Отличите сетевые проблемы от прикладных (инцидент 6).
- Сформулируйте порядок диагностики (снизу вверх или сверху вниз).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Инцидент | Вероятная причина | Как диагностировать` + порядок диагностики.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) имя не резолвится → DNS (nslookup/dig).
- [ ] (2) сертификат → TLS (проверить срок/цепочку CA).
- [ ] (3) 504 → таймаут на gateway/upstream (медленный сервис/нет ответа).
- [ ] (4) часть 502 → нестабильный backend/балансировщик (один из серверов «мёртв», нет health check).
- [ ] (5) часть на старый сервер → DNS TTL/кэш не истёк.
- [ ] (6) 200 с ошибкой в теле → прикладная проблема, не сетевая.
- [ ] Предложен порядок диагностики и инструменты.

## ПОДСКАЗКА
Маркеры: «не найден сервер» → DNS; «сертификат» → TLS; 504 → таймаут upstream; нестабильные 502 → часть backend недоступна (health checks!); «часть на старый IP» → DNS TTL; 200 с ошибкой в теле → приложение, а не сеть. Диагностируйте по слоям: DNS → TLS → соединение/таймаут → балансировщик → приложение.
