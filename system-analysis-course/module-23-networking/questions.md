## Ключ (эталонные ответы)

1. **OSI** — эталонная сетевая модель из 7 уровней (от физического до прикладного), описывающая, как данные передаются по сети.
2. HTTP — прикладной (L7); TLS — уровень представления (поверх TCP); IP — сетевой (L3).
3. **TCP** — с установлением соединения, надёжный, гарантирует порядок (HTTP, БД). **UDP** — без установления, без гарантий доставки/порядка, быстрый (видео, игры, DNS-запросы).
4. Метод, URI, заголовки (headers), тело (body).
5. GET (чтение), POST (создание/действие), PUT (полная замена), PATCH (частичное изменение), DELETE (удаление).
6. **Идемпотентность** — повтор запроса даёт тот же результат, что однократный (состояние не меняется от повторов).
7. Идемпотентны: GET, PUT, DELETE. Не идемпотентны: POST (создаёт новое каждый раз), обычно PATCH.
8. Через **idempotency key**: клиент передаёт уникальный ключ, сервер запоминает его и при повторе возвращает прежний результат, не создавая дубль.
9. 1xx (информационные), 2xx (успех), 3xx (перенаправление), 4xx (ошибка клиента), 5xx (ошибка сервера).
10. **401 Unauthorized** — не аутентифицирован (нет/неверный токен). **403 Forbidden** — аутентифицирован, но нет прав на ресурс.
11. **400 Bad Request** — некорректный синтаксис/формат запроса. **422 Unprocessable Entity** — формат корректен, но нарушено бизнес-правило/валидация.
12. **404 Not Found**. Не 200 с пустым телом, потому что 200 означает успех и наличие ресурса — это вводит клиента в заблуждение.
13. **409 Conflict** — запрос конфликтует с текущим состоянием ресурса (например, недопустимый переход статуса, конкурентное изменение).
14. **HTTPS** — HTTP поверх TLS. TLS даёт шифрование (конфиденциальность), целостность (защита от подмены) и аутентификацию сервера (подлинность по сертификату).
15. Клиент шлёт ClientHello (шифры) → сервер присылает сертификат и ServerHello → клиент проверяет сертификат у доверенного CA → стороны согласуют симметричный сеансовый ключ → дальше обмен шифруется этим ключом.
16. **Сертификат** подтверждает подлинность домена и содержит публичный ключ. **CA (центр сертификации)** — доверенная сторона, выдающая и заверяющая сертификаты, которым доверяют клиенты.
17. **mTLS (взаимный TLS)** — обе стороны предъявляют и проверяют сертификаты (не только сервер, но и клиент); используется для межсервисной аутентификации.
18. Имя резолвится в IP через иерархию: проверка кэша → DNS-резолвер → корневой сервер → TLD → авторитетный сервер домена → возврат IP, кэшируемого по TTL.
19. **TTL** — время жизни DNS-записи в кэше. После смены записи старый IP остаётся в кэшах клиентов до истечения TTL, поэтому переключение не мгновенно (TTL снижают заранее перед миграцией).
20. A (IPv4), AAAA (IPv6), CNAME (псевдоним/алиас), MX (почтовые серверы), TXT (произвольный текст/верификация).
21. Распределяет входящие запросы между несколькими серверами — даёт горизонтальное масштабирование, доступность и отказоустойчивость (через health checks).
22. Round-robin (по очереди), least connections (на наименее загруженный), IP hash/sticky (привязка клиента к серверу), weighted (с учётом мощности).
23. **L4** балансирует по транспортным данным (IP/порты), быстро, не смотрит в HTTP. **L7** — по содержимому HTTP (путь, заголовки), позволяет маршрутизацию и A/B.
24. **ETag** — идентификатор версии ресурса. Клиент шлёт `If-None-Match: <etag>`; если ресурс не изменился, сервер отвечает **304 Not Modified** без тела (экономия трафика).
25. **HTTP-кэш** (браузер, CDN, прокси) работает на уровне HTTP-ответов через Cache-Control/ETag. **Кэш данных** (Redis, cache-aside) — на уровне приложения/БД (модуль 20). Это разные слои с разными механизмами.
