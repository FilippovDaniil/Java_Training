# Модуль 23 — Сети для аналитика

## Зачем этот модуль

Клиент и сервер (модуль 22) общаются **через сеть**. Аналитик не настраивает маршрутизаторы, но обязан понимать транспорт: как устроен HTTP-запрос, какие методы и статусы выбрать, что делает TLS, как работает DNS и балансировка. Это нужно, чтобы проектировать REST API (модуль 29), читать логи/трейсы, формулировать NFR (latency, доступность), диагностировать «почему не работает» и говорить на одном языке с backend/devops.

На собеседовании middle+ обязательно: модель OSI/TCP-IP, методы и статусы HTTP, идемпотентность методов, как работает HTTPS/TLS, DNS-резолюция, стратегии балансировки, HTTP-кэширование.

---

## 1. Модель OSI и TCP/IP

**OSI** — 7 уровней, эталонная модель. На практике чаще мыслят стеком **TCP/IP** (4 уровня). Аналитику важны верхние уровни.

| OSI | Уровень | Что делает | Примеры |
|-----|---------|------------|---------|
| 7 | Прикладной (Application) | протоколы приложений | HTTP, DNS, SMTP, gRPC |
| 6 | Представления (Presentation) | кодирование, шифрование | TLS, форматы |
| 5 | Сеансовый (Session) | сессии | — |
| 4 | Транспортный (Transport) | доставка между процессами | TCP, UDP |
| 3 | Сетевой (Network) | маршрутизация между сетями | IP |
| 2 | Канальный (Data Link) | передача в пределах сети | Ethernet, MAC |
| 1 | Физический (Physical) | биты по среде | кабель, радио |

```
TCP/IP (4 уровня):  Application | Transport (TCP/UDP) | Internet (IP) | Link
```

- **TCP** — надёжный, с установлением соединения, гарантия порядка (HTTP, БД). 
- **UDP** — без гарантий, быстрый (видео, игры, DNS-запросы).

> Аналитику нужно уверенно отличать уровни и знать, где живёт что: HTTP — прикладной, TLS — поверх TCP, IP — маршрутизация. «На каком уровне работает X» — частый вопрос.

---

## 2. HTTP: запрос и ответ

HTTP — текстовый протокол «запрос-ответ» прикладного уровня поверх TCP.

```
Запрос:                          Ответ:
POST /orders HTTP/1.1            HTTP/1.1 201 Created
Host: api.foodexpress.ru        Content-Type: application/json
Authorization: Bearer ...       Location: /orders/1024
Content-Type: application/json
                                {"id":1024,"status":"created"}
{"clientId":1,"items":[...]}
```

Составляющие: **метод**, **URI**, **заголовки** (headers), **тело** (body), в ответе — **статус-код**.

---

## 3. HTTP-методы и идемпотентность

| Метод | Назначение | Идемпотентный | Безопасный* |
|-------|-----------|---------------|-------------|
| **GET** | чтение | да | да |
| **POST** | создание/действие | **нет** | нет |
| **PUT** | полная замена | да | нет |
| **PATCH** | частичное изменение | обычно нет | нет |
| **DELETE** | удаление | да | нет |

\* Безопасный = не меняет состояние.

- **Идемпотентность** — повтор запроса даёт тот же результат, что и однократный. `PUT /orders/1 {...}` дважды → то же состояние. `POST /orders` дважды → два заказа (не идемпотентно).

> Идемпотентность критична для надёжности: при сетевом сбое/ретрае идемпотентные методы (GET/PUT/DELETE) можно безопасно повторять. Для неидемпотентного POST применяют **idempotency key** (модуль 25), чтобы повтор не создал дубль.

---

## 4. HTTP-статусы

| Класс | Значение | Примеры |
|-------|----------|---------|
| **1xx** | информационные | 100 Continue |
| **2xx** | успех | 200 OK, 201 Created, 204 No Content |
| **3xx** | перенаправление | 301 Moved, 304 Not Modified |
| **4xx** | ошибка клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable, 429 Too Many Requests |
| **5xx** | ошибка сервера | 500 Internal, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout |

> Частые ошибки выбора: 401 (не аутентифицирован) vs 403 (нет прав); 400 (синтаксис) vs 422 (валидация бизнес-правил); 404 (нет ресурса) vs 204 (есть, но тела нет). Ресурс не найден — **404**, не 200 с пустым телом. Эти различения — типовой вопрос (и закреплены в REST, модуль 29).

## 5. HTTPS и TLS

**HTTPS = HTTP поверх TLS**. TLS обеспечивает: **шифрование** (никто не прочитает), **целостность** (не подменят), **аутентификацию сервера** (сертификат подтверждает, что это тот сервер).

Упрощённое TLS-рукопожатие (handshake):
```
1. Клиент → ClientHello (поддерживаемые шифры)
2. Сервер → сертификат (с публичным ключом) + ServerHello
3. Клиент проверяет сертификат у доверенного CA
4. Согласование сеансового (симметричного) ключа
5. Дальше — шифрованный обмен симметричным ключом
```

- **Сертификат** выдаётся центром сертификации (CA), подтверждает подлинность домена.
- **mTLS** (взаимный TLS) — обе стороны предъявляют сертификаты (межсервисная аутентификация, модуль 34).

> Аналитику нужно понимать словами: TLS даёт конфиденциальность + целостность + подлинность сервера; асимметричное шифрование используется для обмена симметричным ключом, дальше — симметричное (быстрее). Истёкший/невалидный сертификат → ошибка соединения.

## 6. DNS

**DNS** переводит доменное имя в IP-адрес.

```
api.foodexpress.ru → резолвер → корневой → .ru → foodexpress.ru → IP 203.0.113.10
```

Шаги (упрощённо): браузер/ОС проверяет кэш → запрос к DNS-резолверу → иерархия серверов (root → TLD → авторитетный) → возврат IP → кэш по TTL.

Типы записей: **A** (IPv4), **AAAA** (IPv6), **CNAME** (псевдоним), **MX** (почта), **TXT**.

> DNS кэшируется по TTL — изменение записи распространяется не мгновенно (важно при миграциях/переключении). DNS также используется для балансировки (несколько A-записей, geo-DNS).

---

## 7. Балансировка нагрузки

**Балансировщик (load balancer)** распределяет запросы между несколькими серверами.

| Стратегия | Суть |
|-----------|------|
| **Round-robin** | по очереди на каждый сервер |
| **Least connections** | на наименее загруженный |
| **IP hash / sticky** | привязка клиента к серверу (сессии) |
| **Weighted** | с учётом мощности серверов |

- **L4 (транспортный)** балансировщик — по IP/портам (быстро, не смотрит в HTTP).
- **L7 (прикладной)** — по содержимому HTTP (путь, заголовки) — маршрутизация, A/B.

> Балансировка даёт горизонтальный масштаб и доступность (упал один сервер — трафик на другие). Sticky-сессии нужны, если сервер хранит состояние клиента (но лучше делать серверы stateless — модуль 26). Health checks убирают «мёртвые» серверы из ротации.

---

## 8. HTTP-кэширование

Кэширование на уровне HTTP уменьшает трафик и нагрузку:
- **Cache-Control** (`max-age`, `no-cache`, `private/public`) — политика кэширования.
- **ETag** — версия ресурса; клиент шлёт `If-None-Match` → сервер отвечает **304 Not Modified** без тела, если не изменилось.
- **Last-Modified / If-Modified-Since** — по времени изменения.

> HTTP-кэш (браузер, CDN, прокси) — отдельный слой от кэша БД (модуль 20). ETag/304 экономят трафик: ресурс не пересылается, если не менялся. Для статики — длинный max-age; для динамики — аккуратно (no-cache/короткий TTL).

---

## ⚠️ Подводные камни

- **Путать уровни.** HTTP — прикладной (L7), TLS — поверх TCP, IP — сетевой (L3). «На каком уровне» спрашивают часто.
- **Неверный статус-код.** 401≠403, 400≠422, 404≠«200 с пустым телом». Корректные коды — часть контракта (модуль 29).
- **POST как идемпотентный.** POST создаёт каждый раз новое; для безопасного повтора — idempotency key (модуль 25).
- **Игнорировать DNS TTL.** После смены записи старый IP живёт до истечения TTL — учитывать при переключениях.
- **Sticky-сессии вместо stateless.** Привязка к серверу мешает масштабированию и отказоустойчивости; лучше stateless + внешнее хранилище сессий.
- **HTTP вместо HTTPS для чувствительных данных.** Без TLS трафик читается/подменяется. Чувствительное — только по HTTPS.
- **Путать HTTP-кэш и кэш данных.** Это разные слои; ETag/Cache-Control — про HTTP, Redis/cache-aside — про данные.
