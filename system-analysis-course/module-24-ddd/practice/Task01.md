# Задача 01 — Модуль 24: DDD

## ЗАДАНИЕ
Выделите bounded context'ы из домена FoodExpress: определите границы, внутри которых модель и язык согласованы.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, функциональность: оформление и обработка заказов, каталог ресторанов и меню, оплаты и биллинг, организация доставки (курьеры, трекинг), поддержка клиентов, аналитика/отчётность, уведомления.

## ТРЕБОВАНИЯ
- Выделите bounded context'ы (Заказы, Каталог, Биллинг, Доставка, Поддержка, …).
- Для каждого опишите, что входит в его модель и язык.
- Покажите, что одно понятие (например, «Клиент») в разных контекстах моделируется по-разному.
- Объясните, почему не делать «одну модель на всё».

## ФОРМАТ РЕЗУЛЬТАТА
Список контекстов с кратким описанием модели + пример разной трактовки «Клиента» по контекстам.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выделены осмысленные bounded context'ы по бизнес-областям.
- [ ] Для каждого описана своя модель/язык.
- [ ] Показано, что «Клиент» в Заказах ≠ Клиент в Биллинге ≠ в Поддержке.
- [ ] Объяснён вред «универсальной модели на всё».
- [ ] Контексты соответствуют автономным областям (кандидаты в сервисы).
- [ ] Высокая связность внутри контекста.

## ПОДСКАЗКА
Bounded context = граница, где термины однозначны и модель согласована. «Клиент» в Заказах (адрес, история), в Биллинге (плательщик, баланс), в Поддержке (обращающийся) — три разные модели одного слова, и это правильно. Универсальная модель «Клиент на всё» раздувается и не подходит никому.
