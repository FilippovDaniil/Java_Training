# Задача 02 — Модуль 24: DDD

## ЗАДАНИЕ
Постройте context map для FoodExpress: определите отношения между bounded context'ами.

## ВХОДНЫЕ ДАННЫЕ
Контексты из задачи 01: Заказы, Каталог, Биллинг, Доставка, Поддержка. Известно: Заказы зависят от Каталога (берут блюда/цены) и инициируют оплату в Биллинге; Доставка получает заказы от Заказов; Биллинг — это интеграция со старой системой (legacy) с неудобной моделью; Каталог отдаёт данные многим потребителям.

## ТРЕБОВАНИЯ
- Определите тип отношения между парами контекстов (Customer-Supplier, Conformist, ACL, Shared Kernel, Open Host Service, Partnership).
- Обоснуйте каждый выбор.
- Для интеграции с legacy-биллингом примените ACL и объясните зачем.
- Для Каталога как поставщика многим — Open Host Service.

## ФОРМАТ РЕЗУЛЬТАТА
Context map (текст/диаграмма) + таблица `Пара контекстов | Тип отношения | Обоснование`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Заказы↔Доставка — Customer-Supplier (upstream/downstream).
- [ ] Заказы↔Каталог — потребление (Customer-Supplier или Conformist).
- [ ] Каталог→многие — Open Host Service.
- [ ] Заказы/Биллинг(legacy) — ACL (защита от чужой модели).
- [ ] Каждое отношение обосновано.
- [ ] Объяснена роль ACL при интеграции с legacy.

## ПОДСКАЗКА
Context map = отношения между контекстами. Customer-Supplier: upstream поставляет downstream. Conformist: downstream принимает чужую модель как есть. ACL: downstream защищается слоем-переводчиком (критично для legacy/внешних — не пускать их модель внутрь). Open Host Service: контекст публикует общий API для многих потребителей.
