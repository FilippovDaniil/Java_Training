# Задача 08 — Модуль 24: DDD

## ЗАДАНИЕ
Найдите ошибки выделения контекстов в предложенной декомпозиции и предложите корректную.

## ВХОДНЫЕ ДАННЫЕ
Предложенное деление FoodExpress на bounded context'ы с проблемами:
1. Единый контекст «Данные» — содержит ВСЕ сущности системы (одна модель на всё).
2. Контекст «Клиент» — пытается объединить клиента для заказов, биллинга и поддержки в одну модель.
3. Контексты «Создание», «Чтение», «Обновление» (по операциям, не по домену).
4. Контекст «Утилиты» — мешанина из авторизации, нотификаций и логирования.
5. Заказы и Доставка слиты в один контекст, хотя у них разные команды и язык.

## ТРЕБОВАНИЯ
- Для каждого пункта назовите ошибку.
- Объясните, чем она вредит (раздутая модель, ложные границы, низкая связность).
- Предложите корректное выделение контекстов.
- Сформулируйте признаки хорошего bounded context.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Проблема | Ошибка | Как правильно` + признаки хорошего контекста.

## КРИТЕРИИ ПРИЁМКИ
- [ ] «Данные на всё» → одна модель на всё; разделить по доменам.
- [ ] Универсальный «Клиент» → своя модель Клиента в каждом контексте.
- [ ] Деление по CRUD-операциям → ложные границы; делить по домену.
- [ ] «Утилиты» (мешанина) → разнести generic-поддомены/выделить по назначению.
- [ ] Слияние Заказы+Доставка с разным языком/командами → разделить.
- [ ] Сформулированы признаки хорошего контекста (единый язык, связность, автономия).

## ПОДСКАЗКА
Ошибки: одна модель на всё; универсальная сущность; границы по техническим операциям (CRUD); «свалка» из несвязанного; слияние областей с разным языком. Хороший контекст: единый язык внутри, высокая связность, автономия, соответствие бизнес-области и (часто) команде. Делить по домену, не по операциям/технике.
