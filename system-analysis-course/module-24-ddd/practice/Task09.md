# Задача 09 — Модуль 24: DDD

## ЗАДАНИЕ
Свяжите bounded context'ы с микросервисами: покажите, как стратегический DDD задаёт границы сервисов, и где границы НЕ совпадают 1:1.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, bounded context'ы (из задач 01–02): Заказы, Каталог, Биллинг, Доставка, Поддержка, Уведомления (generic). Стоит вопрос: делать ли сервис на каждый контекст, и что с данными.

## ТРЕБОВАНИЯ
- Сопоставьте контексты с сервисами (где 1:1, где иначе).
- Объясните, почему bounded context — естественный кандидат в сервис.
- Покажите, что database per service вытекает из границ контекста.
- Отметьте случаи, когда контекст ≠ один сервис (мелкий контекст в монолите; крупный контекст из нескольких сервисов).
- Свяжите с транзакциями между сервисами (saga, модуль 25).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Контекст | Сервис(ы) | Свои данные` + пояснение соответствия и исключений + связь с saga.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Большинство контекстов → отдельные сервисы со своими данными.
- [ ] Объяснено, почему bounded context — естественная граница сервиса.
- [ ] Database per service выведено из границ контекста.
- [ ] Отмечены исключения (контекст в монолите; крупный контекст → несколько сервисов).
- [ ] Generic (Уведомления) — возможно готовое решение/общий сервис.
- [ ] Транзакции между контекстами → saga/eventual (модуль 25).

## ПОДСКАЗКА
Bounded context — естественный кандидат в микросервис: автономная модель + свой язык + свои данные (database per service). Но не догма 1:1: мелкий контекст может остаться модулем монолита, очень крупный — разбиться на несколько сервисов. Между контекстами нет общих транзакций — согласованность через saga/события (модуль 25).
