## Ключ (эталонные ответы)

1. **DDD** — подход к проектированию вокруг предметной области и единого языка с бизнесом. Оправдан для **сложных доменов**; для простого CRUD избыточен.
2. **Стратегический** — про границы и контексты (bounded context, context map, поддомены). **Тактический** — про строительные блоки модели (entity, value object, aggregate).
3. Общий словарь терминов домена, на котором говорят бизнес и разработка, отражённый в коде, моделях, документации.
4. Убирает «испорченный телефон» между бизнесом и ИТ: один термин — одно понятие везде, меньше недопонимания и дефектов.
5. **Bounded context** — граница, внутри которой термины единого языка однозначны и модель согласована; за границей те же слова могут значить иное.
6. Универсальная модель раздувается, пытаясь угодить всем контекстам, и не подходит никому; растёт связанность и сложность.
7. Да. «Клиент» в Заказах (адрес, история), в Биллинге (плательщик, баланс), в Поддержке (обращающийся) — разные модели, и это нормально.
8. **Context map** — карта bounded context'ов и отношений между ними (как они интегрируются на уровне моделей).
9. Partnership, Customer-Supplier, Conformist, Anti-Corruption Layer, Shared Kernel, Open Host Service, Published Language.
10. **ACL** — слой-переводчик на границе: не пускает чужую (legacy/внешнюю) модель внутрь, переводя её в термины вашего контекста; защищает чистоту вашей модели.
11. **Customer-Supplier** — upstream поставляет downstream, downstream влияет на приоритеты. **Conformist** — downstream принимает модель upstream как есть, без влияния.
12. **Open Host Service** — контекст публикует общий протокол/API для многих потребителей. **Published Language** — общий формализованный язык обмена (схемы/форматы).
13. **Core** — даёт конкурентное преимущество. **Supporting** — нужен, но не уникален. **Generic** — стандартная функция (авторизация, уведомления, платежи).
14. Core — строить самим лучшими силами; supporting — проще своими силами; generic — купить/взять готовое, не изобретать.
15. Вкладывать ресурсы в generic (изобретать авторизацию/уведомления) вместо концентрации на core.
16. **Entity** — объект с идентичностью, живущей во времени (тот же объект при изменении полей); сравнивается по id.
17. **Value object** — объект, определяемый значениями, без идентичности, неизменяемый; сравнивается по значениям (Деньги, Адрес, Период).
18. Entity имеет идентичность (id) и меняется во времени; VO не имеет идентичности, неизменяем и сравнивается по значениям.
19. **Aggregate** — кластер связанных сущностей/VO как единое целое; **aggregate root** — единственная точка входа, через которую изменяют агрегат и которая держит инварианты.
20. Граница агрегата определяет область согласованности: инварианты держатся внутри, внешние ссылаются только на корень (по id), не на внутренности.
21. Потому что граница агрегата = граница согласованности: одна транзакция атомарно меняет один агрегат; согласованность между агрегатами — eventual/saga.
22. Bounded context — естественный кандидат в микросервис (автономная модель + свой язык + свои данные, database per service); но не обязательно 1:1.
23. Между контекстами/агрегатами нет общих ACID-транзакций — согласованность через saga, события, eventual consistency (модуль 25).
24. Концептуальная модель (модуль 15) — основа: её сущности становятся entity/value object/aggregate внутри bounded context.
25. Одна модель на всё; DDD для простого CRUD; игнор единого языка; интеграция без ACL; огромные агрегаты; изменение внутренностей агрегата мимо корня; путаница core и generic.
