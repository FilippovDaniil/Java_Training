# Модуль 24 — Domain-Driven Design (DDD)

## Зачем этот модуль

В модуле 21 границы сервисов проводились «по bounded context». Что это такое — даёт **DDD (Domain-Driven Design)**: подход к проектированию сложных систем вокруг **предметной области (домена)** и единого языка с бизнесом. DDD делится на **стратегический** (границы, контексты, отношения — для аналитика и архитектора) и **тактический** (entity, value object, aggregate — ближе к коду). Аналитику особенно важен стратегический DDD: он связывает требования, концептуальную модель (модуль 15) и архитектуру (модуль 21).

На собеседовании middle+ обязательно: ubiquitous language, bounded context, context map и виды отношений, агрегат и его границы, entity vs value object, виды поддоменов (core/supporting/generic).

---

## 1. Ubiquitous language (единый язык)

**Единый язык** — общий словарь терминов домена, на котором говорят **и бизнес, и разработка**, и который отражён в коде, моделях, документации.

```
Бизнес говорит «заказ отменён» → в коде Order.cancel(), статус CANCELLED,
в БД order.status='CANCELLED', в API PATCH /orders/{id} {"status":"cancelled"}.
Один термин — везде одинаково.
```

> Единый язык убирает «испорченный телефон» между бизнесом и ИТ. Если аналитик пишет «заявка», разработчик — «тикет», а бизнес — «обращение», и это одно и то же — система обрастёт путаницей. Язык фиксируется в глоссарии (модуль 11) и живёт в каждом bounded context.

---

## 2. Bounded context (ограниченный контекст)

**Bounded context** — граница, внутри которой термины единого языка имеют **однозначный смысл** и модель согласована. За границей те же слова могут значить другое.

```
Контекст «Заказы»:   Клиент = тот, кто оформил заказ (адрес, история заказов)
Контекст «Биллинг»:  Клиент = плательщик (реквизиты, баланс)
Контекст «Поддержка»:Клиент = обращающийся (тикеты, контакты)
```

Одно слово «Клиент» — три разные модели в трёх контекстах. Это нормально: каждый контекст моделирует своё.

> Bounded context — ключевая идея DDD. Попытка сделать «одну модель Клиента на всю систему» (universal model) приводит к раздутой сущности, которая не подходит никому. Лучше — своя модель в каждом контексте. Bounded context ≈ кандидат в микросервис (модуль 21).

---

## 3. Context map (карта контекстов)

**Context map** показывает, как bounded context'ы связаны между собой и какие у них **отношения**:

| Отношение | Суть |
|-----------|------|
| **Partnership** | два контекста развиваются совместно, взаимозависимо |
| **Customer-Supplier** | вышестоящий (upstream) поставляет нижестоящему (downstream); downstream влияет на приоритеты |
| **Conformist** | downstream принимает модель upstream как есть (нет влияния) |
| **Anti-Corruption Layer (ACL)** | downstream защищается слоем-переводчиком от чужой модели |
| **Shared Kernel** | общая часть модели, разделяемая двумя контекстами |
| **Open Host Service** | контекст публикует общий протокол/API для многих |
| **Published Language** | общий формализованный язык обмена (схемы) |

```
[Заказы] ──Customer-Supplier──► [Доставка]
[Заказы] ──ACL──► [Legacy-биллинг]   (защита от старой модели)
[Каталог] ──Open Host Service──► (многие потребители)
```

> Context map — карта интеграции на уровне моделей. **ACL** особенно важен: при интеграции с legacy/внешней системой не пускайте их модель внутрь — переводите на свой язык через слой-адаптер (связь с интеграциями, модуль 27).

---

## 4. Поддомены: core / supporting / generic

Домен делится на **поддомены** по бизнес-ценности:

| Поддомен | Что это | Стратегия |
|----------|---------|-----------|
| **Core (ядро)** | то, что даёт конкурентное преимущество | вкладывать лучшие силы, строить самим |
| **Supporting (поддерживающий)** | нужен, но не уникален | можно проще, своими силами |
| **Generic (общий)** | стандартная функция (авторизация, нотификации) | купить/готовое решение |

```
FoodExpress:  Core = матчинг заказов и логистика доставки (преимущество)
              Supporting = каталог/меню
              Generic = аутентификация, отправка email
```

> Различение поддоменов — про **куда вкладывать ресурсы**. Core строят сами и тщательно; generic (авторизация, платёжный шлюз, email) — берут готовое, не изобретают. Частая ошибка — тратить силы на generic вместо core.

---

## 5. Тактический DDD: entity, value object, aggregate

Внутри контекста модель строится из:

- **Entity (сущность)** — объект с **идентичностью**, живущей во времени (Заказ №1024 — тот же, даже если поля изменились). Сравнивается по id.
- **Value Object (объект-значение)** — определяется **значениями**, без идентичности, неизменяем (Деньги, Адрес, Период). Два адреса с одинаковыми полями — равны.
- **Aggregate (агрегат)** — кластер связанных сущностей/VO, рассматриваемый как единое целое; имеет **корень (aggregate root)** — единственную точку входа, через которую меняют агрегат.

```
Агрегат «Заказ»:  Order (root) ──► OrderItem ──► (Money: VO)
  Менять позиции можно только через Order (root), который держит инварианты
  (например, «сумма = сумме позиций», «минимум 1 позиция»).
```

| | Entity | Value Object |
|--|--------|--------------|
| Идентичность | да (id) | нет |
| Изменяемость | да | нет (заменяется целиком) |
| Равенство | по id | по значениям |
| Пример | Заказ, Клиент | Деньги, Адрес, Координаты |

> **Aggregate root** — граница согласованности: инварианты агрегата держатся внутри него, внешние ссылаются только на корень (по id), не на внутренности. Это напрямую влияет на транзакции (одна транзакция = один агрегат) и на API (операции над корнем).

---

## 6. Связь DDD с архитектурой и моделями

- **Bounded context → микросервис** (модуль 21): естественная граница сервиса.
- **Aggregate → транзакционная граница** (модуль 20): изменения агрегата — в одной транзакции; между агрегатами/контекстами — eventual consistency, saga (модуль 25).
- **Концептуальная модель (модуль 15) → тактический DDD**: сущности домена становятся entity/VO/aggregate.
- **Ubiquitous language → глоссарий и контракты** (модули 11, 28): язык контекста = словарь и схемы обмена.

> DDD — связующее звено Части 5: оно объясняет, **где** проводить границы (контексты → сервисы) и **как** держать согласованность (агрегаты → транзакции). Стратегический DDD — инструмент аналитика для декомпозиции сложного домена.

---

## ⚠️ Подводные камни

- **Одна модель на всю систему.** «Универсальный Клиент» для всех контекстов — раздутая сущность, неудобная всем. У каждого контекста — своя модель.
- **DDD ради DDD.** Тяжёлый аппарат оправдан для **сложного домена**; для CRUD-приложения избыточен.
- **Игнорировать единый язык.** Разные термины для одного понятия у бизнеса и ИТ → дефекты и недопонимание.
- **Пускать чужую модель внутрь.** Интеграция с legacy/внешним без ACL заражает вашу модель их понятиями.
- **Огромные агрегаты.** Слишком большой агрегат → конкурентные конфликты и тяжёлые транзакции; агрегат должен быть минимальным держателем инварианта.
- **Менять внутренности агрегата напрямую.** Только через корень — иначе инварианты не держатся.
- **Путать core и generic.** Силы на generic (авторизация, email) вместо core (конкурентное преимущество).
