# Задача 01 — Модуль 25: Распределённые системы

## ЗАДАНИЕ
Спроектируйте идемпотентную операцию оплаты: защитите от двойного списания при повторах.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, оплата заказа через внешний платёжный сервис. Проблема: клиент нажал «оплатить», ответ не дошёл (таймаут), клиент/приложение повторяет запрос. Без защиты — двойное списание.

## ТРЕБОВАНИЯ
- Спроектируйте операцию оплаты с idempotency key.
- Опишите, что сервер делает при первом запросе и при повторе с тем же ключом.
- Укажите, где хранится ключ и результат, как долго.
- Покажите, как ключ генерируется клиентом и передаётся (заголовок).
- Отметьте связь с гарантиями доставки и retry (задачи 06, 07).

## ФОРМАТ РЕЗУЛЬТАТА
Схема обработки оплаты с idempotency key (первый запрос/повтор) + где хранится ключ + формат передачи.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Клиент генерирует idempotency key и шлёт в заголовке.
- [ ] Первый запрос: ключ не встречался → выполнить списание, сохранить результат под ключом.
- [ ] Повтор с тем же ключом → вернуть сохранённый результат, НЕ списывать повторно.
- [ ] Указано хранение ключа+результата (БД/кэш) и срок хранения.
- [ ] Объяснено, что это защищает от двойного списания при таймауте/ретрае.
- [ ] Отмечена связь с at-least-once и retry.

## ПОДСКАЗКА
Idempotency key: клиент создаёт уникальный ключ операции и шлёт его. Сервер проверяет, обрабатывал ли ключ: нет — выполняет и сохраняет результат под ключом; да — возвращает прежний результат без повторного списания. Ключ+результат хранят в БД/кэше с TTL. Это основа безопасных повторов (модуль 23).
