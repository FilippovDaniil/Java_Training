# Задача 04 — Модуль 25: Распределённые системы

## ЗАДАНИЕ
Спроектируйте Transactional Outbox для надёжной публикации события: решите проблему dual write.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервис Заказы: при создании заказа надо записать заказ в свою БД И опубликовать событие `OrderCreated` в брокер (его ждут Доставка, Уведомления). Проблема: если записать в БД, а публикация события упадёт (или наоборот) — рассинхрон.

## ТРЕБОВАНИЯ
- Объясните проблему dual write (два независимых действия без атомарности).
- Спроектируйте outbox: запись события в таблицу outbox в той же транзакции, что и заказ.
- Опишите relay/процесс, который читает outbox и публикует в брокер.
- Укажите гарантию (at-least-once) и следствие (потребители идемпотентны).
- Отметьте связь с CDC как альтернативой.

## ФОРМАТ РЕЗУЛЬТАТА
Описание проблемы dual write + схема outbox (транзакция + relay) + гарантия доставки + следствие для потребителей.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснена проблема dual write (БД и брокер — два независимых ресурса).
- [ ] Событие пишется в таблицу outbox в ОДНОЙ транзакции с заказом.
- [ ] Отдельный relay читает outbox и публикует в брокер, помечает отправленным.
- [ ] Гарантия at-least-once (событие не потеряется, но возможны дубли).
- [ ] Следствие: потребители должны быть идемпотентны (дедупликация).
- [ ] Упомянут CDC как альтернатива чтению outbox.

## ПОДСКАЗКА
Dual write: «запиши в БД И отправь событие» без атомарности → один из шагов может упасть. Outbox: событие пишется в таблицу outbox в той же транзакции, что бизнес-данные (атомарно). Отдельный процесс (relay) читает outbox и публикует в брокер. Гарантия — at-least-once → возможны дубли → потребитель идемпотентен (модуль 32).
