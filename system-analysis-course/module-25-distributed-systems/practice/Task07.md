# Задача 07 — Модуль 25: Распределённые системы

## ЗАДАНИЕ
Разберите гарантии доставки и их последствия: для сценариев выберите at-least/at-most/exactly-once и спроектируйте дедупликацию.

## ВХОДНЫЕ ДАННЫЕ
Сценарии обмена сообщениями (FoodExpress):
1. Событие «заказ оплачен» → начислить бонусы (нельзя начислить дважды).
2. Метрика «показ баннера» для аналитики (потеря единичных событий некритична).
3. Команда «списать оплату» (нельзя ни потерять, ни выполнить дважды).

## ТРЕБОВАНИЯ
- Для каждого сценария выберите гарантию доставки и обоснуйте.
- Где at-least-once — спроектируйте дедупликацию (по id события/ключу).
- Объясните, почему exactly-once дёшево недостижим и как его эмулируют.
- Свяжите гарантии с идемпотентностью потребителя.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Гарантия | Обоснование | Дедупликация` + пояснение про exactly-once.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Начисление бонусов → at-least-once + дедупликация (по id события) = эффект exactly-once.
- [ ] Метрика показа → at-most-once допустимо (потеря некритична).
- [ ] Списание оплаты → at-least-once + идемпотентность (нельзя потерять и нельзя дважды).
- [ ] Спроектирована дедупликация по id/ключу события.
- [ ] Объяснено, что exactly-once строго недостижим дёшево; эмуляция = at-least-once + идемпотентность.
- [ ] Связь с идемпотентностью потребителя.

## ПОДСКАЗКА
At-most-once: можно потерять (метрики). At-least-once: возможны дубли (норма брокеров) → дедупликация по id события на потребителе. Exactly-once дёшево недостижим — его эмулируют at-least-once + идемпотентный обработчик (дубль обрабатывается как один). «Нельзя потерять и нельзя дважды» = at-least-once + дедуп/идемпотентность.
