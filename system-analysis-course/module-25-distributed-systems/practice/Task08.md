# Задача 08 — Модуль 25: Распределённые системы

## ЗАДАНИЕ
Спроектируйте отражение eventual consistency в UX: что показывать пользователю, пока данные «сходятся».

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сценарии с временным рассинхроном из-за распределённости:
1. Пользователь оформил заказ, но статус «оплачен» появится через 1–2 секунды (асинхронная обработка платежа).
2. Курьер обновил геопозицию, но клиент видит её с задержкой.
3. Пользователь оставил отзыв — он появится в общей ленте не сразу.
4. Бонусы за заказ начисляются асинхронно после доставки.

## ТРЕБОВАНИЯ
- Для каждого сценария предложите UX-приём на время «схождения» данных (оптимистичный UI, «обрабатывается», read-your-writes, индикатор).
- Объясните, где допустима задержка, а где нужен read-your-writes (пользователь видит своё сразу).
- Покажите, как не вводить пользователя в заблуждение (не показывать «оплачено», пока не подтверждено).
- Свяжите с моделями согласованности и репликацией (модули 19, 20).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | UX-приём | Обоснование` + правило: где RYW, где допустима задержка.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Статус оплаты → промежуточное состояние «обрабатывается», не «оплачено» раньше подтверждения.
- [ ] Геопозиция → допустима задержка + индикатор «обновлено N сек назад».
- [ ] Свой отзыв → read-your-writes (автор видит сразу, остальные — позже).
- [ ] Бонусы → «начислятся после доставки» (понятное ожидание).
- [ ] Объяснено, где RYW, где допустима задержка.
- [ ] Подчёркнуто: не показывать неподтверждённое как факт.

## ПОДСКАЗКА
Eventual consistency требует честного UX: показывать промежуточные состояния («обрабатывается»), а не выдавать неподтверждённое за факт (нельзя «оплачено», пока платёж не подтверждён). Read-your-writes: автор видит своё действие сразу (свой отзыв/заказ), даже если для других оно появится позже. Индикаторы свежести для задержек. Связь с модулями 19/20.
