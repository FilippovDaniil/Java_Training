# Задача 09 — Модуль 25: Распределённые системы

## ЗАДАНИЕ
Найдите точки распределённого сбоя в сценарии и предложите паттерны устойчивости для каждой.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, сценарий видеоконсультации (синхронные вызовы между сервисами и внешними системами):
- Запись → сервис Платежи (внешний провайдер) → сервис Видеосвязь (внешний провайдер) → сервис Рецепты → ГИС рецептов (государственная, нестабильная) → Уведомления.
Каждый внешний вызов может зависнуть, упасть, ответить с задержкой или продублироваться.

## ТРЕБОВАНИЯ
- Найдите точки распределённого сбоя (каждый сетевой/внешний вызов).
- Для каждой предложите паттерн устойчивости (timeout, retry+backoff, circuit breaker, fallback, idempotency, DLQ).
- Отметьте, где синхронный вызов лучше заменить асинхронным (ГИС нестабильна).
- Покажите, что произойдёт без защиты (каскадный отказ, зависание, дубли).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Точка сбоя | Риск | Паттерн устойчивости` + рекомендация sync→async где уместно.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каждый внешний/сетевой вызов отмечен как точка сбоя.
- [ ] Платёжный провайдер: timeout + retry(идемпотентный) + circuit breaker.
- [ ] Видеосвязь: timeout + fallback (переподключение/перенос).
- [ ] ГИС рецептов (нестабильна): асинхронно + retry + DLQ + circuit breaker.
- [ ] Уведомления: async, at-least-once + дедуп.
- [ ] Показаны последствия без защиты (каскад, зависание, дубли).
- [ ] Учтена регуляторика (рецепты/медданные).

## ПОДСКАЗКА
Каждый внешний вызов — точка сбоя: ставим timeout (не ждать вечно), retry+backoff (для временных, с идемпотентностью), circuit breaker (не добивать мёртвый сервис), fallback (запасной путь). Нестабильную ГИС лучше дёргать асинхронно (очередь + DLQ + ретраи), а не держать на ней синхронный пользовательский запрос. Без защиты — каскадные отказы и зависания.
