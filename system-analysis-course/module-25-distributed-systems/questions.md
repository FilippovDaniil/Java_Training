## Ключ (эталонные ответы)

1. Нет единого commit/rollback: операция охватывает несколько сервисов с отдельными БД, между шагами сеть может оборваться, узлы — упасть.
2. Сервисы владеют своими БД (database per service), общего транзакционного менеджера нет; распределённые транзакции (2PC) дороги, медленны и снижают доступность.
3. Реплики/сервисы приходят к согласованному состоянию «со временем». Норма, потому что строгая согласованность между сервисами дорога и снижает доступность (CAP).
4. **Идемпотентность** — повтор операции даёт тот же результат, что однократное выполнение (состояние не меняется от повторов).
5. Уникальный ключ операции от клиента; сервер запоминает обработанные ключи и при повторе возвращает прежний результат, не выполняя операцию заново.
6. Потому что повторы неизбежны (таймауты, ретраи, дубли сообщений at-least-once); без идемпотентности они порождают дубли (двойное списание).
7. At-most-once (максимум раз, возможна потеря), at-least-once (минимум раз, возможны дубли), exactly-once (ровно раз).
8. Сообщение доставляется минимум один раз, но возможны **дубли** → потребитель обязан дедуплицировать (по id/ключу).
9. В распределённой системе с ненадёжной сетью строгий exactly-once дорог/недостижим дёшево; на практике эмулируется at-least-once + идемпотентный обработчик.
10. **Saga** — последовательность локальных транзакций в разных сервисах; при сбое выполняются компенсирующие действия для отката бизнес-эффектов.
11. **Компенсация** — бизнес-действие, отменяющее эффект шага (вернуть деньги), а не технический rollback БД; часто «грязное» (эффект уже произошёл, его отменяют новой операцией).
12. **Оркестрация** — центральный оркестратор командует шагами. **Хореография** — сервисы реагируют на события друг друга без центра.
13. Оркестрация — для сложных процессов с ветвлениями и компенсациями (явный контроль, отслеживаемость). Хореография — для простых независимых реакций (слабая связанность).
14. **Dual write** — необходимость атомарно и записать в БД, и опубликовать событие; без атомарности один из шагов может упасть → рассинхрон данных и событий.
15. Событие пишется в таблицу outbox в **той же транзакции**, что бизнес-данные; отдельный процесс (relay) читает outbox и публикует в брокер. Атомарность «данные + событие» обеспечена транзакцией БД.
16. At-least-once (событие не потеряется, но возможны дубли) → потребители должны быть идемпотентны.
17. **Circuit Breaker** — «предохранитель» для вызовов. Состояния: Closed (пропускает), Open (после порога ошибок не зовёт, сразу ошибка/fallback), Half-Open (пробный запрос; успех → Closed, ошибка → Open).
18. Чтобы не ждать вечно зависший ответ: без таймаута вызов держит потоки/соединения, что ведёт к исчерпанию ресурсов и каскадному отказу.
19. Без backoff лавина повторов (retry storm) добивает уже перегруженный сервис; экспоненциальный backoff + jitter разносит повторы во времени.
20. Повтор неидемпотентной операции создаёт дубли (двойное списание/двойной заказ); retry безопасен только для идемпотентных операций (или с idempotency key).
21. **Retry storm** — лавина одновременных повторов от многих клиентов, усугубляющая перегрузку сервиса (часто добивает его окончательно).
22. **Bulkhead** — изоляция ресурсов (отдельные пулы на зависимости), чтобы сбой одной не утопил всю систему. **Fallback** — запасной ответ при недоступности (кэш, заглушка, деградация).
23. Повторять временные/преходящие (таймаут, 5xx, сетевые сбои); не повторять 4xx и бизнес-ошибки (повтор не поможет — запрос некорректен).
24. Показывать промежуточные состояния («обрабатывается»), не выдавать неподтверждённое за факт; read-your-writes (автор видит своё действие сразу); индикаторы свежести для задержек.
25. Ждать ACID между сервисами; retry без идемпотентности; at-least-once без дедупликации; retry без backoff; dual write без outbox; отсутствие таймаутов; вера в дешёвый exactly-once; компенсация как rollback БД; огромные синхронные цепочки.
