# Модуль 25 — Распределённые системы ⭐

## Зачем этот модуль

Микросервисы (модуль 21) и полиглот-хранилища (модуль 19) делают систему **распределённой**: компоненты общаются по сети, у каждого свои данные, нет общих ACID-транзакций. Сеть ненадёжна (CAP, модуль 19), узлы падают. Распределённые системы требуют отдельного набора решений: как держать согласованность без общих транзакций (saga), как переживать сбои (circuit breaker, retry, timeout), как не создавать дубли при повторах (идемпотентность).

Это один из самых частых и важных блоков на собеседовании middle+. Обязательно: модели согласованности, идемпотентность, гарантии доставки (at-least/at-most/exactly once), saga (оркестрация vs хореография), outbox, circuit breaker, retry с backoff.

---

## 1. Главная проблема: нет общих транзакций

В монолите перевод денег = одна ACID-транзакция. В распределённой системе «создать заказ + списать оплату + зарезервировать доставку» — это **три сервиса, три БД**, общего commit/rollback нет. Сеть может оборваться между шагами. Отсюда все паттерны этого модуля.

```
Монолит:        BEGIN; заказ; оплата; доставка; COMMIT  (атомарно)
Распределённо:  заказ(сервис A) → оплата(сервис B) → доставка(сервис C)
                каждый шаг — отдельная транзакция; между ними может всё сломаться
```

---

## 2. Модели согласованности

(см. также модуль 19)

- **Strong consistency** — все видят последнюю запись сразу (дорого, требует координации).
- **Eventual consistency** — реплики/сервисы сойдутся со временем (норма для распределёнки).
- **Causal consistency** — причинно-связанные события видны по порядку.

> В распределённой системе по умолчанию — eventual consistency между сервисами. Задача аналитика: определить, где это допустимо (статус, лента), а где нужна strong (внутри агрегата — деньги, остаток), и спроектировать UX под «данные обновятся через момент».

---

## 3. Идемпотентность

**Идемпотентная операция** — повтор даёт тот же результат, что и однократное выполнение. Критична, потому что в распределёнке **повторы неизбежны** (таймаут, ретрай, дубль сообщения).

**Idempotency key** — уникальный ключ операции от клиента; сервер запоминает обработанные ключи и при повторе возвращает прежний результат, не выполняя операцию заново.

```
POST /payments  Idempotency-Key: abc-123
→ сервер: ключ abc-123 не видел → выполнить списание, сохранить результат под ключом
повтор с тем же ключом → вернуть сохранённый результат, НЕ списывать второй раз
```

> Без идемпотентности ретрай оплаты = двойное списание. Idempotency key — стандартное решение для неидемпотентных операций (POST, оплата). Связь с HTTP-методами (модуль 23).

## 4. Гарантии доставки сообщений

| Гарантия | Суть | Последствие |
|----------|------|-------------|
| **At-most-once** | максимум один раз (может потеряться) | простые, но возможна потеря |
| **At-least-once** | минимум один раз (возможны дубли) | надёжно, но **нужна дедупликация** |
| **Exactly-once** | ровно один раз | идеал, но дорого/сложно (часто эмулируется at-least-once + идемпотентность) |

> На практике большинство брокеров дают **at-least-once** → потребитель **обязан** быть идемпотентным (дедупликация по ключу/id события). «Exactly-once» в распределённой системе строго недостижим дёшево — его эмулируют at-least-once + идемпотентный обработчик. Связь с модулем 32 (Kafka/RabbitMQ).

## 5. Saga — распределённые транзакции

**Saga** — последовательность локальных транзакций в разных сервисах; при сбое шага выполняются **компенсирующие** действия (откат «назад» бизнес-операциями, не rollback БД).

```
Заказ:   создать заказ → списать оплату → зарезервировать доставку
Сбой на доставке → компенсации: вернуть оплату → отменить заказ
```

Два способа координации:

| | Оркестрация | Хореография |
|--|-------------|-------------|
| Управление | центральный оркестратор командует шагами | сервисы реагируют на события друг друга |
| Плюсы | явная логика, легко отслеживать | слабая связанность, нет центра |
| Минусы | оркестратор — точка сложности/связности | трудно проследить общий поток |
| Когда | сложные процессы с ветвлениями | простые цепочки событий |

> Saga заменяет ACID-транзакцию набором локальных транзакций + компенсаций. Ключевое: компенсация — это **бизнес-действие** (вернуть деньги), а не откат БД. Оркестрация — явный «дирижёр»; хореография — события без центра. Связь с BPMN/процессами (модуль 12) и async (модуль 32).

## 6. Outbox — надёжная публикация событий

**Проблема dual write:** сервису надо и записать в свою БД, и опубликовать событие. Если между ними сбой — данные и события рассинхронизируются (записал заказ, но событие не ушло).

**Transactional Outbox:** событие пишется в таблицу `outbox` **в той же транзакции**, что и бизнес-данные; отдельный процесс читает outbox и публикует в брокер (at-least-once).

```
BEGIN;
  INSERT INTO orders ...
  INSERT INTO outbox (event) VALUES ('OrderCreated ...');  -- та же транзакция
COMMIT;
→ relay читает outbox → публикует в Kafka → помечает отправленным
```

> Outbox решает атомарность «изменил данные И отправил событие». Без него возможна потеря событий или «фантомные» события при откате. Часто сочетается с CDC (Change Data Capture).

## 7. Паттерны устойчивости (resilience)

| Паттерн | Назначение |
|---------|------------|
| **Timeout** | не ждать вечно ответа — ограничить время |
| **Retry + backoff** | повтор при сбое с нарастающей паузой (+ jitter) |
| **Circuit Breaker** | «предохранитель»: после серии ошибок перестать звать падающий сервис |
| **Bulkhead** | изоляция ресурсов (пул на зависимость), чтобы один сбой не утопил всё |
| **Fallback** | запасной ответ при недоступности (кэш, заглушка) |

**Circuit Breaker** — три состояния:
```
Closed (норма) ──много ошибок──► Open (не зовём, сразу ошибка/fallback)
   ▲                                  │ по таймеру
   └──── успех ─── Half-Open ◄────────┘ (пробный запрос)
```

> Retry **обязательно** сочетать с идемпотентностью (иначе повтор = дубль) и backoff (иначе «retry storm» добьёт сервис). Circuit breaker не даёт слать запросы в заведомо мёртвый сервис (защита от каскадных отказов). Timeout — основа всего: без него зависший вызов держит ресурсы.

## ⚠️ Подводные камни

- **Ждать ACID между сервисами.** Его нет — нужна saga + eventual consistency. Не моделируйте распределённую операцию как одну транзакцию.
- **Retry без идемпотентности.** Повтор неидемпотентной операции = дубли (двойное списание). Сначала идемпотентность, потом retry.
- **At-least-once без дедупликации.** Дубли сообщений неизбежны — потребитель обязан дедуплицировать по id/ключу.
- **Retry без backoff.** Лавина повторов (retry storm) добивает уже перегруженный сервис; нужен экспоненциальный backoff + jitter.
- **Dual write без outbox.** «Записал в БД и отправил событие» без атомарности → рассинхрон. Outbox/CDC.
- **Нет таймаутов.** Зависший синхронный вызов держит потоки/соединения → каскадный отказ. Таймаут на каждый внешний вызов.
- **«Exactly-once» как данность.** Дёшево недостижимо; эмулируется at-least-once + идемпотентность.
- **Компенсация = rollback БД.** Компенсация saga — бизнес-действие (вернуть деньги), часто «грязное» (деньги уже ушли) — продумывать.
