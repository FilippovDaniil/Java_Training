# Задача 02 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Спроектируйте stateless-сервис для горизонтального масштабирования: вынесите состояние и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервис приложений сейчас stateful: хранит сессии пользователей и корзину в памяти процесса, загруженные данные кэширует локально. При росте нагрузки нельзя свободно добавлять узлы (запрос должен попасть на «свой» сервер — sticky).

## ТРЕБОВАНИЯ
- Определите, какое состояние сейчас в памяти узла (сессии, корзина, локальный кэш).
- Вынесите состояние во внешние хранилища (сессии/корзина → Redis, кэш → общий).
- Покажите, как stateless-сервис масштабируется горизонтально за балансировщиком.
- Объясните, что это даёт для отказоустойчивости (падение узла не теряет состояние).
- Отметьте, где sticky-сессии всё же остаются и почему это хуже.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Состояние | Было (в памяти) | Стало (внешне)` + схема stateless + балансировщик + обоснование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выявлено состояние в памяти (сессии, корзина, локальный кэш).
- [ ] Сессии/корзина вынесены во внешнее хранилище (Redis).
- [ ] Кэш сделан общим/внешним (не на узле).
- [ ] Показано: любой узел обрабатывает любой запрос → масштаб +/− узлы.
- [ ] Объяснена отказоустойчивость (падение узла не теряет состояние).
- [ ] Отмечено, почему sticky-сессии — хуже (привязка мешает масштабу/failover).

## ПОДСКАЗКА
Stateless = узел не хранит состояние клиента между запросами. Сессии/корзину — в Redis, кэш — общий. Тогда любой из N узлов за балансировщиком обработает любой запрос: добавляем/убираем узлы свободно, падение узла не теряет данные. Sticky-сессии (привязка к узлу) — костыль для stateful, мешает масштабу и failover.
