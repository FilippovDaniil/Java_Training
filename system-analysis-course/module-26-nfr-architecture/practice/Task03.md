# Задача 03 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Определите SLI и SLO для сервиса и отделите их от SLA.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервис оформления заказов. Бизнес-требование: «сервис должен быть быстрым и надёжным». Нужно превратить в измеримые SLI/SLO и понять, что обещать клиентам (SLA).

## ТРЕБОВАНИЯ
- Определите 3–4 SLI (доступность, latency p95/p99, частота ошибок, успешность оформления).
- Задайте SLO (целевые значения) для каждого SLI.
- Сформулируйте SLA (что обещаем внешне) и покажите, что SLA ≤ SLO (запас).
- Объясните разницу SLI/SLO/SLA своими словами.
- Свяжите доступность SLO с допустимым простоем (девятки → минуты).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `SLI | SLO (цель) | SLA (обещание)` + пояснение разницы + перевод SLO доступности в простой.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Определены измеримые SLI (доступность, latency p95, % ошибок, успешность).
- [ ] Заданы конкретные SLO (например, доступность ≥ 99.9%, p95 ≤ 300 мс, ошибки ≤ 0.1%).
- [ ] SLA сформулирован и ниже/равен SLO (запас перед клиентом).
- [ ] Объяснена разница: SLI (метрика) / SLO (цель) / SLA (контракт+штрафы).
- [ ] SLO доступности переведён в простой (99.9% ≈ 43 мин/мес).
- [ ] Значения измеримы и проверяемы.

## ПОДСКАЗКА
SLI — что меряем (доступность, p95 latency, % ошибок). SLO — внутренняя цель (≥99.9%, p95≤300мс). SLA — обещание клиенту со штрафами, обычно ниже SLO (запас: обещаем 99.9%, целимся 99.95%). Доступность переводится в простой по таблице девяток (99.9% ≈ 43 мин/мес). Связь с измеримостью NFR (модуль 09).
