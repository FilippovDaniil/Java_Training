# Задача 04 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Спроектируйте graceful degradation при отказе зависимости: система работает частично, а не падает целиком.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, главный экран зависит от нескольких сервисов: профиль (критичен), список врачей (критичен), рекомендации врачей (некритичны), баннеры/акции (некритичны), отзывы (некритичны). Любой сервис может временно отказать.

## ТРЕБОВАНИЯ
- Классифицируйте зависимости на критичные и некритичные.
- Для каждой некритичной опишите поведение при отказе (скрыть блок, заглушка, кэш, значение по умолчанию).
- Для критичных — что делать (без них экран не имеет смысла → понятная ошибка/ретрай).
- Свяжите с circuit breaker/fallback (модуль 25).
- Покажите, как degradation повышает воспринимаемую доступность.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Зависимость | Критичность | Поведение при отказе` + связь с CB/fallback.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зависимости разделены на критичные (профиль, врачи) и некритичные (рекомендации, баннеры, отзывы).
- [ ] Для некритичных — degradation (скрыть/заглушка/кэш), экран работает.
- [ ] Для критичных — понятная ошибка/ретрай (без них смысла нет).
- [ ] Указана связь с circuit breaker/fallback (модуль 25).
- [ ] Объяснено, что degradation повышает воспринимаемую доступность.
- [ ] Отказ некритичного сервиса не роняет весь экран.

## ПОДСКАЗКА
Graceful degradation: при отказе НЕкритичной зависимости показать остальное (скрыть рекомендации/баннеры, отдать кэш или заглушку), а не падать целиком. Критичные (профиль, список врачей) — без них экран бессмысленен → ретрай/понятная ошибка. Технически это fallback за circuit breaker (модуль 25). Итог: воспринимаемая доступность выше реальной доступности всех зависимостей.
