# Задача 05 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Определите, что логировать, мерить и трейсить для сценария: спроектируйте наблюдаемость по трём столпам.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, распределённый сценарий оформления заказа (Gateway → Заказы → Платежи → Доставка). Нужно обеспечить наблюдаемость, чтобы при инциденте быстро находить причину и где именно затык.

## ТРЕБОВАНИЯ
- **Logs:** какие события логировать (с контекстом), на каком уровне.
- **Metrics:** какие метрики собирать («золотые сигналы»: latency, traffic, errors, saturation).
- **Traces:** как трейсить сквозной путь через сервисы, роль correlation ID.
- Покажите, как три столпа вместе локализуют проблему («где в цепочке тормозит/падает»).
- Свяжите с SLI (метрики питают SLI/SLO).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица по трём столпам `Столп | Что собираем | Зачем` + роль correlation ID + связь с SLI.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Logs: ключевые события сценария с контекстом (id заказа, correlation ID), уровни.
- [ ] Metrics: золотые сигналы (latency p95/p99, RPS, % ошибок, насыщение ресурсов).
- [ ] Traces: сквозной трейс через сервисы, correlation ID связывает логи/трейсы.
- [ ] Показано, как три столпа локализуют проблему (трейс → где, логи → что, метрики → насколько).
- [ ] Метрики питают SLI/SLO (модуль 09/26).
- [ ] Correlation ID проходит через все сервисы.

## ПОДСКАЗКА
Три столпа: logs (что произошло — с correlation ID), metrics (сколько/как быстро — золотые сигналы latency/traffic/errors/saturation → SLI), traces (путь запроса — где затык). Correlation ID связывает их в единую картину. В распределёнке именно trace показывает, на каком сервисе проблема, а логи — детали. Связь с эксплуатацией (модуль 37).
