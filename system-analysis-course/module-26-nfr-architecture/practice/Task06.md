# Задача 06 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Выполните capacity planning под прогноз нагрузки: оцените ресурсы с запасом на пик.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, прогноз: в обычный день ~200 заказов/мин в пик, в Чёрную пятницу ожидается ×5. Один сервер приложений обрабатывает ~100 запросов/сек на комфортной загрузке (~70%). Один заказ = ~3 запроса к сервису. Нужно спланировать число узлов под пик с запасом.

## ТРЕБОВАНИЯ
- Переведите прогноз заказов в RPS (запросы/сек) для обычного пика и для Чёрной пятницы.
- Рассчитайте число узлов под нагрузку при целевой загрузке (~70%).
- Заложите запас (headroom) на всплески и отказ узла (N+1).
- Свяжите с автоскейлингом (горизонтальный масштаб, stateless).
- Отметьте узкие места не только в app-серверах (БД, внешние сервисы).

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт RPS → число узлов (обычный/пиковый) + запас + замечание про автоскейлинг и другие узкие места.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Прогноз переведён в RPS (обычный пик и ×5), с учётом ~3 запроса/заказ.
- [ ] Рассчитано число узлов при целевой загрузке ~70%.
- [ ] Заложен запас на всплеск и отказ узла (N+1).
- [ ] Упомянут автоскейлинг (добавлять узлы под нагрузку, требует stateless).
- [ ] Отмечено, что узким местом может быть БД/внешний сервис, не только app.
- [ ] Расчёт прозрачен (формулы и числа).

## ПОДСКАЗКА
Capacity planning: прогноз → нагрузка (RPS) → ресурсы с запасом. Пример: 200 заказов/мин ×5 (ЧП) = 1000 заказов/мин ≈ 16.7/сек × 3 запроса ≈ 50 RPS… (приведите свой расчёт под исходные числа). Делите на ёмкость узла при ~70% загрузке, добавляете N+1 на отказ и headroom на всплеск. Автоскейлинг масштабирует stateless-узлы. Проверьте, что БД/внешние сервисы тоже выдержат пик.
