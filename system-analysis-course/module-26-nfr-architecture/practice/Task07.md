# Задача 07 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Примените error budget для решения «релизить или стабилизировать».

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервис заказов с SLO доступности 99.9% в месяц. Две ситуации:
1. С начала месяца было 5 минут простоя; команда хочет выкатить рискованную фичу.
2. С начала месяца было 40 минут простоя; та же фича на очереди.

## ТРЕБОВАНИЯ
- Рассчитайте error budget из SLO 99.9% (в минутах/месяц).
- Для каждой ситуации определите остаток бюджета и решение (релизить/стабилизировать).
- Объясните, как error budget балансирует скорость релизов и надёжность.
- Сформулируйте политику: что делать при исчерпании бюджета.

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт error budget + решение по двум ситуациям + политика при исчерпании.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Error budget = 1 − 0.999 = 0.1% ≈ 43 мин/мес.
- [ ] Ситуация 1 (5 мин потрачено) → бюджет есть → можно релизить рискованное.
- [ ] Ситуация 2 (40 мин потрачено) → бюджет почти исчерпан → стабилизировать, заморозить рискованные релизы.
- [ ] Объяснено: error budget = управляемый компромисс надёжность vs скорость.
- [ ] Сформулирована политика при исчерпании (freeze фич, фокус на надёжность).
- [ ] Расчёт корректен.

## ПОДСКАЗКА
Error budget = 1 − SLO. При 99.9% это ~43 мин простоя/мес. Пока бюджет не исчерпан — допустимы рискованные релизы (есть «запас на ошибки»); исчерпан — стоп фичам, фокус на стабильность. Это переводит спор «продукт vs надёжность» в числовое правило. 5 мин из 43 — катим; 40 из 43 — стабилизируем.
