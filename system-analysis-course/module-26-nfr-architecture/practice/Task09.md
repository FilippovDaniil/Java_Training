# Задача 09 — Модуль 26: Нефункциональная архитектура

## ЗАДАНИЕ
Спроектируйте health checks (liveness/readiness) и поведение системы при них.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сервис в кластере за балансировщиком/оркестратором. Сервис зависит от БД и кэша. Нужно настроить проверки здоровья, чтобы балансировщик/оркестратор корректно направлял трафик и перезапускал «зависшие» экземпляры.

## ТРЕБОВАНИЯ
- Разграничьте **liveness** (жив ли процесс) и **readiness** (готов ли принимать трафик).
- Определите, что проверять в каждом (liveness — простой ping; readiness — доступность БД/зависимостей).
- Опишите поведение оркестратора: liveness fail → перезапуск; readiness fail → убрать из ротации (не убивать).
- Разберите ловушку: тяжёлые проверки/каскад (readiness падает из-за внешней зависимости → весь сервис вне ротации).
- Свяжите с балансировкой (модуль 23) и graceful degradation (задача 04).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Проверка | Что проверяет | Реакция системы` + разбор ловушки каскада.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Liveness — жив ли процесс (лёгкая проверка), fail → перезапуск.
- [ ] Readiness — готов ли к трафику (зависимости доступны), fail → убрать из ротации (не убивать).
- [ ] Разграничены реакции: перезапуск vs исключение из балансировки.
- [ ] Разобрана ловушка: readiness, завязанный на внешнюю зависимость, может вывести весь кластер из ротации.
- [ ] Связь с балансировкой/health checks (модуль 23).
- [ ] Отмечено, как не превратить временный сбой зависимости в полный отказ.

## ПОДСКАЗКА
Liveness = «процесс жив?» (простой эндпоинт); провал → оркестратор перезапускает экземпляр. Readiness = «готов принимать трафик?» (проверка зависимостей); провал → балансировщик убирает из ротации, но НЕ убивает. Ловушка: если readiness зависит от общей внешней зависимости, её сбой выведет ВСЕ экземпляры из ротации одновременно — иногда лучше degradation, чем «не готов». Связь с балансировкой (модуль 23).
