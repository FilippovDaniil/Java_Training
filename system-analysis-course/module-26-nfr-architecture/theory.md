# Модуль 26 — Нефункциональная архитектура ⭐

## Зачем этот модуль

Стили, слои, границы и распределённость заданы (21–25). Но система должна не только «работать», а выдерживать нагрузку, быть доступной, переживать сбои и быть наблюдаемой. Это **нефункциональная архитектура** — как архитектурные решения обеспечивают NFR (модуль 09): масштабируемость, доступность (SLA/SLO/SLI), отказоустойчивость, наблюдаемость, capacity planning.

Это завершение Части 5 и мост между «архитектурой» и «эксплуатацией». Аналитик переводит NFR-требования в архитектурные решения и обратно — объясняет, почему «99.99%» стоит денег. На собеседовании middle+ обязательно: расчёт SLA из компонентов, stateless для горизонтального масштаба, SLI/SLO/error budget, три столпа наблюдаемости, устранение SPOF.

---

## 1. Масштабируемость

**Масштабируемость** — способность системы расти под нагрузку.

| Тип | Суть | Плюс/минус |
|-----|------|------------|
| **Вертикальное (scale up)** | мощнее железо (CPU/RAM) | просто, но предел и дорого; SPOF |
| **Горизонтальное (scale out)** | больше узлов | почти безграничный масштаб, отказоустойчивость; нужен stateless |

**Stateless** — сервер не хранит состояние клиента между запросами (сессии — во внешнем хранилище). Только stateless-сервисы масштабируются горизонтально свободно: любой узел обработает любой запрос, можно добавлять/убирать узлы.

```
Stateful:  запрос клиента ДОЛЖЕН попасть на «свой» сервер (sticky) → плохо масштабируется
Stateless: любой узел обработает любой запрос → +узлы за балансировщиком → линейный масштаб
```

> Горизонтальное масштабирование — основа облачной архитектуры, но требует stateless-сервисов (состояние — в Redis/БД, не в памяти узла). «Сделать stateless» — частая архитектурная задача и вопрос на собеседовании.

## 2. Доступность: SLA / SLO / SLI

| Термин | Что это | Кто |
|--------|---------|-----|
| **SLI (Indicator)** | измеряемая метрика (доступность, latency p95, % ошибок) | измеряем |
| **SLO (Objective)** | целевое значение SLI (например, доступность ≥ 99.9%) | ставим себе |
| **SLA (Agreement)** | договорное обязательство перед клиентом + штрафы | обещаем клиенту |

**Доступность в «девятках»** → допустимый простой:

| Доступность | Простой в год | в месяц |
|-------------|---------------|---------|
| 99% (две девятки) | ~3.65 дня | ~7.2 ч |
| 99.9% (три) | ~8.76 ч | ~43 мин |
| 99.99% (четыре) | ~52.6 мин | ~4.3 мин |
| 99.999% (пять) | ~5.26 мин | ~26 сек |

```
SLI = «что меряем» (факт), SLO = «цель» (внутренняя), SLA = «обещание клиенту» (контракт).
Обычно SLA < SLO (запас): обещаем 99.9%, целимся в 99.95%.
```

> Каждая девятка — кратно дороже (резервирование, мультизона). Аналитик не пишет «100% доступность» (недостижимо/бессмысленно дорого) — а обоснованное значение под бизнес-критичность.

---

## 3. Расчёт SLA из компонентов

Доступность системы зависит от того, как соединены компоненты:

- **Последовательно (зависимость):** доступности **перемножаются** → итог **ниже** каждого.
  `0.99 × 0.99 × 0.99 = 0.970` (97%, хотя каждый 99%).
- **Параллельно (резервирование):** считается через отказ: `1 − (1−A)ⁿ` → итог **выше**.
  Два узла по 99%: `1 − (0.01 × 0.01) = 0.9999` (99.99%).

```
Цепочка из N зависимостей роняет доступность (умножение).
Резервирование (дубли) повышает (избыточность).
```

> Ключевой вывод: чем длиннее цепочка обязательных зависимостей, тем ниже общая доступность. Поэтому минимизируют синхронные зависимости (async, graceful degradation) и резервируют критичные узлы. Это любимый расчётный вопрос на собеседовании.

## 4. Отказоустойчивость

- **Redundancy (резервирование)** — дублирование узлов/зон, нет единой точки отказа.
- **SPOF (Single Point of Failure)** — компонент, отказ которого роняет всю систему. Устраняется резервированием.
- **Graceful degradation** — при отказе зависимости система работает частично, а не падает целиком (показать каталог без рекомендаций, если сервис рекомендаций недоступен).
- **Failover** — автоматическое переключение на резерв.
- **Multi-AZ / Multi-region** — размещение в разных зонах доступности/регионах.

> Связь с модулем 25 (circuit breaker, fallback) — это тактика graceful degradation. Архитектурно: найти SPOF (один узел БД, один балансировщик, одна зона) и зарезервировать критичные. Health checks (liveness/readiness) — основа автоматического failover.

---

## 5. Наблюдаемость (observability)

Три столпа — чтобы понимать, что происходит в системе (особенно распределённой):

| Столп | Что | Отвечает на вопрос |
|-------|-----|--------------------|
| **Logs (логи)** | события с контекстом | «что именно произошло?» |
| **Metrics (метрики)** | числовые показатели во времени | «сколько/как часто/насколько быстро?» |
| **Traces (трейсы)** | путь запроса через сервисы | «где в цепочке проблема?» |

- **Correlation ID** — сквозной идентификатор запроса через все сервисы (связывает логи/трейсы).
- **Метрики «золотых сигналов»:** latency, traffic, errors, saturation.

> В распределённой системе без наблюдаемости отладка невозможна: запрос проходит через много сервисов. Trace + correlation ID показывают, где затык. Аналитик задаёт: что логировать, какие метрики и SLI собирать, как трейсить сквозной сценарий (связь с модулями 25, 37).

## 6. Error budget и capacity planning

- **Error budget** = `1 − SLO`. При SLO 99.9% бюджет ошибок — 0.1% (≈43 мин/мес). Пока бюджет не исчерпан — можно релизить рискованное; исчерпан — фокус на стабилизацию.
- **Capacity planning** — расчёт ресурсов под прогноз нагрузки: пиковый RPS, рост, запас. Связь с NFR пиков (Чёрная пятница, модуль 09).

```
Error budget превращает «надёжность vs скорость релизов» в управляемый числовой компромисс:
есть бюджет → катим фичи; нет → стабилизируем.
```

> Error budget — инструмент диалога между продуктом (скорость) и надёжностью. Capacity planning переводит прогноз нагрузки в число узлов/ресурсов с запасом на пики.

## ⚠️ Подводные камни

- **«100% доступность».** Недостижима и бессмысленно дорога; ставить обоснованную цель (SLO) под критичность.
- **Путать SLA/SLO/SLI.** SLI — метрика, SLO — внутренняя цель, SLA — обещание клиенту (со штрафами). SLA обычно ниже SLO (запас).
- **Игнорировать перемножение доступностей.** Длинная синхронная цепочка зависимостей роняет общую доступность ниже каждого звена.
- **Stateful-серверы при горизонтальном масштабе.** Состояние в памяти узла мешает масштабу и failover; выносить наружу (Redis/БД).
- **Не искать SPOF.** Один узел БД/балансировщик/зона = единая точка отказа; резервировать критичные.
- **Падать целиком вместо degradation.** Отказ некритичной зависимости (рекомендации) не должен ронять весь сервис.
- **Нет наблюдаемости в распределёнке.** Без логов/метрик/трейсов и correlation ID отладка невозможна.
- **NFR без измеримости.** «Быстро/надёжно» без чисел не проверяется (модуль 09): SLI/SLO с конкретными значениями.
