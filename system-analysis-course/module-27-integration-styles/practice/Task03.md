# Задача 03 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Объясните, почему интеграция через «общую БД» — антипаттерн, и предложите корректную замену.

## ВХОДНЫЕ ДАННЫЕ
Текущее решение FoodExpress: сервисы Заказы, Доставка и Аналитика интегрированы через **общую таблицу** `order` — все три читают и пишут в неё напрямую. Команды жалуются: нельзя изменить схему таблицы, не сломав других; блокировки; неясно, кто отвечает за данные.

## ТРЕБОВАНИЯ
- Перечислите проблемы интеграции через общую БД (связанность, схема, владение, блокировки).
- Определите владельца данных заказа (один сервис).
- Предложите замену: как Доставка и Аналитика получают данные (API/события вместо прямого доступа).
- Покажите, как это снижает связанность и возвращает инкапсуляцию.
- Свяжите с database per service (модуль 21).

## ФОРМАТ РЕЗУЛЬТАТА
Список проблем + назначение владельца + схема замены (API/события) + объяснение снижения связанности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Перечислены проблемы: скрытая связанность через схему, нет инкапсуляции, неясное владение, блокировки.
- [ ] Назначен единственный владелец данных заказа (сервис Заказы).
- [ ] Доставка получает данные через API/события, не из общей таблицы.
- [ ] Аналитика получает данные через события/реплику/выгрузку, не пишет в общую таблицу.
- [ ] Объяснено снижение связанности и возврат инкапсуляции.
- [ ] Связь с database per service (модуль 21).

## ПОДСКАЗКА
Общая БД = скрытая связанность через схему: один владелец не может менять таблицу, не сломав других; инварианты обходятся прямым доступом. Лечение: один владелец данных (Заказы), остальные — через его API или события (Доставка слушает «заказ создан», Аналитика — через события/реплику для чтения). Это database per service (модуль 21).
