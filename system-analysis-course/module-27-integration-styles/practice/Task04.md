# Задача 04 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Выберите между синхронным RPC и асинхронным messaging для набора взаимодействий и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Взаимодействия FoodExpress:
1. Проверка доступности блюда (стоп-лист) при оформлении — нужен ответ сейчас.
2. Отправка чека и push-уведомления после оплаты — можно с задержкой.
3. Получение курса оплаты от платёжного провайдера — синхронный запрос-ответ.
4. Обновление поискового индекса при изменении меню — допустима задержка.
5. Начисление бонусов после доставки — фоновая операция.

## ТРЕБОВАНИЯ
- Для каждого взаимодействия выберите sync RPC или async messaging.
- Обоснуйте через: нужен ли немедленный ответ, развязка, устойчивость к недоступности.
- Покажите, чем плох неверный выбор (sync для фонового → лишняя связанность; async для realtime-проверки → нет ответа сейчас).
- Свяжите с устойчивостью (модуль 25): sync требует timeout/CB, async — гарантий доставки.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Взаимодействие | Sync/Async | Обоснование` + замечание о цене неверного выбора.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проверка стоп-листа, курс провайдера → sync RPC (нужен ответ сейчас).
- [ ] Чек/уведомления, индекс, бонусы → async messaging (развязка, можно позже).
- [ ] Обоснование через немедленность ответа vs развязку.
- [ ] Показана цена неверного выбора в обе стороны.
- [ ] Sync → timeout/circuit breaker; async → гарантии доставки/DLQ (модуль 25/32).
- [ ] Выбор соответствует требованиям каждого взаимодействия.

## ПОДСКАЗКА
Sync RPC — когда нужен немедленный ответ для продолжения (стоп-лист, курс оплаты), но это временная связанность (B должен быть жив) → защищать timeout/CB. Async — для фоновых/развязанных операций (уведомления, индекс, бонусы): отправитель не ждёт, получатель обработает позже → нужны гарантии доставки/DLQ. Неверный выбор: sync для фонового добавляет связанность; async для realtime-проверки не даёт ответа вовремя.
