# Задача 06 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Примените content-based router и message translator для интеграции с разными партнёрами.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** принимает заказы от нескольких партнёров-агрегаторов: партнёр A шлёт JSON в своём формате, партнёр B — XML, партнёр C — другой JSON. Все заказы нужно привести к внутреннему формату и направить в обработку. Часть партнёров шлёт ещё и отмены.

## ТРЕБОВАНИЯ
- Примените message translator: перевод каждого внешнего формата во внутренний (≈ ACL, модуль 24).
- Примените content-based router: разделить «новый заказ» и «отмена» по содержимому.
- Покажите, где translator, где router в потоке.
- Объясните, почему внешние форматы не должны протекать во внутреннюю обработку.
- Свяжите с ACL (модуль 24) и форматами/схемами (модуль 28).

## ФОРМАТ РЕЗУЛЬТАТА
Схема потока (вход партнёров → translator → router → обработка) + примеры перевода + пояснение про изоляцию форматов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Translator переводит каждый внешний формат (A-JSON, B-XML, C-JSON) во внутренний.
- [ ] Router разделяет тип сообщения (новый заказ vs отмена) по содержимому.
- [ ] Показан порядок: сначала перевод во внутренний формат, затем маршрутизация.
- [ ] Объяснено, почему форматы партнёров не должны протекать внутрь (изоляция).
- [ ] Связь с ACL (модуль 24) — translator на границе.
- [ ] Связь с форматами/схемами (модуль 28).

## ПОДСКАЗКА
Message Translator (EIP) переводит чужой формат во внутренний — это тот же принцип, что ACL в DDD (модуль 24): внешние модели не пускаем внутрь. Поток: вход партнёра → translator (в наш формат) → content-based router (новый заказ/отмена → разные каналы) → обработка. Внутренняя обработка работает с одним форматом, не зная о partner-specific. Форматы/схемы — модуль 28.
