# Задача 07 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Примените aggregator: соберите единый ответ из нескольких сервисов.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, экран «детали заказа» собирает данные из нескольких сервисов: Заказы (состав, сумма), Доставка (статус, курьер, ETA), Платежи (статус оплаты), Отзывы (рейтинг ресторана). Нужен единый ответ клиенту.

## ТРЕБОВАНИЯ
- Спроектируйте aggregator: запросить несколько сервисов и собрать единый ответ.
- Решите, синхронно (parallel-вызовы + сборка) или через события (предсобранная проекция).
- Обработайте частичную недоступность сервиса (один не ответил → graceful degradation, модуль 26).
- Задайте таймаут на сбор (не ждать самый медленный бесконечно).
- Сравните с BFF (модуль 22) — где здесь граница.

## ФОРМАТ РЕЗУЛЬТАТА
Схема aggregator (источники → сборка → единый ответ) + обработка частичной недоступности + таймаут + связь с BFF.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Aggregator запрашивает несколько сервисов и собирает единый ответ.
- [ ] Выбран подход (синхронная параллельная сборка или предсобранная проекция через события) с обоснованием.
- [ ] Частичная недоступность → degradation (показать без недоступного блока), не падать целиком.
- [ ] Задан таймаут на сбор (не ждать медленный бесконечно).
- [ ] Связь с BFF (модуль 22) — агрегация под клиента.
- [ ] Ответ собирается из независимых источников.

## ПОДСКАЗКА
Aggregator (EIP) собирает несколько сообщений/ответов в один. Для «деталей заказа»: параллельно запросить сервисы и склеить (быстрее, но связанность) или держать предсобранную проекцию, обновляемую событиями (read-модель, без связанности на чтении). Частичный сбой → degradation (без блока отзывов, например). Таймаут обязателен. Это часто живёт в BFF (модуль 22).
