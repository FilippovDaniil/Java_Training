# Задача 08 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Выберите стиль интеграции с внешней платёжной системой и спроектируйте устойчивое взаимодействие.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** интегрируется с внешним платёжным провайдером (Альфа-Банк): инициировать оплату, получить статус, обработать callback о результате. Провайдер — внешняя система, иногда недоступна/медленна. Оплата критична (деньги).

## ТРЕБОВАНИЯ
- Выберите стиль для каждого взаимодействия: инициация оплаты (sync RPC), получение результата (callback/webhook = async входящий + опрос статуса).
- Спроектируйте надёжность: idempotency для инициации, таймаут/ретрай, проверка статуса (на случай потери callback).
- Обработайте callback (входящее сообщение от провайдера) и его возможные дубли/потери.
- Учтите безопасность интеграции (подпись callback, секреты) — связь с модулем 34.
- Свяжите с устойчивостью (модуль 25): idempotency, retry, circuit breaker.

## ФОРМАТ РЕЗУЛЬТАТА
Схема взаимодействия (инициация + callback + опрос статуса) + механизмы надёжности + безопасность.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Инициация оплаты → sync RPC с idempotency key (защита от двойного списания).
- [ ] Результат → обработка callback/webhook + резервный опрос статуса (на случай потери callback).
- [ ] Callback обрабатывается идемпотентно (возможны дубли).
- [ ] Таймаут/ретрай/circuit breaker на вызовы провайдера (модуль 25).
- [ ] Безопасность: проверка подписи callback, хранение секретов (модуль 34).
- [ ] Критичность денег учтена (не терять/не задваивать).

## ПОДСКАЗКА
Платёжная интеграция = sync инициация (idempotency key!) + асинхронный результат (callback/webhook) + резервный опрос статуса (callback может потеряться — не полагаться только на него). Callback идемпотентен (дубли), с проверкой подписи (модуль 34). Внешний провайдер ненадёжен → timeout/retry/circuit breaker (модуль 25). Деньги — нельзя ни потерять, ни задвоить.
