# Задача 09 — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Оцените связанность (coupling) вариантов интеграции и выберите наименее связанный под требования.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, нужно интегрировать сервис Заказы и сервис Доставка (Доставка реагирует на новые заказы). Варианты:
1. Доставка читает таблицу `order` сервиса Заказы напрямую (общая БД).
2. Заказы синхронно вызывают API Доставки при создании заказа.
3. Заказы публикуют событие «OrderCreated», Доставка подписана.

## ТРЕБОВАНИЯ
- Оцените каждый вариант по видам связанности: временная, схемная, по местоположению.
- Покажите последствия: что ломается при изменении/недоступности одной стороны.
- Выберите наименее связанный вариант под требование «развязка + устойчивость».
- Отметьте, чем снижать остаточную связанность (версионирование схемы события).
- Свяжите с распределённым монолитом (модуль 21).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Вариант | Временная | Схемная | По месту | Последствия` + выбор с обоснованием.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Вариант 1 (общая БД): очень высокая схемная связанность; изменение схемы ломает Доставку.
- [ ] Вариант 2 (sync RPC): высокая временная связанность; недоступность Доставки ломает создание заказа.
- [ ] Вариант 3 (событие): низкая связанность; Доставка может быть временно недоступна (событие подождёт).
- [ ] Выбран вариант 3 под «развязка + устойчивость».
- [ ] Остаточная связанность (схема события) снижается версионированием.
- [ ] Связь с распределённым монолитом (вариант 1-2 → высокая связанность).

## ПОДСКАЗКА
Виды связанности: временная (оба живы сейчас — sync), схемная (зависимость от структуры — общая БД), по месту (жёсткие адреса). Общая БД — максимум схемной; sync RPC — максимум временной; событие — минимум обеих (развязка во времени, контракт = схема события). Для «развязки + устойчивости» — события. Остаточную связанность (схема) снижают версионированием (модуль 28/30). Высокая связанность = распределённый монолит (модуль 21).
