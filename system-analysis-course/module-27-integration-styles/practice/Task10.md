# Задача 10 (мини-проект) — Модуль 27: Способы интеграции (EIP)

## ЗАДАНИЕ
Составьте карту интеграций MedConnect: внешние системы, способы интеграции, EIP-паттерны, связанность и устойчивость. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** (телемедицина, 152-ФЗ). Внешние и внутренние интеграции: платёжный провайдер, провайдер видеосвязи, государственная система рецептов (legacy, только файлы/SOAP), сервис уведомлений (SMS/email/push), поисковый индекс врачей, аналитика, партнёрские клиники (разные форматы). Внутренние сервисы: Записи, Консультации, Рецепты, Платежи.

## ТРЕБОВАНИЯ
Сдайте карту интеграций:
1. **Перечень интеграций** — для каждой: участники, что обменивается.
2. **Способ** — file / RPC / messaging (общую БД не использовать); обоснование под требования.
3. **Sync vs async** — где немедленный ответ, где развязка.
4. **EIP-паттерны** — router/translator/aggregator/DLQ где уместно; translator/ACL для legacy и партнёров.
5. **Связанность** — оценка и как снижается (события, версионирование, ACL).
6. **Устойчивость** — для внешних: timeout/retry/CB, idempotency, гарантии (связь с модулем 25).
7. **152-ФЗ** — где идут медданные, защита канала/доступа.

## ФОРМАТ РЕЗУЛЬТАТА
Карта интеграций (таблица/диаграмма) `Интеграция | Способ | Sync/Async | EIP-паттерны | Связанность | Устойчивость` + раздел 152-ФЗ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все интеграции перечислены с участниками и предметом обмена.
- [ ] Способ выбран обоснованно (общая БД не используется).
- [ ] Sync/async распределены по требованиям.
- [ ] EIP-паттерны применены (translator/ACL для ГИС и партнёров, router, aggregator, DLQ).
- [ ] Связанность оценена и снижается осознанно.
- [ ] Устойчивость внешних вызовов (timeout/retry/CB/idempotency) учтена.
- [ ] Медданные/152-ФЗ: защищённые каналы и доступ отмечены.

## ПОДСКАЗКА
Синтез: выбор способа (зад.01), файлы (02), общая БД (03), sync/async (04), router (05), translator/ACL (06), aggregator (07), платёжная интеграция (08), связанность (09). Главные проверки middle+: способ под требования (не общая БД), translator/ACL для legacy и разноформатных партнёров, async для развязки, устойчивость внешних вызовов, низкая связанность, учёт 152-ФЗ. Стыкуется с DDD/ACL (24), распределёнкой (25), форматами (28), async (32), безопасностью (34).
