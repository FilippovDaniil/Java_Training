## Ключ (эталонные ответы)

1. File Transfer (обмен файлами), Shared Database (общая БД — антипаттерн), RPC (синхронный вызов API), Messaging (асинхронные сообщения через брокер).
2. Обмен файлами (выгрузка/загрузка по расписанию). Уместна для пакетного обмена, отчётов, интеграции с legacy/партнёрами, умеющими только файлы. Минус — запаздывание.
3. Создаёт скрытую связанность через схему (нельзя менять таблицу, не сломав других), обходит инкапсуляцию/инварианты, неясное владение данными, блокировки. Нарушает database per service.
4. Синхронный вызов чужого API (REST/gRPC): «запрос-ответ» в реальном времени; вызывающий ждёт результат.
5. Асинхронный обмен сообщениями через брокер: отправитель публикует, получатель обрабатывает позже; развязка во времени.
6. **Синхронная** — вызывающий ждёт ответ, оба должны быть живы (временная связанность). **Асинхронная** — отправил и продолжил, получатель обработает позже (развязка).
7. Зависимость во времени: системы должны работать одновременно (характерна для sync RPC); недоступность одной ломает другую.
8. Временная (оба живы сейчас), схемная/по формату (структура данных), по местоположению (адреса/хосты), по технологии (стек/протокол).
9. Чтобы системы можно было менять, деплоить, масштабировать и заменять независимо; высокая связанность ведёт к хрупкости и каскадным отказам (распределённый монолит).
10. **EIP (Enterprise Integration Patterns)** — каталог паттернов интеграции (Hohpe & Woolf), общий словарь для messaging-интеграции.
11. **Message Channel** — канал (очередь/топик), по которому передаются сообщения между системами.
12. **Message Router** направляет сообщение нужному получателю по правилу. **Content-Based Router** — по содержимому сообщения (поля/флаги).
13. **Message Translator** преобразует формат/структуру между системами; соответствует Anti-Corruption Layer в DDD (перевод чужой модели в свою на границе).
14. **Aggregator** собирает несколько сообщений/ответов в одно (например, единый ответ из нескольких сервисов).
15. **Splitter** разбивает одно сообщение на несколько (обратное aggregator).
16. **Dead Letter Channel (DLQ)** — канал для сообщений, которые не удалось обработать (для разбора, чтобы не терять и не блокировать поток).
17. **Sync** — когда нужен немедленный ответ для продолжения (проверка, запрос данных). **Async** — для развязки, устойчивости, фоновых задач, нескольких подписчиков, сглаживания пиков.
18. Создаёт временную связанность и каскадные отказы: недоступность/замедление получателя ломает вызывающего; цепочки множат латентность.
19. Нужны брокер, гарантии доставки, дедупликация (at-least-once), DLQ, обработка порядка/дублей; сложнее отлаживать и рассуждать о согласованности (eventual).
20. Один владелец данных; остальные получают их через его API (sync) или через подписку на его события (async); никакого прямого доступа к чужой БД (database per service).
21. Перейти на асинхронное взаимодействие (messaging): получатель может быть временно недоступен, сообщение подождёт в очереди.
22. Версионируемые контракты/схемы (JSON Schema, OpenAPI, схемы событий), обратная совместимость — модули 28/30; ACL/translator на границе.
23. Timeout на вызовы, retry с backoff (+ идемпотентность), circuit breaker, fallback; для критичного — async + резервный опрос статуса; DLQ для async (модуль 25).
24. Метаданные (число записей, контрольная сумма), расписание/окно, обработка ошибок (нет файла/битый/частичный), идемпотентность загрузки (распознавать повторы по id/хешу), квитанции.
25. Общая БД как интеграция; sync везде; async без нужды; файловый обмен без обработки ошибок; игнор связанности; нет translator/ACL для чужого формата; нет DLQ для async.
