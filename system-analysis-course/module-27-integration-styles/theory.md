# Модуль 27 — Способы интеграции (EIP)

## Зачем этот модуль

Система разбита на сервисы/модули (Часть 5), вокруг неё — внешние системы. Часть 6 — про **интеграцию**: как компоненты и системы обмениваются данными. Базовый выбор — **способ интеграции** (файлы, общая БД, RPC, сообщения) и **паттерны** (Enterprise Integration Patterns, EIP). Это определяет связанность, надёжность и масштаб.

Аналитик проектирует интеграции постоянно: с платёжными системами, госсервисами, партнёрами, между своими сервисами. На собеседовании middle+ обязательно: четыре способа интеграции и их trade-offs, sync vs async, почему «общая БД» — антипаттерн, базовые EIP-паттерны, как оценивать связанность (coupling).

---

## 1. Четыре способа интеграции

Классификация (по Hohpe & Woolf, «Enterprise Integration Patterns»):

| Способ | Суть | Связанность | Когда |
|--------|------|-------------|-------|
| **File Transfer** | обмен файлами (выгрузка/загрузка) | низкая, но запаздывает | пакетный обмен, legacy, отчёты |
| **Shared Database** | общая БД, через неё «общаются» | **очень высокая** (антипаттерн) | почти никогда (см. ниже) |
| **RPC / Remote Procedure Call** | синхронный вызов чужого API (REST/gRPC) | высокая (временная) | запрос-ответ в реальном времени |
| **Messaging** | асинхронный обмен сообщениями через брокер | низкая | развязка, надёжность, масштаб |

```
File:      Система A → файл → (расписание) → Система B   (медленно, просто)
Shared DB: A ──► [общая БД] ◄── B                        (связаны через схему — плохо)
RPC:       A ──вызов──► B ──ответ──► A                   (быстро, но A зависит от B сейчас)
Messaging: A → [брокер] → B (когда сможет)               (развязка во времени)
```

> Способ интеграции — фундаментальное решение: оно определяет, насколько системы связаны и что будет при сбое/изменении одной из них. Тренд: от файлов/общей БД — к messaging для развязки.

## 2. Sync vs Async

| | Синхронная (RPC) | Асинхронная (Messaging) |
|--|------------------|--------------------------|
| Ожидание | вызывающий ждёт ответ | отправил и продолжил |
| Связанность во времени | высокая (оба должны быть живы) | низкая (получатель обработает позже) |
| Простота | проще (запрос-ответ) | сложнее (брокер, гарантии) |
| Сбой получателя | вызов падает | сообщение ждёт в очереди |
| Когда | нужен немедленный ответ | развязка, пики, фоновые задачи |

> Синхронность удобна (сразу ответ), но создаёт **временную связанность**: A не работает, если B недоступен (каскадные отказы, модуль 25). Асинхронность развязывает: B может быть временно недоступен, сообщение подождёт. Выбор — баланс «нужен ли ответ сейчас» vs «развязка/устойчивость».

## 3. Почему «общая БД» — антипаттерн интеграции

Интеграция через общую БД (несколько систем читают/пишут одни таблицы) кажется простой, но:

- **скрытая связанность через схему** — нельзя изменить таблицу, не сломав другие системы;
- **нет инкапсуляции** — бизнес-логика и инварианты обходятся прямым доступом к данным;
- **неясное владение** — кто отвечает за данные, кто их меняет;
- **блокировки/конкуренция** — системы мешают друг другу на уровне БД.

> Это прямое следствие database per service (модуль 21). Правильно: у данных один владелец, остальные получают их через API/события, а не лезут в чужую БД. «Общая БД» допустима лишь как осознанный временный компромисс с планом ухода.

## 4. Базовые EIP-паттерны

Из каталога Enterprise Integration Patterns — кирпичики messaging-интеграции:

| Паттерн | Назначение |
|---------|------------|
| **Message Channel** | канал, по которому идут сообщения (очередь/топик) |
| **Message** | единица обмена (команда/событие/документ) |
| **Message Router** | направляет сообщение нужному получателю по правилу |
| **Content-Based Router** | маршрутизация по содержимому сообщения |
| **Message Translator** | преобразует формат/структуру между системами |
| **Message Filter** | пропускает только подходящие сообщения |
| **Aggregator** | собирает несколько сообщений в одно |
| **Splitter** | разбивает одно сообщение на несколько |
| **Dead Letter Channel** | канал для необработанных сообщений (DLQ) |

```
Content-Based Router:  заказ ──► [router] ──(если еда)──► Кухня
                                        └──(если товар)──► Склад
Aggregator:  ответы 3 сервисов ──► [aggregator] ──► единый ответ
Translator:  формат партнёра ──► [translator] ──► наш формат (≈ ACL, модуль 24)
```

> EIP — общий словарь интеграции. Аналитику важно узнавать паттерны в задачах: «направить по типу» → content-based router; «собрать ответ из нескольких» → aggregator; «перевести формат» → translator (родственник ACL из DDD). Детали messaging — модуль 32.

## 5. Связанность (coupling)

**Связанность** — степень зависимости систем друг от друга. Цель интеграции — **низкая связанность (loose coupling)**.

Виды связанности, которые вносит интеграция:
- **временная** (temporal) — оба должны работать одновременно (sync RPC);
- **по формату/схеме** (format) — зависимость от структуры данных партнёра;
- **по местоположению** (location) — жёсткие адреса/хосты;
- **по технологии** — общий стек/протокол.

| Способ | Временная | Схемная | Снижают чем |
|--------|-----------|---------|-------------|
| Shared DB | высокая | очень высокая | уйти от неё |
| Sync RPC | высокая | средняя | таймауты, CB, версионирование контрактов |
| Messaging | низкая | средняя | схемы событий, версионирование |

> Низкая связанность = системы можно менять/деплоить/масштабировать независимо. Async (messaging) снижает временную связанность; версионируемые контракты/схемы (модуль 28, 30) — схемную; service discovery — по местоположению. Связь с распределённым монолитом (модуль 21) — это интеграция с высокой связанностью.

## ⚠️ Подводные камни

- **Общая БД как интеграция.** Скрытая связанность через схему; правильно — API/события, один владелец данных.
- **Везде синхронный RPC.** Временная связанность и каскадные отказы; для развязки/устойчивости — async.
- **Async везде «для развязки».** Async сложнее (брокер, гарантии, дедуп); если нужен немедленный ответ — sync проще и уместнее.
- **Файловый обмен без обработки ошибок.** Битый/опоздавший/дублированный файл — продумать формат, расписание, идемпотентность загрузки.
- **Игнорировать связанность.** Интеграция, жёстко завязанная на формат/хост/время партнёра, ломается при его изменениях.
- **Нет translator/ACL для чужого формата.** Чужая модель протекает внутрь; нужен перевод на границе (модуль 24).
- **Нет DLQ для async.** Необрабатываемые сообщения теряются или блокируют поток; нужен dead letter channel (модуль 32).
