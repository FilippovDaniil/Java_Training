# Задача 03 — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Выберите формат данных под канал/задачу для набора кейсов и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Кейсы:
1. Публичный REST API для мобильного и веб-клиента.
2. Высоконагруженное межсервисное взаимодействие (gRPC) с минимальной задержкой.
3. Интеграция с государственной системой, требующей XML + XSD.
4. Поток событий в Kafka с эволюцией схемы и большим объёмом.
5. Конфигурационный файл, который правят руками.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите формат (JSON / XML / Protobuf / Avro) и обоснуйте.
- Учтите: читаемость, объём/скорость, наличие схемы, эволюция, требования партнёра.
- Отметьте, где бинарный формат оправдан, а где навредит (отлаживаемость).
- Сформулируйте правило выбора формата.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Кейс | Формат | Обоснование` + правило выбора.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Публичный REST → JSON (читаемость, повсеместность).
- [ ] Высоконагруженный gRPC → Protobuf (компактность, скорость, строгая схема).
- [ ] Госсистема с XSD → XML (требование партнёра, строгость).
- [ ] Поток Kafka с эволюцией → Avro (компактность + эволюция схемы + реестр).
- [ ] Конфиг руками → JSON/YAML (читаемость).
- [ ] Сформулировано правило выбора (читаемость/совместимость vs производительность/строгость).

## ПОДСКАЗКА
Текстовые (JSON/XML) — читаемы, для веба/отладки/публичных API. Бинарные (Protobuf/Avro) — компактны/быстры, для нагрузки и объёма (gRPC/Kafka), но нечитаемы. XML — когда требует партнёр/нужны XSD/namespaces. Avro — для стримов с эволюцией схемы (реестр схем). Правило: публичное/отлаживаемое → JSON; нагрузка/объём → бинарь; требование партнёра → его формат.
