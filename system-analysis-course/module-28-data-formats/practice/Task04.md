# Задача 04 — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Классифицируйте изменения схемы на ломающие/не ломающие и спланируйте эволюцию без поломки потребителей.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, схема события `OrderCreated` версии 1: `{orderId, clientId, total, items[]}`. Предлагаемые изменения:
1. Добавить опциональное поле `discount` (с дефолтом 0).
2. Переименовать `total` → `amount`.
3. Сделать `clientId` обязательным (был опциональным).
4. Изменить тип `orderId` с числа на строку (UUID).
5. Добавить новое значение в enum статуса.
6. Удалить неиспользуемое поле `legacyCode`.

## ТРЕБОВАНИЯ
- Для каждого изменения определите: ломающее (breaking) или нет.
- Для ломающих предложите безопасную стратегию (новая версия, deprecation, параллельная поддержка).
- Объясните backward/forward совместимость на этих примерах.
- Сформулируйте правило безопасной эволюции.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Изменение | Breaking? | Почему | Безопасная стратегия` + правило эволюции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Добавить опциональное `discount` → не ломающее.
- [ ] Переименовать `total`→`amount` → ломающее (удаление+добавление).
- [ ] Сделать `clientId` обязательным → ломающее.
- [ ] Сменить тип `orderId` → ломающее.
- [ ] Добавить значение enum → потенциально ломающее для старых потребителей (обработка «неизвестно»).
- [ ] Удалить поле → ломающее для тех, кто его читает; безопасно через deprecation.
- [ ] Сформулировано правило (добавлять опционально — ок; удалять/менять/обязательность — ломает → версия).

## ПОДСКАЗКА
Не ломающее: добавить опциональное поле. Ломающее: удалить/переименовать (= удалить+добавить), сменить тип, сделать обязательным. Enum: добавление значения может сломать старых потребителей. Стратегия для ломающих: новая версия + параллельная поддержка + deprecation старой. Backward — новый читатель читает старые данные; forward — старый читает новые. Связь с версионированием API (модуль 30).
