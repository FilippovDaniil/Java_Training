# Задача 07 — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Обоснуйте выбор Avro/Protobuf для event streaming и роль реестра схем.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, поток событий в Kafka: миллионы событий `OrderEvent` в сутки, несколько потребителей, схема будет эволюционировать (новые поля). Сейчас события шлются как JSON — объём большой, нет контроля совместимости схемы.

## ТРЕБОВАНИЯ
- Обоснуйте переход на Avro/Protobuf для стрима (компактность, скорость, строгая схема).
- Объясните роль реестра схем (schema registry): хранение версий, контроль совместимости.
- Покажите, как реестр предотвращает несовместимые изменения (backward/forward).
- Сравните с JSON: что выигрываем и что теряем (читаемость).
- Свяжите с эволюцией схемы (задача 04) и async (модуль 32).

## ФОРМАТ РЕЗУЛЬТАТА
Обоснование формата + роль реестра схем + сравнение с JSON + связь с совместимостью.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Обоснован бинарный формат для стрима (объём/скорость).
- [ ] Объяснена роль schema registry (версии схем, проверка совместимости при публикации).
- [ ] Показано предотвращение несовместимых изменений (backward/forward проверки).
- [ ] Сравнение с JSON (компактность/контроль vs читаемость).
- [ ] Связь с правилами эволюции схемы (задача 04).
- [ ] Связь с async/Kafka (модуль 32).

## ПОДСКАЗКА
Для больших потоков (Kafka) бинарные Avro/Protobuf компактнее и быстрее JSON. Schema Registry хранит версии схем и проверяет совместимость при регистрации новой версии (отклоняет ломающие изменения) — гарантия, что потребители не сломаются. Avro особенно удобен для эволюции. Платим читаемостью (бинарь), выигрываем объём + контроль совместимости. Связь с модулем 32.
