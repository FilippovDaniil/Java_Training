# Задача 08 — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Сравните вложенную и плоскую структуру данных: для сценариев выберите подходящую и обоснуйте trade-offs.

## ВХОДНЫЕ ДАННЫЕ
Сценарии (FoodExpress):
1. Передача карточки заказа целиком клиенту (заказ + клиент + позиции + адрес).
2. Выгрузка заказов в аналитическую систему/CSV для отчётов.
3. Событие в Kafka для потоковой обработки.
4. Конфигурация со сгруппированными по смыслу секциями.

## ТРЕБОВАНИЯ
- Для каждого сценария выберите вложенную или плоскую структуру.
- Обоснуйте через: цельность объекта, табличность, простоту парсинга/запросов.
- Покажите пример вложенного и плоского представления одного заказа.
- Отметьте антипаттерн «поля-через-индекс» (item_1, item_2) и почему это не плоскость, а ошибка.
- Сформулируйте правило выбора.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Сценарий | Структура | Обоснование` + пример вложенно/плоско + правило.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Карточка заказа клиенту → вложенная (цельный объект, document).
- [ ] Выгрузка в аналитику/CSV → плоская (табличность).
- [ ] Событие Kafka → зависит от обработки (часто умеренно плоское/типизированное).
- [ ] Конфиг → вложенная (группировка по секциям).
- [ ] Отмечен антипаттерн полей-через-индекс (нужен массив).
- [ ] Сформулировано правило (цельность → вложенно; табличность → плоско; список → массив, не индексы).

## ПОДСКАЗКА
Вложенность — для цельного объекта (карточка, document-модель, модуль 19) и группировки (конфиг). Плоскость — для табличных/аналитических данных (CSV, отчёты, потоки). Список всегда → массив, никогда `item_1, item_2` (это и не плоскость, а нарушение, родственное 1NF, модуль 17). Правило: цельность → вложенно; таблица → плоско; коллекция → массив.
