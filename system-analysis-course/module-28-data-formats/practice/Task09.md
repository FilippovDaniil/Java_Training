# Задача 09 — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Спроектируйте маппинг между форматами XML↔JSON для интеграции и обработайте краевые случаи.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** получает рецепты от ГИС в XML, а внутри работает с JSON. Особенности XML от ГИС: атрибуты (`<recipe id="..." type="...">`), namespaces, повторяющиеся элементы (`<position>` несколько раз), отсутствие необязательных элементов, всё текстом (типы — по XSD).

## ТРЕБОВАНИЯ
- Спроектируйте правила маппинга XML→JSON (атрибуты, элементы, массивы, типы).
- Обработайте краевые случаи: атрибут vs вложенный элемент, namespace, повторяющийся элемент → массив, отсутствие элемента → отсутствие/null.
- Покажите пример: фрагмент XML и результат JSON.
- Объясните, почему маппинг не «один-в-один» и где теряется информация.
- Свяжите с translator/ACL (модули 27/24).

## ФОРМАТ РЕЗУЛЬТАТА
Правила маппинга (таблица) + пример XML→JSON + разбор краевых случаев.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Правила маппинга: атрибуты XML → поля JSON, элементы → поля/объекты.
- [ ] Повторяющиеся элементы → JSON-массив.
- [ ] Namespace обработан (отброшен/сохранён осознанно).
- [ ] Типы приведены (текст XML → число/булево по XSD).
- [ ] Отсутствие элемента → отсутствие поля/null (осознанно).
- [ ] Объяснено, почему маппинг не «один-в-один» (XML богаче: атрибуты/namespaces).
- [ ] Связь с translator/ACL.

## ПОДСКАЗКА
XML богаче JSON структурно: атрибуты, namespaces, разница элемент/атрибут — прямого аналога в JSON нет. Правила: атрибут → поле, повторяющийся элемент → массив, текст+XSD → типизированное значение, отсутствие → null/нет поля. Маппинг не «один-в-один» — фиксируйте решения по краевым случаям. Это работа Message Translator / ACL на границе (модули 27/24).
