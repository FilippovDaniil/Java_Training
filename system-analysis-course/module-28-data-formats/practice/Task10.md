# Задача 10 (мини-проект) — Модуль 28: Форматы данных и схемы

## ЗАДАНИЕ
Спроектируйте контракты данных (JSON Schema) ключевых сообщений FoodExpress с продуманной эволюцией и выбором форматов. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, ключевые сообщения: (1) REST-запрос/ответ «создать заказ»; (2) событие `OrderCreated` в Kafka; (3) интеграция с платёжным провайдером; (4) выгрузка заказов в аналитику. Требования: чёткие контракты, корректные типы, безопасная эволюция, разумный выбор форматов под каналы.

## ТРЕБОВАНИЯ
Сдайте пакет контрактов:
1. **JSON Schema для создания заказа** (REST) — типы, required, ограничения, валидные/невалидные примеры.
2. **Схема события `OrderCreated`** — формат (JSON/Avro) + обоснование, правила эволюции (что можно добавлять).
3. **Выбор форматов по каналам** — REST (JSON), Kafka (Avro/JSON), провайдер (его формат), аналитика (плоско/CSV).
4. **Эволюция и совместимость** — какие изменения safe, какие breaking; стратегия версионирования (связь с модулем 30).
5. **Корректность типов** — деньги, даты, статусы-enum, массивы; без антипаттернов.
6. **Маппинг к внешним** — где нужен translator/ACL (провайдер, аналитика).

## ФОРМАТ РЕЗУЛЬТАТА
JSON Schema создания заказа + схема события + таблица «канал → формат» + правила эволюции + перечень корректных типов + точки маппинга.

## КРИТЕРИИ ПРИЁМКИ
- [ ] JSON Schema создания заказа валидна, с типами/required/ограничениями + примеры.
- [ ] Схема события OrderCreated + обоснование формата + правила эволюции.
- [ ] Форматы выбраны под каналы (REST/Kafka/провайдер/аналитика) обоснованно.
- [ ] Классификация изменений safe/breaking + стратегия версионирования.
- [ ] Корректные типы (деньги не float, ISO-даты, enum-статусы, массивы) без антипаттернов.
- [ ] Отмечены точки маппинга/translator к внешним системам.
- [ ] Контракты согласованы между собой (одни и те же сущности).

## ПОДСКАЗКА
Синтез: JSON+типы (зад.01), JSON Schema (02), выбор формата (03), эволюция (04), XML/XSD (05), ошибки контракта (06), Avro/реестр (07), вложенность/плоскость (08), маппинг (09). Главные проверки middle+: схема как контракт, корректные типы, безопасная эволюция (добавлять опционально), формат под канал, translator к внешним. Эти контракты — прямой вход в OpenAPI (модуль 30) и события (модуль 32).
