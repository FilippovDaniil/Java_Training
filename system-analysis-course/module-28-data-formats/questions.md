## Ключ (эталонные ответы)

1. JSON (текстовый), XML (текстовый), Protobuf (бинарный), Avro (бинарный).
2. **JSON** — для веба, REST API, конфигов, читаемости и повсеместности. **XML** — когда нужна строгость, namespaces, валидация XSD, или этого требует партнёр (гос/банки, SOAP, документооборот).
3. При больших объёмах и требованиях к скорости/компактности: межсервисное взаимодействие (gRPC/Protobuf), потоки событий (Kafka/Avro), мобильный трафик. Ценой читаемости.
4. Компактностью, встроенной поддержкой эволюции схемы и работой со schema registry (контроль совместимости версий).
5. **Схема** — формальное описание структуры данных (поля, типы, обязательность, ограничения). Нужна как контракт и для валидации.
6. JSON — **JSON Schema**; XML — **XSD** (и DTD); Protobuf — `.proto`; Avro — Avro schema.
7. Типы полей (`type`), обязательность (`required`), ограничения (`minimum`, `maximum`, `enum`, `minItems`, `pattern`), вложенные объекты и массивы.
8. Проверка соответствия данных схеме. Ловит структурные ошибки (нет обязательного поля, неверный тип, нарушение ограничения) на границе, до бизнес-логики.
9. Новый **читатель** (новая схема) корректно читает данные, созданные по **старой** схеме.
10. Старый **читатель** (старая схема) корректно читает данные, созданные по **новой** схеме (игнорируя новые поля).
11. Добавление **опционального** поля (с дефолтом); часто — расширение ограничений. Главное — не удалять и не менять существующее.
12. Удаление/переименование поля, смена типа, превращение опционального поля в обязательное, сужение ограничений (например, новый строгий enum).
13. Старые потребители не знают нового значения и могут на нём упасть/повести себя некорректно; нужна обработка «неизвестного значения».
14. Robustness Principle: «будь консервативен в том, что отправляешь, и либерален в том, что принимаешь» — шли строго по схеме, принимай терпимо (игнорируй лишнее).
15. Чтобы добавление поля производителем не ломало потребителей (forward-совместимость): незнакомое поле игнорируется, а не вызывает ошибку.
16. Float — двоичная плавающая точка, теряет точность на десятичных дробях (копейки «плывут»). Использовать строку или целое в минимальных единицах (копейки).
17. В стандартном формате ISO 8601 с указанием таймзоны (например, `2026-06-25T10:00:00Z`), а не в произвольном локальном формате.
18. Это коллекция, замаскированная под отдельные поля; не масштабируется, ломает схему при росте числа элементов, усложняет обработку. Нужен массив (родственно нарушению 1NF).
19. **Вложенная** — для цельного объекта (карточка, document-модель) и группировки по смыслу. **Плоская** — для табличных/аналитических данных (CSV, отчёты, потоки).
20. XML богаче структурно (атрибуты vs элементы, namespaces, всё текстом без типов), нет прямого соответствия в JSON; нужны явные правила для повторяющихся элементов (→ массив), отсутствия (→ null), типизации.
21. **Schema registry** хранит версии схем и проверяет совместимость при регистрации новой версии (отклоняет ломающие изменения), гарантируя, что потребители не сломаются. Используется в Kafka с Avro/Protobuf.
22. Схема **является** контрактом данных: стороны договариваются о ней, по ней валидируют, генерируют код/документацию; для API схемы оформляются в спецификации (OpenAPI, модуль 30).
23. **XSD (XML Schema Definition)** — язык описания структуры и типов XML-документов для их валидации.
24. Только не ломающими изменениями (добавлять опциональные поля); ломающие — через новую версию с параллельной поддержкой и deprecation; проверять backward/forward совместимость (реестр схем/тесты).
25. Деньги как float; даты в произвольном формате; статус магическим числом; поля-через-индекс вместо массива; булево строкой; несколько значений в одном поле; формат без схемы; ломающие изменения без версионирования; парсер, падающий от незнакомых полей.
