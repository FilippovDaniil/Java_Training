# Модуль 28 — Форматы данных и схемы

## Зачем этот модуль

Интеграции (модуль 27) обмениваются **данными** — значит, нужен формат и контракт на их структуру. Какой формат выбрать (JSON, XML, Protobuf, Avro), как зафиксировать структуру (JSON Schema, XSD) и как менять её, **не ломая** потребителей (эволюция схемы) — ежедневные задачи аналитика при проектировании API и событий.

На собеседовании middle+ обязательно: сравнение форматов и когда что, что такое схема и валидация, обратная/прямая совместимость, какие изменения схемы ломающие, выбор формата под канал.

---

## 1. Форматы данных

| Формат | Вид | Сильно в | Слабо | Когда |
|--------|-----|----------|-------|-------|
| **JSON** | текстовый | читаемость, веб, повсеместность | объём, нет схемы «из коробки» | REST API, конфиги, веб |
| **XML** | текстовый | строгость, namespaces, XSD, зрелость | многословность | гос/банки, SOAP, документооборот |
| **Protobuf** | бинарный | компактность, скорость, строгая схема | нечитаем глазами | gRPC, межсервисное, мобайл |
| **Avro** | бинарный | компактность, эволюция схемы | нужен реестр схем | event streaming (Kafka) |

```
JSON:     {"id":1,"name":"Ролл","price":300}            (текст, читаемо)
XML:      <dish><id>1</id><name>Ролл</name></dish>      (текст, многословно)
Protobuf: бинарь по .proto-схеме                         (компактно, быстро)
Avro:     бинарь + схема (часто в реестре)               (для стримов)
```

> Текстовые (JSON/XML) — читаемы, удобны для отладки и веба. Бинарные (Protobuf/Avro) — компактны и быстры, нужны при больших объёмах/нагрузке (gRPC, Kafka). Выбор — баланс читаемость/совместимость vs производительность/строгость.

---

## 2. Схемы и валидация

**Схема** — формальное описание структуры данных: какие поля, типы, обязательность, ограничения. Позволяет **валидировать** данные и служит **контрактом**.

| Формат | Язык схемы |
|--------|------------|
| JSON | **JSON Schema** |
| XML | **XSD** (XML Schema Definition), DTD |
| Protobuf | `.proto` (схема встроена в формат) |
| Avro | Avro schema (JSON-описание) |

Пример JSON Schema (фрагмент):
```json
{
  "type": "object",
  "required": ["id", "items"],
  "properties": {
    "id": {"type": "integer"},
    "total": {"type": "number", "minimum": 0},
    "status": {"type": "string", "enum": ["created","paid","delivered"]},
    "items": {"type": "array", "minItems": 1}
  }
}
```

> Схема — это **контракт данных**: на ней договариваются стороны, по ней валидируют вход, генерируют код/документацию. JSON Schema задаёт типы, обязательность (`required`), ограничения (`minimum`, `enum`, `minItems`), вложенность. Валидация по схеме ловит ошибки на границе, до бизнес-логики.

---

## 3. Эволюция схемы и совместимость

Схемы меняются. Главное — **не сломать существующих потребителей**.

| Совместимость | Кто может меняться | Правило |
|---------------|--------------------|---------| 
| **Backward (обратная)** | новый **читатель** читает старые данные | можно удалять опциональные/добавлять с дефолтом |
| **Forward (прямая)** | старый **читатель** читает новые данные | можно добавлять опциональные поля |
| **Full** | оба направления | пересечение правил |

**Не ломающие (safe) изменения:**
- добавить **опциональное** поле (с дефолтом);
- добавить значение в enum (осторожно — старые потребители его не знают);
- расширить ограничение.

**Ломающие (breaking) изменения:**
- удалить/переименовать поле;
- сделать опциональное поле обязательным;
- изменить тип поля;
- сузить ограничение (было `string`, стало `enum` из 3 значений).

```
Safe:  добавить "discount" (опционально, default 0)  → старые клиенты игнорируют
Break: переименовать "sum" → "total"                  → старые клиенты сломались
```

> Правило: **добавлять — можно, удалять/менять/делать обязательным — ломает**. Старые потребители не должны падать от новых данных (forward), новые — читать старые (backward). Это основа версионирования API (модуль 30).

## 4. Принцип Postel (robustness)

«Будь консервативен в том, что отправляешь, и либерален в том, что принимаешь» (Robustness Principle).

- **Потребитель:** игнорируй неизвестные поля (не падай от лишнего) — это даёт forward-совместимость.
- **Производитель:** шли строго по схеме, не присылай лишнего без нужды.

> Практический вывод: парсеры должны **игнорировать незнакомые поля**, а не падать. Тогда добавление поля производителем не ломает потребителей. Это ключ к безопасной эволюции.

---

## 5. Вложенность vs плоскость (trade-offs)

| | Вложенная структура | Плоская структура |
|--|---------------------|-------------------|
| Читаемость | группировка по смыслу | проще для таблиц/CSV |
| Гибкость | вложенные объекты/массивы | повторение полей |
| Запросы/маппинг | сложнее парсить пути | проще фильтровать |
| Когда | document-модель, богатые объекты | аналитика, экспорт, потоки |

```
Вложенно:  {"order":{"client":{"name":"..."},"items":[...]}}
Плоско:    {"order_id":1,"client_name":"...","item_1":"...","item_2":"..."}  ← хуже масштабируется
```

> Вложенность удобна для передачи цельного объекта (карточка, document — модуль 19). Чрезмерная вложенность усложняет маппинг и запросы. Плоскость — для табличных/потоковых данных. «Поля через индекс» (item_1, item_2) — антипаттерн (нарушает и 1NF, модуль 17).

## 6. Маппинг между форматами

При интеграции часто нужен перевод XML↔JSON, формат партнёра ↔ внутренний (это Message Translator/ACL, модули 27/24). Сложности маппинга:
- **атрибуты vs элементы** в XML (нет прямого аналога в JSON);
- **namespaces** XML;
- **типы** (всё строка в XML без XSD; типизация в JSON);
- **массивы** (повторяющиеся элементы XML ↔ JSON array);
- **null/отсутствие** поля.

> Маппинг — не «один-в-один»: XML богаче структурно (атрибуты, namespaces). Аналитик фиксирует правила соответствия полей и обработку краевых случаев (отсутствие/null, повторяющиеся элементы). Это часть контракта интеграции.

## ⚠️ Подводные камни

- **Формат без схемы.** JSON «как есть» без JSON Schema — нет контракта, ошибки уходят в бизнес-логику. Фиксировать схему.
- **Ломающие изменения схемы.** Удаление/переименование/смена типа/обязательность ломают потребителей; добавлять опционально.
- **Парсер падает от незнакомого поля.** Нарушает forward-совместимость; игнорировать неизвестные поля (принцип Postel).
- **Деньги как float в JSON.** Потеря точности; строка или целое в минимальных единицах (копейки) + схема.
- **enum без запаса.** Добавление значения enum ломает старых потребителей, которые его не знают; продумывать обработку «неизвестно».
- **Чрезмерная вложенность / поля-через-индекс.** Усложняет маппинг; `item_1, item_2…` — антипаттерн (массив).
- **Бинарный формат для отлаживаемого веб-API.** Protobuf нечитаем глазами; для публичного REST обычно JSON.
- **Маппинг XML↔JSON «один-в-один».** Игнор атрибутов/namespaces/типов → ошибки; фиксировать правила.
