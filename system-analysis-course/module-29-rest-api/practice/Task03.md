# Задача 03 — Модуль 29: REST API

## ЗАДАНИЕ
Спроектируйте идемпотентность POST через idempotency key для создания заказа/оплаты.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, `POST /orders` и `POST /payments`. Проблема: при сетевом сбое клиент не получил ответ и повторяет POST — создаётся дубль заказа / двойное списание.

## ТРЕБОВАНИЯ
- Опишите, почему POST не идемпотентен и зачем нужен ключ.
- Спроектируйте механизм idempotency key: заголовок, хранение ключа+результата, поведение при повторе.
- Покажите, что возвращается при повторном запросе с тем же ключом (тот же результат, тот же статус).
- Укажите срок жизни ключа и где хранить.
- Свяжите с распределённой устойчивостью (модуль 25) и retry (модуль 23).

## ФОРМАТ РЕЗУЛЬТАТА
Схема обработки POST с idempotency key (первый/повтор) + заголовок + хранение + TTL.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено, почему POST не идемпотентен.
- [ ] Клиент передаёт `Idempotency-Key` в заголовке.
- [ ] Первый запрос: выполнить, сохранить результат под ключом.
- [ ] Повтор с тем же ключом: вернуть прежний результат (тот же id/статус), без дубля.
- [ ] Указаны хранение ключа+результата и TTL.
- [ ] Связь с retry/устойчивостью (модули 23, 25).

## ПОДСКАЗКА
POST создаёт новое каждый раз → повтор = дубль. Idempotency key: клиент генерирует уникальный ключ, шлёт в заголовке; сервер запоминает ключ→результат; при повторе возвращает сохранённый результат, не выполняя операцию. Хранить с TTL. Это делает безопасными ретраи (модули 23, 25). Для оплаты — обязательно.
