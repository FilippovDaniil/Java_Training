# Задача 05 — Модуль 29: REST API

## ЗАДАНИЕ
Спроектируйте формат ошибок по RFC 7807 (problem+json) для API.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, ошибочные ситуации:
1. Заказ не найден (404).
2. Невалидное тело запроса — несколько полей не прошли валидацию (422).
3. Недопустимый переход статуса (отмена доставленного) (409).
4. Превышен лимит запросов (429).

## ТРЕБОВАНИЯ
- Спроектируйте структуру ответа об ошибке по RFC 7807 (`type`, `title`, `status`, `detail`, `instance`).
- Для случая 2 (валидация) расширьте формат списком невалидных полей.
- Приведите JSON-примеры для всех четырёх случаев.
- Объясните, зачем единый формат ошибок (предсказуемость для клиента).
- Укажите content-type `application/problem+json`.

## ФОРМАТ РЕЗУЛЬТАТА
4 JSON-примера ошибок (RFC 7807) + пояснение полей + расширение для валидации.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все примеры содержат `type`, `title`, `status`, `detail`, `instance`.
- [ ] 404, 422, 409, 429 — корректные статусы в `status`.
- [ ] Для 422 добавлен список невалидных полей (расширение).
- [ ] Указан `Content-Type: application/problem+json`.
- [ ] Объяснена ценность единого формата (клиент парсит одинаково).
- [ ] JSON валиден.

## ПОДСКАЗКА
RFC 7807: `type` (URI типа ошибки), `title` (название), `status` (HTTP-код), `detail` (детали случая), `instance` (ссылка на ресурс). Можно расширять: для 422 добавить `errors: [{field, message}]`. Content-Type — `application/problem+json`. Единый формат = клиент обрабатывает ошибки предсказуемо, без разнобоя.
