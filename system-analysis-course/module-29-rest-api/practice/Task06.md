# Задача 06 — Модуль 29: REST API

## ЗАДАНИЕ
Найдите нарушения REST в готовом API и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент API FoodExpress с проблемами:
```
GET    /getOrders?clientId=77
POST   /orders/1024/cancel
GET    /order/1024                      → 200 {"error":"not found"}  (когда нет)
POST   /updateOrderStatus               body: {"id":1024,"status":"paid"}
DELETE /orders/1024                      → 200 {"deleted":true}
GET    /orders                           → отдаёт все 2 млн заказов сразу
POST   /orders                           → 200 (при создании)
GET    /clients/77/orders/1024/items/5/options/3/values/1
```

## ТРЕБОВАНИЯ
- Найдите каждое нарушение (глаголы в URI, неверные статусы, отсутствие пагинации, глубокая вложенность, единственное число).
- Объясните, чем плохо.
- Предложите исправленный вариант (метод/URI/статус).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Нарушение | Чем плохо | Исправление`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] `/getOrders` → глагол в URI → `GET /orders?clientId=77`.
- [ ] `/orders/{id}/cancel` → action-глагол → `PATCH /orders/{id}` `{"status":"cancelled"}`.
- [ ] `200 {"error":"not found"}` → должно быть 404.
- [ ] `POST /updateOrderStatus` → глагол + id в теле → `PATCH /orders/{id}`.
- [ ] `DELETE → 200 {deleted:true}` → должно быть 204 No Content.
- [ ] `GET /orders` без пагинации → добавить пагинацию/limit.
- [ ] `POST /orders → 200` → должно быть 201 + Location.
- [ ] Глубокая вложенность → сократить (≤ 2 уровня).

## ПОДСКАЗКА
Типовые нарушения REST: глаголы в URI (createOrder/cancel/updateStatus); единственное число; статусы (200 вместо 404/201/204); смена статуса action-эндпоинтом вместо PATCH; коллекция без пагинации; вложенность > 2 уровней. Исправления — по конвенциям: существительные мн.ч., методы по семантике, корректные статусы, PATCH для статуса, пагинация.
