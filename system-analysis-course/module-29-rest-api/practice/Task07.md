# Задача 07 — Модуль 29: REST API

## ЗАДАНИЕ
Спроектируйте смену статуса через PATCH (vs action-эндпоинт) с проверкой допустимости перехода.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, заказ со статусами (state machine, модуль 13): Создан → Оплачен → Принят → Готовится → В доставке → Доставлен; отмена возможна до «В доставке». Нужно: отменить заказ, перевести в доставку, подтвердить рестораном.

## ТРЕБОВАНИЯ
- Спроектируйте смену статуса через `PATCH /orders/{id}` с телом `{"status":"...","reason":"..."}`.
- Покажите, как сервер проверяет допустимость перехода (по state machine) и возвращает 409 при недопустимом.
- Сравните с action-эндпоинтами (`POST /orders/{id}/cancel`) — плюсы/минусы.
- Укажите, кто имеет право на переход (связь с авторизацией, модуль 34) и статус 403.
- Приведите примеры: успешный переход и недопустимый (отмена доставленного → 409).

## ФОРМАТ РЕЗУЛЬТАТА
Дизайн PATCH-смены статуса + проверка перехода + примеры (успех/409) + сравнение с action-эндпоинтом.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Смена статуса через `PATCH /orders/{id}` с телом `{"status":...}`.
- [ ] Сервер проверяет допустимость перехода (state machine) → 409 при недопустимом.
- [ ] Право на переход проверяется (403 при отсутствии прав).
- [ ] Приведены примеры успеха и 409 (отмена доставленного).
- [ ] Сравнение PATCH vs action-эндпоинт с обоснованием выбора PATCH.
- [ ] Связь со state machine (модуль 13) и авторизацией (модуль 34).

## ПОДСКАЗКА
Конвенция: смена статуса = `PATCH /orders/{id}` с `{"status":"cancelled","reason":"..."}`, а не `POST /orders/{id}/cancel` (глагол в URL). Сервер проверяет допустимость по state machine (модуль 13): недопустимый переход → 409 Conflict. Право инициировать переход → авторизация (модуль 34), иначе 403. Action-эндпоинты оставляют для сложных операций, не сводимых к смене атрибута.
