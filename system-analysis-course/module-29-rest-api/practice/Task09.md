# Задача 09 — Модуль 29: REST API

## ЗАДАНИЕ
Спроектируйте вложенные ресурсы (макс 2 уровня) для MedConnect и решите, где вкладывать, а где ссылаться.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, сущности: пациенты, врачи, расписания/слоты (у врача), записи/консультации (у пациента и врача), рецепты (по консультации), позиции рецепта.

## ТРЕБОВАНИЯ
- Спроектируйте URI с вложенностью ≤ 2 уровней (слоты врача, записи пациента, позиции рецепта).
- Определите, где вложенность оправдана (принадлежность), а где лучше ресурс верхнего уровня + фильтр.
- Покажите чтение коллекций через вложенный URI и через фильтр (`?doctorId=`).
- Объясните, почему глубокая вложенность (>2) — проблема.
- Учтите 152-ФЗ: доступ к медданным (связь с модулем 34).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица URI (вложенные/плоские) + обоснование выбора + примеры + замечание о доступе.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Слоты врача: `/doctors/{id}/slots` (вложенность 2).
- [ ] Записи пациента: `/patients/{id}/appointments` или `/appointments?patientId=`.
- [ ] Позиции рецепта: `/prescriptions/{id}/items` (2 уровня), не глубже.
- [ ] Объяснено, где вложенность, где плоский URI + фильтр.
- [ ] Объяснена проблема вложенности > 2 уровней.
- [ ] Отмечен контроль доступа к медданным (152-ФЗ, модуль 34).

## ПОДСКАЗКА
Вложенность отражает принадлежность: `/doctors/{id}/slots`, `/prescriptions/{id}/items` (2 уровня — ок). Глубже 2 — неудобно: лучше ресурс верхнего уровня + фильтр (`/appointments?patientId=77`). Одни и те же данные можно отдавать и так, и так. Медданные — строгий контроль доступа (152-ФЗ, модуль 34): кто может читать чужие записи/рецепты.
