# Задача 10 (мини-проект) — Модуль 29: REST API

## ЗАДАНИЕ
Спроектируйте полный REST API оформления и обработки заказа FoodExpress: ресурсы, методы, статусы, идемпотентность, смена статуса, пагинация/фильтры, ошибки. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, поток заказа: клиент создаёт заказ, оплачивает, отслеживает статус (Создан→Оплачен→Принят→Готовится→В доставке→Доставлен), может отменить (до «В доставке»), смотрит историю заказов. Есть мобильный и веб-клиент.

## ТРЕБОВАНИЯ
Сдайте дизайн API:
1. **Ресурсы и URI** — заказы, позиции, оплаты, история; существительные, ≤ 2 уровня.
2. **Методы и статусы** — CRUD + создание (201+Location), удаление (204), ошибки (404/422/409).
3. **Идемпотентность** — idempotency key для создания заказа и оплаты.
4. **Смена статуса** — PATCH с проверкой допустимости перехода (state machine) → 409.
5. **Список/история** — пагинация (offset/cursor — выбрать), фильтры, сортировка, проекция.
6. **Ошибки** — формат RFC 7807 (problem+json), примеры для 404/422/409.
7. **Уровень зрелости** — целиться в уровень 2 Ричардсона; отметить, где мог бы помочь HATEOAS.

## ФОРМАТ РЕЗУЛЬТАТА
Спецификация API (таблица эндпоинтов: метод/URI/статусы/тело) + примеры ошибок RFC 7807 + дизайн пагинации + механизм идемпотентности и смены статуса.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ресурсы/URI: существительные мн.ч., без глаголов, ≤ 2 уровня.
- [ ] Методы и статусы корректны (201+Location, 204, 404/422/409/401/403).
- [ ] Idempotency key для создания заказа/оплаты.
- [ ] Смена статуса через PATCH + проверка перехода (409 при недопустимом).
- [ ] Список/история: пагинация (выбор offset/cursor обоснован) + фильтры/сортировка.
- [ ] Ошибки в формате RFC 7807 с примерами.
- [ ] Целевой уровень зрелости (2) указан; отмечена роль HATEOAS.

## ПОДСКАЗКА
Синтез: ресурсы (зад.01), методы/статусы (02), идемпотентность (03), пагинация (04), ошибки (05), нарушения REST избегать (06), PATCH-статус (07), фильтры (08), вложенность (09). Главные проверки middle+: предсказуемый ресурсный дизайн, корректные методы/статусы, идемпотентность критичных POST, PATCH-смена статуса с проверкой перехода, единый формат ошибок. Это прямой вход в OpenAPI-спецификацию (модуль 30).
