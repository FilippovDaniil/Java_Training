## Ключ (эталонные ответы)

1. **REST** — архитектурный стиль API поверх HTTP. Принципы: ресурсы с URI, единый интерфейс (HTTP-методы), stateless, представления (JSON), опционально HATEOAS.
2. **Ресурс** — сущность (заказ, клиент). **URI** — его адрес: существительное во множественном числе (`/orders`, `/clients/{id}`).
3. Действие задаётся HTTP-методом, а не URL. Глагол в URI (`/createOrder`) дублирует семантику метода и ломает единообразие.
4. Множественное число (`/orders`, `/clients`): URI обозначает коллекцию ресурсов, отдельный — по id (`/orders/{id}`). Единообразие.
5. Максимум 2 уровня (`/clients/{id}/orders`). Глубже — неудобно; вместо этого ресурс верхнего уровня + фильтр.
6. Create → POST, Read → GET, Update (полное) → PUT, Update (частичное) → PATCH, Delete → DELETE.
7. **201 Created** с заголовком **Location**, указывающим URI созданного ресурса.
8. **204 No Content** (успех без тела ответа).
9. **404 Not Found**. Не 200, потому что 200 означает успех и наличие ресурса — это вводит клиента в заблуждение.
10. **400 Bad Request** — некорректный синтаксис/формат. **422 Unprocessable Entity** — формат верный, но нарушено бизнес-правило/валидация.
11. **401 Unauthorized** — не аутентифицирован (нет/неверный токен). **403 Forbidden** — аутентифицирован, но нет прав.
12. При конфликте с текущим состоянием ресурса: недопустимый переход статуса (по state machine), конкурентное изменение, нарушение уникальности.
13. Идемпотентны: GET, PUT, DELETE. Не идемпотентен: POST (и обычно PATCH).
14. Через **idempotency key**: клиент шлёт уникальный ключ в заголовке, сервер при повторе возвращает прежний результат, не создавая дубль.
15. Через **PATCH** `/orders/{id}` с телом `{"status":"..."}` (смена атрибута ресурса), а не `POST /orders/{id}/cancel`. Сервер проверяет допустимость перехода → 409 при недопустимом.
16. **Offset/limit** — пропускает N строк (номера страниц, прыжки), но медленный на больших данных и нестабилен при вставках. **Cursor** — указатель на позицию (keyset), стабилен и быстр, но без прыжков на произвольную страницу.
17. Для больших и/или часто изменяющихся коллекций, бесконечных лент, где важны стабильность и производительность (не нужны прыжки на страницу).
18. Через query-параметры: фильтр (`?status=delivered&minTotal=500`), сортировка (`?sort=-createdAt`), проекция (`?fields=id,total`). Разрешать только допустимые поля.
19. **RFC 7807 (Problem Details)** — стандартный формат ошибок (`application/problem+json`): поля `type`, `title`, `status`, `detail`, `instance` (можно расширять).
20. Чтобы клиент обрабатывал ошибки предсказуемо и единообразно, без разнобоя (`{"error"}` vs `{"message"}`); упрощает интеграцию и обработку.
21. Модель зрелости REST: 0 — RPC поверх HTTP (один эндпоинт); 1 — ресурсы (разные URI); 2 — + методы и статусы по назначению (практический REST); 3 — + HATEOAS.
22. **HATEOAS** — гипермедиа: ответы содержат ссылки на доступные связанные действия/ресурсы, чтобы клиент «навигировал» по API (уровень 3 Ричардсона).
23. Сервер не хранит состояние клиента между запросами (состояние — в токене/хранилище). Даёт горизонтальную масштабируемость и отказоустойчивость (любой узел обработает запрос).
24. Большая коллекция перегружает сеть/память клиента и сервера, медленно работает; всегда нужна пагинация с limit.
25. Глаголы в URI; единственное число; неверные статусы (200 вместо 404/201/204); смена статуса action-эндпоинтом вместо PATCH; коллекция без пагинации; глубокая вложенность; разнобой в формате ошибок; stateful-сервер.
