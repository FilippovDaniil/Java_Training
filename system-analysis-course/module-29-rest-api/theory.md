# Модуль 29 — REST API

## Зачем этот модуль

Способ интеграции выбран, формат и схемы заданы (27–28). Самый распространённый стиль API для синхронного взаимодействия — **REST**. Аналитик проектирует REST API постоянно: ресурсы, URI, методы, статусы, ошибки, пагинацию. Хорошо спроектированный REST — это предсказуемый контракт; плохой (глаголы в URL, неверные статусы) — источник путаницы и багов.

На собеседовании middle+ обязательно: принципы REST, ресурсная модель и именование URI, выбор методов и статусов, идемпотентность, PATCH vs action-эндпоинт, пагинация (offset vs cursor), формат ошибок (RFC 7807), уровни Ричардсона.

---

## 1. Принципы REST

**REST (Representational State Transfer)** — архитектурный стиль для API поверх HTTP. Ключевые принципы:

| Принцип | Суть |
|---------|------|
| **Ресурсы** | всё — ресурс, у каждого свой URI (существительное) |
| **Единый интерфейс** | стандартные HTTP-методы (GET/POST/PUT/PATCH/DELETE) |
| **Stateless** | сервер не хранит состояние клиента между запросами |
| **Представления** | ресурс отдаётся в формате (обычно JSON) |
| **HATEOAS** | ответы содержат ссылки на связанные действия (высший уровень зрелости) |

> REST — про **ресурсы и операции над ними** стандартными методами, а не про «вызов функций». Глаголы выражаются HTTP-методами, а не в URL. Stateless (модуль 26) — основа масштабируемости.

---

## 2. Ресурсы и URI

- **Ресурс** — сущность (заказ, клиент, блюдо). **URI** — его адрес.
- **Существительные во множественном числе:** `/orders`, `/clients`, `/dishes`.
- **Без глаголов в URI:** не `/createOrder`, `/getOrder` — действие задаёт метод.
- **Иерархия:** `/clients/{id}/orders` — заказы клиента (вложенность **максимум 2 уровня**).
- **Идентификатор:** `/orders/{orderId}`.

```
✅ GET    /orders/1024            (получить заказ)
✅ POST   /orders                 (создать заказ)
✅ GET    /clients/77/orders      (заказы клиента — вложенность 2 уровня)
❌ GET    /getOrder?id=1024       (глагол в URI)
❌ POST   /orders/1024/cancel     (глагол-действие — обычно PATCH статуса)
❌ /clients/77/orders/1024/items/5/options/3   (слишком глубокая вложенность)
```

> Правило: URI — это **существительные-ресурсы**, операции — методы. Множественное число, вложенность ≤ 2 уровней, нет глаголов. Это соглашение делает API предсказуемым.

---

## 3. Методы и статусы (CRUD)

| Операция | Метод | Успех | Идемпотентен |
|----------|-------|-------|--------------|
| Создать | `POST /orders` | **201 Created** (+ Location) | нет |
| Читать | `GET /orders/{id}` | 200 OK | да |
| Полная замена | `PUT /orders/{id}` | 200 / 204 | да |
| Частичное изменение | `PATCH /orders/{id}` | 200 / 204 | обычно нет |
| Удалить | `DELETE /orders/{id}` | **204 No Content** | да |

Ключевые статусы (см. модуль 23):
- **201** — создан (с `Location`); **204** — успех без тела; **200** — успех с телом.
- **400** — синтаксис; **422** — валиден, но бизнес-правило; **404** — нет ресурса.
- **401** — не аутентифицирован; **403** — нет прав; **409** — конфликт состояния; **429** — лимит.

> Корректные методы и статусы — это контракт. Создание → 201 (+Location), удаление → 204, «ресурс не найден» → 404 (не 200 с пустым телом). Различение 401/403, 400/422, 409 — частые вопросы.

---

## 4. Идемпотентность в REST

- **GET, PUT, DELETE** — идемпотентны: повтор безопасен.
- **POST** — не идемпотентен: повтор создаёт дубль → **idempotency key** (модуль 25): клиент шлёт заголовок `Idempotency-Key`, сервер при повторе возвращает прежний результат.

```
POST /payments   Idempotency-Key: abc-123
повтор с тем же ключом → тот же результат, без второго списания
```

> Идемпотентность важна для надёжных повторов при сетевых сбоях (модули 23, 25). Для критичных POST (оплата, создание заказа) — idempotency key обязателен.

---

## 5. Смена статуса: PATCH vs action-эндпоинт

Как «отменить заказ»? Два подхода:

| Подход | Пример | Оценка |
|--------|--------|--------|
| **PATCH статуса** | `PATCH /orders/{id}` `{"status":"CANCELLED","reason":"..."}` | RESTful: меняем атрибут ресурса |
| **Action-эндпоинт** | `POST /orders/{id}/cancel` | глагол в URL, не чисто REST, но иногда прагматично |

> Рекомендация (и конвенция проекта): смена статуса — через **PATCH** с телом `{"status":"..."}`, а не `POST /orders/{id}/cancel`. Сервер проверяет допустимость перехода (state machine, модуль 13) → 409 при недопустимом. Action-эндпоинты допустимы для сложных операций, не сводящихся к смене атрибута, но злоупотреблять не стоит.

## 6. Пагинация, фильтрация, сортировка

Коллекции (`GET /orders`) не отдают всё сразу:

**Пагинация:**
| Тип | Как | Когда |
|-----|-----|-------|
| **Offset/limit** | `?offset=40&limit=20` или `?page=3&size=20` | простые случаи, прыжки на страницу |
| **Cursor (keyset)** | `?cursor=<token>&limit=20` | большие/изменяющиеся данные, ленты, стабильность |

> **Offset** прост, но на больших данных медленный (БД пропускает N строк) и нестабилен при вставках (сдвиг). **Cursor** стабилен и быстр для лент/больших объёмов, но нельзя прыгнуть на произвольную страницу. Для бесконечных лент — cursor; для админок с номерами страниц — offset.

**Фильтрация/сортировка/проекция:**
```
GET /orders?status=delivered&minTotal=500   (фильтр)
GET /orders?sort=-createdAt                  (сортировка, - = убыв.)
GET /orders?fields=id,total,status           (проекция полей)
```

## 7. Формат ошибок: RFC 7807 (problem+json)

Единый машиночитаемый формат ошибок — **Problem Details** (`application/problem+json`):

```json
{
  "type": "https://api.foodexpress.ru/errors/invalid-status-transition",
  "title": "Invalid status transition",
  "status": 409,
  "detail": "Cannot cancel a delivered order",
  "instance": "/orders/1024"
}
```

| Поле | Смысл |
|------|-------|
| `type` | URI типа ошибки (документация) |
| `title` | краткое человекочитаемое название |
| `status` | HTTP-статус |
| `detail` | детали конкретного случая |
| `instance` | ссылка на конкретный ресурс/запрос |

> Единый формат ошибок (RFC 7807) делает API предсказуемым: клиент парсит ошибки одинаково. Лучше, чем разнобой `{"error":"..."}` / `{"message":"..."}`. Можно расширять своими полями (например, список невалидных полей при 422).

## 8. Уровни зрелости Ричардсона

Модель зрелости REST (Richardson Maturity Model):

| Уровень | Что |
|---------|-----|
| **0** | один эндпоинт, POST на всё (RPC-стиль поверх HTTP) |
| **1** | ресурсы (разные URI), но методы не используются по смыслу |
| **2** | + HTTP-методы и статусы по назначению (**практический REST**) |
| **3** | + HATEOAS (гипермедиа: ссылки на действия в ответах) |

> Большинство «REST API» на практике — **уровень 2** (ресурсы + правильные методы/статусы), и этого обычно достаточно. Уровень 3 (HATEOAS) полезен, но редок. Аналитику важно целиться минимум в уровень 2.

## ⚠️ Подводные камни

- **Глаголы в URI.** `/createOrder`, `/getOrder`, `/orders/{id}/cancel` — действие задаёт метод, не URL. Смена статуса — PATCH.
- **Неверные статусы.** 404 для отсутствующего (не «200 пустой»); 201 для создания; 204 для удаления; 401≠403, 400≠422, 409 для конфликта.
- **POST без идемпотентности.** Критичные POST (оплата) — с idempotency key, иначе дубли при повторах.
- **Offset-пагинация на больших данных.** Медленно и нестабильно при вставках; для лент — cursor.
- **Возврат всей коллекции.** `GET /orders` без пагинации не масштабируется; всегда limit + пагинация.
- **Глубокая вложенность.** `>2` уровней в URI — неудобно; ограничивать.
- **Разнобой в ошибках.** Без единого формата (RFC 7807) клиент парсит ошибки по-разному.
- **Stateful-сервер.** REST подразумевает stateless (масштабируемость, модуль 26).
