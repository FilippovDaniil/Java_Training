# Задача 04 — Модуль 30: Contract-first / OpenAPI

## ЗАДАНИЕ
Выберите стратегию версионирования API и обоснуйте под контекст.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** API используют: мобильное приложение (долгий цикл обновления у пользователей), веб (обновляется мгновенно), партнёрские интеграции (внешние, медленно мигрируют). Планируются ломающие изменения в будущем. Нужна стратегия версионирования.

## ТРЕБОВАНИЯ
- Сравните способы версионирования (URI, заголовок, media-type) по критериям (явность, кэшируемость, чистота URI, простота для клиентов).
- Выберите способ под контекст и обоснуйте.
- Опишите политику поддержки версий (сколько держать, deprecation, срок миграции).
- Учтите разные клиенты (мобайл/партнёры мигрируют медленно).

## ФОРМАТ РЕЗУЛЬТАТА
Сравнительная таблица способов + выбор с обоснованием + политика версий (deprecation/срок).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сравнение URI / заголовок / media-type по ≥3 критериям.
- [ ] Выбран способ (обычно URI `/v1/`) с обоснованием.
- [ ] Политика: параллельная поддержка версий + deprecation + срок миграции.
- [ ] Учтены медленно мигрирующие клиенты (мобайл, партнёры).
- [ ] Версионирование применяется к ломающим изменениям, не к каждому.
- [ ] Связь с breaking changes (задача 05, модуль 28).

## ПОДСКАЗКА
URI-версионирование (`/v1/orders`) — явно, просто, кэшируемо, понятно клиентам (самый частый выбор). Заголовок/media-type — чище URI, но менее заметны. Под разнородных клиентов (медленный мобайл/партнёры) важна параллельная поддержка старой версии + объявленный deprecation + срок миграции. Новую версию заводят только при ломающих изменениях (модуль 28).
