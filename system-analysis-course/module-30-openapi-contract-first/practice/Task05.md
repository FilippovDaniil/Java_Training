# Задача 05 — Модуль 30: Contract-first / OpenAPI

## ЗАДАНИЕ
Классифицируйте изменения API на breaking / non-breaking и определите, нужна ли новая версия.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** API v1, предлагаемые изменения:
1. Добавить опциональное поле `discount` в ответ `Order`.
2. Добавить новый эндпоинт `GET /orders/{id}/history`.
3. Переименовать поле `total` → `amount` в Order.
4. Сделать ранее опциональный параметр `restaurantId` обязательным в `POST /orders`.
5. Добавить опциональный query-параметр `?includeItems=true`.
6. Изменить тип `id` с integer на string (UUID).
7. Убрать эндпоинт `GET /orders-legacy`.
8. Добавить значение `refunded` в enum статуса.

## ТРЕБОВАНИЯ
- Для каждого изменения: breaking или non-breaking.
- Для breaking — нужна ли новая версия и как мигрировать.
- Объясните, почему добавление обычно безопасно, а удаление/обязательность/смена типа — нет.
- Отметьте «коварные» случаи (добавление значения enum).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица `Изменение | Breaking? | Новая версия? | Стратегия`.

## КРИТЕРИИ ПРИЁМКИ
- [ ] +опциональное поле, +эндпоинт, +опциональный параметр → non-breaking.
- [ ] Переименование поля, смена типа id, обязательность параметра, удаление эндпоинта → breaking → новая версия.
- [ ] Добавление значения enum → потенциально breaking (старые клиенты не знают) — отмечено.
- [ ] Для breaking — стратегия (новая версия + deprecation + параллельная поддержка).
- [ ] Объяснён принцип (добавлять безопасно, удалять/менять/обязательность — ломает).
- [ ] Согласуется с эволюцией схемы (модуль 28).

## ПОДСКАЗКА
Non-breaking: добавить эндпоинт/опциональное поле/опциональный параметр. Breaking: переименовать/удалить, сменить тип, сделать обязательным, убрать эндпоинт. Enum: добавление значения может сломать старых потребителей. Breaking → новая версия (`/v2`) + deprecation `/v1`. Это применение правил эволюции схемы (модуль 28) к API.
