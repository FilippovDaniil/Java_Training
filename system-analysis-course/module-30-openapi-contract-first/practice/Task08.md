# Задача 08 — Модуль 30: Contract-first / OpenAPI

## ЗАДАНИЕ
Примените contract-first: из согласованной OpenAPI-спеки выведите параллельные задачи для фронта и бэка.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, согласована OpenAPI-спека модуля «Записи»: `POST /appointments` (создать запись), `GET /appointments/{id}`, `GET /doctors/{id}/slots` (свободные слоты), `PATCH /appointments/{id}` (отмена/перенос). Фронт и бэк должны работать параллельно по контракту.

## ТРЕБОВАНИЯ
- Опишите contract-first workflow для этой спеки (контракт → параллельная работа → контрактные тесты).
- Выведите задачи фронта (по мокам спеки) и бэка (реализация по спеке) параллельно.
- Покажите роль мок-сервера (фронт не ждёт бэк).
- Опишите, как проверяется соответствие реализации контракту (контрактные тесты).
- Отметьте роль аналитика как владельца контракта.

## ФОРМАТ РЕЗУЛЬТАТА
Workflow contract-first + таблица параллельных задач `Фронт | Бэк` + роль моков и контрактных тестов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Описан workflow: спека → ревью → параллельная работа → контрактные тесты.
- [ ] Задачи фронта (вёрстка/логика по мок-серверу спеки) и бэка (реализация эндпоинтов) идут параллельно.
- [ ] Показана роль мок-сервера (генерируется по OpenAPI, фронт не ждёт бэк).
- [ ] Контрактные тесты проверяют соответствие реализации спеке.
- [ ] Отмечена роль аналитика как владельца/ревьюера контракта.
- [ ] Связь с тестированием (модуль 35).

## ПОДСКАЗКА
Contract-first: сначала согласовать OpenAPI, затем фронт и бэк работают одновременно. Фронт поднимает мок-сервер по спеке (Prism/Swagger) и верстает, не дожидаясь бэка. Бэк реализует по той же спеке. Контрактные тесты (по OpenAPI) ловят расхождение реализации с контрактом. Аналитик владеет контрактом — это его артефакт. Главный выигрыш — параллельность и меньше интеграционных сюрпризов.
