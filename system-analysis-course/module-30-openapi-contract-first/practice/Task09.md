# Задача 09 — Модуль 30: Contract-first / OpenAPI

## ЗАДАНИЕ
Спроектируйте эволюцию API без поломки клиентов: проведите ломающее изменение через версионирование и deprecation.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** API v1: в ответе `Order` поле `total` (строка). Бизнес требует разделить на `subtotal`, `discount`, `total` и переименовать структуру сумм — это ломающее изменение. Клиенты: веб (быстро мигрирует), мобайл и партнёры (медленно).

## ТРЕБОВАНИЯ
- Покажите, почему изменение ломающее.
- Спроектируйте план эволюции: ввести v2 с новой структурой, сохранить v1 параллельно.
- Опишите deprecation v1 (заголовок Deprecation/Sunset, документация, срок).
- Покажите, как клиенты мигрируют постепенно, без одномоментной поломки.
- Рассмотрите альтернативу без новой версии (добавить новые поля опционально, не трогая старые) и когда она лучше.

## ФОРМАТ РЕЗУЛЬТАТА
План эволюции (v1→v2, параллельная поддержка, deprecation, срок) + альтернатива «только добавление» + рекомендация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено, почему переименование/реструктуризация — ломающее.
- [ ] План: v2 с новой структурой + параллельная поддержка v1.
- [ ] Deprecation v1: уведомление (Deprecation/Sunset заголовки, докиn), срок миграции.
- [ ] Постепенная миграция клиентов (веб быстро, мобайл/партнёры — по сроку).
- [ ] Рассмотрена альтернатива «только добавить новые опциональные поля» (non-breaking) и когда она предпочтительнее.
- [ ] Рекомендация обоснована.

## ПОДСКАЗКА
Ломающее изменение → новая версия (v2) + параллельная поддержка v1, объявленный deprecation (заголовки `Deprecation`/`Sunset`, документация) и срок миграции для медленных клиентов. Часто лучше **не ломать**: добавить новые поля (`subtotal`, `discount`) опционально, оставив `total` — тогда версия не нужна (non-breaking, модуль 28). Версионируют, только когда добавлением не обойтись.
