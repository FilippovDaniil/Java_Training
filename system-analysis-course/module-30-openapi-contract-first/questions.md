## Ключ (эталонные ответы)

1. **OpenAPI** — стандарт описания REST API (YAML/JSON). **Swagger** — набор инструментов вокруг него (Editor, UI, Codegen). Раньше спецификация называлась Swagger, сейчас — OpenAPI.
2. OpenAPI **3.x** (3.0/3.1).
3. `info` (метаданные), `servers` (URL/среды), `paths` (эндпоинты), `components` (переиспользуемые части), `security` (применение схем аутентификации).
4. Эндпоинты: пути, HTTP-методы, параметры, тело запроса (`requestBody`), ответы (`responses`) со статусами и схемами.
5. Переиспользуемые элементы: `schemas` (модели), `responses` (типовые ответы), `parameters` (типовые параметры), `securitySchemes`. Чтобы не дублировать и менять в одном месте (DRY).
6. **`$ref`** — ссылка на переиспользуемый компонент (`#/components/schemas/Order`). Убирает дублирование моделей/ответов.
7. Через JSON Schema (type, properties, required, enum, ограничения) — те же конструкции, что в модуле 28.
8. В `components/securitySchemes` (тип: http bearer/basic, apiKey, oauth2, openIdConnect) и применяется через `security` (глобально или на эндпоинт).
9. Нет: OpenAPI описывает **как аутентифицироваться** (где токен, тип), но не саму логику проверки прав — это реализация (модуль 34).
10. **Code-first** — сначала код, спека генерируется из него. **Contract-first** — сначала спецификация (контракт), потом код по ней.
11. Параллельная работа фронта и бэка по согласованному контракту: фронт по мокам не ждёт бэк; меньше интеграционных сюрпризов; единый источник правды.
12. Сервер, автоматически отдающий ответы по OpenAPI-спеке (без реализации), чтобы фронт мог разрабатываться до готовности бэка.
13. Тесты, проверяющие, что реальная реализация соответствует контракту (OpenAPI): структура запросов/ответов, статусы, схемы.
14. В URI (`/v1/orders`), в заголовке (`Accept-Version`), в media-type (`application/vnd.x.v2+json`).
15. **Версия в URI** (`/v1/`): явно, просто, кэшируемо, понятно клиентам.
16. При **ломающих (breaking)** изменениях: удаление/переименование, смена типа, обязательность параметра, изменение семантики.
17. Добавить новый эндпоинт; добавить опциональное поле; добавить опциональный параметр.
18. Удалить/переименовать поле или эндпоинт; сделать поле/параметр обязательным; изменить тип; сузить ограничения.
19. Старые клиенты не знают нового значения и могут на нём упасть/повести себя неверно — нужна обработка «неизвестного значения».
20. **Deprecation** — объявление, что версия/эндпоинт устарел и будет удалён. Уведомляют через документацию и заголовки (`Deprecation`, `Sunset`), дают срок миграции.
21. Чтобы клиенты (особенно медленно мигрирующие — мобайл, партнёры) успели перейти на новую версию без одномоментной поломки.
22. Клиент должен знать, как обрабатывать ошибки (какие статусы и формат, RFC 7807); неполная спека (только 200) оставляет обработку ошибок недокументированной.
23. Клиенты интегрируются по неверному контракту → баги и сюрпризы; поэтому OpenAPI делают источником правды и проверяют соответствие контрактными тестами.
24. Выносить повторяющиеся модели/ответы/параметры в `components` и ссылаться через `$ref`; не копировать inline.
25. Нет `info.version`; глаголы в путях; неверные статусы; отсутствие схем ответов и ошибок; необъявленные path-параметры; неверные типы; дублирование вместо `$ref`; «мёртвые» неиспользуемые схемы; невалидный YAML (не проверен в Swagger Editor).
