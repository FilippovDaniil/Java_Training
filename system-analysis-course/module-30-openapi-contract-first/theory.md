# Модуль 30 — Contract-first / OpenAPI ⭐

## Зачем этот модуль

REST спроектирован (модуль 29), форматы и схемы заданы (модуль 28). Теперь нужно **формализовать контракт** в машиночитаемом виде — это **OpenAPI** (бывший Swagger). Спецификация OpenAPI — единый источник правды для API: по ней генерируют документацию, клиентский/серверный код, моки, тесты. Подход **contract-first** (сначала контракт, потом код) позволяет фронту и бэку работать параллельно по согласованной спецификации.

Для аналитика OpenAPI — ключевой артефакт: он пишет/ревьюит спецификацию, ведёт версионирование, следит за совместимостью. На собеседовании middle+ обязательно: структура OpenAPI (paths/components/schemas/security), contract-first процесс, стратегии версионирования, breaking vs non-breaking изменения.

---

## 1. Что такое OpenAPI

**OpenAPI Specification (OAS)** — стандарт описания REST API в YAML/JSON. **Swagger** — набор инструментов вокруг неё (Swagger Editor, Swagger UI, Codegen).

```
OpenAPI-спецификация (openapi.yaml)
   ├── документация (Swagger UI)
   ├── генерация клиента/сервера (Codegen)
   ├── моки (мок-сервер по спеке)
   └── контрактные тесты
```

> OpenAPI — это контракт **в коде**: один файл, из которого порождается всё остальное. Это убирает расхождение «документация vs реальное API». Текущая версия — OpenAPI 3.x.

## 2. Структура спецификации (OpenAPI 3.x)

```yaml
openapi: 3.0.3
info:
  title: FoodExpress API
  version: 1.0.0
servers:
  - url: https://api.foodexpress.ru/v1
paths:                              # эндпоинты
  /orders:
    post:
      summary: Создать заказ
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateOrderRequest'
      responses:
        '201':
          description: Заказ создан
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Order'
        '422':
          $ref: '#/components/responses/ValidationError'
components:                         # переиспользуемые части
  schemas: { ... }                 # модели данных (JSON Schema)
  responses: { ... }               # типовые ответы
  parameters: { ... }              # типовые параметры
  securitySchemes: { ... }         # схемы аутентификации
security: [ ... ]                  # применение security
```

| Секция | Что описывает |
|--------|---------------|
| `info` | название, версия, описание API |
| `servers` | базовые URL (среды) |
| `paths` | эндпоинты: методы, параметры, тела, ответы |
| `components` | переиспользуемые schemas/responses/parameters/securitySchemes |
| `security` | какие схемы аутентификации применяются |

> Главная идея: описать **каждый эндпоинт** (метод, параметры, тело запроса, возможные ответы со статусами и схемами) и вынести повторяющееся в `components`. Схемы данных в OpenAPI — это по сути JSON Schema (модуль 28).

---

## 3. Components: переиспользование (DRY)

Чтобы не дублировать модели, их выносят в `components` и ссылаются через `$ref`:

```yaml
components:
  schemas:
    Order:
      type: object
      required: [id, status, total]
      properties:
        id: { type: integer }
        status: { type: string, enum: [created, paid, delivered, cancelled] }
        total: { type: string, description: "сумма, копейки/строка" }
  responses:
    ValidationError:
      description: Ошибка валидации
      content:
        application/problem+json:
          schema: { $ref: '#/components/schemas/Problem' }
```

> `$ref` — ссылка на переиспользуемый компонент. Одна модель `Order` используется во многих эндпоинтах; типовая ошибка `ValidationError` — везде. Это DRY для контракта: меняешь в одном месте.

## 4. Security в спецификации

Схемы аутентификации описываются в `securitySchemes` и применяются глобально или к эндпоинту:

```yaml
components:
  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
security:
  - bearerAuth: []          # применяется ко всем эндпоинтам
```

Типы: `http` (bearer/basic), `apiKey` (в заголовке/query), `oauth2`, `openIdConnect`.

> Security в OpenAPI описывает **как аутентифицироваться** (тип, где токен), но не саму проверку прав — это реализация (модуль 34). Аналитик фиксирует в спеке: какие эндпоинты публичные, какие требуют токен, какие scope.

## 5. Contract-first процесс

| Подход | Суть |
|--------|------|
| **Code-first** | сначала код, спецификация генерируется из него | 
| **Contract-first** | сначала спецификация (контракт), потом код по ней |

**Contract-first workflow:**
```
1. Аналитик/команда пишут OpenAPI-спеку (контракт)
2. Согласование контракта (ревью)
3. Фронт и бэк работают параллельно:
   - фронт → мок-сервер по спеке
   - бэк → реализует по спеке
4. Контрактные тесты проверяют соответствие реализации спеке
```

> **Contract-first** — главный практический выигрыш: фронт не ждёт бэк, оба кодят по согласованному контракту, моки генерируются автоматически. Аналитик владеет контрактом. Это снижает интеграционные сюрпризы «в конце».

## 6. Версионирование API

Способы указать версию API:

| Способ | Пример | Плюс / минус |
|--------|--------|--------------|
| **В URI** | `/v1/orders`, `/v2/orders` | явно, просто кэшировать / «грязный» URI |
| **В заголовке** | `Accept-Version: 2` | чистый URI / менее заметно |
| **В media-type** | `Accept: application/vnd.foodexpress.v2+json` | по REST-духу / сложнее |

> Самый распространённый — **версия в URI** (`/v1/`): явно, просто, кэшируемо. Версионируют при **ломающих** изменениях (модуль 28). Не-ломающие (добавление опциональных полей) не требуют новой версии. Несколько версий поддерживают параллельно с планом вывода старой (deprecation).

## 7. Обратная совместимость и breaking changes

(основа — модуль 28, здесь — применительно к API)

**Non-breaking (можно без новой версии):**
- добавить новый эндпоинт;
- добавить опциональное поле в ответ/запрос;
- добавить опциональный параметр;
- добавить значение в enum (осторожно).

**Breaking (нужна новая версия):**
- удалить/переименовать эндпоинт или поле;
- сделать параметр/поле обязательным;
- изменить тип поля;
- изменить семантику/коды ответов;
- сузить ограничения.

```
Non-breaking: + поле "discount" (опционально) в Order  → клиенты не ломаются
Breaking:     переименовать "total" → "amount"          → /v2 + deprecation /v1
```

> Правило: **добавлять — обычно безопасно, удалять/менять — ломает → новая версия**. Стратегия: поддерживать старую версию параллельно, объявить deprecation, дать клиентам срок на миграцию. Контракт OpenAPI делает изменения видимыми (diff спецификаций).

## ⚠️ Подводные камни

- **Документация ≠ реальное API.** Без contract-first/контрактных тестов спека расходится с кодом. OpenAPI как источник правды + проверка соответствия.
- **Дублирование моделей.** Не выносить в `components` → копипаста и рассинхрон; использовать `$ref`.
- **Ломающие изменения без версии.** Удаление/переименование/обязательность ломают клиентов; новая версия + deprecation.
- **Версия без плана вывода старой.** Бесконечная поддержка всех версий; нужен deprecation и срок.
- **Security забыта в спеке.** Не указано, какие эндпоинты требуют токен/scope — клиенты не знают правил.
- **Неполные ответы.** Описан только 200, забыты 4xx/5xx — клиент не знает, как обрабатывать ошибки (описать через RFC 7807, модуль 29).
- **Спека невалидна.** Не проверена в Swagger Editor → ошибки структуры; всегда валидировать.
- **Code-first там, где нужен contract-first.** Параллельная работа фронт/бэк требует согласованного контракта заранее.
