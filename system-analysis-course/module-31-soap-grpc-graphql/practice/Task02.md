# Задача 02 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Обоснуйте, когда выбор SOAP оправдан, на конкретном кейсе интеграции.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** (телемедицина) интегрируется с государственной системой ЕГИСЗ/страховым шлюзом, который предоставляет **только SOAP-сервис с готовым WSDL**, требует подпись сообщений (WS-Security) и формальный контракт.

## ТРЕБОВАНИЯ
- Объясните, почему здесь SOAP оправдан (минимум 3 причины из контекста).
- Назовите, какие элементы SOAP/стандартов задействованы (Envelope, Header/WS-Security, WSDL, Fault).
- Укажите, что было бы проблемой, если бы потребовали REST/gRPC вместо SOAP.
- Зафиксируйте, что аналитик берёт из WSDL для проектирования интеграции.

## ФОРМАТ РЕЗУЛЬТАТА
Короткое обоснование (буллеты) + список задействованных элементов SOAP + 2–3 предложения про альтернативы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Минимум 3 причины «за SOAP» привязаны к контексту (контрагент диктует, WSDL готов, WS-Security/подпись, регуляторика).
- [ ] Названы Envelope/Header/Body/Fault и роль WSDL.
- [ ] Упомянута WS-Security (подпись/безопасность по стандарту).
- [ ] Объяснено, почему «просто перейти на REST» нельзя (контракт задаёт контрагент).
- [ ] Указано, что аналитик извлекает из WSDL (операции, типы, адрес).

## ПОДСКАЗКА
Главный довод — **контракт диктует контрагент/регулятор**: если шлюз отдаёт только WSDL и требует WS-Security, выбора «стиля» нет, его задают снаружи. SOAP здесь = совместимость + формальные гарантии. Опора — §1 теории.
