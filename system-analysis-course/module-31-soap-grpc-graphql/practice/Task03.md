# Задача 03 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Спроектируйте `.proto`-контракт gRPC-сервиса для внутреннего межсервисного взаимодействия.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: сервис заказов вызывает внутренний **сервис расчёта стоимости доставки** (Delivery). Нужны операции: рассчитать стоимость по адресу и составу заказа; получить статус курьера.

## ТРЕБОВАНИЯ
- Опишите `.proto` (proto3): `service DeliveryService` с 2 методами (unary).
- Опишите `message`-типы запросов и ответов с **нумерованными** полями.
- Поля денег — в копейках (целое), как в курсе.
- Кратко обоснуйте, почему здесь gRPC, а не REST (2–3 причины).

## ФОРМАТ РЕЗУЛЬТАТА
Фрагмент `.proto` (syntax, package, service, messages) + 2–3 строки обоснования выбора gRPC.

## КРИТЕРИИ ПРИЁМКИ
- [ ] `syntax = "proto3"` + `package`.
- [ ] `service DeliveryService` с 2 rpc-методами (unary: запрос→ответ).
- [ ] Сообщения с корректно нумерованными полями (1,2,3…).
- [ ] Типы осмысленны (int64 для id/сумм, string для статуса/адреса).
- [ ] Обоснование gRPC привязано к контексту (межсервис, скорость, строгий контракт, кодоген).

## ПОДСКАЗКА
Опора — §2 теории. Структура: `service { rpc Method(Req) returns (Resp); }` + `message Req {...}`. Нумерация полей обязательна и неизменна. Почему gRPC: внутренний вызов, низкая латентность, бинарный protobuf, контракт + кодогенерация на обоих сервисах.
