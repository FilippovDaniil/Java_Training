# Задача 04 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Спроектируйте GraphQL-схему (SDL) для гибких клиентских запросов и приведите пример клиентского запроса.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: мобильный и веб-клиенты показывают заказ по-разному (моб — кратко: статус + сумма; веб — детально: позиции, ресторан, курьер). Нужна схема, позволяющая каждому брать ровно нужные поля.

## ТРЕБОВАНИЯ
- Опишите типы: `Order`, `OrderItem`, `Restaurant`, `Courier` (связи между ними).
- Опишите `type Query` (получить заказ по id, список заказов) и `type Mutation` (создать заказ).
- Используйте корректную нотацию SDL (`!` — обязательность, `[Type!]!` — список).
- Приведите **2 разных** клиентских запроса (краткий «моб» и детальный «веб») к одной схеме.

## ФОРМАТ РЕЗУЛЬТАТА
GraphQL SDL (типы + Query + Mutation) + 2 примера `query` с разным набором полей.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Типы Order/OrderItem/Restaurant/Courier со связями.
- [ ] Корректные модификаторы: `ID!`, `String!`, `[OrderItem!]!`.
- [ ] `Query` (order(id), orders) и `Mutation` (createOrder с input).
- [ ] Два запроса берут **разный** набор полей одной схемы (демонстрация гибкости).
- [ ] Видно, что нет over/underfetching: клиент берёт ровно нужное.

## ПОДСКАЗКА
Опора — §3 теории. Тип: `type Order { id: ID! status: String! items: [OrderItem!]! }`. `Query { order(id: ID!): Order }`. Гибкость показывается двумя запросами: моб берёт `status total`, веб — `items{...} restaurant{...} courier{...}` к той же схеме.
