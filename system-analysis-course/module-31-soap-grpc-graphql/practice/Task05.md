# Задача 05 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Объясните проблему N+1 в GraphQL на конкретном запросе и предложите митигацию.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, запрос списка заказов с вложенными позициями:
```graphql
query { orders { id items { dishName qty } } }
```
Резолвер `items` обращается к БД для каждого заказа отдельно. В выдаче 50 заказов.

## ТРЕБОВАНИЯ
- Посчитайте число обращений к БД при наивной реализации и объясните, откуда «N+1».
- Опишите минимум 2 способа митигации (DataLoader/батчинг, джойн в резолвере, лимит сложности/глубины).
- Покажите, как DataLoader сворачивает N запросов в 1 (`IN (...)`).
- Укажите, чем N+1 опасен под нагрузкой.

## ФОРМАТ РЕЗУЛЬТАТА
Разбор «откуда N+1» (с числом запросов) + список митигаций + короткое пояснение DataLoader.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Корректно посчитано: 1 (за списком) + 50 (за позициями) = 51 запрос.
- [ ] Объяснено, что резолвер вложенного поля вызывается на каждый элемент списка.
- [ ] Минимум 2 способа митигации.
- [ ] DataLoader описан как батчинг id в один запрос + кэш в рамках запроса.
- [ ] Указана опасность (рост запросов к БД, деградация/DoS под нагрузкой).

## ПОДСКАЗКА
Опора — §3 «Проблема N+1». Формула: 1 запрос за родителями + N запросов за детьми. DataLoader собирает все `orderId` и делает один `... WHERE order_id IN (...)`. Дополнительно — depth/complexity limits, чтобы клиент не заказал тяжёлый граф.
