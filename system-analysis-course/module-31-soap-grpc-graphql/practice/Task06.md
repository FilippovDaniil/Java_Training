# Задача 06 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Выберите стиль API (REST / SOAP / gRPC / GraphQL) под каждый из 4 кейсов и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
1. Публичное API каталога ресторанов для сторонних разработчиков (нужны кэш, простота, широкая совместимость).
2. Внутренний вызов между сервисом заказов и сервисом расчёта доставки под высокой нагрузкой, нужен стриминг статусов.
3. Интеграция с государственным шлюзом, который отдаёт только WSDL и требует подпись сообщений.
4. Мобильное и веб-приложение с разными экранами, которым нужны разные наборы полей по одним и тем же сущностям.

## ТРЕБОВАНИЯ
- Для каждого кейса: выбранный стиль + 1–2 решающих критерия.
- Решение должно опираться на свойства стиля, а не «нравится».
- Где уместно — отметьте отвергнутую альтернативу и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: Кейс | Стиль | Решающий критерий | (опц.) Почему не альтернатива.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Кейс 1 → REST (публичный CRUD, HTTP-кэш, совместимость).
- [ ] Кейс 2 → gRPC (межсервис, low-latency, стриминг).
- [ ] Кейс 3 → SOAP (контракт диктует контрагент: WSDL + WS-Security).
- [ ] Кейс 4 → GraphQL (разные клиенты, нет over/underfetching).
- [ ] Каждый выбор обоснован свойством стиля, а не вкусом.

## ПОДСКАЗКА
Опора — §5 «Критерии выбора». Сопоставляйте: публичный кэшируемый CRUD→REST; внутренний быстрый стрим→gRPC; навязанный WSDL/WS-Security→SOAP; гибкие выборки многих клиентов→GraphQL.
