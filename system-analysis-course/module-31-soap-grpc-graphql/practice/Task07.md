# Задача 07 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Определите, где уместен gRPC-стриминг, и сопоставьте сценарии с типами вызовов.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, набор взаимодействий:
- A. Получить заказ по id.
- B. Подписка клиента на изменения статуса заказа (готовится → в пути → доставлен).
- C. Курьерское приложение шлёт поток GPS-координат на сервер.
- D. Живая трекинг-сессия: клиент шлёт запросы, сервер шлёт обновления карты в обе стороны.

## ТРЕБОВАНИЯ
- Сопоставьте каждый сценарий с типом gRPC-вызова: unary / server streaming / client streaming / bidirectional.
- Обоснуйте выбор по направлению потока данных.
- Отметьте, какие из сценариев плохо ложатся на классический REST request/response и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: Сценарий | Тип вызова | Обоснование (направление потока).

## КРИТЕРИИ ПРИЁМКИ
- [ ] A → unary; B → server streaming; C → client streaming; D → bidirectional.
- [ ] Обоснование через направление потока (один/много, кто→кому).
- [ ] Отмечено, что B/C/D неудобны на чистом REST (нужны WS/SSE/поллинг).
- [ ] Корректная терминология (4 типа из теории).

## ПОДСКАЗКА
Опора — §2, таблица типов вызовов. Логика по потоку: один→один = unary; один→много = server streaming; много→один = client streaming; много↔много = bidirectional. REST request/response плохо отдаёт длительные потоки — отсюда стриминг gRPC.
