# Задача 08 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Покажите на конкретном экране проблему overfetching/underfetching в REST и как GraphQL её решает.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, экран «карточка заказа» в мобильном приложении: нужны только `status`, `total` и имена 3 первых блюд. В REST есть эндпоинты: `GET /orders/{id}` (возвращает заказ целиком со всеми полями), `GET /orders/{id}/items` (все позиции с полным описанием), `GET /restaurants/{id}` (ресторан целиком).

## ТРЕБОВАНИЯ
- Покажите overfetching в REST (что лишнего приходит) для этого экрана.
- Покажите underfetching в REST (сколько запросов нужно, чтобы собрать экран).
- Приведите эквивалентный GraphQL-запрос, берущий ровно нужное за один раз.
- Сформулируйте, при каком числе клиентов/экранов выигрыш GraphQL значим, а когда REST достаточно.

## ФОРМАТ РЕЗУЛЬТАТА
Разбор overfetching + underfetching в REST (с числом запросов) + GraphQL-запрос + вывод о применимости.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Overfetching показан (REST отдаёт лишние поля заказа/позиций/ресторана).
- [ ] Underfetching показан (≥2–3 REST-запроса для одного экрана).
- [ ] GraphQL-запрос берёт ровно `status`, `total`, имена блюд за 1 запрос.
- [ ] Сформулировано, когда выигрыш значим (много разнородных клиентов), а когда REST хватает (один-два стабильных клиента).
- [ ] Нет утверждения «GraphQL всегда лучше».

## ПОДСКАЗКА
Опора — §3 и §4. Overfetching = REST-эндпоинт фиксирован, отдаёт всё; underfetching = данные на экран собираются из нескольких эндпоинтов. GraphQL: один запрос с нужными полями. Но помни цену GraphQL (кэш, сложность) — для простых стабильных клиентов REST проще.
