# Задача 09 — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Опишите стратегию эволюции контракта (версионирование/совместимость) для gRPC и для GraphQL.

## ВХОДНЫЕ ДАННЫЕ
Два контракта меняются:
- **gRPC `.proto`**: в `message Order` нужно добавить поле `discount` и удалить устаревшее поле `legacy_code`.
- **GraphQL schema**: в типе `Order` нужно добавить поле `discount` и вывести из обращения поле `legacyCode`.

## ТРЕБОВАНИЯ
- Для gRPC: как добавить поле без поломки (номера полей), как корректно удалить (`reserved`), что нельзя делать с номерами.
- Для GraphQL: как добавить поле, как пометить устаревшее (`@deprecated`), почему обычно без версий URI.
- Классифицируйте каждое изменение как breaking/non-breaking.
- Сформулируйте общий принцип (как в модулях 28/30).

## ФОРМАТ РЕЗУЛЬТАТА
Два блока (gRPC, GraphQL) с конкретными шагами + классификация breaking/non-breaking + общий принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] gRPC: новое поле = новый номер (non-breaking); удаление = `reserved` номера/имени; номера не менять/не переиспользовать.
- [ ] GraphQL: добавление поля non-breaking; устаревание через `@deprecated(reason)`; удаление после миграции.
- [ ] Объяснено, почему GraphQL обычно без версий URI.
- [ ] Удаление поля корректно классифицировано как breaking (для существующих потребителей).
- [ ] Общий принцип: добавлять безопасно, удалять/менять — ломает.

## ПОДСКАЗКА
Опора — §6 теории. gRPC: совместимость держится на стабильных номерах полей; удаление — `reserved 7; reserved "legacy_code";`. GraphQL: схема эволюционирует, `@deprecated` вместо `/v2`. Принцип единый с REST/форматами (модули 28/30).
