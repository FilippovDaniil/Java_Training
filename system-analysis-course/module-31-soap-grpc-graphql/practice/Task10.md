# Задача 10 (мини-проект) — Модуль 31: SOAP / gRPC / GraphQL

## ЗАДАНИЕ
Выберите стиль API для 3 взаимодействий одной системы и обоснуйте каждый выбор. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** (телемедицина), три взаимодействия:
1. **Внешнее**: интеграция с государственным шлюзом проверки полиса ОМС — шлюз предоставляет только WSDL и требует подпись сообщений.
2. **Внутреннее**: сервис записи на приём вызывает сервис расписания врачей под высокой нагрузкой; нужна подписка на освобождающиеся слоты в реальном времени.
3. **Клиентское**: мобильное и веб-приложение пациента показывают разные наборы данных по приёмам, врачам, документам; команда хочет избежать множества узких эндпоинтов и over/underfetching.

## ТРЕБОВАНИЯ
Для каждого взаимодействия:
1. Выбранный стиль (REST / SOAP / gRPC / GraphQL).
2. Контракт-артефакт стиля (WSDL / `.proto` / GraphQL schema / OpenAPI) — назвать.
3. 2–3 решающих критерия выбора (из свойств стиля и контекста).
4. Отвергнутая альтернатива + почему не она.
5. Один риск/подводный камень выбранного стиля и как его смягчить (например: gRPC→браузер; GraphQL→N+1/сложность; SOAP→тяжесть).

Итог: короткая «карта стилей API системы» (таблица) + вывод о том, что в одной системе стили сочетаются.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица по 3 взаимодействиям (стиль, контракт, критерии, отвергнутая альтернатива, риск+смягчение) + 2–3 строки вывода о смешанном ландшафте.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Взаимодействие 1 → SOAP (WSDL диктует контрагент, WS-Security/подпись).
- [ ] Взаимодействие 2 → gRPC (межсервис, low-latency, стриминг/подписка на слоты).
- [ ] Взаимодействие 3 → GraphQL (разные клиенты, нет over/underfetching, один эндпоинт).
- [ ] Для каждого назван контракт-артефакт (WSDL / `.proto` / schema).
- [ ] У каждого выбора — отвергнутая альтернатива с причиной.
- [ ] У каждого выбора — риск стиля + способ смягчения.
- [ ] Вывод: стили сочетаются (SOAP наружу к гос, gRPC внутри, GraphQL клиентам) — нет «одного на всё».

## ПОДСКАЗКА
Синтез: критерии выбора (§5), контракты (§4), риски (⚠️). Каркас вывода: SOAP — там, где контракт навязан регулятором; gRPC — внутреннее низколатентное со стримингом; GraphQL — клиентский слой/BFF. Главная мысль middle+: **выбор стиля — это trade-off под конкретное взаимодействие, а не один стиль на всю систему**. Покажи это, и риски (gRPC-Web для браузера, DataLoader против N+1, тяжесть SOAP) — с митигацией.
