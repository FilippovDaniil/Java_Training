## Ключ (эталонные ответы)

1. **SOAP** — протокол обмена структурированными сообщениями на базе **XML**; предшественник REST, жив в enterprise/legacy (банки, гос, страхование).
2. **Envelope** (конверт) → **Header** (метаданные: безопасность, маршрутизация, транзакции) + **Body** (полезная нагрузка: операция и параметры); ошибки — блок **Fault** внутри Body.
3. **WSDL** — XML-описание сервиса (операции, типы, привязки, адрес) — формальный контракт SOAP, аналог OpenAPI для REST; по нему генерируют клиент/сервер.
4. **WS-*** (WS-Security, WS-ReliableMessaging, WS-AtomicTransaction) дают по стандарту безопасность, надёжную доставку, распределённые транзакции; важно в банках/гос/B2B, где нужны формальные гарантии.
5. Когда требует контрагент/регулятор (банк, госшлюз, страховая), есть готовый WSDL legacy-системы, нужны WS-* гарантии. Для нового публичного API обычно не выбирают (тяжёл).
6. **gRPC** — RPC-фреймворк (вызов удалённого метода как локального) на транспорте **HTTP/2** и бинарном формате **Protocol Buffers (protobuf)**.
7. `.proto` — файл-контракт gRPC: описывает сервисы, методы (rpc), сообщения (message) с нумерованными полями; по нему генерируют клиент/сервер на многих языках.
8. **Unary** (запрос→ответ), **Server streaming** (запрос→поток ответов), **Client streaming** (поток запросов→ответ), **Bidirectional** (поток↔поток).
9. Бинарный protobuf компактнее и быстрее в сериализации, чем текстовый JSON; HTTP/2 даёт мультиплексирование и низкую латентность.
10. Нативно не работает из браузера — нужен **gRPC-Web** + прокси; не human-readable (бинарный).
11. **GraphQL** — язык запросов и рантайм; один эндпоинт, где **клиент сам указывает нужные поля**; контракт — типизированная schema (SDL).
12. **Query** — чтение (как GET); **Mutation** — изменение (создать/обновить/удалить); **Subscription** — подписка на события в реальном времени (обычно WebSocket).
13. **Overfetching** — сервер вернул лишние поля; **underfetching** — данных не хватило, нужен доп. запрос. GraphQL решает: клиент берёт **ровно нужные поля** в одном запросе.
14. Резолвер вложенного поля вызывается для каждого элемента списка: 1 запрос за списком + N запросов за вложенными данными = **N+1** обращений к источнику.
15. **DataLoader** (батчинг id в один запрос `IN (...)` + кэш в рамках запроса), джойны на уровне резолвера, ограничение глубины/сложности запроса.
16. Сложнее кэшировать (нет HTTP-кэша по URL), легко создать тяжёлый запрос (нужны лимиты сложности), риск N+1, выше порог входа.
17. Запросы идут на один URL методом POST с разным телом → стандартный HTTP-кэш по URL не работает; нужны persisted queries / прикладной кэш.
18. REST → OpenAPI; SOAP → WSDL/XSD; gRPC → `.proto`; GraphQL → schema (SDL). Все — формальные контракты, но разной природы (ресурсы / операции / методы / граф).
19. **REST** — публичный ресурсный CRUD, важны кэш и простота. **GraphQL** — много разнородных клиентов с разными выборками, нужно избегать over/underfetching и плодить эндпоинты.
20. **gRPC** — внутреннее межсервисное взаимодействие, высокая нагрузка/низкая латентность, нужен стриминг, строгий контракт и кодогенерация в полиглот-среде.
21. Да. Типично: REST/GraphQL наружу (клиенты), gRPC внутри (между микросервисами), SOAP — для интеграции с конкретной legacy/гос-системой.
22. Поля имеют стабильные номера (tag): добавлять новые поля с новыми номерами, **не менять/не переиспользовать** существующие, удалять через `reserved`; крупные изменения — новая версия пакета/сервиса.
23. Обычно **без версий**: добавляют новые поля/типы, устаревшее помечают `@deprecated(reason: ...)`, удаляют после миграции клиентов.
24. Номер поля — идентификатор в бинарном формате; смена номера сделает старые данные нечитаемыми/неверно интерпретируемыми → ломает обратную совместимость.
25. Брать gRPC/GraphQL «потому что модно» без нужды; SOAP для нового публичного API; gRPC напрямую в браузер; GraphQL без лимитов сложности и без DataLoader (N+1); версионирование GraphQL через URI вместо `@deprecated`; игнор кэширования.
