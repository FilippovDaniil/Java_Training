# Модуль 31 — SOAP / gRPC / GraphQL

## Зачем этот модуль

REST (модули 29–30) — стиль по умолчанию для публичных API, но не единственный. Есть задачи, где REST неудобен: строгие enterprise-контракты с гарантиями (банки, госуслуги) → **SOAP**; быстрое низколатентное межсервисное взаимодействие со строгой схемой и стримингом → **gRPC**; гибкие клиентские запросы без overfetching/underfetching и множества эндпоинтов → **GraphQL**.

Аналитик middle+ должен **выбирать стиль API под задачу** и понимать trade-offs: транспорт, формат, контракт, производительность, эволюция. На собеседовании это частый блок: «чем gRPC отличается от REST», «когда GraphQL, а когда REST», «что такое N+1 в GraphQL», «зачем ещё используют SOAP». Этот модуль даёт карту альтернатив и критерии выбора.

---

## 1. SOAP (Simple Object Access Protocol)

**SOAP** — протокол обмена структурированными сообщениями на базе **XML**. Появился раньше REST, до сих пор жив в enterprise/legacy: банки, страхование, госсистемы, телеком.

```
HTTP POST /PaymentService
Content-Type: text/xml
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>...токены, маршрутизация...</soap:Header>
  <soap:Body>
    <m:CreatePayment xmlns:m="urn:bank:payments">
      <m:amount>10000</m:amount>
      <m:currency>RUB</m:currency>
    </m:CreatePayment>
  </soap:Body>
</soap:Envelope>
```

| Элемент | Назначение |
|---------|------------|
| **Envelope** | корневой конверт сообщения |
| **Header** | метаданные: безопасность (WS-Security), транзакции, маршрутизация |
| **Body** | полезная нагрузка (вызов операции + параметры) |
| **Fault** | стандартный блок ошибки внутри Body |

**WSDL** (Web Services Description Language) — XML-описание сервиса: операции, типы, привязки, адрес. Это контракт SOAP (аналог OpenAPI для REST). По WSDL генерируют клиент и сервер.

**Особенности SOAP:**
- транспортно-независим (HTTP, но и MQ, SMTP);
- стандарты **WS-*** (WS-Security, WS-ReliableMessaging, WS-AtomicTransaction) — гарантии безопасности, надёжности, распределённых транзакций «из коробки» по стандарту;
- строгая типизация и формальный контракт (WSDL + XSD);
- многословный (XML-конверты), тяжелее REST/JSON.

> SOAP жив там, где нужны **формальные enterprise-гарантии и совместимость с legacy** (банки, гос, B2B-интеграции по стандарту). Для новых публичных API его обычно не выбирают — слишком тяжёл.

## 2. gRPC

**gRPC** — RPC-фреймворк от Google: вызов удалённого метода как локального. Транспорт — **HTTP/2**, формат — **Protocol Buffers (protobuf)** (бинарный, компактный). Контракт — `.proto`-файл.

```protobuf
syntax = "proto3";
package orders;

service OrderService {
  rpc GetOrder (GetOrderRequest) returns (Order);
  rpc WatchOrders (WatchRequest) returns (stream OrderEvent);  // server streaming
}

message GetOrderRequest { int64 id = 1; }
message Order {
  int64 id = 1;
  string status = 2;
  int64 total = 3;       // копейки
}
```

**Четыре типа вызовов (стриминг через HTTP/2):**

| Тип | Описание | Пример |
|-----|----------|--------|
| **Unary** | запрос → ответ (как обычный вызов) | получить заказ |
| **Server streaming** | запрос → поток ответов | подписка на статусы заказа |
| **Client streaming** | поток запросов → ответ | загрузка телеметрии пачкой |
| **Bidirectional** | поток ↔ поток | чат, живая трекинг-сессия |

**Особенности gRPC:**
- **бинарный protobuf** → меньше размер, быстрее сериализация, чем JSON;
- **HTTP/2** → мультиплексирование, стриминг, низкая латентность;
- строгий контракт `.proto` + кодогенерация клиента/сервера на многих языках;
- **минус для браузеров**: нативно не работает из браузера (нужен gRPC-Web + прокси); не human-readable.

> gRPC — выбор для **внутреннего межсервисного взаимодействия** (микросервисы), где важны скорость, строгий контракт и стриминг. Для публичного браузерного API чаще REST/GraphQL.

## 3. GraphQL

**GraphQL** — язык запросов к API и рантайм. Один эндпоинт (`POST /graphql`), клиент сам описывает, **какие именно поля** ему нужны. Контракт — **schema** (SDL).

```graphql
# Schema (SDL)
type Order {
  id: ID!
  status: String!
  total: Int!
  items: [OrderItem!]!
}
type Query   { order(id: ID!): Order }
type Mutation { createOrder(input: CreateOrderInput!): Order }
```

```graphql
# Запрос клиента — берёт только нужные поля
query {
  order(id: "42") {
    status
    items { dishName qty }
  }
}
```

| Операция | Назначение |
|----------|------------|
| **Query** | чтение данных (аналог GET) |
| **Mutation** | изменение (создать/обновить/удалить) |
| **Subscription** | подписка на события (реалтайм, обычно WebSocket) |

**Особенности GraphQL:**
- **клиент берёт ровно нужные поля** → нет overfetching (лишних данных) и underfetching (нехватки → доп. запросы);
- один эндпоинт вместо множества;
- сильная типизированная схема + интроспекция (самодокументируемость);
- **минусы**: сложнее кэшировать (HTTP-кэш по URL не работает), легко создать тяжёлый запрос, риск **проблемы N+1**.

### Проблема N+1

Запрос списка заказов с вложенными позициями: 1 запрос за заказами + N запросов за позициями каждого заказа = **N+1 обращений к БД**.

```
query { orders { id items { name } } }
→ 1 запрос: список из 50 заказов
→ 50 запросов: позиции для каждого заказа     ← N+1
```

**Митигация:** **DataLoader** (батчинг + кэш в рамках запроса: собрать все id и сделать 1 запрос `IN (...)`), джойны на уровне резолвера, ограничение глубины/сложности запроса.

> GraphQL — выбор, когда у вас **много разных клиентов с разными потребностями в данных** (моб/веб/партнёры) и важно избегать over/underfetching. Цена — кэширование и контроль сложности запросов.

## 4. Сравнение с REST

| Критерий | REST | SOAP | gRPC | GraphQL |
|----------|------|------|------|---------|
| **Формат** | JSON (обычно) | XML | Protobuf (бинарный) | JSON |
| **Транспорт** | HTTP/1.1+ | HTTP/MQ/SMTP | HTTP/2 | HTTP (1 эндпоинт) |
| **Контракт** | OpenAPI | WSDL/XSD | `.proto` | Schema (SDL) |
| **Стиль** | ресурсы (CRUD) | операции (RPC) | методы (RPC) | граф данных по запросу |
| **Стриминг** | ограниченно (SSE/WS) | нет (в базе) | да (4 типа) | Subscriptions |
| **Кэширование** | простое (HTTP-кэш) | сложно | нет (бинарный) | сложно |
| **Браузер** | да | да | через gRPC-Web | да |
| **Over/underfetching** | возможны | возможны | фикс. контракт | клиент берёт нужное |
| **Где силён** | публичные API, web | enterprise/legacy/гос | межсервис, low-latency | гибкие клиенты, BFF |

> Это не «лучше/хуже», а **разные ниши**. Часто в одной системе сочетают: REST/GraphQL наружу (клиенты), gRPC внутри (между сервисами), SOAP — для интеграции с конкретной legacy/гос-системой.

## 5. Критерии выбора

**REST** — по умолчанию для публичного ресурсного API, простых CRUD, когда важны HTTP-кэш и широкая совместимость.

**SOAP** — когда требует контрагент/регулятор (банк, гос, страховой шлюз), нужны WS-* гарантии, есть готовый WSDL legacy-системы.

**gRPC** — внутреннее межсервисное взаимодействие, высокая нагрузка/низкая латентность, нужен стриминг, строгий контракт и кодогенерация, полиглот-окружение.

**GraphQL** — много разнородных клиентов с разными выборками данных, нужно избегать over/underfetching и плодить эндпоинты; часто как агрегирующий слой (BFF) над несколькими сервисами.

```
Публичный CRUD, кэш, простота        → REST
Контрагент/регулятор требует, legacy → SOAP
Между микросервисами, скорость, стрим → gRPC
Гибкие выборки многих клиентов        → GraphQL
```

## 6. Версионирование в gRPC и GraphQL

| Стиль | Подход к версионированию |
|-------|--------------------------|
| **gRPC (protobuf)** | поля нумеруются (tag); **не переиспользовать/не менять номера**; добавлять новые поля с новыми номерами; удалять — через `reserved`. Версия пакета/сервиса при больших изменениях. |
| **GraphQL** | обычно **без версий**: схема эволюционирует добавлением полей; устаревшее помечают `@deprecated(reason: ...)`; удаляют после миграции клиентов. |

> Общий принцип тот же, что в модулях 28/30: **добавлять безопасно, удалять/менять — ломает**. protobuf держит совместимость через стабильные номера полей; GraphQL — через `@deprecated` вместо версий URI.

---

## ⚠️ Подводные камни

- **«gRPC/GraphQL всегда лучше REST».** Нет — это ниши. gRPC плох для браузера, GraphQL сложен в кэшировании, SOAP тяжёл. Выбор — по контексту.
- **GraphQL и N+1.** Наивные резолверы порождают N+1; нужен DataLoader (батчинг) и ограничение сложности.
- **GraphQL без лимитов сложности.** Клиент может запросить глубоко вложенный тяжёлый граф → DoS; нужны depth/complexity limits.
- **gRPC наружу в браузер напрямую.** Не работает без gRPC-Web + прокси; для публичного web-API чаще REST/GraphQL.
- **SOAP для нового публичного API.** Избыточно тяжёл; SOAP — для legacy/enterprise-требований, не «по умолчанию».
- **Смена номера поля в protobuf.** Ломает совместимость; номера полей неизменны, удаление — через `reserved`.
- **Версионирование GraphQL через URI `/v2`.** Не в духе GraphQL: эволюционируют схему и `@deprecated`, а не плодят версии.
- **Игнор кэширования в GraphQL/gRPC.** HTTP-кэш по URL не работает; нужны persisted queries / прикладной кэш.
