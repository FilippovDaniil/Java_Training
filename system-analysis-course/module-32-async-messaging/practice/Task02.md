# Задача 02 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте событие (event) и его схему для публикации в топик.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: при оплате заказа сервис заказов публикует событие `OrderPaid` в топик `orders`. Потребители: склад/ресторан, уведомления, аналитика.

## ТРЕБОВАНИЯ
- Спроектируйте схему события (JSON): обязательные поля события.
- Включите: уникальный `eventId` (для дедупликации), `eventType`, `occurredAt` (время), ключ партиционирования (`orderId`), полезную нагрузку (clientId, restaurantId, total в копейках), версию схемы.
- Кратко: почему событие именует **факт в прошлом** (`OrderPaid`, не `PayOrder`).
- Укажите, какой ключ партиционирования и почему.

## ФОРМАТ РЕЗУЛЬТАТА
JSON-пример события + список обязательных полей с назначением + 2 строки про имя-факт и ключ партиции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Есть `eventId` (уникальный, для дедупа), `eventType`, `occurredAt`, версия схемы.
- [ ] Есть ключ партиционирования `orderId` и пояснение (порядок событий заказа).
- [ ] Полезная нагрузка содержит нужные поля; сумма в копейках (целое).
- [ ] Имя события — факт в прошедшем времени (`OrderPaid`), не команда.
- [ ] Указано, кто потребители и что событие — для всех (топик/pub-sub).

## ПОДСКАЗКА
Опора — §2, §4, §6. Событие = факт («произошло»), поэтому прошедшее время. `eventId` нужен потребителям для дедупликации (at-least-once). Ключ партиции = `orderId`, чтобы события одного заказа были упорядочены (модуль 28 — про схему/версионирование).
