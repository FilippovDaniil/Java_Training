# Задача 04 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте партиционирование топика так, чтобы сохранить порядок событий и обеспечить масштабирование.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, топик `orders` с событиями жизненного цикла заказа: `OrderCreated → OrderPaid → OrderDelivered`. Важно: события **одного** заказа должны обрабатываться строго по порядку; заказов очень много, нужен параллелизм.

## ТРЕБОВАНИЯ
- Выберите ключ партиционирования и объясните, как он сохраняет порядок для одного заказа.
- Объясните, почему порядок НЕ гарантируется глобально по топику.
- Свяжите число партиций с параллелизмом потребителей (consumer group).
- Покажите риск: что будет, если ключом сделать, например, `restaurantId` или случайное значение.

## ФОРМАТ РЕЗУЛЬТАТА
Выбор ключа + объяснение порядка (внутри партиции) + связь «партиции ↔ параллелизм» + разбор риска неверного ключа.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ключ партиционирования = `orderId` (события заказа в одну партицию).
- [ ] Объяснено: порядок гарантируется только внутри партиции, не глобально.
- [ ] Связь: параллелизм потребителей ≤ числу партиций.
- [ ] Показан риск неверного ключа (перемешивание/перекос нагрузки).
- [ ] Упомянут перекос (data skew), если ключ распределён неравномерно.

## ПОДСКАЗКА
Опора — §4 теории. Ключ = `orderId` → все события заказа в одной партиции → порядок сохранён. Глобального порядка нет: разные заказы в разных партициях читаются параллельно. Случайный ключ ломает порядок; «горячий» ключ (один ресторан) даёт перекос партиции.
