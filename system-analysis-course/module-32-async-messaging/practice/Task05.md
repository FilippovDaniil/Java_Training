# Задача 05 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте обработку ошибок потребителя: ретраи + DLQ + стратегия «ядовитых» сообщений.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, потребитель события `OrderPaid` обновляет склад. Возможные сбои: временная недоступность БД (восстановимо); битое/невалидное сообщение (невосстановимо — poison).

## ТРЕБОВАНИЯ
- Опишите стратегию ретраев для восстановимых ошибок (backoff, лимит попыток).
- Опишите, куда уходит сообщение после исчерпания попыток (DLQ) и зачем.
- Разделите восстановимые и невосстановимые ошибки (poison → быстро в DLQ, не крутить).
- Опишите, что происходит с DLQ дальше (алерт/разбор/переотправка).
- Поясните, почему без DLQ ядовитое сообщение опасно.

## ФОРМАТ РЕЗУЛЬТАТА
Схема обработки (попытка → backoff-ретраи → DLQ) + разделение типов ошибок + действия по DLQ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ретраи с backoff и конечным числом попыток для восстановимых ошибок.
- [ ] После N неудач — перенос в DLQ (поток не блокируется).
- [ ] Разделены восстановимые (ретрай) и невосстановимые/poison (сразу/быстро в DLQ).
- [ ] Описаны действия с DLQ (алерт, ручной разбор, переотправка).
- [ ] Объяснена опасность отсутствия DLQ (блокировка потока / бесконечные ретраи / потеря).

## ПОДСКАЗКА
Опора — §7 теории. Логика: временная ошибка → retry+backoff (например 3 попытки); если не помогло или сообщение битое → DLQ. DLQ снимает ядовитое сообщение с пути, поток продолжает работу; по DLQ — мониторинг и разбор. Ретраи без лимита + без DLQ = бесконечный цикл/блокировка.
