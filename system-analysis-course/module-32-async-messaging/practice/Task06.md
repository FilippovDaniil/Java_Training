# Задача 06 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте дедупликацию на потребителе для гарантии at-least-once.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, потребитель события `OrderPaid` начисляет бонусы клиенту. Доставка — at-least-once, значит одно и то же событие может прийти дважды. Двойное начисление недопустимо.

## ТРЕБОВАНИЯ
- Объясните, почему at-least-once → возможны дубли.
- Спроектируйте дедупликацию: что использовать как ключ (eventId), как хранить обработанные, как проверять.
- Приведите псевдокод обработчика (проверка → обработка → фиксация id).
- Назовите альтернативу через естественную идемпотентность (где она применима).
- Отметьте, что дедупликация — ответственность потребителя, а не брокера.

## ФОРМАТ РЕЗУЛЬТАТА
Объяснение дублей + псевдокод идемпотентного обработчика + альтернатива (естественная идемпотентность) + 1 строка про зону ответственности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: ретраи at-least-once порождают дубли.
- [ ] Ключ дедупа = уникальный `eventId`; хранилище обработанных id.
- [ ] Псевдокод: если id уже обработан — skip, иначе обработать и зафиксировать.
- [ ] Названа естественная идемпотентность (UPSERT/set-статус) как альтернатива.
- [ ] Указано, что идемпотентность — на стороне потребителя.

## ПОДСКАЗКА
Опора — §6 теории и модуль 25. Псевдокод: `if processed.contains(eventId) return; handle(); processed.add(eventId);` (атомарно с обработкой). Где можно — заменить на естественно идемпотентную операцию (не «+100 бонусов», а «установить баланс = X»).
