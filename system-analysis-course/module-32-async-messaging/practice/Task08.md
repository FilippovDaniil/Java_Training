# Задача 08 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте event-driven сценарий уведомлений на основе топика и подписчиков.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: при смене статуса заказа нужно уведомить клиента (push/SMS), обновить экран ресторана и записать событие в аналитику. Эти действия независимы и не должны блокировать оформление заказа.

## ТРЕБОВАНИЯ
- Спроектируйте поток: продюсер публикует событие `OrderStatusChanged` в топик; 3 независимых подписчика.
- Покажите, почему здесь именно топик (pub/sub), а не синхронные вызовы.
- Опишите, что произойдёт, если сервис уведомлений временно недоступен (развязка во времени).
- Отметьте гарантии: at-least-once + идемпотентность подписчиков; порядок по `orderId`.
- Приведите мини-схему взаимодействия (текст/диаграмма).

## ФОРМАТ РЕЗУЛЬТАТА
Схема event-driven потока (продюсер → топик → 3 подписчика) + обоснование топика + поведение при сбое подписчика + гарантии.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Событие `OrderStatusChanged` публикуется в топик; 3 независимых подписчика.
- [ ] Обосновано, почему топик/async (развязка, независимость, не блокирует заказ).
- [ ] Описано поведение при недоступности подписчика (сообщение ждёт/ретраи, остальные работают).
- [ ] Указаны гарантии (at-least-once + дедуп) и порядок по `orderId`.
- [ ] Есть наглядная схема взаимодействия.

## ПОДСКАЗКА
Опора — §1, §2, §5–6. Идея event-driven: продюсер не знает о подписчиках, публикует факт; подписчики реагируют независимо. Сбой одного не ломает других и не блокирует оформление. Каждый подписчик идемпотентен (дубли) и читает по порядку заказа (ключ `orderId`).
