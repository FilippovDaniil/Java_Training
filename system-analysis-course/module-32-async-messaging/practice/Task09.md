# Задача 09 — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Найдите проблемы в готовом async-дизайне (порядок, дубли, надёжность) и предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, дизайн интеграции записи на приём:
- Все события (`AppointmentCreated`, `AppointmentPaid`, `AppointmentCancelled`) публикуются в один топик с **случайным** ключом партиционирования.
- Потребитель «биллинг» при каждом событии `AppointmentPaid` **списывает оплату** без проверки повторов.
- Доставка настроена **at-least-once**.
- Сбойные сообщения **повторяются бесконечно**, DLQ нет.
- Потребителей в группе — 10 при 3 партициях.

## ТРЕБОВАНИЯ

- Для каждой — почему это проблема и как исправить.
- Особо разберите: нарушение порядка событий приёма, двойное списание, бесконечные ретраи, простаивающие потребители.

## ФОРМАТ РЕЗУЛЬТАТА
Список «проблема → последствие → исправление» (≥4 пункта).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Случайный ключ → нарушение порядка событий приёма; фикс: ключ = `appointmentId`.
- [ ] at-least-once + нет дедупа → двойное списание; фикс: идемпотентный потребитель (eventId/дедуп).
- [ ] Бесконечные ретраи без DLQ → блокировка/потеря; фикс: лимит попыток + DLQ.
- [ ] 10 потребителей при 3 партициях → 7 простаивают; фикс: потребителей ≤ партиций или больше партиций.
- [ ] Каждое исправление обосновано.
