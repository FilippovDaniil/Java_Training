# Задача 10 (мини-проект) — Модуль 32: Асинхронная интеграция

## ЗАДАНИЕ
Спроектируйте событийную интеграцию ключевого сценария: события, топики, ключи, гарантии, дедупликация, DLQ. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, сценарий «оплата и обработка заказа»: после оплаты заказа независимо реагируют склад/ресторан (готовит), сервис уведомлений (сообщает клиенту), бонусный сервис (начисляет баллы), аналитика. Действия не должны блокировать оформление; сервисы могут временно падать.

## ТРЕБОВАНИЯ
Сдайте дизайн async-интеграции:
1. **События** — список событий сценария (имена-факты, в прошедшем времени) и их назначение.
2. **Схема ключевого события** (`OrderPaid`): eventId, eventType, occurredAt, ключ партиции, payload, версия.
3. **Топики/каналы** — что в какой топик; очередь vs топик там, где уместно.
4. **Ключ партиционирования** и обоснование порядка.
5. **Гарантии доставки** (at-least-once) + **дедупликация** на потребителях.
6. **DLQ + ретраи** (backoff, лимит, разбор).
7. **Consumer groups** — как масштабируются потребители.
8. **Поведение при сбое** одного из потребителей (развязка).
Кратко отметьте eventual consistency и что показывать пользователю.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн (события, схема события, карта топиков, ключи, гарантии, дедуп, DLQ, consumer groups) + схема взаимодействия (текст/диаграмма) + 2 строки про eventual consistency в UX.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Перечислены события сценария (имена-факты в прошедшем времени).
- [ ] Схема `OrderPaid` с eventId/occurredAt/ключом/версией/payload.
- [ ] Корректно выбраны топики (события «многим») и, где уместно, очереди (задача «одному»).
- [ ] Ключ партиционирования = `orderId` с обоснованием порядка.
- [ ] Гарантия at-least-once + идемпотентные/дедуплицирующие потребители.
- [ ] DLQ + ретраи с backoff и лимитом.
- [ ] Consumer groups для масштабирования (с учётом числа партиций).
- [ ] Описано поведение при сбое потребителя (поток не ломается).
- [ ] Упомянута eventual consistency и честный промежуточный статус в UX.

## ПОДСКАЗКА
Синтез: события и имена-факты (зад.02), очередь/топик (01), ключ/порядок (04), дедуп (06), DLQ (05), consumer group (07), event-driven (08). Главные проверки middle+: развязка и устойчивость к сбоям, порядок через ключ, дубли через идемпотентность, ядовитые сообщения через DLQ, масштаб через партиции/группы, честный UX при eventual consistency. Это вход для sequence-диаграмм взаимодействия (модуль 33).
