## Ключ (эталонные ответы)

1. Синхронно — вызывающий ждёт ответа, оба должны быть онлайн; асинхронно — «отправил и забыл», сообщение идёт через брокер, получатель читает когда готов (развязка во времени).
2. Развязка во времени (получатель может быть оффлайн), сглаживание пиков нагрузки (буфер), широковещание событий (один→многим).
3. **Очередь** (point-to-point) — сообщение обрабатывает **один** потребитель из группы; **топик** (pub/sub) — **каждый** подписчик получает свою копию.
4. **Очередь** — распределить работу (задачу сделает кто-то один); **топик** — разослать событие всем заинтересованным потребителям.
5. Модель, где издатель публикует событие, а все подписчики независимо получают копию; издатель не знает о подписчиках.
6. **Kafka** — распределённый лог (топики+партиции), сообщения хранятся и перечитываемы; **RabbitMQ** — брокер очередей (exchange→queue), сообщения обычно удаляются после ack.
7. **Kafka** — потоки событий, высокий throughput, реплей истории, event streaming/sourcing; **RabbitMQ** — классические очереди задач, гибкая маршрутизация (routing keys/exchanges), RPC-over-MQ.
8. Партиция — единица параллелизма и хранения топика; сообщения с одним ключом идут в одну партицию; партиции дают масштабирование чтения.
9. Только **внутри партиции** (Kafka) / очереди — не глобально по топику.
10. Выбрать **ключ партиционирования = id сущности** (например `orderId`) → все её события в одной партиции и упорядочены.
11. Группа потребителей, делящих партиции: каждую партицию читает один потребитель группы → горизонтальное масштабирование обработки.
12. Числом партиций: потребителей в группе полезно не больше, чем партиций (лишние простаивают).
13. **at-most-once** (≤1, возможна потеря), **at-least-once** (≥1, возможны дубли), **exactly-once** (ровно 1, дорого/ограниченно).
14. **at-least-once** — гарантирует доставку, но допускает дубли (ретраи); **exactly-once** — ровно один раз без дублей, но сложно и дорого в распределённой среде.
15. Проще и надёжнее: ретраи гарантируют доставку, а дубли гасит идемпотентный потребитель; exactly-once дорог и ограничен.
16. Потому что at-least-once порождает дубли; идемпотентность гарантирует, что повторная обработка не изменит результат (нет двойного списания/письма).
17. Уникальный id события + таблица обработанных (processed-ids): уже виденные пропускать; либо бизнес-ключ/idempotency key; либо естественная идемпотентность (UPSERT, set-статус).
18. Сообщение, которое стабильно падает при обработке (битый формат, баг) и без защиты блокирует поток или крутится в бесконечных ретраях.
19. **Dead Letter Queue** — очередь для сообщений, не обработанных после N попыток; уводит ядовитые сообщения из потока для отдельного разбора, чтобы не блокировать обработку.
20. Чтобы повторять временно неуспешную обработку (недоступность зависимости) с нарастающей паузой, не забивая систему мгновенными ретраями.
21. Дубли обработаются повторно → двойные списания, повторные письма, рассинхрон данных; нужна дедупликация.
22. Потребители читают события по схеме; несовместимое изменение формата ломает их — нужна обратная/прямая совместимость (Avro/JSON Schema, модуль 28).
23. Когда клиенту нужен **немедленный синхронный ответ** (результат прямо сейчас) — async усложняет (callback/поллинг статуса), проще синхронный вызов.
24. Промежуточный статус («обрабатывается», «заказ принят, оплата проверяется») — честно показать, что результат ещё не финальный, вместо ложного «готово».
25. Расчёт на глобальный порядок; игнор дублей (нет идемпотентности); отсутствие DLQ; «exactly-once везде»; путаница очередь/топик; мало партиций; жёсткая схема события без эволюции; async там, где нужен синхронный ответ.
