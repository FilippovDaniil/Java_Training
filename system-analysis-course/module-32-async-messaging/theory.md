# Модуль 32 — Асинхронная интеграция ⭐

## Зачем этот модуль

Синхронный вызов (REST/gRPC, модули 29–31) связывает сервисы жёстко: вызывающий ждёт ответа, а если получатель недоступен — падает весь сценарий. Когда нужна **развязка во времени** (получатель может быть оффлайн), **сглаживание пиков** (всплеск нагрузки), **широковещание событий** (одно событие — много потребителей) — используют **асинхронную интеграцию через брокер сообщений**.

Для аналитика middle+ это обязательный блок: проектирование событий и их схем, выбор «очередь vs топик» и «Kafka vs RabbitMQ», понимание партиций/порядка/гарантий доставки, DLQ и дедупликации. На собеседовании спрашивают: «чем at-least-once отличается от exactly-once», «как обеспечить порядок», «что такое DLQ», «почему потребитель должен быть идемпотентным».

---

## 1. Синхронно vs асинхронно

| | Синхронно (RPC/REST) | Асинхронно (messaging) |
|--|----------------------|------------------------|
| **Связанность** | временная (оба онлайн) | развязка во времени |
| **Ожидание** | вызывающий ждёт ответ | «отправил и забыл» |
| **Сбой получателя** | ломает вызов | сообщение ждёт в брокере |
| **Пики нагрузки** | передаются напрямую | сглаживаются буфером |
| **Сложность** | проще понять | сложнее (порядок, дубли, согласованность) |

```
Sync:  Сервис A ──запрос──▶ Сервис B   (A ждёт, B должен быть жив)
Async: Сервис A ──событие──▶ [Брокер] ──▶ Сервис B  (A не ждёт, B читает когда готов)
```

> Асинхронность — про **развязку и устойчивость**, но платишь сложностью: eventual consistency, возможные дубли и нарушения порядка нужно проектировать явно (модуль 25).

---

## 2. Очередь vs топик

| Модель | Семантика | Кто получает |
|--------|-----------|--------------|
| **Очередь (queue)** | point-to-point | **один** потребитель из группы обрабатывает сообщение |
| **Топик (topic)** | publish/subscribe | **каждый** подписчик получает свою копию |

```
Очередь:  Producer ──▶ [ queue ] ──▶ один из Consumer'ов (распределение работы)
Топик:    Producer ──▶ [ topic ] ──┬─▶ Подписчик 1
                                    ├─▶ Подписчик 2
                                    └─▶ Подписчик 3   (каждому копия)
```

> **Очередь** = распределить работу (задачу обработает кто-то один). **Топик/pub-sub** = разослать событие всем заинтересованным. «Заказ оплачен» как событие в топик → его независимо читают склад, уведомления, аналитика.

---

## 3. Брокеры: Kafka vs RabbitMQ

| Критерий | Apache Kafka | RabbitMQ |
|----------|--------------|----------|
| **Модель** | распределённый лог (топики+партиции) | брокер очередей (exchange→queue) |
| **Хранение** | сообщения хранятся (retention), можно перечитать | обычно удаляются после ack |
| **Сила** | высокий throughput, event streaming, реплей | гибкая маршрутизация, классические очереди задач |
| **Порядок** | в пределах партиции | в пределах очереди |
| **Типовое применение** | потоки событий, аналитика, event sourcing | задачи/команды, RPC-over-MQ, сложный routing |

> Грубо: **Kafka** — когда поток событий, нужен высокий throughput и возможность перечитать историю; **RabbitMQ** — когда классические очереди задач и гибкая маршрутизация (routing keys, exchange-типы). Это не «лучше/хуже», а разные модели.

---

## 4. Партиции, порядок, consumer groups

**Партиция** — единица параллелизма и хранения в Kafka-топике. Сообщения с одним **ключом партиционирования** попадают в одну партицию.

- **Порядок гарантируется только внутри партиции**, не глобально по топику.
- Чтобы события одной сущности шли по порядку — выбирают **ключ** = id сущности (например, `orderId`), тогда все события заказа в одной партиции и упорядочены.
- **Consumer group** — группа потребителей, делящих партиции: каждая партиция читается ровно одним потребителем группы → горизонтальное масштабирование (до числа партиций).

```
topic "orders" (key = orderId):
  partition 0: order#1 created → paid → delivered   (порядок сохранён)
  partition 1: order#2 created → cancelled
consumer group "billing": C1 читает p0, C2 читает p1   (параллельно)
```

> Ключевой нюанс для аналитика: **порядок — только в партиции**. Если порядок важен для сущности — ключ партиции = id сущности. Параллелизм потребителей ограничен числом партиций.

---

## 5. Гарантии доставки

| Гарантия | Смысл | Риск |
|----------|-------|------|
| **at-most-once** | максимум один раз | возможна **потеря** (нет ретраев) |
| **at-least-once** | минимум один раз | возможны **дубли** (ретраи) |
| **exactly-once** | ровно один раз | сложно/дорого, ограниченно |

> На практике чаще всего **at-least-once** + **идемпотентный потребитель**: ретраи гарантируют, что сообщение дойдёт, а дедупликация на стороне получателя гасит дубли. «exactly-once» в распределённой системе дорог и ограничен (часто это at-least-once + дедуп под капотом).

---

## 6. Дедупликация и идемпотентность потребителя

Раз доставка at-least-once → дубли неизбежны. Потребитель должен быть **идемпотентным** (модуль 25): повторная обработка того же сообщения не меняет результат.

Способы:
- **уникальный id сообщения/события** + таблица обработанных (processed-ids) → пропускать уже обработанные;
- бизнес-ключ операции (idempotency key);
- естественная идемпотентность (UPSERT, установка статуса вместо инкремента).

```
Получено событие eventId=abc:
  if processed.contains(abc): skip          # дубль
  else: handle(); processed.add(abc)        # обработать один раз
```

> Правило: **at-least-once → дубли возможны → потребитель обязан дедуплицировать**. Это самая частая ошибка в async-дизайне: «обрабатываем как пришло» → двойное списание/уведомление.

---

## 7. DLQ, ретраи, «ядовитые» сообщения

**Ядовитое сообщение (poison message)** — то, что стабильно падает при обработке (битый формат, баг). Без защиты оно блокирует очередь или крутится в бесконечных ретраях.

- **Retry с backoff** — повторять обработку с нарастающей паузой (модуль 25).
- **DLQ (Dead Letter Queue)** — после N неудачных попыток сообщение уводят в «очередь мёртвых писем» для ручного/отложенного разбора, чтобы не блокировать поток.

```
Обработка не удалась → retry x3 (backoff) → всё ещё ошибка → [DLQ]
DLQ: разбор/алерт/ручная переотправка — поток не стоит
```

> DLQ + ограниченные ретраи — обязательный паттерн надёжной async-интеграции. Без DLQ ядовитое сообщение либо теряется, либо вечно блокирует обработку.

## ⚠️ Подводные камни

- **Порядок «глобально».** В Kafka порядок только в партиции; нужен ключ = id сущности, иначе события придут вперемешку.
- **Игнор дублей.** at-least-once даёт дубли; без идемпотентного потребителя — двойные списания/письма.
- **Нет DLQ.** Ядовитое сообщение блокирует поток или теряется; нужны ограниченные ретраи + DLQ.
- **«exactly-once везде».** Дорого и ограниченно; обычно at-least-once + дедуп.
- **Очередь вместо топика (и наоборот).** Если событие нужно нескольким — это топик; если задачу делает один — очередь. Путаница ломает доставку «всем/одному».
- **Слишком мало партиций.** Параллелизм потребителей ≤ числу партиций; упрётесь в потолок масштабирования.
- **Схема события без эволюции.** Меняете формат события — ломаете потребителей; нужна совместимость схем (модуль 28, Avro/JSON Schema).
- **Async там, где нужен синхронный ответ.** Если клиенту нужен немедленный результат — messaging усложняет (нужны callback/поллинг статуса).
