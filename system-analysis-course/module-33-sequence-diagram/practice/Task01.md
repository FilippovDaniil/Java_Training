# Задача 01 — Модуль 33: Sequence diagram

## ЗАДАНИЕ
Постройте sequence diagram синхронного сценария оплаты заказа.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**: клиент оформляет и оплачивает заказ. Участники: Клиент (актор), Сервис заказов, Платёжный шлюз, Склад. Поток: клиент создаёт заказ → сервис вызывает оплату → при успехе резервирует товар → возвращает подтверждение.

## ТРЕБОВАНИЯ
- Участники с lifeline; актор Клиент отдельно.
- Синхронные сообщения (сплошные) и return (пунктир) на каждый вызов.
- Активации (activation bar) у обрабатывающих участников.
- Последовательность: создать заказ → charge → зарезервировать → подтвердить.
- Только основной поток (happy path); ветвления — задача 03.

## ФОРМАТ РЕЗУЛЬТАТА
Sequence diagram (PlantUML `@startuml…@enduml` ИЛИ текстовая нотация: участники + сообщения вида `A -> B: ...`, return `B --> A: ...`).

## КРИТЕРИИ ПРИЁМКИ
- [ ] 4 участника (включая актора Клиент).
- [ ] Sync-сообщения сплошные, return пунктиром.
- [ ] Корректный порядок сверху вниз (создать → оплатить → резерв → подтвердить).
- [ ] Активации показаны.
- [ ] Каждый sync-вызов имеет return.

## ПОДСКАЗКА
Опора — §1, §2, §4 теории. Структура: `Клиент -> Orders: создать заказ`, `Orders -> Pay: charge`, `Pay --> Orders: ok`, и т.д. Время сверху вниз. Это визуализация REST-вызова из модуля 29.
