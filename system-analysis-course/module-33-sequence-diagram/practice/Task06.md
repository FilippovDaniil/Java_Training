# Задача 06 — Модуль 33: Sequence diagram

## ЗАДАНИЕ
Найдите ошибки в чужой sequence-диаграмме и предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, дана диаграмма оплаты (текстовая нотация):
```
Клиент -> Orders : создать заказ
Orders ->> Pay : charge(amount)        // async-стрелка
(return от Pay отсутствует)
Orders -> Notify : publish OrderPaid    // sync-стрелка, ждёт ответ
Orders -> Клиент : 201 Created
(нет обработки отклонённой оплаты)
```

## ТРЕБОВАНИЯ
- Найдите минимум 4 ошибки.
- Для каждой — почему ошибка и как исправить.
- Особо: неверная sync/async-нотация (charge должна быть sync с return; publish — async); отсутствие return; отсутствие ветки ошибки оплаты (`alt`).

## ФОРМАТ РЕЗУЛЬТАТА
Список «ошибка → почему → исправление» (≥4) + при желании исправленный фрагмент.

## КРИТЕРИИ ПРИЁМКИ
- [ ] `charge` помечена как async, хотя нужен ответ → должна быть sync с return.
- [ ] Отсутствует return от Pay → добавить.
- [ ] `publish OrderPaid` показана как sync с ожиданием → должна быть async (событие).
- [ ] Нет ветки отклонённой оплаты → добавить `alt` (успех/ошибка).
- [ ] Каждое исправление обосновано.

## ПОДСКАЗКА
Опора — §2, §3, §5 теории. Ключевая путаница — sync vs async: вызов с нужным ответом (charge) = sync + return; публикация события (OrderPaid) = async без ожидания. Плюс happy-path без `alt` игнорирует отказ оплаты.
