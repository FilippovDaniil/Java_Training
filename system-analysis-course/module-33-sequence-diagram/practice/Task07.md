# Задача 07 — Модуль 33: Sequence diagram

## ЗАДАНИЕ
Переведите текстовый use case в sequence diagram.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, use case «Отменить заказ» (модуль 07):
- Актор: Клиент.
- Предусловие: заказ оплачен и ещё не передан в доставку.
- Основной поток: клиент запрашивает отмену → система проверяет статус → возвращает оплату через платёжный шлюз → освобождает зарезервированный товар → подтверждает отмену.
- Альтернатива: заказ уже в доставке → отмена невозможна (ошибка).

## ТРЕБОВАНИЯ
- Переведите актора и участников в lifeline.
- Основной поток → последовательность sync-сообщений с return.
- Проверка предусловия и альтернатива → `alt` (можно отменить / уже в доставке).
- Сохраните порядок: проверка → возврат оплаты → освобождение товара → подтверждение.
- Покажите соответствие шагов use case сообщениям.

## ФОРМАТ РЕЗУЛЬТАТА
Sequence diagram (PlantUML или текстовая нотация) + краткая таблица «шаг use case → сообщение(я)».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Актор Клиент + участники (Сервис заказов, Платёж, Склад).
- [ ] Основной поток как последовательность sync-сообщений с return.
- [ ] `alt`: успешная отмена / невозможна (уже в доставке).
- [ ] Корректный порядок шагов (возврат до освобождения и подтверждения).
- [ ] Есть сопоставление шагов use case и сообщений.

## ПОДСКАЗКА
Опора — §6 теории и модуль 07. Каждый шаг основного потока → одно-два sync-сообщения; альтернатива/исключение → ветка `alt`. Это типовая задача аналитика: формализовать UC в точную последовательность вызовов.
