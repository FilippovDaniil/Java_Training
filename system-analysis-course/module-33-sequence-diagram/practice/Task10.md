# Задача 10 (мини-проект) — Модуль 33: Sequence diagram

## ЗАДАНИЕ
Постройте sequence-диаграммы 3 ключевых сценариев одной системы, с фреймами, sync/async и обработкой ошибок. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** (телемедицина), 3 сценария:
1. **Запись на приём** (sync): пациент выбирает слот → проверка доступности → проверка полиса через внешний шлюз (тайм-аут/ретрай) → бронирование слота → подтверждение. Альтернатива: слот занят / полис не подтверждён.
2. **Оплата приёма** (sync с ветвлением): списание через платёжный шлюз; успех/отказ (`alt`).
3. **Уведомления после оплаты** (async): публикация события `AppointmentPaid` в топик; подписчики — напоминания пациенту и кабинет врача (pub/sub, модуль 32).

## ТРЕБОВАНИЯ
Для каждого сценария — отдельная sequence-диаграмма:
1. Корректные участники и акторы (внешний шлюз/топик — как отдельные участники).
2. Правильная sync/async-нотация (вызовы с return vs публикация события).
3. Фреймы там, где нужно: `alt` (успех/ошибка), `opt` (условный шаг), `loop` (ретраи), `par` (параллельные подписчики/действия).
4. Обработка ошибок и внешних сбоев (тайм-аут/ретрай у внешнего шлюза).
5. Для сценария 3 — async-доставка подписчикам без ожидания.
Короткий вывод: какие сценарии лучше как sequence и почему (vs BPMN/activity).

## ФОРМАТ РЕЗУЛЬТАТА
3 sequence-диаграммы (PlantUML или текстовая нотация) + 2–3 строки вывода о выборе нотации.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сценарий 1: sync-поток + внешний шлюз с тайм-аутом/ретраем (`loop`) + `alt` (слот/полис).
- [ ] Сценарий 2: `alt` успех/отказ оплаты с разными ответами.
- [ ] Сценарий 3: async-публикация события + доставка подписчикам без return (pub/sub).
- [ ] Корректная sync/async-нотация во всех диаграммах.
- [ ] return на всех sync-вызовах; активации показаны.
- [ ] Обработка ошибок/сбоев присутствует (не только happy path).
- [ ] Вывод обосновывает выбор sequence для взаимодействия компонентов.

## ПОДСКАЗКА
Синтез: sync-оплата (зад.01,03), async-события (02), loop/par (04), сага/ошибки (05), внешняя система с ретраями (08), выбор нотации (09). Главные проверки middle+: точная sync/async-семантика, фреймы для реальных (не идеальных) сценариев, обработка внешних сбоев, развязка в async. Эти диаграммы — мост между требованиями/интеграциями (модули 27–32) и реализацией; дальше — безопасность этих взаимодействий (модуль 34).
