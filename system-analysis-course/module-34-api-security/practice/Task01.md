# Задача 01 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Разграничьте аутентификацию (AuthN) и авторизацию (AuthZ) на наборе сценариев и назначьте HTTP-статусы.

## ВХОДНЫЕ ДАННЫЕ
Ситуации в **FoodExpress**:
1. Запрос без токена к `GET /orders/{id}`.
2. Клиент с валидным токеном пытается удалить чужой заказ.
3. Истёкший JWT в запросе.
4. Клиент (роль `client`) обращается к админскому `GET /admin/reports`.
5. Корректный вход с валидным токеном к своему ресурсу.

## ТРЕБОВАНИЯ
- Для каждой ситуации: это про AuthN или AuthZ (или успех).
- Назначьте HTTP-статус (401 / 403 / 200).
- Сформулируйте правило: когда 401, когда 403.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: Ситуация | AuthN/AuthZ | Статус | Пояснение. + правило 401 vs 403.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ситуации 1 и 3 → AuthN, 401 (нет/невалиден токен).
- [ ] Ситуации 2 и 4 → AuthZ, 403 (опознан, но нет прав).
- [ ] Ситуация 5 → успех, 200.
- [ ] Правило: 401 — не аутентифицирован; 403 — аутентифицирован, но нет прав.
- [ ] Ситуация 2 отмечена как BOLA-риск (доступ к чужому объекту).

## ПОДСКАЗКА
Опора — §1 теории. AuthN («кто ты») провален → 401; AuthZ («что можно») провален → 403. Истёкший токен — это AuthN (личность не подтверждена). Доступ к чужому заказу — AuthZ (и конкретно BOLA, §6).
