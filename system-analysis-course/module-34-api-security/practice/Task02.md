# Задача 02 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Подберите OAuth2 flow под каждый тип клиента и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Клиенты системы:
1. Серверное веб-приложение (есть бэкенд, можно хранить client secret).
2. SPA в браузере (публичный клиент, секрет хранить негде).
3. Мобильное приложение (публичный клиент).
4. Внутренний сервис, обращающийся к API без участия пользователя.

## ТРЕБОВАНИЯ
- Для каждого клиента: OAuth2 flow + 1–2 причины.
- Для публичных клиентов объясните роль PKCE.
- Отметьте, какие flow устарели и почему их не брать.
- Кратко: где нужен OIDC (id_token), а где достаточно access token.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: Клиент | Flow | Обоснование. + 2 строки про PKCE и устаревшие flows.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Серверный web → Authorization Code.
- [ ] SPA и мобайл → Authorization Code + PKCE.
- [ ] Внутренний сервис → Client Credentials.
- [ ] Объяснена роль PKCE (защита кода у публичных клиентов без секрета).
- [ ] Отмечены устаревшие Implicit/Password grant как небезопасные.

## ПОДСКАЗКА
Опора — §2 теории. Серверный (есть секрет) → Authorization Code; публичный (нет секрета) → + PKCE; машина-машина → Client Credentials. PKCE защищает перехват кода. Implicit/Password — не использовать.
