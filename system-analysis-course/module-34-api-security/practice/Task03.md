# Задача 03 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Спроектируйте JWT (набор claim) и правила его валидации на стороне API.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, API заказов. Нужен access token для клиента: идентификатор пользователя, роли, срок жизни, издатель, аудитория. API должен корректно валидировать токен.

## ТРЕБОВАНИЯ
- Спроектируйте payload JWT: обязательные стандартные claim (`sub`, `iss`, `aud`, `exp`, `iat`) + кастомные (`roles`, при необходимости `scope`).
- Опишите алгоритм валидации на API (подпись, exp, iss, aud).
- Укажите, что НЕ класть в payload и почему (JWT не шифрует).
- Опишите стратегию TTL/refresh и отзыва.

## ФОРМАТ РЕЗУЛЬТАТА
JSON-пример payload + список шагов валидации + 2 строки «что не класть» + стратегия TTL/refresh.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Payload с `sub`, `iss`, `aud`, `exp`, `iat`, `roles`.
- [ ] Валидация: проверка подписи, `exp`, `iss`, `aud`.
- [ ] Указано, что секреты/чувствительное в payload не кладут (только подпись, не шифрование).
- [ ] Стратегия: короткий TTL access + refresh-токен.
- [ ] Упомянут отзыв (blacklist/short TTL) при компрометации.

## ПОДСКАЗКА
Опора — §3 теории. Payload: `{sub, iss, aud, exp, iat, roles}`. Валидация обязательно включает проверку **подписи** (иначе подделка) и `exp/iss/aud`. payload читаем → без секретов. Короткий TTL + refresh решает «невозможность отзыва».
