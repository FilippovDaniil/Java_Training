# Задача 04 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Спроектируйте RBAC-модель ролей и прав для API.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, роли: Клиент, Курьер, Сотрудник ресторана, Администратор. Действия над ресурсами: создать заказ, оплатить, посмотреть свой заказ, изменить статус приготовления, изменить статус доставки, отменить заказ, смотреть отчёты, управлять ресторанами.

## ТРЕБОВАНИЯ
- Постройте матрицу «роль × действие» (кто что может).
- Соблюдайте принцип минимальных привилегий.
- Отметьте, где RBAC недостаточно и нужна проверка владения объектом (переход к BOLA/ABAC).
- Свяжите роли с claim `roles` в JWT (задача 03).

## ФОРМАТ РЕЗУЛЬТАТА
Матрица доступа (роль × действие, ✓/✗) + 2–3 строки: где нужна доп. проверка владения + связь с JWT.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Матрица охватывает 4 роли и все действия.
- [ ] Минимальные привилегии (клиент не управляет рестораном, курьер не меняет приготовление и т.п.).
- [ ] Отчёты/управление ресторанами — только Администратор.
- [ ] Отмечено: «посмотреть свой заказ» требует проверки владения (RBAC + объектный уровень).
- [ ] Роли соответствуют claim `roles` в JWT.

## ПОДСКАЗКА
Опора — §5, §6 теории. RBAC = роль→права (матрица). Но «свой заказ» роль не покрывает: нужна проверка владения объектом (иначе BOLA). Роли в матрице = значения `roles` в токене.
