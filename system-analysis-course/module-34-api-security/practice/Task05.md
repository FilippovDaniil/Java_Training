# Задача 05 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Обоснуйте применение mTLS для межсервисного доступа и сравните с токенами/API key.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, внутренние сервисы (заказы, оплата, склад, доставка) общаются между собой (REST/gRPC, модуль 31). Нужно гарантировать, что вызовы идут только от доверенных сервисов, а не от постороннего.

## ТРЕБОВАНИЯ
- Объясните, что такое mTLS и как обе стороны аутентифицируются по сертификатам.
- Сравните mTLS с JWT и API key для межсервисного сценария (что сильнее и почему).
- Отметьте, где mTLS уместен (внутренний/B2B трафик), а где избыточен.
- Кратко: управление сертификатами (выпуск, ротация).

## ФОРМАТ РЕЗУЛЬТАТА
Объяснение mTLS + сравнительная таблица (mTLS / JWT / API key для межсервиса) + 2 строки про область применения и ротацию сертификатов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснена взаимная аутентификация по сертификатам (обе стороны предъявляют).
- [ ] Сравнение с JWT и API key (mTLS — сильная взаимная аутентификация канала).
- [ ] Указано, где mTLS уместен (внутренний/доверенный B2B), где избыточен (публичный браузерный API).
- [ ] Упомянута ротация/управление сертификатами.
- [ ] Отмечено, что mTLS можно сочетать с токеном (канал + идентичность пользователя).

## ПОДСКАЗКА
Опора — §4 теории. mTLS = обе стороны проверяют сертификаты друг друга → сильная аутентификация сервисов. API key слаб (легко утекает), JWT несёт пользователя; mTLS защищает сам канал между сервисами. Часто mTLS (сервис↔сервис) + JWT (пользователь) вместе.
