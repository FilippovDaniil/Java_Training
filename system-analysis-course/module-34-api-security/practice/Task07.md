# Задача 07 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Спроектируйте защиту от BOLA (Broken Object Level Authorization) на конкретном эндпоинте.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect**, эндпоинт `GET /appointments/{id}` (запись на приём). Сейчас: токен валиден → запись отдаётся. Проблема: пациент может подставить чужой `id` и прочитать чужую запись (персональные/медицинские данные — особо чувствительны, 152-ФЗ).

## ТРЕБОВАНИЯ
- Объясните механику атаки BOLA на этом эндпоинте.
- Спроектируйте проверку владения: как сервер убеждается, что запись принадлежит запрашивающему (или у него есть право — например, врач этой записи).
- Опишите ответ при нарушении (403 vs 404 — что безопаснее раскрывать).
- Свяжите с ролями (пациент видит свои; врач — свои приёмы) — RBAC + объектная проверка.
- Отметьте чувствительность данных (приватность, модуль 09).

## ФОРМАТ РЕЗУЛЬТАТА
Разбор атаки + алгоритм проверки владения + выбор статуса ответа + связь с ролями/приватностью.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснена атака (подстановка чужого id при валидном токене).
- [ ] Проверка владения: запись.ownerId == текущий пользователь (или роль с правом на объект).
- [ ] Выбран и обоснован статус (403; вариант 404, чтобы не раскрывать существование чужого объекта).
- [ ] RBAC (пациент/врач) + объектная проверка вместе.
- [ ] Учтена чувствительность мед-данных (152-ФЗ/приватность).

## ПОДСКАЗКА
Опора — §6, §1 теории. BOLA лечится проверкой владения на каждом доступе к объекту: сверять владельца объекта с субъектом токена (`sub`). 404 вместо 403 не раскрывает факт существования чужого ресурса. Аутентификация без объектной проверки = дыра.
