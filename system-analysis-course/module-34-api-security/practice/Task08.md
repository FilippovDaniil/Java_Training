# Задача 08 — Модуль 34: Безопасность интеграций

## ЗАДАНИЕ
Спроектируйте управление секретами и токенами: хранение, ротация, минимальные привилегии.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress** использует: пароль БД, ключ платёжного шлюза, приватный ключ подписи JWT, API-ключи партнёров. Сейчас часть лежит в `application.yml` в гите.

## ТРЕБОВАНИЯ
- Укажите, где НЕ должны храниться секреты (код/гит/ConfigMap) и где должны (secret-хранилище/env).
- Опишите ротацию (периодичность, возможность быстрого отзыва/перевыпуска).
- Примените принцип минимальных привилегий (scope) к каждому секрету.
- Опишите стратегию для токенов: короткий TTL access + refresh, отзыв при компрометации.
- Отметьте аудит доступа к секретам.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: Секрет | Где хранить | Ротация | Scope/минимум прав. + стратегия токенов + 1 строка про аудит.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Явно: секреты не в коде/гите/ConfigMap; в secret-хранилище (Vault/K8s Secret)/env.
- [ ] Ротация описана (периодичность + быстрый отзыв).
- [ ] Минимальные привилегии/scope для каждого секрета.
- [ ] Токены: короткий TTL + refresh + отзыв при компрометации.
- [ ] Упомянут аудит доступа к секретам.

## ПОДСКАЗКА
Опора — §7 теории и раздел про конфигурацию (CLAUDE.md: секреты → Secret, не ConfigMap). Главное: убрать секреты из гита в secret-хранилище, ротация, минимальный scope, короткий TTL токенов. Это и про приватность данных (152-ФЗ).
