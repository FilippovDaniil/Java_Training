## Ключ (эталонные ответы)

1. **Аутентификация (AuthN)** — «кто ты» (установление личности); **авторизация (AuthZ)** — «что тебе можно» (проверка прав). AuthN идёт первой, AuthZ — после.
2. **401 Unauthorized** — не аутентифицирован (не опознан); **403 Forbidden** — опознан, но нет прав.
3. Протокол **делегированной авторизации**: приложение получает доступ к ресурсам от имени пользователя через access token, без передачи пароля.
4. **OIDC** — слой **аутентификации** поверх OAuth2: добавляет `id_token` (кто пользователь). OAuth2 сам по себе — про авторизацию доступа (access token).
5. **Authorization Code** (обмен кода на токен на бэкенде) — самый безопасный для серверных web/мобайл с бэкендом.
6. **Authorization Code + PKCE** — для публичных клиентов (SPA, мобайл); PKCE защищает код авторизации от перехвата (нет клиентского секрета).
7. **Client Credentials** — приложение получает токен под себя (нет пользователя), для сервис-сервис взаимодействия.
8. Три части через точку: **header** (alg/typ), **payload** (claims), **signature** (подпись).
9. `sub` (субъект/пользователь), `iss` (издатель), `aud` (аудитория), `exp` (истечение), `iat`/`nbf` (выпущен/не ранее), кастомные (`roles`, `scope`).
10. Проверить **подпись** (ключом издателя), `exp` (не истёк) и `nbf`/`iat`, `iss` и `aud` (свой издатель/аудитория); при необходимости — отзыв (blacklist/short TTL).
11. Нет — JWT по умолчанию только **подписан** (целостность/подлинность), payload читаем (base64); секреты в payload класть нельзя.
12. Он самодостаточен и валиден до `exp` — нет простого централизованного отзыва; борются коротким TTL + refresh-токеном и blacklist для критичных случаев.
13. Ключ, идентифицирующий **приложение/партнёра** (не пользователя); ограничения: легко утекает, не привязан к пользователю; нужны ротация, scope, лимиты, привязка к IP.
14. **Mutual TLS** — взаимная аутентификация сторон по сертификатам (обе предъявляют сертификат); применяется для межсервисного и доверенного B2B-доступа.
15. **RBAC** — права по **роли**; **ABAC** — по **атрибутам** (пользователя, ресурса, контекста: владелец, время, регион).
16. Когда решение зависит от контекста/атрибутов, а не только роли: «врач видит карту только своих пациентов», «доступ только в рабочее время/регион».
17. **Broken Object Level Authorization** — доступ к чужому объекту по id (`GET /orders/{чужой}`), потому что проверили аутентификацию, но не владение; угроза №1, т.к. встречается повсеместно и ведёт к утечке данных.
18. Проверять **владение объектом** (принадлежит ли запрашивающему) или наличие права на конкретный объект — на каждом доступе, не только факт аутентификации.
19. API отдаёт лишние поля (фронт их «прячет»); избегать — возвращать только нужные поля (DTO/проекция), не сериализовать всю сущность.
20. Доступ к функции/эндпоинту выше своих прав (обычный юзер вызывает админ-операцию); мера — проверка прав на каждое действие/эндпоинт.
21. Rate limiting, пагинация, тайм-ауты; для GraphQL — лимиты глубины/сложности запроса и persisted queries (модуль 31).
22. Нельзя — в коде/гите/ConfigMap; нужно — в secret-хранилище (Vault, K8s Secret, env), с ограниченным доступом и аудитом.
23. Ротация ограничивает время жизни утёкшего секрета и позволяет быстро отозвать; короткий TTL токена сужает окно угона (плюс refresh).
24. Через `securitySchemes` (тип/где токен) и `security` (какие эндпоинты защищены, какие scope) — контракт фиксирует правила аутентификации (модуль 30).
25. Путаница AuthN/AuthZ; BOLA (нет проверки владения); JWT без проверки подписи/exp; секреты в payload/коде/гите; длинный TTL без отзыва; API key как полная аутентификация; нет лимитов (DoS); устаревшие OAuth2 flows (Implicit/Password).
