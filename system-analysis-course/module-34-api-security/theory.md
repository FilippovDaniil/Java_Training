# Модуль 34 — Безопасность интеграций ⭐

## Зачем этот модуль

API спроектированы (REST/gRPC/GraphQL, модули 29–33), взаимодействия выстроены. Теперь их нужно **защитить**: кто обращается (аутентификация), что ему разрешено (авторизация), как защищён канал и токены, какие угрозы характерны для API. Это завершает Часть 6 «Интеграции и API».

Для аналитика middle+ безопасность — не «магия безопасников», а часть требований и контракта: он фиксирует, какие эндпоинты публичные/защищённые, какие роли и права, какой OAuth2-flow, какие claim в JWT, какие угрозы (OWASP API Top 10) и меры. На собеседовании обязательны: AuthN vs AuthZ, OAuth2/OIDC, структура и валидация JWT, RBAC/ABAC, BOLA и OWASP API Top 10, управление секретами.

## 1. AuthN vs AuthZ

| | Аутентификация (AuthN) | Авторизация (AuthZ) |
|--|------------------------|---------------------|
| **Вопрос** | «Кто ты?» | «Что тебе можно?» |
| **Проверяет** | личность (логин/пароль, токен, сертификат) | права на ресурс/действие |
| **Когда** | первой | после аутентификации |
| **Пример** | вход по JWT | «может ли этот пользователь удалить чужой заказ» |

```
Запрос → [AuthN: кто это?] → [AuthZ: что ему можно?] → доступ/отказ
            401 если не опознан      403 если опознан, но нет прав
```

> Частая путаница на собеседовании. **AuthN** — установление личности (→ 401 Unauthorized если не прошёл). **AuthZ** — проверка прав уже опознанного (→ 403 Forbidden если нет прав). Это два разных шага.

## 2. OAuth2 и OIDC

**OAuth2** — протокол **делегированной авторизации**: приложение получает доступ к ресурсам от имени пользователя без передачи его пароля, через **access token**. **OIDC (OpenID Connect)** — слой **аутентификации** поверх OAuth2 (добавляет `id_token` — кто пользователь).

| Flow (grant) | Для кого | Суть |
|--------------|----------|------|
| **Authorization Code** | веб/мобайл с бэкендом | обмен кода на токен (самый безопасный) |
| **Authorization Code + PKCE** | SPA, мобильные (публичные клиенты) | защита кода от перехвата |
| **Client Credentials** | сервис-сервис (нет пользователя) | приложение получает токен под себя |
| **Device Code** | TV/IoT без браузера | вход через другой девайс |

> OAuth2 = **авторизация доступа** (access token), OIDC = **аутентификация** (id_token, кто пользователь). Выбор flow — по типу клиента: серверный web → Authorization Code; SPA/мобайл → + PKCE; межсервис без пользователя → Client Credentials. Устаревшие Password/Implicit grants не использовать.

## 3. JWT (JSON Web Token)

**JWT** — компактный самодостаточный токен из трёх частей через точку: `header.payload.signature`.

```
header:    { "alg": "RS256", "typ": "JWT" }
payload:   { "sub": "user-42", "roles": ["client"], "exp": 1735000000, "iss": "auth.foodexpress.ru", "aud": "api" }
signature: подпись header+payload секретом/приватным ключом
```

| Claim | Смысл |
|-------|-------|
| `sub` | subject — идентификатор пользователя |
| `iss` | issuer — кто выпустил токен |
| `aud` | audience — для кого токен |
| `exp` | expiration — время истечения |
| `iat`/`nbf` | выпущен / не ранее |
| custom (`roles`, `scope`) | роли/права |

**Валидация JWT** (обязательно на стороне API):
1. проверить **подпись** (по ключу издателя);
2. проверить `exp` (не истёк), `nbf`/`iat`;
3. проверить `iss` и `aud` (свой издатель и аудитория);
4. при необходимости — отзыв (blacklist/short TTL).

> JWT **самодостаточен** (данные внутри) → сервер может не ходить в БД на каждый запрос, но **нельзя «отозвать» легко** (до истечения `exp`). Отсюда: короткий TTL + refresh-токен, blacklist для критичных случаев. Подпись проверять обязательно — иначе токен подделают. Полезную нагрузку JWT **не шифрует** (только подписывает) → не класть в payload секреты.

## 4. API keys и mTLS

| Механизм | Для чего | Особенность |
|----------|----------|-------------|
| **API key** | идентификация приложения/партнёра | простой, но не привязан к пользователю; легко утекает; нужны ротация и ограничение по scope/IP |
| **mTLS** (mutual TLS) | взаимная аутентификация сервисов по сертификатам | обе стороны предъявляют сертификат; сильная аутентификация для межсервисного/B2B-доступа |

> **API key** — слабая идентификация (только «какое приложение»), годится для партнёрских/публичных API с лимитами, но не как полноценная аутентификация пользователя. **mTLS** — сильная взаимная аутентификация по сертификатам, типична для внутреннего межсервисного трафика и доверенных B2B-интеграций (модуль 31, gRPC).

## 5. RBAC vs ABAC

| Модель | Основа решения | Пример |
|--------|----------------|--------|
| **RBAC** (role-based) | роль пользователя | роль `admin` может удалять; `client` — нет |
| **ABAC** (attribute-based) | атрибуты (пользователя, ресурса, контекста) | «врач видит карту только своих пациентов и только в рабочее время» |

> **RBAC** проще и покрывает большинство случаев (роли → права). **ABAC** гибче, когда решение зависит от атрибутов и контекста (владелец ресурса, время, регион), но сложнее. Часто комбинируют: RBAC + проверка владения объектом (это уже про BOLA, §6).

## 6. OWASP API Security Top 10

Главные классы угроз для API (ключевые для аналитика):

| Угроза | Суть | Мера |
|--------|------|------|
| **BOLA** (Broken Object Level Authorization) | доступ к чужому объекту по id (`GET /orders/123` чужого) | проверять **владение** объектом, не только аутентификацию |
| **Broken Authentication** | слабая аутентификация (угон токена, нет TTL) | корректные токены/TTL, защита от brute-force |
| **BOPLA / Excessive Data Exposure** | API отдаёт лишние поля, фронт «прячет» | отдавать только нужные поля (DTO/проекция) |
| **Broken Function Level Authorization** | доступ к админ-функции обычным юзером | проверка прав на каждое действие/эндпоинт |
| **Resource Consumption** | нет лимитов → DoS (в т.ч. тяжёлый GraphQL) | rate limiting, пагинация, лимиты сложности |
| **Security Misconfiguration** | дефолты, открытые отладочные эндпоинты | харденинг, минимум поверхности |

> **BOLA — угроза №1 для API**. Классика: пользователь аутентифицирован, но запрашивает `GET /orders/{чужой_id}` и получает чужие данные, потому что проверили «кто ты», но не «твой ли это объект». Мера: всегда проверять, что объект принадлежит запрашивающему (или есть право).

## 7. Управление секретами

Токены, ключи, пароли БД, сертификаты — **секреты**, и их утечка = компрометация.

- **не хранить в коде/гите** (модуль про конфигурацию): только в secret-хранилище (Vault, K8s Secret, переменные окружения), не в ConfigMap;
- **ротация** — периодическая смена; возможность быстро отозвать/перевыпустить;
- **минимальные привилегии** — токен/ключ только с нужным scope;
- **короткий TTL** для access-токенов + refresh;
- **аудит** доступа к секретам.

> Для аналитика: фиксировать в требованиях, что секреты идут через secret-хранилище (не ConfigMap/не код), имеют ротацию и минимальный scope. Это пересекается с приватностью данных (152-ФЗ/GDPR, модуль 09): персональные данные и согласия — отдельный пласт требований к интеграциям.

## ⚠️ Подводные камни

- **Путаница AuthN/AuthZ.** «Аутентифицирован» ≠ «разрешено»; 401 (не опознан) ≠ 403 (нет прав).
- **BOLA.** Проверяют только аутентификацию, забывают проверку владения объектом → доступ к чужим данным по id.
- **JWT без проверки подписи/exp.** Принятие неподписанного/просроченного токена = подделка/угон.
- **Секреты в JWT payload.** JWT не шифрует payload (только подпись) — секреты туда не класть.
- **Невозможность отзыва JWT.** Длинный TTL без refresh/blacklist → угнанный токен живёт долго; короткий TTL + refresh.
- **API key как полная аутентификация.** Key идентифицирует приложение, не пользователя; легко утекает — нужны ротация/scope/лимиты.
- **Секреты в коде/гите/ConfigMap.** Утечка; только secret-хранилище + ротация.
- **Нет лимитов.** Без rate limiting и лимитов сложности (особенно GraphQL) — DoS.
- **Устаревшие OAuth2 flows.** Implicit/Password grant небезопасны; Authorization Code (+PKCE) / Client Credentials.
