# Задача 06 — Модуль 35: Тестирование для аналитика

## ЗАДАНИЕ
Спроектируйте проверку API в Postman по контракту (статусы, схема, ошибки).

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, эндпоинт `POST /orders` (модули 29–30): тело CreateOrderRequest; ответ 201 со схемой Order + заголовок Location; ошибки 422 (валидация, RFC 7807). Нужно проверить соответствие реализации контракту.

## ТРЕБОВАНИЯ
- Опишите запрос (метод, URL, тело по схеме).
- Опишите Postman-проверки (tests) для успешного кейса: статус 201, заголовок Location, соответствие тела схеме Order, значения полей.
- Опишите проверку негативного кейса: невалидное тело → 422 + формат problem+json.
- Свяжите проверки с контрактом OpenAPI (что именно сверяем).

## ФОРМАТ РЕЗУЛЬТАТА
Описание запроса + список Postman-проверок (позитив + негатив) + связь с контрактом.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Запрос по контракту (тело соответствует CreateOrderRequest).
- [ ] Проверки успеха: статус 201, Location, схема/поля Order.
- [ ] Проверка негатива: 422 + формат RFC 7807 (problem+json).
- [ ] Проверки сверяют именно контракт (статусы/схемы из OpenAPI).
- [ ] Указано, что это контрактная проверка (реализация vs спецификация).

## ПОДСКАЗКА
Опора — §7 теории и модуль 30. В Postman tests: `pm.response.code === 201`, проверка заголовка Location, валидация тела по схеме, проверка 422 для невалидного запроса. Цель — подтвердить, что API соответствует контракту.
