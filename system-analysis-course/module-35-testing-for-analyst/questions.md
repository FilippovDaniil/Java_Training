5. Назовите основные виды тестирования.
6. Что такое приёмочное тестирование (UAT)?
7. Чем функциональное тестирование отличается от нефункционального?

## Ключ (эталонные ответы)

1. Модель соотношения тестов: много дешёвых **unit** внизу, меньше **integration**, мало дорогих **E2E** наверху.
2. **Unit** — отдельный модуль изолированно; **integration** — связка компонентов (сервис+БД, API); **E2E** — сквозной сценарий через весь интерфейс.
3. Unit дёшевы, быстры, стабильны; E2E дороги, медленны и хрупки — их держат мало, только на ключевые сквозные сценарии.
4. Перекос в сторону E2E (много медленных хрупких тестов, мало unit) → долгие прогоны, нестабильность, дорогая поддержка.
5. Unit, integration, system, E2E, приёмочное (UAT), регрессионное, дымовое (smoke); функциональное и нефункциональное.
6. **UAT** — проверка заказчиком/пользователем, что система соответствует ожиданиям и критериям приёмки; финальное подтверждение перед приёмкой.
7. **Функциональное** — что система делает (поведение); **нефункциональное** — как (производительность, безопасность, юзабилити — модуль 09).
8. Проверка, что ранее работавшая функциональность не сломалась после изменений.
9. **Use case** — что система должна делать (сценарий с потоками); **test case** — как проверить это (предусловия, шаги, данные, ожидаемый результат).
10. Обычно **много**: основной поток + альтернативы + исключения + граничные/негативные случаи → набор тест-кейсов на один UC.
11. ID/название, предусловия, тест-данные, шаги, ожидаемый результат (и приоритет).
12. Чтобы тест был однозначным и воспроизводимым: «HTTP 402, статус не изменился» проверяемо, «работает корректно» — нет.
13. Разбиение входных данных на группы с одинаковым поведением системы; достаточно проверить одно значение из каждого класса.
14. Проверка краёв диапазона (min−1, min, max, max+1) — там чаще всего off-by-one ошибки.
15. Границы: 17, 18 (нижний край) и 120, 121 (верхний край).
16. **Позитивный** — корректные данные, система работает (happy path); **негативный** — неверные данные/действия, система корректно их отклоняет/обрабатывает.
17. Happy path не выявляет ошибки валидации, обработки сбоев, прав и границ — большинство дефектов ловят негативные сценарии.
18. Для **контрактной проверки API**: отправляет запросы, сверяет статусы, схему и тело ответа со спецификацией; коллекции/Newman для автоматизации.
19. Сверить реальное поведение API со спецификацией (OpenAPI, модуль 30): корректные статусы, схемы ответов, формат ошибок (RFC 7807).
20. Критерии приёмки — основа приёмочных тестов: каждый критерий → проверка; Gherkin (Given/When/Then) прямо ложится в тест-кейсы.
21. Связь «требование ↔ тест-кейс» (traceability matrix, модуль 10): показывает, что каждое требование покрыто тестом и наоборот.
22. **V-модель** (модуль 02): каждому уровню требований соответствует уровень тестирования (бизнес→приёмочное, системные→системное, и т.д.).
23. Конкретные входные значения для теста; без подготовленных данных тест невоспроизводим и неоднозначен.
24. Быстрая базовая проверка работоспособности после сборки/деплоя (ключевые функции живы) перед более глубоким тестированием.
25. Только happy path; расплывчатый ожидаемый результат; «один UC = один тест»; нет трассировки к требованиям; перевёрнутая пирамида; игнор тест-данных; тест-дизайн постфактум вместо параллельно с требованиями.
