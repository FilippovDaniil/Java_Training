# Модуль 35 — Тестирование для аналитика

## Зачем этот модуль

Требования собраны и специфицированы (Части 2–6). Теперь нужно проверить, что система делает **именно то, что требовалось**. Тестирование — это обратная сторона требований: каждое требование/критерий приёмки должно быть проверяемо. Аналитик не пишет автотесты, но **участвует в тест-дизайне**: переводит use cases и критерии приёмки в тест-кейсы, проверяет покрытие требований, валидирует API по контракту (Postman).

Это начало Части 7 «Качество, поставка, документация». На собеседовании middle+ спрашивают: пирамида тестов, виды тестирования, чем use case отличается от test case, как спроектировать тест-кейсы (классы эквивалентности, граничные значения), что такое позитивный/негативный сценарий.

---

## 1. Пирамида тестов

Соотношение видов тестов по количеству и стоимости.

```
        /\        E2E / UI      ← мало, дорого, медленно, хрупко
       /  \
      /----\      Integration   ← средне (сервисы, БД, API)
     /------\
    /--------\    Unit           ← много, дёшево, быстро
```

| Уровень | Что проверяет | Цена/скорость |
|---------|---------------|---------------|
| **Unit** | отдельный модуль/функцию изолированно | дёшево, быстро, много |
| **Integration** | взаимодействие компонентов (сервис+БД, API) | средне |
| **E2E** | сквозной сценарий через весь интерфейс | дорого, медленно, хрупко, мало |

> Принцип: **много дешёвых unit, меньше integration, мало E2E**. Антипаттерн — «перевёрнутая пирамида» / «рожок мороженого» (много медленных E2E, мало unit) → долгие хрупкие прогоны. Аналитику важно понимать уровни, чтобы критерии приёмки ложились на нужный уровень.

---

## 2. Виды тестирования

| Вид | Цель |
|-----|------|
| **Unit** | корректность отдельного блока кода |
| **Integration** | корректность связки компонентов/интеграций |
| **System** | система целиком против требований |
| **E2E** | сквозной пользовательский сценарий |
| **Приёмочное (UAT)** | соответствие ожиданиям заказчика/критериям приёмки |
| **Регрессионное** | старое не сломалось после изменений |
| **Дымовое (smoke)** | базовая работоспособность после сборки |

Отдельно — **функциональное** (что система делает) vs **нефункциональное** (как: нагрузочное, безопасность, юзабилити — модуль 09).

> Для аналитика ключевое — **приёмочное тестирование (UAT)**: проверка, что реализованное соответствует критериям приёмки требований. Связь с V-моделью (модуль 02): уровням требований соответствуют уровни тестирования.

## 3. Use case vs test case

| | Use case | Test case |
|--|----------|-----------|
| **Отвечает на** | что система должна делать (сценарий) | как проверить, что она это делает |
| **Содержит** | акторы, потоки, pre/postconditions | предусловия, шаги, тест-данные, ожидаемый результат |
| **Сколько** | один UC | **много** тест-кейсов на один UC |

```
Use case "Оплатить заказ"
 ├── TC1: успешная оплата (happy path)
 ├── TC2: недостаточно средств → отказ
 ├── TC3: истёкшая карта → ошибка
 └── TC4: таймаут платёжного шлюза → ретрай/ошибка
```

> Один use case → **набор** тест-кейсов: основной поток + альтернативы + исключения + граничные случаи. Use case описывает поведение, test case — конкретную проверку этого поведения с данными и ожидаемым результатом.

---

## 4. Структура тест-кейса

| Поле | Содержание |
|------|------------|
| **ID / название** | уникальный идентификатор, краткое имя |
| **Предусловия** | состояние системы/данных до теста |
| **Тест-данные** | конкретные входные значения |
| **Шаги** | пронумерованные действия |
| **Ожидаемый результат** | что должно произойти (конкретно, проверяемо) |
| **Приоритет** | важность |

```
TC: Оплата заказа недостаточными средствами
Предусловия: заказ создан, статус CREATED; на карте баланс < суммы заказа
Шаги:
  1. Отправить POST /orders/{id}/payment
Ожидаемый результат:
  - HTTP 402; статус заказа остался CREATED; средства не списаны
```

> Тест-кейс должен быть **однозначным и воспроизводимым**: конкретные данные, конкретный ожидаемый результат (не «всё работает», а «HTTP 402, статус не изменился»). Это требование проверяемости из модуля 04.

## 5. Тест-дизайн: классы эквивалентности и граничные значения

Нельзя проверить все входные значения → выбирают представительные.

- **Классы эквивалентности** — разбить входные данные на группы, внутри которых система ведёт себя одинаково; проверять по одному значению из класса.
- **Граничные значения (boundary)** — проверять края диапазонов (min−1, min, max, max+1), где чаще всего баги.

```
Возраст пользователя, допустимо 18..120:
Классы:   [<18 невалид] [18..120 валид] [>120 невалид]
Границы:  17, 18, 120, 121   ← ключевые точки проверки
```

> Это базовые техники тест-дизайна: **классы эквивалентности** сокращают число тестов, **граничные значения** ловят off-by-one ошибки на краях. Аналитик применяет их, выводя тест-кейсы из требований с диапазонами/правилами.

---

## 6. Позитивные и негативные сценарии

| Сценарий | Проверяет |
|----------|-----------|
| **Позитивный** | система работает на корректных данных (happy path) |
| **Негативный** | система корректно обрабатывает неверные данные/действия (валидация, ошибки) |

> Частая ошибка — тестировать только happy path. Зрелая тест-модель содержит **негативные сценарии**: невалидный ввод, нарушение прав (модуль 34), недоступность зависимости, граничные нарушения. Негативные сценарии обычно выявляют больше дефектов.

## 7. Проверка API в Postman (по контракту)

Аналитик проверяет API против контракта (OpenAPI, модуль 30): отправляет запросы, сверяет статусы, схему и тело ответа.

```
Postman-проверка POST /orders:
  - запрос по контракту (тело по схеме CreateOrderRequest)
  - проверки (tests):
      • статус == 201
      • есть заголовок Location
      • тело соответствует схеме Order (id, status, total)
      • status == "created"
```

> Postman (и коллекции/Newman) — инструмент аналитика для **контрактной проверки API**: соответствует ли реальное поведение спецификации (статусы, схемы, ошибки RFC 7807). Это связывает тестирование с contract-first (модуль 30).

## ⚠️ Подводные камни

- **Только happy path.** Без негативных сценариев и границ — дефекты на краях и при ошибках проходят.
- **Расплывчатый ожидаемый результат.** «Работает корректно» непроверяемо; нужен конкретный результат (статус, состояние).
- **Use case = test case.** Один UC ≠ один тест; нужен набор тестов (потоки + исключения + границы).
- **Нет трассировки к требованиям.** Непонятно, что покрыто; нужна связь требование → тест-кейс (traceability, модуль 10).
- **Перевёрнутая пирамида.** Много E2E, мало unit → медленно и хрупко.
- **Игнор тест-данных.** Без подготовленных данных тест невоспроизводим.
- **Тестирование «постфактум».** Тест-дизайн от критериев приёмки нужно вести вместе с требованиями, а не после разработки.
