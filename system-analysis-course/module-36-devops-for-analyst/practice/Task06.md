# Задача 06 — Модуль 36: DevOps для аналитика

## ЗАДАНИЕ
Составьте план миграции БД без даунтайма по паттерну expand/contract.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, нужно переименовать колонку `total` → `amount` в таблице заказов (по сути — изменить контракт данных). Сервис работает 24/7, во время выката одновременно крутятся старая и новая версии кода.

## ТРЕБОВАНИЯ
- Распишите шаги expand → migrate → contract для этого изменения.
- На каждом шаге укажите, что делает БД и что делает код (двойная запись).
- Объясните, почему нельзя сделать переименование одним шагом.
- Свяжите с breaking changes API (модули 28/30): добавлять безопасно, удалять — поздним шагом.

## ФОРМАТ РЕЗУЛЬТАТА
Пошаговый план (expand/migrate/contract) с действиями БД и кода + объяснение «почему не одним шагом».

## КРИТЕРИИ ПРИЁМКИ
- [ ] EXPAND: добавить новую колонку `amount`, не трогая `total` (совместимо).
- [ ] MIGRATE: код пишет в обе колонки; перенос существующих данных.
- [ ] CONTRACT: когда все на новой версии — удалить `total`.
- [ ] Объяснено, почему одномоментное переименование сломает старую версию во время выката.
- [ ] Связь с breaking changes (добавлять безопасно, удалять поздним шагом).

## ПОДСКАЗКА
Опора — §7 теории. Никогда не ломать одним шагом: expand (добавить amount) → код пишет в оба → contract (убрать total). Это та же логика, что non-breaking/breaking в API (модули 28, 30).
