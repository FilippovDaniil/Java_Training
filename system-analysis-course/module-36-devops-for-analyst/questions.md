## Ключ (эталонные ответы)

1. Изоляция этапов: разработка, тестирование, финальная проверка и боевая эксплуатация не мешают друг другу; проблемы ловятся до прода.
2. Один и тот же неизменный артефакт собирается один раз и продвигается по средам; отличается только конфигурация (а не пересборка под каждую среду).
3. **Stage** — максимально похожая на prod среда для финальной проверки/UAT без риска для пользователей; ловит проблемы, которые проявятся только «как в проде».
4. **Continuous Integration** — каждый коммит автоматически собирается и прогоняется тестами; ранняя интеграция и обнаружение проблем.
5. **Continuous Delivery** — артефакт всегда готов к выкату (релиз по кнопке); **Continuous Deployment** — выкат на prod автоматически после прохождения пайплайна.
6. commit → build → unit-тесты → анализ кода → сборка артефакта/образа → deploy на test → integration/e2e → stage → (approve) → prod.
7. **Контейнер** — приложение со всеми зависимостями в изолированной упаковке; решает «работает у меня» — одинаковое поведение в любой среде.
8. Управляет контейнерами: масштабирование, перезапуск упавших, выкат новых версий, балансировка, самовосстановление (модуль 26).
9. **Образ** — неизменный артефакт версии (шаблон); **контейнер** — запущенный экземпляр образа.
10. Понимание воспроизводимости сред, быстрого отката (предыдущий образ), масштабирования/доступности — влияет на NFR и план поставки.
11. Чтобы один артефакт работал в разных средах за счёт разной конфигурации (build once, deploy many); 12-factor: конфиг в окружении.
12. В защищённом secret-хранилище (Secret/Vault/env), не в коде/гите/ConfigMap (модуль 34).
13. Переключатель, включающий/выключающий функциональность без передеплоя.
14. **Деплой** — код размещён на сервере; **релиз** — функция доступна пользователям. Feature flag разводит их: код задеплоен, но выключен флагом.
15. Можно мгновенно выключить проблемную фичу флагом (OFF), не откатывая деплой.
16. Накапливаются (флаги-зомби), усложняют код и тестирование (комбинации флагов); нужна гигиена и удаление отработавших.
17. Recreate, Rolling, Blue-Green, Canary.
18. **Blue-Green** — две полные среды, мгновенное переключение/откат трафика; **Canary** — постепенный выкат новой версии на растущую долю с наблюдением метрик.
19. **Blue-Green** — когда нужен мгновенный откат и есть ресурсы на дублирующую среду; **Canary** — когда важно раннее обнаружение проблем на малой доле реального трафика.
20. Поэтапная замена реплик старой версии на новую без даунтайма (по чуть-чуть, не всё сразу).
21. Во время выката одновременно работают старая и новая версии кода; ломающее изменение схемы сломает одну из них.
22. **Expand** — добавить новое, не ломая старое (совместимо); **migrate** — код пишет в оба, данные переносятся; **contract** — после перехода всех удалить старое.
23. Та же логика: добавлять безопасно (expand ≈ non-breaking), удалять/менять — отдельным поздним шагом (contract); как версионирование API (модули 28, 30).
24. Оценить влияние на бизнес и пользователей, помочь принять решение об откате/обходе, зафиксировать инцидент и последующие изменения требований; не оставаться в стороне.
25. Критерии приёмки выполнены, тесты пройдены, миграция спланирована (expand/contract), конфигурация/секреты готовы по средам, флаг/стратегия выката и план отката определены, документация/обучение готовы, мониторинг настроен.
