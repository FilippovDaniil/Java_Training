# Модуль 36 — DevOps для аналитика

## Зачем этот модуль

Фича разработана и протестирована (модуль 35). Теперь её нужно **доставить пользователю** — безопасно, предсказуемо, с возможностью отката. Аналитик не настраивает пайплайны, но обязан понимать **путь фичи до прода**: среды, CI/CD, контейнеры, релизные стратегии, миграции БД, feature flags. Это влияет на требования (как выкатываем, как откатываем, что включаем флагом) и на участие аналитика в инцидентах.

На собеседовании middle+ спрашивают: зачем разные среды, что такое CI/CD, чем canary отличается от blue-green, как выкатить миграцию БД без даунтайма, зачем feature flags. Понимание DevOps отличает аналитика, который «пишет ТЗ», от того, кто думает о поставке.

---

## 1. Среды (environments)

Код проходит через изолированные среды, прежде чем попасть к пользователю.

| Среда | Назначение |
|-------|------------|
| **dev** | разработка, эксперименты |
| **test / QA** | тестирование командой/QA |
| **stage (pre-prod)** | копия прода для финальной проверки/UAT |
| **prod** | продакшен, реальные пользователи |

```
dev → test → stage → prod   (промоушн: одно и то же неизменное содержимое продвигается дальше)
```

> Принцип: **один и тот же артефакт продвигается по средам** (build once, deploy many), а отличается только конфигурация. Stage максимально похож на prod, чтобы поймать проблемы до релиза. Аналитик понимает, на какой среде проходит UAT и где данные «боевые».

---

## 2. CI/CD

| | Расшифровка | Суть |
|--|-------------|------|
| **CI** (Continuous Integration) | непрерывная интеграция | каждый коммит автоматически собирается и прогоняется тестами |
| **CD** (Continuous Delivery/Deployment) | непрерывная поставка/развёртывание | артефакт автоматически готов к выкату (delivery) или выкатывается (deployment) |

Типовые этапы пайплайна:
```
commit → build → unit-тесты → анализ кода → сборка артефакта/образа →
deploy на test → integration/e2e тесты → deploy на stage → (approve) → deploy на prod
```

> CI ловит проблемы рано (каждый коммит проверен), CD делает поставку повторяемой и быстрой. Для аналитика: пайплайн — это место, где «проходят» критерии приёмки и тесты (модуль 35); провал на этапе = фича не едет дальше.

---

## 3. Контейнеры (Docker) и оркестрация (K8s)

**Контейнер** — упакованное приложение со всеми зависимостями, одинаково работающее в любой среде («работает у меня» → «работает везде»).

**Оркестратор (Kubernetes)** управляет контейнерами: масштабирование, перезапуск упавших, выкат новых версий, балансировка.

| Понятие | Смысл (для аналитика) |
|---------|------------------------|
| **образ (image)** | неизменный артефакт версии приложения |
| **контейнер** | запущенный экземпляр образа |
| **под/реплика** | единица запуска; несколько реплик → масштаб и доступность |
| **health check** | liveness/readiness — жив ли под, готов ли к трафику (модуль 26) |

> Аналитику не нужно писать Dockerfile, но важно понимать: контейнеризация даёт **воспроизводимость сред** и быстрый откат (вернуть предыдущий образ); оркестратор даёт масштабирование и самовосстановление — это влияет на NFR (доступность, масштабируемость, модуль 26).

---

## 4. Конфигурация vs код

Код одинаков во всех средах — отличается **конфигурация** (URL БД, ключи, лимиты, флаги).

- конфигурация **вынесена** из кода (env-переменные, configmap/secret);
- **секреты** — отдельно и защищённо (Secret, не в коде/гите, модуль 34);
- одна сборка → разное поведение по средам за счёт конфигурации.

> Правило **12-factor**: конфигурация в окружении, не в коде. Это позволяет «build once, deploy many». Аналитик фиксирует, что отличается по средам (эндпоинты внешних систем, лимиты, флаги) — это часть требований к поставке.

---

## 5. Feature flags (переключатели функций)

**Feature flag** — переключатель, включающий/выключающий функциональность **без передеплоя**.

Зачем:
- **постепенный выкат** (раскатить на 5% → 50% → 100%);
- **разделение деплоя и релиза** (код в проде, но выключен, пока не готов);
- **быстрый «откат»** фичи (выключить флаг вместо отката деплоя);
- **A/B-тесты** и таргетинг на сегменты.

```
Деплой ≠ релиз: код выкатан с флагом OFF → включаем флаг для 5% → расширяем → 100%
Проблема? → флаг OFF (мгновенно, без отката деплоя)
```

> Feature flags разводят **деплой** (код на сервере) и **релиз** (функция доступна пользователю). Для аналитика — мощный инструмент: фичу можно включать постепенно/по сегментам и мгновенно выключать. Минус — флаги накапливаются (нужна гигиена/удаление старых).

---

## 6. Релизные стратегии

| Стратегия | Суть | Плюс |
|-----------|------|------|
| **Recreate** | погасить старое, поднять новое | просто, но есть даунтайм |
| **Rolling** | поэтапная замена реплик | без даунтайма, постепенно |
| **Blue-Green** | две среды (blue=старая, green=новая), переключение трафика | мгновенный откат (вернуть трафик на blue) |
| **Canary** | новая версия для малой доли (5%), наблюдение, расширение | раннее обнаружение проблем на малой аудитории |

```
Blue-Green: [blue v1 ← трафик] [green v2 готова] → переключить → [green v2 ← трафик], blue в резерве
Canary:     v2 на 5% → метрики ок? → 25% → 50% → 100% (иначе откат)
```

> **Blue-Green** — мгновенное переключение и откат (две полные среды). **Canary** — постепенный выкат с наблюдением метрик на малой доле. Выбор: нужен мгновенный откат и есть ресурсы на дубль среды → blue-green; нужно раннее обнаружение на реальном трафике → canary. Часто canary + feature flags.

---

## 7. Миграции БД без даунтайма (expand/contract)

Схему БД нельзя менять «на ходу» ломающим образом — старая и новая версии кода работают одновременно во время выката. Паттерн **expand/contract** (parallel change):

```
1. EXPAND   — добавить новое (новая колонка/таблица), не ломая старое (совместимо в обе стороны)
2. MIGRATE  — код пишет и в старое, и в новое; перенести данные
3. CONTRACT — когда все на новой версии — удалить старое
```

> Ключ: **никогда не делать ломающее изменение одним шагом**. Сначала добавить (expand, обратно совместимо), дать коду перейти, потом удалить старое (contract). Это та же логика, что breaking changes в API (модули 28, 30): добавлять безопасно, удалять — отдельным поздним шагом. Аналитик закладывает это в план поставки фич с изменением данных.

---

## ⚠️ Подводные камни

- **Деплой = релиз.** Без feature flags нельзя выкатить код «выключенным» и включать постепенно; деплой и релиз стоит разводить.
- **Ломающая миграция одним шагом.** Падение во время выката; нужен expand/contract.
- **Разные артефакты по средам.** Тестировали одно, выкатили другое; build once, deploy many.
- **Конфигурация/секреты в коде.** Нельзя менять по средам, утечки; конфиг в окружении, секреты в Secret (модуль 34).
- **Нет плана отката.** Релиз без стратегии отката рискован; blue-green/флаги дают быстрый откат.
- **Stage не похож на prod.** Проблемы всплывают только в проде; stage должен быть максимально близок.
- **Накопление старых флагов.** Флаги-зомби усложняют систему; нужна гигиена.
- **Аналитик вне инцидента.** При сбое/откате аналитик нужен для оценки влияния на бизнес и решения; не оставаться в стороне.
