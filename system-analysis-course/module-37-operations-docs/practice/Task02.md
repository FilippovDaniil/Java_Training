# Задача 02 — Модуль 37: Эксплуатация и сопровождение

## ЗАДАНИЕ
Напишите ADR (Architecture Decision Record) по архитектурному решению.

## ВХОДНЫЕ ДАННЫЕ
**FoodExpress**, решение: выбрать способ хранения каталога ресторанов и меню. Рассматривались: реляционная БД, документная (MongoDB), поиск (OpenSearch). Контекст: гибкая структура меню, полнотекстовый поиск, высокая частота чтения.

## ТРЕБОВАНИЯ
- Оформите ADR со всеми полями: заголовок/ID, статус, контекст, решение, последствия (+/−), альтернативы (и почему отвергнуты).
- Решение и обоснование должны быть конкретны и связаны с контекстом.
- Покажите trade-off (что выиграли, чем пожертвовали).
- Отметьте, чем ADR ценнее обычной записи в проектной документации.

## ФОРМАТ РЕЗУЛЬТАТА
ADR по шаблону (ID, статус, контекст, решение, последствия, альтернативы).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все поля ADR присутствуют.
- [ ] Контекст описывает проблему и ограничения.
- [ ] Решение конкретно и обосновано контекстом.
- [ ] Последствия включают и плюсы, и минусы.
- [ ] Рассмотрены альтернативы с причиной отказа.
- [ ] Отмечена ценность ADR (фиксация «почему»).

## ПОДСКАЗКА
Опора — §2 теории. Шаблон: Контекст → Решение → Статус → Последствия(+/−) → Альтернативы. Главное — зафиксировать **почему** выбрали и что отвергли. Используй знания модулей 16–20 (выбор хранилища).
