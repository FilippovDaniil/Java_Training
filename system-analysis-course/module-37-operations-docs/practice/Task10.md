# Задача 10 (мини-проект) — Модуль 37: Эксплуатация и сопровождение

## ЗАДАНИЕ
Соберите пакет эксплуатационной документации и сопровождения для фичи/системы: ADR + миграция + обучение + поддержка. Синтез теории модуля.

## ВХОДНЫЕ ДАННЫЕ
**MedConnect** выводит в эксплуатацию модуль «электронная медкарта»: заменяет бумажный/legacy-учёт, используется врачами и пациентами, данные критичны (152-ФЗ, модуль 34). Нужен пакет для запуска и сопровождения.

## ТРЕБОВАНИЯ
Сдайте пакет:
1. **Карта документации** — какие документы нужны, для кого (виды + аудитории).
2. **ADR** — одно ключевое решение по модулю (контекст/решение/последствия/альтернативы).
3. **План миграции данных** из legacy (этапы, маппинг-фрагмент, очистка, верификация, стратегия запуска).
4. **План обучения** пользователей (врачи + пациенты).
5. **Модель поддержки** — линии L1/L2/L3, SLA по приоритетам, обработка change requests.
6. **Оценка результатов** — метрики «до/после» и критерий успеха внедрения.
7. Учёт приватности (152-ФЗ) в документации и миграции.
Короткий вывод: как пакет обеспечивает устойчивую эксплуатацию и развитие.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный пакет (карта документации, ADR, план миграции, план обучения, модель поддержки+SLA, оценка результатов) + 2–3 строки вывода.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Карта документации с видами и аудиториями.
- [ ] ADR со всеми полями по ключевому решению.
- [ ] План миграции: этапы, маппинг, очистка, верификация, стратегия запуска (для мед-данных — осторожная).
- [ ] План обучения для врачей и пациентов.
- [ ] Модель поддержки L1/L2/L3 + SLA + обработка change requests.
- [ ] Оценка результатов «до/после» с критерием успеха.
- [ ] Учтена приватность (152-ФЗ) в миграции/документации.
- [ ] Вывод об устойчивой эксплуатации и развитии.

## ПОДСКАЗКА
Синтез всего модуля: документация (зад.01,04), ADR (02), миграция (03), поддержка/SLA (05,07), change requests (06), оценка (08), обучение (09). Главные проверки middle+: документация под аудиторию, обоснованные решения (ADR), безопасная миграция критичных данных с верификацией, рабочая модель поддержки, замкнутый цикл «цель → оценка результата». Это завершает Часть 7 — дальше сквозной capstone (модуль 38).
