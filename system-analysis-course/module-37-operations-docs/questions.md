## Ключ (эталонные ответы)

1. У разных читателей разные задачи: пользователю — как работать, админу — как обслуживать, поддержке — что делать при сбое, разработке — как устроено; один документ «на всех» неэффективен.
2. Требования (SRS/ТЗ — команда/заказчик), проектная/архитектурная (разработка), API/контракты (интеграторы), руководство пользователя (конечные пользователи), руководство администратора (эксплуатация), runbook (поддержка/DevOps).
3. **Руководство пользователя** — как работать с функциями системы; **руководство администратора** — установка, настройка, обслуживание, эксплуатация.
4. Эксплуатационный документ для поддержки/DevOps: регламенты и пошаговые действия при инцидентах/типовых операциях.
5. Стандартный набор эксплуатационных документов (руководство пользователя, администратора/оператора, описание применения и др.); актуально для госзаказа.
6. **Architecture Decision Record** — короткая запись одного архитектурного решения: контекст, решение, статус, последствия, альтернативы.
7. Контекст (проблема/ограничения), решение (что выбрали), статус (предложено/принято/заменено), последствия (плюсы/минусы), рассмотренные альтернативы.
8. ADR фиксирует **почему** принято решение (обоснование, отвергнутые альтернативы) — это теряется в обычной документации; лёгкая институциональная память проекта.
9. Анализ/профилирование источника → маппинг полей → очистка → перенос (ETL) → верификация → стратегия запуска (big-bang/параллельно).
10. Legacy-данные обычно грязные (дубли, пропуски, разные форматы); без профилирования миграция перенесёт мусор и аномалии.
11. Сопоставление полей источника с полями целевой модели + правила трансформации значений.
12. **Big-bang** — разовое переключение на новую систему (быстро, рискованно); **параллельный** — обе системы работают одновременно со сверкой (безопаснее, дольше/дороже).
13. Чтобы убедиться, что данные перенесены полно и корректно: сверка количества записей, контрольных сумм, выборочная ручная проверка.
14. **L1** — первая линия, типовые обращения по инструкции и маршрутизация; **L2** — техническая диагностика нетиповых проблем; **L3** — разработка, дефекты кода и сложные инциденты.
15. Снизу вверх: L1 решает типовое или эскалирует на L2; L2 разбирает сложное или эскалирует на L3 (код/баг).
16. Наполнять базу знаний L1/L2 (как отвечать), анализировать инциденты, превращать повторяющиеся проблемы в change requests/новые требования.
17. Пользователи, поддержка (инциденты), бизнес (новые цели), регуляторика, технические потребности.
18. Анализ, что изменение затронет: требования, код, данные, интеграции, NFR, сроки, риски, стоимость.
19. Иначе неконтролируемые правки и scope creep, рассинхрон артефактов; нужно понять последствия и обновить связанное (трассировка).
20. Соглашение об уровне сервиса поддержки: сроки реакции и решения обращений по приоритетам.
21. Чем критичнее (Critical → прод лежит), тем жёстче сроки реакции/решения; Low — по плану. Пример: Critical — реакция минуты, решение часы; Low — дни.
22. Сравнение бизнес-метрик «до/после» внедрения: достигнуты ли цели фичи/проекта (модуль 03).
23. Фичу делают ради бизнес-цели; без проверки по метрикам неясно, дала ли она результат — иначе развитие «вслепую».
24. Повторяющиеся инциденты/обращения и дефекты L3 анализируются и оформляются как change requests / новые требования (замыкание цикла).
25. Документация «для галочки»/неактуальная; решения без ADR; миграция без профилирования/очистки/верификации; big-bang без отката; отсутствие линий поддержки; change request без импакт-анализа; внедрили и не оценили результат.
