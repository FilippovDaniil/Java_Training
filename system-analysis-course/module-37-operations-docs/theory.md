# Модуль 37 — Эксплуатация и сопровождение ⭐

## Зачем этот модуль

Система выкачена (модуль 36). Но жизненный цикл на этом не заканчивается — начинается **эксплуатация и сопровождение**, самая длинная фаза ЖЦ (модуль 02). Здесь аналитик: готовит эксплуатационную документацию, фиксирует архитектурные решения (ADR), планирует миграцию данных из legacy, организует обучение пользователей, понимает линии поддержки (L1–L3), обрабатывает change requests, оценивает результаты внедрения.

Это завершает Часть 7 «Качество, поставка, документация». На собеседовании middle+ спрашивают: какие виды документации и для кого, что такое ADR, как спланировать миграцию данных, чем L1/L2/L3 отличаются, как обрабатывать change request, что такое SLA поддержки. Зрелый аналитик думает не только «как построить», но и «как это будут эксплуатировать и развивать».

---

## 1. Виды документации

Документацию подбирают под **этап и аудиторию**.

| Вид | Для кого | Содержание |
|-----|----------|------------|
| **Требования (SRS/ТЗ)** | команда, заказчик | что система должна делать (модуль 11) |
| **Проектная/архитектурная** | разработка, архитекторы | как устроена (диаграммы, ADR) |
| **API/контракты** | интеграторы, фронт/бэк | OpenAPI/WSDL (модули 30–31) |
| **Руководство пользователя** | конечные пользователи | как работать с системой |
| **Руководство администратора** | эксплуатация/админы | установка, настройка, обслуживание |
| **Эксплуатационная (runbook)** | поддержка/DevOps | действия при инцидентах, регламенты |

По **ГОСТ 34** существует стандартный набор эксплуатационных документов (руководство пользователя, руководство администратора/оператора, описание применения и др.) — актуально для госзаказа.

> Принцип: **документация ≠ один большой документ**, а набор под аудиторию. Пользователю — как работать; админу — как обслуживать; поддержке — что делать при сбое; разработке — как устроено. Аналитик определяет, какие документы нужны на каком этапе.

---

## 2. ADR (Architecture Decision Record)

**ADR** — короткая запись **одного архитектурного решения**: контекст, решение, последствия. Ведётся как история решений проекта.

```
ADR-012: Использовать Kafka для событийной интеграции заказов
Статус: принято (2026-06)
Контекст: нужна развязка сервисов, высокий поток событий, реплей истории.
Решение: брокер Kafka (топик per домен, ключ = orderId).
Последствия:
  + развязка, throughput, реплей
  − операционная сложность, eventual consistency
Альтернативы: RabbitMQ (отклонён — нужен реплей/стриминг), синхронный REST (отклонён — связанность).
```

| Поле ADR | Смысл |
|----------|-------|
| **Контекст** | какая проблема, какие силы/ограничения |
| **Решение** | что выбрали |
| **Статус** | предложено/принято/заменено |
| **Последствия** | плюсы и минусы выбора |
| **Альтернативы** | что рассматривали и почему отвергли |

> ADR отвечает на вопрос «**почему так сделали**» через год, когда контекст забыт. Это лёгкий формат (не толстый документ), фиксирующий решение и его обоснование. Аналитик/архитектор ведёт ADR по ключевым решениям — это институциональная память проекта.

## 3. Миграция данных из legacy

При замене/внедрении системы нужно перенести данные из старой. Это отдельный проект внутри проекта.

Этапы:
```
1. Анализ источника (профилирование: качество, объём, аномалии)
2. Маппинг полей (источник → целевая модель), правила трансформации
3. Очистка (дубли, пустые, несогласованные значения)
4. Перенос (ETL): extract → transform → load
5. Верификация (сверка контрольных сумм/количеств, выборочная проверка)
6. Стратегия запуска: big-bang vs параллельная работа двух систем
```

| Подход к запуску | Суть | Риск |
|------------------|------|------|
| **Big-bang** | разом переключиться на новую | быстрее, но рискованно |
| **Параллельный** | две системы работают параллельно, сверка | безопаснее, дороже/дольше |

> Ключевое для аналитика: **маппинг и качество данных**. Legacy-данные грязные (дубли, пропуски, разные форматы) — без профилирования и очистки миграция перенесёт мусор. Обязательна верификация после переноса (количество записей, контрольные суммы, выборка). Связь с expand/contract (модуль 36) для схемы.

## 4. Линии поддержки L1–L3

| Линия | Кто | Что решает |
|-------|-----|------------|
| **L1** | первая линия (операторы/хелпдеск) | типовые обращения по инструкции, регистрация, маршрутизация |
| **L2** | техническая поддержка/эксперты | нетиповые проблемы, диагностика, известные баги |
| **L3** | разработка/архитекторы | дефекты в коде, сложные инциденты, требуют изменений |

```
Обращение → L1 (решил по инструкции? → закрыть : эскалация) → L2 (диагностика) → L3 (код/баг)
```

> Эскалация снизу вверх: L1 фильтрует типовое, L2 разбирает сложное, L3 правит код. Аналитик помогает наполнять базу знаний L1/L2 (как отвечать), а инциденты L3 часто превращаются в change requests / новые требования.

---

## 5. Change requests и управление изменениями

После запуска требования продолжают меняться. Источники change requests: пользователи, поддержка (инциденты), бизнес (новые цели), регуляторика, технические нужды.

Обработка change request (модуль 10):
```
CR → регистрация → импакт-анализ (что затронет: требования, код, данные, сроки, риски)
   → оценка/приоритизация → решение (принять/отклонить/отложить) → реализация → верификация
```

> Каждое изменение проходит **импакт-анализ**: что затрагивает (функции, интеграции, данные, NFR), стоимость, риски. Нельзя «просто внести правку» — нужно понять последствия и обновить связанные артефакты (трассировка, модуль 10). Это защита от неконтролируемого разрастания (scope creep).

## 6. SLA поддержки и оценка результатов

**SLA поддержки** — соглашение об уровне сервиса: сроки реакции и решения по приоритетам.

| Приоритет | Время реакции | Время решения (пример) |
|-----------|---------------|------------------------|
| **Critical** (прод лежит) | 15 мин | 4 часа |
| **High** | 1 час | 1 день |
| **Medium** | 4 часа | 3 дня |
| **Low** | 1 день | по плану |

**Оценка результатов внедрения** — сравнение «до/после» по бизнес-метрикам (модуль 03): достигли ли целей фичи/проекта.

```
Цель фичи «подписка»: +15% повторных заказов за квартал
Оценка: метрика до = X, после = Y → достигли/нет → решение (развивать/откатить/менять)
```

> SLA задаёт измеримые обязательства поддержки (реакция/решение по приоритету). Оценка результатов закрывает цикл: фичу делали ради бизнес-цели (модуль 03) — нужно проверить по метрикам, достигнута ли она, иначе работа «вслепую». Это вход для новых гипотез и change requests.

## ⚠️ Подводные камни

- **Документация «для галочки».** Не под аудиторию/неактуальная — бесполезна; подбирать вид под читателя, поддерживать в актуальном состоянии.
- **Решения без ADR.** Через год никто не помнит «почему так»; фиксировать ключевые решения в ADR.
- **Миграция без профилирования/очистки.** Перенос мусора из legacy; обязательны анализ качества и очистка.
- **Миграция без верификации.** Не сверили — потеряли/исказили данные; контроль количества/сумм/выборки.
- **Big-bang без отката.** Разовое переключение без плана отката — высокий риск; рассмотреть параллельную работу.
- **Нет линий поддержки.** Всё валится на разработку (L3); нужна фильтрация L1/L2.
- **Change request без импакт-анализа.** Неконтролируемые правки, scope creep; каждый CR — через анализ влияния.
- **Не оценили результат.** Внедрили и забыли; без оценки «до/после» неясно, достигнута ли цель.
