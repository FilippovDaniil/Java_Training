# Задача 04 — Capstone: Модель данных (ER + словарь)

## ЗАДАНИЕ
Постройте модель данных платформы: ER-модель (концептуальная + логическая) и словарь данных, с учётом телеметрии.

## ВХОДНЫЕ ДАННЫЕ
Сущности из домена и требований (задачи 01–03): Пользователь, Дом, Помещение, Устройство, ТипУстройства, Телеметрия, Сценарий, Триггер, Действие, Доступ/Роль.

## ТРЕБОВАНИЯ
- Концептуальная ER: сущности, связи, кратности (1:1, 1:N, M:N + разрешение M:N).
- Логическая модель: PK/FK, ключевые атрибуты, типы.
- Отдельно обоснуйте хранение **телеметрии** (time-series: объём, партиционирование, отличие от справочных данных).
- Словарь данных для 4–5 ключевых сущностей (атрибут, тип, описание, ограничения).
- Нормализация справочной части (≥3NF) + где уместна денормализация.

## ФОРМАТ РЕЗУЛЬТАТА
ER-диаграмма (концепт. + логич., PlantUML/Crow's foot/текст) + словарь данных + заметка о хранении телеметрии.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Концептуальная ER со связями и кратностями (разрешён M:N, напр. Доступ).
- [ ] Логическая: PK/FK, типы для ключевых сущностей.
- [ ] Обосновано отдельное хранение телеметрии (time-series).
- [ ] Словарь данных для 4–5 сущностей.
- [ ] Справочная часть нормализована (≥3NF); отмечена денормализация при необходимости.
- [ ] Сущности согласованы с предыдущими задачами (единый язык).

## ПОДСКАЗКА
Опора — модули 16, 17, 19, 20. Ключевой нюанс IoT: телеметрия — огромный append-only поток (time-series), хранится и масштабируется иначе, чем справочники (партиционирование по времени/устройству). Доступ (пользователь×дом×роль) — разрешение M:N. Эта модель — основа концептуальной модели и API.
