# Задача 05 — Capstone: Концептуальная модель + состояния

## ЗАДАНИЕ
Постройте концептуальную модель домена (class diagram) и state machine ключевой сущности.

## ВХОДНЫЕ ДАННЫЕ
Модель данных из задачи 04. Ключевые сущности с поведением: Устройство (жизненный цикл подключения) и Сценарий (активен/выключен/выполняется).

## ТРЕБОВАНИЯ
- Концептуальная модель (class diagram): классы домена, атрибуты, связи, кратности, агрегация/композиция, инварианты/бизнес-правила.
- Отметьте отличие концептуальной модели от модели БД (задача 04) и от классов кода.
- State machine **Устройства**: состояния (напр. Добавлено → Онлайн ↔ Оффлайн, Ошибка, Выведено) с событиями, guard, действиями.
- Проверьте корректность: нет недостижимых/тупиковых состояний, есть путь к финалу.

## ФОРМАТ РЕЗУЛЬТАТА
Class diagram (концептуальная модель) + state machine Устройства (PlantUML/текст) + заметка «концепт. модель vs БД vs код».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Class diagram с классами, кратностями, агрегацией/композицией.
- [ ] Указаны инварианты/бизнес-правила.
- [ ] Объяснено отличие концептуальной модели от ER/кода.
- [ ] State machine Устройства с событиями/guard/действиями.
- [ ] Нет недостижимых/тупиковых состояний; есть финал.
- [ ] Согласованность с моделью данных (задача 04).

## ПОДСКАЗКА
Опора — модули 13, 15. Концептуальная модель — про предметную область (не таблицы): связи, композиция (Дом ◆—Помещение), инварианты («сценарий не выполняется, если выключен»). State machine устройства отражает неустойчивость связи IoT (Онлайн↔Оффлайн). Состояния → валидация в API (задача 07).
