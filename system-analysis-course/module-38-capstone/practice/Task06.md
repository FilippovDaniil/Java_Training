# Задача 06 — Capstone: Архитектура

## ЗАДАНИЕ
Спроектируйте архитектуру платформы: стиль, границы сервисов / bounded contexts, размещение данных.

## ВХОДНЫЕ ДАННЫЕ
Требования и NFR (задача 02), модель данных (задача 04). Особенности: высокий поток телеметрии, реальное время для критичных сценариев, edge (домашний хаб) + облако.

## ТРЕБОВАНИЯ
- Выберите архитектурный стиль (монолит/модульный монолит/микросервисы) с обоснованием через trade-offs.
- Выделите bounded contexts (DDD): минимум 4 (напр. Устройства, Телеметрия, Автоматизация/Правила, Доступ, Уведомления).
- Постройте context map (отношения между контекстами).
- Разместите данные по сервисам (database per service; телеметрия отдельно).
- Учтите edge/cloud: что исполняется на хабе, что в облаке (для критичных сценариев).
- Зафиксируйте 1 ключевое решение как ADR (контекст/решение/последствия/альтернативы).

## ФОРМАТ РЕЗУЛЬТАТА
Выбор стиля + обоснование + список bounded contexts + context map + размещение данных + edge/cloud + ADR ключевого решения.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Стиль выбран и обоснован через trade-offs (с учётом потока телеметрии).
- [ ] ≥4 bounded contexts по разным языкам/ответственности.
- [ ] Context map (отношения контекстов).
- [ ] Размещение данных (телеметрия отдельно от справочников).
- [ ] Учтён edge/cloud для критичных сценариев (исполнение на хабе).
- [ ] ADR ключевого решения со всеми полями.

## ПОДСКАЗКА
Опора — модули 21, 24, 37 (ADR). Телеметрия (высокий поток) — кандидат на отдельный сервис/контекст. Критичные сценарии — исполнять на edge-хабе (работают без облака). Обоснуй стиль честно: микросервисы дают масштаб, но дороже операционно (возможен модульный монолит как старт). Это вход для интеграций (задача 07).
