# Задача 07 — Capstone: Интеграции и API

## ЗАДАНИЕ
Спроектируйте интеграции и контракты платформы: способы интеграции, REST/async, фрагмент OpenAPI, sequence ключевого сценария.

## ВХОДНЫЕ ДАННЫЕ
Архитектура и контексты (задача 06). Взаимодействия: пользователь ↔ платформа (управление), устройства → платформа (телеметрия), платформа → устройства (команды), платформа ↔ внешние (голосовой ассистент, уведомления).

## ТРЕБОВАНИЯ
- Карта интеграций: для каждого взаимодействия — стиль (sync REST / async messaging) + обоснование.
- Обоснуйте, почему телеметрия — async/streaming, а команда устройству — sync (или async с подтверждением).
- Фрагмент OpenAPI: 1–2 эндпоинта управления (напр. `POST /scenarios`, `GET /devices/{id}`) с схемами, статусами, ошибками (RFC 7807), security.
- Схема события телеметрии (async): eventId, deviceId (ключ партиции), occurredAt, payload, версия.
- Sequence-диаграмма критичного сценария «протечка» (sync/async, фреймы alt/loop для ошибок/ретраев).

## ФОРМАТ РЕЗУЛЬТАТА
Карта интеграций + фрагмент OpenAPI (валидный) + схема события телеметрии + sequence критичного сценария (PlantUML/текст).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Карта интеграций со стилями и обоснованием (sync vs async).
- [ ] Телеметрия — async/streaming с ключом партиции = deviceId; команда — sync/подтверждение.
- [ ] Фрагмент OpenAPI с схемами, статусами, ошибками RFC 7807, security.
- [ ] Схема события телеметрии (eventId/deviceId/occurredAt/версия).
- [ ] Sequence критичного сценария с обработкой ошибок (alt/loop).
- [ ] Согласованность с архитектурой (задача 06) и процессами (задача 03).

## ПОДСКАЗКА
Опора — модули 27, 29, 30, 32, 33. Телеметрия = поток → брокер (async), порядок по deviceId. Управление = REST (sync). Sequence «протечки» — из BPMN задачи 03: датчик→правило→команда клапану (ретрай) + уведомление. OpenAPI должен быть валиден (модуль 30).
