# Задача 09 — Capstone: Тест-модель + план поставки

## ЗАДАНИЕ
Постройте тест-модель ключевой фичи и план её поставки.

## ВХОДНЫЕ ДАННЫЕ
Фича «Сценарий автоматизации» (создание + срабатывание, вкл. критичный «протечка»). Требования/критерии приёмки (задача 02), API (задача 07), устойчивость/безопасность (задача 08).

## ТРЕБОВАНИЯ
- **Тест-модель:** тест-кейсы из требований (позитив + негатив + границы), особое внимание критичному сценарию (ложное/пропущенное срабатывание); трассировка требование → тест (без пробелов); контрактные проверки API (Postman) по OpenAPI задачи 07.
- **Тест-дизайн:** классы эквивалентности/границы для правил сценария (условия триггера).
- **План поставки:** среды, миграция данных (expand/contract при изменении схемы), стратегия выката (canary для рискованной автоматизации) + feature flag, план отката, мониторинг (бизнес + технические метрики).
- Чек-лист готовности к релизу.

## ФОРМАТ РЕЗУЛЬТАТА
Тест-модель (тест-кейсы, трассировка, Postman-проверки, тест-дизайн) + план поставки (среды, миграция, выкат+флаг, откат, мониторинг) + чек-лист готовности.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Тест-кейсы позитив/негатив/границы; критичный сценарий покрыт (ложное/пропущенное срабатывание).
- [ ] Трассировка требование → тест без пробелов.
- [ ] Контрактные Postman-проверки по OpenAPI.
- [ ] Тест-дизайн (классы/границы) для условий триггера.
- [ ] План поставки: среды, expand/contract, canary+флаг, откат, мониторинг.
- [ ] Чек-лист готовности к релизу.
- [ ] Согласованность с требованиями/API/безопасностью.

## ПОДСКАЗКА
Опора — модули 35, 36. Для критичной автоматизации негативные сценарии важнее всего: что если датчик ложно сработал? не сработал? устройство оффлайн? Выкат рискованной логики — canary + флаг (мгновенный откат). Это финальный слой перед сборкой пакета (задача 10).
