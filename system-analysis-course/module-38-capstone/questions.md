## Ключ (эталонные ответы)

1. С **бизнес-контекста** (м.03): стейкхолдеры, цели, метрики, MVP. Требования выводятся из целей; без понимания «зачем» соберём не то.
2. Power/interest grid (м.03): владелец дома, члены семьи, гости, производители устройств, провайдер платформы, регулятор (ПДн), поддержка. Не забыть «нечеловеческих» акторов: сами устройства и автоматизацию.
3. SMART: конкретная, измеримая, достижимая, релевантная, ограниченная по времени (напр. «увеличить долю домов с ≥1 активным сценарием до 60% за 6 мес»). North Star — отражает ценность (напр. число активных автоматизаций/дом).
4. По ценности и риску: ключевые сценарии (управление устройством, базовый сценарий, критичные датчики), отложить второстепенное (сложные правила, аналитика). MVP проверяет гипотезу ценности (м.03).
5. **Функциональное** — что делает («создать сценарий», «включить лампу»); **NFR** — как (срабатывание сценария < 1 с; доступность 99.9%; защита доступа к замку). 
6. Latency срабатывания (p95), доступность (SLA), пропускная способность телеметрии (события/сек), надёжность критичных сценариев, безопасность; измеримость — конкретные пороги (м.09, ISO 25010).
7. **BPMN** — бизнес-процесс с ролями для бизнес-аудитории; **sequence** — техническое взаимодействие компонентов/устройств во времени (м.12, 33).
8. Старт по событию датчика → шлюз проверки (протечка?) → параллельно: команда на клапан (перекрыть) + уведомление; обработка ошибок (клапан не ответил → эскалация/повтор), таймер (м.12).
9. Пользователь, Дом, Помещение, Устройство, ТипУстройства, Телеметрия, Сценарий, Триггер, Действие, Доступ. Телеметрия связана с Устройством (1:N), большой объём (м.16).
10. Телеметрия — append-only time-series, огромный объём, запросы по времени/устройству → специализированное/партиционированное хранилище; справочные данные — реляционные, нормализованные (м.17, 19, 20).
11. Концептуальная модель — сущности предметной области и связи безотносительно реализации; модель БД — таблицы/ключи/типы/индексы (м.15, 16).
12. Состояния устройства (Добавлено → Онлайн → Оффлайн → Ошибка → Выведено) с переходами по событиям (подключилось, потеряло связь, сбой) + guard/действия (м.13).
13. По границам контекстов и нагрузке: телеметрия (высокий поток) отдельно от управления/правил; обосновать через trade-offs (операционная сложность vs масштаб), возможен модульный монолит как старт (м.21).
14. Bounded contexts: Управление устройствами, Телеметрия, Автоматизация/Правила, Доступ/Пользователи, Уведомления — по разным языкам и ответственности (м.24).
15. **Sync** — команда устройству (включить, требуется ответ), управление через API; **async** — поток телеметрии, события датчиков, уведомления (м.27, 32).
16. Огромный поток показаний, развязка, буферизация пиков, много потребителей (правила, аналитика, история) → брокер/стриминг; REST на каждое показание не масштабируется (м.32).
17. Идемпотентность по eventId/дедуп на потребителе; порядок — ключ партиции = deviceId (события устройства упорядочены) (м.25, 32).
18. SLA из требований к критичности: для критичных сценариев (протечка/дым) жёсткие latency/доступность; рассчитать доступность из компонентов (цепочка/резерв), задать SLO/SLI (м.26).
19. Graceful degradation: edge-хаб исполняет критичные сценарии локально при потере облака; буфер телеметрии; статус «оффлайн» в UI; повтор отправки команд (м.25, 26).
20. Несанкционированный доступ к замкам/камерам, перехват команд, утечка данных о присутствии, компрометация устройства, DoS на платформу; критичность выше среднего (физическая безопасность) (м.34).
21. Строгая authn (OAuth2/OIDC, MFA для критичных действий), authz по ролям (владелец/член/гость) + проверка владения объектом (анти-BOLA: не управлять чужим домом), mTLS устройств, аудит (м.34).
22. ПДн: данные о присутствии людей, видео с камер, расписания. Приватность: согласие, минимизация, шифрование, ограничение доступа, право на удаление; гости — ограниченные права (м.34, 09).
23. Тест-кейсы из требований/критериев приёмки (позитив+негатив, границы), трассировка требование → тест (нет пробелов), контрактные проверки API (м.35).
24. Среды, миграция данных (expand/contract), стратегия выката (canary для рискованной автоматизации) + feature flag, план отката, мониторинг (м.36); для критичных сценариев — особая осторожность.
25. Защитить = обосновать выбор (trade-offs/ADR), назвать риски/ограничения/допущения, показать соответствие целям и NFR, продемонстрировать связность (цель→требование→модель→API→тест) и единый язык; честно указать неопределённость.
