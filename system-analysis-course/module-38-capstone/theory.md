# Модуль 38 — Capstone: сквозной кейс «Умный дом / IoT»

## Зачем этот модуль

Все инструменты освоены (Части 1–7): от роли аналитика и требований до моделирования, данных, архитектуры, интеграций, тестирования и эксплуатации. **Capstone** собирает их в один сквозной проект на **новом домене** — вы проходите полный цикл аналитической работы «с нуля» и формируете цельный **аналитический пакет уровня middle+**.

Это не новая теория, а **методичка прохождения** и **чек-лист пакета**. 10 задач связаны одной системой: каждая опирается на предыдущие и на конкретные части курса. На выходе — комплект артефактов и умение **защитить решение** (обосновать выбор, trade-offs, риски). Именно это проверяют на финальном собеседовании middle+: не знание отдельных нотаций, а способность пройти путь от бизнес-цели до плана поставки связно.

---

## 1. Домен: «Умный дом / IoT-платформа»

Единый контекст для всех 10 задач (новый домен — не FoodExpress/MedConnect).

**Продукт.** Платформа умного дома: пользователи управляют устройствами (лампы, замки, термостаты, камеры, датчики движения/температуры/протечки/дыма) через мобильное приложение и голос; настраивают **сценарии автоматизации** (правила «если…то…»); видят **телеметрию** (поток данных с датчиков) и историю.

**Ключевые сущности (предварительно):** Пользователь, Дом, Помещение, Устройство, Тип устройства, Телеметрия (показания), Сценарий/Правило, Триггер, Действие, Событие, Доступ/роль (владелец/член семьи/гость).

**Характерные черты домена (важны для анализа):**
- **IoT-специфика:** много устройств, большой поток телеметрии (time-series), edge (домашний хаб) + облако, неустойчивая связь устройств.
- **Реальное время:** срабатывание сценариев должно быть быстрым; критичные события (протечка, дым) — приоритетны.
- **Безопасность и приватность:** доступ к замкам/камерам — чувствителен; данные о присутствии людей дома — ПДн.
- **Надёжность:** ложное/пропущенное срабатывание критичного сценария недопустимо; деградация при потере связи.
- **Разные акторы:** владелец, члены семьи, гости (ограниченные права), сама платформа (автоматизация), внешние системы (голосовой ассистент, погодный сервис; уведомления пользователю).

> Этот контекст намеренно «сырой»: часть деталей вы доопределяете сами (как в реальном проекте). Все задачи ведут одну систему — не выдумывайте новый домен в каждой.

## 2. Как проходить capstone (методичка)

Последовательность повторяет логику курса — от «зачем» к «как поставить». Каждая задача = слой аналитического пакета.

```
Бизнес-контекст (зачем)        → задача 01  [Часть 1, 3]
   ↓
Требования (что)               → задача 02  [Часть 2]
   ↓
Процессы (как идёт работа)     → задача 03  [Часть 3: BPMN]
   ↓
Данные (что храним)            → задача 04  [Часть 4: ER, словарь]
   ↓
Поведение/структура (модели)   → задача 05  [Часть 3: концепт. модель, состояния]
   ↓
Архитектура (из чего строим)   → задача 06  [Часть 5: стиль, DDD]
   ↓
Интеграции и API (как связано) → задача 07  [Часть 6: REST/async, OpenAPI, sequence]
   ↓
Устойчивость/NFR/безопасность  → задача 08  [Части 5–6: распределёнка, NFR, security]
   ↓
Качество и поставка            → задача 09  [Часть 7: тесты, релиз]
   ↓
Сборка пакета + защита         → задача 10  [синтез всего]
```

> Идёте сверху вниз: каждый слой опирается на предыдущий (требования → из них модели → из моделей архитектура → …). Артефакты должны быть **согласованы между собой** (трассировка цель → требование → модель → тест) — это главный признак зрелого пакета.

## 4. Чек-лист аналитического пакета middle+

Готовый пакет по итогам capstone должен содержать:

- [ ] **Бизнес-контекст:** стейкхолдеры (power/interest), бизнес-цели (SMART), метрики (North Star + дерево), границы MVP.
- [ ] **Требования:** каталог (бизнес → пользовательские → функциональные), NFR (измеримые, ISO 25010), приоритизация (MoSCoW/RICE), трассировка.
- [ ] **Процессы:** BPMN ключевых сценариев (с ошибками/таймерами).
- [ ] **Данные:** ER (концепт. + логич.), словарь данных, нормализация.
- [ ] **Модели:** концептуальная модель домена, state machine ключевой сущности.
- [ ] **Архитектура:** стиль + обоснование, границы сервисов/bounded contexts, размещение данных.
- [ ] **Интеграции/API:** способы интеграции, REST/async-контракты, фрагмент OpenAPI, sequence ключевых сценариев.
- [ ] **Устойчивость/NFR/безопасность:** SLA/SLO, паттерны устойчивости (saga/idempotency/CB), модель безопасности (authn/authz, угрозы), приватность.
- [ ] **Качество/поставка:** тест-модель (трассировка к требованиям), план поставки (среды, миграция, выкат, откат).
- [ ] **Связность:** сквозная трассировка цель → требование → модель → API → тест; единый язык терминов.

## 5. Защита решения (что проверяют на middle+)

Аналитический пакет мало собрать — нужно **защитить**:
- **Обоснование выбора:** почему этот стиль/хранилище/стратегия, а не альтернатива (trade-offs, как в ADR, м.37).
- **Риски и ограничения:** что может пойти не так, что вне scope, какие допущения.
- **Соответствие целям:** как решение закрывает бизнес-цели и NFR.
- **Связность:** требования ↔ модели ↔ API ↔ тесты не противоречат друг другу.
- **Честность неопределённости:** где данных не хватает — прямо сказать, а не выдумать.

> Признак middle+ — не «нарисовал все диаграммы», а **связное обоснованное решение с осознанными trade-offs**: вы понимаете, почему сделали так, что это стоит и где риски.

## ⚠️ Подводные камни

- **Несогласованность артефактов.** Сущности в ER не совпадают с моделью/API; термины «плавают». Держите единый язык и трассировку.
- **Смена домена между задачами.** Каждая задача — про ту же IoT-платформу, а не новый кейс.
- **Диаграммы ради диаграмм.** Артефакт без связи с требованиями/решением бесполезен; каждый должен работать на цель.
- **Happy-path везде.** Игнор ошибок/сбоев/безопасности (особенно критично для IoT: протечка/замки). Закладывайте негатив и устойчивость.
- **NFR «на словах».** «Быстро/надёжно» без измеримости (м.09); для IoT — latency срабатывания, доступность, поток телеметрии.
- **Игнор приватности/безопасности.** Камеры/замки/присутствие людей — чувствительно; authn/authz, BOLA, ПДн обязательны.
- **Нет защиты решения.** Собрали артефакты, но не обосновали выбор/риски; middle+ — про обоснование, не объём.
- **Перфекционизм/масштаб.** Не утоните: фокус на MVP и ключевых сценариях, а не «вся платформа целиком».
