# Модуль 39 — ИИ для системного аналитика (бонус)

## Зачем этот модуль

Весь аналитический цикл пройден (Части 1–8). Этот бонусный модуль — про то, как **ускорять** эту работу с помощью больших языковых моделей (LLM): генерировать черновики артефактов (SQL, требования, user stories, диаграммы, тест-кейсы), структурировать «сырьё», делать предварительное ревью. Но с одним жёстким правилом: **ИИ создаёт черновик — аналитик верифицирует и несёт ответственность**.

LLM не заменяет системного аналитика: он не знает вашего контекста целиком, выдумывает факты (галлюцинации) и не отвечает за результат. На собеседовании middle+ всё чаще спрашивают: как вы используете ИИ, как формулируете промпт, как ловите галлюцинации, что нельзя отдавать в ИИ. Этот модуль даёт практический и **критический** взгляд: ИИ как инструмент-ускоритель под контролем аналитика.

---

## 1. Что LLM может и чего не может в работе СА

| Хорошо подходит (черновик) | Плохо / опасно (без проверки) |
|----------------------------|-------------------------------|
| структурировать сырые заметки в требования | принимать решения за аналитика |
| черновик user stories / критериев приёмки | факты о вашем домене/данных (выдумает) |
| генерация SQL по описанию схемы | точные цифры/НПА/специфика проекта |
| черновик диаграмм (PlantUML/BPMN-текст) | гарантии корректности и полноты |
| предварительное ревью текста (дефекты, неоднозначности) | конфиденциальные данные (см. §6) |
| переформулировать, суммировать, перевести | контекст, который вы не дали в промпт |

> Ключевая модель использования: **LLM — генератор черновиков и «второй пары глаз», не источник истины**. Он ускоряет рутину (структурирование, переформулирование, болванки артефактов), но финальную корректность, полноту и соответствие контексту обеспечивает аналитик.

---

## 2. Промпт-инжиниринг: роль · контекст · задача · формат

Качество ответа определяется качеством промпта. Базовая структура эффективного промпта:

| Элемент | Что задаёт | Пример |
|---------|-----------|--------|
| **Роль** | экспертная рамка | «Ты системный аналитик уровня middle+» |
| **Контекст** | предметная область, ограничения, входные данные | домен, фрагмент требований, схема БД |
| **Задача** | что именно сделать | «составь критерии приёмки в Gherkin» |
| **Формат** | структура ответа | таблица / Gherkin / JSON / PlantUML |
| **Примеры** (few-shot) | образец желаемого результата | 1–2 примера готового артефакта |
| **Ограничения** | чего избегать, на чём фокус | «не выдумывай поля, отметь допущения» |

```
[Роль] Ты системный аналитик middle+.
[Контекст] Домен — доставка еды. Требование: клиент оплачивает заказ картой.
[Задача] Составь критерии приёмки.
[Формат] Gherkin (Given/When/Then), таблица позитив/негатив.
[Ограничения] Отметь допущения; не придумывай несуществующие статусы.
```

> Чем точнее роль/контекст/формат — тем полезнее ответ. Слабый промпт («напиши требования») → размытый общий результат. **Дай контекст и попроси отметить допущения** — это снижает галлюцинации и делает ответ проверяемым. Few-shot (пример желаемого) резко повышает соответствие формату.

## 3. Генерация артефактов (черновики)

LLM полезен для болванок, которые аналитик доводит:

- **SQL** по описанию схемы и вопроса («выручка по ресторанам за месяц») → проверить на схеме и данных (модуль 18).
- **User stories + критерии приёмки** из требования → проверить по INVEST/Gherkin (модуль 08).
- **Диаграммы** (PlantUML/BPMN-текст) по словесному сценарию → проверить нотацию и логику (модули 12, 13, 33).
- **Тест-кейсы** по критериям приёмки → проверить покрытие, добавить негатив/границы (модуль 35).
- **Структурирование** сырых заметок интервью в каталог требований → сверить с источником (модуль 05).

> Правило: **ИИ даёт 70% болванки за минуты, аналитик доводит оставшееся и проверяет 100%**. Экономия — на рутине (структура, формулировки, синтаксис), а не на мышлении и ответственности. Сгенерированный SQL/диаграмму всегда прогонять/валидировать, а не принимать на веру.

---

## 4. Ревью и структурирование требований

LLM как «второе мнение» по тексту:
- найти **неоднозначности, противоречия, неполноту** в требовании/ТЗ (модуль 04);
- проверить требования на проверяемость/измеримость;
- предложить недостающие сценарии/негативные кейсы;
- суммировать длинный документ, выделить термины для глоссария.

> Это предварительный фильтр: ИИ быстро подсветит «слабые места» текста, но **итоговую оценку даёт аналитик** — модель может и пропустить дефект, и придумать несуществующий. Используйте как чек-лист-ускоритель, не как финального ревьюера.

---

## 5. Риск 1: галлюцинации

**Галлюцинация** — уверенно звучащий, но неверный/выдуманный ответ (несуществующее поле, неправильный синтаксис, выдуманная норма закона, ложная «лучшая практика»).

Почему опасно для СА: ошибочное требование/SQL/факт уходит в артефакт и дальше в разработку.

Как защищаться:
- **давать контекст** (модель меньше домысливает, когда есть данные);
- **просить отмечать допущения** и неуверенность;
- **верифицировать** всё фактическое (SQL — прогнать, факты — проверить в источнике, диаграмму — сверить с требованием);
- не задавать «наводящие» вопросы, провоцирующие выдумку;
- перепроверять цифры, имена полей, ссылки на стандарты/НПА.

> Базовое правило: **ничего фактического от ИИ не принимать без проверки**. Модель оптимизирована на правдоподобие, не на истинность. Особенно подозрительны: точные числа, названия (полей/API/законов), «всегда/никогда»-утверждения.

---

## 6. Риск 2: конфиденциальность

Отправка данных во внешний ИИ-сервис = передача их третьей стороне (может логироваться/использоваться). Нельзя отдавать в публичный ИИ:
- персональные данные (152-ФЗ/GDPR): ФИО, телефоны, адреса, мед-данные;
- коммерческую тайну, реальные ключи/секреты, внутренние документы под NDA;
- боевые данные/дампы БД.

Что делать:
- **обезличивание/анонимизация**: заменить реальные данные синтетическими/маскированными перед отправкой;
- работать на **абстрактных примерах** (схема без реальных строк);
- использовать **корпоративный/локальный** контур ИИ для чувствительного (политики компании);
- проверять политику провайдера (хранение/обучение на ваших данных).

> Правило: **не отправляй в публичный ИИ то, что не отправил бы постороннему**. Перед промптом — обезличить. Это часть приватности и безопасности (модуль 34): данные о клиентах и секреты не покидают контур.

---

## 7. Верификация — главное правило

Любой выход ИИ — **черновик до проверки**. Уровни верификации:

| Артефакт | Как проверить |
|----------|---------------|
| **SQL** | прогнать на схеме/тестовых данных, сверить результат |
| **Требования/US** | сверить с источником/стейкхолдером, проверить INVEST/измеримость |
| **Диаграмма** | проверить нотацию и соответствие сценарию/требованию |
| **Тест-кейсы** | проверить покрытие (трассировка), добавить негатив/границы |
| **Факты/цифры/НПА** | проверить в первоисточнике |

> Цикл работы с ИИ: **промпт → черновик → критическая проверка → правка промпта/доработка → финал под ответственностью аналитика**. ИИ ускоряет, но не снимает ответственности: за артефакт отвечает аналитик, а не модель.

---

## ⚠️ Подводные камни

- **Доверие без проверки.** Принять SQL/факт/требование «как есть» → ошибка в артефакте; верифицировать всегда.
- **Слабый промпт.** Без роли/контекста/формата — общий бесполезный ответ; структурировать промпт.
- **Утечка данных.** Отправка ПДн/секретов в публичный ИИ; обезличивать, использовать защищённый контур.
- **Галлюцинации в фактах.** Выдуманные поля/нормы/цифры звучат уверенно; проверять первоисточник.
- **ИИ принимает решения.** Делегировать выбор/ответственность модели нельзя; решает и отвечает аналитик.
- **Нет контекста — больше домыслов.** Чем меньше дано, тем больше выдумано; давать данные и просить отмечать допущения.
- **Few-shot проигнорирован.** Без примера формат «плывёт»; дать образец.
- **Замена мышления.** ИИ ускоряет рутину, но анализ/понимание домена — за аналитиком.
