# System Design — практический курс (middle+ → senior)

Пятый раздел репозитория `Java_Training` — рядом с [`../java-course`](../java-course/README.md), [`../patterns-architecture`](../patterns-architecture/README.md), [`../algorithms-course`](../algorithms-course/README.md) и [`../system-analysis-course`](../system-analysis-course/README.md).

Это **логическое продолжение системного анализа**. Если системный анализ отвечает на вопрос «**что** строить» (требования, моделирование, артефакты), то System Design отвечает на вопрос «**как** построить это на масштабе под нагрузкой и ограничениями» — и как **защитить дизайн на интервью**.

Как и системный анализ, курс **не про написание кода**. Вместо кода — артефакты в `.md`: схемы компонентов (C4), расчёты ёмкости, ADR, диаграммы последовательностей, контракты API, таблицы trade-off, дизайн-документы. Вместо компиляции — проверка по **критериям приёмки** и валидаторам.

Построен по той же переносимой методике — [`../LEARNING-METHODOLOGY.md`](../LEARNING-METHODOLOGY.md):

> **Маленькая тема → большая теория → задачи без готовых решений → проверяемый результат → следующая тема.**

- **49 атомарных модулей**, сквозная нумерация `01 → 49` (основы → количественная оценка → строительные блоки → данные → распределённые системы → надёжность → безопасность → 14 референсных архитектур → интервью → capstone → ИИ).
- На каждый модуль — **большая `theory.md`**, **10 задач** (`Task01 → Task10`, десятая — мини-проект) и **`questions.md` с 25 вопросами в стиле собеседования** (подготовка к интервью) с ключом эталонных ответов.
- **Готовых решений нет:** условие + каркас артефакта + критерии приёмки. Решение пишет студент.

> Статус генерации модулей и детальный план — в [`PROGRESS.md`](PROGRESS.md). Это источник истины: при потере контекста работа продолжается оттуда.

## 🎯 Для кого и зачем

Цель курса: после прохождения можно **спроектировать систему с нуля под заданную нагрузку и обосновать каждое решение** — и **пройти design-интервью** на грейд middle+/senior.

Курс полезен трём ролям:

| Роль | Что забирает |
|------|--------------|
| **Архитектор** | язык trade-off, количественное обоснование решений, референсные архитектуры |
| **Разработчик** | понимание «большой картины» за своим сервисом, рост в senior/staff, подготовка к design-интервью |
| **Системный аналитик** | мост от требований к технической реализуемости, диалог с архитектором на одном языке |

Собран на основе анализа трёх рыночных программ (enterprise solution architect; системный аналитик → дизайн; System-Design-интервью) с **добавлением слоя глубины**, которого нет в каждой по отдельности: количественная оценка (estimation), расчёт стоимости, специализированные блоки (consistent hashing, rate limiting, typeahead, гео), данные и аналитика на масштабе (OLAP/Lakehouse/Mesh), 14 разборов реальных систем и отдельная методика интервью с рубрикой по грейдам.

**Связь с системным анализом.** Темы, которые СА уже разбирает (CAP/NoSQL, репликация/шардинг, стили, распределённые паттерны, NFR, REST/async, безопасность), здесь даются под **другим углом** — «дизайн-решение под нагрузку + trade-off в интервью», а не пересказ азов. Где нужны азы — стоит ссылка на соответствующий модуль СА. Поэтому курс **самодостаточен**: проходить СА заранее полезно, но не обязательно.

## 🧩 Анатомия задачи

Шапка задачи в `TaskNN.md`:

```
# Задача NN — Модуль NN: <название>

## ЗАДАНИЕ
что нужно спроектировать/рассчитать/описать (1–3 предложения).

## ВХОДНЫЕ ДАННЫЕ
контекст: фрагмент требований, числа нагрузки, готовая схема, ограничения.

## ТРЕБОВАНИЯ
ограничения и обязательные условия (нотация, формат, объём, что учесть).

## ФОРМАТ РЕЗУЛЬТАТА
в каком виде сдать (расчёт / диаграмма C4 / таблица trade-off / дизайн-документ / ADR / API-контракт).

## КРИТЕРИИ ПРИЁМКИ
объективный чек-лист «сделано/нет» — это и есть проверка вместо компиляции.

## ПОДСКАЗКА
направление, но НЕ ответ.

---
<место для твоего решения>
```

**Правило:** готового решения нет. Есть условие, каркас и критерии, по которым студент сам проверяет себя.

### Две оси нарастающей сложности

```
Внутри темы (01 → 10):
  01–03  базовый приём (расчёт по образцу, чтение готового дизайна, один блок)
  04–07  комбинирование блоков, нестандартные случаи, поиск узких мест, сравнение вариантов
  08–09  спорные кейсы, trade-offs под жёсткими ограничениями, конфликты NFR
  10     мини-проект — синтез темы в дизайн фрагмента системы

Между модулями (01 → 49):
  Часть 1–2   основы дизайна, количественная оценка
  Часть 3–4   строительные блоки, данные и аналитика на масштабе
  Часть 5–7   распределённые системы, надёжность, безопасность и эксплуатация
  Часть 8     14 референсных архитектур (ядро подготовки к интервью)
  Часть 9–10  методика интервью, capstone, ИИ
```

## ✅ Объективная проверяемость без кода

Вместо «компилируется/нет» — критерии под тип артефакта:

| Тип задачи | Чем проверяется |
|------------|-----------------|
| Расчёт нагрузки/ёмкости/стоимости | порядок величин, единицы, прозрачные допущения, перепроверка обратным счётом |
| Диаграмма C4 / компонентов / sequence | чек-лист обязательных элементов + правила корректности нотации |
| Дизайн-документ / HLD | покрытие FR и NFR, обоснование выбора, явные trade-offs, узкие места и их устранение |
| Выбор технологии/паттерна | сравнительная таблица по заданным критериям + обоснование под контекст |
| ADR | контекст → варианты → решение → последствия; альтернативы не выкинуты молча |
| API-контракт / схема данных | валидация линтером/схемой (Swagger Editor, JSON Schema validator) |
| 25 вопросов модуля (интервью) | вопросы в стиле собеседования + ключ эталонных ответов в конце `questions.md` |

## 🧵 Сквозные примеры и референсные системы

В отличие от системного анализа (где тянулись FoodExpress/MedConnect), здесь применение блоков показывается **сразу на мировых системах** — так, как это звучит на реальном интервью.

| Уровень | Что это |
|---------|---------|
| **Модули-блоки (Части 3–7)** | каждый блок иллюстрируется мини-примером из реальной системы (кэш — на ленте соцсети, шардинг — на хранилище фото, очереди — на агрегации кликов) |
| **Часть 8 — референсные архитектуры (14)** | полный сквозной дизайн знаковых систем: shortener, KV-store, лента, мессенджер, уведомления, видео, файлы, такси, платежи, реклама, краулер, leaderboard, планировщик, метрики |
| **Capstone (модуль 48)** | новый домен «с нуля» — проверка самостоятельного прохождения полного цикла дизайна |

## 🚀 Как учиться

1. **Идите сверху вниз** — модули выстроены по нарастанию сложности.
2. **Сначала теория** (`theory.md`): разберите примеры, схемы, «подводные камни».
3. **Потом практика** (`Task01 → Task10`): проектируйте сами, сверяясь с критериями приёмки, а не с ответом.
4. **К каждому блоку держите числа** — дизайн без оценки нагрузки/ёмкости не дизайн, а картинка.
5. **Часть 8 — главный тренажёр**: каждую систему проектируйте целиком, вслух проговаривая trade-offs.
6. **Проверяйте себя 25 вопросами** модуля (формат собеседования) перед переходом дальше.
7. **Capstone (48)** — пройдите полный цикл самостоятельно на новом домене.
