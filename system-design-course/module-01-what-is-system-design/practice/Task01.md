# Задача 01 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Разграничьте три дисциплины — системный анализ, System Design и архитектуру ПО — в сравнительной таблице.

## ВХОДНЫЕ ДАННЫЕ
Проект: сервис доставки еды. Набор активностей: (а) написать use case «оформление заказа»; (б) решить, как хранить 50 млн заказов и выдерживать пик 5000 заказов/мин; (в) выбрать паттерн внутри сервиса заказов (например, гексагональная архитектура); (г) собрать NFR по latency и доступности; (д) решить, нужен ли кэш меню и какой; (е) описать структуру классов сервиса оплаты.

## ТРЕБОВАНИЯ
- Постройте таблицу: дисциплина × (главный вопрос / типовые артефакты / пример из списка а–е).
- Распределите все 6 активностей по дисциплинам.
- Объясните в 1–2 предложениях, где проходит граница между System Design и архитектурой ПО.

## ФОРМАТ РЕЗУЛЬТАТА
Сравнительная таблица + распределение активностей + краткий вывод о границах.

## КРИТЕРИИ ПРИЁМКИ
- [ ] В таблице 3 дисциплины с вопросами «что / как на масштабе / как устроен код».
- [ ] Активности распределены: (а)→СА, (б)(д)→System Design, (в)(е)→архитектура ПО, (г)→СА/Design (NFR — стык).
- [ ] Указано, что System Design оперирует блоками системы и NFR, а архитектура ПО — внутренним устройством кода сервиса.
- [ ] Вывод явно называет критерий границы (масштаб системы vs внутренняя структура компонента).

## ПОДСКАЗКА
Системный анализ — «что» (требования). System Design — «как на масштабе и почему» (блоки, данные, NFR, trade-offs). Архитектура ПО — «чем» внутри одного сервиса (слои, паттерны, классы). NFR живут на стыке СА (зафиксировать) и Design (удовлетворить).
