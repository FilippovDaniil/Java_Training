# Задача 02 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Классифицируйте набор проектных решений по уровням: HLD (высокоуровневый дизайн) или LLD (детальный дизайн).

## ВХОДНЫЕ ДАННЫЕ
Решения по системе ленты соцсети:
1. Лента собирается по модели fan-out-on-write.
2. Таблица `posts(id BIGINT, author_id BIGINT, body TEXT, created_at TIMESTAMP)`.
3. Между сервисом постов и сервисом ленты — очередь сообщений.
4. Индекс по `(author_id, created_at DESC)` для выборки постов автора.
5. Горячие данные ленты кэшируются в Redis.
6. Метод `getFeed(userId, cursor, limit)` возвращает страницу по cursor-пагинации.
7. Чтение масштабируется read-репликами БД.
8. Поле `created_at` хранится в UTC, формат ISO-8601.

## ТРЕБОВАНИЯ
- Для каждого из 8 решений укажите уровень (HLD / LLD) и обоснуйте одним предложением.
- Сгруппируйте результат в таблицу.
- Поясните, почему на design-интервью основное время уходит на HLD.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «решение → уровень → обоснование» + краткий вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] HLD: пункты 1, 3, 5, 7 (блоки системы, потоки, стратегии масштабирования).
- [ ] LLD: пункты 2, 4, 6, 8 (схема таблицы, индекс, сигнатура метода, формат поля).
- [ ] Каждое отнесение обосновано (блок системы vs деталь компонента).
- [ ] Вывод: HLD задаёт архитектуру и проверяет умение видеть систему целиком; LLD — детали, к которым переходят в deep dive.

## ПОДСКАЗКА
HLD — это то, что видно на схеме системы «с высоты»: какие сервисы/хранилища/очереди и как связаны. LLD — то, что внутри прямоугольника: таблицы, индексы, сигнатуры, форматы. Если решение можно нарисовать стрелкой между блоками — HLD; если оно про устройство одного блока — LLD.
