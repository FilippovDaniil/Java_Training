# Задача 03 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Прочитайте готовый фрагмент дизайна и выделите ключевые решения вместе с осями, на которых они лежат.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент HLD сервиса коротких ссылок:
> «Запись ссылки идёт в основную БД (PostgreSQL). Сам редирект — самый горячий путь (≈ 50 000 RPS на чтение против 100 RPS на запись), поэтому маппинг `short → long` кэшируется в Redis с TTL 24 ч; при промахе читаем из БД и прогреваем кэш. Короткий код генерируется как base62 от инкрементного ID, выданного отдельным сервисом ID-генерации, чтобы избежать коллизий. Аналитика кликов пишется асинхронно через очередь, чтобы не замедлять редирект. Допускается, что счётчик кликов отстаёт на несколько секунд.»

## ТРЕБОВАНИЯ
- Выделите минимум 5 ключевых решений.
- Для каждого назовите ось дизайна (нагрузка / латентность / согласованность / стоимость / данные).
- Для каждого назовите, какой компромисс принят (что выиграли, чем заплатили).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «решение → ось → выигрыш → плата».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выделено ≥ 5 решений (кэш редиректа, TTL, генерация ID через отдельный сервис, async-аналитика, разделение read/write путей).
- [ ] У каждого названа корректная ось.
- [ ] Async-аналитика отнесена к жертве строгой согласованности ради латентности редиректа.
- [ ] Замечено, что система явно read-heavy (50000 RPS чтение ≫ 100 RPS запись) и дизайн оптимизирует путь чтения.

## ПОДСКАЗКА
Ищите формулировки «поэтому», «чтобы», «допускается». Каждая из них — след принятого компромисса. «Допускается отставание на секунды» — это явный выбор eventual consistency на оси согласованности ради латентности.
