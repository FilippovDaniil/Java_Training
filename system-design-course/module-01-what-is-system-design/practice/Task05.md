# Задача 05 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Найдите в дизайне необоснованные решения (заявлены без trade-off) и переформулируйте их как обоснованный выбор с альтернативой.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайн-документа новостного сайта (MVP, ожидается ~1000 пользователей в первый месяц):
> «Архитектура: 12 микросервисов. Каждый сервис — со своей БД. Между сервисами — Kafka. Кэш — Redis Cluster из 6 нод. Поиск — Elasticsearch-кластер. Развёртывание — Kubernetes в трёх регионах с active-active. Данные дублируются между регионами для отказоустойчивости.»

## ТРЕБОВАНИЯ
- Укажите минимум 4 решения, заявленных без обоснования.
- Для каждого объясните, почему оно может быть преждевременным под данный контекст (1000 пользователей, MVP).
- Переформулируйте каждое в формате «решение, потому что …, альтернатива … отвергнута, потому что …».
- Предложите соразмерный контексту вариант дизайна системы.

## ФОРМАТ РЕЗУЛЬТАТА
Список «проблемное решение → почему преждевременно → обоснованная переформулировка» + соразмерный вариант MVP.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Найдено ≥ 4 необоснованных решения (12 микросервисов, Kafka, Redis Cluster, мультирегион active-active).
- [ ] Объяснено, что для 1000 пользователей это преждевременная сложность (операционная цена ≫ выгоды).
- [ ] Переформулировки содержат явную альтернативу и причину отказа.
- [ ] Соразмерный вариант: монолит/модульный монолит + одна БД + при необходимости один Redis; мультирегион и дробление — когда появится нагрузка.

## ПОДСКАЗКА
Сложность — это цена, а не достоинство. Спросите по каждому пункту: «какую измеримую проблему под текущей нагрузкой это решает?». Если ответа нет — решение преждевременно. Хороший дизайн соразмерен контексту и эволюционирует с ростом нагрузки.
