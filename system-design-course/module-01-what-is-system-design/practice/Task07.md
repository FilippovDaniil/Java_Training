# Задача 07 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Разложите процесс дизайна на шаги применительно к конкретной задаче: для каждого шага укажите, что именно вы сделаете.

## ВХОДНЫЕ ДАННЫЕ
Задача: спроектировать сервис коротких ссылок (URL-shortener). Контекст пока не уточнён — это часть упражнения.

## ТРЕБОВАНИЯ
- Перечислите 8 шагов процесса дизайна (из §5 теории).
- Для каждого шага напишите 1–2 конкретных действия именно для URL-shortener (не абстрактно).
- На шаге 1 сформулируйте 3 уточняющих вопроса по требованиям.
- На шаге 2 укажите, какие 2–3 числа надо прикинуть.
- Подчеркните, почему нельзя начать с шага 5 (high-level), минуя 1–2.

## ФОРМАТ РЕЗУЛЬТАТА
Нумерованный список 8 шагов, под каждым — конкретные действия для URL-shortener + вывод о порядке.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Все 8 шагов на месте и в правильном порядке (требования → estimation → API → data → high-level → deep dive → bottlenecks → wrap-up).
- [ ] Действия конкретны для URL-shortener (например, API: `POST /urls`, `GET /{code}`; data: маппинг code→url; deep dive: генерация уникального кода).
- [ ] Шаг 1 содержит 3 уточняющих вопроса (объём ссылок, кастомные алиасы, срок жизни, аналитика).
- [ ] Шаг 2 называет конкретные числа (RPS чтения/записи, объём хранения).
- [ ] Объяснено: без требований и чисел нельзя обоснованно выбрать схему хранения, нужду в кэше и стратегию генерации кода.

## ПОДСКАЗКА
Процесс одинаков для любой системы — меняется наполнение шагов. URL-shortener почти всегда read-heavy, и это выясняется на шаге 2 (estimation), что и определяет акцент на кэше/чтении в шаге 5. Поэтому порядок «числа → блоки» обязателен.
