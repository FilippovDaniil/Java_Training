# Задача 08 — Модуль 01: Что такое System Design

## ЗАДАНИЕ
Отделите функциональные требования от нефункциональных и покажите, как именно NFR двигает дизайн (меняет архитектурные решения).

## ВХОДНЫЕ ДАННЫЕ
Требования к сервису онлайн-голосования на музыкальной премии:
1. Пользователь голосует за артиста.
2. Один пользователь — один голос за номинацию.
3. Результаты видны в реальном времени.
4. Во время финала шоу — пик до 1 млн голосов в минуту за 5 минут.
5. Итоговый подсчёт должен быть точным (нельзя потерять или задвоить голос).
6. Страница результатов открывается < 1 сек по всему миру.
7. Система доступна во время эфира (99.99% в окно шоу).

## ТРЕБОВАНИЯ
- Разделите 7 пунктов на FR и NFR.
- Для каждого NFR укажите ось дизайна.
- Для 3 NFR покажите конкретно, как они меняют дизайн по сравнению с «наивной» реализацией (одна БД, синхронная запись).
- Сформулируйте один конфликт между NFR.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица FR/NFR + «NFR → влияние на дизайн» (3 шт.) + один named-конфликт.

## КРИТЕРИИ ПРИЁМКИ
- [ ] FR: 1, 2 (что делает); NFR: 3, 4, 5, 6, 7 (качества). Пункт 3 — на стыке (FR-формулировка с NFR-сутью «реального времени»).
- [ ] Пик 1 млн/мин (нагрузка) → буферизация через очередь/батчинг, шардирование счётчиков, защита от дублей по idempotency/уникальности голоса.
- [ ] «<1 сек по миру» (латентность/география) → CDN/кэш результатов, реплики ближе к пользователю, eventual consistency для отображения.
- [ ] «Точный подсчёт» (надёжность) → конфликтует с «<1 сек в реальном времени»: точность требует строгого учёта, а скорость толкает к кэшу/eventual.
- [ ] Назван минимум один конфликт NFR (точность/согласованность ↔ латентность отображения).

## ПОДСКАЗКА
FR — «голосует», «один голос». NFR — «реальное время», «пик», «точно», «<1 сек», «доступна». Наивная синхронная запись в одну БД ляжет на пике в 1 млн/мин — именно NFR нагрузки заставляет ввести очередь и шардинг. Это и есть «NFR двигает дизайн».
