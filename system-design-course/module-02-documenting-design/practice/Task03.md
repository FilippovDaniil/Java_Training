# Задача 03 — Модуль 02: Документирование дизайна

## ЗАДАНИЕ
Напишите ADR на спорное решение: выбор типа хранилища (SQL vs NoSQL) для конкретного сценария.

## ВХОДНЫЕ ДАННЫЕ
Контекст: проектируется хранилище профилей пользователей социального приложения. ~100 млн пользователей, профиль читается на каждый показ страницы (read-heavy), структура профиля относительно стабильна, но изредка добавляются поля; нужны выборки по точному ключу (userId), сложные транзакции между пользователями не требуются; важна горизонтальная масштабируемость и низкая latency чтения.

## ТРЕБОВАНИЯ
- Оформите ADR по шаблону: Статус / Контекст / Варианты / Решение / Последствия.
- Рассмотрите минимум 2 варианта (реляционная БД и документная/KV-NoSQL).
- В «Последствиях» обязательно укажите и плюсы, и минусы выбранного варианта.
- Решение должно следовать из контекста (доступ по ключу, масштаб, read-heavy).

## ФОРМАТ РЕЗУЛЬТАТА
ADR-документ по шаблону §3 теории.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Присутствуют все поля ADR (Статус, Контекст, Варианты, Решение, Последствия).
- [ ] Рассмотрено ≥ 2 варианта с их сильными/слабыми сторонами.
- [ ] Решение обосновано контекстом (доступ по userId, горизонтальный масштаб, read-heavy).
- [ ] Последствия содержат и плюсы, и минусы (например, минус NoSQL — слабее ad-hoc-запросы/джойны).
- [ ] Статус указан (Принято/Предложено).

## ПОДСКАЗКА
Здесь нет единственно верного ответа — есть обоснованный. Доступ только по ключу + масштаб + read-heavy склоняют к KV/документной модели, но честный ADR назовёт, чем за это платим (гибкость запросов, отсутствие джойнов/транзакций). Важна не «угаданная» БД, а структура рассуждения.
