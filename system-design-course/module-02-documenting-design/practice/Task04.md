# Задача 04 — Модуль 02: Документирование дизайна

## ЗАДАНИЕ
Проведите ревью чужого ADR: найдите пропущенные альтернативы и неполные последствия, предложите исправления.

## ВХОДНЫЕ ДАННЫЕ
ADR на ревью:
> **ADR-0012: Используем Kafka для всех межсервисных взаимодействий**
> **Статус:** Принято
> **Контекст:** Нам нужно, чтобы сервисы обменивались данными.
> **Решение:** Все сервисы общаются через Kafka.
> **Последствия:** Сервисы развязаны, можно масштабировать.

## ТРЕБОВАНИЯ
- Перечислите дефекты этого ADR по структуре (что отсутствует/неполно).
- Назовите минимум 2 непрорассмотренные альтернативы.
- Допишите отсутствующие минусы/последствия выбора Kafka «для всех взаимодействий».
- Уточните контекст: какие вопросы надо было задать (sync vs async, объём, требования к согласованности).

## ФОРМАТ РЕЗУЛЬТАТА
Список дефектов + недостающие альтернативы + дополненные последствия (с минусами) + уточняющие вопросы контекста.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Отмечено: нет раздела «Варианты»; контекст размыт (нет чисел, типа взаимодействий); последствия однобоки (только плюсы).
- [ ] Названы альтернативы: синхронный REST/gRPC, другой брокер/очередь, прямые вызовы для sync-сценариев.
- [ ] Добавлены минусы: Kafka неуместна для синхронных запрос-ответ; операционная сложность; eventual consistency; не для всех взаимодействий.
- [ ] Уточняющие вопросы про sync/async, объём событий, требования к порядку/согласованности.
- [ ] Вывод: «для всех взаимодействий» — чрезмерное обобщение; решение должно зависеть от типа взаимодействия.

## ПОДСКАЗКА
Хороший ADR узнаётся по наличию отвергнутых альтернатив и честных минусов. Здесь нет ни того, ни другого, а контекст не содержит сил, в которых принималось решение. «Один инструмент на всё» — частый запах: разные взаимодействия (sync команда vs async событие) требуют разных решений.
