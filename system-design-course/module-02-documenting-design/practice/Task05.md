# Задача 05 — Модуль 02: Документирование дизайна

## ЗАДАНИЕ
Подберите уровень C4 (или их сочетание) под три разные аудитории и обоснуйте выбор.

## ВХОДНЫЕ ДАННЫЕ
Три ситуации презентации одной и той же системы (платформа онлайн-обучения):
1. Питч топ-менеджменту: «как система встроена в бизнес и с кем взаимодействует».
2. Онбординг нового backend-разработчика в команду: «что из чего состоит, какие сервисы и хранилища».
3. Разбор внутри команды: «как устроен конкретный сервис рекомендаций изнутри».

## ТРЕБОВАНИЯ
- Для каждой ситуации укажите подходящий уровень C4 (Context / Container / Component) и почему.
- Объясните, что будет лишним/непонятным, если показать не тот уровень.
- Сформулируйте правило выбора уровня под аудиторию.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «ситуация → уровень C4 → обоснование → антипример (не тот уровень)» + правило.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ситуация 1 → Context (бизнес-аудитория, граница и окружение).
- [ ] Ситуация 2 → Container (состав системы, сервисы/хранилища/протоколы).
- [ ] Ситуация 3 → Component (внутреннее устройство одного контейнера).
- [ ] Для каждой указано, чем плох неподходящий уровень (Code топ-менеджменту — шум; Context разработчику — мало деталей).
- [ ] Сформулировано правило: уровень детализации = потребность аудитории в детализации.

## ПОДСКАЗКА
C4 — это зум. Бизнесу нужен общий план (Context), команде — состав (Container), внутри сервиса — компоненты (Component). Показать классы топ-менеджеру — утопить в деталях; показать Context разработчику — оставить без нужной информации.
