# Задача 08 — Модуль 02: Документирование дизайна

## ЗАДАНИЕ
Оформите архитектурную диаграмму в подходе Doc-as-Code: запишите её текстовой нотацией (PlantUML или Mermaid), пригодной для хранения в репозитории и ревью.

## ВХОДНЫЕ ДАННЫЕ
Поток оформления заказа: клиент → API → сервис заказов → (синхронно) сервис оплаты; сервис заказов публикует событие `OrderCreated` в очередь; воркер уведомлений читает событие и шлёт письмо через внешний email-провайдер.

## ТРЕБОВАНИЯ
- Запишите **sequence-диаграмму** этого потока текстовой нотацией (PlantUML или Mermaid).
- Явно покажите синхронный вызов оплаты и асинхронную публикацию события.
- Объясните в 2–3 предложениях преимущества Doc-as-Code над «картинкой из графического редактора».
- Укажите, в каком месте репозитория и как это ревьюится.

## ФОРМАТ РЕЗУЛЬТАТА
Блок кода с sequence-нотацией + краткое обоснование Doc-as-Code + размещение в репо.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диаграмма записана валидной текстовой нотацией (PlantUML `@startuml/@enduml` или Mermaid `sequenceDiagram`).
- [ ] Показаны все участники: Client, API, OrderSvc, PaymentSvc, Queue, NotificationWorker, EmailProvider.
- [ ] Синхронный вызов оплаты и асинхронная публикация/чтение события визуально различимы (например, сплошная vs пунктир / явная очередь).
- [ ] Названы преимущества Doc-as-Code: версионирование с кодом, ревью в PR, нет расхождения с реальностью.
- [ ] Указано размещение (например, `docs/diagrams/order-flow.puml`) и ревью через pull request.

## ПОДСКАЗКА
Текстовая нотация диаграмм — ключ к Doc-as-Code: её можно дифать, ревьюить и менять в одном PR с кодом. Для последовательности шагов во времени sequence нагляднее структурных C4-диаграмм. В Mermaid используйте `->>` (sync) и `-->>` (ответ), очередь покажите отдельным участником.
