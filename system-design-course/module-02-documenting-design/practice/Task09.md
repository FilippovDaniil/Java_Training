# Задача 09 — Модуль 02: Документирование дизайна

## ЗАДАНИЕ
Свяжите набор архитектурных решений в журнал ADR: пронумеруйте, проставьте статусы и покажите отношения между ними (в т.ч. замену устаревшего решения).

## ВХОДНЫЕ ДАННЫЕ
Хронология решений по одному сервису:
1. (раннее) Сессии храним в памяти приложения.
2. (позже) Поняли, что нужно переживать рестарт и шарить между инстансами — переходим на Redis.
3. Выбрали REST для публичного API.
4. Для внутренней связи двух сервисов выбрали gRPC.
5. (предложение, ещё не принято) Рассмотреть вынос аутентификации в отдельный сервис.

## ТРЕБОВАНИЯ
- Оформите список как журнал ADR: номер, заголовок, статус.
- Корректно отразите, что решение про Redis **заменяет** решение про in-memory сессии (статусы «Заменено» / «Заменяет ADR-XXXX»).
- Для предложения (п.5) проставьте статус «Предложено».
- Объясните, почему старый ADR не удаляют, а помечают статусом.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица/список журнала ADR (№, заголовок, статус, связь) + пояснение про неудаление.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каждое решение — отдельный пронумерованный ADR с заголовком.
- [ ] ADR про in-memory сессии — статус «Заменено ADR-NN»; ADR про Redis — «Принято, заменяет ADR-MM».
- [ ] REST и gRPC — два отдельных принятых ADR (разные контексты: публичный vs внутренний).
- [ ] Предложение про auth-сервис — статус «Предложено».
- [ ] Пояснено: старый ADR сохраняют для истории решений (почему система пришла к текущему состоянию); удаление стёрло бы контекст.

## ПОДСКАЗКА
Журнал ADR — это история «почему система такая». Решения append-only: новое не затирает старое, а ссылается на него статусом. Это позволяет через год понять, что in-memory отвергли осознанно, а не забыли. Обратите внимание: REST и gRPC не конфликтуют — это решения для разных границ.
