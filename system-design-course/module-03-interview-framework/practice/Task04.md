# Задача 04 — Модуль 03: Фреймворк design-интервью

## ЗАДАНИЕ
Для заданной системы перечислите FR и NFR, которые нужно подтвердить **до** начала проектирования, и покажите, как ответы развилкой меняют дизайн.

## ВХОДНЫЕ ДАННЫЕ
Задача интервью: спроектировать систему уведомлений (push / email / SMS) для крупного продукта.

## ТРЕБОВАНИЯ
- Перечислите ключевые FR, которые надо уточнить (какие каналы, кто инициирует, шаблоны, дедупликация, приоритеты).
- Перечислите ключевые NFR (объём уведомлений/с, latency доставки, гарантии доставки, порядок, доступность).
- Для 3 уточнений постройте «развилку»: если ответ A — дизайн такой, если B — иной.
- Отметьте, какие уточнения сильнее всего влияют на архитектуру.

## ФОРМАТ РЕЗУЛЬТАТА
Списки FR/NFR для подтверждения + 3 развилки «ответ → следствие для дизайна» + вывод о самых влиятельных уточнениях.

## КРИТЕРИИ ПРИЁМКИ
- [ ] FR: набор каналов, источники событий, шаблоны/персонализация, дедуп, пользовательские настройки/отписки.
- [ ] NFR: throughput (уведомлений/с, пики), latency, гарантии доставки (at-least-once?), приоритеты, доступность.
- [ ] Минимум 3 развилки, где разный ответ ведёт к разной архитектуре (например: «нужны ли гарантии доставки?» → очередь с ретраями и DLQ vs fire-and-forget).
- [ ] Указано, что гарантии доставки и объём/пики — самые архитектурно-влиятельные уточнения.
- [ ] Не начато проектирование до уточнений.

## ПОДСКАЗКА
Уточнения ценны, когда разный ответ ведёт к разному дизайну. «At-least-once с дедупом» против «можно потерять» — это два разных архитектурных мира (очереди, ретраи, идемпотентность vs простой вызов). Ищите именно такие развилочные вопросы, а не косметические.
