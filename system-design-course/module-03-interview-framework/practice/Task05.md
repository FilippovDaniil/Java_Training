# Задача 05 — Модуль 03: Фреймворк design-интервью

## ЗАДАНИЕ
Найдите ошибки в транскрипте поведения кандидата на design-интервью и предложите, как нужно было действовать.

## ВХОДНЫЕ ДАННЫЕ
Транскрипт (кандидат K, интервьюер I):
> **I:** Спроектируйте сервис коротких ссылок.
> **K:** Так, заводим таблицу в PostgreSQL: `id, short_code, long_url`. Ставим Spring Boot, кладём nginx впереди. Ещё добавим Redis — он всегда полезен. И Kafka на всякий случай. Так, что ещё... наверное, шардируем базу. Какой следующий вопрос?
> **I:** А сколько ссылок в день ожидаем?
> **K:** Ну, много. Не считал. Давайте просто сделаем масштабируемо.
> *(кандидат молча рисует схему 3 минуты, потом)* **K:** Вот, готово.

## ТРЕБОВАНИЯ
- Перечислите все ошибки поведения кандидата (по типам из теории).
- Для каждой ошибки укажите, как нужно было поступить.
- Перепишите первые 2 минуты ответа кандидата «как надо».

## ФОРМАТ РЕЗУЛЬТАТА
Список ошибок + правильное действие к каждой + переписанное начало ответа.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Отмечено: прыжок в решение без требований; нет уточняющих вопросов; нет чисел («много», «не считал»); технологии «на всякий случай» (Redis/Kafka без обоснования); молчаливое рисование; нет trade-offs.
- [ ] Для каждой ошибки дано корректное действие (сначала требования → estimation → обоснованные блоки, думать вслух).
- [ ] Переписанное начало стартует с уточняющих вопросов и проговаривания плана, а не с таблицы.
- [ ] Подчёркнуто, что «сделаем масштабируемо» без чисел — анти-сигнал (масштабируемость ради масштабируемости).

## ПОДСКАЗКА
Этот транскрипт — концентрат типичных ошибок. Ключевой разворот: не «какие технологии я знаю», а «какие требования и числа, и какое решение из них следует». Redis и Kafka «на всякий случай» — классический red flag: инструменты без задачи.
