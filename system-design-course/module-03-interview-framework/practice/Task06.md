# Задача 06 — Модуль 03: Фреймворк design-интервью

## ЗАДАНИЕ
Выберите 1–2 компонента для deep dive в заданной системе и обоснуйте выбор; наметьте, что именно разбирать вглубь.

## ВХОДНЫЕ ДАННЫЕ
Уже нарисован HLD ленты соцсети: клиенты → балансировщик → API → сервис постов (БД постов), сервис ленты (кэш ленты), сервис подписок (БД графа), очередь для fan-out, воркеры fan-out. Интервьюер: «Давайте углубимся в один компонент».

## ТРЕБОВАНИЯ
- Назовите 1–2 компонента, в которые стоит углубиться, и почему именно они (риск/интерес/нагрузка).
- Для выбранного компонента перечислите, какие вопросы deep dive вы раскроете (алгоритм, данные, краевые случаи, масштабирование).
- Покажите, почему углубляться в тривиальный компонент — слабый ход.
- Назовите хотя бы один trade-off внутри выбранного компонента.

## ФОРМАТ РЕЗУЛЬТАТА
Выбор компонента(ов) + обоснование + план deep dive (что разбираем) + named trade-off.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбран архитектурно-богатый компонент (например, стратегия fan-out ленты: on-write vs on-read; или генерация/хранение ленты для «звёзд» с миллионами подписчиков).
- [ ] Обоснование опирается на риск/нагрузку/интерес, а не на простоту.
- [ ] План deep dive перечисляет конкретику: алгоритм fan-out, обработка hot users, хранение/инвалидация кэша ленты, пагинация.
- [ ] Назван trade-off (fan-out-on-write: быстрое чтение, дорогая запись для популярных авторов ↔ fan-out-on-read наоборот; гибрид).
- [ ] Отмечено, что углубление в тривиальный CRUD не демонстрирует глубину.

## ПОДСКАЗКА
Deep dive — место показать глубину. Выбирайте компонент, где есть нетривиальный выбор и краевые случаи (fan-out ленты — классика: «звёзды» ломают наивный on-write). Тривиальный компонент не даст материала для trade-off-разговора.
