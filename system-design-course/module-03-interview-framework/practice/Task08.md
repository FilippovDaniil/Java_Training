# Задача 08 — Модуль 03: Фреймворк design-интервью

## ЗАДАНИЕ
Постройте рубрику самооценки по грейдам и оцените по ней заданный ответ кандидата.

## ВХОДНЫЕ ДАННЫЕ
Краткое описание ответа кандидата на «спроектируйте rate limiter»:
> Уточнил scope (на пользователя или на IP, какой лимит). Прикинул нагрузку (50k RPS). Предложил алгоритм token bucket, объяснил почему не fixed window (всплески на границе окна). Нарисовал HLD с Redis для счётчиков, обсудил распределённость (рассинхрон счётчиков между нодами) и предложил компромисс. Сам отметил слабое место: точность лимита при гонках, предложил Lua-скрипт для атомарности. Вёл диалог, думал вслух.

## ТРЕБОВАНИЯ
- Постройте рубрику: для каждого критерия оценки (управление неопределённостью, числа, широта, глубина, trade-offs, коммуникация) — что соответствует junior/middle/senior.
- Оцените описанный ответ по каждому критерию с указанием грейда.
- Выставите итоговый грейд и обоснуйте.
- Назовите, чего не хватило бы до следующего грейда.

## ФОРМАТ РЕЗУЛЬТАТА
Рубрика (критерий × грейд) + оценка ответа по критериям + итоговый грейд + зона роста.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Рубрика покрывает ≥ 5 критериев с описанием уровней junior/middle/senior.
- [ ] Ответ оценён по каждому критерию (большинство — на senior: уточнения, числа, trade-offs, самокритика, диалог).
- [ ] Итоговый грейд — senior — обоснован конкретными сигналами из ответа.
- [ ] Названа зона роста до staff+ (например: несколько архитектур с обоснованием, мультирегион/эволюция, влияние на смежные системы).
- [ ] Оценка опирается на наблюдаемое поведение, а не на «знание технологий».

## ПОДСКАЗКА
Грейд определяется глубиной и самостоятельностью, а не списком терминов. Здесь кандидат сам ведёт процесс, приводит числа, называет trade-off и сам находит слабое место — это senior-паттерн. До staff+ обычно не хватает работы с несколькими архитектурами и системного/организационного контекста.
