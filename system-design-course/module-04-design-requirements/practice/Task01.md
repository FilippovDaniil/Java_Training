# Задача 01 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Извлеките из размытой постановки три категории: функциональные требования, нефункциональные требования и ограничения.

## ВХОДНЫЕ ДАННЫЕ
Запрос заказчика:
> «Нужен сервис, где пользователи загружают видео и смотрят их. Должно быть быстро и не падать, у нас уже миллионы пользователей. Бюджет ограничен, команда маленькая. Мы в России, надо соблюдать закон о персональных данных. Запуск через 3 месяца.»

## ТРЕБОВАНИЯ
- Выделите ключевые FR (2–4 архитектурно-определяющих).
- Выделите NFR (даже если в тексте они размыты — отметьте их как «требуют уточнения и измерения»).
- Выделите ограничения (constraints) отдельным списком.
- Для каждого размытого места сформулируйте уточняющий вопрос.

## ФОРМАТ РЕЗУЛЬТАТА
Три списка (FR / NFR / ограничения) + список уточняющих вопросов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] FR: загрузка видео, просмотр/стриминг видео (ядро).
- [ ] NFR: латентность («быстро»), доступность («не падать»), масштаб («миллионы пользователей») — помечены как требующие измеримых значений.
- [ ] Ограничения: бюджет, размер команды, 152-ФЗ (хранение/обработка ПДн в РФ), срок 3 месяца.
- [ ] Уточняющие вопросы бьют в числа и измеримость (сколько загрузок/день, целевой p99, целевой SLA, объём видео).
- [ ] Constraints не перемешаны с FR/NFR.

## ПОДСКАЗКА
Слова-маркеры: «загружают/смотрят» → FR; «быстро/не падать/миллионы» → NFR (пока неизмеримые); «бюджет/команда/закон/срок» → ограничения. Ограничение отличается тем, что его нельзя выбрать — его принимают как данность.
