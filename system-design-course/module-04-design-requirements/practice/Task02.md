# Задача 02 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Превратите размытые нефункциональные пожелания в измеримые NFR, пригодные для дизайна.

## ВХОДНЫЕ ДАННЫЕ
Размытые формулировки про сервис интернет-магазина:
1. «Сайт должен открываться быстро».
2. «Система должна быть надёжной».
3. «Должны выдерживать большие нагрузки, особенно в распродажи».
4. «Данные не должны теряться».
5. «Поиск товаров должен быть быстрым».

## ТРЕБОВАНИЯ
- Для каждого пункта дайте измеримую формулировку (число + метрика + условие).
- Латентность выражайте в перцентилях (p95/p99), а не в среднем — объясните почему.
- Для нагрузки укажите average и peak.
- Для «надёжности» разделите доступность (SLA, %) и durability (потеря данных).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «размытое → измеримое NFR» + пояснение про перцентили.

## КРИТЕРИИ ПРИЁМКИ
- [ ] «Быстро» → конкретный p95/p99 (например, «p99 загрузки страницы < 300 мс»).
- [ ] «Надёжно» разложено на доступность (например, 99.95% SLA) и durability (например, «нет потери подтверждённых заказов»).
- [ ] «Большие нагрузки» → average RPS + peak RPS с коэффициентом для распродаж.
- [ ] «Не теряться» → durability-гарантия (репликация/бэкап, RPO).
- [ ] Объяснено, почему перцентили лучше среднего (среднее прячет хвосты, ощущаемые пользователем).

## ПОДСКАЗКА
Измеримый NFR = метрика + число + условие. «Надёжность» — частая ловушка: это два разных свойства (доступность и сохранность данных), их нельзя сливать. Перцентили нужны потому, что p99 = 2 c при среднем 100 мс означает, что каждый сотый пользователь страдает.
