# Задача 05 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Разделите перемешанный список на требования (что нужно обеспечить) и ограничения (что нельзя выбрать / принято как данность).

## ВХОДНЫЕ ДАННЫЕ
Список пунктов про проект корпоративного мессенджера:
1. Сообщения доставляются за < 1 c (p99).
2. Используем только наш частный дата-центр (нет публичного облака).
3. Поддержка групповых чатов до 5000 участников.
4. Доступность 99.9%.
5. Команда разработки — 5 человек.
6. Все данные хранятся внутри страны (регуляторное требование).
7. История сообщений хранится бессрочно.
8. Релиз первой версии — через 4 месяца.
9. Совместимость с уже используемым корпоративным SSO.

## ТРЕБОВАНИЯ
- Разнесите 9 пунктов на «требования» и «ограничения».
- Для каждого ограничения укажите, какое пространство дизайн-решений оно сужает.
- Объясните принципиальную разницу между требованием и ограничением.

## ФОРМАТ РЕЗУЛЬТАТА
Две колонки (требования / ограничения) + влияние каждого ограничения на дизайн + определение различия.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Требования: 1, 3, 4, 7 (что система обеспечивает: latency, ёмкость групп, доступность, хранение).
- [ ] Ограничения: 2, 5, 6, 8, 9 (частный ДЦ, размер команды, данные в стране, срок, SSO).
- [ ] Для ограничений показано сужение пространства: частный ДЦ → нет managed-сервисов облака; данные в стране → размещение; SSO → готовая интеграция аутентификации.
- [ ] Сформулировано различие: требование — что обеспечить (можно достичь по-разному); ограничение — внешняя рамка, которую принимают.
- [ ] Пункт 9 (SSO) корректно отнесён к ограничению (готовая система, к которой подключаемся).

## ПОДСКАЗКА
Тест: «могу ли я это выбрать/спроектировать иначе?» Если да — требование (есть свобода реализации). Если нет, это навязано извне — ограничение. Ограничения ценны тем, что отсекают целые классы решений ещё до старта, экономя время.
