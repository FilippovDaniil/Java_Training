# Задача 06 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Зафиксируйте явные допущения для системы при недостатке данных и покажите, как изменится дизайн, если допущение окажется неверным.

## ВХОДНЫЕ ДАННЫЕ
Задача: спроектировать систему коротких ссылок. Никаких чисел не дано. Нужно начать дизайн «здесь и сейчас», делая обоснованные допущения.

## ТРЕБОВАНИЯ
- Сформулируйте 5–7 явных допущений (по нагрузке, read/write, объёму, сроку хранения, длине кода, кастомным алиасам).
- Каждое допущение обоснуйте (откуда взято разумное значение).
- Для 3 ключевых допущений покажите чувствительность: «если на самом деле X, дизайн меняется так-то».
- Отметьте, какое допущение наиболее критично для архитектуры.

## ФОРМАТ РЕЗУЛЬТАТА
Список допущений с обоснованием + 3 анализа чувствительности + самое критичное допущение.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Допущения явные и обоснованные (например: 100M новых ссылок/мес, read/write ≈ 100:1, хранение 5 лет, код 7 символов base62).
- [ ] Read/write-ratio помечен как ключевой (определяет read-heavy → акцент на кэше).
- [ ] Анализ чувствительности: если запись на порядок выше — меняется стратегия БД/генерации ID; если хранение бессрочно — растут требования к storage.
- [ ] Указано самое критичное допущение и почему (обычно read/write-ratio и объём).
- [ ] Допущения проговорены как явные, а не скрытые.

## ПОДСКАЗКА
Допущения — мост через неопределённость. Хорошее допущение разумно (порядок величины правдоподобен) и явно (его можно оспорить). Анализ чувствительности показывает зрелость: вы понимаете, какие числа критичны, а какие можно прикинуть грубо. Read/write-ratio почти всегда в числе критичных.
