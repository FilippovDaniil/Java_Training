# Задача 07 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Покажите на примерах, как три конкретных NFR двигают дизайн: для каждого NFR — какое архитектурное решение из него следует и какой была бы «наивная» альтернатива без этого NFR.

## ВХОДНЫЕ ДАННЫЕ
Система: лента активности фитнес-приложения (тренировки друзей). NFR:
1. Лента открывается за p99 < 150 мс при 30 000 RPS на чтение.
2. Доступность 99.99% (приложение — основной экран).
3. Допустимо отставание ленты на несколько секунд (свежесть не критична).

## ТРЕБОВАНИЯ
- Для каждого NFR назовите следующее из него дизайн-решение.
- Для каждого покажите «наивный» вариант, который был бы при отсутствии этого NFR.
- Свяжите NFR №3 (допустимая задержка) с выбором согласованности.
- Сформулируйте общий принцип «NFR → решение».

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «NFR → дизайн-решение → наивная альтернатива без NFR» + вывод-принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] NFR №1 (p99 + 30k RPS) → предрасчёт/кэш ленты, read-реплики; наивно — сборка на лету из БД на каждый запрос.
- [ ] NFR №2 (99.99%) → избыточность, отсутствие SPOF, мультизона; наивно — один инстанс/одна БД.
- [ ] NFR №3 (задержка ок) → eventual consistency, async fan-out через очередь; наивно — синхронное обновление всех лент при каждом событии.
- [ ] Явно связано: допустимая задержка разрешает eventual consistency, что удешевляет и ускоряет дизайн.
- [ ] Вывод: каждое архитектурное решение отслеживается к конкретному NFR.

## ПОДСКАЗКА
Это центральная идея модуля. Двигайтесь от NFR к решению, а не наоборот. NFR №3 — подарок дизайнеру: «можно отставать» снимает требование строгой согласованности и открывает дешёвые async-решения. Если бы свежесть была критична — дизайн был бы дороже и сложнее.
