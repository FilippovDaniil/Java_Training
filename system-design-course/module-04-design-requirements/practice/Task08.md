# Задача 08 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Выявите конфликты между NFR в наборе требований и предложите способ их разрешения (приоритизацию или компромисс).

## ВХОДНЫЕ ДАННЫЕ
NFR банковского приложения (перевод средств):
1. Перевод подтверждается мгновенно (p99 < 200 мс).
2. Строгая согласованность баланса (никаких «фантомных» денег).
3. Доступность 99.99%, в т.ч. при отказе одного дата-центра.
4. Стоимость инфраструктуры под контролем.
5. Глобальная работа (клиенты на разных континентах).

## ТРЕБОВАНИЯ
- Найдите минимум 2 пары конфликтующих NFR.
- Для каждого конфликта объясните механизм противоречия.
- Предложите разрешение: приоритет (что важнее в этом домене) или компромисс (как удовлетворить обе стороны частично).
- Свяжите один из конфликтов с CAP-теоремой.

## ФОРМАТ РЕЗУЛЬТАТА
Список конфликтов (пара NFR + механизм) + разрешение каждого + связь с CAP.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Конфликт 1: строгая согласованность (№2) ↔ доступность при отказе ДЦ (№3) / глобальность (№5) — это CAP.
- [ ] Конфликт 2: строгая согласованность + глобальность ↔ низкая latency (№1) — синхронная репликация между континентами добавляет задержку (это PACELC: latency vs consistency).
- [ ] Конфликт 3 (доп.): доступность/глобальность ↔ стоимость (№4) — избыточность и мультирегион дороги.
- [ ] Разрешение опирается на домен: в банке согласованность баланса приоритетнее, чем минимальная latency; выбирают CP с разумной деградацией.
- [ ] Явно назван CAP/PACELC как рамка конфликта согласованность↔доступность/латентность.

## ПОДСКАЗКА
NFR почти всегда конфликтуют — «всё сразу» невозможно. Банковский домен задаёт приоритет: лучше чуть медленнее или временно недоступно, чем рассогласованный баланс. Это прямое применение CAP (при сетевом разделении выбираем согласованность) и PACELC (иначе — компромисс latency/consistency). Разрешение = осознанная приоритизация под домен.
