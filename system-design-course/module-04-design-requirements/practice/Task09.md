# Задача 09 — Модуль 04: Требования для дизайна

## ЗАДАНИЕ
Оцените масштаб системы из бизнес-контекста: выведите DAU/MAU и пиковую нагрузку, опираясь на разумные допущения.

## ВХОДНЫЕ ДАННЫЕ
Бизнес-описание: «Запускаем приложение для заказа такси в городе с населением 3 млн человек. Рассчитываем за 2 года охватить 20% взрослого населения. Поездки случаются неравномерно: утренний и вечерний час пик дают основную долю заказов.»

## ТРЕБОВАНИЯ
- Сформулируйте допущения: доля взрослых, проникновение, активность (поездок на пользователя в день), доля активных в день (DAU/MAU).
- Оцените MAU и DAU.
- Оцените число заказов в день и переведите в average RPS (заказы/сек).
- Оцените peak RPS с учётом часов пик (коэффициент к average).
- Покажите расчёт прозрачно (с единицами и порядками).

## ФОРМАТ РЕЗУЛЬТАТА
Список допущений + пошаговый расчёт MAU → DAU → заказы/день → average RPS → peak RPS.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Допущения явные и разумные (например: взрослых ~70%, проникновение 20%, DAU/MAU ~20–30%, ~1 поездка/активный/день).
- [ ] MAU ≈ 3M × 0.7 × 0.2 ≈ 420k; DAU ≈ 420k × ~0.25 ≈ ~100k (порядок величины).
- [ ] Заказы/день → average RPS = заказы/день / 86400 (получится единицы заказов/с — корректный порядок).
- [ ] Peak RPS = average × коэффициент часов пик (например, ×5–10), с обоснованием неравномерности.
- [ ] Все переходы прозрачны, единицы указаны, числа округлены до порядка.

## ПОДСКАЗКА
Двигайтесь по цепочке: население → взрослые → проникновение (MAU) → активность (DAU) → действия/день → /86400 (average RPS) → ×peak factor (peak RPS). Точные числа не важны — важен порядок величины и прозрачность допущений. Час пик в такси — классический случай высокого peak factor.
