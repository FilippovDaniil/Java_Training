# Задача 10 — Модуль 04 (мини-проект): Requirements-документ для дизайна

## ЗАДАНИЕ
Соберите компактный документ требований для дизайна заданной системы: ключевые FR, измеримые NFR, ограничения, явные допущения и scope. Это вход для всего последующего дизайна.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) система уведомлений; (б) сервис коротких ссылок; (в) лента соцсети. Числа, которых нет, закройте явными обоснованными допущениями.

## ТРЕБОВАНИЯ
Документ должен содержать разделы:
1. **Контекст и scope** — что система делает (1–2 предложения), что в границах, что вне.
2. **Ключевые FR** — 3–4 архитектурно-определяющие операции.
3. **NFR** — измеримые: масштаб (DAU, RPS read/write), латентность (p95/p99), доступность (SLA), согласованность, durability; по возможности безопасность.
4. **Ограничения** — технические/бюджетные/регуляторные/временные (если применимо).
5. **Допущения** — явные предположения с обоснованием.
6. **Read/write-профиль** — read-heavy или write-heavy (вывод из NFR).
- NFR обязаны быть измеримыми (число + метрика).

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный requirements-документ из 6 разделов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; scope разделяет «в границах / вне».
- [ ] FR — 3–4 ключевые операции, не весь продукт.
- [ ] Все NFR измеримы (есть числа: RPS, p99, %, durability-гарантия).
- [ ] Латентность выражена в перцентилях.
- [ ] Ограничения и допущения присутствуют и явные.
- [ ] Сделан вывод read-heavy / write-heavy с обоснованием из NFR.
- [ ] Документ внутренне согласован и готов служить входом для estimation (модуль 05).

## ПОДСКАЗКА
Это артефакт шага 1 фреймворка, оформленный полностью. Хороший requirements-документ делает дальнейший дизайн почти механическим: из измеримых NFR и read/write-профиля прямо следуют блоки. Проверьте связность: каждый NFR должен быть таким, что из него можно вывести дизайн-решение. Этот документ — мост к модулям 05–07 (estimation) и к полному дизайну в Части 8.
