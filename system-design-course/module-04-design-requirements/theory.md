# Модуль 04 — Требования для дизайна (FR/NFR/ограничения)

## Зачем этот модуль

Часть 1 дала рамку: что такое дизайн, как его документировать, как держаться на интервью. Часть 2 наполняет первые два шага процесса — **требования** (этот модуль) и **количественную оценку** (модули 05–07). Без них дизайн повисает в воздухе: нельзя выбрать БД, не зная нагрузки; нельзя обосновать кэш, не зная read/write-ratio.

Системный анализ собирает требования полно и формально. Здесь угол другой: **что из требований реально двигает архитектуру под нагрузкой**. Дизайнеру не нужна вся SRS — ему нужны несколько ключевых FR в фокусе, измеримые NFR, жёсткие ограничения и явные допущения. Этот минимальный, но точный набор и превращает «спроектируйте Twitter» в задачу, которую можно решить и защитить.

На интервью это шаг 1 фреймворка — самый частый провал. Этот модуль учит задавать **правильные** вопросы и фиксировать требования так, чтобы из них следовал дизайн.

---

## 1. FR для дизайна — не всё, а ключевое

**Функциональные требования (FR)** — что система делает. Для дизайна берут не полный список, а **2–4 ключевые операции**, на которых строится архитектура.

```
Twitter, полный список FR: твит, лента, подписки, лайки, ретвиты,
   DM, поиск, тренды, медиа, нотификации, закладки...
Для дизайна (фокус):  публикация твита  +  домашняя лента  +  подписки
```

> Цель — не описать продукт, а выбрать **архитектурно-определяющие** операции. Лента и публикация задают весь скелет (хранилище, fan-out, кэш); лайки и закладки — производные. Выбор фокуса вслух — часть управления scope на интервью.

---

## 2. NFR — то, что двигает дизайн

**Нефункциональные требования (NFR)** — качества под нагрузкой. Именно они определяют форму архитектуры. Ключевые для дизайна:

| NFR | Вопрос | Влияние на дизайн |
|-----|--------|-------------------|
| **Масштаб** | DAU, RPS чтения/записи | шардинг, число инстансов, очереди |
| **Латентность** | p50 / p95 / p99 целевые | кэш, CDN, реплики, индексы |
| **Доступность** | SLA (99.9 / 99.99%) | избыточность, мультирегион, failover |
| **Согласованность** | strong vs eventual | синхронная репликация vs async (CAP) |
| **Durability** | можно ли терять данные | репликация, WAL, бэкапы, гарантии очередей |
| **Безопасность** | приватность, регуляторика | шифрование, изоляция, аудит |

> Измеряйте latency в **перцентилях (p95/p99)**, а не в среднем: среднее прячет хвосты, а пользователи чувствуют именно хвост. «Быстро» → «p99 < 200 мс». «Надёжно» → «99.99% и durability: не теряем подтверждённые записи». Неизмеримый NFR бесполезен для дизайна.

---

## 3. Ограничения (constraints) ≠ требования

**Ограничение (constraint)** — внешняя рамка, которую нельзя выбрать; её принимают как данность. В отличие от требования (что нужно), constraint — что **нельзя или обязательно**.

| Тип ограничения | Пример |
|-----------------|--------|
| Технические | «только наш облачный провайдер», «существующая БД PostgreSQL» |
| Бюджетные | «инфраструктура ≤ $10k/мес» |
| Регуляторные | «данные граждан ЕС хранятся в ЕС (GDPR)», «152-ФЗ» |
| Организационные | «команда из 3 человек», «релиз через 2 месяца» |
| Временные | «MVP к чёрной пятнице» |

> Ограничения сужают пространство решений до старта дизайна. «Бюджет $10k/мес» сразу отсекает мультирегион active-active; «данные в ЕС» — определяет размещение. Спрашивайте о constraints на шаге 1: они дешевле всего меняют дизайн в самом начале.

---

## 4. Допущения (assumptions) — обязательны при неполных данных

Когда данных нет (а на интервью их почти всегда нет), вы делаете **явные допущения** и проговариваете их. Это не слабость — это инженерная честность.

```
"Точного DAU не знаю — предположим 100 млн, из них активны 50%.
 Read/write-ratio для ленты возьму 100:1 (чтений много больше).
 Если эти числа сильно другие, дизайн придётся пересмотреть — отмечу это."
```

> Допущение, проговорённое вслух, превращает неопределённость в управляемую: интервьюер либо подтвердит, либо поправит число. Молчаливое допущение («я просто решил, что…») — источник дизайна «не про то». Правило: **любое число, которое вы используете, должно иметь источник — данные или явное допущение.**

---

## 5. Scope — что в границах, что вне

**Scope** фиксирует периметр одной итерации дизайна (или интервью): какие фичи внутри, какие явно отложены.

```
В границах (итерация 1):  публикация поста, генерация ленты, подписки
Вне границ:               поиск, реклама, рекомендации, аналитика, DM
                          («при наличии времени/в следующей итерации»)
```

> Явный out-of-scope — не отговорка, а управление ожиданиями. Он показывает, что вы видите весь продукт, но сознательно сузили задачу до выполнимой. Без scope легко утонуть, пытаясь спроектировать всё сразу.

---

## 6. Как NFR двигают дизайн (главная идея)

Один и тот же FR при разных NFR даёт **разные архитектуры**. Это и есть суть System Design.

| FR | NFR | Дизайн-следствие |
|----|-----|------------------|
| Показать ленту | latency p99 < 100 мс, 50k RPS | предрасчёт ленты (fan-out-on-write) + кэш |
| Показать ленту | latency не критична, мало RPS | сборка на лету (fan-out-on-read), без кэша |
| Принять платёж | durability: ни одной потери | синхронная запись, репликация, идемпотентность |
| Записать клик | можно потерять единицы | async через очередь, батчинг, eventual |

> Запомните связку: **NFR → архитектурное решение**. На интервью каждое решение в дизайне должно отслеживаться к требованию. «Зачем здесь очередь?» → «NFR: пик 1М/мин и допустимая задержка — буферизуем через очередь». Решение без NFR-обоснования — угадывание.

---

## ⚠️ Подводные камни

- **Собирать все FR вместо ключевых.** Для дизайна нужны 2–4 архитектурно-определяющие операции, не весь продукт.
- **Неизмеримые NFR.** «Быстро/надёжно/масштабируемо» бесполезны; нужны числа (p99, %, RPS, durability-гарантия).
- **Среднее вместо перцентилей.** Latency меряют p95/p99 — среднее прячет хвосты, которые и чувствует пользователь.
- **Путать constraints и requirements.** Ограничение нельзя выбрать (его принимают); требование — что нужно обеспечить.
- **Скрытые допущения.** Любое число — из данных или из явно озвученного допущения; молчаливые догадки ведут к дизайну «не про то».
- **Нет scope.** Без явных границ дизайн расплывается; обозначьте out-of-scope.
- **Игнорировать NFR в дизайне.** Если решение не отслеживается к NFR — оно не обосновано.
- **Забыть про durability и согласованность.** Их часто не спрашивают, а они радикально меняют дизайн (платёж vs клик).
