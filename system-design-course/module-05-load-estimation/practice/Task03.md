# Задача 03 — Модуль 05: Оценка нагрузки

## ЗАДАНИЕ
Оцените read/write-ratio для нескольких систем и выведите из него направление дизайна.

## ВХОДНЫЕ ДАННЫЕ
Четыре системы:
1. Новостной портал (статьи читают многие, публикуют редакторы).
2. Система сбора метрик с серверов (агенты постоянно шлют данные, дашборды смотрят изредка).
3. Чат-мессенджер (пишут и читают примерно одинаково активно).
4. Система логирования (сервисы пишут логи непрерывно, читают при инцидентах).

## ТРЕБОВАНИЯ
- Для каждой системы оцените порядок read/write-ratio (с допущениями).
- Классифицируйте: read-heavy / write-heavy / сбалансированная.
- Для каждой назовите соответствующее направление дизайна.
- Объясните, почему write-heavy системы проектируют иначе, чем read-heavy.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «система → ratio → класс → направление дизайна» + вывод про write-heavy.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Портал — read-heavy (например, 1000:1) → кэш, CDN, реплики чтения, денормализация.
- [ ] Метрики — write-heavy → буферизация, батчинг, time-series/колоночное хранилище.
- [ ] Чат — близко к сбалансированному (≈1:1) → внимание и к записи, и к чтению.
- [ ] Логи — write-heavy → append-only, очереди, специализированное хранилище, ретеншн.
- [ ] Объяснено: write-heavy упирается в пропускную способность записи (буферизация/очереди/шардинг записи), read-heavy — в чтение (кэш/реплики).

## ПОДСКАЗКА
Ratio — это компас дизайна. Read-heavy «лечится» кэшем и репликами; write-heavy этим не спасти — там нужна буферизация записи, батчинг и хранилища, оптимизированные под запись (LSM, time-series). Метрики и логи — каноничные write-heavy примеры.
