# Задача 10 — Модуль 05 (мини-проект): Load estimation системы

## ЗАДАНИЕ
Проведите полную оценку нагрузки для заданной системы: от DAU до peak RPS по компонентам, с учётом read/write-ratio и fan-out, с проверкой и выводами для дизайна.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) лента соцсети; (б) сервис коротких ссылок; (в) мессенджер. Недостающие числа закройте явными допущениями.

## ТРЕБОВАНИЯ
Расчёт должен содержать:
1. **Допущения** — DAU, действий/пользователя/день, read/write, peak factor, fan-out (если есть).
2. **Average RPS** — отдельно чтение и запись (через 1 сутки ≈ 10^5 c).
3. **Peak RPS** — с обоснованным peak factor.
4. **Read/write-ratio** и вывод read-heavy/write-heavy.
5. **Fan-out** (если применим к системе) — внутреннее усиление нагрузки.
6. **Проверка** — обратный счёт или сравнение с ориентиром.
7. **Выводы для дизайна** — что эти числа означают (кэш? шардинг? очереди? сколько серверов порядка?).

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный расчёт из 7 пунктов выше, с единицами и порядками.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Допущения явные и обоснованные.
- [ ] Average RPS чтения и записи посчитаны корректно (степени десятки, единицы).
- [ ] Peak RPS получен с обоснованным factor.
- [ ] Read/write-ratio выведен, сделан вывод read-heavy/write-heavy.
- [ ] Fan-out учтён (если система его имеет) — внутренний RPS отличается от внешнего.
- [ ] Есть проверка (обратный счёт/ориентир).
- [ ] Выводы связывают числа с конкретными дизайн-решениями.

## ПОДСКАЗКА
Это шаг 2 фреймворка целиком. Главное — не точность, а связная цепочка допущения → расчёт → проверка → вывод. Расчёт существует ради последнего пункта: если в конце нет «значит, дизайн такой-то», работа не закончена. Эти числа станут входом для capacity (модуль 06) и cost (модуль 07), а затем для полного дизайна системы в Части 8.
