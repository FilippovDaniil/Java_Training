# Задача 06 — Модуль 06: Оценка ёмкости

## ЗАДАНИЕ
Оцените размер записи по её полям и выведите суммарный объём хранения за период.

## ВХОДНЫЕ ДАННЫЕ
Таблица событий аналитики кликов. Поля одной записи: event_id (UUID, 16 B), user_id (BIGINT, 8 B), item_id (BIGINT, 8 B), event_type (короткая строка, ~16 B), timestamp (8 B), context (JSON ~200 B). Частота: 50 000 событий/с. Срок хранения: 90 дней. Replication factor = 2.

## ТРЕБОВАНИЯ
- Сложите размер полей и оцените размер записи (добавьте ~20–30% на накладные расходы хранилища/индексы).
- Посчитайте объём в день и за 90 дней (логический).
- Примените replication factor.
- Сделайте вывод о классе хранилища (колоночное/time-series, сжатие, ретеншн).

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт размера записи по полям + объём/день и /90 дней + ×replication + вывод о хранилище.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сумма полей ≈ 16+8+8+16+8+200 = 256 B; с overhead ~300–330 B.
- [ ] События/день = 50 000 × 86 400 ≈ 4.3×10^9; объём/день ≈ 4.3×10^9 × ~300 B ≈ ~1.3 TB/день.
- [ ] За 90 дней ≈ ~115 TB (логический); с replication ×2 ≈ ~230 TB.
- [ ] Вывод: огромный поток write-heavy → колоночное/time-series хранилище со сжатием, обязательный ретеншн 90 дней, партиционирование по времени.
- [ ] Учтены и накладные расходы, и replication.

## ПОДСКАЗКА
Размер записи = сумма полей + overhead (индексы, заголовки строк, ~20–30%). Здесь доминирует поле context (JSON ~200 B) — оптимизация формата (сжатие, бинарный формат) сильно влияет на объём. Огромный поток событий с ретеншном — классика для колоночных/time-series баз со сжатием.
