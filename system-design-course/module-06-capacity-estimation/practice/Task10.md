# Задача 10 — Модуль 06 (мини-проект): Capacity estimation системы

## ЗАДАНИЕ
Проведите полную оценку ёмкости для заданной системы: storage на горизонте 5 лет, bandwidth (ingress/egress), память под кэш — с выводами для дизайна.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) сервис фото; (б) сервис коротких ссылок; (в) лента соцсети. Используйте нагрузку из своих допущений (можно взять числа из задачи 10 модуля 05). Недостающее закройте явными допущениями.

## ТРЕБОВАНИЯ
Расчёт должен содержать:
1. **Допущения** — размеры записей/объектов, частоты, срок хранения, replication factor, hit rate/hot set.
2. **Storage** — объём/день, /год, /5 лет; логический и физический (×replication).
3. **Bandwidth** — ingress и egress (на основе RPS из модуля 05).
4. **Memory** — горячий набор (80/20 или иной), число кэш-серверов.
5. **Latency-контекст** — где в системе самые дорогие по задержке операции (диск/сеть) и почему.
6. **Выводы для дизайна** — нужен ли шардинг? CDN? распределённый кэш? сколько нод порядка?

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный capacity-расчёт из 6 пунктов с единицами и порядками.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Допущения явные; учтены replication factor и срок хранения.
- [ ] Storage посчитан на 5 лет, логический и физический.
- [ ] Bandwidth разделён на ingress и egress; для медиа egress отмечен как доминирующий.
- [ ] Memory под кэш оценена (hot set), посчитано число кэш-серверов.
- [ ] Назван источник наибольшей latency (диск/межрегиональная сеть) с опорой на latency numbers.
- [ ] Выводы связывают ёмкость с решениями (шардинг при больших ТБ, CDN при большом egress, распределённый кэш при большом hot set).

## ПОДСКАЗКА
Это вторая половина estimation — вместе с модулем 05 она даёт полный количественный портрет системы. Связность важна: bandwidth опирается на RPS из load-оценки, memory — на hot set, storage — на частоту записи. Каждая оценка должна заканчиваться выводом для дизайна. Эти числа плюс нагрузка станут входом для cost (модуль 07) и для полного дизайна в Части 8.
