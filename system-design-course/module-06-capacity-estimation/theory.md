# Модуль 06 — Оценка ёмкости (storage/bandwidth/memory)

## Зачем этот модуль

Модуль 05 дал нагрузку (RPS). Этот модуль превращает её и размеры данных в **ёмкость**: сколько хранить (storage), какой канал нужен (bandwidth), сколько памяти под кэш (memory). Это вторая половина estimation — и она определяет выбор хранилища, число дисков, нужду в CDN и шардинге.

Здесь же — **«числа, которые надо знать»** (latency numbers every programmer should know): сколько стоит обращение к L1-кэшу, RAM, SSD, диску, сети. Без интуиции этих порядков нельзя оценить, где узкое место и почему кэш в памяти на порядки быстрее диска.

Навык тот же — back-of-the-envelope, но теперь в байтах и секундах задержки. На интервью «прикиньте, сколько данных за 5 лет» и «уместится ли горячий набор в память» — стандартные вопросы. Ошибка в порядке здесь стоит выбора неверного класса хранилища.

---

## 1. Единицы: байты и биты

| Единица | Степень | Примерно |
|---------|---------|----------|
| KB (килобайт) | 10^3 B | тысяча байт |
| MB (мегабайт) | 10^6 B | миллион |
| GB (гигабайт) | 10^9 B | миллиард |
| TB (терабайт) | 10^12 B | триллион |
| PB (петабайт) | 10^15 B | — |

```
1 byte = 8 bit          ← сеть часто в битах (Mbps, Gbps), хранение в байтах (MB, GB)
1 символ ASCII ≈ 1 B    1 символ UTF-8 ≈ 1–4 B
```

> Частая путаница: канал измеряют в **битах в секунду** (Mbps), а данные — в **байтах** (MB). Перевод — делить/умножать на 8. «100 Мбит/с» ≈ 12.5 МБ/с. Считаем степенями десятки, как с нагрузкой.

---

## 2. Типичные размеры данных (ориентиры)

| Что | Размер (порядок) |
|-----|------------------|
| BIGINT / timestamp / int64 | 8 B |
| UUID | 16 B |
| короткая строка (имя, код) | ~10–50 B |
| короткое текстовое сообщение/твит | ~100–300 B |
| строка БД (метаданные, неск. полей) | ~100 B – 1 KB |
| лог-строка | ~1 KB |
| фото (сжатое) | ~100 KB – 1 MB |
| минута видео (среднее качество) | ~10–50 MB |

> Эти числа — ориентиры для прикидки, не точные значения. Для оценки записи складывают размеры полей плюс накладные расходы (индексы, заголовки). Точность до байта не нужна — нужен порядок.

---

## 3. Storage estimation

Объём хранения = **размер записи × частота создания × срок хранения × replication factor**.

```
Пример (URL-shortener):
  Новых ссылок/день = 10^8 (100M)
  Размер записи ≈ 500 B (short_code, long_url, метаданные)
  В день  = 10^8 × 500 B = 5×10^10 B = 50 GB/день
  За год  = 50 GB × 365 ≈ 18 TB/год
  За 5 лет ≈ 90 TB
  × replication factor 3 ≈ 270 TB
```

| Множитель | Зачем |
|-----------|-------|
| Размер записи | объём одной единицы данных |
| Частота | сколько единиц в день/сек |
| Срок хранения | ретеншн (год, 5 лет, бессрочно) |
| Replication factor | копии для надёжности (обычно ×3) |

> Не забывайте **replication factor** и **срок хранения** — их часто упускают, и оценка занижается в разы. ×3 репликация и «храним 5 лет» легко превращают 18 ТБ в сотни ТБ. Это влияет на выбор: одна нода, шардинг или объектное хранилище.

---

## 4. Bandwidth estimation

Пропускная способность = **средний размер данных × RPS**. Считают отдельно **ingress** (входящий, запись) и **egress** (исходящий, чтение).

```
Пример (сервис фото):
  Загрузок/c (запись) = 1 000;  размер фото ≈ 500 KB
  Ingress = 1 000 × 500 KB = 5×10^8 B/c = 500 MB/c
  Просмотров/c (чтение) = 50 000;  отдаём то же фото
  Egress = 50 000 × 500 KB = 2.5×10^10 B/c = 25 GB/c  ← огромный исходящий поток!
```

> Egress (исходящий трафик) в read-heavy системах с медиа — обычно главная статья по объёму и по деньгам (модуль 07). Именно поэтому медиа отдают через **CDN**, а не из основного сервиса: CDN снимает egress с origin и приближает данные к пользователю.

## 5. Memory estimation (под кэш)

Сколько памяти нужно кэшу = размер **горячего набора** (hot data set). Часто применяют правило **80/20** (Парето): ~20% данных дают ~80% обращений — их и держат в памяти.

```
Пример (кэш ленты/постов):
  Всего активных постов в день = 10^8, размер поста ≈ 1 KB
  Кэшируем горячие 20%: 0.2 × 10^8 × 1 KB = 2×10^7 KB = 20 GB
  Один кэш-сервер ≈ 64 GB RAM → горячий набор влезает в 1 сервер (с запасом на overhead)
```

> Правило 80/20 — допущение для прикидки, не закон. Оно даёт быстрый ответ «влезает ли горячий набор в память одного/нескольких серверов». Если горячий набор — десятки ТБ, нужен распределённый кэш (модуль 09, 33). Память дорогая, поэтому кэшируют не всё, а горячее.

---

## 6. «Числа, которые надо знать» (latency numbers)

Порядки задержек разных операций — фундамент интуиции о производительности:

| Операция | Порядок задержки |
|----------|------------------|
| L1 cache reference | ~1 нс |
| L2 cache reference | ~4 нс |
| Обращение к RAM | ~100 нс |
| Чтение 1 МБ из RAM | ~10 мкс |
| SSD random read | ~100 мкс (0.1 мс) |
| Чтение 1 МБ с SSD | ~1 мс |
| Round-trip в пределах ДЦ | ~0.5 мс |
| Чтение 1 МБ с HDD | ~10–20 мс |
| Disk seek (HDD) | ~10 мс |
| Round-trip между континентами | ~150 мс |

```
Иерархия (грубо):  RAM  ≪  SSD  ≪  HDD  ≪  сеть между ДЦ  ≪  межконтинентальная сеть
                  ~100нс   ~0.1мс  ~10мс     ~единицы мс        ~150мс
```

> Ключевые выводы: **RAM на ~5 порядков быстрее межконтинентальной сети**; SSD на 1–2 порядка быстрее HDD; обращение к памяти ≈ в 100 000 раз быстрее, чем round-trip между континентами. Отсюда — почему кэшируют в RAM, почему география даёт latency, почему диск — узкое место. Точные числа знать не нужно — нужны **порядки и их соотношение**.

---

## ⚠️ Подводные камни

- **Забыть replication factor.** Реальный объём = логический × число копий (обычно ×3); без него занижение в разы.
- **Забыть срок хранения.** «Храним 5 лет» / «бессрочно» радикально меняет storage.
- **Путать биты и байты.** Сеть в Mbps, данные в MB; перевод ×/÷ 8.
- **Игнорировать egress.** В медиа-системах исходящий трафик — главная статья (объём и деньги); CDN обязателен.
- **Кэшировать всё.** Память дорогая; кэшируют горячий набор (80/20), а не весь датасет.
- **Не знать порядки latency.** Без интуиции RAM/SSD/диск/сеть нельзя найти узкое место.
- **Излишняя точность.** Порядок важнее цифр; «~20 ГБ», а не «20 481 МБ».
- **Считать только storage.** Bandwidth и memory не менее важны и часто определяют CDN/кэш.
