# Задача 06 — Модуль 07: Cost Estimation

## ЗАДАНИЕ
Примите решение build vs buy для брокера сообщений, сравнив TCO managed-сервиса и self-hosted на двух масштабах.

## ВХОДНЫЕ ДАННЫЕ
Нужна очередь/стриминг (Kafka-подобное). Два контекста:
- Стартап: команда 4 человека, умеренный объём событий, нет выделенных DevOps.
- Крупный продукт: команда 50+ человек, есть платформенная команда, очень большой объём событий круглосуточно.
Ориентиры: managed-стриминг ≈ $X/мес (включает эксплуатацию провайдером); self-hosted = дешевле по железу, но требует ~1–2 инженеров на поддержку (зарплаты — существенная статья).

## ТРЕБОВАНИЯ
- Для каждого контекста сравните TCO managed vs self-hosted (инфраструктура + люди + операционка).
- Учтите стоимость людей и операционную сложность, а не только цену ресурса.
- Сделайте рекомендацию для каждого контекста.
- Сформулируйте общий принцип выбора build vs buy.

## ФОРМАТ РЕЗУЛЬТАТА
TCO-сравнение для двух контекстов (с учётом людей) + рекомендации + общий принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Стартап: managed выгоднее — экономит дефицитное время 4 инженеров; зарплата DevOps на self-hosted перевешивает экономию на железе.
- [ ] Крупный продукт: self-hosted может окупиться — на огромном объёме экономия на железе перевешивает зарплаты платформенной команды, плюс нужен контроль/гибкость.
- [ ] В сравнении явно учтены люди и операционная сложность (TCO, а не цена ресурса).
- [ ] Рекомендации: стартап → managed (buy); крупный → рассмотреть self-hosted (build).
- [ ] Принцип: решают по TCO под контекст и масштаб; на старте обычно buy, на большом масштабе build может окупиться.

## ПОДСКАЗКА
Build vs buy — сравнение TCO, а не цены за ресурс. Managed дороже за единицу, но дешевле по людям и быстрее стартует. Для маленькой команды время инженеров дороже денег за сервис → buy. На огромном масштабе экономия на железе перевешивает зарплаты → build окупается. Это классический ADR.
