# Задача 08 — Модуль 07: Cost Estimation

## ЗАДАНИЕ
Сравните TCO двух архитектур для одной задачи и выберите экономичную под заданный масштаб, учитывая не только инфраструктуру.

## ВХОДНЫЕ ДАННЫЕ
Задача: бэкенд для среднего SaaS (умеренная нагрузка, команда 6 человек).
- Архитектура А: монолит + одна managed-БД + managed-кэш. Инфраструктура ≈ $3 000/мес. Эксплуатация — силами команды разработки, отдельного DevOps нет.
- Архитектура Б: 12 микросервисов + Kafka + свой Kubernetes + self-hosted БД-кластер. Инфраструктура ≈ $2 200/мес (дешевле железо), но требует ~1.5 выделенных инженера на эксплуатацию и замедляет разработку фич.

## ТРЕБОВАНИЯ
- Сравните инфраструктурную стоимость А и Б.
- Добавьте в TCO стоимость людей и операционную сложность (оцените порядок зарплат / замедления).
- Сделайте вывод, какая архитектура дешевле по TCO под этот масштаб.
- Свяжите вывод с принципом соразмерности дизайна (модуль 01).

## ФОРМАТ РЕЗУЛЬТАТА
Сравнение инфраструктуры + TCO с людьми/сложностью + вывод + связь с соразмерностью.

## КРИТЕРИИ ПРИЁМКИ
- [ ] По инфраструктуре Б чуть дешевле ($2 200 vs $3 000), но разница мала.
- [ ] В TCO добавлена стоимость ~1.5 инженера на эксплуатацию Б (зарплаты — десятки тыс. $/мес, на порядок больше разницы в инфраструктуре).
- [ ] Учтено замедление разработки фич в Б (операционная сложность как скрытый TCO).
- [ ] Вывод: А дешевле по TCO под этот масштаб — экономия на железе Б ничтожна против стоимости людей и потери скорости.
- [ ] Связь: преждевременная сложность (12 микросервисов на 6 человек) дорога не столько по железу, сколько по TCO — это и есть несоразмерный дизайн.

## ПОДСКАЗКА
Ловушка — сравнивать только счёт за инфраструктуру. Б «дешевле» по железу на $800/мес, но требует ~1.5 инженера (десятки тысяч $/мес) и тормозит разработку. TCO переворачивает картину: простой соразмерный монолит дешевле сложной микросервисной архитектуры на среднем масштабе. Сложность — это деньги.
