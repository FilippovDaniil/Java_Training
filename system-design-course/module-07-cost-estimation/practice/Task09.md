# Задача 09 — Модуль 07: Cost Estimation

## ЗАДАНИЕ
Оптимизируйте стоимость системы, не нарушая её NFR: предложите изменения и проверьте каждое на соответствие требованиям.

## ВХОДНЫЕ ДАННЫЕ
Сервис с счётом ≈ $50 000/мес. NFR: p99 чтения < 100 мс; durability — нельзя терять подтверждённые данные; доступность 99.95%. Текущий дизайн: всё в горячем хранилище, все инстансы on-demand, отдача статики из origin, БД с избыточным числом реплик «на всякий случай».

## ТРЕБОВАНИЯ
- Предложите 4–5 оптимизаций стоимости.
- Для каждой проверьте, не нарушает ли она NFR (latency/durability/доступность).
- Отметьте оптимизацию, которая выглядит заманчиво, но нарушила бы NFR (и почему её нельзя).
- Сделайте вывод о принципе «оптимизация под NFR, а не вопреки».

## ФОРМАТ РЕЗУЛЬТАТА
Список оптимизаций с проверкой на NFR + одна отвергнутая (нарушает NFR) + вывод-принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Безопасные оптимизации: tiering хранилища (холодные данные в архив — не влияет на p99 горячих), CDN для статики (улучшает latency и снижает egress), spot-инстансы для stateless/фоновой части (не для критичных), пересмотр избыточных реплик до разумного запаса.
- [ ] Каждая проверена на NFR (например, CDN не вредит durability; spot нельзя для stateful БД).
- [ ] Отвергнутая оптимизация: убрать репликацию БД ради экономии — нарушает durability/доступность; или перенести горячие данные в холодное хранилище — сломает p99.
- [ ] Вывод: оптимизируют под NFR; нельзя резать стоимость ценой нарушения latency/durability/доступности.
- [ ] Оценён порядок экономии без ущерба требованиям.

## ПОДСКАЗКА
Оптимизация стоимости — это поиск «жира», не затрагивающего NFR: tiering холодных данных, CDN, spot для некритичной части, разумный (а не избыточный) запас реплик. Граница: всё, что трогает durability (репликация БД) или p99 (горячие данные в архив), — под запретом. Дешевле — да, но не ценой SLA.
