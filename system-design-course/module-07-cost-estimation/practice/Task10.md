# Задача 10 — Модуль 07 (мини-проект): Cost-model системы

## ЗАДАНИЕ
Постройте модель стоимости для заданной системы: разбивку $/месяц по статьям, $/запрос, краткий TCO и список оптимизаций. Это завершение количественного фундамента (Часть 2).

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) сервис фото; (б) сервис коротких ссылок; (в) лента соцсети. Используйте нагрузку и ёмкость из своих расчётов модулей 05–06. Цены — как порядки/ориентиры.

## ТРЕБОВАНИЯ
Модель должна содержать:
1. **Допущения по ценам** — ориентиры $/инстанс, $/GB хранения, $/GB egress, managed-сервисы.
2. **Разбивка $/месяц** — по статьям: compute, storage, egress, managed, прочее.
3. **Главная статья** — что доминирует и почему.
4. **$/запрос и $/пользователя** — юнит-экономика (связь с бизнесом, если ARPU известен/предположен).
5. **TCO-замечание** — учтены ли люди/операционная сложность (build vs buy, если уместно).
6. **Оптимизации** — 3–4 способа снизить счёт без нарушения NFR, с оценкой эффекта.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированная cost-model из 6 разделов с прозрачными расчётами.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; ценовые допущения явные (как порядки).
- [ ] Разбивка $/мес по статьям; egress учтён (для медиа — выделен как крупный).
- [ ] Определена главная статья расходов.
- [ ] Посчитаны $/запрос и $/пользователя; сделана связь с экономикой.
- [ ] Есть TCO-замечание (люди/сложность), при уместности — build vs buy.
- [ ] Оптимизации направлены в главную статью и не нарушают NFR; оценён порядок экономии.

## ПОДСКАЗКА
Это финал Части 2: requirements (04) → load (05) → capacity (06) → cost (07). Связность обязательна: статьи стоимости опираются на ёмкость (storage, egress) и нагрузку (compute), а оптимизации — на профиль системы (read-heavy → кэш/CDN). Хорошая cost-model заканчивается не суммой, а пониманием, где деньги и как их сэкономить, не сломав SLA. Этот количественный портрет — вход в Часть 3 (строительные блоки).
