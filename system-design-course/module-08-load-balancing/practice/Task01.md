# Задача 01 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Выберите алгоритм балансировки под заданный профиль запросов и обоснуйте выбор сравнением с альтернативами.

## ВХОДНЫЕ ДАННЫЕ
Три сервиса с разным профилем:
- (А) stateless REST API: запросы однородны, ~10 мс каждый, бэкенды одинаковы;
- (Б) сервис экспорта отчётов: запросы очень разной длительности (от 50 мс до 30 с);
- (В) пул кэш-серверов: ключи должны стабильно попадать на «свой» узел, узлы иногда добавляют/выводят.

## ТРЕБОВАНИЯ
- Для каждого сервиса (А/Б/В) выберите алгоритм и объясните, почему.
- Для каждого назовите алгоритм, который здесь был бы **ошибкой**, и почему.
- Сведите в таблицу: сервис → алгоритм → причина → анти-выбор.

## ФОРМАТ РЕЗУЛЬТАТА
Сравнительная таблица из 3 строк + 1–2 предложения обоснования на каждую.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → round-robin (бэкенды равны, запросы однородны); анти-выбор — усложнять без нужды.
- [ ] (Б) → least connections / least response time (разная длительность → round-robin даёт перекос).
- [ ] (В) → consistent hashing (стабильная привязка ключ→узел, минимум переноса при изменении пула); анти-выбор — обычный hash/round-robin.
- [ ] Для каждого назван анти-выбор с причиной.
- [ ] Выбор увязан именно с профилем запросов, а не «по умолчанию».

## ПОДСКАЗКА
Алгоритм выбирают по двум осям: равны ли бэкенды и однородны ли запросы. Однородные + равные → round-robin. Разнородные по длительности → least connections. Нужна липкость по ключу → хеш, а при меняющемся пуле — consistent hashing (модуль 14).
