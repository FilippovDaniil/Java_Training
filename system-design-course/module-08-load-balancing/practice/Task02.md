# Задача 02 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Решите, на каком уровне балансировать (L4 или L7) для двух систем, и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) API-gateway микросервисов: запросы `/users/**`, `/orders/**`, `/payments/**` должны идти в разные сервисы; нужна TLS-termination и canary-выкатка по заголовку.
- (Б) прокси для протокола реального времени (игровой/голосовой трафик поверх UDP) с очень высокой пропускной способностью, контент не разбирается.

## ТРЕБОВАНИЯ
- Для каждой системы выберите L4 или L7 и перечислите, какие возможности уровня вы используете (или почему они не нужны).
- Укажите цену выбора (CPU/латентность/функциональность).
- Опишите, когда комбинируют оба уровня.

## ФОРМАТ РЕЗУЛЬТАТА
Решение по (А) и (Б) + таблица «возможность → нужна ли» + абзац про комбинирование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → L7: маршрутизация по пути, TLS-termination, routing по заголовку для canary — всё это требует разбора HTTP.
- [ ] (Б) → L4: не-HTTP/UDP, максимальная пропускная способность, контент не нужен; L7 был бы лишними затратами CPU.
- [ ] Названа цена L7 (разбор HTTP — больше CPU/латентности) и ограничение L4 (нет content-routing/TLS-termination).
- [ ] Описано комбинирование: L4 на входе (грубое распределение) + L7 глубже (умная маршрутизация).

## ПОДСКАЗКА
L7 «понимает» HTTP — отсюда content-routing, TLS-termination, sticky по cookie, но дороже по CPU. L4 работает с TCP/UDP-пакетами — быстро и для любых протоколов, но «слепо». Нужна маршрутизация по содержимому → L7; нужна сырая скорость и не-HTTP → L4.
