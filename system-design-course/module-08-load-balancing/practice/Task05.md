# Задача 05 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Спроектируйте отказоустойчивый слой балансировки так, чтобы сам LB не был единой точкой отказа. Изобразите схему.

## ВХОДНЫЕ ДАННЫЕ
Сейчас: клиенты → один nginx-LB → пул из 8 серверов. Требование: система должна переживать отказ балансировщика без простоя; точка входа — один домен.

## ТРЕБОВАНИЯ
- Объясните, почему текущая схема — SPOF, несмотря на 8 серверов.
- Предложите HA-схему балансировщика (минимум один вариант детально: active-passive или active-active).
- Нарисуйте текстовую/ASCII или Mermaid-диаграмму: как трафик доходит до пула при здоровом LB и при отказе LB.
- Укажите механизм переключения (floating IP/VRRP, DNS, anycast) и его время реакции/ограничения.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз SPOF + диаграмма HA + описание механизма failover.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Явно сказано: один LB → его отказ кладёт всю систему, число серверов за ним не спасает.
- [ ] Предложена HA-схема: active-passive (floating VIP/VRRP) или active-active (несколько LB + DNS/anycast).
- [ ] Диаграмма показывает оба состояния: норма и после отказа LB (куда переезжает трафик).
- [ ] Назван механизм failover и его свойство: VRRP — быстро, в пределах сети; DNS — есть TTL-задержка; anycast — переключение на сетевом уровне.
- [ ] Точка входа остаётся одним доменом/VIP для клиента.

## ПОДСКАЗКА
«А что если упадёт сам балансировщик?» — стандартный follow-up на интервью. Минимум — пара LB с плавающим IP. DNS-failover прост, но TTL задерживает переключение; VRRP внутри сети быстрее; anycast решает и гео, и HA сразу.
