# Задача 06 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне слоя балансировки и переформулируйте решения с обоснованием.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Ставим один HAProxy перед 12 серверами — этого достаточно для HA».
2. «Алгоритм — round-robin; часть запросов это быстрые чтения, часть — тяжёлые отчёты по 20 секунд».
3. «Сессии храним в памяти инстанса, поэтому включаем sticky sessions по IP».
4. «Health-check: TCP-коннект на порт прошёл → сервер здоров».
5. «При деплое просто выводим инстанс из пула в момент рестарта».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите, в чём ошибка и к чему она приводит.
- Предложите исправление с кратким обоснованием.
- Отметьте, какая из ошибок самая критичная (с точки зрения доступности).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод о самой критичной.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Один LB = SPOF, не HA → нужна HA-пара/active-active.
- [ ] (2) Round-robin при разнородной длительности даёт перекос → least connections / least response time.
- [ ] (3) Sticky лечит симптом; причина — stateful-инстанс → вынести сессию во внешний стор/JWT, сделать stateless.
- [ ] (4) TCP-коннект не значит «сервис здоров» (процесс жив, но БД мертва) → L7-readiness `GET /health` с разумной проверкой зависимостей.
- [ ] (5) Резкий вывод рвёт активные соединения → connection draining (graceful).
- [ ] Указана самая критичная ошибка для доступности (обычно п.1 — SPOF, кладёт всю систему).

## ПОДСКАЗКА
Ищите по чек-листу модуля: HA самого LB, алгоритм под профиль, stateless vs sticky, осмысленный health-check, graceful drain. «12 серверов за одним LB» — классическая ловушка: избыточность есть на слое приложений, но не на точке входа.
