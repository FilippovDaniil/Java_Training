# Задача 08 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Спроектируйте глобальную (geo) балансировку для системы, развёрнутой в нескольких регионах.

## ВХОДНЫЕ ДАННЫЕ
Сервис с пользователями в Европе, США и Азии. Развёрнут в 3 регионах. Требования: пользователь должен попадать в ближайший регион (latency), при отказе региона трафик должен уходить в другой, точка входа — один домен.

## ТРЕБОВАНИЯ
- Опишите двухуровневую балансировку: глобальный уровень (выбор региона) + локальный (выбор сервера).
- Выберите механизм глобального уровня (GSLB/DNS geo-routing или anycast) и обоснуйте.
- Опишите поведение при отказе целого региона (failover) и его ограничения (TTL DNS, «прогрев»).
- Нарисуйте схему: пользователь → глобальный уровень → регион → локальный LB → пул.

## ФОРМАТ РЕЗУЛЬТАТА
Описание двух уровней + выбор механизма + сценарий региона-failover + диаграмма.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Чётко разделены уровни: глобальный выбирает **регион** (по гео/здоровью), локальный — **сервер** внутри региона.
- [ ] Выбран механизм глобального уровня с обоснованием (DNS/GSLB — гибкая geo-логика, но TTL-задержка; anycast — переключение на сетевом уровне, быстрее).
- [ ] Описан региональный failover: при падении региона глобальный уровень перестаёт его отдавать, трафик идёт в живой регион.
- [ ] Названы ограничения (TTL DNS задерживает failover; перелив трафика грузит соседний регион — нужен запас ёмкости).
- [ ] Диаграмма отражает оба уровня.

## ПОДСКАЗКА
Локальный LB не знает про другие регионы — geo и регион-failover решает глобальный уровень. DNS-geo проще, но TTL замедляет переключение; anycast быстрее, но менее гибок в логике. Учтите: при отказе региона его трафик «переливается» в соседние — там нужен запас ёмкости (связь с задачей 07 и модулем 25).
