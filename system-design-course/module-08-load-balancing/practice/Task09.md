# Задача 09 — Модуль 08: Балансировка нагрузки

## ЗАДАНИЕ
Напишите ADR (Architecture Decision Record) по выбору типа балансировщика: hardware vs software vs cloud-managed.

## ВХОДНЫЕ ДАННЫЕ
Стартап на ранней стадии, небольшая команда без сетевых специалистов, развёртывание в публичном облаке, нагрузка пока умеренная, но ожидается рост. Рассматриваются: аппаратный LB, self-hosted software LB (nginx/HAProxy/Envoy), облачный managed LB.

## ТРЕБОВАНИЯ
Оформите ADR со структурой:
1. **Контекст** — ситуация и движущие силы (команда, облако, рост).
2. **Варианты** — hardware / software self-hosted / cloud-managed, по каждому плюсы и минусы (цена за ресурс, цена за людей/эксплуатацию, скорость старта, контроль, HA из коробки).
3. **Решение** — что выбрано.
4. **Последствия** — плюсы, минусы, точка пересмотра (при каком масштабе вернуться к вопросу).

## ФОРМАТ РЕЗУЛЬТАТА
ADR из 4 разделов (контекст → варианты → решение → последствия).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Контекст отражает ключевые силы: малая команда, облако, неопределённый рост.
- [ ] Все 3 варианта разобраны с плюсами/минусами; hardware отсеян (on-prem, дорого, не для облака), self-hosted даёт контроль ценой эксплуатации.
- [ ] Решение обосновано через TCO/контекст (для раннего этапа обычно cloud-managed: авто-HA, быстрый старт, экономия дефицитного времени инженеров).
- [ ] В последствиях есть минусы выбора (вендор-лок, цена на масштабе) и **точка пересмотра** (при каком объёме оценить self-hosted).
- [ ] Решение увязано со стоимостью как осью дизайна (модуль 07), а не «модно/привычно».

## ПОДСКАЗКА
Это решение про TCO, а не про цену за ресурс (модуль 07): managed дороже за трафик, но дешевле по людям и даёт HA из коробки. Хороший ADR не «выбирает раз навсегда», а фиксирует контекст и условие пересмотра.
