## Ключ (эталонные ответы)

1. LB распределяет запросы между серверами и решает: **масштабирование** (пул из N серверов держит суммарный RPS), **доступность** (выводит упавшие серверы), **скрытие топологии** (клиент знает один адрес, пул за ним меняется незаметно).
2. Вертикальное — наращивать мощность одного сервера (упирается в потолок, дорого). Горизонтальное — добавлять одинаковые серверы (почти линейно). LB — обязательное условие горизонтального: он распределяет трафик по пулу.
3. **L4** работает на транспортном уровне (IP/TCP/UDP, порты), маршрутизирует «вслепую», очень быстро. **L7** разбирает HTTP (URL, заголовки, cookies), умеет content-routing, TLS-termination, sticky по cookie, но дороже по CPU.
4. **L7** — когда нужна маршрутизация по содержимому (микросервисы, разные пути), TLS-termination, canary/A-B. **L4** — когда нужна сырая пропускная способность и/или не-HTTP-протоколы (UDP, голос, игры, DB-прокси).
5. L7 разбирает каждый HTTP-запрос → больше CPU и латентности на запрос. L4 лишь перекладывает пакеты → дёшево и быстро, но «слепо» (нет доступа к содержимому).
6. Round-robin, weighted round-robin, least connections, least response time, IP/ключевой hash, consistent hashing, random + power-of-two-choices.
7. Когда запросы разной длительности (быстрые чтения и долгие отчёты): round-robin раздаёт поровну по числу, но не по нагрузке → перекос. Заменяют на **least connections / least response time**.
8. Least connections направляет на сервер с наименьшим числом активных соединений — подходит для запросов разной длительности (стримы, long-poll), где важно фактическое количество «занятых» соединений, а не равный счёт.
9. **Consistent hashing** — отображение ключей на кольцо, при котором добавление/удаление узла переносит лишь малую долю ключей. В балансировке применяют для пулов кэша/шардов, чтобы изменение пула не «промахивало» весь кэш (подробно — модуль 14).
10. **Active** — LB сам периодически шлёт проверочный запрос (`/health`). **Passive** — LB судит по реальным ответам (ошибки/таймауты реального трафика). Active обнаруживает раньше, но создаёт нагрузку; passive реагирует только когда трафик уже пошёл.
11. Время обнаружения ≈ interval × unhealthy_threshold. Агрессивные параметры → быстрое обнаружение, но риск флаппинга (сервер скачет in/out от случайных сбоев). Вялые → стабильно, но дольше шлём трафик на нездоровый сервер.
12. **Liveness** — жив ли процесс (если нет — перезапустить). **Readiness** — готов ли принимать трафик (если нет — не слать запросы, но не убивать). Перегруженный/прогревающийся инстанс жив, но не готов.
13. Поверхностный (TCP-коннект или «200 OK» без проверки зависимостей) маскирует сбой — шлём трафик на сервер с мёртвой БД. Слишком глубокий (дёргает все downstream) даёт каскад: упала одна зависимость → весь пул помечен нездоровым.
14. **Connection draining** — при выводе сервера (деплой, scale-in) LB перестаёт слать новые запросы, но даёт текущим завершиться. Без него вывод рвёт активные соединения → ошибки у пользователей.
15. **Sticky sessions** — привязка клиента к одному серверу (по cookie/IP). Минусы: неравномерность нагрузки, потеря сессии при падении/выводе сервера, помеха масштабированию и деплою.
16. Потому что sticky нужен только если сервер хранит состояние в памяти. Это лечит симптом, а не причину; правильнее убрать состояние из сервера, тогда привязка не нужна.
17. Вынести сессию во внешний стор (Redis) или в подписанный токен (JWT). Тогда любой инстанс обслуживает любой запрос → можно убрать sticky, свободно балансировать, масштабировать и катить релизы.
18. Если LB один, его отказ делает недоступной всю систему, сколько бы серверов ни стояло за ним — весь трафик идёт через него. Это единая точка отказа.
19. Дублировать точку входа: HA-пара (active-passive с floating IP/VRRP) или несколько активных LB (active-active) с распределением через DNS/anycast.
20. **Active-passive** — основной LB работает, резервный ждёт; при отказе VIP переезжает на резерв. **Active-active** — несколько LB обслуживают трафик одновременно (через DNS/anycast), отказ одного снижает ёмкость, но не роняет вход.
21. **GSLB (Global Server LB)** — глобальная балансировка между регионами/ДЦ: DNS отдаёт IP ближайшего/здорового региона (geo-routing, latency-based, регион-failover).
22. **DNS-based** — отдаёт разные IP в зависимости от гео/здоровья, но переключение тормозит TTL кэша DNS. **Anycast** — один IP анонсируется из многих точек, сеть сама ведёт к ближайшей; переключение на сетевом уровне, быстрее, но логика менее гибкая.
23. **Глобальный уровень** выбирает регион (по близости/здоровью), **локальный LB** внутри региона выбирает конкретный сервер. Локальный не знает про другие регионы — geo и регион-failover решает глобальный.
24. **Аппаратный** (F5) — очень высокая нагрузка, on-prem, дорого. **Software** (nginx/HAProxy/Envoy) — гибко, дёшево, дефолт. **Cloud-managed** (ALB/NLB) — быстрый старт, авто-HA и масштабирование; выбор обычно по TCO и контексту (ранний этап → managed).
25. LB лишь распределяет трафик; если весь пул перегружен, он равномерно раздаёт боль, а не создаёт ёмкость. Поэтому пул считают с запасом **N+1** (переживание отказа сервера) и закладывают резерв под всплески, деплой и потерю зоны.
