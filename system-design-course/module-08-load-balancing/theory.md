# Модуль 08 — Балансировка нагрузки

## Зачем этот модуль

Часть 2 дала числа: нагрузка, ёмкость, стоимость. Из этих чисел почти всегда следует один вывод — **один сервер не справится**. Нужно много одинаковых серверов и нечто, что распределяет запросы между ними. Это «нечто» — **балансировщик нагрузки (load balancer, LB)**, первый и самый частый строительный блок любого дизайна на масштабе.

LB решает сразу три задачи: **масштабирование** (горизонтально добавляем серверы), **доступность** (упал сервер — трафик идёт на живые) и **скрытие топологии** (клиент знает один адрес, за ним — пул, который меняется незаметно). Почти на каждом design-интервью балансировщик появляется на high-level-диаграмме перед слоем приложений. Уметь обосновать его тип, алгоритм, проверки здоровья и собственную отказоустойчивость — базовый навык.

Этот модуль — про то, **как** распределять трафик, а не про конкретный продукт. Nginx, HAProxy, Envoy, ELB — это реализации одних и тех же принципов.

---

## 1. Зачем нужен балансировщик

| Задача | Как решает LB |
|--------|---------------|
| **Горизонтальное масштабирование** | пул из N одинаковых серверов обслуживает суммарный RPS; добавили сервер — выросла ёмкость |
| **Доступность** | health-check выводит упавший сервер из ротации, трафик идёт на живые |
| **Скрытие топологии** | клиент знает один VIP/домен; пул за ним меняется (деплой, autoscaling) прозрачно |
| **Распределение по близости** | geo/GSLB направляет к ближайшему дата-центру (latency) |

> Без LB горизонтальное масштабирование невозможно: некому распределять. LB — это точка, превращающая «один мощный сервер» (vertical scaling, упирается в потолок) в «много дешёвых» (horizontal scaling, почти линейно).

---

## 2. Уровни: L4 vs L7

Балансировка работает на разных уровнях модели OSI. На практике важны два.

| | **L4 (transport)** | **L7 (application)** |
|--|--------------------|----------------------|
| Видит | IP-пакеты, TCP/UDP, порты | HTTP-запрос: URL, заголовки, cookies, метод |
| Решение о маршруте | по IP/порту (часто NAT/проброс соединения) | по содержимому (путь, хост, заголовок) |
| Скорость | очень быстро, мало CPU | медленнее (разбирает HTTP) |
| Возможности | простое распределение TCP/UDP | content-based routing, TLS-termination, sticky по cookie, заголовки, перезапись путей |
| Примеры | AWS NLB, LVS, L4-режим HAProxy | nginx, Envoy, AWS ALB, HAProxy в HTTP-режиме |

```
L4:  клиент → [LB: по порту/IP] → сервер   (не заглядывает внутрь запроса)
L7:  клиент → [LB: GET /api/* → сервис A, /img/* → сервис B] → сервис
```

> **Правило выбора:** L7 — когда нужна маршрутизация по содержимому (микросервисы, разные пути → разные сервисы), TLS-termination, A/B, canary. L4 — когда нужна максимальная пропускная способность и не-HTTP-протоколы (game/voice/DB-прокси) или когда контент не важен. Часто их комбинируют: L4 на входе (грубое распределение), L7 глубже (умная маршрутизация).

## 3. Алгоритмы балансировки

| Алгоритм | Как работает | Когда уместен |
|----------|--------------|---------------|
| **Round-robin** | по кругу, поровну | бэкенды равны, запросы однородны |
| **Weighted RR** | по кругу с весами | бэкенды разной мощности |
| **Least connections** | туда, где меньше активных соединений | запросы разной длительности (long-poll, стримы) |
| **Least response time** | минимум (соединения + latency) | чувствительность к latency, разнородная нагрузка |
| **IP hash / hash по ключу** | сервер = hash(client IP / ключ) | нужна привязка клиента к серверу без cookie |
| **Consistent hashing** | hash на кольце | кэш-серверы: добавление/удаление узла переносит мало ключей (см. модуль 14) |
| **Random + power-of-two-choices** | выбрать 2 случайных, отдать менее загруженному | большой пул; почти как least-conn, но дёшево |

> Round-robin — разумный дефолт для равных stateless-бэкендов. Как только запросы становятся разной длительности (одни 5 мс, другие 5 с), round-robin создаёт перекос — переходят к **least connections / least response time**. Хеш-алгоритмы дают «липкость» без cookie, но ломают равномерность при изменении пула (лечится consistent hashing).

## 4. Health checks — проверки здоровья

LB должен направлять трафик только на живые серверы. Для этого он постоянно проверяет их состояние.

| Тип | Как | Плюсы / минусы |
|-----|-----|----------------|
| **Active** | LB сам шлёт периодический запрос (`GET /health`) | явный сигнал; нагрузка проверок; задержка обнаружения = интервал × порог |
| **Passive** | LB смотрит на реальные ответы (ошибки, таймауты) | без лишних запросов; реагирует только когда трафик уже пошёл |

Ключевые параметры active-проверки:
```
interval        — как часто проверять (напр. 5 с)
timeout         — сколько ждать ответ
unhealthy_threshold — сколько провалов подряд → вывести из ротации (напр. 3)
healthy_threshold   — сколько успехов подряд → вернуть (напр. 2)
```

> **Health-эндпоинт должен проверять зависимости с умом.** Поверхностный `/health` (отвечает «200 OK», даже если БД недоступна) маскирует проблему. Слишком глубокий (дёргает все downstream) даёт каскадные false-positive: упала одна зависимость — все инстансы помечены нездоровыми, хотя могли бы обслуживать часть трафика. Разделяют **liveness** (жив ли процесс) и **readiness** (готов ли принимать трафик). Связь с K8s-probes — та же модель.

**Connection draining (graceful shutdown):** перед выводом сервера (деплой, scale-in) LB перестаёт слать новые запросы, но даёт текущим завершиться — иначе обрыв активных соединений.

## 5. Session affinity (sticky sessions)

Иногда нужно, чтобы запросы одного клиента шли на **тот же** сервер — например, если сервер хранит сессию в памяти.

| Способ | Как |
|--------|-----|
| По cookie (L7) | LB ставит/читает cookie с ID сервера |
| По IP (L4, IP hash) | hash(client IP) → сервер |

**Минусы sticky sessions:**
- неравномерность (тяжёлый клиент привязан к одному серверу);
- потеря сессии при падении/выводе сервера;
- мешает свободному масштабированию и деплою.

```
❌ Stateful сервер + sticky:  сессия в памяти инстанса → падёт инстанс → пользователь разлогинен
✅ Stateless сервер:          сессия во внешнем сторе (Redis/JWT) → любой инстанс обслужит любого
```

> **Sticky sessions — это «запах» stateful-сервера.** Предпочтительная архитектура — **stateless-серверы**: состояние сессии выносят во внешний стор (Redis) или в подписанный токен (JWT). Тогда любой инстанс обслуживает любой запрос, балансировка свободна, масштабирование и деплой просты. Sticky оставляют как вынужденное решение (legacy, WebSocket-привязка), а не дизайн по умолчанию.

## 6. Сам LB — это SPOF. Делаем его отказоустойчивым

Если балансировщик один — он **единая точка отказа (SPOF)**: упал LB → недоступна вся система, сколько бы серверов ни было за ним.

| Схема | Как работает |
|-------|--------------|
| **Active-passive** | основной LB + резервный; floating IP/VRRP переезжает на резерв при отказе |
| **Active-active** | несколько LB работают одновременно; распределение через DNS/anycast |
| **DNS round-robin / anycast** | несколько IP/точек входа; клиент попадает на один из живых LB |

```
       ┌─ floating VIP ─┐
клиент → LB-active   LB-passive   (VRRP: VIP переезжает при отказе active)
            │
       пул серверов
```

> Классический промах на интервью — нарисовать один LB и забыть, что он сам может упасть. Senior всегда показывает, что **точка входа дублирована** (HA-пара или несколько активных LB + DNS/anycast). «А что если упадёт сам балансировщик?» — стандартный follow-up.

---

## 7. Глобальная балансировка: DNS, GSLB, anycast

Внутри одного дата-центра — локальный LB. Между регионами/ДЦ — **глобальная** балансировка:

- **DNS-based / GSLB** (Global Server Load Balancing): DNS отдаёт IP ближайшего/здорового ДЦ (geo-routing, latency-based, failover между регионами).
- **Anycast**: один IP анонсируется из многих точек; сеть сама ведёт клиента к ближайшей (используется CDN, DNS).

```
Глобально:  пользователь → GSLB/DNS → ближайший здоровый регион
Локально:   регион → LB → пул сервисов
```

> Двухуровневая модель: **глобальный LB выбирает регион** (по гео/здоровью), **локальный LB выбирает сервер** внутри региона. Это основа multi-region (модуль 25) и работы CDN (модуль 10).

## 8. Типы реализаций

| Тип | Примеры | Когда |
|-----|---------|-------|
| **Аппаратный** | F5 BIG-IP, Citrix | очень высокая нагрузка, on-prem, дорого |
| **Программный** | nginx, HAProxy, Envoy, LVS | гибко, дёшево, дефолт в большинстве систем |
| **Облачный managed** | AWS ALB/NLB, GCP LB, Azure LB | быстрый старт, авто-HA и масштабирование, оплата по трафику |

**Reverse proxy vs LB:** reverse proxy (nginx) принимает запросы от имени серверов и может попутно балансировать, кэшировать, терминировать TLS. LB — частный случай/функция. На практике один nginx/Envoy совмещает обе роли.

## ⚠️ Подводные камни

- **Один LB без HA.** Балансировщик сам — SPOF; нужна active-passive/active-active схема.
- **Round-robin при разнородных запросах.** Создаёт перекос; для разной длительности — least connections.
- **Sticky sessions как дизайн по умолчанию.** Признак stateful-сервера; правильнее stateless + внешняя сессия.
- **Поверхностный health-check.** «200 OK» при мёртвой БД маскирует сбой; слишком глубокий даёт каскадные false-positive. Разделяй liveness/readiness.
- **Нет connection draining.** Вывод сервера рвёт активные соединения; нужен graceful drain.
- **L7 там, где хватило бы L4 (и наоборот).** L7 дороже по CPU; L4 не умеет content-routing/TLS-termination.
- **Забыть про глобальный уровень.** Локальный LB не решает geo-latency и регион-failover — нужен GSLB/anycast.
- **LB маскирует нездоровый пул.** Если все серверы перегружены, LB лишь равномерно раздаёт боль; балансировка не заменяет ёмкость.
