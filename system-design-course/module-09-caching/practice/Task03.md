# Задача 03 — Модуль 09: Кэширование

## ЗАДАНИЕ
Выберите стратегию инвалидации кэша (TTL vs событийная vs комбинация) под разные данные и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Три набора данных:
- (А) каталог товаров: меняется несколько раз в день, пользователи не должны видеть старые цены дольше пары минут;
- (Б) лента курсов валют: обновляется раз в минуту, секундная задержка терпима;
- (В) баланс счёта пользователя: устаревшее значение недопустимо.

## ТРЕБОВАНИЯ
- Для каждого набора выберите стратегию инвалидации и обоснуйте через допустимую staleness.
- Для (А) и (В) опишите механизм событийной инвалидации (что триггерит, удаление vs обновление ключа).
- Укажите окно устаревания (stale window) для выбранной стратегии.
- Объясните, зачем комбинируют событийную инвалидацию с TTL-страховкой.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: данные → допустимая staleness → стратегия → окно устаревания + абзац про комбинацию.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Стратегия выводится из допустимой staleness, а не наоборот.
- [ ] (Б) → TTL (≈60 c терпимо, просто); (А) → короткий TTL или событийная (окно ≈ минуты); (В) → событийная/write-through или не кэшировать (окно ≈ задержка события).
- [ ] Описана событийная инвалидация: изменение → удаление ключа (надёжнее обновления из-за гонок).
- [ ] Для каждой стратегии указано окно устаревания.
- [ ] Объяснено: событийная + TTL-страховка ловит пропущенные события (защита от «вечного» стейла).

## ПОДСКАЗКА
Первый вопрос — «насколько устаревшее терпимо?». Терпим секунды → TTL. Нужна свежесть → событийная (сложнее). Критичные данные иногда не кэшируют вовсе. Удаление ключа надёжнее обновления (меньше гонок read-modify-write).
