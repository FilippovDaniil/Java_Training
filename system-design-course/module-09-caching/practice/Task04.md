# Задача 04 — Модуль 09: Кэширование

## ЗАДАНИЕ
Выберите политику вытеснения (eviction) под профиль доступа и объясните связь с TTL.

## ВХОДНЫЕ ДАННЫЕ
- (А) кэш сессий: доступ равномерный во времени, недавно использованные — вероятнее снова;
- (Б) кэш «топ-100 товаров»: небольшой стабильный набор горячих ключей плюс длинный хвост случайных;
- (В) кэш с требованием: никакое значение не должно жить дольше 10 минут (свежесть).

## ТРЕБОВАНИЯ
- Для (А) и (Б) выберите политику (LRU/LFU/FIFO) и обоснуйте через профиль доступа.
- Для (В) объясните роль TTL и почему он ортогонален политике вытеснения.
- Покажите, как TTL и eviction-политика работают вместе на одной записи.
- Назовите риск неверного выбора (напр. LRU при стабильно горячем наборе).

## ФОРМАТ РЕЗУЛЬТАТА
Решение по (А)/(Б)/(В) + объяснение «TTL + политика вместе» + риск.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → LRU (временна́я локальность: недавнее снова вероятно).
- [ ] (Б) → LFU (стабильно частые ключи не должны вытесняться всплеском «хвоста»; LRU тут хуже).
- [ ] (В) → TTL ограничивает свежесть; объяснено, что TTL (по времени) и eviction (по давлению памяти) — разные оси и работают вместе.
- [ ] Показано: у записи есть и TTL (протухание), и участие в политике вытеснения при нехватке памяти.
- [ ] Назван риск: LRU при горячем наборе может выкинуть горячий ключ под напором случайного хвоста.

## ПОДСКАЗКА
LRU хорош при временно́й локальности; LFU — когда есть стабильно горячие ключи, которые не должны вымываться редкими всплесками. TTL — это не про память, а про свежесть; он работает поверх любой eviction-политики.
