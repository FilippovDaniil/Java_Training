# Задача 05 — Модуль 09: Кэширование

## ЗАДАНИЕ
Спроектируйте защиту от cache stampede (thundering herd) для популярного ключа.

## ВХОДНЫЕ ДАННЫЕ
Главная страница использует кэшированный результат тяжёлого запроса (агрегация, ~800 мс в БД). Ключ один, его читают 20 000 RPS. При истечении TTL наблюдается всплеск: тысячи запросов одновременно промахиваются и бьют в БД, та захлёбывается.

## ТРЕБОВАНИЯ
- Объясните механику stampede на этом примере (что именно происходит в момент истечения TTL).
- Предложите минимум 2 способа защиты и опишите, как каждый работает.
- Выберите основной способ для этого кейса и обоснуйте.
- Укажите, что произойдёт без защиты при 20 000 RPS и пересчёте 800 мс.

## ФОРМАТ РЕЗУЛЬТАТА
Механика → 2+ способа защиты → выбор основного + обоснование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Механика: TTL истёк → все запросы видят промах одновременно → лавина одинаковых тяжёлых запросов в БД.
- [ ] Предложены ≥2 способа: блокировка/lease (один поток пересчитывает, остальные ждут/отдают старое), request coalescing (склейка одинаковых запросов), probabilistic early expiration (обновлять чуть раньше TTL).
- [ ] Выбран основной способ с обоснованием (для горячего ключа — lock/lease + отдавать старое значение во время пересчёта).
- [ ] Оценён ущерб без защиты: ~20 000 одновременных тяжёлых запросов в БД за окно пересчёта → перегрузка.
- [ ] Решение сохраняет доступность (пользователь получает старое значение, а не ошибку, пока идёт пересчёт).

## ПОДСКАЗКА
Суть защиты — чтобы пересчитывал **только один** запрос, а остальные либо ждали его результат, либо получали чуть устаревшее значение. Probabilistic early expiration размазывает обновление по времени до истечения, убирая синхронный «обрыв».
