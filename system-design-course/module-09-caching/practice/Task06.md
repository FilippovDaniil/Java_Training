# Задача 06 — Модуль 09: Кэширование

## ЗАДАНИЕ
Защитите систему от cache penetration и cache avalanche.

## ВХОДНЫЕ ДАННЫЕ
- Penetration: на API `GET /products/{id}` идёт поток запросов с несуществующими id (вероятно, перебор/атака). Каждый промахивает кэш и бьёт в БД.
- Avalanche: при ночном прогреве кэша всем записям выставляют одинаковый TTL = 1 час. Ровно через час масса ключей истекает одновременно — пик нагрузки на БД.

## ТРЕБОВАНИЯ
- Для penetration предложите защиту (минимум 2 механизма) и их trade-off.
- Для avalanche объясните причину и предложите исправление.
- Опишите, как отличить penetration от обычных промахов (нужна ли защита всегда).
- Кратко свяжите обе патологии с защитой от stampede (чем отличаются).

## ФОРМАТ РЕЗУЛЬТАТА
Раздел penetration (2 механизма + trade-off) + раздел avalanche (причина + фикс) + сравнение патологий.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Penetration: кэширование «пусто» (null с коротким TTL) и/или bloom filter существующих ключей; trade-off (null-кэш прост, но память на мусор; bloom — компактно, но ложные срабатывания).
- [ ] Avalanche: причина — одинаковый TTL у массы ключей; фикс — разброс TTL (jitter), разные сроки, прогрев.
- [ ] Объяснено, что penetration характерен потоком несуществующих ключей (часто атака), в отличие от штатных промахов.
- [ ] Показано различие патологий: stampede — один горячий ключ истёк; avalanche — много ключей разом; penetration — ключей вообще нет в БД.
- [ ] Защита не нарушает корректность (null-кэш с коротким TTL, чтобы появившийся позже ключ стал виден).

## ПОДСКАЗКА
Три патологии путают, но они разные: penetration — ключа нет в БД (кэшируем «нет»), avalanche — много ключей истекли вместе (разносим TTL), stampede — один горячий ключ истёк (пересчитывает один). Bloom filter заранее отсекает заведомо отсутствующие ключи, не доходя до БД.
