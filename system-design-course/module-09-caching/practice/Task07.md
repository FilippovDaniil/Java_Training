# Задача 07 — Модуль 09: Кэширование

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне кэширования и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Кэшируем всё, что читается, без TTL — пусть живёт, пока есть память».
2. «Данные в кэше обновляем только при перезапуске сервиса».
3. «Каждый инстанс держит свою локальную копию справочника прав доступа в памяти».
4. «Всем ключам ставим TTL = 1 час, прогреваем кэш при старте».
5. «Hit rate не меряем — кэш же стоит, значит, работает».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка опаснее всего для корректности данных.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод о самой опасной.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Без TTL и инвалидации → вечно устаревшие данные + засорение памяти холодным → TTL + кэшировать только горячее.
- [ ] (2) Обновление лишь при рестарте → пользователи видят устаревшее произвольно долго → событийная инвалидация/TTL.
- [ ] (3) Локальные копии прав у каждого инстанса → рассинхрон (один видит старые права) → распределённый кэш/общий источник, особенно для авторизации.
- [ ] (4) Одинаковый TTL + прогрев → cache avalanche через час → jitter TTL.
- [ ] (5) Не мерить hit rate → кэш может быть бесполезен (низкий hit rate = чистые расходы) → мониторить hit rate.
- [ ] Указана самая опасная для корректности (обычно п.3 — устаревшие права доступа = риск безопасности; или п.2 — произвольно старые данные).

## ПОДСКАЗКА
Чек-лист модуля: определена ли staleness, есть ли инвалидация, локальный vs распределённый для общих данных, jitter TTL против avalanche, меряется ли hit rate. Кэш прав доступа без инвалидации — это уже про безопасность, а не только про свежесть.
