# Задача 08 — Модуль 09: Кэширование

## ЗАДАНИЕ
Спроектируйте многоуровневый кэш для системы и распределите данные по уровням.

## ВХОДНЫЕ ДАННЫЕ
Новостной портал: статичные ассеты (CSS/JS/картинки), HTML-страницы статей (меняются редко), персонализированная лента (для каждого пользователя своя), счётчики просмотров (часто пишутся). Пользователи по всему миру.

## ТРЕБОВАНИЯ
- Распределите каждый тип данных по уровням кэша: client/browser, CDN/edge, reverse-proxy, application local, distributed (Redis).
- Для каждого уровня укажите, что кэшируется, с каким TTL/инвалидацией и почему именно здесь.
- Объясните, что НЕ стоит кэшировать на CDN/edge (и почему).
- Нарисуйте схему пути запроса через уровни.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «тип данных → уровень(и) → TTL/инвалидация → причина» + диаграмма уровней.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Статичные ассеты → browser + CDN (versioned URL, длинный TTL).
- [ ] HTML статей → CDN/reverse-proxy (умеренный TTL + purge при правке).
- [ ] Персонализированная лента → НЕ на CDN (общий кэш для всех сломает приватность/персонализацию); distributed/app-кэш по пользователю.
- [ ] Счётчики → distributed (write-back/инкремент) или отдельный механизм; не кэшировать в CDN.
- [ ] Объяснено, почему персональные/часто меняющиеся данные не кладут на shared edge.
- [ ] Диаграмма отражает путь client → CDN → proxy → app → distributed → DB.

## ПОДСКАЗКА
Принцип: чем ближе к пользователю и чем более «общие» данные, тем выше уровень кэша. Персонализированное и приватное нельзя класть в shared CDN-кэш (увидят чужое). Versioned URL снимает проблему инвалидации статики (новый файл — новый URL).
