# Задача 09 — Модуль 09: Кэширование

## ЗАДАНИЕ
Рассчитайте размер распределённого кэша (память и число узлов) под горячий набор данных.

## ВХОДНЫЕ ДАННЫЕ
- Всего объектов: 200 млн, средний размер записи в кэше ≈ 1 КБ.
- Профиль доступа подчиняется правилу 80/20: ~20% объектов дают ~80% запросов (горячий набор).
- Replication factor кэша = 2 (для доступности горячих ключей).
- Накладные расходы (метаданные, фрагментация) ≈ 30%.
- Узел кэша: ~64 ГБ полезной памяти.

## ТРЕБОВАНИЯ
- Оцените размер горячего набора (объектов и в байтах).
- Добавьте replication factor и накладные расходы.
- Посчитайте число узлов кэша (с округлением вверх).
- Сделайте вывод: что будет, если памяти меньше hot set (эффект на hit rate).

## ФОРМАТ РЕЗУЛЬТАТА
Прозрачный расчёт по шагам + число узлов + вывод про недостаток памяти.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Hot set: 20% × 200 млн = 40 млн объектов × 1 КБ = 40 ГБ (как порядок).
- [ ] С replication ×2 = 80 ГБ; с накладными +30% ≈ 104 ГБ.
- [ ] Узлов: ⌈104 / 64⌉ = 2 узла (с запасом — 3, чтобы не на пределе).
- [ ] Вывод: если памяти меньше hot set, eviction начнёт вытеснять горячее → hit rate падает → нагрузка возвращается в БД.
- [ ] Числа трактуются как порядки; кэшируется горячий набор, а не все 200 ГБ (модуль 06).

## ПОДСКАЗКА
Кэшируют не все данные, а горячий набор (80/20). Размер = hot set × replication × (1 + накладные). Узлов столько, чтобы hot set влезал с запасом: впритык → eviction выкидывает нужное и hit rate проседает. Связь с capacity-estimation (модуль 06).
