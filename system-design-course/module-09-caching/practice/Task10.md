# Задача 10 — Модуль 09 (мини-проект): Стратегия кэширования read-heavy системы

## ЗАДАНИЕ
Спроектируйте полную стратегию кэширования для read-heavy системы — синтез всего модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) лента/таймлайн соцсети; (б) каталог интернет-магазина; (в) сервис профилей пользователей. Нагрузку и hot set возьмите из расчётов модулей 05–06 (или задайте явно как допущение). Read/write-ratio — выраженно read-heavy.

## ТРЕБОВАНИЯ
Стратегия должна покрывать:
1. **Что кэшируем** — какие данные и почему (горячий набор), что НЕ кэшируем.
2. **Допустимая staleness** — для каждого типа данных; из неё выводится остальное.
3. **Уровни кэша** — где (client/CDN/proxy/app/distributed) и что на каждом.
4. **Паттерны чтения/записи** — cache-aside/through, write-through/back/around с обоснованием.
5. **Инвалидация** — TTL / событийная / комбинация; окно устаревания.
6. **Защита от патологий** — stampede, penetration, avalanche, hot key.
7. **Размер и эффект** — оценка hot set / числа узлов и ожидаемого hit rate с эффектом на БД.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов с прозрачными расчётами и обоснованиями.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; нагрузка/hot set заданы (своя оценка или допущение).
- [ ] Для каждого типа данных указана допустимая staleness, и из неё выведена стратегия инвалидации.
- [ ] Уровни кэша распределены осмысленно; персональные/приватные данные не кладутся в shared edge.
- [ ] Паттерны чтения/записи выбраны под профиль с обоснованием.
- [ ] Адресованы все три патологии (stampede/penetration/avalanche) + hot key.
- [ ] Посчитаны hot set / число узлов и ожидаемый hit rate с эффектом на нагрузку БД (связь с модулями 06–07).
- [ ] Видна связность: read-heavy профиль → высокий приоритет кэша → меньше реплик БД → ниже стоимость.

## ПОДСКАЗКА
Это синтез модуля и мост к следующим блокам: за кэшем — CDN (модуль 10), БД и репликация (модули 11–12). Хорошая стратегия начинается с «какая staleness допустима», а заканчивается числами: hit rate, разгрузка БД, экономия. Кэш — главный рычаг для read-heavy систем: он бьёт по latency, нагрузке и стоимости одновременно.
