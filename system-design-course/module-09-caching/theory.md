# Модуль 09 — Кэширование на масштабе

## Зачем этот модуль

Балансировка (модуль 08) даёт горизонтальный масштаб, но не уменьшает **объём работы**: каждый запрос всё равно идёт в БД. Кэш — это блок, который **уменьшает работу**: хранит горячие данные ближе и быстрее, чтобы не пересчитывать и не перечитывать одно и то же. Из трёх осей он бьёт сразу по всем: **latency** (память вместо диска/сети), **нагрузка на БД** (кэш снимает большинство чтений) и **стоимость** (память дешевле десятка read-реплик — модуль 07).

Кэш кажется простым («положили в Redis»), но именно здесь живут самые коварные баги дизайна: устаревшие данные, расхождение кэша и БД, лавины запросов при истечении TTL, перегрев одного «горячего» ключа. Недаром Фил Карлтон сформулировал: *«В Computer Science есть только две трудные вещи — инвалидация кэша и именование сущностей»*.

Этот модуль — про то, **где** кэшировать, **по какому паттерну** писать и читать, **как инвалидировать** и **как пережить патологии** (stampede, penetration, avalanche).

---

## 1. Зачем кэшировать и какой ценой

| Выигрыш | Цена / риск |
|---------|-------------|
| Latency: память ≈ десятки нс — мкс vs диск/сеть мс | Данные в кэше могут устареть (staleness) |
| Разгрузка БД: кэш снимает 80–95% чтений | Сложность: инвалидация, согласованность |
| Стоимость: кэш дешевле, чем +N read-реплик | Память ограничена → eviction, выбор что хранить |
| Сглаживание пиков: горячие данные не доходят до БД | Патологии: stampede, penetration, avalanche |

> Кэш — это **обмен свежести на скорость и стоимость**. Поэтому первый вопрос дизайна не «какой Redis», а «насколько устаревшие данные допустимы для этих данных?». Для курса валют — секунды терпимо; для баланса счёта — почти нет. Допустимая staleness задаёт всю стратегию.

---

## 2. Где кэшировать — уровни

Кэш есть на каждом «слое» пути запроса:

| Уровень | Где | Что кэширует | Пример |
|---------|-----|--------------|--------|
| **Client / browser** | у пользователя | статику, ответы API | HTTP cache, `Cache-Control` |
| **CDN / edge** | ближе к пользователю | статику, медиа, иногда API | CloudFront (модуль 10) |
| **Reverse proxy** | перед приложением | ответы, фрагменты страниц | nginx/Varnish |
| **Application (local / in-process)** | в памяти инстанса | вычисления, hot-объекты | Caffeine, локальная Map |
| **Distributed cache** | отдельный кластер | общий горячий набор | Redis, Memcached |
| **Database cache** | внутри БД | планы, страницы, буферы | buffer pool |

```
client → CDN → reverse-proxy → app(local cache) → distributed cache → DB
   ↑ ближе и дешевле                                          дальше и дороже ↑
```

> **Local vs distributed.** Локальный (in-process) кэш — самый быстрый (без сети), но у каждого инстанса свой → дублирование и риск рассогласования. Распределённый (Redis) — общий для всех инстансов (консистентнее), но платим сетевой хоп. Часто комбинируют (**near cache**): локальный поверх распределённого.

## 3. Паттерны чтения и записи

### Чтение

| Паттерн | Как | Заметка |
|---------|-----|---------|
| **Cache-aside (lazy loading)** | приложение само: смотрит кэш → промах → читает БД → кладёт в кэш | самый частый; кэшируется только то, что реально запросили |
| **Read-through** | кэш сам загружает из БД при промахе (через провайдер) | логика загрузки спрятана в кэш-слой |

### Запись

| Паттерн | Как | Trade-off |
|---------|-----|-----------|
| **Write-through** | пишем в кэш и БД синхронно | кэш всегда свеж; запись медленнее (два хопа) |
| **Write-back (write-behind)** | пишем в кэш, в БД — асинхронно позже | быстрая запись; риск потери при сбое кэша до сброса |
| **Write-around** | пишем только в БД, кэш обновится при следующем чтении | не засоряет кэш редко-читаемыми данными; первое чтение — промах |

```
Cache-aside (типовое чтение):
  GET key → HIT → вернуть
          → MISS → читать БД → положить в кэш (с TTL) → вернуть
```

> **Дефолт — cache-aside + write-around** (или write-through, если важна свежесть). Write-back даёт максимальную скорость записи, но это опасный паттерн: при падении кэша до асинхронного сброса данные теряются — применять только там, где допустима потеря (метрики, счётчики) или с durable-очередью.

---

## 4. Eviction — вытеснение

Память ограничена; когда кэш заполнен, надо что-то удалить.

| Политика | Удаляет | Когда уместна |
|----------|---------|---------------|
| **LRU** (least recently used) | давно не использованное | дефолт; временна́я локальность доступа |
| **LFU** (least frequently used) | редко используемое | есть стабильно «горячие» ключи |
| **FIFO** | самое старое по вставке | простая, редко оптимальна |
| **TTL** (time-to-live) | по истечении срока | ограничение свежести; комбинируется с любой политикой |

> TTL и eviction-политика — **ортогональны и работают вместе**: TTL ограничивает staleness (данные протухают по времени), LRU/LFU решают, что выкинуть при нехватке памяти. Почти всегда у записи есть и TTL, и кэш под давлением вытесняет по LRU.

## 5. Инвалидация — «одна из двух трудных задач»

Как поддерживать согласованность кэша и БД, когда данные меняются?

| Стратегия | Как | Trade-off |
|-----------|-----|-----------|
| **TTL (passive)** | данные живут N секунд, потом перечитываются | просто; окно устаревания = TTL |
| **Событийная (active)** | при изменении в БД — удалить/обновить ключ | свежо; нужна доставка событий, сложнее |
| **Write-through** | запись сразу обновляет кэш | свежо; см. паттерны записи |

```
Окно устаревания (stale window):
  TTL = 60 c  →  данные могут быть устаревшими до 60 секунд
  событийная инвалидация → окно ≈ задержка доставки события
```

> **Инвалидация — источник самых тонких багов.** Удалять ключ обычно надёжнее, чем обновлять (меньше гонок: при гонке «прочитал старое → записал в кэш» обновление может зафиксировать устаревшее значение). Выбор стратегии диктует допустимая staleness: терпим секунды → TTL (просто); нужна свежесть → событийная (сложно, но точно). Часто комбинируют: событийная инвалидация + TTL как страховка от пропущенных событий.

## 6. Метрики кэша: hit rate и 80/20

```
hit rate  = попаданий / всего запросов
miss rate = 1 − hit rate

Запросов в БД после кэша = всего_запросов × miss_rate
```

Правило **80/20 (Парето)**: малая доля данных («горячий набор», hot set) обслуживает бо́льшую часть запросов. Поэтому кэш небольшого hot set даёт высокий hit rate.

> **Hit rate — главная метрика кэша.** При hit rate 90% кэш снимает 90% чтений: БД видит лишь 10% трафика (+ промахи). Рост hit rate с 90% до 95% **уполовинивает** нагрузку на БД (10% → 5%) — поэтому за последние проценты борются. Низкий hit rate означает, что кэш почти бесполезен (платим за память и сложность, а толку нет): надо пересмотреть, что и насколько кэшируем.

## 7. Патологии кэша и защита

| Патология | Что происходит | Защита |
|-----------|----------------|--------|
| **Cache stampede / thundering herd** | популярный ключ истёк → лавина одновременных промахов бьёт в БД | блокировка/lease на пересчёт (один поток грузит), request coalescing, **probabilistic early expiration** (обновлять чуть раньше TTL) |
| **Cache penetration** | запросы несуществующих ключей всегда промахивают и бьют в БД (часто атака) | кэшировать «пусто» (null с коротким TTL), **bloom filter** на существующие ключи |
| **Cache avalanche** | у массы ключей TTL истекает одновременно → разом всё в БД | разброс TTL (**jitter**), прогрев кэша, разные сроки жизни |
| **Hot key** | один ключ настолько горяч, что перегружает узел кэша | реплики горячего ключа, локальный кэш поверх, разбиение ключа |

> Эти четыре — обязательный чек-лист на интервью при упоминании кэша. Stampede лечат тем, что **только один** запрос пересчитывает значение, остальные ждут или отдают старое. Penetration — тем, что «отсутствие» тоже кэшируется. Avalanche — тем, что TTL не выставляют всем одинаковый (добавляют случайный разброс).

---

## 8. Распределённый кэш: топология

- **Шардирование ключей** между узлами — обычно через **consistent hashing** (модуль 14), чтобы добавление/удаление узла переносило мало ключей и не обнуляло весь кэш.
- **Репликация** горячих ключей — для доступности и снятия нагрузки с hot key.
- **Клиентское vs прокси-шардирование** — клиент сам выбирает узел (Memcached-стиль) или есть прокси/кластер-режим (Redis Cluster).
- **Near cache** — локальный кэш в инстансе перед распределённым (быстрее, но риск рассогласования между инстансами).

> Размер распределённого кэша считают по hot set (модуль 06): объём горячих данных × replication factor + накладные. Узлов столько, чтобы hot set помещался в память с запасом, иначе eviction начнёт выкидывать ещё нужное и hit rate упадёт.

## ⚠️ Подводные камни

- **Кэшировать, не определив допустимую staleness.** Сначала «насколько устаревшее терпимо?», потом стратегия.
- **Игнорировать инвалидацию.** TTL «навсегда» или его отсутствие → пользователи видят устаревшее; продумать инвалидацию с самого начала.
- **Write-back без понимания риска.** Асинхронная запись теряет данные при сбое кэша — только для толерантных к потере данных.
- **Не защититься от stampede/penetration/avalanche.** Истечение популярного ключа или поток несуществующих ключей кладут БД.
- **Низкий hit rate, который никто не меряет.** Кэш с низким hit rate — чистые расходы; hit rate надо мониторить.
- **Одинаковый TTL у всех ключей.** Прямой путь к avalanche; добавляйте jitter.
- **Локальный кэш там, где нужна согласованность.** У каждого инстанса своя копия → расхождение; для общих данных — распределённый.
- **Кэшировать всё подряд.** Холодные данные засоряют память и снижают hit rate; кэшируют горячий набор.
