# Задача 04 — Модуль 10: CDN и хранение медиа

## ЗАДАНИЕ
Спроектируйте загрузку файлов через presigned URL (direct upload в объектное хранилище).

## ВХОДНЫЕ ДАННЫЕ
Сервис загрузки фото: сейчас файл идёт через app-сервер (клиент → app → blob), app стал узким местом по egress и памяти при больших файлах. Требование: разгрузить app, сохранить контроль доступа и метаданные, поддержать файлы до 2 ГБ.

## ТРЕБОВАНИЯ
- Опишите поток direct upload через presigned URL по шагам (кто что делает).
- Укажите, какие ограничения накладывает подпись (срок, операция, размер/тип) и зачем.
- Опишите, как app узнаёт о завершении загрузки и где хранятся метаданные.
- Учтите крупные файлы (multipart/chunked) и устойчивость к обрыву.
- Нарисуйте схему потока.

## ФОРМАТ РЕЗУЛЬТАТА
Пошаговый поток + ограничения подписи + механизм уведомления + схема.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Поток: клиент просит у app presigned PUT URL → app выдаёт (с TTL/ограничениями) → клиент льёт файл напрямую в blob → app получает событие/коллбэк → сохраняет метаданные.
- [ ] Подпись ограничивает срок, операцию (PUT), размер/тип — это и безопасность, и защита от злоупотреблений.
- [ ] Метаданные (владелец, размер, ключ, статус) хранятся в БД; сам файл — в blob (не в БД).
- [ ] Для 2 ГБ предусмотрен multipart/chunked upload (части параллельно, докачка при обрыве).
- [ ] Тяжёлый трафик идёт клиент↔blob напрямую, app лишь выдаёт ссылки → узкое место снято.
- [ ] Схема отражает direct path (мимо app) + уведомление о завершении.

## ПОДСКАЗКА
Presigned URL = app выдаёт «пропуск», а байты летят мимо него. App перестаёт быть трубой для файлов: остаётся только выдача подписей и хранение метаданных. Multipart обязателен для крупных файлов: обрыв на 2 ГБ без чанков = заново.
