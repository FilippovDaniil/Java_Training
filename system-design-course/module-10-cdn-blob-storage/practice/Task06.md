# Задача 06 — Модуль 10: CDN и хранение медиа

## ЗАДАНИЕ
Спроектируйте конвейер обработки и доставки медиа (media pipeline) для видео-сервиса. Изобразите схему.

## ВХОДНЫЕ ДАННЫЕ
Пользователи загружают видео (до нескольких ГБ). Нужно: принять оригинал, перекодировать в несколько качеств (480p/720p/1080p), сделать превью, отдавать зрителям по всему миру с низкой latency.

## ТРЕБОВАНИЯ
- Опишите этапы конвейера: upload → store(raw) → trigger → transcode/preview → store(derived) → serve.
- Объясните, почему обработка должна быть асинхронной (через очередь), а не в HTTP-запросе.
- Укажите, что хранится в blob, что в БД, что раздаётся через CDN.
- Учтите крупные файлы (multipart upload) и масштабирование транскодинга (пул воркеров).
- Нарисуйте схему конвейера (текст/ASCII/Mermaid).

## ФОРМАТ РЕЗУЛЬТАТА
Описание этапов + обоснование асинхронности + распределение «blob/БД/CDN» + схема.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Этапы: presigned upload оригинала в blob → событие в очередь → воркеры транскодят/делают превью → производные в blob → раздача через CDN.
- [ ] Обоснована асинхронность: транскодинг занимает секунды-минуты, держать HTTP-запрос нельзя → очередь + воркеры (модуль 15).
- [ ] Распределение: оригинал и производные — в blob; метаданные/статусы — в БД; раздача — через CDN (blob как origin).
- [ ] Учтён multipart upload для крупных файлов и масштабирование пула воркеров под нагрузку.
- [ ] Схема отражает поток от загрузки до раздачи через CDN.

## ПОДСКАЗКА
Канонический конвейер: «залил оригинал напрямую в blob → событие → асинхронная обработка воркерами → производные в blob → CDN зрителям». БД хранит только метаданные и статус обработки. Транскодинг — классический случай очереди (модуль 15): развязывает приём и тяжёлую обработку.
