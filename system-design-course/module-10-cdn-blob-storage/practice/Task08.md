# Задача 08 — Модуль 10: CDN и хранение медиа

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне хранения и доставки медиа и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Картинки и видео храним в реляционной БД в колонке BLOB — так проще транзакции».
2. «Файлы отдаём пользователям напрямую из app-сервера через эндпоинт `/download`».
3. «Загрузку тоже ведём через app: клиент шлёт файл на сервер, сервер кладёт в хранилище».
4. «Транскодинг видео делаем синхронно в обработчике запроса загрузки».
5. «Для обновления логотипа просто перезаписываем `logo.png`, TTL на CDN — 30 дней».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка сильнее всего бьёт по стоимости, а какая — по пользовательскому опыту.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод (стоимость/UX).

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Медиа в реляционной БД раздувает её и ломает бэкапы → файл в blob, в БД метаданные/ссылка.
- [ ] (2) Раздача из app → дорогой egress + узкое место → раздавать через CDN (blob как origin).
- [ ] (3) Загрузка через app → app-труба для байтов → presigned direct upload в blob.
- [ ] (4) Синхронный транскодинг → таймауты/блокировка → асинхронно через очередь и воркеры.
- [ ] (5) Перезапись `logo.png` при TTL 30 дней → пользователи до 30 дней видят старый логотип → versioned URL (`logo.<hash>.png`) или purge.
- [ ] Указано: по стоимости сильнее всего бьют (1)/(2) (egress+БД), по UX — (4)/(5) (таймауты, устаревший контент).

## ПОДСКАЗКА
Канон медиа-дизайна: файл → blob, метаданные → БД, раздача → CDN, загрузка → presigned, обработка → асинхронно, инвалидация статики → versioned URL. Каждый пункт фрагмента нарушает один из этих принципов.
