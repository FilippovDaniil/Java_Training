# Задача 09 — Модуль 10: CDN и хранение медиа

## ЗАДАНИЕ
Выберите стратегию инвалидации контента на CDN (versioned URL vs purge vs TTL) под разные случаи.

## ВХОДНЫЕ ДАННЫЕ
- (А) сборка фронтенда: CSS/JS меняются с каждым релизом, нужны мгновенно и без риска «полустарых» комбинаций;
- (Б) срочный отзыв: на CDN попал файл с ошибкой/утёкшими данными, нужно убрать его немедленно со всех edge;
- (В) лента новостей в JSON: обновляется каждые несколько минут, секундная-минутная задержка терпима.

## ТРЕБОВАНИЯ
- Для каждого случая выберите стратегию и обоснуйте.
- Объясните, почему versioned URL предпочтительнее purge для регулярных обновлений статики.
- Укажите ограничения purge (скорость распространения, лимиты/стоимость API).
- Опишите риск «полустарых» комбинаций ассетов и как versioned URL его решает.

## ФОРМАТ РЕЗУЛЬТАТА
Решение по (А)/(Б)/(В) + сравнение versioned URL vs purge + ограничения purge.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → versioned URL (новый hash в имени = новый URL, долгий TTL, без purge; атомарная смена набора ассетов).
- [ ] (Б) → purge/invalidation (явный срочный сброс на всех edge).
- [ ] (В) → короткий TTL (терпимая задержка, не нужен ни purge, ни версии).
- [ ] Объяснено: versioned URL не требует purge и убирает гонку «полустарых» комбинаций (старые URL просто не запрашиваются).
- [ ] Названы ограничения purge: распространение не мгновенно, есть лимиты/стоимость API → не для рутинных обновлений.

## ПОДСКАЗКА
Versioned URL — для регулярных обновлений (релизы): контент меняется только вместе с именем, поэтому ставят максимальный TTL и не вызывают purge. Purge — аварийный инструмент для срочного отзыва. TTL — для часто меняющейся динамики с терпимой задержкой.
