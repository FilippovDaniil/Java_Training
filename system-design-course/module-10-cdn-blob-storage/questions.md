## Ключ (эталонные ответы)

1. **CDN (Content Delivery Network)** — географически распределённая сеть кэширующих серверов, отдающих контент из точки, ближайшей к пользователю. Задачи: низкая latency, экономия egress, разгрузка origin, поглощение всплесков/DDoS.
2. **Edge / PoP (point of presence)** — кэширующий сервер CDN, размещённый близко к пользователям (во многих городах/регионах). Контент отдаётся с ближайшего edge.
3. Потому что данные приходят с географически близкого edge, а не из далёкого origin-дата-центра — меньше сетевой путь и round-trip time.
4. Дважды: тариф egress через CDN дешевле за единицу, и edge-кэш снимает повторные запросы с origin (origin отдаёт только при промахах — offload ratio). Чем выше offload, тем меньше дорогого origin-egress.
5. **Pull** — edge тянет контент из origin при первом запросе (промах → fetch → кэш), само-наполнение по спросу. **Push** — вы заранее заливаете контент на CDN.
6. Когда контент заранее известен, крупный и нельзя допустить «холодного» первого запроса: релизы, VOD-видео, ожидаемые пики (премьеры). Push гарантирует, что файл уже на edge.
7. Промах edge → edge запрашивает объект из origin, кэширует и отдаёт пользователю. Первый запрос медленнее (идём в origin), последующие — быстрые с edge.
8. **Cache key** — по чему различается контент (обычно URL, иногда + заголовки/query). Неверный ключ → либо отдаём чужой/неподходящий контент, либо не кэшируем вовсе (всё валится в origin).
9. `Cache-Control` задаёт TTL и правила: `max-age`/`s-maxage` — сколько хранить, `public`/`private` — можно ли в shared-кэш, `no-store` — не кэшировать. Так разделяют статику (долго) и динамику/персональное (коротко/никогда).
10. Кэш на shared edge общий для многих пользователей — персональный контент увидят чужие. Персонализированное помечают `private`/`no-store` или кэшируют по пользователю в app/distributed-кэше, не на shared CDN.
11. TTL-истечение (ждём протухания), purge/invalidation (явный сброс на всех edge), versioned URL / cache busting (меняем URL при изменении контента).
12. Versioned URL (`app.<hash>.js`) делает имя уникальным для каждой версии: можно ставить максимальный TTL и не вызывать purge — старый URL просто перестаёт запрашиваться. Это убирает и гонку «полустарых» комбинаций ассетов.
13. Для срочного/аварийного удаления конкретного объекта со всех edge (отозвать ошибочный или утёкший файл). Не для рутинных обновлений: распространение не мгновенно, есть лимиты/стоимость.
14. Хранилище файлов как **объектов** (bucket + key) в плоском пространстве имён, с метаданными, доступом по HTTP-API. Свойства: immutable-объекты, очень высокая durability («11 nines»), дёшево за ГБ, практически безграничный масштаб.
15. **Object/blob** — объекты по HTTP, immutable, плоско (медиа, бэкапы, статика). **Block** — сырые блоки как диск, низкая latency, частые правки (под БД/ОС). **File** — сетевая ФС с POSIX и иерархией, shared-доступ нескольких машин.
16. Большие BLOB раздувают БД, ломают её бэкапы и кэш, дороги и не масштабируются так, как blob. БД оптимизирована под структурированные записи и транзакции, а не под мегабайты бинарей.
17. Файл — в объектное хранилище (blob), а в БД — только метаданные и ссылка (URL/key). Раздача — через CDN. Это масштабируемо, дёшево и не грузит БД.
18. **Presigned URL** — временная подписанная ссылка, дающая клиенту работать с объектным хранилищем напрямую (PUT/GET), минуя app-сервер. Подпись ограничивает срок, операцию, размер/тип.
19. Тяжёлый файловый трафик идёт напрямую клиент↔хранилище, а app лишь выдаёт подписанные ссылки и хранит метаданные. App перестаёт быть «трубой» для байтов — снимается узкое место по egress и памяти.
20. Для крупных файлов: их режут на части, грузят параллельно (быстрее) и могут докачать при обрыве. Без multipart обрыв на больших файлах означает повтор всей загрузки.
21. upload (presigned, multipart) → оригинал в blob → событие в очередь → асинхронная обработка воркерами (transcode/resize/preview) → производные в blob → раздача через CDN. Метаданные/статус — в БД.
22. Транскодинг/обработка занимают секунды-минуты — держать на этом HTTP-запрос нельзя (таймауты, блокировка). Поэтому приём развязывают с обработкой через очередь и пул воркеров (модуль 15).
23. **Storage classes** — уровни хранения с разной ценой/скоростью доступа (hot/cool/archive). **Tiering** — перемещение данных между ними (часто автоматически по lifecycle-политике), чтобы холодные данные хранить дёшево.
24. Архив очень дёшев в хранении, но извлечение дорогое и медленное (минуты-часы) + плата за retrieval. Поэтому в архив переносят только точно «холодные» данные; ошибочный перенос «тёплых» бьёт по latency и стоимости извлечения.
25. Хранение (blob + метаданные в БД, не медиа в БД); загрузка (presigned direct upload, multipart); обработка (асинхронный конвейер через очередь); доставка (CDN, pull/push, TTL, инвалидация versioned URL/purge); tiering под профиль доступа; оценка стоимости (egress-экономия CDN, storage по классам); схема потока.
