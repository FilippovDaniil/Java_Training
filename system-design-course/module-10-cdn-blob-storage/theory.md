# Модуль 10 — CDN и хранение объектов/медиа

## Зачем этот модуль

Кэш (модуль 09) приближает данные к приложению. **CDN** приближает их к самому пользователю — на тысячи серверов по всему миру (edge). А **объектное хранилище** (blob storage, S3-подобное) решает, *где* физически лежат большие файлы — картинки, видео, бэкапы, статика. Эти два блока почти всегда идут в паре: файлы хранят в blob-хранилище, а раздают через CDN.

Зачем это senior'у. Во-первых, **latency**: пользователь в Токио не должен тянуть видео из дата-центра в Вирджинии. Во-вторых, **деньги**: egress (исходящий трафик) — частый главный сюрприз счёта (модуль 07), а CDN режет его в разы. В-третьих, **разгрузка**: CDN снимает с origin-серверов основную массу запросов на статику и медиа, оставляя приложению только динамику. В-четвёртых, **корректный дизайн хранения**: класть медиа в реляционную БД — типичная ошибка; для этого есть blob.

Этот модуль — про то, **как работает CDN** (pull/push, edge, инвалидация), **что такое объектное хранилище** (и чем оно отличается от block/file), и **как собрать конвейер медиа** (upload → store → serve).

---

## 1. CDN: зачем и как работает

**CDN (Content Delivery Network)** — географически распределённая сеть кэширующих серверов (edge / PoP, points of presence), которые отдают контент из точки, ближайшей к пользователю.

| Что даёт CDN | Как |
|--------------|-----|
| **Низкая latency** | контент с ближайшего edge, а не из далёкого origin |
| **Экономия egress** | трафик отдаёт CDN (дешевле за единицу) + кэш снимает повторные запросы с origin |
| **Разгрузка origin** | основная масса запросов на статику не доходит до серверов приложения |
| **Поглощение всплесков/DDoS** | edge-сеть распределяет и поглощает пики |

```
Без CDN:   пользователь (Токио) ─────── origin (Вирджиния)   ← далеко, дорого, грузит origin
С CDN:     пользователь (Токио) ── edge (Токио) ⋯ origin     ← близко; origin дёргается лишь при промахе edge
```

> CDN — это **глобальная балансировка (модуль 08) + кэш (модуль 09) на краю сети**. Первое, что выносят на CDN, — статика и медиа: они одинаковы для всех, легко кэшируются и составляют основную массу байтов и egress.

## 2. Pull vs Push CDN

Как контент попадает на edge:

| | **Pull CDN** | **Push CDN** |
|--|--------------|--------------|
| Как наполняется | edge тянет из origin при первом запросе (промах → fetch → кэш) | вы заранее заливаете контент на CDN |
| Первый запрос | медленный (промах, идём в origin) | быстрый (уже на edge) |
| Управление | простое (CDN сам решает, что хранить) | вы управляете, что и когда залить |
| Когда | большой каталог, нельзя предсказать спрос (популярная статика, картинки) | известный набор, крупные файлы, редко меняются (релизы, видео-VOD) |

> **Pull — дефолт** для большинства систем: проще, само-наполняется по спросу, не хранит лишнее. **Push** выбирают, когда важно избежать «холодного» первого запроса для крупных файлов или когда контент заранее известен. Многие CDN поддерживают оба режима.

## 3. Управление кэшем на edge: TTL, cache key, инвалидация

- **Cache key** — по чему различается контент (обычно URL; иногда + заголовки/query). Неверный ключ → либо отдаём чужое, либо не кэшируем.
- **Cache-Control / TTL** — заголовки задают, сколько edge хранит объект (`max-age`, `s-maxage`, `no-store`, `private`).
- **Static vs dynamic**: статику кэшируют надолго; динамику/персональное — `private`/`no-store` или короткий TTL.

**Инвалидация на CDN — те же «две трудные задачи» (модуль 09):**

| Способ | Как | Trade-off |
|--------|-----|-----------|
| **TTL-истечение** | ждём, пока объект протухнет | просто; окно устаревания = TTL |
| **Purge / invalidation** | явно сбросить объект на всех edge | точно, но медленно/ограниченно (API, может тарифицироваться) |
| **Versioned URL (cache busting)** | менять URL при изменении (`app.v2.js`, hash в имени) | **лучший приём для статики**: новый контент = новый URL, старый просто перестаёт запрашиваться |

> **Versioned URL — главный приём инвалидации статики.** Вместо того чтобы «протухать» `style.css`, публикуют `style.<hash>.css`: при изменении меняется имя, поэтому можно ставить очень долгий TTL и не делать purge вообще. Purge оставляют для срочных правок (например, отозвать утёкший файл).

## 4. Объектное хранилище (blob / S3-подобное)

**Объектное хранилище** хранит файлы как **объекты** в плоском пространстве имён (bucket + key), с метаданными, доступом по HTTP-API.

| Свойство | Описание |
|----------|----------|
| **Immutable-объекты** | объект обычно перезаписывают целиком, не «правят кусок» |
| **Плоское пространство имён** | bucket + key (нет настоящих директорий, «папки» — префиксы) |
| **Metadata** | у объекта есть метаданные (тип, размер, кастомные) |
| **HTTP API** | GET/PUT по URL, легко отдавать в CDN |
| **Durability** | очень высокая («11 nines» — 99.999999999%) за счёт внутренней репликации |
| **Дёшево** | $/ГБ ниже, чем у блочных дисков; storage classes для холодных данных |
| **Масштаб** | практически безграничный, без управления дисками |

### Blob vs Block vs File

| Тип | Что это | Когда |
|-----|---------|-------|
| **Object/Blob** (S3) | объекты по HTTP, immutable, плоско | медиа, бэкапы, статика, логи, data lake, артефакты |
| **Block** (EBS/диск) | сырые блоки, как диск под ОС/БД | БД, файловые системы, низкая latency, частые правки |
| **File** (NFS/EFS) | сетевая файловая система, POSIX, иерархия | shared-доступ нескольких машин к файлам |

> **Не храните медиа в реляционной БД.** Большие BLOB в БД раздувают её, ломают бэкапы и кэш, дороги. Правильный паттерн: **файл — в объектное хранилище, в БД — только метаданные и ссылка (URL/key)**. Это один из самых частых «правильных ответов» на интервью про фото/видео/файлы.

## 5. Presigned URLs — прямая загрузка/скачивание

Гонять файлы через сервер приложения дорого и узко: каждый байт проходит через ваш compute и egress. **Presigned URL** — временная подписанная ссылка, позволяющая клиенту работать с объектным хранилищем **напрямую**, минуя app-сервер.

```
Upload (direct):
  1. клиент → app: «хочу загрузить файл»
  2. app → клиент: presigned PUT URL (с TTL, ограничениями)
  3. клиент → blob storage: PUT файл напрямую (мимо app)
  4. (опц.) blob → app: событие «файл загружен» → запустить обработку

Download (direct):
  app выдаёт presigned GET URL (или раздаём публично через CDN)
```

> Presigned URL **снимает с app-серверов нагрузку и egress** по передаче файлов: тяжёлый трафик идёт клиент↔хранилище напрямую, приложение лишь выдаёт подписанные ссылки и хранит метаданные. Подпись ограничивает доступ (срок, операция, размер) — это и безопасность, и масштабируемость.

## 6. Конвейер медиа (media pipeline)

Типовой путь файла в системе фото/видео:

```
upload (presigned) → blob (raw) → событие → обработка (transcode/resize/
  thumbnails) → blob (производные) → раздача через CDN
```

| Этап | Что происходит |
|------|----------------|
| **Upload** | клиент льёт оригинал напрямую в blob (presigned, multipart для крупных) |
| **Trigger** | хранилище шлёт событие «загружен» (в очередь — модуль 15) |
| **Process** | асинхронно: транскодинг видео в несколько битрейтов, ресайз картинок, превью |
| **Store** | производные версии — обратно в blob |
| **Serve** | раздача через CDN (pull из blob как origin) |

> Обработка медиа — **асинхронная** (через очередь): транскодинг видео секунды-минуты, держать на нём HTTP-запрос нельзя. **Multipart/chunked upload** — для крупных файлов: режут на части, грузят параллельно, докачивают при обрыве.

## 7. Storage classes и tiering

Не все данные одинаково «горячи». Объектные хранилища дают **классы хранения** с разной ценой/скоростью доступа:

| Класс | Доступ | Цена | Для чего |
|-------|--------|------|----------|
| **Hot / Standard** | мгновенный, частый | дороже | активный контент |
| **Cool / Infrequent** | мгновенный, редкий | дешевле + плата за извлечение | редко читаемое |
| **Cold / Archive** | минуты-часы на извлечение | очень дёшево | бэкапы, compliance, старое |

> **Tiering** (часто автоматический по политике жизненного цикла) экономит в разы: старые загрузки, к которым давно не обращались, переезжают в архив. Trade-off: чем дешевле хранение, тем дороже/медленнее извлечение — переносят только то, что точно «холодное» (связь со стоимостью, модуль 07).

## ⚠️ Подводные камни

- **Раздавать медиа из app-сервера.** Дорого по egress и compute, грузит origin; медиа — через CDN.
- **Хранить файлы в реляционной БД.** Раздувает БД, ломает бэкапы; файл → blob, в БД — метаданные/ссылка.
- **Нет стратегии инвалидации CDN.** Правки не доходят до пользователя; используйте versioned URL (для статики) + purge (срочное).
- **Кэшировать персональное/приватное на shared edge.** Чужие увидят чужое; `private`/`no-store` или не на CDN.
- **Грузить файлы через app, а не presigned.** Лишний egress и узкое место; прямая загрузка в blob.
- **Синхронная обработка медиа.** Транскодинг в HTTP-запросе — таймауты; делать асинхронно через очередь.
- **Один TTL/класс хранения на всё.** Холодные данные дорого лежат в hot-классе; нужен tiering.
- **Игнорировать multipart для крупных файлов.** Обрыв на 2 ГБ = заново; режьте на части.
- **Путать pull и push.** Pull прост и само-наполняется; push — для заранее известных крупных файлов.
