# Задача 01 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Подберите тип БД (реляционная или конкретное NoSQL-семейство) под набор кейсов и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Пять подсистем:
- (А) банковские счета и переводы между ними;
- (Б) пользовательские сессии и кэш горячих профилей по id;
- (В) каталог товаров с разнородными атрибутами у разных категорий;
- (Г) граф «друзья/подписки» с запросами «друзья друзей»;
- (Д) телеметрия датчиков IoT: огромный поток записи, чтение по устройству и времени.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите тип/семейство БД и обоснуйте моделью данных и паттерном доступа.
- Для каждого назовите неудачный выбор и почему.
- Сведите в таблицу.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → тип БД → обоснование (модель + доступ) → анти-выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → реляционная (ACID-транзакции, целостность переводов); анти-выбор — eventual-consistent NoSQL.
- [ ] (Б) → key-value (Redis): доступ по ключу, скорость; анти-выбор — реляционная под кэш.
- [ ] (В) → document: гибкая схема под разнородные атрибуты; анти-выбор — жёсткая реляционная схема с десятками NULL-колонок.
- [ ] (Г) → graph: обход связей «друзья друзей»; анти-выбор — реляционная с рекурсивными JOIN.
- [ ] (Д) → wide-column (Cassandra): write-heavy, ключ (устройство, время); анти-выбор — реляционная под поток телеметрии.
- [ ] Каждый выбор увязан с моделью данных и паттерном доступа, а не «по привычке».

## ПОДСКАЗКА
Алгоритм: только по ключу → key-value; самодостаточный документ → document; гигантская запись/time-series → wide-column; главное связи → graph; транзакции/связи/целостность → реляционная. Анти-выбор обычно — натянуть один инструмент на чужую задачу.
