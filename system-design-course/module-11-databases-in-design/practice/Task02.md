# Задача 02 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Обоснуйте выбор SQL vs NoSQL для конкретной системы через пять осей решения.

## ВХОДНЫЕ ДАННЫЕ
Система бронирования отелей: бронь связывает пользователя, номер и даты; нельзя продать один номер дважды (конфликт дат недопустим); сложные запросы (доступность по фильтрам); масштаб — региональный, не гигантский; нужны отчёты.

## ТРЕБОВАНИЯ
- Пройдите по пяти осям: модель данных, паттерны доступа, масштаб, согласованность, транзакции — и для каждой укажите, к чему она склоняет.
- Сделайте вывод (SQL или NoSQL) и обоснуйте.
- Назовите, при каком изменении требований выбор стоило бы пересмотреть.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «ось → к чему склоняет» + итоговый выбор + условие пересмотра.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Разобраны все пять осей применительно к кейсу.
- [ ] Согласованность/транзакции выделены как решающие: «нельзя продать дважды» → нужны ACID-транзакции и сильная согласованность → реляционная.
- [ ] Учтены связи (пользователь/номер/даты) и сложные запросы как аргумент за реляционную.
- [ ] Вывод: реляционная (SQL), обоснованно.
- [ ] Названо условие пересмотра (напр. выход на глобальный масштаб с экстремальной записью → рассмотреть NewSQL/шардирование).

## ПОДСКАЗКА
Двойное бронирование — классический признак потребности в транзакциях и сильной согласованности. Региональный масштаб реляционная держит легко. Дефолт — реляционная; отклоняются от него только при доказанной причине (обычно экстремальный масштаб записи или специфичная модель данных).
