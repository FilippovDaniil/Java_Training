# Задача 04 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Определите, где нужны ACID-гарантии, а где допустим BASE (eventual consistency), для подсистем одной системы.

## ВХОДНЫЕ ДАННЫЕ
Маркетплейс:
- (А) кошелёк продавца и списания/зачисления;
- (Б) счётчик просмотров товара;
- (В) оформление заказа (резерв товара + создание заказа атомарно);
- (Г) лента «недавно просмотренные»;
- (Д) рейтинг/звёзды товара (агрегат по отзывам).

## ТРЕБОВАНИЯ
- Для каждой подсистемы укажите: ACID или BASE и почему.
- Для BASE-подсистем назовите, какая именно несогласованность допустима и на какое время.
- Объясните, чем грозит перепутать (ACID там, где хватило бы BASE, и наоборот).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: подсистема → ACID/BASE → обоснование → (для BASE) допустимая несогласованность.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) кошелёк → ACID (деньги нельзя «в итоге»).
- [ ] (В) заказ+резерв → ACID (атомарность: либо и резерв, и заказ, либо ничего).
- [ ] (Б) счётчик просмотров, (Г) лента, (Д) рейтинг → BASE (eventual: «в итоге» сойдётся, секунды задержки терпимы).
- [ ] Для BASE-подсистем названа конкретная допустимая несогласованность и её окно.
- [ ] Объяснены последствия путаницы: лишний ACID → дорого/не масштабируется; BASE на деньгах → потеря/задвоение средств.

## ПОДСКАЗКА
Вопрос-фильтр: «что будет, если значение временно неверно?». Деньги/заказы — недопустимо (ACID). Счётчики/ленты/рейтинги — «в итоге сойдётся» терпимо (BASE), зато доступно и масштабируемо. Это прямой предвестник CAP (модуль 21).
