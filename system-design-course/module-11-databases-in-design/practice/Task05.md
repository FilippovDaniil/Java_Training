# Задача 05 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Решите, нормализовать или денормализовать модель данных под заданный профиль, и спроектируйте схему.

## ВХОДНЫЕ ДАННЫЕ
Лента постов соцсети. Паттерн: пост показывается с именем и аватаркой автора; чтение ленты — крайне частое (read-heavy, миллионы показов), смена имени/аватарки — редкая. Сейчас данные строго нормализованы: `post(id, author_id, text)` + `user(id, name, avatar)`, при показе — JOIN.

## ТРЕБОВАНИЯ
- Оцените текущую нормализованную схему под профиль (чем плоха на чтении).
- Предложите денормализацию и покажите изменённую схему.
- Опишите цену денормализации: что и как синхронизировать при смене имени/аватарки.
- Сделайте вывод: оправдана ли денормализация для этого профиля.

## ФОРМАТ РЕЗУЛЬТАТА
Оценка → денормализованная схема → стоимость синхронизации → вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Названа проблема нормализованной схемы: JOIN на каждом показе при read-heavy профиле — дорого в масштабе.
- [ ] Предложена денормализация: дублирование `author_name`/`avatar` в записи поста (или в материализованной ленте).
- [ ] Описана цена: при смене имени/аватарки нужно обновить копии (фоновая задача/событие), есть окно несогласованности.
- [ ] Вывод: для read-heavy + редкой записи денормализация оправдана (платим редко при записи, выигрываем на частом чтении).
- [ ] Упомянуто, что в SQL это точечная оптимизация, в NoSQL — норма (нет дешёвых JOIN).

## ПОДСКАЗКА
Денормализация переносит стоимость с частого чтения на редкую запись. Для read-heavy с редкими изменениями это выгодно. Главная цена — синхронизация дублей: обычно через фоновое обновление по событию «профиль изменён», с допустимым окном устаревания.
