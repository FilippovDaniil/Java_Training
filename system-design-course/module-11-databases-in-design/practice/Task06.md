# Задача 06 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Спроектируйте модель данных в NoSQL методом query-driven: от паттернов доступа к структуре.

## ВХОДНЫЕ ДАННЫЕ
Сервис мероприятий на wide-column БД (Cassandra-подобной). Паттерны доступа:
1. показать все мероприятия города на дату (часто);
2. показать детали мероприятия по его id;
3. показать мероприятия, на которые записан пользователь.

## ТРЕБОВАНИЯ
- Объясните, чем query-driven моделирование отличается от реляционного подхода.
- Для каждого паттерна спроектируйте таблицу/ключ так, чтобы запрос обслуживался одной эффективной операцией.
- Объясните, почему здесь допустимо (и нужно) дублирование данных между таблицами.
- Назовите риск моделирования без знания запросов.

## ФОРМАТ РЕЗУЛЬТАТА
Объяснение подхода + таблица «паттерн → ключ (partition/clustering) → структура» + про дублирование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: в NoSQL сначала паттерны доступа, потом модель под них (в отличие от «сначала сущности» в SQL).
- [ ] Для (1): партиция (city, date), кластеризация по времени/названию — одна выборка отдаёт список.
- [ ] Для (2): таблица по event_id.
- [ ] Для (3): таблица по user_id (список мероприятий пользователя).
- [ ] Явно сказано: одни и те же данные дублируются в нескольких таблицах под разные запросы — это норма (нет JOIN).
- [ ] Назван риск: модель без знания запросов → данные нельзя достать эффективно (full scan/scatter-gather).

## ПОДСКАЗКА
В wide-column «как читаем» определяет partition key (что в одной партиции) и clustering key (порядок внутри). Одна таблица = один паттерн доступа; дублирование между таблицами компенсирует отсутствие JOIN. Моделировать «сущности вообще», как в SQL, здесь нельзя.
