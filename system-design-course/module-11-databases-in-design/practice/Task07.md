# Задача 07 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Найдите ошибки в решениях по выбору и использованию БД и переформулируйте их.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Берём MongoDB для всего, потому что она масштабируется и без схемы».
2. «Связи между сущностями делаем JOIN-ами в коде приложения поверх документов».
3. «Деньги храним в той же eventual-consistent NoSQL, что и ленту — меньше зоопарка».
4. «Соцграф (друзья друзей, до 4 уровней) держим в реляционной БД на рекурсивных JOIN».
5. «Таблицы в Cassandra спроектировали по сущностям, запросы придумаем позже».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка опаснее всего (для корректности данных).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод о самой опасной.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) «MongoDB для всего» — выбор от хайпа, не от данных → выбирать по модели/запросам; реляционные данные → реляционная БД.
- [ ] (2) JOIN в коде поверх документов → медленно и хрупко → либо реляционная БД, либо денормализация под запрос.
- [ ] (3) Деньги в eventual-consistent NoSQL → риск потери/задвоения средств → ACID-хранилище для финансов.
- [ ] (4) Глубокий обход в реляционной на рекурсивных JOIN → дорого/не масштабируется → graph DB.
- [ ] (5) Cassandra без знания запросов → данные не достать эффективно → query-driven моделирование заранее.
- [ ] Указана самая опасная для корректности (обычно п.3 — деньги в eventual-consistent).

## ПОДСКАЗКА
Чек-лист модуля: выбор от данных/запросов (не от моды), реляционные данные → реляционная БД, деньги → ACID, обход связей → graph, NoSQL → query-driven. Каждый пункт нарушает один принцип; «деньги в BASE» — нарушение, бьющее по корректности.
