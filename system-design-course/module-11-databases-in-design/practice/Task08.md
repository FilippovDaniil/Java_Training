# Задача 08 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Спроектируйте polyglot persistence для системы: распределите подсистемы по подходящим БД и оцените цену такого зоопарка.

## ВХОДНЫЕ ДАННЫЕ
Платформа онлайн-обучения: пользователи и оплаты; кэш сессий; каталог курсов с разнородными метаданными; полнотекстовый поиск по курсам; лента активности студентов (большой поток); рекомендации «похожие курсы».

## ТРЕБОВАНИЯ
- Распределите каждую подсистему по типу БД с кратким обоснованием.
- Объясните, что лучше оставить в одной реляционной БД, а что действительно требует отдельного хранилища.
- Оцените цену polyglot (TCO: эксплуатация, экспертиза, бэкапы, согласованность между хранилищами).
- Сформулируйте принцип: когда добавлять новое хранилище.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «подсистема → БД → обоснование» + раздел про цену polyglot + принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Пользователи/оплаты → реляционная (ACID); сессии → key-value (Redis); каталог → document/реляционная; поиск → поисковый движок; лента → wide-column; рекомендации → graph.
- [ ] Явно сказано, что часть подсистем можно держать в одной реляционной БД, не плодя хранилища без нужды.
- [ ] Оценена цена polyglot: каждый стор — эксплуатация, экспертиза, мониторинг, бэкапы, согласованность между ними (модуль 07).
- [ ] Сформулирован принцип: добавлять хранилище, когда конкретная подсистема упёрлась в ограничения текущей БД, а не заранее.
- [ ] Решения увязаны с моделью данных и паттернами доступа.

## ПОДСКАЗКА
Polyglot мощен, но каждый новый стор удваивает операционную нагрузку. Правильный путь: стартовать с реляционной, выносить подсистему в специализированное хранилище только когда она доказанно в нём нуждается (масштаб, модель, поиск). «Минимум хранилищ, решающих задачу».
