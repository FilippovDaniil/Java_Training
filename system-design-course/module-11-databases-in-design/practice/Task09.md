# Задача 09 — Модуль 11: Базы данных в дизайне

## ЗАДАНИЕ
Напишите ADR по выбору между реляционной БД и NoSQL (или NewSQL) для ядра системы.

## ВХОДНЫЕ ДАННЫЕ
Сервис заказов растущего маркетплейса. Сейчас всё в одной PostgreSQL. Прогноз: запись по заказам вырастет на порядок, нужны транзакции (заказ+резерв), но появляются опасения по горизонтальному масштабу записи. Рассматриваются: остаться на реляционной + read-реплики + будущее шардирование; перейти на document NoSQL; перейти на NewSQL (CockroachDB/Spanner).

## ТРЕБОВАНИЯ
Оформите ADR:
1. **Контекст** — рост записи, потребность в транзакциях, опасения масштаба.
2. **Варианты** — реляционная (+реплики/шардинг), document NoSQL, NewSQL: плюсы/минусы каждого (транзакции, масштаб записи, сложность, зрелость, TCO).
3. **Решение** — что выбрано и почему.
4. **Последствия** — плюсы, минусы, точка пересмотра.

## ФОРМАТ РЕЗУЛЬТАТА
ADR из 4 разделов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Контекст отражает ключевой конфликт: нужны и ACID-транзакции, и горизонтальный масштаб записи.
- [ ] Все три варианта разобраны: document NoSQL отсекается из-за потребности в транзакциях; реляционная сильна по транзакциям, но горизонтальная запись болезненна; NewSQL даёт и то, и другое ценой сложности/зрелости.
- [ ] Решение обосновано (напр. остаться на реляционной + read-реплики сейчас, шардирование/NewSQL — при достижении порога), увязано с TCO.
- [ ] В последствиях есть минусы выбора и **точка пересмотра** (метрика/объём записи, при котором мигрировать).
- [ ] Решение не «навсегда»: зафиксирован триггер пересмотра.

## ПОДСКАЗКА
Конфликт «транзакции + масштаб записи» — именно та ниша, где появился NewSQL. Но не прыгайте в него преждевременно: реляционная + реплики тянет долго, а NewSQL добавляет сложность. Хороший ADR фиксирует порог, при котором решение пересматривают (модуль 02).
