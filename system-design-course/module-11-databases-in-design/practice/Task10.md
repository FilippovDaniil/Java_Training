# Задача 10 — Модуль 11 (мини-проект): Выбор хранилищ системы

## ЗАДАНИЕ
Спроектируйте выбор хранилищ для целой системы: модель данных, БД на подсистему и обоснование. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) сервис коротких видео (TikTok-подобный); (б) платёжная система; (в) маркетплейс объявлений. Нагрузку/объёмы возьмите из расчётов модулей 05–07 (или задайте явно как допущение).

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Подсистемы** — выделите ключевые подсистемы и их данные.
2. **Модель данных** — для главных сущностей (связи, агрегаты).
3. **Выбор БД на подсистему** — тип/семейство с обоснованием по пяти осям (модель/доступ/масштаб/согласованность/транзакции).
4. **ACID vs BASE** — где какие гарантии и почему.
5. **Нормализация vs денормализация** — где и зачем.
6. **Polyglot** — оправдан ли набор хранилищ; цена (TCO).
7. **Открытые вопросы / точки пересмотра** — что может изменить выбор при росте.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов с таблицей выбора хранилищ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; нагрузка/объёмы заданы (своя оценка или допущение).
- [ ] Выделены подсистемы; для главных сущностей показана модель данных.
- [ ] Для каждой подсистемы выбран тип БД с обоснованием по пяти осям (не «по привычке»).
- [ ] Явно разнесены ACID- и BASE-подсистемы; финансы/критичное — ACID.
- [ ] Указано, где применяется денормализация и какой ценой (синхронизация).
- [ ] Набор хранилищ обоснован (минимум под задачу); оценён TCO polyglot.
- [ ] Есть точки пересмотра при росте (триггеры миграции/шардирования).

## ПОДСКАЗКА
Это синтез модуля и вход в следующие два: выбранную БД дальше надо масштабировать (read-реплики — модуль 12) и шардировать (модуль 13). Хороший выбор хранилищ начинается с реляционного дефолта и отклоняется от него только под доказанную причину, разнося ACID и BASE по подсистемам. Решения опираются на числа Части 2 и на пять осей выбора.
