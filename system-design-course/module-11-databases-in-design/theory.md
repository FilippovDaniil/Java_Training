# Модуль 11 — Базы данных в дизайне (SQL vs NoSQL)

## Зачем этот модуль

Балансировка, кэш и CDN (модули 08–10) распределяют и приближают данные, но рано или поздно запрос доходит до **источника правды** — базы данных. Выбор БД — одно из самых дорогих и трудно-обратимых решений дизайна: он определяет, какие гарантии согласованности вы получите, как система масштабируется, как моделируются данные и какие запросы дёшевы, а какие — нет. Сменить БД в работающей системе — это, как правило, миграция на месяцы.

На интервью «какую БД возьмём и почему?» — обязательный вопрос. Слабый ответ — «PostgreSQL, потому что я его знаю» или «MongoDB, потому что масштабируется». Сильный — выводит выбор из **модели данных, паттернов доступа, требуемой согласованности, масштаба и потребности в транзакциях**.

Этот модуль — карта семейств БД (реляционные и четыре семейства NoSQL), осей выбора (ACID vs BASE, схема, нормализация), подхода query-driven и идеи polyglot persistence. Глубже согласованность разбирается в модулях 12 и 21.

## 1. Реляционные БД (SQL)

| Свойство | Что даёт |
|----------|----------|
| **Таблицы + схема** | строгая структура, типы, ограничения (schema-on-write) |
| **ACID-транзакции** | атомарность, согласованность, изоляция, durability |
| **JOIN** | связывание сущностей по ключам без дублирования |
| **Нормализация** | данные хранятся один раз, целостность через FK |
| **SQL** | мощный декларативный язык запросов |
| **Сильная согласованность** | чтение видит последнюю запись |

**Масштаб:** прежде всего вертикальный (мощнее сервер) + read-реплики для чтений; горизонтальная запись требует шардирования (модуль 13), которое реляционным БД даётся тяжелее.

> Реляционная БД — **дефолт для большинства систем**: транзакции, связи, целостность, сложные запросы «из коробки», зрелые инструменты. Берите её, если нет конкретной причины не брать. NoSQL выбирают под конкретную проблему (масштаб записи, модель данных, гибкая схема), а не «вместо устаревшего SQL» — это миф.

---

## 2. Семейства NoSQL

«NoSQL» — это не одна вещь, а четыре разных семейства под разные задачи:

| Семейство | Модель | Сильная сторона | Примеры | Когда |
|-----------|--------|-----------------|---------|-------|
| **Key-Value** | ключ → значение | сверхбыстрый доступ по ключу, простой масштаб | Redis, DynamoDB, Memcached | кэш, сессии, счётчики, профили по id |
| **Document** | JSON/BSON-документы | гибкая схема, агрегат целиком | MongoDB, Couchbase | контент, каталоги, когда «сущность = документ» |
| **Wide-column** | строки с динамическими колонками, партиции | колоссальная запись и масштаб | Cassandra, HBase, ScyllaDB | time-series, логи, write-heavy, IoT |
| **Graph** | узлы + рёбра как первоклассные | обход связей, многоуровневые связи | Neo4j, JanusGraph | соцграф, рекомендации, fraud, графы знаний |

```
Доступ только по ключу?         → key-value
Данные = самодостаточный документ? → document
Гигантская запись / time-series?   → wide-column
Главное — связи и их обход?        → graph
Связи + транзакции + сложные запросы? → реляционная
```

> Самая частая ошибка — взять document-БД (MongoDB) как «масштабируемую SQL» для данных с богатыми связями, а потом делать JOIN в коде приложения. Если данные реляционны — берите реляционную. NoSQL силён там, где модель данных и паттерн доступа точно совпадают с семейством.

## 3. ACID vs BASE

| ACID (обычно SQL) | BASE (обычно NoSQL) |
|-------------------|---------------------|
| **A**tomicity | **BA**sically **A**vailable |
| **C**onsistency | **S**oft state |
| **I**solation | **E**ventually consistent |
| **D**urability | |
| строгие гарантии, сильная согласованность | доступность и масштаб ценой временной несогласованности |

> **ACID vs BASE — это про компромисс согласованности и доступности/масштаба** (предвестник CAP, модуль 21). Финансовая транзакция требует ACID: нельзя «в итоге» списать деньги. Лента лайков терпит BASE: счётчик «в итоге» сойдётся, зато система доступна и масштабируема. Многие современные БД дают настраиваемую согласованность — выбор не всегда бинарен.

---

## 4. Схема: on-write vs on-read

| | **Schema-on-write** (SQL) | **Schema-on-read** (NoSQL гибкая) |
|--|---------------------------|-----------------------------------|
| Когда проверяется структура | при записи (БД отвергает несоответствие) | при чтении (приложение интерпретирует) |
| Плюс | целостность, предсказуемость | гибкость, быстрая эволюция, разнородные данные |
| Минус | миграции при изменении схемы | риск «мусора», ответственность на приложении |

> «NoSQL = нет схемы» — миф: схема есть, она просто **неявная** и живёт в коде приложения (schema-on-read). Гибкость удобна на раннем этапе и для разнородных данных, но её цена — отсутствие гарантий целостности на уровне БД: за консистентность данных отвечает каждый разработчик.

---

## 5. Нормализация vs денормализация

- **Нормализация** (SQL-идиома): каждый факт хранится один раз, связи через ключи. Плюс — нет аномалий обновления, целостность. Минус — чтение требует JOIN.
- **Денормализация** (частая в NoSQL): данные дублируются и хранятся «готовыми к чтению». Плюс — быстрое чтение без JOIN. Минус — дублирование, надо синхронизировать копии при изменении.

```
Нормализовано:  user(id,name) + order(id,user_id) → JOIN при чтении
Денормализовано: order(id, user_name, ...) → читаем одной операцией, но при смене имени правим везде
```

> Выбор диктуется профилем: **read-heavy** склоняет к денормализации (платим при редкой записи, выигрываем на частом чтении), **write-heavy** и важность целостности — к нормализации. В NoSQL денормализация — норма, потому что нет дешёвых JOIN; в SQL её применяют точечно как оптимизацию.

---

## 6. Query-driven design (моделирование под запросы)

В реляционной БД сначала моделируют сущности (нормализация), а запросы пишут потом — JOIN и индексы справятся. В NoSQL — **наоборот**: сначала перечисляют паттерны доступа, потом проектируют модель так, чтобы каждый паттерн обслуживался одной эффективной операцией.

```
SQL:    модель сущностей → потом любые запросы (JOIN/индексы)
NoSQL:  сначала паттерны доступа → модель/ключи под них (часто дублирование)
```

> В wide-column и key-value «как мы будем читать» определяет shard/partition key и структуру записи **до** того, как что-то записано. Спроектировать таблицу Cassandra, не зная запросов, — почти гарантированный провал (данные нельзя будет достать эффективно). Это ключевая смена мышления при переходе на NoSQL.

## 7. Polyglot persistence и NewSQL

**Polyglot persistence** — в одной системе разные подсистемы используют разные БД, каждую под свою задачу:

```
заказы/платежи → PostgreSQL (ACID)
сессии/кэш     → Redis (key-value)
каталог/поиск  → document / поисковый движок
лента событий  → Cassandra (wide-column)
соцграф        → graph DB
```

**NewSQL** (Google Spanner, CockroachDB, YugabyteDB) — попытка совместить несовместимое: SQL и ACID-транзакции **с** горизонтальным масштабом и распределённостью. Цена — сложность и (иногда) latency распределённых транзакций.

> Polyglot мощен, но не бесплатен: каждая БД — это эксплуатация, экспертиза, бэкапы, мониторинг (TCO, модуль 07). Не плодите хранилища без нужды. NewSQL уместен, когда реально нужны и реляционные гарантии, и горизонтальный масштаб записи одновременно (раньше — взаимоисключающие).

## 8. Как выбирать БД — оси решения

| Ось | Вопрос |
|-----|--------|
| **Модель данных** | связи (граф/реляц.), документы, простые пары, широкие строки? |
| **Паттерны доступа** | по ключу / range / сложные запросы / обход связей? read vs write heavy? |
| **Масштаб** | влезает в один узел? нужен горизонтальный масштаб записи? |
| **Согласованность** | нужна сильная (ACID) или терпима eventual (BASE)? |
| **Транзакции** | нужны многооператорные ACID-транзакции? |

> Идите от данных и запросов, а не от модного имени. Большинство систем стартуют на реляционной БД и добавляют специализированные хранилища точечно, когда конкретная подсистема упирается в её ограничения. «Сначала измерь — потом выбирай» (Часть 2) применимо и здесь.

## ⚠️ Подводные камни

- **«NoSQL, потому что масштабируется/модно».** Выбор — из модели данных и запросов, а не из хайпа.
- **Document-БД для реляционных данных.** Если данные связаны — JOIN в коде хуже реляционной БД.
- **Миф «NoSQL = нет схемы».** Схема есть, она в приложении (schema-on-read); целостность — ваша забота.
- **Моделировать NoSQL без знания запросов.** Wide-column/KV требуют query-driven дизайна заранее.
- **Реляционная БД для глубокого обхода графа.** Многоуровневые JOIN дороги; для связей — graph DB.
- **Денормализация без нужды.** Дублирование требует синхронизации; делайте под доказанный read-профиль.
- **ACID там, где хватило бы BASE (и наоборот).** Финансы → ACID; лайки/счётчики → можно eventual.
- **Polyglot ради polyglot.** Каждая БД — отдельный TCO; добавляйте хранилище под реальную проблему.
- **Считать выбор БД легко-обратимым.** Миграция БД в проде — дорого и долго; решайте осознанно.
