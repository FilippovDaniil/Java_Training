# Задача 01 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Выберите топологию репликации (single-leader / multi-leader / leaderless) под кейсы и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) основная БД заказов: запись умеренная, нужна простота и отсутствие конфликтов, чтений много;
- (Б) система в 3 регионах, где пользователи должны писать в локальный регион с низкой latency;
- (В) хранилище корзин/сессий с приоритетом на доступность и устойчивость к отказам узлов.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите топологию и обоснуйте.
- Назовите главный минус выбранной топологии для этого кейса.
- Для каждого укажите неудачную альтернативу и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → топология → обоснование → главный минус → анти-выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → single-leader (просто, нет конфликтов записи, чтения с реплик); минус — лидер как узкое место/точка отказа (нужен failover).
- [ ] (Б) → multi-leader (локальная запись по регионам); минус — конфликты записи между регионами.
- [ ] (В) → leaderless (высокая доступность, кворумы); минус — согласованность сложнее (eventual без настройки кворума).
- [ ] Для каждого назван анти-выбор (напр. multi-leader для (А) добавляет конфликты без нужды).
- [ ] Выбор увязан с приоритетом (простота/локальная запись/доступность).

## ПОДСКАЗКА
Single-leader — дефолт (прост, без конфликтов). Multi-leader — только ради локальной записи в нескольких регионах (ценой конфликтов). Leaderless — ради максимальной доступности с настраиваемой согласованностью через кворумы.
