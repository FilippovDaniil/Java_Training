# Задача 02 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Выберите режим репликации (sync / async / полусинхронный) под требования и обоснуйте через trade-off.

## ВХОДНЫЕ ДАННЫЕ
- (А) финансовые записи: потеря подтверждённой записи недопустима даже при отказе лидера;
- (Б) лента активности: важна низкая latency записи, единичная потеря при сбое терпима;
- (В) система, где нужен баланс: не терять данные, но и не блокировать запись, если одна реплика тормозит.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите режим и обоснуйте через durability/latency/доступность записи.
- Объясните, почему чистый sync ко всем репликам непрактичен.
- Для (А) укажите, что произойдёт с подтверждённой записью при отказе лидера в выбранном режиме.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → режим → обоснование (durability/latency/доступность) + абзац про непрактичность full-sync.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → sync/полусинхронный (минимум одна реплика подтвердила) — подтверждённая запись переживает отказ лидера.
- [ ] (Б) → async (низкая latency, единичная потеря терпима).
- [ ] (В) → полусинхронный (ждём одну реплику для durability, остальные async для скорости).
- [ ] Объяснено: full-sync ко всем — любая медленная реплика тормозит, любая упавшая блокирует запись.
- [ ] Для (А) явно: при sync/полусинхронном подтверждённая запись уже на реплике → не теряется при failover.
- [ ] Trade-off durability↔latency↔доступность проговорён.

## ПОДСКАЗКА
Async быстр, но при отказе лидера незареплицированное теряется. Sync надёжен, но full-sync ко всем непрактичен. Полусинхронный (ждём ≥1 реплику) — частый компромисс: durability без блокировки на медленных репликах.
