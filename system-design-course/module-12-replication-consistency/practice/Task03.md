# Задача 03 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Диагностируйте симптом, вызванный replication lag, и предложите решение.

## ВХОДНЫЕ ДАННЫЕ
Жалоба пользователей: «Изменил имя в профиле, нажал сохранить — страница перезагрузилась, имя старое. Обновил ещё раз — новое». Архитектура: запись в лидера, чтение распределяется по async read-репликам.

## ТРЕБОВАНИЯ
- Объясните механику бага через replication lag (что именно произошло по шагам).
- Назовите, какая гарантия нарушена (read-your-writes / monotonic reads).
- Предложите минимум 2 решения и их trade-off.
- Объясните, почему это не «баг БД», а следствие async-репликации.

## ФОРМАТ РЕЗУЛЬТАТА
Механика по шагам → нарушенная гарантия → 2+ решения с trade-off → вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Механика: запись ушла в лидера → следующее чтение попало на отстающую реплику (lag) → вернула старое значение.
- [ ] Нарушена read-your-writes (пользователь не видит собственную запись); «прыжок» при повторе — отсутствие monotonic reads.
- [ ] Решения: читать свои свежие данные с лидера; либо с реплики, догнавшей до момента записи; либо «прилепить» сессию к одной реплике (sticky read).
- [ ] Названы trade-off (нагрузка на лидера vs сложность отслеживания позиции репликации).
- [ ] Вывод: это ожидаемое следствие async-репликации, требующее проектирования гарантии, а не дефект.

## ПОДСКАЗКА
Классика read-your-writes: собственная свежая запись «теряется» при чтении с отстающей реплики. Лечение — собственные недавние записи читать с лидера (или с реплики, гарантированно догнавшей). Привязка к одной реплике лечит ещё и «прыжки» (monotonic reads).
