# Задача 04 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Обеспечьте гарантии read-your-writes и monotonic reads для пользовательского сценария, не отказываясь от read-реплик.

## ВХОДНЫЕ ДАННЫЕ
Соцсеть: пользователь публикует комментарий и сразу видит ленту обсуждения; чтения масштабируются async read-репликами. Требование: автор всегда видит свой только что добавленный комментарий и не наблюдает «исчезновения» данных при последующих чтениях.

## ТРЕБОВАНИЯ
- Опишите, как обеспечить read-your-writes для автора (что и откуда читать после записи).
- Опишите, как обеспечить monotonic reads (отсутствие «отката» при следующих чтениях).
- Сохраните масштабирование чтений для прочих пользователей (не «всё с лидера»).
- Укажите цену каждого приёма.

## ФОРМАТ РЕЗУЛЬТАТА
Решение для read-your-writes + решение для monotonic reads + как сохранить масштаб чтений + цена.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Read-your-writes: после записи читать собственные свежие данные с лидера или с реплики, догнавшей до позиции записи (например, в течение N секунд после публикации).
- [ ] Monotonic reads: привязать сессию пользователя к одной реплике (sticky), чтобы не «скакать» между разной свежестью.
- [ ] Масштаб сохранён: с лидера читаются только собственные свежие данные автора; остальные чтения — по репликам.
- [ ] Названа цена: доп. нагрузка на лидера для свежих чтений; сложность маршрутизации/отслеживания позиции.
- [ ] Гарантии применяются точечно (для автора/свежих данных), а не «strong для всех».

## ПОДСКАЗКА
Не нужно делать всё strong. Достаточно: «свои свежие записи — с лидера/догнавшей реплики» (read-your-writes) и «один пользователь — одна реплика на сессию» (monotonic reads). Остальные чтения спокойно масштабируются репликами.
