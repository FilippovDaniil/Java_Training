# Задача 07 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Выберите стратегию разрешения конфликтов для multi-leader/leaderless под разные типы данных.

## ВХОДНЫЕ ДАННЫЕ
Multi-leader система (2 региона), один ключ может быть изменён в обоих одновременно. Типы данных:
- (А) счётчик «количество товара на складе» (инкременты/декременты);
- (Б) поле «последний выбранный язык интерфейса»;
- (В) корзина покупок (набор товаров, добавления из разных устройств).

## ТРЕБОВАНИЯ
- Для каждого типа выберите стратегию (LWW / vector clocks / CRDT / application merge) и обоснуйте.
- Объясните, почему LWW недопустим для (А) и (В).
- Назовите общий принцип: как лучше всего «решить» проблему конфликтов.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: тип данных → стратегия → обоснование + вывод про лучший способ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) счётчик → CRDT-счётчик (или merge сложением), НЕ LWW (LWW потеряет один из инкрементов → неверный остаток).
- [ ] (Б) язык → LWW приемлем (потеря старого выбора некритична).
- [ ] (В) корзина → merge/объединение множеств (как исторически у Amazon), НЕ LWW (иначе товары пропадают).
- [ ] Объяснено, почему LWW молча теряет данные и зависит от синхронности часов.
- [ ] Вывод: лучший способ — не создавать конфликт (single-leader), а при multi-leader выбирать слияние под семантику данных.

## ПОДСКАЗКА
LWW годится, только когда терять одну из версий не страшно (язык, тема). Для счётчиков и наборов он недопустим — теряет обновления. CRDT сливаются корректно по построению. И главное: конфликта лучше избегать (single-leader его не создаёт вовсе).
