# Задача 08 — Модуль 12: Репликация и согласованность

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне репликации и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Чтобы масштабировать запись, добавим больше read-реплик».
2. «Все критичные чтения (баланс, статус платежа) распределяем по async-репликам — так быстрее».
3. «Репликация полностью синхронная ко всем 5 репликам — для надёжности».
4. «При отказе лидера любая реплика сразу объявляет себя новым лидером».
5. «Конфликты в multi-leader разрешаем last-write-wins для всех данных, включая счётчики остатков».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какие ошибки бьют по корректности, а какие — по доступности/производительности.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + классификация (корректность/доступность/перф).

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Read-реплики не масштабируют запись (запись идёт в лидера) → для масштаба записи нужно шардирование (модуль 13).
- [ ] (2) Критичные чтения с async-реплик → видят устаревшее (lag) → читать с лидера/sync-реплики.
- [ ] (3) Full-sync ко всем 5 → медленно и хрупко (медленная/упавшая реплика блокирует) → полусинхронная.
- [ ] (4) Любая реплика — лидер без кворума → split-brain → кворум выборов + фенсинг (модуль 22).
- [ ] (5) LWW для счётчиков остатков → потеря инкрементов, неверный остаток → CRDT/merge сложением.
- [ ] Классифицировано: (2)/(5) — корректность; (3) — доступность/перф; (1) — непонимание масштаба; (4) — корректность+доступность.

## ПОДСКАЗКА
Ключевые ловушки модуля: репликация ≠ масштаб записи; критичное — с лидера; full-sync непрактичен; failover без кворума = split-brain; LWW на счётчиках теряет данные. Каждый пункт нарушает один из этих принципов.
