## Ключ (эталонные ответы)

1. Держать несколько копий данных. Цели: **доступность** (упал узел — копия обслуживает), **масштаб чтений** (read-реплики), **гео-близость** (читать локально), **durability** (данные не теряются с одним диском).
2. Нет. Запись всё равно идёт в источник правды (лидера). Репликация масштабирует чтения и даёт доступность; масштаб записи — это шардирование (модуль 13).
3. **Single-leader** (вся запись в лидера, реплики копируют), **multi-leader** (несколько лидеров, обычно по регионам), **leaderless** (запись/чтение в несколько узлов с кворумами, Dynamo-style).
4. По умолчанию: просто, нет конфликтов записи, достижима сильная согласованность. Дефолт для PostgreSQL/MySQL/MongoDB replica set. Минус — лидер как узкое место записи и точка отказа (нужен failover).
5. Когда нужна локальная запись в нескольких регионах с низкой latency (multi-DC). Главный минус — **конфликты записи**: один ключ может быть изменён в двух лидерах одновременно.
6. Репликация без выделенного лидера: клиент пишет в несколько узлов и читает из нескольких, согласованность настраивается кворумами (W/R/N). Высокая доступность ценой сложности (Cassandra, DynamoDB).
7. **Синхронная** — клиент получает подтверждение после записи на реплику(и) (высокая durability, выше latency). **Асинхронная** — подтверждение сразу после лидера (низкая latency, но lag и риск потери незареплицированного при отказе лидера).
8. Любая медленная реплика тормозит каждую запись, а любая упавшая — блокирует запись вовсе. Поэтому ждать подтверждения от всех реплик непрактично.
9. Компромисс: ждать подтверждения хотя бы одной реплики (durability), остальные реплицировать асинхронно (скорость). Подтверждённая запись переживает отказ лидера, но не блокируется на медленных репликах.
10. Отставание реплики от лидера во времени (от миллисекунд до секунд под нагрузкой) при async-репликации. Чтение с отстающей реплики возвращает устаревшие данные.
11. Гарантия, что пользователь видит результат собственной записи. Нарушается, когда сразу после записи в лидера чтение попадает на отстающую async-реплику — пользователь «не видит» своё изменение.
12. Гарантия, что последующие чтения не возвращают данные «старее» уже прочитанных (нет отката во времени). Нарушается при «скачках» между репликами с разным lag.
13. Собственные свежие записи читать с лидера (или с реплики, догнавшей до позиции записи), а прочие чтения масштабировать репликами. Для monotonic reads — привязать сессию к одной реплике (sticky).
14. Уровни — от strong (чтение всегда видит последнюю запись) через read-your-writes/monotonic к eventual (сойдётся «в итоге»). Это спектр, потому что согласованность торгуется с latency/доступностью — выбирают уровень под данные.
15. **Кворум** — минимальное число узлов для записи (W) и чтения (R) из N реплик. Условие **W+R>N** гарантирует пересечение читающего и писавшего множеств → чтение видит последнюю запись.
16. При N=3: W+R=4>3 → чтение пересекается с записью (видит свежее) и система переживает отказ одного узла и на запись, и на чтение. Классический сбалансированный выбор.
17. Читающее множество может не пересечься с писавшим → чтение может не увидеть последнюю запись (eventual consistency). Например, W=1, R=1 при N=3.
18. **Failover** — повышение реплики до нового лидера при отказе текущего. Опасности: потеря незареплицированных данных (async), split-brain (два лидера), ложный failover (сеть мигнула).
19. **Split-brain** — два узла одновременно считают себя лидером и оба принимают запись → расхождение данных. Защита: кворум на выборы (лидер легитимен лишь при согласии большинства) + фенсинг (старого лидера изолируют/глушат).
20. Использовать синхронную/полусинхронную репликацию для критичных записей, чтобы подтверждённая запись уже была на реплике до подтверждения клиенту — тогда она переживёт отказ лидера.
21. Потому что нельзя локально/«на глаз» решать, кто лидер: при разделении сети это ведёт к split-brain. Нужно согласие большинства (консенсус, модуль 22), гарантирующее единственного легитимного лидера.
22. **LWW** (побеждает запись с большим timestamp), **vector clocks** (отслеживают причинность, выявляют конфликты), **CRDT** (сливаются без конфликтов по построению), **application merge** (приложение решает само).
23. Он молча теряет одну из одновременных записей и зависит от синхронности часов. Для счётчиков, корзин, наборов это недопустимо (теряются инкременты/элементы). Годится лишь там, где потеря версии некритична.
24. Не создавать конфликт: single-leader-репликация не имеет конфликтов записи вовсе. Если multi-leader неизбежен (multi-DC), выбирать стратегию слияния под семантику данных (CRDT/merge), а не LWW для всего.
25. Топология и обоснование; режим sync/async; уровни согласованности по данным (strong/eventual/read-your-writes); число read-реплик и ограничения; безопасный failover (защита от split-brain и потери данных); стратегия конфликтов (если multi-leader/leaderless); перечень закрытых аномалий lag.
