# Задача 03 — Модуль 13: Шардирование и партиционирование

## ЗАДАНИЕ
Диагностируйте hot shard (celebrity problem) и предложите смягчение.

## ВХОДНЫЕ ДАННЫЕ
Соцсеть шардирует данные по `user_id`. В целом распределение ровное, но несколько аккаунтов-знаменитостей (десятки миллионов подписчиков, шквал записей/чтений их активности) перегружают шарды, где они находятся: latency на этих шардах растёт, остальные простаивают.

## ТРЕБОВАНИЯ
- Объясните, почему равномерный по кардинальности ключ `user_id` всё равно даёт hotspot.
- Предложите минимум 3 способа смягчения и опишите каждый.
- Укажите trade-off выбранного основного способа.
- Объясните, почему простое «добавить шардов» не решает эту проблему.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз → 3+ способа смягчения → основной способ с trade-off → почему «больше шардов» не помогает.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: кардинальность высокая, но нагрузка на ключ распределена неравномерно («звезда» = один ключ с огромным трафиком).
- [ ] Способы: разбить «звезду» (composite key `celebrity_id + bucket`), вынести знаменитостей в отдельный кластер/шарды, агрессивно кэшировать их данные (модуль 09), реплицировать горячий шард для чтений.
- [ ] Указан trade-off основного способа (напр. composite key усложняет чтение/сборку; кэш помогает чтениям, но не записи).
- [ ] Объяснено: «больше шардов» не помогает, т.к. горячий — один ключ, и он всё равно попадёт в один шард.
- [ ] Решение направлено именно в горячий ключ, а не размазывает всё.

## ПОДСКАЗКА
Celebrity problem — нагрузка концентрируется на одном ключе, а не распределяется. Добавление шардов не делит один ключ. Лечат разбиением самого ключа (бакетирование), выносом «звёзд» и кэшированием/репликацией их горячих данных.
