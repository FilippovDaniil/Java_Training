# Задача 04 — Модуль 13: Шардирование и партиционирование

## ЗАДАНИЕ
Сравните range- и hash-шардирование для конкретного набора запросов и выберите.

## ВХОДНЫЕ ДАННЫЕ
Система логов. Запросы:
- (1) «все логи за последний час» (очень часто, мониторинг);
- (2) «логи конкретного сервиса за период»;
- (3) запись — постоянный поток с текущим timestamp.
Кандидаты: range по `timestamp`; hash по `service_id`.

## ТРЕБОВАНИЯ
- Для каждого кандидата оцените, как обслуживаются запросы (1)(2) и запись (3).
- Выявите ключевой конфликт: range удобен для (1), но что с записью (3)?
- Предложите решение, совмещающее достоинства (например, композитный ключ).
- Сделайте обоснованный выбор.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «кандидат → запрос (1)/(2)/запись (3)» + выявленный конфликт + решение + выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Range по timestamp: (1) дёшев (range), но запись (3) — все свежие логи летят в один «последний» шард → hotspot записи.
- [ ] Hash по service_id: запись равномерна, (2) по сервису — ок, но (1) «за час» → scatter-gather по всем шардам.
- [ ] Выявлен конфликт: range убивает запись hotspot'ом, hash убивает запрос «за час».
- [ ] Предложен композит (напр. `service_id` как партиция + `timestamp` как clustering, или time-bucket + hash-префикс) для баланса.
- [ ] Сделан обоснованный выбор под доминирующий паттерн; компромисс назван явно.

## ПОДСКАЗКА
Это каноничный конфликт range vs hash. Чистый range по времени = hotspot записи на «сейчас». Чистый hash = дорогой запрос «за период по всем». Решение — композитный ключ: партиционировать по сущности/бакету, сортировать по времени внутри (как в wide-column, модуль 11).
