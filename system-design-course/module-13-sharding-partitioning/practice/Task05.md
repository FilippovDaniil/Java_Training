# Задача 05 — Модуль 13: Шардирование и партиционирование

## ЗАДАНИЕ
Обработайте cross-shard запрос и cross-shard транзакцию: предложите, как избежать или выполнить их корректно.

## ВХОДНЫЕ ДАННЫЕ
E-commerce, данные шардированы по `user_id`. Два проблемных места:
- (А) запрос «топ-100 самых продаваемых товаров по всей платформе» — данные о продажах разбросаны по всем шардам;
- (Б) операция «перевод бонусных баллов от пользователя X пользователю Y» — X и Y в разных шардах, нужна атомарность.

## ТРЕБОВАНИЯ
- Для (А) объясните проблему scatter-gather и предложите решение (не «запрос вживую по всем шардам»).
- Для (Б) объясните, почему cross-shard 2PC нежелателен, и предложите альтернативу.
- Сформулируйте общий принцип проектирования границ шардов относительно запросов/транзакций.

## ФОРМАТ РЕЗУЛЬТАТА
Решение (А) + решение (Б) + общий принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А): scatter-gather по всем шардам дорог и медленный (latency = худший шард) → предрасчёт/материализованный рейтинг (агрегация офлайн/в аналитическом хранилище, модуль 18), а не live-запрос по всем.
- [ ] (Б): cross-shard 2PC медленный, хрупкий, снижает доступность → Saga/eventual с идемпотентностью и компенсацией (модуль 23), либо вынести «кошельки баллов» в отдельный сервис с собственной транзакционностью.
- [ ] Сформулирован принцип: проектировать границы так, чтобы частые запросы и транзакции жили в одном шарде; cross-shard — исключение, обслуживаемое отдельно.
- [ ] Явно сказано: избегать cross-shard как нормы, а не «оптимизировать» её.

## ПОДСКАЗКА
Cross-shard — это то, чего избегают дизайном, а не то, что оптимизируют. Глобальные агрегации считают офлайн (аналитика/предрасчёт). Межшардовые «транзакции» делают через Saga/eventual (модуль 23), а не 2PC. Границы шардов выбирают под транзакционные единицы.
