# Задача 06 — Модуль 13: Шардирование и партиционирование

## ЗАДАНИЕ
Спроектируйте план resharding (роста числа шардов) и объясните, почему наивный подход недопустим.

## ВХОДНЫЕ ДАННЫЕ
Система шардирована как `shard = hash(user_id) mod N`, сейчас N = 4. Данные выросли, нужно перейти к 8 шардам без длительного простоя.

## ТРЕБОВАНИЯ
- Объясните, что произойдёт с распределением ключей при смене N с 4 на 8 в схеме `mod N`.
- Предложите стратегию, минимизирующую перемещение данных (consistent hashing и/или pre-sharding).
- Опишите процесс безопасного rebalancing (как переселять данные без простоя и без потери записей).
- Оцените, какую долю ключей придётся перемещать в наивном и в предложенном подходе.

## ФОРМАТ РЕЗУЛЬТАТА
Анализ `mod N` → стратегия → процесс rebalancing → сравнение доли перемещения.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Показано: при `mod N` смена N перетасовывает почти все ключи (≈ доля, остающаяся на месте, мала) → массовый перелив, фактически простой.
- [ ] Предложен consistent hashing (модуль 14): добавление шарда переносит лишь ~1/N долю ключей; и/или pre-sharding (много логических шардов поверх немногих физических, переселяем логические).
- [ ] Процесс rebalancing: двойная запись/копирование + переключение чтения по мере готовности, без потери новых записей.
- [ ] Дано сравнение: наивный `mod N` ≈ перемещение почти всего; consistent hashing ≈ малая доля.
- [ ] Явно: resharding надо закладывать заранее, а не изобретать в момент кризиса.

## ПОДСКАЗКА
`hash % N` коварен: меняешь N — меняется остаток почти у всех ключей. Consistent hashing переносит лишь долю, приходящуюся на новый узел. Pre-sharding (например, 1024 логических шарда на 4 физических) делает рост дешёвым — переселяешь логические шарды, не пересчитывая ключи.
