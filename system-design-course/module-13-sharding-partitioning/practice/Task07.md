# Задача 07 — Модуль 13: Шардирование и партиционирование

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне шардирования и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Шардируем по `country`, чтобы данные были рядом с пользователями».
2. «Используем `shard = hash(id) mod N`, при росте просто увеличим N».
3. «Каждый шард — один узел без реплик, экономим».
4. «Отчёты по всей системе делаем live-запросом во все шарды на каждый показ дашборда».
5. «Шардируем с самого старта на 64 шарда — масштаб должен быть заложен сразу».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) `country` → низкая кардинальность + страны-гиганты → hot shard → ключ высокой кардинальности (или geo + sub-sharding).
- [ ] (2) `mod N` → resharding перетасует почти всё → consistent hashing / pre-sharding.
- [ ] (3) Шард без реплик → шард = SPOF, отказ теряет часть данных → реплицировать каждый шард (модуль 12).
- [ ] (4) Live scatter-gather на каждый дашборд → дорого/медленно → предрасчёт/аналитическое хранилище (модуль 18).
- [ ] (5) Шардирование с нуля на 64 → преждевременная сложность → сначала вертикальный масштаб/реплики/кэш, шардировать под доказанную нужду.
- [ ] Классифицировано: (3) — доступность; (2)/(5) — масштабируемость/эксплуатация; (1) — и то, и другое; (4) — перф.

## ПОДСКАЗКА
Чек-лист модуля: ключ высокой кардинальности под запросы; consistent hashing вместо `mod N`; шард + реплики (не SPOF); cross-shard агрегации офлайн; не шардировать преждевременно. Каждый пункт нарушает один принцип.
