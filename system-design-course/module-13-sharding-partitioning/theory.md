# Модуль 13 — Шардирование и партиционирование

## Зачем этот модуль

Репликация (модуль 12) масштабирует чтения и даёт доступность, но запись и весь объём данных по-прежнему должны помещаться в один узел: каждая реплика хранит **полную копию**. Когда данные перестают влезать на один сервер или поток записи превышает его ёмкость — упёрлись в стену, которую репликацией не пробить. Лекарство — **шардирование (sharding)**: разбить данные на части (шарды) и разложить по разным узлам, чтобы каждый держал лишь свою долю.

Шардирование — самый сложный из строительных блоков и самое трудно-обратимое решение: неудачно выбранный **shard key** аукается годами (hotspot'ы, дорогие cross-shard-запросы, болезненный resharding). Поэтому на интервью «как будете шардировать?» проверяет зрелость: понимаете ли вы, что ключ выбирают под паттерны запросов, что cross-shard-транзакции надо избегать, и что наивный `hash % N` превращает добавление узла в катастрофу.

Этот модуль — про стратегии шардирования (range/hash/directory/consistent), выбор shard key, проблемы (hotspot, cross-shard, resharding) и их смягчение.

---

## 1. Зачем шардировать и чем это отличается от репликации

| | **Репликация** | **Шардирование (партиционирование)** |
|--|----------------|--------------------------------------|
| Что делает | копирует **все** данные на каждый узел | делит данные на части, каждый узел хранит **свою** |
| Решает | доступность, масштаб чтений | масштаб **записи** и **объёма данных** |
| Узел хранит | полную копию | подмножество (шард) |

```
Репликация:   [ABC] [ABC] [ABC]   — три копии всего
Шардирование: [A]   [B]   [C]     — данные поделены
Вместе:       [A][A'] [B][B'] [C][C']  — каждый шард ещё и реплицирован
```

> Репликация и шардирование **ортогональны и применяются вместе**: шардирование даёт масштаб, репликация — доступность каждого шарда. Без репликации шарда отказ одного узла теряет целую часть данных (шард = SPOF). Признак, что пора шардировать: данные/запись не влезают в один узел даже с вертикальным масштабом.

---

## 2. Vertical vs horizontal partitioning

- **Vertical partitioning** — деление по **колонкам/полям** (редкие/тяжёлые колонки выносят в отдельную таблицу/хранилище). Пример: BLOB-поля отдельно от горячих атрибутов.
- **Horizontal partitioning (sharding)** — деление по **строкам**: разные записи на разные узлы по ключу. Это и есть шардирование в смысле масштаба.

> Дальше речь о горизонтальном шардировании — раскладывании строк по узлам. Vertical partitioning — это скорее приём моделирования (связан с денормализацией, модуль 11), чем механизм масштаба.

## 3. Стратегии шардирования

| Стратегия | Как | Плюсы | Минусы |
|-----------|-----|-------|--------|
| **Range-based** | по диапазону ключа (A–F, G–M, …) | эффективные range-запросы, сортировка | **hotspot** при неравномерном/последовательном ключе |
| **Hash-based** | `shard = hash(key) mod N` | равномерное распределение | **нет** range-запросов; resharding ломает `mod N` |
| **Consistent hashing** | hash на кольце (модуль 14) | минимум перемещения при изменении числа шардов | сложнее; всё ещё нет range-запросов |
| **Directory / lookup** | таблица соответствия ключ→шард | гибкость, произвольное размещение | lookup-сервис — узкое место/SPOF |
| **Geo / entity-based** | по региону/арендатору | данные рядом с пользователем, изоляция | неравномерность по регионам/тенантам |

```
Range:   [user A–F]→shard1  [G–M]→shard2  ...   (range-запросы дёшевы; «свежие» строки → один шард = hotspot)
Hash:    hash(user)→shard                       (равномерно; «все за вчера» → scatter по всем шардам)
```

> **Range vs hash — фундаментальный компромисс.** Range даёт дешёвые диапазонные запросы (по времени, по алфавиту), но рискует hotspot'ом, если ключ растёт монотонно (все новые записи летят в последний шард). Hash распределяет равномерно, но убивает range-запросы (соседние ключи разлетаются). Consistent hashing решает проблему resharding'а у hash-подхода, но не возвращает range-запросы.

## 4. Выбор shard key — критическое решение

Shard key (ключ партиционирования) определяет, в какой шард попадёт запись. Хороший ключ:

| Критерий | Почему |
|----------|--------|
| **Высокая кардинальность** | много различных значений → можно мелко делить |
| **Равномерность** | нагрузка и объём распределяются ровно, без hotspot |
| **Совпадение с паттернами запросов** | частые запросы попадают в один шард, не в scatter-gather |

```
Плохой ключ:  "country" (низкая кардинальность; страна-гигант → hot shard)
Плохой ключ:  "created_at" (монотонный; вся свежая запись → один шард)
Хороший ключ: "user_id" (высокая кардинальность, равномерно — если нет «звёзд»)
```

> Выбор shard key — **самое трудно-обратимое решение** шардирования: сменить его в проде = перелив всех данных. Идеальный ключ одновременно равномерен и совпадает с тем, как вы читаете. Часто эти требования конфликтуют (равномерный hash ломает range-запросы) — тогда заводят несколько представлений данных под разные запросы (как query-driven в модуле 11).

## 5. Проблемы шардирования

### Hotspot / hot shard / celebrity problem
Один шард получает непропорционально много нагрузки: плохой ключ (низкая кардинальность, монотонность) или «звезда» — пользователь-знаменитость с миллионами подписчиков, чьи данные в одном шарде.

**Смягчение:** добавить компонент в ключ (composite key, `celebrity_id + bucket`), вынести «звёзд» отдельно, кэшировать горячее (модуль 09), реплицировать горячий шард.

### Cross-shard запросы (scatter-gather)
Запрос, которому нужны данные из многих шардов (агрегация, сортировка по всем, поиск без shard key), идёт во все шарды и собирает результат — **дорого и медленно** (latency = самый медленный шард).

**Смягчение:** проектировать ключ под частые запросы; денормализация/вторичные индексы под другие паттерны; избегать запросов без shard key.

### Cross-shard транзакции
Транзакция, затрагивающая несколько шардов, требует распределённого протокола (2PC) — медленно, хрупко, снижает доступность.

**Смягчение:** проектировать границы так, чтобы транзакция жила в одном шарде; где нельзя — Saga/eventual (модуль 23), а не 2PC.

### Resharding / rebalancing
При росте надо добавлять шарды. Наивный `hash % N` при смене N перетасовывает **почти все** ключи (катастрофа). 

**Смягчение:** **consistent hashing** (модуль 14) — добавление шарда переносит лишь долю ключей; или заранее много логических шардов поверх немногих физических (pre-sharding).

> **Главные грабли — выбор ключа и resharding.** Большинство боли шардирования (hotspot, scatter-gather, болезненное масштабирование) — следствие ключа, выбранного без учёта запросов и роста. `hash % N` особенно коварен: работает на демо, но делает добавление узла операцией «перелить всё».

---

## 6. Когда НЕ шардировать

Шардирование добавляет огромную сложность: cross-shard-запросы, распределённые транзакции, resharding, эксплуатация. Поэтому:

```
Сначала исчерпай: вертикальный масштаб → read-реплики → кэш → архивация старого
Шардируй, только когда данные/запись доказанно не влезают в один узел.
```

> Преждевременное шардирование — частая ошибка senior-кандидатов «на показ». Один большой узел + реплики + кэш тянут очень далеко. Шардирование — это ответ на доказанную проблему масштаба объёма/записи, а не дефолт. «Соразмерность дизайна» (модуль 01) применима и здесь.

## ⚠️ Подводные камни

- **Плохой shard key.** Низкая кардинальность/монотонность → hotspot; ключ должен быть равномерным и под запросы.
- **Cross-shard-запросы как норма.** Scatter-gather дорог; частые запросы должны жить в одном шарде.
- **Cross-shard-транзакции (2PC) как норма.** Медленно и хрупко; границы — внутри шарда, иначе Saga (модуль 23).
- **`hash % N` для шардирования.** Resharding перетасовывает всё; используйте consistent hashing (модуль 14).
- **Шард без реплик.** Шард становится SPOF; шардирование комбинируют с репликацией.
- **Игнорировать celebrity/hot key.** «Звёзды» перегружают шард; выносите/кэшируйте/реплицируйте.
- **Преждевременное шардирование.** Сначала вертикальный масштаб, реплики, кэш; шардируйте под доказанную нужду.
- **Ключ, не совпадающий с запросами.** Всё уходит в scatter-gather; проектируйте под паттерны доступа.
- **Нет плана resharding.** Рост числа шардов должен быть продуман заранее (pre-sharding/consistent hashing).
