# Задача 01 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Покажите на числах проблему `hash % N` и то, как consistent hashing её снимает.

## ВХОДНЫЕ ДАННЫЕ
Распределённый кэш из N = 4 узлов, ключи распределены `node = hash(key) mod N`. Добавляется 5-й узел (N: 4 → 5).

## ТРЕБОВАНИЯ
- Объясните, почему при `mod N` смена N = 4 → 5 переносит почти все ключи (оцените долю остающихся на месте).
- Опишите, что произойдёт с кэшем в момент такого изменения (эффект на БД).
- Покажите, какую долю ключей переносит consistent hashing при добавлении одного узла.
- Сделайте вывод, почему для эластичных кластеров нужен consistent hashing.

## ФОРМАТ РЕЗУЛЬТАТА
Анализ `mod N` (доля перемещения) → эффект на кэш → доля при consistent hashing → вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Показано: при `mod N` смена N меняет остаток почти у всех ключей (на месте остаётся лишь малая доля).
- [ ] Эффект: массовый промах кэша → лавина в БД (cache stampede, модуль 09).
- [ ] Consistent hashing при добавлении узла переносит ~1/N ключей (только участок нового узла).
- [ ] Вывод: для растущих/сжимающихся кластеров `mod N` непригоден; consistent hashing минимизирует перемещение.
- [ ] Числа трактуются как порядки.

## ПОДСКАЗКА
`mod N` завязан на конкретное N: меняешь N — меняется остаток почти всюду. Consistent hashing трогает лишь участок, «отрезанный» новым узлом. Разница — «перелить всё» против «перелить 1/N».
