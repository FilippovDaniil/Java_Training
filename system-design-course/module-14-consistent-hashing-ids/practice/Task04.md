# Задача 04 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Выберите схему генерации идентификаторов под набор кейсов и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) ID сообщений мессенджера: миллионы/с, нужна сортируемость по времени (пагинация «новые сверху»), шардированная БД;
- (Б) идемпотентный ключ запроса, генерируемый клиентом офлайн без обращения к серверу;
- (В) первичный ключ в одиночной (нешардированной) реляционной БД небольшого сервиса.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите подход (UUID v4 / auto-increment / Snowflake / ULID / ticket server) и обоснуйте.
- Учтите оси: координация, сортируемость, масштаб, простота.
- Назовите неудачный выбор для каждого и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → схема → обоснование → анти-выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → Snowflake/ULID: сортируемый, без координации, масштаб; анти-выбор — auto-increment (SPOF на шардах) и UUID v4 (не сортируем → фрагментация индекса).
- [ ] (Б) → UUID v4 (или ULID): генерируется клиентом без сервера, уникален без координации; анти-выбор — auto-increment (нужен сервер).
- [ ] (В) → auto-increment приемлем (одна БД, просто, сортируем); анти-выбор — Snowflake без нужды (избыточно).
- [ ] Для каждого выбор увязан с координацией/сортируемостью/масштабом.
- [ ] Названы анти-выборы с причинами.

## ПОДСКАЗКА
Три вопроса: нужна ли сортируемость (→ Snowflake/ULID), есть ли координация/один узел (→ auto-increment ок), генерит ли клиент офлайн (→ UUID/ULID). Auto-increment прост, но не для распределёнки; Snowflake — для масштаба с сортировкой; UUID — когда сортировка не нужна, а координация невозможна.
