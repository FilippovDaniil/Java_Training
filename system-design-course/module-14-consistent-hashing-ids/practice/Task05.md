# Задача 05 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Составьте сравнительную таблицу UUID v4 vs Snowflake vs AUTO_INCREMENT по ключевым свойствам и сделайте рекомендацию под требования.

## ВХОДНЫЕ ДАННЫЕ
Требования системы: распределённая генерация без единой точки; сортируемость по времени желательна; компактный первичный ключ (влияет на размер индексов); высокая скорость; устойчивость к отказам.

## ТРЕБОВАНИЯ
- Сравните три подхода по осям: уникальность, сортируемость, координация/SPOF, размер, скорость, поведение как PK (фрагментация индекса).
- На основе требований дайте рекомендацию.
- Объясните, почему UUID v4 плох именно как кластерный первичный ключ.
- Объясните, почему AUTO_INCREMENT не подходит для распределённой генерации.

## ФОРМАТ РЕЗУЛЬТАТА
Сравнительная таблица (3 подхода × оси) + рекомендация + два пояснения.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Таблица покрывает: уникальность, сортируемость, координация/SPOF, размер (128 vs 64 бита), скорость, фрагментация индекса.
- [ ] UUID v4: уникален, но не сортируем, 128 бит, случайные вставки → фрагментация индекса.
- [ ] Snowflake: уникален, сортируем, 64 бита, без координации, локальная генерация.
- [ ] AUTO_INCREMENT: сортируем, но централизован (SPOF/узкое место), не для шардов.
- [ ] Рекомендация под требования — Snowflake/ULID, обоснованно.
- [ ] Объяснены слабости UUID v4 как PK и непригодность AUTO_INCREMENT в распределёнке.

## ПОДСКАЗКА
Ключевые оси: координация (Snowflake/UUID — нет, auto-inc — да), сортируемость (Snowflake/auto-inc — да, UUID v4 — нет), размер и влияние на индекс (UUID 128 бит + random → фрагментация). Под «распределённо + сортируемо + компактно» побеждает Snowflake/ULID.
