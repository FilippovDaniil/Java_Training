# Задача 06 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Спроектируйте Snowflake-подобный 64-битный ID: раскладку битов под заданную нагрузку и число узлов.

## ВХОДНЫЕ ДАННЫЕ
- Пиковая генерация: до ~50 000 ID/с суммарно, но возможны всплески на одном узле до нескольких тысяч ID в одну миллисекунду.
- Число узлов-генераторов: ожидается до ~500, с запасом до 1024.
- Срок жизни схемы: десятки лет (запас по timestamp).

## ТРЕБОВАНИЯ
- Распределите 64 бита: знак, timestamp, machine id, sequence; обоснуйте размер каждого поля.
- Проверьте, хватает ли sequence на всплеск ID в одну миллисекунду.
- Проверьте, хватает ли machine-битов на число узлов.
- Прикиньте, на сколько лет хватает timestamp-битов.
- Объясните, как обеспечивается сортируемость и отсутствие коллизий между узлами.

## ФОРМАТ РЕЗУЛЬТАТА
Раскладка битов (поле → бит → ёмкость) + проверки sequence/machine/timestamp + про сортируемость.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Раскладка: 1 бит знак + 41 timestamp + 10 machine + 12 sequence = 64 (или обоснованный вариант).
- [ ] sequence 12 бит = 4096 ID/мс на узел ≥ требуемого всплеска → хватает.
- [ ] machine 10 бит = 1024 узла ≥ 500 с запасом → хватает.
- [ ] timestamp 41 бит ≈ 2^41 мс ≈ ~69 лет от выбранной эпохи → покрывает десятки лет.
- [ ] Объяснено: timestamp в старших битах → сортируемость; machine id → нет коллизий между узлами; sequence → нет коллизий внутри мс.
- [ ] Числа трактуются как порядки.

## ПОДСКАЗКА
Классическая раскладка: 41 бит времени (≈69 лет), 10 бит узла (1024), 12 бит счётчика (4096/мс). Проверь каждое поле против требований: всплеск/мс ≤ 2^seq, узлы ≤ 2^machine, срок ≤ 2^41 мс. Старшие биты = время → монотонность и сортируемость.
