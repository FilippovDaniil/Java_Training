# Задача 07 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Найдите ошибки в решениях по распределению ключей и генерации ID и переформулируйте их.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Распределяем ключи по кэш-узлам через `hash(key) mod N`, при росте увеличим N».
2. «Consistent hashing внедрили, но каждый узел — одна точка на кольце; нагрузка скачет».
3. «ID заказов генерируем AUTO_INCREMENT в одной БД, потом будем шардировать».
4. «Первичный ключ — UUID v4, кластеризуем таблицу по нему для быстрых вставок».
5. «Сортировку „новые сверху“ делаем по UUID, раз он уникальный».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка больнее всего ударит при росте/шардировании.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) `mod N` → resharding перетасует почти всё → consistent hashing.
- [ ] (2) Нет vnodes → перекос → добавить virtual nodes для равномерности.
- [ ] (3) AUTO_INCREMENT при будущем шардировании → коллизии/SPOF → распределённый ID (Snowflake/ULID) заранее.
- [ ] (4) UUID v4 как кластерный PK → случайные вставки фрагментируют индекс → сортируемый ID (Snowflake/ULID).
- [ ] (5) UUID v4 не сортируем → «новые сверху» по нему не работает → сортируемый ID или отдельное поле времени.
- [ ] Отмечено: п.1 и п.3 особенно болезненны при росте/шардировании.

## ПОДСКАЗКА
Чек-лист модуля: consistent hashing вместо `mod N`; vnodes против перекоса; распределённый ID вместо auto-increment; сортируемый ID вместо UUID v4 как PK; сортировка по времени требует k-sortable ID. Каждый пункт нарушает один принцип.
