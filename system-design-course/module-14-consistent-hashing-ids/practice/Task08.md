# Задача 08 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Проанализируйте влияние clock skew на сортируемые ID (Snowflake-подобные) и спроектируйте защиту.

## ВХОДНЫЕ ДАННЫЕ
Система генерирует Snowflake-ID на множестве узлов (timestamp в старших битах). Наблюдения: (1) часы узлов слегка расходятся (десятки мс); (2) изредка NTP откатывает локальные часы назад.

## ТРЕБОВАНИЯ
- Объясните, как рассинхрон часов между узлами влияет на глобальный порядок ID.
- Объясните, чем опасен откат часов назад на одном узле (коллизии/непорядок).
- Предложите защиту от обоих случаев.
- Уточните, в каком смысле сортируемость остаётся «приближённой» (k-sortable) и когда нужен строгий порядок.

## ФОРМАТ РЕЗУЛЬТАТА
Влияние рассинхрона → опасность отката → меры защиты → про k-sortable и строгий порядок.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Рассинхрон: ID с разных узлов могут нарушать строгий глобальный порядок в пределах величины skew (но коллизий нет — machine id различает узлы).
- [ ] Откат часов: повтор того же timestamp → риск дубликатов/непорядка на узле.
- [ ] Защита: NTP-синхронизация; при откате — не уменьшать счётчик (ждать достижения прежнего времени или использовать монотонные часы); machine id исключает межузловые коллизии.
- [ ] Объяснено: сортируемость k-sortable (верна с точностью до skew), не абсолютна.
- [ ] Для строгого порядка нужен консенсус/логические часы (модуль 22).

## ПОДСКАЗКА
ID со временем в битах наследуют проблему синхронизации часов (как LWW в модуле 12). Между узлами коллизий нет (разный machine id), но строгий порядок нарушается на величину skew. Откат часов лечат ожиданием/монотонными часами. Абсолютный порядок — только через консенсус.
