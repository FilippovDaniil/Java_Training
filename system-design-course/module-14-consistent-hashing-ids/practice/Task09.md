# Задача 09 — Модуль 14: Consistent hashing и уникальные ID

## ЗАДАНИЕ
Примените consistent hashing к конкретной задаче распределения и оцените эффект.

## ВХОДНЫЕ ДАННЫЕ
Распределённый кэш Redis из 10 узлов обслуживает 200 млн горячих ключей. Планируется рост: периодически добавлять/выводить узлы (autoscaling). Сейчас используется клиентское `hash(key) mod N`.

## ТРЕБОВАНИЯ
- Спроектируйте размещение ключей через consistent hashing с virtual nodes.
- Оцените, какая доля ключей переедет при добавлении одного узла (11-й).
- Сравните с текущим `mod N` (что переедет там).
- Объясните, как vnodes обеспечат равномерность при 10 узлах и как учесть узел увеличенной памяти.

## ФОРМАТ РЕЗУЛЬТАТА
Дизайн размещения (CH + vnodes) + оценка переезда при +1 узле (CH vs mod N) + про равномерность/веса.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Размещение: ключи и узлы на кольце, каждый узел — много vnodes.
- [ ] При добавлении 11-го узла переезжает ~1/11 ключей (только его участок), остальное на месте.
- [ ] При `mod N` (10→11) переехало бы почти всё → массовый промах кэша.
- [ ] vnodes выравнивают распределение при 10 узлах; узлу с большей памятью дают больше vnodes (вес).
- [ ] Эффект увязан с разгрузкой БД (высокий hit rate сохраняется при масштабировании, модуль 09).

## ПОДСКАЗКА
Это прямое применение модуля к кэшу (связь с модулем 09). consistent hashing превращает «добавить узел = обнулить кэш» в «добавить узел = подвинуть 1/N ключей». vnodes дают равномерность и веса. Главный выигрыш — hit rate не рушится при autoscaling.
