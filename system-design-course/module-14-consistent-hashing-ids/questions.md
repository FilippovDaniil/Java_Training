## Ключ (эталонные ответы)

1. При изменении N остаток `mod N` меняется почти у всех ключей → почти все данные надо перераспределить. Схема пригодна лишь при постоянном N, а узлы добавляют и теряют постоянно.
2. Массовый промах: ключи «переезжают» на другие узлы, кэш фактически обнуляется → лавина запросов в БД (cache stampede).
3. Узлы и ключи отображаются хеш-функцией на замкнутый диапазон `[0, 2^k)` — кольцо. И те, и другие лежат на одном кольце.
4. Первому узлу по часовой стрелке от позиции ключа (ближайшему «вперёд»).
5. В среднем ~1/N — только ключи участка, который новый узел «отрезал» у соседа. Остальные остаются на месте.
6. Они переходят следующему узлу по часовой стрелке; остальное кольцо не затрагивается.
7. При немногих узлах участки кольца получаются разной длины → одни узлы получают больше ключей и нагрузки, чем другие.
8. **Virtual nodes** — каждый физический узел представлен многими точками на кольце. Решают: равномерность (усреднение участков) и гетерогенность (больше vnodes → больше доля, веса под мощность).
9. Нагрузка отказавшего узла (его vnodes) распределяется по многим соседям, а не сваливается на одного следующего узла.
10. **Rendezvous (HRW) hashing** — для каждого ключа считают `hash(key, node)` по всем узлам и выбирают максимум. Тоже минимизирует перемещение, проще для небольшого числа узлов.
11. Распределённый кэш (Redis/Memcached), шарды БД (Cassandra/DynamoDB/Riak), балансировка по ключу, P2P/распределённые хранилища.
12. Нужны глобально уникальные ID на масштабе без единой точки координации; централизованный счётчик становится узким местом и SPOF и не работает на шардах.
13. Это единая точка генерации (SPOF, узкое место), требует координации (убивает масштаб записи) и не работает на шардах (каждый шард сгенерил бы «1,2,3…» → коллизии).
14. Уникальность, высокая скорость без координации, отсутствие единой точки, часто — сортируемость по времени, компактность (размер PK влияет на индексы).
15. **Плюс:** уникален без координации, генерируется где угодно (в т.ч. клиентом). **Минус:** не сортируем, 128 бит, как кластерный PK вызывает фрагментацию индекса из-за случайных вставок.
16. Случайные значения → вставки попадают в произвольные места индекса (фрагментация, падение производительности). Для PK лучше сортируемый ID (Snowflake/ULID) — вставки локальны.
17. Сервис, выдающий узлам диапазоны ID батчами (Flickr-подход): узел получает блок (напр. 1000 ID) и раздаёт локально, обращаясь к серверу лишь периодически. Уменьшает координацию.
18. 64 бита: 1 бит знак + ~41 бит timestamp (мс от эпохи) + ~10 бит machine id (1024 узла) + ~12 бит sequence (4096 ID/мс на узел).
19. Timestamp лежит в старших битах, поэтому численное значение ID растёт со временем → ID естественно сортируются по времени создания (k-sortable).
20. Каждый узел имеет уникальный machine id (в своих битах) → ID разных узлов не пересекаются; sequence исключает коллизии внутри одной миллисекунды. Координация между узлами не нужна.
21. Сортируемые альтернативы UUID: кодируют timestamp в старших битах + случайность, дают уникальность без координации и сортируемость по времени (в отличие от UUID v4).
22. **Clock skew** — рассинхрон часов между узлами. Для ID со временем в битах он нарушает строгий глобальный порядок в пределах величины skew (коллизий нет — спасает machine id).
23. Откат часов назад может повторить тот же timestamp → риск дубликатов/непорядка на узле. Защита: не уменьшать счётчик (ждать/монотонные часы), NTP-синхронизация.
24. **k-sortable** — порядок верен с точностью до рассинхрона часов (приближённо), а не абсолютно. Строгий глобальный порядок требует консенсуса/логических часов (модуль 22).
25. Распределение ключей (consistent hashing + vnodes, правило принадлежности, доля переезда); равномерность и веса; выбор схемы ID под требования; раскладка битов (для Snowflake-подобных); защита от clock skew; связи с кэшем/шардированием/репликацией.
