# Модуль 14 — Consistent hashing и уникальные ID

## Зачем этот модуль

В модулях 12–13 дважды всплыла одна и та же боль: при шардировании и распределённом кэшировании наивный `hash(key) % N` превращает добавление узла в катастрофу — перетасовывается почти всё. Этот модуль закрывает вопрос: **consistent hashing** — механизм распределения ключей по узлам, при котором изменение числа узлов переносит лишь малую долю данных. Это «клей», без которого не собрать ни распределённый кэш (модуль 09), ни шардированную БД (модуль 13).

Вторая тема — оборотная сторона распределённости: если данных и узлов много, нужны **глобально уникальные идентификаторы**. Привычный `AUTO_INCREMENT` — это единая точка координации (а значит, узкое место и SPOF), он не работает на шардах. Как генерировать миллионы уникальных ID в секунду без координации, да ещё и сортируемых по времени? Это классическая задача дизайна (Snowflake, UUID, ULID).

Оба сюжета — фундаментальные «строительные примитивы», которые всплывают почти в каждой референсной архитектуре Части 8.

---

## 1. Проблема naive hashing (`hash % N`)

```
shard = hash(key) mod N      (N = число узлов)

N=4:  hash(key)=10 → 10 mod 4 = 2 → узел 2
Добавили узел, N=5:  10 mod 5 = 0 → узел 0   ← ключ переехал!
```

При смене N остаток `mod N` меняется почти у **всех** ключей. Для кэша это означает массовый промах (cache stampede, модуль 09); для шардов БД — перелив почти всех данных (фактически простой).

> `hash % N` отлично работает, пока N постоянно, и разваливается при любом добавлении/удалении узла. Поскольку узлы добавляют (рост) и теряют (отказ) постоянно, для распределённых данных он непригоден. Нужна схема, где изменение числа узлов трогает лишь малую часть ключей.

---

## 2. Consistent hashing: кольцо

Идея: отобразить и **узлы**, и **ключи** на одно кольцо (диапазон хеша `[0, 2^k)`, замкнутый в круг). Ключ принадлежит **первому узлу по часовой стрелке**.

```
          0/2^k
            │
   nodeC ●──┴──● nodeA
        │       │
   ─────┤  ring ├─────   ключ k → идём по часовой → первый узел = его владелец
        │       │
        └───●───┘
          nodeB

Добавили nodeD между B и C → переезжают только ключи между B и D,
                              остальные остаются на месте.
```

| Операция | Что переносится |
|----------|-----------------|
| Добавить узел | только ключи участка, который он «отрезал» у соседа (~1/N) |
| Удалить узел | его ключи уходят следующему по часовой стрелке |

> Ключевое свойство: добавление/удаление узла переносит в среднем лишь **1/N** ключей, а не почти все. Это и делает возможными эластичный распределённый кэш и resharding без простоя. Consistent hashing используют Cassandra, DynamoDB, Riak, Memcached-клиенты, многие LB.

---

## 3. Virtual nodes (vnodes)

Наивное кольцо неравномерно: при немногих узлах участки получаются разного размера → перекос нагрузки. Решение — **виртуальные узлы**: каждый физический узел представлен **многими** точками на кольце.

| Без vnodes | С vnodes |
|------------|----------|
| 1 узел = 1 точка | 1 узел = десятки/сотни точек |
| участки неравные → перекос | усреднение → равномерность |
| мощность узлов не учесть | больше точек = больше доля (веса) |

> Virtual nodes решают сразу две задачи: **равномерность** (много мелких участков усредняют распределение) и **гетерогенность** (мощному узлу дают больше vnodes → больше ключей). Плюс при отказе узла его нагрузка распределяется по многим соседям, а не валится на одного. Vnodes — стандарт в реальных системах (Cassandra, Dynamo).

**Альтернатива:** *rendezvous hashing (HRW)* — для каждого ключа считают `hash(key, node)` по всем узлам и берут максимум. Тоже даёт минимум перемещения, проще в реализации для небольшого числа узлов.

---

## 4. Где применяется consistent hashing

| Применение | Что даёт |
|------------|----------|
| Распределённый кэш (Redis/Memcached) | добавление узла не обнуляет весь кэш (модуль 09) |
| Шарды БД (Cassandra, Dynamo) | resharding переносит долю данных, не всё (модуль 13) |
| Балансировка по ключу | стабильная привязка клиент/ключ → узел (модуль 08) |
| P2P / распределённые хранилища | размещение данных без центрального каталога |

## 5. Уникальные ID: зачем и требования

В распределённой системе нужны идентификаторы (для заказов, сообщений, событий). Привычный `AUTO_INCREMENT`:

```
❌ AUTO_INCREMENT:
   - единая точка генерации → узкое место и SPOF
   - не работает на шардах (каждый шард сгенерит «1, 2, 3…» → коллизии)
   - координация убивает масштаб записи
```

Требования к распределённому ID-генератору:

| Требование | Зачем |
|------------|-------|
| **Уникальность** | без коллизий глобально |
| **Высокая скорость** | миллионы ID/с без координации |
| **Без единой точки** | не SPOF, не узкое место |
| **Сортируемость по времени** (часто) | пагинация, «новые сверху», локальность индекса |
| **Компактность** | размер PK влияет на индексы/хранение |

---

## 6. Подходы к генерации ID

| Подход | Уникальность | Сортируемость | Координация | Заметка |
|--------|--------------|---------------|-------------|---------|
| **UUID v4** (random) | да | **нет** | не нужна | 128 бит; плох как кластерный PK (random insert → фрагментация индекса) |
| **AUTO_INCREMENT** | да | да | централизованная | SPOF, не для шардов |
| **Ticket server / range** (Flickr) | да | да (грубо) | периодическая | сервис выдаёт диапазоны узлам (батчами) |
| **Snowflake** (Twitter) | да | **да** (k-sortable) | не нужна | 64 бита: time + machine + sequence |
| **ULID / KSUID** | да | да | не нужна | сортируемая замена UUID |

### Snowflake — раскладка 64 бит

```
┌─1─┬──── 41 бит timestamp ────┬─ 10 бит machine ─┬─ 12 бит sequence ─┐
│ 0 │  миллисекунды от эпохи    │  ID узла (1024)  │ счётчик в той же мс│
└───┴───────────────────────────┴──────────────────┴───────────────────┘
```
- **timestamp** (старшие биты) → ID растут во времени (сортируемость).
- **machine id** → разные узлы не пересекаются (без координации).
- **sequence** → до 4096 ID на узел в одну миллисекунду.

> Snowflake — каноничный ответ на «сгенерируйте уникальные сортируемые ID на масштабе»: компактный (64 бита, влезает в `bigint`), сортируемый по времени, генерируется локально без координации. Сортируемость важна для пагинации «новые сверху» и для локальности вставки в индекс (в отличие от UUID v4, который раскидывает вставки случайно и фрагментирует индекс).

---

## 7. Clock skew — расплата за сортируемость

Snowflake и ULID кладут время в старшие биты → их корректность зависит от **часов узла**. Проблемы:

- **Рассинхрон часов** между узлами → нарушение глобального порядка.
- **Перевод часов назад** (NTP-коррекция) → возможны дубликаты/непорядок.

**Защита:** NTP-синхронизация; не уменьшать локальный счётчик при откате часов (ждать или использовать монотонные часы); machine id гарантирует отсутствие коллизий между узлами даже при skew.

> Любая схема, кодирующая время в ID, наследует проблему синхронизации часов (та же, что мешает LWW в модуле 12). Поэтому «сортируемость по времени» — приближённая (k-sortable: порядок верен с точностью до рассинхрона), а не абсолютная. Для строгого порядка нужен консенсус/логические часы (модуль 22).

---

## ⚠️ Подводные камни

- **`hash % N` для распределённых данных.** Любое изменение N перетасует почти всё; используйте consistent hashing.
- **Consistent hashing без vnodes.** Неравномерные участки → перекос нагрузки; vnodes выравнивают.
- **AUTO_INCREMENT как глобальный ID в распределёнке.** SPOF, узкое место, коллизии на шардах.
- **UUID v4 как кластерный PK.** Случайные вставки фрагментируют индекс; для PK берите сортируемый ID (Snowflake/ULID).
- **Игнорировать сортируемость, когда она нужна.** Пагинация/«новые сверху»/локальность индекса требуют k-sortable ID.
- **Не учитывать clock skew в Snowflake-подобных.** Откат часов → дубли/непорядок; нужна NTP и защита счётчика.
- **Считать k-sortable строгим порядком.** Это приближение с точностью до рассинхрона часов, не консенсус.
- **Один узел генерации ID «для простоты».** Возвращает SPOF, который Snowflake как раз убирает.
