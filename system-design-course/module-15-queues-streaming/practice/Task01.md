# Задача 01 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Выберите между message queue и event stream (log) под кейсы и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) отправка письма-подтверждения после регистрации: задача, должна выполниться один раз;
- (Б) событие «заказ создан», на которое реагируют биллинг, склад, уведомления и аналитика; нужна возможность перечитать историю при добавлении нового потребителя;
- (В) поток кликов для real-time-аналитики с хранением и повторной обработкой.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите queue или stream и обоснуйте моделью (задача vs событие).
- Укажите, нужен ли replay/retention и сколько потребителей.
- Назовите неудачный выбор для каждого.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → queue/stream → обоснование (задача/событие, потребители, replay) → анти-выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → message queue (задача одному consumer'у, после ack исчезает).
- [ ] (Б) → event stream/log (одно событие многим независимым потребителям + replay при добавлении нового).
- [ ] (В) → stream (retention + повторная обработка для аналитики).
- [ ] Для каждого указано число потребителей и нужен ли replay.
- [ ] Названы анти-выборы (напр. queue для (Б) — каждый потребитель требовал бы свою копию/очередь, нет replay).

## ПОДСКАЗКА
Queue = задача, которую один consumer выполнит и забудет. Stream/log = событие, интересное многим, с хранением и возможностью перечитать. «Много независимых читателей + история» — однозначно stream (Kafka).
