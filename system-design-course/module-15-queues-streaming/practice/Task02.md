# Задача 02 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Определите, какие шаги обработки оставить синхронными, а какие развязать через очередь.

## ВХОДНЫЕ ДАННЫЕ
Сценарий загрузки видео: (1) проверка авторизации и размера файла; (2) сохранение файла; (3) транскодинг в несколько качеств; (4) генерация превью; (5) отправка уведомления подписчикам; (6) обновление поисковой выдачи. Сейчас всё выполняется синхронно в одном HTTP-запросе — пользователь ждёт минуты.

## ТРЕБОВАНИЯ
- Разделите шаги на синхронные (в пути запроса) и асинхронные (через очередь).
- Обоснуйте каждую границу (что нужно пользователю немедленно, что может подождать).
- Опишите, что возвращается пользователю сразу и как он узнаёт о завершении фоновой части.
- Объясните, какое свойство системы меняется после развязки (eventual consistency).

## ФОРМАТ РЕЗУЛЬТАТА
Разбивка шагов sync/async + обоснование границ + ответ пользователю + замечание про eventual consistency.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Sync: (1) авторизация/валидация, (2) приём/сохранение файла — нужно для ответа «принято».
- [ ] Async (очередь): (3) транскодинг, (4) превью, (5) уведомления, (6) индексация — тяжёлое/не нужно немедленно.
- [ ] Пользователю сразу возвращается «загружено, обрабатывается»; о завершении — через статус/событие/уведомление.
- [ ] Обосновано: тяжёлое и не-немедленное уходит из пути запроса (модуль 10 — media pipeline).
- [ ] Отмечено: видео становится доступным не мгновенно (eventual) — это цена развязки.

## ПОДСКАЗКА
Граница проста: что нужно пользователю для ответа «принято» — sync; что можно сделать потом — async. Транскодинг (минуты) держать в HTTP нельзя. Пользователь получает «обрабатывается», а готовность приходит позже. Это связь с media pipeline (модуль 10).
