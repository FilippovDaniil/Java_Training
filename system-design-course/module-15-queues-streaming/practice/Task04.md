# Задача 04 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Спроектируйте идемпотентный consumer для at-least-once-очереди.

## ВХОДНЫЕ ДАННЫЕ
Consumer обрабатывает сообщения «начислить кэшбэк пользователю». Очередь гарантирует at-least-once → возможны повторные доставки (consumer обработал, упал до ack, сообщение пришло снова). Сейчас обработчик делает `balance += amount` — при дубле начисляет дважды.

## ТРЕБОВАНИЯ
- Объясните, почему текущий обработчик не идемпотентен.
- Предложите механизм дедупликации/идемпотентности (минимум один детально).
- Опишите, где и как хранить «уже обработано» и как это атомарно с самим действием.
- Назовите разницу между естественно идемпотентными и неидемпотентными операциями.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз → механизм идемпотентности → хранение «обработано» + атомарность → про природу операций.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: `balance += amount` не идемпотентен — повтор меняет результат (двойное начисление).
- [ ] Механизм: дедупликация по message-id / idempotency key (хранить обработанные id и игнорировать повтор) либо естественно идемпотентная формулировка.
- [ ] Хранение «обработано» атомарно с действием (одна транзакция: проверить id + начислить + записать id), иначе гонка.
- [ ] Объяснена разница: `SET status=X` / upsert — естественно идемпотентны; `+=`/insert — нет.
- [ ] Решение устраняет двойное начисление при повторной доставке.

## ПОДСКАЗКА
Идемпотентность = повтор даёт тот же результат. Либо переформулируй операцию (естественная идемпотентность), либо дедуплицируй по id. Ключевая тонкость — атомарность «отметка обработки + само действие» в одной транзакции, иначе сбой между ними вернёт дубли. Детали — модуль 23.
