# Задача 05 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Спроектируйте партиционирование топика и гарантии порядка через partition key.

## ВХОДНЫЕ ДАННЫЕ
Stream «события заказов» (Kafka-подобный). Для каждого заказа события идут в строгой последовательности: `created → paid → shipped → delivered`. Нагрузка высокая, нужен параллелизм обработки. Требование: события одного заказа обрабатываются строго по порядку, разные заказы — параллельно.

## ТРЕБОВАНИЯ
- Объясните, почему глобальный порядок в распределённом стриме недостижим/дорог.
- Выберите partition key так, чтобы события одного заказа шли по порядку.
- Объясните связь числа партиций с параллелизмом и consumer group.
- Назовите ловушку: что будет при partition key, не совпадающем с единицей порядка.

## ФОРМАТ РЕЗУЛЬТАТА
Про глобальный порядок + выбор partition key + партиции/параллелизм/consumer group + ловушка.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: порядок гарантируется только в пределах партиции, не глобально.
- [ ] partition key = `order_id` → все события одного заказа в одной партиции → строгий порядок.
- [ ] Разные заказы попадают в разные партиции → обрабатываются параллельно.
- [ ] Параллелизм = число партиций; в consumer group один consumer на партицию.
- [ ] Ловушка: если ключ не `order_id` (напр. случайный), события заказа разбредутся по партициям → порядок нарушится.

## ПОДСКАЗКА
Это та же идея, что shard key (модуль 13): связанные по порядку события должны иметь общий partition key → одна партиция → порядок. Параллелизм ограничен числом партиций (один consumer на партицию в группе). Глобального порядка между партициями нет — и не нужен.
