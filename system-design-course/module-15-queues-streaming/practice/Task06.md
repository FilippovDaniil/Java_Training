# Задача 06 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Спроектируйте обработку сбоев: retry с backoff и dead letter queue (DLQ) для «ядовитых» сообщений.

## ВХОДНЫЕ ДАННЫЕ
Consumer обрабатывает сообщения. Бывает два типа сбоев: (1) временный (downstream-сервис на секунду недоступен) и (2) постоянный («ядовитое» сообщение — некорректный формат, обработать невозможно). Сейчас при любой ошибке сообщение просто перечитывается бесконечно.

## ТРЕБОВАНИЯ
- Объясните, чем опасно бесконечное перечитывание ядовитого сообщения.
- Спроектируйте retry с экспоненциальным backoff для временных сбоев.
- Спроектируйте DLQ: когда сообщение туда уходит и что с ним дальше.
- Объясните, как отличать временный сбой от постоянного на практике (счётчик попыток).

## ФОРМАТ РЕЗУЛЬТАТА
Опасность poison message → retry с backoff → DLQ (условие + дальнейшая судьба) → различение сбоев.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Опасность: ядовитое сообщение зацикливает consumer'а (падает → перечитывает → падает) → поток встаёт.
- [ ] Retry с backoff: повтор с растущей задержкой (напр. 1с, 2с, 4с…) для временных сбоев, чтобы не долбить downstream.
- [ ] DLQ: после N неудачных попыток сообщение уходит в отдельную очередь для ручного/отложенного разбора, не блокируя основной поток.
- [ ] Различение: счётчик попыток (retry count) — превысил порог → считаем постоянным → DLQ.
- [ ] Основной поток продолжает работать, ядовитые сообщения изолированы.

## ПОДСКАЗКА
Временные сбои лечат retry с backoff (растущая пауза щадит downstream). Постоянные («ядовитые») нельзя ретраить вечно — после N попыток в DLQ для разбора. Счётчик попыток отличает одно от другого. Без DLQ один битый месседж останавливает всю обработку.
