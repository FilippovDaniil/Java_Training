# Задача 08 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Спроектируйте pub/sub fan-out: одно событие → несколько независимых потребителей.

## ВХОДНЫЕ ДАННЫЕ
Событие «заказ оформлен» должно запустить: списание оплаты (биллинг), резерв товара (склад), письмо клиенту (уведомления), запись в аналитику. Сейчас сервис заказов синхронно вызывает все четыре сервиса по очереди — медленно и хрупко (падение одного ломает оформление).

## ТРЕБОВАНИЯ
- Объясните проблемы текущего синхронного fan-out (latency, связанность, отказы).
- Спроектируйте pub/sub: продьюсер публикует событие, подписчики реагируют независимо.
- Объясните, как добавить нового потребителя (например, антифрод), не трогая продьюсера.
- Учтите доставку и идемпотентность подписчиков; нарисуйте схему.

## ФОРМАТ РЕЗУЛЬТАТА
Проблемы sync fan-out → дизайн pub/sub → добавление потребителя → доставка/идемпотентность + схема.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблемы sync: latency складывается, продьюсер связан со всеми, падение одного ломает оформление.
- [ ] Pub/sub: сервис заказов публикует «order.created», подписчики (биллинг/склад/уведомления/аналитика) читают независимо.
- [ ] Новый потребитель (антифрод) подписывается на событие без изменения продьюсера.
- [ ] Указаны at-least-once + идемпотентность подписчиков; падение одного подписчика не влияет на других.
- [ ] Схема показывает один продьюсер → событие → много независимых подписчиков.

## ПОДСКАЗКА
Pub/sub развязывает источник события от потребителей: продьюсер не знает, кто слушает. Добавить подписчика = подписаться, не трогая продьюсера. Это ядро event-driven-архитектуры: сервисы реагируют на события, а не вызывают друг друга напрямую (stream/log даёт fan-out натурально).
