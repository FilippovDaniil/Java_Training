# Задача 09 — Модуль 15: Очереди и стриминг

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне с очередями/стримингом и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Нам нужен exactly-once — включим его в брокере, и про дубли можно забыть».
2. «Consumer делает `INSERT` заказа на каждое сообщение; доставка at-least-once».
3. «Сообщения храним в очереди как основное хранилище состояния заказов».
4. «Ошибочные сообщения просто перечитываем, пока не обработаются».
5. «Рассчитываем на глобальный порядок всех событий в топике из 8 партиций».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка опаснее всего для корректности данных.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) exactly-once как «галочка» → ложная уверенность → at-least-once + идемпотентность.
- [ ] (2) `INSERT` на at-least-once → дубли заказов → идемпотентность (дедуп по id / upsert).
- [ ] (3) Очередь как БД → потеря состояния/неудобный доступ → состояние в БД, очередь — транспорт (или log с retention для replay, но не как primary store).
- [ ] (4) Бесконечный перечит ядовитых → поток встаёт → DLQ + ограничение попыток.
- [ ] (5) Глобальный порядок в 8 партициях → его нет → порядок только в партиции, ключ по единице порядка.
- [ ] Отмечена самая опасная для корректности (обычно п.2 — дубли заказов или п.1 — ложная уверенность).

## ПОДСКАЗКА
Чек-лист модуля: at-least-once + идемпотентность (не «exactly-once-галочка»); idempotent consumer вместо голого INSERT; очередь — транспорт, не БД; DLQ вместо вечного ретрая; порядок только в партиции. Каждый пункт нарушает один принцип.
