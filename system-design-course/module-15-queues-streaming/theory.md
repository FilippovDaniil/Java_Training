# Модуль 15 — Очереди и стриминг сообщений

## Зачем этот модуль

До сих пор все блоки обслуживали **синхронный** путь «запрос → ответ»: клиент ждёт, пока система обработает. Но огромная часть реальной работы не должна происходить «прямо сейчас, пока пользователь ждёт»: транскодинг видео (модуль 10), рассылка уведомлений, аналитика, интеграция между сервисами. Для этого нужен **асинхронный** костяк — очереди и стримы сообщений.

Очередь делает три вещи, которые невозможно получить синхронным вызовом: **развязывает** producer и consumer (они не знают друг о друге и не падают вместе), **сглаживает пики** (буфер принимает всплеск, обработка идёт ровно) и **повышает надёжность** (сообщение не теряется, обработку можно повторить). Это превращает хрупкую цепочку синхронных вызовов в устойчивую event-driven-систему.

На интервью «как развяжете эти сервисы?», «куда денем тяжёлую обработку?», «что если consumer упал?» — прямые сигналы к очередям. Этот модуль — про queue vs stream, семантику доставки, идемпотентность, порядок, DLQ и pub/sub.

---

## 1. Зачем асинхронность

| Выигрыш | Как очередь даёт |
|---------|------------------|
| **Decoupling** (развязка) | producer пишет в очередь и не знает/не ждёт consumer'а; падение одного не роняет другого |
| **Load leveling** (сглаживание пиков) | буфер принимает всплеск, consumer обрабатывает в своём темпе |
| **Надёжность** | сообщение хранится до обработки; упал consumer → повтор |
| **Масштаб обработки** | несколько consumer'ов разбирают очередь параллельно |
| **Фоновая работа** | тяжёлое (transcode, отчёты, рассылки) уходит из пути запроса |

```
Sync:   client → service A → service B → service C   (A ждёт всю цепочку; B упал → всё упало)
Async:  client → service A → [queue] → service B      (A ответил сразу; B разбирает в своём темпе)
```

> Главная ценность очереди — **развязка во времени и в отказах**. Producer и consumer работают независимо: можно перезапускать, масштабировать, держать на ремонте consumer, а producer продолжает писать. Это фундамент устойчивых и масштабируемых систем — но цена в том, что система становится **eventually consistent** (результат появляется не мгновенно).

---

## 2. Sync vs async — когда что

| Используйте sync | Используйте async (очередь) |
|------------------|------------------------------|
| нужен немедленный ответ (чтение, валидация) | результат не нужен прямо сейчас |
| низкая latency запроса критична | тяжёлая/долгая обработка |
| простая операция, без фоновой работы | развязка сервисов, сглаживание пиков |
| клиент ждёт результат | «принято, обработаем» достаточно |

> Не всё надо делать асинхронно: очередь добавляет сложность (доставка, идемпотентность, мониторинг, eventual consistency). Синхронный вызов проще, когда клиенту реально нужен ответ сейчас. Очередь — когда работа может (и должна) уйти из пути запроса.

## 3. Message queue vs event streaming

Два разных примитива, которые часто путают:

| | **Message queue** (RabbitMQ, SQS) | **Event stream / log** (Kafka, Kinesis, Pulsar) |
|--|-----------------------------------|--------------------------------------------------|
| Модель | «задача» — сообщение для обработки | «событие» — факт, который случился |
| Потребление | сообщение берёт **один** consumer, после ack удаляется | append-only лог; **много** consumer'ов читают независимо |
| Хранение | до обработки (затем исчезает) | retention (часы/дни/вечно), можно перечитать (replay) |
| Позиция чтения | брокер отслеживает | consumer хранит свой **offset** |
| Типичное применение | task queue, RPC-разгрузка, рассылки | event sourcing, аналитика, межсервисные события, CDC |

```
Queue:   [msg1 msg2 msg3] → consumer берёт msg1, ack → msg1 удалён
Stream:  log: [e1 e2 e3 e4 ...]  (хранится)
           ├─ consumerA offset=2
           └─ consumerB offset=4   (читают независимо, могут перечитать)
```

> **Queue** — когда сообщение = задача, которую надо выполнить один раз и забыть (отправить письмо, обработать платёж). **Stream** — когда сообщение = событие, которое интересно многим и которое нужно хранить/переигрывать (одно событие «заказ создан» читают биллинг, аналитика, уведомления, поиск). Kafka силён именно тем, что событие читают много независимых потребителей и можно перечитать историю.

## 4. Семантика доставки

| Гарантия | Что значит | Цена |
|----------|------------|------|
| **At-most-once** | каждое сообщение ≤ 1 раз (возможна потеря) | просто, быстро; теряет при сбое |
| **At-least-once** | каждое сообщение ≥ 1 раз (возможны дубли) | надёжно, но **нужна идемпотентность** consumer'а |
| **Exactly-once** | ровно один раз | сложно и дорого; ограничено условиями |

```
At-least-once: consumer обработал, но упал до ack → сообщение придёт снова → дубль
```

> **At-least-once + идемпотентный consumer — практический стандарт.** Exactly-once часто звучит как «галочка», но в общем случае распределённо он недостижим дёшево (нужны транзакционные гарантии end-to-end). Зрелый ответ: «гарантирую at-least-once и делаю обработку идемпотентной» — это даёт эффект exactly-once без его цены.

---

## 5. Идемпотентность consumer'а

При at-least-once сообщение может прийти дважды → обработчик должен давать **тот же результат при повторе**.

| Приём | Как |
|-------|-----|
| Дедупликация по ID | хранить обработанные message-id, повтор игнорировать |
| Idempotency key | операция с ключом выполняется один раз (как в платежах) |
| Естественная идемпотентность | `SET status=PAID` (повтор безвреден), в отличие от `balance += 10` |
| Upsert вместо insert | повторная вставка не дублирует |

> Идемпотентность — **обязательное** свойство consumer'а в at-least-once-системе, а не опция. Без неё дубли приводят к двойным спискам, двойным письмам, двойным заказам. Деталь раскрыта в модуле 23 (надёжность операций); здесь важно: проектируя async, сразу проектируйте идемпотентность.

## 6. Порядок сообщений и партиции

Глобальный порядок в распределённой очереди дорог. Реальная гарантия — **порядок в пределах партиции** (Kafka) или одной очереди.

```
Topic "orders" → партиции по partition key = order_id
  партиция 0: [order_A:created, order_A:paid, order_A:shipped]  ← порядок гарантирован
  партиция 1: [order_B:created, order_B:paid]
Между партициями порядка нет; параллелизм = число партиций.
```

| Свойство | Как достигается |
|----------|-----------------|
| Порядок для связанных событий | один partition key → одна партиция → порядок |
| Параллелизм | больше партиций → больше consumer'ов в группе |
| Consumer group | партиции распределяются между consumer'ами группы |

> **Partition key — ключевое решение стриминга** (как shard key в модуле 13). События, для которых важен порядок (жизненный цикл одного заказа), должны иметь общий ключ → попасть в одну партицию. Параллелизм ограничен числом партиций: один consumer на партицию в группе. Ожидать глобальный порядок между партициями — частая ошибка.

## 7. Надёжность: DLQ, retry, backpressure

| Механизм | Зачем |
|----------|-------|
| **Retry с backoff** | временный сбой → повторить с растущей задержкой |
| **Dead Letter Queue (DLQ)** | «ядовитое» сообщение (не обрабатывается) после N попыток → в отдельную очередь, не блокирует поток |
| **Backpressure** | producer быстрее consumer'а → ограничение/буфер, чтобы не переполнить |
| **Мониторинг lag** | отставание consumer'а (растёт очередь) — ключевая метрика здоровья |

> Без **DLQ** одно непарсимое («ядовитое») сообщение зацикливает consumer'а: он падает, перечитывает, снова падает — поток встаёт. DLQ изолирует такие сообщения для ручного разбора. **Consumer lag** (насколько consumer отстал) — главный сигнал: растёт → обработка не справляется, нужен масштаб или очередь переполнится.

---

## 8. Pub/Sub (fan-out)

Один producer → **много** независимых подписчиков, каждый получает копию события.

```
"order.created" → ┬→ биллинг (списать)
                  ├→ склад (зарезервировать)
                  ├→ уведомления (письмо)
                  └→ аналитика (записать)
```

> Pub/Sub (натуральный для stream/log) развязывает источник события от его потребителей: добавить нового подписчика (например, новый сервис рекомендаций) можно, не трогая producer. Это основа event-driven-архитектуры — сервисы реагируют на события, а не вызывают друг друга напрямую.

## ⚠️ Подводные камни

- **Exactly-once как «галочка».** В общем случае дорог/недостижим; берите at-least-once + идемпотентность.
- **Consumer не идемпотентен при at-least-once.** Дубли → двойные письма/заказы/списания.
- **Очередь как база данных.** Очередь — транспорт, не хранилище состояния; не держите в ней данные «навсегда» (кроме log с retention под replay).
- **Нет DLQ.** Ядовитое сообщение зацикливает consumer'а и останавливает поток.
- **Ожидать глобальный порядок.** Порядок только в пределах партиции/очереди; между партициями его нет.
- **Не мониторить consumer lag.** Очередь растёт незаметно → переполнение/задержки.
- **Нет backpressure.** Producer заваливает consumer'а → неограниченный рост и сбой.
- **Async там, где нужен немедленный ответ.** Не всё стоит развязывать; sync проще, когда клиент ждёт результат.
- **Путать queue и stream.** Задача одному → queue; событие многим с историей → stream.
