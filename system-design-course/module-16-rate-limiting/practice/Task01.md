# Задача 01 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Выберите алгоритм rate limiting под требования и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) публичный API: реальный трафик «дёргается», нужно допускать короткие всплески, но держать средний rate;
- (Б) защита чувствительного downstream, который требует строго равномерного потока запросов;
- (В) простейший лимит для внутреннего сервиса, где допустима грубая точность и минимум памяти.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите алгоритм (token bucket / leaky bucket / fixed window / sliding window) и обоснуйте.
- Назовите главный недостаток выбранного алгоритма.
- Для (В) укажите, какой риск несёт fixed window и приемлем ли он здесь.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → алгоритм → обоснование → недостаток.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → token bucket (допускает burst до размера ведра, держит средний rate).
- [ ] (Б) → leaky bucket (сглаживает в строго равномерный поток на выходе).
- [ ] (В) → fixed window (просто, минимум памяти), с оговоркой про всплеск на границе окна.
- [ ] Для каждого назван недостаток.
- [ ] Выбор увязан с требованием (burst / ровный поток / простота), а не «по умолчанию».

## ПОДСКАЗКА
Token bucket — дефолт, разрешает разумные всплески. Leaky bucket — когда нужен ровный выход (защита downstream). Fixed window — простейший, но дырявый на границе. Sliding window — когда нужна точность без дыры границы.
