# Задача 02 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Сравните token bucket и leaky bucket для конкретного кейса и выберите.

## ВХОДНЫЕ ДАННЫЕ
API мобильного приложения: клиенты периодически синхронизируются пачками (несколько запросов подряд после открытия приложения), затем затишье. Средний rate невысок, но всплески нормальны и желательны. Альтернатива — downstream, который хотелось бы нагружать ровно.

## ТРЕБОВАНИЯ
- Опишите поведение token bucket и leaky bucket на этом «пачечном» трафике.
- Объясните, что произойдёт с легитимными пачками при leaky bucket.
- Выберите алгоритм под приоритет «не наказывать нормальные всплески» и обоснуйте.
- Укажите, при каком приоритете выбор был бы обратным.

## ФОРМАТ РЕЗУЛЬТАТА
Поведение обоих на пачках + эффект leaky на легитимные всплески + выбор + условие обратного выбора.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Token bucket: накопленные токены позволяют пройти пачке сразу (burst), затем восстановление по rate.
- [ ] Leaky bucket: пачка «вытекает» равномерно → легитимные всплески искусственно задерживаются/режутся.
- [ ] Выбор: token bucket (нормальные пачки не наказываются), обоснованно.
- [ ] Условие обратного выбора: если приоритет — строго ровная нагрузка на downstream, тогда leaky bucket.
- [ ] Trade-off «допускать burst ↔ ровный поток» проговорён явно.

## ПОДСКАЗКА
Token bucket копит «право на всплеск» в простое и тратит его на пачку — идеально для «пачечного» клиентского трафика. Leaky bucket превращает любой всплеск в ровный ручеёк — хорошо для защиты downstream, но наказывает легитимные пачки.
