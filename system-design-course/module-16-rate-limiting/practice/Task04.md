# Задача 04 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Выберите ключ лимитирования под разные сценарии и обойдите ловушку NAT.

## ВХОДНЫЕ ДАННЫЕ
- (А) аутентифицированный API с тарифными планами (Free/Pro/Enterprise);
- (Б) анонимная форма входа (защита от brute-force паролей);
- (В) публичный поиск без авторизации, к которому ходят пользователи из больших корпоративных сетей и мобильных операторов (общий внешний IP).

## ТРЕБОВАНИЯ
- Для каждого сценария выберите ключ лимита (per-user/API-key, per-IP, глобально, комбинация) и обоснуйте.
- Для (В) объясните ловушку per-IP за NAT и как её смягчить.
- Для (Б) объясните, какой ключ корректнее (по аккаунту или по IP) и почему, возможно, оба.
- Назовите общий принцип выбора ключа.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: сценарий → ключ → обоснование + разбор ловушки NAT в (В) + принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → per-API-key/per-user (нужно для тарифных квот).
- [ ] (Б) → комбинация: per-account (защита конкретного аккаунта от подбора) + per-IP (защита от распределённого перебора), с осторожностью к NAT.
- [ ] (В) → per-IP грубо наказывает многих за NAT; смягчение — лимит помягче для анонимных + per-user после авторизации, либо дополнительные сигналы (cookie/device).
- [ ] Объяснена ловушка NAT (много пользователей за одним IP → бьём невиновных).
- [ ] Сформулирован принцип: ключ должен максимально точно идентифицировать «клиента» (per-user точнее per-IP).

## ПОДСКАЗКА
per-user/API-key — точно и нужно для квот, но требует аутентификации. per-IP — для анонимных, но за NAT бьёт многих. Часто комбинируют: анонимный per-IP (мягче) + per-user (точно) + глобальный потолок. Для brute-force полезны оба ключа сразу.
