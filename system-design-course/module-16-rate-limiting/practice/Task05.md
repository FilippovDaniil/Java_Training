# Задача 05 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Спроектируйте распределённый rate limiting для сервиса из многих инстансов.

## ВХОДНЫЕ ДАННЫЕ
API-сервис: 10 инстансов за балансировщиком. Требуемый лимит — 1000 запросов/с на пользователя (общий, не на инстанс). Сейчас каждый инстанс считает локально по 1000/с.

## ТРЕБОВАНИЯ
- Объясните, почему локальные счётчики дают неверный (завышенный) общий лимит.
- Предложите дизайн общего счётчика (Redis) с атомарным инкрементом.
- Обсудите trade-off: точность vs latency (хоп в Redis на каждый запрос) и зависимость/SPOF.
- Предложите компромисс, если строгая точность не критична.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз локальных счётчиков → дизайн общего счётчика → trade-off → компромисс.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: 10 инстансов × 1000 = 10 000/с фактически — в 10 раз выше задуманного.
- [ ] Дизайн: общий счётчик в Redis, атомарный инкремент (с TTL/окном), решение по лимиту централизованно.
- [ ] Trade-off: точность ценой сетевого хопа на каждый запрос (latency) + Redis как зависимость/узкое место (нужна HA).
- [ ] Компромисс: локальные счётчики с долей лимита + периодическая синхронизация, либо приближённый алгоритм (точность не критична для защиты).
- [ ] Отмечено: сам rate limiter не должен стать SPOF/узким местом.

## ПОДСКАЗКА
Главная ловушка: локальные счётчики не складываются в глобальный лимит (умножаются на число инстансов). Точный общий лимит = общий стор (Redis, атомарно), ценой хопа и зависимости. Если точность не критична — приближение с локальными счётчиками + сверка дешевле и быстрее.
