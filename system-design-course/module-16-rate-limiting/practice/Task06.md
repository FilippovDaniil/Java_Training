# Задача 06 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Спроектируйте корректный ответ API при превышении лимита и поведение клиента.

## ВХОДНЫЕ ДАННЫЕ
Публичный API ввёл rate limiting. Сейчас при превышении сервер просто возвращает 500 или молча обрывает соединение, и клиенты тут же повторяют запросы, усугубляя нагрузку.

## ТРЕБОВАНИЯ
- Укажите корректный HTTP-статус и заголовки при превышении лимита.
- Объясните роль Retry-After и информационных заголовков лимита.
- Опишите правильное поведение клиента при получении этого ответа (backoff).
- Объясните, чем плох текущий ответ (500/обрыв) для нагрузки и UX.
- Кратко: когда уместен soft limit (throttling/деградация) вместо жёсткого отказа.

## ФОРМАТ РЕЗУЛЬТАТА
Статус+заголовки → роль Retry-After → поведение клиента → критика текущего → про soft limit.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Статус: HTTP 429 Too Many Requests; заголовок Retry-After (когда повторять).
- [ ] Информационные заголовки лимита (limit/remaining/reset) помогают клиенту планировать.
- [ ] Клиент при 429 делает exponential backoff (модуль 15), а не немедленный повтор.
- [ ] Критика: 500/обрыв не сообщает «это лимит» и не даёт времени → клиент долбит сразу, усиливая нагрузку.
- [ ] Soft limit: throttling/урезанный ответ мягче для UX, когда жёсткий отказ нежелателен.

## ПОДСКАЗКА
Стандарт — 429 + Retry-After. Это говорит клиенту «притормози и повтори через N секунд». Без этого клиент ретраит мгновенно (хуже всего при перегрузке). На клиенте — backoff. Иногда мягче не отказать, а замедлить/урезать (graceful degradation).
