# Задача 07 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Рассчитайте параметры rate limiting так, чтобы они защищали реальную ёмкость downstream.

## ВХОДНЫЕ ДАННЫЕ
- За API стоит downstream (БД + внешний платёжный сервис), который устойчиво держит ~5 000 RPS суммарно.
- Клиентов (API-ключей) ~1 000 активных одновременно.
- Нужно: защитить downstream от перегрузки и обеспечить fair use; допустимы короткие всплески.

## ТРЕБОВАНИЯ
- Предложите глобальный лимит, согласованный с ёмкостью downstream (с запасом).
- Предложите per-key лимит (rate + размер burst у token bucket) с учётом числа клиентов.
- Проверьте, что суммарно лимиты не превышают ёмкость downstream в худшем случае.
- Объясните, почему нельзя ставить лимиты «с потолка» без привязки к downstream.

## ФОРМАТ РЕЗУЛЬТАТА
Глобальный лимит + per-key лимит (rate+burst) + проверка суммарной нагрузки + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Глобальный лимит ≤ ёмкости downstream с запасом (напр. ~4 000–4 500 RPS при потолке 5 000).
- [ ] per-key лимит задан с учётом числа клиентов (напр. rate, не дающий нескольким клиентам в сумме пробить глобальный потолок) + разумный burst.
- [ ] Проверено: в худшем случае суммарная нагрузка не превышает ёмкость downstream (глобальный потолок как страховка поверх per-key).
- [ ] Объяснено: лимит «с потолка» либо не защищает (слишком высокий), либо вредит UX (слишком низкий); привязка к измеренной ёмкости downstream (Часть 2).
- [ ] Числа трактуются как порядки.

## ПОДСКАЗКА
Лимиты выводят из ёмкости downstream, а не из головы. Глобальный потолок защищает downstream целиком; per-key обеспечивает fair use. Глобальный действует как страховка: даже если per-key лимиты теоретически суммируются выше потолка, глобальный не пустит больше ёмкости.
