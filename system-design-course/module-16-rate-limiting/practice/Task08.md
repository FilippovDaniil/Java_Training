# Задача 08 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне rate limiting и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Лимит — fixed window 1000/мин, этого достаточно, всплески неважны».
2. «Каждый из 20 инстансов считает лимит локально».
3. «Лимитируем только по IP, в том числе для авторизованного API с тарифами».
4. «При превышении возвращаем 403 без дополнительных заголовков».
5. «Один лимит на все эндпоинты, включая тяжёлый отчётный экспорт».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка сильнее всего подрывает саму защиту (лимит фактически не работает).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) fixed window → всплеск до ×2 на границе → sliding window, если всплески важны.
- [ ] (2) Локальные счётчики на 20 инстансов → фактический лимит ×20 → общий счётчик (Redis) или приближение со сверкой.
- [ ] (3) Только per-IP для API с тарифами → не различает клиентов + NAT → per-user/API-key (+ per-IP для анонимных).
- [ ] (4) 403 без Retry-After → клиент не знает, когда повторять → 429 + Retry-After.
- [ ] (5) Один лимит на всё → тяжёлый экспорт надо лимитировать жёстче → per-endpoint лимиты для дорогих операций.
- [ ] Отмечено: п.2 сильнее всего подрывает защиту (реальный лимит в 20 раз выше задуманного).

## ПОДСКАЗКА
Чек-лист модуля: sliding вместо fixed (если важны всплески); общий счётчик вместо локальных; per-user/API-key вместо голого per-IP; 429+Retry-After; отдельные лимиты на дорогие эндпоинты. Локальные счётчики — самая коварная ошибка: лимит молча умножается на число инстансов.
