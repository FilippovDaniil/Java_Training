# Задача 09 — Модуль 16: Rate limiting & throttling

## ЗАДАНИЕ
Спроектируйте многоуровневый rate limiting с тарифными квотами.

## ВХОДНЫЕ ДАННЫЕ
SaaS-API с тремя планами: Free (низкая квота), Pro (средняя), Enterprise (высокая). Нужно: защитить сервис в целом, обеспечить квоты по планам, жёстче лимитировать дорогие эндпоинты, защититься от анонимного abuse на публичных ручках.

## ТРЕБОВАНИЯ
- Спроектируйте уровни лимитов: глобальный (edge), per-plan (квоты), per-endpoint (дорогие операции), per-IP (анонимные).
- Объясните, как уровни сочетаются (порядок применения, что отсекает первым).
- Опишите, где хранить квоты планов и счётчики.
- Объясните, как квоты связаны с монетизацией и с защитой downstream.

## ФОРМАТ РЕЗУЛЬТАТА
Схема уровней лимитов + порядок применения + хранение квот/счётчиков + связь с монетизацией.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Уровни: глобальный потолок (edge) + per-plan квоты (Free/Pro/Enterprise) + per-endpoint (дорогие ручки жёстче) + per-IP для анонимных.
- [ ] Объяснён порядок: раньше всего отсекает edge/глобальный; затем per-plan/per-key; дорогие эндпоинты — отдельно.
- [ ] Квоты планов хранятся в конфиге/БД, счётчики — в общем сторе (Redis); связаны с API-ключом/планом.
- [ ] Квоты = продуктовый механизм (монетизация: выше план — выше лимит) и защита downstream одновременно.
- [ ] Уровни не противоречат: per-plan ≤ глобальный потолок.

## ПОДСКАЗКА
Реальный rate limiting многоуровневый: глобальный потолок (защита сервиса) + per-plan (квоты-монетизация) + per-endpoint (дорогие операции) + per-IP (анонимный abuse). Лимиты применяются от широкого к узкому; per-plan связан с API-ключом, счётчики — в Redis.
