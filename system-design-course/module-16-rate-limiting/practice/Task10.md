# Задача 10 — Модуль 16 (мини-проект): Rate limiting публичного API

## ЗАДАНИЕ
Спроектируйте полную систему rate limiting для публичного API. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Публичный API на выбор контекста (зафиксируйте): (а) платёжный API; (б) API соцсети/контента; (в) API геокодирования/поиска. Есть авторизованные клиенты (с тарифами) и анонимный трафик. Ёмкость downstream и нагрузку возьмите из расчётов модулей 05–07 (или задайте явно как допущение).

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Цели** — что защищаем (downstream, fair use, стоимость, anti-abuse, квоты).
2. **Алгоритм** — выбор (token bucket и др.) с обоснованием.
3. **Где** — точка лимитирования (edge/gateway/per-service), многоуровневость.
4. **Ключ** — per-user/API-key/IP/глобально + обработка NAT.
5. **Распределённость** — общий счётчик vs локальный, точность/latency.
6. **Ответ** — 429 + Retry-After + поведение клиента (backoff).
7. **Параметры** — лимиты, согласованные с ёмкостью downstream (расчёт); квоты планов.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов + схема прохождения запроса через лимиты.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирован контекст; ёмкость downstream/нагрузка заданы (своя оценка или допущение).
- [ ] Названы цели; выбран алгоритм с обоснованием.
- [ ] Точка лимитирования (edge раньше всего) и многоуровневость.
- [ ] Ключ(и) выбран(ы) с обработкой NAT; для тарифов — per-API-key.
- [ ] Решён распределённый rate limiting (общий счётчик/компромисс) с trade-off точность/latency.
- [ ] Ответ при превышении — 429 + Retry-After; на клиенте backoff.
- [ ] Параметры лимитов согласованы с ёмкостью downstream (расчёт); квоты планов; есть схема.

## ПОДСКАЗКА
Это синтез модуля и обязательный компонент публичного API. Хорошая система rate limiting отвечает на: что защищаем, каким алгоритмом, где (edge), по какому ключу (с оговоркой NAT), как считаем в кластере (общий счётчик), что отвечаем (429+Retry-After) и как лимиты привязаны к реальной ёмкости downstream. Дальше — модуль 17: поиск, автодополнение, гео (финал Части 3).
