## Ключ (эталонные ответы)

1. Чтобы защитить систему от перегрузки и abuse, обеспечить fair use (один клиент не съедает ёмкость всех), защитить downstream, контролировать стоимость и реализовать тарифные квоты.
2. **Rate limiting** защищает систему **от клиента** (ограничивает входящие запросы). **Backpressure** защищает систему **изнутри** (тормозит producer, когда consumer не успевает). Разные направления одной идеи.
3. Часто синонимы. Иногда throttling — замедление запросов сверх лимита (а не жёсткий отказ), rate limiting — собственно ограничение частоты с отказом сверх лимита.
4. Есть ведро ёмкостью B токенов, пополняется r токенов/с. Каждый запрос забирает токен; нет токенов → отказ. В среднем держит rate r, но допускает всплеск до B.
5. Накопленные в простое токены позволяют пройти короткой пачке сразу. Это хорошо, потому что реальный трафик «дёргается» (пачки после открытия приложения) — нормальные всплески не наказываются.
6. Запросы попадают в очередь и «вытекают» с постоянной скоростью → на выходе строго ровный поток. Уместен, когда нужно защитить чувствительный downstream от любых всплесков.
7. На стыке двух окон возможен всплеск почти до 2× лимита (полный лимит в конце одного окна + полный лимит в начале следующего за короткий интервал) — окна независимы.
8. Учитывает запросы за реальный скользящий интервал, а не за фиксированное окно, поэтому всплеск на границе попадает в подсчёт и режется. Нет «дыры» между окнами.
9. **Log** хранит timestamps всех запросов в окне — точно, но память растёт с числом запросов. **Counter** взвешивает текущее и прошлое окно (аппроксимация) — почти так же точно, но мало памяти.
10. **Token bucket** — прост, эффективен, допускает разумные всплески реального трафика, мало памяти. Дефолт для большинства API.
11. Как можно раньше на пути запроса — на edge / API gateway, чтобы отсечь лишнюю нагрузку до того, как она дойдёт до дорогих внутренних сервисов. Часто многоуровнево.
12. Потому что клиента можно обойти/подделать. Client-side — это вежливость и снижение лишних запросов, но реальную защиту делают на сервере (edge/gateway).
13. per-user / per-API-key, per-IP, глобально (на сервис), per-endpoint — и их комбинации.
14. За NAT/прокси много пользователей делят один внешний IP (офис, мобильный оператор) → лимит по IP наказывает многих невиновных. Грубая защита для анонимного трафика.
15. Для аутентифицированных API и тарифных квот: per-user/API-key точно идентифицирует клиента (в отличие от IP) и позволяет различать планы.
16. Каждый из N инстансов считает свой лимит независимо → суммарно лимит умножается на N (10 инстансов × 1000 = 10 000 вместо 1000). Глобальный лимит не соблюдается.
17. Использовать общий счётчик в централизованном сторе (Redis) с атомарным инкрементом, чтобы лимит был общим для всех инстансов; либо приближение с локальными счётчиками + периодической синхронизацией.
18. Точный общий лимит ценой сетевого хопа в Redis на каждый запрос (latency) и зависимости от Redis (узкое место/SPOF — нужна HA). Иногда жертвуют точностью ради скорости.
19. HTTP 429 Too Many Requests с заголовком Retry-After (и желательно X-RateLimit-Limit/Remaining/Reset), чтобы клиент понял, что произошло и когда повторять.
20. Сообщает клиенту, через сколько секунд можно повторить запрос. Без него клиент ретраит немедленно, усугубляя перегрузку.
21. Не повторять немедленно, а применить exponential backoff (растущие паузы), ориентируясь на Retry-After. Это разгружает сервис вместо «долбления».
22. Мягкая реакция на превышение: вместо жёсткого отказа (429) — замедление запросов или урезанный/деградированный ответ. Лучше для UX, когда полный отказ нежелателен.
23. Чтобы лимит реально защищал: слишком высокий лимит не спасает downstream, слишком низкий вредит UX. Лимиты выводят из измеренной ёмкости downstream (Часть 2), а не «с потолка».
24. Запросы не равны: дорогие операции (тяжёлый поиск, экспорт отчётов) нагружают систему сильнее и должны лимитироваться жёстче дешёвых. Per-endpoint лимиты учитывают эту разницу.
25. Цели защиты; алгоритм; точка лимитирования (edge, многоуровневость); ключ(и) с обработкой NAT; распределённый счётчик с trade-off; ответ 429+Retry-After и backoff клиента; параметры лимитов, привязанные к ёмкости downstream, и квоты планов.
