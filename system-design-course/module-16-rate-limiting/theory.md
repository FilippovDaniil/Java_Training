# Модуль 16 — Rate limiting & throttling

## Зачем этот модуль

Очередь (модуль 15) защищает обработку, сглаживая пики *внутри* системы. Но систему надо защищать и *снаружи* — от клиентов, которые шлют слишком много запросов: по ошибке (взбесившийся ретрай), злонамеренно (DDoS, brute-force, scraping) или просто потому что популярны. **Rate limiting** — это контроль того, сколько запросов клиент может сделать за единицу времени, и отказ сверх лимита.

Зачем это в дизайне. Без ограничения один клиент может исчерпать ёмкость на всех (нарушить **fair use**), обвалить downstream (каскадный отказ), взвинтить счёт (egress/compute, модуль 07) или подобрать пароль перебором. Rate limiting — обязательный компонент любого публичного API, основа тарифных квот и первая линия защиты от abuse.

На интервью «как защитите API от перегрузки/abuse?», «как сделаете тарифные планы?» — прямые сигналы к rate limiting. Этот модуль — про алгоритмы (token/leaky bucket, fixed/sliding window), где и по какому ключу лимитировать, распределённый rate limiting и корректный ответ клиенту (429/Retry-After).

---

## 1. Зачем ограничивать частоту

| Цель | Что предотвращает |
|------|-------------------|
| **Защита от перегрузки** | один всплеск не кладёт сервис |
| **Fair use** | один клиент не съедает ёмкость всех |
| **Защита downstream** | не передаём перегрузку в БД/внешние сервисы (каскад) |
| **Контроль стоимости** | abuse не взвинчивает счёт (egress/compute) |
| **Тарифные квоты** | разные планы = разные лимиты (монетизация) |
| **Anti-abuse** | DDoS, brute-force паролей, scraping, спам |

> Rate limiting защищает систему **от клиента** (в отличие от backpressure, который защищает от внутренней перегрузки). Это и техническая защита (стабильность), и продуктовый механизм (квоты тарифов). «Throttling» и «rate limiting» используют как синонимы; иногда throttling — замедление (а не жёсткий отказ) сверх лимита.

---

## 2. Алгоритмы rate limiting

| Алгоритм | Как работает | Burst | Точность | Память |
|----------|--------------|-------|----------|--------|
| **Token bucket** | токены пополняются с rate; запрос берёт токен, нет токенов → отказ | **допускает** всплеск (до размера ведра) | хорошая | низкая |
| **Leaky bucket** | запросы в очередь, «вытекают» с постоянной скоростью | сглаживает в ровный поток | хорошая | средняя |
| **Fixed window** | счётчик за фиксированное окно (минута), сброс в начале | **всплеск ×2 на границе** | грубая | очень низкая |
| **Sliding window log** | хранит timestamps запросов в скользящем окне | точно | точная | **высокая** |
| **Sliding window counter** | взвешенная комбинация текущего и прошлого окна | сглаживает границу | хорошая | низкая |

### Token bucket (самый популярный)
```
Ведро ёмкостью B токенов, пополняется r токенов/с.
Запрос: есть токен → пропустить, забрать токен; нет → отказ (429).
→ допускает burst до B, в среднем держит rate r.
```

### Проблема fixed window
```
Лимит 100/мин. Окно [12:00:00–12:00:59].
99 запросов в 12:00:59 + 100 в 12:01:00 = 199 запросов за ~1 секунду на границе окон.
→ фактический всплеск вдвое выше лимита.
```

> **Token bucket — дефолт**: прост, эффективен, допускает разумные всплески (реальный трафик «дёргается»). **Leaky bucket** — когда нужен строго ровный поток на выходе (защита чувствительного downstream). **Fixed window** прост, но имеет дыру на границе окна; **sliding window counter** убирает её при низкой памяти.

## 3. Где лимитировать

```
client (сам себя) → edge / API gateway → per-service → downstream
                          ↑ основное место (раньше всего отсекает)
```

| Уровень | Зачем |
|---------|-------|
| **Client-side** | вежливость, не главная защита (клиент можно обойти) |
| **Edge / API gateway** | главная точка: отсекает до того, как нагрузка дошла до сервисов |
| **Per-service** | защита конкретного сервиса/эндпоинта (разные лимиты на дорогие операции) |

> Лимитировать надо **как можно раньше** на пути запроса (edge/gateway), чтобы перегрузка не доходила до дорогих внутренних слоёв. Часто лимиты многоуровневые: глобальный на edge + точечные на тяжёлые эндпоинты. Client-side rate limiting — вежливость, но не защита (его обходят).

---

## 4. По какому ключу лимитировать

| Ключ | Когда | Ловушка |
|------|-------|---------|
| **per-user / per-API-key** | аутентифицированные API, тарифные квоты | требует идентификации |
| **per-IP** | анонимный трафик, базовая anti-abuse | **NAT/прокси**: много пользователей за одним IP → бьём невиновных |
| **глобально** | защита ёмкости сервиса в целом | не различает клиентов |
| **per-endpoint** | дорогие операции лимитируют жёстче | |

> Выбор ключа критичен. **per-IP за NAT** наказывает многих пользователей за одним адресом (офис, мобильный оператор) — грубая защита. **per-user/API-key** точнее и нужен для тарифных квот, но требует аутентификации. Часто комбинируют: per-IP для анонимных + per-user для авторизованных + глобальный потолок.

## 5. Распределённый rate limiting

Сервис из многих инстансов за балансировщиком: лимит должен быть **общим**, а не «по инстансу».

| Подход | Плюс | Минус |
|--------|------|-------|
| **Локальный счётчик на инстанс** | быстро, без сети | N инстансов × лимит → реальный лимит в N раз выше задуманного |
| **Централизованный счётчик** (Redis) | точный общий лимит | сетевой хоп на каждый запрос (latency), Redis — узкое место/зависимость |
| **Локальный + периодическая синхронизация** | компромисс | приближённая точность |

```
Лимит 1000/с, 10 инстансов:
  локально по 1000 каждому → фактически 10 000/с (в 10 раз больше!)
  общий счётчик в Redis → честные 1000/с (ценой хопа в Redis)
```

> Главная ловушка распределённого rate limiting: **локальные счётчики не складываются в глобальный лимит**. Для точного общего лимита нужен общий стор (Redis) с атомарным инкрементом — ценой latency и зависимости. Часто идут на компромисс: приближённый лимит с локальными счётчиками + периодической сверкой (точность не критична для защиты). Сам rate limiter не должен стать SPOF/узким местом.

---

## 6. Ответ при превышении

Клиент должен **понять**, что его ограничили и когда повторять:

```
HTTP 429 Too Many Requests
Retry-After: 5            ← через сколько секунд можно повторить
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: <ts>
```

> Стандарт — **HTTP 429** + заголовок **Retry-After**. Без них клиент не знает, что произошло и когда повторять → ретраит немедленно, усугубляя нагрузку. Хороший API сообщает лимит, остаток и время сброса. На стороне клиента — exponential backoff (модуль 15) при получении 429.

**Soft vs hard limits:** иногда вместо жёсткого отказа применяют *throttling* (замедление) или *graceful degradation* (урезанный ответ) — мягче для UX, чем 429.

---

## ⚠️ Подводные камни

- **Fixed window без понимания границы.** Всплеск до ×2 лимита на стыке окон; берите sliding window.
- **Локальные счётчики в кластере.** Реальный лимит = лимит × число инстансов; нужен общий счётчик.
- **per-IP за NAT.** Наказывает многих за одним адресом; для авторизованных — per-user/API-key.
- **Нет 429/Retry-After.** Клиент не знает, когда повторять → ретраит сразу, усиливая нагрузку.
- **Rate limiter как SPOF/узкое место.** Если общий счётчик упал/тормозит — страдает весь трафик; нужны HA и быстрый путь.
- **Один лимит на все запросы.** Дорогие операции (тяжёлый поиск) надо лимитировать жёстче дешёвых.
- **Синхронный Redis на каждый запрос без оглядки на latency.** Добавляет хоп; оцените цену или используйте локальный кэш счётчика.
- **Лимит без связи с ёмкостью downstream.** Лимит должен защищать реальный потолок сервиса/БД, а не быть «с потолка».
