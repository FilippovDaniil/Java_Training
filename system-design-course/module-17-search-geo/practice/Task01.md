# Задача 01 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Объясните, почему SQL `LIKE` не годится для поиска, и спроектируйте решение через inverted index.

## ВХОДНЫЕ ДАННЫЕ
Интернет-магазин: 10 млн товаров, нужен поиск по названию и описанию с ранжированием по релевантности. Сейчас реализовано как `WHERE name LIKE '%query%' OR description LIKE '%query%'`, поиск тормозит.

## ТРЕБОВАНИЯ
- Объясните, почему текущий `LIKE`-подход медленный и неполноценный (минимум 3 причины).
- Предложите архитектуру с inverted index / поисковым движком.
- Опишите, что хранится в движке, а что в основной БД.
- Укажите, как товар попадает в индекс при создании/изменении.

## ФОРМАТ РЕЗУЛЬТАТА
Причины негодности LIKE → архитектура с поисковым движком → распределение БД/индекс → индексация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Названы причины: `LIKE '%...%'` не использует индекс (full scan); нет ранжирования; нет языковой обработки (стемминг/синонимы); не масштабируется на 10 млн.
- [ ] Предложен inverted index / движок (Elasticsearch/OpenSearch).
- [ ] Распределение: источник правды (товары) — в БД; поисковый индекс — в движке (производный).
- [ ] Описана индексация: при создании/изменении товара событие → обновление индекса (модуль 15).
- [ ] Явно: индекс — отдельное eventually consistent хранилище, не замена БД.

## ПОДСКАЗКА
`LIKE '%...%'` всегда читает всю таблицу и не ранжирует. Поиск как функция продукта = inverted index в движке. БД остаётся источником правды, движок — производный индекс, синхронизируемый через события.
