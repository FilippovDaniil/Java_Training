# Задача 03 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Спроектируйте синхронизацию поискового индекса с источником правды (БД) и учтите её ограничения.

## ВХОДНЫЕ ДАННЫЕ
Товары хранятся в реляционной БД (источник правды), поиск — в Elasticsearch. Требуется: при создании/изменении/удалении товара индекс обновляется; при рассинхроне индекс можно пересобрать; пользователи терпят небольшую задержку появления в поиске.

## ТРЕБОВАНИЯ
- Опишите механизм синхронизации БД → индекс (через события/CDC и очередь).
- Объясните, почему индекс — eventually consistent и что такое индексационный lag.
- Опишите процедуру полного перестроения индекса из БД (зачем нужна).
- Назовите, что считается источником правды при расхождении и почему.

## ФОРМАТ РЕЗУЛЬТАТА
Механизм синхронизации + про lag/eventual + перестроение индекса + источник правды.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Синхронизация: изменение в БД → событие/CDC → очередь → индексатор обновляет ES (модуль 15).
- [ ] Объяснено: индексация near-real-time → между записью и появлением в поиске есть lag; индекс eventually consistent.
- [ ] Описано полное перестроение индекса из БД (на случай рассинхрона/смены схемы индекса).
- [ ] Источник правды — БД; индекс производный, при расхождении пересобирается из БД.
- [ ] Учтено в UX: свежесозданный товар появляется в поиске с небольшой задержкой.

## ПОДСКАЗКА
Индекс — производная копия. Синхронизация через события/CDC + очередь (модуль 15). Lag неизбежен → eventual consistency, это нормально для поиска. Обязательно предусмотреть полное перестроение из БД: индекс можно потерять/пересоздать, БД — нельзя.
