# Задача 05 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Спроектируйте кэширование автодополнения и оцените эффект на latency и нагрузку.

## ВХОДНЫЕ ДАННЫЕ
Typeahead-сервис: огромная доля запросов приходится на короткие популярные префиксы («a», «th», «sys»). Каждый символ ввода — это запрос. Нагрузка большая, latency критична.

## ТРЕБОВАНИЯ
- Объясните, почему распределение префиксов идеально подходит под кэш (80/20).
- Спроектируйте кэш популярных префиксов: что кэшировать, TTL, где (client/CDN/distributed).
- Оцените эффект: как кэш снижает latency и нагрузку на trie-сервис.
- Учтите инвалидацию: когда подсказки меняются (новые тренды).

## ФОРМАТ РЕЗУЛЬТАТА
Почему кэш применим (80/20) + дизайн кэша (что/TTL/уровень) + эффект + инвалидация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: короткие префиксы дают основную долю запросов (80/20, модуль 09) → высокий hit rate кэша.
- [ ] Кэш: пары «префикс → топ-K подсказок», умеренный TTL; уровни — client/edge для самых частых + distributed (Redis).
- [ ] Эффект: популярные префиксы отдаются из кэша мгновенно, разгружая trie-сервис (меньше нагрузки, ниже latency).
- [ ] Инвалидация: TTL/событие при обновлении популярности (новые тренды), окно устаревания терпимо.
- [ ] Связь с модулем 09 (кэширование) явная.

## ПОДСКАЗКА
Префиксы подчиняются 80/20: горстка коротких префиксов = большинство запросов. Кэш «префикс → топ-K» с высоким hit rate снимает основную нагрузку. Это прямое применение модуля 09: горячий набор маленький, кэш эффективен. TTL/событие держат подсказки достаточно свежими.
