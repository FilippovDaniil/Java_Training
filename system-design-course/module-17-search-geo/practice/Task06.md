# Задача 06 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Спроектируйте геопоиск «в радиусе»: объясните негодность линейного перебора и выберите подход.

## ВХОДНЫЕ ДАННЫЕ
Сервис доставки: нужно находить рестораны/курьеров в радиусе ~3 км от пользователя. Объектов — миллионы, запросов «рядом со мной» — очень много. Сейчас на каждый запрос считается расстояние до всех объектов.

## ТРЕБОВАНИЯ
- Объясните, почему линейный перебор O(N) и обычный индекс по (lat,lng) не годятся.
- Выберите подход (geohash / quadtree / S2 / H3 или геоиндекс БД) и обоснуйте.
- Опишите, как с его помощью выполняется запрос «в радиусе R».
- Назовите, где хранить гео-данные (специализированный индекс) и про синхронизацию.

## ФОРМАТ РЕЗУЛЬТАТА
Почему перебор плох → выбор подхода → выполнение radius-запроса → хранение/синхронизация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: линейный перебор O(N) не масштабируется; обычный B-tree-индекс не отвечает на 2D-запрос «в радиусе».
- [ ] Выбран подход (например, geohash или S2/Redis GEO) с обоснованием.
- [ ] Описано выполнение: определить ячейку(и) нужной точности, взять кандидатов из ячейки + соседних, отфильтровать по точному расстоянию.
- [ ] Гео-данные — в специализированном индексе (Redis GEO/ES geo/PostGIS), синхронизируемом с источником.
- [ ] Учтено: кандидаты из ячеек — приближение, финальная фильтрация по реальному расстоянию.

## ПОДСКАЗКА
2D-поиск не ложится на 1D-индекс, а перебор всех точек — O(N). Пространственные структуры (geohash/quadtree/S2) сводят «рядом» к ячейкам: берём свою ячейку + соседние как кандидатов, потом фильтруем по точному расстоянию. Хранят в гео-индексе, синхронизируемом как поисковый.
