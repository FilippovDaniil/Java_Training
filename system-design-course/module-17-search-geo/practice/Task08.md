# Задача 08 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Спроектируйте шардирование и индексацию гео-данных под nearby-запросы для движущихся объектов.

## ВХОДНЫЕ ДАННЫЕ
Сервис такси: миллионы водителей, координаты обновляются каждые несколько секунд; запросы «ближайшие водители» — постоянно; нагрузка неравномерна (центры мегаполисов — плотные «горячие» зоны).

## ТРЕБОВАНИЯ
- Предложите гео-индекс/структуру с дешёвым обновлением позиции (движущиеся объекты).
- Предложите шардирование гео-данных (по ячейке/региону) и обработку nearby-запросов.
- Объясните проблему «горячей зоны» центра города и как её смягчить (связь с celebrity problem).
- Учтите синхронизацию/частые апдейты и их нагрузку.

## ФОРМАТ РЕЗУЛЬТАТА
Выбор структуры (дешёвый апдейт) + шардирование + nearby-запрос + смягчение hot zone.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбрана структура с дешёвым обновлением позиции (напр. Redis GEO / geohash-ячейки), т.к. координаты часто меняются.
- [ ] Шардирование по гео-ячейке/региону; nearby = своя ячейка + соседние → фильтр по расстоянию.
- [ ] Проблема hot zone: плотный центр города перегружает ячейку/шард (гео-версия celebrity problem, модуль 13).
- [ ] Смягчение: мельче ячейки в плотных зонах (адаптивная гранулярность/quadtree), кэш, доп. реплики горячих ячеек.
- [ ] Учтена нагрузка частых апдейтов позиции (выбор структуры с дешёвой записью).

## ПОДСКАЗКА
Движущиеся объекты → нужен дешёвый апдейт (Redis GEO/ячейки). Шардирование по ячейке естественно, но центр города = hot zone (как celebrity problem, модуль 13): лечат адаптивной гранулярностью (quadtree мельчит плотное), кэшем и репликами. Частые апдейты позиции — отдельная статья нагрузки.
