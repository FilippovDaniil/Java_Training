# Задача 09 — Модуль 17: Поиск, автодополнение и геосервисы

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне поиска/автодополнения/гео и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Поиск по 50 млн товаров делаем через `LIKE '%query%'` в основной БД».
2. «Поисковый движок — наш источник правды, пишем сразу только в него».
3. «Автодополнение: на каждый символ обходим всё поддерево trie и сортируем на лету».
4. «Ближайшие объекты ищем, считая расстояние до всех точек в БД».
5. «Geohash берём фиксированной длины и ищем только в своей ячейке».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка опаснее всего для корректности данных, а какая — для производительности.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + классификация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) `LIKE` на 50 млн → full scan, нет ранжирования → inverted index/движок.
- [ ] (2) Движок как источник правды → потеря данных (индекс производный, eventually consistent) → источник правды в БД, индекс синхронизируется.
- [ ] (3) Обход поддерева на каждый символ → не уложиться в latency → предрасчёт топ-K в узлах + кэш.
- [ ] (4) Расстояние до всех точек → O(N) → geohash/quadtree/S2 (ячейки + фильтр).
- [ ] (5) Фиксированная длина + только своя ячейка → граничная проблема (теряем близкие точки) → проверять соседние ячейки, подбирать гранулярность под радиус.
- [ ] Классифицировано: (2) — корректность (источник правды); (1)/(3)/(4) — производительность; (5) — корректность результатов поиска.

## ПОДСКАЗКА
Чек-лист модуля: inverted index вместо LIKE; индекс не источник правды; предрасчёт topK + кэш для typeahead; geohash/quadtree вместо перебора; соседние ячейки против граничной проблемы. Самая опасная для данных — «движок как источник правды».
