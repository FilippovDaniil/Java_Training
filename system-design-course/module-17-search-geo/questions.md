## Ключ (эталонные ответы)

1. `LIKE '%...%'` не использует B-tree-индекс (full scan по всей таблице), не ранжирует по релевантности, не делает языковую обработку (стемминг/синонимы) и не масштабируется на миллионы документов.
2. Структура «term → список документов, где он встречается» (как предметный указатель книги). Позволяет искать «взять список по терму» вместо просмотра всех документов.
3. Токенизация (разбить на слова) → нормализация (lowercase) → стемминг/лемматизация (designing→design) → удаление стоп-слов (the, a) → запись термов в индекс.
4. Стемминг объединяет формы одного слова («designing/designed/design») → запросы находят все варианты. Стоп-слова (частые бесполезные) убирают, чтобы не раздувать индекс и не портить релевантность.
5. Запрос токенизируется так же, как документы, затем по каждому терму берутся списки документов и комбинируются (пересечение для AND), после чего результаты ранжируются по релевантности.
6. **TF-IDF** — мера релевантности: TF (частота терма в документе, чаще → релевантнее) × IDF (обратная частота по корпусу, реже слово вообще → весомее, т.к. лучше различает документы).
7. **BM25** — улучшение TF-IDF с насыщением частоты (после некоторого TF прирост убывает) и нормализацией по длине документа. Даёт более качественное ранжирование, стандарт в Lucene/ES.
8. Индекс — производная копия данных под поиск (eventually consistent). Источник правды — основная БД; при расхождении индекс пересобирается из неё.
9. Через события/CDC и очередь: изменение в БД → событие → индексатор обновляет движок (модуль 15). Индексация near-real-time.
10. Задержка между записью в БД и появлением данных в поисковом индексе. Из-за неё свежесозданный объект находится в поиске не мгновенно (eventual consistency).
11. На случай рассинхрона индекса с БД, смены схемы индекса, потери/повреждения индекса. Индекс можно безопасно пересоздать из источника правды; данные не теряются.
12. Elasticsearch/OpenSearch, Apache Solr (оба поверх Lucene) и подобные — они реализуют inverted index, ранжирование и распределённость из коробки.
13. Подсказки по префиксу на каждый вводимый символ. Latency-критично, потому что подсказки должны появляться быстрее, чем человек печатает следующий символ (десятки мс).
14. **Trie** (префиксное дерево): спуск по символам префикса за его длину даёт поддерево всех продолжений — естественно для префиксного поиска.
15. Хранить в каждом узле trie предрассчитанный топ-K популярных предложений, чтобы не обходить всё поддерево на каждый запрос; обновлять топ-K в фоне при изменении популярности.
16. Потому что вычислять топ-K обходом поддерева на каждый символ слишком медленно для latency-бюджета. Предрасчёт топ-K в узлах + кэш популярных префиксов (80/20) дают мгновенный ответ.
17. Он O(N) — считать расстояние до всех объектов на каждый запрос не масштабируется при миллионах точек и многих запросах.
18. «В радиусе» — двумерный запрос, а B-tree-индекс одномерный. Индекс по (lat,lng) не позволяет эффективно выбрать точки внутри круга/прямоугольника.
19. **Geohash** кодирует координаты (lat,lng) в строку так, что у близких точек общий префикс. Это сводит 2D-поиск к префиксному (1D): берут точки из нужной ячейки.
20. Две близкие точки по разные стороны границы ячейки имеют разные префиксы и «не находят» друг друга. Решение — проверять не только свою ячейку, но и 8 соседних, затем фильтровать по точному расстоянию.
21. По радиусу запроса: длиннее geohash → мельче ячейка. Подбирают так, чтобы размер ячейки был соизмерим с радиусом (баланс «лишние кандидаты при крупной ↔ много ячеек при мелкой»).
22. Quadtree (рекурсивное деление на квадранты, мельчит плотные зоны), S2 (Google, кривая Гильберта → 1D-ключи), H3 (Uber, гексагональная сетка), геоиндексы БД (PostGIS R-tree/GiST, Redis GEO, ES geo).
23. Плотная зона (центр мегаполиса) перегружает ячейку/шард — гео-версия celebrity problem. Смягчают адаптивной гранулярностью (quadtree мельчит плотное), кэшем и репликами горячих ячеек.
24. Все три требуют специальной структуры данных (inverted index / trie / geohash) и обычно отдельного eventually consistent хранилища, синхронизируемого с источником правды через события, с кэшем популярного и шардированием.
25. Полнотекстовый поиск (inverted index/движок, конвейер, ранжирование); индексация (CDC/очередь, lag, перестроение, индекс ≠ источник правды); автодополнение (trie + предрасчёт топ-K + кэш); геопоиск (geohash/quadtree/S2, граничная проблема); шардирование/хранение и hot zone; кэширование популярного; схема потоков.
