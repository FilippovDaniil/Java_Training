# Модуль 17 — Поиск, автодополнение и геосервисы

## Зачем этот модуль

Реляционная БД отлично достаёт запись по ключу, но три задачи ей не по силам на масштабе: **полнотекстовый поиск** («найди статьи про system design»), **автодополнение** («показать подсказки, пока пользователь печатает») и **геопоиск** («рестораны в радиусе 2 км»). Все три ломают обычный B-tree-индекс: `LIKE '%...%'` не использует индекс и делает full scan; подсказки нужны за миллисекунды на каждый нажатый символ; «ближайшие точки» — это 2D-запрос, который не ложится на одномерный индекс.

Эти три блока завершают Часть 3 и постоянно встречаются в референсных архитектурах (поиск в e-commerce, typeahead в любом UI, «рядом со мной» в такси/доставке). Объединяет их одна идея: для специальной задачи нужна **специальная структура данных** (inverted index, trie, geohash) и обычно **отдельное хранилище** (поисковый движок), синхронизируемое с источником правды.

Этот модуль — про обратный индекс и ранжирование, trie для typeahead и геоиндексы (geohash/quadtree/S2).

---

## 1. Полнотекстовый поиск: почему не SQL LIKE

```
SELECT * FROM articles WHERE body LIKE '%system design%';
```
| Проблема LIKE | Почему |
|---------------|--------|
| Full scan | `LIKE '%...%'` не использует B-tree-индекс → читает всю таблицу |
| Нет релевантности | возвращает совпадения, но не ранжирует по важности |
| Нет языковой обработки | не знает про стемминг, синонимы, опечатки |
| Не масштабируется | на миллионах документов — неприемлемо медленно |

> `LIKE` годится для разовых запросов по малой таблице, но не для поиска как функции продукта. Полнотекстовый поиск требует другой структуры — **обратного индекса** — и обычно отдельного движка (Elasticsearch/OpenSearch, Solr — поверх Lucene).

---

## 2. Inverted index (обратный индекс)

Вместо «документ → его слова» хранят «слово → список документов» (как предметный указатель в книге):

```
Документы:  d1="system design basics"   d2="design patterns"   d3="system scaling"

Inverted index (term → postings):
  system  → [d1, d3]
  design  → [d1, d2]
  basics  → [d1]
  scaling → [d3]

Запрос "system design" → пересечение [d1,d3] ∩ [d1,d2] = [d1]
```

**Конвейер индексации текста:**
```
текст → токенизация (разбить на слова) → нормализация (lowercase) →
        стемминг (designing→design) → удаление стоп-слов (the, a) → термы в индекс
```

| Шаг | Зачем |
|-----|-------|
| Токенизация | разбить на единицы поиска |
| Нормализация | «Design» = «design» |
| Стемминг/лемматизация | «designing/designed» → «design» |
| Стоп-слова | убрать частые бесполезные слова |

> Inverted index превращает поиск из «просмотреть все документы» в «взять список по терму» — поэтому он быстр на масштабе. Качество поиска во многом определяется конвейером обработки текста (стемминг, синонимы, языки), а не только структурой индекса.

---

## 3. Ранжирование (релевантность)

Найти документы мало — надо отсортировать по релевантности. Классика — **TF-IDF** и **BM25**:

```
TF (term frequency)        — чаще слово в документе → релевантнее
IDF (inverse doc frequency)— реже слово во всех документах → весомее (редкие слова важнее частых)
BM25 — улучшенная версия TF-IDF (с насыщением и нормализацией по длине)
```

> **TF-IDF/BM25** — базовые алгоритмы ранжирования: документ тем релевантнее, чем чаще в нём искомое слово (TF) и чем оно реже встречается вообще (IDF — редкое слово различает документы сильнее). На практике к этому добавляют сигналы (свежесть, популярность, персонализация, ML-ранжирование), но понимать TF-IDF/BM25 на интервью обязательно.

## 4. Индексация и согласованность: индекс ≠ источник правды

Поисковый индекс — **вторичное** хранилище, производное от основной БД. Его надо синхронизировать:

```
БД (источник правды) ──изменение──> [очередь / CDC] ──> обновление индекса (ES)
                                                          ↑ near-real-time, есть lag
```

| Свойство | Следствие |
|----------|-----------|
| Индекс производный | при расхождении источник правды — БД, индекс перестраивается |
| Индексация near-real-time | между записью в БД и появлением в поиске есть задержка (lag) |
| Синхронизация через CDC/события | изменения БД → поток → индексатор (модуль 15) |

> Индекс — **eventually consistent** копия данных под поиск. Это нормально: пользователь терпит, что только что созданная статья появится в поиске через секунды. Но проектировать надо осознанно: индекс не источник правды, синхронизация идёт через очередь/CDC, lag существует. Перестроение индекса из БД должно быть возможно (на случай рассинхрона).

## 5. Автодополнение (typeahead)

Подсказки по префиксу за миллисекунды на каждый символ. Структура — **trie** (префиксное дерево):

```
trie для [car, card, care, cat]:
        c
        └─ a
           ├─ r ──(car)
           │   ├─ d ──(card)
           │   └─ e ──(care)
           └─ t ──(cat)

Префикс "car" → спуститься по c→a→r → собрать поддерево [car, card, care]
```

| Приём | Зачем |
|-------|-------|
| Trie по префиксу | быстрый поиск всех продолжений префикса |
| Топ-K в узлах | хранить в узле заранее посчитанные топ-K популярных предложений → не обходить всё поддерево |
| Кэш популярных префиксов | частые запросы («a», «th») отдавать из кэша (модуль 09) |
| Ограничение длины/числа подсказок | latency-бюджет на каждый символ |

> Typeahead жёстко **latency-критичен**: подсказки нужны быстрее, чем человек печатает следующий символ (десятки мс). Поэтому топ-K результатов часто **предрассчитывают** и хранят в узлах trie или в кэше, а не вычисляют на лету. Ранжирование подсказок — по популярности/частоте запросов.

## 6. Геосервисы: «найти ближайшее»

Задача: «объекты в радиусе R» или «K ближайших». Проблема — координаты двумерны, а индексы одномерны:

```
❌ Линейный перебор: посчитать расстояние до всех N точек → O(N), не масштабируется
❌ Обычный индекс по (lat, lng): не отвечает на «в радиусе» эффективно (2D-запрос)
```

**Решения — пространственные структуры/кодирования:**

| Подход | Идея |
|--------|------|
| **Geohash** | кодирует (lat,lng) в строку; **общий префикс = близость**; запрос по префиксу |
| **Quadtree** | рекурсивно делит плоскость на 4 квадранта; плотные области делятся мельче |
| **S2** (Google) | проекция сферы на кривую Гильберта → 1D-ключи с сохранением близости |
| **H3** (Uber) | гексагональная сетка ячеек (равные соседи) |
| **БД-геоиндексы** | PostGIS (R-tree/GiST), Redis GEO, Elasticsearch geo |

```
Geohash: близкие точки имеют общий префикс
  точка1 = "u4pruyd"   точка2 = "u4pruyf"   → общий префикс "u4pruy" → рядом
Запрос "в радиусе": берём ячейку нужной точности + соседние ячейки.
```

> Geohash сводит 2D-поиск к префиксному (1D) — близкие точки попадают в одну/соседние ячейки. **Граничная проблема:** две близкие точки по разные стороны границы ячейки имеют разные префиксы → надо проверять **соседние ячейки**, а не только свою. Гранулярность (длина geohash) задаёт размер ячейки: слишком крупная → много лишних кандидатов, слишком мелкая → много ячеек для проверки.

---

## 7. Шардирование и индексация гео-данных

- Шардирование по гео-ячейке/региону: данные региона рядом (но «звёздные» зоны — центр мегаполиса — дают hotspot, модуль 13).
- Геоиндекс в специализированном хранилище (Redis GEO, ES geo, PostGIS), синхронизируемом с источником (как поисковый индекс).
- Для «движущихся» объектов (такси) — частые обновления позиции → выбор структуры с дешёвым апдейтом.

> Геопоиск — это поиск + специальная структура + (часто) отдельное хранилище, с теми же оговорками про синхронизацию и lag, что и текстовый поиск. Центр города как hot zone — гео-версия celebrity problem (модуль 13).

---

## ⚠️ Подводные камни

- **SQL `LIKE`/full scan для поиска.** Не масштабируется, нет ранжирования; нужен inverted index/движок.
- **Поисковый индекс как источник правды.** Он производный и eventually consistent; правда — в БД.
- **Игнорировать lag индексации.** Свежие данные появляются в поиске с задержкой; учитывать в UX.
- **Нет пути перестроения индекса.** При рассинхроне индекс надо уметь пересобрать из БД.
- **Автодополнение без trie/кэша и предрасчёта.** Вычисление на лету не уложится в latency-бюджет.
- **Линейный перебор для «ближайших».** O(N) не масштабируется; нужны geohash/quadtree/S2.
- **Geohash без проверки соседних ячеек.** Граничная проблема: близкие точки в разных ячейках теряются.
- **Неверная гранулярность geohash.** Слишком крупная → лишние кандидаты; слишком мелкая → много ячеек.
- **Гео-hotspot центра города.** Плотные зоны перегружают шард/ячейку (как celebrity problem).
