# Задача 02 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Выберите тип хранения (row-oriented vs column-oriented) под запросы и объясните, почему колоночное быстро для аналитики.

## ВХОДНЫЕ ДАННЫЕ
Таблица `events(id, user_id, event_type, country, ts, amount)`, миллиарды строк. Два запроса:
- (А) `SELECT * FROM events WHERE id = 12345` (точечный);
- (Б) `SELECT country, SUM(amount) FROM events GROUP BY country` (аналитический).

## ТРЕБОВАНИЯ
- Для каждого запроса укажите, какое хранение (row/column) эффективнее и почему.
- Объясните три причины, по которым колоночное хранение быстро для (Б).
- Объясните, почему колоночное плохо подходит для точечных update отдельной строки.
- Сделайте вывод о том, какое хранилище под какую нагрузку.

## ФОРМАТ РЕЗУЛЬТАТА
Запрос → хранение → причина + три причины скорости column + минус column для update + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → row store (вся строка рядом, точечный доступ быстр); (Б) → column store.
- [ ] Три причины скорости column для (Б): читает только нужные колонки (country, amount, не всю строку); отличное сжатие (однотипные значения колонки); векторизация операций над колонкой.
- [ ] Объяснено: точечный update строки в column store дорог (значения строки разбросаны по колонкам) → column для OLAP, не OLTP.
- [ ] Вывод: row store → OLTP, column store → OLAP.
- [ ] Связь со сжатием как фактором IO/стоимости.

## ПОДСКАЗКА
Аналитика трогает мало колонок из многих — column store читает только их. Однотипные значения колонки сжимаются в разы (меньше IO). Векторизация ускоряет агрегации. Но строка «размазана» по колонкам → точечная запись дорога.
