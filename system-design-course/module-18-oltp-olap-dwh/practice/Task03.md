# Задача 03 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Диагностируйте проблему «аналитика на проде OLTP» и предложите архитектурное решение.

## ВХОДНЫЕ ДАННЫЕ
Симптом: каждую ночь и в часы построения отчётов latency пользовательских операций (создание заказа) резко растёт, появляются таймауты. Расследование: аналитики и BI-дашборды выполняют тяжёлые агрегатные запросы напрямую на прод-БД заказов (OLTP).

## ТРЕБОВАНИЯ
- Объясните механику проблемы (почему аналитический запрос ломает OLTP-latency).
- Предложите варианты разделения (read-реплика для лёгкого / DWH+ETL для тяжёлого) и когда какой.
- Опишите, что меняется для аналитиков (свежесть данных, лаг).
- Нарисуйте целевую архитектуру.

## ФОРМАТ РЕЗУЛЬТАТА
Механика проблемы → варианты разделения + когда какой → эффект на свежесть → диаграмма.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Механика: тяжёлый скан/агрегация конкурирует за CPU/IO/память с пользовательскими операциями → их latency растёт.
- [ ] Варианты: лёгкая аналитика → read-реплика (свежо, умеренный объём); тяжёлая/историческая → отдельный DWH через ETL/ELT.
- [ ] Объяснено: при DWH данные не realtime (лаг конвейера) — аналитики работают с задержкой, это приемлемо.
- [ ] Диаграмма: OLTP → реплика / → ETL → DWH → BI.
- [ ] Прод OLTP остаётся чистым под пользователей.

## ПОДСКАЗКА
Корень — общие ресурсы: аналитический скан забивает БД, страдают пользователи. Лёгкое уносят на реплику, тяжёлое/историческое — в DWH через конвейер (модуль 20). Цена — лаг свежести в DWH, но это нормально для аналитики.
