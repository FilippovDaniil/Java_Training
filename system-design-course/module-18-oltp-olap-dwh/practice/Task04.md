# Задача 04 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Спроектируйте star schema для аналитики заданного домена.

## ВХОДНЫЕ ДАННЫЕ
Розничная сеть. Нужно анализировать продажи в разрезах: по времени (день/месяц/год), по товару (категория/бренд), по магазину (город/регион), по способу оплаты. Метрики: количество, сумма, скидка.

## ТРЕБОВАНИЯ
- Определите центральную fact table: какие метрики и какие ключи измерений.
- Определите dimension tables и их атрибуты.
- Нарисуйте star schema (факты в центре, измерения по краям).
- Покажите, как запрос «выручка по категориям и регионам за месяц» использует эту схему.

## ФОРМАТ РЕЗУЛЬТАТА
fact table (метрики+ключи) + dimension tables (атрибуты) + диаграмма звезды + пример запроса.

## КРИТЕРИИ ПРИЁМКИ
- [ ] fact_sales: метрики (quantity, amount, discount) + внешние ключи (date_id, product_id, store_id, payment_id).
- [ ] Измерения: dim_date (день/месяц/год), dim_product (категория/бренд), dim_store (город/регион), dim_payment.
- [ ] Диаграмма звезды: fact в центре, измерения по краям.
- [ ] Пример запроса: агрегация fact_sales с join к dim_product и dim_store, group by категория/регион, фильтр по dim_date.
- [ ] Факты = что измеряем (числа), измерения = в каком разрезе режем (контекст).

## ПОДСКАЗКА
Факты — это «что произошло и сколько» (числовые метрики + ключи). Измерения — «в каком разрезе смотрим» (контекст для group by/фильтра). Запрос = агрегация факта + join к нужным измерениям. Денормализация измерений (звезда) ускоряет аналитические join'ы.
