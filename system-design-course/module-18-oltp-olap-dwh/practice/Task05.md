# Задача 05 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Выберите хранилище под аналитическую задачу и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) интерактивные BI-дашборды по агрегированным продажам, запросы редкие, но тяжёлые; данные интегрируются из 5 источников;
- (Б) лёгкая оперативная статистика «заказы за сегодня» для админки, нужна почти свежая, объём небольшой;
- (В) ad-hoc исследовательские запросы аналитиков по сырым событиям, объём огромный.

## ТРЕБОВАНИЯ
- Для каждой задачи выберите подход (read-реплика OLTP / колоночный DWH / др.) и обоснуйте.
- Учтите свежесть данных, объём, тип запросов.
- Назовите неудачный выбор для каждой.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: задача → хранилище → обоснование (свежесть/объём/тип) → анти-выбор.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → колоночный DWH (тяжёлая аналитика, интеграция источников, исторические данные); анти-выбор — прод OLTP.
- [ ] (Б) → read-реплика OLTP (свежесть важна, объём мал, лёгкие запросы); анти-выбор — полноценный DWH (избыточно, лаг).
- [ ] (В) → колоночное аналитическое хранилище/DWH (огромный объём, сканы); анти-выбор — row-store OLTP.
- [ ] Выбор увязан со свежестью, объёмом и типом запросов.
- [ ] Названы анти-выборы с причинами.

## ПОДСКАЗКА
Свежесть + малый объём + лёгкие запросы → read-реплика. Тяжёлая/историческая/интегрированная аналитика → колоночный DWH. Не каждой аналитике нужен DWH (для «заказы за сегодня» реплика проще), и не всякую аналитику тянет реплика.
