# Задача 07 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне аналитики и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «BI-дашборды ходят запросами прямо в прод-БД заказов — так данные всегда свежие».
2. «В DWH кладём те же нормализованные таблицы, что в OLTP, ничего не меняя».
3. «Для аналитики берём ту же row-store БД, что и для транзакций».
4. «Обещаем бизнесу real-time-аналитику, хотя данные грузятся в DWH раз в сутки».
5. «Оперативную метрику „заказов за минуту“ для админки считаем тяжёлым сканом в DWH с суточным лагом».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) BI на проде OLTP → убивает latency пользователей → DWH/реплика.
- [ ] (2) Нормализованные OLTP-таблицы в DWH → медленные аналитические join'ы → dimensional model (star schema).
- [ ] (3) Row store под аналитику → медленные агрегации → колоночное хранилище.
- [ ] (4) «Real-time» при суточной загрузке → ложные ожидания → честно коммуницировать лаг или внедрить стриминг (модуль 20).
- [ ] (5) Свежая оперативная метрика из DWH с суточным лагом → данные неактуальны → считать на реплике/OLTP (свежесть), не в DWH.
- [ ] Классифицировано: (1) бьёт по пользователям; (4)/(5) — по доверию/корректности данных.

## ПОДСКАЗКА
Чек-лист модуля: не гонять BI на прод; star schema в DWH; колоночное под аналитику; честно про лаг; свежую оперативную метрику — на реплике, а не в DWH. Пункты 4–5 — про несоответствие свежести задаче.
