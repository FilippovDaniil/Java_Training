# Задача 08 — Модуль 18: OLTP vs OLAP, DWH

## ЗАДАНИЕ
Оцените объём DWH и выигрыш колоночного сжатия для аналитического хранилища событий.

## ВХОДНЫЕ ДАННЫЕ
- События: 500 млн/день, средний размер записи в row-формате ≈ 400 байт.
- Хранить в DWH: 2 года истории.
- Колоночное сжатие даёт коэффициент ≈ 8× (как порядок, за счёт однотипных значений колонок).
- Аналитические запросы читают в среднем ~4 колонки из 20.

## ТРЕБОВАНИЯ
- Посчитайте объём за 2 года в row-формате (без сжатия).
- Примените колоночное сжатие — итоговый объём в DWH.
- Оцените выигрыш по IO для типичного запроса (4 из 20 колонок + сжатие).
- Сделайте вывод о роли колоночного хранения для стоимости и скорости.

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт объёма (row → сжатый) + выигрыш IO на запрос + вывод. (Числа — порядки.)

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объём/день: 500 млн × 400 Б = 200 ГБ/день; за 2 года ≈ 200 ГБ × 730 ≈ 146 ТБ (row, без сжатия).
- [ ] Со сжатием 8×: ≈ 146 / 8 ≈ ~18 ТБ в DWH.
- [ ] Выигрыш IO на запрос: читаем ~4/20 колонок (×5 меньше) и сжато (×8) → совокупно на порядок(и) меньше IO, чем полный скан row-store.
- [ ] Вывод: колоночное хранение кратно снижает и объём (стоимость), и читаемый IO (скорость) для аналитики.
- [ ] Числа трактуются как порядки; связь со стоимостью (модуль 07).

## ПОДСКАЗКА
Колоночное хранилище выигрывает дважды: сжатие (однотипные значения колонок) уменьшает объём, а чтение только нужных колонок (4 из 20) сокращает IO на запрос. Совокупно — кратное снижение стоимости хранения и времени запроса.
