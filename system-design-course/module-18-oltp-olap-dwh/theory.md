# Модуль 18 — OLTP vs OLAP, DWH, колоночные БД

## Зачем этот модуль

Часть 3 строила систему, обслуживающую пользователей: быстрые операции над отдельными записями — создать заказ, прочитать профиль, списать деньги. Это **транзакционная** (OLTP) нагрузка. Но у бизнеса есть и второй мир: «сколько продали за квартал по регионам?», «какая воронка конверсии?», «топ товаров за год». Это **аналитическая** (OLAP) нагрузка — и она устроена принципиально иначе.

Ключевой урок модуля: **транзакционная и аналитическая нагрузки конфликтуют**, и попытка обслужить обе одной БД ломает обе. Тяжёлый аналитический запрос (просканировать миллиард строк) на проде OLTP кладёт latency пользовательских операций. Поэтому в дизайне их разделяют: OLTP-БД для пользователей, отдельное аналитическое хранилище (DWH/колоночная БД) для аналитики, и конвейер между ними (модуль 20).

Этот модуль — про различие OLTP/OLAP, почему колоночное хранение быстрее для аналитики, что такое Data Warehouse и dimensional modeling (star schema).

---

## 1. OLTP vs OLAP — две разные нагрузки

| | **OLTP** (transaction processing) | **OLAP** (analytical processing) |
|--|-----------------------------------|----------------------------------|
| Запросы | много мелких: чтение/запись отдельных строк | мало, но тяжёлые: агрегации по миллионам строк |
| Доступ | по ключу, точечно (`WHERE id=...`) | сканы, группировки (`GROUP BY`, `SUM`) |
| Колонки | вся строка целиком | несколько колонок из многих |
| Метрика | latency (мс на операцию) | throughput (объём за запрос) |
| Схема | нормализованная | денормализованная (звезда) |
| Хранение | row-oriented | column-oriented |
| Свежесть | актуальные данные | часто с задержкой (ETL) |
| Примеры | заказы, платежи, профили | отчёты, BI, дашборды, ML-фичи |

```
OLTP: "создай заказ #123"           → 1 строка, миллисекунды
OLAP: "выручка по странам за 2025"  → скан миллионов строк, секунды-минуты
```

> OLTP — это всё, что вы строили в Части 3 (модули 11–13). OLAP — новый мир: запросы редкие, но огромные, важна не latency одной операции, а способность переварить объём. Эти профили **противоположны**, поэтому их разделяют физически.

## 2. Почему нельзя смешивать на одной БД

```
Аналитик запускает "SUM по всем заказам за 3 года" на проде OLTP
  → запрос сканирует миллиард строк, забивает CPU/IO/память
  → latency пользовательских заказов взлетает
  → пользователи видят таймауты, пока считается отчёт
```

> Тяжёлый аналитический запрос на OLTP-проде конкурирует за ресурсы с пользовательскими операциями и убивает их latency. Это одна из самых частых ошибок. Решение — **разделение**: аналитику уносят на read-реплику (для лёгких случаев) или в отдельное аналитическое хранилище (DWH), наполняемое конвейером. Прод OLTP остаётся чистым под пользователей.

## 3. Row-oriented vs column-oriented хранение

Фундаментальное отличие, объясняющее, почему аналитические БД быстры:

```
Таблица: id | name | country | amount

Row store (OLTP):  [1,Ann,US,100][2,Bob,UK,200][3,Cyd,US,50]...
  → вся строка рядом → быстро достать/изменить запись целиком

Column store (OLAP): id:[1,2,3...] name:[Ann,Bob,Cyd...] country:[US,UK,US...] amount:[100,200,50...]
  → каждая колонка рядом → читать только нужные колонки
```

| | Row store | Column store |
|--|-----------|--------------|
| Сильно в | достать/изменить целую строку | агрегации по нескольким колонкам |
| `SELECT * WHERE id=5` | быстро (одна строка рядом) | медленно (собирать из колонок) |
| `SUM(amount)` по всем | медленно (читает все строки целиком) | быстро (читает одну колонку) |
| Сжатие | хуже (разнотипные данные в строке) | **отличное** (однотипные значения рядом) |

> **Почему column store быстр для аналитики (3 причины):** (1) читает только нужные колонки, а не всю строку (аналитика трогает 3–5 колонок из 50); (2) превосходное **сжатие** — однотипные значения колонки рядом (страны, даты повторяются) сжимаются в разы, меньше IO; (3) **векторизация** — операции над колонкой как над массивом. Обратная сторона: точечные `UPDATE`/`INSERT` отдельной строки в колоночном хранилище дороги — поэтому оно для OLAP, не OLTP.

## 4. Data Warehouse (DWH)

**Data Warehouse** — централизованное аналитическое хранилище, которое интегрирует данные из множества источников (OLTP-БД, логи, внешние системы) в единую модель для анализа.

| Свойство DWH | |
|--------------|--|
| Назначение | аналитика, BI, отчётность |
| Хранение | колоночное, оптимизировано под сканы/агрегации |
| Данные | интегрированные, исторические, из многих источников |
| Загрузка | через ETL/ELT-конвейеры (модуль 20), с задержкой |
| Примеры | Snowflake, Google BigQuery, Amazon Redshift, ClickHouse |

> DWH — это «единый источник правды для аналитики»: данные из заказов, платежей, маркетинга, логов стекаются в одно место и приводятся к общей модели. Облачные DWH (Snowflake, BigQuery) разделяют хранение и вычисления, масштабируя их независимо. Данные в DWH **не realtime** — между событием в OLTP и его появлением в DWH есть лаг конвейера.

## 5. Dimensional modeling: star schema

Аналитические данные моделируют не как нормализованные таблицы, а как **звезду**: центральная таблица фактов + таблицы измерений.

```
              dim_date
                 │
dim_product ─ FACT_SALES ─ dim_customer
                 │
              dim_store

fact_sales: date_id, product_id, customer_id, store_id, quantity, amount  ← метрики (что мерим)
dim_*:      контекст (имя продукта, регион клиента, дата...)              ← измерения (по чему режем)
```

| | **Fact table** (факты) | **Dimension table** (измерения) |
|--|------------------------|----------------------------------|
| Что | измеримые события/метрики | контекст для группировки/фильтрации |
| Содержит | числа (quantity, amount) + ключи измерений | атрибуты (имя, категория, регион) |
| Размер | огромная (растёт со временем) | небольшие |
| Пример | продажа, клик, транзакция | продукт, клиент, дата, магазин |

> **Star schema** — стандарт DWH: факты в центре («что произошло и сколько»), измерения по краям («в каком разрезе смотрим»). Запрос «выручка по категориям за месяц» = агрегация fact_sales с join к dim_product и dim_date. Денормализация измерений (звезда) ускоряет аналитические join'ы. **Snowflake schema** — нормализованный вариант (измерения разбиты дальше), компактнее, но join'ов больше.

## 6. Разделение OLTP и OLAP в дизайне

```
                          ┌── read-реплика (лёгкая аналитика)
OLTP (PostgreSQL) ────────┤
  пользователи            └── ETL/ELT (модуль 20) ──> DWH (колоночный) ──> BI/дашборды
```

| Подход | Когда |
|--------|-------|
| Read-реплика OLTP | лёгкая аналитика, данные нужны свежими, объём умеренный |
| Отдельный DWH + ETL | тяжёлая аналитика, исторические данные, интеграция источников |
| **HTAP** (гибрид) | системы, претендующие обслуживать обе нагрузки (TiDB, SingleStore) — реже, со своими компромиссами |

> Базовый паттерн: прод OLTP не трогают аналитикой; лёгкие отчёты — с read-реплики, тяжёлая/историческая аналитика — в DWH через конвейер (модуль 20). HTAP обещает «и то, и другое в одной БД», но в дизайне по умолчанию разделяют — это проще и предсказуемее.

## ⚠️ Подводные камни

- **Аналитика на OLTP-проде.** Тяжёлый скан убивает latency пользователей; уносите на реплику/DWH.
- **Колоночная БД под точечные update/транзакции.** Column store плох для OLTP-записи; он для сканов/агрегаций.
- **Нормализованная реляционная схема под аналитику.** Много join'ов на больших объёмах медленны; нужна звезда.
- **Смешивать OLTP и OLAP в одной БД «для простоты».** Разные профили конфликтуют; разделяйте.
- **Забыть про лаг ETL.** Данные в DWH не realtime; не обещайте «свежую» аналитику без стриминга.
- **Не строить dimensional model.** Сырые нормализованные таблицы в DWH → медленные/сложные запросы; используйте star schema.
- **Тащить всё в DWH без нужды.** DWH под аналитику, не под оперативные запросы приложения.
- **Игнорировать сжатие как фактор.** Колоночное сжатие — ключевой выигрыш аналитики по объёму/стоимости.
