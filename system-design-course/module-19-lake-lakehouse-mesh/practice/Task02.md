# Задача 02 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Предотвратите превращение Data Lake в data swamp: спроектируйте governance.

## ВХОДНЫЕ ДАННЫЕ
Компания год складывала в Data Lake всё подряд из десятков источников без правил. Сейчас: никто не знает, что где лежит, дублируются датасеты, неясно качество и актуальность, аналитики не доверяют данным и просят инженеров каждый раз.

## ТРЕБОВАНИЯ
- Объясните, что такое data swamp и почему он возник.
- Предложите меры governance (минимум 4): каталог, владение, качество, метаданные/lineage, слои.
- Объясните, как medallion-слои (bronze/silver/gold) помогают навести порядок.
- Опишите, как вернуть доверие аналитиков к данным.

## ФОРМАТ РЕЗУЛЬТАТА
Что такое swamp и причина → меры governance (≥4) → роль medallion → восстановление доверия.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: data swamp — озеро без управления (всё свалено, не найти, не доверять); причина — отсутствие каталога/владения/качества.
- [ ] Меры governance: data catalog (что где лежит), владельцы датасетов, контроль качества/валидация, метаданные и lineage (модуль 20), стандарты форматов.
- [ ] Medallion: сырьё в bronze, очищенное в silver, готовое в gold → потребители работают с проверенными слоями.
- [ ] Доверие: документированные, провалидированные, версионированные датасеты с владельцем и SLA.
- [ ] Решение системно (правила + слои + ответственность), а не «почистить один раз».

## ПОДСКАЗКА
Data swamp = Lake без дисциплины. Лечится governance: каталог (находимость), владельцы (ответственность), качество (доверие), lineage (происхождение) и слои medallion (сырьё отдельно от витрин). Без этого озеро бесполезно, сколько в нём ни данных.
