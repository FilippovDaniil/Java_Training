# Задача 03 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Объясните, какую проблему решает Lakehouse, и обоснуйте переход на него.

## ВХОДНЫЕ ДАННЫЕ
Компания держит два хранилища: Data Lake (сырьё, ML) и отдельный DWH (BI). Данные дублируются: сырьё в озере → ETL → копия в DWH. Жалобы: дублирование, рассинхрон между озером и DWH, двойная стоимость, сложный конвейер.

## ТРЕБОВАНИЯ
- Объясните, почему исторически держали два хранилища (Lake + DWH).
- Опишите, как Lakehouse устраняет дублирование (одно хранилище для ML и BI).
- Объясните роль table format (Iceberg/Delta/Hudi): что он добавляет к «файлам в S3».
- Назовите, какие гарантии DWH появляются у озера в Lakehouse.

## ФОРМАТ РЕЗУЛЬТАТА
Почему два хранилища → как Lakehouse устраняет дублирование → роль table format → гарантии.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: Lake хорош для сырья/ML, но плох для производительного BI → держали ещё DWH; отсюда дублирование/рассинхрон/двойная стоимость.
- [ ] Lakehouse: одно дешёвое объектное хранилище обслуживает и ML/сырьё, и BI → нет дублирования в два места.
- [ ] Table format добавляет поверх файлов: ACID-транзакции, schema evolution, time travel (версии), эффективные апдейты/удаления, метаданные для производительности.
- [ ] Гарантии DWH у озера: транзакционность, схема, производительные запросы — при дешёвом хранении озера.
- [ ] Вывод: Lakehouse убирает дилемму «Lake или DWH».

## ПОДСКАЗКА
Проблема двух хранилищ — дублирование и рассинхрон. Lakehouse = дешёвое озеро + table format (Iceberg/Delta), который накладывает ACID/схему/time travel на «просто файлы», давая гарантии DWH. Итог — одно хранилище и для ML, и для BI.
