# Задача 04 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Определите, где уместен schema-on-write, а где schema-on-read, и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
- (А) финансовые проводки для регуляторной отчётности: структура строго фиксирована, ошибки недопустимы;
- (Б) приём разнородных логов и событий из множества меняющихся источников «на будущее»;
- (В) данные с эволюционирующей схемой, где новые поля добавляются часто, а старые потребители не должны ломаться.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите schema-on-write или schema-on-read и обоснуйте.
- Объясните trade-off каждого подхода (целостность vs гибкость).
- Для (В) объясните, как table format (schema evolution) совмещает гибкость с гарантиями.
- Назовите риск schema-on-read без контроля качества.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: кейс → подход → обоснование + trade-off + про schema evolution + риск.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → schema-on-write (строгая структура, целостность, БД отвергает несоответствие).
- [ ] (Б) → schema-on-read (принимаем любой формат, интерпретируем при чтении, гибко).
- [ ] (В) → Lakehouse/table format со schema evolution (гибкость добавления полей + гарантии).
- [ ] Trade-off назван: schema-on-write — целостность ценой жёсткости; schema-on-read — гибкость ценой ответственности за качество в приложении.
- [ ] Риск schema-on-read без контроля: мусор/несогласованность, недоверие к данным.

## ПОДСКАЗКА
Schema-on-write — целостность при записи (для строгих/регуляторных данных). Schema-on-read — гибкость приёма «как есть» (для сырья/меняющихся источников), но качество на совести читателя. Table format добавляет schema evolution: меняй схему, не ломая потребителей.
