# Задача 05 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Примените четыре принципа Data Mesh к крупной организации.

## ВХОДНЫЕ ДАННЫЕ
Крупная компания: домены «Заказы», «Платежи», «Логистика», «Маркетинг». Сейчас одна центральная дата-команда обрабатывает запросы всех доменов и стала узким местом: задержки, она не знает специфику доменов, очередь задач растёт.

## ТРЕБОВАНИЯ
- Объясните, почему централизованная дата-команда стала узким местом.
- Примените каждый из 4 принципов Data Mesh к этой организации (конкретно).
- Объясните, что меняется в ответственности (кто за что отвечает).
- Подчеркните, что Data Mesh — организационный подход, а не покупка инструмента.

## ФОРМАТ РЕЗУЛЬТАТА
Причина узкого места → 4 принципа применительно к доменам → смена ответственности → про «не инструмент».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Причина: центр-команда не знает специфику доменов и не масштабируется на их число → очередь/задержки.
- [ ] Domain ownership: домены (Заказы/Платежи/...) сами владеют своими данными.
- [ ] Data as a product: каждый домен публикует данные как продукт (владелец, SLA, документация, качество).
- [ ] Self-serve platform: общая платформа, чтобы домены публиковали/потребляли без центра.
- [ ] Federated governance: общие стандарты (безопасность, форматы), исполняемые децентрализованно.
- [ ] Подчёркнуто: Mesh — смена ответственности/организации, а не «купить продукт».

## ПОДСКАЗКА
Data Mesh решает организационное узкое место: центр-команда не тянет десятки доменов. Четыре принципа: домены владеют данными (ownership), данные = продукт с SLA, общая self-serve платформа, федеративное управление (общие правила, локальное исполнение). Это про людей и ответственность, не про инструмент.
