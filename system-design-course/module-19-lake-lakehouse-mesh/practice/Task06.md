# Задача 06 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Определите, когда Data Mesh оправдан, а когда это overkill.

## ВХОДНЫЕ ДАННЫЕ
- (А) стартап из 15 человек, одна продуктовая команда, аналитика в зачаточном состоянии;
- (Б) корпорация с десятками автономных доменных команд, где центральная дата-команда — постоянное узкое место;
- (В) компания среднего размера, 3–4 команды, центральная дата-команда пока справляется, но рост ускоряется.

## ТРЕБОВАНИЯ
- Для каждого случая решите: Mesh оправдан, overkill или «рано/наблюдать».
- Обоснуйте через организационный масштаб (а не технологию).
- Для (А) и (В) предложите более простую альтернативу.
- Сформулируйте принцип: какой сигнал говорит, что пора задуматься о Mesh.

## ФОРМАТ РЕЗУЛЬТАТА
Решение по (А)/(Б)/(В) + обоснование + альтернативы + сигнал к Mesh.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → overkill: централизованный DWH/Lakehouse достаточен и проще.
- [ ] (Б) → Mesh оправдан: организационный масштаб (десятки доменов) делает центр-команду узким местом.
- [ ] (В) → рано/наблюдать: пока централизованно, следить за сигналами роста.
- [ ] Альтернатива для (А)/(В): централизованный DWH/Lakehouse.
- [ ] Сигнал к Mesh: центральная дата-команда стабильно не успевает за числом доменов (организационное узкое место), а не «модно».
- [ ] Обоснование через организацию, не через инструменты.

## ПОДСКАЗКА
Mesh — ответ на организационный масштаб, не на технический. Маленьким и средним проще централизованный DWH/Lakehouse. Сигнал к Mesh — когда центральная команда становится хроническим узким местом для множества автономных доменов. Не вводить «на вырост» преждевременно.
