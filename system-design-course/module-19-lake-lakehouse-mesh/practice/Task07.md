# Задача 07 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне дата-платформы и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Складываем всё сырьё в Data Lake без каталога и владельцев — потом разберёмся».
2. «BI-дашборды строим прямо поверх сырых файлов в озере без слоёв и структуры».
3. «Внедряем Data Mesh, купив платформу X — это сразу решит наши проблемы с данными».
4. «У нас 10 человек и один продукт, но сразу строим Data Mesh с доменными командами».
5. «Принимаем данные schema-on-read и нигде не валидируем качество — гибкость важнее».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Lake без каталога/владельцев → data swamp → governance (каталог, владельцы, lineage) с самого начала.
- [ ] (2) BI поверх сырых файлов → медленно/хаос → слои medallion (gold-витрины) или Lakehouse/DWH для BI.
- [ ] (3) «Купить Data Mesh» → Mesh не инструмент → это организационный подход (владение/ответственность), платформа лишь часть.
- [ ] (4) Mesh для 10 человек → overkill → централизованный DWH/Lakehouse.
- [ ] (5) Schema-on-read без валидации → мусор/недоверие → контроль качества (валидация, тесты данных).
- [ ] Отмечено: (1) и (5) вернее всего ведут к swamp/недоверию.

## ПОДСКАЗКА
Чек-лист модуля: Lake требует governance (иначе swamp); BI — на структурированных слоях/DWH, не на сыром; Mesh — организация, не покупка; Mesh не для малых; schema-on-read требует контроля качества. Пункты 1 и 5 — прямой путь к болоту.
