# Задача 08 — Модуль 19: Data Lake, Lakehouse, Data Mesh

## ЗАДАНИЕ
Спроектируйте medallion-архитектуру (bronze/silver/gold) для дата-платформы.

## ВХОДНЫЕ ДАННЫЕ
Маркетплейс: сырые данные из источников — события кликстрима (JSON), выгрузки заказов из OLTP, логи, данные партнёров (CSV разного качества). Потребители: BI-дашборды (выручка, конверсия), ML (рекомендации), аналитики ad-hoc.

## ТРЕБОВАНИЯ
- Опишите три слоя: что попадает в bronze, silver, gold и какие преобразования между ними.
- Объясните, зачем хранить сырьё (bronze), если есть очищенные данные.
- Покажите, какие потребители работают с каким слоем.
- Объясните, как medallion защищает от data swamp.

## ФОРМАТ РЕЗУЛЬТАТА
Описание bronze/silver/gold (что + преобразования) + зачем bronze + потребители по слоям + защита от swamp.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Bronze: сырые данные как пришли (кликстрим/заказы/логи/CSV), неизменные, для аудита/переобработки.
- [ ] Silver: очищенные, провалидированные, нормализованные (унифицированные форматы, убран мусор).
- [ ] Gold: готовые витрины/агрегаты под потребителей (выручка, фичи для ML).
- [ ] Объяснено, зачем bronze: возможность переобработать историю и аудит (источник правды сырья).
- [ ] Потребители: ad-hoc/инженеры — bronze/silver; BI — gold; ML — silver/gold.
- [ ] Medallion даёт дисциплину (слои возрастающего качества) → не swamp.

## ПОДСКАЗКА
Bronze = сырьё как есть (храним для переобработки/аудита). Silver = очищенное/провалидированное. Gold = готовые витрины под BI/ML. Потребители работают с проверенными слоями, а не с кучей. Это рецепт против болота: данные проходят слои качества.
