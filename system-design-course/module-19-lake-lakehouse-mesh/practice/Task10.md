# Задача 10 — Модуль 19 (мини-проект): Data-платформа компании

## ЗАДАНИЕ
Спроектируйте дата-платформу для компании: хранение, слои, governance, организация. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Компания на выбор (зафиксируйте): (а) растущий e-commerce-скейлап (BI + ML); (б) крупная корпорация с автономными доменами; (в) data-driven стартап. Источники данных, объёмы и потребности (BI/ML/ad-hoc) задайте явно как допущение (или из модулей 05–07).

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Выбор платформы** — DWH / Lake / Lakehouse / Mesh с обоснованием (данные, зрелость, размер).
2. **Хранение** — где и как (объектное хранилище, table format при Lakehouse).
3. **Слои** — medallion (bronze/silver/gold) или эквивалент.
4. **Governance** — каталог, владение, качество, lineage (против data swamp).
5. **Потребители** — BI/ML/ad-hoc и с каким слоем работают.
6. **Организация** — централизованная команда или Mesh (по масштабу).
7. **Эволюция** — точки пересмотра при росте (когда менять платформу/подход).

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов + схема платформы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна компания; источники/объёмы/потребности заданы.
- [ ] Выбор платформы обоснован размером/зрелостью/данными (не «по моде»).
- [ ] Описано хранение (объектное + table format при Lakehouse).
- [ ] Заложены слои качества (medallion) и governance против swamp.
- [ ] Потребители сопоставлены со слоями.
- [ ] Организационный аспект (центр vs Mesh) соответствует масштабу.
- [ ] Есть точки пересмотра при росте; решения соразмерны (не overkill).

## ПОДСКАЗКА
Это синтез модуля и мост к модулю 20 (конвейеры наполняют платформу). Хорошая дата-платформа: выбор от размера/данных (DWH/Lakehouse/Mesh), дешёвое объектное хранение + table format, слои medallion, governance против swamp, потребители на проверенных слоях, организация по масштабу. Не строить Mesh «на вырост». Дальше — модуль 20: конвейеры (ETL/ELT/CDC).
