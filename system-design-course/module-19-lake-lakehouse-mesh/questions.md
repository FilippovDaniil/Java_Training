## Ключ (эталонные ответы)

1. DWH → Data Lake → Lakehouse → Data Mesh. Это не «новое лучше старого», а разные ответы на разные проблемы (BI, дешёвое сырьё/ML, объединение, организационный масштаб).
2. DWH дорог, требует жёсткой схемы заранее (schema-on-write) и плохо подходит для сырых/полуструктурированных данных и ML.
3. Хранилище сырых данных любого формата (структурированных/полу/неструктурированных) в дешёвом объектном хранилище, со schema-on-read.
4. **Schema-on-write** — структура проверяется при записи (DWH, целостность). **Schema-on-read** — структура интерпретируется при чтении (Lake, гибкость, но качество на совести читателя).
5. DWH: структурированные/обработанные данные, schema-on-write, дороже, для BI, высокая производительность. Lake: сырые любого формата, schema-on-read, дёшево, для ML/research, риск swamp.
6. **Data swamp** — Data Lake без управления: данные свалены без каталога, владельцев и качества, их не найти и им не доверяют. Возникает из-за отсутствия governance.
7. Governance: data catalog (находимость), владельцы датасетов (ответственность), контроль качества/валидация, метаданные и lineage, слои medallion (bronze/silver/gold).
8. Архитектура, объединяющая дешёвое хранение Data Lake с транзакционными гарантиями и производительностью DWH, через table formats поверх файлов в объектном хранилище.
9. Устраняет необходимость держать два хранилища (Lake для сырья/ML + отдельный DWH для BI) с дублированием и рассинхроном — одно хранилище обслуживает и ML, и BI.
10. Формат таблиц поверх файлов в объектном хранилище: Delta Lake, Apache Iceberg, Apache Hudi.
11. ACID-транзакции, schema evolution, time travel (версии данных), эффективные апдейты/удаления, метаданные для производительных запросов — превращает «файлы в S3» в таблицы с гарантиями DWH.
12. Организация данных по слоям качества: bronze (сырьё как пришло), silver (очищенное/провалидированное), gold (готовые витрины/агрегаты под потребителей).
13. Чтобы можно было переобработать историю (при изменении логики/багах) и для аудита — bronze хранит сырьё как источник правды, из которого пересобираются silver/gold.
14. Организационно-архитектурный подход к данным: децентрализация ответственности, домены владеют своими данными как продуктами. Не технология, а способ организации.
15. Domain ownership (домены владеют данными), data as a product (данные — продукт с SLA/качеством), self-serve data platform (общая платформа для публикации/потребления), federated computational governance (общие правила, децентрализованное исполнение).
16. Потому что его суть — смена ответственности и владения (домены отвечают за свои данные), а не конкретный инструмент. «Купить Data Mesh» нельзя — платформа лишь часть подхода.
17. Организационное узкое место: централизованная дата-команда не успевает обслуживать множество автономных доменов и не знает их специфику. Mesh децентрализует ответственность.
18. Для маленьких и средних организаций: централизованный DWH/Lakehouse проще и достаточен. Mesh добавляет организационную сложность, оправданную лишь при масштабе многих доменов.
19. Данные публикуются как продукт: с владельцем, документацией, SLA, гарантиями качества и удобным доступом для потребителей — а не «свалка, разбирайтесь сами».
20. Общие правила и стандарты (безопасность, форматы, совместимость), которые задаются централизованно, но исполняются децентрализованно каждым доменом — баланс автономии и порядка.
21. DWH — для структурированной BI-аналитики (надёжно, быстро, дорого), когда схема и использование известны. Lake — для сырья любого формата, дешёвого хранения «на будущее» и ML (с governance).
22. Когда нужны и BI, и ML без дублирования в два хранилища: Lakehouse даёт дешёвое хранение озера + гарантии DWH (через table format), обслуживая обе нагрузки из одного места.
23. Когда централизованная дата-команда стабильно становится узким местом для растущего числа автономных доменов (организационный масштаб), а не когда это просто «модно».
24. Без валидации в озеро/хранилище затекает мусор и несогласованные данные, потребители теряют доверие — гибкость schema-on-read оборачивается data swamp.
25. Выбор платформы (от размера/данных/зрелости); хранение (объектное + table format при Lakehouse); слои качества (medallion); governance против swamp (каталог/владение/качество/lineage); потребители по слоям; организация (центр vs Mesh по масштабу); точки пересмотра при росте.
