# Модуль 19 — Data Lake, Lakehouse, Data Mesh

## Зачем этот модуль

Data Warehouse (модуль 18) отлично решает BI-аналитику над структурированными данными, но у него есть пределы: он дорог, требует жёсткой схемы заранее (schema-on-write) и плохо подходит для сырых/полуструктурированных данных и ML. Когда объёмы и разнообразие данных взорвались (логи, события, изображения, JSON из десятков источников), родился **Data Lake** — дешёвое хранилище сырья любого формата. Затем — **Lakehouse**, гибрид, добавляющий озеру гарантии DWH. А когда централизованная дата-команда стала узким местом целой организации, появился **Data Mesh** — не технология, а способ организации.

Зачем это senior'у. Выбор дата-платформы — стратегическое решение уровня «SQL vs NoSQL» (модуль 11), но для аналитики. Понимать различие DWH/Lake/Lakehouse и знать, что Data Mesh — про организацию, а не про инструмент, — признак зрелости. И знать главную ловушку: Data Lake без управления превращается в **data swamp** (болото), куда всё свалили и ничего не найти.

Этот модуль — про эволюцию хранения данных, table formats (Iceberg/Delta), четыре принципа Data Mesh и medallion-архитектуру.

---

## 1. Эволюция аналитического хранения

```
DWH ──────────> Data Lake ──────────> Lakehouse ──────────> Data Mesh
структура,       сырьё, дёшево,         озеро + гарантии      децентрализация,
schema-on-write  schema-on-read,        DWH (ACID/схема)      данные как продукт
дорого, BI       ML/raw, риск swamp     (table formats)       (организационно)
```

> Это не «новое лучше старого» — это **разные ответы на разные проблемы**. DWH жив и хорош для BI. Lake появился ради дешёвого сырья и ML. Lakehouse — чтобы не держать два хранилища (озеро + DWH). Mesh — ответ на организационное масштабирование, а не технический. Выбор зависит от данных, зрелости и размера организации.

---

## 2. Data Lake

**Data Lake** — хранилище сырых данных **любого формата** (структурированные, полуструктурированные, неструктурированные) в дешёвом объектном хранилище (blob, модуль 10), со **schema-on-read**.

| Свойство | |
|----------|--|
| Формат | любой: CSV, JSON, Parquet, логи, изображения, видео |
| Схема | schema-on-read (интерпретируется при чтении) |
| Хранение | дешёвое объектное (S3/GCS), практически безграничное |
| Сильно в | сырьё, ML, исследовательская аналитика, дешёвое хранение всего |
| Слабо в | governance, качество, производительные BI-запросы |

```
DWH:       сначала схема → потом грузим (schema-on-write, структурно, дорого)
Data Lake: грузим как есть → схема при чтении (schema-on-read, гибко, дёшево)
```

> Data Lake решает то, что не умеет DWH: хранить **дёшево** и **любой формат**, не зная заранее, как данные понадобятся (важно для ML и исследований). Но гибкость опасна: без каталога, качества и управления озеро превращается в **data swamp** — свалку, где данные есть, но их не найти, им не доверяют и не используют.

---

## 3. DWH vs Data Lake

| | **Data Warehouse** | **Data Lake** |
|--|--------------------|---------------|
| Данные | структурированные, обработанные | сырые, любой формат |
| Схема | schema-on-write | schema-on-read |
| Стоимость | дороже | дёшево |
| Потребители | BI-аналитики, отчёты | data scientists, ML, инженеры |
| Производительность запросов | высокая | переменная |
| Риск | жёсткость, дорого | data swamp |

> Классический выбор: **DWH — для структурированной BI-аналитики** (надёжно, быстро, дорого), **Lake — для сырья, ML и дешёвого хранения всего** (гибко, дёшево, требует дисциплины). Долго организации держали оба (Lake для сырья → ETL в DWH для BI), что дублирует данные и сложно. Это и подтолкнуло к Lakehouse.

---

## 4. Lakehouse

**Lakehouse** — архитектура, объединяющая дешёвое хранение Data Lake с транзакционными гарантиями и производительностью DWH. Достигается через **table formats** поверх файлов в объектном хранилище.

| Table format | Что добавляет поверх файлов в blob |
|--------------|-------------------------------------|
| **Delta Lake**, **Apache Iceberg**, **Apache Hudi** | ACID-транзакции, schema evolution, time travel (версии), эффективные апдейты/удаления, метаданные для быстрых запросов |

```
Lakehouse = дешёвое объектное хранилище (как Lake)
          + table format (Iceberg/Delta) → ACID, схема, time travel, производительность (как DWH)
          → одно хранилище и для ML/сырья, и для BI
```

> Lakehouse убирает дилемму «Lake или DWH»: одно хранилище обслуживает и сырьё/ML, и структурированную BI-аналитику, без дублирования данных в два места. **Table format** — ключ: он накладывает на «просто файлы в S3» транзакционность, версионирование и схему, превращая озеро в нечто с гарантиями DWH. Это доминирующий тренд современных дата-платформ.

---

## 5. Medallion architecture (bronze/silver/gold)

Способ организации данных в Lake/Lakehouse по слоям качества:

```
Bronze (raw)    → сырые данные как пришли из источников (неизменные, аудит)
   ↓ очистка, валидация
Silver (cleaned)→ очищенные, нормализованные, провалидированные
   ↓ агрегация, бизнес-логика
Gold (curated)  → готовые витрины/агрегаты под потребителей (BI/ML)
```

> Medallion даёт дисциплину, которой не хватает «голому» озеру: сырьё хранится (bronze, для переобработки/аудита), но потребители работают с очищенными (silver) и готовыми (gold) данными. Это прямой рецепт против data swamp — данные проходят слои возрастающего качества, а не сваливаются в кучу.

---

## 6. Data Mesh

**Data Mesh** — не технология, а **организационно-архитектурный подход**: децентрализация ответственности за данные. Возникает, когда централизованная дата-команда становится узким местом для всей организации.

Четыре принципа:

| Принцип | Суть |
|---------|------|
| **Domain ownership** | данными владеют доменные команды (кто их создаёт), не центральная дата-команда |
| **Data as a product** | данные — продукт с владельцем, SLA, документацией, качеством для потребителей |
| **Self-serve data platform** | общая платформа, чтобы команды сами публиковали/потребляли данные без центра |
| **Federated computational governance** | общие правила/стандарты (безопасность, форматы), но исполняются децентрализованно |

> Главное про Data Mesh: это **про организацию, а не про инструмент**. Нельзя «купить Data Mesh» — это смена ответственности (домены владеют своими данными как продуктами). Он решает проблему масштаба организации, где центральная дата-команда не успевает обслуживать десятки доменов. Для маленькой компании это **overkill** — централизованный DWH/Lakehouse проще и достаточен.

---

## 7. Что выбрать

```
Нужна структурированная BI-аналитика, данные чистые     → DWH
Нужно дёшево хранить сырьё/любой формат, ML/research     → Data Lake (+ governance!)
Хочется и то, и другое без двух хранилищ                  → Lakehouse
Организация большая, центр-команда — узкое место          → Data Mesh (организационно)
```

> Идите от размера и зрелости: большинство стартует с DWH или Lakehouse. Lake без governance — риск болота. Mesh — только когда организационный масштаб реально требует децентрализации. Как и с polyglot persistence (модуль 11): не плодите сложность раньше нужды.

---

## ⚠️ Подводные камни

- **Data Lake без governance → data swamp.** Каталог, качество, владение обязательны, иначе свалка.
- **Lake как замена DWH для BI.** Без структуры BI-запросы медленны и хаотичны; для BI — DWH/Lakehouse/gold-слой.
- **Воспринимать Data Mesh как технологию.** Это организационный подход (владение/ответственность), не инструмент.
- **Data Mesh для маленькой команды.** Overkill; централизованный DWH/Lakehouse проще и достаточен.
- **Schema-on-read без контроля качества.** Гибкость без валидации → мусор и недоверие к данным.
- **Дублировать всё везде (Lake + DWH + копии).** Сложно и дорого; Lakehouse устраняет дублирование.
- **Не разделять слои качества.** Сваливать сырьё и витрины вместе; нужен medallion (bronze/silver/gold).
- **Гнаться за модным форматом без нужды.** Table format/Lakehouse — под реальную потребность, не ради хайпа.
