# Задача 03 — Модуль 20: Конвейеры данных

## ЗАДАНИЕ
Спроектируйте CDC-конвейер для синхронизации OLTP с аналитикой вместо дорогого полного ETL.

## ВХОДНЫЕ ДАННЫЕ
Сейчас: каждую ночь `SELECT *` из таблицы заказов OLTP (сотни млн строк) → загрузка в DWH. Проблемы: тяжёлая нагрузка на прод-БД ночью, данные в DWH отстают на сутки, экспорт растёт с объёмом.

## ТРЕБОВАНИЯ
- Объясните проблемы полного ETL (нагрузка на OLTP, свежесть, рост).
- Спроектируйте CDC: источник изменений, как доставляются, куда.
- Объясните, почему CDC не нагружает OLTP запросами и даёт near-real-time.
- Назовите тонкость: что делать при эволюции схемы источника.

## ФОРМАТ РЕЗУЛЬТАТА
Проблемы полного ETL → дизайн CDC (источник→доставка→цель) → почему дёшево/свежо → schema evolution.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблемы: `SELECT *` грузит OLTP, суточный лаг, экспорт растёт линейно с таблицей.
- [ ] CDC: читать лог транзакций БД (binlog/WAL) → поток изменений (Debezium → Kafka, модуль 15) → DWH/индекс.
- [ ] Объяснено: CDC читает лог, а не выполняет SELECT'ы → не нагружает запросами; поток изменений → near-real-time.
- [ ] Передаются только изменения (insert/update/delete), а не вся таблица каждый раз.
- [ ] Тонкость: эволюция схемы источника должна обрабатываться (иначе CDC сломается); планировать schema evolution.

## ПОДСКАЗКА
Полный ETL = «выгрузить всё каждый раз» (дорого, грузит OLTP, не свежо). CDC = «читать лог изменений БД» (binlog/WAL через Debezium → Kafka → цель): только дельты, без SELECT-нагрузки, near-real-time. Тот же механизм синхронизирует поисковый индекс (модуль 17).
