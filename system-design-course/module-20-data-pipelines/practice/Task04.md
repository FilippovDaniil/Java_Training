# Задача 04 — Модуль 20: Конвейеры данных

## ЗАДАНИЕ
Спроектируйте оркестрацию конвейера с зависимостями, расписанием и обработкой сбоев.

## ВХОДНЫЕ ДАННЫЕ
Ночной конвейер: извлечь данные из трёх источников (заказы, пользователи, события) → соединить → посчитать агрегаты → загрузить в DWH → уведомить аналитиков. Сейчас это пять отдельных cron-задач, надеющихся, что предыдущая успела; периодически ломается из-за гонок.

## ТРЕБОВАНИЯ
- Объясните проблему «cron-ада» (неявные зависимости).
- Спроектируйте DAG: задачи и их зависимости.
- Опишите обработку сбоев (retry, алерты) и расписание.
- Объясните роль backfill в оркестраторе.

## ФОРМАТ РЕЗУЛЬТАТА
Проблема cron → DAG (задачи+зависимости) → retry/расписание → роль backfill.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблема: отдельные cron-задачи имеют неявные зависимости (надеются на тайминг) → гонки/поломки.
- [ ] DAG: extract (3 источника параллельно) → join → aggregate → load_to_dwh → notify; зависимости явные.
- [ ] Обработка сбоев: retry упавшей задачи, алерты; задача-наследник не стартует, пока не готова родительская.
- [ ] Расписание задаётся на DAG, зависимости — оркестратором (Airflow/Dagster), не таймингом.
- [ ] Backfill: возможность переобработать прошлые даты (новая логика/пропуск/баг) через оркестратор.

## ПОДСКАЗКА
Cron-ад = задачи координируются временем, а не зависимостями (гонки). Оркестратор (Airflow/Dagster) делает зависимости явными через DAG: задача стартует, когда готовы родительские, а не «по часам». Плюс retry, мониторинг и backfill.
