# Задача 06 — Модуль 20: Конвейеры данных

## ЗАДАНИЕ
Спроектируйте контроль качества данных (data quality) в конвейере.

## ВХОДНЫЕ ДАННЫЕ
Конвейер принимает данные из внешних партнёров (CSV/JSON разного качества) и из внутренних источников. Жалобы: в аналитику попадают отрицательные суммы, дубли, записи с битой схемой и пропущенными обязательными полями → отчёты ломаются, аналитики не доверяют данным.

## ТРЕБОВАНИЯ
- Предложите проверки качества на входе и между слоями (минимум 4 типа).
- Решите, что делать с невалидными записями (отбросить / в карантин / алерт).
- Объясните, где в конвейере ставить проверки (вход, между bronze→silver).
- Объясните связь качества с доверием к данным.

## ФОРМАТ РЕЗУЛЬТАТА
Типы проверок (≥4) → обработка невалидных → где ставить → связь с доверием.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проверки: соответствие схеме; диапазоны/бизнес-правила (amount ≥ 0); обязательные поля (не null); отсутствие дублей по ключу; (опц.) свежесть/полнота.
- [ ] Невалидные записи: отбраковка/карантин (dead-letter для данных) + алерт, а не молчаливый пропуск в аналитику.
- [ ] Проверки на входе и при переходе bronze→silver (модуль 19) — мусор не проходит дальше.
- [ ] Объяснено: без качества мусор затекает в аналитику → недоверие (как schema-on-read без контроля, модуль 19).
- [ ] Качество — встроенный этап конвейера, а не разовая чистка.

## ПОДСКАЗКА
Data quality — фильтр на входе и между слоями: схема, диапазоны/правила, обязательные поля, дубли. Невалидное — в карантин + алерт (не в аналитику). Ставить при bronze→silver. Без этого мусор рушит доверие — как schema-on-read без контроля (модуль 19).
