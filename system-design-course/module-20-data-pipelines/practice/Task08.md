# Задача 08 — Модуль 20: Конвейеры данных

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне data pipeline и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Каждую ночь делаем `SELECT *` всей таблицы заказов из прод-OLTP и грузим в DWH».
2. «Загрузка в витрину — простой `INSERT`; если упало, просто перезапускаем задачу».
3. «Задачи конвейера запускаем независимыми cron-джобами, рассчитывая на тайминг».
4. «Данные из внешних источников грузим без всякой валидации — доверяем партнёрам».
5. «Делаем streaming-конвейер для месячного финансового отчёта — это современно».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка бьёт по проду, какая — по корректности/доверию, какая — по лишней сложности.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + классификация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) `SELECT *` всей таблицы → грузит OLTP, не свеж → CDC (лог изменений).
- [ ] (2) `INSERT` + перезапуск → дубли → идемпотентность (перезапись партиции/upsert).
- [ ] (3) Cron-ад → неявные зависимости/гонки → DAG-оркестратор.
- [ ] (4) Нет валидации внешних данных → мусор в аналитике → data quality (схема/правила/дубли + карантин).
- [ ] (5) Streaming для месячного отчёта → лишняя сложность → batch (свежесть не нужна).
- [ ] Классифицировано: (1) бьёт по проду; (2)/(4) — корректность/доверие; (3) — надёжность; (5) — лишняя сложность.

## ПОДСКАЗКА
Чек-лист модуля: CDC вместо полного ETL; идемпотентность вместо голого INSERT; DAG вместо cron; data quality на входе; batch вместо streaming, когда свежесть не нужна. Каждый пункт нарушает один принцип.
