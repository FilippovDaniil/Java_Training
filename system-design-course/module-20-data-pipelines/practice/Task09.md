# Задача 09 — Модуль 20: Конвейеры данных

## ЗАДАНИЕ
Спроектируйте архитектуру, обслуживающую и real-time, и историческую аналитику (Lambda или Kappa).

## ВХОДНЫЕ ДАННЫЕ
Система требует: (1) живой дашборд метрик «прямо сейчас» (секундная свежесть) и (2) точную историческую аналитику по тем же событиям за годы (для отчётов и переобработки при изменении логики).

## ТРЕБОВАНИЯ
- Объясните конфликт: real-time vs точная история — почему один путь обычно не покрывает оба.
- Опишите Lambda-архитектуру (batch layer + speed layer) и её цену.
- Опишите Kappa-архитектуру (всё через стрим + replay) и её упрощение.
- Выберите подход для этого кейса и обоснуйте; назовите trade-off.

## ФОРМАТ РЕЗУЛЬТАТА
Конфликт real-time vs история → Lambda (+цена) → Kappa (+упрощение) → выбор + trade-off.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Конфликт объяснён: streaming даёт свежесть, но точную переобработку истории проще делать batch'ем; отсюда два подхода.
- [ ] Lambda: batch layer (точная история) + speed layer (свежие данные) → объединение; цена — две системы, дублирование логики.
- [ ] Kappa: единый стриминг, история — переигрыванием лога (модуль 15); проще (одна система), но требует надёжного хранения лога и replay.
- [ ] Сделан выбор с обоснованием и назван trade-off (простота Kappa vs зрелость/раздельность Lambda).
- [ ] Связь со стримингом и replay лога (модуль 15).

## ПОДСКАЗКА
Lambda = две системы (batch для точной истории + speed для свежести), цена — дублирование логики. Kappa = одна система: всё через стрим, история через replay лога (модуль 15). Kappa проще, если лог надёжно хранится и переигрывается. Выбор — простота против раздельной оптимизации.
