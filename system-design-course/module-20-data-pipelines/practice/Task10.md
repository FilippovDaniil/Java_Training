# Задача 10 — Модуль 20 (мини-проект): Data pipeline системы

## ЗАДАНИЕ
Спроектируйте полный конвейер данных для системы: от источников до аналитического хранилища. Синтез модуля и завершение Части 4.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) e-commerce; (б) видеосервис; (в) финтех. Источники (OLTP, события, логи, внешние), объёмы и требования к свежести задайте явно как допущение (или из модулей 05–07, 18–19).

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Источники и цель** — откуда данные, в какое хранилище (DWH/Lake/Lakehouse, модули 18–19).
2. **ETL vs ELT** — выбор и где трансформация.
3. **Batch vs streaming** — по требуемой свежести; Lambda/Kappa при необходимости.
4. **Извлечение** — CDC из OLTP (вместо полного ETL) где уместно.
5. **Оркестрация** — DAG, зависимости, расписание, retry, backfill.
6. **Надёжность и качество** — идемпотентность, data quality, обработка сбоев.
7. **Governance** — lineage; слои (medallion) при Lake/Lakehouse; схема потока.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов + диаграмма потока данных.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; источники/объёмы/свежесть заданы.
- [ ] Указаны источники и целевое хранилище (связь с модулями 18–19).
- [ ] Выбраны ETL/ELT и batch/streaming с обоснованием по свежести.
- [ ] CDC применён для извлечения из OLTP, где уместно (вместо полного ETL).
- [ ] Заложена оркестрация (DAG, retry, backfill).
- [ ] Учтены идемпотентность, data quality, обработка сбоев.
- [ ] Есть lineage/слои; диаграмма потока; решения опираются на числа Части 2 и связаны с модулями 15/17/18/19.

## ПОДСКАЗКА
Это синтез Части 4: источники → конвейер → хранилище (модули 18–19) → потребители. Хороший конвейер: ELT для мощных хранилищ, CDC вместо полного ETL, batch/streaming по свежести, оркестрация DAG, идемпотентность + backfill, data quality + lineage. Дальше — Часть 5: распределённые системы (CAP, консенсус, надёжность операций).
