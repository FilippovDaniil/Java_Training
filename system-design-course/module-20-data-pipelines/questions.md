## Ключ (эталонные ответы)

1. Чтобы перемещать и преобразовывать данные из источников (OLTP, логи, события, API) в аналитические хранилища (DWH/Lake/Lakehouse) — без конвейера хранилище пусто, а озеро превращается в болото.
2. Extract (извлечь из источников), Transform (преобразовать: очистка, агрегация, обогащение), Load (загрузить в целевое хранилище).
3. **ETL** — преобразовать до загрузки (трансформация на промежуточном движке). **ELT** — загрузить сырое, преобразовать внутри целевого хранилища его вычислительными ресурсами.
4. Современные хранилища (Snowflake/BigQuery/Lakehouse) дёшевы и мощны: проще залить сырьё и трансформировать на месте, сохранив исходник (bronze-слой) для гибкости и переобработки.
5. Когда схема строгая, целевое хранилище со слабыми вычислениями, или нельзя хранить сырые/чувствительные данные в целевом хранилище (нужно обработать/замаскировать до загрузки).
6. **Batch** обрабатывает периодически большими порциями (высокая latency, проще). **Streaming** — непрерывно по событию (низкая latency, real-time, но сложнее: окна, порядок, состояние).
7. **Batch** — для исторических отчётов и когда достаточно суточной свежести (проще, дешевле). **Streaming** — для real-time (фрод в момент транзакции, живые метрики, алерты).
8. **Lambda** — batch layer (точная история) + speed layer (быстрая свежая) с объединением результатов. Цена — две системы и дублирование логики обработки.
9. **Kappa** — всё проходит через единый стриминг, а история восстанавливается переигрыванием лога (replay). Проще: одна система вместо двух, но требует надёжного хранения и replay лога.
10. **CDC (Change Data Capture)** — захват изменений (insert/update/delete) из OLTP-БД через её лог транзакций и стриминг их в аналитику/индексы/другие сервисы.
11. CDC не нагружает OLTP запросами (читает лог, а не делает `SELECT *`), передаёт только изменения (а не всю таблицу каждый раз) и даёт near-real-time свежесть вместо суточного лага.
12. Из лога транзакций БД: binlog (MySQL), WAL (PostgreSQL) — журнала, который БД и так ведёт для репликации/восстановления.
13. Debezium (часто в связке с Kafka) и подобные — читают лог транзакций и публикуют поток изменений.
14. Реальный конвейер — множество зависимых задач; оркестратор делает зависимости явными, управляет расписанием, retry, мониторингом и backfill (вместо хрупких независимых cron-задач).
15. **DAG** (directed acyclic graph) — граф задач конвейера с зависимостями: задаёт, что после чего выполняется; задача стартует, когда готовы её родительские.
16. **Cron-ад** — координация задач через тайминг отдельных cron-расписаний (надежда, что предыдущая успела). Плох неявными зависимостями и гонками: при сдвиге времени задачи ломаются.
17. **Backfill** — переобработка прошлых периодов (при изменении логики, исправлении бага, пропуске). Нужен, чтобы пересчитать историю, а не только обрабатывать новые данные.
18. Потому что конвейеры падают и перезапускаются (retry неизбежен); неидемпотентный конвейер при повторе дублирует данные. Та же логика, что у consumer'а очереди (модуль 15).
19. Перезаписью партиции по дате (idempotent overwrite) или upsert вместо накопительного `INSERT` — повторный запуск даёт тот же результат; загрузка партиции атомарна (целиком или никак).
20. Валидация данных на входе и между слоями: соответствие схеме, диапазоны/бизнес-правила (amount ≥ 0), обязательные поля (не null), отсутствие дублей по ключу, полнота/свежесть.
21. Не пропускать молча в аналитику: отбраковывать или отправлять в карантин (dead-letter для данных) с алертом, чтобы разобраться, а не загрязнять хранилище.
22. **Data lineage** — прослеживание происхождения данных: из каких источников и через какие трансформации получены датасет/метрика.
23. Отладка (где сломалась метрика — видно цепочку источников/шагов), impact analysis (что затронет изменение источника), compliance/GDPR (где хранятся и как используются данные пользователя).
24. Потому что backfill пересчитывает историю заново — для этого нужны исходные данные (bronze-слой, модуль 19). Если сырьё не сохранено, переобработать прошлое не из чего.
25. Источники и целевое хранилище; ETL vs ELT; batch vs streaming (по свежести, Lambda/Kappa); CDC для извлечения; оркестрация (DAG/retry/backfill); идемпотентность и data quality; governance (lineage, слои medallion) и схема потока.
