# Модуль 20 — Конвейеры данных (ETL/ELT, streaming, CDC, lineage)

## Зачем этот модуль

Модули 18–19 описали, *куда* складывать аналитические данные (DWH, Lake, Lakehouse). Этот модуль — про то, *как* они туда попадают: **конвейеры данных (data pipelines)**, перемещающие и преобразующие данные из источников (OLTP-БД, логи, события, внешние API) в аналитические хранилища. Без конвейера DWH пуст, а Data Lake — болото.

Это завершающий блок Части 4 и связующее звено: конвейер берёт данные из OLTP (Часть 3), часто через очереди/стримы (модуль 15), и наполняет аналитический слой (модули 18–19). От его архитектуры зависит свежесть аналитики (batch vs streaming), нагрузка на источники (полный ETL vs CDC), надёжность (идемпотентность, retry) и доверие к данным (качество, lineage).

На интервью «как данные попадают в аналитику?», «real-time или batch?», «как не грузить прод полным экспортом?» — прямые сигналы. Этот модуль — про ETL vs ELT, batch vs streaming, CDC, оркестрацию, качество и lineage.

---

## 1. Зачем нужны конвейеры

Конвейер делает три вещи: **извлекает** данные из источников, **преобразует** (очистка, агрегация, обогащение) и **загружает** в целевое хранилище.

```
Источники (OLTP, логи, события, API) → [pipeline: extract → transform → load] → DWH/Lake/Lakehouse → потребители
```

> Конвейер — это «кровеносная система» аналитики. Его свойства определяют всё: свежесть (как быстро данные доходят), нагрузку на источники (как извлекаем), надёжность (что при сбое), качество (что валидируем) и прослеживаемость (откуда данные). Дальше — ключевые оси проектирования конвейера.

## 2. ETL vs ELT

| | **ETL** (Extract-Transform-Load) | **ELT** (Extract-Load-Transform) |
|--|----------------------------------|----------------------------------|
| Порядок | преобразовать **до** загрузки | загрузить сырое, преобразовать **в** хранилище |
| Где трансформация | на промежуточном движке | внутри целевого хранилища (его вычислительная мощь) |
| Эпоха | классический DWH (схема фиксирована) | современная (Lake/Lakehouse/облачный DWH) |
| Гибкость | жёстче (схема заранее) | гибче (сырьё хранится, трансформируй как нужно) |
| Когда | строгая схема, ограниченное целевое хранилище | мощное масштабируемое хранилище, сырьё нужно сохранить |

```
ETL: extract → TRANSFORM → load (в хранилище кладём уже готовое)
ELT: extract → load (сырое) → TRANSFORM (внутри хранилища, по необходимости)
```

> Сдвиг от ETL к **ELT** произошёл потому, что современные хранилища (Snowflake/BigQuery/Lakehouse) дёшевы и мощны: проще залить сырьё и трансформировать на месте их вычислительными ресурсами, сохранив исходник (bronze-слой, модуль 19). ETL остаётся, когда схема строгая, целевое хранилище слабое или нужно не хранить сырые/чувствительные данные.

## 3. Batch vs streaming

| | **Batch** | **Streaming** |
|--|-----------|---------------|
| Когда обрабатывает | периодически порциями (ночной job) | непрерывно, по событию (модуль 15) |
| Latency | высокая (часы) | низкая (секунды) |
| Throughput | высокий | высокий, но сложнее |
| Сложность | проще | сложнее (порядок, окна, состояние) |
| Когда | исторические отчёты, тяжёлая обработка | real-time дашборды, фрод, алерты |

**Lambda и Kappa архитектуры:**
```
Lambda: batch layer (точная история) + speed layer (быстрая свежая) → объединение
        ↑ две системы, дублирование логики
Kappa:  всё через единый стриминг (и история — переигрыванием лога)
        ↑ проще, одна система
```

> Выбор по требуемой **свежести**: batch для исторической аналитики (просто, дёшево), streaming для real-time (фрод, живые метрики). **Lambda** даёт и точность batch, и скорость stream ценой двух систем; **Kappa** упрощает до одного стрима (история — replay лога, модуль 15). Не делайте streaming там, где хватает суточного batch — это лишняя сложность.

## 4. CDC (Change Data Capture)

**CDC** — захват изменений из OLTP-БД (insert/update/delete) через её **лог транзакций (binlog/WAL)** и стриминг их в аналитику/индексы/другие сервисы.

```
Полный ETL:  каждую ночь SELECT * всей таблицы → дорого, грузит OLTP, не свежо
CDC:         читаем лог транзакций OLTP → стрим только изменений → дёшево, свежо, не грузит запросами
             (Debezium → Kafka → DWH/индекс/сервисы)
```

| Свойство CDC | |
|--------------|--|
| Источник изменений | лог транзакций БД (binlog MySQL, WAL Postgres) |
| Нагрузка на OLTP | минимальная (читаем лог, не выполняем SELECT'ы) |
| Свежесть | near-real-time (поток изменений) |
| Инструмент | Debezium (+ Kafka, модуль 15) |
| Применение | синхронизация DWH, поисковых индексов (модуль 17), кэшей, других сервисов |

> CDC — элегантная альтернатива дорогому полному ETL: вместо «выгрузить всю таблицу» читаем поток изменений из лога БД, не нагружая её запросами и получая near-real-time. Это тот же механизм, что синхронизирует поисковый индекс (модуль 17) и реализует Outbox-паттерн (модуль 23). Тонкость — учитывать эволюцию схемы источника.

## 5. Оркестрация конвейеров

Реальный конвейер — это множество зависимых задач (extract из 5 источников → join → агрегация → загрузка). Их координирует **оркестратор**.

| Возможность оркестратора | |
|--------------------------|--|
| **DAG** (граф задач) | задаёт зависимости: что после чего |
| Расписание | запуск по времени/событию |
| Retry/обработка сбоев | повтор упавшей задачи, алерты |
| Мониторинг | статус, длительность, SLA задач |
| Backfill | переобработка исторических периодов |

```
Airflow/Dagster DAG:
  extract_orders ─┐
  extract_users  ─┼→ join → aggregate → load_to_dwh → notify
  extract_events ─┘
```

> Без оркестратора зависимые задачи превращаются в «cron-ад»: десятки расписаний, надеющихся, что предыдущая задача успела. Оркестратор (Airflow, Dagster) делает зависимости явными (DAG), управляет retry, расписанием и backfill. Это инфраструктура надёжности конвейера.

## 6. Надёжность: идемпотентность и backfill

Конвейеры падают и перезапускаются — значит, повторный запуск **не должен дублировать данные**.

| Свойство | Зачем |
|----------|-------|
| **Идемпотентность** | повторный запуск задачи даёт тот же результат (upsert/перезапись партиции, а не слепой insert) |
| **Backfill** | возможность переобработать прошлые периоды (новая логика, исправление, пропуск) |
| Атомарность загрузки | партиция загружается целиком или никак (не «полузагрузка») |

> Идемпотентность — то же требование, что у consumer'а очереди (модуль 15): retry неизбежен, повтор не должен дублировать. На практике — перезапись партиции по дате (idempotent overwrite) вместо накопительного insert. **Backfill** обязателен: при изменении логики или баге нужно пересчитать историю из сырья (bronze, модуль 19).

## 7. Data quality и lineage

| | **Data quality** | **Data lineage** |
|--|-------------------|------------------|
| Что | валидация/тесты данных на входе и между слоями | прослеживание происхождения данных |
| Зачем | не пустить мусор в аналитику (схема, диапазоны, null, дубли) | отладка, доверие, compliance, оценка влияния изменений |
| Пример | «amount ≥ 0», «схема соответствует», «нет дублей по id» | «эта метрика собрана из таблиц X,Y через трансформации A,B» |

> **Data quality** — фильтр на входе: без валидации мусор затекает в аналитику и рушит доверие (та же проблема, что schema-on-read без контроля, модуль 19). **Data lineage** отвечает на «откуда взялось это число и как посчитано» — критично для отладки («почему отчёт сломался?»), доверия и compliance (GDPR — где чьи данные). Оба — часть governance (модуль 19).

## ⚠️ Подводные камни

- **ETL там, где ELT гибче (и наоборот).** ELT для мощных хранилищ с сохранением сырья; ETL — строгая схема/слабое хранилище.
- **Streaming там, где хватает batch.** Лишняя сложность; real-time только под реальную потребность в свежести.
- **Полный ETL вместо CDC.** `SELECT *` всей таблицы грузит OLTP и не свеж; CDC читает лог изменений.
- **Неидемпотентный конвейер.** Retry дублирует данные; используйте перезапись партиции/upsert.
- **Нет data quality.** Мусор затекает в аналитику, рушит доверие; валидируйте на входе и между слоями.
- **Нет lineage.** Невозможно отладить/доверять/выполнить compliance; отслеживайте происхождение.
- **Cron-ад вместо оркестрации.** Неявные зависимости ломаются; используйте DAG-оркестратор.
- **Игнорировать эволюцию схемы источника.** CDC/ETL сломаются при изменении схемы; планируйте schema evolution.
- **Нет backfill.** Невозможно пересчитать историю при изменении логики/баге; стройте на сырье (bronze).
