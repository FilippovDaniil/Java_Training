# Задача 04 — Модуль 21: CAP/PACELC

## ЗАДАНИЕ
Разберите систему по PACELC: поведение и при partition, и в нормальной работе.

## ВХОДНЫЕ ДАННЫЕ
Две системы:
- (А) глобальная соцсеть, выбравшая высокую доступность и низкую latency (Cassandra-подобная);
- (Б) финансовая система учёта, выбравшая строгую согласованность (Spanner/традиционный SQL-подобная).

## ТРЕБОВАНИЯ
- Для каждой системы укажите выбор при partition (A vs C) и в нормальной работе (L vs C) — классификация PACELC.
- Объясните, почему даже без partition сильная согласованность стоит latency.
- Объясните, почему AP-система при этом обычно ещё и быстра (низкая latency).
- Сопоставьте: PA/EL vs PC/EC.

## ФОРМАТ РЕЗУЛЬТАТА
PACELC-классификация (А) и (Б) + почему strong = latency-цена + почему AP быстра + сопоставление.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) → PA/EL (при partition — доступность; в норме — низкая latency, жертвует согласованностью).
- [ ] (Б) → PC/EC (при partition — согласованность; в норме — согласованность ценой latency).
- [ ] Объяснено: strong consistency требует координации (кворум/синхронная репликация) → сетевые round-trips → latency даже без partition.
- [ ] Объяснено: AP-система не координирует на каждый запрос → ниже latency.
- [ ] Сопоставление PA/EL vs PC/EC корректно.

## ПОДСКАЗКА
PACELC = CAP + «а что в нормальной работе». Главное добавление: strong consistency стоит latency всегда (нужна координация узлов), не только доступности при partition. Поэтому AP-системы (PA/EL) и доступны, и быстры; strongly-consistent (PC/EC) платят latency.
