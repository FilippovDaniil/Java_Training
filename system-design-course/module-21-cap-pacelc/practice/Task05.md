# Задача 05 — Модуль 21: CAP/PACELC

## ЗАДАНИЕ
Разнесите данные одной системы по уровням спектра согласованности (strong / read-your-writes / eventual).

## ВХОДНЫЕ ДАННЫЕ
Маркетплейс. Данные: (1) баланс кошелька продавца; (2) лента рекомендаций товаров; (3) только что написанный пользователем отзыв (он должен сразу видеть его сам); (4) счётчик «N человек смотрят этот товар»; (5) остаток товара при покупке.

## ТРЕБОВАНИЯ
- Для каждого типа данных назначьте уровень согласованности (strong / read-your-writes / monotonic / eventual) и обоснуйте.
- Объясните, почему не назначить «strong для всего».
- Покажите, где промежуточные уровни (read-your-writes) уместнее, чем крайности.
- Свяжите с ценой (latency/доступность) каждого уровня.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: данные → уровень → обоснование + почему не «strong для всего» + про цену.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) баланс, (5) остаток → strong (несогласованность недопустима).
- [ ] (3) свой отзыв → read-your-writes (автор сразу видит своё, остальным — eventual).
- [ ] (2) рекомендации, (4) счётчик смотрящих → eventual (устаревание терпимо).
- [ ] Объяснено: «strong для всего» — лишняя latency и хуже доступность там, где не нужно.
- [ ] Показано, что read-your-writes — полезная середина (не strong, но и не чистый eventual).
- [ ] Уровни увязаны с ценой (strong дороже по latency/доступности).

## ПОДСКАЗКА
Согласованность — спектр, не рубильник. Деньги/остатки — strong. «Своё только что созданное» — read-your-writes (модуль 12). Рекомендации/счётчики — eventual. Назначать «strong везде» — дорого по latency и доступности без нужды.
