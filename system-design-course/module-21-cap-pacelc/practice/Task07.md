# Задача 07 — Модуль 21: CAP/PACELC

## ЗАДАНИЕ
Оцените latency-цену сильной согласованности (PACELC, ветка Else) и обоснуйте компромисс.

## ВХОДНЫЕ ДАННЫЕ
Глобальный сервис с репликами в 3 регионах (Европа, США, Азия). Для критичных данных рассматривается строгая согласованность через кворум записи по регионам (нужно подтверждение большинства реплик в разных регионах). Межрегиональный round-trip ≈ 100–150 мс.

## ТРЕБОВАНИЯ
- Объясните, почему strong consistency здесь добавляет latency даже без сетевого сбоя (PACELC Else).
- Прикиньте порядок добавочной latency на запись при кворуме по регионам.
- Сопоставьте: что выиграем (согласованность) и что потеряем (скорость) для критичных данных.
- Предложите, какие данные оставить eventual ради latency, а какие держать strong.

## ФОРМАТ РЕЗУЛЬТАТА
Почему strong = latency (PACELC Else) + оценка добавочной latency + компромисс + per-data решение.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: strong consistency требует координации (кворум по регионам) → межрегиональные round-trips на каждую запись, даже без partition.
- [ ] Оценка: запись ждёт подтверждения большинства регионов → добавочная latency порядка межрегионального RTT (~100+ мс), числа как порядок.
- [ ] Сопоставление: согласованность критичных данных ценой заметной latency записи.
- [ ] Per-data: деньги/критичное — strong (платим latency); лента/счётчики — eventual/локально (быстро).
- [ ] Вывод увязан с PACELC (Else: Latency vs Consistency).

## ПОДСКАЗКА
PACELC Else: даже без partition strong consistency = latency, потому что кворум по регионам ждёт межрегиональные подтверждения (~RTT). Это объясняет, почему глобально-согласованные системы (Spanner) медленнее на запись. Решение — strong только для критичного, остальное eventual/локально.
