# Задача 08 — Модуль 21: CAP/PACELC

## ЗАДАНИЕ
Спроектируйте смешанную модель согласованности: разные гарантии для разных данных одной системы.

## ВХОДНЫЕ ДАННЫЕ
Сервис заказов и платежей с социальными элементами: платежи/балансы, оформление заказа с резервом товара, лента «друзья купили», счётчики популярности товаров, профили пользователей. Глобальный масштаб, partition возможен.

## ТРЕБОВАНИЯ
- Для каждой группы данных назначьте CP/AP (и уровень согласованности) с обоснованием.
- Объясните, почему смешивание гарантий — норма, а не компромисс «недоделки».
- Покажите, как сосуществуют CP- и AP-подсистемы в одной системе.
- Назовите, какие данные допускают eventual ради доступности/latency.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: данные → CP/AP + уровень → обоснование + про сосуществование подсистем.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Платежи/балансы, резерв товара → CP/strong.
- [ ] Лента «друзья купили», счётчики популярности → AP/eventual.
- [ ] Профили → eventual или read-your-writes (своё видишь сразу).
- [ ] Объяснено: смешивание гарантий — правильный дизайн (per-data), а не недоработка.
- [ ] Показано сосуществование: критичные подсистемы CP, социальные AP, в одной системе.
- [ ] Названы данные, осознанно оставленные eventual ради доступности/latency.

## ПОДСКАЗКА
Реальные системы почти всегда смешанные: финансовое ядро CP, социальная обвязка AP. Это не компромисс, а грамотный per-data-выбор по цене несогласованности. Платежи нельзя «в итоге», ленту — можно. Подсистемы с разными гарантиями спокойно сосуществуют.
