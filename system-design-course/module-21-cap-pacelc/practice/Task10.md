# Задача 10 — Модуль 21 (мини-проект): Модель согласованности системы

## ЗАДАНИЕ
Постройте модель согласованности для системы: per-data CAP/PACELC-выбор с обоснованием. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) глобальный маркетплейс; (б) платёжная платформа с социальными функциями; (в) сервис бронирования с лентой и отзывами. Масштаб глобальный, partition возможен.

## ТРЕБОВАНИЯ
Модель должна покрывать:
1. **Инвентаризация данных** — ключевые типы данных системы.
2. **CP/AP per-data** — для каждого типа: CP или AP, с обоснованием ценой несогласованности.
3. **Уровень согласованности** — точка на спектре (strong/read-your-writes/eventual).
4. **PACELC** — что выбираете в норме (Latency vs Consistency) для критичных данных.
5. **Поведение при partition** — что делает система с каждой группой данных.
6. **Выбор хранилищ** — CP/AP-БД под группы данных.
7. **Trade-offs** — что осознанно жертвуете и почему (доступность/latency/согласованность).

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов + таблица «данные → гарантии → хранилище».

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; данные инвентаризированы.
- [ ] Для каждого типа данных назначен CP/AP и уровень согласованности с обоснованием.
- [ ] Указан PACELC-выбор (поведение в норме) для критичных данных, с latency-ценой.
- [ ] Описано поведение каждой группы при partition (отказ vs устаревание).
- [ ] Подобраны хранилища под CP/AP-профили.
- [ ] Названы осознанные trade-offs per-data; нет «strong для всего» без нужды.
- [ ] Связь с ACID/BASE (11) и репликацией (12).

## ПОДСКАЗКА
Это синтез модуля: модель согласованности = per-data-карта (что CP, что AP, какой уровень, что при partition, какое хранилище). Финансовое ядро — CP/strong (платим latency и отказом при partition), социальная обвязка — AP/eventual. Дальше — модуль 22: как технически достигают согласия (консенсус).
