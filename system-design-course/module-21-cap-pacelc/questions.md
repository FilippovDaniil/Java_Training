6. Что такое AP-система?
7. Как выбрать между CP и AP для данных?

## Ключ (эталонные ответы)

1. В распределённой системе при сетевом разделении (partition) нельзя одновременно обеспечить и согласованность (C), и доступность (A) — приходится выбирать одно.
2. **C** — каждое чтение видит последнюю запись (линеаризуемость, все узлы согласованы). **A** — каждый запрос получает ответ без ошибки. **P** — система продолжает работать несмотря на потерю сообщений между узлами.
3. Потому что сети ненадёжны: пакеты теряются, связь рвётся, узлы изолируются. Это происходит всегда, поэтому распределённая система обязана переживать partition — P не опциональна.
4. Не «2 из 3», а C **vs** A **во время** partition: система либо отказывает в обслуживании ради согласованности (CP), либо отвечает ценой устаревания (AP).
5. **CP** — при partition предпочитает согласованность: отказывает в обслуживании, лишь бы не отдать несогласованное (банки, инвентарь, координация).
6. **AP** — при partition предпочитает доступность: продолжает отвечать, мирясь с временной несогласованностью (лента, лайки, корзина, метрики).
7. По цене несогласованности: если устаревшее/несогласованное значение опасно (двойное списание, овербукинг) → CP; если терпимо (лента, счётчики) → AP.
8. Потому что P в распределёнке неизбежна — её нельзя «не выбрать». «CA без P» означало бы одноузловую систему. Реальный выбор — C или A при partition, а не любые 2 из 3.
9. Нет. CAP описывает поведение **во время** partition. Вне partition система может обеспечивать и C, и A одновременно. Это и достраивает PACELC.
10. **CAP-C** — линеаризуемость (одна актуальная копия данных среди узлов). **ACID-C** — соблюдение инвариантов/целостности транзакции. Это разные понятия, несмотря на общее слово.
11. PACELC достраивает CAP для нормальной работы: if **P**artition → A vs C; **E**lse (сеть исправна) → **L**atency vs **C**onsistency.
12. В ветке Else (сеть исправна) выбирается между низкой latency и сильной согласованностью — даже без сбоя нельзя иметь и максимально быстро, и строго согласованно.
13. Потому что для согласования последнего значения между узлами нужна координация (кворум, синхронная репликация) — а это сетевые round-trips, добавляющие latency на каждую операцию.
14. Потому что они не координируют узлы на каждый запрос (не ждут кворума/синхронной репликации) — отвечают локально, жертвуя строгой согласованностью ради скорости.
15. **PA/EL** — при partition выбирает доступность, в нормальной работе — низкую latency (жертвуя согласованностью). Пример: Cassandra, DynamoDB.
16. **PC/EC** — при partition выбирает согласованность, в нормальной работе — тоже согласованность ценой latency. Пример: HBase, BigTable, традиционные SQL/Spanner.
17. Согласованность — не «вкл/выкл», а диапазон гарантий от strong (линеаризуемость) через промежуточные уровни до eventual (сойдётся «в итоге»). Выбирают точку под конкретные данные.
18. Causal consistency, read-your-writes (видишь свои записи), monotonic reads (нет отката во времени) — полезные промежуточные гарантии между strong и eventual (модуль 12).
19. Strong consistency дороже: добавляет latency (координация) и ухудшает доступность при partition. Там, где несогласованность терпима, это лишняя цена без выгоды.
20. Назначать уровень согласованности отдельно каждому типу данных по цене несогласованности: деньги — strong/CP, лента — eventual/AP, свои посты — read-your-writes. Не «вся система одного типа».
21. **CP**: HBase, Zookeeper, etcd, традиционные SQL. **AP**: Cassandra, DynamoDB, Riak.
22. Что приложение обязано само разгребать несогласованность: разрешать конфликты записи, обрабатывать устаревшие чтения (LWW/CRDT/merge, модуль 12), а не полагаться, что «БД уладит».
23. CP/AP — это тот же выбор, что ACID vs BASE, но в момент сетевого сбоя: ACID/CP — строгие гарантии, BASE/AP — доступность и масштаб ценой временной несогласованности.
24. Strong consistency достигается синхронной репликацией и кворумами (W+R>N, модуль 12) — это и есть механизм CP, оплачиваемый latency (PACELC); async/низкие кворумы дают AP/eventual.
25. Инвентаризация данных; CP/AP и уровень согласованности per-data с обоснованием; PACELC-выбор для критичных данных (latency-цена); поведение каждой группы при partition; выбор CP/AP-хранилищ; осознанные trade-offs.
