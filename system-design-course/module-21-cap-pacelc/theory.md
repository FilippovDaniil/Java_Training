# Модуль 21 — CAP/PACELC и выбор согласованности

## Зачем этот модуль

Части 3–4 строили распределённую систему: данные реплицируются (модуль 12), шардируются (модуль 13), синхронизируются через очереди (модуль 15) и конвейеры (модуль 20). Но у распределённости есть фундаментальный закон, который нельзя обойти инженерией: **при сбое сети нельзя одновременно гарантировать и согласованность, и доступность**. Это CAP-теорема — и понимание её следствий отличает инженера, который проектирует осознанно, от того, кто надеется на удачу.

Зачем это senior'у. CAP — это язык, на котором обосновывают выбор «банк должен отказать в операции при сбое сети, а лента — продолжить работать с устаревшими данными». PACELC идёт дальше: даже когда сеть исправна, сильная согласованность стоит **latency**. Эти теоремы переводят расплывчатое «надёжность» в конкретный компромисс под каждый тип данных.

Это первый модуль Части 5. Он связывает ACID/BASE (модуль 11) и репликацию/кворумы (модуль 12) в единую рамку выбора согласованности, на которой строятся консенсус (модуль 22) и надёжность операций (модуль 23).

## 1. CAP-теорема

Три свойства распределённой системы:

| Свойство | Значит |
|----------|--------|
| **C** — Consistency | каждое чтение видит последнюю запись (линеаризуемость), все узлы согласованы |
| **A** — Availability | каждый запрос получает ответ (без ошибки), система отвечает |
| **P** — Partition tolerance | система работает несмотря на потерю/задержку сообщений между узлами (сетевое разделение) |

**Теорема:** при сетевом разделении (**P**) нельзя одновременно обеспечить и **C**, и **A** — приходится выбирать.

```
Сеть разделилась на две половины, которые не видят друг друга:
  - продолжить отвечать обеим? → рискуешь отдать рассогласованное (выбрал A, пожертвовал C)
  - отказать, пока не восстановится связь? → недоступен (выбрал C, пожертвовал A)
```

> CAP — не про «выбери 2 из 3». P (сетевое разделение) в распределённой системе **неизбежна** — сети рвутся, пакеты теряются. Поэтому P не опциональна: реальный выбор — что делать **во время** partition: предпочесть **C** (CP) или **A** (AP).

---

## 2. CP vs AP

| | **CP** (consistency over availability) | **AP** (availability over consistency) |
|--|----------------------------------------|----------------------------------------|
| При partition | отказывает в обслуживании, чтобы не отдать несогласованное | продолжает обслуживать, мирясь с устареванием |
| Гарантия | данные всегда согласованы (или ошибка) | система всегда отвечает (но, может, устаревшим) |
| Когда | деньги, инвентарь, бронирование, конфигурация | лента, лайки, корзина, профили, метрики |
| Примеры БД | HBase, Zookeeper, etcd, традиционные SQL | Cassandra, DynamoDB, Riak |

```
CP-пример: банк при partition не даёт списать (лучше ошибка, чем двойное списание)
AP-пример: соцсеть при partition показывает чуть устаревшую ленту (лучше старое, чем недоступность)
```

> Выбор CP/AP — про последствия несогласованности. Если устаревшее значение опасно (списать деньги дважды, продать последний товар двоим) → **CP** (лучше отказать). Если устаревшее терпимо (лента, счётчик лайков) → **AP** (лучше доступность). Это тот же вопрос, что ACID vs BASE (модуль 11), но в момент сетевого сбоя.

## 3. Частые заблуждения о CAP

| Миф | Правда |
|-----|--------|
| «Выбери любые 2 из 3» | P неизбежна; выбор — только C vs A при partition |
| «CAP описывает обычную работу системы» | CAP — про поведение **во время** partition; вне partition можно иметь и C, и A |
| «P можно не выбирать (сделать систему без partition)» | В распределёнке сеть ненадёжна всегда; P обязательна |
| «C в CAP = C в ACID» | CAP-C = линеаризуемость (одна актуальная копия); ACID-C = инварианты/целостность транзакции — это разные вещи |

> Самое вредное заблуждение — «2 из 3», будто можно выбрать CA, отказавшись от P. В распределённой системе так нельзя: отказ от P означает «работаем на одном узле» (не распределённая система). Запомните: **P дана, выбираешь C или A при сбое сети.**

## 4. PACELC — расширение CAP

CAP молчит о том, что происходит, когда сеть **исправна**. PACELC закрывает пробел:

```
PACELC:
  if Partition (P):  выбирай между Availability (A) и Consistency (C)
  Else (E):          выбирай между Latency (L) и Consistency (C)
```

| Классификация | Значит |
|---------------|--------|
| **PA/EL** | при partition — доступность; в норме — низкая latency (Cassandra, DynamoDB) |
| **PC/EC** | при partition — согласованность; в норме — согласованность ценой latency (HBase, BigTable, традиционные SQL) |
| **PA/EC**, **PC/EL** | смешанные настройки (зависит от конфигурации) |

> **Главное добавление PACELC:** даже без сбоя сети сильная согласованность **стоит latency** — чтобы все узлы согласовали последнее значение, нужна координация (кворум, синхронная репликация — модуль 12), а это сетевые round-trips. То есть «strong consistency» вы оплачиваете не только доступностью при partition, но и скоростью в обычной работе. Это объясняет, почему AP-системы (Cassandra) ещё и быстры, а strongly-consistent (Spanner) платят latency.

---

## 5. Спектр согласованности

Согласованность — не «вкл/выкл», а спектр (связь с модулем 12):

```
Strong (linearizable)  — чтение всегда видит последнюю запись
   ↑ дороже: latency, координация, хуже доступность при partition
Causal / read-your-writes / monotonic  — частичные гарантии
   ↓ дешевле: быстрее, доступнее
Eventual               — копии сойдутся «в итоге», без гарантий когда
```

> CAP/PACELC — не бинарный рубильник, а выбор точки на спектре под конкретные данные. Между strong и eventual есть полезные промежуточные уровни (causal, read-your-writes — модуль 12). Хороший дизайн назначает каждому типу данных свой уровень: деньги — strong, лента — eventual, «свои» посты пользователя — read-your-writes.

---

## 6. Как выбирать: разные данные — разные гарантии

```
Одна система, разные данные:
  баланс счёта        → CP / strong  (несогласованность недопустима)
  лента активности    → AP / eventual (устаревание терпимо, важна доступность)
  свои недавние посты  → read-your-writes (вижу то, что только что написал)
  счётчик просмотров   → AP / eventual (приблизительно — ок)
```

> Главный навык: не выбирать «всю систему CP» или «всю AP», а назначать гарантии **по типу данных** исходя из цены несогласованности. Это прямое продолжение разнесения ACID/BASE (модуль 11) и уровней согласованности (модуль 12). На интервью именно per-data-обоснование («деньги CP, потому что…; лента AP, потому что…») — признак зрелости.

## ⚠️ Подводные камни

- **Миф «2 из 3».** P неизбежна; выбор — только C vs A при partition, не «откажемся от P».
- **Думать, что CAP про всё время.** CAP — про поведение во время partition; вне него можно иметь и C, и A.
- **Путать CAP-C и ACID-C.** CAP-C — линеаризуемость; ACID-C — инварианты транзакции; это разное.
- **Игнорировать PACELC latency-цену.** Даже без partition strong consistency стоит latency (координация).
- **Одна гарантия на всю систему.** Назначайте уровень per-data по цене несогласованности.
- **AP без понимания последствий.** Приложение должно разгребать несогласованность (конфликты, устаревание — модуль 12).
- **CP там, где хватило бы AP.** Лишняя строгость = хуже доступность и latency без нужды.
- **Считать согласованность бинарной.** Есть полезный спектр (causal, read-your-writes), не только strong/eventual.
