# Задача 01 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Определите, где в системе нужен консенсус, а где можно обойтись без него.

## ВХОДНЫЕ ДАННЫЕ
Задачи в распределённой системе:
- (А) выбрать единственного лидера среди реплик БД при отказе текущего;
- (Б) обработать независимые HTTP-запросы пользователей (stateless);
- (В) согласовать текущую конфигурацию кластера (какие узлы активны);
- (Г) инкрементировать счётчик лайков (устаревание терпимо);
- (Д) гарантировать, что только один воркер выполняет периодическую задачу.

## ТРЕБОВАНИЯ
- Для каждой задачи укажите, нужен ли консенсус, и обоснуйте.
- Для задач без консенсуса предложите более дешёвую альтернативу.
- Объясните общий критерий «нужен ли консенсус».

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: задача → нужен консенсус? → обоснование → (если нет) альтернатива.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (А) лидер, (В) конфигурация, (Д) единственный воркер (distributed lock) → нужен консенсус (единственное согласованное решение).
- [ ] (Б) независимые stateless-запросы → не нужен (нет общего решения).
- [ ] (Г) счётчик лайков → не нужен (eventual consistency, модуль 21).
- [ ] Для (Б)/(Г) предложены альтернативы (балансировка/идемпотентность/eventual).
- [ ] Критерий назван: консенсус нужен, когда требуется ОДНО согласованное решение для группы узлов, переживающее отказы.

## ПОДСКАЗКА
Консенсус нужен там, где требуется единственное согласованное решение (лидер, конфиг, кто держит lock). Независимые операции и eventual-данные в нём не нуждаются — там дешевле балансировка/идемпотентность/eventual.
