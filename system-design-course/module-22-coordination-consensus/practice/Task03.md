# Задача 03 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Опишите работу Raft «на пальцах» для кластера: leader election и репликация лога.

## ВХОДНЫЕ ДАННЫЕ
Кластер из 5 узлов на Raft хранит реплицируемый лог команд (например, изменения конфигурации). Нужно: выбрать лидера, реплицировать команды, пережить отказ лидера.

## ТРЕБОВАНИЯ
- Опишите leader election: как из follower'ов выбирается лидер (роль candidate, term, голоса большинства).
- Опишите log replication: как команда от клиента попадает в лог и когда считается зафиксированной (commit).
- Опишите, что происходит при отказе лидера (новые выборы, новый term).
- Объясните роль term (эпохи) в защите от «старого» лидера.

## ФОРМАТ РЕЗУЛЬТАТА
Leader election → log replication (+commit) → отказ лидера → роль term.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Leader election: по таймауту follower становится candidate, увеличивает term, просит голоса; набрал большинство (3 из 5) → leader.
- [ ] Log replication: лидер пишет команду в лог и реплицирует follower'ам; команда commit'ится, когда её приняло большинство.
- [ ] Отказ лидера: нет heartbeat → follower'ы начинают новые выборы с новым term.
- [ ] Term: монотонно растёт; старый лидер с меньшим term отвергается (защита от split-brain).
- [ ] Зафиксированное (commit) не теряется (safety).

## ПОДСКАЗКА
Raft = «лидер + большинство + term». Лидер выбирается голосами большинства (нечётное N → нет ничьи). Команды идут через лидера в лог и commit'ятся при репликации на большинство. Term отсекает устаревших лидеров. Этого уровня понимания достаточно для дизайна.
