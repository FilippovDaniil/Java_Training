# Задача 04 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Спроектируйте безопасный leader election, исключающий split-brain (продолжение модуля 12).

## ВХОДНЫЕ ДАННЫЕ
Single-leader БД (модуль 12): при отказе лидера нужно выбрать нового. Текущая наивная схема: реплика, не видящая лидера, объявляет лидером себя. При partition это даёт двух лидеров.

## ТРЕБОВАНИЯ
- Объясните, почему наивная схема приводит к split-brain при partition.
- Спроектируйте выбор лидера через консенсус (кворум большинства).
- Объясните роль term/эпохи и fencing старого лидера.
- Покажите, что происходит при partition: какая сторона выбирает лидера, какая — нет.

## ФОРМАТ РЕЗУЛЬТАТА
Почему split-brain → выбор через кворум → роль term + fencing → поведение при partition.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: «реплика сама себя назначает» → при partition обе стороны независимо выберут лидера → два лидера (split-brain).
- [ ] Решение: лидер легитимен только при голосах большинства (кворум) — через консенсус (etcd/Zookeeper/Raft).
- [ ] Term/эпоха: новый лидер увеличивает term; старый с меньшим term отвергается.
- [ ] Fencing: старого лидера изолируют/заглушают, чтобы он не принимал запись.
- [ ] При partition: только сторона с кворумом (большинством) выбирает лидера; меньшая сторона блокируется.

## ПОДСКАЗКА
Split-brain невозможен, если лидером становятся только с согласия большинства — две половины не могут обе иметь кворум. Term отсекает «ожившего» старого лидера, fencing мешает ему писать. «Безопасный failover» = «выбор лидера через консенсус» (связь с модулем 12).
