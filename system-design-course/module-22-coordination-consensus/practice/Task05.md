# Задача 05 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Спроектируйте distributed lock с защитой от упавшего держателя (TTL/lease).

## ВХОДНЫЕ ДАННЫЕ
Нужно гарантировать, что периодическую задачу (например, ночной биллинг) одновременно выполняет только один воркер из пула. Используется distributed lock. Наблюдение: иногда воркер берёт lock и падает — и задача больше никогда не выполняется (lock висит вечно).

## ТРЕБОВАНИЯ
- Объясните, почему distributed lock без TTL опасен (упавший держатель).
- Спроектируйте lock с lease/TTL: как lock автоматически освобождается.
- Объясните, как держатель продлевает lease, пока работает (heartbeat/renew).
- Назовите остаточную проблему «TTL истёк, а работа ещё идёт» (мост к fencing token, задача 06).

## ФОРМАТ РЕЗУЛЬТАТА
Опасность lock без TTL → дизайн с lease/TTL → продление lease → остаточная проблема.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Опасность: держатель упал с lock → без TTL блокировка вечная, задача не выполняется.
- [ ] Решение: lock с lease/TTL — автоматически истекает, если держатель не продлевает.
- [ ] Продление: держатель периодически продлевает lease (renew/heartbeat), пока работает.
- [ ] Названа остаточная проблема: если держатель «завис» (GC-пауза) и TTL истёк, lock уйдёт другому, а первый ещё может писать → нужен fencing token (задача 06).
- [ ] Решение использует coordination service (etcd/Zookeeper), а не самописное.

## ПОДСКАЗКА
Distributed lock ≠ мьютекс: держатель может умереть, не отпустив lock. Лечит lease/TTL — lock сам истекает. Держатель продлевает его, пока жив. Но остаётся щель: «завис, TTL истёк, lock у другого, а я ещё пишу» — это закрывает fencing token (задача 06).
