# Задача 06 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Решите проблему «зависшего держателя» distributed lock с помощью fencing token.

## ВХОДНЫЕ ДАННЫЕ
Сценарий: воркер A взял lock (TTL), начал писать в общий ресурс (БД), но попал в долгую GC-паузу. TTL истёк, lock перешёл воркеру B, который тоже начал писать. Затем A «проснулся» и продолжил писать, не зная, что lock уже не его → порча данных.

## ТРЕБОВАНИЯ
- Объясните, почему TTL сам по себе не спасает в этом сценарии.
- Спроектируйте fencing token: что выдаётся при захвате lock и как используется.
- Покажите, как ресурс (БД/хранилище) отвергает запись «зависшего» A.
- Объясните, почему проверка токена должна быть на стороне ресурса.

## ФОРМАТ РЕЗУЛЬТАТА
Почему TTL недостаточно → дизайн fencing token → отвержение «зависшего» → где проверять.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: TTL освобождает lock, но «зависший» A не знает об этом и продолжает писать после потери lock.
- [ ] Fencing token: lock-сервис выдаёт монотонно растущий токен при каждом захвате (A=33, B=34).
- [ ] Ресурс принимает запись только с токеном ≥ последнего виденного → запись A (токен 33) отвергается после того, как B (34) уже писал.
- [ ] Проверка токена — на стороне ресурса (БД), т.к. только он может отсечь устаревшего держателя.
- [ ] Решение закрывает щель, которую TTL оставляет открытой.

## ПОДСКАЗКА
TTL освобождает lock, но не уведомляет «зависшего» держателя. Fencing token — монотонный номер захвата: ресурс отвергает запись с токеном меньше последнего виденного. Так старый A (токен 33) не сможет писать после B (34). Проверка обязательно на стороне ресурса.
