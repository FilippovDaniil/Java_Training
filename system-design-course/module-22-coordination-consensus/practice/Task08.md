# Задача 08 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне координации и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Напишем свой простой алгоритм выбора лидера: кто первый заметил отказ — тот и лидер».
2. «Кластер консенсуса из 4 узлов — чётное число надёжнее».
3. «Distributed lock делаем простым флагом в Redis без TTL».
4. «Lock без fencing token — TTL же есть, этого достаточно».
5. «Каждую операцию записи проводим через консенсус для надёжности».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка вернее всего приведёт к split-brain или порче данных.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Самописный «кто первый» → split-brain при partition → выбор через кворум большинства (etcd/Zookeeper).
- [ ] (2) Чётное N=4 → возможна «ничья» 2+2, та же отказоустойчивость, что N=3 → нечётное число узлов.
- [ ] (3) Lock без TTL → упавший держатель блокирует навечно → lease/TTL.
- [ ] (4) Lock без fencing token → зависший держатель повредит данные после TTL → fencing token на стороне ресурса.
- [ ] (5) Консенсус на каждую запись → дорого по latency → консенсус только для критичных решений; обычные операции — идемпотентность/eventual.
- [ ] Отмечено: (1) ведёт к split-brain, (4) — к порче данных.

## ПОДСКАЗКА
Чек-лист модуля: не писать консенсус самому; нечётное число узлов; lock с TTL/lease; lock с fencing token; консенсус только для критичного. Пункт 1 — прямой split-brain, пункт 4 — порча данных зависшим держателем.
