# Задача 09 — Модуль 22: Координация и консенсус

## ЗАДАНИЕ
Определите, где консенсус избыточен, и предложите более дешёвые альтернативы.

## ВХОДНЫЕ ДАННЫЕ
Команда внедрила консенсус (через etcd) буквально на каждую операцию ради «максимальной надёжности». Симптомы: высокая latency, etcd стал узким местом, throughput низкий. Операции: (А) обработка повторно доставленных сообщений; (Б) подсчёт уникальных посетителей; (В) запись независимых событий аудита; (Г) выбор лидера (это реально нужно).

## ТРЕБОВАНИЯ
- Объясните, почему консенсус на каждую операцию убивает производительность.
- Для (А), (Б), (В) предложите дешёвую альтернативу консенсусу.
- Подтвердите, что (Г) — обоснованное применение консенсуса.
- Сформулируйте правило: когда консенсус оправдан.

## ФОРМАТ РЕЗУЛЬТАТА
Почему дорого → альтернативы для (А)/(Б)/(В) → подтверждение (Г) → правило.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: консенсус = кворумные round-trips на операцию → latency + узкое место координатора.
- [ ] (А) повторные сообщения → идемпотентность (модуль 23), не консенсус.
- [ ] (Б) уникальные посетители → eventual/приближённые счётчики (HyperLogLog), не консенсус.
- [ ] (В) независимый аудит → независимая запись (партиционирование), не консенсус.
- [ ] (Г) лидер → консенсус обоснован (единственное согласованное решение).
- [ ] Правило: консенсус — только для критичных согласованных решений (лидер/конфиг/фиксация/lock), не для каждой операции.

## ПОДСКАЗКА
Консенсус дорог: согласие большинства на каждую операцию = latency + bottleneck. Идемпотентность (модуль 23) и eventual consistency (модуль 21) решают многое дешевле. Консенсус берегите для критичных решений (лидер, конфиг, lock), а не для рутины.
