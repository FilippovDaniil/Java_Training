# Задача 10 — Модуль 22 (мини-проект): Координационный слой системы

## ЗАДАНИЕ
Спроектируйте координационный слой системы: leader election, distributed locks, конфигурация. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) распределённый планировщик задач; (б) шардированная БД с лидерами шардов; (в) микросервисная платформа с динамической конфигурацией. Масштаб распределённый, узлы могут падать, partition возможен.

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Где нужен консенсус** — какие решения требуют согласия (лидер/конфиг/lock), а какие нет.
2. **Coordination service** — выбор (Zookeeper/etcd/Consul) с обоснованием; почему не самописное.
3. **Кворум** — размер кластера консенсуса (нечётный), сколько отказов переживает.
4. **Leader election** — как выбирается лидер, защита от split-brain (term/fencing).
5. **Distributed locks** — где нужны, TTL/lease + fencing token.
6. **Конфигурация** — как распространяется согласованная конфигурация.
7. **Где НЕ консенсус** — что вынесено на идемпотентность/eventual ради производительности.

## ФОРМАТ РЕЗУЛЬТАТА
Структурированный дизайн-документ из 7 разделов + схема координационного слоя.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; выделены решения, требующие консенсуса.
- [ ] Выбран coordination service с обоснованием; явно — не писать консенсус самому.
- [ ] Кластер консенсуса нечётный; указано, сколько отказов переживает.
- [ ] Leader election через кворум, защита от split-brain (term + fencing).
- [ ] Distributed locks с TTL/lease и fencing token.
- [ ] Описано распространение согласованной конфигурации.
- [ ] Явно названо, что вынесено из консенсуса (идемпотентность/eventual) ради производительности.

## ПОДСКАЗКА
Это синтез модуля: координационный слой = etcd/Zookeeper (нечётный кластер) для критичных решений (лидер через кворум + term/fencing, locks с TTL+fencing, конфиг), а рутина — мимо консенсуса (идемпотентность/eventual). Дальше — модуль 23: надёжность операций (идемпотентность, Saga, Outbox, resilience).
