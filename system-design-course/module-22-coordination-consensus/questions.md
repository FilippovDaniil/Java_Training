## Ключ (эталонные ответы)

1. Задача заставить группу узлов согласовать одно значение/решение, переживая отказы части узлов (и сетевые сбои).
2. Leader election (кто лидер при failover), фиксация записи (strong consistency), конфигурация кластера, distributed locks, membership/service discovery.
3. При сетевом разделении узлы независимо примут противоречащие решения (например, оба объявят себя лидером) → split-brain и расхождение данных. Нужно согласие, гарантирующее единственное решение.
4. В полностью асинхронной системе с возможностью отказа узла нельзя гарантировать достижение консенсуса за конечное время. Практические алгоритмы обходят это таймаутами: всегда корректны (safety), прогрессируют при стабильной сети (liveness).
5. Принцип «решение легитимно при согласии большинства узлов (> N/2)». Кворум = ⌊N/2⌋+1.
6. Чётное число допускает «ничью» (например, 2+2 при partition — ни у кого нет большинства) и не улучшает отказоустойчивость на узел; нечётное оптимально.
7. N − кворум = N − (⌊N/2⌋+1). N=3 → 1 отказ, N=5 → 2 отказа, N=4 → тоже 1 (как N=3, но дороже).
8. Две непересекающиеся половины не могут обе иметь большинство. При partition только сторона с кворумом принимает решения, другая блокируется → не возникает двух «правд».
9. Paxos — первый строгий алгоритм консенсуса, но печально сложный для понимания/реализации. Raft спроектирован ради понятности и стал де-факто стандартом (etcd, Consul).
10. Leader election (выбор единственного лидера), log replication (репликация лога команд через лидера), safety (зафиксированное не теряется; лидером может стать только узел с актуальным логом).
11. **Term** — монотонно растущий номер эпохи; новый лидер = новый term. Защищает от «старого» лидера: узел с меньшим term отвергается (анти-split-brain).
12. Когда команда реплицирована на большинство узлов (кворум приняло). После этого она считается зафиксированной и не теряется.
13. Если реплика «сама себя» назначает лидером, не видя текущего, то при partition обе стороны независимо выберут лидера → два лидера принимают запись → split-brain.
14. Лидером можно стать только при голосах большинства (кворум, через консенсус) + увеличение term + fencing старого лидера. Тогда при partition только сторона с кворумом выберет лидера.
15. Он может «протухнуть» (истечь по TTL), а держатель — упасть или зависнуть и не знать, что lock уже не его. Локальный мьютекс таких проблем не имеет (один процесс, общая память).
16. Чтобы упавший держатель не блокировал всех навечно: lock автоматически истекает (lease/TTL), если держатель не продлевает его.
17. «Зависший» держатель (например, GC-пауза): TTL истёк, lock перешёл другому, но первый «проснулся» и продолжает писать, не зная, что lock уже не его → порча данных.
18. Монотонно растущий токен, выдаваемый при каждом захвате lock. Ресурс принимает запись только с токеном ≥ последнего виденного → запись «зависшего» старого держателя (меньший токен) отвергается.
19. Потому что только ресурс (БД/хранилище) может фактически отсечь устаревшего держателя в момент записи; lock-сервис не контролирует, что держатель реально делает с ресурсом.
20. Zookeeper (leader election, locks, config — Kafka/HBase), etcd (Raft, KV+config — основа Kubernetes), Consul (service discovery, KV, locks, health-checks).
21. Это источник тончайших, трудноуловимых багов — даже эксперты ошибаются. Готовые сервисы дают корректные, проверенные примитивы поверх Raft/ZAB.
22. Каждая операция консенсуса требует согласия большинства (кворумные round-trips) → высокая latency и узкое место координатора. Это убивает throughput при массовых операциях.
23. Идемпотентностью (повторы безопасны без координации, модуль 23), eventual consistency (модуль 21), партиционированием независимых операций, логическими часами для порядка. Консенсус — только для критичных решений.
24. **Логические часы** (Lamport/vector) задают порядок событий между узлами без синхронизации физических часов и без полного консенсуса — полезны для причинного порядка (связь с clock skew, модуль 14).
25. Какие решения требуют консенсуса; выбор coordination service (не самописное); нечётный кластер и его отказоустойчивость; безопасный leader election (term/fencing); distributed locks с TTL+fencing; распространение конфигурации; что вынесено из консенсуса (идемпотентность/eventual).
