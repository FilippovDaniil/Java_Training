# Модуль 22 — Координация и консенсус (Raft, leader election, distributed locks)

## Зачем этот модуль

В модуле 12 мы столкнулись со split-brain: при наивном failover два узла объявляют себя лидером, и данные расходятся. В модуле 21 увидели, что strong consistency требует согласия узлов. Оба упираются в одну фундаментальную задачу: **как заставить группу узлов согласовать одно решение, когда часть из них может упасть или потерять связь?** Это задача **консенсуса** — сердце распределённых систем.

Зачем это в дизайне. Консенсус нужен везде, где требуется единственное согласованное решение: кто лидер (failover), зафиксирована ли запись (strong consistency), какая текущая конфигурация, кто держит distributed lock. Нельзя решать это «на глаз» — при сбоях это ведёт к split-brain и порче данных. На интервью «как выберете лидера без split-brain?», «как сделаете distributed lock?» — прямые сигналы к консенсусу.

Этот модуль — про кворум/majority, алгоритм Raft, leader election, distributed locks (и их ловушки: TTL, fencing) и готовые coordination services (Zookeeper/etcd).

---

## 1. Зачем нужен консенсус

**Консенсус** — задача: заставить группу узлов согласовать **одно** значение/решение, переживая отказы части узлов.

| Где нужен консенсус | Что согласовывают |
|---------------------|-------------------|
| Leader election | кто текущий лидер (failover, модуль 12) |
| Фиксация записи | зафиксирована ли транзакция/запись (strong consistency) |
| Конфигурация кластера | какие узлы в кластере, текущие настройки |
| Distributed lock | кто владеет блокировкой |
| Membership / service discovery | какие сервисы живы |

> Нельзя решать эти вопросы локально «на глаз»: при сетевом разделении два узла независимо решат «я лидер» → split-brain → расходящиеся данные. Консенсус гарантирует, что даже при сбоях все согласятся на **одно** решение. Это фундамент безопасного failover (модуль 12) и strong consistency (модуль 21).

**FLP impossibility** (кратко): в полностью асинхронной системе с возможностью отказа узла невозможно гарантировать консенсус за конечное время. Практические алгоритмы (Raft, Paxos) обходят это допущениями о таймаутах — они корректны (safety) всегда и прогрессируют (liveness) при достаточной стабильности сети.

## 2. Quorum / majority

Консенсус строится на **большинстве**: решение легитимно, если за него согласилось **> N/2** узлов.

```
N узлов, кворум = ⌊N/2⌋ + 1 (большинство)
  N=3 → кворум 2 (переживает отказ 1)
  N=5 → кворум 3 (переживает отказ 2)
```

Почему **нечётное** число узлов:
```
N=4: большинство = 3. Переживает отказ 1 (как N=3), но требует +1 узел. И при делении 2+2 — нет большинства ни у кого.
N=3: большинство = 2. Переживает отказ 1, дешевле.
→ нечётное число даёт лучшую отказоустойчивость на узел и исключает «ничью».
```

> Большинство — ключ к недопущению split-brain: **две непересекающиеся половины не могут обе иметь большинство**. Поэтому при partition только одна сторона (с кворумом) принимает решения, другая блокируется. Нечётное число узлов исключает «ничью» (50/50) и даёт лучшую отказоустойчивость на каждый узел.

## 3. Consensus-алгоритмы: Raft

| Алгоритм | |
|----------|--|
| **Paxos** | первый строгий алгоритм консенсуса; корректный, но печально сложный для понимания/реализации |
| **Raft** | спроектирован ради понятности; де-факто стандарт (etcd, Consul) |

**Raft в трёх частях:**
```
1. Leader election: узлы — follower'ы; по таймауту один становится candidate,
   запрашивает голоса; набрал большинство → leader (на текущий "term"/эпоху).
2. Log replication: клиентские команды лидер пишет в свой лог и реплицирует
   follower'ам; команда commit'ится, когда её приняло большинство.
3. Safety: только узел с актуальным логом может стать лидером; commit'нутое не теряется.
```

| Понятие Raft | Смысл |
|--------------|-------|
| Term (эпоха) | монотонно растущий номер; новый лидер = новый term (отсекает «старых» лидеров) |
| Heartbeat | лидер шлёт пульс; нет пульса → follower начинает выборы |
| Commit при большинстве | запись считается зафиксированной, когда её реплицировало большинство |

> Raft решает три вещи: выбирает единственного лидера, реплицирует через него лог команд и гарантирует, что зафиксированное не теряется. **Term** (эпоха) — ключ к защите от split-brain: старый «ожившный» лидер с устаревшим term будет отвергнут. Понимать Raft на уровне «лидер + большинство + term» достаточно для дизайна.

## 4. Leader election и защита от split-brain

Безопасный выбор лидера = частный случай консенсуса:

```
Наивно (плохо):   узел не видит лидера → объявляет лидером себя → при partition ДВА лидера (split-brain)
Через консенсус:  кандидат становится лидером ТОЛЬКО при голосах большинства
                  → при partition сторона без кворума не выберет лидера → один лидер
```

| Защита | Как |
|--------|-----|
| Кворум на выборы | лидер легитимен лишь при согласии большинства |
| Term/эпоха | новый лидер увеличивает term; старый лидер с меньшим term отвергается |
| Fencing старого лидера | изолировать/заглушить прежнего, чтобы не принимал запись (см. ниже) |

> Это прямое решение проблемы из модуля 12: split-brain невозможен, если лидером становятся только с согласия большинства (две половины не могут обе иметь кворум). Поэтому «безопасный failover» = «выбор лидера через консенсус», а не «реплика сама решает, что она теперь главная».

## 5. Distributed locks и их ловушки

**Distributed lock** — взаимное исключение в распределёнке (только один процесс делает что-то одновременно). Куда сложнее локального мьютекса.

| Ловушка | Проблема | Решение |
|---------|----------|---------|
| Держатель упал | взял lock и умер → вечная блокировка | **lease / TTL**: lock автоматически истекает |
| TTL истёк, а работа идёт | держатель «завис» (GC-пауза), lock перешёл другому, но первый ещё пишет | **fencing token** |
| Сетевая задержка | держатель думает, что владеет, а уже нет | fencing token проверяется на стороне ресурса |

**Fencing token:**
```
Lock-сервис выдаёт монотонно растущий токен при захвате (33, 34, 35...).
Ресурс (БД/хранилище) принимает запись только с токеном ≥ последнего виденного.
→ "зависший" старый держатель с токеном 33 будет отвергнут, когда новый (34) уже писал.
```

> **Distributed lock не равен локальному мьютексу** — он может «протухнуть» (TTL), а держатель — зависнуть и не знать, что lock уже не его. Без **lease/TTL** упавший держатель блокирует всех навечно; без **fencing token** зависший держатель повредит данные, записав после потери lock. Оба механизма обязательны для корректного distributed lock.

## 6. Coordination services

На практике консенсус не пишут сами — используют готовые системы:

| Сервис | Даёт |
|--------|------|
| **Zookeeper** | leader election, locks, config, membership (Kafka, HBase) |
| **etcd** | KV + consensus (Raft); конфигурация, locks, leader election (Kubernetes) |
| **Consul** | service discovery, KV, locks, health-checks |

Они предоставляют примитивы: **leader election, distributed locks, конфигурация, service discovery, membership** — поверх собственного консенсуса (Raft/ZAB).

> **Не реализуйте консенсус самостоятельно** — это источник тончайших багов (даже эксперты ошибаются). Берите Zookeeper/etcd/Consul: они дают корректные примитивы. На интервью правильный ответ на «как сделать leader election/lock» — «через etcd/Zookeeper», а не «напишу свой алгоритм».

## 7. Когда консенсус НЕ нужен

Консенсус **дорог** (кворумные round-trips, латентность) — применяйте его точечно:

```
Нужен консенсус:   единственное согласованное решение (лидер, конфиг, фиксация, lock)
НЕ нужен:          независимые операции, eventual-данные, идемпотентные ретраи
```

| Вместо консенсуса | Когда |
|-------------------|-------|
| Идемпотентность (модуль 23) | повторы безопасны без координации |
| Eventual consistency (модуль 21) | согласованность «в итоге» допустима |
| Партиционирование без общего состояния | операции независимы |

> Консенсус — мощный, но дорогой инструмент: каждая операция требует согласия большинства (latency). Используйте его только для **критичных согласованных решений** (лидер, конфигурация, фиксация), а не для каждой операции. Часто задачу можно решить дешевле — идемпотентностью (модуль 23) или eventual consistency (модуль 21). Логические часы (Lamport/vector) дают порядок событий без полного консенсуса.

## ⚠️ Подводные камни

- **Самодельный консенсус.** Источник тончайших багов; берите Zookeeper/etcd/Consul.
- **Чётное число узлов.** Возможна «ничья» 50/50 и хуже отказоустойчивость на узел; берите нечётное.
- **Leader election «на глаз».** Без кворума → split-brain; лидер — только с согласия большинства.
- **Distributed lock без TTL/lease.** Упавший держатель блокирует всех навечно.
- **Lock без fencing token.** Зависший старый держатель повредит данные после потери lock.
- **Считать distributed lock равным мьютексу.** Он протухает и держатель может не знать, что lock уже не его.
- **Консенсус на каждую операцию.** Дорого по latency; применяйте только для критичных решений.
- **Игнорировать term/эпоху.** Без неё старый «ожившный» лидер примет запись (split-brain).
