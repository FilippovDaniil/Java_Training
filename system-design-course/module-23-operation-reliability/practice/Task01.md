# Задача 01 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Спроектируйте идемпотентную операцию через idempotency key для платежа.

## ВХОДНЫЕ ДАННЫЕ
API «списать оплату за заказ». Проблема: клиент отправляет запрос, не получает ответ (сеть потеряла), повторяет → возможно двойное списание. Текущая реализация просто делает `INSERT payment + balance -= amount` на каждый запрос.

## ТРЕБОВАНИЯ
- Объясните, почему текущая операция не идемпотентна и откуда берётся двойное списание.
- Спроектируйте idempotency key: что присылает клиент, как сервер его использует.
- Опишите атомарность «проверка ключа + выполнение» (почему в одной транзакции).
- Объясните, что вернётся при повторе с тем же ключом.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз → дизайн idempotency key → атомарность → поведение при повторе.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: ответ теряется → клиент не знает, выполнилось ли → повтор → второе списание.
- [ ] Idempotency key: клиент генерирует уникальный ключ операции и шлёт его; сервер выполняет операцию один раз на ключ.
- [ ] Атомарность: проверка «ключ уже обработан?» + само списание в одной транзакции (иначе гонка/сбой между ними даёт дубль).
- [ ] При повторе с тем же ключом: возвращается прежний результат, без второго списания.
- [ ] Решение делает повтор безопасным (не «запрет повторов»).

## ПОДСКАЗКА
Корень — клиент не знает, выполнилась ли операция (ответ потерян), и повторяет. Лечение не «запретить повтор», а сделать его безвредным: idempotency key + выполнение один раз на ключ. Критична атомарность отметки и действия в одной транзакции.
