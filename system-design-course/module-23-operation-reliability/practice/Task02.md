# Задача 02 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Объясните, почему 2PC плох для распределённой транзакции через сервисы, и предложите альтернативу.

## ВХОДНЫЕ ДАННЫЕ
Процесс оформления заказа затрагивает три сервиса с собственными БД: платежи, склад, доставка. Команда предлагает обернуть всё в распределённую транзакцию через 2PC (two-phase commit).

## ТРЕБОВАНИЯ
- Перечислите проблемы 2PC в этом сценарии (минимум 3).
- Объясните, почему 2PC противоречит распределённому дизайну (доступность, масштаб).
- Предложите альтернативу (Saga) и объясните, что меняется в гарантиях (atomicity → eventual).
- Назовите узкие случаи, где 2PC всё же приемлем.

## ФОРМАТ РЕЗУЛЬТАТА
Проблемы 2PC (≥3) → почему противоречит дизайну → альтернатива Saga → когда 2PC уместен.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблемы 2PC: блокирующий (участники держат блокировки), координатор — SPOF (упал после prepare → зависание), не масштабируется, отказ любого участника блокирует всё.
- [ ] Объяснено: 2PC снижает доступность и плохо масштабируется → противоречит распределённому дизайну (модуль 21).
- [ ] Альтернатива: Saga (локальные транзакции + компенсации); гарантия меняется с атомарности на eventual consistency.
- [ ] Узкие случаи 2PC: внутри одной БД/тесно связанные ресурсы, малый масштаб.
- [ ] Вывод: для cross-service процессов — Saga, а не 2PC.

## ПОДСКАЗКА
2PC даёт атомарность ценой блокировок, SPOF-координатора и плохой доступности — это анти-паттерн для распределёнки. Замена — Saga: пошагово с компенсациями, eventual consistency. 2PC оставляют для узких случаев (одна БД), не для процессов через сервисы.
