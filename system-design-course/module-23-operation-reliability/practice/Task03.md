# Задача 03 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Спроектируйте Saga для многошагового процесса: шаги, компенсации, выбор choreography vs orchestration.

## ВХОДНЫЕ ДАННЫЕ
Процесс бронирования путешествия: (1) забронировать авиабилет; (2) забронировать отель; (3) арендовать авто; (4) списать оплату. Если любой шаг падает, уже сделанные брони/списания нужно откатить.

## ТРЕБОВАНИЯ
- Опишите последовательность локальных транзакций (шаги Saga).
- Для каждого шага определите компенсирующее действие.
- Покажите сценарий сбоя (например, упал шаг 3) и порядок компенсаций.
- Выберите choreography или orchestration для этого случая и обоснуйте.

## ФОРМАТ РЕЗУЛЬТАТА
Шаги Saga + компенсации + сценарий сбоя (порядок компенсаций) + выбор координации.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Шаги: бронь авиа → бронь отеля → аренда авто → оплата (локальные транзакции).
- [ ] Компенсации: отменить бронь авиа, отменить отель, отменить авто, вернуть оплату.
- [ ] Сценарий сбоя (упал шаг 3 «авто»): компенсации в обратном порядке для шагов 1–2 (отменить отель, отменить авиа).
- [ ] Выбор choreography (события) vs orchestration (координатор) обоснован: для сложного процесса с явным контролем — orchestration; для простого/слабосвязанного — choreography.
- [ ] Отмечено: Saga даёт eventual consistency и проходит через временно несогласованные состояния.

## ПОДСКАЗКА
Saga = цепочка локальных транзакций + компенсации при сбое (в обратном порядке). Компенсация не всегда тривиальна («вернуть деньги» ≠ «отменить списание»). Choreography (события) гибче, но запутаннее; orchestration (координатор) нагляднее для сложных процессов.
