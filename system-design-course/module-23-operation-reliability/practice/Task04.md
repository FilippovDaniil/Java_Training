# Задача 04 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Решите проблему dual-write с помощью Outbox pattern.

## ВХОДНЫЕ ДАННЫЕ
Сервис заказов должен: сохранить заказ в БД И опубликовать событие «order.created» в Kafka (для биллинга/склада/уведомлений). Сейчас код делает `db.save(order); kafka.publish(event)`. Проблема: если упало между этими строками — заказ есть в БД, но события нет (downstream не узнали) или наоборот.

## ТРЕБОВАНИЯ
- Объясните dual-write problem (почему два хранилища нельзя записать атомарно наивно).
- Спроектируйте Outbox: что пишется в одной транзакции, кто публикует событие.
- Объясните, как Outbox гарантирует «если заказ сохранён → событие будет опубликовано».
- Укажите, почему consumer события должен быть идемпотентным (at-least-once).

## ФОРМАТ РЕЗУЛЬТАТА
Dual-write problem → дизайн Outbox → гарантия → идемпотентность consumer'а.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: БД и Kafka — разные хранилища без общей транзакции; сбой между записями → рассинхрон.
- [ ] Outbox: в ОДНОЙ транзакции БД сохранить заказ + запись «событие» в таблицу outbox.
- [ ] Отдельный процесс (relay/CDC, модуль 20) читает outbox и публикует в Kafka, помечая опубликованным.
- [ ] Гарантия: раз заказ и outbox-запись в одной транзакции, то при сохранении заказа событие точно будет опубликовано (рано или поздно).
- [ ] Consumer идемпотентен: публикация at-least-once → возможны дубли событий.

## ПОДСКАЗКА
Dual-write нельзя сделать атомарным наивно (два хранилища). Outbox переносит атомарность в одну БД: данные + событие пишутся вместе, а публикация идёт отдельно из надёжно сохранённого outbox (как CDC, модуль 20). Связка «Outbox + идемпотентный consumer» — канон.
