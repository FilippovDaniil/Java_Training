# Задача 05 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Спроектируйте retry с backoff и jitter, предотвращающий retry storm.

## ВХОДНЫЕ ДАННЫЕ
Сервис вызывает downstream-API. При временном сбое downstream тысячи клиентов одновременно ретраят немедленно и синхронно → добивают только начавший восстанавливаться сервис (retry storm), он снова падает. Сейчас retry — это «повторить сразу, до 5 раз».

## ТРЕБОВАНИЯ
- Объясните, почему немедленный синхронный retry создаёт retry storm.
- Спроектируйте exponential backoff (растущая пауза между попытками).
- Объясните роль jitter (рандомизации) — почему без него backoff недостаточен.
- Свяжите с обработкой 429/Retry-After (модуль 16) и ограничением числа попыток.

## ФОРМАТ РЕЗУЛЬТАТА
Почему retry storm → exponential backoff → роль jitter → связь с 429/лимитом попыток.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: немедленные синхронные ретраи всех клиентов сразу добивают восстанавливающийся downstream.
- [ ] Exponential backoff: пауза растёт (1с, 2с, 4с, 8с…), снижая давление на downstream.
- [ ] Jitter: рандомизация пауз, чтобы клиенты не ретраили синхронно (даже с backoff без jitter всплески остаются выровненными).
- [ ] Ограничено число попыток (не бесконечно) + уважение Retry-After при 429 (модуль 16).
- [ ] Решение даёт downstream восстановиться, а не добивает его.

## ПОДСКАЗКА
Retry storm = все ретраят сразу и синхронно. Backoff растягивает паузы, jitter их рассинхронизирует (иначе всплески просто сдвигаются, оставаясь синхронными). Плюс лимит попыток и уважение Retry-After (модуль 16). Цель — дать downstream восстановиться.
