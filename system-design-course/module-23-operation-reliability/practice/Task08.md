# Задача 08 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Найдите ошибки в предложенном дизайне надёжности операций и переформулируйте решения.

## ВХОДНЫЕ ДАННЫЕ
Фрагмент дизайна (как есть):
1. «Распределённую транзакцию через 3 сервиса делаем 2PC — это надёжно».
2. «Обработчик платежа делает `balance -= amount` на каждое сообщение; доставка at-least-once».
3. «Заказ сохраняем в БД, затем публикуем в Kafka двумя отдельными вызовами».
4. «При сбое downstream повторяем немедленно и без ограничений, пока не получится».
5. «Вызовы к внешним сервисам делаем без таймаута и без circuit breaker — упростим код».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и последствие.
- Предложите исправление с обоснованием.
- Отметьте, какая ошибка ведёт к каскадному отказу, а какая — к порче данных/дублям.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: пункт → ошибка → последствие → исправление + классификация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) 2PC через сервисы → блокировки/SPOF/масштаб → Saga + компенсации.
- [ ] (2) `balance -= amount` при at-least-once → двойное списание → идемпотентность (idempotency key).
- [ ] (3) Раздельные db.save + kafka.publish → dual-write рассинхрон → Outbox pattern.
- [ ] (4) Немедленный безлимитный retry → retry storm → backoff+jitter + лимит попыток.
- [ ] (5) Нет timeout/circuit breaker → каскадный отказ → timeout + circuit breaker.
- [ ] Классифицировано: (5)/(4) → каскад; (2)/(3) → порча данных/дубли/рассинхрон.

## ПОДСКАЗКА
Чек-лист модуля: Saga вместо 2PC; идемпотентность при at-least-once; Outbox вместо dual-write; backoff+jitter+лимит вместо немедленного retry; timeout+circuit breaker против каскада. Пункты 4–5 ведут к каскаду, 2–3 — к порче/рассинхрону данных.
