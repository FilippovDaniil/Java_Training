# Задача 09 — Модуль 23: Надёжность операций

## ЗАДАНИЕ
Соберите полный resilience-стек для критичной операции и обоснуйте соразмерность.

## ВХОДНЫЕ ДАННЫЕ
Критичная операция: «провести платёж» — вызывает внешний платёжный шлюз, обновляет баланс, публикует событие. Требования: не списать дважды, пережить временные сбои шлюза, не повиснуть, не уронить себя при отказе шлюза, надёжно опубликовать результат.

## ТРЕБОВАНИЯ
- Соберите стек надёжности из изученных паттернов и объясните роль каждого в этой операции.
- Покажите, как паттерны дополняют друг друга (идемпотентность ↔ retry и т.д.).
- Объясните принцип соразмерности: что из стека НЕ нужно тривиальной операции (например, лайк).
- Кратко свяжите с CAP/eventual (модуль 21).

## ФОРМАТ РЕЗУЛЬТАТА
Стек паттернов с ролью каждого + как дополняют друг друга + соразмерность + связь с eventual.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Стек: idempotency key (повтор безопасен) + timeout (не повиснуть) + retry с backoff+jitter (пережить сбой шлюза) + circuit breaker (не добивать упавший шлюз) + Outbox (надёжно опубликовать событие).
- [ ] Показана взаимодополняемость: идемпотентность делает retry безопасным; circuit breaker + timeout предотвращают каскад; Outbox решает dual-write.
- [ ] Соразмерность: тривиальной операции (лайк) весь стек не нужен — достаточно минимума (идемпотентность/eventual).
- [ ] Связь с CAP: результат может быть eventual (событие публикуется асинхронно), это приемлемо.
- [ ] Стек обоснован критичностью операции.

## ПОДСКАЗКА
Паттерны комбинируются: idempotency (повтор безопасен) → позволяет retry+backoff+jitter → circuit breaker+timeout против каскада → Outbox для надёжного события. Для денег нужен весь стек; для лайка — минимум. Соразмерность критичности — ключевой принцип (модуль 01).
