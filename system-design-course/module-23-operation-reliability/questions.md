## Ключ (эталонные ответы)

1. Потому что в распределёнке отказ — норма: вызовы падают, повторяются, выполняются частично, downstream отказывает. Без паттронов один сбой роняет всё (каскад), ретраи плодят дубли, процессы застревают на полпути.
2. Свойство, при котором повторное выполнение операции даёт тот же результат, что однократное. Повтор безвреден.
3. Потому что после сетевого сбоя вы часто **не знаете**, выполнилась ли операция (ответ потерян), и клиент повторяет. Единственный безопасный путь — сделать повтор безвредным.
4. Уникальный ключ операции, который присылает клиент. Сервер выполняет операцию один раз на ключ; повтор с тем же ключом возвращает прежний результат без второго эффекта.
5. Идемпотентная: `SET status=PAID`, upsert (повтор безвреден). Неидемпотентная: `balance -= 100`, голый `INSERT` (повтор меняет результат/дублирует).
6. Если отметить «выполнено» и сделать действие не атомарно (в разных транзакциях), сбой между ними даёт либо дубль (действие без отметки), либо потерю; атомарность в одной транзакции это исключает.
7. **Two-Phase Commit** — протокол распределённой транзакции (prepare → commit через координатора). Проблемы: блокирующий (держит блокировки), координатор — SPOF, не масштабируется, отказ участника блокирует всё.
8. Он снижает доступность (блокировки, зависание при отказе координатора) и плохо масштабируется (синхронная координация всех участников) — это противоречит целям распределённой системы (модуль 21).
9. Распределённая «транзакция» как последовательность локальных транзакций; при сбое шага выполняются компенсирующие действия, откатывающие уже сделанное. Даёт eventual consistency вместо атомарности.
10. Действие, отменяющее эффект ранее выполненного шага (отменить бронь, вернуть оплату). Не всегда тривиально: «вернуть деньги» — это новая операция, а не откат.
11. **Choreography** — координация через события (каждый сервис слушает и публикует следующее, нет центра; гибко, но сложно отследить). **Orchestration** — центральный оркестратор управляет шагами и компенсациями (нагляднее).
12. Eventual consistency: система проходит через временно несогласованные промежуточные состояния, а согласованность достигается по завершении (или после компенсаций) — не атомарное «всё или ничего мгновенно».
13. Необходимость атомарно сделать два действия в разных хранилищах (записать в БД И опубликовать в Kafka) без общей транзакции. Сбой между ними → рассинхрон (в БД есть, в Kafka нет).
14. В одной транзакции БД записывают данные + «событие» в таблицу outbox; отдельный процесс (relay/CDC) читает outbox и публикует в Kafka. Атомарность переносится в одну БД.
15. Потому что публикация из outbox идёт at-least-once (возможны повторные публикации при сбоях relay) → consumer должен безопасно обрабатывать дубли.
16. Ситуация, когда после сбоя downstream все клиенты ретраят немедленно и синхронно, добивая восстанавливающийся сервис. Избегают: backoff (растущие паузы) + jitter (рандомизация) + лимит попыток.
17. Без jitter даже при backoff клиенты, начавшие одновременно, ретраят синхронно (всплески выровнены). Jitter рандомизирует паузы, рассинхронизируя ретраи и размазывая нагрузку.
18. Чтобы зависший/медленный downstream не держал вызовы вечно: без таймаута висящие вызовы копятся, исчерпывают пулы потоков/соединений и ведут к каскадному отказу.
19. Паттерн, «размыкающий цепь» при потоке сбоев. Состояния: closed (зовём downstream) → много ошибок → open (быстрый отказ, не зовём) → через паузу half-open (пробные запросы) → успех → closed.
20. При open он перестаёт звать упавший сервис: не ждёт таймауты и не копит висящие вызовы, отдавая быстрый отказ/fallback. Это сохраняет ресурсы вызывающего и не тянет всю цепочку вниз.
21. Изоляция ресурсов (отдельные пулы потоков/соединений) по зависимостям, чтобы сбой/замедление одной зависимости не исчерпал ресурсы, нужные другим (как переборки в корабле).
22. Деградированный ответ при сбое зависимости: вернуть кэш, заглушку, частичный результат или пропустить некритичную часть — частичная функциональность вместо полного отказа.
23. Они дополняют друг друга: idempotency делает retry безопасным; retry+backoff+jitter переживают временные сбои; timeout+circuit breaker предотвращают каскад; Saga управляет многошаговым процессом; Outbox надёжно публикует события.
24. Что объём стека соответствует критичности операции: для денег нужен весь арсенал (идемпотентность+Saga+Outbox+resilience), для тривиального (лайк) — минимум. Не навешивать всё на каждую операцию.
25. Шаги и границы локальных транзакций; Saga + компенсации вместо 2PC (choreography/orchestration); идемпотентность шагов; Outbox для событий; resilience-паттерны на downstream (timeout/retry+jitter/circuit breaker/fallback); описание eventual-состояний; соразмерность стека критичности.
