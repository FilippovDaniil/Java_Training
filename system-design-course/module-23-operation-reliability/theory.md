# Модуль 23 — Надёжность операций (идемпотентность, Saga/Outbox, resilience)

## Зачем этот модуль

Во всех предыдущих модулях всплывала одна и та же фраза: «нужна идемпотентность» (модули 12, 15, 20), «избегайте 2PC» (модуль 13), «backoff при 429» (модуль 16). Этот модуль собирает разрозненные упоминания в единый арсенал **паттернов надёжности операций** — то, чем распределённая система защищается от суровой реальности: вызовы падают, повторяются, выполняются частично, downstream-сервисы отказывают.

Зачем это senior'у. В распределённой системе **отказ — норма, а не исключение**. Сеть теряет ответы (вы не знаете, выполнилась операция или нет), сервисы падают посреди многошагового процесса, медленный downstream тянет за собой всех. Без паттернов надёжности система превращается в карточный домик: один сбой роняет всё (каскад), ретраи плодят дубли (двойные списания), процесс застревает на полпути (деньги списали, товар не отправили).

Это финальный модуль Части 5. Он отвечает на «как сделать операцию надёжной, когда вызовы ненадёжны»: идемпотентность, Saga вместо 2PC, Outbox против dual-write, и resilience-паттерны (retry, timeout, circuit breaker, bulkhead).

---

## 1. Идемпотентность

**Идемпотентность** — повторное выполнение операции даёт **тот же результат**, что однократное. Обязательна везде, где возможны повторы (retry, at-least-once доставка — модуль 15).

```
Проблема: клиент шлёт "списать $100", не получает ответ (сеть потеряла), повторяет.
  Неидемпотентно: списали дважды ($200).
  Идемпотентно:   повтор распознан → списали один раз ($100).
```

| Приём | Как |
|-------|-----|
| **Idempotency key** | клиент шлёт уникальный ключ операции; сервер выполняет её один раз на ключ, повтор возвращает прежний результат |
| Дедупликация по ID | хранить обработанные message-id, повтор игнорировать |
| Естественная идемпотентность | `SET status=PAID` (повтор безвреден), в отличие от `balance -= 100` |
| Upsert вместо insert | повторная вставка не дублирует |

> Идемпотентность — **фундамент надёжности в распределёнке**, потому что вы часто **не знаете**, выполнилась ли операция (ответ потерян). Единственный безопасный путь — сделать повтор безвредным. Idempotency key — стандарт для платежей и любых критичных операций: повтор с тем же ключом не создаёт второй эффект. Тонкость — атомарность «отметка о выполнении + само действие» (одна транзакция), иначе сбой между ними вернёт дубли.

---

## 2. Почему 2PC плох

**Two-Phase Commit (2PC)** — классический протокол распределённой транзакции: координатор спрашивает всех участников «готовы?» (prepare), затем «commit». Проблемы:

| Проблема 2PC | |
|--------------|--|
| **Блокирующий** | участники держат блокировки до решения координатора |
| **Координатор — SPOF** | упал координатор после prepare → участники «зависли» в неопределённости |
| **Не масштабируется** | синхронная координация всех участников, медленно |
| **Снижает доступность** | отказ любого участника блокирует всю транзакцию |

> 2PC даёт атомарность через несколько сервисов/БД, но ценой блокировок, SPOF-координатора и плохой доступности — это противоречит распределённому дизайну (модуль 21). Поэтому в дизайне **избегают cross-service/cross-shard 2PC** (модуль 13), предпочитая Saga + eventual consistency. 2PC уместен лишь в узких случаях (внутри одной БД/тесно связанные ресурсы).

## 3. Saga

**Saga** — распределённая «транзакция» как последовательность **локальных** транзакций; при сбое шага выполняются **компенсирующие** действия (откатывающие предыдущие).

```
Заказ: [списать оплату] → [зарезервировать товар] → [создать доставку]
Сбой на доставке → компенсации в обратном порядке:
  отменить резерв → вернуть оплату
```

| Тип Saga | Как координируется |
|----------|--------------------|
| **Choreography** | через события: каждый сервис слушает событие и публикует следующее (нет центра; гибко, но сложно отследить) |
| **Orchestration** | центральный оркестратор управляет шагами и компенсациями (явно, проще отслеживать; оркестратор — точка управления) |

> Saga заменяет атомарность 2PC на **eventual consistency с компенсациями**: вместо «всё или ничего мгновенно» — «выполняем по шагам, при сбое откатываем сделанное». Цена — система проходит через временно несогласованные состояния и нужны **компенсирующие** действия (не всегда тривиальные — «вернуть деньги» ≠ «отменить списание»). Choreography гибче, но запутаннее; orchestration нагляднее. Saga — стандартный ответ на «распределённая транзакция через сервисы».

## 4. Outbox pattern (проблема dual-write)

**Dual-write problem:** надо атомарно сделать два действия — записать в БД и опубликовать событие в Kafka. Но это два разных хранилища, общей транзакции нет:

```
❌ Наивно:  write(DB); publish(Kafka)
   - упало между ними → в БД есть, в Kafka нет (или наоборот) → рассинхрон
```

**Outbox** решает это:
```
1. В ОДНОЙ транзакции БД: записать данные + записать "событие" в таблицу outbox.
2. Отдельный процесс (relay / CDC, модуль 20) читает outbox и публикует в Kafka.
3. Событие помечается опубликованным (at-least-once → consumer идемпотентен).
```

> Outbox решает dual-write тем, что «бизнес-запись» и «событие к отправке» пишутся **атомарно в одну БД** (одна транзакция), а публикация в Kafka происходит отдельно из надёжно сохранённого outbox. Это гарантирует: если запись в БД случилась — событие точно будет опубликовано (рано или поздно). Тот же механизм, что CDC (модуль 20): читать изменения БД и публиковать. Связка «Outbox + идемпотентный consumer» — каноничный способ надёжно соединить БД и события.

## 5. Resilience-паттерны

Защита от отказов downstream-сервисов:

| Паттерн | Что делает | Зачем |
|---------|------------|-------|
| **Timeout** | ограничивает ожидание ответа | зависший downstream не копит висящие вызовы |
| **Retry + backoff + jitter** | повтор с растущей рандомизированной паузой | пережить временный сбой, не устроив retry storm |
| **Circuit breaker** | при потоке сбоев «размыкает цепь» — перестаёт звать падающий сервис | не добивать упавший downstream, не ждать таймауты, быстрый отказ |
| **Bulkhead** | изолирует ресурсы (пулы) по зависимостям | сбой одной зависимости не съедает все потоки/соединения |
| **Fallback / graceful degradation** | при сбое — деградированный ответ (кэш, заглушка) | частичная функциональность вместо полного отказа |

```
Retry storm:  downstream мигнул → тысячи клиентов синхронно ретраят → добивают его
  Лечение: backoff (растущая пауза) + jitter (рандомизация, чтобы не синхронно)

Circuit breaker:  closed (работаем) → много ошибок → open (быстрый отказ, не зовём) → half-open (пробуем) → closed
```

> Три обязательных: **timeout** (никогда не ждать вечно), **retry с backoff+jitter** (повтор без retry storm — синхронные ретраи добивают сервис, jitter их рассинхронизирует), **circuit breaker** (перестать звать явно упавший downstream, давая ему восстановиться и отвечая клиенту быстрым отказом вместо накопления таймаутов). Вместе они предотвращают **каскадный отказ** — когда падение одного сервиса по цепочке роняет всех.

## 6. Как это собирается вместе

```
Надёжная критичная операция (например, платёж):
  idempotency key (повтор безопасен)
  + timeout (не ждать вечно)
  + retry с backoff+jitter (пережить временный сбой)
  + circuit breaker (не добивать упавший downstream)
  + Saga (если многошаговый процесс через сервисы) с компенсациями
  + Outbox (надёжно опубликовать событие о результате)
```

> Паттерны не взаимоисключающие — их комбинируют. Идемпотентность делает retry безопасным; retry+circuit breaker переживают сбои downstream; Saga управляет многошаговым процессом; Outbox надёжно соединяет БД и события. Это «стек надёжности» распределённой операции. Не каждой операции нужен весь стек — масштабируйте под критичность (деньги → весь, лента → минимум).

## ⚠️ Подводные камни

- **2PC как дефолт для распределённых транзакций.** Блокирующий, SPOF-координатор, не масштабируется; берите Saga.
- **Неидемпотентные операции при retry/at-least-once.** Повтор → дубли (двойные списания/письма).
- **Dual-write без Outbox.** «Записал в БД, но не в Kafka» → рассинхрон; используйте Outbox/CDC.
- **Retry без backoff+jitter.** Синхронные ретраи устраивают retry storm и добивают downstream.
- **Нет timeout.** Зависшие вызовы копятся, исчерпывают пулы → каскадный отказ.
- **Нет circuit breaker.** Продолжаем звать упавший сервис → ждём таймауты, тянем всех вниз (каскад).
- **Saga без компенсаций.** Сбой на середине оставляет несогласованное состояние; компенсации обязательны.
- **Путать Saga с атомарной транзакцией.** Saga eventual: есть временно несогласованные состояния, не «всё или ничего мгновенно».
- **Весь resilience-стек на каждую операцию.** Соразмерность: критичное — весь стек, тривиальное — минимум.
