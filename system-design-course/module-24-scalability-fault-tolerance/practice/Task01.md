# Задача 01 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Выберите способ масштабирования (vertical vs horizontal) для каждого из компонентов и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Растущий сервис: (а) stateless REST API (трафик ×10 за полгода); (б) реляционная primary-БД (запись растёт, чтения умеренно); (в) аналитические тяжёлые батч-расчёты раз в сутки. Бюджет ограничен, простой нежелателен.

## ТРЕБОВАНИЯ
- Для каждого компонента выберите vertical / horizontal (или комбинацию) и обоснуйте.
- Назовите пределы выбранного подхода и момент, когда он перестанет работать.
- Укажите, что должно быть верно, чтобы горизонтальный масштаб был возможен (для API).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «компонент → подход → обоснование → предел» + краткий вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] API → horizontal (stateless, за LB); назван предел (нужен LB, внешняя сессия).
- [ ] primary-БД (запись) → сначала vertical/read-реплики, затем шардирование как горизонтальный предел.
- [ ] Батч → vertical или горизонтальный батч-кластер под профиль (разовая тяжёлая задача).
- [ ] Для каждого подхода назван его потолок (железо для vertical, координация/состояние для horizontal).
- [ ] Указано, что для horizontal API сервис должен быть stateless.

## ПОДСКАЗКА
Дефолт для веб/сервисного слоя — horizontal (нет потолка, встроена избыточность), но требует stateless. Vertical прост и уместен там, где горизонталь дорога/сложна (БД-primary до шардирования). Каждый подход имеет потолок — назовите его.
