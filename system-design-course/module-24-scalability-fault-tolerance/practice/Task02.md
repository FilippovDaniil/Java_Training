# Задача 02 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Сделайте stateful-сервис горизонтально масштабируемым, вынеся состояние.

## ВХОДНЫЕ ДАННЫЕ
Веб-приложение хранит пользовательскую сессию в памяти инстанса; LB настроен на sticky sessions (запросы клиента липнут к одному инстансу). Локально на инстансе также лежит загруженный пользователем временный файл и in-memory кэш справочников. При отказе инстанса пользователь «вылетает».

## ТРЕБОВАНИЯ
- Объясните, почему текущая схема не масштабируется честно и почему отказ инстанса теряет пользователя.
- Для каждого вида состояния (сессия, файл, кэш) предложите, куда его вынести.
- Опишите, как после выноса инстансы становятся взаимозаменяемыми (stateless).
- Объясните, почему можно убрать sticky sessions.

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз → таблица «состояние → куда вынести» → как достигается stateless → судьба sticky sessions.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: состояние в памяти → запрос привязан к инстансу (sticky), отказ инстанса = потеря состояния.
- [ ] Сессия → внешнее хранилище сессий (Redis/БД); файл → blob/объектное хранилище; кэш справочников → distributed-кэш (или допускается локальный read-only с TTL).
- [ ] После выноса любой инстанс обрабатывает любой запрос → взаимозаменяемость.
- [ ] Sticky sessions больше не нужны (состояние внешнее) → честный horizontal-масштаб, безболезненный отказ инстанса.
- [ ] Отмечено, что состояние не исчезает, а переезжает в специально масштабируемый/реплицируемый слой.

## ПОДСКАЗКА
Ключ к scale-out — stateless-сервис. Состояние неизбежно «где-то живёт»; задача — чтобы оно жило во внешнем масштабируемом слое (сессия→Redis, файл→blob, данные→БД), а не в памяти случайного сервера. Тогда sticky не нужен, отказ инстанса безболезнен.
