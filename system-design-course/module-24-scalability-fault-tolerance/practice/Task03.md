# Задача 03 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Найдите все SPOF в архитектуре и устраните их избыточностью.

## ВХОДНЫЕ ДАННЫЕ
Архитектура: клиент → один nginx (LB) → 3 инстанса сервиса (stateless) → одна primary-БД (PostgreSQL) → один Redis (сессии/кэш). Конфигурация раздаётся одним config-сервисом. Всё в одной зоне доступности (AZ).

## ТРЕБОВАНИЯ
- Пройдите по схеме и перечислите все SPOF (узлы в единственном экземпляре на пути запроса).
- Для каждого SPOF предложите устранение (схема избыточности).
- Отметьте «скрытые» SPOF, которые легко пропустить.
- Укажите, почему 3 инстанса сервиса не спасают, если есть SPOF ниже по пути.

## ФОРМАТ РЕЗУЛЬТАТА
Список SPOF → таблица «SPOF → устранение» → вывод про самое слабое звено.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Перечислены SPOF: nginx/LB, primary-БД, Redis, config-сервис, единственная AZ.
- [ ] Устранение: LB → HA-пара (floating IP); БД → реплика + failover; Redis → реплика/кластер; config → ≥3 узла/кворум; AZ → multi-AZ.
- [ ] Отмечено, что 3 stateless-инстанса не устраняют SPOF ниже (БД/LB/Redis) — отказоустойчивость по слабому звену.
- [ ] Зона доступности названа как SPOF уровня инфраструктуры (отсылка к модулю 25).
- [ ] Решения сослались на соответствующие модули (8/12/22).

## ПОДСКАЗКА
Метод: пройти по каждому узлу на пути запроса и спросить «что упадёт, если он умрёт?». Любой компонент в единственном экземпляре — кандидат в SPOF. Скрытые SPOF — общий LB/БД/конфиг под резервированным сервисным слоем. Отказоустойчивость = по самому слабому звену.
