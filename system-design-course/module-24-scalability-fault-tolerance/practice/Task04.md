# Задача 04 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Рассчитайте доступность цепочки последовательно соединённых компонентов и сравните с целевым SLA.

## ВХОДНЫЕ ДАННЫЕ
Запрос проходит цепочку: CDN (99.99%) → LB (99.95%) → сервис (99.9%) → primary-БД (99.9%) → внешний платёжный API (99.5%). Все компоненты на критическом пути (отказ любого = отказ запроса). Целевой SLA системы — 99.9%.

## ТРЕБОВАНИЯ
- Посчитайте итоговую доступность цепочки (произведение).
- Сравните с целевым SLA 99.9% — выполняется ли?
- Определите главное слабое звено и его вклад.
- Предложите, как поднять итог до цели (минимум одно решение: убрать звено из критического пути / зарезервировать / сделать опциональным).

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт (произведение по шагам) → вывод по SLA → слабое звено → решение.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Итог посчитан как произведение: ≈0.9999×0.9995×0.999×0.999×0.995 ≈ 0.9924 → ~99.24%.
- [ ] Вывод: цель 99.9% НЕ достигается (цепочка ниже самого слабого звена).
- [ ] Слабое звено — платёжный API (99.5%) как главный вклад в недоступность.
- [ ] Предложено решение: вынести платёж из критического пути (async/очередь), сделать опциональным (graceful degradation) или зарезервировать провайдером-fallback.
- [ ] Явно отмечено: последовательная цепочка всегда ниже любого своего звена.

## ПОДСКАЗКА
Последовательное соединение перемножает доступности и даёт итог НИЖЕ слабейшего звена. Чтобы поднять: либо укоротить критический путь (убрать/сделать async звено), либо зарезервировать слабое звено. Здесь явный кандидат — внешний платёжный API 99.5%.
