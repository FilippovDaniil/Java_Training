# Задача 07 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Сравните active-passive и active-active failover для двух компонентов и выберите схему для каждого.

## ВХОДНЫЕ ДАННЫЕ
Два компонента: (а) реляционная primary-БД (источник правды, строгая согласованность записи); (б) stateless слой API за балансировщиком. Требование: минимальный простой при отказе, разумная стоимость, отсутствие конфликтов данных.

## ТРЕБОВАНИЯ
- Для каждого компонента выберите active-passive или active-active и обоснуйте.
- Назовите trade-off выбранной схемы (простой резерва / лаг переключения vs утилизация / сложность конфликтов).
- Для БД объясните, почему active-active для записи — это multi-leader со своими проблемами.
- Опишите, что происходит при отказе в каждой выбранной схеме.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «компонент → схема → обоснование → поведение при отказе» + trade-off.

## КРИТЕРИИ ПРИЁМКИ
- [ ] БД (запись, согласованность) → active-passive (один primary, промоут реплики при отказе); назван лаг переключения.
- [ ] API (stateless) → active-active (все инстансы обслуживают, отказ части без паузы); назван плюс утилизации.
- [ ] Для БД объяснено: active-active запись = multi-leader → конфликты/сложность (модуль 12), поэтому не выбрана.
- [ ] Описано поведение при отказе: passive — детект+промоут (пауза), active — трафик на оставшихся (без паузы).
- [ ] Учтена стоимость (passive-резерв простаивает) и отсутствие конфликтов как требование.

## ПОДСКАЗКА
Active-passive проще и без конфликтов (один «главный»), но резерв простаивает и есть лаг промоута — подходит для записи в БД. Active-active утилизирует всё и переживает отказ без паузы, но для данных это multi-leader с конфликтами — подходит для stateless-сервисов.
