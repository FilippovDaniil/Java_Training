# Задача 08 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Найдите ошибки в дизайне, который заявлен как «масштабируемый и отказоустойчивый», и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из дизайн-документа:
1. «У нас 6 инстансов сервиса за LB — система отказоустойчива».
2. «Сессии храним в памяти инстанса, LB со sticky sessions — это даёт производительность».
3. «Все 6 инстансов и обе реплики БД в одной зоне — так ниже сетевая latency».
4. «Доступность каждого из 4 последовательных компонентов 99.9%, значит у системы 99.9%».
5. «Поставили standby-БД, failover настроим потом, в проде проверим».
6. «Под нагрузку нужно ровно 4 сервера — столько и держим (экономия)».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и корректное решение.
- Свяжите ошибки с понятиями модуля (SPOF, stateless, независимость отказов, последовательная доступность, тестирование failover, N+1).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → ошибка → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Резервирован только сервис; LB/БД/прочее на пути — скрытые SPOF. Отказоустойчивость по слабому звену.
- [ ] (2) Stateful + sticky → отказ инстанса теряет сессию, нет честного масштаба → вынести сессию во внешнее хранилище.
- [ ] (3) Одна зона = общая точка отказа; «независимость отказов» нарушена → multi-AZ.
- [ ] (4) Последовательная цепочка перемножается: 0.999⁴≈0.996 → ~99.6%, а не 99.9%.
- [ ] (5) Непротестированный failover часто не срабатывает в реальный отказ → тестировать (game day, модуль 25).
- [ ] (6) N без запаса: отказ любого → перегруз → минимум N+1.

## ПОДСКАЗКА
Типовые ловушки: резервирование только одного слоя (скрытые SPOF ниже), stateful+sticky как «оптимизация», резерв в той же точке отказа, забытое перемножение последовательной доступности, нетестированный failover, N вместо N+1.
