# Задача 09 — Модуль 24: Масштабируемость и отказоустойчивость

## ЗАДАНИЕ
Спроектируйте ограничение blast radius (радиуса поражения отказа) для multi-tenant системы через изоляцию.

## ВХОДНЫЕ ДАННЫЕ
SaaS обслуживает тысячи клиентов (tenants) на общем пуле сервисов и одной большой БД. Проблема: один «тяжёлый» клиент (всплеск трафика, тяжёлый запрос, отравленное сообщение) деградирует сервис для всех; редкий баг в общей БД задевает всех клиентов сразу.

## ТРЕБОВАНИЯ
- Объясните текущий blast radius (почему отказ затрагивает всех).
- Предложите изоляцию для ограничения радиуса: cell-based architecture (нарезка на ячейки) и/или bulkhead (изоляция ресурсов).
- Опишите, как распределить клиентов по ячейкам и что значит отказ одной ячейки.
- Укажите trade-off изоляции (стоимость, дублирование, маршрутизация tenant→cell).

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз blast radius → схема изоляции (cells/bulkhead) → распределение tenants → trade-off.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: общий пул + общая БД → отказ/всплеск одного клиента бьёт по всем (blast radius = вся система).
- [ ] Cell-based: система нарезана на независимые ячейки (свой пул сервисов + своя БД на cell), tenant закреплён за ячейкой.
- [ ] Отказ ячейки затрагивает только её клиентов (например, 1/N клиентов), не всех.
- [ ] Bulkhead/изоляция ресурсов как дополнительный/альтернативный уровень (отдельные пулы, лимиты на tenant).
- [ ] Trade-off: дублирование инфраструктуры, маршрутизация tenant→cell, сложность операций — против ограниченного blast radius.

## ПОДСКАЗКА
Зрелая отказоустойчивость ограничивает радиус поражения: один сбойный клиент/компонент не валит всех. Cell-based нарезает систему на независимые ячейки (отказ одной = лишь её клиенты); bulkhead изолирует ресурсы по нагрузке. Цена — дублирование и маршрутизация tenant→cell.
