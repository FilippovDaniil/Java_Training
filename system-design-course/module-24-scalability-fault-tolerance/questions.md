## Ключ (эталонные ответы)

1. Vertical (scale up) — мощнее одна машина (CPU/RAM/диск); horizontal (scale out) — больше машин. Vertical прост, но с потолком и SPOF; horizontal почти без предела и со встроенной избыточностью, но требует LB, stateless и координации.
2. Потолок мощности железа (нельзя бесконечно наращивать), резко растущая цена на верхних конфигурациях и сохранение SPOF (одна машина).
3. Потому что у него нет жёсткого потолка (растём добавлением узлов) и встроена избыточность (несколько взаимозаменяемых инстансов), что даёт и масштаб, и отказоустойчивость.
4. Сервис, не хранящий состояние между запросами (запрос самодостаточен). Любой инстанс обработает любой запрос → добавление инстанса = масштаб, отказ инстанса безболезнен.
5. Состояние в памяти инстанса (сессия, локальный кэш/файлы) и sticky sessions: запрос «привязан» к инстансу, отказ теряет состояние, инстансы не взаимозаменяемы.
6. Сессии — во внешнее хранилище (Redis/БД), файлы — в blob/объектное хранилище, данные — в БД (реплики/шарды), кэш — в distributed-кэш. Состояние переезжает в специально масштабируемый/реплицируемый слой.
7. Архитектура, где узлы не делят память/диск и общаются только по сети, каждый автономен. Нет общего ресурса → нет общего узкого места и общего SPOF; фундамент горизонтального масштаба.
8. Способность продолжать работать (возможно деградированно) при отказе компонентов. Достигается избыточностью — дублированием так, чтобы отказ одного не ронял систему.
9. Запас сверх рабочей мощности N: N+1 переживает 1 одновременный отказ, N+2 — два, 2N — полное дублирование (отказ половины/целого набора).
10. N+1 как минимум: переживает один отказ и плановое обслуживание узла. N без запаса означает перегруз при любом отказе; 2N дороже и нужен критичным системам.
11. Компонент, отказ которого роняет всю систему. Искать: пройти по схеме и для каждого узла на пути запроса спросить «что упадёт, если он умрёт?»; любой компонент в единственном экземпляре — кандидат.
12. Единственный LB, единственная primary-БД без реплики, один кэш-узел, единственный config/DNS-сервис, единственная зона доступности — часто скрыты под резервированным сервисным слоем.
13. Потому что отказоустойчивость определяется самым слабым звеном: 10 инстансов сервиса бесполезны, если они ходят в единственную primary-БД или через единственный LB (SPOF ниже по пути).
14. Автоматическое переключение на резерв при отказе основного компонента (промоут реплики, перенаправление трафика).
15. Active-passive: резерв простаивает и активируется при отказе (проще, нет конфликтов, но простой резерва и лаг переключения). Active-active: оба обслуживают трафик (полная утилизация, отказ части без паузы, но сложнее — синхронизация/конфликты).
16. Потому что два активных «главных» по записи — это multi-leader: нужно решать конфликты записи и согласованность (модуль 12), что сложно и рискованно для источника правды.
17. Доля времени работоспособности. 99.9% ≈ 8.76 ч/год, 99.99% ≈ 52.6 мин/год, 99.999% ≈ 5.26 мин/год. Каждая девятка кратно дороже предыдущей.
18. Перемножением: A = a₁ × a₂ × … × aₙ (система жива, только если живы все звенья на критическом пути).
19. Потому что произведение чисел < 1 всегда меньше любого сомножителя: каждая дополнительная зависимость на пути тянет итог вниз. Отсюда — короче критический путь.
20. A = 1 − (1−a)ⁿ для n независимых резервов (система жива, если жив хоть один). Доступность растёт с каждым резервом (например, два узла по 99% → 99.99%).
21. MTBF — среднее время между отказами, MTTR — среднее время восстановления. Доступность A = MTBF / (MTBF + MTTR).
22. Потому что в формуле A = MTBF/(MTBF+MTTR) важна доля времени в ремонте: быстрое восстановление (↓MTTR) сокращает простой так же, как редкие отказы (↑MTBF). Отсюда ценность автоматического failover и быстрого детекта.
23. Сохранение ядра функциональности при отказе части: урезанный ответ вместо полного отказа (лента без рекомендаций, поиск без подсказок, кэш вместо живого источника).
24. Радиус поражения отказа — насколько широко расходится сбой. Ограничивают изоляцией: bulkhead (отдельные пулы ресурсов), cell-based (независимые ячейки), лимиты на tenant — чтобы один сбой не задел всех.
25. Архитектура, где система нарезана на независимые «ячейки» (каждая — свой набор сервисов и данных), клиент закреплён за ячейкой; отказ одной ячейки затрагивает лишь её клиентов, а не всю систему (ограничивает blast radius).
