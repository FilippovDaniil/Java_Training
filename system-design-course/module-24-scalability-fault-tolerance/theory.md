# Модуль 24 — Масштабируемость и отказоустойчивость

## Зачем этот модуль

Части 3–5 дали строительные блоки (LB, кэш, шарды, репликация) и распределённые паттерны (CAP, консенсус, надёжность операций). Этот модуль связывает их двумя сквозными свойствами, которые senior обязан уметь **считать и обосновывать**: **масштабируемость** (система держит рост нагрузки добавлением ресурсов) и **отказоустойчивость** (система продолжает работать при отказе компонентов).

Зачем это senior'у. На интервью и в проектировании недостаточно сказать «добавим серверов» или «сделаем отказоустойчиво». Нужно: отличать вертикальный масштаб от горизонтального и знать их пределы; понимать, почему **stateless** масштабируется тривиально, а **stateful** — нет; находить **SPOF** и закрывать его избыточностью; и — главное — **считать доступность в числах**: сколько «девяток» даёт схема, как падает доступность при последовательном соединении компонентов и как растёт при резервировании.

Это первый модуль Части 6. Он отвечает на «как система растёт и как переживает отказы», переводя расплывчатые «надёжно/масштабируемо» в измеримые решения: scale-out, redundancy N+1/2N, failover, математика 9-к.

## 1. Вертикальное vs горизонтальное масштабирование

| | Vertical (scale up) | Horizontal (scale out) |
|--|---------------------|------------------------|
| Что делаем | мощнее машина (CPU/RAM/диск) | больше машин |
| Предел | потолок железа, дорого на верхах | практически без потолка |
| Доступность | одна машина = SPOF | избыточность встроена |
| Сложность | простая (код не меняется) | нужен LB, stateless, координация |
| Когда | быстрый выигрыш, БД-primary, простота | рост за пределы одной машины |

```
Scale up:  [2 CPU] → [32 CPU]      одна машина, упёрлись в потолок
Scale out: [m1] → [m1][m2]...[mN]  добавляем машины, растём дальше
```

> Вертикальный масштаб прост (ничего не переписываем), но имеет потолок и оставляет SPOF. Горизонтальный — основа масштаба на больших нагрузках: почти без предела и с встроенной избыточностью, ценой сложности (нужен балансировщик, stateless-сервисы, координация состояния). В дизайне дефолт для веб/сервисного слоя — **scale out**; vertical уместен там, где горизонталь дорога (часто — БД-primary до момента шардирования).

## 2. Stateless vs stateful

**Ключ к горизонтальному масштабу — отсутствие состояния в сервисе.**

```
Stateless: запрос самодостаточен; любой инстанс обработает любой запрос
  → добавил инстанс за LB — масштаб готов, отказ инстанса безболезнен

Stateful: инстанс хранит состояние (сессия в памяти, локальный кэш)
  → запрос «привязан» к инстансу (sticky), отказ = потеря состояния
```

| Что мешает горизонтальному масштабу | Решение |
|-------------------------------------|---------|
| сессия в памяти инстанса | вынести в Redis/внешнее хранилище сессий |
| sticky sessions на LB | stateless + внешняя сессия (модуль 8) |
| локальное состояние/файлы | в БД / blob / распределённый кэш |
| единая БД как общий узел | реплики (12), шардирование (13) |

> Правило: **делайте сервисный слой stateless**, выносите состояние во внешние хранилища (сессии — в Redis, файлы — в blob, данные — в БД). Тогда инстансы взаимозаменяемы: добавление = масштаб, отказ = безболезненная потеря одного из равных. Состояние неизбежно «где-то живёт» — задача в том, чтобы оно жило в специально масштабируемом/реплицируемом слое, а не в памяти случайного сервера.

**Shared-nothing** — архитектура, где узлы не делят память/диск и общаются только по сети; каждый автономен. Это фундамент горизонтального масштаба: нет общего ресурса → нет общего узкого места и общего SPOF.

## 3. Отказоустойчивость и избыточность

**Fault tolerance** — способность системы продолжать работать (возможно, деградированно) при отказе компонентов. Достигается **избыточностью** (redundancy): дублированием компонентов так, чтобы отказ одного не ронял систему.

| Уровень | Что значит | Переживает |
|---------|-----------|-----------|
| **N** | ровно столько мощности, сколько нужно | 0 отказов (нет запаса) |
| **N+1** | один резервный сверх нужного | 1 одновременный отказ |
| **N+2** | два резервных | 2 одновременных отказа |
| **2N** | полное дублирование | отказ половины (целого набора) |

```
Нужно 4 сервера под нагрузку:
  N    = 4   (отказ любого → перегруз/деградация)
  N+1  = 5   (один упал → оставшиеся 4 держат нагрузку)
  2N   = 8   (полная вторая линия, напр. для DR/обновлений без простоя)
```

> Избыточность — это **запас сверх рабочей мощности**, рассчитанный на отказы. N+1 — минимальный разумный дефолт (переживает один отказ и плановое обслуживание узла). 2N дороже вдвое, но даёт полное дублирование (часто для критичных систем / обновлений без простоя). Цена избыточности линейна — поэтому уровень выбирают под требуемую доступность и стоимость отказа, а не «по максимуму везде».

---

## 4. SPOF — единая точка отказа

**SPOF (Single Point of Failure)** — компонент, отказ которого роняет всю систему. Главная цель отказоустойчивого дизайна — **выявить и устранить все SPOF**.

```
Скрытые SPOF (классика):
  единственный LB · единственная primary-БД без реплики · один кэш-узел
  один экземпляр сервиса · единственный DNS/конфиг-сервис · одна сетевая зона
```

| Устранение SPOF | Как |
|-----------------|-----|
| LB | HA-пара (active-passive/active-active), floating IP (модуль 8) |
| primary-БД | реплики + failover (модуль 12) |
| сервис | ≥2 инстанса за LB, stateless |
| координатор | кворум из ≥3 узлов (модуль 22) |
| зона/ДЦ | multi-AZ / multi-region (модуль 25) |

> Поиск SPOF — обязательный шаг дизайна: пройти по схеме и спросить «что упадёт, если умрёт этот узел?». Любой компонент в единственном экземпляре на пути запроса — кандидат в SPOF. Внимание к **скрытым** SPOF: резервированный сервисный слой бесполезен, если он ходит в единственную primary-БД или через единственный LB. Отказоустойчивость определяется самым слабым звеном цепочки.

## 5. Failover

**Failover** — автоматическое переключение на резерв при отказе основного компонента.

| Схема | Как работает | Trade-off |
|-------|-------------|-----------|
| **Active-passive** | резерв простаивает, активируется при отказе основного | проще, нет конфликтов; резерв простаивает, есть лаг переключения |
| **Active-active** | оба обслуживают трафик одновременно | полная утилизация, мгновенный отказ части; сложнее (синхронизация состояния/данных, конфликты) |

```
Active-passive:  [PRIMARY ▶]  [standby ⏸]   отказ → промоут standby (лаг)
Active-active:   [node1 ▶]   [node2 ▶]      отказ → трафик на оставшийся (без лага)
```

> Active-passive проще и безопаснее (нет конфликтов записи, один «главный»), но резерв простаивает и переключение занимает время (детект отказа + промоут + перенаправление). Active-active утилизирует все ресурсы и переживает отказ без паузы, но требует решать синхронизацию и конфликты (для данных — это multi-leader, модуль 12). Выбор: для записи в БД часто active-passive (один primary), для stateless-сервисов — active-active.

## 6. Математика доступности («девятки»)

**Availability** — доля времени, когда система работоспособна. Считается в «девятках»:

| Доступность | Downtime в год | Downtime в месяц |
|-------------|----------------|------------------|
| 99% («две девятки») | ~3.65 дня | ~7.2 ч |
| 99.9% («три») | ~8.76 ч | ~43.8 мин |
| 99.99% («четыре») | ~52.6 мин | ~4.4 мин |
| 99.999% («пять») | ~5.26 мин | ~26 с |

**Последовательное соединение (зависимость):** общая доступность = **произведение** доступностей. Цепочка надёжна не более слабого звена и **всегда ниже** любого из них.

```
LB(99.9%) → app(99.9%) → БД(99.9%)
  A = 0.999 × 0.999 × 0.999 ≈ 0.997  → ~99.7% (хуже каждого звена!)
```

**Параллельное соединение (резервирование):** доступность **растёт**. Для n независимых резервов: A = 1 − (1−a)ⁿ (система жива, если жив хоть один).

```
Один узел a=99% → недоступность 1%
Два в параллели: 1 − (0.01)² = 1 − 0.0001 = 99.99%  (две девятки → четыре)
```

> Две формулы, которые senior считает в уме: **последовательность перемножает и снижает** (каждая зависимость на пути запроса тянет доступность вниз — поэтому меньше звеньев в критическом пути), **параллель/резервирование повышает** (1−(1−a)ⁿ — каждый резерв добавляет девяток). Отсюда два рычага: сокращать длину последовательной цепочки и резервировать критичные узлы. Цель по девяткам диктует и стоимость (5 девяток радикально дороже 3).

**MTBF / MTTR.** Доступность через надёжность и ремонт: **A = MTBF / (MTBF + MTTR)**, где MTBF — среднее время между отказами, MTTR — среднее время восстановления. Вывод: доступность растёт не только реже-ломаясь (↑MTBF), но и **быстрее-восстанавливаясь** (↓MTTR) — отсюда ценность автоматического failover и быстрого детекта.

## 7. Деградация и изоляция отказа

| Приём | Суть |
|-------|------|
| **Graceful degradation** | при отказе части — урезанная функциональность, не полный отказ (лента без рекомендаций, поиск без подсказок) |
| **Blast radius** | насколько широко расходится отказ; цель — минимизировать |
| **Failure isolation / bulkhead** | изолировать сбой в пределах одного отсека (модуль 23) |
| **Cell-based architecture** | система нарезана на независимые «ячейки» (cells), отказ одной задевает лишь её клиентов, не всех |

> Зрелая отказоустойчивость — не только «не падать», но и **ограничивать радиус поражения**: один сбойный компонент/регион/клиент не должен валить всех. Graceful degradation сохраняет ядро функциональности ценой второстепенного; cell-based и bulkhead ограничивают blast radius. Это переводит отказ из «вся система легла» в «часть пользователей увидела деградацию».

## ⚠️ Подводные камни

- **«Добавим серверов» без stateless.** Stateful-сервис (сессия в памяти, sticky) не масштабируется горизонтально честно; сначала вынести состояние.
- **Резервированный слой через единый узел.** Десять инстансов сервиса + одна primary-БД/один LB = SPOF не устранён.
- **Игнор последовательной формулы.** Цепочка из «надёжных» 99.9%-компонентов даёт заметно меньше 99.9%; длинный критический путь убивает доступность.
- **Резервирование без независимости.** Два узла в одной стойке/зоне/на одном питании отказывают вместе — формула 1−(1−a)ⁿ требует независимых отказов.
- **N вместо N+1.** «Ровно под нагрузку» означает отсутствие запаса: отказ одного → перегруз остальных (каскад).
- **Failover, который не тестировали.** Непротестированное переключение часто не срабатывает в реальный отказ (см. модуль 25 — game days).
- **Active-active для записи без решения конфликтов.** Два «главных» пишут несогласованно (это multi-leader — модуль 12).
- **Vertical как стратегия роста.** Упрётесь в потолок железа и останетесь с SPOF.
- **5 девяток «на всякий случай».** Каждая девятка кратно дороже; выбирайте под бизнес-цену простоя.
