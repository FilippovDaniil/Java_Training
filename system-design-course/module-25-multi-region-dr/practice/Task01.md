# Задача 01 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Обоснуйте переход с single-region на multi-region: какие проблемы это решает и что усложняет.

## ВХОДНЫЕ ДАННЫЕ
Сервис развёрнут в одном регионе (multi-AZ внутри). Симптомы: пользователи из другого полушария жалуются на высокую latency; бизнес требует пережить полный отказ региона; юристы сообщают, что данные клиентов из ЕС должны храниться в ЕС.

## ТРЕБОВАНИЯ
- Сопоставьте каждый симптом с драйвером multi-region (latency / катастрофоустойчивость / residency).
- Объясните, почему текущий multi-AZ не закрывает эти задачи.
- Перечислите, что multi-region усложняет (данные через WAN, конфликты, стоимость, операции).
- Сделайте вывод: оправдан ли переход здесь.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «симптом → драйвер» → почему multi-AZ недостаточно → усложнения → вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Latency → гео-близость; отказ региона → катастрофоустойчивость; данные ЕС в ЕС → data residency.
- [ ] Объяснено: multi-AZ даёт HA внутри региона, но не спасает от отказа целого региона, не приближает к дальним юзерам и не решает residency.
- [ ] Названы усложнения: WAN-репликация (latency/лаг), конфликты записи при active-active, ×стоимость, сложность операций/DR.
- [ ] Вывод обоснован: три независимых драйвера присутствуют → переход оправдан.
- [ ] Отмечено, что данные — главная сложность перехода.

## ПОДСКАЗКА
Multi-region решает три задачи: latency (близкий регион), катастрофоустойчивость (отказ региона), residency (данные в юрисдикции). Multi-AZ закрывает только HA внутри региона. Цена — сложность данных через WAN, конфликты, стоимость. Здесь все три драйвера налицо.
