# Задача 02 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Разграничьте multi-AZ и multi-region и определите, какой уровень нужен под каждое требование.

## ВХОДНЫЕ ДАННЫЕ
Набор требований к системе: (а) пережить отказ одного дата-центра без простоя; (б) пережить отказ целого облачного региона; (в) низкая latency для пользователей на двух континентах; (г) минимальная стоимость при достижении базовой HA.

## ТРЕБОВАНИЯ
- Для каждого требования укажите минимально достаточный уровень (multi-AZ или multi-region).
- Объясните, что именно изолирует каждый уровень.
- Отметьте, почему репликация в multi-AZ дешевле, чем в multi-region.
- Сформулируйте дефолт-рекомендацию для серьёзного продакшна.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «требование → уровень → обоснование» + вывод о дефолте.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (а) отказ одного ДЦ → multi-AZ; (б) отказ региона → multi-region; (в) две части света → multi-region (гео-близость); (г) базовая HA дёшево → multi-AZ.
- [ ] Multi-AZ изолирует отказ зоны в регионе (HA), multi-region — отказ региона (катастрофоустойчивость/residency/latency).
- [ ] Объяснено: зоны близко → синхронная репликация дёшева по latency; регионы далеко → WAN дорог/медленен.
- [ ] Дефолт: multi-AZ как почти бесплатная HA; multi-region добавлять осознанно под DR/latency/residency.
- [ ] Не путается HA (multi-AZ) с катастрофоустойчивостью (multi-region).

## ПОДСКАЗКА
Multi-AZ = HA внутри региона (зоны рядом, дешёвая синхронная репликация, переживаем отказ ДЦ). Multi-region = катастрофоустойчивость + гео-близость + residency (регионы далеко, WAN-репликация дорога). Дефолт — multi-AZ; multi-region осознанно.
