# Задача 04 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Определите RPO и RTO для разных классов данных одной системы, исходя из бизнес-критичности.

## ВХОДНЫЕ ДАННЫЕ
Платёжно-маркетплейс система с данными разной критичности: (а) финансовые транзакции/балансы; (б) пользовательские профили и заказы; (в) аналитические логи кликов; (г) кэш рекомендаций. Бюджет ограничен — нельзя дать всем RPO/RTO≈0.

## ТРЕБОВАНИЯ
- Для каждого класса данных назначьте целевые RPO и RTO (с обоснованием по цене потери/простоя).
- Объясните разницу между RPO (потеря данных) и RTO (время простоя) на этих примерах.
- Покажите, как требование RPO=0 влияет на тип репликации.
- Сделайте вывод: почему нельзя ставить всем RPO/RTO≈0.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «класс данных → RPO → RTO → обоснование» + выводы про репликацию и стоимость.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Финансы → RPO≈0, RTO минут (потеря денег недопустима); профили/заказы → RPO секунды–минуты, RTO минуты; логи → RPO минуты–часы; кэш → RPO большой/не критичен (восстановим), RTO низкий не нужен.
- [ ] Чётко разделены RPO (окно потери данных) и RTO (время недоступности).
- [ ] Показано: RPO≈0 требует синхронной репликации (дорогой по WAN-latency); большее RPO допускает async/бэкапы.
- [ ] Вывод: RPO/RTO≈0 для всех неоправданно дорого → дифференцировать по критичности.
- [ ] Обоснования привязаны к цене потери/простоя, а не «по умолчанию строго».

## ПОДСКАЗКА
RPO — сколько данных потеряем (0 → синхронная репликация; минуты → async; часы → бэкапы). RTO — за сколько восстановимся. Назначайте по цене потери/простоя: финансам — близко к нулю, кэшу/логам — щедро. Всем ≈0 — неоправданно дорого.
