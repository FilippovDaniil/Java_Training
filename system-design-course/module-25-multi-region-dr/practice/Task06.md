# Задача 06 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Спроектируйте слой данных для multi-region с учётом WAN-репликации, конфликтов и data residency.

## ВХОДНЫЕ ДАННЫЕ
Глобальный сервис в двух регионах (ЕС и США). Требования: пользователи ЕС обслуживаются из ЕС, пользователи США — из США; данные граждан ЕС не должны покидать ЕС (GDPR); цель — низкая latency записи в своём регионе; пережить отказ региона.

## ТРЕБОВАНИЯ
- Предложите схему размещения данных: что шардируется по региону (residency), что (если что-то) реплицируется глобально.
- Объясните, почему синхронная запись через регионы дорога (WAN-latency) и как это влияет на RPO.
- Решите вопрос конфликтов: возникают ли они при вашей схеме; если active-active глобально — как разрешать.
- Покажите, как схема переживает отказ региона при ограничении residency.

## ФОРМАТ РЕЗУЛЬТАТА
Схема размещения данных → WAN/RPO → конфликты → поведение при отказе региона.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Данные пользователей шардируются по региону (ЕС-данные в ЕС, US-данные в US) — выполняет residency и даёт локальную latency записи.
- [ ] Объяснено: синхронная межрегиональная запись дорога по WAN-latency; async-репликация → лаг → ненулевой RPO при failover.
- [ ] Конфликты: при шардировании по региону каждая запись имеет один «домашний» регион → конфликтов записи нет (не глобальный multi-leader); если глобальная active-active — назван механизм (LWW/CRDT, модуль 12).
- [ ] Отказ региона: для residency-ограниченных данных DR в пределах разрешённой географии (резерв в той же юрисдикции / задокументированное ограничение доступности).
- [ ] Учтена связь с модулями 12 (репликация/конфликты) и 13 (шардирование по региону).

## ПОДСКАЗКА
В multi-region именно данные — главная сложность. Residency решается шардированием по региону (данные не покидают юрисдикцию), что заодно даёт локальную latency и избегает глобального multi-leader (у каждой записи один домашний регион). WAN-async → RPO>0. DR для residency-данных ограничен той же географией.
