# Задача 07 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Спроектируйте маршрутизацию трафика по регионам и механику failover при отказе региона.

## ВХОДНЫЕ ДАННЫЕ
Active-passive multi-region: основной регион A (primary-БД), резервный B (async-реплика, warm standby). Нужно: направлять пользователей в живой регион и автоматически/контролируемо переключаться на B при катастрофе A. Цель RTO — 15 минут, RPO — 1 минута.

## ТРЕБОВАНИЯ
- Выберите механизм маршрутизации (GeoDNS / anycast / global LB) и обоснуйте.
- Опишите пошаговую механику failover на регион B.
- Учтите ловушку DNS TTL и её влияние на RTO.
- Оцените достижимость RPO=1 мин при async-репликации и опишите failback.

## ФОРМАТ РЕЗУЛЬТАТА
Выбор маршрутизации → шаги failover → роль DNS TTL → RPO-оценка → failback.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Маршрутизация выбрана (GeoDNS/global LB с health-проверкой региона) и обоснована.
- [ ] Шаги failover: детект отказа A → промоут реплики B в primary → переключить трафик на B → домасштабировать warm standby.
- [ ] DNS TTL: высокий TTL кэширует старый IP → трафик не уйдёт сразу; TTL надо снижать заранее, иначе RTO срывается.
- [ ] RPO: при async-репликации возможна потеря «хвоста» за lag; для RPO=1 мин лаг репликации должен быть ≤1 мин (мониторить лаг).
- [ ] Описан failback: контролируемый возврат на A с ресинхронизацией данных (не «само вернётся»).

## ПОДСКАЗКА
Маршрутизация — GeoDNS/global LB с health-проверкой. Failover: детект → промоут реплики → переключить трафик → домасштабировать. Ловушка — DNS TTL (высокий TTL держит трафик на упавшем регионе; снижать заранее). Async-лаг определяет фактический RPO. Failback — отдельная контролируемая операция.
