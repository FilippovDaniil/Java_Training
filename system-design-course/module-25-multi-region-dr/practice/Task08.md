# Задача 08 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Найдите ошибки в multi-region/DR-дизайне и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из дизайн-документа:
1. «У нас multi-AZ в одном регионе — значит мы переживём любой отказ, включая региональный».
2. «DR у нас есть: бэкапы раз в сутки в другой регион. Восстановимся мгновенно».
3. «RPO/RTO не считали — просто сделаем понадёжнее».
4. «Failover на резервный регион настроен, но ни разу не запускали — в проде сработает».
5. «DNS TTL у нас 24 часа, это снижает нагрузку на DNS».
6. «Будем писать в оба региона одновременно (active-active), конфликты сами разрулятся».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и корректное решение.
- Свяжите с понятиями модуля (multi-AZ vs multi-region, RPO/RTO, тестирование DR, DNS TTL, конфликты multi-leader).

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Multi-AZ ≠ multi-region: переживает отказ зоны, но не целого региона → для региональной катастрофы нужен multi-region.
- [ ] (2) Суточные бэкапы = RTO часы и RPO до 24ч, не «мгновенно» → для мгновенного нужен hot standby/active-active.
- [ ] (3) Без RPO/RTO нельзя выбрать DR-стратегию → задать числа по критичности.
- [ ] (4) Непротестированный failover обычно не срабатывает → регулярные game days/DR-учения.
- [ ] (5) TTL 24ч → после переключения трафик ещё сутки идёт на упавший регион → снижать TTL заранее.
- [ ] (6) Active-active запись = межрегиональный multi-leader, конфликты сами не разрешаются → нужен механизм (LWW/CRDT) или single-writer.

## ПОДСКАЗКА
Типовые ловушки DR: путать multi-AZ с multi-region, называть бэкапы «мгновенным DR», не задавать RPO/RTO, не тестировать failover, высокий DNS TTL, active-active без решения конфликтов. Каждая — прямое нарушение принципов модуля.
