# Задача 09 — Модуль 25: Multi-region и Disaster Recovery

## ЗАДАНИЕ
Спроектируйте размещение данных под требования data residency (GDPR и локальные законы) для глобального сервиса.

## ВХОДНЫЕ ДАННЫЕ
Сервис работает в ЕС, США и (по новому требованию) в стране с законом о локализации персональных данных (данные граждан должны физически храниться в стране). Сейчас всё в одной глобальной БД с глобальной репликацией во все регионы.

## ТРЕБОВАНИЯ
- Объясните, почему текущая глобальная репликация нарушает residency-требования.
- Предложите схему: какие данные локализуются по региону/стране, какие могут быть глобальными.
- Опишите, как это влияет на архитектуру (шардирование по региону, маршрутизация запросов к «домашнему» региону данных).
- Укажите последствия для DR (можно ли реплицировать residency-данные в другой регион для DR).

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз нарушения → схема размещения (локальное vs глобальное) → влияние на архитектуру → последствия для DR.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: глобальная репликация копирует ПД во все регионы → данные «покидают» юрисдикцию → нарушение residency/GDPR.
- [ ] Персональные/регулируемые данные шардируются и хранятся в стране/регионе пользователя; глобальными могут быть нечувствительные справочники/каталог.
- [ ] Архитектура: шардирование по региону (модуль 13) + маршрутизация запроса к «домашнему» региону данных пользователя.
- [ ] DR для residency-данных ограничен разрешённой географией: резерв в той же юрисдикции или задокументированный компромисс доступности.
- [ ] Отмечено, что residency ограничивает топологию (нельзя свободно реплицировать всё глобально).

## ПОДСКАЗКА
Residency запрещает данным покидать юрисдикцию → глобальная репликация ПД незаконна. Решение: шардировать персональные данные по региону/стране (данные остаются дома), глобально держать лишь нечувствительное. Это ограничивает и топологию, и DR (резерв — в той же юрисдикции).
