# Задача 10 — Модуль 25 (мини-проект): Multi-region + DR-план

## ЗАДАНИЕ
Спроектируйте multi-region архитектуру и полный DR-план системы с обоснованными RPO/RTO. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) глобальный SaaS (пользователи на 3 континентах); (б) платёжная платформа с регуляторикой; (в) маркетплейс с пиками распродаж. Дано: требование пережить отказ целого региона, часть данных под residency, ограниченный, но не нулевой бюджет.

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **Уровни** — multi-AZ (HA) + multi-region (DR/latency): что и зачем.
2. **Топология** — active-passive или active-active (по компонентам) + обоснование.
3. **RPO/RTO** — целевые значения по классам данных, обоснованные критичностью.
4. **DR-стратегия** — выбор на спектре (backup/pilot/warm/hot) под RPO/RTO + стоимость.
5. **Данные** — WAN-репликация, конфликты, residency (что шардируется по региону).
6. **Маршрутизация и failover** — GeoDNS/global LB, шаги переключения, DNS TTL, failback.
7. **Тестирование DR** — как проверяете план (game days), что именно валидируете.

## ФОРМАТ РЕЗУЛЬТАТА
Дизайн-документ из 7 разделов + диаграмма (регионы, потоки данных, точка failover) + таблица RPO/RTO по классам данных.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; разграничены multi-AZ (HA) и multi-region (DR/latency).
- [ ] Топология выбрана по компонентам и обоснована (особенно запись в БД).
- [ ] RPO/RTO заданы по классам данных и обоснованы критичностью (не «всем ≈0»).
- [ ] DR-стратегия выбрана под RPO/RTO + бюджет (место на спектре обосновано).
- [ ] Данные: WAN-репликация/лаг→RPO, конфликты (если active-active), residency через шардирование по региону.
- [ ] Маршрутизация + механика failover с учётом DNS TTL + failback.
- [ ] План тестирования DR (game days), валидирующий заявленные RPO/RTO; связь с модулями 12/13/24.

## ПОДСКАЗКА
Синтез модуля: multi-AZ (HA) + multi-region (DR/latency), топология по компонентам, RPO/RTO по классам данных → DR-стратегия на спектре (не платить за hot, где хватит warm), данные через WAN (async→RPO>0, residency→шард по региону), failover с учётом DNS TTL + failback, и обязательное тестирование (game days).
