# Задача 01 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Найдите узкое место системы по симптомам и метрикам, не угадывая.

## ВХОДНЫЕ ДАННЫЕ
Сервис под ростом нагрузки. Метрики: API-инстансы — CPU 30%, память 40%; БД — CPU 95%, медленные запросы в slow log, connection pool часто исчерпан; кэш Redis — CPU 15%; сеть — далеко от лимита. p99 запроса вырос с 80 мс до 600 мс, при этом трейсинг показывает: из 600 мс ~540 мс проводится в обращении к БД.

## ТРЕБОВАНИЯ
- Определите узкое место по данным и обоснуйте каждым показателем.
- Объясните, почему добавление API-инстансов НЕ поможет.
- Сформулируйте, что подтверждает диагноз (а не интуиция).
- Назовите следующий шаг (что проверить в БД до выбора решения).

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз (узкое место) → доказательства по метрикам → почему не API → следующий шаг.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Узкое место — БД (CPU 95%, slow log, исчерпан пул, 540/600 мс времени запроса в БД).
- [ ] Объяснено: API недогружен (CPU 30%) → добавление инстансов не сдвинет общую производительность (теория ограничений).
- [ ] Диагноз опирается на метрики/трейсинг, а не на догадку.
- [ ] Следующий шаг: разобрать планы медленных запросов/индексы/N+1, проверить размер пула (а не сразу «шардировать»).
- [ ] Отмечено: система не быстрее своего узкого места.

## ПОДСКАЗКА
Узкое место находят по данным: высокая утилизация ресурса + время запроса именно там. Здесь всё указывает на БД (CPU 95%, slow log, пул, 540 из 600 мс). API недогружен → его масштаб бесполезен. Сначала разобрать запросы/индексы/пул, потом выбирать решение.
