# Задача 02 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Диагностируйте БД как узкое место и предложите решения по каждой типичной причине.

## ВХОДНЫЕ ДАННЫЕ
Read-heavy сервис каталога. Наблюдения: страница товара делает 1 запрос списка + по запросу на каждый из ~50 элементов (видно в трейсинге); часть запросов делает полный скан таблицы (нет индекса по фильтру); под пиком исчерпывается connection pool, запросы ждут соединение.

## ТРЕБОВАНИЯ
- Идентифицируйте три проблемы (N+1, отсутствие индекса, исчерпание пула) по описанию.
- Для каждой предложите конкретное решение.
- Расположите решения по принципу «сначала дёшево/локально, потом дорого/структурно».
- Укажите, что проверить до перехода к шардированию/репликам.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «проблема → причина → решение» + порядок действий.

## КРИТЕРИИ ПРИЁМКИ
- [ ] N+1: 1+50 запросов → заменить на джойн/батч-запрос (IN/один запрос).
- [ ] Полный скан → добавить индекс по полю фильтра (проверить план запроса).
- [ ] Исчерпание пула → увеличить пул и/или сократить время удержания соединения (быстрее запросы); расчёт по Литтлу.
- [ ] Порядок: сначала индексы/устранение N+1/настройка пула (дёшево), затем read-реплики/кэш, и лишь потом шардирование.
- [ ] Отмечено: часто проблема в запросах/индексах, а не в «мало серверов» — проверить до структурных изменений.

## ПОДСКАЗКА
БД — узкое место №1. Типовые причины: N+1 (запрос в цикле → джойн/батч), нет индекса (полный скан → индекс по фильтру), исчерпание пула (увеличить/сократить удержание). Сначала дешёвые фиксы (индексы/запросы/пул), потом реплики/кэш, шардирование — в последнюю очередь.
