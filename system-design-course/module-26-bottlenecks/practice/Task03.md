# Задача 03 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Определите, что оптимизировать — latency или throughput — для разных кейсов, и выберите средства.

## ВХОДНЫЕ ДАННЫЕ
Три кейса:
- K1: интерактивный поиск — пользователь ждёт ответ, важно «мгновенно» (отдельный запрос медленный).
- K2: ночная обработка 100M записей — важно успеть за окно, отдельная запись может быть «медленной».
- K3: API под резкий рост числа клиентов — отдельные запросы быстрые, но система «захлёбывается» на пике.

## ТРЕБОВАНИЯ
- Для каждого кейса укажите, что главное — latency или throughput.
- Подберите средства под выбранное измерение.
- Назовите случай, где средства конфликтуют (улучшение одного ухудшает другое).
- Объясните разницу latency/throughput на этих примерах.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «кейс → latency/throughput → средства» + пример конфликта.

## КРИТЕРИИ ПРИЁМКИ
- [ ] K1 → latency (убрать шаги из критического пути, кэш, индексы, ближе данные).
- [ ] K2 → throughput (параллелизм, батчинг, больше воркеров; latency отдельной записи не важна).
- [ ] K3 → throughput (горизонтальный масштаб, async-очередь для сглаживания пика).
- [ ] Назван конфликт: батчинг повышает throughput, но добавляет latency (ждём набор пачки) — нельзя путать кейсы.
- [ ] Чётко разделены latency (время одного запроса) и throughput (запросов в единицу времени) как ортогональные.

## ПОДСКАЗКА
Latency (время одного запроса) и throughput (rps) — ортогональны, лечатся разным. Интерактив → latency (кэш/меньше шагов). Батч/пик по объёму → throughput (параллелизм/батчинг/масштаб/async). Конфликт: батчинг ↑throughput, но ↑latency. Сначала определить, что болит.
