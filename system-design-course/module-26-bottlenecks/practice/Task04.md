# Задача 04 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Рассчитайте размер пула (потоков/соединений) по закону Литтла и оцените эффект роста latency.

## ВХОДНЫЕ ДАННЫЕ
Сервис: пропускная λ = 800 rps, каждый запрос держит соединение к БД в среднем W = 50 мс. Текущий connection pool = 50. Отдельно рассмотрите сценарий, где downstream-БД замедлилась и W вырос до 150 мс при той же λ.

## ТРЕБОВАНИЯ
- Посчитайте по закону Литтла число одновременных запросов «в системе» при W=50 мс.
- Сравните с текущим пулом (50) — достаточно ли его.
- Посчитайте L при W=150 мс и объясните, что произойдёт с прежним пулом.
- Свяжите результат с каскадным отказом (исчерпание пула → очередь/таймауты).

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт L (W=50) → вывод по пулу → расчёт L (W=150) → последствие → связь с resilience.

## КРИТЕРИИ ПРИЁМКИ
- [ ] L = λ·W = 800 × 0.05 = 40 одновременных запросов при W=50 мс.
- [ ] Вывод: пул 50 > 40 → достаточно, но запас невелик (нет headroom под рост).
- [ ] При W=150 мс: L = 800 × 0.15 = 120 → пул 50 исчерпан (нужно ≥120).
- [ ] Последствие: новые запросы встают в очередь и ловят таймауты → деградация/каскад, хотя сам сервис исправен.
- [ ] Связь: медленный downstream «съедает» пул (Литтл) → timeout/bulkhead против каскада (модуль 23).

## ПОДСКАЗКА
Закон Литтла: L = λ·W (число в системе = rps × время). Считаем пул: при W=50мс L=40; пул 50 ок, но без запаса. При W=150мс L=120 → пул исчерпан → очередь/таймауты. Рост latency downstream роняет сервис через исчерпание пула — отсюда timeout/bulkhead.
