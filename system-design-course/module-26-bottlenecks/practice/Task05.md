# Задача 05 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Объясните, почему среднее обманывает, а p99 важен, и как fan-out усиливает хвост latency.

## ВХОДНЫЕ ДАННЫЕ
Сервис ленты собирает ответ, параллельно обращаясь к 100 шардам и ожидая все ответы. Метрики одного шарда: p50 = 5 мс, p99 = 200 мс, среднее = 12 мс. Руководство смотрит на «среднюю latency 12 мс» и считает, что всё хорошо.

## ТРЕБОВАНИЯ
- Объясните, почему среднее (12 мс) скрывает проблему, и что показывает p99.
- Покажите, почему запрос к 100 шардам с ожиданием всех делает «редкий» хвост частым (оценка вероятности зацепить медленный шард).
- Сделайте вывод о фактической latency запроса ленты.
- Предложите меры против хвоста при fan-out.

## ФОРМАТ РЕЗУЛЬТАТА
Среднее vs p99 → расчёт эффекта fan-out → вывод о latency запроса → меры.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: среднее маскирует хвост; p99=200 мс означает, что каждый сотый ответ шарда — 200 мс (страдают реальные пользователи).
- [ ] Fan-out: запрос ждёт max из 100; вероятность, что хотя бы один шард попадёт в свой p99 ≈ 1−(0.99)¹⁰⁰ ≈ 63% → у ~63% запросов есть «медленный» шард.
- [ ] Вывод: latency запроса ленты определяется хвостом шардов (≈p99 одного шарда), а не средним 12 мс.
- [ ] Меры: hedged requests (дублировать медленные), таймаут на шард + частичный ответ, ограничить fan-out, устранить источник хвоста (GC/блокировки).
- [ ] Подчёркнуто: смотреть p99/p999, не среднее.

## ПОДСКАЗКА
Среднее прячет хвост — смотрят перцентили (p99/p999). При fan-out общая latency = max по всем вызванным, поэтому редкий хвост (p99=1%) на 100 бэкендах становится частым: 1−0.99¹⁰⁰≈63%. Лечение: hedged requests, таймаут+частичный ответ, меньше fan-out, убрать источник хвоста.
