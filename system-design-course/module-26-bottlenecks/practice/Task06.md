# Задача 06 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Примените методологию оптимизации (измерить → найти → устранить → повторить) к кейсу и покажите, как узкое место «переезжает».

## ВХОДНЫЕ ДАННЫЕ
Стартовое состояние: сервис держит 1000 rps, узкое место — БД (CPU 100% на чтениях горячих данных). Команда просит «просто ускорить систему». Бюджет на итеративную работу есть.

## ТРЕБОВАНИЯ
- Опишите цикл методологии для этого кейса по шагам.
- Покажите минимум два витка: устранили БД-узкое место → появилось следующее.
- На каждом витке: что измерили, что стало узким местом, какой блок применили.
- Объясните, почему нельзя «оптимизировать всё сразу» и где остановиться (headroom/цель).

## ФОРМАТ РЕЗУЛЬТАТА
Виток 1 (измерить→найти→устранить) → виток 2 (новое узкое место) → когда остановиться.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Виток 1: измерили (CPU БД 100% на чтениях) → узкое место БД → кэш/read-реплики горячих чтений → пропускная выросла.
- [ ] Виток 2: перемерили → новое узкое место (например, CPU сервиса или сеть/пул) → применили соответствующий блок (горизонтальный масштаб/настройка).
- [ ] Явно показано, что узкое место «переехало» после устранения первого.
- [ ] Объяснено: оптимизируют только текущее узкое место (остальное не даст эффекта); цикл повторяется.
- [ ] Остановка по цели/headroom: довести до целевого RPS с запасом, не оптимизировать бесконечно.

## ПОДСКАЗКА
Методология: измерить → найти узкое место → устранить → повторить. После устранения БД-узкого места появляется следующее (CPU/сеть/пул) — это нормально, оно переезжает. Работают только с текущим ограничением. Останавливаются по достижении цели с запасом (headroom), а не «оптимизируют всё».
