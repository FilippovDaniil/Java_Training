# Задача 07 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Устраните узкое место подбором правильного строительного блока под каждый диагноз.

## ВХОДНЫЕ ДАННЫЕ
Набор подтверждённых измерениями узких мест в разных системах:
- A: 95% запросов читают одни и те же 1000 «горячих» записей, БД перегружена чтениями.
- B: данные выросли до 20 ТБ, запись не вмещается в один узел, primary упирается в диск/CPU.
- C: запрос синхронно делает тяжёлую генерацию отчёта (3–5 с), блокируя поток.
- D: пользователи в Азии жалуются на медленную загрузку статики и видео (origin в США).
- E: сервис исправен, но не хватает пропускной — CPU всех инстансов под 100%.

## ТРЕБОВАНИЯ
- Для каждого узкого места подберите блок и обоснуйте.
- Сошлитесь на соответствующий модуль.
- Отметьте, почему другой блок здесь не помог бы.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «узкое место → блок → обоснование → модуль».

## КРИТЕРИИ ПРИЁМКИ
- [ ] A → кэш горячих данных (модуль 9): повторные чтения малого набора.
- [ ] B → шардирование (модуль 13): объём/запись не вмещаются в узел (реплики не решат запись/объём).
- [ ] C → async/очередь (модуль 15): вынести тяжёлую генерацию из синхронного запроса.
- [ ] D → CDN (модуль 10): приблизить статику/медиа к пользователям, разгрузить origin.
- [ ] E → горизонтальный масштаб (модуль 24): добавить инстансы (сервис stateless, упёрлись в CPU).
- [ ] Для 1–2 случаев отмечено, почему неверный блок не сработал бы (например, кэш не решит объём данных B).

## ПОДСКАЗКА
Блок выбирается под диагноз: повторные чтения → кэш; объём/запись не вмещаются → шард; тяжёлая синхронная работа → async; дальние пользователи/статика → CDN; нехватка пропускной stateless-сервиса → горизонтальный масштаб. Неверный блок не сдвинет нужную метрику.
