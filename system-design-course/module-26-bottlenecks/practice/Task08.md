# Задача 08 — Модуль 26: Анализ узких мест и оптимизация

## ЗАДАНИЕ
Найдите ошибки в подходе к оптимизации и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из обсуждения производительности:
1. «Система тормозит — давайте добавим инстансов API» (метрики не смотрели).
2. «Средняя latency 15 мс, всё отлично» (про сервис с fan-out на 50 бэкендов).
3. «Перепишем сериализацию на ассемблере для скорости» (профиль не снимали, узкое место — БД).
4. «Запас не нужен, держим ровно под текущую нагрузку — экономия».
5. «Добавим 100 узлов и ускоримся в 100 раз» (есть общая блокировка/единый координатор).
6. «Узкое место устранили, на этом с производительностью закончили» (после одного фикса).

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и корректный подход.
- Свяжите с понятиями модуля (измерять до оптимизации, p99/fan-out, не то узкое место, headroom, закон Амдала, итеративность).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → ошибка → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Оптимизация наугад без измерения → сначала найти узкое место по метрикам (возможно, не API).
- [ ] (2) Среднее маскирует хвост; при fan-out важен p99/p999 (общая latency = max).
- [ ] (3) Оптимизация не того узкого места (premature/микро-оптимизация) → профилировать, работать с БД.
- [ ] (4) Нет headroom → пик кладёт систему; держать запас ёмкости.
- [ ] (5) Закон Амдала: последовательная часть (блокировка/координатор) ограничивает ускорение, 100× не будет.
- [ ] (6) Узкое место переезжает → методология итеративна (повторить цикл до цели).

## ПОДСКАЗКА
Типовые ошибки: оптимизировать без измерения, смотреть среднее вместо p99 (особенно с fan-out), оптимизировать не то место, работать без headroom, верить в линейное ускорение (Амдал), считать работу законченной после одного фикса (узкое место переезжает).
