## Ключ (эталонные ответы)

1. Компонент, ограничивающий пропускную способность или latency всей системы; именно он определяет потолок производительности.
2. Система не быстрее своего узкого места: общая производительность ограничена самым слабым звеном, а не суммой возможностей компонентов.
3. Потому что потолок задаёт узкое место: усиление любого другого компонента не сдвинет общую производительность (потратили усилия — метрика не изменилась).
4. Измерением: метрики/мониторинг (утилизация ресурсов, latency по слоям, очереди), профилирование, нагрузочное тестирование, распределённый трейсинг — найти, где время и что упирается первым.
5. Потому что интуиция часто ошибается: реальное узкое место обычно не там, где кажется. Оптимизация по догадке улучшает не то и не даёт эффекта.
6. БД — чаще всего: блокировки, медленные запросы, N+1, отсутствие индексов, исчерпание connection pool. Через неё проходит состояние, и она хуже масштабируется горизонтально.
7. Блокировки (contention), медленные запросы (полный скан без индекса), N+1 (запрос в цикле вместо джойна/батча), отсутствие нужных индексов, исчерпание пула соединений.
8. Потому что при исчерпании пула запросы встают в очередь и ловят таймауты, маскируясь под «медленный сервис», хотя сам сервис исправен; размер пула надо считать (по Литтлу), а не задавать наугад.
9. Latency — время одного запроса (мс); throughput — число запросов в единицу времени (rps/qps). Это ортогональные измерения: высокий throughput может сочетаться с высокой latency и наоборот.
10. Latency: убрать шаги из критического пути, кэш, индексы, ближе данные (CDN/реплики). Throughput: параллелизм, горизонтальный масштаб, батчинг, async-очереди.
11. Батчинг: накопление запросов в пачку повышает throughput, но добавляет latency (каждый запрос ждёт, пока наберётся пачка). Async-буферизация — аналогично.
12. L = λ × W: среднее число запросов «в системе» = пропускная (rps) × среднее время в системе (latency, сек). Связывает нагрузку, latency и число одновременных запросов.
13. Число одновременно «в работе» L = λ·W; пул потоков/соединений должен быть ≥ L, иначе запросы встают в очередь и таймаутят (например, 500 rps × 0.2 с = 100 → пул ≥ 100).
14. Растёт: при той же λ рост W увеличивает L = λ·W → больше одновременных запросов → пул исчерпывается → очередь/таймауты. Поэтому медленный downstream «съедает» пул и роняет исправный сервис.
15. Потому что среднее маскирует хвост: несколько очень медленных запросов почти не сдвигают среднее, но это реальные страдающие пользователи. «Среднее 20 мс» может скрывать p99=1000 мс.
16. 99-й перцентиль latency: значение, ниже которого укладывается 99% запросов (каждый сотый — медленнее). Senior смотрит на него (и p999), т.к. хвост определяет реальный UX, а среднее обманывает.
17. «Хвост» распределения latency — самые медленные запросы (высокие перцентили p99/p999). Именно он определяет худший пользовательский опыт и, при fan-out, общую latency.
18. При запросе к N бэкендам/шардам с ожиданием всех общая latency = максимум из N. Чем больше N, тем выше вероятность зацепить чей-то хвостовой ответ: при p99 у одного бэкенда и N=100 у ~63% запросов будет медленный бэкенд (1−0.99¹⁰⁰).
19. Hedged requests (дублировать/перезапрашивать медленные), таймаут на каждый вызов + частичный ответ, ограничение fan-out, устранение источника хвоста (GC-паузы, удержание блокировок, холодный кэш).
20. Измерить → найти узкое место → устранить (применить блок) → повторить. Сначала локализовать по данным, потом работать только с текущим ограничением, затем перепроверить (узкое место сместилось).
21. Потому что после устранения одного ограничения потолок задаёт следующий компонент. Следствие: оптимизация итеративна — цикл повторяют до достижения цели, а не «один фикс и готово».
22. По диагнозу: повторные чтения горячих данных → кэш; перегруз чтениями → реплики/индексы/денормализация; объём/запись не вмещаются → шардирование; тяжёлая синхронная работа → async/очередь; статика/дальние юзеры → CDN; нехватка пропускной → горизонтальный масштаб.
23. Оптимизация до измерения и до того, как это стало реальной проблемой. Вредна: усложняет код, добавляет техдолг и риск без подтверждённой выгоды; часто бьёт мимо настоящего узкого места.
24. Запас ёмкости сверх текущей нагрузки (не работать на 100% утилизации). Нужен, чтобы пик/всплеск/отказ узла не клал систему; без headroom любой скачок ведёт к деградации.
25. Максимальное ускорение от параллелизма ограничено долей последовательной части s: speedup ≤ 1/s, сколько бы узлов ни добавляли. Вывод: масштабирование упирается в непараллелизуемую часть (общая блокировка/единый координатор) — её и нужно сокращать.
