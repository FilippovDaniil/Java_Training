# Модуль 26 — Анализ узких мест и оптимизация

## Зачем этот модуль

Предыдущие модули Части 6 дали масштаб (24) и географическую устойчивость (25). Но прежде чем масштабировать, нужно понять — **что именно ограничивает систему**. «Добавить серверов» бесполезно, если упёрлись в единственную БД; кэш не поможет, если узкое место в сети. Этот модуль про дисциплину поиска и устранения **узких мест (bottlenecks)** — навык, отличающий senior'а от того, кто оптимизирует наугад.

Зачем это senior'у. Система работает не быстрее своего узкого места (теория ограничений). Оптимизировать что угодно, кроме него, — потраченное время: ускорили не-узкое место → общая производительность не изменилась. Senior умеет: **найти** узкое место по метрикам (а не угадать), отличить проблему **latency** от проблемы **throughput**, посчитать нужный размер пула/очереди (**закон Литтла**), понять, почему **p99** важнее среднего, и применить правильный строительный блок к найденному ограничению. И — не оптимизировать преждевременно.

Это третий модуль Части 6. Он даёт методологию (измерить → найти → устранить → повторить) и связывает её со всем арсеналом блоков: кэш, реплики, шардирование, async, CDN, горизонтальный масштаб.

---

## 1. Что такое узкое место

**Bottleneck** — компонент, ограничивающий пропускную способность или latency всей системы. **Закон (теория ограничений): система не быстрее своего узкого места.** Усиление любого другого компонента не даёт прироста.

```
Конвейер: [API 10k rps] → [БД 2k rps] → [кэш 50k rps]
  Узкое место — БД (2k rps). Система держит 2k rps.
  Удвоить API или кэш → НИЧЕГО не изменится. Удвоить БД → система до 4k rps.
```

> Главная мысль модуля: **производительность определяется самым слабым звеном**. Поэтому первая задача — не «ускорить систему вообще», а **найти конкретное узкое место** и работать с ним. Оптимизация не-узкого места — иллюзия прогресса (потратили усилия, метрика не сдвинулась). После устранения одного узкого места появляется следующее (оно «переезжает») — отсюда итеративность.

---

## 2. Как находить узкое место

Не угадывать, а **измерять**:

| Инструмент | Что показывает |
|------------|----------------|
| Метрики/мониторинг (модуль 28) | утилизация ресурсов (CPU/mem/IO/сеть), latency по слоям, очереди |
| Профилирование | где код/запросы проводят время |
| Нагрузочное тестирование (модуль 27) | при каком RPS что упирается первым |
| Распределённый трейсинг | latency по шагам запроса через сервисы |

```
Симптом → гипотеза → подтверждение метрикой → устранение → перепроверка
  (не «кажется, БД медленная» → а «p99 запроса = 400 мс, из них 380 мс в БД»)
```

> Узкое место **находят по данным**, а не по интуиции. Признак приближения к пределу — высокая утилизация ресурса (CPU ~100%, пул соединений исчерпан, очередь растёт, диск насыщен) и рост latency именно на этом шаге. Метрики, трейсинг и нагрузочное тестирование показывают, **где** время и **что** упирается первым. Без измерения оптимизация — угадайка.

## 3. Типичные узкие места

| Ресурс | Частые причины | Куда смотреть |
|--------|----------------|---------------|
| **БД** (самое частое) | блокировки, медленные запросы, N+1, нет индексов, исчерпание connection pool | план запроса, slow log, пул |
| **CPU** | тяжёлые вычисления, сериализация, неэффективные алгоритмы | профиль, нагрузка ядра |
| **Память** | утечки, большой рабочий набор, GC-паузы | heap, частота/длительность GC |
| **Сеть / bandwidth** | большие ответы, болтливость (chatty), egress | объём трафика, RTT |
| **Disk I/O** | случайные чтения, fsync, лог | iops, latency диска |
| **Пулы (потоки/соединения)** | исчерпание → очередь/таймауты | размер пула vs нагрузка |
| **Внешние зависимости** | медленный downstream, отсутствие timeout | latency вызовов (модуль 23) |

> **БД — узкое место №1** в большинстве систем: блокировки, медленные запросы (нет индекса, полный скан), N+1 (запрос в цикле вместо джойна/батча), исчерпание connection pool. Прежде чем шардировать или добавлять реплики — проверьте планы запросов и индексы (часто проблема там, а не в «мало серверов»). Пулы (потоки/соединения) — частый скрытый предел: при исчерпании запросы встают в очередь и ловят таймауты, маскируясь под «медленный сервис».

---

## 4. Latency vs Throughput

Два разных измерения производительности — их часто путают:

| | Latency | Throughput |
|--|---------|-----------|
| Что | время одного запроса | запросов в единицу времени |
| Единица | мс | rps / qps |
| Улучшают | быстрее обработка, кэш, ближе данные | параллелизм, больше узлов, батчинг |

```
Аналогия: труба.
  Latency   = время прохода одной молекулы (длина трубы).
  Throughput = сколько литров в секунду (диаметр трубы).
Можно иметь высокий throughput при высокой latency (широкая длинная труба, батч-конвейер)
и низкую latency при низком throughput (короткая узкая труба).
```

> Latency и throughput — ортогональны, и оптимизируются разными средствами. Снижение latency: убрать шаги из критического пути, кэш, индексы, ближе данные (CDN/реплики). Рост throughput: параллелизм, горизонтальный масштаб, батчинг, async-очереди (сглаживание). Иногда они конфликтуют: батчинг повышает throughput, но добавляет latency (ждём, пока наберётся пачка). Сначала определите, **что именно** болит — медленные отдельные запросы (latency) или нехватка пропускной (throughput).

## 5. Закон Литтла

**L = λ × W**, где L — среднее число запросов «в системе», λ — пропускная (rps), W — среднее время в системе (latency, сек).

Практическое применение — расчёт размера пула/числа параллельных обработчиков:

```
Нужно обслужить λ = 500 rps, каждый запрос держит поток W = 0.2 с.
  L = 500 × 0.2 = 100 одновременных запросов «в работе».
  → пул потоков/соединений должен быть ≥ 100 (иначе очередь и таймауты).
Если W вырос до 0.4 с (downstream замедлился): L = 200 → прежний пул (100) исчерпан → деградация.
```

> Закон Литтла — карманный инструмент capacity-расчётов: связывает пропускную, latency и число одновременных запросов. Им считают размер пула потоков/соединений, длину очереди, число воркеров. Важное следствие: при **росте latency** (W↑) при той же нагрузке (λ) растёт число одновременных запросов (L↑) → пул исчерпывается → новые запросы встают в очередь и таймаутят. Поэтому медленный downstream «съедает» пул и роняет сервис, даже если сам сервис исправен (связь с модулем 23: timeout/bulkhead).

## 6. Tail latency (p99) и fan-out

Среднее (mean) скрывает проблему — смотрят на **перцентили**: p50 (медиана), p95, **p99**, p999.

```
1000 запросов: 990 по 10 мс, 10 по 1000 мс.
  Среднее ≈ 20 мс (выглядит отлично).
  p99 = 1000 мс (каждый сотый пользователь ждёт секунду!).
Хвост (tail) определяет реальный UX, среднее — обманывает.
```

**Fan-out усиливает хвост.** Если запрос обращается к N сервисам/шардам параллельно и ждёт всех, его latency = максимум из N. Чем больше N, тем вероятнее зацепить чей-то «хвостовой» ответ:

```
p99 одного бэкенда = 1%. Запрос ждёт 100 бэкендов → вероятность,
что хотя бы один медленный ≈ 1−(0.99)¹⁰⁰ ≈ 63%. p99 одного → ~p63 общего.
```

> Senior смотрит на **p99/p999**, а не на среднее: среднее маскирует хвост, а именно хвост — это реальные страдающие пользователи (и, при fan-out, почти все запросы). **Fan-out** (лента, агрегация по шардам, микросервисный запрос к десяткам зависимостей) **усиливает хвост**: общая latency = max по всем вызванным, поэтому редкий медленный ответ становится частым на уровне запроса. Лечение хвоста: hedged requests (дублировать медленные), таймауты, ограничение fan-out, борьба с источником хвоста (GC-паузы, holdка на блокировке).

## 7. Методология оптимизации и выбор решения

**Цикл:** измерить → найти узкое место → устранить → **повторить** (узкое место переезжает).

| Найденное узкое место | Типовое решение (блок) |
|-----------------------|------------------------|
| повторные чтения горячих данных | кэш (модуль 9) |
| перегруз чтениями БД | read-реплики (12), индексы, денормализация (11) |
| данные/запись не вмещаются в узел | шардирование (13) |
| тяжёлая синхронная обработка в запросе | вынести в async/очередь (15) |
| статика/медиа, дальние пользователи | CDN (10) |
| не хватает пропускной сервиса | горизонтальный масштаб (24) |
| медленные запросы/N+1 | индексы, переписать запрос, батч/джойн |

**Антипаттерны:** **premature optimization** (оптимизировать до измерения и до того, как это стало проблемой — усложняет код без выгоды); оптимизация **не того** узкого места (метрика не сдвинется); забывать про **headroom** (запас ёмкости — не работать на 100% утилизации, иначе пик кладёт систему).

**Закон Амдала** — предел ускорения от параллелизма: если доля последовательной (непараллелизуемой) части = s, максимальное ускорение ≤ 1/s, сколько бы ядер/узлов ни добавляли. Вывод: масштабирование упирается в последовательную часть (общая блокировка, единый координатор) — её и надо устранять.

> Методология важнее отдельных приёмов: **сначала измерить и локализовать**, потом применить блок к конкретному ограничению, потом перепроверить (узкое место сместилось). Выбор блока — прямое следствие диагноза (кэш для повторных чтений, шард для объёма, async для тяжёлой обработки). Избегайте premature optimization (усложнение без подтверждённой нужды) и держите headroom (запас под пик). Закон Амдала напоминает: параллелизм не безграничен — последовательная часть ставит потолок.

## ⚠️ Подводные камни

- **Оптимизировать наугад.** Без измерения улучшаете не-узкое место → метрика не меняется. Сначала найти.
- **Оптимизировать не то узкое место.** Ускорили API, а упёрлись в БД → ноль эффекта.
- **Смотреть на среднее, не на p99.** Среднее маскирует хвост; страдают реальные пользователи на p99/p999.
- **Игнор fan-out при хвосте.** При запросе к N бэкендам общая latency = max; редкий хвост становится частым.
- **Premature optimization.** Усложнение кода до подтверждённой проблемы — техдолг без выгоды.
- **Забыть про пулы.** Исчерпание connection/thread pool маскируется под «медленный сервис»; считать по Литтлу.
- **Не учитывать рост latency downstream.** W↑ → L↑ → пул исчерпан (Литтл) → каскад (timeout/bulkhead, модуль 23).
- **Работать на 100% утилизации.** Нет headroom → любой пик кладёт систему; держать запас.
- **Верить, что «добавим узлов» решит всё.** Закон Амдала: последовательная часть ограничивает; единый координатор/блокировка останется узким местом.
- **Считать узкое место устранённым навсегда.** Оно переезжает — повторять цикл.
