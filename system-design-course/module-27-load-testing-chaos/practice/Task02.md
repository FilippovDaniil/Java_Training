# Задача 02 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Спроектируйте сценарий нагрузочного теста для сервиса: методология, профиль нагрузки, метрики и критерии успеха.

## ВХОДНЫЕ ДАННЫЕ
REST API интернет-магазина. Реальный паттерн трафика: ~70% GET каталога/товара, ~20% поиск, ~10% оформление заказа (запись). Целевая нагрузка — 8000 RPS, SLO: p99 < 300 мс, error rate < 0.5%. Есть отдельный стенд, по топологии повторяющий прод.

## ТРЕБОВАНИЯ
- Опишите методологию: baseline → ramp-up → измерение → анализ.
- Задайте реалистичный профиль нагрузки (микс эндпоинтов в указанных пропорциях, не один эндпоинт).
- Перечислите метрики, которые снимаете.
- Сформулируйте критерии успеха/провала теста относительно SLO.
- Учтите прогрев (кэши/JIT) и реалистичные данные.

## ФОРМАТ РЕЗУЛЬТАТА
План теста: этапы → профиль нагрузки → метрики → критерии успеха.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Методология: baseline (без/малой нагрузки) → плавный ramp-up → удержание целевой → анализ.
- [ ] Профиль реалистичен: микс 70/20/10 (каталог/поиск/заказ), а не один эндпоинт.
- [ ] Метрики: throughput, latency p50/p95/p99, error rate, утилизация ресурсов.
- [ ] Критерии успеха привязаны к SLO (p99 < 300 мс и error rate < 0.5% при 8000 RPS).
- [ ] Учтён warm-up (прогрев кэшей/JIT) и реалистичные данные/объёмы.
- [ ] Упомянут поиск точки насыщения (knee) при превышении целевой нагрузки.

## ПОДСКАЗКА
План: baseline → ramp-up → удержание → анализ. Нагрузка должна повторять реальный микс эндпоинтов (70/20/10), а не бить в один. Метрики — throughput/p99/error rate/утилизация. Критерии — относительно SLO. Не забыть прогрев и реалистичные данные, иначе картина ложная.
