# Задача 03 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Интерпретируйте результаты нагрузочного теста: найдите точку насыщения и узкое место.

## ВХОДНЫЕ ДАННЫЕ
Результаты ramp-up теста (offered load → наблюдаемые метрики):
| Offered RPS | Throughput | p99 latency | Error rate | CPU БД | CPU app |
|-------------|-----------|-------------|-----------|--------|---------|
| 1000 | 1000 | 40 мс | 0% | 30% | 25% |
| 3000 | 3000 | 60 мс | 0% | 70% | 45% |
| 5000 | 4900 | 250 мс | 0.5% | 95% | 55% |
| 7000 | 5000 | 1800 мс | 12% | 99% | 58% |
| 9000 | 4800 | таймауты | 40% | 100% | 60% |

## ТРЕБОВАНИЯ
- Определите точку насыщения (knee) по данным.
- Назовите узкое место и обоснуйте метриками.
- Объясните, почему при 7000–9000 throughput не растёт (даже падает), а latency/ошибки взлетают.
- Дайте рекомендацию (что оптимизировать), сославшись на модуль 26.

## ФОРМАТ РЕЗУЛЬТАТА
Точка насыщения → узкое место (с доказательством) → объяснение коллапса → рекомендация.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Knee ≈ 5000 RPS: дальше throughput плато/падение (~5000), p99 и error rate резко растут.
- [ ] Узкое место — БД (CPU БД 95–100% у knee, тогда как CPU app лишь ~55–60%).
- [ ] Объяснено: после насыщения БД запросы копятся в очереди → latency взлетает, таймауты/ошибки растут, throughput не растёт (упёрлись).
- [ ] Рекомендация направлена на БД (индексы/кэш/реплики/шард — модуль 26), а не на добавление app-инстансов.
- [ ] Отмечено: добавлять app бесполезно (он недогружен) — система не быстрее узкого места.

## ПОДСКАЗКА
Knee — где throughput выходит на плато, а latency/ошибки взлетают (здесь ~5000 RPS). Узкое место ищут по утилизации: БД 95–100% при недогруженном app (55%) → БД. После насыщения очередь растёт → latency/таймауты. Оптимизировать БД, не app.
