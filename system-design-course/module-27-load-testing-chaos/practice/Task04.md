# Задача 04 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Объясните разницу open vs closed модели нагрузки и выберите правильную модель для цели теста.

## ВХОДНЫЕ ДАННЫЕ
Команда прогнала тест в инструменте с настройкой «100 виртуальных пользователей, каждый шлёт запрос и ждёт ответа, затем следующий». Результат: «при замедлении сервиса RPS сам снижался, коллапса не увидели — всё стабильно». Цель теста была — проверить поведение при реальном наплыве пользователей в пиковый день.

## ТРЕБОВАНИЯ
- Определите, какая модель использовалась (open или closed), и почему.
- Объясните, почему в этой модели «коллапс не увидели» — что именно она маскирует.
- Объясните, как ведут себя реальные пользователи (какая модель их описывает).
- Укажите, какую модель надо было использовать для цели и почему.
- Кратко свяжите с coordinated omission.

## ФОРМАТ РЕЗУЛЬТАТА
Определение модели → что маскирует → модель реальных пользователей → правильный выбор → coordinated omission.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Использована closed-модель (фиксированное число VU, каждый ждёт ответа перед следующим запросом).
- [ ] Объяснено: при замедлении VU просто ждут дольше → фактический rate падает сам → перегруз маскируется («защищён» числом VU).
- [ ] Реальные пользователи приходят по open-модели (фиксированный rate прибытия, им всё равно, ответили ли предыдущим).
- [ ] Для проверки наплыва нужна open-модель (фиксированный rate λ независимо от ответов) → реалистичный перегруз/коллапс.
- [ ] Coordinated omission: измерение latency только по завершённым занижает хвост (медленные «не успели» отправиться).

## ПОДСКАЗКА
Closed: N VU, каждый ждёт ответа → при замедлении rate падает сам → перегруз скрыт. Open: фиксированный rate прибытия независимо от ответов → реалистичный перегруз. Реальные юзеры = open. Для наплыва нужна open-модель. Coordinated omission занижает хвост latency.
