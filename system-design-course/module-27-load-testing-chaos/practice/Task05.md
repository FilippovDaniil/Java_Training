# Задача 05 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Спроектируйте soak (endurance) тест и объясните, какие проблемы он ловит, недоступные короткому тесту.

## ВХОДНЫЕ ДАННЫЕ
Сервис успешно прошёл 15-минутный load-test на целевой нагрузке (p99 и error rate в пределах SLO). Но в проде через 2–3 дня работы наблюдаются: постепенный рост latency, периодические перезапуски подов (OOM), медленное «забивание» пула соединений.

## ТРЕБОВАНИЯ
- Объясните, почему 15-минутный тест не поймал эти проблемы.
- Спроектируйте soak-тест (длительность, нагрузка, что мониторить во времени).
- Перечислите классы проблем, которые ловит именно длительный тест.
- Укажите, на какие тренды метрик смотреть (а не на мгновенные значения).

## ФОРМАТ РЕЗУЛЬТАТА
Почему короткий тест слеп → дизайн soak-теста → классы ловимых проблем → тренды метрик.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: за 15 минут утечки/накопление не успевают проявиться; деградация во времени видна только на длинной дистанции.
- [ ] Soak-тест: штатная (целевая) нагрузка, удержание часы/сутки, непрерывный мониторинг.
- [ ] Классы проблем: утечки памяти (OOM/рост GC), накопление в пулах (соединения/потоки/файловые дескрипторы), фрагментация, рост latency во времени, переполнение логов/диска.
- [ ] Акцент на тренды (память/latency/GC во времени растут?), а не мгновенные значения.
- [ ] Связь с тем, что эти проблемы — деградация во времени, не пиковая нагрузка.

## ПОДСКАЗКА
Утечки и накопление (память, пулы, дескрипторы) проявляются только со временем — короткий тест их не ловит. Soak: штатная нагрузка, но часы/сутки, смотреть на тренды (память/GC/пул/latency растут?). Симптомы из условия (OOM, рост latency, забитый пул) — классика soak-теста.
