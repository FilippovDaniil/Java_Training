# Задача 06 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Спроектируйте chaos-эксперимент по принципам chaos engineering для проверки заявленной отказоустойчивости.

## ВХОДНЫЕ ДАННЫЕ
Система заявлена как отказоустойчивая: 4 stateless-инстанса сервиса за LB (переживает отказ инстанса), primary-БД с репликой и автоматическим failover (заявлено ~30с), circuit breaker на вызовах внешнего платёжного API. Команда хочет проверить, что это работает на деле, а не на бумаге.

## ТРЕБОВАНИЯ
Спроектируйте минимум два эксперимента по полному циклу:
- **Steady state** — измеримое нормальное поведение (какие SLI).
- **Гипотеза** — что должно произойти при отказе.
- **Внедряемый отказ** — конкретное событие.
- **Проверка** — как поймёте, держится ли steady state.
- **Blast radius** — как ограничите ущерб.

## ФОРМАТ РЕЗУЛЬТАТА
2 эксперимента, каждый по структуре: steady state → гипотеза → отказ → проверка → blast radius.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Steady state задан через измеримые SLI (например, error rate < 1%, p99 < 300 мс, throughput стабилен).
- [ ] Эксперимент 1: убить инстанс сервиса → гипотеза «LB перераспределит, steady state держится» → проверка по SLI.
- [ ] Эксперимент 2: убить primary-БД → гипотеза «failover за ~30с, error rate всплеснёт кратко и вернётся» → проверка времени восстановления и SLI.
- [ ] (Опц.) Эксперимент 3: задержка/отказ платёжного API → проверка circuit breaker/fallback (модуль 23).
- [ ] У каждого эксперимента есть гипотеза ДО внедрения отказа (не «сломаем и посмотрим»).
- [ ] Указан контроль blast radius (staging/малая доля, kill switch) и наблюдаемость (модуль 28).

## ПОДСКАЗКА
Цикл: steady state (измеримые SLI) → гипотеза («failover за 30с, SLI держится») → внедрить отказ (убить инстанс/БД, задержка API) → проверить по SLI → ограничить blast radius. Главное — гипотеза ДО поломки и наблюдаемость, иначе это вандализм, а не эксперимент.
