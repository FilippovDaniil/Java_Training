# Задача 07 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Спроектируйте контроль blast radius для проведения chaos-эксперимента в продакшене.

## ВХОДНЫЕ ДАННЫЕ
Команда созрела до chaos-экспериментов в проде (staging уже не выявляет нового — слишком отличается от прода по нагрузке и данным). Первый прод-эксперимент — внесение сетевой задержки в вызовы к сервису рекомендаций (некритичная функция). Руководство опасается задеть реальных пользователей.

## ТРЕБОВАНИЯ
- Опишите стратегию постепенного расширения радиуса (от минимального к большему).
- Предложите способы ограничить, кого/что затрагивает эксперимент.
- Спроектируйте kill switch (мгновенная остановка) и условия его срабатывания.
- Укажите роль наблюдаемости (как поймёте, что пора останавливать).
- Объясните, почему рекомендации — удачный выбор для первого прод-эксперимента.

## ФОРМАТ РЕЗУЛЬТАТА
Стратегия расширения → ограничение радиуса → kill switch + триггеры → роль наблюдаемости → почему рекомендации.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Постепенное расширение: малая доля трафика/пользователей → шире при подтверждённой безопасности.
- [ ] Ограничение радиуса: % пользователей, одна ячейка/регион, один инстанс (cells, модуль 24), некритичная функция.
- [ ] Kill switch: механизм мгновенно прекратить инъекцию отказа; триггеры — выход SLI за порог (error rate/p99) или ручная остановка.
- [ ] Наблюдаемость (модуль 28): метрики/алерты в реальном времени, чтобы видеть выход за пределы и принять решение.
- [ ] Обоснование: рекомендации некритичны → graceful degradation (без них продукт работает) → ограниченный ущерб при первом эксперименте.

## ПОДСКАЗКА
Контроль blast radius: начинать с малой доли трафика, ограничить радиус (% юзеров/ячейка/инстанс), иметь kill switch с автотриггером по SLI-порогу, вести эксперимент с включённой наблюдаемостью. Некритичная функция (рекомендации) — удачный первый выбор: её деградация не ломает ядро.
