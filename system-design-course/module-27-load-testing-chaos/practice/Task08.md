# Задача 08 — Модуль 27: Нагрузочное тестирование и Chaos Engineering

## ЗАДАНИЕ
Найдите ошибки в подходе к тестированию нагрузки и chaos, исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из плана тестирования:
1. «Прогнали load-test на целевой нагрузке — прошло, значит готовы к проду» (stress/soak не делали).
2. «Тестировали 200 виртуальными пользователями, каждый ждёт ответа — перегруза не нашли».
3. «Latency меряем по завершённым запросам, среднее = 30 мс, отлично».
4. «Тест бил только в один эндпоинт /health — быстро и просто».
5. «Chaos: зашли в прод и выключили случайный сервис — посмотрим, что будет».
6. «Chaos-эксперимент запустили, но дашборды и алерты ещё не настроены».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и корректный подход.
- Свяжите с понятиями модуля (виды тестов, closed/open, coordinated omission, реалистичность, гипотеза/steady state, наблюдаемость, blast radius).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → ошибка → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) Только load → нет точки отказа (stress) и поведения во времени (soak); добавить stress + soak.
- [ ] (2) Closed-модель маскирует перегруз → для проверки наплыва нужна open-модель (фиксированный rate).
- [ ] (3) Среднее по завершённым → coordinated omission занижает хвост; смотреть p95/p99 с учётом планового старта.
- [ ] (4) Нереалистичная нагрузка (/health ≠ реальный микс) → тестировать реальный профиль эндпоинтов.
- [ ] (5) Chaos без гипотезы/steady state/контроля → начать со steady state + гипотезы, ограничить blast radius + kill switch.
- [ ] (6) Chaos без наблюдаемости бессмыслен → сначала настроить метрики/алерты (модуль 28), потом эксперимент.

## ПОДСКАЗКА
Типовые ошибки: только load (нет stress/soak), closed для перегруза, coordinated omission (среднее по завершённым), нереалистичный эндпоинт, chaos без гипотезы/steady state, chaos без наблюдаемости и контроля blast radius. Каждая — прямое нарушение принципов модуля.
