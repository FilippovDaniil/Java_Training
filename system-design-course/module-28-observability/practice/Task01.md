# Задача 01 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Разграничьте monitoring и observability и определите, что из требований к какому относится.

## ВХОДНЫЕ ДАННЫЕ
Требования команды к эксплуатации сервиса:
- R1: «Хотим дашборд с CPU/памятью и алерт, если CPU > 80%».
- R2: «При инциденте хотим понять, почему именно эти 3% запросов от клиента X стали медленными — заранее такой вопрос не предвидели».
- R3: «Нужен график error rate за неделю».
- R4: «Хотим уметь по новому, ранее не заданному вопросу разобраться в проблеме, не выкатывая новый код».

## ТРЕБОВАНИЯ
- Для каждого требования укажите, это про monitoring или observability.
- Объясните разницу подходов (известные вопросы vs новые/неизвестные).
- Объясните, почему observability шире и почему в распределёнке важны unknown-unknowns.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «требование → monitoring/observability → почему» + вывод о разнице.

## КРИТЕРИИ ПРИЁМКИ
- [ ] R1, R3 → monitoring (известные заранее вопросы/пороги, дашборды по заданным метрикам).
- [ ] R2, R4 → observability (новые, заранее не заданные вопросы о внутреннем состоянии).
- [ ] Объяснено: monitoring отвечает на known-unknowns, observability позволяет задавать новые вопросы (unknown-unknowns).
- [ ] Отмечено, что observability шире (monitoring — её подмножество).
- [ ] Связь: в распределёнке большинство инцидентов — unknown-unknowns, их нельзя предусмотреть дашбордом.

## ПОДСКАЗКА
Monitoring — известные вопросы (пороги/дашборды), ловит known-unknowns. Observability — свойство задавать новые вопросы о внутреннем состоянии по внешним сигналам (unknown-unknowns), без выката кода. Observability шире, monitoring — подмножество. В распределёнке unknown-unknowns — норма.
