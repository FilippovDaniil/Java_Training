# Задача 02 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Определите, какой из трёх столпов observability отвечает на каждый вопрос, и спроектируйте их связку через correlation ID.

## ВХОДНЫЕ ДАННЫЕ
Микросервисная система (gateway → order → payment → notification). Возникают задачи:
- T1: «Когда вырос общий error rate и до какого уровня — за месяц?»
- T2: «Почему именно заказ #12345 завершился ошибкой — что произошло внутри?»
- T3: «На каком из 4 сервисов теряется latency у медленных запросов?»
- T4: «Как связать запись в метриках, лог и трейс одного и того же запроса?»

## ТРЕБОВАНИЯ
- Для T1–T3 укажите подходящий столп (metrics / logs / traces) и почему.
- Для T4 опишите роль correlation/trace ID.
- Отметьте сильную сторону и стоимость каждого столпа.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «задача → столп → почему» + объяснение correlation ID + столбец «стоимость/сила».

## КРИТЕРИИ ПРИЁМКИ
- [ ] T1 → metrics (агрегаты/тренды во времени, дёшево, долгое хранение).
- [ ] T2 → logs (детали конкретного события, максимум контекста; дорого по объёму → структурировать/сэмплировать).
- [ ] T3 → traces (путь запроса по span'ам, видно, где копится latency; ценны при fan-out — модуль 26).
- [ ] T4 → correlation/trace ID: единый идентификатор, протянутый через метрики/логи/трейсы одного запроса, связывает их.
- [ ] Для каждого столпа названа сильная сторона и относительная стоимость.

## ПОДСКАЗКА
Три столпа отвечают на разное: metrics — ЧТО/КОГДА (тренды/алерты, дёшево); logs — ДЕТАЛИ события (дорого, структурировать); traces — ГДЕ latency в распределёнке. Связывает их correlation/trace ID — один идентификатор через все три для одного запроса.
