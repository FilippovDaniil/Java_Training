# Задача 03 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Спроектируйте SLI и SLO для сервиса, выбрав показатели, важные пользователю.

## ВХОДНЫЕ ДАННЫЕ
API оформления заказов в e-commerce. Пользовательский путь: открыть товар → добавить в корзину → оформить заказ. Команда предлагает кандидатов в SLI: (а) CPU серверов, (б) доля успешных (не-5xx) запросов оформления, (в) доля запросов оформления быстрее 500 мс, (г) длина очереди в брокере, (д) количество логов в час.

## ТРЕБОВАНИЯ
- Выберите 2–4 SLI, важных пользователю, и отбросьте неподходящие с объяснением.
- Для выбранных SLI сформулируйте SLO (целевое значение за период).
- Объясните, почему внутренние метрики (CPU, очередь, логи) — плохие SLI.
- Сформулируйте SLI в форме «доля хороших событий ко всем».

## ФОРМАТ РЕЗУЛЬТАТА
Выбор SLI (с отбраковкой) → формулировка SLO → почему не внутренние метрики.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Выбраны (б) доступность оформления и (в) latency оформления как SLI (важны пользователю).
- [ ] Отброшены (а) CPU, (г) очередь, (д) логи — внутренние метрики, пользователю безразличны.
- [ ] SLO сформулированы измеримо за период (например, «99.9% оформлений успешны за 30 дней», «99% оформлений < 500 мс за 30 дней»).
- [ ] Объяснено: SLI отражает пользовательскую боль (успех/скорость), а не внутреннее состояние.
- [ ] SLI выражены как доля «хороших» событий ко всем; выбрано 2–4, а не десятки.

## ПОДСКАЗКА
SLI = то, что важно пользователю (успех и скорость на его пути), в форме «доля хороших ко всем». CPU/очередь/логи — внутреннее, пользователю всё равно → плохие SLI. SLO = целевое значение SLI за период. Брать 2–4 ключевых, не десятки.
