# Задача 04 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Рассчитайте error budget из SLO и опишите политику его использования.

## ВХОДНЫЕ ДАННЫЕ
Сервис с SLO доступности 99.9% за 30 дней. Объём — 200 млн запросов в месяц. Команда продукта хочет катить рискованные фичи каждую неделю; SRE опасаются за стабильность. Нужно объективное правило вместо спора.

## ТРЕБОВАНИЯ
- Посчитайте error budget: в процентах, в минутах простоя/месяц и в числе «допустимо ошибочных» запросов.
- Опишите политику: что можно, когда бюджет НЕ исчерпан, и что — когда исчерпан.
- Объясните, как error budget связывает надёжность и скорость разработки и снимает конфликт продукт↔SRE.
- Объясните, почему цель не 100% надёжности.

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт бюджета (3 формы) → политика использования → связь надёжность↔скорость → почему не 100%.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Error budget = 1 − 0.999 = 0.1%.
- [ ] В минутах: 0.1% от 30 дней ≈ 43 минуты допустимой недоступности/месяц.
- [ ] В запросах: 0.1% от 200 млн = 200 000 запросов могут «ошибиться» без нарушения SLO.
- [ ] Политика: бюджет есть → катим рискованные фичи/эксперименты/chaos; исчерпан → стоп фичам, только работа над надёжностью до восстановления.
- [ ] Объяснено: error budget — объективное правило, снимающее конфликт «фичи vs стабильность».
- [ ] Объяснено: 100% недостижимо и не нужно (пользователь не заметит, цена огромна) → SLO < 100%, бюджет > 0.

## ПОДСКАЗКА
Error budget = 1 − SLO. Для 99.9%: 0.1% → ~43 мин/месяц → 200k из 200M запросов. Политика: есть бюджет — катим риск; исчерпан — заморозка фич, чиним надёжность. Это объективно разрешает спор продукт↔SRE. 100% не цель: недостижимо/дорого, разница незаметна.
