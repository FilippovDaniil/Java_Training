# Задача 05 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Примените методики RED и USE: определите набор метрик для сервиса и для ресурса.

## ВХОДНЫЕ ДАННЫЕ
Система: сервис заказов (обрабатывает HTTP-запросы) поверх БД PostgreSQL (ресурс с пулом соединений и диском). Команда инструментирует систему «как придётся» и не уверена, что собирает нужное.

## ТРЕБОВАНИЯ
- Для сервиса заказов задайте метрики по RED (Rate/Errors/Duration) — что именно и как измеряется.
- Для БД (ресурса) задайте метрики по USE (Utilization/Saturation/Errors).
- Объясните, на какой вопрос отвечает каждая методика (сервис vs ресурс).
- Покажите, как эти метрики кормят поиск узких мест (модуль 26) и SLI.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица RED для сервиса + таблица USE для БД + объяснение «сервис vs ресурс» + связь с узкими местами/SLI.

## КРИТЕРИИ ПРИЁМКИ
- [ ] RED для сервиса: Rate (запросов/с), Errors (доля ошибок/5xx), Duration (latency p50/p95/p99).
- [ ] USE для БД: Utilization (загрузка CPU/диска/пула), Saturation (очередь к ресурсу/ожидание соединения), Errors (ошибки запросов/отказы соединений).
- [ ] Объяснено: RED — «как сервису» (что чувствует пользователь), USE — «как ресурсу» (CPU/диск/пул).
- [ ] Показано: RED→SLI (rate/errors/duration = доступность/latency), USE→поиск узких мест (saturation/utilization выявляют bottleneck, модуль 26).
- [ ] Отмечено, что это готовые чек-листы: каждому сервису — RED, каждому ресурсу — USE.

## ПОДСКАЗКА
RED (сервисы): Rate, Errors, Duration — то, что чувствует пользователь → кормит SLI. USE (ресурсы): Utilization, Saturation, Errors — состояние CPU/диска/пула → выявляет узкое место (модуль 26). Чек-листы: сервису — RED, ресурсу — USE.
