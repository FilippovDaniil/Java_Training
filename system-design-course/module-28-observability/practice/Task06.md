# Задача 06 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Используйте распределённый трейсинг, чтобы локализовать, где теряется latency в цепочке сервисов.

## ВХОДНЫЕ ДАННЫЕ
Запрос «оформить заказ» проходит: gateway → order-service → (параллельно: payment-service, inventory-service) → notification-service. Пользователи жалуются на медленное оформление (p99 ≈ 2 с). Метрики показывают высокий общий p99, но не говорят, какой сервис виноват. Трейс одного медленного запроса (span'ы):

```
gateway            [■ 8мс]
 order-service     [■■ 30мс]
   payment-service [■■■■■■■■■■■■■■■■ 1850мс]
   inventory-svc   [■ 25мс]
 notification-svc  [■ 15мс]
```

## ТРЕБОВАНИЯ
- По трейсу определите, где копится latency.
- Объясните, почему метрики «общий p99 высокий» этого не показывали, а трейс — да.
- Объясните роль span'ов и trace ID.
- Свяжите с fan-out (модуль 26): почему параллельный вызов ждёт самого медленного.
- Дайте следующий шаг расследования по payment-service.

## ФОРМАТ РЕЗУЛЬТАТА
Локализация latency → почему трейс, а не метрики → роль span/trace ID → связь fan-out → следующий шаг.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Latency локализована в payment-service (1850 из ~1900 мс запроса).
- [ ] Объяснено: агрегированные метрики дают «общий p99», но не разбивку по сервисам; трейс показывает путь и время каждого span'а.
- [ ] Роль span (отрезок работы сервиса/операции) и trace ID (связывает все span'ы одного запроса).
- [ ] Fan-out: order ждёт max(payment, inventory) — параллельный вызов завершается по самому медленному (payment), модуль 26.
- [ ] Следующий шаг: углубиться в payment-service (его внутренние span'ы/логи по trace ID — внешний API? БД? блокировка?).

## ПОДСКАЗКА
Трейс показывает время каждого span'а → видно, что почти вся latency в payment-service (1850 мс). Метрики дают общий p99, но не разбивку по сервисам — для этого нужны traces. Trace ID связывает span'ы. Fan-out: параллельный вызов ждёт самого медленного. Дальше — внутрь payment по trace ID.
