# Задача 08 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Найдите ошибки в стратегии observability и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из дизайн-документа:
1. «SLO у нас — CPU серверов не выше 70%».
2. «Цель — 100% доступности, никаких ошибок».
3. «В метрики добавим label с user_id и request_id — удобно фильтровать».
4. «SLA клиенту обещали 99.99%, внутренний SLO тоже 99.99% — пусть совпадают».
5. «Алертим на каждую аномалию любой метрики, чтобы ничего не упустить».
6. «Логи пишем простым текстом в stdout, потом grep'ом разберёмся».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите ошибку и корректное решение.
- Свяжите с понятиями модуля (SLI про пользователя, 100% не цель, high-cardinality, SLA<SLO, alert fatigue, структурированные логи).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → ошибка → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) SLO на CPU — внутренняя метрика, не про пользователя → SLO на доступность/latency (пользовательский SLI).
- [ ] (2) 100% недостижимо/не нужно → SLO < 100%, error budget > 0.
- [ ] (3) user_id/request_id как label метрики = high-cardinality → взрыв временных рядов; идентификаторы — в логи/трейсы, не в метки метрик.
- [ ] (4) SLA должен быть слабее SLO (буфер), иначе нарушение SLO = сразу нарушение контракта → сделать SLA, например, 99.9% при SLO 99.99%.
- [ ] (5) Алерт на всё → alert fatigue → пропуск реального; алертить на симптомы/SLO-burn.
- [ ] (6) Неструктурированный текст нельзя надёжно запрашивать/агрегировать → структурированные логи (JSON, поля), correlation ID.

## ПОДСКАЗКА
Типовые ошибки: SLO на внутренней метрике (а не на пользовательском SLI), цель 100%, high-cardinality в метках метрик, SLA не слабее SLO, алерт на всё (fatigue), неструктурированные логи. Каждая — прямое нарушение принципов модуля.
