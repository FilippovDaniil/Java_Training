# Задача 09 — Модуль 28: Observability для дизайна

## ЗАДАНИЕ
Разграничьте SLI, SLO, SLA и error budget на одном сервисе и покажите, как они связывают надёжность и скорость разработки.

## ВХОДНЫЕ ДАННЫЕ
Платёжный сервис. Бизнес обещает клиентам определённый уровень в договоре; команда хочет внутренний таргет строже договора и объективное правило, когда замораживать фичи. Термины SLI/SLO/SLA/error budget в команде путают и используют как синонимы.

## ТРЕБОВАНИЯ
- Дайте чёткое определение каждого термина на примере этого сервиса (с конкретными числами).
- Покажите правильное соотношение SLO и SLA (что строже) и зачем буфер.
- Выведите error budget из SLO и опишите политику фич.
- Объясните, как вся связка превращает надёжность в управляемый ресурс.

## ФОРМАТ РЕЗУЛЬТАТА
Определения SLI/SLO/SLA (с числами) → соотношение SLO↔SLA → error budget + политика → надёжность как ресурс.

## КРИТЕРИИ ПРИЁМКИ
- [ ] SLI — измеримый показатель (например, «доля успешных платежей»); SLO — цель для него («99.95% за 30 дней»); SLA — контракт с клиентом + последствия («99.9% или компенсация»).
- [ ] Показано: SLA слабее SLO (буфер), чтобы нарушить внутреннюю цель и среагировать ДО нарушения контракта.
- [ ] Error budget = 1 − SLO (для 99.95% → 0.05%); политика: есть бюджет → катим фичи/риск, исчерпан → заморозка фич, работа над надёжностью.
- [ ] Объяснено: связка делает надёжность измеримым ресурсом, которым «платят» за скорость; снимает конфликт продукт↔SRE.
- [ ] Термины не используются как синонимы; роли (число / цель / контракт / бюджет) различены.

## ПОДСКАЗКА
SLI = число (что меряем), SLO = цель (внутренний таргет), SLA = контракт с последствиями (внешнее обещание). SLA слабее SLO (буфер → реагируем до нарушения договора). Error budget = 1−SLO управляет балансом «фичи vs стабильность»: есть бюджет — катим, исчерпан — чиним.
