# Модуль 28 — Observability для дизайна (SLI/SLO, error budget)

## Зачем этот модуль

Часть 6 дала масштаб (24), DR (25), поиск узких мест (26) и проверку под нагрузкой/отказами (27). У всего этого общий фундамент: **нельзя эксплуатировать, оптимизировать или тестировать то, что не видишь**. Узкое место находят по метрикам (26), steady state в chaos измеряют по SLI (27), а инцидент в проде диагностируют по логам и трейсам. Этот модуль — про **observability**: как спроектировать систему так, чтобы её внутреннее состояние было видно снаружи, и как через SLI/SLO/error budget управлять надёжностью на языке бизнеса.

Зачем это senior'у. Observability — не «прикрутить Grafana потом», а **проектное решение**: что инструментировать, какие SLI выбрать, какие SLO пообещать, как алертить. Senior формулирует **SLI** (что измеряем), **SLO** (целевое значение), понимает **error budget** (он связывает надёжность и скорость разработки) и проектирует алертинг на симптомы, а не на причины. Это завершающий модуль Части 6: он замыкает цикл «спроектировал → нагрузил/сломал контролируемо → увидел → улучшил».

Он отвечает на «как видеть, что система здорова, и как управлять надёжностью измеримо» — три столпа (metrics/logs/traces), RED/USE, SLI/SLO/SLA и error budget.

## 1. Зачем observability и чем отличается от мониторинга

| | Monitoring | Observability |
|--|-----------|---------------|
| Вопросы | известные заранее («CPU > 80%?») | новые, неизвестные заранее («почему именно эти запросы медленные?») |
| Подход | дашборды по заданным метрикам | исследование внутреннего состояния по внешним сигналам |
| Хорош для | known-unknowns (ожидаемые проблемы) | unknown-unknowns (неожиданные, в распределёнке — норма) |

> **Мониторинг** отвечает на вопросы, которые вы задали заранее (пороги, дашборды) — он ловит **известные** проблемы. **Observability** — это свойство системы, позволяющее **задавать новые вопросы** о её внутреннем состоянии по внешним сигналам, не выкатывая новый код: в распределёнке большинство инцидентов — это unknown-unknowns, которые нельзя предусмотреть дашбордом. Observability шире мониторинга: мониторинг — подмножество. Для дизайна это значит: инструментировать систему так, чтобы можно было разобраться в *новой* проблеме постфактум.

## 2. Три столпа observability

| Столп | Что это | Сильная сторона | Стоимость |
|-------|---------|-----------------|-----------|
| **Metrics** | числовые агрегаты во времени (счётчики, гистограммы) | дёшево, тренды, алерты, долгое хранение | низкая (агрегат) |
| **Logs** | дискретные события с деталями | максимум контекста о конкретном событии | высокая (объём) |
| **Traces** | путь одного запроса через сервисы (span'ы) | где именно тратится latency в распределёнке | средняя–высокая |

```
Metrics:  «error rate вырос до 5% в 14:03»          (ЧТО и КОГДА, дёшево)
Logs:     «14:03 order=123 → NullPointer в PaymentSvc» (ДЕТАЛИ конкретного события)
Traces:   «запрос: gateway 5мс → order 10мс → payment 480мс → ...» (ГДЕ latency)
```

> Три столпа дополняют друг друга и отвечают на разные вопросы. **Metrics** дёшевы и хороши для трендов/алертов («что-то не так и когда») — с них начинают. **Logs** дают детали конкретного события («почему именно этот запрос упал»), но дороги в объёме — структурируют (JSON) и сэмплируют. **Traces** показывают путь запроса через сервисы и где копится latency — незаменимы в микросервисах/fan-out (модуль 26). Связывает их **correlation/trace ID**: один идентификатор, протянутый через метрики, логи и трейсы запроса.

**RED и USE** — какие метрики собирать:
- **RED** (для сервисов/запросов): **R**ate (запросов/с), **E**rrors (доля ошибок), **D**uration (latency, перцентили).
- **USE** (для ресурсов): **U**tilization (загрузка), **S**aturation (очередь/насыщение), **E**rrors (ошибки ресурса).

> RED отвечает на «как сервису» (rate/errors/duration — то, что чувствует пользователь), USE — на «как ресурсу» (utilization/saturation/errors — CPU/память/диск/пул). Это готовые чек-листы: для каждого сервиса — RED, для каждого ресурса — USE. Они напрямую кормят поиск узких мест (модуль 26) и SLI.

## 3. SLI — что измеряем

**SLI (Service Level Indicator)** — измеримый показатель качества сервиса с точки зрения пользователя.

| Типичный SLI | Определение |
|--------------|-------------|
| Доступность | доля успешных запросов (не-5xx) от всех |
| Latency | доля запросов быстрее порога (например, < 300 мс) |
| Error rate | доля ошибочных запросов |
| Качество/свежесть | доля ответов из актуальных данных |

> SLI должен отражать то, что **важно пользователю**, а не «всё, что можно измерить». Хороший SLI обычно формулируется как доля «хороших» событий ко всем: «доля запросов, обслуженных за < 300 мс и без ошибки». Частая ошибка — делать SLI из внутренней метрики (CPU, длина очереди): пользователю всё равно, какой у вас CPU, ему важны успех и скорость ответа. Выбирайте 2–4 ключевых SLI на пользовательский путь, а не десятки.

## 4. SLO и SLA

| Термин | Что это | Пример |
|--------|---------|--------|
| **SLI** | измеряемый показатель | «доля запросов < 300 мс» |
| **SLO** | целевое значение SLI (внутренняя цель) | «99.9% запросов < 300 мс за 30 дней» |
| **SLA** | контракт с клиентом + последствия за нарушение | «99.5% или возврат денег» |

```
SLI  — факт измерения (что есть)
SLO  — цель (к чему стремимся; внутреннее обещание)
SLA  — юридический контракт (внешнее обещание + штраф)
Обычно: SLA слабее SLO (буфер): SLO 99.9%, SLA 99.5% — чтобы нарушить SLO раньше, чем SLA.
```

> Три уровня легко спутать: **SLI** — это число (измеряем), **SLO** — цель для этого числа (внутренний таргет надёжности), **SLA** — внешний контракт с финансовыми/юридическими последствиями. Ключевое правило: **SLA должен быть слабее SLO** (буфер), чтобы команда нарушала внутреннюю цель и реагировала *до* нарушения контракта с клиентом. SLO — главный рабочий инструмент (по нему живёт команда); SLA — то, что обещано наружу.

## 5. Error budget

**Error budget = 1 − SLO** — допустимая доля «плохих» событий за период.

```
SLO = 99.9% за 30 дней  →  error budget = 0.1%
0.1% от 30 дней ≈ 43 минуты допустимой недоступности в месяц.
0.1% от 100M запросов = 100 000 запросов могут «ошибиться» без нарушения SLO.
```

**Зачем он нужен — связывает надёжность и скорость разработки:**
```
Бюджет НЕ исчерпан  → есть запас → можно катить рискованные фичи/эксперименты/chaos.
Бюджет исчерпан     → стоп фичам → команда работает только над надёжностью,
                       пока не восстановит бюджет.
```

> Error budget — самая мощная идея модуля: он превращает надёжность в **измеримый ресурс**, которым можно «платить» за скорость. 100% надёжности недостижимы и не нужны (пользователь не заметит разницу 99.99% и 100%, а цена огромна — модуль 24); SLO задаёт **приемлемый** уровень, а error budget — сколько «права на ошибку» осталось. Это снимает вечный конфликт «продукт хочет фичи vs SRE хочет стабильность»: пока бюджет есть — катим; исчерпан — замораживаем фичи и чиним. Объективное правило вместо спора. Связь с chaos (модуль 27): эксперименты «тратят» бюджет осознанно.

## 6. Алертинг

| Принцип | Почему |
|---------|--------|
| Алертить на **симптомы**, не на причины | пользователю важен симптом (медленно/ошибки), причин много и они меняются |
| Алертить на **SLO-burn** (расход бюджета) | алерт срабатывает, когда страдает то, что обещано пользователю |
| **Burn rate** алерты | быстрый расход бюджета → срочный алерт; медленный → тикет |
| Бороться с **alert fatigue** | шумные алерты на всё → их игнорируют → пропускают важное |

```
❌ На причину: «CPU узла 92%»     (ну и что? пользователь, может, не страдает)
✅ На симптом: «p99 > 300мс 5 мин» / «error budget burn rate ×14» (пользователь страдает)
```

> Зрелый алертинг — **на симптомы и расход бюджета**, а не на каждую внутреннюю метрику. Алерт на «CPU 90%» бесполезен, если пользователь не страдает (а если страдает — увидим по SLI). Алертят на нарушение SLO / скорость прожигания error budget (**burn rate**): резкий прожиг → буди дежурного, медленный → заведи тикет. Главный враг — **alert fatigue**: если алертов много и большинство ложные, команда перестаёт реагировать и пропускает реальный инцидент. Меньше алертов, но значимых.

## 7. Ловушки и эксплуатация

| Ловушка | Суть |
|---------|------|
| **High-cardinality labels** | метка с огромным числом значений (user_id, request_id в метриках) → взрыв числа временных рядов, дорого/падает; высокая кардинальность — для логов/трейсов, не для метрик |
| SLO не про пользователя | SLO на CPU/внутреннее → не отражает боль пользователя |
| Слишком много SLI | десятки показателей вместо 2–4 ключевых на путь |
| Логи без структуры | неструктурированный текст → нельзя запросить/агрегировать |
| Алерт на всё | alert fatigue → реальные инциденты тонут |
| 100% надёжность как цель | недостижимо и не нужно; SLO < 100%, error budget > 0 |

> Практические грабли: **high-cardinality** в метриках (user_id как label) взрывает хранилище — идентификаторы место в логах/трейсах, не в метках метрик; SLO должен мерить пользовательскую боль, а не CPU; SLI держать в количестве 2–4 на ключевой путь; логи — структурированными (JSON), чтобы их можно было запрашивать. И мировоззренческое: цель — не 100%, а **достаточная** надёжность с ненулевым error budget.

## ⚠️ Подводные камни

- **«Добавим observability потом».** Инструментирование — проектное решение; ретроспективно дороже и хуже.
- **Путать monitoring и observability.** Дашборды по известным метрикам не дают отвечать на новые вопросы (unknown-unknowns).
- **SLI из внутренних метрик.** Пользователю важны успех/скорость, не CPU/очередь; SLI — про пользователя.
- **Путать SLI/SLO/SLA.** SLI — число, SLO — цель, SLA — контракт; SLA должен быть слабее SLO (буфер).
- **Цель 100% надёжности.** Недостижимо и не нужно; SLO < 100%, error budget > 0.
- **Не использовать error budget.** Без него вечный конфликт «фичи vs стабильность»; бюджет даёт объективное правило.
- **Алертить на причины.** Шумно и бесполезно; алертить на симптомы/SLO-burn.
- **Alert fatigue.** Алерты на всё → игнорируют → пропускают реальное.
- **High-cardinality в метриках.** user_id/request_id как label → взрыв рядов; их место в логах/трейсах.
- **Неструктурированные логи.** Текст без полей нельзя агрегировать/запрашивать.
- **Нет correlation/trace ID.** Без сквозного ID метрики/логи/трейсы запроса не связать.
