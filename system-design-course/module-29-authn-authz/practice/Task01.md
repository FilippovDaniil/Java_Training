# Задача 01 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Разграничьте аутентификацию (AuthN) и авторизацию (AuthZ) на наборе ситуаций и определите HTTP-ответ.

## ВХОДНЫЕ ДАННЫЕ
Ситуации в API:
- S1: запрос без токена к защищённому эндпоинту.
- S2: валидный токен обычного пользователя, который пытается удалить чужой заказ.
- S3: истёкший токен.
- S4: валидный токен admin'а, удаляющий любой заказ.
- S5: валидный токен пользователя, читающий свой профиль.

## ТРЕБОВАНИЯ
- Для каждой ситуации укажите, это вопрос AuthN или AuthZ (или обоих).
- Определите HTTP-ответ (401 / 403 / 200).
- Сформулируйте правило: когда 401, когда 403.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «ситуация → AuthN/AuthZ → HTTP-код → почему» + правило 401 vs 403.

## КРИТЕРИИ ПРИЁМКИ
- [ ] S1 → AuthN провал (нет личности) → 401.
- [ ] S2 → AuthN ок, AuthZ провал (нет прав на чужой объект) → 403.
- [ ] S3 → AuthN провал (невалидная/истёкшая личность) → 401.
- [ ] S4 → AuthN ок + AuthZ ок (роль admin) → 200.
- [ ] S5 → AuthN ок + AuthZ ок (свой объект) → 200.
- [ ] Правило: нет/невалидная личность → 401; личность есть, прав нет → 403.

## ПОДСКАЗКА
AuthN — кто ты (личность установлена?), AuthZ — что можно (права на действие/объект?). Порядок: сначала AuthN, потом AuthZ. Нет/невалидная личность → 401; личность есть, но прав нет → 403.
