# Задача 02 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Выберите session-based или token-based аутентификацию под кейс и обоснуйте trade-off.

## ВХОДНЫЕ ДАННЫЕ
Два кейса:
- A: классическое серверное веб-приложение (монолит), один сервер сейчас, нужен мгновенный «выход со всех устройств» (отзыв сессии), низкая сложность.
- B: микросервисная система с stateless-сервисами за gateway, API для мобильного приложения и партнёрских интеграций, горизонтальное масштабирование.

## ТРЕБОВАНИЯ
- Для каждого кейса выберите session-based или token-based (JWT).
- Обоснуйте через ключевой trade-off (statelessness ↔ мгновенный отзыв, нужно ли общее хранилище сессий).
- Для выбора token в кейсе B опишите, как смягчить проблему отзыва.
- Свяжите выбор с масштабируемостью (модуль 24).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «кейс → выбор → обоснование» + смягчение отзыва для token + связь с масштабом.

## КРИТЕРИИ ПРИЁМКИ
- [ ] A → session-based (мгновенный отзыв удалением сессии, простота; один сервер не требует распределённого хранилища).
- [ ] B → token-based/JWT (stateless-сервисы, любой инстанс проверяет подписью без хранилища, удобно для API/мобайла).
- [ ] Назван trade-off: session требует общего session store при масштабе; token самодостаточен, но отзыв сложен.
- [ ] Для B смягчение отзыва: короткий access token + refresh token (отзывается на auth-сервере) и/или блок-лист.
- [ ] Связь с модулем 24: token → stateless горизонтальный масштаб; session → нужно общее хранилище/sticky.

## ПОДСКАЗКА
Session: состояние на сервере, мгновенный отзыв, но нужен общий store при масштабе. Token (JWT): stateless (проверка подписью без БД), удобно для микросервисов/API, но отзыв сложен → короткий access + refresh. A (монолит, нужен отзыв) → session; B (микросервисы, масштаб, API) → token.
