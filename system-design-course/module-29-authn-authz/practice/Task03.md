# Задача 03 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Спроектируйте JWT-флоу для API: содержимое токена, подпись, проверка, обновление и отзыв.

## ВХОДНЫЕ ДАННЫЕ
Stateless API за gateway, обслуживает несколько микросервисов. Требования: сервисы проверяют токен без обращения к auth-серверу; нужен механизм «выход» и блокировка скомпрометированного токена; access-токены не должны жить долго.

## ТРЕБОВАНИЯ
- Опишите, что положить в claims, а что — нельзя.
- Выберите способ подписи (HMAC vs RSA/EC) и обоснуйте для микросервисов.
- Опишите проверку токена сервисом (без БД).
- Спроектируйте access + refresh токены (сроки жизни, роли) и отзыв.

## ФОРМАТ РЕЗУЛЬТАТА
Claims (что/чего нельзя) → подпись → проверка → access/refresh + отзыв.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Claims: sub (id пользователя), роли/scopes, exp/iat, iss/aud; НЕ класть пароли/секреты/чувствительные PII (payload читаем).
- [ ] Подпись: RSA/EC (асимметричная) для микросервисов — сервисы проверяют публичным ключом, не зная приватного (HMAC требует общего секрета у всех).
- [ ] Проверка: валидация подписи + exp/iss/aud локально, без обращения к auth-серверу.
- [ ] Access token короткоживущий (минуты) + refresh token долгоживущий (обновляет access; отзывается на auth-сервере).
- [ ] Отзыв: через refresh (revoke на сервере) и/или короткий TTL access + блок-лист по jti для скомпрометированных.
- [ ] Отмечено: подпись ≠ шифрование (payload читаем).

## ПОДСКАЗКА
В claims — id/роли/exp/iss/aud, но НЕ секреты (payload читаем). Для микросервисов — асимметричная подпись (RSA/EC): проверка публичным ключом без общего секрета. Проверка локальна (подпись+exp). Отзыв — короткий access + refresh (revoke на сервере) + блок-лист по jti.
