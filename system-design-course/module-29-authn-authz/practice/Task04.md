# Задача 04 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Выберите OAuth2 grant type под каждый сценарий и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Сценарии интеграции:
- C1: одностраничное приложение (SPA) в браузере, пользователь логинится и работает с API.
- C2: мобильное приложение, пользователь логинится.
- C3: бэкенд-сервис планировщика, который дёргает внутренний API без участия пользователя (machine-to-machine).
- C4: «Войти через Google» в вебе для установления личности пользователя.

## ТРЕБОВАНИЯ
- Для C1–C3 выберите grant type (authorization code + PKCE / client credentials).
- Для C4 укажите, что это OIDC, а не «голый» OAuth2, и почему.
- Объясните, зачем PKCE для публичных клиентов (SPA/mobile).
- Объясните, почему client credentials не подходит, где есть пользователь.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «сценарий → grant/протокол → обоснование» + роль PKCE.

## КРИТЕРИИ ПРИЁМКИ
- [ ] C1 (SPA) → authorization code + PKCE (публичный клиент без секрета).
- [ ] C2 (mobile) → authorization code + PKCE (публичный клиент).
- [ ] C3 (M2M) → client credentials (нет пользователя, сервис↔сервис).
- [ ] C4 → OIDC (нужен id_token = кто пользователь), поверх authorization code; не «голый» OAuth2.
- [ ] PKCE: защищает публичных клиентов (не могут хранить секрет) от перехвата authorization code.
- [ ] Client credentials не подходит для C1/C2 — там есть пользователь, нужен его вход/согласие.

## ПОДСКАЗКА
Интерактивный вход пользователя (SPA/mobile/веб) → authorization code; для публичных клиентов (нет секрета) обязателен PKCE (защита code от перехвата). Сервис↔сервис без пользователя → client credentials. «Войти через…» (узнать личность) → OIDC (id_token), не голый OAuth2.
