# Задача 05 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Разграничьте OAuth2 и OIDC и определите, что использовать для каждой задачи.

## ВХОДНЫЕ ДАННЫЕ
Задачи продукта:
- T1: приложение должно узнать, кто залогинившийся пользователь (имя, email, id) для персонализации.
- T2: приложение должно получить доступ к календарю пользователя в стороннем сервисе от его имени.
- T3: команда говорит «используем OAuth2 access token, чтобы понять, кто пользователь — там же есть sub».

## ТРЕБОВАНИЯ
- Для T1 и T2 укажите, что нужно (OIDC / OAuth2) и какой токен (id_token / access_token).
- Разберите T3: почему использовать access token для установления личности — ошибка.
- Сформулируйте принцип «OAuth2 = доступ, OIDC = личность».

## ФОРМАТ РЕЗУЛЬТАТА
T1/T2 → протокол + токен → обоснование; разбор ошибки T3; принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] T1 → OIDC, id_token (предназначен для установления личности: кто пользователь).
- [ ] T2 → OAuth2, access_token (делегированный доступ к ресурсу от имени пользователя).
- [ ] T3 разобрано: access token — для доступа к ресурсам, не для идентификации; его аудитория/назначение иные, полагаться на его claims для «кто пользователь» неверно (для этого есть id_token/OIDC).
- [ ] Сформулирован принцип: OAuth2 — про доступ (что можно), OIDC — про личность (кто это).
- [ ] (Опц.) Отмечено, что OIDC построен поверх OAuth2.

## ПОДСКАЗКА
OAuth2 даёт access token для ДОСТУПА к ресурсам (что можно). OIDC поверх OAuth2 даёт id_token для ЛИЧНОСТИ (кто пользователь). Узнать «кто» → OIDC/id_token; доступ к чужому ресурсу от имени юзера → OAuth2/access_token. Access token не предназначен для идентификации.
