# Задача 06 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Спроектируйте SSO (Single Sign-On) для набора приложений компании.

## ВХОДНЫЕ ДАННЫЕ
Компания имеет 5 внутренних приложений (веб + мобильное). Сейчас в каждом своя система логина — пользователи заводят по 5 паролей, при увольнении доступ отзывают вручную в каждом. Нужно: один вход на все приложения, единая точка управления учётками, быстрый отзыв доступа.

## ТРЕБОВАНИЯ
- Опишите архитектуру SSO: роль Identity Provider (IdP) и приложений (relying parties).
- Выберите протокол (OIDC/SAML) и обоснуйте.
- Опишите флоу: пользователь заходит в приложение, как происходит единый вход.
- Покажите, как SSO решает проблемы из условия (5 паролей, ручной отзыв).

## ФОРМАТ РЕЗУЛЬТАТА
Архитектура (IdP + RP) → протокол → флоу единого входа → решение проблем.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Центральный IdP аутентифицирует пользователя; приложения (relying parties) доверяют токенам IdP.
- [ ] Протокол: OIDC (современный, для веб/мобайл) или SAML (легаси-enterprise); выбор обоснован.
- [ ] Флоу: приложение редиректит на IdP → пользователь логинится один раз → IdP выдаёт токен → приложение доверяет; повторный вход в другое приложение — без повторного логина (сессия у IdP).
- [ ] Решение: один набор учётных данных вместо пяти; централизованное управление и мгновенный отзыв в IdP отключает доступ ко всем приложениям.
- [ ] (Опц.) Упомянута связь с MFA на уровне IdP (единая точка усиления).

## ПОДСКАЗКА
SSO: центральный IdP аутентифицирует пользователя один раз, приложения (relying parties) доверяют его токенам (OIDC/SAML). Флоу — редирект на IdP, единый логин, токен приложениям. Решает: один пароль вместо пяти, централизованное управление + мгновенный отзыв во всех системах разом.
