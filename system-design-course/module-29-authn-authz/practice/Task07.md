# Задача 07 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Выберите модель авторизации (RBAC / ABAC / ReBAC) под каждое требование и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Требования к доступу в разных системах:
- A: внутренняя админка — есть роли admin/editor/viewer с фиксированными правами.
- B: файловое хранилище — пользователь видит свои файлы и те, что ему расшарили; владелец может дать доступ другим.
- C: корпоративная система — доступ зависит от отдела, уровня допуска, времени суток и гео.
- D: команда хочет выразить «пользователь редактирует документ, если он его владелец» через роли.

## ТРЕБОВАНИЯ
- Для A–C выберите модель (RBAC/ABAC/ReBAC) и обоснуйте.
- Разберите D: почему «доступ к своим объектам» плохо ложится на чистый RBAC.
- Кратко опишите, чем модели отличаются по основанию решения.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «требование → модель → обоснование» + разбор D.

## КРИТЕРИИ ПРИЁМКИ
- [ ] A → RBAC (фиксированные роли → права; просто и достаточно).
- [ ] B → ReBAC (доступ через отношения: владелец/расшарено; Google Zanzibar-стиль).
- [ ] C → ABAC (решение по атрибутам: отдел/допуск/время/гео).
- [ ] D разобрано: «свой объект» — это отношение (владение), роли его не выражают (роль не привязана к конкретному объекту) → нужен ReBAC (или проверка владения в коде).
- [ ] Основания различены: RBAC — роли, ABAC — атрибуты, ReBAC — отношения.

## ПОДСКАЗКА
RBAC — по ролям (просто, фиксированные права). ABAC — по атрибутам (отдел/время/гео, гибко). ReBAC — по отношениям (владелец/расшарено, Zanzibar). «Доступ к своим объектам» — это отношение владения, ролями не выражается → ReBAC. Фиксированные роли → RBAC; контекстные политики → ABAC.
