# Задача 08 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Спроектируйте проброс identity и проверку авторизации в микросервисной системе по принципу zero trust.

## ВХОДНЫЕ ДАННЫЕ
Система: gateway → order-service → (вызывает) payment-service и inventory-service. Сейчас: gateway проверяет токен, а внутренние сервисы «доверяют любому запросу из внутренней сети» (никакой проверки между сервисами). Тонкая проверка прав на конкретный заказ нигде не делается.

## ТРЕБОВАНИЯ
- Распределите проверки: что делает gateway, что — каждый сервис.
- Опишите проброс identity по цепочке вызовов (token propagation).
- Объясните принцип zero trust и почему «доверие внутренней сети» опасно.
- Укажите, где проверять доступ к конкретному объекту (тонкая authZ).

## ФОРМАТ РЕЗУЛЬТАТА
Распределение проверок (gateway/сервис) → проброс identity → zero trust → где тонкая authZ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Gateway: аутентификация (валиден ли токен) + грубая авторизация (роль/scope).
- [ ] Каждый сервис проверяет identity (не доверяет слепо) и тонкую авторизацию на свой ресурс.
- [ ] Token propagation: identity пользователя пробрасывается по цепочке вызовов (токен/подписанные claims), каждый сервис валидирует.
- [ ] Zero trust: «из внутренней сети» ≠ доверенный; скомпрометированный сервис иначе ходит везде → проверять на каждом хопе.
- [ ] Тонкая авторизация (доступ к конкретному заказу) делается в сервисе, который владеет объектом (связь с BOLA, модуль 30).
- [ ] Отмечено: аутентификацию удобно централизовать на gateway, но объектную авторизацию — нет.

## ПОДСКАЗКА
Gateway — AuthN + грубая authZ (роль/scope). Сервис — проверка identity + тонкая (объектная) authZ. Identity пробрасывается по цепочке (token propagation), каждый сервис валидирует сам. Zero trust: не доверять «внутренней сети» — иначе один взломанный сервис ходит везде. Доступ к конкретному объекту — в сервисе-владельце.
