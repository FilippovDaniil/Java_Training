# Задача 09 — Модуль 29: AuthN/AuthZ в дизайне

## ЗАДАНИЕ
Найдите ошибки в дизайне аутентификации/авторизации и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из дизайн-документа:
1. «JWT храним в localStorage браузера — удобно доставать из JS».
2. «В payload JWT кладём пароль пользователя и номер карты — токен же подписан».
3. «Проверку прав делаем на фронтенде: прячем кнопку 'удалить', если не админ».
4. «Access token живёт 30 дней, refresh не нужен — меньше возни».
5. «SPA использует authorization code без PKCE».
6. «Внутренние сервисы не проверяют токен — раз пришло из нашей сети, значит свои».
7. «Секрет для подписи JWT захардкожен в коде и в гите».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите уязвимость и корректное решение.
- Свяжите с понятиями модуля (XSS/localStorage, payload читаем, authZ на сервере, отзыв, PKCE, zero trust, secrets).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → уязвимость → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) localStorage уязвим к XSS → httpOnly-cookie (с защитой от CSRF) или короткий токен в памяти.
- [ ] (2) Payload JWT читаем (Base64, не шифр) → не класть пароль/карту/секреты; подпись защищает целостность, не конфиденциальность.
- [ ] (3) AuthZ только на фронте обходится → проверять права на сервере всегда (фронт — лишь UX).
- [ ] (4) 30-дневный access нельзя отозвать → короткий access + refresh token (отзываемый).
- [ ] (5) SPA — публичный клиент → authorization code обязателен с PKCE (защита code от перехвата).
- [ ] (6) Доверие внутренней сети нарушает zero trust → каждый сервис проверяет identity/токен.
- [ ] (7) Секрет в коде/гите → в секрет-хранилище (vault), ротация; асимметричная подпись снимает общий секрет.

## ПОДСКАЗКА
Классические ошибки: JWT в localStorage (XSS), секреты в payload (он читаем), authZ на фронте (обходится), вечный access без refresh (нет отзыва), SPA без PKCE, доверие внутренней сети (нарушение zero trust), секрет в коде/гите. Каждая — прямое нарушение принципов модуля.
