# Задача 10 — Модуль 29 (мини-проект): Дизайн AuthN/AuthZ системы

## ЗАДАНИЕ
Спроектируйте полную схему аутентификации и авторизации для системы. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Система на выбор (зафиксируйте одну): (а) маркетплейс (покупатели/продавцы/админы, мобильное + веб + партнёрский API); (б) SaaS B2B с организациями и ролями; (в) соцсеть с приватностью и расшариванием. Архитектура микросервисная, за gateway.

## ТРЕБОВАНИЯ
Дизайн должен покрывать:
1. **AuthN** — session vs token (обоснование под масштаб), формат токена (JWT: claims/подпись).
2. **Протоколы** — OAuth2 grant types под клиентов (веб/мобайл/M2M), OIDC для логина, SSO/IdP если нужно.
3. **Токены** — access/refresh, сроки, отзыв.
4. **Модель авторизации** — RBAC/ABAC/ReBAC под правила доступа системы (обоснование).
5. **Размещение проверок** — что на gateway (AuthN/грубая authZ), что в сервисах (тонкая/объектная authZ).
6. **Проброс identity** — token propagation + zero trust между сервисами.
7. **Секреты и хранение** — где ключи подписи, как хранить токены на клиенте.

## ФОРМАТ РЕЗУЛЬТАТА
Дизайн-документ из 7 разделов + диаграмма потока аутентификации (клиент → IdP/authZ server → gateway → сервисы).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Зафиксирована одна система; выбран session/token с обоснованием под масштаб; описан JWT (claims без секретов, подпись).
- [ ] OAuth2 grant types сопоставлены клиентам; OIDC для установления личности; SSO/IdP при необходимости.
- [ ] Access + refresh с сроками и механизмом отзыва.
- [ ] Модель авторизации выбрана под правила системы и обоснована (RBAC/ABAC/ReBAC).
- [ ] Проверки распределены: gateway (AuthN/грубая) vs сервис (тонкая/объектная — BOLA).
- [ ] Проброс identity (token propagation) + zero trust между сервисами.
- [ ] Секреты в хранилище (не в коде); безопасное хранение токенов на клиенте; связи с модулями 24/30.

## ПОДСКАЗКА
Синтез модуля: AuthN (token для stateless-микросервисов, JWT без секретов в payload) → OAuth2 grant под клиентов + OIDC для «кто» + SSO/IdP → access+refresh с отзывом → модель авторизации под правила (RBAC/ABAC/ReBAC) → проверки (gateway грубо, сервис тонко/объектно) → token propagation + zero trust → секреты в vault. Дальше — Модуль 30: защита API (BOLA — это и есть объектная authZ).
