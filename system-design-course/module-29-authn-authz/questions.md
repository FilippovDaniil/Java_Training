# Вопросы для самопроверки — Модуль 29: AuthN/AuthZ в дизайне (OAuth2/OIDC/JWT, SSO)

25 вопросов в стиле design-интервью (middle+/senior). Сначала ответьте сами, потом сверьтесь с ключом.

1. Чем аутентификация (AuthN) отличается от авторизации (AuthZ)?
2. Какой HTTP-код при провале AuthN, а какой — при провале AuthZ?
3. Почему AuthZ нельзя проверять раньше AuthN?
4. Чем session-based аутентификация отличается от token-based?
5. Почему token-based лучше масштабируется горизонтально?
6. В чём преимущество session-based по отзыву?
7. Из чего состоит JWT?
8. Что такое claims и что нельзя класть в payload JWT?
9. Чем HMAC-подпись отличается от RSA/EC для микросервисов?
10. В чём главная слабость JWT и как её смягчают?
11. Чем access token отличается от refresh token?
12. Что такое OAuth2 и какую задачу он решает?
13. Назовите роли в OAuth2.
14. Когда использовать authorization code + PKCE, а когда client credentials?
15. Зачем нужен PKCE?
16. Что такое OIDC и чем он отличается от OAuth2?
17. Что такое id_token?
18. Что такое SSO и как он устроен?
19. Что такое Identity Provider (IdP)?
20. Чем различаются RBAC, ABAC и ReBAC?
21. Почему «доступ к своим объектам» плохо ложится на RBAC?
22. Где проверять аутентификацию, а где — тонкую авторизацию?
23. Что такое token propagation в микросервисах?
24. Что такое zero trust и почему нельзя доверять внутренней сети?
25. Где хранить секреты для подписи токенов и почему не в коде?

---

## Ключ (эталонные ответы)

1. AuthN устанавливает личность («кто ты»), AuthZ проверяет права уже установленной личности («что тебе разрешено»). Сначала AuthN, потом AuthZ.
2. Провал AuthN (нет/невалидная личность) → 401 Unauthorized. Провал AuthZ (личность есть, прав нет) → 403 Forbidden.
3. Потому что нельзя проверить права субъекта, не зная, кто он. Авторизация работает над уже установленной личностью.
4. Session-based: состояние хранится на сервере (session store), у клиента — cookie с session-id. Token-based: состояние в самом токене у клиента (самодостаточный), сервер проверяет подпись.
5. Потому что токен самодостаточен и проверяется подписью без обращения к хранилищу → сервисы stateless, любой инстанс обработает запрос (не нужен общий session store/sticky sessions).
6. Сессию можно мгновенно отозвать, удалив её из хранилища; выданный токен действителен до истечения, отозвать его сложнее.
7. Из трёх частей (Base64url): header (алгоритм подписи), payload (claims), signature (подпись header.payload секретом/ключом).
8. Claims — утверждения о субъекте (id, роли, срок exp, iss/aud). В payload нельзя класть секреты/пароли/чувствительные PII, т.к. он только закодирован (Base64), а не зашифрован — читаем.
9. HMAC — симметричная подпись на общем секрете (все проверяющие знают секрет). RSA/EC — асимметричная: подписывает приватным ключом, сервисы проверяют публичным, не зная приватного. Для микросервисов предпочтительна асимметричная (не раздавать секрет всем).
10. Невозможность мгновенного отзыва (токен валиден до exp). Смягчают коротким access token + долгоживущим refresh token (отзываемым на сервере) и/или блок-листом по jti.
11. Access token — короткоживущий, даёт доступ к ресурсам. Refresh token — долгоживущий, служит для получения нового access token; хранится надёжнее и отзывается на auth-сервере.
12. Протокол делегированной авторизации: позволяет приложению получить ограниченный доступ к ресурсам пользователя без передачи ему пароля (через access token со scopes).
13. Resource Owner (пользователь-владелец), Client (приложение), Authorization Server (выдаёт токены), Resource Server (хранит ресурсы, принимает access token).
14. Authorization code + PKCE — для интерактивного входа пользователя (веб/SPA/mobile). Client credentials — для machine-to-machine (нет пользователя, сервис↔сервис).
15. PKCE защищает публичных клиентов (SPA/mobile, которые не могут хранить секрет) от перехвата authorization code: клиент доказывает, что именно он инициировал запрос.
16. OIDC — слой аутентификации поверх OAuth2, добавляющий id_token (кто пользователь). OAuth2 — про доступ к ресурсам (authZ), OIDC — про установление личности (authN).
17. Токен (JWT) в OIDC, содержащий информацию о пользователе (id/sub, email и т.д.) — предназначен именно для установления личности, в отличие от access token.
18. Single Sign-On — один вход для многих приложений. Пользователь аутентифицируется у центрального Identity Provider один раз, приложения (relying parties) доверяют его токенам и не требуют повторного логина.
19. Сервис, централизованно аутентифицирующий пользователей и выдающий токены/утверждения о личности, которым доверяют приложения (через OIDC/SAML). Единая точка управления учётками, отзыва, MFA.
20. RBAC — доступ по ролям пользователя. ABAC — по атрибутам (юзера/ресурса/контекста: отдел, время, гео). ReBAC — по отношениям между сущностями (владелец/расшарено; Google Zanzibar).
21. Потому что «свой объект» — это отношение владения между пользователем и конкретным ресурсом, а роль не привязана к конкретному объекту. Это естественно выражается в ReBAC (или проверкой владения), а не ролями.
22. Аутентификацию и грубую авторизацию (роль/scope) удобно делать на gateway/edge (единая точка). Тонкую (объектную) авторизацию — в самом сервисе, который владеет ресурсом и знает свои правила доступа.
23. Передача identity пользователя по цепочке межсервисных вызовов (токен/подписанные claims), чтобы каждый сервис мог проверить, кто инициатор, и применить свои правила авторизации.
24. Модель «никому не доверяй по умолчанию»: запрос не считается доверенным только потому, что пришёл из внутренней сети. Иначе скомпрометированный сервис получает доступ ко всему — каждый сервис должен проверять identity/права.
25. В секрет-хранилище (vault/secret manager) с ротацией, не в коде/гите/логах (утечка = подделка токенов). Асимметричная подпись дополнительно снимает необходимость раздавать общий секрет всем сервисам.
