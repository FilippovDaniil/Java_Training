# Модуль 29 — AuthN/AuthZ в дизайне (OAuth2/OIDC/JWT, SSO)

## Зачем этот модуль

Части 1–6 построили систему, которая держит нагрузку, переживает отказы и наблюдаема. Но любая реальная система должна знать, **кто** к ней обращается и **что** этому кому-то можно. Этот модуль открывает Часть 7 (безопасность и эксплуатация) с самого фундаментального: **аутентификация** (AuthN — кто ты) и **авторизация** (AuthZ — что тебе разрешено). Это не «прикрутить безопасность в конце», а **архитектурное решение**: где система аутентифицирует запрос, в каком виде носит identity и как пробрасывает его через сервисы.

Зачем это senior'у. Identity пронизывает всю архитектуру: выбор session vs token определяет, нужна ли общая сессия (а значит — stateful или stateless сервисы, модуль 24); где проверять authZ (на gateway или в сервисе) влияет на связность; как пробрасывать identity между микросервисами — это вопрос доверия (zero trust). Senior говорит на языке OAuth2/OIDC/JWT, отличает делегирование доступа от установления личности, выбирает RBAC/ABAC/ReBAC под требования и не делает классических ошибок (JWT в localStorage, authZ только на фронте, секреты в коде).

Он отвечает на «кто ты и что тебе можно» в распределённой системе — токены vs сессии, OAuth2/OIDC, SSO, модели авторизации и проброс identity.

## 1. AuthN vs AuthZ

| | Authentication (AuthN) | Authorization (AuthZ) |
|--|------------------------|----------------------|
| Вопрос | **Кто ты?** | **Что тебе разрешено?** |
| Когда | первым (установить личность) | после (проверить права) |
| Пример | логин/пароль, токен, биометрия | «может ли этот юзер удалить этот заказ?» |
| Провал → | 401 Unauthorized | 403 Forbidden |

```
Запрос → [AuthN: кто это?] → установлена личность → [AuthZ: что можно?] → разрешено/403
         нет личности → 401                          нет прав → 403
```

> Это базовое и часто путаемое различие: **AuthN устанавливает личность**, **AuthZ проверяет права** уже установленной личности. Порядок строгий — нельзя авторизовать неизвестного. На уровне HTTP: нет/невалидная личность → **401**, личность есть, но прав не хватает → **403**. В дизайне это два разных слоя: аутентификацию часто выносят на edge/gateway (единая точка), а авторизацию — ближе к ресурсу (сервис знает свои правила доступа).

## 2. Session-based vs Token-based

Два способа «помнить», что пользователь аутентифицирован между запросами (HTTP без состояния):

| | Session-based | Token-based (JWT) |
|--|---------------|-------------------|
| Где состояние | на сервере (session store) + cookie с session-id у клиента | в самом токене у клиента (самодостаточный) |
| Проверка | поиск сессии в хранилище | проверка подписи токена (без БД) |
| Масштаб | нужно общее хранилище сессий (Redis) — stateful | stateless, любой инстанс проверит |
| Отзыв | мгновенный (удалить сессию) | сложно (токен валиден до истечения) |

> Выбор напрямую влияет на масштабируемость (модуль 24): **session-based** требует общего хранилища сессий (иначе sticky sessions), **token-based** делает сервисы stateless (токен самодостаточен, проверяется подписью без обращения к хранилищу). Зато сессию можно мгновенно отозвать (удалить из store), а выданный токен живёт до истечения — отзыв требует ухищрений (короткий TTL + refresh, блок-лист). Это классический trade-off statelessness ↔ мгновенный отзыв.

## 3. JWT

**JWT (JSON Web Token)** — самодостаточный подписанный токен: `header.payload.signature` (Base64url).

```
header:    {"alg":"RS256","typ":"JWT"}        — алгоритм подписи
payload:   {"sub":"user42","role":"admin",    — claims (кто, что, до когда)
            "exp":1700000000}
signature: подпись(header.payload, ключ)       — целостность + подлинность
```

| Плюсы | Минусы |
|-------|--------|
| stateless: проверка подписи без БД | нельзя мгновенно отозвать (валиден до `exp`) |
| самодостаточный (claims внутри) | размер (едет в каждом запросе) |
| подходит для микросервисов/API | утечка = доступ до истечения |

> JWT несёт **claims** (утверждения: кто пользователь, его роли, срок) и **подпись** (HMAC на общем секрете или RSA/EC на паре ключей — тогда сервисы проверяют публичным ключом, не зная приватного). Главная сила — проверка **без обращения к хранилищу** (подходит для stateless-сервисов и межсервисных вызовов). Главная слабость — **отзыв**: выданный JWT действителен до `exp`. Лечение — короткоживущий **access token** (минуты) + долгоживущий **refresh token** (обновляет access; его можно отозвать на auth-сервере). Подпись ≠ шифрование: payload читаем — **не кладите туда секреты**.

## 4. OAuth2

**OAuth2** — протокол **делегированной авторизации**: дать приложению доступ к ресурсам пользователя **без передачи пароля**.

| Роль | Кто |
|------|-----|
| Resource Owner | пользователь (владелец данных) |
| Client | приложение, запрашивающее доступ |
| Authorization Server | выдаёт токены (после согласия владельца) |
| Resource Server | хранит ресурсы, принимает access token |

| Grant type | Когда |
|------------|-------|
| **Authorization Code + PKCE** | веб/SPA/mobile-приложения (PKCE защищает публичных клиентов) |
| **Client Credentials** | machine-to-machine (нет пользователя, сервис↔сервис) |

```
Authorization Code (упрощённо):
  user → client → authZ server (логин+согласие) → code → client меняет code на access token
       → client зовёт resource server с access token
```

> OAuth2 решает задачу «дать стороннему приложению доступ к моим данным, не отдавая ему пароль» (кнопка «войти через…», доступ к API от имени пользователя). Ключевая идея — **access token** с ограниченными правами (scopes) вместо пароля. Для дизайна важны два grant'а: **authorization code + PKCE** (интерактивный вход пользователя; PKCE обязателен для SPA/mobile — публичных клиентов без секрета) и **client credentials** (сервис↔сервис без пользователя). OAuth2 — про **доступ** (authZ), не про то, кто пользователь.

## 5. OIDC и SSO

**OpenID Connect (OIDC)** — слой **аутентификации** поверх OAuth2. Добавляет **id_token** (тоже JWT) — кто пользователь.

```
OAuth2:  access_token → «этому клиенту разрешён доступ к ресурсам» (ЧТО можно)
OIDC:    id_token     → «пользователь — это user42, email…»        (КТО это)
```

**SSO (Single Sign-On)** — один вход для многих приложений: пользователь аутентифицируется у **Identity Provider (IdP)** один раз, остальные приложения доверяют IdP.

> Частая путаница: **OAuth2 — про доступ, OIDC — про личность**. Если нужно «войти через Google» (узнать, кто пользователь) — это OIDC (id_token), а не «голый» OAuth2 (его access token не предназначен для установления личности). **SSO** строится на OIDC/SAML: централизованный **IdP** аутентифицирует пользователя, приложения (relying parties) доверяют его токенам — один вход, единая identity, проще отзыв/политики. Для дизайна с многими сервисами SSO + центральный IdP — стандарт.

## 6. Модели авторизации

| Модель | Решение основано на | Пример |
|--------|---------------------|--------|
| **RBAC** (Role-Based) | роли пользователя | «admin может удалять» |
| **ABAC** (Attribute-Based) | атрибуты (юзер/ресурс/контекст) | «доступ, если отдел совпадает и рабочее время» |
| **ReBAC** (Relationship-Based) | отношения между сущностями | «может, т.к. владелец документа» (Google Zanzibar) |

> Модель авторизации — проектное решение под сложность правил. **RBAC** прост и покрывает большинство (роли → права), но плохо выражает «доступ к *своим* объектам». **ABAC** гибок (политики на атрибутах: департамент, время, гео), но сложнее в управлении. **ReBAC** моделирует доступ через **отношения** (владелец/член/расшарено) — то, что нужно для «пользователь видит свои документы и те, что ему расшарили» (соцсети, файловые сервисы; Google Zanzibar). Начинают с RBAC, переходят к ABAC/ReBAC, когда правила перерастают роли.

## 7. Где проверять AuthZ и проброс identity

```
Клиент → [API Gateway: AuthN + грубая authZ] → [Сервис A: тонкая authZ] → [Сервис B]
              проверка токена,                   проверка прав на
              coarse-grained (роль/scope)         КОНКРЕТНЫЙ объект
```

| Уровень | Что проверяет |
|---------|---------------|
| **Gateway/edge** | аутентификация (валиден ли токен), грубая авторизация (роль/scope) |
| **Сервис** | тонкая авторизация (доступ к конкретному объекту — модуль 30, BOLA) |

**Проброс identity в микросервисах (token propagation):** identity пользователя передаётся по цепочке вызовов (токен/подписанные claims), каждый сервис проверяет сам.

> Распределение проверок: **аутентификацию** и грубую авторизацию удобно делать на **gateway** (единая точка, разгружает сервисы), но **тонкую авторизацию** (доступ к конкретному ресурсу) обязан делать **сам сервис** — только он знает свои правила и владение объектами. Identity пробрасывается по цепочке (token propagation). Принцип **zero trust**: не доверять запросу только потому, что он «из внутренней сети» — каждый сервис проверяет identity/права. «Аутентифицировались на периметре, внутри всё своим» — устаревшая и опасная модель.

## ⚠️ Подводные камни

- **Путать AuthN и AuthZ.** Аутентифицирован ≠ имеет права; проверять и личность, и доступ к объекту (401 vs 403).
- **AuthZ только на фронтенде.** Клиент не доверенный; авторизацию проверять на сервере всегда.
- **Секреты/PII в payload JWT.** Payload читаем (Base64, не шифр); подпись защищает целостность, не конфиденциальность.
- **JWT в localStorage.** Уязвим к XSS; для браузера — httpOnly-cookie (с защитой от CSRF) или короткий токен в памяти.
- **Долгоживущий access token.** Нельзя отозвать → утечка опасна; короткий access + refresh.
- **OAuth2 вместо OIDC для логина.** Access token не для установления личности; «кто пользователь» — id_token (OIDC).
- **Публичный клиент без PKCE.** SPA/mobile не хранят секрет; authorization code без PKCE перехватываем.
- **Доверие внутренней сети.** Без zero trust скомпрометированный сервис ходит везде; пробрасывать и проверять identity.
- **Секреты в коде/гите.** Ключи подписи/клиентские секреты — в секрет-хранилище (модуль 30), не в репозитории.
- **RBAC там, где нужен ReBAC.** «Доступ к своим объектам» ролями не выразить; брать ReBAC/ABAC.
