# Задача 01 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Спроектируйте эшелонированную оборону (defense in depth) для публичного API, расписав слои.

## ВХОДНЫЕ ДАННЫЕ
Публичный REST API финтех-сервиса: мобильные/веб-клиенты + партнёрские интеграции. Сейчас защита сводится к «проверяем токен на gateway». Нужно показать многослойную защиту, чтобы пробой одного слоя не открывал всю систему.

## ТРЕБОВАНИЯ
- Перечислите слои обороны от клиента до данных (минимум 5).
- Для каждого слоя укажите, от чего он защищает.
- Объясните принцип «пробой одного слоя ≠ компрометация всего» на примере украденного токена.
- Назовите сопутствующие принципы (least privilege, fail secure, минимизация поверхности).

## ФОРМАТ РЕЗУЛЬТАТА
Схема слоёв (клиент→данные) → таблица «слой → от чего защищает» → пример с украденным токеном → принципы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Слои: TLS (транспорт) → WAF/DDoS-щит (периметр) → gateway (authN + rate limit) → сервис (валидация + объектная authZ) → БД (параметризация + least privilege) → данные (шифрование at-rest, минимизация PII).
- [ ] Для каждого слоя названа угроза, которую он закрывает.
- [ ] Пример: украденный токен проходит authN, но объектная authZ, rate limit и минимум прав БД всё ещё ограничивают ущерб.
- [ ] Названы принципы: least privilege, fail secure (deny при ошибке), минимизация поверхности.
- [ ] Подчёркнуто: безопасность — свойство каждого слоя, не одна «стена».

## ПОДСКАЗКА
Defense in depth — независимые слои: TLS → WAF/DDoS → gateway (authN+rate limit) → сервис (валидация+объектная authZ) → БД (параметризация+least privilege) → данные (шифрование/минимизация PII). Пробой одного (украли токен) сдерживается остальными. Принципы: least privilege, fail secure, минимизация поверхности.
