# Задача 02 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Найдите BOLA/IDOR-уязвимость в API и спроектируйте корректную объектную авторизацию.

## ВХОДНЫЕ ДАННЫЕ
Эндпоинты сервиса:
- `GET /api/orders/{orderId}` — возвращает заказ; сервер проверяет только валидность токена.
- `GET /api/users/{userId}/invoices` — счета пользователя; проверяется только токен.
- Команда «защитилась», заменив числовые id на UUID: «теперь не переберёшь».

## ТРЕБОВАНИЯ
- Объясните, в чём BOLA-уязвимость текущих эндпоинтов.
- Объясните, почему замена id на UUID не закрывает дыру.
- Спроектируйте корректную проверку: что именно сервер должен проверять при обращении по id.
- Свяжите с разграничением AuthN/AuthZ (модуль 29).

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз BOLA → почему UUID не помогает → корректная объектная authZ → связь с AuthN/AuthZ.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Диагноз: токен валиден, но не проверяется, принадлежит ли объект (заказ/счета) этому пользователю → доступ к чужим объектам по id.
- [ ] UUID не закрывает: это лишь усложняет перебор, но не проверяет доступ; UUID утекают (логи, ссылки, рефереры).
- [ ] Корректно: при каждом обращении по id проверять на сервере, что объект принадлежит/доступен текущему пользователю (объектная authZ).
- [ ] Связь: аутентифицирован (AuthN) ≠ имеет доступ к этому объекту (AuthZ на объект); BOLA — провал второго.
- [ ] Отмечено, что это серверная проверка, не «спрятать id».

## ПОДСКАЗКА
BOLA: токен валиден, но сервер не проверяет владение объектом → перебор id даёт чужие данные. UUID лишь усложняет перебор (и утекает), но не авторизует. Решение — серверная проверка «этот объект принадлежит/доступен этому пользователю» при каждом запросе по id. Это объектная authZ из модуля 29.
