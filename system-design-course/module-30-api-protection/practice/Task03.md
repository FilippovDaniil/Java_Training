# Задача 03 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Найдите и устраните уязвимости к инъекциям в дизайне доступа к данным.

## ВХОДНЫЕ ДАННЫЕ
Фрагменты дизайна:
1. Поиск: запрос строится конкатенацией — `"SELECT * FROM products WHERE name LIKE '%" + q + "%'"`.
2. Экспорт: имя файла из ввода передаётся в shell-команду генерации отчёта.
3. Фильтр: поле сортировки берётся из query-параметра и подставляется в SQL как есть.

## ТРЕБОВАНИЯ
- Для каждого фрагмента укажите тип инъекции и риск.
- Предложите корректное решение (разделение данных и кода).
- Объясните разницу подходов для значений (параметризация) и для идентификаторов (имя колонки — allowlist).
- Сформулируйте общий принцип защиты от инъекций.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «фрагмент → тип инъекции → решение» + принцип.

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) SQL-инъекция → параметризованный запрос/prepared statement (значение как параметр, не часть строки).
- [ ] (2) Command injection → не передавать ввод в shell; использовать безопасный API/библиотеку, валидировать/экранировать.
- [ ] (3) SQL-инъекция через имя колонки → параметры не работают для идентификаторов; проверять поле сортировки по allowlist разрешённых колонок.
- [ ] Объяснена разница: значения → параметризация; идентификаторы (колонки/таблицы) → allowlist (их нельзя биндить параметром).
- [ ] Принцип: разделять данные и код; недоверенный ввод никогда не интерпретируется как код/команда.

## ПОДСКАЗКА
Инъекция = смешение данных и кода. Значения → параметризованные запросы (биндинг). Идентификаторы (имя колонки/таблицы) параметром не биндятся → allowlist разрешённых. Shell → не передавать ввод напрямую. Принцип: недоверенный ввод ≠ код.
