# Задача 04 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Спроектируйте валидацию ввода и защиту от mass assignment / excessive data exposure.

## ВХОДНЫЕ ДАННЫЕ
Эндпоинт `POST /api/users` создаёт пользователя. Сейчас: тело запроса целиком маппится в сущность User и сохраняется; ответ возвращает всю сущность User (включая passwordHash, internalFlags, isAdmin). Валидация — только на клиенте (форма).

## ТРЕБОВАНИЯ
- Найдите проблемы: mass assignment, excessive data exposure, валидация только на клиенте.
- Спроектируйте серверную валидацию ввода (allowlist полей и правил).
- Защитите от mass assignment (что нельзя принимать из тела).
- Защитите от excessive data exposure (что нельзя возвращать).

## ФОРМАТ РЕЗУЛЬТАТА
Диагноз (3 проблемы) → серверная валидация → защита mass assignment → фильтрация ответа.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Mass assignment: клиент может прислать `isAdmin=true`/internalFlags → принимать только явный allowlist разрешённых полей (не маппить тело в сущность целиком).
- [ ] Excessive data exposure: ответ не должен содержать passwordHash/internalFlags/isAdmin → возвращать DTO с разрешёнными полями (фильтрация на сервере).
- [ ] Валидация переносится/дублируется на сервер (клиентская — только UX, обходится).
- [ ] Валидация по allowlist (разрешённые поля/форматы/диапазоны), а не blocklist.
- [ ] Отмечено: фильтрацию входа и выхода делает сервер, не клиент.

## ПОДСКАЗКА
Mass assignment — маппить тело в сущность целиком (клиент пришлёт isAdmin) → явный allowlist полей. Excessive data exposure — вернуть всю сущность (passwordHash) → отдавать DTO с разрешёнными полями. Валидация — на сервере (клиентская обходится) и по allowlist. Фильтр входа/выхода — на сервере.
