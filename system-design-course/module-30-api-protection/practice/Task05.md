# Задача 05 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Спроектируйте secrets management для системы, убрав секреты из кода и снизив риск утечки.

## ВХОДНЫЕ ДАННЫЕ
Сейчас: ключ подписи JWT, пароль БД и API-ключ платёжного провайдера захардкожены в `application.yml` и закоммичены в гит; в логах при ошибке печатается полный объект запроса с токеном; один и тот же ключ используется год без ротации.

## ТРЕБОВАНИЯ
- Перечислите проблемы текущего подхода и их риски.
- Спроектируйте корректное хранение секретов (где и как).
- Опишите доставку секретов в приложение (без попадания в образ/гит).
- Добавьте ротацию и маскирование в логах.

## ФОРМАТ РЕЗУЛЬТАТА
Проблемы/риски → хранилище секретов → доставка → ротация + маскирование логов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблемы: секреты в гите (утечка навсегда в истории), токен в логах (утечка), отсутствие ротации (долгая компрометация).
- [ ] Хранилище: vault/secret manager (или K8s Secret/облачный secret store), не код/гит/образ.
- [ ] Доставка: секреты инъектируются в рантайме (env/смонтированный секрет), не зашиты в образ и не в репозитории.
- [ ] Ротация: регулярная смена ключей/паролей; для подписи — поддержка нескольких ключей (key id) при ротации.
- [ ] Маскирование: не логировать токены/пароли/карты; маскировать чувствительные поля.
- [ ] Отмечено: утечка ключа подписи = подделка токенов (связь с модулем 29); удалить секреты из истории гита/ротировать скомпрометированные.

## ПОДСКАЗКА
Секреты — в vault/secret manager, не в коде/гите/образе/логах. Доставка — в рантайме (env/смонтированный секрет). Ротация регулярная (+ key id для плавной смены ключа подписи). Логи — маскировать токены/пароли. Секрет в гите = утечка навсегда (история) → удалить и ротировать.
