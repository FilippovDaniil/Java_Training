# Задача 07 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Спроектируйте корректную настройку транспорта и браузерной защиты: TLS, CORS, CSRF.

## ВХОДНЫЕ ДАННЫЕ
Система: SPA на `app.example.com` + API на `api.example.com`. Текущее состояние: внутренние вызовы между сервисами идут по HTTP; CORS настроен как `Access-Control-Allow-Origin: *` с `Allow-Credentials: true`; аутентификация частично на cookie, частично на Bearer-токене; про CSRF не думали.

## ТРЕБОВАНИЯ
- Исправьте TLS: где он должен быть (включая внутренние вызовы при zero trust).
- Исправьте CORS: почему `*` с credentials опасен/некорректен и как правильно.
- Разберите CSRF: для какой схемы аутентификации он актуален и как защититься.
- Объясните, почему Bearer-в-заголовке не подвержен CSRF, а cookie — подвержен.

## ФОРМАТ РЕЗУЛЬТАТА
TLS (где) → CORS (правильно) → CSRF (когда актуален + защита) → cookie vs Bearer.

## КРИТЕРИИ ПРИЁМКИ
- [ ] TLS: везде, включая внутренние межсервисные вызовы (zero trust, против MITM/перехвата токенов); HSTS на публичном.
- [ ] CORS: `*` вместе с `Allow-Credentials: true` некорректно/опасно → явный allowlist origin'ов (например, `app.example.com`).
- [ ] CSRF актуален для cookie-based аутентификации (браузер шлёт cookie автоматически) → SameSite-cookie + CSRF-токены.
- [ ] Bearer-токен в заголовке не подвержен CSRF: его надо явно подставить из JS, браузер сам не отправит.
- [ ] Рекомендация согласовать схему (не смешивать cookie и Bearer бессистемно); для cookie — защита CSRF, для Bearer — защита от XSS/хранения.

## ПОДСКАЗКА
TLS везде (вкл. внутренние вызовы при zero trust) + HSTS. CORS `*` с credentials некорректен → явный allowlist origin'ов. CSRF актуален для cookie-auth (cookie шлётся автоматически) → SameSite + CSRF-токены. Bearer-в-заголовке от CSRF не страдает (надо явно подставить), но требует защиты от XSS.
