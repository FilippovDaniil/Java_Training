# Задача 08 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Найдите уязвимости в дизайне API по OWASP и исправьте их.

## ВХОДНЫЕ ДАННЫЕ
Цитаты из дизайн-документа:
1. «Эндпоинт `/api/orders/{id}` проверяет токен — этого достаточно».
2. «Запросы строим конкатенацией строк, так гибче».
3. «Возвращаем всю сущность пользователя, фронт сам покажет нужное».
4. «На проде включён debug-режим: при ошибке отдаём полный стектрейс — удобно дебажить».
5. «БД-пользователь приложения имеет права суперпользователя — чтобы не возиться с грантами».
6. «Лимитов на запросы нет — у нас и так мало трафика».

## ТРЕБОВАНИЯ
- Для каждого пункта укажите класс уязвимости (по OWASP) и исправление.
- Свяжите с понятиями модуля (BOLA, injection, excessive data exposure, misconfiguration, least privilege, rate limiting).

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «пункт → класс уязвимости → исправление».

## КРИТЕРИИ ПРИЁМКИ
- [ ] (1) BOLA → проверять владение объектом на сервере, не только токен.
- [ ] (2) Injection → параметризованные запросы/ORM, не конкатенация.
- [ ] (3) Excessive data exposure → возвращать DTO с разрешёнными полями, фильтрация на сервере.
- [ ] (4) Security misconfiguration (verbose errors) → в проде скрывать стектрейсы, отдавать обобщённую ошибку, детали — в логи.
- [ ] (5) Нарушение least privilege → БД-пользователю минимум необходимых прав, не суперпользователь.
- [ ] (6) Lack of rate limiting → ввести лимиты (brute-force/перебор/DDoS возможны при любом трафике).

## ПОДСКАЗКА
Классический OWASP-набор: только токен без объектной проверки (BOLA), конкатенация SQL (injection), вся сущность в ответ (excessive data exposure), стектрейсы в проде (misconfiguration), суперправа БД (нарушение least privilege), нет лимитов (lack of rate limiting). Каждая закрывается конкретной мерой модуля.
