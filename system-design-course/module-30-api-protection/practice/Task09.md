# Задача 09 — Модуль 30: Защита API и противодействие атакам

## ЗАДАНИЕ
Спроектируйте обращение с чувствительными данными (PII): минимизация, маскирование, шифрование at-rest.

## ВХОДНЫЕ ДАННЫЕ
Сервис хранит и обрабатывает: ФИО, email, телефон, номер карты, паспортные данные, историю платежей. Сейчас: всё пишется в одну таблицу без шифрования; при ошибках в логи попадает полный объект запроса (включая карту и паспорт); собираются все поля «на всякий случай»; доступ к данным имеют все сервисы.

## ТРЕБОВАНИЯ
- Примените минимизацию данных: что собирать/хранить, а что — нет.
- Спроектируйте маскирование в логах (что нельзя писать).
- Опишите шифрование at-rest и в транзите для чувствительных данных.
- Ограничьте доступ (least privilege) к PII.

## ФОРМАТ РЕЗУЛЬТАТА
Минимизация → маскирование логов → шифрование (at-rest/in-transit) → ограничение доступа.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Минимизация: собирать/хранить только необходимое; не хранить полный номер карты (токенизация/PCI), не дублировать паспорт без нужды («что не собрали — не утечёт»).
- [ ] Маскирование: в логи не писать карту/паспорт/токены; маскировать (например, last4 карты), а не полный номер.
- [ ] Шифрование: чувствительные данные — at-rest (диски/бэкапы/поля) и in-transit (TLS); ключи в vault (модуль 30 §6).
- [ ] Least privilege: доступ к PII только у сервисов, которым он нужен, не у всех.
- [ ] Отмечено: минимизация PII снижает ущерб от утечки и регуляторный риск (связь с residency/GDPR, модуль 25).

## ПОДСКАЗКА
PII: минимизировать (не собирать/хранить лишнее; карты — токенизация/PCI, «что не собрали — не утечёт»), маскировать в логах (last4, не полный номер/паспорт), шифровать at-rest и in-transit (ключи в vault), ограничить доступ least privilege. Меньше PII + шифрование = меньше ущерб от утечки и регуляторный риск.
