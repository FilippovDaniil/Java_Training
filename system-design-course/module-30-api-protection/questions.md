## Ключ (эталонные ответы)

1. Потому что всё, что выставлено наружу (эндпоинты, параметры), доступно злоумышленникам для попыток взлома; это входная точка в систему, которую надо защищать.
2. Эшелонированная оборона: несколько независимых слоёв защиты, чтобы пробой одного слоя не означал компрометацию всей системы (TLS, WAF, authN, объектная authZ, параметризация, шифрование данных).
3. Каждому компоненту/пользователю/сервису давать минимум прав, необходимых для работы. Снижает ущерб при компрометации (взломанный компонент мало что может).
4. При ошибке/сбое система должна закрываться (deny по умолчанию), а не открывать доступ. Сбой не должен превращаться в обход защиты.
5. Это стандартный чек-лист типичных классов уязвимостей API, по которому проходят при ревью дизайна, чтобы узнавать и закрывать распространённые провалы.
6. Broken Object Level Authorization (он же IDOR): запрос аутентифицирован, но сервер не проверяет, что пользователь имеет доступ именно к запрашиваемому объекту. Частая, т.к. путают аутентификацию (кто) с авторизацией на объект (можно ли этот объект).
7. Потому что UUID лишь усложняет перебор, но не проверяет доступ; UUID утекают (логи, ссылки, рефереры). Защита — авторизация, а не «спрятать id».
8. Проверять на сервере при каждом обращении по id, что объект принадлежит/доступен текущему пользователю (объектная авторизация), а не полагаться на необнаружимость id.
9. Когда недоверенный ввод интерпретируется как код/команда (SQL, shell, NoSQL). Корень — смешение данных и кода.
10. Параметризованными запросами / prepared statements / ORM: значения передаются как параметры, а не конкатенируются в SQL-строку (данные отделены от кода).
11. Потому что параметры биндят значения, а не идентификаторы (имена колонок/таблиц). Поле сортировки/колонку нужно проверять по allowlist разрешённых имён.
12. Потому что клиент не доверенный: клиентскую валидацию обходят (прямой запрос к API мимо формы). Валидация должна быть на сервере (клиентская — только UX).
13. Allowlist разрешает только известно-хорошее (всё остальное запрещено), blocklist запрещает известно-плохое и всегда неполон (новые обходы). Allowlist безопаснее по умолчанию.
14. Когда клиент передаёт поля, которые не должен (например, `isAdmin=true`), и они маппятся в сущность. Защита — принимать только явный allowlist разрешённых полей, не маппить тело запроса в сущность целиком.
15. Когда API возвращает больше данных, чем нужно (вся сущность с passwordHash/внутренними полями) в расчёте, что клиент отфильтрует. Фильтровать должен сервер — возвращать DTO с разрешёнными полями.
16. Ограничивает brute-force (перебор паролей/токенов), перебор объектов (BOLA-сканирование) и abuse, а также сглаживает объёмные атаки. Это слой защиты, а не только справедливость распределения.
17. Поглощением на периметре: CDN/anycast распределяют и фильтруют объём, scrubbing-центры чистят трафик, WAF режет вредоносные паттерны, автоскейл/лимиты сдерживают нагрузку — до приложения.
18. Чтобы шифровать трафик в транзите и не дать перехватить данные и токены (MITM). При zero trust TLS нужен и для внутренних межсервисных вызовов (внутренняя сеть не считается доверенной). HSTS заставляет браузер всегда использовать HTTPS.
19. Указывать явный allowlist разрешённых origin'ов вместо `*`. `*` вместе с `Allow-Credentials: true` некорректно/опасно — открывает доступ с кредами произвольным сайтам.
20. Для cookie-based аутентификации (браузер автоматически прикрепляет cookie к запросу). Защита — SameSite-cookie + CSRF-токены.
21. Потому что Bearer-токен надо явно подставить в заголовок из JS; браузер не отправляет его автоматически, поэтому сторонний сайт не может выполнить запрос от имени пользователя (но появляется риск XSS/хранения токена).
22. В секрет-хранилище (vault/secret manager) с ротацией. Не в коде/гите (утечка навсегда в истории)/образе/логах. Утечка ключа подписи = подделка токенов.
23. Минимизировать сбор и хранение (что не собрали — не утечёт; карты токенизировать), маскировать в логах (не писать полный номер/паспорт/токены), шифровать at-rest и in-transit, ограничить доступ least privilege.
24. Небрежность конфигурации, открывающая доступ: дефолтные креды (admin/admin), открытые порты управления, verbose-ошибки/стектрейсы в проде, забытые debug-эндпоинты, избыточные права сервиса/БД-пользователя. Закрывается дисциплиной и прод-hardening.
25. API gateway — единая точка применения сквозных политик (TLS-termination, аутентификация, rate limiting, логирование), уменьшает дублирование. WAF — слой фильтрации известных вредоносных паттернов (инъекции, сканеры) до приложения.
