# Модуль 30 — Защита API и противодействие атакам

## Зачем этот модуль

Модуль 29 установил, кто пользователь и что ему можно. Но API — это **поверхность атаки**: всё, что выставлено наружу, кто-то попытается сломать. Этот модуль — про защиту API: какие классы атак существуют (OWASP), как строить **эшелонированную оборону** (defense in depth) и какие конкретные меры закрывают конкретные угрозы — от инъекций и BOLA до DDoS и утечки секретов.

Зачем это senior'у. Безопасность — не отдельная фаза, а свойство дизайна на каждом слое: транспорт (TLS), вход (валидация), авторизация на объект (BOLA — самая частая API-уязвимость), нагрузка (rate limiting против brute-force/DDoS), секреты (vault, не код), данные (шифрование, минимизация PII). Senior держит в голове модель угроз и OWASP API Top 10, проектирует защиту слоями (один пробитый слой не открывает всё) и не делает классических ошибок (конкатенация SQL, секреты в гите, CORS `*`, проверка только аутентификации без проверки доступа к объекту).

Он отвечает на «как защитить выставленный наружу API» — defense in depth, OWASP, инъекции, BOLA, rate limiting/DDoS, secrets, PII.

## 1. Поверхность атаки и defense in depth

**Любой выставленный наружу endpoint — потенциальная цель.** Принцип защиты — **defense in depth (эшелонированная оборона)**: несколько независимых слоёв, чтобы пробой одного не открывал систему.

```
Клиент → [TLS] → [WAF/DDoS-щит] → [API Gateway: authN + rate limit]
       → [Сервис: валидация ввода + authZ на объект] → [БД: параметризация, least privilege]
       → [Данные: шифрование at-rest, минимизация PII]
```

| Принцип | Суть |
|---------|------|
| Defense in depth | много слоёв; пробой одного ≠ компрометация всего |
| Least privilege | каждому компоненту/пользователю — минимум прав |
| Fail secure | при ошибке — закрываться (deny), а не открываться |
| Минимизация поверхности | не выставлять лишнего (порты, эндпоинты, данные) |

> Безопасность строится **слоями**, а не одной «стеной»: TLS на транспорте, WAF/щит на периметре, authN + rate limit на gateway, валидация + объектная authZ в сервисе, параметризация + least privilege в БД, шифрование данных. Если злоумышленник пробил один слой (например, украл токен), остальные (объектная авторизация, rate limit, минимум прав БД) всё ещё сдерживают ущерб. Это основная ментальная модель модуля.

## 2. OWASP API Security Top 10 (ориентир)

Стандартный чек-лист классов уязвимостей API:

| Класс | Суть |
|-------|------|
| **BOLA** (Broken Object Level Authorization) | доступ к чужому объекту по id (самая частая) |
| Broken Authentication | слабая/обходимая аутентификация |
| Excessive Data Exposure | API отдаёт лишние поля (фильтровать должен сервер, не клиент) |
| Mass Assignment | клиент передаёт поля, которые не должен (например, `isAdmin=true`) |
| Injection | SQL/command/NoSQL-инъекции |
| Security Misconfiguration | дефолтные креды, открытые порты, verbose-ошибки |
| Lack of Rate Limiting | нет защиты от brute-force/перебора/DDoS |

> OWASP API Top 10 — это **карта типичных провалов**, по которой проходят при ревью дизайна. Senior'у не нужно помнить все наизусть, но нужно узнавать паттерны: **BOLA** (доступ к чужому по id — №1), **excessive data exposure** (вернули весь объект пользователя с хэшем пароля «клиент сам отфильтрует»), **mass assignment** (приняли `role` из тела запроса), **misconfiguration** (debug-эндпоинт в проде). Каждый класс закрывается конкретной мерой (ниже).

## 3. BOLA / IDOR — авторизация на объект

**BOLA (он же IDOR — Insecure Direct Object Reference)** — самая частая API-уязвимость: запрос аутентифицирован, но сервер **не проверяет, что пользователь имеет доступ именно к этому объекту**.

```
❌ GET /api/orders/12345
   сервер: «токен валиден → отдаём заказ 12345»   ← НЕ проверил, чей это заказ!
   атакующий перебирает id: 12346, 12347... → читает чужие заказы.

✅ сервер: «токен валиден (AuthN) И заказ 12345 принадлежит этому пользователю (AuthZ на объект) → отдаём»
```

> BOLA — прямое следствие путаницы AuthN и AuthZ (модуль 29): аутентификация подтверждает «кто ты», но **не** «можно ли тебе именно этот объект». Защита — **проверять владение/доступ к конкретному ресурсу на сервере** при каждом обращении по id (а не полагаться на то, что клиент «не угадает» id). Не помогает «прятать» id или использовать UUID вместо инкремента (это лишь усложняет перебор, но не закрывает дыру). Только серверная объектная авторизация. Это та самая «тонкая authZ в сервисе» из модуля 29.

## 4. Инъекции и валидация ввода

**Инъекция** — недоверенный ввод интерпретируется как код/команда (SQL, shell, NoSQL).

```
❌ "SELECT * FROM users WHERE name = '" + input + "'"
   input = "'; DROP TABLE users; --"   → SQL-инъекция

✅ параметризованный запрос / prepared statement / ORM:
   "SELECT * FROM users WHERE name = ?"  с привязкой параметра (данные ≠ код)
```

| Мера | Против чего |
|------|-------------|
| Параметризованные запросы / ORM | SQL-инъекции (не конкатенировать ввод в запрос) |
| Валидация ввода **allowlist** (что разрешено), на **сервере** | мусор/инъекции/mass assignment |
| Экранирование вывода | XSS (если API кормит веб) |
| Не передавать ввод в shell | command injection |

> Корень инъекций — **смешение данных и кода**. Лечение универсально: **разделять их** — параметризованные запросы (данные едут как параметры, не как часть SQL-строки), а не конкатенация. Валидация ввода — по **allowlist** (разрешаем известно-хорошее), а не blocklist (запрещаем известно-плохое — всегда неполон), и **на сервере** (клиентская валидация — UX, её обходят). Mass assignment закрывается явным списком разрешённых к записи полей (не маппить тело запроса в сущность целиком).

## 5. Rate limiting, DDoS и транспорт

| Угроза | Защита | Связь |
|--------|--------|-------|
| Brute-force (перебор паролей/токенов) | rate limiting per-user/IP, блокировки, 429 | модуль 16 |
| Перебор объектов (BOLA-сканирование) | rate limiting + объектная authZ | 16, §3 |
| DDoS (объёмная атака) | CDN/scrubbing, anycast, WAF, автоскейл | модуль 10 |
| Перехват трафика (MITM) | TLS везде (in-transit), HSTS | — |

> **Rate limiting** (модуль 16) — не только про справедливость, но и про безопасность: ограничивает brute-force паролей, перебор id (BOLA-сканирование), abuse. **DDoS** поглощают на периметре — CDN/anycast распределяют и фильтруют объём (модуль 10), WAF режет вредоносные паттерны до приложения. **TLS обязателен везде** (включая внутренние вызовы при zero trust): без шифрования в транзите трафик и токены перехватываются (MITM). HSTS заставляет браузер всегда использовать HTTPS.

## 6. Secrets, CORS, CSRF, данные

| Тема | Правило |
|------|---------|
| **Secrets management** | ключи/пароли/токены — в vault/secret manager, не в коде/гите/логах; ротация |
| **CORS** | явный allowlist origin'ов, не `*` для приватных API с кредами |
| **CSRF** | для cookie-based auth — CSRF-токены/SameSite; token-in-header (Bearer) не подвержен |
| **Шифрование at-rest** | чувствительные данные/диски/бэкапы шифровать в покое |
| **PII** | минимизировать сбор, маскировать в логах, ограничивать доступ |

> Практические рубежи: **секреты** — в специализированном хранилище (vault), никогда в репозитории/образе/логах; утечка ключа подписи = подделка токенов (модуль 29). **CORS** — это политика браузера; для приватных API указывать конкретные origin'ы, не `*`. **CSRF** актуален для cookie-based аутентификации (браузер шлёт cookie автоматически) → SameSite-cookie + CSRF-токены; Bearer-токен в заголовке от CSRF не страдает (его надо явно подставить). **Данные**: шифровать at-rest, минимизировать PII (что не собрали — не утечёт), маскировать в логах (не писать карты/пароли/токены).

## 7. API Gateway, WAF и misconfiguration

```
API Gateway как единая точка применения политик:
  TLS-termination · аутентификация · rate limiting · базовая валидация · логирование
WAF (Web Application Firewall):
  фильтрует известные вредоносные паттерны (инъекции, сканеры) до приложения
```

**Security misconfiguration** — частый и недооценённый класс:
- дефолтные креды (admin/admin), открытые порты управления;
- verbose-ошибки/стектрейсы в проде (раскрывают внутренности);
- debug-эндпоинты, открытые в прод;
- избыточные права сервиса/БД-пользователя (нарушение least privilege).

> **API Gateway** — естественное место централизации сквозных политик безопасности (TLS, authN, rate limit, логирование) — меньше дублирования и единая точка контроля. **WAF** добавляет слой фильтрации вредоносных паттернов. Но самый дешёвый источник взлома — **misconfiguration**: дефолтные пароли, открытые порты, подробные ошибки в проде, забытые debug-эндпоинты, БД-пользователь с правами суперпользователя. Это не «дыры в коде», а небрежность конфигурации — закрывается дисциплиной (least privilege, прод-настройки, скрытие деталей ошибок).

## ⚠️ Подводные камни

- **BOLA: проверять только аутентификацию.** Аутентифицирован ≠ имеет доступ к этому объекту; проверять владение на сервере при каждом обращении по id.
- **«Спрятать» id вместо authZ.** UUID/обфускация не заменяют объектную авторизацию.
- **Конкатенация ввода в SQL.** → инъекция; только параметризованные запросы/ORM.
- **Валидация только на клиенте / blocklist.** Клиентскую обходят; blocklist неполон → серверный allowlist.
- **Excessive data exposure / mass assignment.** Не возвращать весь объект «клиент отфильтрует»; не маппить тело запроса в сущность целиком.
- **Секреты в коде/гите/логах.** → утечка; vault + ротация, маскирование в логах.
- **Нет TLS (особенно внутри).** Перехват трафика/токенов; TLS везде, HSTS.
- **CORS `*` для приватного API.** Открывает доступ чужим origin'ам; явный allowlist.
- **Нет rate limiting.** Brute-force/перебор/DDoS; лимиты per-user/IP + DDoS-щит на периметре.
- **Misconfiguration.** Дефолтные креды, verbose-ошибки, debug в проде, избыточные права; дисциплина least privilege и прод-hardening.
- **Безопасность «потом».** Это свойство каждого слоя на этапе дизайна, не финальная фаза.
