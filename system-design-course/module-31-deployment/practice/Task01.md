# Задача 01 — Модуль 31: Развёртывание

## ЗАДАНИЕ
Обоснуйте выбор контейнеров vs VM и необходимость оркестрации для системы.

## ВХОДНЫЕ ДАННЫЕ
Система: ~15 микросервисов, нагрузка переменная (пики ×5), требуется быстрое масштабирование, самовосстановление при падении инстанса и обновления без простоя. Сейчас всё деплоится вручную на несколько VM, обновления — с простоем, упавший сервис поднимают руками.

## ТРЕБОВАНИЯ
- Обоснуйте переход на контейнеры (vs VM): что это даёт.
- Объясните, почему 15 микросервисов с такими требованиями нуждаются в оркестрации.
- Сопоставьте требования из условия с возможностями оркестратора (self-healing, HPA, rolling).
- Отметьте, что важно из гарантий K8s для дизайна (а не админская кухня).

## ФОРМАТ РЕЗУЛЬТАТА
Контейнеры vs VM → зачем оркестрация → таблица «требование → возможность оркестратора» → вывод.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Контейнеры: immutable-образ с зависимостями, лёгкий/быстрый старт, воспроизводимость («работает у меня → везде») — против тяжёлых VM.
- [ ] Оркестрация нужна: ручное управление 15 сервисами на нескольких узлах невозможно при таких требованиях.
- [ ] Сопоставление: самовосстановление → self-healing; пики ×5 → HPA-автомасштабирование (модуль 24); обновления без простоя → rolling update.
- [ ] Отмечена декларативность (описать желаемое состояние, система поддерживает) и service discovery.
- [ ] Фокус на гарантиях (self-healing/HPA/rolling), а не на деталях администрирования.

## ПОДСКАЗКА
Контейнеры — immutable-образ, лёгкий старт, воспроизводимость (vs тяжёлые VM). Оркестрация (K8s) для многих контейнеров: self-healing (упал→поднял), HPA (пики→масштаб), rolling (без простоя), service discovery, декларативность. Сопоставьте требования из условия с этими возможностями.
