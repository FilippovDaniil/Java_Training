# Задача 03 — Модуль 31: Развёртывание

## ЗАДАНИЕ
Спроектируйте zero-downtime deployment через health checks и корректный вывод старых инстансов.

## ВХОДНЫЕ ДАННЫЕ
Сервис за балансировщиком, rolling update. Проблема: при выкате новые инстансы начинают получать трафик до того, как прогрелись (кэш/пул/соединение к БД) → всплеск ошибок; старые инстансы убиваются мгновенно, обрывая текущие запросы пользователей.

## ТРЕБОВАНИЯ
- Объясните роль liveness и readiness проб (чем отличаются, что делают при провале).
- Покажите, как readiness обеспечивает, что трафик идёт только в готовый инстанс.
- Опишите корректный вывод старого инстанса (connection draining, модуль 8).
- Сформулируйте, как из этого складывается zero-downtime.

## ФОРМАТ РЕЗУЛЬТАТА
Liveness vs readiness → роль readiness при старте → draining старых → итог zero-downtime.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Liveness: жив ли процесс → провал → перезапуск контейнера. Readiness: готов ли принимать трафик → провал → убрать из балансировки.
- [ ] Новый инстанс получает трафик только после прохождения readiness (после прогрева кэша/пула) → нет всплеска ошибок при старте.
- [ ] Старый инстанс: сначала выводится из балансировки, доделывает текущие запросы (connection draining), затем останавливается — без обрыва.
- [ ] Итог: трафик всегда идёт только в готовые инстансы, старые не обрываются → обновление без простоя.
- [ ] Связь с probes (модуль 24) и draining (модуль 8).

## ПОДСКАЗКА
Liveness (жив?) → рестарт; readiness (готов?) → вывод из балансировки. Трафик идёт в инстанс только после readiness (после прогрева) — нет ошибок при старте. Старый инстанс выводят из балансировки и дают доделать запросы (draining), потом гасят. Это и есть zero-downtime.
