# Задача 04 — Модуль 31: Развёртывание

## ЗАДАНИЕ
Спроектируйте canary-выкат с опорой на observability и автоматическим решением о расширении/откате.

## ВХОДНЫЕ ДАННЫЕ
Рискованный релиз ключевого сервиса. Нужно: выкатить на малую долю трафика, понаблюдать по метрикам, и либо расширить, либо откатить — желательно автоматически, без ручного «кажется, ок».

## ТРЕБОВАНИЯ
- Опишите этапы canary (доли трафика и шаги расширения).
- Определите метрики/SLI, по которым сравнивается canary с базовой версией (связь с модулем 28).
- Сформулируйте критерии автоматического расширения и автоматического отката.
- Свяжите с error budget (модуль 28) и chaos/наблюдаемостью (модуль 27).

## ФОРМАТ РЕЗУЛЬТАТА
Этапы canary → метрики сравнения → критерии расширения/отката → связь с error budget.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Этапы: малая доля (например, 1–5%) → при успехе расширение (25% → 50% → 100%) по шагам.
- [ ] Метрики/SLI canary сравниваются с baseline: error rate, latency p99, бизнес-метрики (модуль 28).
- [ ] Критерий расширения: SLI canary не хуже baseline в пределах порога; критерий отката: превышение порога error rate/latency → автоматический откат.
- [ ] Решение автоматизировано на метриках, а не «на глаз».
- [ ] Связь: canary тратит часть error budget осознанно; опирается на observability (модуль 28); родственно chaos-подходу (гипотеза + наблюдение, модуль 27).

## ПОДСКАЗКА
Canary: 1–5% трафика → сравнить SLI (error rate/p99/бизнес) с baseline → при не-хуже расширять (25→50→100%), при превышении порога — автооткат. Решение на метриках (модуль 28), не «на глаз». Canary осознанно тратит error budget; по духу — как chaos-гипотеза с наблюдением (модуль 27).
