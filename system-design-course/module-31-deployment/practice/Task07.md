# Задача 07 — Модуль 31: Развёртывание

## ЗАДАНИЕ
Спроектируйте использование feature flags, отделив выкатку кода от включения фичи.

## ВХОДНЫЕ ДАННЫЕ
Команда готовит крупную фичу. Раньше выкат кода = немедленная активация для всех; при проблеме приходилось срочно откатывать весь релиз (редеплой). Нужно: катить код безопасно, включать фичу постепенно и мгновенно гасить при проблеме без редеплоя.

## ТРЕБОВАНИЯ
- Объясните разницу «выкат кода» vs «включение фичи» и зачем их разделять.
- Опишите механику feature flag (код за флагом, управление флагом).
- Покажите постепенное включение (когорты/проценты) и мгновенное отключение.
- Сравните с canary: что общего и в чём разница уровня.

## ФОРМАТ РЕЗУЛЬТАТА
Разделение выкат/включение → механика флага → постепенное включение/гашение → сравнение с canary.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Разделение: код выкатывается «тёмным» (за выключенным флагом) — в проде, но неактивен; включение фичи — отдельное действие.
- [ ] Зачем: разделяет риск деплоя кода и риск включения фичи; гашение без редеплоя.
- [ ] Механика: проверка флага в коде; управление флагом снаружи (конфиг/сервис флагов) без передеплоя.
- [ ] Постепенное включение для когорты/процента пользователей; мгновенное отключение при проблеме.
- [ ] Сравнение с canary: оба — постепенное раскрытие с быстрым откатом; canary — на уровне версий/трафика, feature flag — на уровне функциональности (можно в одной версии).

## ПОДСКАЗКА
Feature flag отделяет выкат кода (за выключенным флагом, «тёмный») от включения фичи (отдельно, постепенно, мгновенно гасится без редеплоя). Это разделяет два риска. Похоже на canary, но canary — на уровне версий/трафика, а флаг — на уровне функциональности внутри одной версии.
