# Задача 09 — Модуль 31: Развёртывание

## ЗАДАНИЕ
Спроектируйте инфраструктуру как код (IaC) и immutable infrastructure, обеспечив паритет сред.

## ВХОДНЫЕ ДАННЫЕ
Сейчас: инфраструктура поднята вручную через веб-консоль облака; staging заметно отличается от prod (другие размеры машин, версии, конфиги); серверы патчат на месте, из-за чего конфигурации «разъезжаются» (на одном узле одно, на другом другое). Нагрузочные тесты на staging не предсказывают поведение prod.

## ТРЕБОВАНИЯ
- Опишите переход на IaC: что это даёт (воспроизводимость/версионирование/ревью).
- Опишите immutable infrastructure: чем заменить ручной патчинг.
- Объясните, как IaC + immutable устраняют конфигурационный дрейф и «снежинки».
- Свяжите паритет сред с достоверностью тестов (модуль 27).

## ФОРМАТ РЕЗУЛЬТАТА
IaC (что даёт) → immutable (вместо патчинга) → устранение дрейфа → паритет сред и тесты.

## КРИТЕРИИ ПРИЁМКИ
- [ ] IaC: инфраструктура описана декларативным кодом (Terraform и т.п.), версионируется, ревьюится, применяется воспроизводимо.
- [ ] Immutable: не патчить серверы на месте — собирать новый образ и пересоздавать инстансы; откат = развернуть прошлый образ.
- [ ] Устранение дрейфа: одинаковые инстансы из одного описания/образа → нет «снежинок» и расхождений конфигурации.
- [ ] Паритет: staging создаётся тем же IaC, что prod (близкие размеры/версии/конфиги) → нагрузочные тесты предсказывают prod (модуль 27).
- [ ] Отмечена связь с контейнерами (immutable-образ) и security в инфре (ревью кода, модуль 30).

## ПОДСКАЗКА
IaC — инфраструктура как декларативный код (воспроизводимо/версионируемо/ревьюируемо). Immutable — не патчить, а пересоздавать из нового образа (нет дрейфа/«снежинок», лёгкий откат). Паритет: staging из того же IaC, что prod → тесты достоверны (модуль 27).
