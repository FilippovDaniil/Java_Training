## Ключ (эталонные ответы)

1. Потому что способ развёртывания определяет ключевые свойства: можно ли обновляться без простоя (доступность), быстро ли откатиться, безопасно ли катить рискованное, воспроизводима ли инфраструктура. Это архитектурные решения.
2. VM виртуализирует целую ОС через гипервизор (тяжёлая, старт минуты); контейнер изолирует процесс, разделяя ядро ОС (лёгкий, старт секунды, immutable-образ приложения с зависимостями).
3. Воспроизводимость («работает у меня → работает везде») через immutable-образ с зависимостями, лёгкий и быстрый старт, переносимость между средами.
4. Чтобы управлять множеством контейнеров на многих узлах: расписание, самовосстановление, автомасштабирование, service discovery, обновления без простоя, декларативное поддержание желаемого состояния — вручную это невозможно.
5. Способность автоматически восстанавливать состояние: упал контейнер → перезапустить; упал узел → переместить нагрузку; не хватает реплик → создать недостающие.
6. Вы описываете желаемое состояние (например, «N реплик такой версии»), а система сама приводит и поддерживает фактическое состояние к желаемому, а не выполняет императивные шаги.
7. Постепенная замена инстансов старой версии новой. Без простоя и без удвоения ресурсов; trade-off — откат постепенный, и во время выката сосуществуют две версии (нужна совместимость).
8. Две полные среды (blue=текущая, green=новая); трафик переключается разом. Сила — мгновенное переключение и мгновенный откат (вернуть на blue). Цена — две полные среды (≈×2 ресурсы).
9. Выкат новой версии сначала на малую долю трафика с расширением при успехе. Лучше всего для рискованных изменений: ограничивает blast radius и позволяет проверить на реальном трафике по метрикам.
10. На метрики/SLI canary в сравнении с baseline (error rate, latency p99, бизнес-метрики) и observability (модуль 28); решение автоматизируют по порогам, а не «на глаз». Связано с расходом error budget.
11. Liveness — жив ли процесс (провал → перезапуск контейнера). Readiness — готов ли принимать трафик (провал → убрать из балансировки, но не перезапускать).
12. Новый инстанс получает трафик только после прохождения readiness (после прогрева) — нет ошибок при старте; старый сначала выводится из балансировки и доделывает запросы (draining), потом гасится — нет обрыва. Итог — обновление без простоя.
13. Корректный вывод инстанса из эксплуатации: его убирают из балансировки и дают завершить уже принятые запросы, прежде чем остановить, чтобы не обрывать активные соединения пользователей.
14. Механизм включать/выключать функциональность без редеплоя. Разделение нужно, чтобы отделить риск деплоя кода (выкатить «тёмным» за флагом) от риска включения фичи (активировать постепенно, мгновенно гасить при проблеме).
15. Canary раскрывает новую версию/трафик постепенно (на уровне версий и долей трафика). Feature flag раскрывает функциональность внутри одной версии (на уровне фичи/когорты). Оба дают постепенность и быстрый откат, но на разных уровнях.
16. CI (Continuous Integration) — автоматическая сборка и тесты при каждом коммите (раннее обнаружение интеграционных поломок). CD (Continuous Delivery/Deployment) — автоматическая доставка/выкат проверенного артефакта по средам.
17. Build → test (unit/integration) → security scan → package (образ) → deploy (staging → prod). Тесты и скан выступают «воротами»: их провал блокирует выкат.
18. Чтобы безопасность была встроена в процесс, а не «ревью в конце»: проверка зависимостей на уязвимости, сканирование образа, статический анализ — до попадания в прод (модуль 30).
19. Описание инфраструктуры декларативным кодом (Terraform и др.). Даёт воспроизводимость (та же среда из того же описания), версионирование (история), ревьюируемость (как код), предсказуемость.
20. Подход, при котором серверы/образы не патчат на месте, а пересоздают из нового образа: вместо изменения работающего инстанса разворачивают новый и гасят старый.
21. Все инстансы создаются из одного образа/описания и не меняются на месте → нет уникальных «снежинок» и расхождений конфигурации между узлами; откат = развернуть предыдущий образ.
22. Максимальная близость сред (особенно staging к prod) по размерам, версиям, конфигам. Важен, потому что без него нагрузочные тесты и проверки на staging не предсказывают поведение prod (модуль 27).
23. Конфиг в окружении (env, не в коде/образе → один образ для всех сред), stateless-процессы (состояние снаружи), логи как потоки в stdout (сбор снаружи), dev/prod parity.
24. Потому что образ может утечь (реестр, копия) — вместе с ним утекут зашитые секреты; и один образ нельзя переиспользовать между средами. Секреты инъектируют в рантайме (env/secret manager).
25. Потому что любое изменение может оказаться сбойным в проде; готовый быстрый откат (предыдущий образ / переключение blue-green / гашение canary) минимизирует простой и ущерб. Обратимость релиза — часть отказоустойчивости на уровне поставки.
