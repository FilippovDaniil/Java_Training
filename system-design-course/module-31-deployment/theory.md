# Модуль 31 — Развёртывание (контейнеры/оркестрация/CI-CD топология)

## Зачем этот модуль

Система спроектирована, масштабируема (24), отказоустойчива (25), наблюдаема (28) и защищена (29–30). Остаётся вопрос, который senior закладывает **в дизайн**, а не оставляет «на потом»: **как код доезжает до прода и обновляется**. Топология развёртывания — часть архитектуры: упаковка (контейнеры), запуск и самовосстановление (оркестрация), стратегия выката (rolling/blue-green/canary) и конвейер доставки (CI/CD). Этот модуль завершает Часть 7 (безопасность и эксплуатация).

Зачем это senior'у. Способ развёртывания определяет, можно ли обновляться **без простоя** (связь с доступностью, модуль 24), быстро ли **откатиться** при проблеме, безопасно ли катить рискованное (canary + observability + error budget, модули 27–28), воспроизводима ли инфраструктура (IaC). Senior выбирает стратегию выката под требования к доступности и риску, проектирует CI/CD-конвейер (включая security-скан, модуль 30) и избегает классических ошибок (нет health checks → рестарт-цикл, секреты в образе, нет rollback, ручной патчинг прода).

Он отвечает на «как система попадает в прод и обновляется надёжно» — контейнеры, оркестрация, стратегии выката, CI/CD, IaC, immutable infrastructure.

## 1. Контейнеры и зачем оркестрация

**Контейнер (Docker)** — изолированная, воспроизводимая упаковка приложения с зависимостями в **immutable-образ**.

| | VM | Контейнер |
|--|----|-----------| 
| Изолирует | целую ОС (гипервизор) | процесс (общее ядро ОС) |
| Вес/старт | тяжёлый, минуты | лёгкий, секунды |
| Воспроизводимость | образ ОС | immutable-образ приложения |

Один контейнер — просто. Сотни контейнеров на десятках машин — нужна **оркестрация**:

| Оркестрация (Kubernetes) даёт | Зачем |
|-------------------------------|-------|
| Расписание (scheduling) | разместить контейнеры по узлам |
| Self-healing | упал контейнер/узел → перезапустить/переместить |
| Автомасштабирование (HPA) | под нагрузку добавить/убрать реплики (модуль 24) |
| Service discovery | найти сервис без ручных адресов |
| Rolling update | обновление без простоя |
| Декларативность | «хочу N реплик» — система приводит к состоянию |

> Контейнеры решают «работает у меня → работает везде» (immutable-образ с зависимостями, лёгкий и быстрый старт против VM). Но управлять руками сотнями контейнеров невозможно — **оркестратор (Kubernetes)** автоматизирует размещение, самовосстановление (упало → поднял), автомасштабирование (модуль 24), обнаружение сервисов и обновления. Ключевая идея — **декларативность**: вы описываете желаемое состояние, система его достигает и поддерживает. Для дизайна важна не админская кухня K8s, а его гарантии: self-healing, HPA, rolling update без простоя.

## 2. Стратегии развёртывания

Как выкатить новую версию с минимальным риском и простоем:

| Стратегия | Как | Откат | Стоимость |
|-----------|-----|-------|-----------|
| **Rolling** | постепенно заменять инстансы старой версии новой | постепенный (катим старую назад) | низкая (без удвоения) |
| **Blue-Green** | две полные среды (blue=текущая, green=новая); переключить трафик разом | мгновенный (переключить обратно) | высокая (две среды) |
| **Canary** | новая версия на малой доле трафика → расширять при успехе | быстрый (убрать canary) | средняя |

```
Rolling:    [v1][v1][v1] → [v2][v1][v1] → [v2][v2][v1] → [v2][v2][v2]  (без даунтайма)
Blue-Green: blue(v1) ▶ trafffic   →   переключить →   green(v2) ▶ traffic   (мгновенный откат)
Canary:     95% → v1, 5% → v2  → наблюдаем метрики/SLO → расширяем v2 или откатываем
```

> Выбор стратегии — компромисс простоя, риска, скорости отката и стоимости. **Rolling** — дефолт K8s: без простоя и без удвоения ресурсов, но откат постепенный и какое-то время сосуществуют две версии (нужна совместимость). **Blue-Green** — мгновенное переключение и мгновенный откат, но держит две полные среды (дорого). **Canary** — самый безопасный для рискованных изменений: новая версия сначала на малой доле, решение о расширении — по метрикам/SLO (модуль 28) и поведению (связь с canary-анализом). Canary + observability + error budget — зрелый способ катить риск.

## 3. Zero-downtime, health checks и feature flags

**Health checks** (связь с probes, модуль 24):

| Проверка | Вопрос | Действие при провале |
|----------|--------|----------------------|
| **Liveness** | жив ли процесс? | перезапустить контейнер |
| **Readiness** | готов ли принимать трафик? | убрать из балансировки (не слать запросы) |

**Zero-downtime deployment** опирается на них: новый инстанс получает трафик только после **readiness**; старый выводится из балансировки и доделывает текущие запросы (connection draining, модуль 8) до остановки.

**Feature flags** — отделить **выкатку кода** от **включения фичи**:
```
Деплой кода (фича за выключенным флагом) → код в проде, но неактивен
→ включить флаг (для % юзеров / когорты) → выключить мгновенно при проблеме (без редеплоя)
```

> Развёртывание без простоя — это **health checks + грамотный вывод старых инстансов**: readiness гарантирует, что трафик идёт только в готовый инстанс (иначе ошибки во время старта), liveness перезапускает зависшие. **Feature flags** — мощный приём: код выкатывается «тёмным» (за флагом), а фича включается отдельно и постепенно (для когорты), и мгновенно гасится при проблеме без редеплоя. Это разделяет два риска — риск деплоя кода и риск включения фичи — и даёт контроль, близкий к canary, но на уровне функциональности.

## 4. CI/CD

**CI/CD** — конвейер от коммита до прода:

```
commit → [BUILD] → [TEST] → [SECURITY SCAN] → [PACKAGE: образ] → [DEPLOY: staging → prod]
         CI: сборка + тесты + скан           CD: доставка/выкат
```

| | Continuous Integration (CI) | Continuous Delivery/Deployment (CD) |
|--|-----------------------------|-------------------------------------|
| Про что | автоматическая сборка + тесты при каждом коммите | автоматическая доставка/выкат проверенного артефакта |
| Цель | рано ловить поломки интеграции | быстро и надёжно доставлять в прод |

> CI/CD автоматизирует путь «код → прод» так, чтобы он был **быстрым, повторяемым и проверенным**. **CI** при каждом коммите собирает и прогоняет тесты (рано ловит интеграционные поломки). **CD** доставляет проверенный артефакт по средам (staging → prod). Важный шаг — **security scan** в конвейере (модуль 30): проверка зависимостей на уязвимости, сканирование образа, статический анализ — безопасность встроена в пайплайн, а не «ревью в конце». Автоматизация конвейера снижает человеческие ошибки и ускоряет доставку (связь с error budget — частые мелкие безопасные релизы).

## 5. IaC и immutable infrastructure

**Infrastructure as Code (IaC)** — инфраструктура описана **декларативным кодом** (Terraform и др.), версионируется и применяется воспроизводимо.

**Immutable infrastructure** — серверы/образы **не патчат на месте**, а **пересоздают** из нового образа:
```
❌ Mutable: зашёл на сервер, обновил пакет, поправил конфиг  → дрейф конфигурации, «снежинки»
✅ Immutable: новый образ → выкатил новые инстансы → старые погасил  → воспроизводимо, предсказуемо
```

| Подход | Эффект |
|--------|--------|
| IaC | воспроизводимая, версионируемая, ревьюируемая инфраструктура |
| Immutable | нет конфигурационного дрейфа; откат = развернуть прошлый образ |

> **IaC** превращает инфраструктуру в код: воспроизводимо (та же среда из того же описания), версионируемо (история изменений), ревьюируемо (как код, модуль 30 — security в инфре). **Immutable infrastructure** убирает «снежинки» (уникальные руками настроенные серверы) и **конфигурационный дрейф**: ничего не патчим на работающем сервере — собираем новый образ и заменяем инстансы. Откат становится тривиальным (развернуть предыдущий образ). Вместе с контейнерами (§1) это даёт предсказуемость и воспроизводимость прода.

## 6. Среды, 12-factor и секреты в деплое

**Environments:** dev → staging → prod, с **паритетом** (staging максимально близок к prod, иначе тесты/нагрузка не переносятся, модуль 27).

**12-factor (избранное, релевантное дизайну):**
| Принцип | Суть |
|---------|------|
| Config в окружении | конфиг — в env-переменных, не в коде/образе (один образ для всех сред) |
| Stateless-процессы | состояние во внешних хранилищах (модуль 24) |
| Logs как потоки | приложение пишет в stdout, сбор — снаружи (модуль 28) |
| Dev/prod parity | минимизировать различия сред |

**Секреты в деплое** (связь с модулем 30): секреты инъектируются в рантайме (env/смонтированный секрет/secret manager), **не зашиты в образ** и не в репозитории — иначе образ с секретом утекает целиком.

> Зрелое развёртывание следует **12-factor**: один и тот же immutable-образ едет во все среды, а отличается только **конфигом из окружения** (env) — это и есть переносимость. Состояние — снаружи (stateless, модуль 24), логи — в stdout как поток (сбор снаружи, модуль 28). **Паритет сред** критичен: если staging не похож на prod, ни нагрузочные тесты (модуль 27), ни проверки не дают гарантий. И **секреты не кладут в образ** — только инъекция в рантайме (иначе утечка образа = утечка секретов).

## 7. Rollback и связь всего вместе

```
Безопасный релиз = воспроизводимый артефакт + стратегия выката + наблюдаемость + быстрый откат:
  immutable-образ (один на все среды, конфиг из env)
  + CI/CD (build→test→scan→deploy) с security-сканом
  + стратегия выката под риск (rolling / blue-green / canary)
  + health checks (readiness/liveness) для zero-downtime
  + observability (SLO/error budget — модуль 28) как критерий «катить дальше / откатить»
  + готовый rollback (предыдущий образ / переключение blue-green / гашение canary)
```

> Всё сходится в одном: **релиз должен быть обратимым**. Immutable-образы делают откат тривиальным (развернуть прошлую версию), стратегия выката ограничивает blast radius изменения (canary/rolling), observability даёт сигнал, что что-то не так (нарушение SLO/прожиг error budget — модули 27–28), а готовый rollback быстро возвращает рабочее состояние. Senior проектирует не только «как выкатить», но и «как мгновенно откатить» — это часть отказоустойчивости (модуль 24) на уровне поставки.

## ⚠️ Подводные камни

- **Нет health checks (readiness/liveness).** Трафик в неготовый инстанс → ошибки; зависшие не перезапускаются; нет zero-downtime.
- **Секреты в образе/гите.** Утечка образа = утечка секретов; инъекция в рантайме (модуль 30).
- **Нет стратегии отката.** Проблема в проде → долгий простой; готовить rollback (образ/blue-green/canary).
- **Ручной патчинг прода.** Конфигурационный дрейф, «снежинки», нет воспроизводимости; immutable + IaC.
- **Rolling без совместимости версий.** Во время выката сосуществуют v1/v2 (и схемы БД) — нужна обратная совместимость.
- **Canary без observability.** Не на что опереться при решении «расширять/откатывать»; нужны метрики/SLO (модуль 28).
- **Staging ≠ prod (нет паритета).** Тесты/нагрузка не переносятся (модуль 27).
- **Конфиг в образе, не в env.** Разные образы под среды → теряется переносимость и воспроизводимость (12-factor).
- **Нет security-скана в CI/CD.** Уязвимые зависимости/образы доезжают до прода; встроить скан (модуль 30).
- **Деплой = включение фичи.** Связывает два риска; разделять через feature flags.
