# Задача 01 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Соберите требования (FR/NFR) к URL-shortener и определите профиль нагрузки, который задаст весь дизайн.

## ВХОДНЫЕ ДАННЫЕ
Постановка от интервьюера размытая: «спроектируйте сервис коротких ссылок типа bit.ly». Нужно превратить её в чёткий scope до начала дизайна.

## ТРЕБОВАНИЯ
- Сформулируйте ядро FR (что система обязана делать) и опциональные FR (уточнить у интервьюера).
- Сформулируйте NFR в измеримых терминах.
- Определите ключевой профиль нагрузки (read/write-ratio) и объясните, почему он определяет дизайн.
- Перечислите 3–4 уточняющих вопроса по scope.

## ФОРМАТ РЕЗУЛЬТАТА
FR (ядро + опц.) → NFR (измеримые) → профиль нагрузки → уточняющие вопросы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ядро FR: сократить URL → код, редирект код → URL. Опц.: кастомный alias, TTL, аналитика.
- [ ] NFR измеримы: read-heavy (~100:1), низкая latency редиректа, высокая доступность, масштаб (млрд ссылок).
- [ ] Профиль: read/write ≈ 100:1 → дизайн упирается в масштаб чтений (кэш/реплики), а не записи.
- [ ] Объяснено, почему read-heavy и латентность редиректа определяют упор на кэш + реплики.
- [ ] 3–4 уточняющих вопроса (alias? TTL? аналитика? ожидаемый объём/масштаб?).

## ПОДСКАЗКА
Не прыгать в решение: сначала scope. Ядро — сокращение+редирект; alias/TTL/аналитика опциональны (спросить). Ключ — профиль read/write ≈ 100:1 и латентно-критичный редирект: это сразу диктует кэш+реплики, а не масштаб записи.
