# Задача 03 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Спроектируйте API URL-shortener и обоснуйте выбор кода редиректа (301 vs 302).

## ВХОДНЫЕ ДАННЫЕ
Нужны два эндпоинта: создание короткой ссылки (с опциональным alias/TTL) и редирект по коду. Продукт хочет считать клики (аналитику).

## ТРЕБОВАНИЯ
- Спроектируйте эндпоинт создания (метод, путь, тело, статус-код, ответ).
- Спроектируйте эндпоинт редиректа (метод, путь, поведение, коды ошибок).
- Сравните 301 vs 302 по влиянию на нагрузку и аналитику.
- Выберите код редиректа под требование «считать клики» и обоснуйте.

## ФОРМАТ РЕЗУЛЬТАТА
API создания → API редиректа → таблица 301 vs 302 → выбор + обоснование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Создание: POST /urls с телом {longUrl, alias?, ttl?} → 201 {shortUrl} (POST на коллекцию, без глагола в URL).
- [ ] Редирект: GET /{code} → 301/302 Location: longUrl; 404 если не найден/истёк.
- [ ] 301: браузер кэширует редирект → меньше хитов на сервер, но клики не доходят (нет аналитики).
- [ ] 302: каждый клик доходит до сервера → аналитика возможна, но выше нагрузка.
- [ ] Под «считать клики» выбран 302 (с обоснованием trade-off нагрузки).

## ПОДСКАЗКА
Создание — POST /urls (коллекция, не глагол), 201+shortUrl. Редирект — GET /{code}, 301/302, 404 при отсутствии. 301 кэшируется браузером (меньше нагрузки, нет аналитики), 302 — каждый клик на сервер (аналитика есть, нагрузка выше). Нужна аналитика → 302.
