# Задача 04 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Сравните подходы к генерации короткого кода и спроектируйте надёжное решение без коллизий и единой точки координации.

## ВХОДНЫЕ ДАННЫЕ
Нужен короткий код (как можно короче, но без коллизий) для миллиардов ссылок, при записи ~100/с (пик). Кандидаты: counter+base62, hash(URL)+срез, random+проверка, distributed ID (Snowflake), Key Generation Service.

## ТРЕБОВАНИЯ
- Посчитайте, сколько комбинаций даёт base62 при длине 7 символов, и хватит ли на масштаб.
- Сравните подходы по коллизиям, предсказуемости, координации.
- Объясните проблему hash (коллизии) и counter (предсказуемость/централизация).
- Выберите решение и обоснуйте (KGS как зрелый вариант).

## ФОРМАТ РЕЗУЛЬТАТА
Расчёт base62 → таблица сравнения подходов → выбор + обоснование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] base62: 62^7 ≈ 3.5 трлн комбинаций на 7 символов — с запасом на 6 млрд ссылок.
- [ ] Сравнение: counter (коротко, но предсказуем + точка координации), hash (коллизии → проверка/перехэш), random (коллизии растут с заполнением), Snowflake (уникален, без координации, но длиннее), KGS (уникальны by design).
- [ ] Объяснена проблема hash (разные URL → один срез) и counter (предсказуемость + централизация/SPOF).
- [ ] Выбор обоснован: KGS пред-генерирует пул уникальных кодов → нет коллизий и координации на горячем пути (модуль 14).
- [ ] (Опц.) Отмечено, что неугадываемость (если нужна) исключает counter.

## ПОДСКАЗКА
base62 (a-zA-Z0-9): 62^7 ≈ 3.5T — хватит. Counter предсказуем+централизован, hash даёт коллизии (проверка/перехэш), random — коллизии при заполнении. Зрелое решение — KGS: пред-генерирует пул уникальных кодов, раздаёт app-серверам → без коллизий и координации (модуль 14).
