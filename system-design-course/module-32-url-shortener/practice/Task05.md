# Задача 05 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Спроектируйте модель данных и выберите тип БД под паттерн доступа URL-shortener.

## ВХОДНЫЕ ДАННЫЕ
Основная операция — получить длинный URL по короткому коду (точечный lookup). Дополнительно: метаданные (дата создания, владелец, TTL, счётчик кликов). Масштаб — миллиарды записей, чтений тысячи/с.

## ТРЕБОВАНИЯ
- Опишите модель данных (ключ, поля).
- Определите паттерн доступа (по чему всегда идёт запрос).
- Сравните SQL vs KV под этот паттерн.
- Выберите тип БД и обоснуйте через query-driven подход (модуль 11).

## ФОРМАТ РЕЗУЛЬТАТА
Модель данных → паттерн доступа → SQL vs KV → выбор + обоснование.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Модель: short_code (PK/ключ) → long_url, created_at, expires_at?, owner?, clicks?.
- [ ] Паттерн доступа: всегда точечный lookup по ключу (code → url), без JOIN/связей/сложных запросов.
- [ ] SQL: сработает, но связи/транзакции не нужны; KV: идеально под lookup по ключу + горизонтальный масштаб.
- [ ] Выбор: key-value хранилище (модуль 11), обоснованный query-driven (от паттерна доступа к модели).
- [ ] Отмечено: выбор БД — следствие паттерна доступа, а не «по привычке SQL».

## ПОДСКАЗКА
Паттерн — точечный lookup code→url, без связей/JOIN. Это эталон key-value (модуль 11): простая модель, lookup по ключу, горизонтальный масштаб. SQL сработает, но его сильные стороны не нужны. Выбор БД — от паттерна доступа (query-driven).
