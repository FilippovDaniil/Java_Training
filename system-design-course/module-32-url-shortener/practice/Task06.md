# Задача 06 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Спроектируйте high-level архитектуру URL-shortener: компоненты и оба потока (создание и редирект).

## ВХОДНЫЕ ДАННЫЕ
Известно: read-heavy профиль (~100:1), KV-БД, Key Generation Service для кодов, требование низкой latency редиректа. Нужно собрать компоненты в связную схему.

## ТРЕБОВАНИЯ
- Перечислите компоненты (LB, app, cache, KV-БД, реплики, KGS).
- Опишите поток редиректа (GET /code) по шагам, включая cache hit/miss.
- Опишите поток создания (POST /urls) по шагам.
- Покажите, как архитектура отражает read-heavy профиль.

## ФОРМАТ РЕЗУЛЬТАТА
Список компонентов → поток редиректа → поток создания → связь с read-heavy + диаграмма.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Компоненты: LB (модуль 8), app-серверы, cache (Redis), KV-БД + реплики чтения, KGS.
- [ ] Редирект: GET /code → LB → app → cache (hit → редирект; miss → KV-БД → закэшировать → редирект); 404 при отсутствии.
- [ ] Создание: POST /urls → app → взять код у KGS → записать в БД → закэшировать → вернуть shortUrl.
- [ ] Архитектура отражает read-heavy: кэш перед БД + реплики чтения на горячем пути редиректа.
- [ ] Диаграмма со стрелками потоков (создание vs редирект).

## ПОДСКАЗКА
Компоненты: LB→app→cache(Redis)→KV-БД+реплики, отдельно KGS. Редирект (горячий): cache hit→редирект, miss→БД→закэшировать. Создание: код от KGS→запись в БД→кэш. Кэш+реплики перед БД отражают read-heavy профиль.
