# Задача 07 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Спроектируйте стратегию кэширования для горячего пути редиректа.

## ВХОДНЫЕ ДАННЫЕ
Чтения ~4000/с (пик ~10k/с), запись ~40/с. Распределение доступа неравномерно: небольшая доля «вирусных» ссылок даёт большую часть редиректов. Latency редиректа критична.

## ТРЕБОВАНИЯ
- Обоснуйте, почему кэш здесь обязателен (read-heavy + горячие ключи).
- Выберите, что кэшировать и какой паттерн записи/инвалидации (TTL, eviction).
- Прикиньте эффект кэша на нагрузку БД при разумном hit rate.
- Учтите согласованность кэш↔БД (что при обновлении/удалении ссылки).

## ФОРМАТ РЕЗУЛЬТАТА
Зачем кэш → что кэшируем + паттерн/eviction → эффект на БД (hit rate) → согласованность.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Обосновано: read-heavy + правило 80/20 (вирусные ссылки) → кэш резко снижает нагрузку на БД и latency (модуль 9).
- [ ] Кэшируется отображение code→long_url; cache-aside (lazy) + TTL; eviction LRU (горячие остаются).
- [ ] Эффект: при hit rate, например, 90% к БД доходит лишь ~10% чтений (~400/с вместо 4000/с).
- [ ] Согласованность: при удалении/смене ссылки или истечении TTL — инвалидация/обновление записи в кэше (иначе стейл-редирект).
- [ ] Связь с модулем 9 (паттерны/eviction/инвалидация, hot keys).

## ПОДСКАЗКА
Read-heavy + 80/20 (вирусные ссылки) → кэш обязателен (модуль 9). Кэшируем code→url, cache-aside + TTL, eviction LRU. При hit rate 90% к БД доходит ~10% чтений. Согласованность: инвалидировать кэш при удалении/смене ссылки и по TTL.
