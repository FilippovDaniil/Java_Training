# Задача 08 — Модуль 32: URL-shortener

## ЗАДАНИЕ
Найдите узкие места дизайна URL-shortener при росте и спроектируйте их устранение.

## ВХОДНЫЕ ДАННЫЕ
Система растёт: чтения и объём данных кратно увеличиваются, KGS становится критичным компонентом, продукт хочет надёжную аналитику кликов без замедления редиректа.

## ТРЕБОВАНИЯ
- Перечислите узкие места (чтения, KGS, объём БД, аналитика на горячем пути).
- Для каждого предложите решение со ссылкой на модуль.
- Объясните, почему KGS — потенциальный SPOF и как это закрыть.
- Опишите, как считать клики, не замедляя редирект.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «узкое место → решение → модуль» + разбор KGS-SPOF + аналитика async.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Чтения → кэш горячих кодов + реплики чтения (модули 9, 12).
- [ ] KGS → SPOF: пред-генерация пула кодов + HA-резерв (active-passive/несколько инстансов с непересекающимися диапазонами) (модули 14, 24).
- [ ] Объём БД → шардирование по short_code (hash-based) (модуль 13).
- [ ] Аналитика → 302 + async-запись событий кликов в очередь (не блокировать редирект) (модуль 15).
- [ ] Для каждого узкого места указан модуль; решения соразмерны (не «всё сразу»).

## ПОДСКАЗКА
Узкие места предсказуемы: чтения→кэш+реплики (9/12); KGS как SPOF→пред-генерация пула + HA с непересекающимися диапазонами (14/24); объём→шард по коду (13); аналитика→async-очередь, чтобы не тормозить редирект (15).
