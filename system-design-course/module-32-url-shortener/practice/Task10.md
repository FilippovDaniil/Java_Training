# Задача 10 — Модуль 32 (мини-проект): Полный HLD URL-shortener

## ЗАДАНИЕ
Соберите полный high-level design URL-shortener по фреймворку design-интервью. Синтез модуля.

## ВХОДНЫЕ ДАННЫЕ
Условие интервью: «Спроектируйте сервис коротких ссылок (TinyURL/bit.ly), масштаб ~100M новых URL/мес, read/write ~100:1, нужна аналитика кликов». 45–60 минут.

## ТРЕБОВАНИЯ
Дизайн-документ по шагам:
1. **Требования** — FR (ядро + опц.), NFR (измеримые), scope.
2. **Estimation** — запись/чтение (avg/peak), хранение 5 лет.
3. **API** — создание + редирект, 301 vs 302 (под аналитику).
4. **Модель данных и БД** — модель + выбор KV (обоснование).
5. **Генерация кода** — подход (KGS/base62), коллизии, длина.
6. **High-level** — компоненты + оба потока (диаграмма).
7. **Deep dive + узкие места** — кэш/реплики, KGS-SPOF, шардирование, аналитика async.

## ФОРМАТ РЕЗУЛЬТАТА
HLD из 7 разделов + диаграмма (LB→app→cache→KV-БД+реплики, KGS) + расчёты estimation.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Требования: FR (сокращение+редирект+аналитика), NFR (read-heavy, latency, HA), scope зафиксирован.
- [ ] Estimation прозрачен: запись ~40–100/с, чтение ~4–10k/с, ~3 ТБ/5лет.
- [ ] API: POST /urls + GET /code; выбран 302 под аналитику (с обоснованием).
- [ ] Модель данных + выбор KV (точечный lookup, query-driven).
- [ ] Генерация кода: KGS/base62, обсуждены коллизии и длина (62^7).
- [ ] High-level со схемой; оба потока (создание/редирект, cache hit/miss).
- [ ] Узкие места: чтения→кэш/реплики, KGS→HA/пул, объём→шард, аналитика→async; связи с модулями 9/11/12/13/14/15.

## ПОДСКАЗКА
Идите по фреймворку (модуль 03): требования→estimation→API→data→генерация кода→high-level→узкие места. Ключевые решения: read/write 100:1 → кэш+реплики; KV под lookup; KGS под коды (коллизии/координация); 302 под аналитику; узкие места — чтения/KGS/объём/аналитика. Дальше — Модуль 33: distributed KV-store изнутри.
