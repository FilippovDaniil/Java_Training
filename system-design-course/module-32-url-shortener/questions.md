## Ключ (эталонные ответы)

1. Ядро: сократить URL → код и редирект код → URL. Опциональны: кастомный alias, TTL/срок жизни, аналитика кликов (уточнить у интервьюера).
2. Read-heavy (~100:1 чтений к записям) и латентно-критичный редирект. Это сразу диктует упор на кэш и реплики чтения, а не на масштаб записи.
3. Запись: 100M/(30×86400) ≈ ~40/с, пик ×2–3 ≈ ~100/с. Чтение: ×100 → ~4000/с среднее, пик ~10k/с.
4. 100M×12×5 = 6 млрд записей; ~500 байт каждая → ≈3 ТБ (× replication factor).
5. POST /urls с телом {longUrl, alias?, ttl?} → 201 {shortUrl}; GET /{code} → 301/302 Location: longUrl (404, если код не найден/истёк).
6. 301 (Moved Permanently) браузер кэширует — последующие переходы не доходят до сервера (меньше нагрузки, но нет аналитики). 302 (Found) — каждый клик доходит до сервера (аналитика возможна, но выше нагрузка).
7. 302: тогда каждый клик доходит до сервера и его можно посчитать (ценой большей нагрузки, которую закрывают кэшем).
8. base62 = a–z A–Z 0–9 (62 символа), компактно и URL-безопасно. 62^7 ≈ 3.5 трлн комбинаций на 7 символов — с запасом на миллиарды ссылок.
9. Counter+base62 (последовательный ID), hash(URL)+срез, random+проверка занятости, distributed ID (Snowflake/ticket server), Key Generation Service (пред-генерация пула кодов).
10. Коллизии: разные URL могут дать одинаковый срез хэша → нужна проверка занятости и перехэширование/удлинение, что усложняет горячий путь.
11. Код предсказуем (последователен → можно перебирать чужие ссылки) и счётчик — точка координации/SPOF, плохо масштабируется в шардированной системе.
12. Отдельный сервис, заранее генерирующий пул уникальных кодов и раздающий их app-серверам пачками. Снимает коллизии (коды уникальны by design) и координацию на горячем пути создания.
13. Key-value хранилище: паттерн доступа — точечный lookup по ключу (code → url), без связей/JOIN/транзакций; KV даёт простую модель и горизонтальный масштаб. SQL сработает, но его сильные стороны не нужны.
14. Потому что система всегда запрашивает длинный URL по короткому коду (и наоборот при создании) — это обращение по ключу, без сложных запросов и связей. Выбор БД следует из паттерна (query-driven).
15. Read-heavy + правило 80/20 (вирусные ссылки дают большую часть трафика) → кэш резко снижает нагрузку на БД и latency редиректа. Кэшируют отображение code → long_url.
16. При hit rate H к БД доходит доля (1−H) чтений. Например, при 90% и 4000 чтений/с к БД идёт лишь ~400/с.
17. Инвалидировать/обновлять запись в кэше при удалении или изменении ссылки и использовать TTL; иначе возможен стейл-редирект на старый URL.
18. Кэш горячих кодов + реплики чтения (масштаб чтений), при росте объёма — шардирование по коду; LB перед app-серверами.
19. Если генерация кодов идёт через единственный KGS — его отказ останавливает создание ссылок. Закрывают пред-генерацией пула (app работает с запасом кодов) и HA-резервом (несколько инстансов с непересекающимися диапазонами).
20. Когда объём перестаёт умещаться/обслуживаться одним узлом — шардировать по short_code (hash-based) для равномерности; реплики на шард для чтений/доступности.
21. Использовать 302 и писать события кликов асинхронно — в очередь (модуль 15), а агрегировать в фоне. Это не блокирует горячий путь редиректа.
22. Скелет тот же (code → ресурс, KGS, кэш, KV для метаданных), но хранится текстовый фрагмент вместо URL; объём данных существенно больше, поэтому добавляется object storage и CDN.
23. Маленькие фрагменты — в БД; большие — в object/blob storage (модуль 10), а в KV-БД хранится метаданные (code → ссылка на блоб). Раздача больших/популярных — через CDN.
24. Проверять уникальность alias при создании (он занимает то же пространство имён, что и коды) и возвращать ошибку/предлагать другой при конфликте; генерируемые коды и alias не должны пересекаться.
25. Хранить expires_at в записи; при редиректе проверять срок (истёкший → 404) и использовать TTL в кэше; фоновая чистка/ленивое удаление просроченных записей.
