# Модуль 32 — URL-shortener / Pastebin (референсная архитектура)

## Зачем этот модуль

Части 1–7 дали строительные блоки и принципы. Часть 8 собирает их в **целостные дизайны реальных систем** по фреймворку design-интервью (модуль 03): требования → estimation → API → данные → high-level → deep dive → узкие места → trade-off. Первая система — **URL-shortener** (TinyURL/bit.ly): классическая «разминочная» задача интервью, на которой отрабатывается весь процесс на простом, но не тривиальном примере.

Зачем это senior'у. URL-shortener обманчиво прост («таблица из двух колонок»), но вскрывает ключевые решения: как **генерировать короткий код без коллизий и без единой точки координации** (модуль 14), как держать **read-heavy** нагрузку (кэш + реплики, модули 9, 12), какую **БД** выбрать (KV под профиль доступа, модуль 11) и где **узкие места** (модуль 26). Это эталон того, как раскладывать систему по шагам, а не прыгать в решение.

Он показывает полный HLD на простой системе: от требований и чисел до генерации ключей, кэширования и масштабирования; в конце — вариант Pastebin.

## 1. Требования (FR/NFR)

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Сократить длинный URL → короткий код | **Read-heavy** (~100:1 чтений к записям) |
| Редирект по короткому коду → исходный URL | Низкая latency редиректа (это «горячий путь») |
| (опц.) кастомный alias | Высокая доступность (битая ссылка = плохо) |
| (опц.) TTL/срок жизни ссылки | Масштаб (миллиарды ссылок) |
| (опц.) аналитика кликов | (опц.) код неугадываем (не перебрать) |

> Правильный старт — **уточнить scope** (модуль 04): сокращение + редирект — ядро; alias/TTL/аналитика — опциональны (уточнить у интервьюера). Ключевое наблюдение, определяющее весь дизайн: система **сильно read-heavy** (редиректов кратно больше, чем создания ссылок) и редирект — **латентно-критичный горячий путь**. Это сразу диктует упор на кэш и реплики чтения, а не на масштаб записи.

---

## 2. Оценка нагрузки и ёмкости (estimation)

```
Допущение: 100M новых URL/месяц.
Запись:  100M / (30 × 86400 с) ≈ 40 запросов/с (среднее), пик ×2–3 ≈ ~100/с.
Чтение:  read/write 100:1 → ~4000 чтений/с (среднее), пик ~10k/с.

Хранение (5 лет): 100M/мес × 12 × 5 = 6 млрд записей.
Размер записи ≈ short_code + long_url + метаданные ≈ ~500 байт.
Storage ≈ 6e9 × 500 B ≈ 3 ТБ (× replication factor).
```

> Estimation (модули 5–7) сразу отсекает паникёрские решения: **запись мизерна** (~40–100/с — одна машина справится), а **чтения** на 2 порядка выше (тысячи/с) — вот где нужен масштаб (кэш + реплики). Объём (~3 ТБ за 5 лет) умещается, но при росте — кандидат на шардирование. Главное число, формирующее дизайн: **read/write ≈ 100:1**.

## 3. API

```
POST /urls
  body: { "longUrl": "...", "alias": "my-link"?, "ttl": 86400? }
  → 201 { "shortUrl": "https://sho.rt/aB3xZ9" }

GET /{code}
  → 301 или 302 Location: <longUrl>   (редирект)
  → 404 если код не найден/истёк
```

> API минимален (модуль REST-соглашений): **POST /urls** создаёт (201 + короткий URL), **GET /{code}** редиректит. Создание — POST на коллекцию (не глагол в URL). Кастомный alias и TTL — опциональные поля тела. Важное проектное решение — **код редиректа** (см. §5): 301 (permanent) или 302 (found/temporary) — влияет на аналитику и нагрузку.

---

## 4. Модель данных и выбор БД

```
Запись:  short_code (PK) → long_url, created_at, expires_at?, owner?, clicks?
```

| Вариант | Подходит? |
|---------|-----------|
| Реляционная (SQL) | можно, но связи/JOIN/транзакции не нужны |
| **Key-Value (NoSQL)** | **идеально**: запрос всегда «по ключу» (code→url), огромный масштаб чтений, простая модель |

> Паттерн доступа — **точечный lookup по ключу** (`code → long_url`), без связей, JOIN и сложных запросов. Это эталонный кейс для **key-value хранилища** (модуль 11): простая модель, горизонтальное масштабирование, быстрый доступ по ключу. Реляционная БД сработает, но её сильные стороны (связи, транзакции) здесь не нужны. Выбор БД — следствие паттерна доступа (query-driven, модуль 11).

## 5. Генерация короткого кода (ключевое решение)

Короткий код — обычно **base62** (`a–z A–Z 0–9` = 62 символа): `62^7 ≈ 3.5 трлн` комбинаций на 7 символов.

| Подход | Плюсы / минусы |
|--------|----------------|
| **Counter + base62** | счётчик → base62; коротко и без коллизий, но **последователен/предсказуем** и счётчик — точка координации |
| **Hash(URL) + срез** | хэшировать URL, взять N символов; **коллизии** (разные URL → один срез) → проверять и перехэшировать |
| **Random + проверка** | случайный код + проверка занятости; коллизии редки, но растут с заполнением |
| **Distributed ID** (Snowflake/ticket) | уникальный ID без единой точки (модуль 14) → base62 |
| **Key Generation Service (KGS)** | **пред-генерирует** уникальные коды пачками, раздаёт app-серверам |

```
301 vs 302 (важно для аналитики!):
  301 Moved Permanently → браузер кэширует редирект → меньше хитов на сервер, НО нет аналитики кликов.
  302 Found (temporary)  → каждый клик доходит до сервера → можно считать клики, но выше нагрузка.
```

> Генерация кода — **сердце дизайна**. Наивный hash даёт коллизии (нужна проверка/перехэш), counter предсказуем и централизован. Зрелое решение — **Key Generation Service** (модуль 14): отдельный сервис заранее генерирует пул уникальных кодов и раздаёт их app-серверам пачками. Это снимает коллизии (коды уникальны by design) и координацию на горячем пути (app берёт готовый код). Выбор **301 vs 302** — продуктовое решение с архитектурным эффектом: 301 разгружает сервер ценой потери аналитики, 302 включает аналитику ценой нагрузки.

## 6. High-level дизайн

```
                ┌────────── cache (Redis): code → long_url (hot) ──────────┐
                ▼                                                            │
Client → [LB] → [App-серверы] ──(miss)──► [KV-БД: code → long_url] ◄─ реплики чтения
   │             │   ▲
 GET /code       │   └── при создании: получить код (KGS) → записать в БД → закэшировать
 POST /urls ─────┘
                                    [KGS] пред-генерирует пул кодов
```

> Поток **редиректа** (горячий путь): `GET /code` → LB → app → **cache** (hit → редирект; miss → KV-БД → закэшировать → редирект). Поток **создания**: `POST /urls` → app → взять код у **KGS** → записать в БД → закэшировать. Архитектура прямо отражает read-heavy профиль: кэш перед БД (модуль 9), реплики чтения (модуль 12), LB (модуль 8). Создание редко и просто.

## 7. Deep dive, узкие места и масштабирование

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| Латентность редиректа / нагрузка чтений | кэш горячих кодов + реплики чтения | 9, 12 |
| KGS как единая точка | пред-генерация пула + резервирование KGS (HA) | 14, 24 |
| Объём данных растёт | шардирование по `short_code` (hash-based) | 13 |
| Коллизии кодов | KGS (уникальны by design) / проверка+перехэш | 14 |
| Аналитика кликов | 302 + async-запись событий в очередь (не блокировать редирект) | 15 |

**Pastebin-вариант:** то же, но хранится **текстовый блоб** (а не URL). Маленькие — в БД, большие — в **object/blob storage** (модуль 10), метаданные (code→ссылка на блоб) в KV-БД; раздача через CDN. Генерация кода — та же (KGS).

> Узкие места предсказуемы: **чтения** (закрываются кэшем + репликами), **KGS** (единая точка → пред-генерация + HA-резерв, модули 14/24), **объём** (шардирование по коду, модуль 13). Аналитику кликов выносят в **async** (очередь, модуль 15), чтобы не замедлять редирект. **Pastebin** — почти тот же дизайн с заменой «короткий URL» на «ссылка на текстовый блоб в object storage» (модуль 10) + CDN для раздачи. Это демонстрирует переиспользование одного скелета под родственные задачи.

## ⚠️ Подводные камни

- **Прыжок в решение без уточнения scope.** Сначала FR/NFR (alias/TTL/аналитика опц.), read/write-ratio (модуль 03/04).
- **Игнор read-heavy профиля.** Без кэша/реплик горячий путь редиректа не держит нагрузку.
- **Hash без обработки коллизий.** Разные URL → один срез; нужна проверка/перехэш или KGS.
- **Counter как ID в шардах.** Предсказуем и централизован (точка координации/SPOF); KGS/Snowflake (модуль 14).
- **KGS без HA.** Единая точка генерации → SPOF; резервировать + пред-генерировать пул (модуль 24).
- **301 при нужной аналитике.** Браузер кэширует редирект → клики не доходят; для аналитики — 302.
- **Аналитика синхронно в редиректе.** Замедляет горячий путь; писать async через очередь (модуль 15).
- **Реляционная БД «по привычке».** Паттерн — lookup по ключу; KV проще и масштабируемее (модуль 11).
- **Pastebin: большой текст в БД.** Большие блобы — в object storage + CDN (модуль 10), не в строке БД.
