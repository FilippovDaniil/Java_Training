# Задача 01 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Сформулируйте требования к распределённому KV-хранилищу и обоснуйте, почему KV-модель выбрана и когда она уместна.

## ВХОДНЫЕ ДАННЫЕ
Нужно хранилище для огромного масштаба (петабайты, тысячи узлов) с высокой доступностью и низкой latency: сессии, профили, каталог, кэш-подобные данные. Доступ — по ключу, без запросов по значению и связей.

## ТРЕБОВАНИЯ
- Сформулируйте FR (операции) и NFR (масштаб, доступность, latency, согласованность).
- Объясните, почему минимальная KV-модель позволяет неограниченный горизонтальный масштаб.
- Укажите, когда KV уместен, а когда нет (нужны JOIN/транзакции/запросы по значению).
- Назовите ключевой NFR Dynamo-стиля и его следствие (AP).

## ФОРМАТ РЕЗУЛЬТАТА
FR/NFR → почему KV масштабируется → когда уместен/нет → ключевой NFR + следствие.

## КРИТЕРИИ ПРИЁМКИ
- [ ] FR: get/put/delete по ключу; нет запросов по значению/JOIN.
- [ ] NFR: огромный масштаб, высокая доступность (always-writeable), низкая latency p99, горизонтальный масштаб, настраиваемая согласованность.
- [ ] Объяснено: простота модели (нет связей/JOIN/межузловых транзакций) убирает координацию → неограниченный горизонтальный масштаб.
- [ ] Когда уместен: доступ по ключу, нужен масштаб/доступность; когда нет: нужны связи/транзакции/запросы по атрибутам.
- [ ] Ключевой NFR — высокая доступность + настраиваемая согласованность → AP-выбор (модуль 21).

## ПОДСКАЗКА
KV: get/put/delete по ключу, без запросов по значению/JOIN — эта минимальность убирает межузловую координацию → масштаб без границ. Уместен при доступе по ключу + потребности в масштабе/доступности; не уместен, где нужны связи/транзакции. Ключевой NFR — HA + настраиваемая согласованность (AP).
