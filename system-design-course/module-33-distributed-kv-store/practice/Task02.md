# Задача 02 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Спроектируйте распределение данных по узлам через consistent hashing и обоснуйте virtual nodes.

## ВХОДНЫЕ ДАННЫЕ
Кластер из узлов разной мощности, который должен эластично расти/сжиматься. Наивный вариант — `hash(key) % N` — уже показал проблему: при изменении числа узлов почти все ключи переезжают.

## ТРЕБОВАНИЯ
- Объясните проблему `hash % N` и как consistent hashing её решает.
- Опишите размещение ключей и узлов на кольце и определение владельца ключа.
- Обоснуйте virtual nodes (vnodes): неравномерность + узлы разной мощности + rebalancing.
- Объясните, почему не нужен центральный каталог размещения.

## ФОРМАТ РЕЗУЛЬТАТА
Проблема hash%N → кольцо (размещение/владелец) → vnodes (зачем) → отсутствие центрального каталога.

## КРИТЕРИИ ПРИЁМКИ
- [ ] hash%N: смена N пересортирует почти все ключи; consistent hashing перемещает лишь долю (модуль 14).
- [ ] Кольцо: ключи и узлы хэшируются на одно кольцо; владелец ключа — первый узел по часовой стрелке.
- [ ] vnodes: каждый физический узел = много точек на кольце → равномерность, учёт разной мощности (больше vnodes мощным), плавный rebalancing.
- [ ] Центральный каталог не нужен: ключ детерминированно отображается на узел по хэшу.
- [ ] Связь с модулем 14.

## ПОДСКАЗКА
hash%N перетасует всё при смене N → consistent hashing (кольцо): ключ → первый узел по часовой; добавление/удаление двигает лишь долю. vnodes (узел = много точек) дают равномерность, учёт мощности и плавный rebalancing. Размещение детерминировано по хэшу — без центрального каталога.
