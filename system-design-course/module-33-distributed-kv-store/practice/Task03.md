# Задача 03 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Спроектируйте репликацию: preference list, число реплик N и выбор sync/async.

## ВХОДНЫЕ ДАННЫЕ
Нужно обеспечить durability и доступность чтения при отказах узлов/стоек. Узлы расположены в нескольких стойках/зонах. Требуется баланс между durability и latency записи.

## ТРЕБОВАНИЯ
- Опишите, как формируется preference list для ключа (на основе кольца).
- Объясните роль N (replication factor) и почему реплики кладут на разные физические узлы/стойки.
- Сравните sync vs async репликацию (durability vs latency, модуль 12).
- Свяжите репликацию с независимостью отказов (модуль 24).

## ФОРМАТ РЕЗУЛЬТАТА
Preference list → роль N → sync vs async → независимость отказов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Preference list: владелец ключа + следующие (N−1) узла по кольцу; пропуск повторных физических узлов.
- [ ] N — число реплик ключа; реплики на разных узлах/стойках → переживание отказа узла/стойки.
- [ ] sync: выше durability, но выше latency; async: ниже latency, но риск потери незареплицированного хвоста (модуль 12).
- [ ] Реплики на независимых точках отказа (разные стойки/зоны) → отказ одной не теряет все копии (модуль 24).
- [ ] Отмечено: репликация + кворум (следующая задача) дают доступность и согласованность.

## ПОДСКАЗКА
Preference list = владелец + следующие N−1 узлов по кольцу (пропуская повторные физические узлы). N — число реплик; класть на разные стойки/зоны (независимость отказов, модуль 24). sync — durability↑/latency↑; async — latency↓/риск потери хвоста (модуль 12).
