# Задача 04 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Спроектируйте настраиваемую согласованность через кворум и подберите W/R под разные профили данных.

## ВХОДНЫЕ ДАННЫЕ
Хранилище с N=3 репликами обслуживает разные данные: (а) read-heavy справочник (нужны быстрые чтения, свежесть важна); (б) write-heavy лог событий (always-writeable, нужны быстрые записи); (в) сбалансированные данные, переживающие отказ одного узла со свежими чтениями.

## ТРЕБОВАНИЯ
- Сформулируйте правило кворума и почему `W + R > N` гарантирует свежесть чтения.
- Подберите W/R для (а), (б), (в) и обоснуйте.
- Объясните trade-off (latency/доступность записи vs чтения) через PACELC (модуль 21).
- Покажите, как один движок обслуживает разные требования настройкой W/R.

## ФОРМАТ РЕЗУЛЬТАТА
Правило кворума → таблица «профиль → W/R → обоснование» → trade-off (PACELC).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Правило: W+R>N → множества чтения и записи пересекаются → читатель захватывает ≥1 реплику со свежим значением.
- [ ] (а) read-heavy/свежесть → W=3, R=1 (быстрые свежие чтения, дороже записи).
- [ ] (б) write-heavy/always-writeable → W=1, R=3 (быстрые записи, слабее согласованность чтения).
- [ ] (в) баланс + переживание отказа узла → W=2, R=2 (W+R=4>3, кворумная согласованность).
- [ ] Trade-off объяснён через PACELC: настройка W/R = выбор latency vs consistency (модуль 21).
- [ ] Показано: один движок под разные профили настройкой W/R.

## ПОДСКАЗКА
W+R>N → пересечение множеств → свежее чтение. N=3: read-heavy/свежесть → W=3,R=1; write-heavy/always-writeable → W=1,R=3; баланс+переживание отказа узла → W=2,R=2. Настройка W/R = выбор latency vs consistency (PACELC, модуль 21).
