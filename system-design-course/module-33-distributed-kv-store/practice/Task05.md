# Задача 05 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Спроектируйте разрешение конфликтов при расхождении реплик и выберите механизм под кейс.

## ВХОДНЫЕ ДАННЫЕ
При async-репликации и W<N реплики расходятся. Два кейса: (а) пользовательская настройка (последнее значение должно победить, простота важнее); (б) корзина покупок (нельзя терять добавленные товары при конкурентных изменениях).

## ТРЕБОВАНИЯ
- Объясните, откуда берутся конфликты (AP + async + W<N).
- Сравните механизмы: LWW, vector clocks, read-repair, application merge/CRDT.
- Подберите механизм для (а) и (б) с обоснованием.
- Укажите проблему LWW при clock skew.

## ФОРМАТ РЕЗУЛЬТАТА
Источник конфликтов → сравнение механизмов → выбор для (а)/(б) → проблема LWW/clock skew.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Источник: AP-выбор + async-репликация + W<N → реплики временно расходятся.
- [ ] LWW: по timestamp побеждает последняя (просто, но теряет данные); vector clocks: отслеживают причинность, выявляют конкурентные версии; read-repair: чинит расхождения при чтении; CRDT/merge: тип данных/приложение сливает версии.
- [ ] (а) настройка → LWW (последнее побеждает, простота).
- [ ] (б) корзина → vector clocks + merge / CRDT (объединение, чтобы не терять товары).
- [ ] Проблема LWW: при clock skew (рассинхрон часов, модуль 14/22) «последняя» определяется неверно → потеря записи.
- [ ] Read-repair упомянут как попутное лечение расхождений.

## ПОДСКАЗКА
Конфликты от AP+async+W<N. LWW — последняя по timestamp (просто, теряет данные, опасно при clock skew). Vector clocks — причинность + конкурентные версии → merge/CRDT (корзина = объединение). Read-repair чинит при чтении. Настройка → LWW; корзина → vector clocks/CRDT.
