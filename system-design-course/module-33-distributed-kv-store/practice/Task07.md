# Задача 07 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Выберите storage engine (LSM-tree vs B-tree) под профиль нагрузки KV-хранилища и обоснуйте.

## ВХОДНЫЕ ДАННЫЕ
Два сценария: (а) write-heavy хранилище (телеметрия/события, огромный поток записей, чтения преимущественно по ключу); (б) read-heavy с диапазонными запросами (часто читают отсортированные диапазоны ключей).

## ТРЕБОВАНИЯ
- Кратко опишите принцип LSM-tree и B-tree.
- Сопоставьте их сильные стороны (запись vs чтение/диапазоны).
- Выберите движок для (а) и (б) с обоснованием.
- Укажите, почему Dynamo-style хранилища обычно берут LSM.

## ФОРМАТ РЕЗУЛЬТАТА
Принцип LSM/B-tree → сильные стороны → выбор для (а)/(б) → почему LSM в Dynamo-style.

## КРИТЕРИИ ПРИЁМКИ
- [ ] LSM: запись в память + последовательный лог, фоновое слияние (compaction) → высокая пропускная записи.
- [ ] B-tree: обновление на месте, хорош для чтений и диапазонных сканов.
- [ ] (а) write-heavy → LSM (последовательная запись, высокая пропускная).
- [ ] (б) read-heavy/диапазоны → B-tree (или LSM с учётом диапазонов) — обоснование по чтениям/сканам.
- [ ] Dynamo-style обычно LSM: профиль write-heavy и масштаб записи; отмечен компромисс (read/space amplification от compaction).

## ПОДСКАЗКА
LSM — запись в память+лог, фоновое слияние → отлично для write-heavy. B-tree — обновление на месте → лучше для чтений/диапазонов. Write-heavy → LSM; read-heavy/диапазоны → B-tree. Dynamo-style берёт LSM (масштаб записи), цена — compaction (read/space amplification).
