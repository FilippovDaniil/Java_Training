# Задача 08 — Модуль 33: Distributed KV-store

## ЗАДАНИЕ
Опишите путь записи и чтения в KV-хранилище, где любой узел может быть координатором.

## ВХОДНЫЕ ДАННЫЕ
Кластер с consistent hashing, N=3, настраиваемый кворум. Клиент обращается к произвольному узлу. Нужно описать, что происходит при put и get, включая сбор кворума и разрешение конфликтов.

## ТРЕБОВАНИЯ
- Опишите путь PUT по шагам (хэш ключа → preference list → реплики → W ack).
- Опишите путь GET по шагам (чтение с R реплик → разрешение конфликтов → ответ).
- Объясните роль координатора и почему любой узел может им быть.
- Покажите, как кворум W/R встроен в путь запроса.

## ФОРМАТ РЕЗУЛЬТАТА
Путь PUT → путь GET → роль координатора → встроенный кворум.

## КРИТЕРИИ ПРИЁМКИ
- [ ] PUT: координатор хэширует ключ → определяет preference list → шлёт запись на N реплик → ждёт W подтверждений → отвечает клиенту.
- [ ] GET: координатор хэширует ключ → читает с R реплик → разрешает конфликты (versions/vector clocks, read-repair) → отвечает.
- [ ] Координатор — любой узел (получивший запрос); он маршрутизирует к preference list по хэшу ключа.
- [ ] Любой узел может координировать → нет центрального узкого места/SPOF.
- [ ] Кворум встроен: ответ после W (запись) / R (чтение); связь с W+R>N.

## ПОДСКАЗКА
PUT: хэш ключа → preference list → послать на N → ждать W ack → ответ. GET: хэш → читать R реплик → разрешить конфликты (read-repair) → ответ. Координатор — любой узел (по хэшу маршрутизирует), нет центра/SPOF. Кворум W/R встроен в ожидание ответов.
