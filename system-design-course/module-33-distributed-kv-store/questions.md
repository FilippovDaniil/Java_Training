## Ключ (эталонные ответы)

1. get(key), put(key,value), delete(key) — доступ только по ключу. Нет запросов по значению, JOIN, сложных запросов и (как правило) межузловых транзакций.
2. Потому что простота модели убирает межузловую координацию (нет JOIN/транзакций между узлами), которая иначе ограничивала бы масштаб. Данные независимо распределяются по узлам — можно добавлять узлы почти без предела.
3. Через consistent hashing: ключи и узлы отображаются на кольцо, владелец ключа — первый узел по часовой стрелке; виртуальные узлы выравнивают распределение.
4. `hash % N` при изменении числа узлов пересортировывает почти все ключи (массовое перемещение). Consistent hashing перемещает лишь долю ключей при добавлении/удалении узла.
5. Каждый физический узел представлен многими точками (vnodes) на кольце. Дают равномерное распределение, учёт узлов разной мощности (больше vnodes — мощным) и плавный rebalancing.
6. Список узлов, отвечающих за реплики ключа: владелец ключа плюс следующие узлы по кольцу (с пропуском повторных физических узлов).
7. Сколько всего реплик хранится у каждого ключа (N). Влияет на durability и доступность чтения.
8. Чтобы отказ одного узла/стойки/зоны не уничтожил все копии сразу — реплики должны быть на независимых точках отказа (модуль 24).
9. W — число реплик, подтверждающих запись; R — число читаемых реплик; при W + R > N множества чтения и записи пересекаются.
10. Потому что при пересечении множеств любое чтение захватывает хотя бы одну реплику, участвовавшую в последней успешной записи → видит свежее значение.
11. Read-heavy/свежесть: W=3, R=1 (быстрые свежие чтения). Write-heavy/always-writeable: W=1, R=3 (быстрые записи). Баланс с переживанием отказа узла: W=2, R=2.
12. Настройка W/R — это выбор точки в спектре согласованность↔доступность/latency: больше W/R — строже согласованность, выше latency/ниже доступность записи (PACELC: latency vs consistency); Dynamo-стиль обычно AP.
13. Из AP-выбора и асинхронной репликации при W<N: реплики временно расходятся, пока обновления распространяются, и могут быть конкурентные записи.
14. LWW выбирает «победителя» по timestamp (просто, но теряет проигравшую запись). Vector clocks отслеживают причинность и выявляют конкурентные (неупорядочиваемые) версии, которые сливает приложение.
15. «Последняя» определяется по часам узлов; при их рассинхроне (clock skew) более ранняя по реальному времени запись может получить больший timestamp и затереть более новую → потеря данных.
16. Механизм: при чтении обнаружены расходящиеся реплики → устаревшие обновляются актуальным значением на лету (попутное лечение расхождений).
17. Протокол, в котором узлы периодически обмениваются информацией о членстве и состоянии кластера, распространяя её эпидемически. Даёт децентрализованное знание о составе/сбоях без мастера (нет SPOF).
18. При недоступности целевой реплики запись принимает другой (соседний) узел и хранит «подсказку», доставляя данные целевому узлу, когда тот вернётся. Сохраняет доступность записи (always-writeable).
19. Фоновый процесс приведения реплик к согласию. Merkle-деревья (деревья хэшей) позволяют быстро сравнить реплики и найти именно расходящиеся диапазоны, не пересылая и не сравнивая все данные.
20. LSM-tree: запись в память + последовательный лог + фоновое слияние (высокая пропускная записи). B-tree: обновление на месте (хорош для чтений/диапазонов). Для write-heavy брать LSM.
21. Узел, обрабатывающий клиентский запрос: хэширует ключ, маршрутизирует к preference list, собирает W/R ответов. Любой узел может им быть, т.к. размещение определяется хэшом — нет центрального узла (нет SPOF).
22. Координатор хэширует ключ → определяет preference list → отправляет запись на N реплик → ждёт W подтверждений → отвечает клиенту (остальные реплики догоняют асинхронно).
23. Координатор хэширует ключ → читает с R реплик → разрешает конфликты (сравнивает версии/vector clocks, при необходимости read-repair) → возвращает значение.
24. Hot key: кэш перед стором, репликация горячего ключа, разбиение ключа. Tail latency: hedged requests (дублировать медленный запрос на другую реплику), таймауты (модуль 26).
25. Распределённый кэш — in-memory с eviction (LRU/LFU/TTL), данные эфемерны, цель — latency (ускоритель перед БД). Персистентный KV-store — durable источник правды с репликацией и согласованностью. Скелет (consistent hashing) общий, но кэш не гарантирует durability.
