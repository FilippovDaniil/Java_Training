# Модуль 33 — Distributed KV-store / распределённый кэш (референсная архитектура)

## Зачем этот модуль

В модуле 32 мы выбрали «key-value хранилище» как готовый блок. Этот модуль проектирует его **изнутри**: как построить распределённое KV-хранилище в стиле Amazon Dynamo / Cassandra — то, на чём держатся URL-shortener, сессии, кэши и каталоги по всему курсу. Это самая «распределённая» референсная система: она синтезирует consistent hashing (14), репликацию (12), кворумы и CAP (21), разрешение конфликтов (12) в один связный дизайн.

Зачем это senior'у. KV-store — это лакмус понимания распределённых систем. Здесь нет «спрятать за managed-сервис»: нужно самому решить, как **распределить данные** (consistent hashing), как **реплицировать** и какую **согласованность** дать (настраиваемый кворум W+R>N), как **разрешать конфликты** (vector clocks / LWW / read-repair), как **обнаруживать отказы** (gossip, hinted handoff, anti-entropy) и какой **storage engine** взять (LSM vs B-tree). Это разговор на уровне staff-инженера.

Он показывает полный дизайн Dynamo-style KV-хранилища: от модели и распределения данных до кворумов, конфликтов и членства кластера; в конце — вариант распределённого кэша.

## 1. Требования и почему KV

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| `get(key)` | Огромный масштаб (петабайты, тысячи узлов) |
| `put(key, value)` | Высокая доступность (always writeable) |
| `delete(key)` | Низкая latency (p99) |
| — (нет запросов по значению/JOIN) | Горизонтальный масштаб (добавил узел → больше ёмкости) |
| — | **Настраиваемая** согласованность (под данные) |

> KV-модель намеренно **минимальна** (get/put/delete по ключу, без запросов по значению и связей) — именно эта простота позволяет **горизонтально масштабироваться без границ** (нет JOIN/транзакций между узлами, которые требовали бы координации). Ключевой NFR Dynamo-стиля — **высокая доступность и настраиваемая согласованность**: система должна принимать записи даже при отказах и разделениях (AP-выбор, модуль 21), а строгость согласованности — настраиваться под данные.

---

## 2. Распределение данных: consistent hashing

```
Кольцо (модуль 14): ключи и узлы хэшируются на одно кольцо.
  key → hash(key) → первый узел по часовой стрелке → его «владелец».
  vnodes (виртуальные узлы): каждый физический узел = много точек на кольце
    → равномерность + учёт разной мощности + плавный rebalancing.
```

> Данные распределяются через **consistent hashing** (модуль 14): добавление/удаление узла перемещает лишь долю ключей (а не пересортировывает всё, как `hash % N`). **Виртуальные узлы (vnodes)** решают неравномерность и позволяют узлам разной мощности нести пропорциональную долю. Это фундамент эластичности: кластер растёт/сжимается, перемещая минимум данных. Любой ключ детерминированно отображается на узел-владельца — не нужен центральный каталог размещения.

---

## 3. Репликация и preference list

```
Реплицируем каждый ключ на N узлов (replication factor):
  владелец ключа + следующие (N−1) узла по кольцу = preference list.
  (пропускаем повторные физические узлы, чтобы реплики были на разных машинах/стойках)
```

| Параметр | Смысл |
|----------|-------|
| **N** | сколько реплик ключа всего |
| sync vs async | sync — durability/latency↑; async — latency↓, риск потери хвоста (модуль 12) |
| preference list | список узлов, отвечающих за реплики ключа |

> Каждый ключ реплицируется на **N узлов** — это владелец плюс следующие по кольцу (**preference list**), с пропуском повторных физических узлов (чтобы реплики были на разных машинах/стойках — независимость отказов, модуль 24). N даёт durability и доступность чтения (модуль 12). Репликация по WAN/между стойками обычно асинхронна (latency), отсюда — возможные расхождения реплик, которые разрешаются кворумом и conflict resolution (§4–5).

---

## 4. Настраиваемая согласованность: кворум W+R>N

```
N — число реплик, W — сколько подтверждают запись, R — сколько читаем.
  W + R > N  → читающее и пишущее множества ПЕРЕСЕКАЮТСЯ → читатель видит свежую запись.

Примеры (N=3):
  W=3, R=1 → быстрые чтения, медленные/менее доступные записи (read-heavy)
  W=1, R=3 → быстрые записи, медленные чтения (write-heavy, always-writeable)
  W=2, R=2 → сбалансированно, переживает отказ 1 узла (кворумная согласованность)
```

> **Кворум** даёт **настраиваемую согласованность** (модуль 21): правило `W + R > N` гарантирует пересечение множеств чтения и записи → читатель захватит хотя бы одну реплику со свежим значением. Крутя W и R, систему настраивают под данные: `W=2,R=2,N=3` — баланс с переживанием отказа одного узла; `R=1` — быстрые чтения; `W=1` — always-writeable (но слабее согласованность). Это и есть «настраиваемость» Dynamo: один движок обслуживает и строгие, и доступностно-ориентированные сценарии — выбор по PACELC (latency vs consistency, модуль 21).

## 5. Разрешение конфликтов

При async-репликации и `W<N` реплики расходятся → нужно разрешать конфликты:

| Механизм | Как |
|----------|-----|
| **Last-Write-Wins (LWW)** | по timestamp выигрывает последняя запись (просто; теряет данные при clock skew) |
| **Version vectors / vector clocks** | отслеживают причинность; обнаруживают конкурентные версии (модуль 12) |
| **Read-repair** | при чтении обнаружены расходящиеся реплики → обновить устаревшие на лету |
| **Application merge / CRDT** | приложение/тип данных сливает версии (корзина = объединение) |

> Цена доступности (AP) — **временная несогласованность реплик**, которую надо разрешать. **LWW** прост, но при clock skew (модуль 14/22) теряет записи. **Vector clocks** честно отслеживают причинность и выявляют **конкурентные** версии (которые нельзя автоматически упорядочить) — их сливает приложение или **CRDT** (например, корзина = объединение добавлений). **Read-repair** чинит расхождения попутно при чтении. Выбор механизма — продуктовое решение: «потерять запись при гонке» (LWW) vs «слить версии» (vector clocks + merge).

## 6. Членство кластера и обнаружение отказов

| Механизм | Назначение |
|----------|------------|
| **Gossip-протокол** | узлы периодически обмениваются информацией о членстве/состоянии → децентрализованно, без мастера |
| **Hinted handoff** | целевой узел временно недоступен → запись принимает другой узел и хранит «подсказку», доставляет позже |
| **Anti-entropy (Merkle-деревья)** | фоновая сверка реплик: Merkle-дерево быстро находит расходящиеся диапазоны для синхронизации |

> Кластер без мастера узнаёт о составе и сбоях через **gossip** (эпидемическое распространение — нет SPOF-координатора, модуль 22). **Hinted handoff** держит систему **always-writeable**: если реплика недоступна, запись временно принимает сосед и «подсказкой» досылает её, когда узел вернётся (повышает доступность записи). **Anti-entropy** с **Merkle-деревьями** периодически приводит реплики к согласию в фоне: дерево хэшей позволяет логарифмически быстро найти именно расходящиеся диапазоны, не сравнивая все данные. Вместе это даёт самовосстановление без централизованного управления.

## 7. Storage engine, путь запроса и узкие места

**Storage engine:** **LSM-tree** (log-structured merge) — записи в память + последовательный лог, фоновое слияние; оптимален для **write-heavy** (последовательная запись, высокая пропускная). **B-tree** — лучше для read-heavy/диапазонных чтений. Dynamo-style обычно берёт LSM.

```
Путь запроса (любой узел = координатор):
  PUT key,val → hash(key) → preference list → послать на N реплик → ждать W ack → ответ.
  GET key     → hash(key) → читать с R реплик → разрешить конфликты (versions) → ответ.
```

| Узкое место | Решение |
|-------------|---------|
| **Hot key** (горячий ключ) | кэш перед стором, репликация хот-ключа, разбиение ключа |
| Rebalancing при росте | vnodes минимизируют перемещение (модуль 14) |
| **Tail latency** (p99) при чтении с R реплик | hedged requests (послать дубль на 2-ю реплику) (модуль 26) |

**Вариант: распределённый кэш** (Redis/Memcached-кластер) — тот же скелет (consistent hashing для шардирования), но **in-memory**, с **eviction** (LRU/LFU/TTL, модуль 9) вместо durability; согласованность слабее, цель — latency. Это «KV без персистентности» как ускоритель перед БД.

> Путь запроса децентрализован: **любой узел — координатор** (хэширует ключ, обращается к preference list, собирает W/R ответов) — нет узкого центра. **LSM-tree** под write-heavy (последовательная запись), **B-tree** под чтения/диапазоны. Узкие места: **hot keys** (кэш/репликация/разбиение), **tail latency** (hedged requests, модуль 26), **rebalancing** (сглажен vnodes). **Распределённый кэш** — упрощённый родственник: тот же consistent hashing, но in-memory + eviction вместо durability (модуль 9).

## ⚠️ Подводные камни

- **`hash % N` вместо consistent hashing.** Добавление узла пересортирует все ключи; нужен hash ring + vnodes (модуль 14).
- **Игнор vnodes.** Неравномерное распределение и узлы разной мощности; vnodes выравнивают.
- **Кворум без W+R>N.** Множества не пересекаются → читатель может не увидеть свежую запись.
- **LWW при clock skew.** Теряет записи при рассинхроне часов; vector clocks для причинности.
- **Не разрешать конфликты.** AP + async → расхождения реплик; нужны read-repair / vector clocks / merge.
- **Централизованный координатор членства.** SPOF; gossip децентрализует.
- **Нет hinted handoff.** Падает доступность записи при недоступной реплике.
- **Без anti-entropy.** Реплики «дрейфуют» без фоновой синхронизации (Merkle).
- **B-tree под write-heavy.** Хуже LSM по пропускной записи; для записей — LSM.
- **Hot key без защиты.** Один ключ перегружает узел; кэш/репликация/разбиение (модуль 26).
- **Путать distributed cache и persistent KV.** Кэш — in-memory + eviction (latency), стор — durable.
