# Задача 05 — Модуль 34: Newsfeed

## ЗАДАНИЕ
Спроектируйте модель данных ленты: посты, граф подписок, feed cache.

## ВХОДНЫЕ ДАННЫЕ
Гибридная лента (push + pull). Нужно хранить посты, связи подписок (для доставки и для подтягивания постов celebrity) и предвычисленные ленты. Масштаб — сотни миллионов пользователей.

## ТРЕБОВАНИЯ
- Опишите хранение постов (модель, по чему доступ).
- Опишите граф подписок (почему нужны обе стороны связи).
- Опишите feed cache (что хранит, где, ограничение размера).
- Объясните, почему в feed cache хранят post_id (ссылку), а не копию контента.

## ФОРМАТ РЕЗУЛЬТАТА
Посты → граф подписок (обе стороны) → feed cache → почему ссылки, а не копии.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Посты: post_id → author_id, content, created_at; доступ по post_id (модуль 11).
- [ ] Граф подписок: нужны и «на кого подписан» (подтянуть посты), и «кто подписчики» (кому доставлять при push); при масштабе — отдельное/шардированное хранилище.
- [ ] Feed cache: user_id → список post_id (предвычисленная лента) в Redis (модуль 9); хранить окно N постов.
- [ ] В feed cache — post_id (ссылки), контент хранится отдельно по id; копирование контента в каждую ленту раздуло бы память.
- [ ] Отмечено: один пост попадает в ленты многих → дедупликация по ссылке.

## ПОДСКАЗКА
Посты: post_id→author/content/created_at. Граф: обе стороны (подписки — для pull, подписчики — для push-доставки), при масштабе шардировать. Feed cache: user_id→[post_id] в Redis, окно N. Хранить post_id (ссылки), не копии контента — иначе память раздувается (один пост в лентах многих).
