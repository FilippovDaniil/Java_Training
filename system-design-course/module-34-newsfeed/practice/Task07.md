# Задача 07 — Модуль 34: Newsfeed

## ЗАДАНИЕ
Спроектируйте fan-out через асинхронную очередь и обеспечьте надёжность доставки.

## ВХОДНЫЕ ДАННЫЕ
Публикация поста требует разложить его по лентам подписчиков (push). Раньше это делалось синхронно в обработчике POST → публикация тормозила, пики валили систему, при сбоях посты терялись/задваивались.

## ТРЕБОВАНИЯ
- Объясните, зачем выносить fan-out в очередь (модуль 15) и что это даёт.
- Опишите работу воркеров fan-out (что делают с событием).
- Обеспечьте надёжность: retry/at-least-once и идемпотентность (модуль 23), чтобы повтор не задвоил пост в ленте.
- Назовите цену подхода (eventual consistency / fan-out lag) и почему она приемлема.

## ФОРМАТ РЕЗУЛЬТАТА
Зачем очередь → работа воркеров → надёжность (retry+идемпотентность) → цена (lag).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Очередь развязывает публикацию и доставку: POST отвечает мгновенно, fan-out — в фоне; сглаживание пиков (load leveling); масштаб воркерами (модуль 15).
- [ ] Воркеры: берут событие «пост X автора A» → находят подписчиков → дописывают post_id в их feed cache (батчами; celebrity исключены).
- [ ] Надёжность: at-least-once доставка + идемпотентность (дедуп по post_id), чтобы retry не задвоил пост в ленте (модуль 23).
- [ ] Цена: eventual consistency — пост появляется с задержкой (fan-out lag); приемлемо для ленты (модуль 21).
- [ ] Отмечено сглаживание всплесков публикаций очередью.

## ПОДСКАЗКА
Очередь (модуль 15) развязывает POST и доставку: быстрый ответ, load leveling, масштаб воркерами. Воркеры раскладывают post_id по feed cache подписчиков. Надёжность: at-least-once + идемпотентность (дедуп по post_id, модуль 23), иначе retry задвоит. Цена — fan-out lag (eventual), для ленты ок.
