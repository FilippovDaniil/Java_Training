# Задача 08 — Модуль 34: Newsfeed

## ЗАДАНИЕ
Спроектируйте ранжирование ленты и кэширование, сравнив хронологию и ML-relevance.

## ВХОДНЫЕ ДАННЫЕ
Лента может показываться хронологически или ранжироваться по релевантности (engagement). Продукт хочет повысить вовлечённость, но не усложнять чрезмерно на старте. Чтение ленты — горячий путь, latency критична.

## ТРЕБОВАНИЯ
- Сравните хронологическое и ML-ранжирование (плюсы/минусы, сложность).
- Опишите feed cache: что хранит, ограничение размера, пагинация.
- Объясните, почему контент постов кэшируют отдельно от лент (по post_id).
- Дайте рекомендацию по эволюции ранжирования (с чего начать).

## ФОРМАТ РЕЗУЛЬТАТА
Хроно vs ML (сравнение) → feed cache (структура/окно) → отдельный кэш контента → рекомендация по эволюции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Хронологическое: просто/предсказуемо; ML/relevance: выше вовлечённость, но сложнее (фичи, модель, A/B).
- [ ] Feed cache: user_id → список post_id в памяти (Redis); хранить окно N постов, остальное — пагинацией из БД (модуль 9).
- [ ] Контент постов кэшируется отдельно по post_id (один пост в лентах многих → дедупликация по ссылке, не копия).
- [ ] Рекомендация: начать с хронологии (проще, быстрее запустить), затем эволюционировать к ML-ранжированию под вовлечённость.
- [ ] Учтено, что чтение — горячий путь (кэш обязателен).

## ПОДСКАЗКА
Хроно — просто/предсказуемо; ML — вовлечённее, но сложнее (фичи/модель/A-B). Feed cache: user_id→[post_id], окно N, пагинация из БД (модуль 9). Контент — отдельно по post_id (дедуп по ссылке). Начать с хронологии, эволюционировать к ML.
