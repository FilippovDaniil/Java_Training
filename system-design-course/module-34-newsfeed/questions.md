## Ключ (эталонные ответы)

1. Опубликовать пост; показать ленту постов тех, на кого подписан; ранжирование (хронология или relevance). Опционально — лайки, комментарии.
2. Потому что небольшая задержка появления поста в ленте (секунды) приемлема для пользователя — не нужна мгновенная согласованность; это AP-выбор (модуль 21), позволяющий асинхронную доставку.
3. Записи: 300M×2/86400 ≈ ~7000 постов/с. Чтения: 300M×10/86400 ≈ ~35000 чтений/с — лента read-heavy.
4. Fan-out — доставка одного поста многим подписчикам. Стержень, т.к. число доставок резко различается (средний ~200, celebrity ~десятки млн) и определяет, как генерировать ленту.
5. Стратегия, при которой пост при публикации сразу вписывается в предвычисленные ленты всех подписчиков. Чтение ленты быстрое, запись дорогая.
6. Стратегия, при которой лента собирается в момент чтения из постов тех, на кого подписан пользователь. Запись дешёвая, чтение дорогое.
7. Push: дорогая запись (N доставок), быстрое чтение (готовая лента). Pull: дешёвая запись (один пост), дорогое чтение (сборка из подписок на лету).
8. Celebrity (один пост → миллионы записей в ленты, всплеск/задержка) и трата ресурсов на доставку неактивным подписчикам.
9. Активный пользователь с тысячами подписок: сборка ленты из множества источников на лету даёт высокую latency чтения.
10. Проблема, что push для пользователя с десятками миллионов подписчиков порождает десятки миллионов записей на один пост — всплеск нагрузки и задержка доставки.
11. Применяет стратегию по типу автора: для обычных пользователей — push (предвычисление в ленты), для celebrity — pull (их посты не рассылаются, а подтягиваются при чтении ленты подписчика).
12. Как объединение предвычисленной части (push от обычных авторов из feed cache) и свежих постов celebrity (pull при чтении), которые сливаются и ранжируются.
13. По числу подписчиков выше определённого порога (или вручную помеченные/верифицированные аккаунты).
14. Посты: post_id → author_id, content, created_at (доступ по post_id). Граф подписок — отдельное (при масштабе шардированное/специализированное) хранилище связей «кто на кого подписан».
15. Нужны «на кого подписан пользователь» (чтобы подтянуть посты celebrity при pull) и «кто подписчики автора» (чтобы доставить пост при push).
16. Предвычисленная лента пользователя в памяти (Redis): user_id → список post_id. Делает чтение ленты быстрым (один запрос).
17. Потому что один пост попадает в ленты многих пользователей; хранение ссылок (post_id) вместо копий контента экономит память, контент хранится отдельно по id (дедупликация).
18. Чтобы развязать публикацию и доставку: POST отвечает мгновенно, fan-out идёт в фоне воркерами; очередь сглаживает всплески (load leveling) и масштабируется воркерами (модуль 15).
19. Потому что доставка at-least-once и retry при сбоях могут повторно обработать событие; без идемпотентности (дедуп по post_id) пост задвоится в ленте подписчика (модуль 23).
20. Задержка появления поста в лентах из-за асинхронной фоновой доставки. Приемлем, т.к. лента допускает eventual consistency — пользователю не критично увидеть пост через секунды.
21. Хронологическое — новейшие сверху (просто, предсказуемо). ML/relevance — ранжирование по вероятности интереса (engagement, близость, свежесть): выше вовлечённость, но сложнее (фичи, модель, A/B).
22. Хранят ограниченное окно (например, последние N постов) в кэше, остальное подгружают пагинацией из БД; применяют TTL/eviction.
23. Контент кэшируют (горячий пост читают многие), медиа раздают через CDN/object storage (модуль 10), а не через сервис ленты; при необходимости — реплики/доп. кэш.
24. Шардированием графа (модуль 13) и/или специализированным хранилищем для обхода связей; обе стороны связи индексируются для push-доставки и pull-подтягивания.
25. Read-heavy + eventual → гибрид fan-out: push для обычных (предвычисленная лента в feed cache), pull для celebrity (подмешивание при чтении); доставка через асинхронную очередь с идемпотентными воркерами; контент по ссылкам + CDN для медиа; ранжирование от хронологии к ML; узкие места (celebrity, fan-out lag, hot content) закрываются гибридом, масштабом воркеров и кэшем.
