# Модуль 34 — Newsfeed / лента соцсети (референсная архитектура)

## Зачем этот модуль

Лента соцсети (Twitter/Facebook/Instagram) — одна из самых частых и поучительных задач system-дизайна, потому что она строится вокруг **одного фундаментального решения**: как генерировать ленту — **на запись (push)** или **на чтение (pull)**. Этот выбор и его гибрид определяют всю архитектуру и упираются в знаменитую **celebrity problem**. Модуль синтезирует fan-out (модули 5/26), очереди (15), кэш (9) и моделирование графа (11).

Зачем это senior'у. Newsfeed учит думать о **fan-out** — самой коварной оси масштаба: наивный push разваливается на пользователе с миллионами подписчиков, наивный pull — на пользователе, читающем ленту из тысяч источников. Senior знает trade-off push/pull, умеет вывести **гибрид** (push для большинства, pull для celebrity), спроектировать **fan-out через асинхронную очередь** и **feed cache**, и обсудить ранжирование (хронология vs ML). Это эталон задачи, где «правильного» ответа нет — есть осознанный компромисс.

Он показывает полный HLD ленты: требования, estimation, ключевую развилку push vs pull, celebrity problem и гибрид, fan-out-конвейер, кэш и ранжирование.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Опубликовать пост | **Read-heavy** (лент читают кратно чаще, чем постят) |
| Показать ленту постов тех, на кого подписан | Низкая latency загрузки ленты (горячий путь) |
| Ранжирование (хронология или relevance) | Высокая доступность |
| (опц.) лайки/комментарии | Eventual consistency допустима (пост может появиться с задержкой) |

```
Estimation (пример): 300M DAU, в среднем 2 поста/день, ~10 загрузок ленты/день.
  Записи постов: 300M×2 / 86400 ≈ ~7000 постов/с.
  Чтения ленты:  300M×10 / 86400 ≈ ~35000 чтений/с (read-heavy).
  Fan-out: пост среднего юзера (200 подписчиков) → 200 записей в ленты;
           пост celebrity (50M подписчиков) → 50M «доставок» — вот где боль.
```

> Лента — **read-heavy** с допустимой **eventual consistency** (если пост появится в ленте через секунды — это нормально; модуль 21). Estimation сразу высвечивает главную проблему — **fan-out**: средний пост порождает сотни доставок, а пост знаменитости — десятки миллионов. Именно асимметрия «обычный пользователь vs celebrity» делает задачу нетривиальной и определяет всю дальнейшую архитектуру.

## 2. Ключевая развилка: fan-out on write vs on read

| | Fan-out on write (**push**) | Fan-out on read (**pull**) |
|--|------------------------------|----------------------------|
| Когда работа | при **публикации**: вписать пост в ленты всех подписчиков | при **чтении**: собрать ленту из постов тех, на кого подписан |
| Чтение ленты | быстро (лента предвычислена) | медленно (собирать на лету) |
| Запись поста | дорого (N доставок, N=число подписчиков) | дёшево (одна запись поста) |
| Боль | **celebrity** (миллионы доставок); трата на неактивных | пользователь, читающий из тысяч источников |

```
Push:  POST → записать в feed каждого подписчика (предвычисленная лента) → GET ленты = просто чтение.
Pull:  POST → записать только пост → GET ленты = найти подписки → собрать их посты → отранжировать.
```

> Это **центральное решение** дизайна. **Push (fan-out on write)** делает чтение ленты мгновенным (она уже собрана), но запись дорогой — и катастрофически дорогой для celebrity (один твит → миллионы записей), плюс впустую работает для неактивных подписчиков. **Pull (fan-out on read)** делает запись дешёвой (просто сохранить пост), но чтение дорогим (собирать и ранжировать на лету) — плохо для активных пользователей с тысячами подписок. Ни один чистый подход не масштабируется на всех — отсюда гибрид (§3).

## 3. Celebrity problem и гибрид

```
Проблема: pop-star с 50M подписчиков постит → push = 50M записей в ленты (всплеск, перегрузка).
Решение — ГИБРИД:
  • Обычные пользователи → push (fan-out on write): их посты заранее раскладываются в ленты подписчиков.
  • Celebrity (много подписчиков) → pull (fan-out on read): их посты НЕ рассылаются;
    при чтении ленты подписчик дополнительно подтягивает свежие посты celebrity, на которых подписан.
  Итоговая лента = предвычисленная (push от обычных) + подмешанные посты celebrity (pull).
```

> **Celebrity problem** — почему чистый push не работает: рассылка одного поста миллионам — это всплеск нагрузки и задержка. **Гибрид** решает её, применяя стратегию по типу автора: для **обычных** пользователей — push (дёшево по числу подписчиков, быстрое чтение), для **celebrity** — pull (не рассылать миллионам, подтянуть при чтении). Лента читателя собирается как «предвычисленная часть (push) + свежие посты celebrity (pull)». Это классический ответ senior-уровня: не «push или pull», а «push для хвоста, pull для головы распределения».

## 4. Модель данных и high-level

```
Данные:
  posts:       post_id → author_id, content, created_at        (модуль 11)
  follow-граф: who follows whom (нужны и подписки, и подписчики)
  feed cache:  user_id → список post_id (предвычисленная лента, в Redis — модуль 9)

High-level (поток публикации, push):
  POST → Post Service → записать пост → положить событие в ОЧЕРЕДЬ (модуль 15)
       → Fan-out Service (воркеры) → для каждого подписчика дописать post_id в его feed cache.

High-level (поток чтения):
  GET feed → Feed Service → читать feed cache (push-часть)
           → подмешать свежие посты celebrity (pull-часть) → отранжировать → отдать.
```

> Модель: **посты**, **граф подписок** (нужны обе стороны — кому доставлять и чьи посты тянуть) и **feed cache** (предвычисленная лента в Redis, модуль 9). Ключевой приём — **fan-out через асинхронную очередь** (модуль 15): публикация не ждёт раскладки по подписчикам (это делают воркеры в фоне), поэтому POST быстрый, а доставка — eventual. Чтение читает готовый feed cache и подмешивает pull-часть от celebrity. Граф подписок при масштабе — отдельное хранилище (часто специализированное под обход связей, модуль 11).

## 5. Fan-out через асинхронную очередь

```
POST (быстрый ответ) → пост сохранён + событие "fan-out post X author A" в очередь.
Очередь → пул воркеров fan-out:
   найти подписчиков A → для каждого: feed_cache[follower] += post_id
   (батчами; celebrity исключены из push — их тянут на чтении)
```

| Зачем очередь (модуль 15) | Эффект |
|---------------------------|--------|
| Развязать публикацию и доставку | POST отвечает мгновенно, fan-out — в фоне |
| Сгладить всплески (load leveling) | пик публикаций не валит систему |
| Масштабировать воркерами | добавить воркеров под рост fan-out |
| Надёжность (retry/at-least-once) | доставка переживает сбои (идемпотентность — модуль 23) |

> **Fan-out выносят в асинхронную очередь** (модуль 15) — это превращает дорогую операцию «разложить пост по N лентам» из синхронной (блокирует публикацию) в фоновую (воркеры обрабатывают из очереди). Выгоды: быстрый POST, сглаживание пиков (load leveling), горизонтальное масштабирование воркерами, надёжность через retry (с идемпотентностью — модуль 23, чтобы повтор не задвоил пост в ленте). Цена — **eventual consistency**: пост появляется в лентах с небольшой задержкой (fan-out lag), что для ленты приемлемо.

## 6. Ранжирование и кэширование

| Ранжирование | Суть |
|--------------|------|
| **Хронологическое** | новейшие сверху; просто, предсказуемо |
| **Relevance / ML** | ранжировать по вероятности интереса (engagement, близость, свежесть) — сложнее, но вовлечённее |

**Feed cache** (модуль 9): предвычисленная лента (`user_id → список post_id`) в памяти (Redis) — горячий путь чтения. Хранят ограниченный «хвост» (например, последние N постов), старое — догружать из БД. Сам контент постов — отдельно (по post_id), часто тоже в кэше.

> Два слоя оптимизации чтения. **Ранжирование**: хронология проста, но соцсети идут к **ML-ранжированию** (relevance) — больше вовлечённость ценой сложности (нужны фичи, модель, A/B). **Feed cache** — то, что делает ленту быстрой: лента предвычислена и лежит в памяти, читается за один запрос; хранят ограниченное окно (пагинация догружает остальное из БД). Контент постов кэшируют отдельно по id (один пост — в лентах многих, дедупликация по ссылке, а не копированием).

## 7. Узкие места и масштабирование

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Celebrity fan-out** | гибрид (pull для celebrity, не рассылать миллионам) | §3 |
| **Fan-out lag** (отставание доставки) | масштабировать воркеры; приоритет активным; eventual ок | 15 |
| **Hot users / горячие посты** | кэш контента, реплики, CDN для медиа | 9, 10 |
| **Размер feed cache** | хранить окно (N постов), остальное из БД; TTL | 9 |
| **Tail latency сборки ленты** | ограничить fan-out источников, hedged requests | 26 |
| Граф подписок на масштабе | шардирование/специализированное хранилище | 11, 13 |

> Главное узкое место — **celebrity fan-out** (решён гибридом). Дальше: **fan-out lag** (доставка отстаёт на пике — масштабировать воркеры, приоритизировать активных пользователей, eventual допустима), **hot content** (вирусный пост — кэш + CDN для медиа, модуль 10), **размер feed cache** (хранить окно + пагинация), **tail latency** при сборке pull-части (модуль 26). Граф подписок при огромном масштабе шардируют (модуль 13). Все узкие места — продолжение fan-out-проблемы в разных проявлениях.

## ⚠️ Подводные камни

- **Чистый push без учёта celebrity.** Один пост знаменитости → миллионы записей → всплеск/перегрузка; нужен гибрид.
- **Чистый pull для активных пользователей.** Сборка ленты из тысяч подписок на лету → высокая latency чтения.
- **Синхронный fan-out в публикации.** POST блокируется на раскладке по N лентам; выносить в async-очередь (модуль 15).
- **Fan-out без идемпотентности.** Retry задвоит пост в ленте; idempotency/дедуп (модуль 23).
- **Копировать контент поста в каждую ленту.** Раздувает память; хранить post_id (ссылку), контент — отдельно.
- **Неограниченный feed cache.** Лента бесконечной длины в памяти; хранить окно N + пагинация из БД.
- **Push для неактивных подписчиков.** Трата на тех, кто не заходит; приоритизировать активных / lazy.
- **Strong consistency для ленты.** Не нужна и дорога; eventual (fan-out lag) приемлема (модуль 21).
- **Игнор медиа.** Картинки/видео — через CDN/object storage (модуль 10), не через сервис ленты.
