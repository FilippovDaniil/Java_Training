# Задача 01 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Соберите требования (FR/NFR) к мессенджеру и выделите свойство, отличающее его от REST-систем.

## ВХОДНЫЕ ДАННЫЕ
Постановка: «спроектируйте мессенджер типа WhatsApp». Нужно зафиксировать scope и понять, чем дизайн принципиально отличается от ранее разобранных систем (URL-shortener, лента).

## ТРЕБОВАНИЯ
- Сформулируйте ядро FR (1:1, группы, статусы, presence, история, offline) и опциональные.
- Сформулируйте NFR (latency real-time, доступность, порядок, durability, масштаб).
- Назовите свойство, отличающее чат от запрос-ответных систем.
- Объясните, почему это свойство ведёт к постоянным соединениям.

## ФОРМАТ РЕЗУЛЬТАТА
FR (ядро+опц.) → NFR → отличающее свойство → следствие (persistent-соединения).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ядро FR: 1:1 сообщения, групповые чаты, статусы (sent/delivered/read), presence, история, offline-доставка.
- [ ] NFR: низкая latency (real-time), высокая доступность, порядок в диалоге, durability сообщений, масштаб (млрд юзеров/млн соединений).
- [ ] Отличающее свойство: сервер должен сам push'ить сообщения получателю в реальном времени (не ждать его запроса).
- [ ] Объяснено: server push → нужно постоянное соединение (WebSocket), а не модель запрос-ответ.
- [ ] Отмечен масштаб одновременных соединений как ключевой вызов.

## ПОДСКАЗКА
Ядро: 1:1/группы/статусы/presence/история/offline. NFR: real-time latency, порядок, durability, масштаб соединений. Отличие от REST: сервер push'ит сообщения сам (real-time), а не отвечает на запрос → нужны постоянные соединения (WebSocket) и удержание миллионов сокетов.
