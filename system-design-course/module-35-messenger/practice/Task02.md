# Задача 02 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Выполните estimation для мессенджера, выделив масштаб одновременных соединений как ключевой вызов.

## ВХОДНЫЕ ДАННЫЕ
Допущения: 1 млрд зарегистрированных, 200M DAU, в среднем 40 сообщений/день на активного, средний размер сообщения — порядка сотни байт; пиковый коэффициент учесть.

## ТРЕБОВАНИЯ
- Посчитайте поток сообщений (сообщений/с, среднее и пик).
- Оцените порядок одновременных открытых соединений.
- Оцените объём хранения за год (с историей).
- Объясните, почему «нагрузка соединений» здесь важнее RPS.

## ФОРМАТ РЕЗУЛЬТАТА
Сообщения/с → одновременные соединения → storage/год → вывод про соединения vs RPS.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Сообщения: 200M×40 / 86400 ≈ ~100k/с (среднее), пик ×несколько.
- [ ] Одновременные соединения: десятки–сотни млн открытых WebSocket'ов (значимая доля DAU онлайн).
- [ ] Storage: 200M×40×365 ≈ ~3e12 сообщений/год × ~сотни байт → петабайтный масштаб с историей.
- [ ] Объяснено: основная нагрузка — удержание миллионов persistent-соединений (память/сокеты на connection-серверах), а не только RPS.
- [ ] Расчёты прозрачны (единицы/порядки, модули 5–6).

## ПОДСКАЗКА
День ≈ 86400 с. Сообщения: 200M×40/86400 ≈ 100k/с. Соединения: десятки–сотни млн открытых WS. Storage: ~3e12 сообщений/год → петабайты. Вывод: вызов — удержание миллионов соединений (сокеты/память на connection-серверах), а не только RPS.
