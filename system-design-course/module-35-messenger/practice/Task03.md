# Задача 03 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Сравните способы получения сообщений в реальном времени (polling / long-polling / WebSocket) и обоснуйте выбор.

## ВХОДНЫЕ ДАННЫЕ
Получатель должен видеть входящие сообщения немедленно. Кандидаты: HTTP polling (периодический опрос), long-polling (держать запрос открытым), WebSocket (постоянное двунаправленное соединение).

## ТРЕБОВАНИЯ
- Опишите каждый подход и его проблему для чата.
- Объясните, почему чату нужен server push (а не опрос клиентом).
- Выберите WebSocket и обоснуйте (двунаправленность, latency, push).
- Назовите цену WebSocket (что сервер обязан делать).

## ФОРМАТ РЕЗУЛЬТАТА
Сравнение 3 подходов → почему server push → выбор WebSocket → цена (удержание соединений).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Polling: периодический опрос → задержка + лишний трафик (большинство опросов пустые).
- [ ] Long-polling: запрос висит до события → лучше, но накладно и односторонне.
- [ ] WebSocket: одно постоянное двунаправленное соединение, server push, низкая latency.
- [ ] Объяснено: чату нужен server push (сервер инициирует доставку), опрос клиентом не даёт real-time.
- [ ] Цена WebSocket: сервер удерживает миллионы открытых соединений (stateful по соединению) → connection servers.

## ПОДСКАЗКА
Polling — задержка + пустые опросы; long-polling — лучше, но односторонне/накладно; WebSocket — постоянное двунаправленное, server push, низкая latency. Чату нужен push (сервер инициирует). Выбор — WebSocket; цена — удержание миллионов открытых соединений (connection servers).
