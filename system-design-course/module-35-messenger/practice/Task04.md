# Задача 04 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Спроектируйте маршрутизацию сообщения к получателю, который подключён к другому connection-серверу.

## ВХОДНЫЕ ДАННЫЕ
Миллионы соединений распределены по множеству connection-серверов. Alice подключена к серверу A, Bob — к серверу B. Alice отправляет сообщение Bob. Нужно доставить сообщение на сервер, где висит соединение Bob.

## ТРЕБОВАНИЯ
- Объясните проблему: почему отправитель и получатель обычно на разных серверах.
- Спроектируйте session registry (что хранит, когда обновляется).
- Опишите поток сообщения от Alice до push'а Bob по WS.
- Укажите роль connection-серверов (stateful по соединению) и где хранится «кто где».

## ФОРМАТ РЕЗУЛЬТАТА
Проблема → session registry → поток сообщения → роль connection-серверов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблема: соединения размазаны по многим серверам → получатель почти всегда на другом сервере, чем отправитель.
- [ ] Session registry: `user_id → connection_server`, обновляется при подключении/переподключении (обычно в Redis).
- [ ] Поток: Alice → её connection server → message service (сохранить) → найти сервер Bob в реестре → переслать → connection server Bob push'ит по WS.
- [ ] Connection servers stateful по соединению (держат сокет), но состояние «кто где» вынесено в реестр → горизонтальный масштаб.
- [ ] Отмечено, что message service сохраняет сообщение до/при маршрутизации.

## ПОДСКАЗКА
Соединения размазаны по серверам → получатель на другом сервере. Session registry (`user_id→сервер`, в Redis) обновляется при подключении. Поток: отправитель→message service (сохранить)→найти сервер получателя→переслать→push по WS. Connection servers держат сокеты, «кто где» — в реестре.
