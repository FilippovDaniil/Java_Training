# Задача 05 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Спроектируйте доставку сообщений для online и offline получателей с надёжной семантикой и статусами.

## ВХОДНЫЕ ДАННЫЕ
Получатель может быть онлайн (есть WS-соединение) или офлайн (нет). Сеть ненадёжна: ответы теряются, клиент переотправляет. Нужны статусы sent/delivered/read и гарантия, что сообщение не потеряется и не задвоится.

## ТРЕБОВАНИЯ
- Опишите доставку для online-получателя.
- Опишите доставку для offline-получателя (хранение + reconnect + push).
- Выберите семантику доставки и защиту от дублей.
- Опишите ack-цепочку статусов (sent/delivered/read).

## ФОРМАТ РЕЗУЛЬТАТА
Online-доставка → offline-доставка → семантика+дедуп → статусы.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Online: получатель найден в реестре → push по WS → ack.
- [ ] Offline: нет соединения → сохранить в inbox/очередь получателя → доставить при reconnect → отправить push-уведомление (модуль 36).
- [ ] Семантика: at-least-once (повтор лучше потери) + дедупликация по message_id (генерируется клиентом) — идемпотентность (модуль 23).
- [ ] Статусы: sent (сервер принял) → delivered (доставлено устройству) → read (прочитано) — ack на этапах.
- [ ] Отмечено: дедуп защищает от дублей при retry/потере ack.

## ПОДСКАЗКА
Online → push по WS + ack. Offline → сохранить в inbox, доставить при reconnect + push-уведомление (модуль 36). Семантика at-least-once + дедуп по message_id (идемпотентность, модуль 23). Статусы — ack-цепочка sent→delivered→read.
