# Задача 08 — Модуль 35: Чат / мессенджер

## ЗАДАНИЕ
Спроектируйте presence (онлайн/офлайн) и решите проблему fan-out статуса.

## ВХОДНЫЕ ДАННЫЕ
Нужно показывать статус контактов (онлайн/офлайн/«печатает»). У активного пользователя могут быть сотни контактов. Наивно: при каждом изменении статуса push'ить его всем контактам → при масштабе это лавина обновлений.

## ТРЕБОВАНИЯ
- Спроектируйте определение онлайн/офлайн (heartbeat).
- Объясните проблему fan-out статуса (почему дорого).
- Предложите решение (ленивое/по запросу обновление вместо push всем).
- Укажите trade-off выбранного подхода (свежесть статуса vs нагрузка).

## ФОРМАТ РЕЗУЛЬТАТА
Heartbeat → проблема fan-out статуса → решение (ленивое) → trade-off.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Heartbeat: клиент периодически шлёт сигнал; отсутствие N heartbeat'ов → офлайн; presence service хранит статус.
- [ ] Проблема: изменение статуса одного пользователя × сотни контактов × миллионы пользователей = лавина push-обновлений presence.
- [ ] Решение: обновлять статус лениво (при открытии чата/списка контактов, по запросу), а не push'ить всем контактам сразу.
- [ ] Trade-off: ленивое обновление снижает нагрузку, но статус может быть слегка несвежим (приемлемо для presence).
- [ ] Отмечено: presence — отдельный сервис, не нагружающий путь сообщений.

## ПОДСКАЗКА
Presence: heartbeat (нет N сигналов → офлайн). Проблема fan-out: статус одного × сотни контактов × миллионы = лавина push. Решение — ленивое обновление (по запросу/при открытии чата), не push всем. Trade-off: чуть менее свежий статус, зато без лавины (для presence приемлемо).
