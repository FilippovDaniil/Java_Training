## Ключ (эталонные ответы)

1. 1:1 сообщения, групповые чаты, статусы доставки/прочтения (sent/delivered/read), presence (онлайн/офлайн/«печатает»), история сообщений, offline-доставка.
2. Сервер должен сам push'ить сообщения получателю в реальном времени, а не отвечать на его запрос. Это требует постоянных соединений и удержания состояния соединения на сервере.
3. Огромное число одновременных открытых соединений (десятки–сотни млн WebSocket'ов) — нагрузка на удержание соединений (память/сокеты), а не только на RPS.
4. Периодический опрос даёт задержку (сообщение ждёт следующего опроса) и лишний трафик (большинство опросов пустые); не обеспечивает real-time.
5. WebSocket — одно постоянное двунаправленное соединение с server push и низкой latency; сервер доставляет сообщение немедленно, без опроса клиентом.
6. Серверы, удерживающие persistent WS-соединения (миллионы). Stateful по соединению (держат открытый сокет конкретного пользователя), хотя бизнес-логика может быть stateless; масштабируются добавлением серверов.
7. Чтобы знать, на каком connection-сервере висит соединение каждого пользователя (`user_id → сервер`): отправитель и получатель почти всегда на разных серверах, и сообщение нужно доставить на нужный сервер.
8. Отправитель → его connection server → message service (сохраняет сообщение) → ищет сервер получателя в session registry → пересылает сообщение этому серверу → тот push'ит получателю по WS.
9. Получатель найден онлайн в реестре → сообщение push'ится ему по WebSocket → возвращается ack (delivered).
10. Нет активного соединения → сообщение сохраняется в его inbox/очереди → доставляется при следующем reconnect; параллельно отправляется push-уведомление, чтобы привлечь пользователя (модуль 36).
11. At-least-once (повтор лучше потери). Дедупликация по message_id (генерируется клиентом) защищает от дублей при retry/потере ack — идемпотентность (модуль 23).
12. Подтверждения на этапах доставки: sent (сервер принял сообщение), delivered (доставлено устройству получателя), read (получатель прочитал). Это ack-цепочка, важная для UX.
13. Глобальный тотальный порядок требует координации между всеми узлами (дорого) и не нужен пользователю; важен лишь порядок сообщений внутри одного диалога.
14. Каждому сообщению присваивают sequence number / логическую метку в рамках conversation_id; получатель сортирует по ней. Партиционирование по conversation_id поддерживает порядок (порядок в партиции, модуль 15).
15. Wide-column (Cassandra-style): нагрузка write-heavy и огромный объём, доступ по conversation_id с хронологией. Даёт высокую пропускную записи и быстрый range-запрос истории; реляционная не тянет объём/запись.
16. conversation_id (partition key) с кластеризацией/сортировкой по времени или sequence — все сообщения диалога в одной партиции, отсортированы.
17. В object storage + CDN (модуль 10); в самом сообщении хранится ссылка, а не блоб, чтобы не раздувать хранилище сообщений.
18. Клиент периодически шлёт heartbeat; отсутствие нескольких heartbeat'ов → офлайн. Presence service хранит текущий статус.
19. Изменение статуса одного пользователя нужно показать всем его контактам; при сотнях контактов и миллионах пользователей это лавина push-обновлений presence.
20. Обновлять статус лениво/по запросу (при открытии чата или списка контактов), а не push'ить всем контактам сразу. Trade-off — статус слегка менее свежий, что для presence приемлемо.
21. Сообщение раскладывается членам группы (fan-out). Маленькие группы — push каждому; большие — отдельная стратегия (не рассылать всем синхронно).
22. Большая группа = одно сообщение порождает много доставок (как пост celebrity → миллионы лент), поэтому требует особой стратегии fan-out, а не наивной рассылки каждому (модуль 34).
23. Горизонтально: много connection-серверов, балансировка соединений (модули 8, 24), состояние «кто где» в session registry; сервисы за ними stateless.
24. Чтобы доставить сообщение/привлечь пользователя, когда у него нет активного соединения (офлайн): сообщение сохранено в inbox, а push-уведомление (модуль 36) сигнализирует о новом сообщении.
25. Real-time через WebSocket + connection servers; маршрутизация к получателю через session registry; доставка online (push)/offline (inbox+reconnect+push) с at-least-once+дедуп и статусами; порядок per-conversation (seq); хранение в wide-column с партицией по conversation_id; presence с ленивым fan-out; большие группы как celebrity; масштаб соединений добавлением серверов.
