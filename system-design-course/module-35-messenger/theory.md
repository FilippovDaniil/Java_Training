# Модуль 35 — Чат / мессенджер (референсная архитектура)

## Зачем этот модуль

Мессенджер (WhatsApp/Telegram/Slack) — система, которая впервые в курсе ломает привычную модель «запрос → ответ»: здесь сервер должен **сам push'ить** сообщения клиенту в реальном времени. Это вводит **постоянные соединения** (WebSocket), **серверы соединений**, **presence** и **маршрутизацию к получателю** — темы, которых не было в URL-shortener или ленте. Модуль синтезирует очереди (15), надёжность доставки (23), шардирование хранилища (13) и балансировку миллионов соединений (8/24).

Зачем это senior'у. Чат вскрывает специфический класс задач — **real-time с persistent-соединениями**: как удержать миллионы открытых WebSocket'ов, как **найти, на каком сервере висит получатель**, как доставить сообщение **online и offline**, как гарантировать **порядок и отсутствие дублей** (at-least-once + дедуп). Это разговор про connection servers, session registry и presence-fan-out — то, что отличает дизайн чата от дизайна REST-сервиса.

Он показывает полный HLD мессенджера: от постоянных соединений и маршрутизации до доставки, порядка, хранения и presence.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| 1:1 сообщения | Низкая latency (real-time, «мгновенно») |
| Групповые чаты | Высокая доступность |
| Статусы (sent / delivered / read) | **Порядок** сообщений в диалоге |
| Presence (онлайн/офлайн, «печатает») | Durability (сообщения не теряются) |
| История + offline-доставка | Масштаб (млрд пользователей, млн соединений) |

```
Estimation (пример): 1 млрд пользователей, 200M DAU, 40 сообщений/день на активного.
  Сообщений: 200M×40 / 86400 ≈ ~100k сообщений/с (пик ×несколько).
  Одновременные соединения: десятки–сотни млн открытых WebSocket'ов.
  Storage: 200M×40 ×365 ≈ ~3e12 сообщений/год × ~порядка сотни байт → петабайты с историей.
```

> Главное отличие от прошлых систем — **real-time с server push**: сервер должен доставить сообщение получателю немедленно, а не ждать его запроса. Estimation высвечивает два масштабных вызова: **число одновременных соединений** (десятки–сотни миллионов открытых WebSocket'ов — нагрузка не на RPS, а на удержание соединений) и **порядок/durability** сообщений. Это сразу диктует connection servers и продуманную маршрутизацию.

## 2. Постоянное соединение: WebSocket

| Подход | Пригодность для чата |
|--------|----------------------|
| HTTP polling | клиент периодически спрашивает «есть новое?» — задержка + лишний трафик |
| Long-polling | держим запрос открытым до события — лучше, но накладно, односторонне |
| **WebSocket** | **двунаправленное** постоянное соединение, server push, низкая latency |

```
Клиент ──(один раз)──► WebSocket handshake ──► постоянное соединение
   ◄──── сервер push'ит сообщения в реальном времени ────►
   ────► клиент шлёт сообщения по тому же соединению ────►
```

> Чат требует, чтобы **сервер инициировал** доставку (получатель не опрашивает сервер каждую секунду). Polling даёт задержку и лишнюю нагрузку, long-polling лучше, но односторонний и накладный. **WebSocket** — стандарт: одно постоянное двунаправленное соединение, по которому сервер push'ит сообщения с минимальной latency. Цена — сервер должен **удерживать** миллионы открытых соединений (stateful по соединению), что и порождает «серверы соединений».

## 3. Серверы соединений и маршрутизация к получателю

```
Проблема: у Alice соединение на connection-server-7, у Bob — на connection-server-42.
  Alice шлёт Bob → как система узнаёт, что Bob висит на сервере 42?

Решение — session registry / presence-routing:
  при подключении клиент регистрируется: user_id → connection_server (в реестре, напр. Redis).
  отправка: найти сервер получателя в реестре → переслать ему сообщение → он push'ит по WS.
```

| Компонент | Роль |
|-----------|------|
| **Connection servers** | удерживают persistent WS-соединения (миллионы); stateful по соединению |
| **Session registry** | `user_id → connection_server` (кто где подключён); обычно в Redis |
| **Message service** | принимает/сохраняет сообщение, маршрутизирует к серверу получателя |

> Ключевая задача — **маршрутизация к получателю**: отправитель и получатель почти всегда висят на **разных** connection-серверах. Система ведёт **session registry** (`user_id → сервер`), куда клиент пишется при подключении. Поток: сообщение → message service (сохранить) → найти сервер получателя в реестре → переслать ему → connection server push'ит по WS. Connection-серверы **stateless по бизнес-логике, но stateful по соединению** (держат сокет), поэтому масштабируются горизонтально, а состояние «кто где» вынесено в реестр.

## 4. Доставка: online, offline, семантика

```
Получатель ONLINE  → найден в реестре → push по WS → ack.
Получатель OFFLINE → нет соединения → сохранить в его «inbox»/очередь сообщений
                     → доставить при следующем reconnect → отправить push-уведомление (модуль 36).
```

| Аспект доставки | Решение |
|-----------------|---------|
| Семантика | **at-least-once** + **дедупликация** по message_id (идемпотентность, модуль 23) |
| Подтверждения | ack-цепочка: sent (сервер принял) → delivered (доставлено получателю) → read |
| Offline | хранить недоставленные, доставлять при reconnect; push через notification (модуль 36) |

> Доставка раздваивается по presence: **online** — push по WS; **offline** — сообщение кладётся в персистентный inbox получателя и доставляется при следующем подключении (плюс push-уведомление, модуль 36, чтобы привлечь пользователя). Семантика — **at-least-once** (повтор лучше потери), а **дедупликация по message_id** (генерируется клиентом) защищает от дублей при retry (модуль 23). Статусы sent/delivered/read — это ack'и на разных этапах, важная часть UX.

## 5. Порядок сообщений

```
Проблема: сообщения в одном диалоге не должны переставляться (ответ раньше вопроса).
Решение: порядок гарантируется per-conversation (не глобально):
  каждому сообщению — sequence number / временная метка в рамках conversation_id;
  клиент сортирует/отображает по нему.
```

> Глобальный порядок в распределённой системе дорог и не нужен — важен порядок **внутри диалога** (модуль 15: порядок только в партиции). Каждому сообщению присваивают **sequence number** (или логическую метку) в рамках `conversation_id`; получатель упорядочивает по нему. Партиционирование хранилища по `conversation_id` (§6) естественно поддерживает per-conversation порядок: все сообщения диалога идут через одну партицию. Это пример «достаточной согласованности» — не глобальный тотальный порядок, а порядок там, где он виден пользователю.

## 6. Хранение сообщений

| Профиль | Выбор |
|---------|-------|
| Нагрузка | **write-heavy** (каждое сообщение пишется), огромный объём |
| Паттерн доступа | по `conversation_id`, последние сообщения, хронология |
| Хранилище | **wide-column** (Cassandra-style): партиция по `conversation_id`, сортировка по времени (модули 11, 13) |

```
messages: partition key = conversation_id, clustering = timestamp/seq → message_id, sender, content
  → все сообщения диалога в одной партиции, отсортированы по времени → быстрый range-запрос «история».
```

> Чат — **write-heavy** (поток сообщений) с доступом по диалогу — эталон для **wide-column** хранилища (модуль 11): партиция по `conversation_id`, кластеризация по времени. Это даёт быструю запись (последовательная по партиции), быстрый range-запрос истории и естественное шардирование по диалогу (модуль 13). Реляционная БД упёрлась бы в объём/запись. Медиа (картинки/видео) — в object storage + CDN (модуль 10), в сообщении — ссылка.

## 7. Presence, группы, high-level и узкие места

**Presence** (онлайн/офлайн): клиент шлёт **heartbeat**; отсутствие heartbeat → офлайн. Проблема — **fan-out статуса**: «Alice онлайн» надо показать всем её контактам → при больших списках это дорого (рассылка изменений presence многим). Решают: обновлять presence лениво (по запросу/при открытии чата), а не push'ить всем сразу.

**Группы:** сообщение в группу → fan-out членам (маленькие группы — push каждому; большие — отдельная обработка, как celebrity в ленте, модуль 34).

```
High-level:
  Client ↔ [WS Gateway / connection servers] ↔ [Message Service] → [wide-column store] + [inbox/очередь]
                         │                              │
                  [Session Registry]             [Notification] (push для offline, модуль 36)
                  [Presence Service]  ← heartbeat
```

| Узкое место | Решение |
|-------------|---------|
| Миллионы persistent-соединений | много connection-серверов + LB по соединениям (модули 8, 24) |
| Presence fan-out | ленивое обновление статуса; не push'ить всем контактам |
| Hot groups (большие/активные) | отдельная стратегия fan-out (как celebrity, модуль 34) |
| Порядок/durability | per-conversation seq + надёжное хранилище |

> Финальная сборка: **WS-gateway/connection servers** держат соединения, **message service** сохраняет и маршрутизирует, **session registry** и **presence** обеспечивают «кто где и онлайн ли», **notification** добивает offline-пользователей (модуль 36). Узкие места специфичны для real-time: **масштаб соединений** (горизонтально — много connection-серверов), **presence fan-out** (ленивое обновление), **hot groups** (стратегия как для celebrity). Это качественно иной набор узких мест, чем у stateless REST-систем.

## ⚠️ Подводные камни

- **Polling вместо WebSocket для real-time.** Задержка + лишний трафик; нужно persistent-соединение с server push.
- **Игнор маршрутизации к получателю.** Отправитель и получатель на разных серверах; нужен session registry (`user_id→сервер`).
- **Считать connection-серверы stateless.** Они держат сокеты (stateful по соединению); масштаб — добавлением серверов + реестр.
- **Терять offline-сообщения.** Если получатель офлайн — сохранять в inbox и доставлять при reconnect + push (модуль 36).
- **Доставка без дедупликации.** At-least-once + retry задвоят сообщение; дедуп по message_id (модуль 23).
- **Глобальный порядок сообщений.** Дорог и не нужен; порядок per-conversation (seq number).
- **Реляционная БД под поток сообщений.** Write-heavy + объём → wide-column, партиция по conversation_id (модули 11/13).
- **Push presence всем контактам сразу.** Fan-out статуса дорог; обновлять лениво.
- **Большая группа = простой fan-out каждому.** Hot groups — отдельная стратегия (как celebrity, модуль 34).
- **Медиа через message service.** Картинки/видео — в object storage + CDN (модуль 10), в сообщении ссылка.
