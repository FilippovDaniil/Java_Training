# Задача 01 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Соберите требования (FR/NFR) к системе уведомлений и выделите два режима нагрузки.

## ВХОДНЫЕ ДАННЫЕ
Постановка: «спроектируйте систему уведомлений» — она шлёт push/SMS/email, обслуживает и срочные сообщения (OTP), и массовые маркетинговые кампании.

## ТРЕБОВАНИЯ
- Сформулируйте ядро FR (каналы, шаблоны, настройки/opt-out, отложенная отправка, трекинг).
- Сформулируйте NFR (пропускная, надёжность, latency срочных, масштаб, дедуп).
- Выделите два режима нагрузки и их различие.
- Объясните, почему различие режимов влияет на дизайн.

## ФОРМАТ РЕЗУЛЬТАТА
FR → NFR → два режима нагрузки → влияние на дизайн.

## КРИТЕРИИ ПРИЁМКИ
- [ ] FR: отправка по каналам (push/SMS/email/in-app), шаблоны+персонализация, настройки/opt-out, отложенная отправка, трекинг доставки.
- [ ] NFR: высокая пропускная, надёжность (at-least-once), низкая latency для срочных, масштаб fan-out, дедупликация.
- [ ] Два режима: ровный поток транзакционных (OTP/алерты) + резкие всплески кампаний (fan-out на миллионы).
- [ ] Объяснено: разные режимы → нужна развязка через очередь + приоритизация (кампания не должна задержать OTP).
- [ ] Зафиксирован scope.

## ПОДСКАЗКА
FR: каналы push/SMS/email, шаблоны, настройки/opt-out, отложенная отправка, трекинг. NFR: пропускная, надёжность, latency срочных, масштаб, дедуп. Два режима — транзакционные (ровно, срочно) и кампании (всплески на миллионы) → развязка очередью + приоритеты.
