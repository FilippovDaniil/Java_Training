# Задача 03 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Спроектируйте работу с каналами и сторонними провайдерами, абстрагировав их различия.

## ВХОДНЫЕ ДАННЫЕ
Каналы: push (APNS/FCM), SMS (Twilio), email (SES/SendGrid), in-app. У каждого провайдера свой API, форматы, rate limits, режимы отказа, стоимость (SMS дорог, email боится спам-репутации).

## ТРЕБОВАНИЯ
- Перечислите каналы и их провайдеров с особенностями.
- Объясните, почему дизайн строится вокруг ненадёжных внешних зависимостей.
- Спроектируйте абстракцию (адаптер на канал/провайдера).
- Укажите, что у каждого канала своя логика (лимиты, отказы, форматы).

## ФОРМАТ РЕЗУЛЬТАТА
Каналы+провайдеры (таблица) → почему ненадёжные зависимости → адаптеры → специфика каналов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Каналы: push (APNS/FCM — токены устройств), SMS (Twilio — дорого, строгие лимиты, регуляторика), email (SES/SendGrid — спам/bounce/репутация), in-app (через WS/feed).
- [ ] Объяснено: провайдеры вне вашего контроля — лимиты, отказы, задержки, стоимость → дизайн вокруг ненадёжных зависимостей.
- [ ] Адаптер на канал/провайдера абстрагирует API/форматы/лимиты/отказы (единый интерфейс отправки).
- [ ] Указано: каждый канал = отдельный воркер/адаптер со своей логикой (лимиты, обработка ошибок).
- [ ] Отмечена возможность fallback-провайдера в рамках канала.

## ПОДСКАЗКА
Каналы: push (APNS/FCM), SMS (Twilio, дорого/лимиты), email (SES, спам-репутация), in-app. Провайдеры вне контроля (лимиты/отказы/стоимость) → дизайн вокруг ненадёжных зависимостей. Адаптер на канал абстрагирует API/лимиты/отказы; каждый канал — свой воркер; возможен fallback-провайдер.
