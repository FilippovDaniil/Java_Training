# Задача 04 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Спроектируйте архитектуру доставки через очередь и воркеров по каналам, объяснив развязку.

## ВХОДНЫЕ ДАННЫЕ
Приём уведомлений быстрый, а доставка медленная и зависит от внешних провайдеров (лимиты, задержки, отказы). Кампания на миллионы создаёт всплеск. Нужно не блокировать приём и масштабировать доставку.

## ТРЕБОВАНИЯ
- Спроектируйте поток: producer → notification service → очередь → воркеры → провайдеры.
- Объясните, что даёт развязка через очередь (модуль 15).
- Обоснуйте специализацию воркеров по каналам.
- Покажите, как очередь сглаживает всплеск кампании.

## ФОРМАТ РЕЗУЛЬТАТА
Поток (компоненты) → что даёт очередь → воркеры по каналам → сглаживание всплеска.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Поток: producer → Notification Service (настройки+шаблон) → очередь → воркеры канала → адаптер провайдера → APNS/Twilio/SES.
- [ ] Развязка через очередь (модуль 15): быстрый приём, асинхронная доставка, надёжность (сообщение переживёт сбой воркера), масштаб воркерами.
- [ ] Воркеры специализированы по каналам (push/SMS/email) — разные провайдеры/лимиты/логика.
- [ ] Всплеск кампании: уведомления встают в очередь (load leveling), воркеры разбирают в темпе лимитов провайдера, не валя его и сервис.
- [ ] Отмечено сходство fan-out с лентой (модуль 34).

## ПОДСКАЗКА
Поток: producer→notification service→очередь→воркеры по каналам→адаптеры→провайдеры. Очередь (модуль 15) развязывает быстрый приём от медленной доставки: load leveling, масштаб воркерами, надёжность. Воркеры по каналам (разные провайдеры/лимиты). Всплеск кампании сглаживается очередью.
