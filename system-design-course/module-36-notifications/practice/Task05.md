# Задача 05 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Спроектируйте надёжность доставки: семантика, дедупликация, retry, DLQ, circuit breaker.

## ВХОДНЫЕ ДАННЫЕ
Провайдеры временно падают и тормозят. Сеть теряет ответы → воркер может повторить отправку. Критично: OTP и платёжные алерты НЕ должны прийти дважды и НЕ должны потеряться.

## ТРЕБОВАНИЯ
- Выберите семантику доставки и обоснуйте.
- Спроектируйте дедупликацию (как не отправить OTP дважды).
- Опишите retry с backoff и роль DLQ.
- Добавьте circuit breaker против упавшего провайдера + fallback.

## ФОРМАТ РЕЗУЛЬТАТА
Семантика → дедупликация → retry+DLQ → circuit breaker+fallback.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Семантика at-least-once (повтор лучше потери), особенно для критичных уведомлений.
- [ ] Дедупликация: idempotency key на уведомление + хранилище обработанных id → повтор не отправляет второй раз (модуль 23).
- [ ] Retry с backoff при временных сбоях/5xx провайдера; после N попыток → DLQ (не блокировать очередь, разобрать отдельно).
- [ ] Circuit breaker: при потоке отказов провайдера разомкнуть (не добивать, не копить таймауты), отдавать fallback/откладывать (модуль 23).
- [ ] Fallback-провайдер в рамках канала при отказе основного.
- [ ] Подчёркнуто: at-least-once + дедуп = не потерять и не задвоить.

## ПОДСКАЗКА
At-least-once (не терять) + дедуп (idempotency key + хранилище id) = не задвоить OTP. Retry+backoff при сбоях провайдера, после N → DLQ. Circuit breaker против упавшего провайдера (не добивать) + fallback-провайдер. Это resilience-стек модуля 23 для внешних зависимостей.
