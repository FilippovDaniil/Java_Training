# Задача 06 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Спроектируйте rate limiting на двух уровнях: лимиты провайдера и анти-спам для пользователя.

## ВХОДНЫЕ ДАННЫЕ
Провайдеры (APNS/Twilio/SES) имеют жёсткие лимиты — при превышении throttle/блокировка вашего аккаунта. Пользователи жалуются на спам (слишком частые уведомления). Кампания может попытаться отправить миллионы за минуту.

## ТРЕБОВАНИЯ
- Спроектируйте rate limiting на стороне провайдера (не превышать его лимиты).
- Спроектируйте пользовательский лимит (анти-спам, digest).
- Объясните, что произойдёт без каждого уровня.
- Свяжите с модулем 16 (алгоритмы/распределённый лимит).

## ФОРМАТ РЕЗУЛЬТАТА
Лимит провайдера → пользовательский лимит/digest → последствия отсутствия → связь с модулем 16.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Провайдерный лимит: воркеры ограничивают исходящий поток под лимит APNS/Twilio/SES (распределённый счётчик, модуль 16).
- [ ] Пользовательский: не более N уведомлений/период на пользователя; объединять в digest при превышении.
- [ ] Без провайдерного лимита: провайдер throttle/блокирует аккаунт → доставка встаёт.
- [ ] Без пользовательского: спам → отписки/жалобы (ухудшение репутации email, отток).
- [ ] Связь с модулем 16: алгоритмы (token bucket), распределённый rate limiting (общий счётчик на воркеров).

## ПОДСКАЗКА
Два уровня: провайдерный (воркеры не превышают лимиты APNS/Twilio/SES, иначе блок) и пользовательский (N/период, digest против спама). Без первого — провайдер блокирует; без второго — спам/отписки. Алгоритмы и распределённый счётчик — модуль 16.
