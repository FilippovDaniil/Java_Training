# Задача 08 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Спроектируйте приоритизацию уведомлений, чтобы срочные не застревали за массовой кампанией.

## ВХОДНЫЕ ДАННЫЕ
Одновременно: маркетинговая кампания на 50M пользователей (можно доставить в течение часа) и поток OTP/алертов безопасности (нужны за секунды). Если они в одной очереди — OTP застрянет за миллионами промо-сообщений.

## ТРЕБОВАНИЯ
- Классифицируйте уведомления по приоритету (транзакционные vs промо).
- Спроектируйте приоритетные очереди (как срочные обходят массовые).
- Опишите, какие требования к latency у каждого класса.
- Объясните, почему нельзя смешивать их в одной очереди.

## ФОРМАТ РЕЗУЛЬТАТА
Классификация → приоритетные очереди → latency-требования → почему нельзя смешивать.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Классы: TRANSACTIONAL (OTP, сброс пароля, платёжные/security-алерты) — высокий приоритет; PROMOTIONAL (маркетинг, дайджесты) — низкий, батч.
- [ ] Приоритетные очереди: отдельные очереди по приоритету (или веса), воркеры обслуживают высокий приоритет первым.
- [ ] Latency: транзакционные — секунды (срочно); промо — можно отложить (минуты/часы).
- [ ] Объяснено: в одной очереди OTP застрянет за миллионами промо → недопустимая задержка срочного.
- [ ] Отмечено: это load-leveling + приоритизация (массовое не вытесняет срочное).

## ПОДСКАЗКА
Классы: транзакционные (OTP/security — секунды) vs промо (можно отложить). Приоритетные очереди (отдельные/веса): срочные обходят массовые. В одной очереди OTP застрянет за миллионами промо. Приоритизация защищает латентно-критичные уведомления.
