# Задача 09 — Модуль 36: Система уведомлений

## ЗАДАНИЕ
Найдите узкие места системы уведомлений (вокруг внешних зависимостей и fan-out) и спроектируйте их устранение.

## ВХОДНЫЕ ДАННЫЕ
Эксплуатация выявила: один провайдер (SMS) периодически «лежит» и тянет воркеров в таймауты; при кампании на миллионы воркеры упираются в лимиты провайдера; дедупликация при огромном объёме тормозит; иногда срочные уведомления застревают.

## ТРЕБОВАНИЯ
- Перечислите узкие места (отказ провайдера, лимиты, дедуп на масштабе, застревание срочных).
- Для каждого дайте решение со ссылкой на модуль.
- Объясните, почему большинство узких мест — вокруг внешних зависимостей.
- Укажите роль fallback-провайдера.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица «узкое место → решение → модуль» + вывод про внешние зависимости.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Отказ провайдера → circuit breaker + retry/backoff + fallback-провайдер (модуль 23).
- [ ] Лимиты провайдера при кампании → rate limiting воркеров + очередь сглаживает (модули 16, 15).
- [ ] Дедуп на масштабе → idempotency key + быстрое хранилище обработанных id (например, Redis с TTL).
- [ ] Застревание срочных → приоритетные очереди (транзакционные отдельно от промо).
- [ ] Вывод: большинство узких мест — вокруг ненадёжных внешних провайдеров; дизайн подчинён надёжности интеграций.
- [ ] Fallback-провайдер: при отказе основного канала переключиться на резервного.

## ПОДСКАЗКА
Узкие места вокруг провайдеров: отказ → circuit breaker+retry+fallback (23); лимиты при кампании → rate limiting воркеров + очередь (16/15); дедуп на масштабе → idempotency key + быстрый стор id (Redis+TTL); застревание срочных → приоритетные очереди. Дизайн подчинён надёжности внешних интеграций.
