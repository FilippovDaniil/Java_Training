## Ключ (эталонные ответы)

1. Отправка по каналам (push/SMS/email/in-app), шаблоны с персонализацией, пользовательские настройки/opt-out, отложенная/запланированная отправка, трекинг доставки.
2. Ровный поток транзакционных уведомлений (OTP, алерты) и резкие всплески массовых маркетинговых кампаний (fan-out на миллионы за короткое окно).
3. Средний: 100M×5/86400 ≈ ~6000/с. Кампания на 50M за ~10 минут: 50M/600 ≈ ~83000/с — на порядок-полтора выше среднего.
4. Push (APNS/FCM), SMS (Twilio/Vonage), email (SES/SendGrid), in-app (через WS/feed). Каждый внешний провайдер со своим API, лимитами, отказами, стоимостью.
5. Потому что доставка идёт через сторонних провайдеров, которые вне вашего контроля: у них свои rate limits, режимы отказа, задержки и стоимость; система должна уважать лимиты и переживать их отказы.
6. Чтобы скрыть различия API/форматов/лимитов/отказов за единым интерфейсом отправки и легко добавлять/заменять провайдеров (в т.ч. fallback) без изменения остальной логики.
7. Чтобы развязать быстрый приём уведомлений от медленной доставки через провайдеров: сглаживание всплесков (load leveling), масштабирование воркерами, надёжность (сообщение переживёт сбой воркера). Модуль 15.
8. Потому что у каналов разные провайдеры, лимиты, форматы и логика обработки ошибок; отдельные воркеры/очереди на канал позволяют их независимо масштабировать и настраивать.
9. At-least-once: лучше доставить повторно, чем потерять (особенно для критичных уведомлений вроде OTP/алертов).
10. Чтобы не отправить уведомление дважды при retry/повторной обработке (двойной OTP недопустим). Реализуется idempotency key на уведомление + хранилищем обработанных id (модуль 23).
11. Retry с backoff переживает временные сбои/throttle провайдера без лавины повторов. DLQ (dead-letter queue) принимает уведомления, не доставленные после N попыток, чтобы разобрать их отдельно и не блокировать основную очередь.
12. Чтобы при потоке отказов провайдера перестать его «добивать» и не копить таймауты у воркеров: разомкнуть цепь, отдавать fallback/откладывать, дать провайдеру восстановиться (модуль 23).
13. Резервный провайдер для канала: при отказе основного (например, основной SMS-шлюз лежит) переключиться на запасного, чтобы канал оставался работоспособным.
14. Чтобы не превысить лимиты APNS/Twilio/SES — иначе провайдер сам начнёт throttle или заблокирует ваш аккаунт, и доставка встанет. Реализуется на стороне воркеров (модуль 16).
15. Чтобы не спамить пользователя (ограничить число уведомлений в период, объединять в digest) — иначе отписки, жалобы, падение репутации email и отток.
16. Чтобы не отправлять отключённые пользователем каналы/категории и соблюдать согласие; проверка до отправки экономит ресурсы и предотвращает нарушение (отправленное не отозвать).
17. Юридические требования к уведомлениям: наличие согласия пользователя, возможность отписки (unsubscribe для email/SMS), аудит; нарушение (GDPR/CAN-SPAM) грозит штрафами.
18. Обычно нет: транзакционные (OTP, сброс пароля, security/платёжные алерты) относятся к работе сервиса и не подпадают под opt-out; отключаемы только промо/маркетинговые.
19. Чтобы отделить контент (текст с плейсхолдерами, локализация) от логики отправки и переиспользовать его с данными персонализации; упрощает изменение текстов без правки кода.
20. Раздельные очереди (или веса) по приоритету: высокоприоритетные (транзакционные) обслуживаются раньше низкоприоритетных (промо), чтобы срочное не застревало за массовым.
21. Транзакционные требуют низкой latency (секунды — OTP бесполезен с задержкой); промо можно отложить (минуты/часы), доставлять батчем.
22. Срочный OTP застрянет в очереди за миллионами промо-сообщений кампании → недопустимая задержка критичного уведомления. Поэтому их разделяют по приоритетным очередям.
23. Через scheduled-очередь/таймеры: уведомление хранится с временем отправки и попадает в рабочую очередь по наступлении срока (или периодический планировщик выбирает «пора»).
24. Через статусы (queued → sent → delivered → failed/bounced), обновляемые ответами и webhooks/callbacks провайдеров; агрегация для аналитики обычно асинхронная.
25. Очередь + воркеры по каналам (развязка от медленных/ненадёжных провайдеров); адаптеры провайдеров + fallback; надёжность (at-least-once + дедуп по idempotency key + retry/backoff + DLQ + circuit breaker); rate limiting (провайдер + пользователь); проверка opt-out/compliance до отправки; шаблоны; приоритетные очереди (транзакционные ≠ промо); трекинг через webhooks.
