# Модуль 36 — Система уведомлений (референсная архитектура)

## Зачем этот модуль

Уведомления — это «клей», которым пользовались прошлые системы: push для offline-получателя в чате (модуль 35), алерты, OTP, маркетинговые рассылки. Система уведомлений выглядит просто («отправить сообщение»), но на масштабе вскрывает целый класс задач: **интеграция со сторонними провайдерами** (APNS/Twilio/SES) с их лимитами и отказами, **fan-out кампании на миллионы**, **надёжность доставки** (at-least-once + дедуп + retry + DLQ) и **пользовательские настройки/opt-out**. Модуль синтезирует очереди (15), надёжность (23) и rate limiting (16).

Зачем это senior'у. Дизайн уведомлений — это в первую очередь дизайн вокруг **ненадёжных внешних зависимостей** (провайдеры падают, имеют лимиты, отвечают с задержкой) и **массового fan-out**. Senior проектирует развязку через очередь, воркеров по каналам, защиту от дублей (нельзя отправить OTP дважды), уважение лимитов провайдера и приоритеты (транзакционные vs промо). Это эталон системы, где главная сложность — не алгоритм, а **надёжность и интеграции**.

Он показывает полный HLD системы уведомлений: каналы и провайдеры, очередь и воркеры, надёжность, настройки, приоритеты.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Отправка по каналам: push / SMS / email / in-app | Высокая пропускная (массовый fan-out) |
| Шаблоны + персонализация | Надёжность (at-least-once, не терять) |
| Пользовательские настройки / opt-out | Низкая latency для срочных (OTP) |
| Отложенная / запланированная отправка | Масштаб (кампании на миллионы) |
| Трекинг доставки | Дедупликация (не отправить дважды) |

```
Estimation (пример): 100M пользователей, в среднем 5 уведомлений/день → ~6000/с среднее.
  Маркетинговая кампания: 50M пользователей за короткое окно → всплеск десятки–сотни тысяч/с.
  По каналам разделить (push/SMS/email) — у каждого свои лимиты провайдера.
```

> Уведомления кажутся тривиальными, но estimation сразу показывает **два режима нагрузки**: ровный поток транзакционных (OTP, алерты) и **резкие всплески** маркетинговых кампаний (fan-out на миллионы за минуты). Плюс — каналы (push/SMS/email) с **разными провайдерами и лимитами**. Это диктует развязку через очередь и приоритизацию, чтобы кампания не задержала срочный OTP.

## 2. Каналы и сторонние провайдеры

| Канал | Провайдер (пример) | Особенности |
|-------|--------------------|-------------|
| Push (mobile) | APNS (iOS), FCM (Android) | токены устройств, доставка best-effort |
| SMS | Twilio, Vonage | дорого, строгие rate limits, регуляторика |
| Email | SES, SendGrid | спам-фильтры, bounce/complaint, репутация |
| In-app | собственный (через WS/feed) | как лента/чат (модули 34/35) |

> Ключевая особенность: уведомления **доставляются через сторонних провайдеров**, которые вы не контролируете — у каждого свои **лимиты (rate limits)**, форматы, режимы отказа и стоимость (SMS дорог, email боится спам-репутации). Это значит, что дизайн строится вокруг **ненадёжных внешних зависимостей**: их надо абстрагировать (адаптер на канал), уважать их лимиты (модуль 16) и переживать их отказы (retry/circuit breaker, модуль 23). Каждый канал — отдельный воркер/адаптер.

## 3. Архитектура: очередь + воркеры по каналам

```
Producer (сервис) → Notification Service → [проверка настроек/opt-out] → [шаблонизация]
   → ОЧЕРЕДЬ (по каналу / по приоритету)  (модуль 15)
   → Воркеры канала (push / SMS / email) → адаптер провайдера → APNS/Twilio/SES
```

| Компонент | Роль |
|-----------|------|
| Notification Service | приём запросов, проверка настроек, шаблонизация, постановка в очередь |
| **Очередь** (модуль 15) | развязка приёма и доставки, сглаживание всплесков, масштаб воркерами |
| Воркеры по каналам | берут из очереди, вызывают провайдера, обрабатывают ответ/ошибку |
| Адаптеры провайдеров | абстракция над APNS/Twilio/SES (форматы, лимиты, отказы) |

> Сердце дизайна — **развязка через очередь** (модуль 15): приём уведомления (быстрый) отделён от доставки (медленной, зависящей от провайдера). Это даёт: **сглаживание всплесков** (кампания на 50M встаёт в очередь, не валит провайдера и сервис), **масштабирование воркерами** (добавить воркеров под рост), **надёжность** (сообщение в очереди переживёт сбой воркера). Воркеры специализированы **по каналам** (у push, SMS, email разные провайдеры/лимиты/логика). Это та же fan-out-механика, что в ленте (модуль 34), но «получатель» — внешний провайдер.

## 4. Надёжность доставки

| Механизм | Назначение |
|----------|------------|
| **At-least-once** | лучше доставить повторно, чем потерять |
| **Дедупликация / идемпотентность** | idempotency key на уведомление → не отправить OTP дважды (модуль 23) |
| **Retry + backoff** | провайдер временно недоступен/ответил 5xx → повтор с паузой (модуль 23) |
| **DLQ** (dead-letter queue) | после N неудач → в DLQ для разбора, не блокировать очередь |
| **Circuit breaker** | провайдер «лежит» → разомкнуть, не добивать, fallback (модуль 23) |

> Надёжность — главная сложность уведомлений. Доставка **at-least-once** (повтор лучше потери), но это требует **дедупликации**: idempotency key на уведомление защищает от двойной отправки при retry (критично для OTP/платёжных алертов — модуль 23). Провайдеры падают и тормозят → **retry с backoff** переживает временные сбои, **DLQ** ловит безнадёжные (после N попыток — в отдельную очередь, чтобы не блокировать поток), **circuit breaker** перестаёт добивать упавший провайдер. Это прямое применение resilience-стека модуля 23 к внешним зависимостям.

## 5. Rate limiting, настройки, шаблоны

**Rate limiting (модуль 16)** — на двух уровнях:
- **Провайдер**: не превышать лимиты APNS/Twilio/SES (иначе блокировка/throttle с их стороны).
- **Пользователь**: не спамить (например, не более N уведомлений/час), объединять (digest).

**Настройки и opt-out:** перед отправкой проверять предпочтения пользователя (какие каналы/категории включены) и **согласие** (compliance: GDPR, законный unsubscribe для email/SMS). Отправка без согласия — юридический риск.

**Шаблоны:** уведомление = шаблон (текст с плейсхолдерами) + данные персонализации; локализация. Разделяет контент и логику отправки.

> Три практических слоя: **rate limiting** дважды — уважать лимиты провайдера (иначе он сам вас заблокирует) и не спамить пользователя (UX + закон); **настройки/opt-out** — проверять согласие и предпочтения **до** отправки (опт-аут и unsubscribe обязательны юридически — GDPR/CAN-SPAM); **шаблоны** — отделить контент (текст, локализация) от механики доставки. Пропуск проверки opt-out — частая и дорогая (штрафы) ошибка.

## 6. Приоритеты и трекинг

```
Приоритетные очереди:
  TRANSACTIONAL (OTP, «платёж прошёл», сброс пароля) → высокий приоритет, низкая latency.
  PROMOTIONAL  (маркетинг, дайджесты)                → низкий приоритет, батч, можно отложить.
  → срочный OTP не должен ждать за миллионной маркетинговой рассылкой.
```

**Трекинг доставки:** статусы (queued → sent → delivered → failed/bounced), агрегация для аналитики (часто async, как просмотры — модуль 26). Provider callbacks/webhooks обновляют статус.

> Критичное продуктовое решение — **приоритизация**: транзакционные уведомления (OTP, алерты безопасности) латентно-критичны и не должны застревать за маркетинговой кампанией на 50M. Решается **раздельными очередями по приоритету** (или весами). **Трекинг** доставки (статусы через provider webhooks) нужен для аналитики и разбора сбоев; агрегацию делают асинхронно. Отложенная/запланированная отправка — через scheduled-очередь/таймеры.

## 7. High-level и узкие места

```
High-level:
  [Producers] → [Notification Service] → preference/opt-out → template
       → [Priority Queues: transactional | promotional]  (модуль 15)
       → [Channel Workers: push | sms | email] → [Provider Adapters] → APNS/Twilio/SES
       → [Tracking] ← webhooks (delivered/bounced)
```

| Узкое место | Решение |
|-------------|---------|
| Отказ/throttle провайдера | retry+backoff, circuit breaker, fallback-провайдер (модуль 23) |
| Лимиты провайдера | rate limiting на стороне воркеров (модуль 16) |
| Всплеск кампании на миллионы | очередь сглаживает + масштаб воркеров; приоритет защищает транзакционные |
| Дедупликация на масштабе | idempotency key + хранилище обработанных id |
| Срочные застревают за промо | приоритетные очереди |

> Финальная сборка: producers → notification service (настройки + шаблон) → **приоритетные очереди** → **воркеры по каналам** → **адаптеры провайдеров** → внешние провайдеры; статусы возвращаются через webhooks. Узкие места — вокруг **внешних зависимостей** (отказы/лимиты провайдеров → resilience-стек модуля 23 + rate limiting модуля 16 + fallback-провайдер) и **масштаба fan-out** (очередь + воркеры + приоритеты). Это система, где архитектура подчинена надёжности интеграций, а не вычислениям.

## ⚠️ Подводные камни

- **Синхронная отправка без очереди.** Кампания на миллионы заблокирует сервис/провайдера; развязывать очередью (модуль 15).
- **Без дедупликации.** At-least-once + retry → двойной OTP/двойное списание-алерт; idempotency key (модуль 23).
- **Игнор лимитов провайдера.** Превышение → провайдер throttle/блокирует вас; rate limiting воркеров (модуль 16).
- **Нет retry/DLQ.** Временный сбой провайдера → потеря уведомления; retry+backoff, DLQ для безнадёжных.
- **Нет circuit breaker.** Добиваем упавший провайдер, копим таймауты; разомкнуть + fallback (модуль 23).
- **Отправка без проверки opt-out/согласия.** Юридический риск (GDPR/CAN-SPAM), штрафы; проверять до отправки.
- **Транзакционные и промо в одной очереди.** OTP застрянет за рассылкой; приоритетные очереди.
- **Спам пользователю.** Нет пользовательского лимита/дайджеста → отписки; ограничивать частоту.
- **Зависимость от одного провайдера.** Его отказ = нет канала; fallback-провайдер.
- **Контент захардкожен в коде.** Нет шаблонов/локализации; отделить шаблон от логики.
