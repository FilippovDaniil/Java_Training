# Задача 03 — Модуль 37: Видеостриминг

## ЗАДАНИЕ
Разделите систему на два пути — upload/ingestion и streaming/playback — и спроектируйте каждый под его профиль.

## ВХОДНЫЕ ДАННЫЕ
Загрузка видео редка, но тяжела (транскодинг). Воспроизведение массово и латентно-критично. Команда сначала хотела «единый сервис, который и принимает, и отдаёт видео».

## ТРЕБОВАНИЯ
- Опишите путь upload/ingestion (что происходит, профиль нагрузки).
- Опишите путь streaming/playback (что происходит, профиль нагрузки).
- Объясните, почему их проектируют отдельно.
- Укажите главную ошибку (отдавать видео из origin/app, а не из CDN).

## ФОРМАТ РЕЗУЛЬТАТА
Путь upload → путь playback → почему раздельно → главная ошибка.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Upload: загрузка → сырое в object storage → транскодинг (CPU-heavy) → варианты → раздача в CDN; профиль — редкий, тяжёлый.
- [ ] Playback: плеер → CDN (edge близко) → чанки видео → ABR; профиль — массовый, латентно-критичный.
- [ ] Раздельно: разные профили и оптимизации (upload — пропускная транскодинга/async; playback — latency и egress через CDN).
- [ ] Главная ошибка: отдавать видео из origin/app → чудовищный egress-cost и latency; playback — из CDN.
- [ ] Отмечено: транскодинг — async (не блокировать загрузку).

## ПОДСКАЗКА
Два пути: upload (редкий, тяжёлый — async-транскодинг, object storage) и playback (массовый, latency-критичный — из CDN, ABR). Проектируют отдельно (разные оптимизации). Главная ошибка — отдавать видео из origin/app вместо CDN (egress-cost + latency).
