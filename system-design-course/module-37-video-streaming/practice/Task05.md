# Задача 05 — Модуль 37: Видеостриминг

## ЗАДАНИЕ
Спроектируйте adaptive bitrate streaming (ABR) с нарезкой на чанки и выберите протокол.

## ВХОДНЫЕ ДАННЫЕ
Пользователи смотрят с разных сетей (нестабильный мобильный интернет, стабильный Wi-Fi). Нужно плавное воспроизведение без «крутилки» буферизации при колебаниях сети.

## ТРЕБОВАНИЯ
- Объясните принцип ABR (как плеер реагирует на изменение сети).
- Опишите подготовку контента (чанки в нескольких битрейтах + manifest).
- Выберите протокол (HLS/DASH vs RTMP) и обоснуйте через CDN-friendliness.
- Объясните, почему HTTP-based протокол подходит для CDN.

## ФОРМАТ РЕЗУЛЬТАТА
Принцип ABR → подготовка (чанки+manifest) → протокол + обоснование → почему HTTP/CDN.

## КРИТЕРИИ ПРИЁМКИ
- [ ] ABR: плеер измеряет пропускную сети и выбирает битрейт следующего чанка; при ухудшении берёт чанк ниже качеством (деградация чёткости вместо буферизации).
- [ ] Подготовка: видео заранее нарезано на чанки (~2–10 с) в нескольких битрейтах; manifest (HLS/DASH) описывает варианты и сегменты.
- [ ] Протокол: HLS/DASH (chunk-based поверх HTTP); RTMP — для live-ingest/persistent, не для массовой раздачи.
- [ ] HTTP-based → чанки это обычные HTTP-объекты → кэшируются на edge CDN (CDN-friendly, модуль 10).
- [ ] Отмечено: переключение качества — по чанкам, плавно.

## ПОДСКАЗКА
ABR: плеер меряет сеть → выбирает битрейт следующего чанка (сеть просела → ниже качество, без буферизации). Контент — чанки (~2–10с) в нескольких битрейтах + manifest (HLS/DASH). Протокол HLS/DASH (HTTP, CDN-friendly: чанки = HTTP-объекты на edge), не RTMP.
