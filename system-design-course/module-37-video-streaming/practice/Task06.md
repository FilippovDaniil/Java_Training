# Задача 06 — Модуль 37: Видеостриминг

## ЗАДАНИЕ
Спроектируйте хранение видео и метаданных с tiering под профиль доступа и стоимость.

## ВХОДНЫЕ ДАННЫЕ
Хранятся: сырые видео, транскодированные варианты, чанки, метаданные (название/автор/длительность). Объём — петабайты и растёт. Доступ неравномерен: малая доля видео даёт большинство просмотров, «длинный хвост» почти не смотрят.

## ТРЕБОВАНИЯ
- Определите, что хранить в object storage, а что — в БД.
- Спроектируйте tiering по популярности (hot/cold/archive) и связь со стоимостью.
- Объясните, почему видео не место в реляционной БД.
- Свяжите профиль 80/20 доступа с выбором tier.

## ФОРМАТ РЕЗУЛЬТАТА
Object storage vs БД → tiering (hot/cold/archive) → почему не реляционная БД → 80/20 и tier.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Object/blob storage: сырые видео, варианты, чанки (петабайты, durable, дёшево — модуль 10); БД/metadata-сервис: название/автор/длительность/теги.
- [ ] Tiering: популярное → hot (быстрый доступ + CDN), непопулярное → cold/archive (дешевле, медленнее) — снижение storage cost (модуль 7).
- [ ] Реляционная БД не для видео: большие блобы → object storage; в БД только метаданные.
- [ ] 80/20: малая доля видео — большинство просмотров (hot/CDN), длинный хвост — редкий доступ (cold/archive).
- [ ] Отмечена связь tiering ↔ стоимость хранения.

## ПОДСКАЗКА
Видео (сырое/варианты/чанки) → object storage (петабайты, дёшево, durable, модуль 10); метаданные → БД. Tiering по популярности: hot (популярное+CDN) / cold / archive (хвост) — снижает storage cost (модуль 7). 80/20: малая доля даёт большинство просмотров → hot; хвост → archive.
