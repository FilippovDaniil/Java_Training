# Задача 08 — Модуль 37: Видеостриминг

## ЗАДАНИЕ
Выберите протокол стриминга (HLS/DASH vs RTMP) под сценарии и обоснуйте через CDN-совместимость.

## ВХОДНЫЕ ДАННЫЕ
Два сценария: (а) массовая раздача VOD (видео по запросу) миллионам зрителей с разными сетями; (б) приём прямой трансляции (live ingest) от стримера на сервер. Нужно выбрать протокол под каждый.

## ТРЕБОВАНИЯ
- Кратко опишите HLS/DASH и RTMP.
- Объясните, почему HTTP-based (HLS/DASH) подходит для массовой раздачи через CDN.
- Выберите протокол для (а) и (б) с обоснованием.
- Свяжите выбор с ABR и CDN-кэшированием.

## ФОРМАТ РЕЗУЛЬТАТА
HLS/DASH vs RTMP → почему HTTP для раздачи → выбор для (а)/(б) → связь с ABR/CDN.

## КРИТЕРИИ ПРИЁМКИ
- [ ] HLS/DASH: chunk-based поверх HTTP (manifest + сегменты); RTMP: persistent-соединение, низкая задержка ingest.
- [ ] HTTP-based: чанки — обычные HTTP-объекты → кэшируются на edge CDN → масштабируемая раздача (CDN-friendly).
- [ ] (а) массовая раздача VOD → HLS/DASH (CDN-кэшируемые чанки, поддержка ABR).
- [ ] (б) live ingest → RTMP (или low-latency-протокол) для приёма от стримера; затем перепаковка в HLS/DASH для раздачи.
- [ ] Связь: HLS/DASH несут ABR (несколько битрейтов в manifest) и кэшируются CDN; RTMP — не для массовой раздачи.

## ПОДСКАЗКА
HLS/DASH — chunk-based поверх HTTP (CDN-friendly: чанки = HTTP-объекты на edge, несут ABR). RTMP — persistent, низкая задержка для ingest. Массовая раздача VOD → HLS/DASH; live ingest → RTMP (затем перепаковать в HLS/DASH для раздачи).
