## Ключ (эталонные ответы)

1. Загрузить видео, транскодировать его, стримить/воспроизводить; опционально — поиск и рекомендации.
2. Storage (петабайты) и bandwidth/egress (миллиарды просмотров × мегабиты). Система оперирует огромными объёмами байтов, а не числом запросов — нагрузка измеряется в данных, не в RPS.
3. Потому что egress повторяется на каждый просмотр (млрд просмотров × мегабиты отдачи), тогда как storage оплачивается единожды за хранение. Отдача байтов из облачного origin дорога — это доминирующая статья.
4. Upload/ingestion (запись: загрузка + тяжёлый CPU-транскодинг, редкий) и streaming/playback (чтение: массовое воспроизведение из CDN, латентно-критичное). Профили нагрузки противоположны.
5. Потому что у них разные профили и оптимизации: upload — под пропускную транскодинга (async), playback — под глобальную latency и стоимость egress (CDN). Единый сервис плохо служит обоим.
6. Преобразование загруженного видео в несколько разрешений/битрейтов/кодеков. Нужен, т.к. устройства (телефон/TV) и сети (4G/Wi-Fi) различны, а кодеки должны быть совместимы.
7. Потому что транскодинг CPU-heavy и долгий (минуты); синхронно он заблокировал бы загрузку. Его выносят в асинхронный конвейер через очередь, а воркеры масштабируются под нагрузку.
8. Чтобы поддержать adaptive bitrate: разные качества для разных устройств и сетей, плеер выбирает подходящий вариант под текущие условия.
9. Adaptive bitrate streaming: видео существует в нескольких качествах (нарезанных на чанки), а плеер на лету выбирает битрейт следующего чанка под текущую пропускную сети.
10. При ухудшении сети плеер берёт следующий чанк более низкого качества — пользователь видит снижение чёткости вместо «крутилки»/буферизации; при улучшении возвращается к высокому.
11. Чтобы реализовать ABR (переключать качество по сегментам) и сделать раздачу CDN-friendly: чанки — обычные HTTP-объекты, кэшируемые на edge.
12. HLS/DASH — chunk-based стриминг поверх HTTP (manifest + сегменты), подходит для массовой раздачи. RTMP — persistent-соединение с низкой задержкой, применяется для live-ingest, а не для массовой раздачи.
13. Потому что чанки — это обычные HTTP-объекты, которые CDN кэширует и отдаёт с edge как любой статический файл, что даёт масштабируемую раздачу.
14. Видеофайлы (сырое, варианты, чанки) — в object/blob storage (петабайты, durable, дёшево). Метаданные (название, автор, длительность, теги) — в БД/metadata-сервисе.
15. Распределение данных по уровням хранения по популярности/доступу: популярное — hot (быстро + CDN), редкое — cold/archive (дешевле, медленнее). Снижает стоимость хранения петабайт.
16. Потому что это большие бинарные файлы; их место — object storage (дёшево, durable, масштабируемо). Реляционная БД для блобов неэффективна; в ней держат только метаданные.
17. Потому что CDN отдаёт видео с edge близко к пользователю (низкая latency) и поглощает egress (origin платит лишь при miss). Без CDN egress-стоимость и latency делают сервис нежизнеспособным.
18. (1) Снижение latency — отдача чанков с географически близкого edge; (2) поглощение egress — origin отдаёт лишь при промахе кэша, что радикально снижает главную статью стоимости.
19. Pull CDN тянет контент с origin при первом запросе и кэширует (хорошо для непредсказуемого/хвоста). Push CDN — контент заранее размещают на edge (pre-positioning) до запросов.
20. Заблаговременное размещение контента на edge-серверах CDN. Применяют для предсказуемо популярного контента (новые релизы), чтобы первый же зритель получал его с близкого edge.
21. Асинхронной агрегацией: события просмотра идут в очередь, фоновый агрегатор суммирует. Синхронный инкремент счётчика на миллиарды просмотров не масштабируется (модуль 26).
22. Egress/bandwidth cost (→ CDN), storage cost (→ tiering), пропускная транскодинга (→ воркеры/очередь), вирусное видео (→ кэш/pre-positioning), подсчёт просмотров (→ async).
23. Агрессивным кэшированием на CDN и pre-positioning по edge, чтобы пиковый спрос обслуживался с edge, а не бил по origin; масштабирование раздачи через CDN.
24. Netflix — ограниченный курируемый каталог: пред-кодирует контент заранее и агрессивно pre-position'ит популярное по edge (Open Connect, серверы у провайдеров). YouTube — огромный UGC-поток: транскодирует на загрузке, оптимизируется под объём и разнообразие.
25. Система про storage/egress: два пути (upload+async-транскодинг vs playback из CDN); транскодинг в несколько вариантов + чанки через очередь; ABR (HLS/DASH поверх HTTP); object storage + tiering; CDN как ядро (latency + поглощение egress) с pre-positioning популярного; async-подсчёт просмотров; стратегия зависит от предсказуемости контента (Netflix vs YouTube).
