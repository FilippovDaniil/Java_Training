# Модуль 37 — Видеостриминг (YouTube/Netflix) (референсная архитектура)

## Зачем этот модуль

Видеостриминг — система, где доминируют **хранение и пропускная способность**, а не запросы в секунду. Один файл — гигабайты, аудитория — миллиарды просмотров, и **egress-трафик становится главной статьёй стоимости**. Это первая система курса, где **CDN не опция, а ядро архитектуры**, а тяжёлый **транскодинг** определяет конвейер загрузки. Модуль синтезирует object storage и CDN (модуль 10), очереди (15) и cost-мышление (модуль 7).

Зачем это senior'у. Видео учит думать о **двух радикально разных путях** — загрузка/обработка (write, тяжёлый транскодинг) и стриминг/воспроизведение (read, чудовищный масштаб) — и о том, что **стоимость egress** диктует архитектуру (CDN как способ и ускорить, и сэкономить). Senior знает про **adaptive bitrate** (ABR), нарезку на чанки (HLS/DASH), пред-кодирование вариантов и распределение популярного контента (push CDN). Это эталон системы, где главные оси — storage, bandwidth, cost.

Он показывает полный HLD видеостриминга: два пути (upload/playback), транскодинг-конвейер, ABR, роль CDN и стоимость egress.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Загрузить видео | Огромное хранилище (петабайты) |
| Транскодинг (разные качества/форматы) | Высокий bandwidth (egress — доминанта) |
| Стриминг/воспроизведение | Низкая latency старта, минимум буферизации |
| Поиск, (опц.) рекомендации | Глобальный охват, высокая доступность |

```
Estimation (пример): загрузки — сотни часов видео/минуту (UGC).
  Storage: видео × несколько вариантов качества → ПЕТАБАЙТЫ, растёт безостановочно.
  Bandwidth (egress): млрд просмотров × мегабиты/с → ДОМИНИРУЮЩАЯ статья стоимости.
  Просмотры ≫ загрузок (read-heavy на уровне байтов, не запросов).
```

> Estimation сразу показывает, что видео — это **система про байты, а не про RPS**: главные оси — **storage** (петабайты, постоянный рост) и **bandwidth/egress** (миллиарды просмотров × мегабиты). Именно **egress — доминирующая статья стоимости** (модуль 7), и это определяет всю архитектуру: CDN нужен не только для latency, но и чтобы **не платить за egress из origin**. Просмотров несравнимо больше загрузок — система read-heavy по объёму данных.

## 2. Два пути: upload/ingestion vs streaming/playback

```
ПУТЬ 1 — Upload / ingestion (запись, редкий, тяжёлый):
  загрузка → сырое видео в object storage → ТРАНСКОДИНГ (CPU-heavy) → варианты → раздача в CDN.

ПУТЬ 2 — Streaming / playback (чтение, массовый):
  плеер → CDN (edge близко к пользователю) → отдача чанков видео → adaptive bitrate.
```

| Путь | Профиль | Главное |
|------|---------|---------|
| Upload | редкий, тяжёлый (CPU транскодинга) | async-конвейер, object storage |
| Playback | массовый, latency-критичный | CDN, ABR, чанки |

> Ключевая ментальная модель — **два совершенно разных пути**. **Upload** редок, но тяжёл (транскодинг — CPU-heavy, минуты на видео) → асинхронный конвейер. **Playback** массов и латентно-критичен → обслуживается из CDN, а не из origin. Их проектируют отдельно: загрузка оптимизируется под пропускную обработки, воспроизведение — под глобальную latency и стоимость egress. Смешивать их (отдавать видео из origin/app) — главная ошибка.

## 3. Транскодинг-конвейер

```
Upload pipeline:
  загрузка (chunked/multipart) → сырое в object storage → событие в ОЧЕРЕДЬ (модуль 15)
  → транскодеры (воркеры): кодируют в НЕСКОЛЬКО разрешений/битрейтов/кодеков + нарезают на ЧАНКИ
  → варианты в object storage → распространение в CDN → видео доступно.
```

| Зачем транскодинг | Причина |
|-------------------|---------|
| Разные устройства/экраны | телефон ≠ TV (разрешения) |
| Разные сети | 4G ≠ Wi-Fi (битрейты) → ABR |
| Разные кодеки/форматы | совместимость (H.264/H.265/AV1) |

> **Транскодинг** превращает одно загруженное видео в **множество вариантов** (разрешения × битрейты × кодеки), потому что устройства и сети различны (телефон в метро ≠ TV на оптике). Это CPU-heavy и долго → выносится в **асинхронный конвейер через очередь** (модуль 15): загрузка завершается быстро, транскодеры-воркеры обрабатывают в фоне и масштабируются горизонтально. Видео также **нарезается на чанки** (сегменты по несколько секунд) — основа adaptive bitrate (§4). Результат складывается в object storage и распространяется в CDN.

## 4. Adaptive bitrate (ABR) и протоколы

```
Видео заранее нарезано на чанки (по ~2–10 с) в НЕСКОЛЬКИХ битрейтах:
  1080p chunks, 720p chunks, 480p chunks, 240p chunks ...
Manifest (HLS/DASH) описывает доступные варианты и чанки.
Плеер: измеряет сеть → выбирает битрейт чанка → при ухудшении сети берёт чанк ниже качеством.
  → плавное воспроизведение без буферизации (деградация качества вместо «крутилки»).
```

| Протокол | Суть |
|----------|------|
| **HLS / DASH** | chunk-based поверх **HTTP**, manifest + сегменты; **CDN-friendly** (обычные HTTP-объекты) |
| RTMP | старый, persistent-соединение; для live-ingest, не для массовой раздачи |

> **Adaptive bitrate (ABR)** — ключ к плавному воспроизведению: видео заранее существует в нескольких качествах, нарезанных на чанки, а **плеер сам переключает битрейт** под текущую сеть (сеть просела → следующий чанк берётся ниже качеством). Пользователь видит снижение чёткости вместо «крутилки» буферизации. Реализуется через **HLS/DASH** — chunk-based стриминг **поверх HTTP**, что делает его **CDN-friendly** (чанки — обычные HTTP-объекты, кэшируются на edge, модуль 10). Это причина, почему стриминг ушёл от RTMP к HTTP-based протоколам.

## 5. Хранение и доставка (CDN — ядро)

| Слой | Решение |
|------|---------|
| Видеофайлы (сырое + варианты + чанки) | **object/blob storage** (модуль 10), петабайты |
| Метаданные (название, автор, длительность) | БД (часто отдельный metadata-сервис) |
| Доставка | **CDN** — отдача чанков с edge близко к пользователю |
| Холодное хранилище | tiering: непопулярное → дешёвый cold/archive tier (модуль 10, cost) |

```
CDN — ядро доставки:
  edge-серверы по миру кэшируют чанки → пользователь тянет видео с близкого edge (низкая latency).
  origin (object storage) отдаёт только при miss → огромная ЭКОНОМИЯ egress (модуль 7).
  Популярный контент: pre-positioning (push CDN) — заранее разложить по edge.
```

> **CDN — не оптимизация, а ядро** видеостриминга (модуль 10): edge-серверы по миру отдают чанки близко к пользователю (низкая latency, нет буферизации) и, что критично, **поглощают egress** — origin отдаёт лишь при промахе кэша, что радикально снижает стоимость (egress из облачного origin дорог, модуль 7). Файлы — в **object storage** (петабаты, дёшево, durable), метаданные — отдельно в БД. **Tiering** (модуль 10) удешевляет хранение: непопулярное видео уходит в cold/archive. Для предсказуемо популярного — **pre-positioning** (push CDN): разложить заранее по edge.

## 6. Метаданные, просмотры, узкие места

**Метаданные/поиск:** отдельный сервис (название, описание, теги, длительность) + поиск (модуль 17). Подсчёт **просмотров** — асинхронная агрегация (события в очередь → агрегатор), как в leaderboard/ad (не инкрементировать счётчик синхронно на каждый просмотр).

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Egress / bandwidth cost** | CDN (poглощает egress), tiering | 10, 7 |
| **Storage cost** (петабайты) | object storage + tiering (hot/cold/archive) | 10, 7 |
| **Transcoding throughput** | масштабировать воркеры, очередь, приоритет | 15 |
| **Viral / hot video** | агрессивное кэширование + pre-positioning в CDN | 10 |
| Подсчёт просмотров на масштабе | async-агрегация (не синхронный инкремент) | 26 |

> Узкие места видео — **про деньги и объём**: egress (главная статья → CDN), storage (петабайты → tiering), пропускная транскодинга (→ воркеры + очередь). **Viral video** решается кэшированием/pre-positioning в CDN. Подсчёт просмотров — async (синхронный инкремент счётчика на млрд просмотров не масштабируется — модуль 26). Это система, где оптимизация = снижение стоимости egress/storage при сохранении глобальной latency.

## 7. Netflix vs YouTube

| | YouTube (UGC) | Netflix (курируемый каталог) |
|--|---------------|------------------------------|
| Контент | пользовательский, огромный объём загрузок | ограниченный каталог фильмов/сериалов |
| Транскодинг | на загрузке, массовый поток | пред-кодирование каталога заранее (один раз, тщательно) |
| CDN | стандартный + свой | агрессивный **pre-positioning** (Open Connect — серверы у провайдеров) |
| Оптимизация | под объём загрузок и разнообразие | под идеальную доставку известного популярного контента |

> Две стратегии под разный профиль контента. **YouTube** — гигантский поток UGC: транскодинг на загрузке, оптимизация под объём и разнообразие. **Netflix** — ограниченный курируемый каталог: можно **пред-кодировать заранее** (тщательно, под все устройства) и **агрессивно pre-position'ить** популярное по edge (вплоть до серверов внутри сетей провайдеров — Open Connect), т.к. известно, что будут смотреть. Урок: архитектура доставки видео зависит от того, **предсказуем ли контент** — это влияет на стратегию транскодинга и CDN.

## ⚠️ Подводные камни

- **Отдавать видео из origin/app, не из CDN.** Чудовищный egress-cost и latency; CDN — ядро доставки (модуль 10).
- **Игнор стоимости egress.** Egress — доминирующая статья; считать её и снижать через CDN/tiering (модуль 7).
- **Синхронный транскодинг в загрузке.** CPU-heavy, минуты → блокирует; async-конвейер через очередь (модуль 15).
- **Один вариант видео для всех.** Разные устройства/сети → нужен транскодинг в несколько качеств + ABR.
- **Без чанков / не HTTP-протокол.** ABR требует чанков; HLS/DASH (HTTP) — CDN-friendly, RTMP — нет.
- **Видео в реляционной БД.** Большие файлы — в object storage (модуль 10), в БД — метаданные.
- **Без tiering.** Хранить всё в hot-tier дорого; непопулярное → cold/archive (модуль 7/10).
- **Синхронный подсчёт просмотров.** Инкремент на млрд просмотров не масштабируется; async-агрегация (модуль 26).
- **Одна стратегия для UGC и каталога.** Предсказуемый контент (Netflix) → пред-кодирование + pre-positioning; UGC (YouTube) → транскод на загрузке.
