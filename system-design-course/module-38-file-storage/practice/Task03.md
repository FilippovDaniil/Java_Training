# Задача 03 — Модуль 38: Разделение metadata vs content

## ЗАДАНИЕ
Спроектируйте разделение системы на слой метаданных и слой содержимого, обоснуйте выбор хранилищ.

## ВХОДНЫЕ ДАННЫЕ
Файл представлен своим содержимым (большое, immutable) и метаданными (имя, папка, версии, права, размер). Нужно решить, где и как хранить каждое.

## ТРЕБОВАНИЯ
- Опишите metadata service: что хранит (file tree, версии, права, список блоков) и в какой БД (обоснуйте, почему часто SQL).
- Опишите content/block storage: блоки в object storage (модуль 10), почему не в реляционной БД.
- Покажите, как файл в метаданных = упорядоченный список ссылок (хэшей) на блоки.
- Объясните, зачем разделение (независимое масштабирование, разные профили).

## ФОРМАТ РЕЗУЛЬТАТА
Слой metadata (что/где) → слой content (что/где) → связь (файл = список хэшей) → зачем разделение.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Metadata: file tree/namespace, версии, права, ссылки на блоки → БД (часто SQL ради согласованности дерева/прав).
- [ ] Content: блоки → object/blob storage (дёшево, durable, петабайты, модуль 10), НЕ в реляционной БД.
- [ ] Файл в метаданных = упорядоченный список хэшей блоков.
- [ ] Обосновано: разные профили (мелкое/часто меняется vs огромное/immutable) → независимое масштабирование.

## ПОДСКАЗКА
Два слоя: metadata (file tree/версии/права/хэши блоков) → БД; content (блоки) → object storage. Файл = упорядоченный список хэшей. Разделение даёт независимое масштабирование и подходящее хранилище под каждый профиль.
