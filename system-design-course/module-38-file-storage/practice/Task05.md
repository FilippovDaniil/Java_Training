# Задача 05 — Модуль 38: Content-addressable storage и дедупликация

## ЗАДАНИЕ
Спроектируйте content-addressable storage блоков и механизм дедупликации.

## ВХОДНЫЕ ДАННЫЕ
Тысячи пользователей хранят один и тот же популярный файл; пользователь повторно загружает уже существующий блок. Нужно избежать дублирования хранения и передачи.

## ТРЕБОВАНИЯ
- Опишите content-addressable storage: ключ блока = хэш содержимого (напр. SHA-256).
- Покажите, как идентичные блоки дают один ключ → хранятся однократно (дедуп).
- Опишите протокол заливки: клиент шлёт хэши → сервер отвечает, какие есть → клиент грузит только новые.
- Объясните, почему блоки immutable и как это связано с дедупом и версиями.

## ФОРМАТ РЕЗУЛЬТАТА
Content-addressable (хэш=ключ) → дедуп идентичных блоков → протокол заливки (проверка хэшей) → immutability.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ключ блока = хэш содержимого (SHA-256) → идентичные блоки → один ключ → хранятся однократно.
- [ ] Дедуп работает между версиями и между пользователями.
- [ ] Протокол: клиент отправляет хэши → сервер сообщает существующие → передаются только новые блоки (экономия bandwidth + storage).
- [ ] Блоки immutable (правка = новый блок/хэш), это основа дедупа и версий.

## ПОДСКАЗКА
Ключ = SHA-256(содержимое). Одинаковые блоки → один ключ → одна копия (дедуп). Перед заливкой клиент шлёт хэши, сервер отвечает «есть/нет», грузятся только новые. Блоки immutable.
