# Задача 06 — Модуль 38: Sync-протокол между устройствами

## ЗАДАНИЕ
Спроектируйте протокол синхронизации файла между несколькими устройствами одного пользователя.

## ВХОДНЫЕ ДАННЫЕ
Устройство A изменило файл. Устройства B и C должны узнать об этом и подтянуть изменения, не качая файл целиком и не опрашивая сервер постоянно.

## ТРЕБОВАНИЯ
- Опишите цепочку: A правит → грузит изменённые чанки → обновляет метаданные (новая версия).
- Опишите, как B/C узнают об изменении (notification service / long-polling / push, модуль 36) — а не постоянный polling.
- Опишите дельта-синхронизацию: B запрашивает дельту метаданных → качает только изменённые блоки.
- Объясните роль локального watcher и версий метаданных.

## ФОРМАТ РЕЗУЛЬТАТА
Цепочка sync (A→metadata→notification→B) → дельта-sync (только изменённые блоки) → роль watcher/версий.

## КРИТЕРИИ ПРИЁМКИ
- [ ] A: правка → заливка изменённых чанков → обновление метаданных (версия).
- [ ] Уведомление B/C через notification service/long-polling/push (модуль 36), не постоянный polling.
- [ ] Дельта-sync: B запрашивает дельту метаданных и качает ТОЛЬКО изменённые блоки (не весь файл).
- [ ] Описаны локальный watcher и догон по версиям метаданных.

## ПОДСКАЗКА
A правит → грузит изменённые чанки → новая версия метаданных → notification service пушит B/C → они берут дельту метаданных и качают только изменённые блоки. Watcher следит локально; постоянный polling на масштабе недопустим.
