# Задача 09 — Модуль 38: Узкие места и масштаб

## ЗАДАНИЕ
Найдите узкие места файлового хранилища на масштабе и предложите решения со ссылками на модули.

## ВХОДНЫЕ ДАННЫЕ
Система выросла: миллионы устройств синхронизируются, петабайты данных, есть популярные общие файлы. Появляются проблемы производительности и стоимости.

## ТРЕБОВАНИЯ
Разберите минимум 4 узких места:
- нагрузка на metadata service (частый sync);
- sync на масштабе (миллионы устройств);
- hot files (популярный общий файл);
- storage cost (петабайты).
Для каждого — решение и ссылка на модуль.

## ФОРМАТ РЕЗУЛЬТАТА
Таблица: узкое место → причина → решение → модуль.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Metadata service → шардирование по юзеру + кэш + лёгкие дельта-запросы (модули 13, 9).
- [ ] Sync на масштабе → notification/long-polling вместо постоянного polling (модуль 36).
- [ ] Hot files → CDN/кэш для блоков; дедуп уже снижает число копий (модуль 10).
- [ ] Storage cost → дедупликация + compression + tiering старых версий (модули 10, 7).
- [ ] Упомянута durability через репликацию object storage (модуль 12).

## ПОДСКАЗКА
Metadata → шард по юзеру + кэш (13/9). Sync → notification, не polling (36). Hot files → CDN/кэш блоков (10). Storage → дедуп + compression + tiering (10/7). Durability → репликация (12).
