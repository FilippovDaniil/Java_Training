## Ключ (эталонные ответы)

1. Загрузка/скачивание файлов, синхронизация между устройствами, шеринг и права доступа, версионирование, организация в папки (namespace).
2. Durability — потеря файла недопустима (это потеря доверия к сервису). Обеспечивается репликацией в object storage («11 nines»).
3. Storage: система оперирует петабайтами данных и их постоянным ростом — главная ось не RPS, а объём хранения.
4. Резко снижает фактический объём: идентичные файлы/блоки (дистрибутивы, картинки, документы) у тысяч пользователей хранятся однократно; плюс compression. Наивная оценка (users×files×size×versions×replication) кратно завышена.
5. Метаданные (file tree, версии, права, ссылки на блоки) — мелкие, часто меняются, требуют согласованности → БД. Содержимое — большие immutable-блоки → object storage. Слои масштабируются независимо.
6. File tree/namespace, имена, версии, права (ACL) и упорядоченные списки ссылок-хэшей на блоки. Часто реляционная БД — ради согласованности дерева и прав в транзакциях.
7. В object/blob storage (модуль 10): дёшево, durable, петабайты. Реляционная БД для больших блобов неэффективна; в ней — только метаданные и ссылки на блоки.
8. Как упорядоченный список ссылок (хэшей) на блоки, из которых файл собирается.
9. Разбиение файла на блоки фиксированного размера. Нужен ради дельта-синхронизации, дедупликации, параллельной и resumable загрузки.
10. Дельта-синхронизация (передавать только изменённые блоки), дедупликация (одинаковые блоки — однократно), параллельная загрузка блоков и resumable upload (продолжение с места обрыва).
11. Передача при изменении файла только изменённых чанков, а не файла целиком; правка одного байта в гигабайте → по сети идёт один блок.
12. Хранение, где ключ блока = хэш его содержимого (напр. SHA-256). Идентичные блоки дают один ключ и хранятся однократно.
13. Перед заливкой клиент отправляет хэши блоков; сервер сообщает, какие уже есть; передаются только новые блоки. Одинаковые блоки (между версиями и пользователями) занимают одну копию.
14. Клиент считает хэши блоков → шлёт их серверу → сервер отвечает, какие существуют → клиент грузит только отсутствующие блоки и обновляет метаданные. Экономит и bandwidth, и storage.
15. Чтобы хэш-ключ был стабилен (основа content-addressable и дедупа) и работали версии: правка не меняет блок, а создаёт новый блок с новым хэшем; старые блоки остаются для прежних версий.
16. Через notification service (модуль 36): после обновления метаданных другим устройством сервер пушит сигнал (long-polling/WebSocket); устройство запрашивает дельту метаданных и качает изменённые блоки.
17. Миллионы устройств, постоянно опрашивающих сервер, перегружают metadata-сервис. Нужен push/long-polling, чтобы сервер сам уведомлял об изменениях.
18. Два устройства правят одну версию оффлайн и расходятся. Разрешение — не терять данные тихо: создать conflict copy и/или хранить обе версии, выбор оставить пользователю.
19. Потому что молча затирает чужую правку — данные теряются без ведома пользователя; для файлов это неприемлемо.
20. Версия — это набор ссылок-хэшей блоков. Неизменённые блоки переиспользуются (дедуп), поэтому новая версия добавляет лишь изменённые блоки — хранение версий дёшево.
21. Права (ACL: чтение/запись на файл/папку) живут в метаданных. Шеринг = ссылки на те же блоки (содержимое не дублируется); общая папка указывает на общие блоки.
22. Как токенизированный/presigned URL (модуль 10): даёт доступ к содержимому без прав на аккаунт, с ограниченным сроком/областью.
23. Возобновляемая загрузка: при обрыве связи клиент продолжает с первого недозагруженного блока, а не с начала файла. Возможно потому, что файл уже разбит на независимые блоки.
24. Нагрузка на metadata service (→ шардирование по юзеру + кэш, модули 13/9); sync на масштабе (→ notification вместо polling, модуль 36); hot files (→ CDN/кэш блоков, модуль 10); storage cost (→ дедуп + compression + tiering, модули 10/7); durability (→ репликация object storage, модуль 12).
25. Два слоя (metadata в БД vs content в object storage); chunking файла на блоки; content-addressable storage (хэш = ключ) → дедупликация; дельта-синхронизация через notification service; resumable/параллельная загрузка; конфликты через conflict copy (не LWW); версии дёшевы через переиспользование блоков; права/шеринг в метаданных; главное NFR — durability.
