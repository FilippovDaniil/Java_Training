# Модуль 38 — Облачное файловое хранилище (Dropbox/Google Drive) (референсная архитектура)

## Зачем этот модуль

Облачное файловое хранилище — система, где главный вызов не «много запросов», а **синхронизация состояния файлов между устройствами** при требовании **никогда не терять данные** (durability). Здесь рождаются три идеи, которые senior обязан знать: **разделение метаданных и содержимого**, **chunking файла на блоки** и **дедупликация через content-addressable storage**. Модуль синтезирует object storage и CDN (модуль 10), очереди/уведомления (модули 15, 36), репликацию/durability (модуль 12).

Зачем это senior'у. File storage учит думать о **двух независимых слоях** — лёгкие, часто меняющиеся **метаданные** (file tree, версии, права) и тяжёлое, неизменяемое **содержимое** (блоки). Их хранят и масштабируют по-разному. Senior знает, почему файл **бьют на чанки** (дельта-синхронизация, resumable upload, дедуп), как **хэш блока становится его ключом** (content-addressable → дедуп идентичных блоков), и как устроена **синхронизация** между устройствами (передавать только изменённые чанки, разрешать конфликты без тихой потери).

Он показывает полный HLD файлового хранилища: requirements/estimation, разделение metadata/content, chunking + дедуп, sync-протокол, конфликты и шеринг.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Загрузка / скачивание файлов | **Durability** — потеря файла недопустима |
| Синхронизация между устройствами | Высокая доступность |
| Шеринг и права доступа | Масштаб storage (петабайты) |
| Версионирование, папки (namespace) | Разумная latency синхронизации |
| | Согласованность метаданных (file tree) |

```
Estimation (пример): десятки млн пользователей, сотни файлов на юзера, средний файл ~МБ.
  Storage: пользователи × файлы × размер × версии × replication → ПЕТАБАЙТЫ.
  Дедупликация и compression РЕЗКО снижают фактический объём (одни и те же файлы у многих).
  Нагрузка: write (загрузки/правки) умеренный, read (скачивания/sync) выше; sync-проверки частые, но лёгкие (метаданные).
```

> Estimation сразу показывает: главная ось — **storage** (петабайты), а главное NFR — **durability** (потеря файла = катастрофа доверия). При этом **дедупликация** (одни и те же файлы — дистрибутивы, картинки, документы — лежат у тысяч пользователей) и **compression** кратно снижают реальный объём. Запросы делятся на **тяжёлые** (передача содержимого) и **лёгкие, но частые** (sync-проверки метаданных) — это намёк на разделение системы на два слоя (§2).

## 2. Ключевая идея: разделение metadata vs content

```
ДВА НЕЗАВИСИМЫХ СЛОЯ:

  Metadata service (БД)          Content / block storage (object storage)
  ─────────────────────          ────────────────────────────────────────
  file tree / namespace          сами блоки файла (immutable)
  имена, папки, размеры          ключ блока = его хэш (content-addressable)
  версии, права доступа          дёшево, durable, петабайты (модуль 10)
  ССЫЛКИ на блоки (хэши)          НЕ хранится в реляционной БД!
  меняется часто, мелкое         меняется редко (immutable), огромное
```

| Слой | Что | Где хранится | Профиль |
|------|-----|--------------|---------|
| **Metadata** | file tree, версии, права, список блоков | БД (часто SQL — согласованность дерева) | мелкое, часто меняется |
| **Content** | блоки файла (immutable) | object/blob storage (модуль 10) | огромное, immutable, дёшево |

> Центральное решение дизайна — **разделить метаданные и содержимое**. Метаданные (структура папок, версии, права, список блоков файла) — мелкие, часто меняются, требуют согласованности → **БД** (нередко реляционная: file tree и права удобны в транзакциях). Содержимое — большие неизменяемые блоки → **object storage** (дёшево, durable «11 nines», петабайты — модуль 10). Файл в метаданных представлен как **упорядоченный список ссылок на блоки** (хэшей). Это разделение позволяет масштабировать слои независимо и делает возможными chunking, дедуп и дельта-синхронизацию.

## 3. Chunking: файл как набор блоков

```
Файл → разбивается на БЛОКИ фиксированного размера (напр. 4 МБ):
  file.bin = [block_1][block_2][block_3][block_4]
  metadata хранит: file → [hash_1, hash_2, hash_3, hash_4] (упорядоченный список)
```

| Зачем chunking | Что даёт |
|----------------|----------|
| **Дельта-синхронизация** | изменился 1 блок → передаём только его, не весь файл |
| **Дедупликация** | одинаковые блоки хранятся один раз (§4) |
| **Параллельная загрузка** | блоки грузятся/качаются параллельно |
| **Resumable upload** | разрыв связи → продолжить с недозагруженного блока, не сначала |

> **Chunking** — разбиение файла на блоки фиксированного размера — фундамент почти всех преимуществ системы. (1) **Дельта-sync**: правка в большом файле меняет один-два блока → по сети идут только они, а не гигабайт целиком. (2) **Дедупликация**: одинаковые блоки (между версиями и между пользователями) хранятся однократно (§4). (3) **Параллелизм**: блоки грузятся одновременно → быстрее. (4) **Resumable upload**: оборвалась загрузка — продолжаем с первого недозагруженного блока, а не с нуля. Метаданные файла = упорядоченный список хэшей его блоков.

## 4. Content-addressable storage и дедупликация

```
Ключ блока = ХЭШ его содержимого (напр. SHA-256):
  block_storage[ sha256(block_bytes) ] = block_bytes

Дедупликация:
  перед загрузкой блока клиент шлёт его хэш → сервер: «такой блок уже есть?»
  есть  → НЕ загружаем повторно, просто ссылаемся (экономия bandwidth + storage)
  нет   → загружаем блок
```

| Свойство | Следствие |
|----------|-----------|
| Ключ = хэш содержимого | идентичные блоки → один ключ → хранятся **один раз** |
| Блоки **immutable** | хэш не меняется; правка = новый блок с новым хэшем |
| Дедуп между юзерами/версиями | один популярный файл у тысяч юзеров → одна копия блоков |

> **Content-addressable storage**: ключом блока служит **хэш его содержимого** (SHA-256). Два одинаковых блока дают один и тот же хэш → один и тот же ключ → хранятся **однократно** — это и есть **дедупликация**. Перед заливкой клиент сначала отправляет хэши блоков; сервер отвечает, какие уже существуют, и клиент **не передаёт** уже известные блоки (экономия и bandwidth, и storage). Блоки **immutable** (правка файла создаёт новые блоки с новыми хэшами, старые остаются для версий). Дедуп работает и между версиями файла, и между разными пользователями.

## 5. Синхронизация между устройствами

```
Как клиент B узнаёт, что устройство A изменило файл?

  A правит файл → загружает изменённые чанки → обновляет metadata (новая версия)
       → NOTIFICATION SERVICE уведомляет другие устройства юзера (long-polling/push)
       → B запрашивает дельту метаданных → видит: изменились блоки X, Y
       → B качает ТОЛЬКО блоки X, Y (дельта-sync) → собирает файл локально.
```

| Механизм | Роль |
|----------|------|
| Локальный watcher (клиент) | следит за изменениями файлов на устройстве |
| Notification service (модуль 36) | пушит «есть изменения» на другие устройства (long-polling/WebSocket) |
| Дельта-синхронизация | передаём только изменённые чанки, не весь файл |
| Версии метаданных | клиент догоняет от своей версии к последней |

> **Синхронизация** — сердце системы. Клиент держит **локальный watcher**: заметил правку → залил изменённые чанки → обновил метаданные. Чтобы другие устройства узнали об этом, **notification service** (модуль 36) пушит им сигнал (long-polling/WebSocket — постоянный опрос дорог на масштабе). Получив сигнал, устройство запрашивает **дельту метаданных** (что изменилось с его версии) и качает **только изменённые блоки** (дельта-sync благодаря chunking). Полная перезаливка файла при каждой правке — главный анти-паттерн.

## 6. Конфликты, версии, шеринг

**Конфликты** (два устройства правят файл оффлайн одновременно):

```
Устройство A и B оба правят v3 оффлайн → оба пытаются создать v4.
  Решение: НЕ терять данные тихо.
  - версионирование: храним обе версии;
  - conflict copy: «file (conflicted copy from B).docx» — пользователь решает сам.
```

**Шеринг и права** (метаданные):

| Аспект | Решение |
|--------|---------|
| Версионирование | старые версии = старые наборы блоков (дёшево благодаря дедупу) |
| Конфликт правок | conflict copy / явная версия — без тихой потери (last-write-wins теряет данные) |
| Шеринг | права в metadata (ACL: кто читает/пишет); общая папка → ссылки на те же блоки |
| Публичная ссылка | токенизированный URL (presigned — модуль 10) |

> **Конфликты** неизбежны (оффлайн-правки на двух устройствах). Главное правило — **не терять данные тихо**: вместо «победил последний» (LWW молча затирает чужую правку) система создаёт **conflict copy** или хранит обе версии, отдавая решение пользователю. **Версионирование** дёшево: старая версия — это просто прежний набор хэшей блоков (неизменённые блоки переиспользуются через дедуп). **Шеринг** живёт в метаданных: ACL (права на чтение/запись), общая папка ссылается на те же блоки; публичные ссылки — токенизированные (presigned URL, модуль 10).

## 7. High-level и узкие места

```
                        ┌──────────────────────┐
   [Клиент + watcher] ──┤  Metadata service    │── БД (file tree, версии, права, хэши блоков)
        │   │           └──────────┬───────────┘
        │   │                      │ уведомления об изменениях
        │   │           ┌──────────┴───────────┐
        │   └───────────┤ Notification service │── push на другие устройства юзера
        │               └──────────────────────┘
        │  блоки (по хэшу, дедуп)
        └──────────────► Block service ──────────► Object storage (immutable блоки, петабайты)
```

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| Нагрузка на **metadata service** (частый sync) | шардирование по юзеру, кэш, лёгкие дельта-запросы | 13, 9 |
| **Sync на масштабе** (миллионы устройств) | notification/long-polling вместо постоянного polling | 36 |
| **Hot files** (популярный общий файл) | CDN/кэш для блоков, дедуп уже снижает копии | 10 |
| **Storage cost** (петабайты) | дедупликация + compression + tiering старых версий | 10, 7 |
| **Durability** | object storage «11 nines», репликация (модуль 12) | 12, 10 |

> High-level: **клиент с watcher** общается с **metadata service** (БД: дерево, версии, права, ссылки-хэши) и **block service** (заливка/скачивание блоков в object storage по хэшу). **Notification service** пушит другим устройствам сигнал об изменениях для дельта-sync. Узкие места — про **частые лёгкие операции с метаданными** (шардирование по юзеру, кэш) и **петабайтное хранилище** (дедуп + compression + tiering удешевляют). Durability обеспечивает object storage с репликацией.

## ⚠️ Подводные камни

- **Хранить содержимое файлов в реляционной БД.** Блобы — в object storage (модуль 10); в БД — только метаданные и ссылки на блоки.
- **Перезаливать весь файл при правке.** Без chunking нет дельта-sync; правка байта → гигабайты по сети. Бить на блоки.
- **Не делать дедупликацию.** Одинаковые блоки у тысяч юзеров без content-addressable storage → петабайты впустую.
- **Постоянный polling вместо notification.** Миллионы устройств, опрашивающих сервер, кладут metadata-сервис; нужен push/long-polling (модуль 36).
- **Last-write-wins для конфликтов.** Молча теряет чужие правки; нужен conflict copy / версии.
- **Игнор resumable upload.** Большой файл + обрыв связи = заливка с нуля; chunking даёт продолжение с места.
- **Mutable-блоки.** Если блок можно изменить, ломаются дедуп и версии; блоки immutable, правка = новый блок/хэш.
- **Слабая durability.** Один диск/регион — потеря файла недопустима; нужна репликация object storage (модуль 12).
- **Метаданные без согласованности.** Рассыпавшееся file tree хуже потери файла; метаданные требуют сильной согласованности (модуль 11).
