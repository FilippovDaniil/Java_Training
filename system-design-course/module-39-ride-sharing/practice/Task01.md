# Задача 01 — Модуль 39: Сервис такси / ride-sharing

## ЗАДАНИЕ
Соберите требования (FR/NFR) к ride-sharing и определите главную ось, отличающую систему от прочих.

## ВХОДНЫЕ ДАННЫЕ
Постановка: «спроектируйте Uber / Lyft». Нужно зафиксировать scope и понять, какая нагрузка доминирует (в отличие от read-heavy или storage-heavy систем).

## ТРЕБОВАНИЯ
- Сформулируйте ядро FR (запрос поездки, матчинг с ближайшим водителем, real-time локация, трекинг/ETA/цена/оплата).
- Сформулируйте NFR (latency матчинга, write-throughput локаций, HA, геораспределённость, согласованность поездки).
- Назовите главную ось (write-heavy real-time гео-поток) и почему не RPS-чтений и не storage.
- Объясните, почему локации эфемерны, а поездка — нет.

## ФОРМАТ РЕЗУЛЬТАТА
FR → NFR → главная ось → эфемерные локации vs durable-поездка → scope.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ядро FR: запрос поездки, матчинг ближайшего водителя, real-time локация водителей, трекинг/ETA/цена/оплата.
- [ ] NFR: низкая latency матчинга, огромный write-throughput location-апдейтов, HA, геораспределённость, согласованность состояния поездки.
- [ ] Главная ось: write-heavy real-time гео-поток (не RPS-чтений, не петабайты).
- [ ] Отмечено: локации эфемерны (важна последняя), поездка — durable (биллинг/споры).
- [ ] Зафиксирован scope.

## ПОДСКАЗКА
FR: запрос/матчинг/локация/трекинг/ETA/цена. NFR: latency матчинга, write-throughput локаций (главное!), HA, гео, согласованность поездки. Главная ось — write-heavy real-time гео-поток. Локации эфемерны, поездка durable.
