# Задача 02 — Модуль 39: Estimation location-throughput

## ЗАДАНИЕ
Оцените write-throughput location-апдейтов и нагрузку матчинга, покажите, что это write-heavy система.

## ВХОДНЫЕ ДАННЫЕ
Допущения (примите явно): миллионы активных водителей, апдейт локации каждые ~4 с, десятки млн поездок/день, гео-запрос на каждый запрос поездки.

## ТРЕБОВАНИЯ
- Посчитайте write QPS location-апдейтов (водители / интервал).
- Посчитайте порядок числа поездок/с и гео-запросов.
- Сравните write-нагрузку локаций с нагрузкой матчинга → покажите, что доминирует.
- Сделайте вывод: что хранить in-memory, а что — durable.

## ФОРМАТ РЕЗУЛЬТАТА
Допущения → write QPS локаций → поездки/гео-запросы → что доминирует → вывод о хранении.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Write QPS локаций = водители / интервал апдейта → сотни тысяч/с.
- [ ] Поездки/с и гео-запросы посчитаны (порядок).
- [ ] Показано: location-write доминирует над матчингом → система write-heavy.
- [ ] Вывод: локации → in-memory/Redis GEO; поездки → durable БД.

## ПОДСКАЗКА
Млн водителей / 4 с ≈ сотни тыс. апдейтов/с (write-heavy ось). Поездки/с и гео-запросы — на порядки меньше. Локации эфемерны → in-memory/Redis GEO; поездки durable → БД.
