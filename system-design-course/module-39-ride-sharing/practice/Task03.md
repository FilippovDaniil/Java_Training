# Задача 03 — Модуль 39: Geospatial index для nearby

## ЗАДАНИЕ
Спроектируйте геопространственный индекс для запроса «ближайшие водители в радиусе».

## ВХОДНЫЕ ДАННЫЕ
На запрос поездки нужно за миллисекунды найти свободных водителей рядом с rider'ом среди миллионов. Линейный перебор недопустим.

## ТРЕБОВАНИЯ
- Покажите, почему линейный перебор и обычный B-tree по координатам не подходят.
- Опишите geospatial index (geohash/quadtree/S2/H3 — модуль 17): как 2D→индексируемый ключ.
- Опишите nearby-запрос: выборка по ячейке rider'а и соседним.
- Учтите граничную проблему (ближайший может быть в соседней ячейке).

## ФОРМАТ РЕЗУЛЬТАТА
Почему не перебор/B-tree → geospatial index (выбор подхода) → nearby-запрос по ячейке+соседям → граничная проблема.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: линейный перебор миллионов недопустим; 2D-координаты не ложатся в обычный индекс.
- [ ] Описан geospatial index (geohash/quadtree/S2/H3), близкие точки → общие/соседние ячейки.
- [ ] Nearby-запрос = ячейка rider'а + соседние ячейки.
- [ ] Учтена граничная проблема (проверять соседей), ссылка на модуль 17.

## ПОДСКАЗКА
Перебор млн на запрос — нельзя; 2D не индексируется B-tree. Geospatial index (geohash/quadtree/S2/H3) кодирует область в ключ → nearby = ячейка + соседи. Граничная проблема: ближайший может быть в соседней ячейке (модуль 17).
