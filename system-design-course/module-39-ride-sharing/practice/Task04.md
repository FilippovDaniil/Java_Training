# Задача 04 — Модуль 39: Location update pipeline

## ЗАДАНИЕ
Спроектируйте конвейер приёма location-апдейтов от водителей (write-heavy путь).

## ВХОДНЫЕ ДАННЫЕ
Миллионы водителей шлют координаты каждые ~4 с (сотни тыс. write/с). Нужно держать актуальную карту для nearby-запросов и не положить хранилище.

## ТРЕБОВАНИЯ
- Опишите путь: водители → location service → geo-index/Redis GEO.
- Обоснуйте, почему НЕ персистить каждый апдейт в durable БД (throughput + эфемерность).
- Покажите, что хранится только последняя позиция водителя.
- Опишите, как добавить историю маршрутов, если нужна (async в очередь/pipeline — модуль 20).

## ФОРМАТ РЕЗУЛЬТАТА
Путь апдейта → почему in-memory/Redis GEO, не durable БД → только последняя позиция → (опц.) история async.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Путь: водитель → location service → in-memory/Redis GEO.
- [ ] Обосновано: durable-БД не выдержит сотни тыс/с и не нужна (позиции эфемерны).
- [ ] Хранится только последняя позиция (история для матчинга не нужна).
- [ ] История маршрутов (если нужна) — async в очередь/data pipeline (модуль 20), отдельно от горячего индекса.

## ПОДСКАЗКА
Водитель → location service → Redis GEO/in-memory (последняя позиция). Durable БД на каждый тик — нельзя (throughput) и не нужно (эфемерно). История маршрутов — async в pipeline (модуль 20).
