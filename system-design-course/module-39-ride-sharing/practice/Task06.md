# Задача 06 — Модуль 39: Состояние поездки (state machine)

## ЗАДАНИЕ
Спроектируйте модель состояния поездки и обоснуйте её durability в контраст с локациями.

## ВХОДНЫЕ ДАННЫЕ
Поездка проходит стадии от запроса до завершения; возможна отмена. Данные нужны для биллинга, разрешения споров и аудита.

## ТРЕБОВАНИЯ
- Опишите state machine: requested→matched→en route→started→completed (+ cancelled).
- Укажите валидные переходы.
- Обоснуйте durability (БД, не in-memory) — в отличие от эфемерных локаций.
- Опишите согласованность переходов (атомарность, идемпотентные обновления).

## ФОРМАТ РЕЗУЛЬТАТА
State machine (состояния+переходы) → durability в БД → контраст с локациями → согласованность переходов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] State machine: requested→matched→en route→started→completed, плюс cancelled; валидные переходы.
- [ ] Поездка durable (в БД), т.к. нужна для биллинга/споров/аудита.
- [ ] Явный контраст: локации эфемерны (in-memory), поездка durable (БД) — разные требования, разные хранилища.
- [ ] Переходы атомарны и идемпотентны.

## ПОДСКАЗКА
State machine: requested→matched→en route→started→completed (+cancelled). Durable в БД (биллинг/споры/аудит) — в отличие от эфемерных локаций. Переходы атомарны и идемпотентны.
