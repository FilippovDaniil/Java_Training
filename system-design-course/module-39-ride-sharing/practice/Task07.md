# Задача 07 — Модуль 39: Real-time связь (трекинг)

## ЗАДАНИЕ
Спроектируйте real-time связь для трекинга водителя на карте rider'а и обмена статусами.

## ВХОДНЫЕ ДАННЫЕ
Rider должен видеть движение водителя на карте «вживую», получать статусы (водитель назначен, прибыл, поездка началась). Нужна низкая latency без перегрузки сервера.

## ТРЕБОВАНИЯ
- Обоснуйте выбор WebSocket/push против polling (модуль 35).
- Опишите поток: позиция водителя → сервер → push на rider'а (и наоборот для статусов).
- Свяжите с connection-серверами/persistent-соединениями мессенджера (модуль 35).
- Отметьте, что трекинг берёт позицию из location-слоя (§location pipeline).

## ФОРМАТ РЕЗУЛЬТАТА
WebSocket vs polling → поток обновлений → связь с модулем 35 → источник позиции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Обосновано: WebSocket/push, а не polling (real-time, низкая latency, без лишней нагрузки) — модуль 35.
- [ ] Поток: позиция водителя → сервер → push rider'у; статусы поездки → push обеим сторонам.
- [ ] Связь с persistent-соединениями/connection-серверами мессенджера (модуль 35).
- [ ] Позиция для трекинга берётся из location-слоя.

## ПОДСКАЗКА
WebSocket/push (не polling) для live-трекинга — как мессенджер (модуль 35). Позиция водителя → сервер → push rider'у; статусы → обеим сторонам. Persistent-соединения держат connection-серверы.
