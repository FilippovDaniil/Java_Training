# Задача 09 — Модуль 39: Шардирование гео и узкие места

## ЗАДАНИЕ
Спроектируйте шардирование гео по регионам и разберите узкие места системы на масштабе.

## ВХОДНЫЕ ДАННЫЕ
Сервис работает в сотнях городов. В мегаполисах в час пик — резкий рост location-апдейтов и запросов поездок (hot regions). Единый глобальный гео-индекс перегружается.

## ТРЕБОВАНИЯ
- Опишите шардирование гео по региону/городу (модуль 13) и почему это естественно (запросы локальны).
- Разберите узкие места: location write throughput, гео-запросы, hot regions, matching latency, surge-расчёт.
- Для каждого — решение и ссылка на модуль.

## ФОРМАТ РЕЗУЛЬТАТА
Шардирование гео (по региону) → таблица узких мест (причина → решение → модуль).

## КРИТЕРИИ ПРИЁМКИ
- [ ] Шардирование гео по региону/городу (модуль 13); обосновано локальностью запросов.
- [ ] Location write throughput → in-memory/Redis GEO (не durable БД на тик).
- [ ] Гео-запросы → geospatial index (модуль 17); matching latency → индекс в памяти + локальность.
- [ ] Hot regions → шардирование гео по региону (модуль 13).
- [ ] Surge-расчёт → периодическая агрегация по зонам (модуль 26).

## ПОДСКАЗКА
Шард гео по региону/городу (модуль 13) — запросы локальны. Узкие места: write throughput (Redis GEO), nearby (geo-index, 17), hot regions (шард по региону, 13), matching latency (индекс в памяти), surge (агрегация по зонам, 26).
