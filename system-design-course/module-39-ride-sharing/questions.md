## Ключ (эталонные ответы)

1. Запрос поездки rider'ом, матчинг с ближайшим водителем, обновление локаций водителей в реальном времени, трекинг поездки, расчёт ETA и цены, оплата.
2. Поток location-апдейтов: миллионы водителей × апдейт каждые ~4 с = сотни тысяч записей/с. Это write-heavy real-time гео-поток, а не чтения и не объём хранения.
3. Для матчинга важна только последняя позиция водителя — история апдейтов не нужна (эфемерны). Поездка же нужна для биллинга, споров и аудита — её терять нельзя (durable).
4. Число активных водителей разделить на интервал апдейта: млн / 4 с ≈ сотни тысяч апдейтов/с. Это доминирующая write-нагрузка.
5. Перебор миллионов водителей на каждый запрос поездки не масштабируется по latency; обычные 2D-координаты не ложатся в B-tree.
6. Геопространственный индекс кодирует 2D-область в индексируемый ключ так, что близкие точки попадают в общие/соседние ячейки. Подходы: geohash, quadtree, S2 (Google), H3 (Uber, гексагоны).
7. Берётся ячейка точки rider'а и соседние ячейки; среди водителей в этих ячейках ищутся ближайшие в радиусе. Сводит запрос к нескольким ячейкам вместо перебора всех.
8. Ближайший водитель может оказаться в соседней ячейке (у границы). Решение — всегда проверять соседние ячейки, а не только свою (модуль 17).
9. Сотни тысяч записей/с durable-БД не выдержит, и это не нужно — позиции эфемерны (важна последняя). Запись каждого тика в durable-хранилище — пустая трата.
10. В памяти / Redis GEO: быстрый гео-индекс, хранящий последнюю позицию каждого водителя, перезаписываемую на каждый апдейт; поддерживает nearby-запросы.
11. Асинхронно: поток позиций отправляют в очередь/data pipeline (модуль 20) для аналитики/биллинга по расстоянию, отдельно от горячего гео-индекса.
12. Nearby-запрос (кандидаты рядом) → выбор (ближайший/ETA/рейтинг) → предложение водителю → подтверждение водителем.
13. Атомарным захватом статуса водителя (lock или compare-and-set): при гонке двух rider'ов на одного водителя выигрывает один, второй идёт к следующему кандидату.
14. Мобильная сеть ненадёжна, клиент повторяет запрос; без идемпотентности (idempotency key, модуль 23) повтор создал бы дублирующую поездку.
15. Сработает таймаут, и система предложит поездку следующему кандидату из списка ближайших.
16. State machine: requested→matched→driver en route→started→completed, плюс cancelled; с валидными переходами, хранится в durable БД.
17. Поездка нужна для биллинга, разрешения споров и аудита — её потеря недопустима. Локации же важны лишь как последняя позиция (эфемерны), поэтому живут в памяти.
18. Локации — высокочастотный эфемерный поток (in-memory/Redis GEO, не персистить), поездка — критичное durable-состояние (БД, согласованность). Разные требования → разные хранилища.
19. Постоянное соединение (WebSocket/push, как в мессенджере — модуль 35): позиция водителя и статусы поездки пушатся rider'у в реальном времени; connection-серверы держат соединения.
20. Polling даёт задержку и лишнюю нагрузку при миллионах клиентов; WebSocket/push обеспечивает низкую latency live-трекинга без постоянных опросов.
21. Отдельным сервисом маршрутизации по графу дорог и текущему трафику — время прибытия водителя и длительность поездки.
22. Динамический множитель цены при превышении спроса над предложением в зоне. Балансирует рынок и стимулирует водителей выезжать туда, где спрос.
23. Surge зависит от соотношения запросов и свободных водителей в гео-зоне; его считают агрегацией по зонам периодически, а не синхронно на каждый запрос (дорого, модуль 26).
24. Запросы локальны для города/региона, поэтому гео-индекс шардируют по региону (модуль 13). Hot regions — мегаполисы в час пик с резким ростом нагрузки; шардирование изолирует их и не даёт перегрузить единый индекс.
25. Главная ось — write-heavy real-time гео-поток; geospatial index (geohash/quadtree/S2/H3) для nearby; эфемерные локации в in-memory/Redis GEO (не durable БД); атомарный матчинг (один водитель ↔ одна поездка) + идемпотентность; durable state machine поездки в БД; WebSocket для трекинга; surge по зонам; шардирование гео по региону против hot regions.
