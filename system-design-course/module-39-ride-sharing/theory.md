# Модуль 39 — Сервис такси / ride-sharing (Uber/Lyft) (референсная архитектура)

## Зачем этот модуль

Ride-sharing — система, где сходятся **реальное время**, **гео** и **огромный write-throughput**. Водители шлют координаты каждые несколько секунд (миллионы апдейтов в секунду), а матчинг «ближайший водитель» — это **геопространственный запрос на масштабе**. Это первая система курса, где доминирует не чтение и не хранение, а **поток гео-апдейтов в реальном времени**. Модуль синтезирует геосервисы (модуль 17), real-time связь (модуль 35), идемпотентность/надёжность операций (модуль 23) и шардирование (модуль 13).

Зачем это senior'у. Ride-sharing учит думать о **двух потоках с противоположными требованиями**: высокочастотные **location updates** (write-heavy, эфемерные — не нужно персистить каждый в durable БД) и критичный по корректности **матчинг/состояние поездки** (durable, один водитель ↔ одна поездка). Senior знает, как хранить и запрашивать локации (**geospatial index** — geohash/quadtree/S2/H3), почему апдейты держат **в памяти/Redis GEO**, как обеспечить **идемпотентный матчинг** и зачем нужен **surge pricing**.

Он показывает полный HLD ride-sharing: requirements/estimation, geo-индекс, location pipeline, матчинг, состояние поездки, real-time связь, ETA/surge.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Запрос поездки (rider) | Низкая latency матчинга |
| Матчинг с ближайшим водителем | **Огромный write-throughput** location-апдейтов |
| Real-time локация водителей | Высокая доступность, геораспределённость |
| Трекинг поездки, ETA, цена, оплата | Согласованность состояния поездки |

```
Estimation (пример): миллионы активных водителей.
  Location updates: водители шлют координаты каждые ~4 с → driverс / 4 → ОГРОМНЫЙ write QPS
    (млн водителей / 4 c ≈ сотни тысяч апдейтов/с — write-heavy ось).
  Поездки/день: десятки млн; гео-запросы «ближайшие» — на каждый запрос поездки.
  Read (rider видит карту/трекинг) — тоже real-time, но матчинг и location-write доминируют.
```

> Estimation сразу вскрывает главную ось — **поток location-апдейтов**: миллионы водителей × апдейт каждые ~4 секунды = **сотни тысяч записей в секунду**. Это не RPS-чтений и не петабайты — это **write-heavy real-time гео-поток**. Ключевой вывод: апдейты локаций **эфемерны** (важна последняя позиция, не история) → их не нужно гонять в durable БД на каждый тик. А вот **состояние поездки** требует согласованности и durability. Два потока с противоположными требованиями (§2, §4–6).

## 2. Гео: хранение и запрос локаций

```
Задача: «найти ближайших водителей в радиусе R от точки rider'а» — БЫСТРО, на масштабе.
Проблема: 2D-координаты не индексируются обычным B-tree; линейный перебор всех водителей недопустим.

Решение — geospatial index (модуль 17):
  geohash / quadtree / S2 (Google) / H3 (Uber) — кодируют 2D-область в индексируемый ключ.
  Близкие точки → близкие/общие префиксы ячеек → быстрый nearby-запрос по ячейке и соседям.
```

| Подход | Идея |
|--------|------|
| **geohash** | префикс кодирует область; общий префикс ≈ близость |
| **quadtree** | рекурсивное деление на 4 квадранта |
| **S2 (Google)** | ячейки на сфере, иерархия |
| **H3 (Uber)** | гексагональная сетка (равные соседи) |

> Ядро системы — **геопространственный запрос** «ближайшие водители в радиусе». Линейный перебор миллионов водителей на каждый запрос недопустим, а 2D-координаты не ложатся в обычный индекс. Решение — **geospatial index** (модуль 17): geohash/quadtree/S2/H3 кодируют область в индексируемый ключ так, что близкие точки попадают в общие/соседние ячейки. Запрос «рядом» сводится к выборке по ячейке rider'а и её соседям (помним про граничную проблему — проверять соседние ячейки). Uber исторически использует **H3** (гексагоны: равноудалённые соседи).

## 3. Location update pipeline (write-heavy)

```
Водители (млн) ──каждые ~4 c──► Location service ──► IN-MEMORY geo-index / Redis GEO
                                                       (последняя позиция каждого водителя)
   НЕ персистим каждый апдейт в durable БД (это сотни тыс/с впустую):
     - важна ПОСЛЕДНЯЯ позиция, история эфемерна;
     - in-memory/Redis GEO держит актуальную карту водителей для nearby-запросов.
```

| Решение | Почему |
|---------|--------|
| In-memory / Redis GEO для позиций | сотни тыс. write/с; durable-БД не выдержит и не нужна |
| Хранить только последнюю позицию | история апдейтов не нужна для матчинга (эфемерны) |
| (Опц.) поток в очередь для аналитики | если нужна история маршрутов — async в data pipeline (модуль 20) |

> **Location pipeline** — главный write-нагруженный путь. Сотни тысяч апдейтов в секунду нельзя писать в durable БД (не выдержит, и не нужно). Позиции **эфемерны** — важна лишь **последняя** координата водителя для матчинга. Поэтому актуальную карту держат **в памяти / Redis GEO** (быстрый гео-индекс с nearby-запросами), перезаписывая позицию на каждый тик. Если бизнесу нужна история маршрутов (аналитика, биллинг по расстоянию) — её отправляют **асинхронно в очередь/data pipeline** (модуль 20), отдельно от горячего гео-индекса. Это классический пример «не персистить высокочастотный эфемерный сигнал синхронно».

## 4. Матчинг / dispatch

```
Запрос поездки rider'а:
  1. nearby-запрос к geo-index → кандидаты-водители рядом;
  2. выбор (ближайший / ETA / рейтинг) → предложение водителю;
  3. водитель принимает → ПОДТВЕРЖДЕНИЕ.

Критично: ОДИН водитель ↔ ОДНА поездка.
  гонка: два rider'а матчатся на одного водителя одновременно.
  решение: атомарный захват (lock / compare-and-set статуса водителя) + ИДЕМПОТЕНТНОСТЬ (модуль 23).
```

| Проблема | Решение |
|----------|---------|
| Два запроса → один водитель | атомарный захват статуса (lock/CAS), один выигрывает |
| Повтор запроса (retry) дублирует поездку | idempotency key (модуль 23) |
| Водитель не ответил | таймаут → следующий кандидат |

> **Матчинг (dispatch)** — критичная по корректности часть: nearby-запрос даёт кандидатов, сервис выбирает (ближайший/ETA/рейтинг) и предлагает поездку. Главное требование — **один водитель ↔ одна поездка**: при гонке (два rider'а на одного водителя) нужен **атомарный захват** статуса водителя (lock или compare-and-set) — выигрывает один. Повторы запроса (ненадёжная мобильная сеть, retry) не должны плодить поездки → **идемпотентность** через idempotency key (модуль 23). Не ответивший водитель — таймаут и переход к следующему кандидату.

## 5. Состояние поездки (state machine)

```
requested → matched → driver en route → started → completed
                  ↘ cancelled (на любом шаге до started)

Поездка — DURABLE (в отличие от эфемерных локаций):
  хранится в БД (надёжно), переходы фиксируются, история нужна (биллинг, споры, аудит).
```

| Аспект | Решение |
|--------|---------|
| Состояние | явная state machine, валидные переходы |
| Durability | поездка в durable БД (не in-memory) |
| Согласованность | переходы атомарны; idempotent-обновления |

> **Состояние поездки** — противоположность location-апдейтам: его **нельзя терять**, нужны история и согласованность (биллинг, споры, аудит). Моделируется **state machine** (requested→matched→en route→started→completed, плюс cancelled) с валидными переходами, хранится в **durable БД**, обновления атомарны и идемпотентны. Контраст с §3 — урок дизайна: в одной системе соседствуют **эфемерный высокочастотный поток** (локации, in-memory) и **критичное durable-состояние** (поездка, БД); их проектируют по-разному.

## 6. Real-time связь, ETA, surge pricing

**Real-time связь** (rider ↔ driver): обновления позиции водителя на карте rider'а, статусы — через **WebSocket/push** (постоянное соединение, как в мессенджере — модуль 35), а не polling.

**ETA**: расчёт по маршруту/трафику (отдельный сервис маршрутизации).

**Surge pricing** (динамическая цена):

```
Спрос (запросы поездок в зоне) ≫ предложение (свободные водители)
  → множитель цены растёт (surge ×1.5, ×2 ...) → балансирует спрос/предложение.
Считается по зонам (geo-ячейкам) на основе текущего соотношения.
```

| Механизм | Роль |
|----------|------|
| WebSocket/push (модуль 35) | real-time трекинг водителя у rider'а |
| ETA-сервис | маршрут/время прибытия |
| Surge pricing | динамическая цена при дисбалансе спроса/предложения по зонам |

> **Real-time связь** — как в мессенджере (модуль 35): постоянное соединение (WebSocket/push) даёт rider'у живой трекинг водителя на карте без polling. **ETA** считает отдельный сервис маршрутизации (граф дорог, трафик). **Surge pricing** — экономический регулятор: когда в гео-зоне спрос (запросы) сильно превышает предложение (свободные водители), множитель цены растёт, балансируя рынок и стимулируя водителей. Считается по **гео-зонам** (тем же ячейкам, что и матчинг).

## 7. High-level и узкие места

```
   [Rider app] ─┐                          ┌─► Geo-index / Redis GEO (позиции, nearby)
                ├─ WebSocket/push ─ Gateway ┤
  [Driver app] ─┘  (location updates,        ├─► Matching service (dispatch, захват водителя)
                    trip events)             ├─► Trip service ─► БД (state machine, durable)
                                             ├─► Pricing/ETA service (surge, маршрут)
                                             └─► Очередь ─► data pipeline (история, аналитика)
```

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Location write throughput** (сотни тыс/с) | in-memory/Redis GEO, не durable БД на каждый тик | — |
| **Гео-запросы на масштабе** | geospatial index (geohash/quadtree/S2/H3) | 17 |
| **Hot regions** (центр города, час пик) | шардирование гео по региону/городу | 13 |
| **Matching latency** | гео-индекс в памяти, локальность по региону | 17, 13 |
| **Surge-расчёт** | агрегирование по зонам, периодический пересчёт | 26 |

> High-level: rider/driver-приложения держат **WebSocket** к gateway (location updates от водителей, trip-события, трекинг для rider'а). За ним — **location service** (пишет в geo-index/Redis GEO), **matching service** (nearby-запрос + атомарный захват), **trip service** (durable state machine в БД), **pricing/ETA**. История — async в очередь/pipeline. Узкие места — про **гео на масштабе**: write-throughput локаций (in-memory), nearby-запросы (geo-индекс), **hot regions** (шардирование гео по региону — модуль 13), matching latency. Surge — async-агрегация по зонам.

## ⚠️ Подводные камни

- **Персистить каждый location update в durable БД.** Сотни тыс/с убьют БД и не нужны; позиции эфемерны → in-memory/Redis GEO.
- **Линейный перебор водителей для nearby.** Не масштабируется; нужен geospatial index (geohash/quadtree/S2/H3, модуль 17).
- **Игнор граничной проблемы geohash.** Ближайший водитель может быть в соседней ячейке; проверять соседей (модуль 17).
- **Не обеспечить «один водитель ↔ одна поездка».** Гонка матчинга → двойная назначка; нужен атомарный захват + идемпотентность (модуль 23).
- **Polling вместо WebSocket для трекинга.** Real-time карта требует push-соединения (модуль 35).
- **Хранить поездку в памяти.** Состояние поездки durable (биллинг/споры/аудит) → БД, в отличие от локаций.
- **Один глобальный geo-индекс без шардирования.** Hot regions (мегаполис в час пик) перегрузят; шардировать гео по региону (модуль 13).
- **Синхронный пересчёт surge на каждый запрос.** Дорого; агрегировать по зонам периодически.
- **Смешивать эфемерный поток и durable-состояние в одном хранилище.** Разные требования → разные решения (Redis GEO vs БД).
