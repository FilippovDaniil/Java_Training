# Задача 03 — Модуль 40: Идемпотентность платежа

## ЗАДАНИЕ
Спроектируйте идемпотентный приём платежа, исключающий двойное списание при повторах.

## ВХОДНЫЕ ДАННЫЕ
Клиент инициирует платёж, но ответ теряется в сети, и клиент (или сервис) повторяет запрос. Без защиты это второе списание (double-charge).

## ТРЕБОВАНИЯ
- Опишите проблему: повтор запроса → двойное списание.
- Спроектируйте idempotency key (модуль 23): клиент генерирует уникальный ключ на операцию.
- Опишите серверную логику: ключ обработан → вернуть прежний результат; новый → выполнить и сохранить.
- Объясните связь at-least-once + дедуп = exactly-once-эффект.

## ФОРМАТ РЕЗУЛЬТАТА
Проблема double-charge → idempotency key → серверная логика (дедуп) → at-least-once+дедуп=exactly-once.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Проблема: повтор запроса без защиты → второе списание.
- [ ] Idempotency key уникален на одну операцию, присылается клиентом (модуль 23).
- [ ] Сервер: ключ уже обработан → прежний результат (не списывать); новый → выполнить + сохранить результат под ключом.
- [ ] Объяснено: at-least-once доставка + дедуп по ключу = exactly-once-эффект (настоящий exactly-once недостижим).

## ПОДСКАЗКА
Повтор → double-charge. Idempotency key (уникален на операцию) → сервер дедуплицирует: обработан → прежний результат; новый → выполнить и сохранить. at-least-once + дедуп = exactly-once-эффект (модуль 23).
