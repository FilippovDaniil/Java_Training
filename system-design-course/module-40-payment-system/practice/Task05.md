# Задача 05 — Модуль 40: Согласованность денег

## ЗАДАНИЕ
Обоснуйте выбор strong consistency (а не eventual) для платёжной системы и его последствия.

## ВХОДНЫЕ ДАННЫЕ
В ленте соцсети (модуль 34) выбрана AP/eventual consistency. Нужно показать, почему для денег выбор противоположный, и что из этого следует.

## ТРЕБОВАНИЯ
- Сформулируйте CP-выбор (модуль 21): при partition — отказать в операции, а не рискнуть балансом.
- Сравните с лентой (AP/eventual): почему для денег eventual недопустима.
- Покажите следствие для хранилища: SQL/ACID-транзакции (атомарность двойной записи, изоляция).
- Отметьте trade-off: strong consistency стоит доступности/latency (PACELC).

## ФОРМАТ РЕЗУЛЬТАТА
CP-выбор → контраст с лентой (AP) → следствие (SQL/ACID) → trade-off (цена согласованности).

## КРИТЕРИИ ПРИЁМКИ
- [ ] CP-выбор (модуль 21): при partition отказать в операции, не показать некорректный баланс.
- [ ] Контраст с лентой: устаревший пост безвреден (AP), неверный баланс — финансовая ошибка (нужен strong).
- [ ] Следствие: SQL/ACID-транзакции для леджера/счетов (атомарность двойной записи, изоляция).
- [ ] Отмечен trade-off: согласованность ценой доступности/latency (PACELC).

## ПОДСКАЗКА
Деньги → CP (модуль 21): при partition отказать, не рискнуть балансом. Лента AP (устаревший пост ок), деньги нет (неверный баланс = ошибка). Следствие — SQL/ACID. Trade-off: strong стоит доступности/latency.
