# Задача 06 — Модуль 40: Распределённая транзакция (Saga + Outbox)

## ЗАДАНИЕ
Спроектируйте распределённую платёжную транзакцию через Saga и Outbox, обоснуйте отказ от 2PC.

## ВХОДНЫЕ ДАННЫЕ
Платёж проходит через несколько сервисов/шагов: списать у плательщика, провести через PSP, начислить получателю, уведомить. Любой шаг может упасть.

## ТРЕБОВАНИЯ
- Объясните, почему 2PC плох (блокирующий, координатор SPOF, не масштабируется — модуль 23).
- Спроектируйте Saga: последовательность локальных транзакций + компенсации при сбое.
- Покажите компенсацию для денег (reversal-проводка) при сбое шага.
- Спроектируйте Outbox: атомарная запись в БД + событие → надёжная публикация (решает dual-write).

## ФОРМАТ РЕЗУЛЬТАТА
Почему не 2PC → Saga (шаги + компенсации) → компенсация = reversal → Outbox (dual-write).

## КРИТЕРИИ ПРИЁМКИ
- [ ] 2PC отвергнут с обоснованием (блокирующий, координатор SPOF, не масштабируется).
- [ ] Saga: локальные транзакции по шагам + компенсирующие действия при сбое.
- [ ] Компенсация для денег = reversal-проводка (откат эффекта уже выполненных шагов).
- [ ] Outbox: запись в БД и событие в одной транзакции → отдельный процесс/CDC публикует → нет dual-write рассогласования.

## ПОДСКАЗКА
2PC: блокирующий, SPOF, не масштабируется → нет. Saga: локальные шаги + компенсации (для денег — reversal). Outbox: БД+событие атомарно → публикация → решает dual-write «записал, но не отправил» (модуль 23).
