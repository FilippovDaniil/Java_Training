# Задача 08 — Модуль 40: Статусная модель платежа

## ЗАДАНИЕ
Спроектируйте статусную модель (state machine) платежа с валидными переходами и аудитом.

## ВХОДНЫЕ ДАННЫЕ
Платёж проходит стадии от инициации до окончательного расчёта; возможны отказ и возврат. Каждый переход важен для аудита и разрешения споров.

## ТРЕБОВАНИЯ
- Опишите состояния: initiated → pending → authorized → captured → settled, плюс failed и refunded.
- Укажите валидные переходы (и тупиковые/терминальные состояния).
- Свяжите переходы с интеграцией PSP (pending до ответа провайдера).
- Опишите аудит каждого перехода (append-only, correlation ID — модуль 28).

## ФОРМАТ РЕЗУЛЬТАТА
Состояния → валидные переходы → связь с PSP → аудит переходов.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Состояния: initiated, pending, authorized, captured, settled, failed, refunded.
- [ ] Валидные переходы и терминальные состояния указаны.
- [ ] Pending связан с ожиданием ответа PSP; failed/refunded — обработаны.
- [ ] Каждый переход аудируется (append-only/лог + correlation ID, модуль 28).

## ПОДСКАЗКА
State machine: initiated→pending→authorized→captured→settled (+failed, +refunded). Pending = ждём PSP. Переходы валидны и аудируются (append-only + correlation ID, модуль 28).
