# Задача 09 — Модуль 40: Безопасность (PCI DSS) и узкие места

## ЗАДАНИЕ
Спроектируйте безопасность платежей по PCI DSS и разберите узкие места системы на масштабе.

## ВХОДНЫЕ ДАННЫЕ
Система обрабатывает карты и деньги: есть требования compliance (PCI DSS), а на масштабе появляются hot accounts, отказы провайдеров и рост стоимости сверки.

## ТРЕБОВАНИЯ
- Опишите PCI DSS-меры: не хранить номера карт, токенизация, шифрование at-rest/in-transit, секреты вне кода (модуль 30).
- Опишите audit log/трассируемость каждой операции.
- Разберите узкие места: согласованность vs throughput, hot accounts, отказ провайдеров, reconciliation на масштабе.
- Для каждого — решение и ссылка на модуль.

## ФОРМАТ РЕЗУЛЬТАТА
PCI DSS (токенизация/шифрование/секреты) → audit → таблица узких мест (причина → решение → модуль).

## КРИТЕРИИ ПРИЁМКИ
- [ ] PCI DSS: номера карт не хранить, токенизация, шифрование at-rest/in-transit, секреты вне кода (модуль 30).
- [ ] Audit log/трассируемость каждой операции (append-only + correlation ID, модуль 28).
- [ ] Согласованность vs throughput → узкие ACID-границы + шардирование по аккаунту (модули 13/21).
- [ ] Hot accounts → оптимизация блокировок/очередь проводок (модуль 26); отказ провайдеров → circuit breaker (модуль 23); reconciliation → батч-сверка (модуль 20).

## ПОДСКАЗКА
PCI DSS: не хранить карты → токенизация, шифрование, секреты вне кода (модуль 30). Audit (append-only + correlation ID, 28). Узкие места: согласованность/throughput (шард по аккаунту, 13/21), hot accounts (блокировки/очередь, 26), провайдеры (circuit breaker, 23), reconciliation (батч, 20).
