# Модуль 40 — Платёжная система (payment system) (референсная архитектура)

## Зачем этот модуль

Платёжная система — система, где **корректность важнее всего**: деньги нельзя ни потерять, ни списать дважды. Это полная противоположность ленте соцсети (модуль 34): не «доступность ценой временной несогласованности» (AP), а **строгая согласованность ценой доступности** (CP — модуль 21). Здесь рождаются три обязательных для senior'а идеи: **идемпотентность платежа** (против двойного списания), **double-entry ledger** (бухгалтерская книга как источник правды) и **reconciliation** (сверка с внешними банками/PSP). Модуль синтезирует CAP/выбор согласованности (модуль 21), надёжность операций — идемпотентность/Saga/Outbox (модуль 23), защиту API/PCI DSS (модуль 30).

Зачем это senior'у. Платежи учат проектировать **под корректность денег**: повтор запроса не должен списать дважды (идемпотентность), баланс обязан сходиться (двойная запись, append-only журнал), а расхождения с банком — выявляться (reconciliation). Senior знает, почему здесь **strong consistency/ACID**, а не eventual; почему распределённую транзакцию строят на **Saga + Outbox**, а не на 2PC; и почему карты **не хранят** (PCI DSS, токенизация).

Он показывает полный HLD платёжной системы: requirements/estimation, идемпотентность, double-entry ledger, согласованность, Saga/Outbox, интеграция с PSP, reconciliation, статусная модель, безопасность.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Приём платежа, перевод средств | **Корректность — деньги не терять/не дублировать** |
| Интеграция с PSP/банками | Strong consistency, **CP-выбор** (модуль 21) |
| Возвраты (refund) | Durability, аудит/трассируемость |
| История транзакций, баланс/леджер | Безопасность/compliance (PCI DSS) |
| | Разумная доступность |

```
Estimation (пример): транзакций — десятки/сотни млн в день; пики (распродажи) ×N.
  Объём не петабайтный и не «миллионы QPS» — но КАЖДАЯ транзакция критична.
  Главное не throughput, а КОРРЕКТНОСТЬ и АУДИТ каждой записи.
  Требование к согласованности: strong (баланс должен сходиться в любой момент).
```

> Estimation у платежей нетипичен: главное — **не масштаб throughput**, а **корректность каждой транзакции**. Объёмы (десятки-сотни млн транзакций/день) велики, но не сравнимы с лентой или видео; зато цена ошибки несоизмерима — двойное списание или потерянный платёж недопустимы. Поэтому ключевое NFR — **strong consistency (CP-выбор, модуль 21)**: при сетевом разделении система скорее **откажет в операции**, чем рискнёт некорректным балансом. Это задаёт всю архитектуру: ACID, идемпотентность, двойная запись, аудит.

## 2. Идемпотентность (против двойного списания)

```
Проблема: сеть ненадёжна → клиент/сервис ПОВТОРЯЕТ запрос платежа.
  Без защиты: повтор → второе списание (double-charge) — катастрофа.

Решение — idempotency key (модуль 23):
  клиент генерирует уникальный ключ на ОДНУ платёжную операцию → шлёт с запросом.
  сервер: ключ уже обработан? → вернуть прежний результат (НЕ списывать снова).
                     новый?    → выполнить, сохранить результат под ключом.
```

| Аспект | Решение |
|--------|---------|
| Повтор запроса (retry) | idempotency key → дедупликация, один эффект |
| Семантика | at-least-once доставка + дедуп = exactly-once-эффект |
| Хранение ключей | таблица обработанных ключей с результатом |

> **Идемпотентность** — первое, что проектируют в платежах. Сеть ненадёжна, клиенты и сервисы повторяют запросы; без защиты повтор означает **двойное списание**. Решение — **idempotency key** (модуль 23): уникальный ключ на одну операцию, который клиент присылает с запросом. Сервер проверяет: ключ уже обработан → возвращает прежний результат, не списывая повторно; новый → выполняет и сохраняет результат под ключом. Так **at-least-once доставка + дедупликация дают exactly-once-эффект** — настоящий exactly-once в распределёнке недостижим, а этот приём — стандарт индустрии.

## 3. Double-entry ledger (источник правды)

```
Каждая транзакция = ДВЕ записи (дебет одного счёта, кредит другого), сумма сходится:
  перевод 100: [счёт A: -100 (debit)]  [счёт B: +100 (credit)]   → итог изменения = 0.

Журнал APPEND-ONLY (неизменяемый):
  записи не редактируют и не удаляют; ошибка → компенсирующая запись (reversal).
  ledger = ИСТОЧНИК ПРАВДЫ; баланс = свёртка (сумма) записей по счёту.
```

| Свойство | Зачем |
|----------|-------|
| Двойная запись (дебет=кредит) | баланс системы всегда сходится; ошибки видны |
| Append-only (immutable) | аудит, нельзя «тихо» переписать историю денег |
| Источник правды | баланс = агрегат записей, а не отдельное мутируемое поле |
| Исправление = reversal | не правка записи, а компенсирующая проводка |

> **Double-entry ledger** (двойная бухгалтерская запись) — финансовый стандарт, прямо ложащийся на дизайн: каждая транзакция порождает **две записи** (дебет и кредит), их сумма равна нулю → баланс системы всегда сходится, а расхождение мгновенно заметно. Журнал — **append-only** (неизменяемый): записи не редактируют и не удаляют (это убило бы аудит); ошибку исправляют **компенсирующей записью** (reversal), а не правкой. Ledger — **источник правды**; баланс счёта вычисляется как свёртка его записей, а не хранится отдельным мутируемым полем (которое можно рассогласовать). Это даёт аудируемость и корректность.

## 4. Согласованность: strong, не eventual

| | Платежи | Лента (модуль 34) |
|--|---------|-------------------|
| Согласованность | **strong / ACID** | eventual |
| CAP-выбор | **CP** (модуль 21) | AP |
| При partition | отказать в операции | обслужить (устаревшие данные ок) |
| Хранилище | SQL/ACID-транзакции | NoSQL/кэш |

> **Согласованность денег — строгая (strong/ACID), не eventual.** Это сознательный **CP-выбор** (модуль 21): при сетевом разделении платёжная система скорее **откажет в операции**, чем покажет некорректный баланс или допустит двойное списание. Контраст с лентой (AP, eventual — устаревший пост безвреден): для денег eventual consistency недопустима, потому что «временно неверный баланс» означает реальную финансовую ошибку. Отсюда выбор **SQL/ACID-транзакций** для леджера и счётов: атомарность двойной записи и изоляция критичны.

## 5. Распределённая транзакция: Saga + Outbox (не 2PC)

```
Платёж затрагивает НЕСКОЛЬКО сервисов/шагов (списать, провести через PSP, начислить, уведомить).
  2PC — плохо: блокирующий, координатор = SPOF, не масштабируется (модуль 23).

Saga (модуль 23): последовательность локальных транзакций + КОМПЕНСАЦИИ при сбое.
  шаг упал → выполнить компенсирующие действия для уже сделанных шагов (откат эффекта).

Outbox (модуль 23): атомарно записать в БД и «событие на отправку» в ОДНОЙ транзакции
  → отдельный процесс/CDC публикует событие → решает dual-write problem
     («списал в БД, но не отправил в очередь» — рассогласование).
```

| Паттерн | Роль |
|---------|------|
| **Saga** | распределённая транзакция через локальные шаги + компенсации |
| **Outbox** | надёжная публикация событий (нет dual-write рассогласования) |
| Почему не 2PC | блокирующий, координатор SPOF, не масштабируется |

> Платёж — **распределённая операция** (списание, проведение через PSP, начисление, уведомление через разные сервисы). **2PC отвергают** (блокирующий, координатор — SPOF, плохо масштабируется — модуль 23). Вместо него — **Saga**: цепочка локальных транзакций с **компенсациями** (шаг упал → откатываем эффект уже выполненных шагов компенсирующими действиями; для денег компенсация = reversal-проводка). Чтобы события между шагами публиковались надёжно (без dual-write проблемы «записал в БД, но не в очередь»), применяют **Outbox** (модуль 23): запись в БД и событие сохраняются в одной транзакции, отдельный процесс/CDC публикует их.

## 6. Интеграция с PSP/банками и reconciliation

```
Внешние PSP/банки — АСИНХРОННЫ и НЕНАДЁЖНЫ:
  payment service → PSP-адаптер → провайдер (Stripe/банк): статус pending → success/failed.
  retry при сбое (+ идемпотентность!), circuit breaker при отказе провайдера (модуль 23).

RECONCILIATION (сверка):
  периодически сверять внутренний ledger с выписками банка/PSP →
  находить расхождения (наш success, у банка failed и наоборот) → расследовать/исправлять.
```

| Механизм | Роль |
|----------|------|
| Async-интеграция | статусы pending/success/failed; не блокировать на ответ провайдера |
| Retry + идемпотентность | повтор к PSP не должен задвоить платёж |
| Circuit breaker (модуль 23) | отказ провайдера не роняет систему |
| **Reconciliation** | периодическая сверка ledger ↔ банк/PSP, выявление расхождений |

> Платёжная система **зависит от внешних PSP/банков**, которые **асинхронны и ненадёжны**: ответ приходит не сразу (статусы pending→success/failed), вызовы падают. Поэтому интеграция async, с **retry** (обязательно идемпотентным — повтор к провайдеру не должен задвоить) и **circuit breaker** на случай отказа провайдера (модуль 23). Ключевой механизм корректности — **reconciliation**: периодическая **сверка внутреннего ledger с выписками банка/PSP**. Расхождения (у нас success, у банка failed, или наоборот) неизбежны в распределённой финансовой системе — их находят сверкой и расследуют/исправляют компенсирующими проводками. Без reconciliation ошибки накапливаются незаметно.

## 7. Статусная модель, безопасность, узкие места

**Статусная модель платежа** (state machine):

```
initiated → pending → authorized → captured → settled
                  ↘ failed        ↘ refunded
```

**Безопасность (PCI DSS):** карты **не хранить** — **токенизация** (хранить токен, не номер); шифрование at-rest/in-transit; секреты вне кода (модуль 30); строгая авторизация и audit log каждой операции.

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| Согласованность vs throughput | ACID-границы узкие; шардировать по аккаунту/региону | 13, 21 |
| **Hot accounts** (популярный счёт) | оптимизация блокировок, очередь проводок по счёту | 26 |
| Отказ внешних провайдеров | circuit breaker + retry + fallback | 23 |
| Reconciliation на масштабе | батч-сверка, выявление расхождений | 20 |
| Аудит/трассируемость | append-only ledger + correlation ID | 28 |

> **Статусная модель** платежа — явная state machine (initiated→pending→authorized→captured→settled, плюс failed/refunded) с валидными переходами и аудитом каждого. **Безопасность** определяется **PCI DSS**: номера карт **не хранят** — используют **токенизацию** (хранится токен, выданный провайдером), данные шифруют at-rest и in-transit, секреты держат вне кода (модуль 30), каждую операцию логируют (audit log). Узкие места — про **корректность под нагрузкой**: узкие ACID-границы и шардирование по аккаунту, **hot accounts** (оптимизация блокировок/очередь проводок), отказ провайдеров (circuit breaker), reconciliation на масштабе (батч), сквозной аудит (correlation ID — модуль 28).

## ⚠️ Подводные камни

- **Нет идемпотентности.** Повтор запроса → двойное списание; обязателен idempotency key (модуль 23).
- **Eventual consistency для денег.** Баланс обязан сходиться всегда; нужен strong/ACID, CP-выбор (модуль 21).
- **Баланс как мутируемое поле.** Рассогласуется и не аудируется; баланс = свёртка append-only ledger (двойная запись).
- **Редактировать/удалять записи ledger.** Убивает аудит; исправление — компенсирующая проводка (reversal), журнал immutable.
- **2PC для распределённой транзакции.** Блокирующий, координатор SPOF; нужны Saga + компенсации + Outbox (модуль 23).
- **Dual-write (БД + очередь раздельно).** Рассогласование «записал, но не отправил»; решает Outbox (модуль 23).
- **Синхронно блокироваться на ответ PSP.** Провайдеры async/ненадёжны; статусы pending + retry + circuit breaker (модуль 23).
- **Отсутствие reconciliation.** Расхождения с банком копятся незаметно; нужна периодическая сверка ledger ↔ PSP.
- **Хранить номера карт.** Нарушение PCI DSS; токенизация, не хранить PAN (модуль 30).
- **Нет audit log.** Финансовая система обязана быть трассируемой; append-only + correlation ID (модуль 28).
