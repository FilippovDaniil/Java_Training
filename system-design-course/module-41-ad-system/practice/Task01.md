# Задача 01 — Модуль 41: Рекламная система / агрегация кликов

## ЗАДАНИЕ
Соберите требования (FR/NFR) к рекламной системе и выделите два пути, определяющих архитектуру.

## ВХОДНЫЕ ДАННЫЕ
Постановка: «спроектируйте рекламную систему: показ объявлений + подсчёт показов/кликов для аналитики и биллинга». Нужно зафиксировать scope и увидеть раскол на два разных пути.

## ТРЕБОВАНИЯ
- Сформулируйте ядро FR (ad serving, таргетинг/выбор, трекинг показов/кликов, агрегация/аналитика, бюджет).
- Сформулируйте NFR (latency ad serving < 100 мс, write-throughput событий, точность подсчёта, eventual consistency для аналитики, масштаб).
- Назовите два пути: ad serving (read, low-latency) vs event aggregation (write-heavy).
- Объясните, почему точность подсчёта критична (биллинг).

## ФОРМАТ РЕЗУЛЬТАТА
FR → NFR → два пути → почему точность подсчёта важна → scope.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Ядро FR: показ рекламы, таргетинг/выбор объявления, трекинг показов/кликов, агрегация/аналитика, управление бюджетом.
- [ ] NFR: низкая latency ad serving (< 100 мс), огромный write-throughput событий, точность подсчёта (биллинг), eventual consistency ок для аналитики, масштаб (млрд/день).
- [ ] Выделены два пути: ad serving (read, low-latency) vs event aggregation (write-heavy).
- [ ] Объяснено: подсчёт = биллинг рекламодателю → точность критична.
- [ ] Зафиксирован scope.

## ПОДСКАЗКА
FR: serving/таргетинг/трекинг/агрегация/бюджет. NFR: latency serving <100мс, write-throughput событий, точность подсчёта, eventual для аналитики. Два пути: serving (read low-latency) vs aggregation (write-heavy). Подсчёт = биллинг → точность.
