# Задача 02 — Модуль 41: Estimation потока событий

## ЗАДАНИЕ
Оцените event QPS (показы/клики) и нагрузку ad serving, покажите, что путь событий — write-heavy.

## ВХОДНЫЕ ДАННЫЕ
Допущения (примите явно): показы млрд/день, CTR ~1% (клики = процент показов), запрос ad serving на каждую загрузку экрана, пики.

## ТРЕБОВАНИЯ
- Посчитайте event QPS (показы + клики) из млрд/день.
- Оцените нагрузку ad serving (запросов/с) и её latency-бюджет.
- Сравните: путь событий (write) vs путь serving — что по объёму, что по latency.
- Сделайте вывод о хранении/обработке (очередь+стрим для событий, кэш для serving).

## ФОРМАТ РЕЗУЛЬТАТА
Допущения → event QPS → нагрузка serving → сравнение путей → вывод об обработке.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Event QPS = млрд/день / 86400 ≈ сотни тысяч событий/с (показы+клики).
- [ ] Нагрузка ad serving оценена + latency-бюджет (< 100 мс).
- [ ] Показано: события — write-heavy (объём), serving — latency-критичен (read).
- [ ] Вывод: события → очередь + стрим-агрегация; serving → кэш.

## ПОДСКАЗКА
Млрд показов/день / 86400 ≈ сотни тыс/с (+ клики ~1%). Serving — latency <100мс. События доминируют по объёму (write-heavy) → очередь+стрим; serving → кэш.
