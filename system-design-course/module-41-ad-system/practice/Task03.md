# Задача 03 — Модуль 41: Ad serving (low-latency)

## ЗАДАНИЕ
Спроектируйте путь выдачи объявления, укладывающийся в latency-бюджет < 100 мс.

## ВХОДНЫЕ ДАННЫЕ
На каждую загрузку страницы/экрана нужно за десятки миллисекунд выбрать релевантное объявление из множества кандидатов с учётом таргетинга.

## ТРЕБОВАНИЯ
- Опишите шаги: выбор кандидатов (таргетинг) → ранжирование → возврат объявления.
- Покажите роль кэша (кандидаты/таргетинг/частые запросы) для latency.
- Объясните, почему этот путь отделён от подсчёта событий.
- Учтите фильтрацию по бюджету (не показывать объявления исчерпанных кампаний).

## ФОРМАТ РЕЗУЛЬТАТА
Шаги serving → роль кэша → отделение от подсчёта → фильтр по бюджету.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Шаги: выбор кандидатов по таргетингу → ранжирование → возврат объявления.
- [ ] Кэш кандидатов/таргетинга для соблюдения latency < 100 мс (модуль 9).
- [ ] Обосновано: serving отделён от event aggregation (разные профили).
- [ ] Объявления исчерпанных кампаний не показываются (фильтр по бюджету).

## ПОДСКАЗКА
Serving: кандидаты (таргетинг) → ранжирование → объявление, за <100мс. Кэш кандидатов/таргетинга (модуль 9). Отделён от подсчёта. Фильтровать исчерпанные кампании (бюджет).
