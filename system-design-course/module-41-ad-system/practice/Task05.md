# Задача 05 — Модуль 41: Агрегация по временным окнам

## ЗАДАНИЕ
Спроектируйте async-агрегацию событий и обоснуйте отказ от синхронного инкремента счётчиков.

## ВХОДНЫЕ ДАННЫЕ
Нужно показывать рекламодателю число показов/кликов по кампаниям/объявлениям. Наивно — инкрементировать счётчик на каждое событие, но событий миллиарды в день.

## ТРЕБОВАНИЯ
- Покажите, почему синхронный инкремент на каждое событие не масштабируется (hot row, блокировки — модуль 26).
- Спроектируйте async-агрегацию по временным окнам (кампания/объявление/минута-час).
- Опишите pre-aggregation: дашборды читают готовые агрегаты, не сырые события.
- Упомяните Lambda/Kappa (real-time приближённо + точный батч — модуль 20).

## ФОРМАТ РЕЗУЛЬТАТА
Почему не синхронный инкремент → async-агрегация по окнам → pre-aggregation → Lambda/Kappa.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объяснено: млрд инкрементов/день → hot row, блокировки, не масштабируется (модуль 26).
- [ ] Async-агрегация по временным окнам (группировка по кампании/объявлению/окну).
- [ ] Pre-aggregation: дашборды читают готовые суммы, не сырьё.
- [ ] Упомянуты Lambda/Kappa (быстро приближённо + точно батчем, модуль 20).

## ПОДСКАЗКА
Синхронный инкремент млрд/день → hot row, блокировки, нельзя (модуль 26). Async-агрегация по окнам (кампания/объявление/час), pre-aggregation для дашбордов. Lambda/Kappa — свежесть + точность (модуль 20).
