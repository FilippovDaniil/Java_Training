# Задача 07 — Модуль 41: OLAP и Lambda/Kappa

## ЗАДАНИЕ
Спроектируйте аналитический слой (OLAP) и архитектуру, дающую и свежесть, и точность.

## ВХОДНЫЕ ДАННЫЕ
Рекламодателям нужны и near-real-time метрики (сколько кликов прямо сейчас), и точные исторические отчёты (агрегаты по кампаниям за период).

## ТРЕБОВАНИЯ
- Обоснуйте OLAP/колоночное хранилище для аналитики (модуль 18), а не OLTP.
- Объясните, почему аналитические агрегаты не гонят на транзакционной БД.
- Опишите Lambda/Kappa (модуль 20): real-time приближённый слой + точный батч-пересчёт.
- Покажите, как replay из очереди даёт пересчёт при ошибке агрегации.

## ФОРМАТ РЕЗУЛЬТАТА
OLAP (почему колоночное) → почему не OLTP → Lambda/Kappa → replay для пересчёта.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Аналитика → OLAP/колоночное хранилище (большие агрегаты по немногим колонкам, модуль 18).
- [ ] Объяснено: аналитика на OLTP убивает транзакционную БД → разделение.
- [ ] Lambda/Kappa: real-time приближённый слой + точный батч (модуль 20).
- [ ] Replay из очереди → пересчёт агрегатов при ошибке.

## ПОДСКАЗКА
Аналитика → OLAP/колоночное (модуль 18), не OLTP. Lambda/Kappa: real-time приближённо + точный батч (модуль 20). Replay из очереди = пересчёт при ошибке агрегации.
