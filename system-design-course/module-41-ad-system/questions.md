## Ключ (эталонные ответы)

1. Показ релевантной рекламы (ad serving), таргетинг/выбор объявления, трекинг показов и кликов, агрегация/аналитика (дашборды), управление бюджетом кампании.
2. Ad serving (read, latency-критичный, < 100 мс — выбрать и показать объявление) и event aggregation (write-heavy — принять и посчитать миллиарды событий). Профили противоположны, поэтому пути проектируют отдельно.
3. Потому что подсчёт показов/кликов = биллинг рекламодателю: ошибка означает неверный счёт за деньги.
4. Число событий в день разделить на 86400: млрд показов/день ≈ сотни тысяч событий/с (плюс клики ~1% от показов). Это write-heavy ось.
5. Десятки миллисекунд (порядка < 100 мс): иначе реклама не успеет отобразиться при загрузке страницы/экрана.
6. Выбор кандидатов по таргетингу → ранжирование → возврат объявления (с фильтрацией исчерпанных по бюджету кампаний).
7. Чтобы уложиться в latency-бюджет: кэшируют кандидатов/таргетинг/частые запросы, ускоряя выбор объявления (модуль 9).
8. У них противоположные профили: serving — read с жёсткой latency, aggregation — write-heavy с упором на пропускную и точность. Единый путь плохо служит обоим.
9. Event collector принимает события показов/кликов → кладёт в очередь (Kafka) → стрим-процессоры агрегируют по временным окнам → результат в OLAP-store.
10. Очередь развязывает приём и обработку, буферизует и сглаживает пики (вирусная кампания), даёт retention и replay (пересчёт при ошибке).
11. Всплеск событий (сотни тысяч/с и пики) положит аналитическую БД. Очередь поглощает поток, обработка идёт в своём темпе.
12. Миллиарды инкрементов в день в одни и те же строки создают hot row, блокировки и не масштабируются (модуль 26). Нужна async-агрегация.
13. Группировка событий по (кампания, объявление, окно времени — минута/час) и суммирование внутри окна вместо инкремента на каждое событие.
14. Заранее посчитанные агрегаты: дашборды читают готовые суммы по окнам, а не сканируют сырые события.
15. Архитектуры, дающие и свежесть, и точность: real-time приближённый слой (быстрые метрики) + точный батч-пересчёт (модуль 20). Нужны, т.к. рекламодателю важны и «сейчас», и точные исторические отчёты.
16. Очередь обеспечивает at-least-once доставку (модуль 15): при retry событие может прийти повторно.
17. Дедупликацией по event_id (идемпотентность, модуль 23): повторное событие с тем же id не учитывается.
18. Настоящий exactly-once недостижим; его эффект дают at-least-once доставка (с повторами) + дедуп по event_id — клик учитывается ровно один раз.
19. Накрутка кликов ботами/спамом, искажающая биллинг. Борются фильтрацией подозрительных событий (детект ботов/аномалий) до подсчёта.
20. Аналитика — большие агрегаты по многим строкам/немногим колонкам (OLAP, колоночное хранилище, модуль 18). На OLTP такие сканы убивают транзакционную БД, поэтому пути разделяют.
21. Near-real-time учётом расхода: показ останавливают при исчерпании бюджета, а ad serving фильтрует объявления исчерпанных кампаний.
22. Равномерное растягивание показов по периоду кампании, чтобы не потратить весь бюджет в первый час.
23. Очередь хранит события (retention); при ошибке агрегации стрим-процессор перечитывает (replay) события и пересчитывает агрегаты.
24. Event write throughput (→ очередь + стрим), hot ads/кампании (→ партиционирование + окна), точность vs latency (→ Lambda/Kappa), exactly-once подсчёт (→ дедуп по event_id), latency ad serving (→ кэш).
25. Два пути (ad serving low-latency vs event aggregation write-heavy); события → очередь → стрим-агрегация по окнам (не синхронный инкремент); exactly-once-подсчёт через дедуп по event_id + click fraud detection; аналитика в OLAP + Lambda/Kappa для свежести и точности; бюджет/pacing с near-real-time учётом; ad serving ускоряется кэшем.
