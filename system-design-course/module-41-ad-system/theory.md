# Модуль 41 — Рекламная система / агрегация кликов (ad-system) (референсная архитектура)

## Зачем этот модуль

Рекламная система — это **две системы в одной**: молниеносная выдача объявлений (read, latency бюджет < 100 мс) и аналитический конвейер, переваривающий **миллиарды событий показов и кликов в день** (write-heavy). Здесь сходятся очереди и стриминг (модуль 15), конвейеры данных и Lambda/Kappa (модуль 20), OLAP (модуль 18), идемпотентность подсчёта (модуль 23) и async-агрегация (модуль 26). Это эталон системы, где **счётчики нельзя инкрементировать синхронно**.

Зачем это senior'у. Ad-system учит **разделять путь выдачи и путь учёта**: ad serving оптимизируют под latency (кэш, ранжирование кандидатов), а подсчёт событий — под пропускную и точность (поток в очередь → стрим-агрегация по окнам, не синхронный счётчик). Senior знает, почему **клик должен считаться ровно один раз** (идемпотентность/дедуп, защита от fraud), как устроены **временные окна** агрегации и почему аналитика живёт в **OLAP**, а не в OLTP.

Он показывает полный HLD рекламной системы: два пути, event ingestion, async-агрегация, exactly-once-подсчёт, OLAP, бюджет/pacing.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Показ релевантной рекламы (ad serving) | Низкая latency ad serving (< 100 мс) |
| Таргетинг и выбор объявления | **Огромный write-throughput событий** |
| Трекинг показов/кликов | Точность подсчёта (биллинг) |
| Агрегация/аналитика (дашборды) | Eventual consistency ок для аналитики |
| Управление бюджетом кампании | Масштаб (млрд событий/день) |

```
Estimation (пример): показы — млрд/день, клики — проценты от показов (CTR ~1%).
  Event QPS: млрд/день / 86400 ≈ сотни тысяч событий/с (показы+клики) → write-heavy.
  Ad serving: запрос на каждую загрузку страницы/экрана, latency-критичен (< 100 мс).
  Аналитика: агрегаты по кампаниям/объявлениям — большие сканы (OLAP), не latency-критичны.
```

> Estimation сразу вскрывает **две разные оси**: путь **выдачи** (read, миллионы запросов с жёстким latency-бюджетом < 100 мс — иначе реклама не успеет показаться) и путь **событий** (сотни тысяч показов/кликов в секунду — write-heavy поток в миллиарды/день). Точность подсчёта критична для **биллинга** (рекламодателю выставляют счёт за клики/показы), а вот аналитические агрегаты допускают eventual consistency. Этот раскол определяет всю архитектуру (§2).

## 2. Два пути: ad serving vs event aggregation

```
ПУТЬ 1 — Ad serving (read, latency-критичный, < 100 мс):
  запрос → выбор кандидатов (таргетинг) → ранжирование → вернуть объявление → ПОКАЗ.

ПУТЬ 2 — Event aggregation (write-heavy, аналитика):
  показ/клик → событие → ОЧЕРЕДЬ → стрим-агрегация по окнам → OLAP-store → дашборды/биллинг.
```

| Путь | Профиль | Оптимизация |
|------|---------|-------------|
| Ad serving | read, low-latency | кэш, быстрый выбор+ранжирование кандидатов |
| Event aggregation | write-heavy, batch/stream | очередь, async-агрегация, OLAP |

> Ключевая ментальная модель — **два пути**. **Ad serving** должен уложиться в десятки миллисекунд: выбрать кандидатов по таргетингу, отранжировать, вернуть объявление — с агрессивным кэшированием. **Event aggregation** — обратное: миллиарды событий в день нужно надёжно принять и посчитать, latency здесь не важна, важны пропускная и точность. Их проектируют отдельно: смешивать (например, синхронно инкрементировать счётчик кликов в момент показа) — фатальная ошибка (§5).

## 3. Event ingestion pipeline

```
Показы/клики (сотни тыс/с) ──► Event collector ──► ОЧЕРЕДЬ (Kafka, модуль 15)
                                                       │
                                       стрим-процессоры (Flink/Spark Streaming)
                                                       │
                                          агрегация по окнам ──► OLAP-store
```

| Компонент | Роль |
|-----------|------|
| Event collector | приём событий показов/кликов (лёгкий, быстрый) |
| Очередь (Kafka, модуль 15) | буфер/развязка, retention, replay, сглаживание пиков |
| Стрим-процессоры | агрегация по временным окнам (модуль 20) |

> События не пишут напрямую в аналитическую БД — их **буферизуют в очередь** (Kafka — модуль 15): это развязывает приём от обработки, сглаживает пики, даёт retention и **replay** (пересчитать при ошибке агрегации). Лёгкий **event collector** только принимает события и кладёт в лог; тяжёлую работу делают **стрим-процессоры** (Flink/Spark Streaming), агрегируя по временным окнам. Очередь здесь — несущая конструкция: без неё всплеск событий (вирусная кампания) положил бы хранилище.

## 4. Агрегация: async по временным окнам

```
НЕ синхронный инкремент счётчика на каждое событие (модуль 26):
  млрд инкрементов/день в одну строку = hot row, блокировки, не масштабируется.

ВМЕСТО — async-агрегация по ОКНАМ:
  стрим группирует события по (кампания, объявление, минута/час) → суммирует в окне →
  пишет агрегат (pre-aggregation) → дашборды читают готовые агрегаты, не сырые события.
```

| Приём | Зачем |
|-------|-------|
| Async-агрегация (не синхронный инкремент) | млрд событий нельзя считать построчно онлайн (модуль 26) |
| Временные окна (минута/час/день) | группировка → один агрегат на окно |
| Pre-aggregation | дашборды читают готовые суммы, а не сырьё |
| Lambda/Kappa (модуль 20) | real-time приближённо + точный пересчёт батчем |

> **Главный урок модуля**: счётчики показов/кликов **нельзя инкрементировать синхронно** на каждое событие (модуль 26) — миллиарды инкрементов в день создают hot row, блокировки и не масштабируются. Вместо этого — **async-агрегация по временным окнам**: стрим-процессор группирует события по (кампания, объявление, окно времени), суммирует и пишет **pre-aggregated** результат. Дашборды читают готовые агрегаты. Для сочетания свежести и точности применяют **Lambda/Kappa** (модуль 20): быстрый приближённый счёт в реальном времени + точный пересчёт батчем.

## 5. Exactly-once / дедуп подсчёта

```
Проблема: событие клика может прийти ДВАЖДЫ (retry, at-least-once доставка очереди — модуль 15).
  Двойной подсчёт клика = неверный биллинг рекламодателю.

Решение:
  - идемпотентность/дедуп по event_id (модуль 23): клик с тем же id считается ОДИН раз;
  - at-least-once доставка + дедуп = exactly-once-эффект подсчёта;
  - click fraud detection: фильтровать боты/спам-клики до подсчёта.
```

| Требование | Решение |
|------------|---------|
| Клик считается один раз | дедуп по event_id (идемпотентность, модуль 23) |
| Дубли из-за retry/at-least-once | at-least-once + дедуп = exactly-once-эффект |
| Накрутка/боты (click fraud) | фильтрация подозрительных событий до подсчёта |

> Точность подсчёта = деньги: рекламодателю выставляют счёт за клики, поэтому **клик должен считаться ровно один раз**. Но очередь даёт **at-least-once** (модуль 15) — событие может прийти дважды (retry). Решение — **дедупликация по event_id** (идемпотентность — модуль 23): повторное событие игнорируется. At-least-once + дедуп = **exactly-once-эффект** подсчёта (как в платежах — модуль 40). Дополнительно — **click fraud detection**: боты и накрутка фильтруются до учёта, иначе биллинг искажается.

## 6. OLAP, бюджет и pacing

**Аналитика → OLAP** (модуль 18): агрегаты по кампаниям/объявлениям/времени — это большие сканы по многим строкам/немногим колонкам → **колоночное хранилище** (DWH), не OLTP. Сырые события и pre-aggregated данные ложатся в OLAP-слой; дашборды строят отчёты там.

**Бюджет/pacing**:

```
Кампания имеет бюджет. Нельзя перерасходовать (показать больше, чем оплачено).
  - учёт расхода в (около)реальном времени → остановить показ при исчерпании бюджета;
  - pacing: равномерно растягивать показы на период (не потратить всё за час).
```

| Аспект | Решение |
|--------|---------|
| Аналитические запросы | OLAP/колоночное хранилище (модуль 18), не OLTP |
| Не перерасходовать бюджет | near-real-time учёт расхода → стоп при исчерпании |
| Pacing | равномерное распределение показов по периоду |

> Аналитика — классический **OLAP** (модуль 18): агрегаты по большим объёмам → колоночное хранилище/DWH, отдельно от latency-критичного ad serving. **Бюджет** требует near-real-time учёта расхода (на стыке быстрого приближённого счёта и точной агрегации): показ останавливают при исчерпании бюджета, чтобы не показать рекламы больше оплаченного. **Pacing** растягивает показы равномерно по периоду кампании, а не тратит весь бюджет в первый час.

## 7. High-level и узкие места

```
   [Клиент] ──► Ad server ──► выбор кандидатов + ранжирование + кэш ──► объявление
                   │
                   ▼ (показ/клик)
   [Event collector] ──► ОЧЕРЕДЬ (Kafka) ──► стрим-агрегаторы (окна, дедуп) ──► OLAP-store
                                                                                  │
                                                              дашборды / биллинг / pacing
```

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Event write throughput** (сотни тыс/с) | очередь + стрим-обработка, не синхронная запись | 15, 20 |
| **Hot ads / кампании** | партиционирование событий, агрегация по окнам | 13, 26 |
| **Точность vs latency** подсчёта | Lambda/Kappa (быстро приближённо + точно батчем) | 20 |
| **Exactly-once подсчёт** | дедуп по event_id на масштабе | 23 |
| Ad serving latency | кэш кандидатов/таргетинга, быстрый ранкер | 9 |

> High-level: **ad server** обслуживает запросы (кандидаты → ранжирование → кэш → объявление); **event collector** принимает показы/клики и кладёт в **очередь**; **стрим-агрегаторы** считают по окнам с дедупом и пишут в **OLAP**, откуда читают дашборды, биллинг и pacing. Узкие места — про **поток событий**: write throughput (очередь + стрим), hot ads (партиционирование), точность vs latency (Lambda/Kappa), exactly-once (дедуп). Ad serving масштабируют кэшем.

## ⚠️ Подводные камни

- **Синхронный инкремент счётчика на каждое событие.** Млрд/день → hot row, блокировки; нужна async-агрегация по окнам (модуль 26).
- **Писать события напрямую в БД.** Всплеск кладёт хранилище; буфер — очередь (Kafka, модуль 15).
- **Подсчёт без дедупа.** At-least-once → двойной клик → неверный биллинг; дедуп по event_id (модуль 23).
- **Игнор click fraud.** Боты/накрутка искажают биллинг; фильтровать до подсчёта.
- **Смешивать ad serving и aggregation.** Разные профили (latency vs throughput) → разные пути.
- **Аналитика на OLTP.** Большие агрегаты убивают транзакционную БД; OLAP/колоночное (модуль 18).
- **Нет pacing/контроля бюджета.** Перерасход = показ рекламы сверх оплаченного; near-real-time учёт + стоп.
- **Только точный батч без real-time.** Свежесть теряется; Lambda/Kappa даёт и то, и другое (модуль 20).
