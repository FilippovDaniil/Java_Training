## Ключ (эталонные ответы)

1. Обход веб-страниц, парсинг и извлечение ссылок и контента, построение индекса, дедупликация, уважение robots.txt, свежесть/recrawl.
2. Масштаб против politeness: нужно обойти миллиарды страниц (высокий общий throughput), но нельзя перегружать отдельные сайты (жёсткий per-domain лимит).
3. Inverted index (поисковый индекс, модуль 17): извлечённый контент токенизируется и индексируется (term → документы).
4. Скорость = целевой объём / срок обхода (тысячи-десятки тысяч страниц/с). Storage = сырые страницы (blob, модуль 10) + индекс + граф ссылок → петабайты.
5. URL frontier (очередь обхода), fetcher (загрузка), parser (извлечение ссылок/контента), dedup (URL и контент), storage + indexer (хранение + построение индекса).
6. Очередь URL на обход. Две цели: приоритизация (важные/свежие страницы раньше) и politeness (не более N запросов/с на домен).
7. Двухуровневой схемой: front queues упорядочивают по приоритету, back queues группируют по хосту с задержкой (crawl-delay) между запросами к одному домену.
8. Потому что он должен одновременно соблюдать приоритет (не строго порядок поступления) и politeness (разносить запросы к одному хосту во времени) — это приоритетно-вежливая очередь.
9. Взять URL из frontier (с учётом приоритета/politeness) → проверить robots.txt → fetch → parse (контент, ссылки) → store → добавить невиденные ссылки во frontier.
10. Это URL-дедуп: проверка, видели ли мы URL. Без неё новые ссылки на уже обойдённые страницы возвращаются в очередь.
11. Обход зациклится на взаимных ссылках страниц и захлебнётся дублями — фактически не продвинется.
12. Обязательство не вредить чужим серверам: уважение robots.txt (Disallow, crawl-delay), per-domain rate limit и back-off при ошибках/5xx.
13. Правила обхода сайта: какие пути запрещены (Disallow) и как часто можно обращаться (crawl-delay).
14. Глобальный лимит либо замедлит обход везде, либо позволит перегрузить отдельный сайт. Нагрузка должна ограничиваться на каждый хост отдельно (модуль 16).
15. Нельзя ускорить обход одного домена (politeness ограничивает per-host). Высокий суммарный throughput достигается только параллелизмом по многим доменам.
16. URL-дедуп (видели ли этот URL) и контент-дедуп (одинаковый/похожий контент по разным URL — зеркала, копии).
17. Хранить точное множество миллиардов URL дорого по памяти. Bloom filter компактен и вероятностен: отвечает «точно не видели» или «возможно видели».
18. Через checksums (точные дубли) и simhash/shingles (near-duplicate detection) — похожий контент даёт близкие хэши, и его не индексируют повторно.
19. Бесконечное пространство URL (календарь до бесконечности, динамические параметры), поглощающее ресурсы. Защита: лимит глубины, лимит URL на домен, детект паттернов, нормализация URL.
20. Приводит разные формы одного URL к канонической → снижает дубли и помогает не попадать в ловушки из вариаций параметров.
21. Извлечённый контент токенизируется и заносится в inverted index (term → документы, модуль 17), который и используется для поиска.
22. Периодический переобход страниц для свежести индекса. Частота зависит от изменчивости страницы: новостные сайты переобходят чаще, архивные — реже.
23. Партиционированием по хосту/домену (модуль 13): воркер владеет набором доменов (это локализует politeness), плюс координация через общий «seen»-стор, чтобы не дублировать работу.
24. Масштаб frontier (→ распределённый frontier, партиция по хосту), politeness vs throughput (→ параллелизм по доменам), дедуп на масштабе (→ bloom + simhash), детект ловушек (→ лимиты/нормализация), распределённая координация (→ партиция + общий seen-стор).
25. Конвейер frontier→fetcher→parser→dedup→storage/indexer (BFS); URL frontier балансирует приоритет и politeness; politeness (robots.txt + per-domain rate limit) обязательна, trade-off с throughput; дедупликация двух видов (bloom для URL, simhash для контента); защита от ловушек (лимиты/нормализация); inverted index (модуль 17) + recrawl для свежести; распределённый обход партицией по хосту.
