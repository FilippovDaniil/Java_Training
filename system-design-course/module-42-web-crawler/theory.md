# Модуль 42 — Веб-краулер / поисковый индекс (web crawler) (референсная архитектура)

## Зачем этот модуль

Веб-краулер — система обхода **миллиардов страниц**, где главные вызовы не похожи на прочие: **politeness** (не перегрузить чужие сайты), **дедупликация на масштабе** (не качать одно и то же дважды), **ловушки** (бесконечные пространства) и **распределённая координация**. Это эталон управляемого BFS-обхода графа веба с приоритетной очередью. Модуль синтезирует rate limiting (модуль 16), inverted index и поиск (модуль 17), шардирование (модуль 13), object storage (модуль 10).

Зачем это senior'у. Краулер учит проектировать **URL frontier** (очередь обхода, балансирующую приоритет и вежливость), **дедупликацию двух видов** (URL — bloom filter; контент — simhash для near-duplicates), **детект ловушек** и **распределённый обход** (партиционирование по хосту). Senior понимает, что краулер — это вход в **поисковый индекс** (inverted index — модуль 17), и что **politeness против throughput** — фундаментальный trade-off этой системы.

Он показывает полный HLD краулера: requirements/estimation, URL frontier, crawl loop, politeness, дедупликация, ловушки, построение индекса, распределённый обход.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Обход веб-страниц (crawl) | **Огромный масштаб** (млрд страниц) |
| Парсинг, извлечение ссылок/контента | **Politeness** — не перегружать сайты |
| Построение индекса | Свежесть (актуальность данных) |
| Дедупликация | Устойчивость к плохому HTML/ошибкам |
| Уважение robots.txt, свежесть/recrawl | Расширяемость (новые парсеры/типы) |

```
Estimation (пример): миллиарды страниц, recrawl периодически.
  Скорость обхода: страниц/с = (целевой объём) / (срок обхода) → тысячи-десятки тыс. страниц/с.
  Storage: страницы (сырой HTML в blob — модуль 10) + индекс + граф ссылок → петабайты.
  Politeness ограничивает скорость на ОДИН домен (нельзя долбить чужой сайт).
```

> Estimation вскрывает напряжение: нужно обойти **миллиарды страниц** за разумный срок (тысячи-десятки тысяч страниц/с суммарно), но при этом **нельзя перегружать отдельные сайты** (politeness — не более N запросов/с на домен). Хранилище петабайтное (сырой HTML в blob — модуль 10, индекс, граф ссылок). Главное противоречие — **масштаб против вежливости**: общий throughput высок, но per-domain жёстко ограничен (§3, §5).

## 2. Архитектура краулера

```
        ┌──────────────┐  URL    ┌─────────┐  HTML   ┌────────┐  ссылки  ┌──────────┐
        │ URL frontier │────────►│ Fetcher │────────►│ Parser │─────────►│  Dedup   │
        │ (очередь)    │◄────────│ (загр.) │         │(извлеч)│  новые   │(bloom/   │
        └──────────────┘ новые URL└─────────┘         └───┬────┘  URL     │ simhash) │
                                                          │ контент       └────┬─────┘
                                                          ▼                     │ уникальные
                                                   Storage (blob) + Indexer ◄───┘
```

| Компонент | Роль |
|-----------|------|
| **URL frontier** | очередь URL на обход (приоритет + politeness) |
| **Fetcher** | загрузка страницы по URL |
| **Parser** | извлечение ссылок и контента из HTML |
| **Dedup** | отсев виденных URL и near-duplicate контента |
| **Storage + Indexer** | хранение страниц + построение inverted index |

> Краулер — конвейер: **frontier** выдаёт URL → **fetcher** качает страницу → **parser** извлекает ссылки и контент → новые ссылки проходят **dedup** и возвращаются во frontier (так растёт обход), контент идёт в **storage** и **indexer**. Это **BFS-обход графа веба**: страница — узел, ссылки — рёбра. Цикл самоподдерживающийся: каждая страница порождает новые URL для обхода.

## 3. URL frontier

```
Frontier балансирует ДВЕ цели:
  ПРИОРИТЕТ — важные/свежие страницы обходить раньше (PageRank, частота изменений);
  POLITENESS — не более N запросов/с на ОДИН домен (отдельная очередь на хост).

Двухуровневая схема:
  front queues (по приоритету) → back queues (по хосту, с задержкой между запросами к хосту).
```

| Цель | Механизм |
|------|----------|
| Приоритизация | front-очереди по важности (свежесть, ранг) |
| Politeness | back-очереди по хосту + crawl-delay между запросами к домену |

> **URL frontier** — сердце краулера, и его сложность в том, что он одновременно решает **две задачи**: обходить важное раньше (**приоритет** — свежие/высокоранговые страницы) и **не перегружать домены** (**politeness**). Классическая двухуровневая схема: **front queues** упорядочивают по приоритету, **back queues** группируют по хосту так, что к одному домену запросы идут с задержкой (crawl-delay). Frontier — это не просто FIFO, а приоритетно-вежливая очередь.

## 4. Crawl loop (BFS-обход)

```
loop:
  url = frontier.next()                  # с учётом приоритета + politeness
  if not allowed(url): continue          # robots.txt
  html = fetch(url)                      # загрузка
  content, links = parse(html)           # извлечение
  store(content)                         # в blob + indexer
  for link in links:
      if not seen(link):                 # URL-дедуп (bloom filter)
          frontier.add(link)             # новые URL — в очередь
```

> **Crawl loop** реализует BFS: взять URL из frontier → проверить robots.txt → загрузить → распарсить → сохранить контент → добавить **новые** (невиденные) ссылки во frontier. Ключевое — `seen(link)`: без проверки виденных URL обход зациклится (страницы ссылаются друг на друга) и захлебнётся дублями. Граф веба обходится «вширь», фактический порядок задаёт приоритет frontier.

---

## 5. Politeness (robots.txt + rate limit)

```
POLITENESS — обязательство не вредить чужим серверам:
  - robots.txt: уважать запреты на обход (Disallow), crawl-delay;
  - per-domain rate limit (модуль 16): не более N запросов/с на ОДИН хост;
  - back-off при ошибках/5xx сервера.
```

| Правило | Зачем |
|---------|-------|
| robots.txt (Disallow/crawl-delay) | сайт явно задаёт, что и как часто можно обходить |
| Per-domain rate limit (модуль 16) | не перегрузить один сервер (даже если общий throughput высок) |
| Back-off при ошибках | не добивать падающий сайт |

> **Politeness** — то, что отличает краулер от DDoS. Даже при высоком суммарном throughput нагрузка **на один домен** строго ограничена: уважается **robots.txt** (Disallow, crawl-delay), применяется **per-domain rate limiting** (модуль 16 — отдельный лимит на хост, не глобальный), при ошибках сервера — back-off. Это и есть фундаментальный trade-off: **politeness против throughput** — нельзя ускорить обход одного сайта, можно лишь параллелить по многим доменам.

## 6. Дедупликация и ловушки

**Два вида дедупликации**:

```
URL-дедуп: видели ли мы этот URL? → BLOOM FILTER (компактный, вероятностный — модуль 9/14):
  «точно нет» или «возможно да»; экономит память на млрд URL.

Контент-дедуп: одинаковый/похожий контент по разным URL? → checksums + SIMHASH/shingles:
  near-duplicate detection (зеркала, копии) — не индексировать дубли.
```

**Crawler traps** (ловушки):

```
Бесконечные пространства (календарь до ∞, динамические параметры) → spider trap.
  Защита: ограничение глубины, лимит URL на домен, детект паттернов, нормализация URL.
```

| Проблема | Решение |
|----------|---------|
| Повтор URL | bloom filter «видели ли URL» (компактно для млрд) |
| Near-duplicate контент | simhash/shingles, checksums |
| Crawler traps | лимит глубины/URL на домен, детект паттернов, нормализация URL |

> **Дедупликация двух видов**. **URL-дедуп**: для миллиардов URL хранить полный set дорого → **bloom filter** (вероятностный, компактный — модуль 9/14): отвечает «точно не видели» или «возможно видели». **Контент-дедуп**: разные URL могут давать одинаковый/похожий контент (зеркала, копии) → **simhash/shingles** для near-duplicate detection, чтобы не индексировать дубли. **Crawler traps** — бесконечные пространства (календари, динамические параметры) — отлавливают лимитом глубины, лимитом URL на домен, нормализацией URL и детектом паттернов.

## 7. Индекс, свежесть, распределённый обход и узкие места

**Inverted index** (модуль 17): извлечённый контент токенизируется и индексируется (term → список документов) — это и есть переход краулер → **поисковый индекс**.

**Свежесть/recrawl**: страницы меняются → периодический переобход; частота recrawl зависит от частоты изменений страницы (новостной сайт чаще, архив реже).

**Распределённый обход**: партиционирование по **хосту/домену** (модуль 13) — один воркер отвечает за набор доменов (упрощает politeness: лимит на домен в одном месте) + координация (что уже обойдено).

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| Масштаб **frontier** (млрд URL) | распределённый frontier, партиция по хосту | 13 |
| **Politeness vs throughput** | параллелить по доменам, не по одному хосту | 16 |
| **Дедуп на масштабе** | bloom filter (URL) + simhash (контент) | 9, 14 |
| Детект **traps** | лимиты глубины/URL, нормализация | — |
| Распределённая координация | партиция по хосту, общий «seen»-стор | 13 |

> Извлечённый контент строит **inverted index** (модуль 17) — краулер питает поиск. **Свежесть** требует **recrawl** с частотой по изменчивости страницы. **Распределённый обход** партиционируют по **хосту** (модуль 13): воркер владеет доменами → politeness решается локально. Узкие места — про масштаб: распределённый frontier, politeness-vs-throughput (параллелизм по доменам), дедуп (bloom + simhash), детект ловушек, координация «что обойдено».

## ⚠️ Подводные камни

- **Игнор robots.txt / per-domain rate limit.** Краулер превращается в DDoS; politeness обязательна (модуль 16).
- **Ускорять обход одного домена.** Politeness ограничивает per-host; throughput растят параллелизмом по многим доменам.
- **Нет URL-дедупа.** Обход зацикливается на взаимных ссылках; bloom filter «видели ли URL» (модуль 9/14).
- **Хранить полный set URL точно.** Млрд URL — дорого по памяти; bloom filter (вероятностный, компактный).
- **Нет контент-дедупа.** Зеркала/копии раздувают индекс; simhash/shingles для near-duplicates.
- **Без детекта ловушек.** Spider trap (бесконечный календарь) поглотит ресурсы; лимит глубины/URL, нормализация.
- **Хранить сырой HTML в реляционной БД.** Это блобы → object storage (модуль 10).
- **Глобальный rate limit вместо per-domain.** Либо медленно везде, либо перегруз отдельных сайтов; нужен per-host.
- **Игнор свежести.** Индекс устаревает; recrawl по частоте изменений страницы.
