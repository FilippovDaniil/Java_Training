# Задача 02 — Модуль 43: Estimation

## ЗАДАНИЕ
Оцените нагрузку лидерборда и покажите, что главное — стоимость ранга, а не объём данных.

## ВХОДНЫЕ ДАННЫЕ
Допущения (примите явно): млн активных игроков, частота обновлений счёта (игровые события), частота чтений (top-K и «мой ранг»).

## ТРЕБОВАНИЯ
- Оцените объём данных (млн записей «пользователь → счёт»).
- Посчитайте write-нагрузку (обновлений счёта/с) и read-нагрузку (запросов ранга/топа/с).
- Покажите, что объём данных мал, а узкое место — стоимость вычисления ранга при таких частотах.
- Сделайте вывод: нужна структура с дешёвыми обновлением и рангом.

## ФОРМАТ РЕЗУЛЬТАТА
Допущения → объём данных → write/read нагрузка → вывод: узкое место — ранг, не storage.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Объём данных оценён (млн записей — мегабайты, мелкие).
- [ ] Write (обновления счёта/с) и read (ранг/топ/с) посчитаны.
- [ ] Показано: данных мало, но частые ранги при O(N) — неподъёмны.
- [ ] Вывод: нужна структура с O(log N) обновлением и рангом.

## ПОДСКАЗКА
Данные: млн × (user+score) ≈ мегабайты (мелкие). Write: события/с; read: ранг/топ часто. Узкое место — не storage, а ранг при O(N). Вывод — нужна O(log N) структура.
