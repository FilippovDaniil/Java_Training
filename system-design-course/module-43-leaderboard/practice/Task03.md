# Задача 03 — Модуль 43: Почему не SQL ORDER BY/COUNT

## ЗАДАНИЕ
Покажите, почему наивное вычисление ранга в реляционной БД не масштабируется.

## ВХОДНЫЕ ДАННЫЕ
Лидерборд хранится как таблица `(user, score)`. Нужно отвечать на «мой ранг» и «top-K» при миллионах игроков и тысячах запросов/с.

## ТРЕБОВАНИЯ
- Покажите наивный SQL: ранг = `COUNT(*) WHERE score > my_score`, top-K = `ORDER BY score DESC LIMIT K`.
- Оцените сложность ранга (O(N) скан/подсчёт на запрос).
- Объясните, почему при частых запросах + обновлениях это не масштабируется.
- Сделайте вывод о необходимости специализированной структуры (sorted set).

## ФОРМАТ РЕЗУЛЬТАТА
Наивный SQL → сложность O(N) → почему не масштабируется → вывод о sorted set.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Наивный SQL приведён (COUNT WHERE score > X; ORDER BY LIMIT K).
- [ ] Сложность ранга = O(N) скан/подсчёт на каждый запрос.
- [ ] Объяснено: млн пользователей × тысячи запросов ранга/с → неподъёмно (каждый запрос трогает большую часть данных).
- [ ] Вывод: нужна структура с O(log N) рангом (Redis Sorted Set).

## ПОДСКАЗКА
Ранг = COUNT(*) WHERE score > X → O(N) скан. Top-K = ORDER BY LIMIT. При млн × тысячи запросов/с — не масштабируется. Вывод: Redis Sorted Set (O(log N)).
