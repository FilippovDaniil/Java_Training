## Ключ (эталонные ответы)

1. Записать/обновить счёт, узнать ранг пользователя, получить top-K лидеров, ранг/счёт вокруг пользователя (соседи), временные лидерборды (день/неделя).
2. Стоимость вычисления ранга, а не storage: данных мало (млн записей «пользователь → счёт» — мегабайты), но запрос ранга наивно дорог.
3. Ранг = «сколько счетов выше моего». Наивно это требует подсчёта/скана по всем счетам — O(N) на каждый запрос.
4. Стоимость ранга: объём данных мал, а узкое место — частые запросы ранга при O(N) и высокая частота обновлений.
5. Ранг = `SELECT COUNT(*) WHERE score > my_score` (O(N) скан), top-K = `ORDER BY score DESC LIMIT K`.
6. При миллионах пользователей и тысячах запросов ранга в секунду каждый запрос сканирует/считает большую часть таблицы — нагрузка не выдерживается.
7. Структура Redis, где элементы с числовым score хранятся отсортированными по score, с быстрыми операциями ранга и диапазона.
8. ZADD (обновить счёт) — O(log N), ZREVRANK (ранг) — O(log N), ZREVRANGE (top-K/диапазон) — O(log N + range), ZSCORE (счёт) — O(1).
9. Skip list (упорядоченность и операции за O(log N)) + hash-таблица (доступ к элементу по ключу за O(1)).
10. Заменяет O(N) скан на O(log N) операции: и обновление счёта, и ранг, и top-K становятся дешёвыми — задача, неподъёмная в SQL, решается напрямую.
11. ZREVRANGE board 0 K-1 (с WITHSCORES при необходимости) — первые K по убыванию счёта.
12. ZREVRANK board user — позиция пользователя сверху рейтинга.
13. Получить ранг (ZREVRANK), затем ZREVRANGE rank-k rank+k — элементы вокруг пользователя.
14. Вторичным критерием (например, временем достижения счёта, закодированным в score), чтобы при равных очках порядок был однозначным.
15. Отдельным sorted set на период (board:дата, board:неделя); событие счёта обновляет все актуальные периоды (daily + weekly + all-time).
16. Чтобы старые периодные лидерборды автоматически истекали и не копились вечно; очистка становится автоматической.
17. Ранг — глобальная величина: чтобы знать позицию, нужны все счета. При разбиении по узлам точный глобальный ранг требует объединения данных всех шардов.
18. Часто достаточно: данные мелкие (млн записей), и один ZSET помещается в память узла; шардирование вводят лишь когда это перестаёт быть так.
19. Партиция + merge: взять top-K из каждого шарда и объединить (top-K мал). Глобальный точный ранг при этом всё равно сложен.
20. Бакеты/перцентили («вы в топ-5%») вместо точной позиции #N. Применяют на огромном масштабе, когда точный ранг не нужен — осознанный trade-off точность/масштаб.
21. Redis in-memory: при сбое узла данные в памяти теряются, а лидерборд (особенно all-time) терять нельзя.
22. Из durable backing: пересобрать ZSET из источника правды (событий счёта/БД) или восстановить через Redis-persistence (RDB/AOF).
23. Репликами ZSET для масштабирования чтений и кэшированием top-K; запись идёт на основную реплику.
24. Hot leaderboard (→ реплики для чтений + кэш top-K), write throughput обновлений (→ пакетирование), шардирование рангов (→ merge для top-K, приближённые ранги для глобального), durability ZSET (→ Redis-persistence + durable backing store).
25. Ранг на масштабе труден (SQL O(N) не масштабируется); Redis Sorted Set даёт O(log N) (skip list + hash) для обновления/ранга/top-K; запросы top-K/ранг/диапазон вокруг; временные лидерборды через отдельные ZSET с TTL; глобальный ранг труден к шардированию → приближённые ранги; персистентность поверх in-memory (durable backing); hot leaderboard масштабируют репликами и кэшем.
