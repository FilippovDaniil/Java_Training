# Модуль 43 — Leaderboard / рейтинги (leaderboard) (референсная архитектура)

## Зачем этот модуль

Лидерборд — обманчиво простая система: «показать топ игроков и ранг пользователя». Но **вычисление ранга на масштабе** — нетривиальная задача: наивный `ORDER BY` / `COUNT(*) WHERE score > X` в SQL — это O(N) на запрос, и при миллионах пользователей он не масштабируется. Это эталон системы, где правильная **структура данных** (Redis Sorted Set) решает задачу, которую неправильная (реляционная таблица) делает неподъёмной. Модуль синтезирует кэширование/in-memory (модуль 9), узкие места (модуль 26), шардирование (модуль 13).

Зачем это senior'у. Лидерборд учит видеть, что **выбор структуры данных — часть дизайна**: O(log N) операции **sorted set** (skip list) против O(N) сканов SQL. Senior знает запросы лидерборда (top-K, ранг, диапазон вокруг пользователя), как делать **временные лидерборды** (TTL), как **шардировать ранги** (трудно!) и когда переходить к **приближённым рангам** на огромном масштабе. И всегда — **персистентность** поверх in-memory.

Он показывает полный HLD лидерборда: requirements/estimation, проблема ранга, Redis Sorted Set, запросы, временные окна, шардирование/приближённость, персистентность.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Записать/обновить счёт пользователя | Низкая latency чтений (ранг/топ) |
| Узнать ранг пользователя | Real-time обновления счёта |
| Top-K лидеров | Масштаб (млн пользователей) |
| Ранг/счёт вокруг пользователя (относительный) | Точность рангов (или осознанная приближённость) |
| Временные лидерборды (daily/weekly) | |

```
Estimation (пример): млн активных пользователей, частые обновления счёта (игровые события).
  Обновления счёта/с: события игры → десятки-сотни тыс/с (write).
  Чтения: top-K и «мой ранг» — частые, latency-критичны.
  Главное — не объём данных (млн записей мелкие), а СТОИМОСТЬ вычисления ранга.
```

> Estimation у лидерборда нетипичен: данных мало (млн записей «пользователь → счёт» — мегабайты), но **дорогая операция — вычисление ранга**. Узнать «какой я по счёту среди миллионов» наивно означает посчитать, сколько счетов выше — это **скан O(N)** на каждый запрос. При частых чтениях ранга и обновлениях счёта (десятки-сотни тыс/с) наивный подход не выживает. Поэтому ось дизайна — **не storage, а структура данных для ранга** (§3, §4).

## 2. Проблема: ранг на масштабе

```
Наивно (SQL):
  ранг = SELECT COUNT(*) FROM scores WHERE score > :my_score    -- O(N) скан!
  top-K = SELECT * FROM scores ORDER BY score DESC LIMIT K       -- сортировка/индекс-скан

При млн пользователей и тысячах запросов/с это НЕ масштабируется:
  каждый запрос ранга сканирует/считает большую часть таблицы.
```

| Подход | Стоимость ранга | Проблема |
|--------|-----------------|----------|
| SQL `COUNT(*) WHERE score > X` | O(N) на запрос | скан таблицы, не масштабируется |
| SQL `ORDER BY score LIMIT K` | сортировка/индекс | дорого при частых запросах + обновлениях |

> **Ключевая проблема** — ранг. В реляционной БД «мой ранг» = `COUNT(*) WHERE score > my_score` — **O(N) скан** на каждый запрос; top-K = `ORDER BY ... LIMIT K`. При миллионах пользователей, частых обновлениях счёта и тысячах запросов ранга в секунду это **не масштабируется** (каждый запрос трогает большую часть данных). Нужна структура, где и обновление счёта, и запрос ранга — дёшевы. Это **Redis Sorted Set** (§3).

---

## 3. Решение: Redis Sorted Set (ZSET)

```
Redis Sorted Set (ZSET) — элементы с числовым score, отсортированы по score:
  ZADD board <score> <user>        # добавить/обновить счёт — O(log N)
  ZREVRANK board <user>            # ранг (позиция сверху) — O(log N)
  ZREVRANGE board 0 K-1            # top-K — O(log N + K)
  ZSCORE board <user>              # счёт пользователя — O(1)
  ZREVRANGE board r-k r+k          # диапазон вокруг ранга — O(log N + range)

Внутри: skip list (упорядоченность, O(log N)) + hash (доступ по элементу за O(1)).
```

| Операция | Команда | Сложность |
|----------|---------|-----------|
| Обновить счёт | `ZADD` | O(log N) |
| Ранг пользователя | `ZREVRANK` | O(log N) |
| Top-K | `ZREVRANGE 0 K-1` | O(log N + K) |
| Диапазон вокруг | `ZREVRANGE r-k r+k` | O(log N + range) |

> **Redis Sorted Set (ZSET)** — структура, созданная ровно под лидерборд: элементы с числовым score хранятся отсортированными. Обновление счёта (`ZADD`), запрос ранга (`ZREVRANK`), top-K (`ZREVRANGE`) — все **O(log N)** вместо O(N). Внутри ZSET — **skip list** (даёт упорядоченность и O(log N)) + **hash** (доступ к элементу по ключу за O(1)). In-memory → низкая latency. Это превращает неподъёмную в SQL задачу в дешёвую: правильная структура данных и есть решение дизайна.

## 4. Запросы лидерборда

```
Top-K:                ZREVRANGE board 0 K-1 WITHSCORES        # топ-10 лидеров
Ранг пользователя:    ZREVRANK board <user>                  # «вы #1234»
Счёт пользователя:    ZSCORE board <user>
Вокруг пользователя:  rank = ZREVRANK; ZREVRANGE board rank-5 rank+5   # «соседи по рейтингу»
```

| Запрос | Как |
|--------|-----|
| Top-K лидеров | `ZREVRANGE 0 K-1` |
| «Мой ранг» | `ZREVRANK <user>` |
| Соседи по рейтингу | ранг → `ZREVRANGE rank±k` |

> Три типичных запроса лидерборда покрываются ZSET напрямую. **Top-K** — `ZREVRANGE 0 K-1`. **Ранг пользователя** — `ZREVRANK`. **Диапазон вокруг пользователя** («покажи тех, кто рядом со мной») — сначала `ZREVRANK` для своего ранга, затем `ZREVRANGE rank-k rank+k`. Все дёшевы. **Ties** (равные счета) решают вторичным критерием (например, временем достижения счёта — кодируется в score, чтобы ранний игрок был выше).

## 5. Временные лидерборды

```
Daily / weekly / monthly лидерборды:
  отдельный sorted set на период:  board:2026-06-28,  board:2026-W26 ...
  установить TTL → старые лидерборды автоматически истекают (не копятся вечно).
  событие счёта пишется во ВСЕ актуальные периоды (daily + weekly + all-time).
```

| Аспект | Решение |
|--------|---------|
| Период | отдельный sorted set на день/неделю/месяц |
| Очистка старых | TTL на ключ (автоистечение) |
| Обновление | счёт пишется во все актуальные периоды |

> **Временные лидерборды** (за день/неделю) делают через **отдельный sorted set на период** (`board:2026-06-28`) с **TTL** — старые лидерборды автоматически истекают и не копятся вечно. Событие счёта обновляет **все актуальные** периоды сразу (daily + weekly + all-time). Это просто и масштабируемо: каждый период независим, очистка автоматическая.

## 6. Шардирование и приближённые ранги

```
Проблема масштаба: один ZSET на ОЧЕНЬ много пользователей → один узел = узкое место.
  Шардировать ранги ТРУДНО: ранг — глобальная величина (нужно знать все счета).

Подходы при огромном масштабе:
  - один большой ZSET, пока влезает (часто достаточно: данные мелкие);
  - партиция + merge для top-K (top из каждого шарда → объединить) — но глобальный ранг сложен;
  - ПРИБЛИЖЁННЫЕ ранги: бакеты/перцентили («вы в топ-5%») вместо точной позиции #N.
```

| Подход | Когда |
|--------|-------|
| Один большой ZSET | пока влезает в память узла (часто достаточно) |
| Партиция + merge для top-K | top-K масштабируется; глобальный ранг — трудно |
| Приближённые ранги (бакеты/перцентили) | огромный масштаб, точный ранг не нужен |

> **Шардирование лидерборда трудно**, потому что ранг — **глобальная величина** (чтобы знать позицию, нужны все счета). Практика: пока один ZSET влезает в память — этого часто достаточно (данные мелкие). Для top-K помогает **партиция + merge** (взять top из каждого шарда → объединить), но **точный глобальный ранг** при шардировании сложен. На огромном масштабе переходят к **приближённым рангам** — бакеты/перцентили («вы в топ-5%»), что снимает необходимость точной позиции #N. Осознанный выбор точность vs масштаб.

## 7. Персистентность, high-level и узкие места

```
   [Score event] ──► Score service ──► Redis ZSET (быстрые ранги/топ)
                          │
                          └──────────► Durable store (источник правды счетов)
   [Чтение ранга/топ] ──► Redis ZSET (O(log N))
```

**Персистентность**: Redis — in-memory, нужен durable backing (события счёта / периодический снапшот / durable БД как источник правды), чтобы при сбое восстановить ZSET. Redis-persistence (RDB/AOF) или восстановление из durable store.

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Hot leaderboard** (один популярный) | реплики ZSET для чтений, кэш top-K | 9, 12 |
| **Write throughput** обновлений | пакетирование, реплика на запись | 26 |
| **Шардирование рангов** | top-K через merge; приближённые ранги | 13 |
| **Точность vs масштаб** | бакеты/перцентили на огромном масштабе | — |
| Durability ZSET | Redis-persistence + durable backing store | 12 |

> High-level: **score service** принимает события счёта → обновляет **Redis ZSET** (быстрые ранги/топ) и **durable store** (источник правды). Чтения ранга/топа идут в ZSET за O(log N). **Персистентность** обязательна: Redis in-memory → нужен durable backing (события/снапшот/БД) для восстановления. Узкие места: hot leaderboard (реплики для чтений + кэш top-K), write throughput (пакетирование), шардирование рангов (merge/приближённость), durability (Redis-persistence + backing store).

## ⚠️ Подводные камни

- **Считать ранг через SQL `COUNT(*) WHERE score > X`.** O(N) скан на запрос, не масштабируется; нужен sorted set (O(log N)).
- **`ORDER BY ... LIMIT K` на каждый запрос топа.** Дорого при частых чтениях + обновлениях; ZSET `ZREVRANGE`.
- **Только in-memory без персистентности.** Сбой Redis → потеря лидерборда; durable backing store/Redis-persistence (модуль 12).
- **Пытаться точно шардировать глобальный ранг.** Ранг глобален → трудно; top-K через merge, для остального — приближённые ранги.
- **Игнор временных периодов.** Вечный all-time лидерборд без daily/weekly; отдельные ZSET с TTL.
- **Не обрабатывать ties.** Равные счета дают неоднозначный порядок; вторичный критерий (время) в score.
- **Хранить вечно все периоды.** Старые лидерборды копятся; TTL на периодные ключи.
- **Ad-hoc структура вместо sorted set.** Изобретать ранжирование руками вместо готового O(log N) ZSET.
