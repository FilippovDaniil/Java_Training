# Задача 03 — Модуль 44: Job store и механизм планирования

## ЗАДАНИЕ
Спроектируйте хранилище задач и эффективный механизм поиска «созревших» задач.

## ВХОДНЫЕ ДАННЫЕ
Среди миллионов задач нужно за каждый тик быстро находить те, чьё время срабатывания наступило (next_run_time <= now). Полный скан недопустим.

## ТРЕБОВАНИЯ
- Опишите job store (durable БД): определения, расписание, состояние, история.
- Спроектируйте поиск due-задач: индекс по next_run_time + polling, ИЛИ timing wheel/бакеты по времени.
- Объясните, почему полный скан не масштабируется.
- Покажите, как механизм превращает «найти due среди млн» в дешёвую операцию.

## ФОРМАТ РЕЗУЛЬТАТА
Job store (что хранит) → поиск due (индекс/timing wheel) → почему не скан → дешевизна операции.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Job store (durable БД): определения, расписание, состояние, история.
- [ ] Поиск due: индекс по next_run_time + polling ИЛИ timing wheel/бакеты по времени.
- [ ] Объяснено: полный скан млн задач на каждый тик не масштабируется.
- [ ] Показано: индекс/timing wheel делает поиск due дешёвым.

## ПОДСКАЗКА
Job store (durable БД): определения/расписание/состояние/история. Поиск due: индекс по next_run_time + polling, или timing wheel (бакеты по времени). Полный скан нельзя.
