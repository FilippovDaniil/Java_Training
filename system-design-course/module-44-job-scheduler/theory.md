# Модуль 44 — Распределённый планировщик задач (job-scheduler) (референсная архитектура)

## Зачем этот модуль

Планировщик задач — система, которая обязана **выполнить задачу в нужный момент и не потерять её**, оставаясь распределённой и без единой точки отказа. Здесь сходятся очереди (модуль 15), идемпотентность и надёжность операций (модуль 23), координация/leader election (модуль 22). Главные вызовы не похожи на read/write-heavy системы: **точность срабатывания**, **at-least-once execution без двойного запуска**, **scheduler как SPOF** и **thundering herd в полночь** (тысячи cron-задач на одно время).

Зачем это senior'у. Job-scheduler учит **развязывать планирование и выполнение** (scheduler решает «что и когда», workers исполняют через очередь), обеспечивать **at-least-once + идемпотентность** (exactly-once срабатывание недостижимо), переживать **смерть воркера** (lease/visibility timeout) и **смерть планировщика** (leader election против двойного триггера). Senior знает, как эффективно находить «созревшие» задачи на масштабе (индекс по next_run / timing wheel) и как сгладить hot time.

Он показывает полный HLD планировщика: requirements/estimation, job store, механизм планирования, развязка с выполнением, надёжность, отказы, leader election, recurring/catch-up.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Запланировать задачу (разовая в T / cron) | **Надёжность** — задача обязана выполниться |
| Выполнить через воркеры | At-least-once execution |
| Retry упавших, DLQ | Масштаб (млн задач), нет SPOF |
| Статус/история, отмена | Точность/низкая latency срабатывания |
| Приоритеты | Durability определений задач |

```
Estimation (пример): млн запланированных задач, всплески в "круглые" времена.
  Срабатываний/с: распределение по времени неравномерно (полночь, начало часа — пики).
  Воркеры: число = пиковая нагрузка срабатываний / производительность воркера.
  Storage: определения задач + история выполнений (durable).
```

> Estimation вскрывает специфику: нагрузка **неравномерна во времени** — множество cron-задач назначено на «круглые» моменты (полночь, начало часа), создавая **пики срабатываний** (thundering herd). Главное NFR — **надёжность**: задача обязана выполниться (at-least-once), её определение durable. Точность срабатывания (задача стартует близко к назначенному времени) и отсутствие SPOF — ключевые. Это не про объём данных, а про **гарантию выполнения в срок в распределённой среде** (§4–7).

## 2. Компоненты

```
   ┌───────────┐   due-задачи   ┌────────────┐   очередь   ┌──────────┐
   │ Job store │◄──────────────►│ Scheduler  │────────────►│  Queue   │──► Workers
   │ (durable) │  next_run_time │ (что/когда)│  (модуль 15)└──────────┘    (исполняют)
   └───────────┘                └────────────┘                  ▲              │
        ▲ определения/состояние/история                          └── статус ◄───┘
```

| Компонент | Роль |
|-----------|------|
| **Job store** (БД) | durable определения задач, расписание, состояние, история |
| **Scheduler** | решает ЧТО и КОГДА запустить (находит «созревшие» задачи) |
| **Queue** (модуль 15) | созревшие задачи → воркерам (развязка) |
| **Workers** | выполняют задачи, тянут из очереди, масштабируются горизонтально |

> Четыре компонента. **Job store** (durable БД) хранит определения, расписание, состояние и историю — терять задачи нельзя. **Scheduler** решает **что и когда** запускать, находя «созревшие» задачи. **Queue** (модуль 15) переносит созревшие задачи воркерам — это и есть **развязка планирования и выполнения** (§4). **Workers** исполняют задачи, тянут из очереди и масштабируются независимо. Ключевой принцип: scheduler не исполняет задачи сам, он только ставит их в очередь.

## 3. Механизм планирования (поиск due-задач)

```
Как эффективно находить «созревшие» задачи (next_run_time <= now) среди миллионов?

  Вариант A — индекс по next_run_time:
    SELECT * FROM jobs WHERE next_run_time <= now ORDER BY next_run_time  (индекс на колонке)
    scheduler периодически опрашивает (polling) и забирает due-задачи.

  Вариант B — timing wheel / бакеты по времени:
    задачи раскладываются в бакеты по времени срабатывания → берём текущий бакет.
```

| Подход | Идея |
|--------|------|
| Индекс по next_run_time + polling | БД-индекс на «когда запускать»; периодический опрос due |
| Timing wheel / бакеты | задачи в бакетах по времени; обрабатываем текущий бакет |

> Главная техническая задача scheduler'а — **эффективно находить созревшие задачи** среди миллионов. Наивный полный скан недопустим. Решения: **индекс по `next_run_time`** (БД-запрос «что пора запускать» по индексу + периодический polling) или **timing wheel** (задачи раскладываются в бакеты по времени, берём текущий бакет — амортизированно дёшево). Это структура, превращающая «найти due среди млн» в дешёвую операцию (концептуально как индекс/sorted-структура из предыдущих модулей).

## 4. Развязка планирования и выполнения

```
Scheduler НЕ исполняет задачу сам:
  scheduler находит due-задачу → кладёт в ОЧЕРЕДЬ (модуль 15) → возвращается к планированию.
  Workers тянут из очереди → исполняют → обновляют статус.

Зачем: scheduler остаётся лёгким и быстрым; исполнение масштабируется воркерами независимо.
```

| Сторона | Ответственность |
|---------|-----------------|
| Scheduler | найти due → поставить в очередь (не блокироваться на исполнении) |
| Workers | взять из очереди → исполнить → обновить статус/историю |

> **Развязка планирования и выполнения** — фундамент дизайна. Scheduler только **находит созревшие задачи и ставит их в очередь** (модуль 15), не исполняя сам — иначе долгая задача заблокировала бы планирование других. **Workers** независимо тянут из очереди, исполняют и обновляют статус. Это позволяет масштабировать исполнение (добавить воркеров) отдельно от планирования и держит scheduler лёгким. Очередь — несущая конструкция между «когда» и «как».

## 5. Надёжность: at-least-once + идемпотентность

```
Задача ОБЯЗАНА выполниться → at-least-once execution (лучше дважды, чем ни разу).
  Следствие: задача может выполниться ПОВТОРНО (retry, переотдача после сбоя воркера).
  → задачи должны быть ИДЕМПОТЕНТНЫМИ (модуль 23): повтор не ломает результат.

  retry с backoff при сбое; DLQ для постоянно падающих (ядовитые задачи).
  exactly-once срабатывание НЕдостижимо → at-least-once + идемпотентность.
```

| Механизм | Роль |
|----------|------|
| At-least-once execution | гарантия «задача выполнится» (лучше дважды, чем потерять) |
| Идемпотентность задач (модуль 23) | повтор безопасен (обязателен при at-least-once) |
| Retry + backoff | повтор при сбое без retry storm |
| DLQ | постоянно падающие задачи в отдельную очередь |

> Раз задача **обязана выполниться**, выбирают **at-least-once** (лучше выполнить дважды, чем потерять). Цена — задача может сработать **повторно** (retry, переотдача после смерти воркера). Поэтому задачи делают **идемпотентными** (модуль 23): повторное выполнение не ломает результат. Сбои обрабатывают **retry с backoff**, а постоянно падающие («ядовитые») задачи отправляют в **DLQ**. Как и в платежах/уведомлениях, **exactly-once недостижим** → at-least-once + идемпотентность.

## 6. Отказ воркера и отказ scheduler'а

**Отказ воркера** (lease / visibility timeout):

```
Воркер взял задачу и УМЕР, не завершив → задача не должна потеряться.
  lease/visibility timeout: задача «арендована» на время T; воркер шлёт heartbeat.
  нет heartbeat / истёк lease → задача ПЕРЕОТДаётся другому воркеру.
```

**Отказ scheduler'а** (SPOF → leader election):

```
Один scheduler = SPOF (упал → ничего не планируется).
Два scheduler'а наивно = ДВОЙНОЙ запуск задачи.
  Решение: LEADER ELECTION (модуль 22) — только лидер триггерит; при падении выбирается новый.
  Альтернатива: ПАРТИЦИОНИРОВАНИЕ задач между scheduler'ами (каждый владеет частью).
```

| Отказ | Защита |
|-------|--------|
| Воркер умер | lease/visibility timeout + heartbeat → переотдача задачи |
| Scheduler — SPOF | leader election (модуль 22): один лидер триггерит, failover |
| Двойной запуск | leader election / партиция задач |

> Два вида отказов. **Воркер умер** не доделав задачу → **lease/visibility timeout**: задача «арендована» на время, воркер шлёт **heartbeat**; нет heartbeat → задача **переотдаётся** другому (потому at-least-once и нужна идемпотентность). **Scheduler — SPOF**: один упал — ничего не планируется; два наивно — **двойной запуск**. Решение — **leader election** (модуль 22): только лидер триггерит, при падении выбирается новый. Альтернатива — **партиционирование** задач между scheduler'ами (каждый владеет своей частью).

## 7. Recurring, catch-up, high-level и узкие места

**Recurring** (cron): после выполнения вычислить следующий `next_run` и обновить в job store.

**Missed jobs / catch-up**: scheduler лежал → пропущены срабатывания. Политика: выполнить пропущенные (catch-up) или пропустить (только последнее) — зависит от семантики задачи.

**Приоритеты**: priority queues (срочные раньше).

```
   [API: schedule/cancel] ──► Job store (БД) ──► Scheduler (LEADER, найти due)
                                                      │ поставить в очередь
                                                 Queue (priority, модуль 15)
                                                      │
                                                 Workers (lease+heartbeat, идемпотентны) ──► статус → Job store
```

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Scheduler SPOF** | leader election / партиция задач | 22 |
| **Поиск due-задач** на масштабе | индекс по next_run / timing wheel | — |
| **Hot time** (полночь, thundering herd) | джиттер времени, размазать срабатывания, масштаб воркеров | 16 |
| Масштаб воркеров | горизонтальное масштабирование, очередь как буфер | 15 |
| Exactly-once | at-least-once + идемпотентность (недостижим строгий) | 23 |

> **Recurring**: после выполнения cron-задачи вычисляется следующий `next_run`. **Missed/catch-up**: если scheduler лежал, пропущенные срабатывания либо догоняются, либо пропускаются (по семантике). **Приоритеты** — priority queues. Узкие места: **scheduler SPOF** (leader election/партиция — модуль 22), **поиск due** (индекс/timing wheel), **hot time** (тысячи задач на полночь → джиттер, размазать срабатывания, масштаб воркеров), масштаб воркеров (очередь как буфер). Строгий exactly-once недостижим → at-least-once + идемпотентность.

## ⚠️ Подводные камни

- **Scheduler исполняет задачи сам.** Долгая задача блокирует планирование; развязать через очередь (модуль 15).
- **Один scheduler без leader election.** SPOF; упал — ничего не планируется. Leader election/партиция (модуль 22).
- **Два scheduler'а без координации.** Двойной запуск задач; нужен leader election.
- **Задачи не идемпотентны при at-least-once.** Повтор/переотдача ломает результат; идемпотентность обязательна (модуль 23).
- **Нет lease/visibility timeout.** Воркер умер → задача потеряна; lease + heartbeat → переотдача.
- **Полный скан job store для поиска due.** Не масштабируется; индекс по next_run / timing wheel.
- **Игнор hot time (полночь).** Thundering herd кладёт систему; джиттер/размазывание + масштаб воркеров.
- **Нет DLQ для ядовитых задач.** Постоянно падающая задача забивает retry; DLQ.
- **Обещать строгий exactly-once.** Недостижим; честно — at-least-once + идемпотентность.
