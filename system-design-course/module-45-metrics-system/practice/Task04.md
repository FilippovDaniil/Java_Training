# Задача 04 — Модуль 45: Проблема high cardinality

## ЗАДАНИЕ
Объясните проблему high cardinality и сформулируйте правило выбора labels.

## ВХОДНЫЕ ДАННЫЕ
Инженер хочет добавить к метрике http_requests label user_id (миллионы пользователей), чтобы видеть запросы по каждому юзеру.

## ТРЕБОВАНИЯ
- Определите cardinality = число уникальных серий (комбинаций metric + labels).
- Покажите, как unbounded label (user_id) рождает взрыв серий.
- Опишите последствия (раздувание индекса/памяти/storage, медленные запросы, падение TSDB).
- Сформулируйте правило: labels только для ограниченных множеств (service, region), не для unbounded (user_id, request_id).

## ФОРМАТ РЕЗУЛЬТАТА
Cardinality (что это) → взрыв от unbounded label → последствия → правило выбора labels.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Cardinality = число уникальных серий (metric + labels).
- [ ] Показано: user_id (млн значений) → млн серий на метрику → взрыв.
- [ ] Последствия: раздувание индекса/памяти/storage, медленные запросы, риск падения TSDB.
- [ ] Правило: labels для ограниченных множеств (service/region/status), НЕ unbounded (user_id/request_id/URL).

## ПОДСКАЗКА
Cardinality = число серий (metric+labels). Unbounded label (user_id) → млн серий → взрыв (память/индекс/storage, медленные запросы, падение). Правило: labels только ограниченные множества.
