# Задача 06 — Модуль 45: TSDB и сжатие time-series

## ЗАДАНИЕ
Обоснуйте специализированную TSDB и спроектируйте сжатие time-series.

## ВХОДНЫЕ ДАННЫЕ
Миллионы точек/с нужно хранить компактно и эффективно агрегировать по времени. Обычная реляционная БД не справляется.

## ТРЕБОВАНИЯ
- Объясните, почему нужна специализированная TSDB, а не обычный SQL (append-heavy, time-ordered, агрегация по диапазонам).
- Опишите колоночное хранение серий для эффективной агрегации.
- Спроектируйте сжатие: delta-of-delta для timestamps, XOR для values (Gorilla).
- Покажите, почему эти приёмы дают кратное снижение storage.

## ФОРМАТ РЕЗУЛЬТАТА
Почему TSDB, не SQL → колоночное хранение → сжатие (delta-of-delta + XOR) → эффект на storage.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Обосновано: TSDB под append-heavy, time-ordered, агрегацию по диапазонам; обычный SQL неэффективен.
- [ ] Колоночное хранение серий для эффективной агрегации по времени.
- [ ] Сжатие: delta-of-delta (timestamps — равные интервалы → почти нули) + XOR (values — близкие значения → мало бит).
- [ ] Показано: кратное снижение storage против наивного хранения (timestamp, value) на точку.

## ПОДСКАЗКА
TSDB (не SQL): append-heavy, time-ordered, агрегация по диапазонам, колоночность. Сжатие Gorilla: delta-of-delta (timestamps) + XOR (values) → кратно меньше storage.
