## Ключ (эталонные ответы)

1. Приём метрик от множества источников/агентов, хранение time-series, запрос/агрегация метрик, алертинг по условиям, дашборды.
2. Write-heavy поток измерений: десятки тысяч хостов × тысячи метрик × частый сбор = миллионы точек/с. Это непрерывный поток, а не запросы пользователей.
3. Модуль 28 — про концепции observability (SLI/SLO, RED/USE, error budget). Этот модуль — про дизайн самой платформы, которая собирает, хранит и запрашивает метрики.
4. Хосты × метрик на хост × частота сбора: например, 10^5 хостов × 10^3 метрик / 10 с ≈ 10^6 точек/с.
5. Имя метрики + набор labels/тегов + timestamp + значение (например, http_requests_total{service,method,status} ts value).
6. Уникальная комбинация имени метрики и набора labels; запросы выбирают серии по labels и агрегируют по времени.
7. Число уникальных серий — комбинаций метрики и значений labels.
8. От добавления label с неограниченным (unbounded) множеством значений (user_id, request_id, URL с параметрами): каждая комбинация — новая серия, их становятся миллионы.
9. Раздувание индекса, памяти и storage, замедление запросов и риск падения TSDB.
10. Labels — только для ограниченных множеств (service, region, status), но не для unbounded-идентификаторов (user_id, request_id, полный URL).
11. Push — агенты сами шлют метрики в коллектор; pull — сервер периодически скрейпит таргеты по HTTP (модель Prometheus).
12. Pull: + явный список таргетов (видно, кто упал), контроль частоты сбора; − нужен сетевой доступ к таргетам и service discovery.
13. Чтобы развязать приём и запись, сгладить пики потока метрик и не положить TSDB при всплеске (модуль 15).
14. Time-series — append-heavy, упорядоченный по времени поток с агрегацией по диапазонам; обычная реляционная БД для миллионов точек/с неэффективна. TSDB оптимизирована под это (колоночность, сжатие).
15. Через delta-of-delta: интервалы сбора почти равномерны, поэтому вторые разности близки к нулю и кодируются минимумом бит.
16. Через XOR соседних значений: близкие значения дают мало значащих бит, что сильно сжимает (алгоритм Gorilla).
17. Агрегация старых данных в более грубое разрешение (raw → 1m → 1h → 1d): хранить агрегаты (min/max/avg по окну) вместо всех сырых точек.
18. Уровни хранения по возрасту данных: свежее — высокое разрешение (raw), старое — downsampled (1m/1h/1d) с разным сроком хранения; снижает storage, сохраняя тренды.
19. Запросы по временным диапазонам с агрегацией по сериям (PromQL-подобный язык: rate(), sum by (service) и т.п.).
20. Правила вычисляются по метрикам (например, error_rate > 5% за 5 минут); при срабатывании создаётся алерт.
21. Дедуплицирует (один инцидент — один алерт, а не шквал), группирует связанные алерты, поддерживает silencing и маршрутизирует по каналам (связь с модулем 36); борется с alert fatigue.
22. Шардированием по серии/метрике (consistent hashing — модуль 14) и федерацией (агрегация из нескольких TSDB); реплики для масштаба чтений и HA.
23. Write throughput (→ буфер + шарды), high cardinality (→ ограничивать labels), запросы по длинным диапазонам (→ downsampling), storage cost (→ сжатие + retention), hot metrics (→ шарды/реплики).
24. Потому что она незаметно (через невинно добавленный label) рождает миллионы серий, раздувает память/storage, замедляет запросы и роняет TSDB — и это самая частая причина деградации систем метрик.
25. Time-series как отдельный класс данных (метрика + labels + ts + value); high cardinality — главная ловушка (ограничивать labels); push vs pull ingestion + буфер; специализированная TSDB со сжатием time-series (delta-of-delta + XOR); downsampling + retention tiers для storage; запросы по диапазонам + алертинг с alert manager (дедуп/группировка); шардирование по серии (consistent hashing) для масштаба.
