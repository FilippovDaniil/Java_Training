# Модуль 45 — Система метрик и мониторинга (metrics-system) (референсная архитектура)

## Зачем этот модуль

Система метрик (Prometheus/Datadog/InfluxDB) — это **дизайн самой платформы мониторинга**: принять миллионы метрик в секунду, сжать и хранить как time-series, быстро запросить по диапазонам и поднять алерты. Если модуль 28 объяснял **зачем** observability (SLI/SLO/RED/USE), то здесь — **как устроена** система, которая всё это собирает. Главные вызовы уникальны: **взрыв cardinality** (комбинации labels), **специализированная TSDB** со сжатием time-series, **downsampling/retention** и **push vs pull** ingestion. Модуль синтезирует очереди (15), OLAP/колоночность (18), уведомления (36), consistent hashing (14).

Зачем это senior'у. Metrics-system учит, что **time-series — отдельный класс данных** (не реляционные строки): write-heavy append-only поток, который сжимается по-особому (delta-of-delta, XOR) и хранится в **TSDB**, а не в обычной БД. Senior знает про **high cardinality** как главную ловушку, разницу **push/pull**, **downsampling** старых данных ради retention и устройство **алертинга** поверх метрик.

Он показывает полный HLD системы метрик: requirements/estimation, time-series модель, cardinality, ingestion, TSDB/сжатие, downsampling, запросы/алертинг, масштаб.

## 1. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Приём метрик от множества источников/агентов | **Огромный write-throughput** (млн метрик/с) |
| Хранение time-series | Масштабируемое хранилище |
| Запрос/агрегация метрик | Быстрые запросы по временным диапазонам |
| Алертинг по условиям | Retention/downsampling |
| Дашборды | Высокая доступность |

```
Estimation (пример): хосты × метрик-на-хост × частота сбора → write QPS.
  10^5 хостов × 10^3 метрик × (1 раз / 10 c) ≈ 10^7 точек / 10 c = 10^6 точек/с → write-heavy.
  Storage: точки/с × срок × размер точки → терабайты (но сжатие time-series кратно снижает).
  Чтения: запросы дашбордов/алертов по диапазонам — агрегации по многим сериям.
```

> Estimation сразу показывает: система **write-heavy** — десятки тысяч хостов × тысячи метрик × частый сбор = **миллионы точек/с**. Это не RPS-запросов, а непрерывный **поток измерений**. Storage кажется огромным, но **сжатие time-series** (§5) снижает его кратно. Чтения — агрегации по временным диапазонам и многим сериям (дашборды, алерты). Главная ось — **write-throughput и эффективное хранение time-series**, плюс скрытая ловушка — cardinality (§4).

## 2. Модель данных time-series

```
Точка time-series:
  metric_name{label1=v1, label2=v2, ...} timestamp value
  пример: http_requests_total{service="api", method="GET", status="200"} 1719_... 42

Серия (series) = уникальная комбинация (metric_name + набор labels).
  Запрос: выбрать серии по labels + диапазон времени → агрегировать.
```

| Элемент | Роль |
|---------|------|
| metric_name | что измеряем (http_requests_total) |
| labels/теги | измерения/разрезы (service, method, status) |
| timestamp + value | момент + значение |
| **series** | уникальная комбинация metric + labels |

> Модель данных — **time-series**: имя метрики + набор **labels** (теги-разрезы) + timestamp + значение. Уникальная комбинация (метрика + labels) — это **серия**. Запросы выбирают серии по labels и временному диапазону, затем агрегируют. Labels дают мощную многомерность (разбить по сервису/методу/статусу), но именно они — источник главной проблемы: **cardinality** (§4). Эта модель принципиально отличается от реляционных строк и требует специального хранилища (§5).

## 3. Ingestion: push vs pull

```
PUSH: агенты сами шлют метрики в коллектор (Datadog, StatsD).
  + работает за NAT/firewall; − коллектор не знает, кто «пропал» (нет таргет-листа).

PULL: сервер периодически СКРЕЙПИТ таргеты по HTTP (Prometheus).
  + явный список таргетов (видно, кто down); − нужен доступ к таргетам, service discovery.

Коллектор/ingestion → буфер/очередь (модуль 15) → TSDB.
```

| Модель | Плюс | Минус |
|--------|------|-------|
| **Push** (агент → коллектор) | за NAT/firewall, ephemeral-задачи | нет явного списка → «кто пропал?» |
| **Pull** (сервер скрейпит) | явные таргеты (видно down), контроль частоты | нужен доступ + service discovery |

> **Ingestion** бывает двух моделей. **Push**: агенты сами отправляют метрики в коллектор (StatsD/Datadog) — работает за NAT и для эфемерных задач, но коллектор не знает, кто «должен был прислать, но молчит». **Pull**: сервер периодически **скрейпит** таргеты по HTTP (модель Prometheus) — есть явный список таргетов (видно, кто упал), контроль частоты, но нужен сетевой доступ и service discovery. Поток метрик буферизуют через очередь (модуль 15) перед записью в TSDB, сглаживая пики.

## 4. Проблема high cardinality

```
Cardinality = число уникальных СЕРИЙ (комбинаций metric + labels).
  http_requests{service, method, status} — умеренно.
  ДОБАВИЛИ label user_id (млн юзеров) → млн серий на метрику → ВЗРЫВ cardinality.

Последствия: раздувание индекса/памяти/storage, замедление запросов, падение TSDB.
Правило: labels — для ОГРАНИЧЕННЫХ множеств (service, region), НЕ для unbounded (user_id, request_id).
```

| Причина | Следствие |
|---------|-----------|
| Unbounded label (user_id, URL с параметрами) | взрыв числа серий |
| Высокая cardinality | раздувание индекса/памяти, медленные запросы, риск падения |
| Контроль | labels только для ограниченных множеств |

> **High cardinality — главная ловушка** систем метрик (упоминалась в модуле 28). **Cardinality** — число уникальных серий (комбинаций метрики и labels). Добавление label с **неограниченным** множеством значений (user_id, request_id, полный URL) рождает **миллионы серий** на одну метрику → раздувание индекса/памяти/storage, замедление запросов, падение TSDB. Правило дизайна: labels — только для **ограниченных** множеств (service, region, status), но **не** для unbounded-идентификаторов. Это ключевое ограничение, которое senior называет первым.

## 5. TSDB и сжатие time-series

```
Зачем СПЕЦИАЛИЗИРОВАННАЯ TSDB (не обычный SQL):
  - данные append-heavy, упорядочены по времени;
  - колоночное хранение серий → эффективная агрегация по времени;
  - СЖАТИЕ time-series (Gorilla):
      timestamps: delta-of-delta (равномерные интервалы → почти нули);
      values:     XOR соседних значений (близкие значения → мало бит).
  → кратное снижение storage против наивного (timestamp,value) на точку.
```

| Приём | Зачем |
|-------|-------|
| Append-heavy, time-ordered | поток точек по времени, не случайные записи |
| Колоночное хранение | агрегация по времени читает нужное |
| delta-of-delta (timestamps) | равные интервалы → почти нули → сжатие |
| XOR (values) | близкие соседние значения → мало бит |

> **TSDB** (Prometheus TSDB, InfluxDB, Cortex/M3) специализирована, потому что time-series — особый класс: **append-heavy**, упорядоченный по времени, с агрегацией по диапазонам. Обычная реляционная БД для миллионов точек/с неэффективна. Ключ — **сжатие** (алгоритм Gorilla): **timestamps** жмут через **delta-of-delta** (равномерные интервалы сбора → дельты почти нулевые), **values** — через **XOR** соседних (близкие значения дают мало значащих бит). Это снижает хранение time-series в разы против наивного хранения каждой пары (timestamp, value).

## 6. Downsampling, retention, запросы, алертинг

**Downsampling/retention**:

```
Свежие данные — высокое разрешение (raw, секунды); старые — агрегаты (1m → 1h → 1d).
  retention tiers: raw 15 дней → 1m 90 дней → 1h год.
  старые точки downsampled (min/max/avg по окну) → резко меньше storage, тренды сохранены.
```

**Запросы**: по временным диапазонам + агрегация по сериям (PromQL-подобный язык): `rate()`, `sum by (service)`.

**Алертинг**:

```
Правила вычисляются по метрикам (например, error_rate > 5% за 5 мин) → алерт.
  Alert manager: ДЕДУП (один инцидент — один алерт), ГРУППИРОВКА, SILENCING, маршрутизация (модуль 36).
```

| Аспект | Решение |
|--------|---------|
| Retention | downsampling старых данных в rollups (1m/1h/1d) |
| Запросы | диапазон + агрегация по сериям (PromQL-подобно) |
| Алертинг | правила → алерт; alert manager (дедуп/группировка/silencing) |

> **Downsampling** удешевляет retention: свежие данные хранят в высоком разрешении, старые — в **rollups** (1m → 1h → 1d агрегаты), что резко снижает storage при сохранении трендов. **Запросы** — по временным диапазонам с агрегацией по сериям (PromQL-подобный язык: `rate`, `sum by`). **Алертинг**: правила вычисляются по метрикам (error_rate > 5% за 5 мин) → алерт; **alert manager** делает **дедуп** (один инцидент = один алерт, не шквал), **группировку**, **silencing** и маршрутизацию (связь с модулем 36).

## 7. Масштаб, high-level и узкие места

```
   [Агенты/таргеты] ──push/pull──► Ingestion ──► буфер (модуль 15) ──► TSDB (сжатие, шарды)
                                                                          │
                                              Query engine ◄──────────────┤
                                                  │                       │
                                              Дашборды            Alerting (правила) ──► Alert manager (дедуп) ──► каналы
```

**Масштаб**: шардирование по серии/метрике (consistent hashing — модуль 14); федерация (агрегация из нескольких TSDB).

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Write throughput** (млн точек/с) | буфер/очередь, шардирование по серии | 15, 14 |
| **High cardinality** | ограничивать labels (не unbounded), cardinality-лимиты | 28 |
| **Запросы по длинным диапазонам** | downsampling/rollups, предагрегация | — |
| **Storage cost** | сжатие time-series + retention tiers | 18 |
| **Hot metrics** | шардирование, реплики для чтений | 14, 12 |

> High-level: **агенты/таргеты** (push/pull) → **ingestion** → буфер (очередь — модуль 15) → **TSDB** (сжатие, шардирование). **Query engine** обслуживает дашборды и **alerting** (правила → alert manager с дедупом). **Масштаб** — шардирование по серии (consistent hashing — модуль 14) + федерация. Узкие места: write throughput (буфер + шарды), **high cardinality** (ограничивать labels — главное!), запросы по длинным диапазонам (downsampling), storage cost (сжатие + retention), hot metrics (шарды/реплики). Этим завершается Часть 8 курса.

## ⚠️ Подводные камни

- **Unbounded labels (user_id, request_id, URL с параметрами).** Взрыв cardinality → падение TSDB; labels только для ограниченных множеств.
- **Хранить метрики в обычной реляционной БД.** Миллионы точек/с неэффективны; нужна TSDB с time-series сжатием.
- **Без сжатия time-series.** Наивное (timestamp, value) на точку раздувает storage; delta-of-delta + XOR (Gorilla).
- **Без downsampling/retention.** Хранить raw вечно дорого; rollups (1m/1h/1d) + retention tiers.
- **Pull без service discovery.** Сервер не знает таргетов; нужен service discovery для скрейпа.
- **Push без учёта пропавших.** Не видно, кто «замолчал»; компенсировать heartbeat/up-метрикой.
- **Алерты без дедупа/группировки.** Шквал алертов (alert fatigue — модуль 28); alert manager (дедуп/группировка/silencing).
- **Синхронная запись без буфера.** Пик метрик кладёт TSDB; буфер/очередь (модуль 15).
- **Игнор шардирования при росте.** Один узел не вмещает поток; шардирование по серии (модуль 14).
