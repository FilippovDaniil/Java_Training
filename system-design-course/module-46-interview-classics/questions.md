## Ключ (эталонные ответы)

1. ≈20 канонических задач покрывают большинство интервью; распознав архетип задачи, вы сразу знаете, какие проверенные блоки применять, вместо проектирования «с нуля».
2. URL-shortener, KV-store/cache, newsfeed/Twitter, chat, notifications, video/YouTube, file-storage/Dropbox, ride-sharing/Uber, payment, ad-aggregation, web-crawler, leaderboard, job-scheduler, metrics, rate-limiter, autocomplete, nearby/proximity, ticket-booking.
3. Доминирующая ось задачи (что в ней главное), определяющая выбор архитектуры и блоков.
4. Read-heavy, write-heavy/ingestion, real-time/messaging, geo/proximity, storage/media-heavy, correctness-critical, search/index, coordination/scheduling.
5. Чтения ≫ записи (newsfeed, URL-shortener); блоки — кэш (9), read-реплики (12), CDN (10), fan-out push/pull (34).
6. Поток событий/записей доминирует (metrics, ad-aggregation, логи); блоки — очередь (15), async-агрегация по окнам (20/26), TSDB/OLAP (18/45); не синхронный инкремент.
7. Persistent-соединения и server push (chat, трекинг, notifications); блоки — WebSocket + connection servers (35), presence, push для offline.
8. Пространственные запросы «рядом» (Uber, Yelp); блоки — geospatial index geohash/quadtree/S2/H3 (17), шардирование по региону (13).
9. Деньги/инвентарь, недопустимы потеря/дублирование (payment, booking); блоки — идемпотентность (23), strong consistency/ACID (21), double-entry ledger (40), Saga/Outbox.
10. Большие байты/egress (YouTube, Dropbox); блоки — object storage + CDN (10), chunking/дедуп (38), async-транскодинг (37).
11. Полнотекстовый поиск/обход (web-crawler, autocomplete); блоки — inverted index (17), URL frontier + дедуп bloom/simhash (42).
12. Соответствие «архетип → набор готовых строительных блоков из Частей 3–7»: распознал архетип — взял блоки, а не изобретаешь заново.
13. Идемпотентность, кэширование, шардирование, async-очередь — «несущие конструкции», встречающиеся почти в каждой задаче.
14. Требования (FR/NFR/scope) → estimation → API → модель данных → high-level → deep dive → bottlenecks/масштаб/trade-offs → wrap-up.
15. Проговаривание рассуждений вслух; интервьюер оценивает ход мысли, а не только итоговую схему, поэтому молчание — потеря сигнала.
16. Чтобы не проектировать «вслепую»: scope и NFR (масштаб, согласованность, latency) определяют всю архитектуру; прыжок в решение без них — главная ошибка.
17. Уточнить требования → определить доминирующую ось → свести к архетипу(ам) → подобрать блоки по карте → зафиксировать явные допущения → проектировать по 8 шагам.
18. Реальные системы имеют несколько важных осей: Uber — geo + real-time + correctness (платёж); YouTube — storage-heavy + read-heavy раздача. Нужно учесть все доминирующие оси.
19. Latency numbers (RAM ~100 нс, SSD ~100 мкс, диск ~10 мс, сеть DC ~0.5 мс, межрегион ~50–150 мс), порядки (день ≈ 10^5 с), peak factor (2–10×), частое read/write 100:1.
20. 301 vs 302; push vs pull fan-out; sync vs async; SQL vs NoSQL; cache-aside vs write-through; strong vs eventual consistency.
21. Deep dive в компонент, масштабирование 10×/100× (13/24), обработка отказов (23/24/25), trade-offs, узкие места (26), согласованность (12/21), стоимость (7) — направления зависят от архетипа.
22. Заученные решения хрупки: незнакомая 21-я задача их ломает. Метод (архетипы + блоки + 8 шагов) применим к любой задаче, включая невиданную.
23. Прыжок в решение без требований, отсутствие estimation, нераспознанный архетип, отсутствие trade-offs, молчание (нет think-aloud), плохой тайм-менеджмент, заучивание вместо метода.
24. Явные допущения заменяют недостающие детали постановки, делают рассуждение прозрачным и позволяют двигаться, не застревая на неизвестных.
25. Уточнить требования → определить архетип(ы) → подобрать блоки по карте → вести по 8 шагам с think-aloud → обосновать числами → углубиться в 1–2 компонента → разобрать узкие места/trade-offs → подытожить; незнакомую задачу свести к архетипу. Метод устойчивее зубрёжки.
