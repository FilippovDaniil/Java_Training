# Модуль 46 — Классические задачи и методика ответа (interview-classics)

## Зачем этот модуль

Большинство system-design-интервью берут задачу из **узкого каталога ≈20 канонических проблем** — и почти все они разобраны в Части 8 этого курса. Но senior'а отличает не зубрёжка решений, а умение **распознать архетип** задачи и **переиспользовать блоки**: увидел «это read-heavy с fan-out» → знаешь, что нужны кэш, реплики и выбор push/pull. Этот модуль — мост между **методикой** (модуль 03 — фреймворк интервью) и **разобранными системами** (Часть 8): каталог классики, классификация по архетипам, карта «архетип → строительные блоки» и метод ответа на **незнакомую** задачу.

Зачем это senior'у. Заучить 20 решений — хрупко: дадут 21-ю, и вы поплыли. Метод устойчив: любую задачу сводишь к **архетипу(ам)**, подбираешь блоки из Частей 3–7, ведёшь по 8-шаговому фреймворку. Этот модуль учит **видеть структуру за условием**, а не вспоминать заученное.

Он даёт каталог, архетипы, карту паттернов, методику и шпаргалку — инструментарий для ответа на любую SD-задачу.

## 2. Архетипы задач

```
Классификация по ДОМИНИРУЮЩЕЙ оси (что главное в задаче):
```

| Архетип | Доминанта | Примеры |
|---------|-----------|---------|
| **Read-heavy** | чтения ≫ записи | newsfeed, URL-shortener |
| **Write-heavy / ingestion** | поток событий/записей | metrics, ad-aggregation, логи |
| **Real-time / messaging** | persistent-соединения, push | chat, трекинг, notifications |
| **Geo / proximity** | пространственные запросы | Uber, Yelp, nearby |
| **Storage / media-heavy** | байты, egress | YouTube, Dropbox |
| **Correctness-critical** | деньги/инвентарь, ACID | payment, booking, inventory |
| **Search / index** | inverted index, обход | web-crawler, autocomplete |
| **Coordination / scheduling** | триггеры, leader election | job-scheduler |

> **Архетип** — доминирующая ось задачи, определяющая дизайн. Это ключевой навык: не «как спроектировать Uber», а «это **geo + real-time**, значит нужны geospatial index и WebSocket». Многие задачи **смешивают** архетипы (Uber = geo + real-time + correctness для платежа; YouTube = storage-heavy + read-heavy раздача). Распознав архетип(ы), вы мгновенно знаете, **какие блоки** в игре (§3), и не изобретаете заново.

## 3. Карта паттернов: архетип → блоки

| Архетип | Строительные блоки (Части 3–7) |
|---------|-------------------------------|
| Read-heavy | кэш (9) + read-реплики (12) + CDN (10); fan-out push/pull (34) |
| Write-heavy / ingestion | очередь (15) + async-агрегация по окнам (20/26) + TSDB/OLAP (18/45) |
| Real-time / messaging | WebSocket + connection servers (35) + presence; push для offline |
| Geo / proximity | geospatial index geohash/quadtree/S2/H3 (17); шард по региону (13) |
| Storage / media-heavy | object storage + CDN (10) + chunking/дедуп (38); async-транскодинг (37) |
| Correctness-critical | идемпотентность (23) + ACID/strong consistency (21) + ledger (40); Saga/Outbox |
| Search / index | inverted index (17); URL frontier + дедуп bloom/simhash (42) |
| Coordination / scheduling | leader election (22) + lease/visibility timeout + at-least-once (44) |

> Это **карта переиспользования**: архетип → готовый набор блоков из Частей 3–7. Распознав «write-heavy ingestion», вы сразу тянете очередь + async-агрегацию + TSDB (а не синхронный инкремент). «Correctness-critical» → идемпотентность + ACID + ledger. Так дизайн собирается из **проверенных кубиков**, а не выдумывается. Сквозные паттерны (идемпотентность, кэш, шардирование, async-очередь) встречаются почти везде — это «несущие конструкции» SD.

## 4. Методика ответа (8-шаговый фреймворк)

```
1. Требования (FR/NFR, scope)          ← уточнить, НЕ прыгать в решение
2. Estimation (load/storage/cost)       ← обосновать масштаб числами
3. API (ключевые эндпоинты)
4. Модель данных (выбор хранилища)
5. High-level дизайн (компоненты, поток)
6. Deep dive (1–2 узких компонента детально)
7. Bottlenecks / масштаб / trade-offs
8. Wrap-up (резюме решений, открытые вопросы)
```

| Принцип | Суть |
|---------|------|
| Уточнять требования первым | не проектировать «вслепую»; зафиксировать scope/NFR |
| Думать вслух (think aloud) | интервьюер оценивает рассуждение, не только ответ |
| Вести разговор | сами двигаете по этапам, не ждёте вопросов |
| Тайм-менеджмент | не утонуть в одном этапе; оставить время на deep dive |

> **Метод важнее решения** (модуль 03). 8 шагов — каркас для **любой** задачи: уточнить требования (никогда не прыгать сразу в решение), посчитать масштаб, дать API и модель данных, нарисовать high-level, углубиться в 1–2 критичных компонента, разобрать узкие места и trade-offs, подытожить. Параллельно — **think aloud** (озвучивать рассуждение), **вести** разговор самому и **держать тайминг** (§модуль 47). Архетип (§2) подсказывает, куда углубляться на шаге 6.

## 5. Распознавание незнакомой задачи

```
Дали НЕЗНАКОМУЮ задачу (нет в каталоге)?
  1. Уточни требования → пойми доминирующую ось (FR/NFR).
  2. Сведи к АРХЕТИПУ(ам) — на что похоже? (read-heavy? geo? correctness?)
  3. Подбери БЛОКИ по карте (§3) для этих архетипов.
  4. Зафиксируй ДОПУЩЕНИЯ явно и проектируй по 8 шагам.
```

> Главная страховка от «задачи, которой не видел»: **сведение к архетипу**. Незнакомая постановка («спроектируйте систему бронирования переговорок», «дизайн системы антифрода») всё равно ложится на один-два архетипа (booking = correctness-critical; антифрод = write-heavy ingestion + real-time scoring). Уточнив требования и определив архетип, вы **подбираете блоки по карте** и проектируете методом — без заученного решения. Явные **допущения** заменяют недостающие детали.

## 6. Шпаргалка чисел и типовых решений

| Категория | Что помнить |
|-----------|-------------|
| Latency numbers | RAM ~100 нс, SSD ~100 мкс, диск ~10 мс, сеть DC ~0.5 мс, межрегион ~50–150 мс |
| Порядки | день ≈ 86400 ≈ 10^5 с; K/M/B; byte = 8 бит |
| Peak factor | пик ≈ 2–10× среднего |
| Read/write | часто 100:1 (read-heavy) |
| Типовые решения | 301 vs 302; push vs pull fan-out; sync vs async; SQL vs NoSQL; cache-aside vs write-through |

> Senior держит в голове **опорные числа** (latency numbers, порядки, peak factor) для быстрой estimation и **типовые развилки** с готовым ответом «когда что»: 301 (кэшируется, нет аналитики) vs 302; push fan-out (read-heavy, не celebrity) vs pull; sync vs async (нужна ли немедленная согласованность); SQL (транзакции/связи) vs NoSQL (масштаб/гибкость); cache-aside vs write-through. Эти развилки повторяются из задачи в задачу — знание их экономит время на интервью.

## ⚠️ Подводные камни

- **Прыжок в решение без требований.** Главная ошибка; сначала уточнить FR/NFR/scope (модуль 03).
- **Не распознал архетип.** Проектируешь «с нуля» вместо переиспользования блоков; сначала классифицируй.
- **Изобретение паттернов заново.** Архетип → готовые блоки (§3); не выдумывай то, что уже есть в Частях 3–7.
- **Нет чисел.** Дизайн без estimation неубедителен; считать масштаб (модули 5–7).
- **Заучивание решений вместо метода.** 21-я задача сломает зубрёжку; держись метода + архетипов.
- **Молчание.** Интервьюер оценивает рассуждение; думать вслух.
- **Утонуть в одном этапе.** Тайм-менеджмент; оставить время на deep dive и wrap-up (модуль 47).
- **Игнор смешанных архетипов.** Uber — это и geo, и real-time, и correctness; учесть все оси.
