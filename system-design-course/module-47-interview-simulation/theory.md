# Модуль 47 — Симуляция интервью + рубрика по грейдам (interview-simulation)

## Зачем этот модуль

Знать метод (модуль 46) — половина дела; вторая половина — **как это оценивается**. System-design-интервью не имеет «правильного ответа»: интервьюер собирает **сигналы** и сопоставляет их с **рубрикой по грейду** (junior / mid / senior / staff). Один и тот же дизайн на 60 минут может быть «mid» или «senior» — разница в **глубине, квантификации, trade-offs, проактивности и коммуникации**. Этот модуль показывает полную **симуляцию** интервью с таймингом, **рубрику по уровням** и метод **самооценки** для mock-практики.

Зачем это senior'у. Понимание рубрики превращает подготовку из «решу побольше задач» в целенаправленную: вы знаете, **какие сигналы** поднимают ответ с mid на senior (квантификация, failure modes, trade-offs вслух, проактивный deep dive) и как **не провалить тайминг** (не утонуть в требованиях, оставить время на глубину). Самооценка по рубрике делает mock-практику измеримой.

Он завершает подготовку к интервью: симуляция, тайм-менеджмент по фазам, рубрика по грейдам, сигналы, восстановление после застревания, mock-практика.

---

## 1. Как оценивают: сбор сигналов

```
Интервью — НЕ поиск «правильного ответа», а СБОР СИГНАЛОВ за 45–60 мин:
  - структурное мышление (есть ли метод/каркас?);
  - requirements first (уточняет ли до решения?);
  - квантификация (обосновывает ли числами?);
  - trade-off-рассуждения (видит ли альтернативы и компромиссы?);
  - глубина (может ли уйти вглубь компонента?);
  - коммуникация (понятно ли излагает, реагирует ли на подсказки?).
Сигналы → сопоставляются с РУБРИКОЙ по грейду (§4).
```

> System-design-интервью **открытое**: нет единственно верного решения. Интервьюер за 45–60 минут собирает **сигналы** о том, как вы мыслите: есть ли структура, уточняете ли требования, обосновываете ли числами, видите ли trade-offs, можете ли углубиться, понятно ли излагаете. Затем сигналы сопоставляются с **рубрикой по грейду** (§4). Поэтому важно не «выдать идеальную схему», а **продемонстрировать сигналы** — через think-aloud, метод и проактивность.

## 2. Полная симуляция (worked example)

```
«Спроектируйте Twitter (newsfeed)» — сжатый ход ответа senior-уровня:

[0–5 мин] Требования: «Уточню scope — лента подписок, публикация, ранжирование?
           NFR: read-heavy, низкая latency ленты, eventual consistency ок?»
[5–10]    Estimation: «100M DAU, ~2 поста/юзер/день → ~2000 write/s; чтений ленты на порядок больше.»
[10–20]   API + данные: «POST /tweets, GET /feed; посты, граф подписок, feed cache (Redis).»
[20–30]   High-level: «client→LB→feed service→feed cache; публикация→fan-out service через очередь.»
[30–45]   Deep dive: «Fan-out: push vs pull. Push для обычных, pull для celebrity (hybrid) —
           проговариваю trade-off, celebrity fan-out дорог.»
[45–55]   Bottlenecks: «celebrity, fan-out lag, hot users; кэш + реплики; деградация.»
[55–60]   Wrap-up: «резюме решений, открытые вопросы: ранжирование ML, мультирегион.»
```

> **Симуляция** показывает метод в действии: senior **уточняет требования** (не прыгает в решение), **квантифицирует** (DAU → RPS), даёт API и данные, рисует high-level, **углубляется** в fan-out с **trade-off вслух** (push vs pull, celebrity problem → hybrid), разбирает **узкие места** и подводит итог. Обратите внимание на **тайминг** (§3) — на deep dive остаётся время, потому что требования не растянуты. Это и есть «senior-ответ»: структура + числа + trade-offs + глубина + коммуникация.

---

## 3. Тайм-менеджмент по фазам (45–60 мин)

| Фаза | Время | Риск |
|------|-------|------|
| Требования | ~5 мин | застрять, уточняя бесконечно |
| Estimation | ~5 мин | пропустить (нет чисел) или утонуть |
| API + модель данных | ~10 мин | излишняя детализация схемы |
| High-level | ~10 мин | — |
| Deep dive (1–2 компонента) | ~10–15 мин | **не дойти** (съели время раньше) |
| Bottlenecks + wrap-up | ~5–10 мин | не остаётся времени |

> **Тайм-менеджмент решает**. Типичный провал — **утонуть в требованиях/высокоуровневой схеме** и не дойти до **deep dive**, где собирается главный senior-сигнал (глубина). Грубый бюджет на 60 мин: требования ~5, estimation ~5, API+данные ~10, high-level ~10, deep dive ~10–15, bottlenecks+wrap-up ~5–10. Следите за временем сами и **двигайте** разговор. Лучше пройти все фазы поверхностно и углубить одну, чем застрять на ранней.

---

## 4. Рубрика по грейдам

| Грейд | Что демонстрирует |
|-------|-------------------|
| **Junior** | знает компоненты (LB, БД, кэш), но не собирает целостный дизайн; нет чисел/trade-offs |
| **Mid** | полный рабочий дизайн, базовые trade-offs, нужны наводящие вопросы |
| **Senior** | **ведёт сам**, квантифицирует, deep dive, trade-offs вслух, failure modes, проактивен |
| **Staff** | нестандартные ограничения, **несколько жизнеспособных архитектур**, cost/эволюция/организация, лидирует обсуждение |

> **Рубрика** объясняет, что отличает уровни. **Junior** знает кубики, но не складывает систему. **Mid** даёт полный дизайн с базовыми trade-offs, но реактивен (ждёт вопросов). **Senior** **сам ведёт**, обосновывает числами, проактивно углубляется, проговаривает trade-offs и **failure modes**, видит узкие места. **Staff** добавляет нестандартные ограничения, **несколько жизнеспособных вариантов** с обоснованным выбором, мыслит cost/эволюцией/организацией и **лидирует** дискуссию. Разница mid→senior — не «больше компонентов», а **проактивность, квантификация, trade-offs и глубина**.

## 5. Сигналы интервьюера и восстановление

**Сигналы, которые надо подавать**:

| Сигнал | Как продемонстрировать |
|--------|------------------------|
| Структурное мышление | явно идти по 8 шагам, называть фазы |
| Requirements first | уточнять scope/NFR до решения |
| Квантификация | estimation числами, обосновывать масштаб |
| Trade-offs | называть альтернативы и компромисс вслух |
| Глубина | уйти в 1–2 компонента детально |
| Реакция на подсказки | слышать намёки интервьюера, корректировать |

**Восстановление после застревания**:

```
Застряли? НЕ молчите:
  - проговорите варианты вслух («вижу два подхода: A и B...»);
  - вернитесь к структуре (на каком шаге я? что дальше по фреймворку?);
  - попросите уточнение/подсказку (это нормально, не провал);
  - зафиксируйте допущение и двигайтесь дальше.
```

> Интервьюер ищет конкретные **сигналы** — подавайте их явно: называйте фазы, уточняйте требования, считайте, проговаривайте trade-offs, углубляйтесь, **реагируйте на подсказки** (намёк интервьюера — это направление, а не ловушка). При **застревании** не молчите: проговорите варианты, вернитесь к структуре фреймворка, попросите уточнение, зафиксируйте допущение и двигайтесь. Управляемое восстановление — само по себе сильный сигнал.

## 6. Самооценка и mock-практика

```
Mock-цикл:
  1. Возьми задачу (каталог, модуль 46) → ответь на время (45–60 мин), запиши/проговори.
  2. Оцени себя по РУБРИКЕ (§4): какой грейд? где просел (нет чисел? нет deep dive?)?
  3. Найди 1–2 слабых сигнала → целенаправленно тренируй.
  4. Повтори на новой задаче того же/другого архетипа.
```

| Шаг | Зачем |
|-----|-------|
| Ответ на время | реалистичность (тайминг — частый провал) |
| Самооценка по рубрике | измеримость (грейд + слабые сигналы) |
| Точечная тренировка | улучшать конкретный сигнал, не «всё подряд» |

> **Mock-практика** делает подготовку измеримой: отвечайте на время, затем **оценивайте себя по рубрике** (§4) — какой грейд, где просел (нет квантификации? не дошли до deep dive? нет trade-offs?). Найдя 1–2 слабых сигнала, тренируйте именно их, а не «решаю больше задач вслепую». Записывайте ответы — это вскрывает молчание, плохой тайминг и пропущенные фазы. Связка «модуль 46 (метод) + рубрика (оценка)» = целенаправленная подготовка.

## ⚠️ Подводные камни

- **Молчание.** Интервьюер не видит мысль; думать вслух (сигнал structured thinking).
- **Нет структуры.** Хаотичный ответ читается как junior; идти по 8 шагам явно.
- **Утонуть в требованиях/high-level.** Не остаётся времени на deep dive (главный senior-сигнал); держать тайминг.
- **Over-engineering.** Лишняя сложность без обоснования — анти-сигнал; решать под реальные NFR.
- **Игнор подсказок интервьюера.** Намёк — это направление; реагировать, а не упираться.
- **Нет trade-offs.** Один вариант без альтернатив — потолок mid; называть компромиссы вслух.
- **Нет чисел.** Без estimation дизайн неубедителен; квантифицировать.
- **Паника при застревании.** Молчание/сдача; проговорить варианты, вернуться к структуре, попросить уточнение.
