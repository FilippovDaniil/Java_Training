# Задача 04 — Модуль 48: Virtual waiting room

## ЗАДАНИЕ
Спроектируйте виртуальную очередь (waiting room) для сглаживания flash-sale спайка.

## ВХОДНЫЕ ДАННЫЕ
Миллионы пользователей бьют в систему в момент старта продаж. Если пустить их прямо в инвентарь, БД захлебнётся в контеншн.

## ТРЕБОВАНИЯ
- Опишите virtual waiting room: входящие → очередь (модуль 15) → позиция/токен.
- Покажите, как система впускает к покупке ограниченный поток (сколько держит инвентарь).
- Обеспечьте fairness (честная позиция, не «кто агрессивнее ретраит»).
- Добавьте rate limiting на входе (16) и кэш/CDN для статики каталога (9/10).

## ФОРМАТ РЕЗУЛЬТАТА
Waiting room (очередь+токен) → ограниченный впуск → fairness → rate limit + кэш/CDN.

## КРИТЕРИИ ПРИЁМКИ
- [ ] Virtual waiting room: входящие в очередь (модуль 15), выдача позиции/токена.
- [ ] Впуск к покупке ограниченным потоком (по ёмкости инвентаря).
- [ ] Fairness: честный порядок по позиции/токену, не обход агрессивными ретраями.
- [ ] Rate limiting на входе (модуль 16) + кэш/CDN каталога (модули 9/10).

## ПОДСКАЗКА
Waiting room: входящие → очередь (15) → позиция/токен → впуск ограниченным потоком (по ёмкости инвентаря). Fairness по позиции. Rate limit (16) + кэш/CDN (9/10).
