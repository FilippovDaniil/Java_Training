## Ключ (эталонные ответы)

1. Проектирование нового домена с нуля как синтез всего курса (метод + блоки + системы). Нет эталонного решения; capstone демонстрирует владение методом на незнакомой задаче.
2. Он смешивает сразу несколько архетипов: correctness-critical (нет оверселлинга), read-heavy (каталог), экстремальную контеншн (flash-sale), payment и search — то есть требует синтеза блоков из разных частей.
3. FR: просмотр/поиск событий, выбор мест, hold/резерв, оплата, подтверждение. NFR: нет оверселлинга (correctness), выдержать flash-sale спайк, read-heavy просмотр, fairness очереди, доступность.
4. Контеншн и оверселлинг: тысячи людей претендуют на одно место одновременно, и его нельзя продать дважды — при экстремальном спайке нагрузки.
5. available → held (с TTL) → booked (оплата прошла) или released (TTL истёк/отмена) → снова available.
6. Атомарным захватом места (locking / compare-and-set / SELECT FOR UPDATE) — гонку тысяч выигрывает один; плюс идемпотентность, чтобы повтор не создал две брони.
7. Чтобы зарезервировать место на время оплаты, но не навечно: при истечении TTL место авто-освобождается, иначе брошенные корзины заморозили бы весь зал.
8. Оверселлинг (продать больше мест, чем есть) недопустим — это реальный провал. Нужна strong consistency/CP (как в payment), а не eventual.
9. Спайк, а не средний трафик: в момент старта продаж популярного события миллионы пользователей бьют одновременно — пик на порядки выше среднего (экстремальный peak factor).
10. Виртуальная очередь: входящие попадают в очередь, получают позицию/токен, а система впускает к покупке ограниченный поток (по ёмкости инвентаря). Сглаживает спайк до посильного уровня.
11. Честной позицией/токеном в очереди — порядок определяется временем входа, а не тем, кто агрессивнее ретраит; rate limiting отсекает ботов.
12. Rate limiting (модуль 16) защищает вход от перегруза/ботов; кэш/CDN (9/10) отдают статику каталога, поглощая read-heavy спайк просмотра перед on-sale.
13. После hold (TTL) пользователь оплачивает; успех → booked, неоплата/таймаут → release места. Платёж переиспользует решения модуля 40.
14. Платёж может повторяться (retry, ненадёжная сеть); idempotency key (модули 40/23) не даёт списать дважды (double-charge).
15. Бронь+оплата — распределённая транзакция; при сбое оплаты Saga выполняет компенсацию — освобождает место (release), приводя систему в согласованное состояние (модуль 23).
16. Просмотр каталога — через кэш + CDN (9/10), поиск событий — через inverted index (модуль 17); это отдельный read-heavy слой.
17. У них разные профили: просмотр — read-heavy и кэшируемый, покупка — correctness-критичная с контеншн. Разделение позволяет масштабировать и оптимизировать каждый независимо.
18. Просмотр→кэш+CDN (9/10); поиск→inverted index (17); спайк→очередь+rate limit (15/16); инвентарь→ACID/locking/идемпотентность (21/23); оплата→ledger/Saga/idempotency (40/23); надёжность→24/25/13; эксплуатация→28/31. Всё переиспользовано из курса.
19. Hot event (один популярный концерт), контеншн на популярных местах, оверселлинг vs availability, fairness waiting room, перегрузка инвентаря спайком.
20. Шардированием по событию (13), virtual waiting room (15) и кэшированием каталога (9) — изолируют и сглаживают пиковый спрос на популярное событие.
21. При конфликте система выбирает корректность: лучше отказать/задержать (CP), чем продать несуществующее место. Availability приносится в жертву ради недопустимости оверселлинга (модуль 21).
22. Надёжность: redundancy/устранение SPOF (24), multi-region/DR (25), шард по событию (13). Безопасность: PCI DSS/PII платежа (40/30), anti-bot/rate limit (16/30). Эксплуатация: observability/SLO (28), деплой canary/blue-green (31).
23. Выбрать новый домен → уточнить требования и определить архетипы (46) → estimation (5–7) → собрать HLD из блоков по карте → углубиться в 1–2 компонента → разобрать узкие места/trade-offs → добавить надёжность/безопасность/эксплуатацию → задокументировать (C4/ADR — 02).
24. Уточнить требования, определить доминирующие архетипы и свести задачу к ним (модуль 46), затем подобрать блоки по карте «архетип → блоки» — даже для невиданного домена.
25. Не знание одной системы, а владение методом: способность взять незнакомый домен, классифицировать его, собрать полный HLD из проверенных блоков, защитить correctness-ядро, выдержать нагрузку и обосновать trade-offs — то, что отличает senior'а.
