# Модуль 48 — Capstone: дизайн нового домена с нуля (capstone)

## Зачем этот модуль

Capstone — выпускной модуль: спроектировать **новый домен, которого нет в Части 8**, с нуля, синтезируя **весь курс** — от requirements и estimation до строительных блоков, распределённых систем, надёжности, безопасности и эксплуатации. Здесь нет эталонного решения; цель — показать, что вы **владеете методом** (модуль 46) и умеете собрать систему из проверенных блоков под незнакомую задачу. В качестве worked-примера берём **систему онлайн-бронирования билетов (flash-sale ticketing)** — домен, который смешивает сразу несколько архетипов и потому идеально проверяет синтез.

Зачем это senior'у. Реальная работа — это постоянный «capstone»: новый домен, нет готового шаблона. Этот модуль тренирует **сведе́ние нового к известному** (архетипы → блоки) и сборку полного HLD под жёсткие, конфликтующие требования (нет оверселлинга **и** выдержать спайк миллионов). Это финальная проверка инженерной зрелости.

Он демонстрирует полный capstone на ticketing и даёт методологию для вашего собственного.

---

## 1. Что такое capstone и почему ticketing

```
Capstone = НОВЫЙ домен с нуля + синтез ВСЕХ частей курса.
  Нет эталона → демонстрация мастерства: метод (46) + блоки (Части 3–7) + системы (Часть 8).

Worked-домен: онлайн-бронирование билетов (flash-sale ticketing, концерты/события).
  Почему отличный capstone — СМЕШИВАЕТ архетипы:
    correctness-critical (нет оверселлинга мест) + read-heavy (просмотр каталога)
    + ЭКСТРЕМАЛЬНАЯ контеншн (flash-sale: миллионы в момент старта продаж)
    + payment + поиск.
```

| Архетип в ticketing | Что приносит |
|---------------------|--------------|
| Correctness-critical | инвентарь мест без двойного бронирования |
| Read-heavy | просмотр событий/каталога |
| Экстремальная контеншн | flash-sale спайк (on-sale moment) |
| Payment | оплата брони |
| Search | поиск событий |

> **Capstone** — проектирование нового домена с нуля как синтез всего курса. Ticketing выбран потому, что он **не сводится к одному архетипу**: это одновременно correctness-critical (оверселлинг недопустим), read-heavy (каталог), задача **экстремальной контеншн** (flash-sale — миллионы одновременно в секунду старта продаж), плюс payment и search. Спроектировать его — значит **собрать блоки из разных частей** в одну систему под конфликтующие требования. Это и есть проверка зрелости (модуль 46: новый домен → архетипы → блоки).

## 2. Требования и estimation

| Функциональные (FR) | Нефункциональные (NFR) |
|---------------------|------------------------|
| Просмотр событий, поиск | **Нет оверселлинга** (correctness) |
| Выбор мест | Выдержать flash-sale спайк (млн одновременно) |
| Hold/резерв места (временный) | Read-heavy просмотр, низкая latency |
| Оплата, подтверждение брони | Честность очереди (fairness), доступность |

```
Estimation: ключевое — СПАЙК, не среднее.
  Обычный день: умеренный трафик просмотра.
  On-sale moment (популярный концерт): МИЛЛИОНЫ пользователей за секунды → пик ≫ среднего на ПОРЯДКИ.
  → проектировать под спайк (§4), не под средний RPS.
```

> Estimation у ticketing нетипичен: доминирует **спайк**, а не средний трафик. В момент старта продаж популярного события **миллионы** пользователей бьют одновременно — пик на порядки выше среднего (модуль 5, peak factor здесь экстремальный). Главное NFR — **нет оверселлинга** (продать больше мест, чем есть, — провал; correctness-critical, модуль 40/21). Это сразу задаёт два конфликтующих фронта: **корректность инвентаря** (§3) и **выживание под спайком** (§4).

## 3. Ключевой вызов: контеншн и оверселлинг

```
Тысячи людей хотят ОДНО место одновременно. Нельзя продать его дважды.

Жизненный цикл места (state machine):
  AVAILABLE → HELD (TTL, напр. 5–10 мин) → BOOKED   (оплата прошла)
                    ↘ released (TTL истёк / отмена) → AVAILABLE

Механика:
  - атомарный захват места (locking / compare-and-set / SELECT FOR UPDATE) — один выигрывает;
  - hold с TTL: место бронируется на время оплаты, не «навечно» (иначе зависнет);
  - идемпотентность (модуль 23): повтор запроса не создаёт две брони;
  - strong consistency на инвентаре (модуль 21): оверселлинг недопустим → CP, не eventual.
```

| Механизм | Роль | Модуль |
|----------|------|--------|
| Атомарный захват места | один покупатель на место (нет double-booking) | 21, 23 |
| Hold с TTL | резерв на время оплаты, авто-release | — |
| Идемпотентность | повтор не создаёт две брони | 23 |
| Strong consistency инвентаря | оверселлинг недопустим → CP | 21 |

> **Сердце capstone** — контеншн на инвентаре. Место проходит **state machine** (available → held с TTL → booked, либо release по таймауту). Захват места **атомарен** (locking/CAS/SELECT FOR UPDATE — один выигрывает гонку тысяч), **hold ограничен TTL** (резерв на время оплаты, иначе брошенные корзины заморозят зал), операции **идемпотентны** (повтор не двоит бронь — модуль 23), а инвентарь требует **strong consistency** (оверселлинг = реальный провал → CP-выбор, не eventual — модуль 21, как в payment модуле 40). Это correctness-ядро, вокруг которого строится всё.

## 4. Сглаживание спайка: virtual waiting room

```
Проблема: млн запросов в секунду старта продаж убьют инвентарь-БД (контеншн + перегруз).

Решение — VIRTUAL WAITING ROOM (виртуальная очередь):
  все входящие → попадают в очередь (модуль 15) → получают позицию/токен →
  система впускает к покупке ОГРАНИЧЕННЫЙ поток (столько, сколько инвентарь держит) →
  остальные ждут с честной позицией (fairness).

  + rate limiting на входе (модуль 16); + кэш/CDN для статики каталога (9/10).
```

| Приём | Роль | Модуль |
|-------|------|--------|
| Virtual waiting room (очередь) | сгладить спайк до посильного потока | 15 |
| Токен/позиция + fairness | честный порядок, не «кто быстрее задолбит» | — |
| Rate limiting на входе | защита от перегруза/ботов | 16 |
| Кэш/CDN каталога | разгрузить read-heavy просмотр | 9, 10 |

> Спайк миллионов нельзя пустить прямо в инвентарь — он захлебнётся в контеншн. **Virtual waiting room** (виртуальная очередь — модуль 15) принимает всех, выдаёт **позицию/токен** и **впускает к покупке ограниченный поток**, который инвентарь способен обслужить; остальные честно ждут (**fairness** — не «кто агрессивнее ретраит»). На входе — **rate limiting** (модуль 16, против ботов/перегруза), статика каталога — из **кэша/CDN** (модули 9/10). Так write-heavy спайк превращается в управляемый поток, а correctness-ядро (§3) работает в посильном режиме.

## 5. Карта синтеза: блоки со всего курса

| Подсистема | Блоки | Часть/модуль |
|-----------|-------|--------------|
| Просмотр каталога | кэш + CDN | 9, 10 |
| Поиск событий | inverted index | 17 |
| Сглаживание спайка | очередь (waiting room) + rate limit | 15, 16 |
| Инвентарь мест | ACID/strong consistency + locking + идемпотентность | 21, 23 |
| Оплата | ledger + Saga/Outbox + idempotency | 40, 23 |
| Estimation/cost | load/capacity/cost | 5–7 |
| Надёжность/масштаб | redundancy, multi-region, шардирование по событию | 24, 25, 13 |
| Эксплуатация | observability (SLO), деплой (canary) | 28, 31 |

> Capstone — это **карта синтеза**: каждая подсистема собирается из готовых блоков Частей 3–7, а payment-ядро — из решений модуля 40. Просмотр → кэш+CDN (read-heavy архетип); поиск → inverted index; спайк → очередь+rate limit; инвентарь → ACID+locking+идемпотентность (correctness); оплата → ledger+Saga+idempotency; сверху — estimation/cost (5–7), надёжность/multi-region (24/25), шардирование по событию (13), observability и деплой (28/31). Ничего не изобретается заново — **всё переиспользуется** (модуль 46).

## 6. High-level, deep dive, узкие места

```
   [Клиент] → CDN/кэш (каталог) ─┐
                                  ├→ Waiting room (очередь) → Booking service → Inventory (ACID, locking)
   [Поиск] → search service ──────┘                                │
                                                          Payment (ledger/Saga, idempotency)
   Hold(TTL) → Booked при оплате / released при таймауте.
```

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| **Hot event** (один популярный концерт) | шардирование по событию, waiting room, кэш каталога | 13, 15, 9 |
| **Контеншн на популярных местах** | атомарный захват, hold TTL, очередь | 21, 23 |
| **Оверселлинг vs availability** | CP-выбор (корректность важнее) | 21 |
| **Fairness waiting room** | честная позиция/токен, не FIFO-обход ботами | 16 |
| Спайк перегружает инвентарь | waiting room ограничивает поток | 15 |

> High-level: каталог отдаётся из **CDN/кэша**, поиск — отдельным сервисом; покупка идёт через **waiting room** → **booking service** → **inventory** (ACID, locking) → **payment** (ledger/Saga). **Deep dive** — жизненный цикл места (available→held(TTL)→booked/released) и waiting-room токен. Узкие места: **hot event** (шардирование по событию, очередь, кэш), контеншн (атомарный захват + hold TTL), **оверселлинг vs availability** (выбираем CP — корректность важнее), fairness очереди. Это синтез всех осей курса в одной системе.

## 7. Как делать свой capstone

```
Методология собственного capstone:
  1. Выбери НОВЫЙ домен (не из Части 8): e-commerce checkout, collaborative editor,
     stock exchange, online judge, IoT-платформа, food delivery...
  2. Уточни требования (FR/NFR), определи доминирующие АРХЕТИПЫ (модуль 46).
  3. Сделай estimation (модули 5–7) — найди главную ось.
  4. Собери HLD из БЛОКОВ по карте (архетип → блоки), углубись в 1–2 критичных.
  5. Разбери узкие места, trade-offs, надёжность/безопасность/эксплуатацию.
  6. Задокументируй (C4 + ADR + sequence — модуль 02).
```

> Ваш capstone строится тем же методом. Выберите **новый домен** (не из Части 8), уточните требования, определите **архетипы** (модуль 46), сделайте estimation, соберите HLD **из блоков по карте**, углубитесь в 1–2 критичных компонента, разберите узкие места и trade-offs, добавьте надёжность/безопасность/эксплуатацию и **задокументируйте** (C4/ADR/sequence — модуль 02). Хороший capstone демонстрирует не знание одной системы, а **владение методом** на незнакомой задаче — именно это отличает senior'а.

## ⚠️ Подводные камни

- **Игнор спайка, проектирование под средний RPS.** Flash-sale — это пик на порядки; waiting room + estimation под спайк (15/5).
- **Eventual consistency на инвентаре.** Оверселлинг недопустим → strong consistency/CP (21), как в payment.
- **Hold без TTL.** Брошенные корзины замораживают места навсегда; резерв с TTL + авто-release.
- **Нет атомарного захвата.** Гонка тысяч за место → double-booking; locking/CAS/идемпотентность (21/23).
- **Пустить спайк прямо в инвентарь.** БД захлебнётся в контеншн; virtual waiting room сглаживает поток (15).
- **Нечестная очередь.** Боты/агрессивные ретраи обходят ожидающих; fairness через позицию/токен + rate limit (16).
- **Изобретать блоки заново.** Capstone = синтез готовых блоков по карте (модуль 46), не изобретение.
- **Capstone без документации.** Дизайн без C4/ADR/sequence нечитаем; документировать (модуль 02).
- **Не свести новый домен к архетипам.** Без классификации проектируешь вслепую; архетипы → блоки (модуль 46).
