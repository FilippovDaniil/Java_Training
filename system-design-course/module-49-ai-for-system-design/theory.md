# Модуль 49 — Бонус: ИИ в System Design (ai-for-system-design)

## Зачем этот модуль

Финальный модуль курса — про то, как **ИИ встраивается в системный дизайн** с двух сторон: (A) вы **проектируете системы, обслуживающие ML/LLM** (inference serving, RAG, vector DB, feature store), и (B) **ИИ помогает вам проектировать** (brainstorm, ревью, генерация ADR/диаграмм). Ключевая мысль: ИИ-компонент **не отменяет фундамент** курса — он добавляет новые блоки и новую дорогую ось (GPU cost), но всё остальное (estimation, кэш, очереди, надёжность) работает как прежде. RAG-ретривер — это всё тот же поиск (модуль 17) плюс vector DB; inference serving — всё тот же сервис под нагрузкой с латентностью и стоимостью.

Зачем это senior'у. ИИ-фичи стали обычной частью продуктов, и senior должен уметь **спроектировать их как нормальную распределённую систему**: где GPU-узкое место, как батчить inference, зачем RAG, как считать стоимость токенов, чем semantic search отличается от keyword. И — пользоваться ИИ как помощником, **проверяя** его (галлюцинации).

Он даёт оба угла: проектирование ИИ-обслуживающих систем и ИИ как инструмент дизайна.

## 1. Два угла: ИИ в System Design

```
(A) Проектировать системы, ОБСЛУЖИВАЮЩИЕ ИИ:
    inference serving, RAG, vector DB, feature store — ML/LLM как компонент системы.

(B) ИИ как ПОМОЩНИК в дизайне:
    brainstorm вариантов, ревью дизайна, генерация ADR/диаграмм — с проверкой (галлюцинации).
```

| Угол | Суть |
|------|------|
| (A) Системы для ИИ | ML/LLM-компонент проектируется как обычная распределённая система + новые блоки |
| (B) ИИ для дизайна | LLM ускоряет дизайн-работу, но требует верификации |

> Два угла. **(A)** — вы **встраиваете ML/LLM в систему**: модель за сервисом, RAG-конвейер, vector DB, feature store. Это проектируется **теми же методами**, что весь курс (requirements, estimation, кэш, очереди, надёжность), плюс новые блоки и новая ось — GPU cost. **(B)** — вы **используете ИИ как помощника** в самом дизайне (накидать варианты, отревьюить, сгенерировать ADR/диаграмму), но обязательно **проверяете** (LLM галлюцинирует). Главное: ИИ — это **дополнение к фундаменту**, а не его замена.

## 2. Inference serving (обслуживание модели)

```
Модель за сервисом, отвечающим на запросы:
  online inference (real-time, низкая latency на запрос) vs batch inference (офлайн, пропускная).
  GPU-узкое место → БАТЧИНГ запросов (несколько запросов в один проход GPU) ↑ throughput.
  autoscaling под нагрузку; модель — тяжёлая (память/прогрев) → не масштабируется мгновенно.
```

| Аспект | Решение |
|--------|---------|
| Online vs batch | real-time (latency) vs офлайн (пропускная) |
| GPU-узкое место | батчинг запросов (dynamic batching) ↑ throughput |
| Масштабирование | autoscaling, но прогрев модели медленный |
| Latency | модель тяжёлая → latency выше обычного сервиса |

> **Inference serving** — выдача предсказаний модели под нагрузкой. Различают **online** (real-time, важна latency на запрос) и **batch** (офлайн, важна пропускная). Главное узкое место — **GPU**: чтобы поднять throughput, запросы **батчат** (несколько в один проход GPU — dynamic batching), балансируя latency и пропускную. Масштабирование осложнено тем, что **модель тяжёлая** (память, медленный прогрев) — не масштабируется мгновенно как stateless-сервис. По сути это знакомый сервис под нагрузкой (модули 8/24/26), но с GPU как дефицитным ресурсом.

## 3. RAG (retrieval-augmented generation)

```
RAG = заземлить LLM на ВАШИХ данных без переобучения:
  1. документы → embeddings (векторы) → vector DB (индекс).
  2. запрос → embedding → ПОИСК похожих (retrieve) в vector DB → релевантный контекст.
  3. контекст + запрос → LLM → ответ, опирающийся на данные.

Зачем: снизить галлюцинации, дать свежие/приватные знания, без дорогого переобучения модели.
```

| Шаг | Что |
|-----|-----|
| Индексация | документы → embeddings → vector DB |
| Retrieval | запрос → embedding → поиск похожих векторов → контекст |
| Generation | LLM генерирует ответ по контексту |

> **RAG** — самый частый паттерн LLM-систем: вместо переобучения модели на ваших данных вы **извлекаете релевантный контекст** и подаёте его в промпт. Документы превращаются в **embeddings** (векторы) и кладутся в **vector DB**; на запрос ищутся похожие векторы (retrieve), найденный контекст идёт в LLM (generation). Зачем: **снизить галлюцинации** (ответ опирается на реальные документы), дать **свежие/приватные знания** без переобучения. Архитектурно это конвейер «поиск (модуль 17, но семантический) → LLM».

## 4. Vector DB и semantic search

```
Keyword search (модуль 17): inverted index, точное совпадение термов.
Semantic search: embeddings → близость по СМЫСЛУ (косинусное расстояние векторов).

Vector DB: хранит векторы + ANN (approximate nearest neighbor) поиск:
  точный k-NN по млрд векторов дорог → ПРИБЛИЖЁННЫЙ поиск (HNSW и др.) — быстро, чуть неточно.
```

| | Keyword (17) | Semantic (vector) |
|--|--------------|-------------------|
| Основа | inverted index, термы | embeddings, близость смысла |
| Совпадение | точное по словам | по смыслу (синонимы, перефраз) |
| Поиск | точный | ANN (HNSW) — приближённый, быстрый |

> **Vector DB** — хранилище embeddings с поиском по **смысловой близости** (а не по словам, как inverted index модуля 17). Запрос-вектор сравнивается с векторами документов (косинусное расстояние). Точный k-NN по миллиардам векторов дорог → используют **ANN** (approximate nearest neighbor, напр. **HNSW**) — приближённый поиск, жертвующий капелькой точности ради скорости (тот же трюк «приближённость vs точность», что и в leaderboard/метриках). Semantic search ловит синонимы и перефразировки, недоступные keyword-поиску.

## 5. Feature store и ML-данные

```
Feature store: централизованное хранилище ML-фич:
  offline (для обучения, батч, история) + online (для inference, низкая latency).
  Проблема — training-serving skew: фича в обучении ≠ фича в проде → деградация модели.
  → единый источник фич для обучения и inference.
```

| Аспект | Роль |
|--------|------|
| Offline store | фичи для обучения (батч, история) |
| Online store | фичи для inference (низкая latency) |
| Training-serving skew | расхождение фич обучение↔прод → деградация; единый источник |

> **Feature store** решает проблему ML-данных: модели нужны **фичи** и при обучении (offline, батч, история — связь с конвейерами модуля 20), и при inference (online, низкая latency). Главная ловушка — **training-serving skew**: если фича считается по-разному в обучении и в проде, модель деградирует незаметно. Feature store даёт **единый источник** фич для обоих путей. Это специализированный случай разделения OLTP/OLAP (модуль 18) и пайплайнов (модуль 20) под ML.

## 6. LLM-специфика в дизайне

| Особенность | Следствие для дизайна |
|-------------|----------------------|
| Генерация медленная | **token streaming** (отдавать по токенам, не ждать весь ответ) |
| Стоимость per-token | cost растёт с длиной prompt+ответа → **GPU/token cost — новая ось** (модуль 7) |
| Лимит context window | нельзя влить всё → RAG/суммаризация/чанкинг |
| Prompt caching | кэшировать повторяющийся префикс промпта → дешевле/быстрее |
| Недетерминизм/галлюцинации | guardrails, валидация вывода, eval |
| Зависимость от внешней модели | fallback при недоступности (модуль 23), timeout, circuit breaker |

> LLM приносит **новые design-факторы**. Генерация медленная → **token streaming** (отдавать ответ по мере генерации, как UX-паттерн). Стоимость считается **за токены** → длина промпта/ответа = деньги, а **GPU/inference cost — новая дорогая ось** (модуль 7). **Context window** ограничен → нельзя подать всё, отсюда RAG/чанкинг/суммаризация. **Prompt caching** удешевляет повторяющиеся префиксы. Вывод недетерминирован → **guardrails** и валидация. Зависимость от модели/провайдера → **fallback/timeout/circuit breaker** (модуль 23), как с любой внешней зависимостью.

## 7. ИИ как помощник в дизайне и новые узкие места

**(B) ИИ-помощник в дизайне**:

```
LLM ускоряет дизайн-работу:
  + brainstorm вариантов архитектуры, ревью дизайна, генерация ADR/диаграмм/boilerplate.
  − КАВЕАТ: галлюцинации — LLM выдаёт правдоподобное, но неверное → ВСЕГДА проверять.
  Правило: ИИ — ускоритель и второй взгляд, не источник истины.
```

**Новые узкие места ИИ-систем**:

| Узкое место | Решение | Модуль |
|-------------|---------|--------|
| Inference latency | батчинг, кэш, меньшая модель/квантизация | 26 |
| **GPU cost** (новая дорогая ось) | батчинг, autoscaling, right-sizing, кэш ответов | 7 |
| Масштаб vector index | ANN/HNSW, шардирование векторов | 13 |
| Eval/качество | оценка ML-системы ≠ обычные тесты (метрики качества) | 27 |

> **(B)** ИИ **ускоряет дизайн**: накидать варианты, отревьюить, сгенерировать ADR/диаграмму/болерплейт — но **галлюцинации** требуют **обязательной проверки** (LLM выдаёт правдоподобное, не обязательно верное; ИИ — ускоритель и второй взгляд, не источник истины). **Новые узкие места** ИИ-систем: inference latency (батчинг/кэш/квантизация — модуль 26), **GPU cost** (новая дорогая ось — модуль 7), масштаб vector index (ANN/шардирование — 13), и **eval** (качество ML-системы оценивается метриками, а не обычными pass/fail тестами — модуль 27). Фундамент курса остаётся — ИИ добавляет блоки, а не заменяет принципы.

## ⚠️ Подводные камни

- **Считать ИИ заменой фундамента SD.** ML/LLM — это компонент; estimation, кэш, очереди, надёжность работают как прежде.
- **Игнор GPU/token cost.** Новая дорогая ось; считать стоимость inference/токенов (модуль 7).
- **Переобучать модель вместо RAG.** Для свежих/приватных знаний дешевле RAG (retrieve + контекст), чем переобучение.
- **Точный k-NN на масштабе.** Дорог; ANN/HNSW (приближённый поиск) для vector DB.
- **Игнор training-serving skew.** Расхождение фич роняет модель; единый feature store.
- **Ждать весь ответ LLM.** Медленно для UX; token streaming.
- **Влить всё в context window.** Лимит контекста; RAG/чанкинг/суммаризация.
- **Нет fallback при недоступности модели.** Внешняя зависимость; timeout/circuit breaker/fallback (модуль 23).
- **Доверять выводу ИИ-помощника без проверки.** Галлюцинации; всегда верифицировать.
- **Тестировать ML как обычный код (pass/fail).** Нужны метрики качества/eval (модуль 27).
